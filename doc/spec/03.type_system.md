## 3.1 Overview

Mizar Evolution adopts a **soft typing system** layered on untyped set theory. Types serve as syntactic guides and are not part of the logical foundation. All expressions ultimately denote sets, and type checking is implemented as generation of proof obligations in first-order logic.

> Freek Wiedijk. [Mizar's Soft Type System](https://cs.ru.nl/F.Wiedijk/mizar/miztype.pdf) TPHOLs 2007

**Design Principles**:

* **Mathematical alignment**: Types match mathematical practice.
* **Separation of concerns**: Types guide readability and verification but are erased for logical core.
* **Efficiency**: Syntax and inference mechanisms are optimized for fast checking.
* **Consistency**: All typing constructs follow unified syntax and semantics.

## 3.2 Core Concepts

| Concept | Description | Defined In |
| --- | --- | --- |
| **Radix-types** | Root types at the base of the type hierarchy: the built-ins `object` and `set`, and user-defined structures | §3.4, Ch.5 |
| **Attributes** | Predicates (possibly parameterized) that refine an existing type | Ch.6 |
| **Mode-types** | Named types introduced by `mode` declarations; each unfolds to an attribute chain applied to a radix-type | Ch.7 |
| **Structures** | Composite radix-types with named fields and properties | Ch.5 |
| **Clusters** | Registration mechanism for type inference and attribute propagation | Ch.17 |

Radix-types and mode-types form two disjoint syntactic categories that both appear at the **head** of a type expression (after the attribute chain). A mode-type is semantically a shorthand for `attribute_chain radix_type` and is unfolded to that form during type checking (§Ch.7).

## 3.3 Type Expressions

A **type expression** specifies how a value is classified.

**Syntax (EBNF)**:

```ebnf
type_expression   ::= attribute_chain type_head ;
type_head         ::= radix_type | mode_type ;
attribute_chain   ::= { [ "non" ] attribute_ref } ;
attribute_ref     ::= [ param_prefix ] [ struct_name "." ] attribute_name ;
param_prefix      ::= parameter "-" | "(" parameter_list ")" "-" ;
radix_type        ::= builtin_type | struct_name [ type_args ] ;
mode_type         ::= mode_name [ type_args ] ;
type_args         ::= ( "of" | "over" ) argument_list ;
argument_list     ::= term_expression { "," term_expression } ;
builtin_type      ::= "object" | "set" ;
```

* `struct_name "." attribute_name` is a struct-qualified attribute reference. The `.` here is interpreted as a namespace separator (Ch.2 §2.5.3, rule 4); a struct name that is also in scope as a variable is shadowed by the variable, following the usual rule.
* `parameter` and `parameter_list` are template parameters defined in Ch.18 (Templates).
* `mode_name`, `struct_name`, and `attribute_name` resolve to `user_symbol` (Ch.2 §2.5.2) entries registered by their defining chapters — Ch.7 (modes), Ch.5 (structs), and Ch.6 (attributes) respectively. A name is recognised in these positions only when it is present in the active lexicon.

**Examples**:

```mizar
set                           :: Built-in radix-type
non empty set                 :: Attribute + radix-type
positive Integer              :: Attribute + mode
Element of G                  :: Dependent mode
associative unital Magma      :: Multiple attributes + mode
Polynomial over R             :: Parameterized structure
```

## 3.4 Built-in Types

Mizar provides two built-in radix-types:

| Type | Description |
| --- | --- |
| `object` | Universal type. All values in the Mizar universe, including structures. |
| `set` | Subtype of `object`. Mathematical sets in the ZFC sense. |

**Type Hierarchy**:

```
object
├── set
│   └── (user-defined modes based on set)
└── (user-defined structures)
```

* Structures defined via `struct` are subtypes of `object` but **not** of `set`.
* This distinction allows separation of abstract algebraic structures from general sets.

## 3.5 Subtyping and Widening

`S` is a **subtype** of `T` when membership in `S` implies membership in `T`.

* **Widening**: Automatically convert a value to a supertype.
* **Narrowing**: Convert to a subtype — requires `reconsider` (Ch.15) with a proof obligation.

**Examples**:

```mizar
:: Widening (automatic)
let n be Nat;
:: n can be used where Integer is expected

:: Narrowing (explicit)
let x be Integer such that x >= 0;
reconsider x as Nat by mml.number.real.Nat_is_nonneg_Real;  :: Proof required
```

**Subtype chains** (each type is a subtype of the one on its right):

* `Prime`, then `Integer`, then `Number`
* `non empty set`, then `set`, then `object`

## 3.6 Clusters (Overview)

Clusters are registration mechanisms that enable automatic type inference and propagation.

| Cluster Type | Purpose | Example |
| --- | --- | --- |
| **Existential** | Prove existence of typed values | `cluster empty finite set;` |
| **Conditional** | Declare attribute implications | `cluster empty -> finite set;` |
| **Functor** | Assign types to functor results | `cluster f(x) -> positive for Real;` |

Detailed syntax and examples are provided in **Chapter 17 (Clusters and Registrations)**.

## 3.7 Logic Encoding Details

This section describes how types map to First-Order Logic.

### 3.7.1 Types as Predicates

Each type `T` is encoded as a unary predicate `is_T(x)`.

| Mizar Source | FOL Encoding |
| --- | --- |
| `x is set` | `is_set(x)` |
| `x is non empty set` | `is_set(x) ∧ ¬ is_empty(x)` |
| `x is Element of G` | `is_Element_of(x, G)` |

(The names `Real`, `Integer`, and similar used elsewhere in this chapter are mode-types defined in the MML, not built-ins; their FOL encodings follow the same `is_T` schema.)

### 3.7.2 Subtyping as Implication

"`S` is a subtype of `T`" is encoded as the universal implication `∀x. is_S(x) ⇒ is_T(x)`.

**Example**: "`Prime` is a subtype of `Integer`" corresponds to `∀x. is_Prime(x) ⇒ is_Integer(x)`.

### 3.7.3 Type Erasure for ATPs

When exporting to ATPs (TPTP format), types are erased:

1. Variable declarations become untyped: `let x be Real` → `x`
2. Type assumptions become hypotheses: `is_Real(x) ⇒ ...`
3. Attribute chains become conjunctions

## 3.8 Complete Syntax (EBNF)

Summary of type-related syntax from this chapter.

```ebnf
(* Type Expressions *)
type_expression   ::= attribute_chain type_head ;
type_head         ::= radix_type | mode_type ;
attribute_chain   ::= { [ "non" ] attribute_ref } ;
attribute_ref     ::= [ param_prefix ] [ struct_name "." ] attribute_name ;
param_prefix      ::= parameter "-" | "(" parameter_list ")" "-" ;

radix_type        ::= builtin_type | struct_name [ type_args ] ;
mode_type         ::= mode_name [ type_args ] ;

type_args         ::= ( "of" | "over" ) argument_list ;
argument_list     ::= term_expression { "," term_expression } ;

builtin_type      ::= "object" | "set" ;

attribute_name    ::= user_symbol ;  (* registered by Chapter 6 definitions *)
mode_name         ::= user_symbol ;  (* registered by Chapter 7 declarations *)
struct_name       ::= user_symbol ;  (* registered by Chapter 5 declarations *)

(* user_symbol: see Chapter 2 §2.5.2 *)
(* parameter, parameter_list: see Chapter 18 (Templates) *)
```
