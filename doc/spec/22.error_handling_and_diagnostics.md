This chapter specifies the diagnostic output of the Mizar Evolution verifier: the format of error messages, the error code system, the classification of errors by phase, and the guidance the verifier provides when proof obligations fail. The design goal is **actionable diagnostics**: every message should tell the author what went wrong, where, and what to try next.

## 22.1 Error Message Design Principles

### 22.1.1 Error Code System (E-series)

Every diagnostic emitted by the verifier carries a **unique, stable code**. Codes are grouped by phase and severity:

| Prefix | Severity | Phase |
|---|---|---|
| `E` | Error — verification cannot proceed | Syntax, type, proof, logical consistency |
| `W` | Warning — verification proceeds, but something is suspicious | All phases |
| `I` | Info — no problem, additional information requested | Display hints (§21.7) |

Codes have the form `Xnnnn` where `X` is the severity prefix and `nnnn` is a four-digit decimal number. Numbers are assigned permanently; a code is never reused for a different diagnostic after it is assigned. When a diagnostic is removed from the verifier, its code is retired and the number is never reissued.

Each code also has a **semantic name** in dot-separated category notation. The semantic name may change across versions (e.g., to fix a typo or improve clarity); the numeric code is the stable identifier. Tools should key on the numeric code; humans may use either form.

```
error[E0101]: type mismatch (type.mismatch)
  --> src/mylib.miz:14:16
   |
14 |    var n := sqrt(2) as Integer;
   |             ^^^^^^^ inferred type: Real; expected: Integer
   |
help: use a narrowing cast with justification: `sqrt(2) as Integer by ...`
```

Code ranges by phase:

| Range | Phase |
|---|---|
| E0001–E0099 | Lexical / syntax errors |
| E0100–E0199 | Type errors |
| E0200–E0299 | Overload resolution and template errors |
| E0300–E0399 | Proof and ATP errors |
| E0400–E0499 | Logical inconsistency errors |
| E0500–E0599 | Algorithm verification errors |
| W0001–W0099 | Structural warnings |
| W0100–W0199 | Proof and ATP warnings |
| W0200–W0299 | Algorithm and contract warnings |
| I0001–I0099 | Informational display (§21.7) |

### 22.1.2 Source Span and Context Display

Every diagnostic identifies its source location with a **span**: a half-open range `[start, end)` in the source file, expressed as line and column numbers (both 1-based).

The display format is:

```
severity[CODE]: short description (semantic.name)
  --> path/to/file.miz:line:col
   |
LL | source text of the relevant line
   | ^^^^^^ annotation under the span
   |
   = note: additional context
```

Rules:
- The `-->` line gives the file path relative to the workspace root (§23.3), the line number, and the column of the **start** of the primary span.
- At most one **primary span** is shown with `^` underlines. Secondary spans (e.g., the definition site of a conflicting overload) are shown with `-` underlines on separate `|` blocks.
- If the span covers multiple lines, the display shows the first and last lines with `...` in between.
- All column numbers are counted in Unicode scalar values, not bytes.

**Secondary spans** cross-reference related source locations:

```
error[E0201]: ambiguous overload (resolve.ambiguous_symbol)
  --> src/algebra.miz:42:5
   |
42 |   a + b
   |   ^^^^^ two candidates match
   |
  --> src/ring.miz:7:3
   |
 7 |   func add(a, b: Element of R) -> Element of R;
   |   ---- candidate 1
   |
  --> src/nat.miz:11:3
   |
11 |   func add(a, b: Nat) -> Nat;
   |   ---- candidate 2
   |
help: use a qualified name or `qua` to disambiguate
```

### 22.1.3 Fix Suggestions and `help:` Annotations

When the verifier can determine a likely correction, it appends one or more `help:` lines to the diagnostic. Fix suggestions are classified by confidence:

| Keyword | Meaning |
|---|---|
| `help:` | High-confidence suggestion; applying it is likely correct. |
| `note:` | Contextual information without a specific fix. |
| `hint:` | Low-confidence suggestion; one possibility among several. |

Fix suggestions are machine-readable when they follow the pattern:

```
help: replace `old text` with `new text`
help: insert `text` after line LL
help: remove the declaration at line LL
```

IDE code actions (§23.6.1) parse these patterns to offer one-click fixes. When the fix cannot be expressed as a simple text operation, the `help:` line gives a natural-language description instead.

```
error[E0102]: narrowing cast requires proof (type.narrowing_requires_proof)
  --> src/mylib.miz:7:15
   |
 7 |   var n := k as Nat;
   |                 ^^^ Integer cannot be narrowed to Nat without proof
   |
help: add a justification: `var n := k as Nat by non_negative_lemma;`
note: alternatively, supply a proof block:
      `var n := k as Nat
       proof
         k >= 0 by ...;
         thus k is Nat;
       end;`
```

## 22.2 Syntax Errors

Syntax errors are detected during parsing, before type-checking or proof verification begins. They carry codes in the E0001–E0099 range.

### 22.2.1 Malformed Tokens and Unexpected Symbols

**E0001 — `syntax.unexpected_token`**. The parser encountered a token that cannot start or continue any syntactic construct in the current context.

```
error[E0001]: unexpected token `:=` (syntax.unexpected_token)
  --> src/lib.miz:5:12
   |
 5 |   theorem := foo;
   |           ^^ expected an identifier after `theorem`
```

**E0002 — `syntax.malformed_literal`**. A numeric or string literal does not conform to the lexical rules of §2.3.

```
error[E0002]: malformed numeric literal (syntax.malformed_literal)
  --> src/lib.miz:3:10
   |
 3 |   const x = 1_000_000_;
   |             ^^^^^^^^^^^ trailing underscore in integer literal
```

**E0003 — `syntax.unexpected_end_of_file`**. The source file ended while an open construct was still pending.

```
error[E0003]: unexpected end of file (syntax.unexpected_end_of_file)
  --> src/lib.miz:20:1
   |
20 | (end of file)
   | ^ expected `end` to close `definition` opened at line 3
```

### 22.2.2 Structural Errors (Missing `end`, Unmatched Delimiters)

**E0010 — `syntax.missing_end`**. A `definition`, `proof`, `do...end`, or similar block was opened but not closed.

```
error[E0010]: missing `end` for `proof` block (syntax.missing_end)
  --> src/lib.miz:18:1
   |
 8 |   proof
   |   ----- `proof` opened here
   |
18 |   theorem next_theorem : ...;
   |   ^^^^^^^ expected `end` before new top-level item
```

**E0011 — `syntax.unmatched_delimiter`**. A parenthesis, bracket, or `do` keyword has no matching close.

```
error[E0011]: unmatched `(` (syntax.unmatched_delimiter)
  --> src/lib.miz:12:8
   |
12 |   func f(a, b -> Nat;
   |          ^ no matching `)`
```

**E0012 — `syntax.reserved_keyword_as_identifier`**. A reserved keyword (Appendix C) is used where an identifier is expected.

> [!NOTE]
> The parser applies **error recovery** after every syntax error: it attempts to skip tokens until it finds a synchronisation point (typically `;`, `end`, or a top-level keyword) and then resumes parsing. This allows a single compilation run to report multiple independent syntax errors rather than stopping at the first one.

## 22.3 Type Errors

Type errors are detected during type-checking and type inference (Chapters 3 and 8). They carry codes in the E0100–E0199 range.

### 22.3.1 Type Mismatch

**E0101 — `type.mismatch`**. An expression of type `T` appears where type `U` is required, and `T` is not a subtype of `U` and no coercion path exists. Unlike E0102 (narrowing cast present but lacking proof), this error fires when the types are entirely unrelated — no cast can bridge them.

```
error[E0101]: type mismatch (type.mismatch)
  --> src/lib.miz:9:7
   |
 9 |   var n as Plane := some_line;
   |            ^^^^^ ^^^^^^^^^^^ inferred type: Line; required: Plane; no coercion path registered
   |
note: `Line` and `Plane` are unrelated modes; no widening, narrowing, or coercion applies
```

**E0102 — `type.narrowing_requires_proof`**. A narrowing coercion (e.g., `Integer` to `Nat`) is written without a justification. Widening coercions (e.g., `Nat` to `Integer`) are accepted silently; narrowing coercions require either a `by` justification or a `proof ... end` block (§20.1.3).

**E0103 — `type.sethood.missing`**. A Fraenkel set comprehension `{ f(x) where x is T : P(x) }` is used, but `sethood` is not registered for type `T` (§21.7.6). The comprehension is rejected at type-check time.

```
error[E0103]: `sethood` not registered for type `MyType` (type.sethood.missing)
  --> src/lib.miz:22:7
   |
22 |   const S = { f(x) where x is MyType : P(x) };
   |              ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ Fraenkel comprehension requires sethood
   |
help: add a `sethood` proof to the mode definition of `MyType`:
      `definition
         mode MyType is ...;
         sethood
         proof
           :: ex S being set st for x being MyType holds x in S
           ...
         end;
       end;`
```

### 22.3.2 Type Inference Failure

**E0110 — `type.inference_conflict`**. Two constraints derived from different parts of an expression require incompatible types. This arises in template/schema inference when the same type parameter is unified with two incompatible types — neither of which is a subtype of the other — so there is no single "wrong" expression site (contrast E0101, which fires at a specific mismatched term).

```
error[E0110]: type inference conflict (type.inference_conflict)
  --> src/lib.miz:8:11
   |
 8 |   const z = combine(some_list, some_nat);
   |             ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ type parameter `T` inferred as both `List of Nat` and `Nat`; no common supertype
   |
   = note: `T` = `List of Nat` (from argument 1 at col 19)
   = note: `T` = `Nat`         (from argument 2 at col 30)
   |
help: supply an explicit type for one or both arguments
```

> [!NOTE]
> E0110 is distinct from E0102: it fires when the inference engine derives contradictory constraints with no narrowing-cast resolution path, not when a cast is present but lacks a proof.
>
> Cases where omitted template arguments cannot be determined from the available argument types are reported as E0203 (`template.argument_omitted_not_inferable`) in §22.3.4, rather than as a separate general-purpose type inference failure.

### 22.3.3 Mode and Attribute Violations

**E0120 — `type.mode_mismatch`**. A term of mode `M` is used where mode `N` is required, and no registered mode widening applies (§7.4).

```
error[E0120]: mode mismatch (type.mode_mismatch)
  --> src/lib.miz:12:22
   |
12 |   const r = ordinal_add(3, omega);
   |                          ^ mode `Nat`; required: `Ordinal`; no registered widening from `Nat` to `Ordinal`
   |
help: register a mode widening: `registration cluster Nat -> Ordinal; end;`
note: alternatively, introduce an explicit coercion at the call site
```

**E0121 — `type.attribute_required`**. An operation requires its argument to carry a specific attribute (e.g., `non zero`, `finite`), but the attribute is not registered for the supplied type. The verifier checks whether a cluster registration could discharge the attribute before emitting this error; it is only emitted when cluster resolution definitively fails. This is distinct from E0102: E0102 concerns subtype narrowing (e.g., `Integer → Nat`), whereas E0121 concerns a missing attribute on an otherwise correct type.

```
error[E0121]: attribute `non zero` required (type.attribute_required)
  --> src/lib.miz:15:16
   |
15 |   const q = a / b;
   |                ^ denominator must be `non zero`
   |
note: `b` has type `Integer`; `Integer` does not carry `non zero` by any registered cluster
help: add a precondition: `requires b <> 0`
```

**E0122 — `type.attribute_contradiction`**. A term is simultaneously required to carry attribute `A` and attribute `B`, and `A` and `B` are mutually exclusive by a registered inconsistency cluster (§17.5).

```
error[E0122]: contradictory attributes (type.attribute_contradiction)
  --> src/lib.miz:7:7
   |
 7 |   var s as infinite finite set := ...;
   |            ^^^^^^^^ ^^^^^^ attributes `infinite` and `finite` are mutually exclusive
   |
   = note: registered inconsistency: `cluster infinite finite -> ;` (§17.5)
```

### 22.3.4 Overload Resolution Failure

**E0201 — `resolve.ambiguous_symbol`**. Overload resolution (Chapter 19) finds two or more candidates that are equally well-typed for the given arguments. The verifier cannot choose between them without additional information.

```
error[E0201]: ambiguous overload for `f` (resolve.ambiguous_symbol)
  --> src/lib.miz:8:11
   |
 8 |   const z = f(1, 1);
   |             ^^^^^^^^ two equally-ranked candidates; argument types are both `Nat`
   |
  --> src/algebra.miz:3:3
   |
 3 |   func f(x: Integer, y: Real) -> Real;
   |   ---- candidate 1
   |
  --> src/algebra.miz:7:3
   |
 7 |   func f(x: Real, y: Integer) -> Real;
   |   ---- candidate 2
   |
help: supply explicit types to disambiguate: `f(1 as Integer, 1 as Real)`
```

**E0202 — `resolve.no_viable_overload`**. No candidate survives type-checking for the given argument types. This may mean the function is not imported, or the arguments have the wrong types.

**E0203 — `template.argument_omitted_not_inferable`**. A template (§18) is used without supplying all schema parameters, and the missing parameters cannot be inferred from context.

```
error[E0203]: template argument cannot be inferred (template.argument_omitted_not_inferable)
  --> src/lib.miz:12:3
   |
12 |   SomeTemplate(x);
   |   ^^^^^^^^^^^^ schema parameter `T` is not inferable from argument types
   |
help: supply the argument explicitly: `SomeTemplate[Nat](x)`
```

## 22.4 Proof Errors and ATP Diagnostics

Proof errors are detected during verification condition (VC) generation and ATP interaction (§21.8). They carry codes in the E0300–E0399 range. ATP diagnostics are designed to be **actionable**: every failure includes a triage classification (§22.4.2) and, where possible, a concrete suggestion.

### 22.4.1 Proof Obligation Failure

**E0301 — `proof.by.search_exhausted`**. A `by` step failed: the verifier exhausted its search strategy without finding a proof. The ATP was called but returned no certificate within the resource budget.

```
error[E0301]: proof obligation not discharged (proof.by.search_exhausted)
  --> src/lib.miz:34:3
   |
34 |   thus n + 0 = n by Nat_add;
   |   ^^^^^^^^^^^^^^^^^^^^^^^^^^ could not prove: n + 0 = n
   |
note: goal: n + 0 = n
note: ATP: Vampire (timeout 10s), E (timeout 10s) — both exhausted
hint: try adding `add_0_right` to the `by` clause
```

**E0302 — `proof.by.missing_fact`**. Premise selection identified the goal as likely provable, but one or more required lemmas are not in scope or not reachable from the current `by` clause.

```
error[E0302]: likely missing lemma (proof.by.missing_fact)
  --> src/lib.miz:41:3
   |
41 |   thus result = Eval(t, v);
   |   ^^^^^^^^^^^^^^^^^^^^^^^^^ goal appears provable but requires additional facts
   |
note: goal: result = Eval(t, v)
note: candidate lemmas not in scope:
        poly_eval_add  (src/poly.miz:88)
        poly_eval_mul  (src/poly.miz:102)
help: add to `by` clause: `by poly_eval_add, poly_eval_mul`
```

**E0303 — `proof.obligation.open`**. A proof block ended with one or more sub-goals still open. Each open goal is reported as a separate attached note.

```
error[E0303]: proof block has 2 open goals (proof.obligation.open)
  --> src/lib.miz:50:3
   |
50 |   end;
   |   ^^^ proof block closed with goals remaining
   |
note: open goal 1: n > 0
note: open goal 2: m divides n
```

### 22.4.2 Failure Triage: Counterexample, Missing Lemma, or Computation Limit

When a proof obligation fails, the verifier classifies the failure into one of three categories before reporting it. The triage result determines which diagnostic code and suggestions are emitted.

| Triage result | Condition | Primary action |
|---|---|---|
| **Counterexample** | A counterexample model was found (§22.4.3). | Fix the statement or add a missing precondition. |
| **Missing lemma** | Premise selection found plausible candidates not in scope. | Add the suggested lemmas to the `by` clause. |
| **Computation limit** | ATP timed out or hit the axiom budget; no counterexample found. | Increase budget, split the goal, or add intermediate `assert` steps. |

The triage is performed after the first ATP call fails. If the counterexample search (§22.4.3) returns a model within its time budget, the triage is `Counterexample`; otherwise, premise selection runs and if it finds ranked candidates the triage is `Missing lemma`; otherwise the triage is `Computation limit`.

### 22.4.3 Counterexample Generation Integration

When a `by` step or an obligation within a `proof ... end` block fails, the verifier optionally invokes a **counterexample generator** (such as Nitpick or the finite-model finder of CVC5) to search for a model that falsifies the goal. This is controlled by the verifier configuration; it is enabled by default for goals with finite-domain quantifiers and disabled for large algebraic goals.

If a counterexample is found:

- The diagnostic code is **E0310 — `proof.counterexample.found`**.
- The model is displayed as a set of concrete value assignments.
- The diagnostic message describes which hypothesis or precondition is likely missing.

```
error[E0310]: counterexample found (proof.counterexample.found)
  --> src/lib.miz:27:3
   |
27 |   thus a + b = b + a;
   |   ^^^^^^^^^^^^^^^^^^ falsified by the model below
   |
note: counterexample model:
        type: Integer
        a := 1
        b := 2
        a + b = 3
        b + a = 3  (true in this case — statement is actually true)
note: counterexample search inconclusive: all tested models satisfy the goal
note: triage: computation limit (not a genuine counterexample)
```

> [!NOTE]
> A failed counterexample search does not imply the goal is false. It means the goal could not be refuted within the search budget. The triage then falls through to the missing-lemma or computation-limit category.

If a genuine counterexample is found (a model that falsifies the goal), the verifier emits:

```
error[E0310]: statement is false — counterexample found (proof.counterexample.found)
  --> src/lib.miz:15:3
   |
15 |   thus n * 2 = n + n + 1;
   |   ^^^^^^^^^^^^^^^^^^^^^^^ falsified
   |
note: counterexample:
        n := 1
        n * 2 = 2
        n + n + 1 = 3
        2 ≠ 3
help: the statement is incorrect; check the formula
```

### 22.4.4 Premise Selection and Lemma Suggestion

When the triage result is `Missing lemma`, the verifier runs a **premise selection** step. Premise selection scores all lemmas reachable from the current module by syntactic overlap with the goal (using a SInE-style heuristic) and returns the top candidates.

The ranked candidates are attached to the diagnostic as `note:` lines with their source locations:

```
note: candidate lemmas (by relevance score):
  1. add_0_right       (src/nat.miz:45)   — score 0.92
  2. add_comm          (src/nat.miz:51)   — score 0.87
  3. add_assoc         (src/nat.miz:58)   — score 0.71
```

The verifier then attempts a **retry** using the top-N candidates (default: N = 8) as an additional axiom set. If the retry succeeds, the error is upgraded to a suggestion:

```
help: the goal is provable with the following addition:
      `thus n + 0 = n by add_0_right;`
```

If the retry also fails, the full list of candidates is shown and the suggestion is downgraded to a `hint:`.

The `@missing_facts` build artifact (§23.5.2) records the full ranked candidate list for all open obligations in machine-readable form, enabling AI tools to propose `by` clause completions without re-running the verifier.

### 22.4.5 ATP Timeout and Computation Limit Handling

**E0320 — `proof.atp.timeout`**. Every ATP backend has a per-obligation time budget (default: 30 seconds per backend, configurable in the manifest, §23.1). When the budget is exhausted without a certificate, the obligation is reported as a timeout.

```
error[E0320]: ATP timeout (proof.atp.timeout)
  --> src/lib.miz:88:3
   |
88 |   thus thesis;
   |   ^^^^^^^^^^^^ Vampire (30s), E (30s) — all backends timed out
   |
note: goal: for n being Nat holds Fib(n) + Fib(n+1) = Fib(n+2)
note: triage: computation limit — no counterexample found
hint: split into subgoals with `assert` statements
hint: increase the ATP budget: `@proof_hint(solver: vampire)` with longer timeout in mizar.pkg
hint: add intermediate lemmas to reduce the goal complexity
```

**E0321 — `proof.atp.axiom_budget_exceeded`**. The number of candidate axioms for the obligation exceeds the configured `max_axioms` limit (§21.4.2). The verifier truncated the axiom set, which may have excluded the required lemma.

```
warning[W0101]: axiom budget exceeded — set was truncated (proof.atp.axiom_budget_exceeded)
  --> src/lib.miz:44:3
   |
44 |   thus thesis by ...;
   |   ^^^^^^^^^^^^^^^^^^^ 312 candidates → truncated to 128 (max_axioms limit)
   |
hint: cite specific lemmas in the `by` clause to reduce the candidate set
hint: increase `max_axioms` in mizar.pkg if the current limit is too restrictive
```

### 22.4.6 Known ATP Limitations and Honest Disclosure

The verifier does not pretend that ATP failure implies the statement is false, nor that ATP success implies the statement is true without kernel validation. The following disclosures are made explicit in diagnostic output:

- **Timeout does not imply falsity.** When `proof.atp.timeout` is emitted, the diagnostic includes `note: a timeout does not mean the goal is false`. The author should attempt to assist the ATP with intermediate steps.
- **Externally attested proofs are flagged.** When a prover cannot produce a kernel-verifiable certificate (§21.7.5), the proof is marked `externally_attested` in the build artifact (§23.5.1) and a `W0102` warning is emitted.
- **AC reasoning limitations.** When an obligation involves AC operators but the selected backend does not support native AC reasoning, the verifier emits `hint: switch to a backend with AC support (vampire or e)`.
- **Undecidable fragments.** When the goal contains constructs known to lie outside any decidable fragment (e.g., full second-order quantification), the verifier emits a `note:` explaining the limitation rather than simply reporting a timeout.

## 22.5 Logical Inconsistencies

Logical inconsistencies are detected when the verifier can prove that the accumulated axioms, registrations, or contracts are contradictory. These are the most serious errors: they indicate that the mathematical content of the file is unsound. They carry codes in the E0400–E0499 range.

### 22.5.1 Contradictory Axioms

**E0401 — `logic.contradictory_axioms`**. The combination of `axiom` declarations in scope allows the verifier (or ATP) to derive `False`. This means at least one axiom is either incorrect or inconsistent with the others.

```
error[E0401]: contradictory axioms detected (logic.contradictory_axioms)
  --> src/lib.miz:30:1
   |
30 |   axiom all_zero : for n being Nat holds n = 0;
   |   ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ this axiom, together with
   |
  --> src/lib.miz:25:1
   |
25 |   axiom succ_ne_zero : for n being Nat holds succ(n) <> 0;
   |   ---- ... makes the system inconsistent (1 = succ(0) <> 0 = 1)
   |
note: ATP derived `False` from these two axioms in 0.3s (Vampire)
```

Contradictory axioms are an error, not a warning, because any theorem becomes provable in an inconsistent system. The verifier halts further proof checking for the affected module.

### 22.5.2 Circular Definitions

**E0410 — `logic.circular_definition`**. A `func`, `pred`, `mode`, or `attr` definition refers to itself outside of an explicit `algorithm` block (which supports recursion with a termination proof, §20.8). Direct circular definitions in the `definition` layer are unconditionally rejected.

```
error[E0410]: circular definition (logic.circular_definition)
  --> src/lib.miz:8:3
   |
 8 |   func f(n) -> Nat equals g(n) + 1;
   |   ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
   |
  --> src/lib.miz:12:3
   |
12 |   func g(n) -> Nat equals f(n) - 1;
   |   ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ `f` and `g` are mutually circular
   |
note: use `algorithm` with `decreasing` for recursive definitions (§20.8)
```

**E0411 — `logic.circular_cluster`**. A cluster registration creates a cycle in the attribute inheritance graph: attribute `A` implies `B`, and `B` implies `A`, where both `A` and `B` carry contradictory requirements. This is distinct from a benign bidirectional registration, which is permitted.

### 22.5.3 Contract Violations (`requires`, `ensures`, `assert`)

**E0420 — `vc.postcondition.return`**. The verifier cannot prove that the `ensures` clause holds at a `return` statement (§20.4.1).

```
error[E0420]: postcondition not provable at return (vc.postcondition.return)
  --> src/lib.miz:18:5
   |
18 |     return -n;
   |     ^^^^^^^^^^ could not prove `result >= 0` when `n < 0`
   |
note: goal: -n >= 0
note: available: n < 0
note: ATP: Vampire — timeout; E — timeout
hint: the postcondition `ensures result >= 0` requires `n < 0 => -n >= 0`
hint: this holds over `Integer` but may not be registered; try `by neg_nonneg`
```

**E0421 — `vc.assert.failed`**. An `assert` statement (§20.4.2) within an algorithm body could not be proved.

**E0422 — `vc.precondition.call_site`**. At a call site, the verifier cannot prove that the callee's `requires` clause is satisfied.

```
error[E0422]: precondition not provable at call site (vc.precondition.call_site)
  --> src/lib.miz:35:15
   |
35 |   const q = divide(a, b);
   |                       ^ cannot prove `b <> 0`
   |
note: `divide` requires: b <> 0
note: available hypotheses do not establish this
help: add `requires b <> 0` to the enclosing algorithm, or add `assert b <> 0 by ...;` before the call
```

**E0423 — `vc.loop.establish`**. A loop invariant (§20.5.1) cannot be proved to hold before the first iteration.

**E0424 — `vc.loop.maintain`**. A loop invariant cannot be proved to be preserved from one iteration to the next.

**E0425 — `vc.loop.decrease`**. The termination measure (§20.5.2) cannot be proved to strictly decrease on each iteration, or cannot be proved to be bounded below.

**E0426 — `vc.recursion.decrease`**. For a recursive algorithm with `decreasing`, the termination measure cannot be proved to decrease at a recursive call site.

### 22.5.4 Cluster Registration Inconsistency

**E0430 — `logic.cluster.inconsistency`**. A conditional cluster registration (§17.3) adds attribute `A` to a type that already provably carries `¬A`. This means the registration, together with existing facts, implies a contradiction.

```
error[E0430]: cluster registration creates inconsistency (logic.cluster.inconsistency)
  --> src/lib.miz:45:1
   |
45 |   cluster non_zero for Integer;
   |   ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ registration implies all Integer values are non_zero
   |
note: but `0` is an `Integer` (from mml.integer) and is not `non_zero`
note: the attribute `non_zero` is defined as `x <> 0`
note: contradiction: 0 is Integer & 0 is non_zero & 0 = 0 implies 0 <> 0
```

**E0431 — `logic.cluster.overlap_ambiguity`**. Two functor cluster registrations overlap on their triggering conditions in a way that produces conflicting attribute sets for the same term. This does not immediately cause a contradiction but leaves attribute resolution undefined for the affected terms.

## 22.6 Warning Messages

Warnings do not prevent verification from completing. They indicate patterns that are likely unintentional, suboptimal, or deprecated. All warnings can be suppressed per-item using `@suppress(Wnnnn)` or globally in the package manifest (§23.1).

### 22.6.1 Unused Variables and Definitions

**W0001 — `warn.unused_variable`**. A `var` or `const` binding inside an algorithm body is never read after its declaration.

```
warning[W0001]: unused variable `tmp` (warn.unused_variable)
  --> src/lib.miz:12:7
   |
12 |   var tmp := compute_something();
   |       ^^^ declared but never used
   |
help: remove the binding, or prefix with `_` to suppress: `var _tmp := ...`
```

**W0002 — `warn.unused_definition`**. A `func`, `pred`, `mode`, or `attr` defined in a module is never referenced within the module or any module that imports it. This warning is emitted only at the package level (§23.4), not per-file.

**W0003 — `warn.unused_hypothesis`**. A hypothesis introduced by `assume` or `let` within a proof block is not referenced in any subsequent step. This is distinct from W0101 (redundant hypothesis, §22.6.2).

### 22.6.2 Redundant Hypotheses

**W0101 — `warn.redundant_hypothesis`**. A `by` clause references a fact that is not used in the ATP proof found. The fact may still be correct, but removing it would produce a simpler and more robust proof.

```
warning[W0101]: `Nat_add_comm` appears redundant in `by` clause (warn.redundant_hypothesis)
  --> src/lib.miz:22:22
   |
22 |   thus n = n by reflexivity, Nat_add_comm;
   |                              ^^^^^^^^^^^^ not used in the ATP proof
   |
help: remove `Nat_add_comm` from the `by` clause
```

**W0102 — `warn.externally_attested_proof`**. A proof obligation was accepted by an ATP but no kernel-verifiable certificate was produced (§21.7.5). The proof is `externally_attested` in the build artifact.

```
warning[W0102]: proof accepted without kernel certificate (warn.externally_attested_proof)
  --> src/lib.miz:55:3
   |
55 |   thus thesis;
   |   ^^^^^^^^^^^^ Z3 accepted the goal but produced no LFSC certificate
   |
note: this proof is marked `externally_attested` in the build artifact
note: the proof cannot be published to the package registry without kernel verification
hint: switch to `cvc5` or `vampire` for a kernel-verifiable certificate
```

### 22.6.3 Deprecated Features

**W0010 — `warn.deprecated_syntax`**. A syntactic construct, annotation, or library function is marked deprecated. The warning includes the replacement and the version in which the deprecated form will be removed.

```
warning[W0010]: `@infer_as` is deprecated (warn.deprecated_syntax)
  --> src/lib.miz:7:3
   |
 7 |   const z = @infer_as(0, Real);
   |             ^^^^^^^^^^^^^^^^^^^ deprecated since version 0.8; removed in version 1.0
   |
help: replace with `const z = 0 as Real;`
```

### 22.6.4 Unreachable Code in Algorithms

**W0201 — `warn.unreachable_code`**. A statement inside an algorithm body can never be executed because all control-flow paths reaching it are guarded by a condition the verifier can statically determine to be always false, or because an earlier `return`, `break`, or `continue` unconditionally transfers control away.

```
warning[W0201]: unreachable statement (warn.unreachable_code)
  --> src/lib.miz:19:5
   |
14 |     if n = 0 do
15 |       return 1;
16 |     end;
17 |     if n > 0 do
18 |       return n * factorial(n - 1);
19 |     end;
   |     ^^^^ this branch is unreachable: `n = 0` and `n > 0` together with `n: Nat` are exhaustive
20 |     return 0;
   |     ^^^^^^^^^ also unreachable
```

**W0202 — `warn.loop_may_not_terminate`**. An algorithm is declared `terminating` but contains a loop whose `decreasing` measure could not be statically verified to satisfy the well-foundedness condition. The verifier accepted the termination claim based on the user's annotation but could not confirm it independently.

### 22.6.5 Weakened Postcondition (`ensures` provable to stronger form)

**W0210 — `warn.weakened_postcondition`**. The verifier detected that the `ensures` clause is provable in a strictly stronger form than stated. This indicates the author may have intended a stronger guarantee, or that the postcondition was simplified incorrectly during editing.

```
warning[W0210]: `ensures` clause weaker than provable (warn.weakened_postcondition)
  --> src/lib.miz:8:5
   |
 8 |   terminating algorithm double(n) -> Nat
 9 |     ensures result >= n
   |             ^^^^^^^^^^^ this is provable, but `result = 2 * n` is also provable
   |
note: the verifier can prove `result = 2 * n` which implies `result >= n`
hint: consider strengthening to `ensures result = 2 * n` for a more precise specification
```

This warning is emitted only when the verifier can identify a **specific** stronger formula, not merely when the stated postcondition is weaker in the abstract. The stronger formula is obtained by running the premise selection heuristic (§22.4.4) in reverse: looking for equalities that the ATP can prove from the algorithm body.
