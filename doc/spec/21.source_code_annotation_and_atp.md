This chapter specifies the annotation system and automated theorem prover (ATP) integration of Mizar Evolution. Annotations are source-level markers that guide type inference, overload resolution, and proof search without altering the core semantics of a definition or theorem. The ATP integration layer translates verification conditions into standard problem formats, dispatches them to external provers, and submits the resulting proof certificates to the minimum kernel for independent validation.

The design follows three principles:

- **Semantic neutrality**: every annotation is ignorable. A source file without any annotations remains logically valid; the verifier may take longer or emit more diagnostic messages, but it will not reject correct code.
- **Erasability**: annotations carry no runtime meaning. They are silently dropped during code extraction (§20.9) and do not appear in proof certificates.
- **Layered integration**: human-written hints, verifier-resolved metadata, and diagnostic stability codes are treated as three distinct concerns, each with its own output channel.

Documentation comments (`::` lines preceding named items) and their structured tags (`@param`, `@returns`, etc.) are specified in §24.1.

## 21.1 Design Principles and Architecture

### 21.1.1 Three-layer AI/IDE Integration Model

AI and IDE tooling requires several kinds of information that the Mizar source language intentionally leaves implicit: inferred types, resolved overloads, `qua` coercions, cluster attribute propagation, and open verification conditions. This information can be made available through three complementary layers.

**Layer 1 — Hand-written hints.** The author places `@`-prefixed annotations directly in the source file. These are the annotations specified in §21.4–§21.8. They reduce the verifier's search space and give AI tools a stable, human-authored summary of intent. Because they are semantically neutral, they can be omitted, added, or removed freely without affecting correctness.

**Layer 2 — Resolved metadata output.** The verifier and type-checker record all internally computed information — inferred types, resolved functor fully-qualified names (FQNs), inserted `qua` coercions, active proof obligations — and emit them as structured artifacts alongside the build output. This is the primary channel for IDE hover and code-action data. The artifact format is described in §23.5.

**Layer 3 — Diagnostic stability codes.** Every diagnostic message carries a machine-readable code in addition to its natural-language description. AI tools consume codes rather than message text; this prevents brittleness when message wording changes across verifier versions. The code vocabulary is defined in §22.7.

### 21.1.2 Phase-based Introduction Plan

Not all annotations need to be introduced simultaneously. The recommended implementation order is:

**Phase 1 (core proof guidance):**
- `@[hint]`, `@[simp]` — theorem database labels
- `@proof_hint` — per-step lemma and solver constraints
- `@show_thesis` — interactive proof tracking
- Diagnostic code stabilization (§22.7)

**Phase 2 (type and resolution):**
- `@type_hint` — type inference guidance
- `@expect_type` — regression detection
- `@show_resolution` — overload and coercion visibility
- `@show_vc` — algorithm verification condition display
- Resolved metadata JSON artifact output (§23.5.1)

**Phase 3 (algorithm and ATP):**
- `@invariant_shape`, `@measure_hint`, `@match_intent` — algorithm hint vocabulary
- `@vc_hint` — per-VC lemma specification
- ATP explanation API and LSP integration (§23.6)

### 21.1.3 Non-recommended Annotations

The following patterns are **not** provided as source annotations, because they risk harming language evolution or conflating specification with implementation.

**Overload locking.** An annotation that forces a specific overload resolution unconditionally would make the source sensitive to library reorganisation. Disambiguation is handled by qualification, `qua`, and the `@resolve` hint (§21.5.1), which remains advisory.

**Cluster expansion ordering.** The order in which cluster registrations propagate is an internal verifier concern. Fixing it in source creates fragile code that breaks when registration rules are refined.

**Pick execution strategy.** The `the Element of S` construct (§20.3) is logically arbitrary. Attaching a source-level annotation such as `smallest` or `first` conflates the mathematical specification with an implementation policy; such policies belong in MVM configuration, not source.

**AI-specific semantics.** Annotations whose sole purpose is to change the meaning of a construct for AI tools — as opposed to guiding the verifier — are explicitly out of scope. AI-facing information must remain erasable metadata.

## 21.2 Library Annotations

Library annotations attach stable attributes to definitions for use by the proof search engine. They are written as `@[label]` immediately before the item they annotate and may appear on definitions, theorems, and registrations.

```ebnf
library_annotation ::= "@[" label_list "]" ;
label_list         ::= label_name { "," label_name } ;
label_name         ::= identifier [ "(" annotation_args ")" ] ;
```

The defined label names are: `hint`, `simp` (§21.6.1), and the algorithm verification labels `vc_hint`, `invariant_shape`, `measure_hint`, `match_intent` (§21.7). Additional labels may be registered by user libraries, but are ignored by the built-in verifier.

## 21.3 LaTeX Expression Annotations

A LaTeX expression annotation supplies the preferred mathematical rendering of a symbol for documentation output and IDE display. It does not affect type-checking or proof search.

```ebnf
latex_annotation ::= "@latex" "(" latex_string ")" ;
latex_string     ::= string_literal ;
```

The annotation is placed immediately before a `func`, `pred`, `mode`, or `attr` definition:

```mizar
@latex("n!")
func factorial(n) -> Nat;

@latex("\gcd(a, b)")
func Gcd(a, b) -> Nat;

@latex("a \leq b")
pred leq(a, b);
```

The documentation generator (§24.3) substitutes the LaTeX string wherever the symbol appears in rendered output. If no `@latex` annotation is present, the symbol name is used verbatim.

## 21.4 Type Annotations

### 21.4.1 `@type_hint` (inference guidance)

`@type_hint` tells the type inference engine (Chapter 8) to try a specific type first when resolving the annotated binding or expression. The hint is advisory: if normal inference reaches the same conclusion, the hint is harmless; if inference would reach a different valid conclusion, the hint biases the choice; if inference cannot unify the hint with any valid type, the hint is ignored and a warning is emitted rather than an error.

```ebnf
type_hint_annotation ::= "@type_hint" "(" type_hint_item { "," type_hint_item } ")" ;
type_hint_item       ::= identifier ":" type_expression ;
```

The annotation is placed immediately before the binding or expression it targets:

```mizar
@type_hint(result: Element of K)
terminating algorithm eval_at(t, v) -> Element of K
  ensures result = Eval(t, v)
  decreasing term_size(t)
do
  ...
end;
```

```mizar
@type_hint(x: Element of PolyRing[K, x])
var x;
```

**Permitted uses:**

- Disambiguating pattern variables in `match` arms where multiple types are structurally compatible.
- Resolving overloaded literals (e.g., `0` where both `Nat` and `Real` are candidates) without forcing an explicit `as` cast.
- Narrowing the candidate set for template instantiation (§18.5) when the algorithm is called in a polymorphic context.

**Restrictions.** `@type_hint` is not a cast. It does not widen or narrow the type of an expression; it only affects the order in which the type inference algorithm explores candidates. When a definitive type annotation is needed, use `as T` (§8.3) or an explicit `let` declaration.

### 21.4.2 `@expect_type` (regression detection)

`@expect_type` asserts that the annotated expression or binding resolves to a specific type. Unlike `@type_hint`, a mismatch between the expected type and the inferred type is a **verifier error**, not a warning. This makes the annotation suitable for regression detection: if a library refactoring accidentally changes how a type is inferred, the error is reported immediately.

```ebnf
expect_type_annotation ::= "@expect_type" "(" identifier ":" type_expression ")" ;
```

```mizar
@expect_type(m: Nat)
const m = n - i;
```

**Permitted uses:**

- Documenting type boundaries in specification files and teaching materials.
- Detecting unintended inference changes during refactoring.
- Giving AI tools an explicit type guarantee at a specific source location.

> [!NOTE]
> `@expect_type` does not imply that the expression is valid at any type other than the one stated. If the inferred type is a subtype of the expected type (e.g., `Nat` where `Integer` is expected), the verifier may still accept the annotation if the subtype relationship is registered. Whether to treat widening as acceptable is a verifier configuration option.

## 21.5 Resolution Hints

### 21.5.1 `@resolve` (overload disambiguation)

`@resolve` advises the overload resolution engine (Chapter 19) to prefer a specific candidate when multiple definitions match. The annotation is placed immediately before the expression or statement containing the ambiguous symbol.

```ebnf
resolve_annotation ::= "@resolve" "(" resolve_item { "," resolve_item } ")" ;
resolve_item       ::= resolve_key ":" fqn ;
resolve_key        ::= "op" | "functor" | "predicate" | "mode" ;
fqn                ::= identifier { "." identifier } ;
```

```mizar
@resolve(op: algebra.ring.add)
a + b;
```

```mizar
@resolve(functor: mml.number.gcd.Gcd)
Gcd(a, b);
```

`@resolve` is an advisory hint. The verifier still checks that the nominated candidate is type-correct in context; if it is not, overload resolution proceeds as if the annotation were absent and a warning is emitted. This prevents `@resolve` from being used to suppress legitimate type errors.

> [!NOTE]
> `@resolve` is intended as an escape hatch for AI-generated code and for environments with unusually large overload sets. In normal human-authored code, qualified names and `qua` coercions are the preferred disambiguation mechanisms.

### 21.5.2 IDE-only Annotations (`@view`, `@template_args`)

`@view` and `@template_args` are **not** source-level annotations. They are metadata labels used by the LSP server and IDE code actions to communicate resolved information back to the editor. They are listed here to document their meaning, since they may appear in IDE hover output and in the `*.mizir.json` artifact (§23.5.1).

**`@view(S: Interface)`.** Records that the expression `S` was accessed through the view named by the interface. Corresponds to a `qua` coercion inserted by the verifier. In source, write `S qua Interface` explicitly.

**`@template_args(params)`.** Records the concrete arguments substituted for a template's schema parameters (§18.5). In source, write `Template[Arg1, Arg2]` explicitly.

Both labels appear in the `inserted_coercions` and `template_args` fields of `*.mizir.json` (§23.5.1) and in LSP hover responses (§23.6.1).

## 21.6 Reasoning Annotations

### 21.6.1 `@[hint]`, `@[simp]`

These library annotations tag theorems and registrations with roles that guide proof search:

| Label | Effect on proof search |
|---|---|
| `@[hint]` | The tagged fact is offered as a priority candidate when `by` performs backward proof search in the enclosing namespace. |
| `@[simp]` | The tagged equation or iff is passed to the ATP translator as a priority rewrite axiom. A `direction` argument controls orientation: `left_to_right` (default) or `right_to_left`. |

Since proof search is entirely delegated to ATP backends (§21.10), `@[simp]` does not trigger a separate pre-ATP normalization pass. Instead, the Translator includes `@[simp]`-tagged lemmas as high-priority axioms in the problem file, allowing the ATP's native rewriting mechanism (superposition, paramodulation) to apply them during proof search. This avoids a goal-less normalization problem: the ATP always works against an explicit proof obligation.

```mizar
@[simp]
theorem add_0_right:
  for n being Nat holds n + 0 = n;

@[simp(direction: right_to_left)]
theorem factorial_unfold:
  for n being Nat holds factorial(n + 1) = (n + 1) * factorial(n);

@[hint]
theorem Gcd_commutes:
  for a, b being Nat holds Gcd(a, b) = Gcd(b, a);
```

Multiple labels may be combined on a single item:

```mizar
@[simp, hint]
theorem mul_1_right:
  for n being Nat holds n * 1 = n;
```

**Scope.** A `@[simp]` or `@[hint]` label has effect within the module where the annotation appears and in any module that imports it. Labels are not inherited transitively across `import` boundaries unless the importing module re-exports the annotated item.

### 21.6.2 `@proof_hint` (lemma set and solver constraints)

`@proof_hint` constrains the proof search for the immediately following `by` step, `thus` statement, or `proof ... end` block. It narrows the set of candidate lemmas, controls the ATP solver selection, and optionally caps the axiom budget.

```ebnf
proof_hint_annotation ::= "@proof_hint" "(" proof_hint_options ")" ;
proof_hint_options    ::= proof_hint_option { "," proof_hint_option } ;
proof_hint_option     ::= "lemmas"     ":" "[" fqn_list "]"
                        | "using"      ":" "[" theory_list "]"
                        | "max_axioms" ":" nat_literal
                        | "solver"     ":" solver_name ;
fqn_list              ::= fqn { "," fqn } ;
theory_list           ::= identifier { "," identifier } ;
solver_name           ::= "vampire" | "e" | "cvc5" | "z3" | "auto" ;
```

```mizar
@proof_hint(
  lemmas: [poly_eval_add, poly_eval_mul],
  max_axioms: 32,
  solver: vampire
)
thus result = Eval(t, v);
```

```mizar
@proof_hint(using: [arithmetic, simp, cluster])
thus thesis by mod_positive;
```

**`lemmas`** lists the exact facts the ATP should consider, in addition to the locally available hypotheses. Facts not in this list are excluded from the axiom set passed to the ATP. This prevents the translator (§21.10.2) from loading unrelated library lemmas and speeds up proof search at the cost of requiring explicit guidance.

**`using`** lists named theory modules (e.g., `arithmetic`, `ring_normalize`, `cluster`) whose contents are included in the axiom set. This is coarser than `lemmas` but requires less maintenance when the library evolves.

**`max_axioms`** caps the total number of axioms sent to the ATP. If the combined `lemmas` and `using` set exceeds this limit, lower-priority facts are pruned.

**`solver`** selects the backend prover (§21.6.3). When set to `auto`, the verifier applies its default selection heuristic.

### 21.6.3 Solver Selection

The default solver selection policy is:

1. If the goal involves only linear arithmetic and equality, prefer **CVC5**.
2. If the goal involves higher-order patterns or AC operators, prefer **Vampire** with AC support.
3. Otherwise, try **E** first and fall back to **Vampire** on timeout.

The policy can be overridden globally in the package manifest (§23.1) and locally with `@proof_hint(solver: ...)`. Individual `solver` overrides take precedence over the global setting.

## 21.7 Algorithm Verification Hints

The annotations in this section apply specifically to algorithm bodies (Chapter 20). They provide the verifier and AI tools with structured metadata about loop invariants, termination measures, and pattern-match intent. All annotations in this section are semantically neutral: they do not change the verification conditions generated (§20.4.3), only the strategy used to discharge them.

### 21.7.1 `@vc_hint` (per-VC lemma guidance)

`@vc_hint` attaches a lemma set to a specific verification condition (VC) generated by a loop or algorithm contract. It is placed immediately before the loop or contract statement to which it applies.

```ebnf
vc_hint_annotation ::= "@vc_hint" "(" vc_kind ":" "[" theory_list "]" ")" ;
vc_kind            ::= "establish" | "maintain" | "exit" | "break" | "continue" | "decrease" ;
```

The VC kinds correspond to the obligations generated for `while` loops (§20.5.1):

| `vc_kind` | Obligation |
|---|---|
| `establish` | Invariant holds before the first iteration. |
| `maintain` | Invariant is preserved from one iteration to the next. |
| `exit` | Postcondition follows from invariant and loop exit condition. |
| `break` | Postcondition follows from invariant at the point of a `break`. |
| `continue` | Invariant holds after `continue` skips the loop tail. |
| `decrease` | Termination measure strictly decreases each iteration. |

```mizar
while i <= n do
  @vc_hint(establish: [arithmetic])
  @vc_hint(maintain: [arithmetic, ring_normalize])
  @vc_hint(decrease: [arithmetic])
  invariant s = i * (i - 1) div 2;
  decreasing n - i + 1;
  s := s + i;
  i := i + 1;
end;
```

Multiple `@vc_hint` annotations may be stacked before a single loop; each applies to the VC kind it names. If no `@vc_hint` is present, the verifier uses the global solver heuristic (§21.6.3).

### 21.7.2 `@invariant_shape` (accumulator, frame, bound, relational)

`@invariant_shape` tags an `invariant` clause with a semantic role, enabling AI tools and IDE display features to classify the invariant without parsing its formula.

```ebnf
invariant_shape_annotation ::= "@invariant_shape" "(" shape_kind { "," shape_option } ")" ;
shape_kind                 ::= "accumulator_over" ":" identifier
                             | "frame"            ":" "[" identifier_list "]"
                             | "bound"
                             | "relational"
                             | "ghost_coupling" ;
shape_option               ::= identifier ":" identifier ;
```

| Shape | Typical invariant pattern |
|---|---|
| `accumulator_over: V` | `total = Sum(V)`, where `V` is a ghost processed-set. |
| `frame: [x, y]` | Only `x` and `y` change; everything else is preserved. |
| `bound` | A numeric quantity remains within a fixed range. |
| `relational` | Relates loop variables to the pre-loop state. |
| `ghost_coupling` | Links a ghost variable to its concrete counterpart. |

```mizar
for x in S processed V do
  @invariant_shape(accumulator_over: V)
  invariant total = Sum(V);
  total := total + x;
end;
```

```mizar
while current <> {} do
  @invariant_shape(frame: [best, visited])
  invariant visited = S \ current & best in S;
  ...
end;
```

`@invariant_shape` is a pure metadata tag. It does not generate any additional proof obligations and does not affect which lemmas the verifier selects.

### 21.7.3 `@measure_hint` (numeric vs. structural termination)

`@measure_hint` classifies a `decreasing` termination measure as numeric or structural and optionally supplies a human-readable description of what is decreasing.

```ebnf
measure_hint_annotation ::= "@measure_hint" "(" measure_kind { "," measure_option } ")" ;
measure_kind            ::= "kind" ":" ( "numeric" | "structural" ) ;
measure_option          ::= "source" ":" string_literal ;
```

```mizar
@measure_hint(kind: structural)
decreasing term_size(t);
```

```mizar
@measure_hint(kind: numeric, source: "remaining interval length")
decreasing r - l + 1;
```

**`kind: structural`** tells the verifier that the measure is a structural size function (e.g., `term_size`, `card`) and that its well-foundedness follows from a registered well-order on the type, without requiring an explicit numeric lower-bound argument.

**`kind: numeric`** indicates that the measure is a non-negative integer expression and that termination follows from the well-order of `Nat`. The verifier generates the obligation `decreasing_expr >= 0` automatically.

The `source` string is advisory only; it appears in IDE tooltips and diagnostic output.

### 21.7.4 `@match_intent` (structural recursion, normalization, etc.)

`@match_intent` tags a `match` statement with its intended role, allowing AI tools to select appropriate lemma templates for the resulting proof obligations.

```ebnf
match_intent_annotation ::= "@match_intent" "(" "kind" ":" match_kind ")" ;
match_kind              ::= "structural_recursion"
                          | "normalization"
                          | "destructor_elimination"
                          | "compiler_lowering" ;
```

```mizar
@match_intent(kind: structural_recursion)
match t do
  case p + q do
    @proof_hint(lemmas: [poly_eval_add])
    return eval_at(p, v) + eval_at(q, v);
  end;
  case p * q do
    @proof_hint(lemmas: [poly_eval_mul])
    return eval_at(p, v) * eval_at(q, v);
  end;
  otherwise
    return Eval(t, v);
  end;
end;
```

| Intent | Meaning |
|---|---|
| `structural_recursion` | Each arm recurses on a structurally smaller subterm. |
| `normalization` | Each arm rewrites toward a canonical form. |
| `destructor_elimination` | Each arm eliminates one level of a composite constructor. |
| `compiler_lowering` | Each arm maps a high-level construct to a lower-level representation. |

## 21.8 AI/IDE Display Hints

The annotations in this section instruct the verifier or LSP server to surface internal state at a specific source location. They are intended for interactive proof development, debugging, and AI-guided editing. They have no effect on the final verification result.

### 21.8.1 `@show_thesis`

`@show_thesis` requests that the verifier emit the current **proof obligation** (thesis) as a diagnostic at the annotated position. This corresponds to the thesis-tracking mechanism described in §16.9, made accessible at the source level.

```mizar
proof
  @show_thesis
  thus n + 0 = n by add_0_right;
end;
```

The emitted diagnostic has code `info.thesis` and includes the formula that must be proved at that point. It appears in the IDE as an informational annotation (not a warning or error) and is included in `@missing_facts` output (§23.5.2) when the step cannot be discharged.

### 21.8.2 `@show_vc`

`@show_vc` requests that the verifier enumerate the verification conditions generated for the annotated algorithm statement. It is placed immediately before a loop, `assert`, or contract clause.

```mizar
@show_vc
while i <= n do
  invariant s = i * (i - 1) div 2;
  decreasing n - i + 1;
  ...
end;
```

The verifier emits one `info.vc` diagnostic per generated obligation, labelled by its kind (§21.7.1) and its current proof status (`proved`, `open`, `timeout`). This is equivalent to the information that the LSP server returns via the `@show_vc` code action (§23.6.1), but triggered directly from source.

### 21.8.3 `@show_resolution`

`@show_resolution` requests that the verifier emit the overload resolution result for the immediately following expression. The diagnostic lists the resolved functor or predicate FQN, the template arguments substituted, and any `qua` coercions inserted.

```mizar
@show_resolution
Product[R](s);
```

Example diagnostic output (code `info.resolution`):

```
resolved functor : algebra.monoid.Product
template args    : R qua MulMagma
result type      : Element of R
inserted coercion: s qua MulMagma
```

This information is always available via LSP hover (§23.6.1); `@show_resolution` makes it visible in the plain-text verifier output for users not running an IDE.

### 21.8.4 `@show_type`

`@show_type` requests that the verifier emit the inferred type of a specific expression without assigning it to a binding.

```ebnf
show_type_annotation ::= "@show_type" "(" term_expression ")" ;
```

```mizar
@show_type(total + x)
```

The diagnostic (code `info.type`) reports the inferred type of the given expression in the current scope. This is a lightweight debugging tool; in interactive development the LSP hover (§23.6.1) is usually sufficient.

## 21.9 Built-in Verification Engine (Pre-ATP)

Before dispatching a proof obligation to an external ATP, the verifier applies a sequence of lightweight built-in decision procedures. These pre-ATP steps discharge many routine obligations without external calls, reducing latency and ATP resource consumption.

### 21.9.1 Term Normalization

The normalizer applies built-in structural property encodings derived from cluster registrations. The following property classes are normalised automatically when the corresponding registration is active:

| Property | Normalisation applied |
|---|---|
| `commutativity` of `f` | Canonical ordering of `f(a, b)` and `f(b, a)` |
| `idempotence` of `f` | Reduction of `f(a, a)` to `a` |
| `involutiveness` of `f` | Reduction of `f(f(a))` to `a` |
| `projectivity` of `f` | Reduction of `f(f(a, b), b)` to `f(a, b)` |

Normalisation runs to a fixed point before any inference step.

### 21.9.2 Extended Congruence Closure

The verifier maintains a congruence closure over the equalities established in the current proof context. When a goal `a = b` is presented, the engine checks whether `a` and `b` are congruent under the accumulated equalities before invoking the ATP. This discharges goals of the form `f(a) = f(b)` given `a = b` without external calls.

The congruence closure is extended with the normalisation equations of §21.9.1. When commutativity is registered for an operator, the closure treats `f(a, b)` and `f(b, a)` as equivalent without generating an explicit symmetry step.

### 21.9.3 Built-in Inference Rules

The following inference rules are applied by the built-in engine without ATP involvement:

| Rule | Applied when |
|---|---|
| Reflexivity | Goal has the form `a = a` or `P(a) <=> P(a)`. |
| Symmetry (equality) | Goal has the form `a = b` and `b = a` is in context. |
| Symmetry (predicate) | Property `symmetry` is registered for `P`; `P(b, a)` follows from `P(a, b)`. |
| Modus ponens | `P` and `P => Q` are in context; `Q` is discharged. |
| Transitivity | `a = b` and `b = c` are in context; `a = c` is discharged. |

These rules are not configurable. They are sound and terminating by construction.

### 21.9.4 Checker Hint Database

The **checker hint database** is a module-scoped table populated by `@[hint]` and `@[simp]` labels. When the built-in engine fails to discharge a goal, it passes the goal and a ranked list of database candidates to the ATP translator (§21.10.2). `@[hint]`-tagged facts are ranked as general priority candidates; `@[simp]`-tagged equations are passed as priority rewrite axioms with their direction. The ranking is based on syntactic overlap between the goal and the candidate's conclusion.

The database is indexed per namespace. Items tagged in imported modules are visible in the importing module unless the import uses a qualified prefix that excludes them. The `@proof_hint(lemmas: [...])` annotation overrides database ranking for the annotated step.

## 21.10 ATP Integration

### 21.10.1 Minimum Kernel Architecture and de Bruijn Criterion

Mizar Evolution adopts the **minimum kernel** design: the trusted core of the verifier performs only proof certificate validation and does not include any inference logic. Inference is entirely delegated to external ATPs. This design satisfies the de Bruijn criterion — the size of the trusted code base is minimised, and the correctness of the system does not depend on the correctness of any external prover.

```
┌──────────────────────────────────────┐
│          User Proof Source           │
│  (proof...end / by references)       │
└──────────────┬───────────────────────┘
               │
       ┌───────▼───────┐
       │  Translator   │  ← encodes properties as axioms / AC declarations
       │               │  ← selects lemmas from hint database
       └───────┬───────┘
               │  TPTP / SMT-LIB
       ┌───────▼───────┐
       │  ATP Backend  │  Vampire / E / CVC5
       │  (external)   │
       └───────┬───────┘
               │  proof certificate
       ┌───────▼───────┐
       │  Minimum      │  ← validates certificate only
       │  Kernel       │  ← performs no inference
       └───────────────┘
```

The ATP is treated as an **untrusted oracle**: it produces a proof certificate that the kernel checks independently. A bug or unsoundness in the ATP can produce an incorrect certificate, but the kernel will reject it. Only a bug in the kernel itself can compromise soundness.

> [!IMPORTANT]
> The kernel never trusts an ATP result unconditionally. Every proof obligation that the ATP claims to have proved must be accompanied by a certificate that the kernel can check without invoking the ATP again. An ATP timeout or certificate rejection is reported as an open proof obligation, not as an accepted proof.

### 21.10.2 Problem Translation Pipeline (Translator)

The Translator converts a Mizar proof obligation into a first-order or SMT problem. The pipeline has four stages:

1. **Scope collection.** The Translator collects all hypotheses, `let`-bound variables, and registered properties in scope at the obligation site. It applies the `@proof_hint` constraints (§21.6.2) to restrict the axiom set.

2. **Property encoding.** Registered properties of functors and predicates (§10.6.1, §9.5.1) are translated to axioms or native declarations (§21.10.4). AC operators are declared using backend-native AC mechanisms when available.

3. **Goal translation.** The proof obligation itself is translated as the negated conjecture (for refutation-based ATPs) or as the assertion to be satisfied (for SMT solvers).

4. **Lemma ranking.** The checker hint database (§21.9.4) provides a ranked candidate list. The Translator selects at most `max_axioms` lemmas (default: 128; overridable by `@proof_hint`) and appends them as additional axioms.

The output is a self-contained problem file in the format required by the selected backend.

### 21.10.3 Backend Provers (Vampire, E, CVC5)

| Prover | Input format | Strengths | Proof output |
|---|---|---|---|
| **Vampire** | TPTP FOF/TFF | General first-order logic, AC reasoning, large axiom sets | TSTP derivation |
| **E** | TPTP FOF | Equational reasoning, AC completeness | TSTP derivation |
| **CVC5** | SMT-LIB 2 | Linear arithmetic, algebraic datatypes, quantifier-free fragments | LFSC / Alethe certificate |
| **Z3** | SMT-LIB 2 | Linear and nonlinear arithmetic, bit-vectors | SMTLIB proof (optional) |

Backend selection follows the policy in §21.6.3 unless overridden by `@proof_hint(solver: ...)`. Multiple backends may be tried in parallel; the first to return a valid certificate wins.

### 21.10.4 Property Encoding for ATP

#### 21.10.4.1 TPTP Encoding (Vampire, E): Axioms and AC Declarations

Properties that have no native TPTP representation are encoded as first-order axioms:

```tptp
fof(idem_union, axiom, ![A]: (union(A,A) = A)).         % idempotence
fof(invol_neg,  axiom, ![A]: (neg(neg(A)) = A)).         % involutiveness
fof(proj_f,     axiom, ![A,B]: (f(f(A,B),B) = f(A,B))). % projectivity
fof(sym_equiv,  axiom, ![A,B]: (equiv(A,B) => equiv(B,A))).  % symmetry
fof(refl_sub,   axiom, ![A]: (subset(A,A))).             % reflexivity
fof(irrefl_lt,  axiom, ![A]: (~less(A,A))).              % irreflexivity
```

Commutativity and associativity are handled differently. When the target prover supports native AC reasoning, the Translator emits a backend-specific AC declaration instead of axioms:

- **Vampire**: `--ac_reasoning on` flag; the prover normalises AC terms internally.
- **E**: AC completeness is enabled via the `--term-ordering` flag; terms are normalised under AC.

Encoding commutativity as a plain FOF axiom (`f(A,B) = f(B,A)`) causes a quadratic blowup in the search space. The native AC mechanism instead works on AC-normal forms, reducing complexity by orders of magnitude for large algebraic problems.

#### 21.10.4.2 SMT-LIB Encoding (CVC5, Z3)

For SMT backends, properties are encoded as universally quantified assertions:

```smt2
; commutativity
(assert (forall ((a T) (b T)) (= (f a b) (f b a))))
; reflexivity
(assert (forall ((a T)) (P a a)))
; symmetry
(assert (forall ((a T) (b T)) (=> (P a b) (P b a))))
; idempotence
(assert (forall ((a T)) (= (f a a) a)))
```

CVC5 and Z3 integrate these assertions into their congruence closure and quantifier-instantiation engines. For purely linear arithmetic goals, quantified assertions are usually unnecessary because the arithmetic decision procedure handles them directly.

#### 21.10.4.3 Encoding Strategy by Property Type

| Property | TPTP (Vampire/E) | SMT-LIB (CVC5/Z3) | Notes |
|---|---|---|---|
| `commutativity` | **AC declaration** (native) | Quantified assertion | AC native is orders of magnitude faster |
| `associativity` | **AC declaration** (native) | Quantified assertion | Combined with commutativity when both hold |
| `idempotence` | Equality axiom | Quantified assertion | No native mechanism |
| `involutiveness` | Equality axiom | Quantified assertion | No native mechanism |
| `projectivity` | Equality axiom | Quantified assertion | No native mechanism |
| `symmetry` | Implication axiom | Quantified assertion | ATP equality handling is effective |
| `reflexivity` | Universal axiom | Quantified assertion | Interacts with ATP equality |
| `asymmetry` | Implication axiom | Quantified assertion | |
| `connectedness` | Implication axiom | Quantified assertion | |
| `irreflexivity` | Negation axiom | Quantified assertion | |

### 21.10.5 Proof Certificate Formats (TSTP, LFSC/Alethe)

The minimum kernel accepts proof certificates in the following formats:

**TSTP** (used by Vampire and E). A TSTP derivation is a sequence of annotated formulae, each labelled with the inference rule used and the indices of the parent formulae. The kernel reconstructs each inference step independently and checks that the final formula is the negation of the original goal.

**LFSC/Alethe** (used by CVC5). An LFSC certificate is a typed proof term in the Edinburgh Logical Framework. An Alethe certificate is a sequence of SMT proof steps. The kernel has a dedicated checker for each format that validates the term structure and all typing judgements.

**Proof log** (fallback). When a prover cannot emit a structured certificate (e.g., Z3 in some configurations), the ATP interaction is logged as a `*.atp-log` file (§23.5.3). In this mode the proof is marked as **externally attested** rather than kernel-verified. Externally attested proofs are accepted for development but are flagged in the build output and do not satisfy the kernel-verification requirement for release builds.

> [!IMPORTANT]
> The distinction between a kernel-verified proof and an externally attested proof is surfaced in the `*.mizir.json` artifact (§23.5.1) as the `proof_status` field (`"kernel_verified"` vs `"externally_attested"`). A module containing externally attested proofs cannot be published to the package registry without explicit author acknowledgement.

### 21.10.6 Sethood Resolution at Type-Check Level

The `sethood` property (§5.8.1) is resolved at type-check time, before any proof obligation is generated and before the ATP is invoked. This avoids a class of ATP timeouts that would otherwise occur when the Fraenkel set comprehension `{ f(x) where x is T : P(x) }` is used with a type `T` that has no registered sethood witness.

The rule is:

- When `{ f(x) where x is T : P(x) }` appears in a term, the type-checker looks up whether `sethood` is registered for `T`.
- If a registration is found, the type-checker inserts the corresponding existence fact as a background axiom for the enclosing proof obligation.
- If no registration is found, the type-checker emits a **compile-time error** with code `type.sethood.missing` and rejects the term. The error is not deferred to the ATP.

This design keeps sethood reasoning entirely within the type-checker's scope, consistent with the kernel's role as a certificate validator rather than an inference engine.
