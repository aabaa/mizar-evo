This chapter specifies the annotation system and automated theorem prover (ATP) integration of Mizar Evolution. Annotations are source-level markers that guide type inference, overload resolution, and proof search without altering the core semantics of a definition or theorem. The ATP integration layer translates verification conditions into standard problem formats, dispatches them to external provers, and submits the resulting proof certificates to the minimum kernel for independent validation.

The design follows three principles:

- **Semantic neutrality**: every annotation is ignorable. A source file without any annotations remains logically valid; the verifier may take longer or emit more diagnostic messages, but it will not reject correct code.
- **Certificate neutrality**: annotations do not appear in proof certificates and are dropped during code extraction (§20.10). Some annotations (`@show_*`, `@eval`) produce diagnostic output at verification time; this output is advisory and has no effect on what is proved.
- **Layered integration**: human-written hints, verifier-resolved metadata, and diagnostic stability codes are treated as three distinct concerns, each with its own output channel.

Documentation comments (`::` lines preceding named items) and their structured tags (`@param`, `@returns`, etc.) are specified in §24.1.

## 21.1 Design Principles and Architecture

### 21.1.1 Three-layer AI/IDE Integration Model

AI and IDE tooling requires several kinds of information that the Mizar source language intentionally leaves implicit: inferred types, resolved overloads, `qua` coercions, cluster attribute propagation, and open verification conditions. This information can be made available through three complementary layers.

**Layer 1 — Hand-written hints.** The author places `@`-prefixed annotations directly in the source file. These are the annotations specified in §21.4–§21.6. They reduce the verifier's search space and give AI tools a stable, human-authored summary of intent. Because they are semantically neutral, they can be omitted, added, or removed freely without affecting correctness.

**Layer 2 — Resolved metadata output.** The verifier and type-checker record all internally computed information — inferred types, resolved functor fully-qualified names (FQNs), inserted `qua` coercions, active proof obligations — and emit them as structured artifacts alongside the build output. This is the primary channel for IDE hover and code-action data. The artifact format is described in §23.5.

**Layer 3 — Diagnostic stability codes.** Every diagnostic message carries a machine-readable code in addition to its natural-language description. AI tools consume codes rather than message text; this prevents brittleness when message wording changes across verifier versions. The code vocabulary is defined in §22.7.

### 21.1.2 Phase-based Introduction Plan

Not all annotations need to be introduced simultaneously. The recommended implementation order is:

**Foundation (all phases):**
- `@[...]` library annotation syntax — §21.2
- `@latex` — documentation rendering for symbols; §21.3

**Phase 1 (core proof guidance):**
- `@proof_hint` — per-step solver and axiom budget control
- Diagnostic code stabilization (§22.7)

**Phase 2 (display hints, §21.5):** available from Phase 1 onward and extended as Phase 2 features are introduced.
- `@show_thesis` — emit current proof obligation as a diagnostic
- `@show_resolution` — report overload resolution result for the following expression
- `@show_type` — report inferred type of an expression
- Resolved metadata JSON artifact output (§23.5.1)

**Phase 3 (algorithm and ATP):**
- `@eval` — verification-time expression evaluation
- ATP explanation API and LSP integration (§23.6)

### 21.1.3 Non-recommended Annotations

The following patterns are **not** provided as source annotations, because they risk harming language evolution or conflating specification with implementation.

**Overload locking.** An annotation that forces a specific overload resolution unconditionally would make the source sensitive to library reorganisation. Disambiguation is handled by qualification, `qua`, and the `@resolve` hint (§21.6.1), which remains advisory.

**Cluster expansion ordering.** The order in which cluster registrations propagate is an internal verifier concern. Fixing it in source creates fragile code that breaks when registration rules are refined.

**Pick execution strategy.** The `the Element of S` construct (§20.3) is logically arbitrary. Attaching a source-level annotation such as `smallest` or `first` conflates the mathematical specification with an implementation policy; such policies belong in MVM configuration, not source.

**AI-specific semantics.** Annotations whose sole purpose is to change the meaning of a construct for AI tools — as opposed to guiding the verifier — are explicitly out of scope. AI-facing information must remain erasable metadata.

## 21.2 Library Annotations

Library annotations attach stable attributes to definitions for use by the proof search engine. They are written as `@[label]` immediately before the item they annotate and may appear on definitions, theorems, and registrations.

```ebnf
library_annotation ::= "@[" label_list "]" ;
label_list         ::= label_name { "," label_name } ;
label_name         ::= identifier [ "(" annotation_args ")" ] ;
```

Additional labels may be registered by user libraries, but are ignored by the built-in verifier. Additional labels may be registered by user libraries, but are ignored by the built-in verifier.

## 21.3 LaTeX Expression Annotations

A LaTeX expression annotation supplies the preferred mathematical rendering of a symbol for documentation output and IDE display. It does not affect type-checking or proof search.

```ebnf
latex_annotation ::= "@latex" "(" latex_string ")" ;
latex_string     ::= string_literal ;
```

The annotation is placed immediately before a `func`, `pred`, `mode`, or `attr` definition:

```mizar
@latex("n!")
func factorial(n) -> Nat;

@latex("\gcd(a, b)")
func Gcd(a, b) -> Nat;

@latex("a \leq b")
pred leq(a, b);
```

The documentation generator (§24.3) substitutes the LaTeX string wherever the symbol appears in rendered output. If no `@latex` annotation is present, the symbol name is used verbatim.

## 21.4 Reasoning Annotations

Proof search is entirely delegated to ATP backends (§21.7). The ATP receives only local hypotheses and lemmas explicitly cited in `by` clauses — there is no automatic premise selection from the global library. Authors are responsible for citing the required lemmas; `@proof_hint` controls solver choice and resource limits (§21.4.1).

### 21.4.1 `@proof_hint` (solver, axiom budget, and timeout)

`@proof_hint` controls the ATP solver selection and resource limits for the immediately following `by` step, `thus` statement, or `proof ... end` block. Lemma selection is handled via `by` citations (including grouped and bulk forms; §16.5).

```ebnf
proof_hint_annotation ::= "@proof_hint" "(" proof_hint_options ")" ;
proof_hint_options    ::= proof_hint_option { "," proof_hint_option } ;
proof_hint_option     ::= "max_axioms" ":" nat_literal
                        | "timeout"    ":" nat_literal
                        | "solver"     ":" solver_name ;
solver_name           ::= "vampire" | "e" | "cvc5" | "z3" | "auto" ;
```

```mizar
@proof_hint(max_axioms: 32, solver: vampire)
thus result = Eval(t, v) by poly_eval_add, poly_eval_mul;

@proof_hint(timeout: 60, solver: cvc5)
thus complexity_bound(n) by complexity_lemma;
```

**`max_axioms`** caps the total number of axioms sent to the ATP. When the axiom set derived from `by` citations and local hypotheses exceeds this limit, lower-relevance items are pruned.

**`timeout`** sets the per-step ATP time limit in seconds. Overrides the global `atp_timeout` setting in `mizar.pkg` (§23.1) for this step only. Useful for steps known to require extended search, or conversely to fail fast on steps expected to be trivial.

**`solver`** selects the backend prover (§21.4.2). When set to `auto`, the verifier applies its default selection heuristic.

### 21.4.2 Solver Selection

The default solver selection policy is:

1. If the goal involves only linear arithmetic and equality, prefer **CVC5**.
2. If the goal involves higher-order patterns or AC operators, prefer **Vampire** with AC support.
3. Otherwise, try **E** first and fall back to **Vampire** on timeout.

The policy can be overridden globally in the package manifest (§23.1) and locally with `@proof_hint(solver: ...)`. Individual `solver` overrides take precedence over the global setting.

## 21.5 AI/IDE Display Hints

The annotations in this section instruct the verifier or LSP server to surface internal state at a specific source location. They are intended for interactive proof development, debugging, and AI-guided editing. They have no effect on the final verification result.

### 21.5.1 `@show_thesis`

`@show_thesis` requests that the verifier emit the current **proof obligation** (thesis) as a diagnostic at the annotated position. This corresponds to the thesis-tracking mechanism described in §16.9, made accessible at the source level.

```mizar
proof
  @show_thesis
  thus n + 0 = n by add_0_right;
end;
```

The emitted diagnostic has code `info.thesis` and includes the formula that must be proved at that point. It appears in the IDE as an informational annotation (not a warning or error) and is included in `@missing_facts` output (§23.5.2) when the step cannot be discharged.

### 21.5.2 `@show_resolution`

`@show_resolution` requests that the verifier emit the overload resolution result for the immediately following expression. The diagnostic lists the resolved functor or predicate FQN, the template arguments substituted, and any `qua` coercions inserted.

```ebnf
show_resolution_annotation ::= "@show_resolution" ;
```

```mizar
@show_resolution
Product[R](s);
```

Example diagnostic output (code `info.resolution`):

```
resolved functor : algebra.monoid.Product
template args    : R qua MulMagma
result type      : Element of R
inserted coercion: s qua MulMagma
```

This information is always available via LSP hover (§23.6.1); `@show_resolution` makes it visible in the plain-text verifier output for users not running an IDE. When the resolution is unexpected, disambiguation should be achieved in source using qualified names or `qua` coercions.

### 21.5.3 `@show_type`

`@show_type` requests that the verifier emit the inferred type of a specific expression without assigning it to a binding.

```ebnf
show_type_annotation ::= "@show_type" "(" term_expression ")" ;
```

```mizar
@show_type(total + x)
```

The diagnostic (code `info.type`) reports the inferred type of the given expression in the current scope. This is a lightweight debugging tool; in interactive development the LSP hover (§23.6.1) is usually sufficient.

## 21.6 Verification-time Expression Evaluation

### 21.6.1 `@eval`

`@eval` requests that the verifier execute the given expression using the computational content of algorithms in scope and emit the result as a diagnostic. Unlike the `@show_*` display annotations (§21.7), `@eval` actively runs code; it is therefore dropped during code extraction (§20.10) but does execute at verification time.

```ebnf
eval_annotation ::= "@eval" "(" term_expression ")" ;
```

`@eval` may appear in three contexts:

**Module level.** Evaluates a closed (ground) term and reports its value and type.

```mizar
@eval(factorial(10))
:: info.eval: 3628800 : Nat

@eval(euclid_gcd(48, 18))
:: info.eval: 6 : Nat
```

**Inside a proof block.** Evaluates an expression in the current scope. Free variables bound by `let` or `assume` with concrete values are substituted; universally quantified variables without concrete bindings produce a type-only result.

```mizar
proof
  let n be Nat;
  @eval(n + 0)
  :: info.eval: n + 0 : Nat  (no concrete value; n is universally quantified)
  thus n + 0 = n by add_0_right;
end;
```

**Inside an algorithm body.** Evaluates an expression at that program point using the current values of all in-scope variables. This is the primary use case: it functions as a non-blocking watch expression for exploring algorithm behaviour without generating a verification condition.

```mizar
terminating algorithm euclid_gcd(a, b) -> Nat
  ...
do
  while b <> 0 do
    @eval(a)   :: displays current value of a at this loop point
    @eval(b)
    const r := a mod b;
    a := b;
    b := r;
  end;
  return a;
end;
```

The diagnostic code is `info.eval`. In an IDE, `@eval` results appear as inline hints at the annotated position (§23.6.1). In plain-text verifier output they are printed immediately after the annotation site.

> [!NOTE]
> `@eval` requires the expression to be of a computable type (a type for which a registered `algorithm` defines the operation). Expressions involving non-computable mathematical objects (e.g., an arbitrary `Element of R` with no concrete instantiation) produce a `W0003`-class warning and no value output.

## 21.7 ATP Integration

### 21.7.1 Minimum Kernel Architecture and de Bruijn Criterion

Mizar Evolution adopts the **minimum kernel** design: the trusted core of the verifier performs only proof certificate validation and does not include any inference logic. Inference is entirely delegated to external ATPs. This design satisfies the de Bruijn criterion — the size of the trusted code base is minimised, and the correctness of the system does not depend on the correctness of any external prover.

```
┌──────────────────────────────────────┐
│          User Proof Source           │
│  (proof...end / by references)       │
└──────────────┬───────────────────────┘
               │
       ┌───────▼───────┐
       │  Translator   │  ← encodes properties as axioms / AC declarations
       │               │  ← selects lemmas from hint database
       └───────┬───────┘
               │  TPTP / SMT-LIB
       ┌───────▼───────┐
       │  ATP Backend  │  Vampire / E / CVC5
       │  (external)   │
       └───────┬───────┘
               │  proof certificate
       ┌───────▼───────┐
       │  Minimum      │  ← validates certificate only
       │  Kernel       │  ← performs no inference
       └───────────────┘
```

The ATP is treated as an **untrusted oracle**: it produces a proof certificate that the kernel checks independently. A bug or unsoundness in the ATP can produce an incorrect certificate, but the kernel will reject it. Only a bug in the kernel itself can compromise soundness.

> [!IMPORTANT]
> The kernel never trusts an ATP result unconditionally. Every proof obligation that the ATP claims to have proved must be accompanied by a certificate that the kernel can check without invoking the ATP again. An ATP timeout or certificate rejection is reported as an open proof obligation, not as an accepted proof.

### 21.7.2 Problem Translation Pipeline (Translator)

The Translator converts a Mizar proof obligation into a first-order or SMT problem. The pipeline has four stages:

1. **Scope collection.** The Translator collects all hypotheses, `let`-bound variables, and registered properties in scope at the obligation site. It applies the `@proof_hint` constraints (§21.4.1) to restrict the axiom set.

2. **Property encoding.** Registered properties of functors and predicates (§10.6.1, §9.5.1) are translated to axioms or native declarations (§21.7.4). AC operators are declared using backend-native AC mechanisms when available.

3. **Goal translation.** The proof obligation itself is translated as the negated conjecture (for refutation-based ATPs) or as the assertion to be satisfied (for SMT solvers).

4. **Axiom set assembly.** The axiom set sent to the ATP consists of: (a) local hypotheses and `let`-bound equalities in scope; (b) lemmas explicitly cited in the `by` clause (§16.5), including grouped (`module.{a,b}`) and bulk (`module.*`) forms; (c) `invariant` and `decreasing` justifications (§20.2.2). No automatic premise selection from the global library is performed — the author is responsible for citing the required lemmas. The `max_axioms` limit (default: 128; overridable by `@proof_hint`) caps the total axiom count; bulk citations are resolved before the cap is applied.

The output is a self-contained problem file in the format required by the selected backend.

### 21.7.3 Backend Provers (Vampire, E, CVC5)

| Prover | Input format | Strengths | Proof output |
|---|---|---|---|
| **Vampire** | TPTP FOF/TFF | General first-order logic, AC reasoning, large axiom sets | TSTP derivation |
| **E** | TPTP FOF | Equational reasoning, AC completeness | TSTP derivation |
| **CVC5** | SMT-LIB 2 | Linear arithmetic, algebraic datatypes, quantifier-free fragments | LFSC / Alethe certificate |
| **Z3** | SMT-LIB 2 | Linear and nonlinear arithmetic, bit-vectors | SMTLIB proof (optional) |

Backend selection follows the policy in §21.4.2 unless overridden by `@proof_hint(solver: ...)`. Multiple backends may be tried in parallel; the first to return a valid certificate wins.

### 21.7.4 Property Encoding for ATP

#### 21.10.4.1 TPTP Encoding (Vampire, E): Axioms and AC Declarations

Properties that have no native TPTP representation are encoded as first-order axioms:

```tptp
fof(idem_union, axiom, ![A]: (union(A,A) = A)).         % idempotence
fof(invol_neg,  axiom, ![A]: (neg(neg(A)) = A)).         % involutiveness
fof(proj_f,     axiom, ![A,B]: (f(f(A,B),B) = f(A,B))). % projectivity
fof(sym_equiv,  axiom, ![A,B]: (equiv(A,B) => equiv(B,A))).  % symmetry
fof(refl_sub,   axiom, ![A]: (subset(A,A))).             % reflexivity
fof(irrefl_lt,  axiom, ![A]: (~less(A,A))).              % irreflexivity
```

Commutativity and associativity are handled differently. When the target prover supports native AC reasoning, the Translator emits a backend-specific AC declaration instead of axioms:

- **Vampire**: `--ac_reasoning on` flag; the prover normalises AC terms internally.
- **E**: AC completeness is enabled via the `--term-ordering` flag; terms are normalised under AC.

Encoding commutativity as a plain FOF axiom (`f(A,B) = f(B,A)`) causes a quadratic blowup in the search space. The native AC mechanism instead works on AC-normal forms, reducing complexity by orders of magnitude for large algebraic problems.

#### 21.10.4.2 SMT-LIB Encoding (CVC5, Z3)

For SMT backends, properties are encoded as universally quantified assertions:

```smt2
; commutativity
(assert (forall ((a T) (b T)) (= (f a b) (f b a))))
; reflexivity
(assert (forall ((a T)) (P a a)))
; symmetry
(assert (forall ((a T) (b T)) (=> (P a b) (P b a))))
; idempotence
(assert (forall ((a T)) (= (f a a) a)))
```

CVC5 and Z3 integrate these assertions into their congruence closure and quantifier-instantiation engines. For purely linear arithmetic goals, quantified assertions are usually unnecessary because the arithmetic decision procedure handles them directly.

#### 21.10.4.3 Encoding Strategy by Property Type

| Property | TPTP (Vampire/E) | SMT-LIB (CVC5/Z3) | Notes |
|---|---|---|---|
| `commutativity` | **AC declaration** (native) | Quantified assertion | AC native is orders of magnitude faster |
| `associativity` | **AC declaration** (native) | Quantified assertion | Combined with commutativity when both hold |
| `idempotence` | Equality axiom | Quantified assertion | No native mechanism |
| `involutiveness` | Equality axiom | Quantified assertion | No native mechanism |
| `projectivity` | Equality axiom | Quantified assertion | No native mechanism |
| `symmetry` | Implication axiom | Quantified assertion | ATP equality handling is effective |
| `reflexivity` | Universal axiom | Quantified assertion | Interacts with ATP equality |
| `asymmetry` | Implication axiom | Quantified assertion | |
| `connectedness` | Implication axiom | Quantified assertion | |
| `irreflexivity` | Negation axiom | Quantified assertion | |

### 21.7.5 Proof Certificate Formats (TSTP, LFSC/Alethe)

The minimum kernel accepts proof certificates in the following formats:

**TSTP** (used by Vampire and E). A TSTP derivation is a sequence of annotated formulae, each labelled with the inference rule used and the indices of the parent formulae. The kernel reconstructs each inference step independently and checks that the final formula is the negation of the original goal.

**LFSC/Alethe** (used by CVC5). An LFSC certificate is a typed proof term in the Edinburgh Logical Framework. An Alethe certificate is a sequence of SMT proof steps. The kernel has a dedicated checker for each format that validates the term structure and all typing judgements.

**Proof log** (fallback). When a prover cannot emit a structured certificate (e.g., Z3 in some configurations), the ATP interaction is logged as a `*.atp-log` file (§23.5.3). In this mode the proof is marked as **externally attested** rather than kernel-verified. Externally attested proofs are accepted for development but are flagged in the build output and do not satisfy the kernel-verification requirement for release builds.

> [!IMPORTANT]
> The distinction between a kernel-verified proof and an externally attested proof is surfaced in the `*.mizir.json` artifact (§23.5.1) as the `proof_status` field (`"kernel_verified"` vs `"externally_attested"`). A module containing externally attested proofs cannot be published to the package registry without explicit author acknowledgement.

### 21.7.6 Sethood Resolution at Type-Check Level

The `sethood` property (§5.8.1) is resolved at type-check time, before any proof obligation is generated and before the ATP is invoked. This avoids a class of ATP timeouts that would otherwise occur when the Fraenkel set comprehension `{ f(x) where x is T : P(x) }` is used with a type `T` that has no registered sethood witness.

The rule is:

- When `{ f(x) where x is T : P(x) }` appears in a term, the type-checker looks up whether `sethood` is registered for `T`.
- If a registration is found, the type-checker inserts the corresponding existence fact as a background axiom for the enclosing proof obligation.
- If no registration is found, the type-checker emits a **compile-time error** with code `type.sethood.missing` and rejects the term. The error is not deferred to the ATP.

This design keeps sethood reasoning entirely within the type-checker's scope, consistent with the kernel's role as a certificate validator rather than an inference engine.
