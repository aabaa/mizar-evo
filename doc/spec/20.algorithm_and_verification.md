Mizar Evolution integrates **executable algorithms** with the formal proof system. An `algorithm` block defines a computation that can be:

- **Run** by the Mizar Virtual Machine (MVM) during verification (`by computation`).
- **Reasoned about** using Hoare-logic contracts (`requires`, `ensures`, `invariant`).
- **Promoted** to a mathematical functor (`func`) when total correctness is established (via `terminating`).

The design follows the principle of **partial correctness by default**: an algorithm is permitted to diverge unless an explicit termination argument is given. This avoids demanding termination proofs for algorithms that are only ever used on concrete inputs or in bounded contexts.

## 20.1 Algorithm Syntax

### 20.1.1 Algorithm Definition Block

An algorithm must be declared **inside a `definition` block**. The surrounding `let` declarations bind the type context; the algorithm's parameter list selects which of those bindings become formal parameters.

```ebnf
algorithm_def ::= [ "terminating" ] "algorithm" identifier [ "[" schema_params "]" ] "(" [ identifier_list ] ")" [ "->" type_expression ]
                  [ "requires" formula ]
                  [ "ensures"  formula ]
                  [ "decreasing" term_list ]
                  algorithm_body ";" ;

schema_params    ::= identifier { "," identifier } ;
identifier_list  ::= identifier { "," identifier } ;

algorithm_body   ::= "do" algo_statement_list "end" ;
```

The `[ "->" type_expression ]` is optional: algorithms with no return type are **void** (procedures). The optional `[ "decreasing" term_list ]` declares the recursion termination measure (§20.5.2) at the header level; for loop termination see §20.2.2.

**Template algorithms.** `algorithm_def` is a valid `template_item` (§18.2.1). When the enclosing `definition` block declares schema parameters — **type** (`let T be type`), **pred** (`let P be pred(...)`), or **func** (`let F be func(...) -> ...`) — those parameters may be referenced in the algorithm body and in `requires`/`ensures` clauses. The algorithm name takes a `[...]` bracket suffix listing the schema parameters, following the same convention as schema theorems (§18.8.4). Instantiation is performed at each call site by supplying concrete type, pred, or func names in the brackets.

```mizar
definition
  let n be Nat;

  terminating algorithm factorial(n) -> Nat
    ensures result >= 1
    decreasing n
  do
    if n = 0 do
      return 1;
    end;
    return n * factorial(n - 1);
  end;
end;
```

The `terminating` modifier (§20.7.2) declares that the algorithm terminates for all inputs satisfying `requires`. It is optional — algorithms without `terminating` are treated as partially defined and cannot be used as first-class functors.

### 20.1.2 Parameters and Return Type

Parameters are positional and **immutable** within the algorithm body (they behave as `const`). To use a mutable version of a parameter, create a `var` copy:

```mizar
definition
  let x be Nat;

  algorithm copy_and_modify(x) -> Nat
  do
    var y := x;
    y := y + 1;
    return y;
  end;
end;
```

The return type specifies the type of every `return` expression. The special identifier `result` in `ensures` clauses refers to the return value:

```mizar
definition
  let n be Integer;

  algorithm abs_val(n) -> Nat
    ensures result >= 0
  do
    if n >= 0 do
      return n;
    end;
    return -n;
  end;
end;
```

### 20.1.3 Variable Declarations (`var`, `const`)

Inside an algorithm body, variables are introduced with `var` (mutable) or `const` (immutable). **Ghost variables** (`ghost var`, `ghost const`) exist only at the specification level and are erased at extraction time.

```ebnf
var_decl         ::= "var"         var_binding   { "," var_binding   } [ "as" type_expression [ justification ] ] ";" ;
ghost_var_decl   ::= "ghost" "var" var_binding   { "," var_binding   } [ "as" type_expression [ justification ] ] ";" ;
const_decl       ::= "const"       const_binding { "," const_binding } [ "as" type_expression [ justification ] ] ";" ;
ghost_const_decl ::= "ghost" "const" const_binding { "," const_binding } [ "as" type_expression [ justification ] ] ";" ;

var_binding   ::= identifier [ ":=" term ] ;
const_binding ::= identifier ":=" term ;

justification ::= simple_justification | proof ;
(* simple_justification, proof, reasoning, references — defined in §15/§16 *)
```

| Keyword | Mutability | Runtime | Use case |
|---|---|---|---|
| `const` | Immutable | Yes | Intermediate results, captured values, picks |
| `var` | Mutable | Yes | Loop counters, accumulators, state |
| `ghost const` | Immutable | No (spec only) | Proof witnesses, specification values |
| `ghost var` | Mutable | No (spec only) | Proof tracking (visited sets, abstract state) |

**Cast rules between variable kinds.** The four kinds form a hierarchy from most concrete to most abstract. A value of a more concrete kind may be used wherever a more abstract kind is expected; the reverse is forbidden.

| From \ To | `const` | `ghost const` | `ghost var` |
|---|---|---|---|
| `var` | ✓ (current value, read-only) | ✓ | ✓ |
| `const` | — | ✓ | — |
| `ghost var` | ✗ | ✓ (current value) | — |
| `ghost const` | ✗ | — | — |

The upward direction (concrete → ghost) is always safe: reading a runtime value in a specification context does not affect execution. The downward direction (ghost → runtime) is always forbidden: a ghost value has no runtime representation and cannot be assigned to a non-ghost variable or passed to a non-ghost call.

Ghost variables may appear only in **specification contexts**: `invariant`, `ensures`, `requires`, and `assert`. They may not appear on the right-hand side of non-ghost assignments, and may not be passed to non-ghost algorithm calls.

The verifier checks that ghost variables do not influence runtime behaviour. The extractor silently removes all ghost declarations and ghost assignments.

```mizar
definition
  let S be finite set of Integer;

  algorithm find_min(S) -> Integer
    requires S <> {}
    ensures result in S & for x being Integer st x in S holds result <= x
  do
    ghost var visited := {} as set of Integer;
    var current := S;
    var best := the Element of S;
    while current <> {} do
      invariant visited = S \ current;
      invariant best in S & for x being Integer st x in visited holds best <= x;
      decreasing card current;
      const x := the Element of current;
      if x < best do best := x; end;
      visited := visited \/ {x};       :: ghost update — erased at extraction
      current := current \ {x};
    end;
    return best;
  end;
end;
```

The optional `as T` annotation corresponds to Mizar's `reconsider x = e as T`: it asserts that the initializer expression has type `T`. Widening coercions (e.g. `Nat` to `Integer`) and disambiguation of overloaded literals (e.g. `0 as Real`) are resolved automatically by the type hierarchy and require no proof. Narrowing coercions (e.g. `Integer` to `Nat`) generate a proof obligation; if the automatic prover cannot discharge it, supply a justification (`by` or `proof ... end`):

```mizar
var n := some_int as Nat by non_negative_lemma;
var m := some_int as Nat
proof
  some_int >= 0 by ...;
  thus some_int is Nat;
end;
const r := 0 as Real;             :: automatic — no justification needed
```

If the annotation is omitted entirely, the type is inferred from the initializer expression (see [Chapter 8, Type Inference](./08.type_inference.md)).

**Multiple bindings** may appear in a single `var` or `const` statement. A trailing `as T` annotation applies to all bindings in the statement. If individual bindings require different type annotations or justifications, use separate statements.

```mizar
var a := 1, b := 2 as Nat;          :: two vars, both typed Nat
const lo := 0, hi := n - 1 as Nat;   :: two constants, shared type
```

**Uninitialized bindings** (no `:=`) leave the variable without a value; the verifier rejects any use before assignment — except as pattern variables in a `match` statement (§20.2.5), where the `match` itself provides the initial assignment on a successful case:

```mizar
var p, q, r as Element of PolyRing[K,x];  :: uninitialized; valid as match pattern variables
match t do
  case p * (q + r) do  :: p, q, r are assigned here
    ...
  end;
  otherwise
    ...
  end;
end;
```

### 20.1.4 Assignment

Mutable variables are updated with `:=`:

```ebnf
assignment ::= lvalue ":=" term ";" ;
lvalue     ::= identifier
             | lvalue "." identifier ;
```

The right-hand side is any term expression. The left-hand side must be a `var`-declared identifier or a field thereof. Assigning to a `const` identifier, a `let`-bound variable, or a parameter is a **compile-time error**.

Assignments to `ghost var` identifiers use the same `:=` syntax; the verifier determines from the declaration that the target is a ghost variable. Assigning a `ghost var` or `ghost const` value to a non-ghost variable is a **compile-time error** — ghost values have no runtime representation and must not influence non-ghost state.

```mizar
counter := counter + 1;              :: runtime assignment
state.value := state.value + 1;      :: field update
S_visited := S_visited \/ {x};       :: ghost assignment — legal, target is ghost var
counter := S_visited;                :: compile-time error — ghost value on rhs of non-ghost assignment
```

## 20.2 Control Flow

### 20.2.1 Conditional (`if`)

```ebnf
if_stmt ::= "if" formula "do" algo_statement_list if_tail ;
if_tail ::= "end" ";"
          | "else" if_stmt
          | "else" algo_statement_list "end" ";" ;
```


```mizar
if n = 0 do
  return 1;
end;

if n mod 2 = 0 do
  return n div 2;
else
  return 3 * n + 1;
end;

:: if-else-if chain
if n < 0 do
  return -1;
else if n = 0 do
  return 0;
else
  return 1;
end;
```

Both branches must return the same type when `return` is used, or both must fall through to subsequent statements.

### 20.2.2 Bounded Loop (`while`)

```ebnf
while_stmt ::= "while" formula "do"
                 { while_annotation ";" }
                 algo_statement_list
               "end" ";" ;

while_annotation ::= "invariant" formula
                   | "decreasing" term_list ;
```

`while_annotation` clauses (`invariant`, `decreasing`) appear at the **top** of the `while` body, before any other statements.

```mizar
var i := 1;
var s := 0;
while i <= n do
  invariant s = i * (i - 1) div 2 & i >= 1;
  decreasing n - i + 1;
  s := s + i;
  i := i + 1;
end;
```

### 20.2.3 Range Loop (`for ... to`)

```ebnf
for_range_stmt ::= "for" identifier "=" expr ( "to" | "downto" ) expr
                   [ "step" expr ]
                   "do" { for_annotation ";" } algo_statement_list "end" ";" ;

for_annotation ::= "invariant" formula ;
```

The loop variable is immutable within the body and scoped to the loop. Bounds and step are evaluated once before the loop begins. The `step` expression must be a positive `Nat` value; the default step is `1`.

- `for i = 1 to n` — iterates i = 1, 2, …, n (ascending).
- `for i = n downto 1` — iterates i = n, n−1, …, 1 (descending).
- `for i = 0 to n step 2` — iterates i = 0, 2, 4, … (ascending by 2).
- `for i = n downto 0 step 3` — iterates i = n, n−3, … (descending by 3).

```mizar
var s := 0;
for i = 1 to n do
  s := s + i;
end;

for i = 10 downto 1 do
  process(i);
end;

:: Every other element, descending
for i = n downto 1 step 2 do
  handle(i);
end;
```

Termination is **automatically guaranteed** — no `decreasing` annotation is needed. An optional `invariant` clause may be placed at the top of the body to assist verification of the loop's accumulated effect:

```mizar
var s := 0;
for i = 1 to n do
  invariant s = i * (i - 1) div 2;
  s := s + i;
end;
```

### 20.2.4 Collection Loop (`for ... in`)

```ebnf
for_collection_stmt ::= "for" identifier "in" expr [ "processed" identifier ] "do"
                          { for_annotation ";" }
                          algo_statement_list
                        "end" ";" ;
```

Iterates over all elements of a **finite set**. The verifier requires the collection expression to have type `finite set of T` (or a type registered as `finite`). Collections with multiplicity (bags, sequences) are not supported — use `for ... to` over an explicitly ordered structure instead.

**Iteration order is implementation-defined.** The loop makes no guarantee about the order in which elements are visited.

**The `processed` binding.** When the optional `processed V` clause is present, `V` is a **loop-scoped ghost binding** of type `set of T`, managed automatically by the loop:

- Before the first iteration: `V = {}`
- After each iteration of element `x`: the loop implicitly extends `V` with `{x}`
- On normal exit: `V = S` is added to the logical context automatically
- `V` is erased at extraction time (no runtime representation)
- If `processed` is omitted, no tracking binding is created

The user names `V` freely and uses it in `invariant` clauses.

**Invariant verification.** The verifier checks the standard three conditions using the loop-managed `V`:

1. **Establishment** — invariant holds with `V = {}`
2. **Maintenance** — for an arbitrary `x ∈ S \ V`, if the invariant holds before the body, it holds after the body (with `V` extended by `{x}`)
3. **Post-condition** — on normal exit, `V = S` and the invariant give the final result

Because condition 2 uses an **arbitrary** `x ∈ S \ V`, the proof never assumes a particular enumeration order. Order-independence is implicit in the invariant being stated in terms of the set `V`, not a sequence.

**`Fold` template and set functors.** The standard way to express order-independent accumulation is via the **`Fold` template functor** (§18), which abstracts the recursive structure of set-valued computation:

```mizar
:: Conceptual definition of the Fold template (defined in §18 Templates)
:: Fold[op, e](S) for a commutative, associative op with unit e
::
:: func Fold[op, e](S) -> T means
::   it = e if S = {}
::   otherwise
::     ex x st x in S
::      & it = op(x, Fold[op, e](S \ {x})) ;
::
:: uniqueness proof: for x ≠ y in S,
::   op(x, op(y, Fold[op,e](S\{x,y})))
::   = op(y, op(x, Fold[op,e](S\{x,y})))  :: by commutativity of op
::   hence both choices yield the same result.
::   Full proof by induction on card S.
```

The **uniqueness condition of `Fold`** is exactly order-independence: it is proved once at the template level from commutativity and associativity of `op`, and all instantiations inherit it automatically.

Standard library set functors are pre-defined instances of `Fold`:

```mizar
:: Sum(S)              = Fold[+, 0](S)
:: Product(S)          = Fold[*, 1](S)
:: BigUnion(F, S)      = Fold[∪, {}](S)    (element mapped by F)
:: BigIntersection(F, S) = Fold[∩, univ](S) (element mapped by F)
Count(S, P)         = Fold[+, 0](S)     (element mapped to 1 if P holds, else 0)
```

Users who need a custom accumulation define their own `Fold` instance and cite the commutativity/associativity of their operation to discharge the uniqueness obligation.

```mizar
:: Sum of elements — processed V tracks the accumulated set.
var total := 0;
for x in S processed V do
  invariant total = Sum(V);
  total := total + x;
end;
:: After normal exit: V = S (automatic), so total = Sum(S).

:: Product of elements.
var prod := 1;
for x in S processed V do
  invariant prod = Product(V);
  prod := prod * x;
end;

:: Existence check — no accumulator; visited clause not needed.
for x in S do
  if x mod 2 = 0 do
    return x;
  end;
end;
```

If the loop body has genuine order dependence (no commutative characterisation exists), `for ... in` is the wrong construct. Use `for ... to` over an explicitly ordered structure instead.

> [!NOTE]
> If `return` or `break` exits the loop early, the context contains `V c= S` and whatever the invariant states about `V`. The post-condition `V = S` is only guaranteed on **normal** exit.

### 20.2.5 AST Pattern Match (`match`)

`match` performs **structural pattern matching** on the AST of a term. Each `case` describes a syntactic form; when a case matches, subterms are bound to pattern variables declared with `var`. This is fundamentally different from `if`/`else`, which tests boolean predicates: `match` simultaneously tests structure and extracts subterms in a single step.

```ebnf
match_stmt ::= "match" expr "do"
               match_case+
               ( "otherwise" algo_statement_list "end" ";"
               | exhaustiveness_proof )
               "end" ";" ;

match_case           ::= "case" term_pattern "do" algo_statement_list "end" ";" ;
exhaustiveness_proof ::= "exhaustive" [ justification ] ";" ;
term_pattern         ::= term_expression ;
```

**Pattern variables** are `var`-declared identifiers appearing in `term_pattern`. Regardless of any prior assigned value, all such variables are treated as **capture variables**: on a successful match each is overwritten with the corresponding subterm. Value constraints on captured subterms should be expressed with `if` inside the case body.

**Matching is syntactic** (AST-level), not semantic. Algebraic identities such as commutativity are not applied: `p * (q + r)` matches only the exact syntactic form whose outermost operation is `*` and whose right operand is `+`.

**Cases are tested in order**; the first matching case executes. `otherwise` (or a proof of `exhaustive`) is mandatory — omitting both is a **compile-time error**.

**Non-linear patterns**: the same variable may appear more than once in a pattern. All occurrences must match syntactically identical subterms.

**Types of pattern variables** are taken from their `var` declarations. If a `var` is declared without an explicit type annotation, the type is inferred from the matched expression and the operator signatures appearing in the pattern.

**Implicit conversions in patterns** are deliberately restricted. When a pattern variable has an explicit type annotation, the matched subterm must have exactly that type, or admit a **unique structure-inheritance upcast** to it (Chapter 19, §19.2.2). **Cluster-based attribute closure is not used** when checking whether a subterm fits a pattern variable type, and ambiguous inheritance paths are not resolved implicitly — use an explicit `qua` in the scrutinee or choose a less specific pattern-variable type. This keeps pattern matching local, predictable, and aligned with the AST-level nature of `match`.

```mizar
:: Evaluate a polynomial expression at a given point
definition
  let K be Field;
  let x be Variable;
  let t be Element of PolyRing[K,x];
  let v be Element of K;

  algorithm eval_at(t, v) -> Element of K
    ensures result = Eval(t, v)
    decreasing term_size(t)
  do
    var p, q as Element of PolyRing[K,x];
    match t do
      case p + q do
        return eval_at(p, v) + eval_at(q, v) by poly_eval_add;
      end;
      case p * q do
        return eval_at(p, v) * eval_at(q, v) by poly_eval_mul;
      end;
      otherwise
        :: t is atomic (the variable x or a scalar constant)
        return Eval(t, v);
      end;
    end;
  end;
end;

:: Non-linear pattern: matches only when both summands are syntactically identical
match t do
  case p + p do
    return 2 * p by ...;
  end;
  otherwise
    return t;
  end;
end;
```

**Verification conditions.** On entry to a case body the verifier adds a propositional equality as a hypothesis — no justification is required, since the equality is witnessed by the structural match itself. Preceding cases contribute negation hypotheses:

| Branch | Hypotheses added |
|---|---|
| First `case π₁ do` | `e = π₁` (with captured vars bound) |
| Second `case π₂ do` | `¬(∃ capture vars. e = π₁)` and `e = π₂` (with captured vars bound) |
| `otherwise` | negation of all preceding patterns |

> [!NOTE]
> Cases may overlap — the **first** matching case wins.

### 20.2.6 Jump Statements (`return`, `break`, `continue`)

```ebnf
return_stmt   ::= "return" [ expr [ justification ] ] ";" ;
break_stmt    ::= "break" ";" ;
continue_stmt ::= "continue" ";" ;
```

| Statement | Effect | Verification obligation |
|---|---|---|
| `return expr` | Exit algorithm with value `expr` | `expr` must satisfy `ensures`; preconditions of any tail call are discharged |
| `return expr by ...` | Shorthand for `assert` + `return` (§20.4.2) | Same as `return expr`, with the justification applied to the `ensures` discharge |
| `return` | Exit a void algorithm | All `ensures` conditions must hold |
| `break` | Exit the immediately enclosing loop | Loop `invariant` must hold at break point |
| `continue` | Skip to the next loop iteration | Loop `invariant` must hold; **`decreasing` measure must have decreased** relative to the start of the current iteration |

**`continue` and termination.** Because `continue` bypasses the end of the loop body where the termination check normally occurs, the verifier checks that the `decreasing` measure has **already decreased** at the `continue` point. If no `decreasing` annotation is present, `continue` generates no termination obligation (the loop is treated as potentially diverging).

```mizar
:: continue: the decreasing measure must have already decreased at the continue point.
while i <= n do
  invariant s = sum_of_odds_up_to(i - 1);
  decreasing n - i + 1;
  if i mod 2 = 0 do
    i := i + 1;
    continue;  :: n - (i + 1) + 1 < n - i + 1 already holds before continue
  end;
  s := s + i;
  i := i + 1;
end;
```

## 20.3 Non-determinism (Pick)

A **Pick** operation selects an arbitrary element from a non-empty set, expressed using Mizar's existing `the` term forms. These are not new Chapter 20-specific expressions: they remain part of the general `term_expression` syntax (Chapter 13), exactly as in classical Mizar. In this chapter we use the name **Pick** only for the algorithmic semantics of Mizar's existing `the`-expressions.

```mizar
const x := the Element of S;    :: pick any element of S
const p := the prime Number;     :: pick any prime
```

**Semantics.** Both forms are treated existentially by the verifier — as *some* element satisfying the membership or type predicate — without fixing a specific choice. The MVM selects one element non-deterministically at runtime.

**Proof safety.** The contracts of an algorithm (`requires`, `ensures`) are the only interface visible to the surrounding proof system. An `ensures` clause that mentions a Pick result must hold for **every** possible choice; the verifier generates a proof obligation to that effect. Properties not stated in `ensures` cannot be assumed by callers regardless of the choice made. This containment guarantees that non-determinism inside an algorithm cannot silently weaken external proofs.

**Non-emptiness obligations.** The verifier generates a non-emptiness obligation at each Pick site, but its form differs by variant:

- *Set-based* (`the Element of S`): the obligation `S <> {}` is a verification condition discharged by proof. If it cannot be discharged statically, the MVM raises a **runtime verification failure** (§20.9.3).
- *Type-based* (`the prime Number`): the obligation is that the type is **inhabited** — i.e., there exists at least one term of that type. This is a standard Mizar type-inhabitation obligation discharged statically; no runtime check is needed.

```mizar
definition
  let S be finite set of Integer;

  terminating algorithm find_even(S) -> Integer
    ensures result = -1 or (result in S & result mod 2 = 0)
  do
    var current := S;
    while current <> {} do
      invariant current c= S;
      decreasing card current;
      const x := the Element of current;
      if x mod 2 = 0 do
        return x;
      end;
      current := current \ {x};
    end;
    return -1;
  end;
end;
```

> [!NOTE]
> Whether a Pick expression can be executed by the MVM depends not on finiteness but on whether the MVM can produce a concrete witness at runtime:
> - `the Element of S` where `S` is a finite set known at runtime: the MVM can enumerate `S` and select an element — **executable**.
> - `the prime Number`: if the Mizar library registers a computable canonical representative (e.g., `2`), the MVM can use it — **executable**.
> - A type or set for which no algorithm can produce a concrete element in finite time: **not executable**. For example, `the Element of S` where membership in `S` is undecidable.
>
> Non-executable Pick expressions are still mathematically sound in algorithm definitions and may be used in proofs where only the logical specification (contracts) matters. An algorithm containing a non-executable Pick cannot be promoted to `terminating`.

## 20.4 Contracts (`requires`, `ensures`, `assert`)

### 20.4.1 Preconditions and Postconditions

- `requires P` — `P` must hold at every call site. The algorithm body may assume `P`.
- `ensures Q` — `Q` holds upon normal termination. `result` in `Q` refers to the return value.

Each clause appears at most once, in the fixed order `requires` → `ensures` → `decreasing`. Multiple conditions within a clause are expressed using `&` or `and`:

```mizar
definition
  let a, b be Real;

  algorithm divide(a, b) -> Real
    requires b <> 0
    ensures result * b = a
  do
    return a / b;
  end;
end;
```

Contracts are **proof obligations**:
- At each call site `divide(x, y)`, the verifier demands a proof that `y <> 0`.
- After `return v`, the verifier demands a proof that `v * b = a`.

If the automatic prover cannot discharge an obligation, the user adds an `assert` (§20.4.2) with a justification (`by` or `proof ... end`) at the relevant site.

### 20.4.2 Inline Assertions

```ebnf
assert_stmt ::= "assert" formula [ justification ] ";" ;
```

`assert P` inserts a proof checkpoint: the verifier must establish `P` at that program point, using the current logical context plus any cited lemmas or an inline proof.

**`return expr by ...` — inline assertion shorthand.**
When a `return` carries a justification, it is syntactic sugar for an `assert` that discharges the `ensures` postcondition for `expr`, followed by the `return`:

```mizar
  ensures result = a
do
  ...
  return b by th1;   :: sugar for: assert a = b by th1; return b;
  ...
```

The asserted formula is the `ensures` clause with `result` substituted by the returned expression.  Using a full `proof ... end` block is also allowed:

```mizar
  return b
  proof
    ...
  end;
```

```mizar
definition
  let arr be FinSequence of Real;
  let lo, hi, target be Integer;

  algorithm binary_search(arr, lo, hi, target) -> Integer
  do
    var l := lo;
    var r := hi;
    while l <= r do
      invariant lo <= l & r <= hi;
      decreasing r - l + 1;
      const mid := (l + r) div 2;
      assert l <= mid & mid <= r by arithmetic;
      if arr.mid = target do
        return mid;
      else if arr.mid < target do
        l := mid + 1;
      else
        r := mid - 1;
      end;
    end;
    return -1;
  end;
end;
```

### 20.4.3 Verification Obligations

| Construct | Obligation |
|---|---|
| Call site `f(args)` | `requires` preconditions of `f`, instantiated with `args` |
| `return expr` | `ensures` postconditions, with `result` bound to `expr` |
| `assert P` | `P` holds at that program point |
| `break` inside loop | loop `invariant` holds at break point |
| `continue` inside loop | loop `invariant` holds at continue point; `decreasing` measure has already decreased relative to start of current iteration |
| `for x in S` with mutable state | loop `invariant` must be stated in terms of a set-based functor (via `ghost var`), proving result is order-independent |
| `for x in S` | `S` is `finite` |
| `the Element of S` | `S <> {}` |
| `decreasing e` (loop) | `e` is `Nat`-valued; `e` decreases strictly on each loop iteration |
| `decreasing e` (header) | `e` is `Nat`-valued; `e` decreases strictly on each recursive call |
| `terminating` | all `decreasing` measures are valid; algorithm returns on all inputs satisfying `requires` |

## 20.5 Loop Verification

### 20.5.1 Loop Invariants (`invariant`)

A loop invariant must hold:
1. **Before** the first iteration (establishment).
2. **At the loop head on every subsequent iteration** (maintenance).
3. **On normal exit** — when the loop condition becomes false.
4. **At every `break` statement** — the invariant must hold at the break point.
5. **At every `continue` statement** — the invariant must hold at the continue point (the loop will re-enter from the top).

```mizar
var i := 1;
var s := 0;
while i <= n do
  invariant s = (i - 1) * i div 2;
  decreasing n - i + 1;
  s := s + i;
  i := i + 1;
end;
:: After loop: i = n + 1, s = n * (n + 1) div 2
```

Multiple `invariant` clauses are interpreted as a conjunction.

### 20.5.2 Termination Measure (`decreasing`)

```ebnf
(* loop termination — inside while body, via while_annotation *)
(* while_annotation ::= "invariant" formula | "decreasing" term_list ; *)

(* recursion termination — in algorithm header *)
(* algorithm_def ::= ... [ "decreasing" term_list ] algorithm_body ";" ; *)

term_list ::= term_expression { "," term_expression } ;
```

The `decreasing` annotation is used in two distinct positions:

- **Algorithm header** (`[ "decreasing" term_list ]` in `algorithm_def`) — declares the recursion termination measure; the tuple must decrease on every recursive call.
- **`while` body** (`while_annotation`) — declares the loop termination measure; the tuple must decrease on every iteration.

The `for` loop does **not** accept `decreasing` — iteration over a finite set is guaranteed to terminate by the finiteness obligation.

In both positions the verifier generates the same obligations:

1. Every expression in the list is **`Nat`-valued** (non-negative integer).
2. The tuple **decreases lexicographically** on every loop iteration or recursive call.

| Form | Ordering | Example |
|---|---|---|
| `decreasing e` | `<` on `Nat` | `decreasing n - i` |
| `decreasing e, f` | Lexicographic `<` on `Nat × Nat` | `decreasing m, n` |

For a custom well-founded ordering, map the measure to `Nat` via a user-defined functor before using `decreasing`:

```mizar
:: User-defined measure mapping a tree to its depth (a Nat)
const d := tree_depth(t);
decreasing d;
```

The non-negativity obligation (`e >= 0`) is discharged at the definition site. If the automatic prover cannot establish it, add an `assert`:

```mizar
assert n - i >= 0 by arithmetic;
decreasing n - i;
```

### 20.5.3 Structural Recursion Measure (`term_size`)

For algorithms that recurse on syntactic subterms via `match`, the language provides a built-in ghost functor `term_size`:

```ebnf
term_size_expr ::= "term_size" "(" expr ")" ;
```

`term_size(t)` is a `ghost const` of type `Nat`. Its concrete value is unspecified — it is an abstract well-founded measure satisfying one axiom family: for any binary operator `⊕`, if a `match` case captures `p` and `q` from `t` via the pattern `p ⊕ q`, then the verifier automatically adds the hypotheses

```
term_size(p) < term_size(t)
term_size(q) < term_size(t)
```

to the case body. No user proof is required for these inequalities; they are witnessed by the structural match itself.

Because `term_size` is a `ghost const`, it may only appear in specification contexts (`decreasing`, `invariant`, `ensures`, `requires`, `assert`). It cannot be passed to non-ghost calls or assigned to non-ghost variables.

```mizar
algorithm eval_at(t, v) -> Element of K
  ensures result = Eval(t, v)
  decreasing term_size(t)    :: ghost measure; verifier knows term_size(p) < term_size(t)
do
  var p, q as Element of PolyRing[K,x];
  match t do
    case p + q do
      :: hypotheses available: term_size(p) < term_size(t),  term_size(q) < term_size(t)
      return eval_at(p, v) + eval_at(q, v) by poly_eval_add;
    end;
    case p * q do
      return eval_at(p, v) * eval_at(q, v) by poly_eval_mul;
    end;
    otherwise
      return Eval(t, v);
    end;
  end;
end;
```

`term_size` is the only built-in AST-level ghost functor. It does not expose the structure of a term beyond what `match` already makes available. If a future use case requires finer AST introspection (e.g., traversal algorithms that need node counts or depths as runtime values), additional ghost functors may be introduced at that time.

## 20.6 Snapshot and Claim

### 20.6.1 Snapshots

A **snapshot** records the complete local state of an algorithm at a specific program point. It is a specification-only construct: snapshots are erased at code extraction (§20.10) and incur no runtime cost.

```ebnf
snapshot_stmt ::= "snapshot" identifier ";" ;
```

A snapshot captures all bindings visible at that point — `var`, `const`, `ghost var`, `ghost const`, and algorithm parameters — under a single name. The snapshot name must be unique within the algorithm body.

### 20.6.2 Claim Blocks

A `claim` block groups theorems about the snapshot states of a single algorithm, expressing internal verification properties that go beyond what `requires`/`ensures` contracts alone can state.

```ebnf
claim_block ::= "claim" identifier "do"
                  { theorem_item }
                "end" ";" ;
```

The identifier names the algorithm whose snapshots are brought into scope. Inside the block, snapshot variables are accessed as `S.x` (snapshot `S`, variable `x`).

**Universal quantification.** Every theorem in a `claim` block is implicitly universally quantified over all valid executions — that is, over all calls satisfying the algorithm's `requires` precondition.

**Snapshot semantics.** A snapshot captures the complete local state: all `var`, `const`, `ghost var`, and `ghost const` bindings visible at that program point, together with the algorithm parameters.

### 20.6.3 Example

The following algorithm counts from `0` to `n`. The snapshot `Init` captures the state before the loop; snapshot `S` captures the state at the top of each iteration.

```mizar
definition
  let n be Nat;

  algorithm count_up(n)
    requires n >= 0
    ensures result = n
  do
    var x := 0;
    var i := 0;
    snapshot Init;
    while i < n do
      snapshot S;
      x := x + 1;
      i := i + 1;
    end;
    return x;
  end;
end;

claim count_up do
  :: state before the loop
  theorem InitState:
    Init.x = 0 & Init.i = 0;

  :: loop invariant: accumulator equals iteration counter at each step
  theorem Progress:
    S.x = S.i;
end;
```

`InitState` states that `x` and `i` are both zero before the loop begins. `Progress` states that at every iteration step the accumulator `x` equals the counter `i` — an internal property that `ensures result = n` alone does not capture.

## 20.7 Termination Proofs

### 20.7.1 Declaring a Termination Measure

For a recursive algorithm, `decreasing` is placed in the **algorithm header**, between the contracts and the body:

```mizar
definition
  let m, n be Nat;

  terminating algorithm ackermann(m, n) -> Nat
    decreasing m, n
  do
    if m = 0 do
      return n + 1;
    end;
    if n = 0 do
      return ackermann(m - 1, 1);
    end;
    return ackermann(m - 1, ackermann(m, n - 1));
  end;
end;
```

The verifier checks that on each recursive call the tuple `(m, n)` decreases lexicographically:

- `ackermann(m - 1, 1)` — first component `m - 1 < m`. ✓
- `ackermann(m, n - 1)` — first components equal, second `n - 1 < n`. ✓
- `ackermann(m - 1, ackermann(m, n - 1))` — first component `m - 1 < m`. ✓

### 20.7.2 Terminating Algorithms and Promotion to `func`

An algorithm prefixed with `terminating` declares that it terminates for every input satisfying its `requires` precondition. This makes the algorithm usable everywhere a `func` is expected — not only in `by computation` proofs, but in arbitrary formal reasoning.

See §20.1.1 for the full `algorithm_def` grammar.

```mizar
definition
  let n be Nat;

  terminating algorithm factorial(n) -> Nat
    decreasing n
  do
    if n = 0 do
      return 1;
    end;
    return n * factorial(n - 1);
  end;
end;
```

Once `terminating` is declared and verified, `factorial` may be used as a functor:

```mizar
theorem : factorial(0) = 1;
theorem : factorial(n + 1) = (n + 1) * factorial(n);
```

> [!NOTE]
> Without `terminating`, the algorithm exists only as a computable procedure. It can be called with `by computation` for concrete values, but it cannot be universally quantified over or used as a first-class functor in proofs.

## 20.8 Recursive Definitions

### 20.8.1 Direct Recursion

An algorithm may call itself recursively. The verifier accepts recursive calls without additional annotation, but requires `decreasing` to prove termination. Without `decreasing`, the algorithm is treated as partially defined.

```mizar
definition
  let n be Nat;

  terminating algorithm fibonacci(n) -> Nat
    decreasing n
  do
    if n = 0 do return 0; end;
    if n = 1 do return 1; end;
    return fibonacci(n - 1) + fibonacci(n - 2);
  end;
end;
```

Each recursive call site generates a proof obligation that the `requires` preconditions are satisfied.

### 20.8.2 Mutual Recursion

Two or more algorithms may call each other mutually. They are placed within the **same `definition` block**; no special declaration is needed — the verifier detects mutual recursion automatically.

`boolean` is the built-in truth-value type; its values are `true` and `false`.

```mizar
definition
  let n be Nat;

  terminating algorithm is_even(n) -> boolean
    decreasing n
  do
    if n = 0 do return true; end;
    return is_odd(n - 1);
  end;

  terminating algorithm is_odd(n) -> boolean
    decreasing n
  do
    if n = 0 do return false; end;
    return is_even(n - 1);
  end;
end;
```

The `decreasing` measures must collectively establish termination across the call cycle. The verifier checks that no infinite call sequence can occur.

## 20.9 Execution Model and Computability

### 20.9.1 The Mizar Virtual Machine (MVM)

The **Mizar Virtual Machine (MVM)** evaluates algorithm expressions during verification. It is invoked when:

- A `by computation` proof step is used (§20.9.2).
- An algorithm is called with fully concrete (ground) arguments in a verification context.

The MVM uses a **call-by-value** model. Side effects are confined to local `var` state; there is no global mutable state.

### 20.9.2 `by computation` Proof Step

```ebnf
computation_proof  ::= "by" "computation" [ "(" computation_option { "," computation_option } ")" ] ;
computation_option ::= "steps"   ":" nat_literal
                     | "timeout" ":" nat_literal
                     | "nest"    ":" nat_literal ;
```

Options:

| Option | Default | Meaning |
|---|---|---|
| `steps` | `0` (unlimited) | Maximum number of MVM execution steps |
| `timeout` | `0` (unlimited) | Wall-clock time limit in milliseconds |
| `nest` | `0` (unlimited) | Maximum function call stack depth |

```mizar
theorem : factorial(5) = 120 by computation;

theorem Fact10 : factorial(10) = 3628800
proof
  thus thesis by computation(steps: 100000);
end;
```

**What `by computation` can prove:**

| Goal form | MVM behaviour |
|---|---|
| `f(v₁, …, vₙ) = c` (ground equality) | Evaluate `f(v₁, …, vₙ)`, check result equals `c` |
| `P(v₁, …, vₙ)` (ground predicate) | Evaluate `P`, check it holds |
| Non-ground formulas | **Not supported** — requires classical proof |

For existential goals of the form `ex x st P(x)`, the recommended approach is to implement a search algorithm explicitly:

```mizar
terminating algorithm find_witness(S) -> Integer
  requires S <> {}
  ensures result in S & P(result)
do
  ...
end;
```

The `ensures` clause then provides the existential witness within the proof system, without requiring special MVM search semantics.

### 20.9.3 Computability and Runtime Failures

Algorithm definitions may reference non-constructive operations (non-computable functions, Pick on abstract sets, etc.). Such definitions are mathematically valid, but the MVM cannot execute them.

When the MVM encounters an unexecutable operation, it raises a **runtime verification failure**:

| Operation | Runtime failure condition |
|---|---|
| `the Element of S` | MVM cannot enumerate `S` (e.g., `S` is infinite and has no registered enumerator) |
| Call to `func f(x)` with no `algorithm` body | `f` has no computable implementation |
| Call to `pred P(x)` or `func F(x)` template parameter | Concrete instantiation has no computable implementation (e.g., a `means`-defined functor) |
| Division by zero (`a / 0`) | Zero denominator |

A runtime failure causes the `by computation` step to **fail** — but does not invalidate the algorithm's definition. The definition remains mathematically meaningful; only its executability via the MVM is limited.

> [!NOTE]
> `the Element of S` where `S` is a mathematically valid infinite set (e.g., `NAT`) is not an error at the definition level. The MVM simply cannot execute it. The algorithm can still be used in classical proofs through its contracts.

### 20.9.4 Gas Limit

To prevent diverging algorithms from hanging the verifier, the MVM enforces a **gas limit**: a maximum number of execution steps per `by computation` invocation.

```mizar
theorem : hard_computation(42) = 123456789
proof
  thus thesis by computation(steps: 1000000);
end;
```

Setting any option to `0` disables that limit (default). Options may be combined:

```mizar
thus thesis by computation(steps: 500000, timeout: 2000, nest: 200);
```

## 20.10 Code Extraction

Code extraction from verified algorithms is provided as a **separate tool**, not part of the core language specification. The extractor takes `terminating` algorithms and produces executable code in a target language (Python, C with GMP, Rust, etc.), preserving all formally verified properties by construction.

Configuration such as target language, numeric type mapping (e.g., `Nat` → GMP `mpz_t` in C), and host-library bindings is supplied via an extractor configuration file rather than inline annotations.

Refer to the extraction tool's documentation for details.

## 20.11 Complete Syntax (EBNF)

```ebnf
(* ── Algorithm definition ─────────────────────────────── *)
algorithm_def ::= [ "terminating" ] "algorithm" identifier [ "[" schema_params "]" ] "(" [ identifier_list ] ")" [ "->" type_expression ]
                  [ "requires" formula ]
                  [ "ensures"  formula ]
                  [ "decreasing" term_list ]
                  algorithm_body ";" ;

identifier_list   ::= identifier { "," identifier } ;

algorithm_body    ::= "do" algo_statement_list "end" ;

(* ── Statement list ────────────────────────────────────── *)
algo_statement_list ::= { algo_statement } ;

algo_statement ::= var_decl
                 | const_decl
                 | ghost_var_decl
                 | ghost_const_decl
                 | assignment
                 | ghost_assignment
                 | if_stmt
                 | while_stmt
                 | for_range_stmt
                 | for_collection_stmt
                 | match_stmt
                 | return_stmt
                 | break_stmt
                 | continue_stmt
                 | assert_stmt
                 | snapshot_stmt ;

(* ── Declarations and assignment ───────────────────────── *)
var_decl         ::= "var"         var_binding   { "," var_binding   } [ "as" type_expression [ justification ] ] ";" ;
ghost_var_decl   ::= "ghost" "var" var_binding   { "," var_binding   } [ "as" type_expression [ justification ] ] ";" ;
const_decl       ::= "const"       const_binding { "," const_binding } [ "as" type_expression [ justification ] ] ";" ;
ghost_const_decl ::= "ghost" "const" const_binding { "," const_binding } [ "as" type_expression [ justification ] ] ";" ;

var_binding   ::= identifier [ ":=" term ] ;
const_binding ::= identifier ":=" term ;

assignment       ::= lvalue ":=" term ";" ;
ghost_assignment ::= "ghost" lvalue ":=" term ";" ;
lvalue           ::= identifier
                   | lvalue "." identifier ;

justification ::= simple_justification | proof ;
(* simple_justification, proof, reasoning, references — defined in §15/§16 *)

(* ── Control flow ──────────────────────────────────────── *)
if_stmt ::= "if" formula "do" algo_statement_list if_tail ;
if_tail ::= "end" ";"
          | "else" if_stmt
          | "else" algo_statement_list "end" ";" ;

while_stmt ::= "while" formula "do"
                 { while_annotation ";" }
                 algo_statement_list
               "end" ";" ;

while_annotation ::= "invariant" formula
                   | "decreasing" term_list ;

term_list ::= term_expression { "," term_expression } ;

for_range_stmt ::= "for" identifier "=" expr ( "to" | "downto" ) expr
                   [ "step" expr ]
                   "do" { for_annotation ";" } algo_statement_list "end" ";" ;

for_collection_stmt ::= "for" identifier "in" expr [ "processed" identifier ] "do"
                          { for_annotation ";" }
                          algo_statement_list
                        "end" ";" ;

for_annotation ::= "invariant" formula ;

match_stmt ::= "match" expr "do"
               match_case+
               ( "otherwise" algo_statement_list "end" ";"
               | exhaustiveness_proof )
               "end" ";" ;

match_case           ::= "case" term_pattern "do" algo_statement_list "end" ";" ;
term_pattern         ::= term_expression ;
exhaustiveness_proof ::= "exhaustive" [ justification ] ";" ;

return_stmt   ::= "return" [ expr [ justification ] ] ";" ;
break_stmt    ::= "break" ";" ;
continue_stmt ::= "continue" ";" ;
assert_stmt   ::= "assert" formula [ justification ] ";" ;
snapshot_stmt ::= "snapshot" identifier ";" ;

(* ── Snapshot and Claim ────────────────────────────────────── *)
claim_block ::= "claim" identifier "do"
                  { theorem_item }
                "end" ";" ;

(* snapshot_access inside claim_block theorems:                  *)
(*   snapshot_name "." identifier                                 *)

(* ── Proof step ─────────────────────────────────────────────
   Used in proof blocks (outside algorithm bodies) as:
     "thus" formula "by" "computation" ...
   or in short-form theorems:
     theorem : formula "by" "computation" ... ";"              *)
computation_proof   ::= "by" "computation"
                        [ "(" computation_option { "," computation_option } ")" ] ;
computation_option  ::= "steps"   ":" nat_literal
                      | "timeout" ":" nat_literal
                      | "nest"    ":" nat_literal ;

(* ── Pick expression ─────────────────────────────────────────
   Used as a term on the right-hand side of const_decl or var_decl
   inside an algorithm body:
     const x := the Element of S ;
     const p := the prime Number ;                              *)
pick_expr ::= "the" type_expression ;
```

## 20.12 Extended Examples

### Fast Exponentiation (Exponentiation by Squaring)

Demonstrates recursion with a non-trivial termination measure and a mathematical `ensures` clause verified by `by computation`.

```mizar
definition
  let x be Nat;
  let n be Nat;

  terminating algorithm fast_power(x, n) -> Nat
    ensures result = x |^ n
    decreasing n
  do
    if n = 0 do
      return 1;
    end;
    if n mod 2 = 0 do
      const half := fast_power(x, n div 2);
      return half * half;
    end;
    return x * fast_power(x, n - 1);
  end;
end;

:: Verified by computation for concrete inputs
theorem : fast_power(2, 10) = 1024 by computation;
theorem : fast_power(3, 5)  = 243  by computation;

:: Promoted to func via terminating; usable in general proofs
theorem fast_power_correct :
  for x, n being Nat holds fast_power(x, n) = x |^ n;
```

### Integer Factorisation (Trial Division)

```mizar
definition
  let n be Nat;

  algorithm trial_factor(n) -> Nat
    requires n >= 2
    ensures result >= 2 & result divides n
  do
    var d := 2;
    while d * d <= n do
      invariant d >= 2 & for k being Nat st 2 <= k & k < d holds not k divides n;
      assert n - d * d >= 0 by arithmetic;
      decreasing n - d * d + 1;
      if n mod d = 0 do
        return d;
      end;
      d := d + 1;
    end;
    return n;   :: n is prime
  end;
end;
```

### Euclidean GCD Algorithm

The algorithm is named `euclid_gcd` to distinguish it from the mathematical functor `Gcd` defined in the standard library. Both the `ensures` clause and the loop invariant use `Gcd` (mathematical) — not the algorithm itself — so there is no circularity.

```mizar
definition
  let a, b be Nat;

  terminating algorithm euclid_gcd(a, b) -> Nat
    requires a >= 1 & b >= 1
    ensures result = Gcd(a, b)
  do
    var x := a;
    var y := b;
    while y <> 0 do
      invariant x >= 1 & y >= 0 & Gcd(a, b) = Gcd(x, y);
      decreasing y;
      const r := x mod y;
      x := y;
      y := r;
    end;
    return x;
  end;
end;

theorem : euclid_gcd(12, 8) = 4 by computation;
theorem : euclid_gcd(100, 75) = 25 by computation;
```

### Symbolic Differentiation

Demonstrates `match` on an expression AST, structural recursion with `term_size`, and `return ... by` shorthand. `Expr[K, x]` is an arithmetic expression type over field `K` and variable `x`. `Diff(t, x)` is the mathematical (formal derivative) functor defined in the supporting theory.

```mizar
definition
  let K be Field;
  let x be Variable;
  let t be Expr[K, x];

  terminating algorithm diff(t) -> Expr[K, x]
    ensures result = Diff(t, x)
    decreasing term_size(t)
  do
    var p, q as Expr[K, x];
    match t do
      case p + q do
        :: D(p + q) = D(p) + D(q)
        return diff(p) + diff(q) by diff_add;
      end;
      case p * q do
        :: D(p * q) = D(p) * q + p * D(q)
        return diff(p) * q + p * diff(q) by diff_mul;
      end;
      case x do
        :: D(x) = 1
        return 1 by diff_var;
      end;
      otherwise
        :: t is a constant (no occurrence of x); D(c) = 0
        return 0 by diff_const;
      end;
    end;
  end;
end;

:: Spot-check on concrete expressions
theorem : diff(x * x) = x + x by computation;           :: D(x^2) = 2x
theorem : diff(x * x + x) = x + x + 1 by computation;
```


## 20.13 Logical Encoding

This section specifies how algorithm constructs map to logical formulas in the Mizar kernel. It complements the operational description in §20.4–§20.7 with the formal axioms generated for `terminating` algorithms and the verification-condition (VC) schema for each statement form.

### 20.13.1 Contract Axioms

Every algorithm call site generates proof obligations from `requires`, and every `return` generates obligations from `ensures`. The table below gives the FOL encoding. Type parameters of the enclosing `definition` block are universally quantified and appear as type guards `is_A(x)`.

| Construct | FOL Encoding |
|---|---|
| `requires P` | Proof obligation at each call site: `Γ ⊢ P(a₁, …, aₙ)` |
| `ensures Q` at `return e` | Proof obligation: `Γ ⊢ Q[result/e]` |
| `ensures Q` (caller side, partial algorithm) | Available only if the call is known to terminate; no unconditional axiom is emitted |
| `assert φ` | Proof obligation: `Γ ⊢ φ` at that program point; `φ` is added to `Γ` afterwards |
| call `f(a₁, …, aₙ)` | (1) `Γ ⊢ requires_f(a₁, …, aₙ)`; (2) `ensures_f(a₁, …, aₙ, f(a₁, …, aₙ))` is added to `Γ` |

Multiple `requires` and `ensures` clauses are conjoined.

### 20.13.2 Promotion Axioms for `terminating`

When `terminating` is declared and verified, the algorithm is promoted to a `func` and the following axioms are emitted into the Mizar kernel:

| Axiom | FOL Schema |
|---|---|
| **Totality** | `∀x₁,…,xₙ. (is_A₁(x₁) ∧ … ∧ P(x₁,…,xₙ) → is_T(f(x₁,…,xₙ)))` |
| **Contract** | `∀x₁,…,xₙ. (is_A₁(x₁) ∧ … ∧ P(x₁,…,xₙ) → Q(x₁,…,xₙ, f(x₁,…,xₙ)))` |

If `requires` is absent, `P` defaults to `⊤` — the functor is total on all inputs of the declared types. Multiple `ensures` clauses are conjoined into a single `Q`.

**Example** — `factorial`:

```
terminating algorithm factorial(n) -> Nat
  ensures result >= 1
  decreasing n
```

Emitted axioms:

| Axiom | Formula |
|---|---|
| Totality | `∀n. (is_Nat(n) → is_Nat(factorial(n)))` |
| Contract | `∀n. (is_Nat(n) → factorial(n) ≥ 1)` |

These axioms become available to the proof system as soon as `terminating` is verified. The algorithm body is not unfolded — further properties are established by explicit theorems (`theorem factorial(0) = 1 by computation;` etc.).

### 20.13.3 Verification Condition Schema

The verifier traverses the algorithm body and generates VCs by a standard Hoare-logic schema. `Γ` denotes the current logical context (accumulated facts at that program point).

| Statement | VC generated | Context after |
|---|---|---|
| `var x := e` | — | `Γ ∪ { x = e, is_inferred_T(x) }` |
| `var x := e as T` | `Γ ⊢ is_T(e)` (auto or via justification) | `Γ ∪ { x = e, is_T(x) }` |
| `const x := e` | — | `Γ ∪ { x = e }` |
| `x := e` | — | `Γ[x ↦ e]` |
| `ghost x := e` | — (spec only) | `Γ_ghost ∪ { x = e }` |
| `return e` | `Γ ⊢ ensures[result/e]` | — (exits) |
| `return` (void) | `Γ ⊢ ensures` | — (exits) |
| `assert φ` | `Γ ⊢ φ` | `Γ ∪ { φ }` |
| call `f(args)` | `Γ ⊢ requires_f(args)` | `Γ ∪ { ensures_f(args, result) }` |
| `if C do S₁ end` | verify `S₁` under `Γ ∪ {C}`; verify tail under `Γ ∪ {¬C}` | `Γ ∪ { ¬C }` (fall-through) |
| `if C do S₁ else S₂ end` | verify `S₁` under `Γ ∪ {C}`; verify `S₂` under `Γ ∪ {¬C}` | join of exit contexts |
| `while C do invariant I; decreasing V; S end` | (1) `Γ ⊢ I` (establishment); (2) `I ∧ C ⊢ wp(S, I)` (maintenance); (3) `I ∧ C ⊢ is_Nat(V)` (non-negativity); (4) `I ∧ C ⊢ V' < V` (decrease) | `Γ ∪ { I ∧ ¬C }` |
| `break` inside loop | `Γ ⊢ I` | — (exits loop) |
| `continue` inside loop | `Γ ⊢ I`; `Γ ⊢ V' < V` | — (re-enters loop head) |
| `for i = a to b do invariant I; S end` | (1) `Γ ⊢ I[i/a]` (establishment); (2) `I ∧ a ≤ i ≤ b ⊢ wp(S, I[i/i+1])` (maintenance); verify `S` under `Γ ∪ {I, a ≤ i ≤ b}` | `Γ ∪ { I ∧ i > b }` |
| `for x in S processed V do invariant I; S_body end` | (1) `Γ ⊢ I[V/∅]` (establishment); (2) `I ∧ x ∈ S ∖ V ⊢ wp(S_body, I[V ↦ V ∪ {x}])` (maintenance, for arbitrary `x`) | `Γ ∪ { I ∧ V = S }` |
| `for x in S do S_body end` (no `processed`) | verify `S_body` under `Γ ∪ { x ∈ S }` | `Γ` |
| `match e do case π₁ do S₁ end; … end` | for case `i`: bind capture vars; verify `Sᵢ` under `Γ ∪ { e = πᵢ, ¬(∃ cvars₁. e = π₁), …, ¬(∃ cvarsᵢ₋₁. e = πᵢ₋₁) }`; verify `otherwise` or `exhaustive` present | join of exit contexts |

Multiple `invariant` clauses are conjoined: `invariant P; invariant Q;` is treated as `invariant P ∧ Q;`.

### 20.13.4 Recursive Algorithm Encoding

Termination of a recursive `terminating` algorithm is established by a **well-founded recursion argument**. The underlying principle is:

> *If for every input `x` satisfying `requires`, every recursive call `f(y)` satisfies `measure(y) < measure(x)` (lexicographically on `Nat`), and `measure` is `Nat`-valued, then `f` is total on all inputs satisfying `requires`.*

This corresponds to the well-founded induction schema on `Nat` (or `Nat × Nat × …`):

```
∀x. (∀y. (measure(y) < measure(x) → P(y))) → P(x)
─────────────────────────────────────────────────────
∀x. P(x)
```

where `P(x)` is the conjunction of totality and contract for input `x`.

For **mutual recursion**, the same principle applies collectively: the verifier checks that no infinite call sequence can occur by verifying that the shared `decreasing` measure decreases on every cross-algorithm call, treating the mutually recursive group as a single induction.

The promotion axioms (§20.13.2) are emitted for each algorithm in the group once termination is verified for all.

### 20.13.5 Ghost Variable Erasure

Ghost variables exist in the logical context during verification but are erased from the runtime program. The erasure is **sound** in the following sense:

> *Any non-ghost contract (`requires`, `ensures`, `assert`) that is verified in the presence of ghost variables also holds in the erased program — provided the erased statements mention no ghost variables.*

| Property | Statement |
|---|---|
| **Contract preservation** | Let `S'` be `S` with all `ghost var`, `ghost const`, and `ghost` assignments removed. If `S` satisfies contracts `P`/`Q` where `P`, `Q` mention no ghost variables, then `S'` also satisfies `P`/`Q`. |
| **Invariant projection** | Loop invariants that mention ghost variables (e.g. `S_visited`) are verified during type-checking only; they generate no runtime obligation. |
| **Ghost isolation** | The verifier statically checks that ghost variables do not appear on the right-hand side of non-ghost assignments and are not passed to non-ghost algorithm calls (§20.1.3). This is the key condition for soundness of erasure. |
