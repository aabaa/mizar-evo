Mizar Evolution integrates **executable algorithms** with the formal proof system. An `algorithm` block defines a computation that can be:

- **Run** by the Mizar Virtual Machine (MVM) during verification (`by computation`).
- **Reasoned about** using Hoare-logic contracts (`requires`, `ensures`, `invariant`).
- **Promoted** to a mathematical functor (`func`) when total correctness is established (via `terminating`).

The design follows the principle of **partial correctness by default**: an algorithm is permitted to diverge unless an explicit termination argument is given. This avoids demanding termination proofs for algorithms that are only ever used on concrete inputs or in bounded contexts.

## 20.1 Algorithm Syntax

### 20.1.1 Algorithm Definition Block

An algorithm must be declared **inside a `definition` block**. The surrounding `let` declarations bind the type context; the algorithm's parameter list selects which of those bindings become formal parameters.

```ebnf
algorithm_def ::= [ "terminating" ] "algorithm" identifier "(" [ identifier_list ] ")" [ "->" type_expression ]
                  { algorithm_contract }
                  [ "decreasing" term_list ]
                  algorithm_body ";" ;

identifier_list  ::= identifier { "," identifier } ;

algorithm_contract ::= "requires" formula
                     | "ensures"  formula ;

algorithm_body   ::= "do" algo_statement_list "end" ;
```

The `[ "->" type_expression ]` is optional: algorithms with no return type are **void** (procedures). The optional `[ "decreasing" term_list ]` declares the recursion termination measure (§20.5.2) at the header level; for loop termination see §20.2.2.

```mizar
definition
  let n be Nat;

  terminating algorithm factorial(n) -> Nat
    ensures result >= 1
    decreasing n
  do
    if n = 0 do
      return 1;
    end;
    return n * factorial(n - 1);
  end;
end;
```

The `terminating` modifier (§20.6.2) declares that the algorithm terminates for all inputs satisfying `requires`. It is optional — algorithms without `terminating` are treated as partially defined and cannot be used as first-class functors.

### 20.1.2 Parameters and Return Type

Parameters are positional and **immutable** within the algorithm body (they behave as `const`). To use a mutable version of a parameter, create a `var` copy:

```mizar
definition
  let x be Nat;

  algorithm copy_and_modify(x) -> Nat
  do
    var y := x;
    y := y + 1;
    return y;
  end;
end;
```

The return type specifies the type of every `return` expression. The special identifier `result` in `ensures` clauses refers to the return value:

```mizar
definition
  let n be Integer;

  algorithm abs_val(n) -> Nat
    ensures result >= 0
  do
    if n >= 0 do
      return n;
    end;
    return -n;
  end;
end;
```

### 20.1.3 Variable Declarations (`var`, `const`)

Inside an algorithm body, variables are introduced with `var` (mutable) or `const` (immutable). **Ghost variables** (`ghost var`, `ghost const`) exist only at the specification level and are erased at extraction time.

```ebnf
var_decl        ::= "var"   identifier ":=" term [ "as" type_expression [ justification ] ] ";" ;
const_decl      ::= "const" identifier "="  term [ "as" type_expression [ justification ] ] ";" ;
ghost_var_decl  ::= "ghost" "var"   identifier ":=" term [ "as" type_expression [ justification ] ] ";" ;
ghost_const_decl ::= "ghost" "const" identifier "="  term [ "as" type_expression [ justification ] ] ";" ;

justification ::= simple_justification | proof ;
(* simple_justification, proof, reasoning, references — defined in §15/§16 *)
```

| Keyword | Mutability | Runtime | Use case |
|---|---|---|---|
| `const` | Immutable | Yes | Intermediate results, captured values, picks |
| `var` | Mutable | Yes | Loop counters, accumulators, state |
| `ghost const` | Immutable | No (spec only) | Proof witnesses, specification values |
| `ghost var` | Mutable | No (spec only) | Proof tracking (visited sets, abstract state) |

Ghost variables may appear only in **specification contexts**: `invariant`, `ensures`, `requires`, and `assert`. They may not appear on the right-hand side of non-ghost assignments, and may not be passed to non-ghost algorithm calls. Ghost updates use the `ghost` prefix on the assignment:

```ebnf
ghost_assignment ::= "ghost" lvalue ":=" term ";" ;
```

The verifier checks that ghost variables do not influence runtime behaviour. The extractor silently removes all ghost declarations and ghost assignments.

```mizar
definition
  let n be Nat;

  algorithm sum_to(n) -> Nat
  do
    var acc := 0;
    for i = 1 to n do
      acc := acc + i;
    end;
    return acc;
  end;
end;
```

The optional `as T` annotation corresponds to Mizar's `reconsider x = e as T`: it asserts that the initializer expression has type `T`. Widening coercions (e.g. `Nat` to `Integer`) and disambiguation of overloaded literals (e.g. `0 as Real`) are resolved automatically by the type hierarchy and require no proof. Narrowing coercions (e.g. `Integer` to `Nat`) generate a proof obligation; if the automatic prover cannot discharge it, supply a justification (`by` or `proof ... end`):

```mizar
var n := some_int as Nat by non_negative_lemma;
var m := some_int as Nat
proof
  some_int >= 0 by ...;
  thus some_int is Nat;
end;
const r := 0 as Real;             :: automatic — no justification needed
```

If the annotation is omitted entirely, the type is inferred from the initializer expression (see [Chapter 8, Type Inference](./08.type_inference.md)).

### 20.1.4 Assignment

Mutable variables are updated with `:=`:

```ebnf
assignment ::= lvalue ":=" term ";" ;
lvalue     ::= identifier
             | lvalue "." identifier ;
```

The right-hand side is any term expression. The left-hand side must be a `var`-declared identifier or a field thereof. Assigning to a `const` identifier, a `let`-bound variable, or a parameter is a **compile-time error**.

Ghost variables are updated with the `ghost` prefix on the assignment. Ghost assignments may only appear where ghost variables are in scope and must not influence non-ghost state.

```mizar
counter := counter + 1;              :: runtime assignment
state.value := state.value + 1;      :: field update
ghost S_visited := S_visited \/ {x}; :: ghost assignment (spec only)
```

## 20.2 Control Flow

### 20.2.1 Conditional (`if`)

```ebnf
if_stmt ::= "if" formula "do" algo_statement_list if_tail ;
if_tail ::= "end" ";"
          | "else" if_stmt
          | "else" algo_statement_list "end" ";" ;
```


```mizar
if n = 0 do
  return 1;
end;

if n mod 2 = 0 do
  return n div 2;
else
  return 3 * n + 1;
end;

:: if-else-if chain
if n < 0 do
  return -1;
else if n = 0 do
  return 0;
else
  return 1;
end;
```

Both branches must return the same type when `return` is used, or both must fall through to subsequent statements.

### 20.2.2 Bounded Loop (`while`)

```ebnf
while_stmt ::= "while" formula "do"
                 { while_annotation ";" }
                 algo_statement_list
               "end" ";" ;

while_annotation ::= "invariant" formula
                   | "decreasing" term_list ;
```

`while_annotation` clauses (`invariant`, `decreasing`) appear at the **top** of the `while` body, before any other statements.

```mizar
var i := 1;
var s := 0;
while i <= n do
  invariant s = i * (i - 1) div 2 & i >= 1;
  decreasing n - i + 1;
  s := s + i;
  i := i + 1;
end;
```

### 20.2.3 Range Loop (`for ... to`)

```ebnf
for_range_stmt ::= "for" identifier "=" expr ( "to" | "downto" ) expr
                   [ "step" expr ]
                   "do" { for_annotation ";" } algo_statement_list "end" ";" ;

for_annotation ::= "invariant" formula ;
```

The loop variable is immutable within the body and scoped to the loop. Bounds and step are evaluated once before the loop begins. The `step` expression must be a positive `Nat` value; the default step is `1`.

- `for i = 1 to n` — iterates i = 1, 2, …, n (ascending).
- `for i = n downto 1` — iterates i = n, n−1, …, 1 (descending).
- `for i = 0 to n step 2` — iterates i = 0, 2, 4, … (ascending by 2).
- `for i = n downto 0 step 3` — iterates i = n, n−3, … (descending by 3).

```mizar
var s := 0;
for i = 1 to n do
  s := s + i;
end;

for i = 10 downto 1 do
  process(i);
end;

:: Every other element, descending
for i = n downto 1 step 2 do
  handle(i);
end;
```

Termination is **automatically guaranteed** — no `decreasing` annotation is needed. An optional `invariant` clause may be placed at the top of the body to assist verification of the loop's accumulated effect:

```mizar
var s := 0;
for i = 1 to n do
  invariant s = i * (i - 1) div 2;
  s := s + i;
end;
```

### 20.2.4 Collection Loop (`for ... in`)

```ebnf
for_collection_stmt ::= "for" identifier "in" expr "do"
                          { for_annotation ";" }
                          algo_statement_list
                        "end" ";" ;
```

`for_annotation` is shared with `for ... to` and accepts only `invariant` — no `decreasing` is needed as termination is guaranteed by finiteness.

Iterates over all elements of a **finite** collection. The verifier requires the collection to be a `finite set` (or a type registered as `finite`).

**Iteration order is implementation-defined.** The `for ... in` loop makes no guarantee about the order in which elements are visited. When the loop body modifies mutable state, the loop invariant must characterise the accumulated result in a way that does not depend on the order of iteration.

**Tracking visited elements with `ghost var`.** When the loop body modifies mutable state, the user declares a `ghost var` to track the set of elements processed so far. Ghost variables exist only at the specification level and are erased at extraction time (§20.1.3). The verifier injects `S_visited = S` as a post-condition when the loop exits normally; after early exit via `return` or `break`, only what the invariant states about `S_visited` is available.

**Order-independence via set functors and template theorems.** The key obligation is that the invariant characterises the accumulated state as a function of `S_visited` as a *set*, not as a sequence. This is achieved using **set functors** such as `Sum` and `Product` from the standard library.

These functors carry an `ensures` clause proving that their result is independent of the enumeration order. This proof is discharged once — at the functor's definition — using a **template theorem** of the form:

> *If `op` is a commutative and associative binary operation with unit `e`, then for any finite set `S` and any two enumerations of `S`, folding `op` over those enumerations with initial value `e` yields the same result.*

Users who define their own set functors for use in `for ... in` invariants must similarly prove order-independence in the functor's `ensures`, citing the appropriate commutativity/associativity theorems. The concrete syntax for the template theorem is to be determined (see Chapter on Templates).

```mizar
:: Sum of elements.
:: S_visited is a ghost var — explicit, user-maintained, erased at extraction.
ghost var S_visited := {} as set of Integer;
var total := 0;
for x in S do
  invariant S_visited c= S & total = Sum(S_visited);
  total := total + x;
  ghost S_visited := S_visited \/ {x};
end;
:: After normal exit: S_visited = S (from loop semantics), so total = Sum(S).

:: Product of elements.
ghost var S_visited := {} as set of Integer;
var prod := 1;
for x in S do
  invariant S_visited c= S & prod = Product(S_visited);
  prod := prod * x;
  ghost S_visited := S_visited \/ {x};
end;

:: Existence check — no mutable accumulator; no ghost var needed.
for x in S do
  if x mod 2 = 0 do
    return x;
  end;
end;
```

Standard library set functors:

| Functor | Meaning |
| --- | --- |
| `Sum(S)` | Sum of elements of a finite set of numbers |
| `Product(S)` | Product of elements |
| `BigUnion(F, S)` | `⋃ { F(x) : x in S }` |
| `BigIntersection(F, S)` | `⋂ { F(x) : x in S }` |
| `Count(S, P)` | `card { x in S : P(x) }` |

If the loop body has genuine order dependence (no commutative characterisation exists), `for ... in` is the wrong construct. Use `for ... to` over an explicitly ordered structure instead.

> [!NOTE]
> If `return`, `break`, or `continue` exits the loop early and no mutable accumulator is updated, no invariant is needed — the loop effect is trivially order-independent. After early exit, only `S_visited c= S` and whatever the invariant states about `S_visited` are available.

### 20.2.5 Pattern Match (`match`)

```ebnf
match_stmt ::= "match" expr "do"
               match_case+
               ( "otherwise" algo_statement_list
               | exhaustiveness_proof )
               "end" ";" ;

match_case           ::= "case" formula "do" algo_statement_list "end" ";" ;
exhaustiveness_proof ::= "exhaustive" [ justification ] ";" ;
```

Cases are tested in order; the first matching case is executed. The `otherwise` branch handles any remaining case and does **not** use `do`. A `match` without `otherwise` must close with `exhaustive`, supplying a proof that the case predicates cover every value; omitting both is a **compile-time error**.

```mizar
:: With otherwise
match n do
  case n > 0 do
    return 1;
  end;
  case n < 0 do
    return -1;
  end;
  otherwise
    return 0;
end;

:: With exhaustive proof
match n do
  case n > 0 do return  1; end;
  case n < 0 do return -1; end;
  case n = 0 do return  0; end;
  exhaustive by integer_trichotomy;
end;

:: Type-based dispatch
match x do
  case x is NatNumber do
    return handle_nat(x);
  end;
  case x is NegInteger do
    return handle_neg(x);
  end;
  otherwise
    return handle_other(x);
end;
```

> [!NOTE]
> `case x is T` is syntactic sugar for the type assertion formula `x : T holds`. Cases may overlap — the **first** matching case wins.

### 20.2.6 Jump Statements (`return`, `break`, `continue`)

```ebnf
return_stmt   ::= "return" [ expr ] ";" ;
break_stmt    ::= "break" ";" ;
continue_stmt ::= "continue" ";" ;
```

| Statement | Effect | Verification obligation |
|---|---|---|
| `return expr` | Exit algorithm with value `expr` | `expr` must satisfy `ensures`; preconditions of any tail call are discharged |
| `return` | Exit a void algorithm | All `ensures` conditions must hold |
| `break` | Exit the immediately enclosing loop | Loop `invariant` must hold at break point |
| `continue` | Skip to the next loop iteration | Loop `invariant` must hold; **`decreasing` measure must have decreased** relative to the start of the current iteration |

**`continue` and termination.** Because `continue` bypasses the end of the loop body where the termination check normally occurs, the verifier checks that the `decreasing` measure has **already decreased** at the `continue` point. If no `decreasing` annotation is present, `continue` generates no termination obligation (the loop is treated as potentially diverging).

```mizar
definition
  let S be finite set of Integer;

  algorithm first_even(S) -> Integer
    ensures result = -1 or result mod 2 = 0
  do
    for x in S do
      if x mod 2 = 0 do
        return x;
      end;
    end;
    return -1;
  end;
end;
```

## 20.3 Non-determinism (Pick)

A **Pick** operation selects an arbitrary element from a non-empty set, expressed using Mizar's `the` construct:

```ebnf
pick_expr ::= "the" type_expression "of" expr
            | "the" type_expression ;
```

```mizar
const x = the Element of S;    :: pick any element of S
const p = the prime Number;     :: pick any prime
```

**Semantics.** The MVM selects one element non-deterministically at runtime. The verifier treats `the Element of S` existentially — as *some* element satisfying `it in S` — without fixing a specific choice.

**Proof safety.** The contracts of an algorithm (`requires`, `ensures`) are the only interface visible to the surrounding proof system. An `ensures` clause that mentions a Pick result must hold for **every** possible choice; the verifier generates a proof obligation to that effect. Properties not stated in `ensures` cannot be assumed by callers regardless of the choice made. This containment guarantees that non-determinism inside an algorithm cannot silently weaken external proofs.

**Emptiness check.** If `S` is empty, the MVM raises a **runtime verification failure** (§20.8.3). The verifier generates an obligation `S <> {}` at each Pick site.

```mizar
definition
  let S be finite set of Integer;

  terminating algorithm find_even(S) -> Integer
    ensures result = -1 or (result in S & result mod 2 = 0)
  do
    var current := S;
    while current <> {} do
      invariant current c= S;
      decreasing card current;
      const x = the Element of current;
      if x mod 2 = 0 do
        return x;
      end;
      current := current \ {x};
    end;
    return -1;
  end;
end;
```

> [!NOTE]
> Pick on an infinite set is permitted in algorithm definitions but the MVM cannot execute it. Such algorithms may still be used in proofs where only their logical specification (contracts) matters.

## 20.4 Contracts (`requires`, `ensures`, `assert`)

### 20.4.1 Preconditions and Postconditions

```ebnf
algorithm_contract ::= "requires" formula
                     | "ensures"  formula ;
```

- `requires P` — `P` must hold at every call site. The algorithm body may assume `P`.
- `ensures Q` — `Q` holds upon normal termination. `result` in `Q` refers to the return value.

Multiple clauses are treated as conjunctions:

```mizar
definition
  let a, b be Real;

  algorithm divide(a, b) -> Real
    requires b <> 0
    ensures result * b = a
  do
    return a / b;
  end;
end;
```

Contracts are **proof obligations**:
- At each call site `divide(x, y)`, the verifier demands a proof that `y <> 0`.
- After `return v`, the verifier demands a proof that `v * b = a`.

If the automatic prover cannot discharge an obligation, the user adds an `assert` (§20.4.2) with a justification (`by` or `proof ... end`) at the relevant site.

### 20.4.2 Inline Assertions

```ebnf
assert_stmt ::= "assert" formula [ justification ] ";" ;
```

`assert P` inserts a proof checkpoint: the verifier must establish `P` at that program point, using the current logical context plus any cited lemmas or an inline proof.

```mizar
definition
  let arr be FinSequence of Real;
  let lo, hi, target be Integer;

  algorithm binary_search(arr, lo, hi, target) -> Integer
  do
    var l := lo;
    var r := hi;
    while l <= r do
      invariant lo <= l & r <= hi;
      decreasing r - l + 1;
      const mid = (l + r) div 2;
      assert l <= mid & mid <= r by arithmetic;
      if arr.item(mid) = target do
        return mid;
      else if arr.item(mid) < target do
        l := mid + 1;
      else
        r := mid - 1;
      end;
    end;
    return -1;
  end;
end;
```

### 20.4.3 Verification Obligations

| Construct | Obligation |
|---|---|
| Call site `f(args)` | `requires` preconditions of `f`, instantiated with `args` |
| `return expr` | `ensures` postconditions, with `result` bound to `expr` |
| `assert P` | `P` holds at that program point |
| `break` inside loop | loop `invariant` holds at break point |
| `continue` inside loop | loop `invariant` holds at continue point; `decreasing` measure has already decreased relative to start of current iteration |
| `for x in S` with mutable state | loop `invariant` must be stated in terms of a set-based functor (via `ghost var`), proving result is order-independent |
| `for x in S` | `S` is `finite` |
| `the Element of S` | `S <> {}` |
| `decreasing e` (loop) | `e` is `Nat`-valued; `e` decreases strictly on each loop iteration |
| `decreasing e` (header) | `e` is `Nat`-valued; `e` decreases strictly on each recursive call |
| `terminating` | all `decreasing` measures are valid; algorithm returns on all inputs satisfying `requires` |

## 20.5 Loop Verification

### 20.5.1 Loop Invariants (`invariant`)

A loop invariant must hold:
1. **Before** the first iteration (establishment).
2. **At the loop head on every subsequent iteration** (maintenance).
3. **On normal exit** — when the loop condition becomes false.
4. **At every `break` statement** — the invariant must hold at the break point.
5. **At every `continue` statement** — the invariant must hold at the continue point (the loop will re-enter from the top).

```mizar
var i := 1;
var s := 0;
while i <= n do
  invariant s = (i - 1) * i div 2;
  decreasing n - i + 1;
  s := s + i;
  i := i + 1;
end;
:: After loop: i = n + 1, s = n * (n + 1) div 2
```

Multiple `invariant` clauses are interpreted as a conjunction.

### 20.5.2 Termination Measure (`decreasing`)

```ebnf
(* loop termination — inside while body, via while_annotation *)
(* while_annotation ::= "invariant" formula | "decreasing" term_list ; *)

(* recursion termination — in algorithm header *)
(* algorithm_def ::= ... [ "decreasing" term_list ] algorithm_body ";" ; *)

term_list ::= term_expression { "," term_expression } ;
```

The `decreasing` annotation is used in two distinct positions:

- **Algorithm header** (`[ "decreasing" term_list ]` in `algorithm_def`) — declares the recursion termination measure; the tuple must decrease on every recursive call.
- **`while` body** (`while_annotation`) — declares the loop termination measure; the tuple must decrease on every iteration.

The `for ... in` loop does **not** accept `decreasing` — iteration over a finite set is guaranteed to terminate by the finiteness obligation.

In both positions the verifier generates the same obligations:

1. Every expression in the list is **`Nat`-valued** (non-negative integer).
2. The tuple **decreases lexicographically** on every loop iteration or recursive call.

| Form | Ordering | Example |
|---|---|---|
| `decreasing e` | `<` on `Nat` | `decreasing n - i` |
| `decreasing e, f` | Lexicographic `<` on `Nat × Nat` | `decreasing m, n` |

For a custom well-founded ordering, map the measure to `Nat` via a user-defined functor before using `decreasing`:

```mizar
:: User-defined measure mapping a tree to its depth (a Nat)
const d = tree_depth(t);
decreasing d;
```

This avoids extending the grammar with a `wrt` clause and makes the well-foundedness argument explicit and readable.

The non-negativity obligation (`e >= 0`) is discharged at the definition site. If the automatic prover cannot establish it, add an `assert`:

```mizar
assert n - i >= 0 by arithmetic;
decreasing n - i;
```

## 20.6 Termination Proofs

### 20.6.1 Declaring a Termination Measure

For a recursive algorithm, `decreasing` is placed in the **algorithm header**, between the contracts and the body:

```mizar
definition
  let m, n be Nat;

  terminating algorithm ackermann(m, n) -> Nat
    decreasing m, n
  do
    if m = 0 do
      return n + 1;
    end;
    if n = 0 do
      return ackermann(m - 1, 1);
    end;
    return ackermann(m - 1, ackermann(m, n - 1));
  end;
end;
```

The verifier checks that on each recursive call the tuple `(m, n)` decreases lexicographically:

- `ackermann(m - 1, 1)` — first component `m - 1 < m`. ✓
- `ackermann(m, n - 1)` — first components equal, second `n - 1 < n`. ✓
- `ackermann(m - 1, ackermann(m, n - 1))` — first component `m - 1 < m`. ✓

### 20.6.2 Terminating Algorithms and Promotion to `func`

An algorithm prefixed with `terminating` declares that it terminates for every input satisfying its `requires` precondition. This makes the algorithm usable everywhere a `func` is expected — not only in `by computation` proofs, but in arbitrary formal reasoning.

See §20.1.1 for the full `algorithm_def` grammar.

```mizar
definition
  let n be Nat;

  terminating algorithm factorial(n) -> Nat
    decreasing n
  do
    if n = 0 do return 1; end;
    return n * factorial(n - 1);
  end;
end;
```

Once `terminating` is declared and verified, `factorial` may be used as a functor:

```mizar
theorem : factorial(0) = 1;
theorem : factorial(n + 1) = (n + 1) * factorial(n);
```

> [!NOTE]
> Without `terminating`, the algorithm exists only as a computable procedure. It can be called with `by computation` for concrete values, but it cannot be universally quantified over or used as a first-class functor in proofs.
>
> The keyword `terminating` was chosen to avoid confusion with Mizar's existing use of `totality` for functors defined on all inputs of a given type. `terminating` is the standard term in formal verification and term-rewriting theory, and directly conveys that the algorithm's execution terminates.

## 20.7 Recursive Definitions

### 20.7.1 Direct Recursion

An algorithm may call itself recursively. The verifier accepts recursive calls without additional annotation, but requires `decreasing` to prove termination. Without `decreasing`, the algorithm is treated as partially defined.

```mizar
definition
  let n be Nat;

  terminating algorithm fibonacci(n) -> Nat
    decreasing n
  do
    if n = 0 do return 0; end;
    if n = 1 do return 1; end;
    return fibonacci(n - 1) + fibonacci(n - 2);
  end;
end;
```

Each recursive call site generates a proof obligation that the `requires` preconditions are satisfied.

### 20.7.2 Mutual Recursion

Two or more algorithms may call each other mutually. They are placed within the **same `definition` block**; no special declaration is needed — the verifier detects mutual recursion automatically.

`boolean` is the built-in truth-value type; its values are `true` and `false`.

```mizar
definition
  let n be Nat;

  terminating algorithm is_even(n) -> boolean
    decreasing n
  do
    if n = 0 do return true; end;
    return is_odd(n - 1);
  end;

  terminating algorithm is_odd(n) -> boolean
    decreasing n
  do
    if n = 0 do return false; end;
    return is_even(n - 1);
  end;
end;
```

The `decreasing` measures must collectively establish termination across the call cycle. The verifier checks that no infinite call sequence can occur.

## 20.8 Execution Model and Computability

### 20.8.1 The Mizar Virtual Machine (MVM)

The **Mizar Virtual Machine (MVM)** evaluates algorithm expressions during verification. It is invoked when:

- A `by computation` proof step is used (§20.8.2).
- An algorithm is called with fully concrete (ground) arguments in a verification context.

The MVM uses a **call-by-value** model. Side effects are confined to local `var` state; there is no global mutable state.

### 20.8.2 `by computation` Proof Step

```ebnf
computation_proof ::= "by" "computation" [ "[" computation_option { "," computation_option } "]" ] ;
computation_option ::= "steps"   "=" nat_literal
                     | "timeout" "=" nat_literal ;
```

```mizar
theorem : factorial(5) = 120 by computation;

theorem Fact10 : factorial(10) = 3628800
proof
  thus thesis by computation[steps = 100000];
end;
```

**What `by computation` can prove:**

| Goal form | MVM behaviour |
|---|---|
| `f(v₁, …, vₙ) = c` (ground equality) | Evaluate `f(v₁, …, vₙ)`, check result equals `c` |
| `P(v₁, …, vₙ)` (ground predicate) | Evaluate `P`, check it holds |
| Non-ground formulas | **Not supported** — requires classical proof |

For existential goals of the form `ex x st P(x)`, the recommended approach is to implement a search algorithm explicitly:

```mizar
terminating algorithm find_witness(S) -> Integer
  requires S <> {}
  ensures result in S & P(result)
do
  ...
end;
```

The `ensures` clause then provides the existential witness within the proof system, without requiring special MVM search semantics.

### 20.8.3 Computability and Runtime Failures

Algorithm definitions may reference non-constructive operations (non-computable functions, Pick on infinite sets, etc.). Such definitions are mathematically valid, but the MVM cannot execute them.

When the MVM encounters an unexecutable operation, it raises a **runtime verification failure**:

| Operation | Runtime failure condition |
|---|---|
| `the Element of S` | MVM cannot enumerate `S` (e.g., `S` is infinite and has no registered enumerator) |
| Call to `func f(x)` with no `algorithm` body | `f` has no computable implementation |
| Division by zero (`a / 0`) | Zero denominator |

A runtime failure causes the `by computation` step to **fail** — but does not invalidate the algorithm's definition. The definition remains mathematically meaningful; only its executability via the MVM is limited.

> [!NOTE]
> `the Element of S` where `S` is a mathematically valid infinite set (e.g., `NAT`) is not an error at the definition level. The MVM simply cannot execute it. The algorithm can still be used in classical proofs through its contracts.

### 20.8.4 Gas Limit

To prevent diverging algorithms from hanging the verifier, the MVM enforces a **gas limit**: a maximum number of execution steps per `by computation` invocation.

```mizar
theorem : hard_computation(42) = 123456789
proof
  thus thesis by computation[steps = 1000000];
end;
```

Setting `steps = 0` disables the limit (use with caution). The `timeout` option sets a wall-clock limit in milliseconds.

## 20.9 Code Extraction

Code extraction from verified algorithms is provided as a **separate tool**, not part of the core language specification. The extractor takes `terminating` algorithms and produces executable code in a target language (Python, C with GMP, Rust, etc.), preserving all formally verified properties by construction.

Configuration such as target language, numeric type mapping (e.g., `Nat` → GMP `mpz_t` in C), and host-library bindings is supplied via an extractor configuration file rather than inline annotations.

Refer to the extraction tool's documentation for details.

## 20.10 Complete Syntax (EBNF)

```ebnf
(* ── Algorithm definition ─────────────────────────────── *)
algorithm_def ::= [ "terminating" ] "algorithm" identifier
                  "(" [ identifier_list ] ")" [ "->" type_expression ]
                  { algorithm_contract }
                  [ "decreasing" term_list ]
                  algorithm_body ";" ;

identifier_list   ::= identifier { "," identifier } ;

algorithm_contract ::= "requires" formula
                     | "ensures"  formula ;

algorithm_body    ::= "do" algo_statement_list "end" ;

(* ── Statement list ────────────────────────────────────── *)
algo_statement_list ::= { algo_statement } ;

algo_statement ::= var_decl
                 | const_decl
                 | ghost_var_decl
                 | ghost_const_decl
                 | assignment
                 | ghost_assignment
                 | if_stmt
                 | while_stmt
                 | for_range_stmt
                 | for_collection_stmt
                 | match_stmt
                 | return_stmt
                 | break_stmt
                 | continue_stmt
                 | assert_stmt ;

(* ── Declarations and assignment ───────────────────────── *)
var_decl         ::= "var"   identifier ":=" term [ "as" type_expression [ justification ] ] ";" ;
const_decl       ::= "const" identifier "="  term [ "as" type_expression [ justification ] ] ";" ;
ghost_var_decl   ::= "ghost" "var"   identifier ":=" term [ "as" type_expression [ justification ] ] ";" ;
ghost_const_decl ::= "ghost" "const" identifier "="  term [ "as" type_expression [ justification ] ] ";" ;
assignment       ::= lvalue ":=" term ";" ;
ghost_assignment ::= "ghost" lvalue ":=" term ";" ;
lvalue           ::= identifier
                   | lvalue "." identifier ;

justification ::= simple_justification | proof ;
(* simple_justification, proof, reasoning, references — defined in §15/§16 *)

(* ── Control flow ──────────────────────────────────────── *)
if_stmt ::= "if" formula "do" algo_statement_list if_tail ;
if_tail ::= "end" ";"
          | "else" if_stmt
          | "else" algo_statement_list "end" ";" ;

while_stmt ::= "while" formula "do"
                 { while_annotation ";" }
                 algo_statement_list
               "end" ";" ;

while_annotation ::= "invariant" formula
                   | "decreasing" term_list ;

term_list ::= term_expression { "," term_expression } ;

for_range_stmt ::= "for" identifier "=" expr ( "to" | "downto" ) expr
                   [ "step" expr ]
                   "do" { for_annotation ";" } algo_statement_list "end" ";" ;

for_collection_stmt ::= "for" identifier "in" expr "do"
                          { for_annotation ";" }
                          algo_statement_list
                        "end" ";" ;

for_annotation ::= "invariant" formula ;

match_stmt ::= "match" expr "do"
               match_case+
               ( "otherwise" algo_statement_list
               | exhaustiveness_proof )
               "end" ";" ;

match_case           ::= "case" formula "do" algo_statement_list "end" ";" ;
exhaustiveness_proof ::= "exhaustive" [ justification ] ";" ;

return_stmt   ::= "return" [ expr ] ";" ;
break_stmt    ::= "break" ";" ;
continue_stmt ::= "continue" ";" ;
assert_stmt   ::= "assert" formula [ justification ] ";" ;

(* ── Proof step ─────────────────────────────────────────────
   Used in proof blocks (outside algorithm bodies) as:
     "thus" formula "by" "computation" ...
   or in short-form theorems:
     theorem : formula "by" "computation" ... ";"              *)
computation_proof   ::= "by" "computation"
                        [ "[" computation_option { "," computation_option } "]" ] ;
computation_option  ::= "steps"   "=" nat_literal
                      | "timeout" "=" nat_literal ;

(* ── Pick expression ─────────────────────────────────────────
   Used as a term on the right-hand side of const_decl or var_decl
   inside an algorithm body:
     const x = the Element of S ;
     const p = the prime Number ;                              *)
pick_expr ::= "the" type_expression "of" expr
            | "the" type_expression ;
```

## 20.11 Extended Examples

### Fast Exponentiation (Exponentiation by Squaring)

Demonstrates recursion with a non-trivial termination measure and a mathematical `ensures` clause verified by `by computation`.

```mizar
definition
  let x be Nat;
  let n be Nat;

  terminating algorithm fast_power(x, n) -> Nat
    ensures result = x |^ n
    decreasing n
  do
    if n = 0 do
      return 1;
    end;
    if n mod 2 = 0 do
      const half = fast_power(x, n div 2);
      return half * half;
    end;
    return x * fast_power(x, n - 1);
  end;
end;

:: Verified by computation for concrete inputs
theorem : fast_power(2, 10) = 1024 by computation;
theorem : fast_power(3, 5)  = 243  by computation;

:: Promoted to func via terminating; usable in general proofs
theorem fast_power_correct :
  for x, n being Nat holds fast_power(x, n) = x |^ n;
```

### Integer Factorisation (Trial Division)

```mizar
definition
  let n be Nat;

  algorithm trial_factor(n) -> Nat
    requires n >= 2
    ensures result >= 2 & result divides n
  do
    var d := 2;
    while d * d <= n do
      invariant d >= 2 & for k being Nat st 2 <= k & k < d holds not k divides n;
      assert n - d * d >= 0 by arithmetic;
      decreasing n - d * d + 1;
      if n mod d = 0 do
        return d;
      end;
      d := d + 1;
    end;
    return n;   :: n is prime
  end;
end;
```

### Euclidean GCD Algorithm

The algorithm is named `euclid_gcd` to distinguish it from the mathematical functor `Gcd` defined in the standard library. The loop invariant uses `Gcd` (mathematical) — not the algorithm itself — to avoid circular reasoning.

```mizar
definition
  let a, b be Nat;

  terminating algorithm euclid_gcd(a, b) -> Nat
    requires a >= 1 & b >= 1
    ensures result divides a & result divides b &
            for d being Nat st d divides a & d divides b holds d divides result
  do
    var x := a;
    var y := b;
    while y <> 0 do
      invariant x >= 1 & y >= 0 & Gcd(a, b) = Gcd(x, y);
      decreasing y;
      const r = x mod y;
      x := y;
      y := r;
    end;
    return x;
  end;
end;

theorem : euclid_gcd(12, 8) = 4 by computation;
theorem : euclid_gcd(100, 75) = 25 by computation;
```

### Rubik's Cube (Group-Theoretic Solver Sketch)

This is a structural sketch. `dfs` is assumed to be a separately defined algorithm that performs depth-limited DFS over the Rubik's cube group and returns `empty_seq` if no solution is found within the given depth. `CubeState`, `MoveSequence`, `is_valid_state`, `applies`, `is_solved`, and `solved_state` are assumed to be defined in a supporting theory.

```mizar
definition
  let state be CubeState;

  :: Helper: depth-limited DFS; returns empty_seq if no solution found within depth d.
  algorithm dfs(state, d) -> MoveSequence
    requires is_valid_state(state) & d >= 0
    ensures result = empty_seq or
            (len(result) <= d & applies(result, state) = solved_state);

  :: Iterative deepening depth-first search (IDDFS).
  :: Termination: depth increases by 1 each iteration and is bounded by 20
  :: (God's number: every cube state is reachable in at most 20 moves).
  algorithm solve_cube(state) -> MoveSequence
    requires is_valid_state(state)
    ensures applies(result, state) = solved_state
  do
    var depth := 0;
    while not is_solved(state) do
      invariant 0 <= depth & depth <= 20;
      decreasing 20 - depth;
      const seq = dfs(state, depth);
      if seq <> empty_seq do
        return seq;
      end;
      depth := depth + 1;
    end;
    return empty_seq;
  end;
end;

```

## 20.12 Logical Encoding

This section specifies how algorithm constructs map to logical formulas in the Mizar kernel. It complements the operational description in §20.4–§20.6 with the formal axioms generated for `terminating` algorithms and the verification-condition (VC) schema for each statement form.

### 20.12.1 Contract Axioms

Every algorithm call site generates proof obligations from `requires`, and every `return` generates obligations from `ensures`. The table below gives the FOL encoding. Type parameters of the enclosing `definition` block are universally quantified and appear as type guards `is_A(x)`.

| Construct | FOL Encoding |
|---|---|
| `requires P` | Proof obligation at each call site: `Γ ⊢ P(a₁, …, aₙ)` |
| `ensures Q` at `return e` | Proof obligation: `Γ ⊢ Q[result/e]` |
| `ensures Q` (caller side, partial algorithm) | Available only if the call is known to terminate; no unconditional axiom is emitted |
| `assert φ` | Proof obligation: `Γ ⊢ φ` at that program point; `φ` is added to `Γ` afterwards |
| call `f(a₁, …, aₙ)` | (1) `Γ ⊢ requires_f(a₁, …, aₙ)`; (2) `ensures_f(a₁, …, aₙ, f(a₁, …, aₙ))` is added to `Γ` |

Multiple `requires` and `ensures` clauses are conjoined.

### 20.12.2 Promotion Axioms for `terminating`

When `terminating` is declared and verified, the algorithm is promoted to a `func` and the following axioms are emitted into the Mizar kernel:

| Axiom | FOL Schema |
|---|---|
| **Totality** | `∀x₁,…,xₙ. (is_A₁(x₁) ∧ … ∧ P(x₁,…,xₙ) → is_T(f(x₁,…,xₙ)))` |
| **Contract** | `∀x₁,…,xₙ. (is_A₁(x₁) ∧ … ∧ P(x₁,…,xₙ) → Q(x₁,…,xₙ, f(x₁,…,xₙ)))` |

If `requires` is absent, `P` defaults to `⊤` — the functor is total on all inputs of the declared types. Multiple `ensures` clauses are conjoined into a single `Q`.

**Example** — `factorial`:

```
terminating algorithm factorial(n) -> Nat
  ensures result >= 1
  decreasing n
```

Emitted axioms:

| Axiom | Formula |
|---|---|
| Totality | `∀n. (is_Nat(n) → is_Nat(factorial(n)))` |
| Contract | `∀n. (is_Nat(n) → factorial(n) ≥ 1)` |

These axioms become available to the proof system as soon as `terminating` is verified. The algorithm body is not unfolded — further properties are established by explicit theorems (`theorem factorial(0) = 1 by computation;` etc.).

### 20.12.3 Verification Condition Schema

The verifier traverses the algorithm body and generates VCs by a standard Hoare-logic schema. `Γ` denotes the current logical context (accumulated facts at that program point).

| Statement | VC generated | Context after |
|---|---|---|
| `var x := e` | — | `Γ ∪ { x = e, is_inferred_T(x) }` |
| `var x := e as T` | `Γ ⊢ is_T(e)` (auto or via justification) | `Γ ∪ { x = e, is_T(x) }` |
| `const x = e` | — | `Γ ∪ { x = e }` |
| `x := e` | — | `Γ[x ↦ e]` |
| `ghost x := e` | — (spec only) | `Γ_ghost ∪ { x = e }` |
| `return e` | `Γ ⊢ ensures[result/e]` | — (exits) |
| `return` (void) | `Γ ⊢ ensures` | — (exits) |
| `assert φ` | `Γ ⊢ φ` | `Γ ∪ { φ }` |
| call `f(args)` | `Γ ⊢ requires_f(args)` | `Γ ∪ { ensures_f(args, result) }` |
| `if C do S₁ end` | verify `S₁` under `Γ ∪ {C}`; verify tail under `Γ ∪ {¬C}` | `Γ ∪ { ¬C }` (fall-through) |
| `if C do S₁ else S₂ end` | verify `S₁` under `Γ ∪ {C}`; verify `S₂` under `Γ ∪ {¬C}` | join of exit contexts |
| `while C do invariant I; decreasing V; S end` | (1) `Γ ⊢ I` (establishment); (2) `I ∧ C ⊢ wp(S, I)` (maintenance); (3) `I ∧ C ⊢ is_Nat(V)` (non-negativity); (4) `I ∧ C ⊢ V' < V` (decrease) | `Γ ∪ { I ∧ ¬C }` |
| `break` inside loop | `Γ ⊢ I` | — (exits loop) |
| `continue` inside loop | `Γ ⊢ I`; `Γ ⊢ V' < V` | — (re-enters loop head) |
| `for i = a to b do invariant I; S end` | (1) `Γ ⊢ I[i/a]` (establishment); (2) `I ∧ a ≤ i ≤ b ⊢ wp(S, I[i/i+1])` (maintenance); verify `S` under `Γ ∪ {I, a ≤ i ≤ b}` | `Γ ∪ { I ∧ i > b }` |
| `for x in S do invariant I; S_body end` | (1) `Γ ⊢ I[S_visited/∅]`; (2) `I ∧ x ∈ S ∖ S_visited ⊢ wp(S_body, I[S_visited ↦ S_visited ∪ {x}])`; (3) `I` is order-independent (via set functor) | `Γ ∪ { I ∧ S_visited = S }` |
| `match e do case C₁ do S₁ end; … end` | verify `Sᵢ` under `Γ ∪ {Cᵢ}` for each `i`; verify `otherwise` or `exhaustive` covers all cases | join of exit contexts |

Multiple `invariant` clauses are conjoined: `invariant P; invariant Q;` is treated as `invariant P ∧ Q;`.

### 20.12.4 Recursive Algorithm Encoding

Termination of a recursive `terminating` algorithm is established by a **well-founded recursion argument**. The underlying principle is:

> *If for every input `x` satisfying `requires`, every recursive call `f(y)` satisfies `measure(y) < measure(x)` (lexicographically on `Nat`), and `measure` is `Nat`-valued, then `f` is total on all inputs satisfying `requires`.*

This corresponds to the well-founded induction schema on `Nat` (or `Nat × Nat × …`):

```
∀x. (∀y. (measure(y) < measure(x) → P(y))) → P(x)
─────────────────────────────────────────────────────
∀x. P(x)
```

where `P(x)` is the conjunction of totality and contract for input `x`.

For **mutual recursion**, the same principle applies collectively: the verifier checks that no infinite call sequence can occur by verifying that the shared `decreasing` measure decreases on every cross-algorithm call, treating the mutually recursive group as a single induction.

The promotion axioms (§20.12.2) are emitted for each algorithm in the group once termination is verified for all.

### 20.12.5 Ghost Variable Erasure

Ghost variables exist in the logical context during verification but are erased from the runtime program. The erasure is **sound** in the following sense:

> *Any non-ghost contract (`requires`, `ensures`, `assert`) that is verified in the presence of ghost variables also holds in the erased program — provided the erased statements mention no ghost variables.*

| Property | Statement |
|---|---|
| **Contract preservation** | Let `S'` be `S` with all `ghost var`, `ghost const`, and `ghost` assignments removed. If `S` satisfies contracts `P`/`Q` where `P`, `Q` mention no ghost variables, then `S'` also satisfies `P`/`Q`. |
| **Invariant projection** | Loop invariants that mention ghost variables (e.g. `S_visited`) are verified during type-checking only; they generate no runtime obligation. |
| **Ghost isolation** | The verifier statically checks that ghost variables do not appear on the right-hand side of non-ghost assignments and are not passed to non-ghost algorithm calls (§20.1.3). This is the key condition for soundness of erasure. |
