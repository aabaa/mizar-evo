The documentation generator (`mizar doc`) extracts structured documentation from source files and produces browsable, cross-referenced HTML output. The source of documentation is the Mizar source itself: item labels, type signatures, theorem statements, and `:::` documentation comments (§2.9.1). No separate documentation language is required.

The generator reads the `exports` array from `*.mizir.json` artifacts (§23.5.1) to determine which items are publicly visible, and uses the `@latex` annotations (§21.3) and resolved types from the same artifacts to render mathematical notation.

## 24.1 Documentation Comments Syntax

A **documentation comment** is a contiguous block of `:::` line comments immediately preceding a named item (definition, theorem, registration, or algorithm). No blank line may appear between the comment block and the item it documents; a blank line ends the association.

```mizar
::: Computes the greatest common divisor of two positive integers
::: using the Euclidean algorithm.
:::
::: The result satisfies `result divides a`, `result divides b`, and
::: for any `d` dividing both `a` and `b`, `d divides result`.
terminating algorithm euclid_gcd(a, b) -> Nat
  requires a > 0 & b > 0
  ensures result = Gcd(a, b)
  ...
```

The body of a documentation comment is interpreted as **Markdown** (CommonMark specification) with the following Mizar-specific extensions:

- Inline code enclosed in backticks (`` `code` ``) is rendered with Mizar syntax highlighting. It is linked only when the contents are an exact label or FQN reference (§24.2).
- Display math delimited by `$$` is rendered as a mathematical formula (§24.3).
- Cross-reference links of the form `[text](ref)` are resolved as described in §24.2.

**Block structure.** A documentation comment may contain multiple paragraphs separated by blank `:::` lines. Lists, code blocks, and block-quote admonitions are supported using standard Markdown syntax within `:::` lines.

```mizar
::: Computes `n!` (n factorial) for a natural number `n`.
:::
::: ## Examples
:::
::: ```mizar
::: factorial(0) = 1
::: factorial(5) = 120
::: ```
:::
::: ## Mathematical definition
:::
::: $$n! = \prod_{k=1}^{n} k$$
:::
::: The algorithm terminates because `n` decreases at each recursive call.
terminating algorithm factorial(n) -> Nat
  ensures result = Factorial(n)
  decreasing n
  ...
```

**Structured tags.** The following tags may appear as the first token of a paragraph within a documentation comment:

| Tag | Usage |
|---|---|
| `@param name` | Documents the parameter named `name`. One tag per parameter. |
| `@returns` | Documents the return value; the identifier `result` refers to the return value. |
| `@requires` | Prose description of the precondition (supplements the formal `requires` clause). |
| `@ensures` | Prose description of the postcondition (supplements the formal `ensures` clause). |
| `@see ref` | Adds a "See also" cross-reference link (§24.2). |
| `@since version` | Records the version in which the item was introduced. |
| `@deprecated version` | Marks the item as deprecated since `version`; the deprecation message follows. |

```mizar
::: Divides `a` by `b` and returns the quotient.
:::
::: @param a  the dividend
::: @param b  the divisor; must be non-zero
::: @returns  the real number `a / b`
::: @requires `b <> 0`
::: @see mml.real.Real_div_mul
definition
  let a, b be Real;

  algorithm divide(a, b) -> Real
    requires b <> 0
    ensures result * b = a
  do
    return a / b;
  end;
end;
```

**Module-level documentation.** A documentation comment at the top of a source file, before any `import` or definition statement, documents the module as a whole. It appears on the module's index page in the generated output.

## 24.2 Cross-Referencing and Hyperlinks

The documentation generator resolves cross-references among labelled items across the entire package and its dependencies. References are label-based rather than expression-based so that documentation links are stable even when an item has overloaded notation or multiple surface syntaxes.

**Inline code links.** When the contents of a backtick span are an exact item label or FQN, the generator links it to the documentation page for that item:

```
::: The result equals `Gcd` as defined in `mml.nat.Gcd`.
```

The generator first attempts to resolve an unqualified label in the current module's exported labels; if not found, it searches imported modules in import order. Ambiguous labels emit a build-time warning (`W0002`-class) and are rendered without a link. Backtick spans that are expressions, theorem fragments, or overloaded surface notation are rendered as code without link resolution. Authors should use an explicit label or FQN when a link is required.

**Explicit links.** The standard Markdown link syntax `[display text](ref)` accepts three forms for `ref`:

| Form | Example | Resolves to |
|---|---|---|
| Label or FQN | `(Gcd)`, `(mml.nat.Gcd)` | Documentation page for the labelled item |
| Section reference | `(§20.5.1)` | Section anchor in the generated specification |
| URL | `(https://...)` | External URL (passed through unchanged) |

```mizar
::: This algorithm implements the approach described in [§20.5.1](§20.5.1).
::: The result type is documented under [Element of R](algebra.struct.Element).
```

**Automatic back-references.** The generator indexes all `by` justification clauses in theorems and proofs. For each theorem `T`, the generated page includes a "Used by" section listing all theorems that cite `T` in a `by` clause. This section is derived from the `exports` array in `*.mizir.json` (§23.5.1) and does not require any annotation.

**`@see` links.** The `@see ref` tag in a documentation comment adds an entry to the "See also" list at the bottom of the item's page. `ref` follows the same three forms as explicit links.

**Anchor naming.** Every documented item receives a stable anchor derived from its exported label, not from rendered notation or overload resolution. The anchor for an item whose FQN is `algebra.ring.AddAssoc` is `#algebra.ring.AddAssoc`; within that module page it may also be addressed as `#AddAssoc` when the label is unique on the page. Section headings receive anchors of the form `#sec-20-5-1`.

Labels are part of the public documentation interface. For definitions, the definition label is used. For theorems and registrations, the theorem or registration label is used. For algorithms, the algorithm identifier is the documentation label; if a future syntax introduces a separate algorithm label, the documentation generator will use that label instead.

## 24.3 Mathematical Formula Rendering

Mathematical notation in generated documentation comes from two sources: `@latex` annotations on symbol definitions (§21.3) and LaTeX expressions embedded in documentation comments.

**`@latex` annotations.** When a functor, predicate, mode, or attribute definition carries a `@latex` annotation, the generator substitutes the given LaTeX string wherever the symbol appears in rendered output — in theorem statements, type signatures, and formula displays. If no `@latex` annotation is present, the symbol name is rendered in a monospace typeface.

```mizar
@latex("n!")
func Factorial(n) -> Nat;

@latex("\gcd(a, b)")
func Gcd(a, b) -> Nat;
```

A theorem whose statement involves both symbols is rendered as:

> **Theorem** `gcd_factorial_coprime`: for all $n \geq 2$, $\gcd(n!, n+1) = 1$.

**Inline formulas** in documentation comments use standard LaTeX delimiters:

- Inline: `$formula$` — rendered inline with the surrounding text.
- Display: `$$formula$$` on its own line — rendered as a centred display equation.

The generator does not validate LaTeX syntax during the build; invalid LaTeX is passed through to the rendering engine and may produce a rendering error in the output.

**Automatic formula generation.** For items without a `@latex` annotation, the generator produces a default mathematical rendering by translating the Mizar surface syntax to LaTeX using the following rules:

| Mizar syntax | Default LaTeX output |
|---|---|
| `a + b` | `a + b` |
| `a * b` | `a \cdot b` |
| `a / b` | `\frac{a}{b}` |
| `a |^ n` | `a^{n}` |
| `a mod b` | `a \bmod b` |
| `a <= b` | `a \leq b` |
| `a <> b` | `a \neq b` |
| `a in S` | `a \in S` |
| `S c= T` | `S \subseteq T` |
| `for x being T holds P` | `\forall x \in T,\; P` |
| `ex x being T st P` | `\exists x \in T,\; P` |

The automatic translation is applied only when no `@latex` annotation overrides it. Authors are encouraged to supply `@latex` annotations for non-standard or domain-specific notation.

**Formula numbering.** Display equations in theorem statements are numbered automatically in the HTML output. The numbering is `(chapter.section.item)`, where `item` is the sequential index of the display equation within the item's documentation page. Formula numbers are hyperlinked and can be referenced with `$$\text{see }(24.3.1)$$` syntax in documentation comments.

## 24.4 HTML Output

The documentation generator writes a static HTML site to the directory specified by `doc_dir` in the package manifest (default: `docs/`).

The HTML output is a static site suitable for hosting on any web server or GitHub Pages. It uses **KaTeX** for client-side mathematical formula rendering, requiring no server-side processing.

Directory structure of the HTML output:

```
docs/
├── index.html           ← package overview and module list
├── search.json          ← full-text search index
├── algebra/
│   ├── index.html       ← module index for `algebra`
│   ├── groups/
│   │   ├── basic.html   ← documentation for `algebra.groups.basic`
│   │   └── ...
│   └── ...
└── assets/
    ├── katex/           ← KaTeX library files
    ├── style.css
    └── search.js
```

**Module index page.** Each module's index page lists all exported items with their type signatures (rendered using `@latex` annotations where available), a one-line summary extracted from the first paragraph of the documentation comment, and links to the full item page.

**Item page.** Each named item has a dedicated page containing:
- The full type signature or theorem statement, rendered mathematically.
- The complete documentation comment, rendered as HTML.
- The `requires` and `ensures` clauses (for algorithms and functions), rendered mathematically.
- The `@param` and `@returns` tags formatted as a parameter table.
- An automatically generated "Used by" section (§24.2).
- The "See also" list from `@see` tags.
- A "Source" link to the relevant line in the source file (using the `source_range` from `*.mizir.json`).

**Search.** The generator produces a `search.json` index containing the FQN, item kind, one-line summary, and type signature for every exported item. The search UI (`search.js`) performs client-side full-text search over this index without requiring a backend.

**Versioned documentation.** When the package manifest includes a `version` field (§23.1), the generator embeds the version string in the page header and in `search.json`. Hosting infrastructure may use this to serve multiple versions concurrently; the standard layout places version-specific output under `docs/v2.3.1/`.
