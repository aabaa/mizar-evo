## 13.1 基本式

基本式は、原子値、参照、およびより複雑な用語の構成要素を形成する基本構成を表します。

**構文 (EBNF)**:

```ebnf
primary_expression ::= variable_identifier
                     | "it"
                     | "(" term_expression ")"
                     | numeral
                     | bracket_functor_application ;
variable_identifier ::= identifier ;
numeral            ::= digit+ ;
bracket_functor_application ::= user_symbol term_list user_symbol
                              | "[" term_list "]" ;
term_list         ::= term_expression { "," term_expression } ;
```

### 13.1.1 変数識別子

変数識別子は、以前に宣言された変数への参照です。これらの使用法は、言語の一般的なスコープ規則に従います。

変数宣言、スコープ、シャドウイング、および予約された識別子の詳細なルールについては、[第4章 (変数と定数)](./04.variables_and_constants.md) を参照してください。

**例**：

```mizar
let x be Nat;
let y be Nat;
x + y;
```

### 13.1.2 `it` キーワード

`it` は、`means` スタイルを使用して `func` ブロック内で現在定義されているオブジェクトを指します。

**例**：

```mizar
definition
  let X, Y be set;
  func UnionDef: X \/ Y -> set means
    for z being object holds z in it iff z in X or z in Y;
end;
```

**制限**：
- `means` を使用した `func` 定義内でのみ許可されます
- 定義されている戻りオブジェクトのみを参照します。
- `equals` 定義では使用できません (値は明示的です)

### 13.1.3 括弧で囲まれた式

グループ化と優先順位制御に使用されます。

**例**：

```mizar
(a + b) * c           :: Override default precedence
(((a + b) * c) + d)   :: Arbitrary nesting
```

**意味論**:
- かっこはデフォルトの演算子の優先順位をオーバーライドします
- 任意のネストがサポートされています
- 式の型は括弧で囲んで保持されます

### 13.1.4 数字

自然数の数値リテラル。

**例**：

```mizar
0
42
1000
```

## 13.2 functorの適用

functor適用は、ユーザー定義のfunctorを引数に適用することによって項を構築します。

**構文 (EBNF)**:

```ebnf
functor_application ::= [ functor_loci ] functor_symbol [ functor_loci ] ;
functor_loci       ::= term_expression | "(" term_list ")" ;
inline_functor_application ::= inline_func_name "(" [ term_list ] ")" ;  (* mandatory parens *)
template_functor_application ::= functor_symbol template_args
                                 [ "(" [ term_list ] ")" ] ;
term_list         ::= term_expression { "," term_expression } ;
```

> [！注記]
> シンボリックfunctorの場合、括弧は **単一引数の場合はオプション**で、**複数の引数の場合は必須**です。インラインfunctorには常に括弧が必要です。
> Template functor application は Ch.16 §16.5 の bracket argument list を使い、
> template-capable name の後では Ch.2 §2.5.1 の規則に従って bracket functor
> notation より先に解析されます。

### 13.2.1 functorの分類

|タイプ |宣言 |範囲 |例 |
| --- | --- | --- | --- |
| **象徴的** | `func Label: pattern -> type ...` |moduleレベル | `x + y`、`|.x.|` |
| **インライン** | `deffunc name(args) -> type equals ...` |証明ローカル | `double(n)` |

### 13.2.2 シンボリックfunctorの適用

シンボリックfunctorは複数の表記スタイルをサポートしています。

|表記法 |パターン |例 |
| --- | --- | --- |
| **プレフィックス** | `symbol x` または `symbol(x, y)` | `- x`、`max(a, b)` |
| **中置** | `x symbol y` | `x + y`、`X \/ Y` |
| **後置** | `x symbol` | `n !`、`f "` |
| **ブラケット** | `left x, y right` | `[: X, Y :]`、`|. x .|`、`[x, y]` |

> [！注記]
> 単一の引数の場合、括弧はオプションです。`sin x` と `sin(x)` は同等です。
> 複数の引数の場合、`max(a, b)` のように括弧が必要です。

```mizar
:: Prefix (single arg - parens optional)
- x
card X
sin x       :: equivalent to sin(x)
sin(x)

:: Prefix (multi-arg - parens required)
max(a, b)
min(x, y, z)

:: Infix
x + y
X \/ Y

:: Postfix
n !

:: Bracket
[: A, B :]
|. x .|
[a, b]
```

予約token `[` と `]` は、parser が term primary を期待する位置でのみ、組み込み bracket functor delimiter pair として許可されます。template可能な名前の後では、同じtokenはtemplate引数 (`Module[R]`, `SomeTheorem[R]`) を区切り、library annotation 形式は別個の複合token `@[` で始まります。`x[y]` のような postfix indexing は定義されていません。

### 13.2.3 インラインfunctor適用

インラインfunctor (証明ブロックで `deffunc` で定義) は必須のかっこで適用されます。

> [！重要]
> インラインfunctorは、引数が 0 個または 1 つであっても、定義と適用の両方で **かっこが必要です**。

**例**：

```mizar
proof
  deffunc double(n be Nat) -> Nat equals n + n;
  
  A1: double(3) = 6;
  ...;
end;
```

**意味論**:
- インラインfunctorは置換によって拡張されます
- 定義された証明ブロックの外側には表示されません
- 証明ローカルの略語に便利

> [！注記]
> `deffunc` 定義構文については、「[第15章 (文), §15.2.3](./15.statements.md#1523-インライン関数定義-deffunc)」を参照してください。

## 13.3 構造式

構造式を使用すると、ユーザー定義の `struct` 型に基づいて構造化データ値の構築と操作が可能になります ([第5章](./05.structures.md) を参照)。

**構文 (EBNF)**:

```ebnf
structure_expression ::= struct_constructor | selector_access ;
struct_constructor   ::= struct_name [ type_args ] "(" [ field_assignment_list ] ")" ;
field_assignment_list ::= field_assignment { "," field_assignment } ;
field_assignment     ::= field_name ":" term_expression ;
selector_access      ::= term_expression "." field_name [ "(" [ term_list ] ")" ] ;
```

### 13.3.1 構造体コンストラクター

構造体の値は、フィールドと値のペアを指定することによって作成されます。

**例**：

```mizar
:: Define a Point structure with default values (from Ch.7)
struct Point where
  field x -> Real := 0;
  field y -> Real := 0;
end;

:: Construct with all fields specified
set p = Point(x: 3, y: 4);

:: Omit fields that have defaults
set origin = Point();             :: x = 0, y = 0
set on_x_axis = Point(x: 5);      :: x = 5, y = 0
```

**デフォルト値の構文**:

フィールドは、構造体定義で `:=` を使用してデフォルト値を宣言できます。

```ebnf
field_decl ::= "field" identifier "->" type_expression [ ":=" term_expression ] ";" ;
```

**ルール**:
- デフォルトのないすべてのフィールドはコンストラクターで指定する必要があります
- デフォルト値を持つフィールドは省略できます (デフォルト値が使用されます)。
- フィールド名は構造体の宣言と一致する必要があります
- コンストラクター内のフィールドの順序は関係ありません
- 各フィールド値の型は宣言された型と一致する必要があります

### 13.3.2 セレクターへのアクセス

構造体のフィールドには、ドット表記を使用してアクセスします。

**例**：

```mizar
set p = Point(x: 3, y: 4);
p.x + p.y = 7;  :: Field access

:: Chained access for nested structures
set line = Line(start: Point(x: 0, y: 0), end: Point(x: 1, y: 1));
line.start.x = 0;
line.end.y = 1;
```

選択されたフィールドまたはプロパティが関数値を持つ場合、セレクターの直後に引数リストを続けることができます。

```mizar
let M be Magma;
let x, y be Element of M.carrier;

M.binop(x, y)
```

これはセレクターアクセスの後に選択された関数値の適用が続く形として解析され、`M.binop` が表す値を `x` と `y` に適用することと意味的に等価です。型検査は、選択されたフィールドまたはプロパティが、与えられた引数を受け入れる関数型を持つことを確認しなければなりません。

> [！重要]
> 従来の Mizar セレクター構文 `the x of p` は **非推奨**です。ドット表記 `p.x` のみを使用してください。これにより、明確な記述演算子 `the` による構文の曖昧さを回避しながら、より簡潔で普遍的になじみのある構文が提供されます。

### 13.3.3 構造の更新

構造値は証明コンテキストでは **不変**です。更新では、機能更新構文を介して新しい値が生成されます。

**構文 (EBNF):**

```ebnf
functional_update ::= term_expression "with" "(" field_update_list ")" ;
field_update_list ::= field_update { "," field_update } ;
field_update      ::= selector ":=" term_expression ;
```

**例 (証明コンテキスト - 不変):**

```mizar
set p = Point(x: 3, y: 4);
set q = p with (x := p.x + 1);   :: New point; p is unchanged
:: p.x = 3, q.x = 4
```

**例 (アルゴリズム コンテキスト - `var` で変更可能):**

アルゴリズム ブロックでは、`var` で宣言された構造体は、インプレース フィールドの変更をサポートします。

```mizar
algorithm
  var p := Point(x: 3, y: 4);
  p.x := p.x + 1;               :: In-place update (p is var)
  :: p.x = 4

  const q := Point(x: 1, y: 2);  :: Immutable (no var)
  :: q.x := 5;                   :: ERROR: q is not var
end;
```

> [！注記]
> 構造体定義の構文については、「[第5章（構造）](./05.structures.md)」を参照してください。
> アルゴリズム ブロックと `var` 宣言については、「[第20章 (algorithm と verification)](./20.algorithm_and_verification.md)」を参照してください。

## 13.4 セット式

セット式は、列挙または内包表記によってセットを構築します。

**構文 (EBNF)**:

```ebnf
set_expression    ::= set_enumeration | set_comprehension ;
set_enumeration   ::= "{" [ term_list ] "}" ;
set_comprehension ::= "{" term_expression "where" typed_var_list [ ":" formula ] "}" ;
term_list         ::= term_expression { "," term_expression } ;
typed_var_list    ::= typed_var { "," typed_var } ;
typed_var         ::= identifier "is" type_expression ;
```

### 13.4.1 セットの列挙

セット要素の明示的なリスト。

**例**：

```mizar
{}              :: empty set
{ a }           :: singleton
{ a, b, c }     :: finite set
```

### 13.4.2 集合内包表記 (フランケル演算子)

セットの要素に対するマッピングとフィルターを指定してセットを構築します。

> [！重要]
> **セット フード制約**: ジェネレーター変数は、任意の型ではなく、**セット** (例: `Element of S`) の要素の範囲である必要があります。これにより、結果が適切なクラスではなくセットになることが保証されます。
>
> ```mizar
> :: OK: x はセットの要素全体に渡ります
> { x ここで、x は NAT の要素です: x < 10 }
>
> :: NG: グループはセットではなく型 (クラス) です
> { G はグループです : G はアーベルです } :: REJECTED
> ```

**例**：

```mizar
:: Even natural numbers
{ n where n is Element of NAT : n mod 2 = 0 }

:: Squares of naturals
{ n * n where n is Element of NAT : n < 10 }

:: All elements (condition omitted)
{ f.x where x is Element of dom f }
```

> [！注記]
> 条件を省略した場合は、ジェネレーター型を満たすすべての要素が含まれます。 `: formula` の部分を省略するだけです。

### 13.4.3 複数のジェネレーター

**例**：

```mizar
:: Cartesian product elements (no filter needed)
{ [x, y] where x is Element of A, y is Element of B }

:: Pairs satisfying a condition
{ [x, y] where x is Element of NAT, y is Element of NAT : x + y = 10 }
```

### 13.4.4 入れ子になった内包表記

**例**：

```mizar
:: Set of sets
{ { y where y is Element of NAT : y < x }
  where x is Element of NAT : x < 5 }
```


## 13.5 選択演算子 (`the`)

`the` 演算子 (明確な説明) は、指定された型の任意の要素を選択します。論理ではヒルベルトのε演算子に相当します。

**構文 (EBNF):**

```ebnf
choice_expression ::= "the" type_expression ;
```

**意味論:**

- `the T` は、`T` 型のオブジェクトを返します。
- **存在要件**: 型 `T` は **空ではない**必要があります。これは、存在登録システム ([第17章](./17.clusters_and_registrations.md) を参照) によって保証されます。
- **決定主義**: 選択は安定しています。 `the T` は常に、特定のコンテキスト内の同じ任意の要素を参照しますが、`is T` 以外の特定のプロパティは想定されていません。

**例：**

```mizar
set n = the Nat;       :: An arbitrary natural number
set x = the Element of X;
```

> [！注記]
> この演算子は、パラメーター型の任意の要素を使用して証人を構築する必要がある、パラメーター化された存在登録に不可欠です。

## 13.6 型式認定 (`qua`)

`qua` 演算子は、**証明を必要とせずに**、用語の特定の型のビューを明示的に選択します。これは 2 つの相補的な状況で有効であり、どちらも静的に検証可能です。

1. **構造継承アップキャスト** — ターゲット型は、用語の `inherit` チェーンを介して到達可能な祖先モードです。
2. **クラスターベースの拡張** — ターゲット型の属性は用語のタイプよりも厳密に少なく、ターゲットの属性はすべて、アクティブなクラスター登録を介して用語の属性から派生できます。

どちらの場合も、値自体は変更されません。overloadの解決、フィールド アクセス、属性チェックに使用される型のみが影響を受けます。

**構文 (EBNF)**:

```ebnf
qua_expression ::= operator_expression "qua" type_expression { "qua" type_expression } ;
```

`qua` は、ユーザー定義のシンボリックfunctorではなく、固定項レベルの型修飾演算子です。これは用語レベルの演算子の中で最も優先順位が低く、左から右に連鎖します。括弧は部分用語のみを修飾するために使用できます。

### 13.6.1 構造継承のアップキャスト

構造が `inherit ... extends ...` チェーンを介して祖先モードから継承する場合、`qua` は目的の祖先ビューを選択します。これは、複数の継承パスが存在し (ダイヤモンド継承)、検証器がパスを自動的に解決できない場合に最も役立ちます。

```mizar
let R be DoubleLoopStr;

:: View R as AddLoopStr to access additive properties
(R qua AddLoopStr) is associative;

:: View R as MulLoopStr to access multiplicative properties
(R qua MulLoopStr) is commutative;
```

```mizar
:: Ring inherits from Magma via both AddGroup and MulMonoid
let R be commutative Ring;

f(R qua AddMagma)   :: selects R's additive Magma structure
f(R qua MulMagma)   :: selects R's multiplicative Magma structure
```

### 13.6.2 クラスタベースの拡張

パラメーターまたはoverloadが必要とする以上の属性が用語にある場合、`qua` は意図した属性の少ない型を選択します。検証器は、ターゲット属性がクラスター クロージャによって用語の型から導出可能な属性のサブセットであることを確認します。証明は生成されません。

```mizar
:: cluster registrations: finite → countable, finite → compact
let S be finite TopSpace;

:: f is overloaded for countable TopSpace and compact TopSpace
f(S qua countable TopSpace)   :: resolves to the countable overload
f(S qua compact TopSpace)     :: resolves to the compact overload
```

### 13.6.3 `qua`のチェーン化

`qua` ステップを左から右に連鎖させて、継承階層の複数のレベルを横断することができます。

```mizar
f(R qua AddGroup qua Magma)
:: first selects R's AddGroup view, then that structure's Magma ancestor
```

各ステップは有効なワイドニング (アップキャストまたはクラスター) である必要があります。連鎖は左から右に評価されます。

### 13.6.4 制限事項

- ターゲット型は、`inherit` チェーン (アップキャスト) またはクラスター クロージャー (属性のサブセット化) を介して、ワイドニングによって到達可能である必要があります。証拠は生成されません。
- ナローイング (より特定のタイプへのキャスト) には、証明付きの `reconsider` が必要です。 `qua`は絞り込みには使用できません。
- template引数 `[...]` では、`qua` は同じ優先順位と結合規則 `PermProduct[R qua AddMagma]` を使用します ([第 18 章、§18.2.2](./18.templates.md#1822-型パラメータ) を参照)。

> [！注記]
> `reconsider` ステートメントの構文については、[第8章 (型推論)](./08.type_inference.md) および [第15章 (文), §15.5.1](./15.statements.md#1551-型の再検討-reconsider) を参照してください。

## 13.7 演算子の優先順位と結合性

ユーザー定義のfunctorには、明示的な優先順位と結合性の宣言を含めることができます。優先順位が解析に影響するため、これらは注釈ではなくスタンドアロンのステートメントです (§10.9 を参照)。

**構文 (EBNF)**:

```ebnf
infix_operator_decl   ::= "infix_operator"   "(" string_literal "," infix_assoc "," nat_literal ")" ";" ;
prefix_operator_decl  ::= "prefix_operator"  "(" string_literal "," nat_literal ")" ";" ;
postfix_operator_decl ::= "postfix_operator" "(" string_literal "," nat_literal ")" ";" ;
infix_assoc           ::= "left" | "right" | "none" ;
```

### 13.7.1 優先規則

各シンボリックfunctorには、関連付けられた優先レベル (0 ～ 255) があります。値が大きいほど、結合がより強くなります。

**例**：

```mizar
infix_operator("+", left, 80);
infix_operator("*", left, 90);
infix_operator("^", right, 95);
```

これらの宣言により、次のようになります。
- `a + b * c` は `a + (b * c)` として解析されます (優先順位が高いほど、より厳密にバインドされます)
- `a * b + c` は `(a * b) + c` として解析されます
- `a + b = c + d` は `(a + b) = (c + d)` として解析されます (`=` の優先順位はコア moduleで固定されています)

### 13.7.2 結合性

|タイプ |グループ化 |例 |
| --- | --- | --- |
| `left_associative` |左から右 | `a + b + c` → `(a + b) + c` |
| `right_associative` |右から左 | `a ^ b ^ c` → `a ^ (b ^ c)` |
| `non_associative` |括弧が必要です | `a = b = c` → **エラー** |

> [！注記]
> `non_associative` は、**同じ** 演算子の連鎖のみを禁止します (例: `a = b = c`)。 `precedence_value` は、オペレーターが **他の** オペレーターとどのように対話するかを決定するため、依然として必要です。たとえば、優先順位 50 の `=` と優先順位 80 の `+` により、`a + b = c` は `(a + b) = c` として解析されます。

### 13.7.3 曖昧さ回避ガイドライン

- 優先順位が不明瞭な場合は括弧を使用してください
- 修飾名を使用してoverloadされたシンボルの曖昧さをなくす
- 複雑な式の暗黙的なグループ化に依存しないようにする

## 13.8 論理エンコーディングの詳細

このセクションでは、項式が一階論理にどのようにマップされるかについて説明します。

### 13.8.1 変数の参照

変数は、タイプ ガード付きの FOL 変数としてエンコードされます。

|Mizar source | FOL エンコーディング |
| --- | --- |
| `let x be Nat; x` | `x` (コンテキスト `is_Nat(x)` あり) |

### 13.8.2 `it` キーワード

`means` 定義では、`it` は関数アプリケーションに置き換えられます。

```
func F(x) -> T means P(it)
⇒  forall x (P(F(x)))
```

### 13.8.3 functorのアプリケーション

functorアプリケーションは、FOL 関数アプリケーションとしてエンコードされます。

|Mizar source | FOL エンコーディング |
| --- | --- |
| `x + y` | `add(x, y)` |
| `- x` | `neg(x)` |
| `n !` | `factorial(n)` |
| `[: A, B :]` | `cartesian_product(A, B)` |

### 13.8.4 構造コンストラクタのエンコーディング

構造体コンストラクターは、FOL 関数アプリケーションとしてエンコードされます。

|Mizar source | FOL エンコーディング |
| --- | --- |
| `Point(x: 3, y: 4)` | `make_Point(3, 4)` |
| `Rect(w: a, h: b)` | `make_Rect(a, b)` |

### 13.8.5 セレクターアクセスエンコーディング

セレクターは FOL 関数としてエンコードされます。

|Mizar source | FOL エンコーディング |
| --- | --- |
| `p.x` | `Point_x(p)` |
| `line.start.x` | `Point_x(Line_start(line))` |

### 13.8.6 式エンコーディングの設定

**列挙**：

|Mizar source | FOL エンコーディング |
| --- | --- |
| `{}` | `empty_set` |
| `{ a }` | `singleton(a)` |
| `{ a, b }` | `union(singleton(a), singleton(b))` |

**理解** (置換公理スキーマを使用してエンコード):

```
{ f(x) where x is Element of S : P(x) }
⇒  ∃Z. ∀y. (y ∈ Z ↔ ∃x. x ∈ S ∧ P(x) ∧ y = f(x))
```

条件を省略した場合：

```
{ f(x) where x is Element of S }
⇒  ∃Z. ∀y. (y ∈ Z ↔ ∃x. x ∈ S ∧ y = f(x))
```

### 13.8.7 `qua` 演算子のエンコーディング

`qua` オペレーターは静的型アサーションを生成します。新しい公理は導入されません。検証器は、コンパイル時に継承チェーンまたはクラスター クロージャを介して有効性条件を解除します。

**継承アップキャスト** — 祖先モードのフィールド投影を選択します。

```
(x qua T).field
⇒  Validity (static): is_T reachable from type_of(x) via inherit chain
⇒  Result: T_field(x)
```

**クラスターベースの拡張** — 拡張タイプに一致するoverload候補を選択します。

```
f(x qua T)
⇒  Validity (static): cl(type_of(x)) ⊇ attributes(T)  (cluster closure subsumes T)
⇒  Overload resolution: f is resolved against parameter type T
```

どちらの場合も、`x` という用語自体は FOL エンコードでは変更されません。 `qua` は、メタ レベルでの type-dispatch にのみ影響します。

#### 有効性条件の FOL エンコーディング

上記の 2 つの有効性条件は、単なるアルゴリズムによるコンパイル時のチェックではありません。それらは FOL でエンコードでき、自動ソルバーによって実行できますし、そうすべきです。これにより、静的チェック自体が形式的に検証可能であり、矛盾がないことが保証されます。

**チェーンの継承 — ベースおよび推移閉包:**

```
inherit(S, T)           -- direct inheritance: S inherits from T
inherit*(S, T)          -- transitive closure

Axioms:
  ∀S.     inherit*(S, S)                                        -- (Refl)
  ∀S T U. inherit*(S, T) ∧ inherit*(T, U) → inherit*(S, U)     -- (Trans)
  ∀S T.   inherit(S, T) → inherit*(S, T)                       -- (Base)
```

**継承チェーンのサイクル自由度 (一貫性チェック):**

```
  ∀S T.   inherit*(S, T) ∧ inherit*(T, S) → S = T              -- (AntiSym)
```

これは署名全体にわたって保持される必要があります。定義時のソルバー チェックは、不健全性を引き起こす前に循環継承を検出します。

**継承アップキャスト (FOL ディスチャージ) の有効条件:**

```
(x qua T) via inheritance is valid
⇔  ⊢_FOL  inherit*(type_radix(x), T)
```

ここで、`type_radix(x)` は、`x` の型の基数モードです。検証器はこの目標を発行し、SMT/FOL ソルバーは署名で宣言された `inherit` 公理からこの目標を排出します。

**クラスターの閉鎖 — ベースと伝播:**

```
has_attr(A, S)          -- attribute A is directly registered for mode S
cl(A, S)                -- A is in the cluster closure of S

Axioms:
  ∀A S.   has_attr(A, S) → cl(A, S)                            -- (ClBase)
  ∀A B S. cluster_implies(A, B) ∧ cl(A, S) → cl(B, S)          -- (ClProp)
```

ここで、`cluster_implies(A, B)` は、登録されたクラスター ルール `A → B` を表します。

**クラスターの影響の一貫性 (矛盾する属性がない):**

```
  ∀A S.   ¬(cl(A, S) ∧ cl(non_A, S))                           -- (ClConsist)
```

ここで、`non_A` は属性 `A` の否定を表します。この条件により、クラスター チェーンが属性とその否定を同時に導出することはできなくなります。

**クラスターベースの拡張 (FOL 放電) の有効条件:**

```
(x qua T) via cluster widening is valid
⇔  ⊢_FOL  ∀A ∈ attributes(T). cl(A, type_radix(x))
```

検証器は、ターゲット型 `T` の属性ごとに 1 つの証明義務を発行し、それぞれがクラスター閉包公理によって実行されます。

**連鎖 `qua` — ステップごとの FOL 排出:**

`x qua T1 qua T2` などの連鎖式は左結合です。

```
(x qua T1 qua T2)
≡  ((x qua T1) qua T2)

Proof obligations emitted:
  Step 1:  inherit*(type_radix(x), T1)      -- x's radix reaches T1
  Step 2:  inherit*(T1, T2)                 -- T1's radix reaches T2
```

各ステップは独立して実行されます。すべてのステップが放電する場合、チェーン全体は有効です。

**`qua` の発生ごとに発行されるソルバー証明義務の概要:**

|ケース |証明義務 |
| --- | --- |
|継承アップキャスト `x qua T` | `inherit*(type_radix(x), T)` |
|クラスターの拡大 `x qua T` |各 `A ∈ attributes(T)` の `cl(A, type_radix(x))` |
|チェーン付き `x qua T1 qua T2` |各ステップごとに独立した義務 |
|サイクルチェック (`inherit` 宣言ごとに 1 回) | `¬inherit+(T, T)` |
|クラスターの整合性 (クラスタールールごとに 1 回) | `¬(cl(A, S) ∧ cl(non_A, S))` |

### 13.8.8 括弧で囲む

括弧は FOL エンコーディングには影響しません。解析のみを制御します。

|Mizar source | FOL エンコーディング |
| --- | --- |
| `(a + b) * c` | `mul(add(a, b), c)` |
| `a + (b * c)` | `add(a, mul(b, c))` |

## 13.9 完全な構文 (EBNF)

すべての用語式構文の概要。

```ebnf
(* Term Expressions *)
term_expression      ::= operator_expression { "qua" type_expression } ;

operator_expression  ::= postfix_expression | functor_application ;
postfix_expression   ::= term_primary { term_postfix } ;

(* Primary Expressions *)
term_primary         ::= variable_identifier
                       | "it"
                       | numeral
                       | "(" term_expression ")"
                       | struct_constructor
                       | set_expression
                       | choice_expression
                       | inline_functor_application
                       | template_functor_application
                       | bracket_functor_application ;
variable_identifier ::= identifier ;
numeral            ::= digit+ ;

term_postfix        ::= "." field_name [ "(" [ term_list ] ")" ]
                      | "with" "(" field_update_list ")" ;

(* Functor Applications *)
functor_application ::= [ functor_loci ] functor_symbol [ functor_loci ] ;
functor_loci       ::= term_expression | "(" term_list ")" ;
inline_functor_application ::= inline_func_name "(" [ term_list ] ")" ;  (* mandatory parens *)
template_functor_application ::= functor_symbol template_args
                                 [ "(" [ term_list ] ")" ] ;
bracket_functor_application ::= user_symbol term_list user_symbol
                              | "[" term_list "]" ;
(* template_args is defined in Chapter 16 §16.5. *)
(* For deffunc definition syntax, see Chapter 15 (Statements) *)

(* Structure Expressions *)
structure_expression ::= struct_constructor | selector_access ;
struct_constructor   ::= struct_name [ type_args ] "(" [ field_assignment_list ] ")" ;
field_assignment_list ::= field_assignment { "," field_assignment } ;
field_assignment     ::= field_name ":" term_expression ;
selector_access      ::= term_expression "." field_name [ "(" [ term_list ] ")" ] ;
(* For field_decl with optional defaults, see Chapter 5 (Structures) *)

(* Set Expressions *)
set_expression       ::= set_enumeration | set_comprehension ;
set_enumeration      ::= "{" [ term_list ] "}" ;
set_comprehension    ::= "{" term_expression "where" typed_var_list [ ":" formula ] "}" ;
term_list            ::= term_expression { "," term_expression } ;
typed_var_list       ::= typed_var { "," typed_var } ;
typed_var            ::= identifier "is" type_expression ;  (* type must be Element of <set> *)

(* Choice Operator *)
choice_expression    ::= "the" type_expression ;

(* Type Qualification *)
qua_expression       ::= operator_expression "qua" type_expression { "qua" type_expression } ;

(* Operator Precedence Declarations *)
infix_operator_decl   ::= "infix_operator"   "(" string_literal "," infix_assoc "," nat_literal ")" ";" ;
prefix_operator_decl  ::= "prefix_operator"  "(" string_literal "," nat_literal ")" ";" ;
postfix_operator_decl ::= "postfix_operator" "(" string_literal "," nat_literal ")" ";" ;
infix_assoc           ::= "left" | "right" | "none" ;
```
