## 18.1 概要

template は **ジェネリック メカニズム**、つまり型、述語、または関数によってパラメーター化された数学的定義と定理スキーマを記述するための機能です。template は、インスタンスのファミリー全体で有効な数学的推論のパターンをキャプチャします。

template は 2 つの異なる役割を果たします。

1. **パラメーター化された定義**: 抽象型パラメーターに対する属性、モード、構造、述語、およびfunctorを定義します。たとえば、実数、複素数、または任意のフィールド K に対して汎用の `VectorSpace[K]` template をインスタンス化できます。 `PolyRing[R]` template は、任意の可換環 R 上に多項式環を生成します。

2. **定理スキーマ (スキーム)**: 述語または関数によってパラメーター化された状態定理。帰納、再帰、および普遍的な構造特性の主要なメカニズムです。これらは、古典的な Mizar `scheme` 構造を包含します。

**templateが必要な場合**

|状況 |アプローチ |
|---|---|
|型パラメータがブロック内の定義から独立している定理 |通常の普遍的に定量化された定理 — template は必要ありません |
|パラメータ化された定義と同時定義された定理 (同じブロック内の `VectorSpace[K]` に関する定理など) |template `definition` ブロック内の `theorem` 項目。囲んでいる `let` 型パラメーターを共有します。
|タイプ | によってパラメーター化された定義 (struct、mode、attr、func、pred)template (`let` 型のパラメーターを持つ `definition` ブロック) |
|述語またはfunctorによってパラメータ化された定理 | `let P be pred(T);` または `let F be func(T) -> S;` のtemplate |

**`of`/`over` 構文との関係**

モード (第 7 章) および構造 (第 5 章) の `of`/`over` パラメーター化は、型templateの **糖衣構文** です。単一パラメータと複数パラメータの両方の形式がサポートされています (1 つのパラメータの場合は `Subset of X`、2 つのパラメータの場合は `Function of X, Y`)。どちらの表記法も受け入れられます。

```mizar
:: Shorthand (Chapter 7 syntax)
definition
  mode Subset of X is X-subset_like set;
end;

:: Equivalent full template form
definition
  let X be type;
  mode Subset[X] is X-subset_like set;
  :: Automatic synonym: Subset of X
end;
```

`of`/`over` の短縮表記は、単純な場合に便利です。完全な `[...]` template形式も同様に有効で、複数パラメータの場合の構文の曖昧さを回避します (型引数と用語引数の区別など)。制約付きtemplateには完全な形式が必要です。

**古典的な Mizar schemeとの関係**

古典的な Mizar `scheme` 宣言は、一般的なtemplate システムと統合されています。 `let P be pred(T);` 項目と `theorem` 項目を含む `definition` ブロックは、一貫した構文を使用して `scheme` メカニズムを組み込みます。

> [！注記]
> template定理を適用する際に検証器をガイドするヒントについては、「[第21章 (source code annotation と ATP integration)](./21.source_code_annotation_and_atp.md)」を参照してください。

## 18.2 template宣言

### 18.2.1 基本形式: `let` パラメータを含む `definition` ブロック

template は、先頭の `let` 宣言がtemplate パラメーターを確立する `definition` ブロックです。ブロック内のすべての項目はスコープを共有します。

**構文 (EBNF):**

```ebnf
template_def    ::= "definition"
                    { let_param }
                    { template_item }
                    "end" ";" ;

let_param       ::= "let" identifier_list "be" type_expression
                    [ "such" "that" formula ( "by" label_ref_list | proof ) ]
                    ";" ;

template_item   ::= attr_def
                  | mode_def
                  | struct_def
                  | pred_def
                  | func_def
                  | algorithm_def
                  | theorem_item
                  | registration_item ;
```

**例 — パラメータ化されたマグマ構造:**

```mizar
definition
  let T be type;
  struct MagmaStr[T] where
    field carrier -> T;
    field binop -> BinOp of T;
  end;
end;
```

パラメータ化された名前 `MagmaStr[T]` は、`[...]` 括弧を使用して、名前に型引数が含まれていることを示します。この括弧表記はパラメータ化されたtemplate名とスキームの適用のために**予約されています**。他の目的 (リストのインデックス付けなど) に使用してはなりません。パラメータがコンテキストから一意に推測できる場合、`[...]` 引数リストは省略できます (§18.2.7 を参照)。

該当する場合、自動同義語が生成されます。

- `Module over R` は `Module[R]` (単一の構造体パラメーター) の同義語になります。
- `Subset of X` は `Subset[X]` (シングル モード パラメータ) の同義語になります

### 18.2.2 型パラメータ

`let X be <type>;` は、型 (モードまたは構造体) にわたる **型パラメーター** を宣言します。

|宣言 |意味 |
|---|---|
| `let T be type;` | T は任意の型 |
| `let G be Group;` | G は `Group` モードを満たす必要があります |
| `let R be commutative Ring;` | R は可換リングでなければなりません |

```mizar
:: Single type parameter: polynomial ring over a commutative ring
definition
  let R be commutative Ring;
  struct PolyRing[R] where
    field carrier -> non empty set;   :: the underlying set of polynomials
    field add -> BinOp of carrier;
    field mul -> BinOp of carrier;
    field coeff -> Function of [:carrier, Nat:], R.carrier;
  end;
  :: Automatic synonym: PolyRing over R
end;

:: Two type parameters: generic homomorphism mode
definition
  let G, H be Group;
  mode Homomorphism[G,H] is
    operation_preserving Function of G.carrier, H.carrier;
  :: Automatic synonym: Homomorphism of G, H
end;
```

**有界型パラメータ**

`let T be type extends M` は **有界型変数** を宣言します。T は、M の特定のインスタンスではなく、(`inherit ... from ...` チェーンを介して) M から継承する任意の構造体を範囲とします。M がモードの場合、T はさらに、M を定義するすべての属性を満たす必要があります。たとえば、`let T be type extends commutative Magma` では、T の構造体が Magma から継承するだけでなく、T が `commutative` 属性を満たすことも必要です。

|宣言 |意味 |
|---|---|
| `let T be type extends Group;` | T は、Group | から継承する任意の構造です。
| `let T be type extends commutative Magma;` |可換マグマによって境界付けられた T |
| `let K be Field; let T be type extends VectorSpace[K];` |パラメーター化モードによって制限される T |

属性フォームは `such that` 制約の糖分です。`let T be type extends commutative Magma` は `let T be type extends Magma such that T is commutative` と同等です。

**`qua`強制によるインスタンス化**

**モード名**を直接指定する場合は、バインドされたモードを明確に識別する必要があります。目的のモード名を次のように記述します。

```mizar
PermProduct[AddMagma]            :: OK — mode name directly specifies additive Magma
PermProduct[commutative MulMagma]  :: OK — multiplicative Magma
PermProduct[Ring]                :: error — Ring reaches Magma via both AddGroup and MulMonoid
```

構造にバインド モードへの複数の継承パスがある **変数**が指定された場合、呼び出し元は `V qua M` を使用して目的のパスを選択する必要があります。

```mizar
let R be commutative Ring;
PermProduct[R qua AddMagma]   :: select R's additive Magma structure
PermProduct[R qua MulMagma]   :: select R's multiplicative Magma structure
```

V のモードが `inherit ... from M` チェーンを介して M から継承している場合、`V qua M` は整形式です。これは、§13.6 の `qua` 型修飾と同じ演算子です。 `[...]` 括弧で囲むことにより、外側の括弧がオプションになります。

**マルチレベル継承のための `qua` のチェーン**

継承チェーンが複数のレベルにまたがっており、中間ステップに明示的に名前を付ける必要がある場合は、`qua` を左から右にチェーンできます。

```mizar
PermProduct[R qua AddGroup qua Magma]
:: first selects R's AddGroup view, then that structure's Magma view
```

各ステップは、`inherit ... extends ...` チェーンに沿った有効な拡幅でなければなりません。

**表記の再マッピング**

`let T be type extends M` のtemplate 内では、M の表記が適用されます。 `qua` 経由でインスタンス化されると、表記法は `inherit` チェーンで宣言されたフィールド マッピングを通じて再マップされます。

```mizar
inherit AddGroup extends Group where
  field add  from mul;
  field zero from unit;
  field neg  from inv;
end;
```

特定のリング変数 R に対して `[R qua AddGroup]` として適用された `*` (グループの `mul` 表記) を使用する定理は、使用サイトでは `+` (AddGroup の `add` 表記) として表示されます。

**例 — 有限積の順列不変性:**

```mizar
definition
  let T be type extends commutative Magma;
  theorem PermProduct[T]:
    for s being FinSequence of T,
        p being Permutation of dom s
    holds Product[T](s) = Product[T](s * p)
  proof
    :: by induction on len s, using commutativity of T
    ...
  end;
end;

:: Application: sum permutation in any Ring (AddMagma directly — unambiguous)
PermProduct[AddMagma]

:: Application: product permutation in any commutative Ring
PermProduct[commutative MulMagma]

:: Application to a specific ring variable (qua-coercion on the variable)
let R be commutative Ring;
PermProduct[R qua AddMagma]   :: use R's additive structure
```

### 18.2.3 述語パラメータ

`let P be pred(T₁, ..., Tₙ);` は、n 項述語にわたる **述語パラメーター** を宣言します。述語パラメーターは、同じ `definition` ブロック内の `theorem` 項目内でのみ有効です。

述語パラメーターは `defpred` 規則に従い、定理本体内で `P(t₁, ..., tₙ)` として適用されます。

```mizar
definition
  let P be pred(Nat);
  theorem NatInduction[P]:
    P(0) & (for n being Nat st P(n) holds P(n+1))
    implies for n being Nat holds P(n)
  proof
    assume Base: P(0);
    assume Step: for n being Nat st P(n) holds P(n+1);
    let n be Nat;
    thus P(n) by Base, Step, mml.number.natural.Nat_induction;
  end;
end;
```

スキーマ定理は通常の含意定理です。呼び出し元は、先行条件を名前付きラベルとして証明し、`by NatInduction[P], Label1, Label2` 経由で引用します。

### 18.2.4 functorパラメータ

`let F be func(T₁, ..., Tₙ) -> S;` は、指定されたシグネチャの関数にわたる **functorパラメーター** を宣言します。functorパラメーターは、同じ `definition` ブロック内の `theorem` 項目内でのみ有効です。

functorパラメーターは `deffunc` 規則に従い、定理本体内で `F(t)` として適用されます。

```mizar
definition
  let A be type;
  let F be func(A) -> A;
  let P be pred(A);
  theorem ClosedUnderCompose[F,P]:
    (for x being A st P(x) holds P(F(x)))
    implies for x being A st P(x) holds P(F(F(x)))
  proof
    assume Step: for x being A st P(x) holds P(F(x));
    let x be A;
    assume Px: P(x);
    thus P(F(F(x))) by Px, Step;
  end;
end;
```

**functorパラメーターと一階関数**

functorパラメーター `let F be func(T) -> S;` は **スキーマ レベルの関数シンボル** です。これは談話領域内のセットではなく、`is` 述語推論をサポートしません。これは、セット要素であるモード (`let phi be Homomorphism[G,H];` など) を介して定義された 一階関数とは区別されます。

|種類 |宣言 | FOLステータス | `is` 述語 |
|---|---|---|---|
|一階関数 (モードベース) | `let phi be Homomorphism[G,H];` |要素を設定 |サポートされている |
|スキーマfunctorパラメータ | `let F be func(T) -> S;` |メタレベルシンボル |サポートされていません |

関数パラメーターに代数プロパティが必要な場合は、モードベースの型宣言 (`let phi be Homomorphism[G,H];`) を使用します。 `func(...)` パラメータは、代数制約が必要ないスキーマ用に予約してください。

### 18.2.5 制約付きパラメータ

型パラメータがモード以外の追加条件を満たす必要がある場合は、`such that` でそれを指定します。呼び出し元は、使用される特定のインスタンスに制約が適用されるという証拠を提供する必要があります。

```mizar
definition
  let F be Field;
  let V be VectorSpace[F] such that V is finite_dimensional;
  :: Constraint: verified at each use site
  mode Basis[V] is finite linearly_independent Subset of V.carrier;
end;
```

使用サイトでは、呼び出し元は制約を名前付きラベルとして解除する必要があります。

```mizar
let F be REAL_FIELD;
let W be finite_dimensional VectorSpace[REAL_FIELD];
let phi be Homomorphism[W, W];
set V = image(phi);
reconsider V as VectorSpace[REAL_FIELD] by mml.algebra.linear.hom_image_vspace;
FinDim: V is finite_dimensional by mml.algebra.linear.image_fin_dim;
:: now Basis[V] is usable — constraint FinDim is discharged
let B be Basis[V] by FinDim;
```

**`such that` が必要な場合**: ブロック内の **定義項目** (モード、属性、functor、構造体) が整形式であるための制約を必要とする場合に使用します。定理のみが条件を使用する場合、**不適切**です。その場合は、代わりに通常の含意仮説を記述します。

```mizar
:: Preferred: express finite-dimensionality as a theorem hypothesis
theorem BasisCard:
  for F being Field, V being VectorSpace[F] st V is finite_dimensional
  holds card (Basis of V) = dim V;
:: No template needed — no definition item depends on the constraint
```


### 18.2.6 スコープ規則

- `definition` ブロックの先頭にある `let` 宣言は、ブロック内の **すべての項目** を対象とします。これは、定義ブロック内の `let IT be ...;` の古典的な Mizar スコープ規則に従います。
- 述語パラメーター (`let P be pred(T);`) とfunctorパラメーター (`let F be func(T) -> S;`) の範囲は、ブロック内の個々の `theorem` および `algorithm` 項目です。 `attr`、`mode`、`struct`、`func`、または `pred` アイテムでは使用できません。
- パラメータ化された名前 `N[params]` は、閉じている `end;` の後の、囲んでいる **module** 内で表示されます。明示的な `import` がなければ、他のmoduleからアクセスすることはできません。
- パラメーター化された定義を持つ `definition` ブロックは、別の `definition` ブロック内または `proof` ブロック内にネストすることはできません。
- パラメータ化された定義間の循環依存関係は禁止されています。

### 18.2.7 templateパラメータの推論

templateが適用されるとき、すべてのtemplate パラメーターが他の引数の型から **一意に推測**できる場合、`[...]` 引数リストは**省略**できます。このルールは、`func` および `pred` templateに適用されます。

```mizar
:: Explicit form — always valid
Product[Ring](s)
a +[Ring] b
s are_congruent[H]

:: Implicit form — valid when type is uniquely determined
Product(s)           :: T inferred from the element type of s
a + b                :: resolved from the types of a and b
s are_congruent(H)   :: H's type determines the pred
```

**推論ルール**: `[T]` 引数が省略された場合、エラボレータは残りの引数の型からパラメータを決定します。パラメータを一意に決定できる場合、暗黙的なインスタンス化は成功します。 2 つ以上のインスタンス化が一致する場合、あいまいさエラーが報告され、明示的な `[T]` が必要になります。

```mizar
:: Ambiguous — error, explicit required
PermProduct[Ring]       :: Ring reaches Magma via AddGroup and via MulMonoid

:: Unambiguous — AddMagma directly names the desired mode
PermProduct[AddMagma]   :: OK

:: When a specific ring variable is in scope, qua-coercion on the variable
let R be commutative Ring;
PermProduct[R qua AddMagma]   :: use R's additive structure specifically
```

**`qua` 強制は常に明示的です**: `V qua M` 強制 (§18.2.2) は決して推論されません。特定の変数 (例: `R qua AddMagma`) に適用して複数の継承パスの 1 つを選択する場合は、明示的に記述する必要があります。モード名自体があいまいでない場合 (`AddMagma` など)、`qua` 強制は必要ありません。

推論メカニズムは、古典的な Mizar オペレーター定義にすでに使用されているoverload解決を完全なtemplate システムに均一に拡張します。

## 18.3 属性のtemplate (attr)

属性template は、型パラメーターによってインデックス付けされた属性のファミリーを定義します。template パラメータは、サブジェクトのタイプから**独立**する必要があります。template パラメータには、サブジェクト自体によってまだ決定されていない情報が含まれます。

**第 6 章の属性構文との関係**

第 6 章のハイフンでつながれた属性フォームは、ルール `G-attrname` ↔ `attrname[G]` に従って、単一パラメータのtemplate フォームの糖衣構文です。

|シュガーフォーム（第6章） |templateフォーム |
|---|---|
| `n is G-valid_exponent` | `n is valid_exponent[G]` |
| `S is M-dense` | `S is dense[M]` |

どちらの形式でも受け付けられます。 `[G]` フォームは複数パラメータ属性に必要です。

**例 1 — 群の指数:**

```mizar
definition
  let G be Group;
  let n be positive Nat;
  attr ExponentDef: n is valid_exponent[G] means
    for g being Element of G holds g^n = G.unit;
  :: Automatic synonym: n is G-valid_exponent
end;
```

`g^n = G.unit` は、G の n 乗されたすべての要素が単位に等しいことを意味します。

**使用法：**

```mizar
let G be finite Group;
assume 12 is valid_exponent[G];
:: equivalently: assume 12 is G-valid_exponent
```

**例 2 — 計量空間内の密なサブセット:**

```mizar
definition
  let M be MetricSpace;
  let S be Subset of M.carrier;
  attr DenseDef: S is dense[M] means
    for x being Element of M, eps being positive Real holds
      ex s being Element of S st dist_M(x, s) < eps;
  :: Automatic synonym: S is M-dense
end;
```

**使用法：**

```mizar
let M be MetricSpace;
let S be Subset of M.carrier;
assume S is dense[M];
:: equivalently: assume S is M-dense
```

## 18.4 モードのtemplate (mode)

モード template は、型パラメーターによってインデックス付けされた型のファミリーを定義し、`of`/`over` パラメーター化を一般化します。

**例 1 — 一般サブオブジェクト モード:**

```mizar
definition
  let A be type;
  mode Subobject[A] is A-subset_like set;
  :: Automatic synonym: Subobject of A
end;

:: Usage
let S be Subobject[G];
:: equivalently: let S be Subobject of G;
```

**例 2 — 一般準同型モード (2 つのパラメーター):**

```mizar
definition
  let G,H be Group;
  mode Homomorphism[G,H] is
    operation_preserving Function of G.carrier, H.carrier;
  :: Automatic synonym: Homomorphism of G, H
end;

:: Usage
let phi be Homomorphism[G, H];
```

**例 3 — 制約モード:**

```mizar
definition
  let F be Field;
  let V be VectorSpace[F] such that V is finite_dimensional;
  mode FinBasis[V] is finite linearly_independent Subset of V.carrier;
  :: FinBasis[V] only makes sense for finite-dimensional V
end;
```

使用サイトでは、呼び出し元は制約を解除する必要があります。

```mizar
let F be REAL_FIELD;
let V be VectorSpace[REAL_FIELD];
FinDim: V is finite_dimensional proof ... end;
let B be FinBasis[V] by FinDim;  :: valid — FinDim discharges the constraint
```

## 18.5 構造体のtemplate (struct)

構造template はパラメータ化された複合型を定義します。

**例 1 — フィールド上の一般的なベクトル空間:**

```mizar
definition
  let K be Field;
  struct VectorSpace[K] where
    field carrier -> non empty set;
    field add -> BinOp of carrier;
    field smul -> Function of [:K.carrier, carrier:], carrier;
  end;
  :: Automatic synonym: VectorSpace over K
end;

:: Usage: instantiated for real numbers
let V be VectorSpace[REAL_FIELD];
:: equivalently: let V be VectorSpace over REAL_FIELD;
```

**例 2 — 継承を伴う一般的な順序構造:**

```mizar
definition
  let T be set;
  struct OrdStr[T] where
    field carrier -> T;
    field order -> Relation of T;
  end;
  :: Automatic synonym: OrdStr over T

  inherit OrdStr[T] from 1-sorted where
    field carrier from carrier;
  end;
end;
```

**例 3 — 複数の型パラメーターを含む構造体:**

```mizar
definition
  let F be Field;
  let V,W be VectorSpace[F];
  struct BilinearMap[F,V,W] where
    field map -> Function of [:V.carrier, W.carrier:], F.carrier;
    property kernel -> Subobject[V];
  end;
  :: Automatic synonym: BilinearMap over F, V, W
end;

:: Usage
let B be BilinearMap[REAL_FIELD, V, W];
```

## 18.6 述語のtemplate (pred)

述語template は、型パラメーターによってインデックス付けされた述語のファミリーを定義します。

**例 1 — 一般的な合同述語:**

```mizar
definition
  let G be Group;
  let H be Subgroup of G;
  let a, b be Element of G;
  pred GroupCongruencePred: a, b are_congruent[H] means
    a * b" in H;
end;

:: Usage
let G be Group;
let H be Subgroup of G;
let a, b be Element of G;
assume a, b are_congruent[H];
```

**例 2 — イデアルを法とする合同:**

```mizar
definition
  let R be commutative Ring;
  let I be Ideal of R;
  let a, b be Element of R;
  pred RingCongruencePred: a, b are_congruent_mod[I] means
    a - b in I.carrier;
  reflexivity
  proof
    let a be Element of R;
    thus a - a in I.carrier by mml.algebra.ring.Ring_sub_self_zero, I.zero_in_ideal;
  end;
  symmetry
  proof
    let a, b be Element of R;
    assume a - b in I.carrier;
    thus b - a in I.carrier by mml.algebra.ring.Ring_neg_closed, I.ideal_neg_closed;
  end;
  transitivity
  proof
    let a, b, c be Element of R;
    assume a - b in I.carrier;
    assume b - c in I.carrier;
    thus a - c in I.carrier by mml.algebra.ring.Ring_add_closed, I.ideal_add_closed;
  end;
end;
```

## 18.7 functorのtemplate (func)

functor template は、型パラメーターによってパラメーター化された操作のファミリーを定義します。

**例 1 — 一般的な恒等関数:**

```mizar
definition
  let T be type;
  func IdDef: id[T] -> Function of T, T equals
    { [x, x] where x is Element of T : x in T };
end;

:: Usage
let f = id[NAT];
```

**例 2 — 一般的な中立要素:**

```mizar
definition
  let G be Group;
  func UnitDef: unit[G] -> Element of G means
    for a being Element of G holds it * a = a & a * it = a;
  existence
  proof
    consider e being Element of G such that
      A1: for a being Element of G holds e * a = a & a * e = a
      by mml.algebra.group.Group_identity_exists;
    take e;
    thus thesis by A1;
  end;
  uniqueness
  proof
    let e1, e2 be Element of G;
    assume A1: for a being Element of G holds e1 * a = a & a * e1 = a;
    assume A2: for a being Element of G holds e2 * a = a & a * e2 = a;
    thus e1 = e1 * e2 by A2
           .= e2 by A1;
  end;
end;
```

## 18.8 パラメータ化された定理とスキーム

パラメーターが **集合** (`Group` や `Ring` などのモードによって制約された集合を含む) である定理は、通常の全称量化された定理として記述することができます。`for G being Group` は `for G being set st is_Group(G)` と同等であるため、一階になります。

```mizar
theorem CommMul:
  for G being commutative Group, a, b being Element of G holds a * b = b * a;
```

定理が次のようにパラメータ化される場合、templateが必要です。
- **タイプ** (`let T be type`) — `for T being type` は通常の定理ステートメントでは**許可されません**。
- **述語** (`let P be pred(T)`) — スキームスタイルのパラメータ化。
- **functor** (`let F be func(T) -> S`) — 関数シンボルによるパラメーター化。

### 18.8.1 型パラメータ化された定理

`definition` ブロック内のパラメーター化された定義と同時定義された定理は、ブロックの `let T be type` パラメーターを共有できます。これは、抽象型にわたる定理に必要な形式でもあります。

```mizar
definition
  let T be type;
  theorem FinSeqConcat[T]:
    for s, t being FinSequence[T]
    holds len (s ^ t) = len s + len t
  proof
    ...
  end;
end;
```

### 18.8.2 述語でパラメータ化された定理 (スキーム)

囲んでいる `definition` ブロック内の `let P be pred(T);` を含む定理は **述語スキーマ** です。これは、一階定理のファミリーを生成し、述語ごとに 1 つが `P` に置き換えられます。証明内で `assume Label: ...` を使用して前件条件が導入されます。

```mizar
definition
  let P be pred(Nat);
  theorem NatInduction[P]:
    P(0) & (for n being Nat st P(n) holds P(n+1))
    implies for n being Nat holds P(n)
  proof
    assume Base: P(0);
    assume Step: for n being Nat st P(n) holds P(n+1);
    let n be Nat;
    thus P(n) by Base, Step, mml.number.natural.Nat_induction;
  end;
end;
```

### 18.8.3 functorでパラメーター化された定理

`let F be func(T) -> S;` の定理は、ブラック ボックスとして扱われる **スキーマ レベルの関数シンボル** によってパラメーター化されます。関数で代数的プロパティが必要な場合は、代わりにモードベースのパラメーターを使用してください (§18.2.4 を参照)。

次の定理は、`F` が `P` および `P(x, F(x))` に関して単調である場合、`P(F(x), F(F(x)))` を示します。これは、上昇チェーンと固定点の理論における基本的なステップであり、どの `F` およびどの次数 `P` にも当てはまります。

```mizar
definition
  let A be type;
  let P be pred(A, A);
  let F be func(A) -> A;
  theorem MonotoneStep[P,F]:
    (for x, y being A st P(x, y) holds P(F(x), F(y)))
    implies for x being A st P(x, F(x)) holds P(F(x), F(F(x)))
  proof
    assume Mono: for x, y being A st P(x, y) holds P(F(x), F(y));
    let x be A;
    assume P(x, F(x));
    thus P(F(x), F(F(x))) by Mono;
  end;
end;
```

### 18.8.4 アルゴリズムtemplate

`algorithm_def` は `template_item` として現れる場合があり、型、述語、またはfunctorによってパラメーター化された **高階アルゴリズム** を提供します。 3 種類のスキーマ パラメーター (`type`、`pred`、`func`) はすべて、アルゴリズム本体および `requires`/`ensures` 句で自由に参照できます。

アルゴリズム名には、スキーマ定理と同じ規則に従って、すべてのスキーマ パラメーター (type、pred、および func など) をリストする `[...]` 括弧接尾辞が付けられます。各呼び出しサイトで、ブラケット引数が具体的な型、pred、または func 名でインスタンス化されます。

pred または func パラメータが実行可能コンテキストで呼び出され、具体的なインスタンス化に計算可能な実装がない場合、MVM は **実行時検証エラー** (§20.9.3) を生成します。アルゴリズム定義自体は数学的に有効なままです。

**例 — `iterate[T, F]`: functorを n 回適用します (type パラメーターと func パラメーター)**

```mizar
definition
  let T be type;
  let F be func(T) -> T;
  let n be Nat;
  let x be T;

  terminating algorithm iterate[T, F](n, x) -> T
    ensures result = F applied n times to x
    decreasing n
  do
    if n = 0 do return x; end;
    return F(iterate[T, F](n - 1, x));
  end;
end;
```

インスタンス化では、型とfunctorの両方が提供されます。

```mizar
deffunc double(i) = 2 * i;
const result = iterate[Integer, double](5, 1);   :: double applied 5 times to 1 = 32
```

**例 — `sigma[F]`: 整数範囲にわたるfunctorの合計**

```mizar
definition
  let F be func(Integer) -> Integer;
  let lo, hi be Integer;

  terminating algorithm sigma[F](lo, hi) -> Integer
    ensures result = Sigma(F, lo, hi)
  do
    var s := 0;
    for i = lo to hi do
      s := s + F(i);
    end;
    return s;
  end;
end;
```

**例 — `count[P]`: 述語を満たす範囲内の整数の数**

```mizar
definition
  let P be pred(Integer);
  let lo, hi be Integer;

  terminating algorithm count[P](lo, hi) -> Nat
    ensures result = card {i being Integer : lo <= i & i <= hi & P(i)}
  do
    var n := 0;
    for i = lo to hi do
      if P(i) do
        n := n + 1;
      end;
    end;
    return n;
  end;
end;
```

**インスタンス化**

呼び出しサイトでは、括弧引数は具体的な pred/func 名です。

```mizar
:: deffunc as functor argument
deffunc square(i) = i * i;
const sum_of_squares = sigma[square](1, 10);   :: 1^2 + 2^2 + ... + 10^2

:: defpred as predicate argument
defpred IsOdd(i) means i mod 2 = 1;
const odd_count = count[IsOdd](1, 100);        :: number of odd integers in [1,100]

:: terminating algorithm (promoted to func) passed directly
const sum_of_factorials = sigma[factorial](1, 5);
```

**二階で十分です。** pred および func パラメータは (一階オブジェクトではなく) スキーマ レベルであるため、それらによってパラメータ化されたアルゴリズムは二階です。これは、専用の関数型コンストラクターを必要とせずに、すべての標準的な高階パターン (`sigma`、`product`、`count`、`fold`、コンパレーター付き `sort` など) をカバーします。

## 18.9 証明におけるスキームの適用

述語またはfunctorのスキーマは、次の方法で証明に適用されます。

1. 述語引数を `defpred` で定義します (またはfunctor引数を `deffunc` で定義します)。
2. 現在の証明コンテキストで名前付きラベルとして含意の先行条件を証明します。
3. `by SchemaName[P], Label1, Label2, ...` によるスキーマ定理の参照。

**構文 (EBNF):**

```ebnf
scheme_app  ::= "by" theorem_label "[" scheme_arg { "," scheme_arg } "]"
                { "," label_reference } ";" ;

scheme_arg  ::= defpred_identifier | deffunc_identifier ;
```

> [！重要]
> インライン λ スタイルの述語引数 (例: `by NatInduction[pred n means n * 2 is even]`) はサポートされません**。スキーマを適用する前に、必ず `defpred` (または `deffunc`) を導入してください。これにより可読性が向上し、検証器が述語定義を明確に解決できるようになります。

**例 — 自然数帰納法を適用する:**

```mizar
theorem NatMulEven:
  for n being Nat holds n * 2 is even
proof
  defpred P(n) means n * 2 is even;
  Base: P(0)
  proof
    thus 0 * 2 is even by mml.number.nat.zero_even;
  end;
  Step: for n being Nat st P(n) holds P(n+1)
  proof
    let n be Nat;
    assume A1: n * 2 is even;
    n * 2 + 2 = (n + 1) * 2 by mml.number.nat.distrib;
    thus (n + 1) * 2 is even by A1, mml.number.nat.even_plus_even;
  end;
  thus for n being Nat holds P(n) by NatInduction[P], Base, Step;
end;
```

## 18.10 論理エンコーディングの詳細

template はメタレベルの構成要素です。各 **使用サイト** (パラメーター化された名前が使用されるか、スキーマが適用される場所) は、一階公理と証明義務のコレクションを生成します。

### 18.10.1 公理スキーマとしてのtemplate

`let` パラメーターを持つ `definition` ブロックは、**公理スキーマ**、つまりパラメーター化された名前が使用されるか、スキーマ定理が参照されるときに FOL 公理を生成するためのレシピとして扱われます。

|templateレベル | FOL ステータス |
|---|---|
| `let` パラメーターを含む `definition` ブロック | FOL 公理は生成されません (スキーマのみ) |
|パラメータ化された名前の使用 (例: `let M be Module[R];`) | `R` のインスタンス化の公理を生成します。
|スキーマ アプリケーション (`by NatInduction[P], Base, Step`) |特定の一階インスタンスを生成します |

**検証保証**: 検証器は、宣言時に各template本体を**シンボル的に 1 回**チェックします。本体内のすべての証明義務は、パラメータ制約に関連して免除されます。これにより、再証明することなく、すべての使用で有効な公理が生成されることが保証されます。

### 18.10.2 型パラメータのエンコーディング

型パラメーター `let T be type;` は、スキーマ内で **解釈されない述語記号** `is_T(x)` としてエンコードされます。

|出典 | FOL スキーマ |
|---|---|
| `let T be type;` |スキーマ変数 `is_T/1` (未解釈) |
| `let G be Group;` |ガード付きスキーマ変数 `is_G/1` `∀x. is_G(x) → is_Group(x)` |
| `let T be type extends M;` |ガード付きスキーマ変数 `is_T/1` `∀x. is_T(x) → is_M(x)` |
| `let x be T;` |ガード `is_T(x)` が先行例として追加されました |
|使用サイトの `v qua N` (v 変数、v の境界の N 祖先) |スキーマに新しい公理は追加されません。検証器は、FOL 有効性義務として `inherit*(type_radix(v), N)` を発行および解除します ([§13.8.7](./13.term_expression.md#1387-qua-演算子のエンコーディング) を参照)。 v の N 構造を選択します |

**例：**

`G, H` が `Group` である `Homomorphism[G, H]` の FOL スキーマ:

```
∀G, H. (∀x. is_G(x) → is_Group(x)) ∧ (∀x. is_H(x) → is_Group(x))
  → ∀f. (is_Hom(f, G, H) ↔ is_operation_preserving(f, G, H))
```

`G := INT_GROUP`、`H := REAL_GROUP` の用途:

```
∀f. (is_Hom(f, INT_GROUP, REAL_GROUP)
       ↔ is_operation_preserving(f, INT_GROUP, REAL_GROUP))
```

### 18.10.3 述語パラメータのエンコーディング

述語パラメータ `let P be pred(T);` は、**解釈されない述語シンボル** `P/1` としてエンコードされます。スキーマは **一階式のファミリー** (`P` に置換される述語ごとに 1 つ) を表します。 FOL は二階の量化をサポートしていないため、各インスタンス化は個別の 一階公理を生成します。

|出典 | FOL スキーマ |
|---|---|
| `let P be pred(T);` |スキーマ変数 `P/1` (未解釈) |
|式中の `P(x)` |生成された式の `P(x)` |
| `for x st P(x) holds Q` |生成された式の `∀x. (P(x) → Q)` |

**例 — NatInduction スキーマ:**

FOL スキーマ (置換された述語 `P` ごとに 1 つのインスタンス):

```
P(0) ∧ (∀n. is_Nat(n) → P(n) → P(succ(n)))
  → ∀n. is_Nat(n) → P(n)
```

これはまさに FOL における **自然数帰納公理スキーマ** です。信頼できるkernel は、高階の推論を必要とせずに、特定の各インスタンス (`P(n) := is_even(n * 2)` など) を検証します。

### 18.10.4 functorパラメータのエンコーディング

functorパラメーター `let F be func(T) -> S;` は、型ガード公理を使用して **解釈されない関数シンボル** `F/1` としてエンコードされます。

|出典 | FOL スキーマ |
|---|---|
| `let F be func(T) -> S;` |ガード付きの未解釈関数 `F/1` `∀x. is_T(x) → is_S(F(x))` |
|期間中 `F(x)` | `F(x)` |

**例 — func パラメーターを使用した `ClosedUnderCompose`:**

`ClosedUnderCompose[F,P]` の FOL スキーマ (§18.2.4)、`F` はスキーマfunctor、`P` はスキーマ述語です。

```
(∀x. is_A(x) ∧ P(x) → is_A(F(x)))          (* F: A → A guard *)
∧ (∀x. is_A(x) ∧ P(x) → P(F(x)))            (* antecedent: F preserves P *)
  → ∀x. is_A(x) ∧ P(x) → P(F(F(x)))         (* conclusion *)
```

各具体的なインスタンス化 (`F := succ`、`P(n) := n is even` など) は、個別の一階公理を生成します。

### 18.10.5 制約エンコーディング

パラメーター制約 (`such that` 句) は **条件付きガード** としてエンコードされます。つまり、証明義務はサイトの使用に延期されます。

|出典 | FOL効果 |
|---|---|
| `let V be T such that φ` |制約 `φ` は、各使用現場で証明する必要があります。生成されたすべての公理において前件として機能します。
| `implies` 節の仮説 |生成された FOL の含意における前件 |

**例 — 定義項目の場合は `such that`:**

```mizar
definition
  let F be Field;
  let V be VectorSpace[F] such that V is finite_dimensional;
  mode FinBasis[V] is finite linearly_independent Subset of V.carrier;
end;
```

使用サイト `F := F₀`、`V := V₀` で生成された公理 (呼び出し元が `V₀ is finite_dimensional` を証明した後):

```
is_Field(F₀) ∧ is_VectorSpace(V₀, F₀) ∧ is_finite_dimensional(V₀)
  → ∀x. (is_FinBasis(x, V₀) ↔ is_finite(x) ∧ is_linearly_independent(x, V₀))
```

**例 — 定理仮説の場合は `implies`:**

```mizar
theorem BasisCard:
  for F being Field, V being VectorSpace[F] st V is finite_dimensional
  holds card (Basis of V) = dim V;
```

FOL エンコーディング (template は必要ありません - 通常の普遍的な定量化):

```
∀F, V. (is_Field(F) ∧ is_VectorSpace(V, F) ∧ is_finite_dimensional(V)
         → card(Basis(V)) = dim(V))
```

## 18.11 完全な構文 (EBNF)

```ebnf
(* === Template Declaration (definition block with let parameters) === *)
template_def    ::= "definition"
                    { let_param }
                    { template_item }
                    "end" ";" ;

let_param       ::= "let" identifier_list "be" let_type
                    [ "such" "that" formula
                      ( "by" label_ref_list | proof ) ]
                    ";" ;

(* === Type in let_param: plain type expression, or abstract type with optional bound === *)
let_type        ::= "type" [ "extends" bound_type ]
                  | type_expression ;

bound_type      ::= [ attribute_chain ] radix_type ;

template_item   ::= attr_def
                  | mode_def
                  | struct_def
                  | pred_def
                  | func_def
                  | algorithm_def
                  | theorem_item
                  | registration_item ;

(* === Predicate Parameters (in let_param; valid in theorem and algorithm items) === *)
pred_param      ::= identifier "be" "pred" "(" type_list ")" ;

type_list       ::= type_expression { "," type_expression } ;

(* === Functor Parameters (in let_param; valid in theorem and algorithm items) === *)
func_param      ::= identifier "be" "func" "(" type_list ")" "->" type_expression ;

(* === Parameterized Names (reserved bracket notation) === *)
(* The "[" type_arg_list "]" part may be omitted when parameters are uniquely
   inferrable from the argument types; see §18.2.7. *)
param_name      ::= identifier [ "[" type_arg_list "]" ] ;
type_arg_list   ::= type_arg { "," type_arg } ;
type_arg        ::= type_expression | qua_arg ;
qua_arg         ::= identifier { "qua" radix_type } ;
    (* identifier must be a variable (not a mode name); each "qua" step is a
       widening along an inherit chain. Multiple qua steps are left-associative:
         R qua AddGroup qua Magma
       first selects R's AddGroup view, then that view's Magma ancestor.
       Well-formed iff each step follows some "inherit ... from ..." chain. *)

(* === Theorem Items with Predicate/Functor Parameters === *)
(* Schema theorems are ordinary implication theorems:
   hypothesis conditions appear as antecedents of the formula itself. *)
theorem_item     ::= [ theorem_status ] theorem_role label_identifier ":"
                     formula
                     [ justification ] ";" ;
theorem_status   ::= "open" | "assumed" | "conditional" ;
theorem_role     ::= "theorem" | "lemma" ;

(* === Scheme Application in Proofs === *)
(* scheme_name is the label of a schema theorem (e.g. NatInduction).
   Arguments are passed via template_args (defined in Chapter 16 §16.5):
     - For predicate/functor parameters: defpred_identifier | deffunc_identifier
     - For type parameters: type_expression | qua_arg
   Additional proof-local labels follow after the template_args. *)
scheme_app      ::= "by" scheme_name template_args
                    { "," label_reference } ";" ;
    (* template_args  ::=  "[" template_arg { "," template_arg } "]"
       template_arg   ::=  type_expression | coercion_arg
                         | defpred_identifier | deffunc_identifier
       — see Chapter 16 §16.5 for the full definition *)

scheme_name     ::= label_identifier ;

(* === Predicate/Functor Application (following defpred/deffunc convention) === *)
pred_app        ::= identifier "(" term_list ")" ;
func_app        ::= identifier "(" term_list ")" ;
```
