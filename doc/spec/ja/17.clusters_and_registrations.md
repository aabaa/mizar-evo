## 17.1 概要

Mizar のソフトタイピング システム ([第3章 (型システム)](./03.type_system.md) を参照) では、型は **基数型** (モードまたは構造) と 0 個以上の **属性** (形容詞) で構成されます。クラスターは、属性の組み合わせに関する事実を登録することで、個々の属性定義と型推論エンジンの間のギャップを埋めます。

登録には次の 4 種類があります。

|登録種類 |目的 |正確性の条件 |
|---|---|---|
| **実存的** |指定された属性を持つ型が存在することを証明する | `existence` |
| **条件付き** |ある属性セットが別の属性セットを暗黙的に示すことを宣言します。 `coherence` |
| **関数** |関手の結果に特定の属性があることを宣言します。 `coherence` |
| **削減** |用語を別の用語に簡略化できることを宣言します | `reducibility` |

すべての登録は `registration ... end` ブロック内に表示され、登録が発効する前に履行する必要がある証明義務を生成します。

有効化は item 順で決定的です。定義 module 内では、登録 item は自身の correctness
condition が受理された後の後続 item に対して有効になります。受理は後続の VC、proof、
kernel、artifact phase により非同期に生成されてよいです。実装はその status が未解決の間、
module または依存する use site を pending に保持してよいですが、完了した verification pass は、
受理済みの先行 registration が source order 上の後続 item に対して有効であるかのように
振る舞わなければなりません。先行 registration の accepted status が後続 phase または artifact
から到着したという理由だけで、後続 use を最終的に reject してはなりません。同じ module の
以前の item や、同じ registration block 内の以前の statement に retroactive には適用されません。
import された registration は、通常の import visibility に従って、importing module の非 import
body の先頭から有効です。証明が pending、unavailable、または failed の registration は、黙って
fallback として使われません。依存する resolution site は、pending のままにするか、期待した
registration が利用できないことを診断しなければなりません。

> [！重要]
> クラスターの登録は **グローバル公理**です。一度登録されると、現在のmoduleとそれをインポートするすべてのmodule全体に適用されます。登録が正しくないと、型システムの健全性が損なわれる可能性があります。このため、各登録には正当性の証明が必要です。

## 17.2 登録ブロック

すべてのクラスターおよびリダクション登録は、`registration ... end` ブロックで囲まれます。

**構文 (EBNF):**

```ebnf
registration_block ::= "registration" { registration_item } "end" ";" ;
registration_item  ::= existential_registration
                     | conditional_registration
                     | functorial_registration
                     | reduction_registration ;
```

**例：**

```mizar
registration
  cluster EmptyFiniteExists: empty finite set;
  existence proof ... end;

  cluster EmptyImpliesFinite: empty -> finite for set;
  coherence proof ... end;
end;
```

1 つの `registration` ブロックに複数の登録項目を含めることができます。各項目には独自の正当性条件があります。

**必須ラベル**

すべての登録アイテムには **ラベル** (登録キーワードの直後にコロンで区切って配置される識別子) が必要です。

```
cluster Label: ...
reduce  Label: ...
```

ラベルは次の理由で必須です (オプションではありません)。

- **トレーサビリティ**: 証明では、`by` の理由で名前による特定の登録を引用できます。
- **エラー メッセージ**: 検証器は、位置ではなくラベルによってエラーを報告します。
- **module インターフェイス**: ラベルはmoduleのパブリック インターフェイスの一部であり、選択的にインポートまたはオーバーライドできます。

登録上のラベルは、定理ラベルと同じ命名規則に従います。これらは、それを囲んでいるmodule内で一意である必要があります。

> [！注記]
> `registration` ブロックはトップレベルの宣言です。 `definition` ブロックと同様に、`let` を使用してパラメーターを導入する場合があります。パラメータ化された登録については、§17.3.3 を参照してください。

## 17.3 存在するクラスタ

**存在クラスタ**は、特定の属性で修飾されたタイプが**存在**しているという事実を登録します。つまり、指定されたすべての属性を満たす、その型のオブジェクトが少なくとも 1 つ存在します。

**構文 (EBNF):**

```ebnf
existential_registration ::= "cluster" label ":" adjective_list type_expression ";"
                              "existence" justification ";" ;
adjective_list           ::= adjective { adjective } ;
adjective                ::= [ "non" ] [ param_prefix ] attribute_name ;
```

### 17.3.1 基本的な存在登録

**例：**

```mizar
registration
  :: Register that there exists an empty finite set
  cluster EmptyFiniteExists: empty finite set;
  existence
  proof
    take {};
    thus {} is empty finite set;
  end;
end;
```

この登録後、型 `empty finite set` は、それ以上の存在証明なしで変数宣言で使用できるようになります。

```mizar
let S be empty finite set;  :: OK — existence registered
```

### 17.3.2 構造物の存在登録

**例：**

```mizar
registration
  :: Register that there exists a non empty Group
  cluster NonEmptyGroupExists: non empty Group;
  existence
  proof
    set G = SymmetricGroup(3);
    take G;
    thus G is non empty Group by GroupTheory.Sym3_is_Group;
  end;
end;
```

### 17.3.3 パラメータ化された存在登録

登録項目は、登録ブロック内の `let` によって導入されたパラメーターに依存する場合があります。

**例：**

```mizar
registration
  let X be non empty set;
  cluster NonEmptyFiniteSubsetExists: non empty finite Subset of X;
  existence
  proof
    set A = { the Element of X };
    take A;
    thus A is non empty finite Subset of X;
  end;
end;
```

> [！注記]
> 式 `the Element of X` は、空ではない型から任意の要素を選択する **明確な記述演算子** (選択演算子) です。 FOLにおけるヒルベルトのε演算子に相当します。構文と意味論は、[第13章 (項式)](./13.term_expression.md) で形式的に定義されています。

### 17.3.4 Inhabitation evidence と existential registration

値を導入または仮定する位置で使われるすべての型式は、その item order 上の
位置で inhabitation evidence を持たなければなりません。existential
registration は、属性を持つ正規化済み型式に必要な evidence です。無属性の
基底は下の表を使います。必要な evidence が欠落している場合、検証器は
hard error を生成します。

|状況 |例 |要件 |
|---|---|---|
|属性付き型の変数の宣言 | `let x be prime Nat;` |完全な正規化済み属性付き型に対する可視 active existential registration |
|RHS が属性を持つ mode の定義 | `mode Prime is prime Integer;` |完全な正規化済み属性付き RHS に対する可視 active existential registration (§7.8) |
|関手戻り型での属性付き型の使用 | `func F(x) -> non zero Real` |完全な正規化済み属性付き戻り型に対する可視 active existential registration |
|属性付き型式のテンプレート型実引数としての使用 | `Inhab[prime Nat]` |実引数に対する可視 active existential registration([第18章 §18.10.2](./18.templates.md#18102-型パラメータのエンコーディング)参照) |

無属性の基底は existential registration を必要としません。代わりに次の
built-in/base-shape inhabitation table を満たさなければなりません。

|型式の形 | inhabitation evidence |
|---|---|
| `object` | 組み込みの inhabited universe。 |
| `set` | 空集合を witness とする組み込み inhabited type。 |
| 追加属性のない accepted mode application `M(args)` | 同じ正規化済み argument tuple における、accepted mode declaration の §7.8 inhabitation obligation。 |
| 追加属性のない structure radix/application `S(args)` | constructor-witness rule: structure parameter を代入し、inheritance/default を考慮した後、必要な各 `field` 引数型が inhabitation evidence を持つ。先行する dependent field witness を使ってよい。`property` member は constructor 引数ではないため除外する(§5.5.1)。field を持たない structure は自明に inhabited。 |
| template 本体内の bare schema type parameter `T` | schema context の `∃x. is_T(x)` 仮定。この仮定は、template type actual がこの表で gate されるためにのみ健全である(§18.10.2)。 |
| template type actual | 実引数はこの同じ表で検査する。属性付き実引数は上の existential-registration 行を使う。schema 内部の bare type parameter は §18.10.2 に従う。 |

> [！重要]
> すべての属性付き型は、使用前に対応する existential registration が必要です。
> bare built-in、accepted mode application、bare structure radix は上の表に
> 依拠できますが、bare-structure 行は `non empty S` や他の attribute chain を
> 許可するものではありません。このポリシーにより、空の型がシステムに導入され
> なくなり、型システムの健全性が維持されます。mode 宣言については
> [第 7 章 (モード)、§7.8](./07.modes.md#78-正確性の条件)を参照してください。

## 17.4 条件付きクラスター

**条件付きクラスタ**は属性の暗示を登録します。特定の型のオブジェクトが特定の属性 (*前件*) を持っている場合、そのオブジェクトは自動的に追加の属性 (*後件*) を持ちます。

**構文 (EBNF):**

```ebnf
conditional_registration ::= "cluster" label ":" antecedent_adjectives "->"
                              consequent_adjectives "for" type_expression ";"
                              "coherence" justification ";" ;
antecedent_adjectives    ::= adjective_list ;
consequent_adjectives    ::= adjective_list ;
```

### 17.4.1 基本的な条件付き登録

**例：**

```mizar
registration
  :: Every empty set is finite
  cluster EmptyImpliesFinite: empty -> finite for set;
  coherence
  proof
    let S be set;
    assume S is empty;
    thus S is finite by mml.set.Empty_is_finite;
  end;
end;
```

この登録後、検証器は、セットが `empty` であることが分かると、自動的に `finite` を推論します。ラベル `EmptyImpliesFinite` は証明の中で明示的に引用することもできます。

```mizar
let S be empty set;
:: The verifier now knows: S is empty AND S is finite (by cluster)
S is finite;  :: automatically true, no proof needed
:: or: S is finite by EmptyImpliesFinite;  :: explicit citation
```

### 17.4.2 複数属性の条件付きクラスタ

前件と後件の両方に複数の属性が含まれる場合があります。

**例：**

```mizar
registration
  :: A finite non empty lattice is complete
  cluster FiniteNonEmptyLatticeComplete: finite non empty -> complete for Lattice;
  coherence
  proof
    let L be Lattice;
    assume L is finite non empty;
    thus L is complete by mml.order.Finite_Lattice_complete;
  end;
end;
```

### 17.4.3 連鎖した条件付きクラスタ

複数の条件付き登録により、検証器が推移的に解決する含意チェーンが形成されることがあります。

```mizar
registration
  cluster EmptyImpliesFinite: empty -> finite for set;
  coherence proof ... end;

  cluster FiniteImpliesCountable: finite -> countable for set;
  coherence proof ... end;
end;

:: Now the verifier can derive:
:: empty set -> finite set -> countable set
let S be empty set;
S is countable;   :: automatically true (by chaining EmptyImpliesFinite, FiniteImpliesCountable)
```

### 17.4.4 パラメータ化された条件付きクラスタ

**例：**

```mizar
registration
  let G be Group;
  cluster CyclicIsAbelian: cyclic -> abelian for Subgroup of G;
  coherence
  proof
    let H be Subgroup of G;
    assume H is cyclic;
    thus H is abelian by mml.group.Cyclic_is_abelian;
  end;
end;
```

### 17.4.5 クラスタ内の否定された属性

`non` 修飾子は、先行位置と後続位置の両方に現れる場合があります。

**例：**

```mizar
registration
  :: A positive Real is non zero
  cluster PositiveIsNonZero: positive -> non zero for Real;
  coherence proof ... end;
end;
```

## 17.5 関数クラスター

**関手クラスター** は、関手アプリケーションの結果が特定の属性を持つことを登録します。これにより、検証器は計算値に属性を自動的に割り当てることができます。

**構文 (EBNF):**

```ebnf
functorial_registration ::= "cluster" label ":" functor_term "->"
                             consequent_adjectives "for" type_expression ";"
                             "coherence" justification ";" ;
functor_term            ::= functor_application ;
```

末尾の `for` 型式は単なる説明ではなく、適用可能性 guard である。functorial
registration は、use site で関手適用結果の既知の正規化型が、parameter と
attribute を含む完全な `for` 型式そのもの、またはその subtype である場合に
だけ後件属性を追加できる。結果が無関係な型としてしか分かっていない場合、
registration 自体は妥当な registration のままだが発火しない。その後の use
site が後件属性を必要とするなら、別の evidence を得なければならない。

同じ guard は registration の `coherence` 義務にも現れる。
`cluster F(args) -> B1 ... Bm for T` の義務は、通常の parameter guard と
`F(args)` が `T` として既知であることを前提に後件を証明する条件付き義務で
あり、それ自体が `F(args)` の型が `T` であることを証明するものではない。

### 17.5.1 基本的な関数登録

**例：**

```mizar
registration
  let X, Y be non empty set;
  cluster CartesianNonEmpty: [: X, Y :] -> non empty for set;
  coherence
  proof
    let X, Y be non empty set;
    thus [: X, Y :] is non empty by mml.set.Cartesian_non_empty;
  end;
end;
```

この登録後:

```mizar
let A, B be non empty set;
set P = [: A, B :];
:: The verifier automatically knows: P is non empty
:: or: P is non empty by CartesianNonEmpty;  :: explicit citation
```

### 17.5.2 複数の属性を使用した関数登録

**例：**

```mizar
registration
  let n be Nat;
  cluster FactorialPositive: n ! -> positive non zero for Nat;
  coherence
  proof
    let n be Nat;
    thus n ! is positive non zero by mml.number.Factorial_positive;
  end;
end;
```

同じfunctor適用に対して、複数のfunctorial clusterが同時に適用されることがあります。これらは、競合する overload 定義とは異なり、互いに競合しません。引数 guard と結果 guard の両方が成り立つ各 cluster は、すでに判明している functor 結果に証明済みの属性を追加するだけであり、検証器は互換な属性をすべてまとめて集めます。`for` result guard が一致しない登録は適用されず、別の結果型を強制しません。競合が報告されるのは、同時に発火した登録が互いに矛盾する属性を導く場合だけです。

### 17.5.3 関数登録の例

**例：**

```mizar
definition
  let G be Group;
  func CenterDef: center(G) -> Subgroup of G means
    for x being Element of G holds
      x in it iff for y being Element of G holds x * y = y * x;
  existence proof ... end;
  uniqueness proof ... end;
end;

registration
  let G be Group;
  cluster CenterIsAbelian: center(G) -> abelian for Subgroup of G;
  coherence
  proof
    let G be Group;
    thus center(G) is abelian by mml.group.Center_is_abelian;
  end;
end;
```

### 17.5.4 入れ子になった関数登録

機能登録は他の登録と組み合わされます。

```mizar
registration
  let A be non empty set;
  cluster PowerSetNonEmpty: bool A -> non empty for set;
  coherence proof ... end;
end;

:: Now with both registrations:
let X be non empty set;
set P = bool [: X, X :];
:: [: X, X :] is non empty (by CartesianNonEmpty)
:: bool [: X, X :] is non empty (by PowerSetNonEmpty, since [: X, X :] is non empty set)
```

## 17.6 削減登録

**削減登録**は、特定の用語を別の用語に簡略化できることを宣言します。これは、型のチェックと証明の検証中に適用される計算の簡略化の一種です。

> [！注記]
> リダクション登録は普遍的等価公理 (§17.9.4 を参照) を生成し、等価公理 (§17.9.6 を参照) として ATP backendにもエクスポートされます。最新の ATP (特に Vampire/E のような重ね合わせベースの証明器) は、用語の順序に基づいてこれらの等式を書き換えルールに自動的に配置し、証明検索中に Knuth-Bendix 補完を効果的に実行します。

**構文 (EBNF):**

```ebnf
reduction_registration ::= "reduce" label ":" term_expression "to" term_expression ";"
                           "reducibility" justification ";" ;
```

### 17.6.1 基本的な削減

**例：**

```mizar
registration
  let n be Nat;
  reduce NatAddZero: n + 0 to n;
  reducibility
  proof
    let n be Nat;
    thus n + 0 = n by mml.number.natural.Nat_add_zero;
  end;
end;
```

この登録後、検証器は型チェック中に `n + 0` を `n` に自動的に単純化します。ラベルは証明でも引用できます。

```mizar
let m be Nat;
set k = m + 0;
:: The verifier recognizes k = m (by NatAddZero)
:: or: k = m by NatAddZero;  :: explicit citation
```

### 17.6.2 ID操作の削減

**例：**

```mizar
registration
  let X be set;
  reduce UnionEmpty: X \/ {} to X;
  reducibility
  proof
    let X be set;
    thus X \/ {} = X by mml.set.Union_empty;
  end;

  reduce IntersectionSelf: X /\ X to X;
  reducibility
  proof
    let X be set;
    thus X /\ X = X by mml.set.Intersection_self;
  end;
end;
```

### 17.6.3 代数演算のリダクション

**例：**

```mizar
registration
  let G be Group;
  let x be Element of G;
  reduce GroupMultUnit: x * (1.G) to x;
  reducibility
  proof
    let G be Group;
    let x be Element of G;
    thus x * (1.G) = x by mml.group.Group_mult_unit;
  end;

  reduce GroupUnitMult: (1.G) * x to x;
  reducibility
  proof
    let G be Group;
    let x be Element of G;
    thus (1.G) * x = x by mml.group.Group_unit_mult;
  end;
end;
```

これらの削減により、検証器は型チェック中に単位要素による乗算を自動的に簡素化できます。

### 17.6.4 停止性・決定性・トレーサビリティ

リダクションは、型チェック中に自動適用されるグローバルな項書き換え系を形成します。この系を予測可能かつ import しても安全に保つため、本言語は「**どのような規則を書けるか**」と「**どの順序で適用するか**」を、ライブラリ作者の自制心に委ねるのではなく言語側で制約します。

**停止性（登録時に強制）。** すべてのリダクションは、システム全体で固定された**簡約順序** `≻` を厳密に減少させなければなりません。すなわち右辺は左辺より小さくなければなりません（`LHS ≻ RHS`）。言語の既定順序は、置換に対して安定な構造的サイズ順序です。登録は、次の条件をすべて満たす場合にのみ受理されます。

1. `RHS` のすべての自由パターン変数が `LHS` にも自由に出現する。
2. 各自由パターン変数 `v` について、`RHS` における `v` の自由出現回数は `LHS` における `v` の自由出現回数以下である。
3. `size(LHS) > size(RHS)` である。ここで `size` は、束縛子を alpha 正規化した後の解決済みの項 AST のノード数であり、各変数出現を 1 ノードとして数える。

既定の順序では、同サイズの向き付けは受理しません。決定的な構造比較は診断や安定ソートには使えますが、停止性を正当化するためには使いません。この順序で `RHS` が `LHS` より厳密に小さくない登録は、**登録時に診断付きで拒否**されます。

すべての書き換えステップが整礎な尺度を厳密に減少させるため、書き換え系は**構成上グローバルに停止**します。ステップ制限は存在せず、サイクル——直接的（`reduce A to B` と `reduce B to A`）であれ間接的（A → B → C → A）であれ——は登録できません。この保証は import 閉包全体にわたって成立し、import された規則が下流モジュールで非停止を引き起こすことはありません。

**決定的正規化。** 正規化は、項、スコープ内の activated reduction
rule 集合、その rewrite site で利用可能な解消済み side-condition 集合の
決定的関数です。解消済み side-condition 集合は、周囲の registration
parameter が導入する `such` guard に対する、安定した local fact、trace、
citation evidence を含みます。この evidence 集合が変われば適用可能な rule は
変わり得ますが、固定された 3 つ組の入力について、結果は登録順や import 順に
**依存しません**。固定戦略は次のとおりです。

|観点 |規則 |
|---|---|
| **方向** |左から右のみ。`LHS` を `RHS` へ書き換える。|
| **redex 走査** |最左最内：部分項を完全に正規化してから親項を対象とする。|
| **マッチング** |パターンは alpha 同値と変数束縛を法として一致する。パターン変数はインスタンス化され、インスタンス化された項は周囲の `let` 宣言が導入する型・属性ガードを満たさなければならない。周囲の `let` に付いた `such` 条件は、解消済み side-condition 集合から読む適用可能性の副条件である。すなわち、その規則が適用され得る前に、記録済み local fact、trace、または明示引用された evidence としてすでに利用可能でなければならないが、specificity の入力ではない。|
| **規則選択** |同一 redex に複数規則が一致した場合、まず適用不能な規則を除外する。残った一致について、最初に `LHS` pattern を pattern subsumption で比較し、厳密な pattern-subsumption の勝者があれば guard 比較の前に選択する。pattern 比較で一意な規則が選べない場合、対応する pattern-variable の type / attribute guard を §19.2.3 の specificity preorder で位置ごとに比較する。一方の規則が guard についてより specific であるのは、対応するすべての位置で少なくとも同じだけ specific で、少なくとも 1 つの位置で厳密により specific な場合だけである。対応が欠ける場合、位置ごとに勝者が逆になる場合、その他の混在ケースは比較不能とする。`such` side condition はこの比較から除外する。残った相互比較不能または同等に特殊な一致は、**FQN**（モジュール FQN + ラベル）の辞書順で最小のものを選ぶ。これは import 順に依存しない。|

pattern-first の選択は、type guard 比較の段階についてのみ overload
resolution（§19）と整合します。pattern subsumption でまだ決まっていない
一致のうち、位置ごとの preorder が一意な勝者を持つ場合は、より specific な
guard 付き変数が優先されます。ただし overload resolution とは異なり、
reduction normalization は全域的で決定的な関数でなければならないため、
比較不能な一致を曖昧性エラーにはしません。FQN による tie-break が、そのような
ケースをマシンやビルド構成をまたいで決定的かつ再現可能にします。

> [!NOTE]
> 戦略正規形はスコープ内規則の*集合*と rewrite site で利用可能な解消済み
> side-condition 集合に依存します。そのため、（例えば新たな import によって）
> 規則を追加した場合や、`such` guard 用の local fact、trace、citation
> evidence を追加・削除した場合、自動正規化の結果は変わり得ます。これらの
> 残る変化要因は下記のトレーサビリティによって監査可能であり、常に明示引用で
> 回復できます。

**回復。** すべてのリダクションは `reducibility` 証明に裏打ちされているため、リダクションが偽の命題を証明可能にすることは決してありません——最悪でも、決定的戦略がどの表層形を生成するかが変わるだけです。自動正規化が証明の想定しない形を生んだ場合、その証明は該当する等式を明示引用（`... by <label>;`）することで**局所的に**修復します。グローバルなオプトイン／オプトアウト機構は提供しません。明示引用が逃げ道です。

**トレーサビリティ。** 自動適用された各リダクションは、その完全修飾名（モジュール FQN + ラベル）で検証トレース（`resolution-trace/`、§23.7.7）に記録されます。上流の変更が規則を改変・削除したとき、トレースは下流の証明がどのリダクションに依存していたかを正確に特定し、本来は暗黙だった破損を、名前の付いた診断可能な破損へと変えます。したがってラベルはすべてのリダクションで必須です（§17.2）。ラベルは規則選択に用いる FQN を構成し、このトレースを意味あるものにします。

> [!NOTE]
> 合流（confluence）は**要求しません**——決定的戦略が、それなしに一意な戦略正規形を保証します。臨界対（合流）リントと、`LHS` 先頭シンボルに対する orphan rule 制約は将来拡張として留保します。これらは canonicity を加えパッケージ間衝突を限定しますが、健全性・停止性・決定性には不要です。旧 Mizar の `identify` 構文（無方向の functor 同一視）は本仕様に意図的に**含めません**。項を簡約する等値は、上記の簡約順序に従わせるため、向き付きの `reduce` として書かねばなりません。

## 17.7 クラスタ解決ルール

クラスター解決は、ATP backendとは独立して、**cluster-db グラフの前方走査** (§23.7.4) によって実行されます。クラスター構文では結合した前件と後件のみが許可されるため (§17.10)、すべての登録は `A₁ ∧ … ∧ Aₙ → B₁ ∧ … ∧ Bₘ` 形式の Horn 節となり、そのグラフのエッジ (またはハイパーエッジ) に直接マッピングされます。外部ソルバーは必要ありません。

解決された属性セットは基礎事実として ATP backendに渡されるため、証明検索は実行時にクラスター公理を連鎖する必要はありません (§21.4.2)。導出ステップは `resolution-trace/` (§23.7.7) に記録され、命題チェーンとして最小kernelによって検証できます。

### 17.7.1 グラフ走査アルゴリズム

先行件と後件は両方とも属性の結合であるため (§17.10)、クラスター ルールの一般的な形式は `A₁ ∧ … ∧ Aₙ → B₁ ∧ … ∧ Bₘ` です。 `A₁ ∧ … ∧ Aₙ → B₁ ∧ B₂` は `A₁ ∧ … ∧ Aₙ → B₁` および `A₁ ∧ … ∧ Aₙ → B₂` のペアと意味的に同等であるため、ロード時に、このような各ルールは **m 個の単一結果ルール** (結果属性ごとに 1 つ) に正規化されます。正規化後、すべてのルールは厳密に 1 つの後件属性を持ち、cluster-db グラフには先行件の数に応じて 2 種類のエッジが含まれます。

|先行詞の数 |グラフ表現 |アルゴリズム |
|---|---|---|
|シングル (`A → B`) |有向刃 `A → B` | BFS/DFS の到達可能性 |
|複数 (`A₁ ∧ A₂ ∧ … → B`) | Hyperedge: すべてのsource頂点がファクト セット内にある場合にのみ起動します。固定点の反復 |

**手順：**

1. すべてのルールを単一結果形式に正規化します (cluster-db のロード時に 1 回限り)。
2. 型の基数型と現在既知の属性からファクト セット `F` を初期化します。
3. 先行頂点がすべて `F` 内にある各エッジについて、その (単一の) 後続頂点を `F` に追加します。
4. `F` (固定点) が変化しなくなるまで、手順 3 を繰り返します。
5. ターゲット属性が `F` にある場合にのみ、解決は成功します。

単一先行チェーン (一般的なケース) では、単純なグラフの到達可能性が低下し、完全な固定点ループなしで BFS によって短絡される可能性があります。

この fixpoint の停止性は、言語契約の load-bearing な部分です。停止性は制限された
cluster adjective 文法(§17.10)から従います。cluster rule は、active registration の有限集合、
registration-local な parameter prefix、そしてそれらの registration に既に現れる numeric
prefix 上の既存 attribute key だけを追加します。closure algorithm は新しい term や任意の
attribute argument を合成しません。したがって existential、conditional、functorial cluster が
使う `adjective` 文法は、第 3 章の一般 `attribute_ref` 文法の真部分集合であり、
`attribute_name(argument_list)` 形式を許しません。将来 cluster adjective に任意の term 引数を
持たせる拡張を行う場合は、言語の一部にする前に新しい停止性論証が必要です。実装上の
saturation bound は resource limit や bug に対する防御的診断であり、成功した truncated
closure semantics を定義するものではありません。

**例：**

```mizar
:: Given these registrations:
registration
  cluster EmptyImpliesFinite: empty -> finite for set;
  coherence proof ... end;

  cluster FiniteImpliesCountable: finite -> countable for set;
  coherence proof ... end;
end;

:: When the verifier encounters:
let S be empty set;
S is countable;  :: resolved by graph traversal

:: Traversal:
::   F = { set, empty }
::   Edge EmptyImpliesFinite fires   → F = { set, empty, finite }
::   Edge FiniteImpliesCountable fires → F = { set, empty, finite, countable }
::   "countable" ∈ F — resolution succeeds.
```

クラスター解決のために ATP 呼び出しは発行されません。

### 17.7.2 解像度におけるサブ型の互換性

型 `T` に登録された条件付きクラスターは、`T` のサブタイプにも適用されます。例えば：

```mizar
registration
  cluster EmptyImpliesFinite: empty -> finite for set;
  coherence proof ... end;
end;

:: Since Subset of X is a subtype of set:
let X be set;
let S be empty Subset of X;
:: The verifier derives: S is finite (EmptyImpliesFinite applies to subtypes of set)
```

### 17.7.3 競合の解決

すべてのクラスター登録には正当性証明 (`existence` または `coherence`) が必要であるため、矛盾するクラスター (たとえば、同じタイプに対して 1 つは `A` を派生し、もう 1 つは `non A` を派生する) が一貫した論理システム内で発生してはなりません。

それでも cluster closure 中に矛盾が検出された場合、それは accepted registration database に
根本的な不整合があることを示します。registration proof が誤って受理されたか、基礎となる
公理が矛盾しています。検証器はこれを fatal な `cluster` 診断として報告します。この診断は、
derived fact を追加すると同じ subject 上に `A` と `non A` の両方が置かれる場合に発生します。
1 つの registration item が verified closure/export result に寄与する前に、矛盾する consequent
へ正規化される static case も含みます。後段で ATP から見える不整合は別個の proof-system
failure であり、cluster-resolution conflict detection の trigger ではありません。

> [！注意]
> クラスター解像度の矛盾 (たとえば、同じ用語から導出された `finite` と `non finite` の両方) は、論理システムが健全でないことを示します。これは回復可能なエラーではありません。ユーザーは、矛盾した登録またはその前提を特定して修正する必要があります。

## 17.8 他の機能との相互作用

### 17.8.1 クラスタとモード

RHS 型式が属性を持つ mode 定義は、整形式であるために existential cluster に
依存します。無属性の基底は §17.3.4 の inhabitation table を使います。
[第 7 章 (モード)、§7.8](./07.modes.md#78-正確性の条件)を参照してください。

```mizar
:: The mode "Prime" requires an existential cluster:
registration
  cluster PrimeIntegerExists: prime Integer;
  existence
  proof
    take 2;
    thus 2 is prime by mml.number.prime.Two_is_prime;
  end;
end;

definition
  mode PrimeDef: Prime is prime Integer;
end;
```

### 17.8.2 クラスタと型推論

型推論エンジン ([第8章 (型推論)](./08.type_inference.md) を参照) は次のようにクラスターを使用します。

|推論フェーズ |クラスターの役割 |
|---|---|
| **型チェック** |条件付きクラスターは、型指定された値の既知の属性を拡張します。
| **サブ型のテスト** |クラスターは、`A₁ ... Aₙ T` が `B₁ ... Bₘ T` のサブタイプであるかどうかを判断するために使用されます。
| **関手の結果の型付け** |関数クラスターは計算値の属性を提供します。
| **用語の簡略化** |リダクションでは、一致する前に用語を正規化します。

### 17.8.3 クラスターと証明義務

クラスター登録では、[第 16 章 (定理と証明)、§16.6.3](./16.theorems_and_proofs.md#1663-クラスター登録の義務) で説明されている証明義務が発生します。各登録のラベルは、生成された理論内の対応する FOL 公理の名前になり、証明で `by Label` を介して登録を明示的に引用できるようになります。次の表に義務をまとめます。

|登録タイプ |正確性の条件 | FOLの義務 |
|---|---|---|
|実存的 | `existence` | `∃x. (is_T(x) ∧ is_A₁(x) ∧ ... ∧ is_Aₙ(x))` |
|条件付き | `coherence` | `∀x. (is_T(x) ∧ is_A₁(x) ∧ ... ∧ is_Aₖ(x) → is_B₁(x) ∧ ... ∧ is_Bₘ(x))` |
|関数 | `coherence` | `∀args. (parameter_guards(args) ∧ is_T(F(args)) → is_B₁(F(args)) ∧ ... ∧ is_Bₘ(F(args)))` |
|削減 | `reducibility` | `∀args. LHS(args) = RHS(args)` |

### 17.8.4 クラスターと個別のコンパイル

moduleで定義されたクラスター登録は、moduleのパブリック インターフェイスの一部としてエクスポートされます (`private` とマークされていない限り)。別のmoduleが定義moduleをインポートすると、登録されたクラスターがインポーターの型推論エンジンで使用できるようになります。

```mizar
:: Module: std.set.finite
registration
  cluster EmptyImpliesFinite: empty -> finite for set;
  coherence proof ... end;
end;

:: Module: std.topology.compact
import std.set.finite;
:: The cluster EmptyImpliesFinite is now available here
```

> [！注記]
> 個別のコンパイルとmodule インターフェイスの詳細については、「[第12章 (module と namespace), §12.6](./12.modules_and_namespaces.md#126-論理エンコーディングと個別コンパイル)」を参照してください。

## 17.9 論理エンコーディングの詳細

このセクションでは、クラスターの登録が一階論理 (FOL) にマップされる方法について説明します。型のエンコード規則については、「[第3章 (型システム), §3.7](./03.type_system.md#37-論理エンコーディングの詳細)」を参照してください。属性のエンコードについては、[第6章 (属性), §6.8](./06.attributes.md#68-論理エンコーディングの詳細) を参照してください。

### 17.9.1 存在するクラスタのエンコーディング

存在クラスタは、属性付き型の非空性を主張する存在公理を生成します。

|Mizar source | FOL エンコーディング |
|---|---|
| `cluster empty finite set;` | `∃x. (is_set(x) ∧ is_empty(x) ∧ is_finite(x))` |
| `cluster non empty Group;` | `∃x. (is_Group(x) ∧ ¬is_empty(x))` |
| `cluster prime Integer;` | `∃x. (is_Integer(x) ∧ is_prime(x))` |

**パラメータ化された形式:**

```
registration
  let X be non empty set;
  cluster NonEmptyFiniteSubsetExists: non empty finite Subset of X;
  existence proof ... end;
end;

⟹  ∀X. (is_set(X) ∧ ¬is_empty(X) →
      ∃x. (is_Subset_of(x, X) ∧ ¬is_empty(x) ∧ is_finite(x)))
```

### 17.9.2 条件付きクラスタエンコーディング

条件付きクラスターは普遍的な含意の公理を生成します。

|Mizar source | FOL エンコーディング |
|---|---|
| `cluster empty -> finite for set;` | `∀x. (is_set(x) ∧ is_empty(x) → is_finite(x))` |
| `cluster finite non empty -> complete for Lattice;` | `∀x. (is_Lattice(x) ∧ is_finite(x) ∧ ¬is_empty(x) → is_complete(x))` |
| `cluster non zero -> non empty for Real;` | `∀x. (is_Real(x) ∧ ¬is_zero(x) → ¬is_empty(x))` |

**パラメータ化された形式:**

```
registration
  let G be Group;
  cluster CyclicIsAbelian: cyclic -> abelian for Subgroup of G;
  coherence proof ... end;
end;

⟹  ∀G. (is_Group(G) →
      ∀x. (is_Subgroup_of(x, G) ∧ is_cyclic(x) → is_abelian(x)))
```

**複数の属性の後件:**

```
cluster FiniteNonEmptyLatticeCompleteBounded: finite non empty -> complete bounded for Lattice;

⟹  ∀x. (is_Lattice(x) ∧ is_finite(x) ∧ ¬is_empty(x)
      → is_complete(x) ∧ is_bounded(x))
```

### 17.9.3 関数クラスタのエンコーディング

関数クラスターは、関数の結果の属性を主張する普遍公理を生成します。

|Mizar source | FOL エンコーディング |
|---|---|
| `cluster [: X, Y :] -> non empty for set;` | `∀X,Y. (¬is_empty(X) ∧ ¬is_empty(Y) ∧ is_set(cartesian_product(X, Y)) → ¬is_empty(cartesian_product(X, Y)))` |
| `cluster n ! -> positive for Nat;` | `∀n. (is_Nat(n) ∧ is_Nat(factorial(n)) → is_positive(factorial(n)))` |
| `cluster center(G) -> abelian for Subgroup of G;` | `∀G. (is_Group(G) ∧ is_Subgroup_of(center(G), G) → is_abelian(center(G)))` |

> [！注記]
> 完全な FOL エンコーディングには、(周囲の `let` 宣言からの) 関手の
> parameter 型 guard と、関手結果に対する `for` 型 guard が含まれる。
> `for` guard は適用可能性を制御する。checker が結果型を guarded type
> expression またはその subtype としてすでに知っている場合に限り、この
> cluster を使える。

### 17.9.4 縮小エンコーディング

還元により普遍的な等価公理が生成されます。

|Mizar source | FOL エンコーディング |
|---|---|
| `reduce n + 0 to n;` | `∀n. (is_Nat(n) → add(n, 0) = n)` |
| `reduce X \/ {} to X;` | `∀X. (is_set(X) → union(X, empty_set) = X)` |
| `reduce X /\ X to X;` | `∀X. (is_set(X) → intersection(X, X) = X)` |

> [!NOTE]
> 公理は対称ですが、検証器は §17.6.4 の決定的で停止する戦略のもと、向き付き書き換え（`LHS → RHS`）としてのみ適用します。この向き付けと簡約順序 `≻` こそが、型チェック中の一意な戦略正規形を保証します。ATP backendへエクスポートされるのは依然として対称な等式です（§17.9.6）。

### 17.9.5 複合エンコーディングの例

FOL エンコーディングを含む完全な登録ブロック:

```mizar
registration
  let R be Ring;

  cluster CommSubgroupIsAbelian: commutative -> abelian for Subgroup of (R qua AddLoopStr);
  coherence proof ... end;

  let I be Ideal of R;
  cluster QuotientIsRingLike: R / I -> Ring-like for DoubleLoopStr;
  coherence proof ... end;
end;
```

FOL エンコーディング:

```
:: Conditional cluster
∀R. (is_Ring(R) →
  ∀x. (is_Subgroup_of(x, R_as_AddLoopStr(R)) ∧ is_commutative(x)
    → is_abelian(x)))

:: Functorial cluster
∀R. (is_Ring(R) →
  ∀I. (is_Ideal_of(I, R) →
    is_DoubleLoopStr(quotient(R, I)) → is_Ring_like(quotient(R, I))))
```

### 17.9.6 ATP 変換戦略

クラスターを ATP backendにエクスポートする場合、次のエンコード戦略が適用されます。

|登録タイプ | TPTP エンコーディング | SMT-LIB エンコーディング |
|---|---|---|
|実存的 | `fof(ex_cluster_N, axiom, ?[X]: (is_T(X) & is_A(X)))` | `(assert (exists ((x T)) (is_A x)))` |
|条件付き | `fof(cond_cluster_N, axiom, ![X]: (is_T(X) & is_A(X) => is_B(X)))` | `(assert (forall ((x T)) (=> (is_A x) (is_B x))))` |
|関数 | `fof(func_cluster_N, axiom, ![Args]: (is_T(f(Args)) => is_A(f(Args))))` | `(assert (forall ((args T)) (=> (is_T (f args)) (is_A (f args)))))` |
|削減 | `fof(reduce_N, axiom, ![Args]: lhs(Args) = rhs(Args))` | `(assert (forall ((args T)) (= (lhs args) (rhs args))))` |

> [！注記]
> ATP 統合の詳細とプロパティのエンコードについては、「[第21章 (source code annotation と ATP integration)](./21.source_code_annotation_and_atp.md)」を参照してください。重ね合わせベースの証明器 (Vampire/E) は、通常、明示的な指示なしで、リダクションからの単位等価性を書き換えルールとして扱います。

## 17.10 完全な構文 (EBNF)

この章のすべてのクラスターと登録構文の概要。

```ebnf
(* === Registration Block === *)
registration_block ::= "registration" { registration_content } "end" ";" ;

registration_content ::= parameter_decl             (* let ... *)
                       | registration_item ;

parameter_decl       ::= "let" qualified_vars [ "such" conditions ] [ "by" references ] ";" ;

registration_item    ::= existential_registration
                       | conditional_registration
                       | functorial_registration
                       | reduction_registration ;

(* === Existential Cluster === *)
existential_registration ::= "cluster" label ":" adjective_list type_expression ";"
                              "existence" justification ";" ;

(* === Conditional Cluster === *)
conditional_registration ::= "cluster" label ":" antecedent_adjectives "->"
                              consequent_adjectives "for" type_expression ";"
                              "coherence" justification ";" ;

(* === Functorial Cluster === *)
functorial_registration  ::= "cluster" label ":" functor_term "->"
                              consequent_adjectives "for" type_expression ";"
                              "coherence" justification ";" ;

(* === Reduction Registration === *)
reduction_registration   ::= "reduce" label ":" term_expression "to" term_expression ";"
                              "reducibility" justification ";" ;

(* === Supporting Rules === *)
adjective_list           ::= adjective { adjective } ;
adjective                ::= [ "non" ] [ param_prefix ] attribute_name ;
antecedent_adjectives    ::= adjective_list ;
consequent_adjectives    ::= adjective_list ;
functor_term             ::= functor_application ;

(* Note: adjective は Chapter 3 attribute_ref の cluster-registration subset である。
   cluster closure の停止性はこの制限形式に依存するため、
   attribute_name(argument_list) は意図的に除外される。

   qualified_vars, conditions, and justification
   are defined in Chapter 15 (Statements).
   type_expression is defined in Chapter 3 (Type System).
   functor_application is defined in Chapter 13 (Term Expressions).
   term_expression is defined in Chapter 13 (Term Expressions).
   param_prefix is defined in Chapter 6 (Attributes). *)
```
