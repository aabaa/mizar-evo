## 17.1 概要

Mizar のソフトタイピング システム ([第3章 (型システム)](./03.type_system.md) を参照) では、型は **基数型** (モードまたは構造) と 0 個以上の **属性** (形容詞) で構成されます。クラスターは、属性の組み合わせに関する事実を登録することで、個々の属性定義と型推論エンジンの間のギャップを埋めます。

登録には次の 4 種類があります。

|登録種類 |目的 |正確性の条件 |
|---|---|---|
| **実存的** |指定された属性を持つ型が存在することを証明する | `existence` |
| **条件付き** |ある属性セットが別の属性セットを暗黙的に示すことを宣言します。 `coherence` |
| **関数** |関手の結果に特定の属性があることを宣言します。 `coherence` |
| **削減** |用語を別の用語に簡略化できることを宣言します | `reducibility` |

すべての登録は `registration ... end` ブロック内に表示され、登録が発効する前に履行する必要がある証明義務を生成します。

> [！重要]
> クラスターの登録は **グローバル公理**です。一度登録されると、現在のmoduleとそれをインポートするすべてのmodule全体に適用されます。登録が正しくないと、型システムの健全性が損なわれる可能性があります。このため、各登録には正当性の証明が必要です。

## 17.2 登録ブロック

すべてのクラスターおよびリダクション登録は、`registration ... end` ブロックで囲まれます。

**構文 (EBNF):**

```ebnf
registration_block ::= "registration" { registration_item } "end" ";" ;
registration_item  ::= existential_registration
                     | conditional_registration
                     | functorial_registration
                     | reduction_registration ;
```

**例：**

```mizar
registration
  cluster EmptyFiniteExists: empty finite set;
  existence proof ... end;

  cluster EmptyImpliesFinite: empty -> finite for set;
  coherence proof ... end;
end;
```

1 つの `registration` ブロックに複数の登録項目を含めることができます。各項目には独自の正当性条件があります。

**必須ラベル**

すべての登録アイテムには **ラベル** (登録キーワードの直後にコロンで区切って配置される識別子) が必要です。

```
cluster Label: ...
reduce  Label: ...
```

ラベルは次の理由で必須です (オプションではありません)。

- **トレーサビリティ**: 証明では、`by` の理由で名前による特定の登録を引用できます。
- **エラー メッセージ**: 検証器は、位置ではなくラベルによってエラーを報告します。
- **module インターフェイス**: ラベルはmoduleのパブリック インターフェイスの一部であり、選択的にインポートまたはオーバーライドできます。

登録上のラベルは、定理ラベルと同じ命名規則に従います。これらは、それを囲んでいるmodule内で一意である必要があります。

> [！注記]
> `registration` ブロックはトップレベルの宣言です。 `definition` ブロックと同様に、`let` を使用してパラメーターを導入する場合があります。パラメータ化された登録については、§17.3.3 を参照してください。

## 17.3 存在するクラスタ

**存在クラスタ**は、特定の属性で修飾されたタイプが**存在**しているという事実を登録します。つまり、指定されたすべての属性を満たす、その型のオブジェクトが少なくとも 1 つ存在します。

**構文 (EBNF):**

```ebnf
existential_registration ::= "cluster" label ":" adjective_list type_expression ";"
                              "existence" justification ";" ;
adjective_list           ::= adjective { adjective } ;
adjective                ::= [ "non" ] [ param_prefix ] attribute_name ;
```

### 17.3.1 基本的な存在登録

**例：**

```mizar
registration
  :: Register that there exists an empty finite set
  cluster EmptyFiniteExists: empty finite set;
  existence
  proof
    take {};
    thus {} is empty finite set;
  end;
end;
```

この登録後、型 `empty finite set` は、それ以上の存在証明なしで変数宣言で使用できるようになります。

```mizar
let S be empty finite set;  :: OK — existence registered
```

### 17.3.2 構造物の存在登録

**例：**

```mizar
registration
  :: Register that there exists a non empty Group
  cluster NonEmptyGroupExists: non empty Group;
  existence
  proof
    set G = SymmetricGroup(3);
    take G;
    thus G is non empty Group by GroupTheory.Sym3_is_Group;
  end;
end;
```

### 17.3.3 パラメータ化された存在登録

登録項目は、登録ブロック内の `let` によって導入されたパラメーターに依存する場合があります。

**例：**

```mizar
registration
  let X be non empty set;
  cluster NonEmptyFiniteSubsetExists: non empty finite Subset of X;
  existence
  proof
    set A = { the Element of X };
    take A;
    thus A is non empty finite Subset of X;
  end;
end;
```

> [！注記]
> 式 `the Element of X` は、空ではない型から任意の要素を選択する **明確な記述演算子** (選択演算子) です。 FOLにおけるヒルベルトのε演算子に相当します。構文と意味論は、[第13章 (項式)](./13.term_expression.md) で形式的に定義されています。

### 17.3.4 存在登録が必要な場合

次の状況では、既存の登録が**必須**となります。登録が欠落している場合、検証器はエラーを生成します。

|状況 |例 |要件 |
|---|---|---|
|属性付き型の変数の宣言 | `let x be prime Nat;` |必須 — 欠落している場合は検証器エラー |
|属性付きのベースを使用したモードの定義 | `mode Prime is prime Integer;` |必須 — 欠落している場合は検証器エラー |
|関手での属性付き型の使用`func F(x) -> non zero Real` |必須 — 欠落している場合は検証器エラー |

> [！重要]
> すべての属性付き型は、使用前に対応する存在登録が必要です。このポリシーにより、空の型がシステムに導入されなくなり、型システムの健全性が維持されます。詳細については、「[第 7 章 (モード)、§7.8](./07.modes.md#78-correctness-conditions)」を参照してください。

## 17.4 条件付きクラスター

**条件付きクラスタ**は属性の暗示を登録します。特定の型のオブジェクトが特定の属性 (*前件*) を持っている場合、そのオブジェクトは自動的に追加の属性 (*後件*) を持ちます。

**構文 (EBNF):**

```ebnf
conditional_registration ::= "cluster" label ":" antecedent_adjectives "->"
                              consequent_adjectives "for" type_expression ";"
                              "coherence" justification ";" ;
antecedent_adjectives    ::= adjective_list ;
consequent_adjectives    ::= adjective_list ;
```

### 17.4.1 基本的な条件付き登録

**例：**

```mizar
registration
  :: Every empty set is finite
  cluster EmptyImpliesFinite: empty -> finite for set;
  coherence
  proof
    let S be set;
    assume S is empty;
    thus S is finite by mml.set.Empty_is_finite;
  end;
end;
```

この登録後、検証器は、セットが `empty` であることが分かると、自動的に `finite` を推論します。ラベル `EmptyImpliesFinite` は証明の中で明示的に引用することもできます。

```mizar
let S be empty set;
:: The verifier now knows: S is empty AND S is finite (by cluster)
S is finite;  :: automatically true, no proof needed
:: or: S is finite by EmptyImpliesFinite;  :: explicit citation
```

### 17.4.2 複数属性の条件付きクラスタ

前件と後件の両方に複数の属性が含まれる場合があります。

**例：**

```mizar
registration
  :: A finite non empty lattice is complete
  cluster FiniteNonEmptyLatticeComplete: finite non empty -> complete for Lattice;
  coherence
  proof
    let L be Lattice;
    assume L is finite non empty;
    thus L is complete by mml.order.Finite_Lattice_complete;
  end;
end;
```

### 17.4.3 連鎖した条件付きクラスタ

複数の条件付き登録により、検証器が推移的に解決する含意チェーンが形成されることがあります。

```mizar
registration
  cluster EmptyImpliesFinite: empty -> finite for set;
  coherence proof ... end;

  cluster FiniteImpliesCountable: finite -> countable for set;
  coherence proof ... end;
end;

:: Now the verifier can derive:
:: empty set -> finite set -> countable set
let S be empty set;
S is countable;   :: automatically true (by chaining EmptyImpliesFinite, FiniteImpliesCountable)
```

### 17.4.4 パラメータ化された条件付きクラスタ

**例：**

```mizar
registration
  let G be Group;
  cluster CyclicIsAbelian: cyclic -> abelian for Subgroup of G;
  coherence
  proof
    let H be Subgroup of G;
    assume H is cyclic;
    thus H is abelian by mml.group.Cyclic_is_abelian;
  end;
end;
```

### 17.4.5 クラスタ内の否定された属性

`non` 修飾子は、先行位置と後続位置の両方に現れる場合があります。

**例：**

```mizar
registration
  :: A positive Real is non zero
  cluster PositiveIsNonZero: positive -> non zero for Real;
  coherence proof ... end;
end;
```

## 17.5 関数クラスター

**関手クラスター** は、関手アプリケーションの結果が特定の属性を持つことを登録します。これにより、検証器は計算値に属性を自動的に割り当てることができます。

**構文 (EBNF):**

```ebnf
functorial_registration ::= "cluster" label ":" functor_term "->"
                             consequent_adjectives "for" type_expression ";"
                             "coherence" justification ";" ;
functor_term            ::= functor_application ;
```

### 17.5.1 基本的な関数登録

**例：**

```mizar
registration
  let X, Y be non empty set;
  cluster CartesianNonEmpty: [: X, Y :] -> non empty for set;
  coherence
  proof
    let X, Y be non empty set;
    thus [: X, Y :] is non empty by mml.set.Cartesian_non_empty;
  end;
end;
```

この登録後:

```mizar
let A, B be non empty set;
set P = [: A, B :];
:: The verifier automatically knows: P is non empty
:: or: P is non empty by CartesianNonEmpty;  :: explicit citation
```

### 17.5.2 複数の属性を使用した関数登録

**例：**

```mizar
registration
  let n be Nat;
  cluster FactorialPositive: n ! -> positive non zero for Nat;
  coherence
  proof
    let n be Nat;
    thus n ! is positive non zero by mml.number.Factorial_positive;
  end;
end;
```

### 17.5.3 関数登録の例

**例：**

```mizar
definition
  let G be Group;
  func CenterDef: center(G) -> Subgroup of G means
    for x being Element of G holds
      x in it iff for y being Element of G holds x * y = y * x;
  existence proof ... end;
  uniqueness proof ... end;
end;

registration
  let G be Group;
  cluster CenterIsAbelian: center(G) -> abelian for Subgroup of G;
  coherence
  proof
    let G be Group;
    thus center(G) is abelian by mml.group.Center_is_abelian;
  end;
end;
```

### 17.5.4 入れ子になった関数登録

機能登録は他の登録と組み合わされます。

```mizar
registration
  let A be non empty set;
  cluster PowerSetNonEmpty: bool A -> non empty for set;
  coherence proof ... end;
end;

:: Now with both registrations:
let X be non empty set;
set P = bool [: X, X :];
:: [: X, X :] is non empty (by CartesianNonEmpty)
:: bool [: X, X :] is non empty (by PowerSetNonEmpty, since [: X, X :] is non empty set)
```

## 17.6 削減登録

**削減登録**は、特定の用語を別の用語に簡略化できることを宣言します。これは、型のチェックと証明の検証中に適用される計算の簡略化の一種です。

> [！注記]
> リダクション登録は普遍的等価公理 (§17.9.4 を参照) を生成し、等価公理 (§17.9.6 を参照) として ATP backendにもエクスポートされます。最新の ATP (特に Vampire/E のような重ね合わせベースの証明器) は、用語の順序に基づいてこれらの等式を書き換えルールに自動的に配置し、証明検索中に Knuth-Bendix 補完を効果的に実行します。

**構文 (EBNF):**

```ebnf
reduction_registration ::= "reduce" label ":" term_expression "to" term_expression ";"
                           "reducibility" justification ";" ;
```

### 17.6.1 基本的な削減

**例：**

```mizar
registration
  let n be Nat;
  reduce NatAddZero: n + 0 to n;
  reducibility
  proof
    let n be Nat;
    thus n + 0 = n by mml.number.natural.Nat_add_zero;
  end;
end;
```

この登録後、検証器は型チェック中に `n + 0` を `n` に自動的に単純化します。ラベルは証明でも引用できます。

```mizar
let m be Nat;
set k = m + 0;
:: The verifier recognizes k = m (by NatAddZero)
:: or: k = m by NatAddZero;  :: explicit citation
```

### 17.6.2 ID操作の削減

**例：**

```mizar
registration
  let X be set;
  reduce UnionEmpty: X \/ {} to X;
  reducibility
  proof
    let X be set;
    thus X \/ {} = X by mml.set.Union_empty;
  end;

  reduce IntersectionSelf: X /\ X to X;
  reducibility
  proof
    let X be set;
    thus X /\ X = X by mml.set.Intersection_self;
  end;
end;
```

### 17.6.3 代数演算のリダクション

**例：**

```mizar
registration
  let G be Group;
  let x be Element of G;
  reduce GroupMultUnit: x * (1.G) to x;
  reducibility
  proof
    let G be Group;
    let x be Element of G;
    thus x * (1.G) = x by mml.group.Group_mult_unit;
  end;

  reduce GroupUnitMult: (1.G) * x to x;
  reducibility
  proof
    let G be Group;
    let x be Element of G;
    thus (1.G) * x = x by mml.group.Group_unit_mult;
  end;
end;
```

これらの削減により、検証器は型チェック中に単位要素による乗算を自動的に簡素化できます。

### 17.6.4 削減のルールと制限

|ルール |説明 |
|---|---|
| **方向** |リダクションは左から右にのみ適用されます。左の項は右の項に簡略化されます。
| **コンフルエンス** |検証器は合流を保証しません。ユーザーは、削減によって曖昧さが生じないことを確認する必要があります。
| **アプリケーション** |リダクションは、型チェックにおける用語の正規化中に積極的に適用されます。
| **範囲** |削減は一度登録されるとグローバルになります。その後のすべての検証に影響します。
| **マッチング** |リダクション パターンはモジュロ変数バインディングと一致します (パターン変数はインスタンス化されます)。
| **歩数制限** |検証器は、非終了を防ぐためにリダクション チェーンにステップ制限を課します。

> [！注意]
> 循環リダクション (`reduce A to B` や `reduce B to A` など) は非終了の原因となります。検証器は登録時に直接サイクルを検出しようとしますが、それらを拒否します。ただし、間接サイクル (A → B → C → A) は静的に検出できない場合があります。これらは検証時にステップ制限によって捕捉され、検証エラーが発生します。

## 17.7 クラスタ解決ルール

クラスター解決は、ATP backendとは独立して、**cluster-db グラフの前方走査** (§23.6.2.1) によって実行されます。クラスター構文では結合した前件と後件のみが許可されるため (§17.10)、すべての登録は `A₁ ∧ … ∧ Aₙ → B₁ ∧ … ∧ Bₘ` 形式の Horn 節となり、そのグラフのエッジ (またはハイパーエッジ) に直接マッピングされます。外部ソルバーは必要ありません。

解決された属性セットは基礎事実として ATP backendに渡されるため、証明検索は実行時にクラスター公理を連鎖する必要はありません (§21.4.2)。導出ステップは `resolution-trace/` (§23.6.2.4) に記録され、命題チェーンとして最小kernelによって検証できます。

### 17.7.1 グラフ走査アルゴリズム

先行件と後件は両方とも属性の結合であるため (§17.10)、クラスター ルールの一般的な形式は `A₁ ∧ … ∧ Aₙ → B₁ ∧ … ∧ Bₘ` です。 `A₁ ∧ … ∧ Aₙ → B₁ ∧ B₂` は `A₁ ∧ … ∧ Aₙ → B₁` および `A₁ ∧ … ∧ Aₙ → B₂` のペアと意味的に同等であるため、ロード時に、このような各ルールは **m 個の単一結果ルール** (結果属性ごとに 1 つ) に正規化されます。正規化後、すべてのルールは厳密に 1 つの後件属性を持ち、cluster-db グラフには先行件の数に応じて 2 種類のエッジが含まれます。

|先行詞の数 |グラフ表現 |アルゴリズム |
|---|---|---|
|シングル (`A → B`) |有向刃 `A → B` | BFS/DFS の到達可能性 |
|複数 (`A₁ ∧ A₂ ∧ … → B`) | Hyperedge: すべてのsource頂点がファクト セット内にある場合にのみ起動します。固定点の反復 |

**手順：**

1. すべてのルールを単一結果形式に正規化します (cluster-db のロード時に 1 回限り)。
2. 型の基数型と現在既知の属性からファクト セット `F` を初期化します。
3. 先行頂点がすべて `F` 内にある各エッジについて、その (単一の) 後続頂点を `F` に追加します。
4. `F` (固定点) が変化しなくなるまで、手順 3 を繰り返します。
5. ターゲット属性が `F` にある場合にのみ、解決は成功します。

単一先行チェーン (一般的なケース) では、単純なグラフの到達可能性が低下し、完全な固定点ループなしで BFS によって短絡される可能性があります。

**例：**

```mizar
:: Given these registrations:
registration
  cluster EmptyImpliesFinite: empty -> finite for set;
  coherence proof ... end;

  cluster FiniteImpliesCountable: finite -> countable for set;
  coherence proof ... end;
end;

:: When the verifier encounters:
let S be empty set;
S is countable;  :: resolved by graph traversal

:: Traversal:
::   F = { set, empty }
::   Edge EmptyImpliesFinite fires   → F = { set, empty, finite }
::   Edge FiniteImpliesCountable fires → F = { set, empty, finite, countable }
::   "countable" ∈ F — resolution succeeds.
```

クラスター解決のために ATP 呼び出しは発行されません。

### 17.7.2 解像度におけるサブ型の互換性

型 `T` に登録された条件付きクラスターは、`T` のサブタイプにも適用されます。例えば：

```mizar
registration
  cluster EmptyImpliesFinite: empty -> finite for set;
  coherence proof ... end;
end;

:: Since Subset of X is a subtype of set:
let X be set;
let S be empty Subset of X;
:: The verifier derives: S is finite (EmptyImpliesFinite applies to subtypes of set)
```

### 17.7.3 競合の解決

すべてのクラスター登録には正当性証明 (`existence` または `coherence`) が必要であるため、矛盾するクラスター (たとえば、同じタイプに対して 1 つは `A` を派生し、もう 1 つは `non A` を派生する) が一貫した論理システム内で発生してはなりません。

それにもかかわらず、ATP 解決中に矛盾が検出された場合、これは公理システムに根本的な矛盾があることを示します。証明が誤って受け入れられたか、基礎となる公理が矛盾しているかのいずれかです。検証器はこれを **致命的エラー** として報告します。

> [！注意]
> クラスター解像度の矛盾 (たとえば、同じ用語から導出された `finite` と `non finite` の両方) は、論理システムが健全でないことを示します。これは回復可能なエラーではありません。ユーザーは、矛盾した登録またはその前提を特定して修正する必要があります。

## 17.8 他の機能との相互作用

### 17.8.1 クラスタとモード

属性付きの基本型を使用したモード定義は、整形式であるために存在クラスターに依存します。 [第 7 章 (モード)、§7.8](./07.modes.md#78-correctness-conditions)を参照してください。

```mizar
:: The mode "Prime" requires an existential cluster:
registration
  cluster PrimeIntegerExists: prime Integer;
  existence
  proof
    take 2;
    thus 2 is prime by mml.number.prime.Two_is_prime;
  end;
end;

definition
  mode PrimeDef: Prime is prime Integer;
end;
```

### 17.8.2 クラスタと型推論

型推論エンジン ([第8章 (型推論)](./08.type_inference.md) を参照) は次のようにクラスターを使用します。

|推論フェーズ |クラスターの役割 |
|---|---|
| **型チェック** |条件付きクラスターは、型指定された値の既知の属性を拡張します。
| **サブ型のテスト** |クラスターは、`A₁ ... Aₙ T` が `B₁ ... Bₘ T` のサブタイプであるかどうかを判断するために使用されます。
| **関手の結果の型付け** |関数クラスターは計算値の属性を提供します。
| **用語の簡略化** |リダクションでは、一致する前に用語を正規化します。

### 17.8.3 クラスターと証明義務

クラスター登録では、[第 16 章 (定理と証明)、§16.6.3](./16.theorems_and_proofs.md#1663-cluster-registration-obligations) で説明されている証明義務が発生します。各登録のラベルは、生成された理論内の対応する FOL 公理の名前になり、証明で `by Label` を介して登録を明示的に引用できるようになります。次の表に義務をまとめます。

|登録タイプ |正確性の条件 | FOLの義務 |
|---|---|---|
|実存的 | `existence` | `∃x. (is_T(x) ∧ is_A₁(x) ∧ ... ∧ is_Aₙ(x))` |
|条件付き | `coherence` | `∀x. (is_T(x) ∧ is_A₁(x) ∧ ... ∧ is_Aₖ(x) → is_B₁(x) ∧ ... ∧ is_Bₘ(x))` |
|関数 | `coherence` | `∀args. is_B₁(F(args)) ∧ ... ∧ is_Bₘ(F(args))` |
|削減 | `reducibility` | `∀args. LHS(args) = RHS(args)` |

### 17.8.4 クラスターと個別のコンパイル

moduleで定義されたクラスター登録は、moduleのパブリック インターフェイスの一部としてエクスポートされます (`private` とマークされていない限り)。別のmoduleが定義moduleをインポートすると、登録されたクラスターがインポーターの型推論エンジンで使用できるようになります。

```mizar
:: Module: std.set.finite
registration
  cluster EmptyImpliesFinite: empty -> finite for set;
  coherence proof ... end;
end;

:: Module: std.topology.compact
import std.set.finite;
:: The cluster EmptyImpliesFinite is now available here
```

> [！注記]
> 個別のコンパイルとmodule インターフェイスの詳細については、「[第12章 (module と namespace), §12.6](./12.modules_and_namespaces.md#126-logic-encoding-and-separate-compilation)」を参照してください。

## 17.9 論理エンコーディングの詳細

このセクションでは、クラスターの登録が一階論理 (FOL) にマップされる方法について説明します。型のエンコード規則については、「[第3章 (型システム), §3.7](./03.type_system.md#37-logic-encoding-details)」を参照してください。属性のエンコードについては、[第6章 (属性), §6.8](./06.attributes.md#68-logic-encoding-details) を参照してください。

### 17.9.1 存在するクラスタのエンコーディング

存在クラスタは、属性付き型の非空性を主張する存在公理を生成します。

|Mizar source | FOL エンコーディング |
|---|---|
| `cluster empty finite set;` | `∃x. (is_set(x) ∧ is_empty(x) ∧ is_finite(x))` |
| `cluster non empty Group;` | `∃x. (is_Group(x) ∧ ¬is_empty(x))` |
| `cluster prime Integer;` | `∃x. (is_Integer(x) ∧ is_prime(x))` |

**パラメータ化された形式:**

```
registration
  let X be non empty set;
  cluster NonEmptyFiniteSubsetExists: non empty finite Subset of X;
  existence proof ... end;
end;

⟹  ∀X. (is_set(X) ∧ ¬is_empty(X) →
      ∃x. (is_Subset_of(x, X) ∧ ¬is_empty(x) ∧ is_finite(x)))
```

### 17.9.2 条件付きクラスタエンコーディング

条件付きクラスターは普遍的な含意の公理を生成します。

|Mizar source | FOL エンコーディング |
|---|---|
| `cluster empty -> finite for set;` | `∀x. (is_set(x) ∧ is_empty(x) → is_finite(x))` |
| `cluster finite non empty -> complete for Lattice;` | `∀x. (is_Lattice(x) ∧ is_finite(x) ∧ ¬is_empty(x) → is_complete(x))` |
| `cluster non zero -> non empty for Real;` | `∀x. (is_Real(x) ∧ ¬is_zero(x) → ¬is_empty(x))` |

**パラメータ化された形式:**

```
registration
  let G be Group;
  cluster CyclicIsAbelian: cyclic -> abelian for Subgroup of G;
  coherence proof ... end;
end;

⟹  ∀G. (is_Group(G) →
      ∀x. (is_Subgroup_of(x, G) ∧ is_cyclic(x) → is_abelian(x)))
```

**複数の属性の後件:**

```
cluster FiniteNonEmptyLatticeCompleteBounded: finite non empty -> complete bounded for Lattice;

⟹  ∀x. (is_Lattice(x) ∧ is_finite(x) ∧ ¬is_empty(x)
      → is_complete(x) ∧ is_bounded(x))
```

### 17.9.3 関数クラスタのエンコーディング

関数クラスターは、関数の結果の属性を主張する普遍公理を生成します。

|Mizar source | FOL エンコーディング |
|---|---|
| `cluster [: X, Y :] -> non empty for set;` | `∀X,Y. (¬is_empty(X) ∧ ¬is_empty(Y) → ¬is_empty(cartesian_product(X, Y)))` |
| `cluster n ! -> positive for Nat;` | `∀n. (is_Nat(n) → is_positive(factorial(n)))` |
| `cluster center(G) -> abelian for Subgroup of G;` | `∀G. (is_Group(G) → is_abelian(center(G)))` |

> [！注記]
> 完全な FOL エンコーディングには、(周囲の `let` 宣言からの) 関手のパラメーターの型ガードが含まれています。

### 17.9.4 縮小エンコーディング

還元により普遍的な等価公理が生成されます。

|Mizar source | FOL エンコーディング |
|---|---|
| `reduce n + 0 to n;` | `∀n. (is_Nat(n) → add(n, 0) = n)` |
| `reduce X \/ {} to X;` | `∀X. (is_set(X) → union(X, empty_set) = X)` |
| `reduce X /\ X to X;` | `∀X. (is_set(X) → intersection(X, X) = X)` |

### 17.9.5 複合エンコーディングの例

FOL エンコーディングを含む完全な登録ブロック:

```mizar
registration
  let R be Ring;

  cluster CommSubgroupIsAbelian: commutative -> abelian for Subgroup of (R qua AddLoopStr);
  coherence proof ... end;

  let I be Ideal of R;
  cluster QuotientIsRingLike: R / I -> Ring-like for DoubleLoopStr;
  coherence proof ... end;
end;
```

FOL エンコーディング:

```
:: Conditional cluster
∀R. (is_Ring(R) →
  ∀x. (is_Subgroup_of(x, R_as_AddLoopStr(R)) ∧ is_commutative(x)
    → is_abelian(x)))

:: Functorial cluster
∀R. (is_Ring(R) →
  ∀I. (is_Ideal_of(I, R) →
    is_Ring_like(quotient(R, I))))
```

### 17.9.6 ATP 変換戦略

クラスターを ATP backendにエクスポートする場合、次のエンコード戦略が適用されます。

|登録タイプ | TPTP エンコーディング | SMT-LIB エンコーディング |
|---|---|---|
|実存的 | `fof(ex_cluster_N, axiom, ?[X]: (is_T(X) & is_A(X)))` | `(assert (exists ((x T)) (is_A x)))` |
|条件付き | `fof(cond_cluster_N, axiom, ![X]: (is_T(X) & is_A(X) => is_B(X)))` | `(assert (forall ((x T)) (=> (is_A x) (is_B x))))` |
|関数 | `fof(func_cluster_N, axiom, ![Args]: is_A(f(Args)))` | `(assert (forall ((args T)) (is_A (f args))))` |
|削減 | `fof(reduce_N, axiom, ![Args]: lhs(Args) = rhs(Args))` | `(assert (forall ((args T)) (= (lhs args) (rhs args))))` |

> [！注記]
> ATP 統合の詳細とプロパティのエンコードについては、「[第21章 (source code annotation と ATP integration)](./21.source_code_annotation_and_atp.md)」を参照してください。重ね合わせベースの証明器 (Vampire/E) は、通常、明示的な指示なしで、リダクションからの単位等価性を書き換えルールとして扱います。

## 17.10 完全な構文 (EBNF)

この章のすべてのクラスターと登録構文の概要。

```ebnf
(* === Registration Block === *)
registration_block ::= "registration" { registration_content } "end" ";" ;

registration_content ::= parameter_decl             (* let ... *)
                       | registration_item ;

parameter_decl       ::= "let" variable_list [ qualification ] [ "such" conditions ] ";" ;

registration_item    ::= existential_registration
                       | conditional_registration
                       | functorial_registration
                       | reduction_registration ;

(* === Existential Cluster === *)
existential_registration ::= "cluster" label ":" adjective_list type_expression ";"
                              "existence" justification ";" ;

(* === Conditional Cluster === *)
conditional_registration ::= "cluster" label ":" antecedent_adjectives "->"
                              consequent_adjectives "for" type_expression ";"
                              "coherence" justification ";" ;

(* === Functorial Cluster === *)
functorial_registration  ::= "cluster" label ":" functor_term "->"
                              consequent_adjectives "for" type_expression ";"
                              "coherence" justification ";" ;

(* === Reduction Registration === *)
reduction_registration   ::= "reduce" label ":" term_expression "to" term_expression ";"
                              "reducibility" justification ";" ;

(* === Supporting Rules === *)
adjective_list           ::= adjective { adjective } ;
adjective                ::= [ "non" ] [ param_prefix ] attribute_name ;
antecedent_adjectives    ::= adjective_list ;
consequent_adjectives    ::= adjective_list ;
functor_term             ::= functor_application ;

(* Note: variable_list, qualification, conditions, and justification
   are defined in Chapter 15 (Statements).
   type_expression is defined in Chapter 3 (Type System).
   functor_application is defined in Chapter 13 (Term Expressions).
   term_expression is defined in Chapter 13 (Term Expressions).
   param_prefix is defined in Chapter 6 (Attributes). *)
```
