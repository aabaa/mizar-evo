## 7.1 モードの定義と目的

モードは、`set` や `struct` などの基数型の上に構築されたユーザー定義型コンストラクターです。彼らは以下を許可します：

* 型としての数学的概念の **抽象化**
* 属性を使用した型の **改良**
* 複合型の **構成**
* 値に対する **依存関係** (つまり、依存型)

モードは Mizar の型分類の中心的なメカニズムであり、構造に代わる抽象的かつmodule式の手段を提供します。

## 7.2 モードの宣言と使用のための構文

**構文 (EBNF)**:

```ebnf
mode_def       ::= "mode" label ":" mode_def_name [ type_params ] "is" attribute_chain radix_type ";" ;
label          ::= identifier ;
mode_def_name  ::= constructor_name ;
type_params    ::= ( "of" | "over" ) type_parameter_list
                 | "[" type_parameter_list "]" ;
type_parameter_list ::= identifier { "," identifier } ;
attribute_chain ::= { [ "non" ] attribute_ref } ;
attribute_ref  ::= [ param_prefix ] [ struct_ref_name "." ] attribute_name
                   [ "(" argument_list ")" ] ;
param_prefix   ::= parameter "-" | "(" parameter_list ")" "-" ;
parameter_list ::= parameter { "," parameter } ;
parameter      ::= identifier | numeral ;
radix_type     ::= builtin_type
                 | mode_ref_name [ type_args ]
                 | struct_ref_name [ type_args ] ;
type_args      ::= ( "of" | "over" ) argument_list
                 | "[" type_arg_list "]" ;
type_arg_list  ::= type_arg { "," type_arg } ;
type_arg       ::= type_expression | qua_arg ;
qua_arg        ::= identifier { "qua" radix_type } ;
argument_list  ::= term_expression { "," term_expression } ;
mode_ref_name  ::= qualified_constructor_name ;
struct_ref_name ::= qualified_constructor_name ;
attribute_name ::= qualified_constructor_name ;
```

* `label`: この定義を参照するための必須の識別子
* `mode_def_name`: 定義するモードの名前。通常の識別子、または読みやすい
  ハイフン区切りのコンストラクタ名を使用できますが、任意の演算子風の記号記法は
  使えません。
* `type_params`: dependent mode の任意のパラメーター。bracket 形を標準形とし、`of` と `over` も省略形として受け入れます
* `attribute_chain`: 基本型を調整する 0 個以上の属性 (パラメーター化される可能性があります)

### モードの定義

```mizar
definition
  mode Label: <mode_name> is <attribute_chain> <type_expression>;
end;
```

ここで、`<attribute_chain>` は、基本型を改良する述語の (おそらく空の) シーケンス (パラメータ化されている可能性があります) です。属性は、`attr` キーワードを使用して定義する必要があります (第 6 章を参照)。

例:

```mizar
definition
  mode IntegerDef: Integer is integer Number;
  mode PrimeDef: Prime is prime Integer;
  mode GroupDef: Group is non empty associative invertible unital Magma;
end;
```

### モードの適用

モードは、変数の宣言、式、および型の制限で使用されます。

```mizar
let G be Group;
let n be Prime;
```

## 7.3 モードによる型推論

Mizar は、モードで宣言された変数の型関係を自動的に推測します。

* 推論は属性クラスターと構造継承に従います
* 推論は依存モードとパラメータ化された型をサポートします

例:

```mizar
let n be Prime;  :: implies n is Integer and Number
let S be Subset of X;  :: implies S is set and S ⊆ X
```

## 7.4 モードの実装

内部的には、各モードは次のように解釈されます。

1. メンバーシップの述語 (おそらく依存モードのパラメーターを含む)
2. 属性と基本型をリンクするクラスター登録

これにより、均一で効率的な型チェック メカニズムがサポートされます。

```mizar
definition
  mode Label: M is <attribute_chain> <type_expression>;
end;
```

これは、述語とクラスター登録を定義することと同じです。

### 7.4.1 プロパティの実装

構造体本体 (第 5 章を参照) で宣言された `property` は **仮想スロット** です。これは派生値の名前と型を宣言しますが、実装は提供しません。実装は、モードでパラメータ化された `definition` ブロック内の `property ... means` または `property ... equals` 定義を介して提供されます。

property implementation は property 値の唯一の供給源です。デフォルトの
構造体コンストラクターは field のみを受け取り、property 引数を取らず、
property の射影公理も生成しません (
[第 5 章 §5.5.1](./05.structures.md#551-デフォルトのコンストラクター) と
[§5.8.4](./05.structures.md#584-コンストラクターの公理) を参照)。

同じ struct property に対する property implementation は 1 つの実装 family
を形成します。異なる mode domain に対して複数の実装を定義できますが、2 つの
domain が重なる場合、それらの実装は `coherence` correctness condition
(§7.8.2) によって関係付けられなければなりません。正規化済み mode domain に
共通インスタンスがあり、それを検証器が静的に否定できない場合、その domain は
重なるものとします。特に、一方の実装 mode が他方の実装 mode を refinement
または inheritance によって狭める場合、狭い側は広い側と重なります。

```mizar
:: means form: define by logical condition
definition
  let M be UnitalMagma;
  property M.unit means
    for x be Element of M.carrier holds
    M.binop(it, x) = x & M.binop(x, it) = x;
  existence proof ... end;
  uniqueness proof ... end;
end;

:: equals form: define by direct term
definition
  let S be FinSequenceStr;
  property S.length equals len S.coeffs;
end;
```

* `it` は、(`means` 形式で) 定義されているプロパティ値を指します。
* `means` には、`existence` および `uniqueness` の証明義務が必要です。詳細については、§7.8.2 を参照してください。
* `equals` には `existence` または `uniqueness` の証明義務はありません。これらは term から直ちに従います。ただし、この実装が同じ property の別実装と重なる場合、`coherence` 義務は別途必要です。

## 7.5 モードと構造の関係

モードと構造は補完的な役割を果たします。内部フィールド (環や群などの代数構造) を含む複合型を定義する必要がある場合は、**構造** を使用します。内部構造 (素数、連続関数など) を追加せずに、論理プロパティによって既存の型を分類または制限したい場合は、**モード** を使用します。

|側面 |モード |構造物 |
| ---------- | ------------------------------------ | ---------------------------------- |
|定義 |要約 (属性経由) |コンクリート (フィールド/プロパティ経由) |
|使い方 |概念的な分類 |データ モデリングとコンポーネント アクセス |
|変換 |構造アクセス用の `qua` 演算子 |フィールド/プロパティへのアクセスは明示的です |

例：

```mizar
definition
  struct MetricStruct where
    field carrier -> set;
    field distance -> Function of [:carrier, carrier:], REAL;
  end;

definition
  mode MetricSpaceDef: MetricSpace is reflective discerning symmetric triangle MetricStruct;
end;
```

## 7.6 統合モードの構文

Mizar はモード定義に `is` 構文を適用し、従来の `means` 形式を置き換えます。これにより、次のことが保証されます。

* 簡素化された解析と推論
* 言語全体での統一的な表現
* 型 (モード) と述語 (論理プロパティ) の明確な分離

の代わりに：

```mizar
definition
  mode Prime means <definition>;
end;
```

使用：

```mizar
definition
  let n be Nat;
  attr PrimeAttr: n is prime means <definition>;
  mode PrimeDef: Prime is prime Integer;
end;
```

## 7.7 依存モード

モードは、`of` または `over` を使用してパラメータ化できます。

```mizar
definition
  mode SubsetDef: Subset of X is X-subset_like set;
  mode MatrixDef: Matrix over R is R-matrix_like set;
end;

let A be Matrix over REAL;
```

これらの形式は、template スタイルの定義の糖衣構文です。たとえば、`Matrix over REAL` は、他の表記法の `Matrix[REAL]` と同等です。

複数のパラメータがサポートされています。

```mizar
definition
  mode FinDimVSDef: FinDimVectorSpace over F, n is n-dimensional vector_space_like structure over F;
end;

let V be FinDimVectorSpace over COMPLEX, 5;
```

> [！警告]
> モード本体で使用されるすべての変数は、`type_params` で宣言する必要があります。以下は、`n` が宣言されていないため、エラーになります。

```mizar
definition
  :: ERROR: 'n' is not declared in type_params
  mode FinDimVSDef: FinDimVectorSpace over F
  is n-dimensional vector_space_like structure over F;
end;
```

## 7.8 正確性の条件

モード定義には、正規化済み右辺型式全体に対する inhabitation evidence が
必要です。非パラメータ mode では、これは次を意味します。

```text
∃x. is_RHS(x)
```

パラメータ化 mode `T(params) is RHS(params)` では、義務は有効なパラメータ
組ごとに guard されます。

```text
∀params. ParamGuard(params) → ∃x. is_RHS(x, params)
```

`ParamGuard` は、宣言された parameter type と attribute guard を正規化した
連言です。RHS が属性を持つ場合、evidence は正規化済み属性付き型全体に
対する可視 active existential registration です(§17.3.4)。RHS が無属性の
場合、検証器は §17.3.4 の base-shape inhabitation table(`object`、`set`、
accepted mode application、structure radix、template 本体内の schema type
parameter)を使います。

典型的な属性付き mode は cluster registration を使います。

```mizar
registration
  cluster prime Integer;
  existence
  proof
    take 2; thus 2 is prime;
  end;
end;

definition
  mode PrimeDef2: Prime is prime Integer;
end;
```

この検査は**必須**です。RHS 型式に対する inhabitation evidence が利用できない
場合、システムは mode 宣言の時点で **ハード エラー**を発生させます。これにより、
使用中のすべての mode が空でないことが保証され、論理の健全性が維持されます。
registration activation は item-ordered (§17.1) なので、後続の existential
registration は先行する mode 宣言を修復できません。

### 7.8.1 モードプロパティ: `sethood`

`sethood` プロパティは、モードのすべてのインスタンスが **セット** (適切なクラスではない) を形成することをアサートします。これは集合論の基礎にとって不可欠であり、モードのインスタンスを理解し、明確に定義された集合を生成することが保証されます。

**構文 (EBNF):**

```ebnf
mode_property ::= "sethood" justification ";" ;
```

**FOL義務:**

|プロパティ | FOLの義務 |
|---|---|
| `sethood` | `∃S. ∀x. (is_T(x) → x ∈ S)` |
| `T(params)` の `sethood` | `∀params. ParamGuard(params) → ∃S. ∀x. (is_T(x, params) → x ∈ S)` |

witness set `S` は正規化済み parameter tuple に依存してよいものとします。
`T(args)` の使用箇所は、同じ resolved mode と正規化済み argument vector に
対する sethood evidence だけを受け取ります。`T(a)` の sethood は、argument が
同じ tuple に正規化されない限り、`T(b)` 上の comprehension を許可しません。

public/exported mode について、accepted `sethood` property は semantic property
fact として module interface の一部になり、downstream availability と semantic
fingerprint に寄与します。`S` の具体的な witness term は使用可能な値としては
export されません。private mode の sethood は private のままです。

**例：**

```mizar
definition
  mode FiniteOrdinalDef: FiniteOrdinal is ordinal finite set;
  sethood
  proof
    :: Must prove: ex S being set st
    ::   for x being FiniteOrdinal holds x in S
    take omega;
    let x be FiniteOrdinal;
    thus x in omega by mml.ordinal.Finite_ordinal_in_omega;
  end;
end;
```

**検証器の効果**: `sethood` がモード T に対して証明されると、検証器は T の要素に対するフレンケル式 (集合内包表記) を許可します。`sethood` がないと、結果が適切なクラスである可能性があるため、内包表記 `{ f(x) where x is T : P(x) }` は拒否されます。

> [！注記]
> セット内包表記の制限については、[第13章 (項式), §13.4.2](./13.term_expression.md#1342-集合内包表記-フランケル演算子) を参照してください。

### 7.8.2 プロパティ実装の正確性

プロパティの実装では、`means` または `equals` のいずれかを使用します (§7.4.1 を参照)。 2 つの形式は、証明義務が異なります。

**構文 (EBNF)**:

```ebnf
property_impl    ::= "definition"
                       "let" identifier "be" mode_application ";"
                       ( property_means_impl | property_equals_impl )
                     "end" ";" ;

property_means_impl  ::= "property" identifier "." identifier "means" formula_definiens ";"
                         existence_block
                         uniqueness_block
                         [ coherence_block ] ;

property_equals_impl ::= "property" identifier "." identifier "equals" term_definiens ";"
                         [ coherence_block ] ;
formula_definiens ::= formula
                    | formula_case { "," formula_case } [ "otherwise" formula ] ;
formula_case      ::= formula "if" formula ;
term_definiens    ::= term_expression
                    | term_case { "," term_case } [ "otherwise" term_expression ] ;
term_case         ::= term_expression "if" formula ;

existence_block  ::= "existence" justification ";" ;
uniqueness_block ::= "uniqueness" justification ";" ;
coherence_block  ::= "coherence" justification ";" ;
```

|フォーム |義務 |理由 |
|------|------------|--------|
| `means` | `existence` + `uniqueness` |定義条件は明らかに一意の値を生成しない可能性があります。
| `equals` | `existence`/`uniqueness` は不要 | term が値を直接提供するため、存在と一意性は即時です |
| 重なる実装 | `coherence` | 同じ object に適用できる同一 property の実装は同じ値を定義しなければなりません |

`let M be <mode>` 句は、アドホック `assume` ステートメントを排除するための鍵です。モードの属性チェーンは、存在に必要な条件をすでに保証しています。

**重なりの coherence。** `I₁` と `I₂` を、同じ struct property `p` の 2 つの
可視実装とし、それぞれの正規化済み mode domain を `D₁` と `D₂` とします。
その重なり `D₁ ∩ D₂` が静的に空でない場合、後から来る実装は `coherence`
block を含まなければなりません。1 つの `coherence` block は、同じ property
について、以前から可視であるすべての重なる実装との義務を discharge します。

各実装は、その mode domain 内の object `x` と property 値 `v` の間の
定義関係 `Defᵢ(x, v)` を提供します。

| 組み合わせ | 義務で使われる関係 |
|---|---|
| `means` / `means` | それぞれの formula definiens で、`it` を `v₁` と `v₂` に置き換えたもの |
| `means` / `equals` | 片側の `means` formula と、`equals` 側の `v = term(x)` |
| `equals` / `equals` | `v₁ = term₁(x)` と `v₂ = term₂(x)` |

生成される coherence 義務は次の形です。

```text
∀x, v₁, v₂.
  (is_D₁(x) ∧ is_D₂(x) ∧ Def₁(x, v₁) ∧ Def₂(x, v₂)) → v₁ = v₂
```

必要な `coherence` 証明が欠落、保留、または拒否されている重なり実装は、
受理済み property implementation になりません。検証器は source order で
一方を選ぶのではなく、property-coherence diagnostic を報告します。

**例** — `UnitalMagma.unit`:

```mizar
:: struct UnitalMagmaStr declares: property unit -> Element of carrier  (virtual)
:: attr unital: MagmaStr is unital means ex e st (identity condition)
:: mode UnitalMagma is unital UnitalMagmaStr

definition
  let M be UnitalMagma;        :: M is unital is guaranteed by the mode
  property M.unit means
    for x be Element of M.carrier holds
    M.binop(it, x) = x & M.binop(x, it) = x;
  existence
  proof
    consider e being Element of M.carrier such that
    A1: for x holds M.binop(e, x) = x & M.binop(x, e) = x by UnitalDef;
    take e;
    thus for x holds M.binop(e, x) = x & M.binop(x, e) = x by A1;
  end;
  uniqueness
  proof
    let e1, e2 be Element of M.carrier;
    assume A1: for x holds M.binop(e1, x) = x & M.binop(x, e1) = x;
    assume A2: for x holds M.binop(e2, x) = x & M.binop(x, e2) = x;
    thus e1 = M.binop(e1, e2) by A2
           .= e2 by A1;
  end;
end;
```

## 7.9 論理エンコーディングの詳細

このセクションでは、モードが一階論理にどのようにマップされるかについて説明します。

### 7.9.1 述語としてのモード

各モード `M` はメンバーシップ述語としてエンコードされます。変数宣言 `let x be M` は以下を生成します。

|Mizar source | FOL エンコーディング |
| --- | --- |
| `let x be Integer;` | `is_Integer(x)` |
| `let G be Group;` | `is_Group(G)` |
| `let S be Subset of X;` | `is_Subset(S, X)` |

### 7.9.2 属性チェーンの拡張

属性チェーンを含むモードは接続詞に拡張されます。

|Mizar source | FOL エンコーディング |
| --- | --- |
| `mode Prime is prime Integer` | `∀x. is_Prime(x) ↔ (is_Integer(x) ∧ is_prime(x))` |
| `mode Group is non empty associative invertible unital Magma` | `∀x. is_Group(x) ↔ (is_Magma(x) ∧ ¬is_empty(x) ∧ is_associative(x) ∧ is_invertible(x) ∧ is_unital(x))` |

### 7.9.3 依存モードのエンコーディング

パラメーターを含む依存モードは、複数引数の述語としてエンコードされます。

|Mizar source | FOL エンコーディング |
| --- | --- |
| `let S be Subset of X;` | `is_Subset(S, X)` |
| `let V be FinDimVectorSpace over F, n;` | `is_FinDimVectorSpace(V, F, n)` |

### 7.9.4 サブ型の関係

モード継承により含意公理が生成されます。

```
mode Prime is prime Integer
⇒  forall x (is_Prime(x) implies is_Integer(x))
```

## 7.10 完全な構文 (EBNF)

モード関連の構文の概要。

```ebnf
(* Mode Definition *)
mode_def       ::= "mode" label ":" mode_def_name [ type_params ] "is" attribute_chain radix_type ";"
                   [ mode_property ] ;
label          ::= identifier ;
mode_def_name  ::= constructor_name ;
type_params    ::= ( "of" | "over" ) type_parameter_list
                 | "[" type_parameter_list "]" ;
type_parameter_list ::= identifier { "," identifier } ;
attribute_chain ::= { [ "non" ] attribute_ref } ;
attribute_ref  ::= [ param_prefix ] [ struct_ref_name "." ] attribute_name
                   [ "(" argument_list ")" ] ;
param_prefix   ::= parameter "-" | "(" parameter_list ")" "-" ;
parameter_list ::= parameter { "," parameter } ;
parameter      ::= identifier | numeral ;
radix_type     ::= builtin_type
                 | mode_ref_name [ type_args ]
                 | struct_ref_name [ type_args ] ;

(* Mode Property *)
mode_property  ::= "sethood" justification ";" ;

(* Property Implementation *)
property_impl        ::= "definition"
                           "let" identifier "be" mode_application ";"
                           ( property_means_impl | property_equals_impl )
                         "end" ";" ;

property_means_impl  ::= "property" identifier "." identifier "means" formula_definiens ";"
                         existence_block
                         uniqueness_block
                         [ coherence_block ] ;

property_equals_impl ::= "property" identifier "." identifier "equals" term_definiens ";"
                         [ coherence_block ] ;
formula_definiens ::= formula
                    | formula_case { "," formula_case } [ "otherwise" formula ] ;
formula_case      ::= formula "if" formula ;
term_definiens    ::= term_expression
                    | term_case { "," term_case } [ "otherwise" term_expression ] ;
term_case         ::= term_expression "if" formula ;
coherence_block   ::= "coherence" justification ";" ;

existence_block  ::= "existence" justification ";" ;
uniqueness_block ::= "uniqueness" justification ";" ;

(* Mode Application *)

attribute_name   ::= qualified_constructor_name ;

                   | "[" type_arg_list "]" ;

qua_arg          ::= identifier { "qua" radix_type } ;
```
