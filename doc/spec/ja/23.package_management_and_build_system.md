この章では、Mizar Evolution のパッケージ管理システム、ビルド pipeline、および IDE 統合レイヤーについて説明します。 **パッケージ** は配布とバージョン管理の単位であり、単一のマニフェストを持つ Mizar source fileの自己完結型コレクションです。ビルド システムは、source fileを、ダウンストリーム パッケージや IDE toolで使用できる検証済みのアーティファクトに変換します。

## 23.1 マニフェストfile形式 (`mizar.pkg`)

すべてのパッケージには、ルート ディレクトリに `mizar.pkg` という名前のfileが含まれている必要があります。マニフェストは TOML 構文を使用します。すべてのフィールドについては以下で説明します。 **必須**とマークされたフィールドは存在する必要があります。その他はすべてオプションであり、存在しない場合は指定されたデフォルトが使用されます。

```toml
[package]
name    = "algebra"           # required; lowercase, hyphens allowed
version = "2.3.1"             # required; semantic version (§23.2)
authors = ["Alice <alice@example.com>"]
license = "MIT"
description = "Algebraic structures"
edition = "2025"              # language edition; defaults to current stable

[dependencies]
mml-core = "^1.0"
topology  = { version = "^0.9", features = ["metric"] }

[dev-dependencies]           # used only during verification of this package
test-utils = "^0.2"

[verifier]
max_axioms    = 128          # global axiom budget per obligation (§21.4.2)
atp_timeout   = 30           # seconds per ATP backend per obligation (§22.4.5)
default_solver = "auto"      # "auto" | "vampire" | "e" | "cvc5" | "z3"
require_kernel_certificates = true   # reject externally attested proofs in release builds

[build]
incremental = true           # reuse cached artifacts for unchanged files
cache_dir   = ".mizar-cache" # relative to package root
artifact_dir = "build"       # where *.mizir.json and logs are written
```

**`[package]`** フィールド:

|フィールド |必須 |デフォルト |説明 |
|---|---|---|---|
| `name` |はい | — |依存関係の宣言とインポート パスで使用されるパッケージ識別子。 `[a-z][a-z0-9-]*` | と一致する必要があります。
| `version` |はい | — | `MAJOR.MINOR.PATCH` セマンティック バージョン管理形式のパッケージ バージョン (§23.2) |
| `authors` |いいえ | `[]` |著者文字列のリスト (従来は `"Name <email@example.com>"` |
| `license` |いいえ | `""` | SPDX ライセンス識別子 (例: `"MIT"`、`"Apache-2.0"`);空の文字列は、ライセンスが宣言されていないことを意味します。
| `description` |いいえ | `""` |人間が読める形式のパッケージの 1 行の説明 |
| `edition` |いいえ | — |言語版。parserと型ルールを選択します。エディションによっては、互換性のない構文変更が導入される場合があります。省略した場合は、現在の安定版を使用します。

**`[dependencies]`** エントリは、検証時に必要なパッケージを指定します。各エントリは、裸のバージョン文字列 (`"^1.0"`)、または `version` とオプションの `features` を含むテーブルのいずれかです。機能は、パッケージのエクスポートされた定義のオプションのサブセットを選択します。

**`[verifier]`** フィールドは、ATP インタラクションのグローバルなデフォルトをオーバーライドします。

|フィールド |デフォルト |説明 |
|---|---|---|
| `max_axioms` | `128` |グローバル公理の義務ごとの予算 (§21.4.2) |
| `atp_timeout` | `30` |義務ごとの ATP backendごとの秒数 (§22.4.5) |
| `default_solver` | `"auto"` | ATP backend: `"auto"` \| `"vampire"` \| `"e"` \| `"cvc5"` \| `"z3"`; `"auto"` はビルド時に利用可能な最速のbackendを選択します。
| `require_kernel_certificates` | `true` | `true` の場合、`externally_attested` (§21.7.5、§22.6.2) とマークされた証明は、警告ではなくビルド エラーとして扱われます。公開を目的としたパッケージに推奨 |

**`[build]`** フィールドは、インクリメンタル ビルド システムを制御します。

|フィールド |デフォルト |説明 |
|---|---|---|
| `incremental` | `true` | `true` の場合、各source file とその推移的な依存関係セットがハッシュされます。変更されていないfileは、キャッシュされたアーティファクトを再検証せずに再利用します。
| `cache_dir` | `".mizar-cache"` |ビルド キャッシュ ディレクトリへのパス (パッケージ ルートからの相対パス) |
| `artifact_dir` | `"build"` | `*.mizir.json` とログが書き込まれるディレクトリ (パッケージ ルートからの相対パス) |

## 23.2 バージョン管理と依存関係の解決

パッケージのバージョンは **セマンティック バージョニング** (SemVer 2.0.0) に従います。

- `MAJOR` — 互換性のない変更 (定義の削除または名前変更、型署名の変更、エクスポートの削除) に対して増分されます。
- `MINOR` — 下位互換性のある追加のために増分されます: 新しい定義、新しい定理、既存の証明を無効にしない新しい登録。
- `PATCH` — 下位互換性のある修正のために増分: 証明の修正、以前は弱かった `ensures` 句の修正、ドキュメントの更新。

> [！注記]
> **クラスターとoverloadの登録。** 新しい条件付きクラスター、機能クラスター、またはoverloadされた定義の追加は、通常は MINOR です。ただし、新しい登録によって、ダウンストリーム パッケージで既に型チェックされている式の **overload解決** (§23.8.3 `PickMostSpecific`) の結果が変更される場合 (たとえば、以前に選択されたものに代わるより厳密に具体的な登録を導入することによって、または以前は明確だった解決があいまいになる原因となる比類のない最小限の要素を導入することによって)、その変更は依存関係から監視可能であり、**MAJOR** として扱われなければなりません。 `UnionAll` 起動属性セットを拡張するだけの登録 (overloadの解決に影響を与えたり、新しい競合を作成したりすることはありません) は、MINOR のままです。

依存関係エントリの **バージョン制約** では、次の構文を使用します。

|制約 |意味 |
|---|---|
| `"1.2.3"` |正確にはバージョン 1.2.3 |
| `"^1.2.3"` | 1.2.3 との互換性: `>= 1.2.3, < 2.0.0` |
| `"~1.2.3"` |約1.2.3: `>= 1.2.3, < 1.3.0` |
| `">= 1.0, < 2.0"` |明示的な範囲 |

**依存関係の解決**は、**PubGrub アルゴリズム** を使用します。これは、バージョン制約解決専用に構築された競合主導のバックトラッキング リゾルバーです (Cargo のリゾルバーと Dart の pub toolで使用されるのと同じアルゴリズム)。このアルゴリズムは、バージョン条件のセット (パッケージごとの下限/上限) を維持し、現在の部分割り当てを一度に 1 パッケージずつ拡張し、アクティブな条件と互換性のある最新バージョンを常に選択します。競合が検出されると、**非互換性** (すべて同時に成り立つことができない最小限の用語のセット) が導出され、競合の原因となった最も初期の決定点に戻ります。この競合主導の学習により、ソルバー内部のヒューリスティックに依存せずに、「最新の互換性のあるもの」が確実に選択されます。満足のいく割り当てが存在しない場合、蓄積された非互換性により、ビルド システムが依存関係の競合エラーとして表面化する、人間が判読できる競合の説明が形成されます。

**Diamond の依存関係** (共有パッケージの異なるバージョンに依存する 2 つのパッケージ) は、バージョン範囲が重複する場合は統合によって解決され、重複しない場合はエラーで拒否されます。 Mizar パッケージは、1 つのワークスペース内で同じパッケージの複数の同時バージョンをサポートしません。

**再現性。** 解決されたバージョンの割り当ては、ワークスペース ルートの **ロック file** (`mizar.lock`) に書き込まれます。後続のビルドでは、リゾルバーを再実行するのではなく、ロック fileの正確なバージョンが使用されます。ロック fileはアプリケーションのバージョン管理にコミットされ、library の場合は除外されます (そのため、ダウンストリーム パッケージは依存関係を自由に解決できます)。

## 23.3 ワークスペースのレイアウト

標準の単一パッケージのワークスペースは次のレイアウトになっています。

```
mypackage/
├── mizar.pkg          ← package manifest
├── mizar.lock         ← resolved dependency lock file
├── src/
│   ├── lib.miz        ← top-level source file (exports)
│   └── ...            ← additional source files
├── build/
│   ├── lib.mizir.json ← resolved metadata artifact (§23.5.1)
│   ├── lib.atp-log    ← ATP interaction log (§23.5.3)
│   ├── missing_facts.json ← premise candidates (§23.5.2)
│   └── proof-witnesses/
│       └── lib/
│           └── vc-42.json ← per-obligation proof witness (§23.5.1)
└── .mizar-cache/
    ├── cluster-db/    ← global cluster and registration indexes (§23.7.7)
    └── ...
```

**マルチパッケージ ワークスペース** は、共通のルートの下にいくつかの関連パッケージをグループ化します。

```
workspace/
├── mizar.workspace    ← workspace manifest (lists member packages)
├── mizar.lock
├── algebra/
│   ├── mizar.pkg
│   └── src/
├── topology/
│   ├── mizar.pkg
│   └── src/
└── .mizar-cache/      ← shared cache across all members
```

ワークスペース マニフェスト (`mizar.workspace`) にはメンバー パッケージがリストされています。

```toml
[workspace]
members = ["algebra", "topology"]
```

メンバー パッケージは、ワークスペース ロック fileとビルド キャッシュを共有します。メンバー間のインポートは、パブリック パッケージ レジストリではなく、ワークスペースの依存関係グラフを通じて解決されます。

**インポート パス。** パッケージ内では、source fileは `src/` を基準とした相対パスによって識別されます。file `src/groups/basic.miz` は次のようにインポートされます。

```mizar
import groups.basic;
```

パッケージ間のインポートでは、パッケージ名が接頭辞として使用されます。

```mizar
import algebra.groups.basic;
```

## 23.4 ビルドのライフサイクルと再現性

ビルド システムは、source fileごとに次のフェーズを順番に実行します。フェーズは独立したfile間で並列化されます。インポート依存関係のあるfileはトポロジ順に処理されます。

|フェーズ |入力 |出力 |故障モード |
|---|---|---|---|
| **解析** | `.miz` source | AST |構文エラー (E0001 ～ E0099、§22.2) |
| **名前解決** | AST + インポートされたシンボル |解決済みのAST |未定義の名前エラー |
| **型チェック** |解決済みのAST |型付き AST + VC リスト |タイプ エラー (E0100 ～ E0299、§22.3) |
| **VC 世代** |型付きAST |証明義務 |論理不一致エラー (E0400 ～ E0499、§22.5) |
| **ATP前** |証明義務 |免責/オープン義務 | — (内蔵エンジン、§21.7) |
| **ATP ディスパッチ** |オープンな義務 |証明書または失敗の証明 |証明エラー (E0300 ～ E0399、§22.4) |
| **kernelチェック** |証明証明書 | `kernel_verified` / `externally_attested` |証明書の拒否 (E0350–E0359、§22.4) |
| **アーティファクトの放出** |完全に処理された AST + 証明書 | `*.mizir.json`、ログ | — |

**クラスターとoverloadの解決。** クラスターの解決 (§23.7) と登録包含 DAG (§23.8) によるoverloadの解決は、**型チェック フェーズ中に** 実行されます。AST が名前解決されると、型チェッカーはインポート フィルター処理されたクラスター インデックスと DAG を参照して、属性セットを導出し、overloadの勝者を選択し、`qua` 強制を挿入します。ここで発行された解決トレース (§23.7.7 `resolution-trace/`) は、後のフェーズと LSP サーバーにフィードされます。

**インクリメンタル ビルド。** 各フェーズの出力は、入力のハッシュとともにキャッシュされます。source fileが変更されると、影響を受けるfileとその逆依存関係コーンのみが再処理されます。クラスタ データベース (§23.7.7) は段階的に更新されます。変更されたfile内の登録のみが再インデックスされます。

**再現性** ビルドは、同一のsource file、ロック file、およびtoolチェーンのバージョンを指定して、ビットごとに同一のアーティファクトを生成する場合、**再現可能**です。ビルド システムは次の方法でこれを実現します。

- ロック fileを使用して依存関係のバージョンを修正します。
- 決定論的なランダム シードを使用して ATP 呼び出しをシードします (`[verifier]` で構成可能)。
- アーティファクト出力内のすべてのセットとマップを正規のキー順序で並べ替えます。

`require_kernel_certificates = true` の場合、再現性を確保するには、ATP backendが決定論的証明証明書を生成する必要もあります。証明書の決定性を保証できないbackend (一部の構成の Z3 など) は、`externally_attested` 証明を生成しますが、このモードでは拒否されます。

## 23.5 検証 toolのアーティファクトとビルド出力

ビルド システムは、検証されたsource fileごとに 3 種類のアーティファクトを `artifact_dir` に書き込みます。これらのアーティファクトは、検証 toolとダウンストリーム tool (IDE、AI アシスタント、ドキュメント ジェネレーター、依存パッケージ) の間の主要なインターフェイスです。

### 23.5.1 `*.mizir.json` アーティファクトの形式

各source file `src/foo.miz` に対して、ビルド システムは `build/foo.mizir.json` を書き込みます。このfileには、検証器のsourceに対する解決されたビュー (推論された型、解決されたシンボル、挿入された強制、アクティブな証明義務、およびそれらの証明ステータス) が記録されます。これは、LSP ホバー応答 (§23.6.1) および AI tool統合の主な入力です。

最上位構造:

```json
{
  "schema_version": "1.0",
  "source_file":    "src/foo.miz",
  "source_hash":    "sha256:a3f4...",
  "verified_at":    "2025-09-01T12:34:56Z",
  "expressions":    [ ... ],
  "obligations":    [ ... ],
  "exports":        [ ... ]
}
```

`verified_at` は、build directory を見る人間や IDE のための local かつ user-visible な metadata です。これは canonical artifact content hashes と publication reproducibility checks から除外されます。`verified_at` だけが異なる二つの artifacts は、dependency resolution and package publication において同等です。

**`expressions` 配列。** 各エントリは、型指定された AST 内の 1 つの式ノードをカバーします。

```json
{
  "expr_id":          "e124",
  "source_range":     "20:14-20:23",
  "kind":             "functor_application",
  "inferred_type":    "Element of PolyRing[K, x]",
  "resolved_symbol":  "algebra.poly.add",
  "template_args":    ["K", "x"],
  "inserted_coercions": ["PolyRing[K, x] qua AddStr"],
  "active_thesis":    "p + q = q + p",
  "overload_resolution": {
    "candidate_count":    3,
    "winning_candidate":  "algebra.poly.add#Def3",
    "selection_reason":   "unique_match"
  }
}
```

|フィールド |コンテンツ |
|---|---|
| `expr_id` |file内の安定した識別子 (不透明な文字列、file バージョンごとに一意) |
| `source_range` | `"line:col-line:col"` (1 ベースの Unicode スカラー値) |
| `kind` | AST ノードの種類: `functor_application`、`predicate`、`type_ascription`、`variable` など |
| `inferred_type` | Mizar サーフェイス構文でタイプ チェッカーによって割り当てられたタイプ |
| `resolved_symbol` |解決された関手、述語、またはモードの完全修飾名 |
| `template_args` |スキーマパラメータの代わりに具体的な引数を指定します。兄弟 `template_args_kind` フィールドによって明確化された 3 つの異なるバインディング sourceをカバーします: 関手引数バインディング (§23.7.3 の `?X₁, …, ?Xₙ`)、構造体パラメーター バインディング (パラメーター化された構造体型 `S₀[?P]` の `?P`、§23.7.1)、およびtemplate属性スキーマ バインディング (template属性の `?X`、§23.7.1)。各エントリはオブジェクト `{ "var": "?X", "value": "…", "source": "functor_arg"|"struct_param"|"attr_schema" }` |
| `inserted_coercions` | `qua` 強制は型チェッカーによって暗黙的に挿入されます。
| `active_thesis` |証明内にある場合、この式の位置における現在の証明義務。
| `overload_resolution` |式にoverloadされたシンボルが含まれる場合に存在します。非overload式の場合は `null` |
| `overload_resolution.candidate_count` |解決中に考慮されるoverload候補の数 |
| `overload_resolution.winning_candidate` |選択した定義の完全修飾名、その後に `#` と、その宣言サイトで特定のoverloadされた定義に割り当てられたラベル (識別子) (例: `algebra.poly.add#Def3`)。
| `overload_resolution.selection_reason` |登録包含 DAG (§23.8.3) のボキャブラリーを使用して、勝者がどのように選択されたか: `"unique_match"` (1 つの登録の引数ガードのみが一致)、`"most_specific_in_dag"` (複数の登録が起動され、DAG が `PickMostSpecific` の下で一意の最小要素を識別)、または `"explicit_qua"` (作成者が選択を強制する明示的な `qua` 強制を作成しました) |

完全なoverload解決トレース (考慮されたすべての候補者とそのスコアの詳細) は、`explain-overload <expr-id>` (§23.7.8) から入手できます。ここの `overload_resolution` フィールドには、ホバー表示および引用toolに必要な概要のみが記録されます。

**`obligations` 配列。** 各エントリは 1 つの検証条件をカバーします。

```json
{
  "obligation_id":  "vc-42",
  "kind":           "vc.loop.maintain",
  "source_range":   "35:3-35:30",
  "goal":           "s + i = i * (i + 1) div 2",
  "hypotheses":     ["i <= n", "s = i * (i - 1) div 2"],
  "proof_status":   "kernel_verified",
  "certificate":    "tstp",
  "prover":         "vampire",
  "elapsed_ms":     412,
  "used_axioms":    ["mml.nat.Nat_triangle_step", "mml.nat.Nat_add_comm"],
  "proof_witness":  {
    "file":          "build/proof-witnesses/foo/vc-42.json",
    "hash":          "sha256:c4d5...",
    "kernel_checked": true
  }
}
```

> [！注記]
> `proof_witness` フィールド形式は **暫定的で検討中** であり、kernel 再生、公理ごとのインスタンス化スコープ、高階バインディング表現、および用語共有をカバーする別の実装仕様に抽出する候補です。同じ証明で使用される複数の公理が同じ変数名を独立してバインドし、単一のフラットな置換マップがあいまいになる可能性があるため、完全なインスタンス化データはインラインではなく別のfile (`proof_witness.file` によって参照される) に保存されます。外部fileには公理ごとのバインディングが保存されるため、各公理の変数スコープは独立しています。以下に示すスキーマは例示的なものです。形式的な形式は、その別の仕様で定義されます。

参照される証人file (`build/proof-witnesses/<source_basename>/vc-<id>.json`) は次の構造になっています。

```json
{
  "schema_version": "1.0",
  "obligation_id":  "vc-42",
  "kernel_checked": true,
  "steps": [
    {
      "axiom":         "mml.nat.Nat_triangle_step",
      "instantiation": { "?n": "i" }
    },
    {
      "axiom":         "mml.nat.Nat_add_comm",
      "instantiation": { "?a": "s", "?b": "i" }
    }
  ]
}
```

`steps` の各エントリは 1 つの公理適用を記録します。 `instantiation` マップはその公理の量化された変数にスコープが設定されているため、2 つの異なる公理で同じ変数名が競合することはありません。正確な表現、特に高階のインスタンス化と用語の共有の処理方法はまだ最終化されておらず、将来のスキーマ バージョンで変更される可能性があります。

`proof_status` は次のいずれかです。

|値 |意味 |
|---|---|
| `"kernel_verified"` |最小限のkernelによって生成および受け入れられた証明書 |
| `"externally_attested"` | ATPは目標を受け入れた。kernel検証可能な証明書が生成されませんでした。
| `"discharged_builtin"` |プレ ATP 内蔵エンジンによって排出されます (§21.7)。 ATP コールはありません |
| `"open"` |まだ証明されていません。エラーまたはtimeoutが報告されました。

`used_axioms` フィールドには、受け入れられた証明証明書に表示される公理の完全修飾名がリストされます。 `proof_status` が `"kernel_verified"` または `"discharged_builtin"` の場合にのみ設定されます。このフィールドは、引用絞り込みの主な入力です (§23.5.4、§23.8)。

`proof_witness` フィールドは、外部監視fileへの参照です。 `proof_witness.file` はfileへのパス (パッケージ ルートからの相対パス)、`proof_witness.hash` は整合性チェック用の SHA-256 コンテンツ ハッシュです。最小kernel がインスタンス化された証明書を正常に再生した場合、`proof_witness.kernel_checked` フラグは `true` になります。 `false` 証明書が ATP によって生成されたが、kernelの再生がまだ実行されていない場合 (例: kernel フェーズが完了する前の増分ビルド パス中)。監視file自体には、公理ごとのインスタンス化置換が保存されます。その形式については、上記の注を参照してください。

**`exports` 配列。** 各エントリは、moduleからエクスポートされた 1 つの項目 (定理項目、定義、登録) を完全修飾名とsource範囲とともに記録します。この配列は、依存関係リゾルバーおよび IDE ナビゲーション機能によって使用されます。

### 23.5.2 `@missing_facts` 出力 (検証器が生成した前提候補)

ステータス `"open"` のすべての証明義務について、検証器は前提選択 (§22.4.4) を実行し、ランク付けされた候補リストを `build/missing_facts.json` に記録します。このfileは、AI 支援による `by` 句補完の主な入力です。

```json
{
  "schema_version": "1.0",
  "source_file":    "src/foo.miz",
  "missing_facts":  [
    {
      "obligation_id": "vc-42",
      "kind":          "vc.loop.maintain",
      "source_range":  "35:3-35:30",
      "goal":          "s + i = i * (i + 1) div 2",
      "triage":        "missing_lemma",
      "candidates": [
        {
          "rank":         1,
          "score":        0.94,
          "name":         "Nat_triangle_step",
          "fqn":          "mml.nat.Nat_triangle_step",
          "source":       "mml/nat.miz:213",
          "conclusion":   "for n being Nat holds n * (n+1) div 2 + (n+1) = (n+1) * (n+2) div 2"
        },
        {
          "rank":         2,
          "score":        0.81,
          "name":         "Nat_div_add",
          "fqn":          "mml.nat.Nat_div_add",
          "source":       "mml/nat.miz:198",
          "conclusion":   "for a, b being Nat holds (a + b) div 2 = a div 2 + b div 2 when 2 divides a+b"
        }
      ],
      "retry_succeeded": false
    }
  ]
}
```

`triage` は、`"counterexample"`、`"missing_lemma"`、または `"computation_limit"` (§22.4.2) のいずれかです。

検証器が上位 N 候補を公理セットに追加して ATP を再実行し、成功した場合、`retry_succeeded` は `true` になります。その場合、`candidates` の最初のエントリが義務を解決する補題となり、IDE はワンクリックで修正を行います。

> [！注記]
> `missing_facts.json` fileは、オープン義務を生成するビルド パスごとに再生成されます。これは、パッケージの公開されたアーティファクトの一部ではありません。これは開発時toolです。オープン義務 (ステータス `"open"`) を持つパッケージの公開は、パッケージ レジストリによって拒否されます。

### 23.5.3 ATP相互作用の概要ログ

ビルド システムは、source fileごとに、現在のビルドのすべての ATP インタラクションを記録する `build/foo.atp-log` を書き込みます。ログは行指向の JSON (1 行に 1 つの JSON オブジェクト) であり、効率的なストリーミングと grep ベースのフィルタリングを実現します。

各行に 1 つの ATP コールが記録されます。

```json
{"obligation_id":"vc-42","prover":"vampire","version":"4.8","elapsed_ms":412,"status":"proved","certificate_format":"tstp","axiom_count":37,"goal_size_literals":5}
{"obligation_id":"vc-43","prover":"vampire","version":"4.8","elapsed_ms":30000,"status":"timeout","axiom_count":128,"goal_size_literals":22}
{"obligation_id":"vc-43","prover":"e","version":"3.1","elapsed_ms":30000,"status":"timeout","axiom_count":128,"goal_size_literals":22}
```

フィールド:

|フィールド |コンテンツ |
|---|---|
| `obligation_id` | `*.mizir.json` の `obligation_id` と一致します。
| `prover` |backend名: `"vampire"`、`"e"`、`"cvc5"`、`"z3"` |
| `version` |ビルド時の証明器バージョン文字列 |
| `elapsed_ms` |この ATP コールの実時間 |
| `status` | `"proved"`、`"timeout"`、`"counterexample"`、`"error"` |
| `certificate_format` | `"tstp"`、`"lfsc"`、`"alethe"`、`"none"` |
| `axiom_count` |証明器に送信された公理の数 |
| `goal_size_literals` |否定された予想内のリテラルの数 |

ログは完全である必要はありません。`incremental = true` の場合、キャッシュされた義務の呼び出しは再発行されません。ログは、パフォーマンス分析、ATP チューニング、検証失敗の事後診断を目的としています。これは証明の有効性を示す形式的な記録ではありません。そのロールは `*.mizir.json` の証明書に属します。

### 23.5.4 使用済み公理の記録

最小kernel が証明証明書を受け入れると、証明書に表示される公理のセットを抽出し、対応する義務エントリ (§23.5.1) の `used_axioms` フィールドに記録します。このセットは常に、ATP に送信される公理のサブセットです。 ATP が検索に冗長な施設を含めたが、最終的な証明ではそのすべてを使用しなかった場合、厳密には小さい可能性があります。

検証器は、`used_axioms` を、対応する証明ステップの `by` 節内の明示的な引用と比較します。 `by` 句が `used_axioms` に現れない補題を指定している場合、検証器は警告 `W0103` (§23.9.1) を発行します。

一括引用 (`module.*`) は、個別に未使用としてフラグが立てられることはありません。その目的は、補題の選択を ATP に委任することです。それらを最小限にすることを要求すると、その目的が果たせなくなります。一括引用が成功すると、`used_axioms` フィールドに実際に使用された特定の補題が記録され、`mizar refine` (§23.9.2) が一括引用を正確なグループ化された引用に置き換えることができます。

> [！注記]
> `proof_status` が `"externally_attested"` の場合、`used_axioms` は空です。この場合、kernel解析可能な証明書は使用できません。改良toolは、`used_axioms` が設定されていなくても、黙って義務をスキップします。

## 23.6 LSP の統合と IDE のサポート

Mizar Evolution Language Server Protocol (LSP) サーバーは、ビルド アーティファクト (§23.5) とクラスター データベース (§23.7.7) を読み取り、対話型 IDE 機能を提供します。サーバーはバックグラウンド プロセスとして実行され、完全なビルドを再実行することなく、エディターからの LSP リクエストに応答します。

### 23.6.1 ホバー、コードアクション、および診断

**ホバー。** ユーザーが式の上にホバーすると、LSP サーバーは `*.mizir.json` 内の対応する `expr_id` を検索し、次を返します。

- 推論されたタイプ (`inferred_type` フィールド)。
- 解決されたシンボル FQN (`resolved_symbol` フィールド)。
- template引数 (`template_args`) (存在する場合)。
- 人間が読めるメモとしてフォーマットされた強制 (`inserted_coercions`) が挿入されました。
- 証明内にある場合は、その位置 (`active_thesis`) にあるアクティブな論文。

ホバー応答の例 (IDE によってレンダリング):

```
Type:     Element of PolyRing[K, x]
Symbol:   algebra.poly.add
          (resolved from `+` via PolyRing[K, x] → RingStr → AddStr)
Coercion: PolyRing[K, x] qua AddStr  (inserted implicitly)
Thesis:   p + q = q + p
```

**診断。** 最後のビルド中に生成されたすべての `error[Ennnn]` および `warning[Wnnnn]` メッセージは、source範囲とともに LSP 診断として表示されます。診断は保存するたびに段階的に更新されます。変更の影響を受けたfileのみが再検証されます。

`@show_thesis`、`@show_resolution`、および `@show_type` からの情報レベルの診断 (`I` シリーズ、§21.7) は、LSP `Hint` 重大度メッセージとして表示され、下線ではなく微妙なインライン注釈として表示されます。

**コード アクション。** LSP サーバーは、診断メッセージ (§22.1.3) 内の `help:` 提案を解析し、LSP コード アクションに変換します。ワンクリック修正は以下に対して提供されます。

- 欠落している `by` 句の補題を追加します (`missing_facts.json` から)。
- あいまいさを解決するために `qua` 強制を挿入します。
- 推論されたタイプ (`as T`) を使用してバインディングに注釈を付けます。
- `@proof_hint` 注釈を挿入します。
- オープン義務の `by` 節に定理を追加します。

**検証条件の表示コード アクション。** カーソルがループまたはアルゴリズム コントラクト上にある場合、LSP サーバーは、`*.mizir.json` の `obligations` 配列から取得されたオープン VC のリストとその証明ステータスを表示する「検証条件の表示」コード アクションを提供します。

## 23.7 クラスター解像度グラフ

クラスター解決 (条件付きクラスター登録、機能クラスター登録、および構造体の継承による属性の伝播) は、人間の作成者と AI toolの両方にとって、Mizar 型システムの最も不透明な側面の 1 つです。検証器は、2 つのクラスター解決インデックスと共有登録包含 DAG (§23.8) を維持します。存在クラスタ (§17.3) は型の存在を確立しますが、属性を伝播せず、どのグラフにも表示されません。

|指標・グラフ |出典 |役割 |
|---|---|---|
| **条件付きクラスター グラフ** |条件付きクラスター登録 (§17.4) |属性の伝播は型を支配します。
| **構造継承グラフ** | `extends` 宣言 (§5) |構造体のサブタイプ関係 |
| **登録包含 DAG** |機能クラスター (§17.5) とoverload登録 |ガードの包摂と射撃セットの選択 (§23.8) |

BFS 解決アルゴリズム (§23.7.4) は、条件付きクラスター グラフと構造体継承グラフを参照して、既知の結果型の属性を閉じます。関数クラスターは、個別の関手クラスター グラフではなく `UnionAll` を使用して、登録包含 DAG (§23.7.3、§23.8) を通じて関手アプリケーションの初期属性セットを提供します。

### 23.7.1 条件付きクラスターグラフ

条件付きクラスター グラフは、型に対する属性伝播ルールをキャプチャします。登録された条件付きクラスター ルールのみが保存され、構造体の継承はエンコードされません。

グラフは **構造体タイプによってインデックス付けされます**。構造体型 `S` ごとに 1 つの有向ハイパーグラフがあり、`G_S` と書き込まれます。各 BFS 実行は単一の具象構造体タイプ (§23.7.4) のコンテキストで動作するため、`G_S` 内のすべての頂点はプレーンな属性名であり、構造体タイプ修飾子は必要ありません。

`G_S` の **頂点**:
- `S` に適用可能なプレーン属性名 (例: `finite`、`non empty`、`complete`)
- template属性パターン — スキーマ引数を含む属性名 (例: `element_like[?X]`、`subset_like[?X]`)。スキーマ引数は、BFS 一致時にファクト セット内の具体的なファクトに対してバインドされます (§23.7.4)。template属性は、パラメーター化された属性名 (§17.4.4) を使用するクラスター登録から生成されます。

モードはグラフの頂点ではありません。 `mode M is A₁ A₂ … Aₙ T` (§7.2) として定義されたモードは、グラフ構築時にその基礎となる構造体型 `T` に展開されます (構造体型に到達するまで再帰的に)。 `for` 句でモード名を指定する条件付きクラスター ルールは、グラフにデータが設定される前に、拡張された構造体タイプとプレーンな属性名を使用して書き換えられます。

**非構造体モードのターゲット。** モード チェーンは、**非構造体プリミティブ**、特に `set` (すべてのセットのユニバース) または `struct` (§5) として宣言されていない抽象祖先で終了する場合があります。その場合、ルールは種類ごとのグラフ `G_set` (または対応するプリミティブ グラフ) に寄与します。プリミティブ グラフは、BFS アルゴリズム全体を通じて構造体グラフで均一に扱われます (§23.7.4)。唯一の実際的な違いは、構造体継承グラフ (§23.7.2) に `extends` エッジがないため、独自のルール セットを通じてのみ参加することです。 BFS は静的に既知のターゲットの種類を必要とするため、型変数、または先頭が登録された構造体またはプリミティブではない項に展開するモードは、登録時にエラーで拒否されます。

> [！注記]
> **パラメータ化された構造体タイプ** (例: `Subgroup of G` と `let G be Group`) は、**単一のtemplateインデックス付きグラフ** `G_{Subgroup[?P]}` によって表され、そのルールには登録のパラメータ制約を指定する `when: ?P is P` ガードが適用されます。登録 `let G be Group; cluster cyclic -> abelian for Subgroup of G` は `{ cyclic } → { abelian }   when: ?P is Group` に寄与し、`let G be abelian Group; cluster some_attr -> another_attr for Subgroup of G` は `{ some_attr } → { another_attr }   when: ?P is abelian Group` に寄与します。両方のルールは同じ `G_{Subgroup[?P]}` グラフ内に存在します。
>
> パラメータ間の伝播 (パラメータが `cyclic Group` の場合にも、`Subgroup[Group]` に登録されたルールが確実に適用されるようにする) は、template属性と同じメカニズムを使用し、**動的パラメータ マッチング** を通じて BFS アルゴリズム (§23.7.4) によって処理されます。 BFS が具体的な `H be Subgroup of G` を `G be cyclic Group` で処理するとき、`?P := cyclic Group` をバインドし、**アップキャストのみ**によって各ルールの `when` 句をチェックします。つまり、構造体継承チェーンとパラメータで**宣言された**属性です。 `cyclic Group is Group` は (同じベース、宣言された属性スーパーセット) を保持するため、`?P is Group` によって保護されるルールが起動されます。 `cyclic Group is abelian Group` は、`cluster cyclic -> abelian for Group` が登録されている場合でも、このガードを**保持しません**。これは、クラスター伝播が意図的にパラメーター型に適用されないためです (理論的根拠については、§23.7.4 のステップ 4a を参照してください)。補助継承エッジは生成されません。

**エッジ:** 各条件付きクラスター登録 (§17.4) は、指定された構造体型のグラフに 1 つのハイパーエッジを提供します。

|エッジの種類 |意味 |
|---|---|
| `conditional_cluster` |条件付きクラスター登録からの属性の伝播 (§17.4) |

複数の先行件属性を持つクラスター (`finite ∧ non empty → complete for LatticeStr` など) は **ハイパーエッジ** として保存されます。source セットにはすべての先行件のプレーンな属性名が含まれます。ハイパーエッジは、すべてのsource頂点が現在のファクト セットに存在する場合にのみ起動します。複数の結果として生じる属性を持つルールは、元の登録構造を維持したままグラフに保存されます。 BFS は、ルールが起動されると、各結果を個別に展開します。

構造体ごとの条件付きクラスター グラフの一般的なフラグメント:

```
(* G_set — rules for struct type `set` *)

(* Single-antecedent: degenerate hyperedge *)
{ empty }
  --conditional_cluster(mml.sets.cluster_empty_finite)-->    { finite }
  --conditional_cluster(mml.sets.cluster_empty_countable)--> { countable }

{ finite }
  --conditional_cluster(mml.sets.cluster_finite_countable)--> { countable }

(* Template attribute: ?X is a schema variable bound at BFS match time.
   Fires when the fact set contains `element_like[v]` for some concrete v
   that satisfies the parameter constraint `v is finite set`. *)
{ element_like[?X] }
  --conditional_cluster(mml.sets.cluster_element_of_finite,
                        when: ?X is finite set)--> { finite }

(* Template attribute in both antecedent and consequent.
   Fires when the fact set contains `element_like[v]` for some `v is set`,
   and adds `element_like[[:v, v:]]` to the fact set. *)
{ element_like[?X], element_like[?Y] }
  --conditional_cluster(mml.sets.cluster_pair_element,
                        when: ?X is set, ?Y is set)--> { element_like[[:?X, ?Y:]] }

(* G_LatticeStr — rules for struct type `LatticeStr` *)

(* Multi-antecedent: proper hyperedge — fires only when ALL sources are in the fact set *)
{ finite, non empty }
  --conditional_cluster(mml.lattice.FiniteNonEmptyComplete)--> { complete }

(* G_RingStr — rules for struct type `RingStr` *)
{ skew_field }
  --conditional_cluster(mml.ring.skew_field_is_domain)--> { domain_like }

{ domain_like }
  --conditional_cluster(mml.ring.domain_is_non_degenerate)--> { non_degenerate }

(* G_{Subgroup[?P]} — template-indexed graph for parameterized `Subgroup of G`.
   Each rule carries a `when: ?P is P` guard naming its parameter constraint.
   At BFS time, ?P is bound to the concrete parameter type and guards are checked. *)
{ cyclic }
  --conditional_cluster(mml.group.cyclic_subgroup_abelian,
                        when: ?P is Group)--> { abelian }

{ normal }
  --conditional_cluster(mml.group.normal_subgroup_of_solvable_is_solvable,
                        when: ?P is solvable Group)--> { solvable }
```

条件付きクラスター グラフは、インポート クロージャごとに物理的に再構築されるのではなく、グローバル登録インデックスとして保存されます。各ルールは、それを導入したパッケージまたはsource アーティファクトを記録します。インポート クロージャ `I` を持つsource fileの場合、`G_S[I]` は、`G_S` を `I` で表示される原点を持つルールにフィルタリングすることによって (または事前計算された可視性参照に従うことによって) 取得されるビューです。したがって、インポート セットが異なる 2 つのsource fileでは、インポート環境ごとにグラフを複製することなく、異なるルール ビューが表示され、同じ式に対して異なる属性結論に達する可能性があります (§23.7.7)。

### 23.7.2 構造体の継承グラフ

構造体継承グラフは、`extends` 経由で宣言された構造体型間のサブタイプ関係をエンコードします。これは、条件付きクラスター グラフとは別の有向非巡回グラフ (DAG) です。

**頂点:** 構造体のタイプ。ここでは、パラメータ化されたインスタンス化は頂点ではありません**。パラメータの変化は、このグラフの補助エッジではなく、動的 BFS マッチング (§23.7.1、§23.7.4) を介してtemplate インデックス付き条件付きクラスター グラフ `G_{S[?P]}` 内で処理されます。

**エッジ — `extends` 宣言 (§5):** 各 `extends` 宣言は、サブ構造体からスーパー構造体への 1 つの有向エッジに寄与します。

```
RingStr        --inheritance--> AddStr
RingStr        --inheritance--> MulStr
```

構造体継承グラフ自体は属性ファクトをエンコードしません。エッジは属性ペイロードを持ちません。その唯一の役割は、どの構造体ごとのクラスター ルール グラフが BFS ルール セットに寄与するかを選択することです (§23.7.4 ステップ 2)。親型のクラスター ルールから子型の式への属性の伝播は発生しますが、特別な「継承転送」パスではなく、ルールユニオン メカニズムを介して一律に行われます。

source コード内の `qua` キーワード (`x qua AddStr` など) は、複数のパスが存在する場合にどの継承パスに従うかを選択します (曖昧さ回避)。このグラフにエッジは追加されません。

構造体継承グラフは、条件付きクラスター ルール インデックスと一緒に `cluster-db` (§23.7.7) に保存されます。

### 23.7.3 機能クラスタの登録

機能クラスター登録では、特定の関手アプリケーションの結果に初期属性を割り当てます。これらは、別個の関手クラスター グラフとしてではなく、共有登録包摂 DAG (§23.8) 内の登録ノードとして保存されます。機能クラスター登録 (§17.5) の一般的な形式は次のとおりです。

```
let x₁ be P₁, …, xₙ be Pₙ;
cluster Label: F(x₁, …, xₙ) -> A₁ A₂ … for T;
```

そして、引数の制約 `x₁ is P₁, …, xₙ is Pₙ` が満たされると、式 `F(x₁, …, xₙ)` の構造体タイプが `T` で、属性が `A₁, A₂, …` であることを宣言します。引数の制約は登録の **ガード** であり、条件付きクラスターの `when` ガード (§23.7.1) と同様に処理されます。

各機能クラスターは `RegistrationNode` (§23.8.1) に寄与します。

```
RegistrationNode {
  symbol            = F,
  argument_guard    = [when: ?X₁ is P₁, …, ?Xₙ is Pₙ],
  consequent        = { result_type = T, attributes = { A₁, A₂, … } },
  resolution_policy = UnionAll,
  assume_clause     = None
}
```

引数制約のない登録 (無条件の場合) は空の `when` 句を運び、常に起動します。

機能クラスター登録ノードの典型的なフラグメント:

```
(* Single guard without attribute refinement — applies to any Ring argument *)
mml.poly.PolyRing_is_Ring:
  symbol=PolyRing
  guard=[?R is Ring]
  consequent={ result_type=RingStr,
               attributes={ non empty, associative, abelian } }
  policy=UnionAll

(* Multiple conditional registrations for the same functor — all may fire *)
mml.sets.CartesianNonEmpty:
  symbol=Cartesian
  guard=[?X is non empty set, ?Y is non empty set]
  consequent={ result_type=set, attributes={ non empty } }
  policy=UnionAll

mml.sets.CartesianCountable:
  symbol=Cartesian
  guard=[?X is countable set, ?Y is countable set]
  consequent={ result_type=set, attributes={ countable } }
  policy=UnionAll

mml.sets.CartesianFinite:
  symbol=Cartesian
  guard=[?X is finite set, ?Y is finite set]
  consequent={ result_type=set, attributes={ finite } }
  policy=UnionAll
```

**ガード評価。** §23.7.4 ステップ 4a のパラメーター ガード (アップキャストのみで評価される) とは異なり、機能クラスター ガードは **各引数の BFS 派生属性セットを参照します**。これは現在の Mizar の動作と一致します。具体的な引数 `A` に対して `?X is finite set` が成立するかどうかをチェックするとき、検証器は `A` の完全に伝播された属性セットを使用します (たとえば、`A` は、`empty → finite` などの条件付きクラスターを介して `finite` を継承しました)。この違いは、2 人の警備員の役割の違いによって正当化されます。

- **パラメータ ガード (§23.7.4 ステップ 4a)** は、*タイプ宣言* (`let G be abelian Group`) (宣言された属性のみがスコープ内とみなされる宣言的行為) に適用するルール ビューを選択します。
- **関数クラスター ガード (こちら)** は、関手アプリケーション サイトの引数 *式* に適用されます。これらの引数は既にアンビエント ファクト セット内の派生ファクトを保持しているため、実際の (BFS クローズド) 属性コンテンツを評価することが適切です。

**他のグラフとの相互作用。** 検証器が式 `e = F(a₁, …, aₙ)` の型を推測するとき:

1. 各引数 `aᵢ` の BFS クローズド属性セット `F_i` を計算します (§23.7.4 の再帰適用)。
2. シンボル `F` の表示されているすべての機能クラスター登録ノードを列挙し、`F₁, …, Fₙ` に対するそのガードをテストします。 `UnionAll` ポリシーに基づいて、ガードが保持されているすべての登録が同時に発火します。登録包含 DAG (§23.8) は、この列挙を加速します。最も具体的な発砲登録が特定されると、そのすべてのアクティブな祖先は、ガードを再評価することなく発砲することがわかります。
3. `A₀ := ∪ { consequent.attributes of each firing registration }` にしましょう。結果の構造体型 `T` は、起動登録から取得されます。同じ関手のすべての起動登録は、`T` に同意する必要があります。この一貫性は現在のインポート クロージャに対してチェックされ、違反した場合はビルド エラーとして報告されます。これは、異なる結果型を持つ 2 つの登録が競合するのは、両方が一緒に表示される場合のみであるためです。
4. `T` の条件付きクラスターを介してさらに伝播するために、`S = T` および `A₀` を使用して BFS (§23.7.4) をシードします。

同じ DAG は、`PickMostSpecific` ポリシー (§23.8.3) によるoverload解決にも使用されます。したがって、ビルド システムは、関数クラスターの起動とoverloadされた関手/モード/述語/属性の選択の両方に対して 1 つのガード包含構造を維持します。

### 23.7.4 BFS 解決アルゴリズム

BFS 解決アルゴリズムは、初期属性セット `A₀` が与えられた場合に、結果の構造体型 `S` がすでにわかっている式の完全な属性セットを決定します。関手アプリケーションの場合、`S` および `A₀` は、登録包含 DAG (§23.7.3、§23.8) を通じて可視の関数クラスター登録ノードを起動することによって最初に取得されます。 BFS 自体は関手のoverloadを解決せず、関数クラスターの登録をトラバースしません。表示される条件付きクラスターと構造体の継承の下でのみ `A₀` を閉じます。

**入力:**
- 具象構造体型 `S` (おそらくパラメーター化されたインスタンス化 `S₀[P₀]` — 例: `G be abelian Group` を使用した `H be Subgroup of G` から派生した `Subgroup[abelian Group]`)
- 初期属性セット `A₀` (プレーンまたはインスタンス化された属性名。該当する場合は機能クラスター登録によって提供され、それ以外の場合は空)
- インポート クロージャ `I` — 検証されるsource fileによって推移的にインポートされるパッケージのセット
- グローバル条件付きクラスター ルール インデックス `G`。各ルールは元のパッケージ/source アーティファクトによってタグ付けされます (§23.7.1)
- グローバル構造体継承グラフ `H`、各エッジはその元のパッケージ/source アーティファクトによってタグ付けされます (§23.7.2)
- 可視述語 `visible_I(origin)`。インポート クロージャから派生し、両方のインデックスをフィルタリングするために使用されます。

**アルゴリズム：**

1. **該当するスーパー構造体を列挙します。** 表示されている継承エッジのみをトラバースします。
   ```
   H[I]      := { e ∈ H | visible_I(e.origin) }
   Supers(S) := { T | S →* T in H[I] }
   ```
   パラメータ化された `S = S₀[P₀]` の場合は、グローバル パラメータ バインディング `?P := P₀` を維持します。パラメータ化されたインスタンス化は、個別の継承頂点には展開されません。

2. **表示されるルール ビューを収集します。** 原点が表示され、ターゲットの種類が `S` に適用されるルールをグローバル条件付きクラスター インデックスから選択して、ローカル ルール ビューを構築します。
   ```
   Sources(S) := { base(S) } ∪ Supers(S)
   if S = S₀[P₀], also include the template source S₀[?P]

   Rules :=
     { r ∈ G
       | visible_I(r.origin)
       ∧ r.target ∈ Sources(S) }
   ```
   これはフィルタリングされたビューであり、再構築されたグラフではありません。 Implementations may materialize it for the current source file, or may keep references into the global index and test `visible_I` lazily.選択された各ルールは、`element_like[?X]` などのプレーン属性またはtemplate属性を持つ `{ B₁, …, Bₙ } → { C₁, …, Cₘ }` の形式を持ち、オプションで `when: ?P is Group` などのパラメーター ガードを持ちます。

3. **ファクト セットを初期化します。** `F := A₀`。

4. **BFS ループ** (`F` が安定するまで実行):

   ａ． **固定パラメータ ガードを準備します。** template source `S₀[?P]` からの `when: ?P is P` ガードを運ぶ各ルールについて:
      - `S = S₀[P₀]` の具象パラメータを使用して `?P := P₀` をバインドします。
      - **アップキャストのみ**によって実行されるこの BFS のガードを 1 回評価します。`H[I]` にパス `base(P₀) →* base(P)` があり、** で宣言された属性セットがある場合、`P₀ is P` は保持されます。** `P₀` は、`P` で宣言された属性セットのスーパーセットです。クラスターの伝播はパラメーターには**適用されません**。直接宣言された属性とモードのみが参加します。
      - ガードが失敗した場合は、この実行に対してルールを非アクティブとしてマークします。これが当てはまる場合は、ルールの結果を `?P` に置き換えて、ルールをアクティブなままにします。

      たとえば、`abelian Group is Group` は (属性スーパーセット、同じ基本構造体) を保持します。 `abelian` は `cyclic Group` の宣言された属性の中にないため、`cluster cyclic -> abelian for Group` が表示されていても、`cyclic Group is abelian Group` はこのガードを**保持しません**。
      - **理論的根拠:** パラメータ型は作成者による意図の宣言であり、推論されるべき事実ではありません。パラメータに対して BFS を再帰的に実行すると、ガード評価が高価になり、インポート クロージャに二重に依存し、「どのルール セットが適用されるか」と「どの属性が派生するか」の分離が曖昧になります。より広範囲の一致を望む作成者は、パラメーター宣言の範囲を広げるか (例: `let G be abelian Group`)、ルールをより弱いレベルで登録する必要があります (例: `for Subgroup[?P] when: ?P is Group`)。

   b. **プレーンなアクティブ ルール。** ファクト レベルのスキーマ変数を持たない各アクティブ ルール `{ B₁, …, Bₙ } → { C₁, …, Cₘ }` について:
      - すべての先行件 `B₁, …, Bₙ` が `F` に存在する場合、後続の各 `C₁, …, Cₘ` を `F` に追加し、新しい先行件が満たされたすべてのルールをキューに入れます。

   c. **template属性のアクティブ ルール。** ファクト レベルのスキーマ変数 `?X` を含む各アクティブ ルール `{ B₁[?X], …, Bₙ[?X] } → { C₁[?X], …, Cₘ[?X] }` について:
      - 各候補バインディング `?X := v` について、`F` 内の形式 `Bᵢ[v]` のファクトを照合することによって取得されます。
        - パラメーター制約 `v is P` (登録で `let ?X be P` が宣言されている場合) を **`v` の BFS クローズド属性セット** に対して検証します。これは、ステップ 4a のアップキャスト専用ポリシーではなく、機能クラスター ガード評価 (§23.7.3) と並行して行われます。ここでの `v` は、一致時にバインドされた具象式であるため、`v` の派生属性はスコープ内にあり、アンビエント ファクト セットと一致します。
        - すべての先行件 `B₁[v], …, Bₙ[v]` が同じバインディングを持つ `F` に存在する場合:
          - インスタンス化された各結果 `C₁[?X := v], …, Cₘ[?X := v]` を `F` に追加します。
          - 新しい先行条件が満たされたばかりのエンキュー ルール。
      - 複数のスキーマ変数 (`?X` や `?Y` など) は個別にバインドされます。ルールが適用される前に、すべてが一貫して満たされている必要があります。
      - ルールでは、固定 `?P` パラメータ ガード (ステップ a、アップキャストのみ) とファクトレベルのスキーマ変数 (ステップ c、BFS クローズ) を組み合わせることができます。両方の一致条件がそれぞれのポリシーの下で成立する必要があります。

   d.ルールごとの先行カウンタ (template ルールのバインディング対応追跡) を使用すると、各反復ですべてのルールを再スキャンすることなく、起動を効率的に検出できます。 `?P` は具体的な入力型 `S` によって固定されるため、パラメーター ガードの結果 (ステップ a) は BFS 実行ごとに 1 回評価されます。

5. **結果。** 最後のファクト セット `F` には、**現在のインポート クロージャ `I`** の下で構造体型 `S` の式に保持されるすべての属性名 (プレーンおよびtemplateでインスタンス化されたもの) が含まれています。

**主要なプロパティ:**
- **インポートに依存します。** 同じ式により、追加のクラスター提供パッケージをインポートするfileで、より大きな `F` が生成される可能性があります。インポートを追加すると、`F` しか成長できません (インポートに関しては単調)。インポートを削除すると、インポートが縮小され、型エラーが発生する可能性があります。
- `F` のすべての頂点は、プレーンまたはインスタンス化された属性名です (スキーマ変数は一致時に削除されます)。構造体型 `S` は暗黙的なコンテキストです。継承は、どの条件付きルール ビューが `Rules` に寄与するかを決定するだけです。
- 反復にわたる単調: 属性は追加されるだけで、削除されることはありません。終了は、登録されたクラスターが終了するように設計されていることに依存しており、これは一貫性証明要件 (§17.4) によって強制されます。
- 属性の除去 (例: `finite DoubleMagmaStr` を `DoubleMagmaStr` として扱う) は暗黙的な型の包含です。ファクト セットには必要以上の属性が含まれており、常に弱い要件を満たします。

### 23.7.5 ローカルサブグラフの抽出(パスビュー、近傍ビュー)

AI toolと IDE 機能は、完全なグローバル グラフではなく、**ローカル サブグラフ**を受け取ります。 2 つの抽出モードが定義されています。

**パス ビュー。** 特定のクエリを解決するために検証器が実際に通過したエッジの最小限のシーケンスを返します。これは、「なぜ `T` には `A` 属性があるのですか?」に対する主な応答です。クエリ。

```json
{
  "query":  "S is countable",
  "status": "resolved",
  "path": [
    {
      "kind":   "conditional_cluster",
      "from":   "empty set",
      "to":     "finite set",
      "source": "mml.sets.cluster_empty_finite",
      "location": "mml/sets.miz:47"
    },
    {
      "kind":   "conditional_cluster",
      "from":   "finite set",
      "to":     "countable set",
      "source": "mml.sets.cluster_finite_countable",
      "location": "mml/sets.miz:112"
    }
  ]
}
```

各パス エントリの `kind` フィールドは、§23.7.1 ～ §23.7.3 および §23.8 で定義されているグラフ エッジおよび登録起動ステップに対応する次の値のいずれかをとります。

| `kind` |sourceセクション |意味 |
|---|---|---|
| `conditional_cluster` | §23.7.1 |構造体ごとの条件付きクラスター グラフ内の属性の伝播 |
| `inheritance` | §23.7.2 | `extends` による構造体のサブタイプ |
| `functional_cluster` | §23.7.3 |関手アプリケーションから設定された初期属性 |
| `subsumption` | §23.8 |同じシンボルの登録間の発火セットの包含 |

**近傍ビュー。** グラフ内の頂点のすぐ隣を、競合する候補およびその候補が選択された理由 (または解像度があいまいな理由) とともに返します。これは、「なぜこれがあいまいなのですか?」に対する主な応答です。クエリ。

```json
{
  "query":  "PermProduct(R)",
  "status": "ambiguous",
  "vertex": "RingStr",
  "candidates": [
    {
      "target": "AddStr",
      "edge_kind": "inheritance",
      "path_length": 2,
      "reason": "RingStr → AddStr → MagmaStr (additive path)"
    },
    {
      "target": "MulStr",
      "edge_kind": "inheritance",
      "path_length": 2,
      "reason": "RingStr → MulStr → MagmaStr (multiplicative path)"
    }
  ],
  "suggested_fix": "insert `qua` annotation in source to disambiguate: `PermProduct(R qua MulStr)`"
}
```

抽出深度には制限があります。デフォルトでは、パス ビューは最大 8 つのエッジを返し、近傍ビューは最大 16 個の候補を返します。これらの制限は、説明 API (§23.7.8) を介して増やすことができます。

### 23.7.6 説明抽出子(成功、曖昧さ、失敗)

**説明エクストラクター** は、グローバル グラフとローカル サブグラフ応答の間のレイヤーです。検証器からの解決結果を受け取り、人間と機械が読める説明を構築します。

次の 3 つの結果クラスが処理されます。

**成功。** 検証器は一意の解決パスを見つけました。エクストラクタは、各ステップ、そのsource登録、および登録が定義されているlibrary内の場所を示すパス ビュー (§23.7.5) を返します。

**曖昧さ** 同じ長さと優先度の解決パスが 2 つ以上見つかりました。エクストラクターは、競合するすべての候補を含む近傍ビューを返します。 `suggested_fix` フィールドは、具体的な曖昧さの解消 (通常は `qua` 強制または修飾名) を提案します。

**失敗。** 解決パスが存在しません。エクストラクターは失敗を分類します。

|失敗の種類 |意味 |提案された修正 |
|---|---|---|
| `unreachable` |source頂点を必要なターゲットに接続するパスがありません |欠落しているクラスターを登録するか、`qua` 強制を追加します。
| `attribute_not_registered` |必須の属性には、このタイプに対して起動するクラスターがありません。タイプ | のクラスター登録を追加します。
| `schema_var_unbound` |スキーマ変数 (パラメーター `?P`、ファクトレベル `?X`、または関手引数) を具体的な値にバインドできませんでした。引数を明示的に指定するか、一致ルールのガードを広げます。
| `qua_required` | `qua` 強制が必要ですが、書き込まれませんでした。関連するサイトに `qua Interface` を挿入します。
| `sethood_missing` |フランケル理解には `sethood` (§21.7.6) が必要です。タイプ | に `sethood` を登録します。

AI toolの場合、失敗の説明は成功のトレースよりも実用的です。これらは、コードの型チェックを行うための登録の欠落や強制を特定します。

### 23.7.7 ストレージフォーマット(`cluster-db`、`resolution-trace`、`diagnostic-explanation`)

クラスター解決インフラストラクチャは、ビルド キャッシュ内の 3 つのストレージ レイヤーを使用します。

**`cluster-db/`.** クラスターおよび登録インデックス。登録ごとの可視性メタデータを含む静的ビルド キャッシュ データとして保存されます。キャッシュはインポート クロージャごとに複製されるわけではありません。検証器は、source fileのインポート クロージャからその起点が見えるルール、宣言、および DAG エッジをフィルタリングすることによって、インポート スコープのビューを導出します。インポートされたパッケージのセットが変更されると、可視性ビューのみが再計算されます。現在のパッケージ内のsource fileが変更されると、影響を受ける登録とその派生インデックス エントリのみが更新されます。

クラスタ データベースは、以下を含むディレクトリです。
- `graph.json` — 条件付きクラスター グラフと構造体継承グラフの完全な頂点とエッジのリスト。コンパクトな隣接形式で、インポート フィルター用の起点識別子が含まれます。
- `subsumption-dag.json` — 登録包含 DAG スケルトン (§23.8): `(symbol, guard)` によってキー設定されたノード、条件付きクラスターの前提条件が記録された強いエッジから弱いエッジへと続くため、クエリ時にインポート クロージャーに対して DAG をフィルターできます。
- `attr-index.json` — 属性名からそれを生成するすべての登録への逆インデックス。
- `type-index.json` — タイプ名からそれによってトリガーされたすべての登録への逆インデックス。
- `reduction-index.json` — 解決済み `LHS` head からリダクション登録への項 head インデックス。各規則の FQN、型・属性ガード要約、`such` 副条件要約、簡約順序メタデータ、可視性起点を含む。

**`resolution-trace/`.** 検証済みsource file (`foo.resolution-trace.json`) ごとに 1 つのfile。そのfileの型チェック中に実際に使用されたクラスター解決パスとリダクション適用を記録します。記録されたクラスター パスは、パスビュー抽出 (§23.7.5) への入力です。記録されたリダクション ステップは、グラフのパスビュー抽出ではなく、kernel 再生と戦略監査 (§17.6.4) に用います。fileは、影響を受けるfileの検証パスごとに再生成されます。

解決トレースは、**自動登録効果の最小kernel検証のための証明証明書**としても機能します。各トレースステップは、クラスター ステップまたはリダクション ステップとしてタグ付けされます。クラスター ステップ群は、BFS トラバーサルによって実際に取得されたルール適用の順序付きシーケンスを構成します。参照される各ルールは `coherence` 証明 (§17.4、§17.5) によってすでに検証されているため、kernel は、参照されたクラスター登録が存在すること、すべての前件が現在のファクト セットに存在することを確認し、後件を追加することでチェーンを検証します。

リダクション ステップでは、2 層の証拠を記録します。kernel 再生層は、リダクション規則の FQN、source redex、インスタンス化された対象項、パターン置換、解消済みの型・属性ガードおよび `such` ガードを記録します。規則は `reducibility` 証明 (§17.6) によってすでに検証されているため、最小 kernel は局所的な書き換え証拠だけを検証します。すなわち、記録された redex が規則の `LHS` に一致すること、記録された対象が対応する `RHS` インスタンスであること、記録された各ガードに、fact/trace 参照または明示引用として安定した証拠があることを確認します。戦略監査層は、そのステップ前の包含項 fingerprint、その項内の redex path、有効なリダクション rule-view fingerprint、選択規則の特殊性/FQN selection key を記録します。このデータは、トレースを発行する前に決定的 normalizer が検査し、artifact validator が再計算してもよいものです。最小 kernel は規則探索やリダクションの再選択を行いません。再生は、外部ソルバーを必要としない線形時間チェックのままです。

**`diagnostic-explanation/`.** source file (`foo.diagnostic-explanation.json`) ごとに 1 つのfile。型チェック中に発生したすべてのあいまいさと失敗に対する説明抽出プログラムの出力を記録します。このfileは、解決エラーに関連する IDE ホバーおよびコードアクション応答の主な入力です。

3 つのfileはすべて、UTF-8 でエンコードされた JSON を使用します。これらのスキーマは、トップレベルの `schema_version` フィールドでバージョン管理されます。toolはこのフィールドをチェックし、サポートされていないスキーマ バージョンのfileの読み取りを拒否する必要があります。

### 23.7.8 説明API

説明 API は、LSP サーバー (IDE 使用用)、CLI (`mizar explain`)、および `*.mizir.json` fileから直接 (AI tool使用用) の 3 つのインターフェイスを通じて公開されます。

5 つのクエリ 型は次のとおりです。

**`explain-type <expr-id>`**。指定された `expr_id` を持つ式の完全な型推論トレースを返します。推論された型に寄与したクラスター駆動型の拡大のパス ビューが含まれます。

CLI: `mizar explain type --file src/foo.miz --expr e124`

LSP: 式の上にマウスを置くとトリガーされます。結果はホバー応答に入力されます。

**`explain-attribute <type> <attribute>`**。 「`<type>` には `<attribute>` 属性があります」というクエリの解決パス (または失敗の説明) を返します。両方の引数に Mizar サーフェス構文を受け入れます。

CLI: `mizar explain attribute "finite set" "countable"`

応答 (成功):

```json
{
  "query":  "finite set is countable",
  "status": "resolved",
  "path": [
    { "kind": "conditional_cluster", "from": "finite set", "to": "countable set",
      "source": "mml.sets.cluster_finite_countable", "location": "mml/sets.miz:112" }
  ]
}
```

**`explain-overload <expr-id>`**。式のoverload解決結果 (勝者候補、考慮されたすべての候補、勝者を決定したスコアの詳細) を返します。 `@show_resolution` (§21.5.2) の出力と同等ですが、構造化された JSON です。

**`explain-qua <expr-id>`**。 `qua` 強制の説明 (どのビューが選択されたか、それが必要な理由、およびどのような代替手段が存在するか) を返します。暗黙的に挿入された強制 (`*.mizir.json` の `inserted_coercions` に記録される) の場合、このクエリは挿入が必要な理由を説明します。

**`explain-failure <diagnostic-id>`**。 `CODE` (例: `E0201`) とsource範囲によって識別される、特定のエラーまたは警告の完全な診断説明を返します。応答には、説明抽出者の分類 (§23.7.6)、競合する候補または欠落している登録、および提案された修正が含まれます。これは、診断出力 (§22.1.3) の `help:` 行とプログラム的に同等です。

CLI: `mizar explain failure --code E0201 --file src/foo.miz --line 42`

すべての API 応答は、`schema_version` フィールドを持つ JSON オブジェクトです。 CLI はデフォルトでこれらを人間が読めるテキストとしてレンダリングします。 `--json` を渡して生の JSON を受け取ります。 LSP サーバーは常に内部で JSON フォームを使用し、IDE のホバーおよびコードアクション形式を使用して表示用にレンダリングします。

> [！注記]
> 説明 API は、バルク ダンプではなく、増分クエリ用に設計されています。 AI toolは、完全なクラスター データベースまたはすべての解決トレースを一度に要求するのではなく、特定の式または失敗についての説明を要求する必要があります。制限された抽出深さ (§23.7.5) はこれを強制します。単一クエリの深さ制限の増加はサポートされていますが、この制限は、大規模なlibrary環境のグラフ データによって AI コンテキスト ウィンドウが圧倒されるのを防ぐために存在します。

### 23.7.9 メモリ設計原則

§23.5 と §23.7 で述べた保存・クエリ設計は、単一の意図的な原則に従います。すなわち、**インターフェイスと index は常駐させ、証明本体・トレース・AI 向けの詳細データは遅延的に読み込む**、という原則です。これは分離コンパイルのメモリモデル（§12.6.3）に対応するビルドシステム／tool 側の対応物であり、検証器と言語サーバーが、メモリを library の規模とともに増大させることなく大規模な library 環境へスケールできるようにするものです。

この原則は、本章で既に規定済みの次の設計ルールによって実現されます。

| ルール | 機構 | 節 |
|---|---|---|
| cluster／登録グラフを import closure ごとに複製しない | 単一の global `cluster-db` から、可視性の origin でフィルタして import スコープのビューを導出 | §23.7.7 |
| 登録データ全体をメモリに読み込まない | オンディスクの逆引き index（`attr-index`、`type-index`、`reduction-index`）とグラフの compact adjacency format | §23.7.7 |
| 解決／簡約トレースを常駐させない | source file ごとに 1 つの `resolution-trace` artifact を各検証パスで再生成し、必要時に読み込む | §23.7.7 |
| proof witness を inline しない | `*.mizir.json` からパスと hash で参照する外部 witness file | §23.5.1 |
| グラフデータを AI tool へ一括ダンプしない | 既定の抽出深さ制限を伴う、有界かつ増分的な説明クエリ | §23.7.5、§23.7.8 |
| 変更分のみ再計算する | cluster-db の増分更新と逆依存コーンの再処理 | §23.4 |

LSP サーバー（§23.6）もこれらの artifact を同様に利用します。すなわち、global な状態をメモリ上に再構築するのではなく、`*.mizir.json`・`cluster-db` ビュー・file ごとのトレースを読み取り、index 化されたデータとオンデマンドの参照とからホバー・コードアクション・説明の要求に応答します。

> [!NOTE]
> これらのルールは定性的なメモリモデルを表すものであり、性能保証ではありません。具体的なメモリ予算やベンチマーク指標（例: file ごと、import closure サイズごとのピーク RSS、あるいは LSP のアイドル時メモリ）は評価方法論に属し、本仕様の対象外とすることを意図しています。

## 23.8 登録包含 DAG

同じシンボルに対する複数の登録は、通常、引数ガードの包含順序を形成します。たとえば、関手 `[:_, _:]` の場合、条件付きクラスター `finite → countable` (§23.7.1) がすべての `finite set` を `countable set` にするため、`{ ?X, ?Y : finite set }` でガードされた登録は `{ ?X, ?Y : countable set }` でガードされた登録よりも厳密に強力です。関手だけでなく、overloadされた **モード**、**述語**、**属性**でも同じ現象が発生します。

この章では、**登録包含 DAG** を、クラスターの起動 (§23.7.3、`UnionAll` ポリシー) と過負荷解決 (`PickMostSpecific` ポリシー) の両方に機能する共有基板として定義します。その目的は 2 つあります。

- **加速。** 最も具体的な発砲登録が特定されると、その祖先はすべて (`UnionAll` の場合) ガードを再評価することなく発砲することがわかります。
- **統合。** 関手、モード、述語、および属性のoverload解決では、同じ包含機構が再利用され、解決ポリシーと `assume` 句の存在のみがコンシューマを区別します。

各overload コンシューマーの言語レベルの意味論は、それぞれの章で指定されます (属性 §6、モード §7、関手 §10、述語 §9)。この章では、共有解像度インフラストラクチャのみを定義します。

### 23.8.1 抽象登録ノード

登録ノードには以下が含まれます。

```
RegistrationNode {
  symbol              : functor | mode | predicate | attribute,
  argument_guard      : [when: ?X₁ is P₁, …, ?Xₙ is Pₙ],
  consequent          : <consumer-specific payload>,   (* cluster: attribute set;
                                                          overload: definition body *)
  resolution_policy   : UnionAll | PickMostSpecific,
  assume_clause       : Option<FOL formula>            (* overload only; None for clusters *)
}
```

- **`symbol`** は、その登録が 1 つの包含格子を形成するoverloadされたエンティティを識別します。
- **`argument_guard`** は条件付きクラスター ガード (§23.7.1) と同じ `when: ?X is P` 形式を使用するため、含意機構は共有されます。
- **`consequent`** は DAG に対して不透明です。消費者向けの各章はその形状を定義します。
- **`resolution_policy`** は、発射セットがどのように消費されるかを決定します (§23.8.3)。
- **`assume_clause`** はoverload ノードに接続されていますが、包含には参加しません (§23.8.4)。

### 23.8.2 包摂エッジとインポートスコープのインスタンス化

**エッジ条件。** `R_strong.argument_guard` が `R_weak.argument_guard` を伴う場合、エッジ `R_strong → R_weak` が追加されます。含意は、クラスター グラフ (§23.7.1) から抽出された (おそらく空の) 条件付きクラスター ルールのセットによって証明されます。これらはエッジの **前提条件** として記録されます。

```
R_finite_finite  --subsumption(via mml.sets.finite_is_countable on {?X, ?Y})--> R_countable_countable
```

`on {?X, ?Y}` アノテーションは、引用された条件付きクラスター ルールが個別に適用されるスキーマ変数をリストします。前提条件に複数の個別のクラスター ルールが必要な場合、それらは `via { rule_a on {?X}, rule_b on {?Y} }` というセットとしてリストされます。

**インポート スコープのインスタンス化。** 前提条件で条件付きクラスター ルールが引用されているため、DAG はインポート クロージャー `I` によってパラメーター化されます。エッジは、引用するすべてのルールが `I` にある場合にのみアクティブになります。完全な DAG スケルトン (エッジ + 前提条件) は一度構築され、`cluster-db` (§23.7.7) に保存されます。クエリ時に、`I` に対してフィルタリングされ、アクティブな DAG が取得されます。

エッジ `R_strong → R_weak` が現在の `I` で非アクティブである場合 (`R_strong.guard ⇒ R_weak.guard` を証明する条件付きクラスター ルールがインポートされていないため)、`R_strong` からのトラバーサルは、`R_weak` の起動を自動的に暗黙的に示すことはできません。その場合、`R_weak` 登録には、具体的な引数に対して独立して評価されるガードが必要です。これは、DAG なしで実行されるのと同じ作業です。したがって、DAG はパフォーマンス アクセラレータであり、正確性への依存ではありません。すべての登録の列挙とそれに続く独立したガード チェックは有効な (遅い場合でも) 実装のままであり、結果は同じです。

**非巡回性** 包含関係は部分順序であり、そのエッジは常に厳密に強いガードから厳密に弱いガードの方向に向けられます (同じシンボルの等価ガードの登録はロード時に検出されるエラーであり、エッジに折りたたまれません)。したがって、結果として得られるグラフは、構造的には DAG です。

### 23.8.3 解決ポリシー

消費者は、発射セットをどのように消費するかによって異なります。

|消費者 | `resolution_policy` |発砲動作 |
|---|---|---|
|機能クラスター (§17.5) | `UnionAll` |すべての射撃登録が貢献します。結果 (属性セット) は結合されます。 |
|overloadされた関手 / モード / 述語 / 属性 | `PickMostSpecific` |アクティブな DAG 内の固有の最小要素を選択します。曖昧さ（比較できない最小限の要素）は重大なエラーです。 |

`UnionAll` の場合、DAG は単一のトラバーサルを可能にします。つまり、最も具体的な一致する登録を起動し、アクティブ エッジを介して祖先に到達します。祖先に対するすべてのガード テストはスキップされます。

`PickMostSpecific` の場合、DAG は選択を直接提供します。最も具体的な起動登録は、アクティブ エッジに沿って起動子孫を持たないものです。曖昧さ (アクティブな DAG の下で比較できない 2 つ以上の最小発火登録) は、解決エラーとして報告されます。作成者は明示的に曖昧さを解消する (例: `qua`) か、包含エッジを提供するパッケージをインポートすることによって解決する必要があります。

### 23.8.4 `assume` 句の扱い

overload登録には、`assume P(x)` 句が含まれる場合があります。これらの節は、次の 3 つの理由により、包含エッジに*​​参加しません**。

1. `assume` 句は任意の FOL 式です。それらの間の論理的な意味合いは決定できないため、安定したエッジ状態を提供できません。
2. 現在の Mizar のoverload解決は、最初に引数ガードによって解決され、使用サイトでの別個の証明義務として `assume` が解除されます。この 2 段階の動作を維持します。
3. `assume` が包含命令に参加した場合、「どの登録が最も具体的であるか」は証明放出に依存し、型推論が定理証明に依存することになります。これは、決定可能な解決策の出発点ではありません。

したがって、`assume` 条項は、選択された `RegistrationNode` に引き継がれ、解決が完了した後、使用サイトで解除されます。放電の失敗は、解決エラーではなく証明エラーです。解決結果は、引数ガードが一致するとすぐに修正されます。

## 23.9 引用の絞り込み

引用の改良は、冗長または大量の `by` 引用を、kernel が実際に使用した最小限の補題セットに置き換えるプロセスです。検証器は、証明成果物 (§23.5.4) で使用された公理を記録し、警告と 2 つの専用 CLI toolによる改良をサポートします。

### 23.9.1 検証器の警告 (W0103)

証明ステップの明示的な `by` 節が、`used_axioms` (§23.5.4) に存在しない補題を指定する場合、検証器は次のように出力します。

```
warning[W0103]: by clause contains unused citations (proof.citation.unused)
  --> src/lib.miz:35:7
   |
35 |   thus thesis by poly_eval_add, poly_eval_mul, poly_eval_const;
   |                                                ^^^^^^^^^^^^^^^ not used in proof certificate
   |
help: remove unused citation, or run `mizar refine` to rewrite automatically
```

この警告は情報提供です。関係なく、検証は成功します。その目的は、著者が段階的に証明を強化できるように、冗長な引用を表面化することです。

一括引用 (`module.*`) は W0103 から除外されます。一括引用の目的は、補題の選択を ATP に委任することです。最小限にすることを要求すると、その目的が果たせなくなります。 `mizar refine` を使用して、成功した一括引用を正確な明示的な形式に変換します。

### 23.9.2 `mizar refine` — 証明証明書からの高速絞り込み

`mizar refine` は、`*.mizir.json` に記録された `used_axioms` セットを使用して、source file内の `by` 引用を書き換えます。 ATP または検証器は再実行されません。既存のビルド アーティファクトを読み取り、source の純粋に構文上の書き換えを実行します。

```
mizar refine [--file <path>] [--dry-run] [--all]
```

|オプション |効果 |
|---|---|
| `--file <path>` |指定されたsource fileのみで引用を絞り込む |
| `--dry-run` |fileを変更せずに提案されたリライトを印刷する |
| `--all` |パッケージ内のすべてのsource fileを調整する |

**動作。** `by` 句を含む各証明ステップについて:

1. `*.mizir.json` で対応する義務エントリを検索します。
2. `proof_status` が `"kernel_verified"` で、`used_axioms` が空でない場合、引用リストを正規にソートされた `used_axioms` 内の名前で置き換えます。
3. 使用されているすべての公理が同じmoduleからのものである場合、一括引用 (`module.*`) はグループ化された引用 (`module.{name1, name2, ...}`) に置き換えられ、それ以外の場合はフラットな明示的な引用に置き換えられます。
4. `proof_status` が `"externally_attested"` または `"open"` であるステップは変更されません。

`mizar refine` を実行した後、後続のビルドは `kernel_verified` ステータスのすべての義務で成功するはずです。失敗した場合、toolは不一致を報告し、元の引用を復元します。

> [！注記]
> `mizar refine` はべき等です。2 回実行すると 1 回実行した場合と同じ結果が得られます。その出力は直接コミットしても安全です。

### 23.9.3 `mizar minimize` — ブルートフォースの最小化

`mizar minimize` は、各 `by` 節から個々の引用を削除して再検証することを繰り返し試みることにより、`mizar refine` よりもさらに進んでいます。証明証明書に何が記録されているかに関係なく、ATP が目標を達成できる最小限の引用のサブセットを見つけます。

```
mizar minimize [--file <path>] [--strategy <greedy|delta>] [--timeout <sec>]
```

|オプション |効果 |
|---|---|
| `--file <path>` |指定されたsource fileのみでの引用を最小限に抑えます。
| `--strategy greedy` |一度に 1 つの引用を削除し、検証を続ける削除は維持します (デフォルト) |
| `--strategy delta` |デルタ デバッグ: 引用サブセットに対する二分探索 (大規模な引用セットでは平均して ATP 呼び出しが少なくなります) |
| `--timeout <sec>` |試行ごとの ATP timeout。デフォルトはパッケージ `atp_timeout` 設定です。

**`mizar refine` よりも `mizar minimize` を優先する場合。**

- `proof_status` が `"externally_attested"` の場合: 証明書が利用できないため、`mizar refine` は動作できませんが、`mizar minimize` はサブセットを使用して ATP を再実行することで、より小さい引用セットを見つけることができます。
- 証明が大規模な明示的な `by` リストを使用して手動で作成された場合: `mizar refine` は、証明書に存在しない補題のみを削除します。 `mizar minimize` は、代替の証明パスを発見することにより、ATP が厳密に小さいサブセットで成功することがわかる場合があります。
- 最大限にコンパクトな証明が必要な場合 (たとえば、パッケージを正式な libraryに公開する前)。

`mizar minimize` は、証明ステップごとに複数の ATP 呼び出しが必要なため、`mizar refine` よりも大幅に遅くなります。これは、通常のビルド サイクルのステップとしてではなく、時折クリーンアップ toolとして使用することを目的としています。
