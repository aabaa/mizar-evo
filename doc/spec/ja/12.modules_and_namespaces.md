## 12.1 moduleの概念と論理エンコーディング

Mizar の **module** は 2 つの異なる役割を果たします。

1. **コンパイル単位**: 単一の `.miz` source fileに対応します。
2. **論理理論**: 一階論理における理論 $T = (\Sigma, \Gamma)$ を定義します。ここで、$\Sigma$ は署名 (functor、述語、型などの語彙)、$\Gamma$ は公理と定理のセットです。

**namespaceの原則**:
論理namespaceと物理file パスの間には **1 対 1 のマッピング**があります。moduleで定義されているすべてのシンボルは、fileの場所から直接派生した **完全修飾名 (FQN)** によって論理的に識別されます。

## 12.2 リポジトリのレイアウトと命名

moduleは階層的に編成されています。信頼レベル、安定性、エコシステムの成長を効果的に管理するために、システムは **5 つの主要なnamespaceルート**を区別します。

### 12.2.1 論理namespaceルート

|ルート |名前 |説明 |ガバナンス |
| --- | --- | --- | --- |
| **`std`** |標準library | Mizar チームによって保守されている信頼できるコア library |Mizar チーム |
| **`pub`** |発行 (JFM) | *Journal of Formalized Mathematics* からの検証済みの記事 | JFM編集委員会 |
| **`pkg`** |パッケージレジストリ |中央レジストリに登録されたサードパーティ パッケージ |コミュニティ (crates.io モデル) |
| **`dev`** |開発 |ローカル開発ワークスペースと実験プロジェクト |ユーザーローカル |
| **`ext`** |外部 |外部リポジトリへの参照 (ミラー、フォーク) |外部メンテナー |

#### `std` (標準library)

標準libraryには、すべてのユーザーが信頼できる基礎的な理論が含まれています。 Mizar チームによって厳格な品質管理が行われています。

```mizar
import std.algebra.group;
import std.topology.metric_space;
import std.set_theory.zfc;
import std.algebra.linear.{eigen_value, jordan};
```

#### `pub` (発行/JFM)

*Journal of Formalized Mathematics* からの検証済みの記事が含まれています。これらは、JFM 出版物の学術的価値と引用の安定性を維持する「凍結された」理論です。

```mizar
import pub.jfm.pepin.jordan_curve;
import pub.jfm.trybulec.boolean_properties;
```

#### `pkg` (パッケージレジストリ)

中央の Mizar パッケージ レジストリに公開されたサードパーティ パッケージ (crates.io と同様)。パッケージ名は登録する必要があり、グローバルに一意です。

**命名規則**:

|パターン |説明 |例 |
| --- | --- | --- |
| `pkg.<package_name>` |登録されたパッケージ名 | `pkg.mathcomp_mizar.algebra` |
| `pkg.<scope>.<name>` |スコープ付きパッケージ | `pkg.nakasho.quantum.hilbert` |

```mizar
import pkg.mathcomp_mizar.algebra.ring;
import pkg.bourbaki.topology.filters as filters;
import pkg.nakasho.quantum.hilbert;
```

**登録ルール**:
* パッケージ名は中央レジストリに先着順に登録されます。
* スコープ (`pkg.` の後の最初のコンポーネント) はアカウントを登録することで予約できます
* パッケージ名は小文字でアンダースコアを付ける必要があります (`snake_case`)

#### `dev` (開発)

未公開のプロジェクト、実験、進行中の作業のためのローカル開発ワークスペース。レジストリとは共有されません。

```mizar
import dev.my_project.utils;
import dev.experiments.category_theory;
```

#### `ext` (外部)

ローカルにミラーリングまたはベンダー化された外部リポジトリへの参照。中央レジストリに公開されていない依存関係に使用されます。

```mizar
import ext.github.user.repo.module;
import ext.gitlab.org.project.submodule;
```

> [！注記]
> `ext` ルートは例外的な場合を想定しています。発見しやすさと依存関係の解決のために、作成者は `pkg` に公開することをお勧めします。

### 12.2.2 fileのnamespaceへのマッピング

パッケージ マネージャーは、これらの論理ルートを物理パスにマップします。論理namespaceは、ディレクトリ構造とfile名を正確に反映します。

* **形式**: `<Root>.<Component>...<Name>`
* **ルール**：
  * パス区切り文字 `/` (Windows では `\`) は、ネームスペースのドット `.` に対応します。
  * namespace識別子では、`.miz` 拡張子が省略されます。

**例**:

|論理namespace |物理パス (ルートからの相対パス) |
| --- | --- |
| `std.algebra.linear.vector_space` | `std/algebra/linear/vector_space.miz` |
| `pub.jfm.pepin.jordan_curve` | `pub/jfm/pepin/jordan_curve.miz` |
| `pkg.mathcomp_mizar.algebra.ring` | `pkg/mathcomp_mizar/algebra/ring.miz` |
| `dev.my_project.utils` | `dev/my_project/utils.miz` |

## 12.3 インポートステートメント

`import` ステートメントは、別のmoduleへの依存関係を宣言します。

import declarations は module の **import prelude** を形成します。comments and layout を無視したとき、`.miz` file 内のすべての `import` statements は、最初の non-import top-level item より前に、連続して現れなければなりません。import prelude は、`export`、`reserve`、`definition`、`registration`、theorem、proof、algorithm、その他の declaration を含む、最初の `import` ではない top-level item で終了します。

Imports は file-scoped です。prelude 内のすべての import は、module body 全体に対する 1 つの active import closure と 1 つの active lexical environment に寄与します。`import` statements は definitions、registrations、proofs、algorithms の内部、または non-import top-level item の後には許可されません。後続の import-shaped statement は syntax error です。

この制約により、lexing and disambiguation は module body を parse する前に、import prelude から active symbol set を一度だけ構築できます。

**構文 (EBNF)**:

```ebnf
import_stmt       ::= "import" module_alias_decl { "," module_alias_decl } ";" ;
module_alias_decl ::= module_path [ "as" module_identifier ]
                    | module_branch_import ;
module_branch_import
                  ::= module_path ".{" module_identifier { "," module_identifier } "}" ;
module_path       ::= [ relative_prefix ] module_identifier { "." module_identifier } ;
relative_prefix   ::= "." | ".." ;
module_identifier ::= identifier ;
```

### 12.3.1 配置ルール

1. import prelude は、optional module-level documentation comments and other trivia の後、file 内の最初の `import` statement から始まります。
2. prelude には `import` statements だけが含まれます。
3. prelude は最初の non-import top-level item で終了します。
4. prelude の後に `import` statement は現れてはいけません。
5. file の active lexical environment は complete prelude から構築され、file の残りの部分で安定します。

### 12.3.2 解決ルール

1. **絶対インポート**:
  * `import std.algebra.group;`
  * `<std_root>/algebra/group.miz` に解決されます。
2. **相対インポート**:
  * `import .utils;` (兄弟: `./utils.miz`)
  * `import ..common;` (親: `../common.miz`)

### 12.3.3 エイリアシング

エイリアスは、冗長性や衝突を避けるためにnamespaceの短い名前を提供します。

```mizar
import std.algebra.group as Grp;
let G be Grp.Group;
```

### 12.3.4 循環インポート

循環依存関係 (例: A が B をインポート、B が A をインポート) は**禁止されています**。コンパイル順序の決定可能性を確保するには、依存関係グラフは有向非巡回グラフ (DAG) である必要があります。

## 12.4 エクスポートステートメント

`export` ステートメントは、インポートされたシンボルまたはnamespaceを再公開し、現在のmoduleを事実上「ファサード」または「集約」moduleにします。これにより、共通の定義を集約する「プレリュード」moduleを作成できます。

`export` statements が存在する場合、それらは import prelude の後、ordinary declarations の前に現れます。

**構文 (EBNF)**:
  ```ebnf
  export_stmt ::= "export" module_path { "," module_path } ";" ;
  ```

* **使用法**: `export std.algebra.group;`

## 12.5 可視性制御(プライベート/パブリック)

デフォルトでは、すべての定義と宣言は **public** です。可視性修飾子は、トップレベルまたは定義ブロック内のいずれかで、個々の項目へのアクセスを制限できます。

**構文 (EBNF)**:
```ebnf
visibility        ::= "private" | "public" ;

(* Top-level declarations *)
top_level_decl    ::= claim_block
                    | [ visibility ] ( theorem_item | notation_decl ) ;

(* Items inside a definition block *)
definition_block_item ::= [ visibility ] ( definitional_item | theorem_item | registration_item ) ;
definitional_item     ::= func_def | pred_def | mode_def | attr_def | struct_def | algorithm_def ;
```

* **パブリック**: moduleの署名 $\Sigma$ の一部。輸入業者が入手可能です。
* **プライベート**: 内部補題またはヘルパー定義。file内でのみ表示されます。

public な **algorithm** (§20.1.1) は、その *署名と契約* — 名前、パラメータと戻り値の型、`requires`、`ensures` — を module 署名 $\Sigma$ の一部として公開します。実装 **本体** は private な詳細であり、インポート側に読み込まれることは決してありません（§12.6.1 の opaque-import モデルと整合）。本体が読み込まれないため、インポート側はインポートした algorithm を `by computation` で評価できません（§20.9.1）。下流の証明はその契約にのみ依存します。この署名 vs 本体の分離が、algorithm 本体のみの編集を非破壊な変更にし、一方で public な algorithm の契約変更は依存側から観測可能になります（§23.2）。

## 12.6 論理エンコーディングと個別コンパイル

モノリシック libraryで見られるパフォーマンスの低下を回避するために、Mizar は **Separate Compilation** モデルを採用しています。

### 12.6.1 不透明なインポート

module A がmodule B をインポートする場合:

1. **再検証なし**: module B の証明は再チェックされません**。
2. **署名のロード**: 論理署名と公定理項目のみがロードされます。

### 12.6.2 ATP のシンボルマングリング

識別子は、FQN に基づいて ATP のグローバルに一意なシンボルに変換されます。

|Mizar source |論理エンコーディング (TPTP の例) |
| --- | --- |
|namespaceプレフィックス `std__algebra__group__` |
|述語 `is_abelian` | `std__algebra__group__is_abelian` |

### 12.6.3 メモリモデル

分離コンパイルは単なるビルド時の最適化ではなく、検証器の**常駐集合モデル**をも規定します。module を import すると、その public な signature と定理項目のみが読み込まれ（§12.6.1）、証明本体・private な補題・elaboration state は決して読み込まれないため、ある module を検証するのに必要なメモリは、その背後にある library の証明内容に応じて増大しません。

具体的には、1 つの module を検証する際のピークメモリは、おおよそ次のようになります。

```
ピークメモリ
  ≈ size(active な source file とその型付き AST)
  + size(import した public インターフェイス)        (* public 定理の言明を含む——§12.6.1、アーキテクチャ §3: ModuleSummary *)
  + size(import でフィルタした登録 index)             (* §23.7: フィルタによる cluster-db ビュー *)
  + size(active な module の検証条件)                 (* 正準 VcIr。discharge 前に生成される——アーキテクチャ §7 *)
  + size(進行中の discharge 状態)                     (* active な VC キューと ATP portfolio——§23.4、アーキテクチャ §14 *)
  + 小さな各種キャッシュ
```

証明義務ごとの ATP 作業——discharge 状態と prover portfolio——は、library 内の全証明義務ではなく、現在進行中の検証条件に限定されます。module 全体の正準 `VcIr` は discharge に先立って実体化される（アーキテクチャ §7）ため、上記の上界に寄与するのは単一の証明義務ではなく、その module 自身の VC 集合です。

そして、この常駐集合は次のものとは**独立**です。

- import した module の証明本体および private な補題（読み込まれない——§12.6.1）。
- 現在の import closure 外の登録データ（`cluster-db` は単一の global index であり、import closure ごとに実体化せず、フィルタによって import スコープのビューを導出する——§23.7.7）。
- import した証明義務の proof witness（hash で参照される外部 artifact として保存される——§23.5.1）。

これは cluster および登録の基盤がその側で従うのと同じ原則です。すなわち、cluster-db は単一の global index であり、import closure ごとに複製するのではなくフィルタによって import スコープのビューを導出し（§23.7.7）、説明データやトレースデータは常駐させるのではなく有界かつ増分的なクエリで供給されます（§23.7.8）。分離コンパイルと import スコープのフィルタリングがあいまって、検証器と言語サーバーはインターフェイスと index のみをメモリに保持し、証明本体・トレース・AI 向けの詳細データは、特定のクエリが要求したときにのみ遅延的に読み込みます。

> [!NOTE]
> 本節は定性的な常駐集合モデルを述べるものであり、性能保証ではありません。具体的なメモリ予算やベンチマーク指標（例: file ごと、あるいは import closure サイズごとのピーク RSS）は評価方法論の領域であり、本仕様の規範的な一部ではありません。

## 12.7 完全な構文 (EBNF)

次の EBNF は、この章で定義されているmodule レベルの構造を要約しています。

```ebnf
compilation_unit   ::= import_prelude export_prelude { annotated_declaration } ;

import_prelude     ::= { import_stmt } ;
export_prelude     ::= { export_stmt } ;

import_stmt        ::= "import" module_alias_decl { "," module_alias_decl } ";" ;
export_stmt        ::= "export" module_path { "," module_path } ";" ;

module_alias_decl  ::= module_path [ "as" module_identifier ]
                     | module_branch_import ;
module_branch_import
                   ::= module_path ".{" module_identifier { "," module_identifier } "}" ;
module_path        ::= [ relative_prefix ] module_identifier { "." module_identifier } ;
relative_prefix    ::= "." | ".." ;
module_identifier  ::= identifier ;

(* Declaration Rules *)
annotated_declaration ::= { annotation } declaration
                        | standalone_diagnostic_annotation ;

declaration        ::= definition_block         (* Visibility is handled inside *)
                     | reserve_decl
                     | claim_block
                     | [ visibility ] theorem_item
                     | [ visibility ] notation_decl
                     | registration_block ;

visibility         ::= "private" | "public" ;

(* Definition Block Structure *)
definition_block   ::= "definition" { definition_content } "end" ";" ;

definition_content ::= { annotation }
                       ( definition_parameter_decl (* ordinary and template let ... *)
                       | assumption               (* assume ... *)
                       | correctness_condition
                       | property_item
                       | [ visibility ] definitional_item
                       | [ visibility ] theorem_item
                       | [ visibility ] registration_item ) ;

definition_parameter_decl ::= "let" definition_parameter_binding
                              [ definition_parameter_constraint ] ";" ;
definition_parameter_binding ::= definition_qualified_vars
                               | pred_param
                               | func_param ;
definition_qualified_vars ::= definition_explicit_vars [ "," definition_implicit_vars ]
                            | definition_implicit_vars ;
definition_explicit_vars ::= definition_qualified_segment
                             { "," definition_qualified_segment } ;
definition_qualified_segment ::= variable_list parameter_qualification ;
definition_implicit_vars ::= variable_list ;
parameter_qualification ::= ( "being" | "be" ) let_type ;
definition_parameter_constraint ::= "such" conditions
                                  | "such" "that" formula
                                    ( "by" references | proof )
                                  | "by" references ;

definitional_item  ::= func_def | pred_def | mode_def | attr_def | struct_def | algorithm_def ;

(* Note: Specific syntax for func_def, theorem_item,
   correctness_condition, registration_item, etc. is defined in subsequent chapters.
   Template definitions are parsed as ordinary definition_block declarations;
   template-specific constraints are checked semantically. *)

(* Namespace-qualified reference to a user-defined symbol (functor, mode,
   struct, attribute, predicate). The leading dotted path is resolved as
   a module path per §2.5.3 rule 4; the final token is a user_symbol
   (§2.5.2) that must be present in the active lexicon. A namespace
   segment that is also in scope as a variable is shadowed. *)
qualified_symbol   ::= { namespace_segment "." } user_symbol ;
namespace_segment  ::= identifier ;
```

## 12.8 例

### 12.8.1 標準librarymodule

**file**: `std/algebra/abelian_group.miz`

```mizar
:: Defines module: std.algebra.abelian_group

:: 1. Imports
import std.algebra.group as Grp;
import std.arithmetics.integers;

:: 2. Exports
export Grp;

:: 3. Private Lemma
private theorem CommutativityLemma:
  for a, b being Element of Grp.Group holds ...;

:: 4. Public Definition (Attribute)
definition
  let G be Grp.Group;
  attr AbelianDef: G is abelian means
    for a, b being Element of G holds a * b = b * a;
end;

:: 5. Public Theorem
theorem AbelianProperties:
  for G being Grp.Group st G is abelian holds ...
  by CommutativityLemma;
```

### 12.8.2 サードパーティパッケージmodule

**file**: `pkg/mathcomp_mizar/algebra/ring_extensions.miz`

```mizar
:: Defines module: pkg.mathcomp_mizar.algebra.ring_extensions

:: Imports from standard library and JFM
import std.algebra.ring;
import pub.jfm.trybulec.polynomial_ring;

:: Imports from same package
import .ring_morphisms;

definition
  let R, S be Ring;
  mode RingExtensionDef: RingExtension of R, S is ...;
end;
```

### 12.8.3 開発module

**file**: `dev/my_research/experimental_lattice.miz`

```mizar
:: Defines module: dev.my_research.experimental_lattice

import std.algebra.lattice;
import pkg.mathcomp_mizar.algebra.ring_extensions;

:: Experimental definitions, not yet ready for publication
private func experimental_closure(L) -> Lattice means ...;

theorem ExperimentalResult:
  ...;
```
