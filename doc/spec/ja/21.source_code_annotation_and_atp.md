この章では、Mizar Evolution のアノテーション システムと自動定理証明器 (ATP) の統合について説明します。注釈は、定義や定理の中核となる意味論を変更することなく、型推論、overload解決、証明検索をガイドするsource レベルのマーカーです。 ATP 統合レイヤーは、検証条件を標準の問題形式に変換し、外部証明器に送信し、結果として得られた証明証明書を独立した検証のために最小kernelに送信します。

設計は次の 3 つの原則に従っています。

- **意味的中立性**: すべての注釈は無視できます。注釈のないsource fileは論理的に有効なままです。検証器は時間がかかるか、より多くの診断メッセージを発行する可能性がありますが、正しいコードを拒否しません。
- **証明書の中立性**: 注釈は証明証明書には表示されず、コード抽出中に削除されます (§20.10)。一部の注釈 (`@show_*`、`@eval`) は、検証時に診断出力を生成します。この出力はアドバイスであり、証明された内容には影響しません。
- **階層化された統合**: 人間が作成したヒント、検証器が解決したメタデータ、および診断安定性コードは、それぞれ独自の出力チャネルを持つ 3 つの異なる懸念事項として扱われます。

ドキュメントコメント (名前付き項目の前の `::` 行) とその構造化タグ (`@param`、`@returns` など) は §24.1 で指定されています。

## 21.1 設計原則とアーキテクチャ

### 21.1.1 3 層 AI/IDE 統合モデル

AI および IDE toolには、Mizar source言語が意図的に暗黙的に残しているいくつかの種類の情報 (推論された型、解決されたoverload、`qua` 強制、クラスター属性の伝播、およびオープンな検証条件) が必要です。この情報は、3 つの相補的なレイヤーを通じて利用可能になります。

**レイヤー 1 — 手書きのヒント。** 作成者は、`@` という接頭辞が付いた注釈をsource fileに直接配置します。これらは、§21.4–§21.6 で指定されている注釈です。これらは検証器の検索スペースを削減し、AI toolに人間が作成した安定した意図の概要を提供します。これらは意味的に中立であるため、正確さに影響を与えることなく自由に省略、追加、または削除できます。

**レイヤー 2 — 解決されたメタデータ出力。** 検証器と型チェッカーは、内部で計算されたすべての情報 (推論された型、解決されたfunctor完全修飾名 (FQN)、挿入された `qua` 強制、アクティブな証明義務) を記録し、ビルド出力と一緒に構造化されたアーティファクトとして出力します。これは、IDE ホバーおよびコードアクション データのプライマリ チャネルです。アーティファクトの形式については、§23.5 で説明されています。

**レイヤー 3 — 診断安定性コード。** すべての診断メッセージには、自然言語の説明に加えて機械可読コードが含まれています。 AI toolはメッセージ テキストではなくコードを消費します。これにより、検証器のバージョン間でメッセージの文言が変更された場合の脆弱性が防止されます。コード語彙は §22.7 で定義されています。

### 21.1.2 フェーズベースの導入計画

すべてのアノテーションを同時に導入する必要はありません。推奨される実装順序は次のとおりです。

**基礎 (全フェーズ):**
- `@[...]` library アノテーション構文 — §21.2
- `@latex` — シンボルのドキュメントレンダリング。 §21.3

**フェーズ 1 (コア証明ガイダンス):**
- `@proof_hint` — ステップごとのソルバーと公理バジェット制御
- 診断コードの安定化 (§22.7)

**フェーズ 2 (ヒントの表示、§21.5):** フェーズ 1 以降で利用可能であり、フェーズ 2 機能の導入に応じて拡張されます。
- `@show_thesis` — 診断として電流証明義務を発行する
- `@show_resolution` — 次の式のoverload解決結果をレポートします。
- `@show_type` — 式の推論された型をレポートする
- 解決されたメタデータ JSON アーティファクト出力 (§23.5.1)

**フェーズ 3 (アルゴリズムと ATP):**
- `@eval` — 検証時の式の評価
- ATP 説明 API と LSP の統合 (§23.6)

### 21.1.3 非推奨のアノテーション

次のパターンは、言語の進化に悪影響を及ぼしたり、仕様と実装を混同したりする危険があるため、source アノテーションとして提供されません**。

**overload ロック** 特定のoverload解決を無条件に強制するアノテーションにより、sourceがlibrary の再編成の影響を受けやすくなります。曖昧さの解消は、修飾 `qua` および `@resolve` ヒント (§21.6.1) によって処理されますが、これは依然として勧告です。

**クラスターの拡張順序。** クラスターの登録が伝播する順序は、内部検証器の関心事項です。これをsource内で修正すると、登録ルールが改良されたときに破損する脆弱なコードが作成されます。

**実行戦略を選択します。** `the Element of S` 構造 (§20.3) は論理的に任意です。 `smallest` や `first` などのsource レベルのアノテーションを付加すると、数学的仕様と実装ポリシーが混同されます。このようなポリシーは、sourceではなく MVM 構成に属します。

**AI 固有の意味論。** 検証器をガイドするのではなく、AI toolの構造の意味を変更することを唯一の目的とするアノテーションは、明示的に範囲外です。 AI に対応する情報は、消去可能なメタデータのままでなければなりません。

## 21.2 library の注釈

library アノテーションは、証明探索エンジンで使用するための安定した属性を定義に付加します。これらは、注釈を付ける項目の直前に `@[label]` として書かれ、定義、定理、および登録に表示される場合があります。

```ebnf
library_annotation ::= "@[" label_list "]" ;
label_list         ::= label_name { "," label_name } ;
label_name         ::= label_identifier [ "(" annotation_args ")" ] ;
annotation_args    ::= annotation_arg { "," annotation_arg } ;
annotation_arg     ::= identifier | nat_literal | string_literal ;
```

追加のラベルはユーザー libraryによって登録できますが、組み込みの検証器によって無視されます。

## 21.3 LaTeX 式の注釈

LaTeX 式の注釈は、ドキュメント出力および IDE 表示用に、シンボルの推奨される数学的レンダリングを提供します。型チェックや校正検索には影響しません。

```ebnf
latex_annotation ::= "@latex" "(" latex_string ")" ;
latex_string     ::= string_literal ;
```

注釈は、`func`、`pred`、`mode`、または `attr` 定義の直前に配置されます。

```mizar
@latex("n!")
func factorial(n) -> Nat;

@latex("\gcd(a, b)")
func Gcd(a, b) -> Nat;

@latex("a \leq b")
pred leq(a, b);
```

ドキュメント ジェネレーター (§24.3) は、レンダリングされた出力内でシンボルが出現する場所を LaTeX 文字列に置き換えます。 `@latex` 注釈が存在しない場合は、シンボル名がそのまま使用されます。

## 21.4 推論の注釈

証拠検索は完全に ATP backendに委任されます (§21.7)。 ATP は、`by` 節で明示的に引用されたローカル仮説と補題のみを受け取ります。グローバル libraryからの自動前提選択はありません。著者は必要な補題を引用する責任があります。 `@proof_hint` は、ソルバーの選択とリソース制限を制御します (§21.4.1)。

### 21.4.1 `@proof_hint` (ソルバー、公理バジェット、およびtimeout)

`@proof_hint` は、直後の `by` ステップ、`thus` ステートメント、または `proof ... end` ブロックの ATP ソルバーの選択とリソース制限を制御します。補題の選択は、`by` 引用 (グループ化された一括形式を含む、§16.5) によって処理されます。

```ebnf
proof_hint_annotation ::= "@proof_hint" "(" proof_hint_options ")" ;
proof_hint_options    ::= proof_hint_option { "," proof_hint_option } ;
proof_hint_option     ::= "max_axioms" ":" nat_literal
                        | "timeout"    ":" nat_literal
                        | "solver"     ":" solver_name ;
solver_name           ::= "vampire" | "e" | "cvc5" | "z3" | "auto" ;
```

```mizar
@proof_hint(max_axioms: 32, solver: vampire)
thus result = Eval(t, v) by poly_eval_add, poly_eval_mul;

@proof_hint(timeout: 60, solver: cvc5)
thus complexity_bound(n) by complexity_lemma;
```

**`max_axioms`** は、ATP に送信される公理の総数を制限します。 `by` の引用とローカル仮説から導出された公理セットがこの制限を超えると、関連性の低い項目は削除されます。

**`timeout`** は、ステップごとの ATP 時間制限を秒単位で設定します。このステップのみ、`mizar.pkg` (§23.1) のグローバル `atp_timeout` 設定をオーバーライドします。拡張検索が必要であることがわかっているステップ、または逆に、簡単であると予想されるステップで高速に失敗することがわかっているステップに役立ちます。

**`solver`** はbackend 証明器を選択します (§21.4.2)。 `auto` に設定すると、検証器はデフォルトの選択ヒューリスティックを適用します。

### 21.4.2 ソルバーの選択とポートフォリオの実行

検証器は、**ポートフォリオ モデル**内の 1 つ以上のbackendに各証明義務をディスパッチします。`solver: auto` が有効な場合、複数のbackendが並行して起動され、最初にkernelで受け入れられた有効な証明書を返したものが勝ちとなります。残りのbackend プロセスは終了されます。これにより、作成者がどちらが成功するかを予測する必要がなく、ATP と SMT backendの補完的な長所が活用されます。

**クラスター属性の解決**は、証明探索の前に個別に処理されます。クラスター構文では結合的な先行詞と後件のみが許可されるため (§17.10)、すべての登録は Horn 節になります。複数の後件を持つルールは、ロード時に単一結果ルールに正規化されます。次に、検証器は、クラスター データベース グラフ上の統合された **ルールごとの先行カウンタを備えた BFS** によって結果のルールを解決します (§23.7.4、§17.7.1)。解決中に外部ソルバーは呼び出されません。導出ステップは、`resolution-trace/` (§23.7.7) にルール適用の順序付けされたシーケンスとして記録され、線形時間トレース再生 (§23.7.7) を介して最小kernelによって検証されます。

解決された属性セットは、後続の証明義務で基礎事実として利用できるため、ATP backendは証明検索時にクラスター公理を連鎖する必要がありません。

**デフォルトのポートフォリオ** (`solver: auto`の場合):

|backend | | のときに発売されました。強み |
|---|---|---|
| **CVC5** |常に |線形/非線形算術演算、代数データ型、定量化された目標のための MBQI |
| **ヴァンパイア** |常に |深い FOL 推論、大規模な公理セット、AC 演算子 |
| **E** |常に |等式推論、AC 完全性 |
| **Z3** |常に |ビットベクトル、非線形算術、量指定子のないフラグメント |

実際には、算術負荷の高い目標 (ループ不変条件、終了手段) は通常、最初に CVC5 または Z3 によって閉じられます。深い補題連鎖を必要とする目標は、最初に Vampire または E によって閉じられます。

ポートフォリオは `@proof_hint(solver: ...)` で制限でき、そのステップの他のすべてのbackendが無効になります。これは、特定のbackendが目標クラスに対してkernel検証可能な証明書を生成することがわかっている場合、または証明書形式の再現性が必要な場合に便利です。

このポリシーは、パッケージ マニフェスト (§23.1) でグローバルに、また `@proof_hint(solver: ...)` を使用してローカルでオーバーライドできます。個々の `solver` オーバーライドは、グローバル設定よりも優先されます。

## 21.5 AI/IDE 表示のヒント

このセクションの注釈は、検証器または LSP サーバーに、特定のsource の場所の内部状態を明らかにするように指示します。これらは、インタラクティブな証明開発、デバッグ、AI ガイド付き編集を目的としています。最終的な検証結果には影響しません。

### 21.5.1 `@show_thesis`

`@show_thesis` は、検証器が現在の **証明義務** (論文) を注釈付きの位置で診断として出力することを要求します。これは、§16.9 で説明されている論文追跡メカニズムに対応しており、source レベルでアクセス可能です。

```mizar
proof
  @show_thesis
  thus n + 0 = n by add_0_right;
end;
```

発行された診断にはコード `info.thesis` があり、その時点で証明する必要がある式が含まれています。これは (警告やエラーではなく) 情報注釈として IDE に表示され、ステップを放電できない場合は `@missing_facts` 出力 (§23.5.2) に含まれます。

### 21.5.2 `@show_resolution`

`@show_resolution` は、検証器が直後の式のoverload解決結果を出力することを要求します。診断には、解決されたfunctorまたは述語 FQN、置換されたtemplate引数、挿入された `qua` 強制、およびその箇所で統合された有効な再定義がリストされます。

```ebnf
show_resolution_annotation ::= "@show_resolution" ;
```

```mizar
@show_resolution
Product[R](s);
```

診断出力の例 (コード `info.resolution`):

```
resolved functor : algebra.monoid.Product
template args    : R qua MulMagma
result type      : Element of R
inserted coercion: s qua MulMagma
refinements      : none
```

この情報は、LSP ホバー (§23.6.1) を介して常に入手できます。 `@show_resolution` を使用すると、IDE を実行していないユーザーのプレーンテキスト検証出力にそれが表示されます。解決が予期しない場合は、修飾名または `qua` 強制を使用してsource内で曖昧さを解消する必要があります。

### 21.5.3 `@show_type`

`@show_type` は、検証器が特定の式の推論された型をバインディングに割り当てずに出力することを要求します。

```ebnf
show_type_annotation ::= "@show_type" "(" term_expression ")" ;
```

```mizar
@show_type(total + x)
```

診断 (コード `info.type`) は、現在のスコープ内の指定された式の推論された型を報告します。これは軽量のデバッグ toolです。インタラクティブな開発では、通常、LSP ホバー (§23.6.1) で十分です。

module レベルでは、`@show_type(expr)` は standalone diagnostic declaration です。後続の item には付着せず、論理内容にも寄与しません。

## 21.6 検証時の式の評価

### 21.6.1 `@eval`

`@eval` は、検証器がスコープ内のアルゴリズムの計算内容を使用して指定された式を実行し、結果を診断として出力することを要求します。 `@show_*` 表示アノテーション (§21.7) とは異なり、`@eval` はコードをアクティブに実行します。したがって、コード抽出中に削除されますが (§20.10)、検証時には実行されます。

```ebnf
eval_annotation ::= "@eval" "(" term_expression ")" ;
standalone_diagnostic_annotation ::= show_type_annotation | eval_annotation ;
```

`@show_type(...)` と `@eval(...)` は診断形式であり、付与型 annotation ではありません。
別の宣言や文の前に現れた場合でも、独立した単独項目として扱われます。

`@eval` は、次の 3 つのコンテキストで表示される場合があります。

**module レベル。** 閉じた (接地) 項を評価し、その値と型を報告します。

```mizar
@eval(factorial(10))
:: info.eval: 3628800 : Nat

@eval(euclid_gcd(48, 18))
:: info.eval: 6 : Nat
```

module レベルでは、`@eval(expr)` は `@show_type(expr)` と同じ standalone diagnostic declaration 形式として解析されます。後続の定義、定理、登録なしで現れることができます。

**証明ブロック内。** 現在のスコープ内の式を評価します。具体的な値を持つ `let` または `assume` によってバインドされた自由変数が置き換えられます。具体的なバインディングのない普遍的に定量化された変数は、型のみの結果を生成します。

```mizar
proof
  let n be Nat;
  @eval(n + 0)
  :: info.eval: n + 0 : Nat  (no concrete value; n is universally quantified)
  thus n + 0 = n by add_0_right;
end;
```

**アルゴリズム本体の内部。** スコープ内のすべての変数の現在値を使用して、そのプログラム ポイントで式を評価します。これは主な使用例です。検証条件を生成せずにアルゴリズムの動作を調査するためのノンブロッキング ウォッチ式として機能します。

```mizar
terminating algorithm euclid_gcd(a, b) -> Nat
  ...
do
  while b <> 0 do
    @eval(a)   :: displays current value of a at this loop point
    @eval(b)
    const r := a mod b;
    a := b;
    b := r;
  end;
  return a;
end;
```

診断コードは`info.eval`です。 IDE では、`@eval` の結果は、注釈が付けられた位置にインライン ヒントとして表示されます (§23.6.1)。プレーンテキスト検証 toolの出力では、注釈サイトの直後に出力されます。

> [！注記]
> `@eval` では、式が計算可能な型 (登録された `algorithm` が演算を定義する型) である必要があります。計算不可能な数学的オブジェクト (具体的なインスタンス化を持たない任意の `Element of R` など) を含む式では、`W0003` クラスの警告が生成され、値は出力されません。

## 21.7 ATP の統合

### 21.7.1 最小kernelアーキテクチャとde Bruijn基準

Mizar Evolution は **最小kernel** 設計を採用しています。検証器の信頼されたコアは証明証明書の検証のみを実行し、推論論理は含まれません。推論は完全に外部 ATP に委任されます。この設計は、de Bruijn の基準を満たしています。つまり、信頼されたコードベースのサイズが最小限に抑えられ、システムの正しさが外部証明器の正しさに依存しません。

```
┌──────────────────────────────────────┐
│          User Proof Source           │
│  (proof...end / by references)       │
└──────────────┬───────────────────────┘
               │
       ┌───────▼───────┐
       │  Type-checker │  ← cluster attribute resolution via SMT (CVC5/Z3)
       │               │  ← resolved attribute sets forwarded to Translator
       └───────┬───────┘
               │
       ┌───────▼───────┐
       │  Translator   │  ← encodes explicit by-citations as axioms
       │               │  ← encodes functor/predicate properties (§21.7.4)
       └───────┬───────┘
               │  TPTP / SMT-LIB (dispatched in parallel)
       ┌───────┴────┬───────────────┐
       │            │               │ 
  ┌────▼────┐  ┌────▼────┐  ┌───────▼───────┐
  │Vampire  │  │   E     │  │  CVC5 / Z3    │
  │(FOL/AC) │  │(equat.) │  │  (SMT/arith.) │
  └────┬────┘  └────┬────┘  └───────┬───────┘
       │            │               │
       └──── first valid cert ──────┘
               │
       ┌───────▼───────┐
       │  Minimum      │  ← validates certificate only
       │  Kernel       │  ← performs no inference
       └───────────────┘
```

ATP は **信頼できないオラクル** として扱われます。ATP は、kernel が独自にチェックする証明証明書を生成します。 ATP のバグまたは不健全性により、不正な証明書が生成される可能性がありますが、kernel はそれを拒否します。kernel自体のバグのみが健全性を損なう可能性があります。

> [！重要]
> kernel は ATP の結果を無条件に信頼することはありません。 ATP が証明したと主張するすべての証明義務には、ATP を再度呼び出すことなくkernel がチェックできる証明書が添付されている必要があります。 ATP timeoutまたは証明書の拒否は、受け入れられた証明としてではなく、オープンな証明義務として報告されます。

### 21.7.2 問題の変換pipeline (トランスレーター)

Translator は、Mizar の証明義務を 一階または SMT 問題に変換します。pipelineには 4 つのステージがあります。

1. **スコープの収集。** Translator は、義務サイトのスコープ内のすべての仮説、`let` 束縛変数、および登録されたプロパティを収集します。 `@proof_hint` 制約 (§21.4.1) を適用して、公理セットを制限します。

2. **プロパティのエンコーディング。** functorおよび述語の登録されたプロパティ (§10.6.1、§9.5.1) は、公理またはネイティブ宣言 (§21.7.4) に変換されます。 AC 演算子は、利用可能な場合はbackend ネイティブの AC メカニズムを使用して宣言されます。

3. **目標の翻訳。** 証明義務自体は、否定された推測 (反駁ベースの ATP の場合) または満たされるべき主張 (SMT ソルバーの場合) として翻訳されます。

4. **公理セットのアセンブリ。** ATP に送信される公理セットは次のもので構成されます。(a) スコープ内のローカル仮説と `let` 限定等式。 (b) グループ化された形式 (`module.{a,b}`) および一括形式 (`module.*`) を含む、`by` 節 (§16.5) で明示的に引用された補題。 (c) `invariant` および `decreasing` の正当化 (§20.2.2)。グローバル libraryからの前提条件の自動選択は実行されません。作成者は必要な補題を引用する責任があります。 `max_axioms` 制限 (デフォルト: 128、`@proof_hint` によってオーバーライド可能) は、合計公理数を制限します。一括引用は、上限が適用される前に解決されます。

出力は、選択したbackendで必要な形式の自己完結型の問題fileです。

### 21.7.3 backend 証明器 (Vampire、E、CVC5、Z3)

|証明器 |種類 |入力形式 |強み |証明出力 |
|---|---|---|---|---|
| **ヴァンパイア** | ATP | TPTP FOF/TFF |深い FOL 推論、AC 演算子、大規模な公理セット | TSTP の導出 |
| **E** | ATP | TPTPFOF |等式推論、AC 完全性 | TSTP の導出 |
| **CVC5** | SMT | SMT-LIB2 |算術、代数データ型、定量化された目標の MBQI、クラスター解決 | LFSC / Alethe 証明書 |
| **Z3** | SMT | SMT-LIB2 |非線形算術演算、ビットベクトル、量指定子のないフラグメント、クラスター解決 | SMT-LIB 証明 (オプション) |

4 つのbackendはすべて、`solver: auto` で並行して起動されます。kernel が受け入れた証明書を最初に返したものが勝ちとなります (§21.4.2)。 CVC5 および Z3 は、証明探索が開始される前のタイプチェック時にクラスター属性を解決するための SMT backendとしても機能します。

### 21.7.4 ATPのプロパティエンコーディング

functorと述語の登録されたプロパティ (§10.6.1、§9.5.1) は、各backendに送信される問題file内の公理としてエンコードされます。エンコードは簡単です。各プロパティは、普遍的に定量化された 一階式 (Vampire/E の場合は TPTP) または普遍的に定量化された SMT アサーション (CVC5/Z3 の場合は SMT-LIB) になります。

**可換性と結合性**は特別な扱いを受けます。それらを単純な公理としてエンコードすると、検索空間で組み合わせ爆発が発生します。ターゲット backendがネイティブ AC 推論をサポートしている場合、トランスレーターは代わりにそのメカニズムを使用します。

- **ヴァンパイア**: `--ac_reasoning on`;項は内部的に AC 正規形に正規化されます。
- **E**: `--term-ordering` による AC の完全性。項は AC で正規化されます。
- **CVC5/Z3**: 定量化されたアサーションとしてエンコードされます。 SMT 合同クロージャがそれらを処理します。

他のすべてのプロパティ (冪等性、包含性、対称性、再帰性、非再帰性など) は単純な公理としてエンコードされ、特別な処理なしでbackendに渡されます。現在の証明の目標に直接関係のないプロパティは、引用された `by` 節またはローカル コンテキストが注釈を付けたfunctorまたは述語を参照する場合、公理セットに引き続き含まれます。それ以外の場合は省略されます。

### 21.7.5 証明証明書の形式 (TSTP、LFSC/Alethe)

最小kernel は、次の形式の証明証明書を受け入れます。

**TSTP** (Vampire と E によって使用されます)。 TSTP 導出は注釈付きの一連の式であり、それぞれに使用される推論ルールと親式のインデックスがラベル付けされています。kernel は各推論ステップを独立して再構築し、最終的な式が元の目標の否定であることを確認します。

**LFSC/Alethe** (CVC5 で使用)。 LFSC 証明書は、エディンバラ論理フレームワークの入力された証明用語です。 Alethe 証明書は、一連の SMT 証明ステップです。kernelには、用語の構造とすべての入力判断を検証する各フォーマット専用のチェッカーがあります。

**証明ログ** (フォールバック)。証明器が構造化証明書 (例: 一部の構成の Z3) を発行できない場合、ATP インタラクションは `*.atp-log` fileとして記録されます (§23.5.3)。このモードでは、証明はkernel検証ではなく **外部証明** としてマークされます。外部で証明された証明は開発には受け入れられますが、ビルド出力でフラグが立てられ、リリース ビルドのkernel検証要件を満たしていません。

> [！重要]
> kernel検証された証明と外部で証明された証明の区別は、`*.mizir.json` アーティファクト (§23.5.1) で `proof_status` フィールド (`"kernel_verified"` 対 `"externally_attested"`) として明らかにされます。外部で証明された証明を含むmoduleは、明示的な作成者の承認なしにパッケージ レジストリに公開できません。

### 21.7.6 型チェックレベルでの Sethood 解決

`sethood` プロパティ (§5.8.1) は、証明義務が生成される前、および ATP が呼び出される前に、型チェック時に解決されます。これにより、Fraenkel 集合内包表記 `{ f(x) where x is T : P(x) }` が、登録された sethood Witness を持たない型 `T` とともに使用されるときに発生するクラスの ATP timeoutが回避されます。

ルールは次のとおりです。

- `{ f(x) where x is T : P(x) }` が用語に出現すると、型チェッカーは `sethood` が `T` に登録されているかどうかを検索します。
- 登録が見つかった場合、型チェッカーは、対応する存在事実を、それを囲む証明義務の背景公理として挿入します。
- 登録が見つからない場合、型チェッカーはコード `type.sethood.missing` の **コンパイル時エラー**を発行し、その用語を拒否します。エラーは ATP に延期されません。

この設計により、sethood 推論は完全に型チェッカーのスコープ内に保持され、推論エンジンではなく証明書検証器としてのkernelの役割と一致します。
