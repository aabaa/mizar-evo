ドキュメント ジェネレーター (`mizar doc`) は、source fileから構造化ドキュメントを抽出し、参照可能な相互参照された HTML 出力を生成します。ドキュメントのsourceは Mizar source自体です: 項目ラベル、型署名、定理ステートメント、および `:::` ドキュメント コメント (§2.9.1)。別のドキュメント言語は必要ありません。

ジェネレーターは、`*.mizir.json` アーティファクト (§23.5.1) から `exports` 配列を読み取り、どのアイテムが公開されるかを決定し、`@latex` 注釈 (§21.3) と同じアーティファクトからの解決された型を使用して数学的表記をレンダリングします。

## 24.1 ドキュメントのコメント構文

**ドキュメント コメント** は、名前付き項目 (定義、定理、登録、またはアルゴリズム) の直前にある `:::` 行コメントの連続ブロックです。コメント ブロックとその項目の間に空白行を入れてはなりません。空白行で関連付けが終了します。

```mizar
::: Computes the greatest common divisor of two positive integers
::: using the Euclidean algorithm.
:::
::: The result satisfies `result divides a`, `result divides b`, and
::: for any `d` dividing both `a` and `b`, `d divides result`.
terminating algorithm euclid_gcd(a, b) -> Nat
  requires a > 0 & b > 0
  ensures result = Gcd(a, b)
  ...
```

ドキュメント コメントの本文は、次の Mizar 固有の拡張子を備えた **Markdown** (CommonMark 仕様) として解釈されます。

- バッククォートで囲まれたインライン コード (`` `code` ``) は、Mizar 構文の強調表示でレンダリングされます。内容が正確なラベルまたは FQN 参照である場合にのみリンクされます (§24.2)。
- `$$` で区切られた表示数学は数式としてレンダリングされます (§24.3)。
- `[文章](ref)` 形式の相互参照リンクは、§24.2 で説明されているように解決されます。

**ブロック構造。** ドキュメント コメントには、空白の `:::` 行で区切られた複数の段落が含まれる場合があります。リスト、コード ブロック、およびブロック引用符の警告は、`:::` 行内の標準の Markdown 構文を使用してサポートされます。

```mizar
::: Computes `n!` (n factorial) for a natural number `n`.
:::
::: ## Examples
:::
::: ```mizar
::: factorial(0) = 1
::: factorial(5) = 120
::: ```
:::
::: ## Mathematical definition
:::
::: $$n! = \prod_{k=1}^{n} k$$
:::
::: The algorithm terminates because `n` decreases at each recursive call.
terminating algorithm factorial(n) -> Nat
  ensures result = Factorial(n)
  decreasing n
  ...
```

**構造化タグ** 次のタグは、ドキュメント コメント内の段落の最初のtokenとして表示される場合があります。

|タグ |使い方 |
|---|---|
| `@param name` | `name` という名前のパラメータについて説明します。パラメータごとに 1 つのタグ。 |
| `@returns` |戻り値を文書化します。識別子 `result` は戻り値を参照します。 |
| `@requires` |前提条件の散文説明 (形式的な `requires` 句を補足します)。 |
| `@ensures` |事後条件の散文的な説明 (形式的な `ensures` 句を補足します)。 |
| `@see ref` | 「関連項目」相互参照リンク (§24.2) を追加します。 |
| `@since version` |アイテムが導入されたバージョンを記録します。 |
| `@deprecated version` | `version` 以降、項目を非推奨としてマークします。非推奨メッセージが続きます。 |

```mizar
::: Divides `a` by `b` and returns the quotient.
:::
::: @param a  the dividend
::: @param b  the divisor; must be non-zero
::: @returns  the real number `a / b`
::: @requires `b <> 0`
::: @see mml.real.Real_div_mul
definition
  let a, b be Real;

  algorithm divide(a, b) -> Real
    requires b <> 0
    ensures result * b = a
  do
    return a / b;
  end;
end;
```

**moduleレベルのドキュメント。** source fileの先頭、`import` または定義ステートメントの前にあるドキュメント コメントは、module全体をドキュメント化します。これは、生成された出力のmoduleのインデックス ページに表示されます。

## 24.2 相互参照とハイパーリンク

ドキュメント ジェネレーターは、パッケージ全体とその依存関係にわたるラベル付き項目間の相互参照を解決します。参照は式ベースではなくラベルベースであるため、アイテムにoverloadされた表記法や複数の表面構文がある場合でも、ドキュメントのリンクは安定しています。

**インライン コード リンク。** バックティック スパンの内容が正確な項目ラベルまたは FQN である場合、ジェネレーターはその項目のドキュメント ページにリンクします。

```
::: The result equals `Gcd` as defined in `mml.nat.Gcd`.
```

ジェネレーターはまず、現在のmoduleのエクスポートされたラベル内の未修飾ラベルの解決を試みます。見つからない場合は、インポートされたmoduleをインポート順に検索します。あいまいなラベルはビルド時の警告 (`W0002` クラス) を発行し、リンクなしでレンダリングされます。式、定理の断片、またはoverloadされた曲面表記であるバックティック スパンは、リンク解決なしでコードとしてレンダリングされます。リンクが必要な場合、作成者は明示的なラベルまたは FQN を使用する必要があります。

**明示的なリンク。** 標準の Markdown リンク構文 `[表示テキスト](ref)` は、`ref` の 3 つの形式を受け入れます。

|フォーム |例 | | に解決されます。
|---|---|---|
|ラベルまたは FQN | `(Gcd)`、`(mml.nat.Gcd)` |ラベル付きアイテムのドキュメント ページ |
|セクション参照 | `(§20.5.1)` |生成された仕様内のセクションアンカー |
| URL | `(https://...)` |外部 URL (変更せずにパススルー) |

```mizar
::: This algorithm implements the approach described in [§20.5.1](§20.5.1).
::: The result type is documented under [Element of R](algebra.struct.Element).
```

**自動逆参照** ジェネレーターは、定理と証明内のすべての `by` 正当化節にインデックスを付けます。定理 `T` ごとに、生成されたページには、`by` 句で `T` を引用するすべての定理をリストする「使用者」セクションが含まれます。このセクションは、`*.mizir.json` (§23.5.1) の `exports` 配列から派生したものであり、注釈は必要ありません。

**`@see` リンク。** ドキュメント コメント内の `@see ref` タグは、アイテムのページの下部にある「関連項目」リストにエントリを追加します。 `ref` は、明示的リンクと同じ 3 つの形式に従います。

**アンカーの名前付け。** 文書化されたすべての項目は、レンダリングされた表記法やoverloadの解決からではなく、エクスポートされたラベルから派生した安定したアンカーを受け取ります。 FQN が `algebra.ring.AddAssoc` である項目のアンカーは `#algebra.ring.AddAssoc` です。そのmodule ページ内では、ラベルがページ上で一意である場合、`#AddAssoc` としてアドレス指定されることもあります。セクション見出しは、`#sec-20-5-1` 形式のアンカーを受け取ります。

ラベルはパブリック ドキュメント インターフェイスの一部です。定義には定義ラベルが使用されます。定理と登録については、定理または登録ラベルが使用されます。アルゴリズムの場合、アルゴリズム識別子はドキュメントのラベルです。将来の構文で別のアルゴリズム ラベルが導入された場合、ドキュメント ジェネレーターは代わりにそのラベルを使用します。

## 24.3 数式のレンダリング

生成されたドキュメントの数学的表記は、シンボル定義の `@latex` 注釈 (§21.3) とドキュメントのコメントに埋め込まれた LaTeX 式の 2 つのsourceから得られます。

**`@latex` 注釈。** functor、述語、モード、または属性定義に `@latex` 注釈が付けられている場合、ジェネレーターは、定理ステートメント、型署名、および数式表示など、レンダリングされた出力にシンボルが表示される場所であればどこでも、指定された LaTeX 文字列を置き換えます。 `@latex` 注釈が存在しない場合、シンボル名は等幅書体で表示されます。

```mizar
@latex("n!")
func Factorial(n) -> Nat;

@latex("\gcd(a, b)")
func Gcd(a, b) -> Nat;
```

ステートメントに両方の記号が含まれる定理は次のように表現されます。

> **定理** `gcd_factorial_coprime`: すべての $n \geq 2$ に対して、$\gcd(n!, n+1) = 1$。

ドキュメント コメント内の **インライン数式** では、標準の LaTeX 区切り文字が使用されます。

- インライン: `$formula$` — 周囲のテキストとともにインラインでレンダリングされます。
- 表示: `$$formula$$` を単独の行に表示 — 中央に配置された表示方程式としてレンダリングされます。

ジェネレーターはビルド中に LaTeX 構文を検証しません。無効な LaTeX はレンダリング エンジンに渡され、出力でレンダリング エラーが発生する可能性があります。

**式の自動生成。** `@latex` 注釈のない項目の場合、ジェネレーターは、次のルールを使用して Mizar 曲面構文を LaTeX に変換することにより、デフォルトの数学的レンダリングを生成します。

| Mizar の構文 |デフォルトの LaTeX 出力 |
|---|---|
| `a + b` | `a + b` |
| `a * b` | `a \cdot b` |
| `a / b` | `\frac{a}{b}` |
| `a |^ n` | `a^{n}` |
| `a mod b` | `a \bmod b` |
| `a <= b` | `a \leq b` |
| `a <> b` | `a \neq b` |
| `a in S` | `a \in S` |
| `S c= T` | `S \subseteq T` |
| `for x being T holds P` | `\forall x \in T,\; P` |
| `ex x being T st P` | `\exists x \in T,\; P` |

自動翻訳は、`@latex` 注釈がオーバーライドしない場合にのみ適用されます。作成者は、非標準またはドメイン固有の表記法に対して `@latex` アノテーションを提供することをお勧めします。

**式の番号付け。** 定理ステートメント内の表示式には、HTML 出力で自動的に番号が付けられます。番号付けは `(chapter.section.item)` です。`item` は、アイテムのドキュメント ページ内の表示式の連続インデックスです。式の番号はハイパーリンクされており、ドキュメントのコメントで `$$\text{see }(24.3.1)$$` 構文を使用して参照できます。

## 24.4 HTML出力

ドキュメント ジェネレーターは、パッケージ マニフェストの `doc_dir` で指定されたディレクトリ (デフォルト: `docs/`) に静的 HTML サイトを書き込みます。

HTML 出力は、Web サーバーまたは GitHub ページでのホスティングに適した静的サイトです。クライアント側の数式レンダリングには **KaTeX** を使用しており、サーバー側の処理は必要ありません。

HTML 出力のディレクトリ構造:

```
docs/
├── index.html           ← package overview and module list
├── search.json          ← full-text search index
├── algebra/
│   ├── index.html       ← module index for `algebra`
│   ├── groups/
│   │   ├── basic.html   ← documentation for `algebra.groups.basic`
│   │   └── ...
│   └── ...
└── assets/
    ├── katex/           ← KaTeX library files
    ├── style.css
    └── search.js
```

**module インデックス ページ。** 各moduleのインデックス ページには、エクスポートされたすべてのアイテムとそのタイプ シグネチャ (可能な場合は `@latex` 注釈を使用してレンダリング)、ドキュメント コメントの最初の段落から抽出された 1 行の概要、および完全なアイテム ページへのリンクがリストされます。

**アイテム ページ。** 各名前付きアイテムには、次の内容を含む専用ページがあります。
- 数学的に表現された完全な型署名または定理ステートメント。
- HTML としてレンダリングされた完全なドキュメント コメント。
- `requires` 句と `ensures` 句 (アルゴリズムと関数用) は数学的にレンダリングされます。
- パラメータ テーブルとしてフォーマットされた `@param` タグと `@returns` タグ。
- 自動的に生成される「使用者」セクション (§24.2)。
- `@see` タグの「関連項目」リスト。
- source file内の関連行への「source」リンク (`*.mizir.json` の `source_range` を使用)。

**検索。** ジェネレーターは、エクスポートされたすべてのアイテムの FQN、アイテムの種類、1 行の概要、およびタイプ シグネチャを含む `search.json` インデックスを生成します。検索 UI (`search.js`) は、backendを必要とせずに、このインデックスに対してクライアント側の全文検索を実行します。

**バージョン付きドキュメント。** パッケージ マニフェストに `version` フィールド (§23.1) が含まれる場合、ジェネレーターはバージョン文字列をページ ヘッダーと `search.json` に埋め込みます。ホスティング インフラストラクチャはこれを使用して、複数のバージョンを同時に提供する場合があります。標準レイアウトでは、バージョン固有の出力が `docs/v2.3.1/` の下に配置されます。
