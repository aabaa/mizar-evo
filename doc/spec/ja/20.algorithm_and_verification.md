Mizar Evolution は、**実行可能なアルゴリズム**を形式的な証明システムと統合します。 `algorithm` ブロックは、次のような計算を定義します。

- **検証中に Mizar 仮想マシン (MVM) によって実行** (`by computation`)。
- **Hoare 論理 コントラクト (`requires`、`ensures`、`invariant`) を使用する理由**。
- (`terminating` 経由で) 完全な正確性が確立されると、**数学functor (`func`) に昇格**します。

この設計は、**デフォルトで部分的正しさ**の原則に従っています。つまり、明示的な終了引数が指定されない限り、アルゴリズムは分岐することが許可されます。これにより、具体的な入力または制限されたコンテキストでのみ使用されるアルゴリズムに対する終了証明の要求が回避されます。

## 20.1 アルゴリズム構文

### 20.1.1 アルゴリズム定義ブロック

アルゴリズムは **`definition` ブロック内**で宣言する必要があります。周囲の `let` 宣言は型コンテキストをバインドします。アルゴリズムのパラメータ リストは、それらのバインディングのうちどれが仮引数になるかを選択します。

`definitional_item` (§12.5) として、アルゴリズムは他の定義と同じ `public` / `private` の可視性規則に従います。**public** なアルゴリズムは、その *署名と契約* — 名前、パラメータと戻り値の型、`requires`、`ensures` — を module 署名 $\Sigma$ に寄与し、実装本体は private のままです (§12.6.1)。したがって public なアルゴリズムへの本体のみの編集は下位互換な変更であり、一方その署名や契約の変更はインポート側から観測可能です。各ケースは §23.2 のパッケージバージョニング規則が分類します。

```ebnf
algorithm_def ::= [ "terminating" ] "algorithm" identifier [ "[" schema_params "]" ] "(" [ identifier_list ] ")" [ "->" type_expression ]
                  [ "requires" formula ]
                  [ "ensures"  formula ]
                  [ "decreasing" term_list ]
                  algorithm_body ";" ;

schema_params    ::= identifier { "," identifier } ;
identifier_list  ::= identifier { "," identifier } ;

algorithm_body   ::= "do" algo_statement_list "end" ;
```

`[ "->" type_expression ]` はオプションです。戻り値の型のないアルゴリズムは **void** (プロシージャ) です。オプションの `[ "decreasing" term_list ]` は、ヘッダー レベルで再帰終了措置 (§20.5.2) を宣言します。ループの終了については、§20.2.2 を参照してください。

**template algorithm。** `algorithm_def` は有効な `template_item` (§18.2.1) です。囲んでいる `definition` ブロックがスキーマ パラメーター (**type** (`let T be type`)、**pred** (`let P be pred(...)`)、または **func** (`let F be func(...) -> ...`)) を宣言する場合、これらのパラメーターはアルゴリズム本体および `requires`/`ensures` 句で参照される場合があります。アルゴリズム名には、スキーマ定理 (§18.8.4) と同じ規則に従って、スキーマ パラメーターをリストする `[...]` 括弧接尾辞が付けられます。インスタンス化は、括弧内に具体的な型、pred、または func 名を指定することによって、各呼び出しサイトで実行されます。

```mizar
definition
  let n be Nat;

  terminating algorithm factorial(n) -> Nat
    ensures result >= 1
    decreasing n
  do
    if n = 0 do
      return 1;
    end;
    return n * factorial(n - 1);
  end;
end;
```

`terminating` 修飾子 (§20.7.2) は、`requires` を満たすすべての入力に対してアルゴリズムが終了することを宣言します。これはオプションです。`terminating` のないアルゴリズムは部分的に定義されているものとして扱われ、ファーストクラスのfunctorとして使用できません。

### 20.1.2 パラメータと戻り値の型

パラメータはアルゴリズム本体内で位置指定されており、**不変**です (`const` として動作します)。パラメータの可変バージョンを使用するには、`var` コピーを作成します。

```mizar
definition
  let x be Nat;

  algorithm copy_and_modify(x) -> Nat
  do
    var y := x;
    y := y + 1;
    return y;
  end;
end;
```

戻り値の型は、すべての `return` 式の型を指定します。 `ensures` 句の特別な識別子 `result` は、戻り値を参照します。

```mizar
definition
  let n be Integer;

  algorithm abs_val(n) -> Nat
    ensures result >= 0
  do
    if n >= 0 do
      return n;
    end;
    return -n;
  end;
end;
```

### 20.1.3 変数宣言(`var`、`const`)

アルゴリズム本体内では、変数は `var` (可変) または `const` (不変) で導入されます。 **ゴースト変数** (`ghost var`、`ghost const`) は仕様レベルでのみ存在し、抽出時に消去されます。

```ebnf
var_decl         ::= "var"         var_binding   { "," var_binding   } [ "as" type_expression [ justification ] ] ";" ;
ghost_var_decl   ::= "ghost" "var" var_binding   { "," var_binding   } [ "as" type_expression [ justification ] ] ";" ;
const_decl       ::= "const"       const_binding { "," const_binding } [ "as" type_expression [ justification ] ] ";" ;
ghost_const_decl ::= "ghost" "const" const_binding { "," const_binding } [ "as" type_expression [ justification ] ] ";" ;

var_binding   ::= identifier [ ":=" term ] ;
const_binding ::= identifier ":=" term ;

justification ::= simple_justification | proof | computation_proof ;
(* simple_justification, proof, computation_proof, reasoning, references — defined in §15/§16/§20.9.2 *)
```

|キーワード |可変性 |ランタイム |使用例 |
|---|---|---|---|
| `const` |不変 |はい |中間結果、取得された値、選択 |
| `var` |可変 |はい |ループ カウンタ、アキュムレータ、状態 |
| `ghost const` |不変 |いいえ (仕様のみ) |証人、仕様値 |
| `ghost var` |可変 |いいえ (仕様のみ) |証明追跡 (訪問済みセット、抽象状態) |

**変数の種類間のキャスト ルール。** 4 つの種類は、最も具体的なものから最も抽象的なものまでの階層を形成します。より抽象的な種類が期待される場合はどこでも、より具体的な種類の値を使用できます。逆は禁止です。

|から \ まで | `const` | `ghost const` | |
|---|---|---|---|
| `var` | ✓ (現在の値、読み取り専用) | ✓ | ✓ |
| `const` | — | ✓ | — |
| `ghost var` | ✗ | ✓ (現在値) | — |
| `ghost const` | ✗ | — | — |

上向き方向 (コンクリート → ゴースト) は常に安全です。仕様コンテキストでのランタイム値の読み取りは実行に影響しません。下方向 (ゴースト→ランタイム) は常に禁止されています。ゴースト値にはランタイム表現がなく、非ゴースト変数に割り当てたり、非ゴースト呼び出しに渡したりすることはできません。

ゴースト変数は、**仕様コンテキスト**: `invariant`、`ensures`、`requires`、および `assert` でのみ表示される場合があります。これらは非ゴースト割り当ての右側に表示されない場合があり、非ゴースト アルゴリズム呼び出しに渡されない場合があります。

検証器は、ゴースト変数が実行時の動作に影響を与えないことをチェックします。エクストラクターは、すべてのゴースト宣言とゴースト割り当てをサイレントに削除します。

```mizar
definition
  let S be finite set of Integer;

  algorithm find_min(S) -> Integer
    requires S <> {}
    ensures result in S & for x being Integer st x in S holds result <= x
  do
    ghost var visited := {} as set of Integer;
    var current := S;
    var best := the Element of S;
    while current <> {} do
      invariant visited = S \ current;
      invariant best in S & for x being Integer st x in visited holds best <= x;
      decreasing card current;
      const x := the Element of current;
      if x < best do best := x; end;
      visited := visited \/ {x};       :: ghost update — erased at extraction
      current := current \ {x};
    end;
    return best;
  end;
end;
```

オプションの `as T` アノテーションは、Mizar の `reconsider x = e as T` に対応します。これは、初期化子式の型が `T` であることをアサートします。強制の拡張 (例: `Nat` から `Integer`) とoverloadされたリテラルの曖昧さの解消 (例: `0 as Real`) は型階層によって自動的に解決され、証明は必要ありません。縮小強制 (例: `Integer` から `Nat`) は証明義務を生じます。自動証明器がそれを無効にできない場合は、正当な理由 (`by` または `proof ... end`) を指定します。

```mizar
var n := some_int as Nat by non_negative_lemma;
var m := some_int as Nat
proof
  some_int >= 0 by ...;
  thus some_int is Nat;
end;
const r := 0 as Real;             :: automatic — no justification needed
```

注釈が完全に省略された場合、型は初期化子の式から推測されます ([第 8 章、型推論](./08.type_inference.md) を参照)。

**複数のバインディング**が 1 つの `var` または `const` ステートメントに出現する場合があります。末尾の `as T` 注釈は、ステートメント内のすべてのバインディングに適用されます。個々のバインディングに異なる型の注釈または正当化が必要な場合は、別のステートメントを使用します。

```mizar
var a := 1, b := 2 as Nat;          :: two vars, both typed Nat
const lo := 0, hi := n - 1 as Nat;   :: two constants, shared type
```

**初期化されていないバインディング** (`:=` なし) は、値のない変数のままになります。検証器は、代入前の使用を拒否します。ただし、`match` ステートメント (§20.2.5) のパターン変数としての使用を除き、`match` 自体が成功した場合の初期代入を提供します。

```mizar
var p, q, r as Element of PolyRing[K,x];  :: uninitialized; valid as match pattern variables
match t do
  case p * (q + r) do  :: p, q, r are assigned here
    ...
  end;
  otherwise
    ...
  end;
end;
```

### 20.1.4 割り当て

可変変数は `:=` で更新されます。

```ebnf
assignment ::= lvalue ":=" term ";" ;
lvalue     ::= identifier
             | lvalue "." identifier ;
```

右辺は任意の項式です。左側は、`var` で宣言された識別子またはそのフィールドでなければなりません。 `const` 識別子、`let` にバインドされた変数、またはパラメーターに代入すると、**コンパイル時エラー**になります。

右辺は代入前状態で評価されます。VC 生成では、代入を蓄積済み fact への
テキスト置換としてモデル化しません。代わりに、変更され得る mutable location
に対して fresh な old-state 名を導入し、target と alias しない free location
だけを含む fact を保存し、post-state equality を追加します。たとえば
`x := x + 1` は pre-state の値 `x₀` を記録して `x = x₀ + 1` を追加します。
既存 fact `x = 5` を `x + 1 = 5` へ書き換えることはありません。field update も
同じ old-state discipline を使います。`p.f := e` は `p` と `e` を pre-state で
評価してから、`p.f` の解決済み lvalue identity を更新します。

この節でいう alias は、同じ mutable storage cell に対する複数の解決済み
lvalue path を指します。通常の値に reference semantics があるという意味ではありません。
`var y := x` は `x` の値をコピーするだけで、`y` を `x` の alias にはしません。

`ghost var` 識別子への割り当てには、同じ `:=` 構文が使用されます。検証器は宣言からターゲットがゴースト変数であると判断します。 `ghost var` または `ghost const` 値を非ゴースト変数に割り当てると、**コンパイル時エラー**になります。ゴースト値には実行時表現がなく、非ゴースト状態に影響を与えてはなりません。

```mizar
counter := counter + 1;              :: runtime assignment
state.value := state.value + 1;      :: field update
S_visited := S_visited \/ {x};       :: ghost assignment — legal, target is ghost var
counter := S_visited;                :: compile-time error — ghost value on rhs of non-ghost assignment
```

## 20.2 制御フロー

### 20.2.1 条件付き(`if`)

```ebnf
if_stmt ::= "if" formula "do" algo_statement_list if_tail ;
if_tail ::= "end" ";"
          | "else" if_stmt
          | "else" algo_statement_list "end" ";" ;
```


```mizar
if n = 0 do
  return 1;
end;

if n mod 2 = 0 do
  return n div 2;
else
  return 3 * n + 1;
end;

:: if-else-if chain
if n < 0 do
  return -1;
else if n = 0 do
  return 0;
else
  return 1;
end;
```

`return` が使用される場合、両方の分岐が同じ型を返すか、両方が後続のステートメントに渡される必要があります。

### 20.2.2 有界ループ(`while`)

```ebnf
while_stmt ::= "while" formula "do"
                 { while_annotation ";" }
                 algo_statement_list
               "end" ";" ;

while_annotation ::= "invariant" formula [ justification ]
                   | "decreasing" term_list [ justification ] ;
```

`while_annotation` 句 (`invariant`、`decreasing`) は、他のステートメントの前に、`while` 本体の **先頭**に表示されます。オプションの `justification` は、句によって生成される検証条件の補題を提供します: `invariant` の確立と維持、`decreasing` の厳密な減少。

```mizar
var i := 1;
var s := 0;
while i <= n do
  invariant s = i * (i - 1) div 2 & i >= 1 by arithmetic, triangle_step;
  decreasing n - i + 1 by arithmetic;
  s := s + i;
  i := i + 1;
end;
```

### 20.2.3 範囲ループ (`for ... to`)

```ebnf
for_range_stmt ::= "for" identifier "=" expr ( "to" | "downto" ) expr
                   [ "step" expr ]
                   "do" { for_annotation ";" } algo_statement_list "end" ";" ;

for_annotation ::= "invariant" formula [ justification ] ;
```

ループ変数は本体内で不変であり、スコープはループ内にあります。境界とステップは、ループが開始される前に 1 回評価されます。 `step` 式は正の `Nat` 値である必要があります。デフォルトのステップは `1` です。

- `for i = 1 to n` — i = 1、2、…、n (昇順) を繰り返します。
- `for i = n downto 1` — i = n, n−1, …, 1 (降順) を繰り返します。
- `for i = 0 to n step 2` — i = 0、2、4、… (2 ずつ昇順) を繰り返します。
- `for i = n downto 0 step 3` — i = n, n−3, … (3 ずつ降順) を繰り返します。

```mizar
var s := 0;
for i = 1 to n do
  s := s + i;
end;

for i = 10 downto 1 do
  process(i);
end;

:: Every other element, descending
for i = n downto 1 step 2 do
  handle(i);
end;
```

終了は **自動的に保証されます** - `decreasing` 注釈は必要ありません。ループの累積効果の検証を支援するために、オプションの `invariant` 句を本文の先頭に配置できます。

```mizar
var s := 0;
for i = 1 to n do
  invariant s = i * (i - 1) div 2;
  s := s + i;
end;
```

### 20.2.4 収集ループ (`for ... in`)

```ebnf
for_collection_stmt ::= "for" identifier "in" expr [ "processed" identifier ] "do"
                          { for_annotation ";" }
                          algo_statement_list
                        "end" ";" ;
```

**有限セット**のすべての要素を反復します。検証器では、コレクション式の型が `finite set of T` (または `finite` として登録されている型) である必要があります。多重度のあるコレクション (バッグ、シーケンス) はサポートされていません。代わりに、明示的に順序付けされた構造上で `for ... to` を使用してください。

**反復順序は実装によって定義されます。** ループは、要素がアクセスされる順序について保証しません。

**`processed` バインディング。** オプションの `processed V` 句が存在する場合、`V` は `set of T` 型の **ループ スコープのゴースト バインディング**であり、ループによって自動的に管理されます。

- 最初の反復の前: `V = {}`
- 要素 `x` の各反復後: ループは `V` を `{x}` で暗黙的に拡張します。
- 通常終了時: `V = S` が論理コンテキストに自動的に追加されます。
- `V` は抽出時に消去されます (実行時表現はありません)
- `processed` が省略された場合、追跡バインディングは作成されません

ユーザーは `V` に自由に名前を付け、`invariant` 句で使用します。

**不変検証。** 検証器は、ループ管理された `V` を使用して、標準の 3 つの条件をチェックします。

1. **確立** — `V = {}` で不変式が成り立つ
2. **メンテナンス** — 任意の `x ∈ S \ V` について、不変式が本体の前で成立する場合、不変式は本体の後にも成立します (`V` を `{x}` で拡張)
3. **事後条件** - 通常の終了時、`V = S` と不変条件により最終結果が得られます。

条件 2 では **任意** `x ∈ S \ V` が使用されるため、証明では特定の列挙順序が仮定されることはありません。順序の独立性は、シーケンスではなく、セット `V` に関して記述される不変条件に暗黙的に含まれます。

**`Fold` templateとセット functor** 順序に依存しない累積を表現する標準的な方法は、**`Fold` template functor** (§18) を使用することです。これは、セット値計算の再帰構造を抽象化します。

```mizar
:: Conceptual definition of the Fold template (defined in §18 Templates)
:: Fold[op, e](S) for a commutative, associative op with unit e
::
:: func Fold[op, e](S) -> T means
::   it = e if S = {}
::   otherwise
::     ex x st x in S
::      & it = op(x, Fold[op, e](S \ {x})) ;
::
:: uniqueness proof: for x ≠ y in S,
::   op(x, op(y, Fold[op,e](S\{x,y})))
::   = op(y, op(x, Fold[op,e](S\{x,y})))  :: by commutativity of op
::   hence both choices yield the same result.
::   Full proof by induction on card S.
```

`Fold`** の **一意性条件は正確に順序独立です。`op` の可換性と結合性からtemplate レベルで一度証明され、すべてのインスタンス化はそれを自動的に継承します。

標準library のセットfunctorは、`Fold` の事前定義されたインスタンスです。

```mizar
:: Sum(S)              = Fold[+, 0](S)
:: Product(S)          = Fold[*, 1](S)
:: BigUnion(F, S)      = Fold[∪, {}](S)    (element mapped by F)
:: BigIntersection(F, S) = Fold[∩, univ](S) (element mapped by F)
Count(S, P)         = Fold[+, 0](S)     (element mapped to 1 if P holds, else 0)
```

カスタム累積を必要とするユーザーは、独自の `Fold` インスタンスを定義し、操作の可換性/結合性を引用して一意性の義務を果たします。

```mizar
:: Sum of elements — processed V tracks the accumulated set.
var total := 0;
for x in S processed V do
  invariant total = Sum(V);
  total := total + x;
end;
:: After normal exit: V = S (automatic), so total = Sum(S).

:: Product of elements.
var prod := 1;
for x in S processed V do
  invariant prod = Product(V);
  prod := prod * x;
end;

:: Existence check — no accumulator; visited clause not needed.
for x in S do
  if x mod 2 = 0 do
    return x;
  end;
end;
```

ループ本体に真の順序依存性がある (可換性の特徴付けが存在しない) 場合、`for ... in` は間違った構成です。代わりに、明示的に順序付けされた構造上で `for ... to` を使用してください。

> [！注記]
> `return` または `break` がループを早期に終了した場合、コンテキストには `V c= S` と、`V` に関する不変式の記述がすべて含まれます。事後条件 `V = S` は **通常** 終了時にのみ保証されます。

### 20.2.5 AST パターン マッチ (`match`)

`match` は、用語の AST に対して **構造パターン マッチング**を実行します。各 `case` は構文形式を記述します。大文字と小文字が一致すると、部分項は `var` で宣言されたパターン変数にバインドされます。これは、ブール述語をテストする `if`/`else` とは根本的に異なります。`match` は、1 つのステップで構造のテストと部分項の抽出を同時に行います。

```ebnf
match_stmt ::= "match" expr "do"
               match_case+
               ( "otherwise" algo_statement_list "end" ";"
               | exhaustiveness_proof )
               "end" ";" ;

match_case           ::= "case" term_pattern "do" algo_statement_list "end" ";" ;
exhaustiveness_proof ::= "exhaustive" [ justification ] ";" ;
term_pattern         ::= term_expression ;
```

**パターン変数**は、`term_pattern` に現れる `var` で宣言された識別子です。事前に割り当てられた値に関係なく、そのような変数はすべて **キャプチャ変数** として扱われます。一致に成功すると、それぞれが対応するサブタームで上書きされます。キャプチャされたサブタームの値制約は、ケース本体内の `if` で表現する必要があります。

**マッチングは構文上** (AST レベル) であり、セマンティックではありません。可換性などの代数恒等式は適用されません。`p * (q + r)` は、最も外側の演算が `*` で、右側のオペランドが `+` である正確な構文形式にのみ一致します。

**ケースは順番にテストされます**;最初に一致したケースが実行されます。 `otherwise` (または `exhaustive` の証明) は必須です。両方を省略すると **コンパイル時エラー**になります。

**非線形パターン**: 同じ変数がパターン内に複数回出現する場合があります。すべての出現箇所は、構文的に同一のサブタームと一致する必要があります。

**パターン変数のタイプ**は、`var` 宣言から取得されます。 `var` が明示的な型注釈なしで宣言されている場合、型は一致した式とパターン内に現れる演算子シグネチャから推測されます。

**パターン内の暗黙的な変換**は意図的に制限されています。パターン変数に明示的な型アノテーションがある場合、一致するサブタームはまさにその型を持つか、**固有の構造継承アップキャスト**をそれに対して許可する必要があります (第 19 章、§19.2.2)。 **クラスターベースの属性クロージャは、部分項がパターン変数タイプに適合するかどうかをチェックするときに使用されません**。また、あいまいな継承パスは暗黙的に解決されません。精査で明示的な `qua` を使用するか、より具体的ではないパターン変数型を選択します。これにより、パターン マッチングがローカルで予測可能になり、`match` の AST レベルの性質に合わせて維持されます。

```mizar
:: Evaluate a polynomial expression at a given point
definition
  let K be Field;
  let x be Variable;
  let t be Element of PolyRing[K,x];
  let v be Element of K;

  algorithm eval_at(t, v) -> Element of K
    ensures result = Eval(t, v)
    decreasing term_size(t)
  do
    var p, q as Element of PolyRing[K,x];
    match t do
      case p + q do
        return eval_at(p, v) + eval_at(q, v) by poly_eval_add;
      end;
      case p * q do
        return eval_at(p, v) * eval_at(q, v) by poly_eval_mul;
      end;
      otherwise
        :: t is atomic (the variable x or a scalar constant)
        return Eval(t, v);
      end;
    end;
  end;
end;

:: Non-linear pattern: matches only when both summands are syntactically identical
match t do
  case p + p do
    return 2 * p by ...;
  end;
  otherwise
    return t;
  end;
end;
```

**検証条件** 事例本体に入る際、検証器は命題的等価性を仮説として追加します。構造的一致自体によって等号性が証明されるため、正当化は必要ありません。前述のケースは否定仮説に寄与します。

|支店 |仮説が追加されました |
|---|---|
|最初の `case π₁ do` | `e = π₁` (キャプチャされた変数がバインドされている) |
| 2番目の`case π₂ do` | `¬(∃ capture vars. e = π₁)` および `e = π₂` (キャプチャされた変数がバインドされている) |
| `otherwise` |先行するすべてのパターンの否定 |

> [！注記]
> ケースは重複する場合があります。**最初に**一致したケースが優先されます。

### 20.2.6 ジャンプステートメント (`return`、`break`、`continue`)

```ebnf
return_stmt   ::= "return" [ expr [ justification ] ] ";" ;
break_stmt    ::= "break" ";" ;
continue_stmt ::= "continue" ";" ;
```

|声明 |効果 |検証義務 |
|---|---|---|
| `return expr` |値 `expr` でアルゴリズムを終了 | `expr` は `ensures` を満たす必要があります。末尾呼び出しの前提条件は解除されます。
| `return expr by ...` | `assert` + `return` (§20.4.2) の短縮形 | `return expr` と同じですが、`ensures` の放電に正当化が適用されます。
| `return` | void アルゴリズムを終了する |すべての `ensures` 条件が満たされる必要があります |
| `break` |すぐに囲まれているループを終了します |ループ `invariant` はブレークポイントで保持する必要があります |
| `continue` |次のループ反復にスキップ |ループ `invariant` は保持する必要があります。 **`decreasing` メジャーは現在の反復の開始と比較して減少している必要があります** |

**`continue` と終了。** `continue` は、終了チェックが通常行われるループ本体の終わりをバイパスするため、検証器は、`decreasing` メジャーが `continue` ポイントで **すでに減少している**ことを確認します。 `decreasing` アノテーションが存在しない場合、`continue` は終了義務を生成しません (ループは分岐する可能性があるものとして扱われます)。

```mizar
:: continue: the decreasing measure must have already decreased at the continue point.
while i <= n do
  invariant s = sum_of_odds_up_to(i - 1);
  decreasing n - i + 1;
  if i mod 2 = 0 do
    i := i + 1;
    continue;  :: n - (i + 1) + 1 < n - i + 1 already holds before continue
  end;
  s := s + i;
  i := i + 1;
end;
```

## 20.3 非決定論（選択）

**Pick** 操作は、Mizar の既存の `the` 用語形式を使用して表現された、空ではないセットから任意の要素を選択します。これらは、新しい第 20 章固有の式ではありません。これらは、従来の Mizar とまったく同様に、一般的な `term_expression` 構文 (第 13 章) の一部のままです。この章では、Mizar の既存の `the` 式のアルゴリズム 意味論に対してのみ **Pick** という名前を使用します。

```mizar
const x := the Element of S;    :: pick any element of S
const p := the prime Number;     :: pick any prime
```

**意味論。** 実行可能な algorithm body の外では、`the` は
[第13章, §13.5](./13.term_expression.md#135-選択演算子-the) で定義される
安定 choice 項です。algorithm body の内部では、`the T` を含む statement 右辺が
`ControlFlowIr` の site-local Pick binding へ低位化されます。これには
`const`/`var` initializer と後続の assignment が含まれます。論理 VC はその
binding を membership または型述語を満たす任意の fresh 値として扱い、その
binding が non-ghost state に流れる場合、MVM は実行時に具体値を非決定的に
1 つ選びます。ghost state だけに流れる Pick binding は仕様専用であり、ghost state
とともに消去されます。それでも同じ non-emptiness obligation を生成し、non-ghost
computation へ流れてはなりません。algorithm specification formula (`requires`、
`ensures`、`assert`、`invariant`、`decreasing`) は runtime Pick site を作らず、
第13章の stable choice semantics を使います。周囲の証明システムから見えるのは
algorithm contract だけであり、内部の生成 choice 名は観測できません。各 Pick site は、
program が結果を明示的に変数へ格納して再利用しない限り、それぞれ fresh binding を導入します。

**証明の安全性。** アルゴリズムのコントラクト (`requires`、`ensures`) は、周囲の証明システムから見える唯一のインターフェイスです。 Pick の結果を記述する `ensures` 句は、**すべて**の可能な選択肢に適用される必要があります。検証器はその旨の証明義務を負います。 `ensures` に記載されていないプロパティは、選択に関係なく、呼び出し元が想定することはできません。この封じ込めにより、アルゴリズム内の非決定性が外部の証明を黙って弱めることができないことが保証されます。

**非空性義務。** 検証器は各ピック サイトで非空性義務を生成しますが、その形式はバリアントによって異なります。

- *セットベース* (`the Element of S`): 義務 `S <> {}` は、証明によって免除される検証条件です。静的に放電できない場合、MVM は **実行時検証エラー** (§20.9.3) を生成します。
- *型ベース* (`the prime Number`): 義務は、型が**存在**であることです。つまり、その型の用語が少なくとも 1 つ存在します。これは、静的に実行される標準的なMizar 型居住義務です。実行時チェックは必要ありません。

```mizar
definition
  let S be finite set of Integer;

  terminating algorithm find_even(S) -> Integer
    ensures result = -1 or (result in S & result mod 2 = 0)
  do
    var current := S;
    while current <> {} do
      invariant current c= S;
      decreasing card current;
      const x := the Element of current;
      if x mod 2 = 0 do
        return x;
      end;
      current := current \ {x};
    end;
    return -1;
  end;
end;
```

> [！注記]
> MVM で Pick 式を実行できるかどうかは、有限性ではなく、MVM が実行時に具体的な監視を生成できるかどうかによって決まります。
> - `the Element of S` ここで、`S` は実行時に既知の有限セットです。MVM は `S` を列挙し、要素 (**実行可能file**) を選択できます。
> - `the prime Number`: Mizar libraryが計算可能な正規表現 (`2` など) を登録している場合、MVM はそれを使用できます (**実行可能file**)。
> - どのアルゴリズムも有限時間内に具象要素を生成できないタイプまたはセット: **実行可能ではありません**。たとえば、`S` のメンバーシップが決定できない `the Element of S` などです。
>
> 実行不可能な Pick 式は、アルゴリズム定義において依然として数学的に適切であり、論理仕様 (契約) のみが重要な証明に使用できます。実行不可能な Pick を含むアルゴリズムは、`terminating` に昇格できません。

## 20.4 契約 (`requires`、`ensures`、`assert`)

### 20.4.1 事前条件と事後条件

- `requires P` — `P` はすべての通話サイトで保持する必要があります。アルゴリズム本体は `P` を想定する場合があります。
- `ensures Q` — `Q` は正常終了時に保持されます。 `Q` の `result` は戻り値を指します。

各句は、固定順序 `requires` → `ensures` → `decreasing` で最大 1 回出現します。句内の複数の条件は、`&` または `and` を使用して表現されます。

```mizar
definition
  let a, b be Real;

  algorithm divide(a, b) -> Real
    requires b <> 0
    ensures result * b = a
  do
    return a / b;
  end;
end;
```

契約は**証明義務**です:
- 各呼び出しサイト `divide(x, y)` で、検証器は `y <> 0` の証明を要求します。
- `return v` の後、検証器は `v * b = a` であるという証明を要求します。

自動証明器が義務を履行できない場合、ユーザーは関連するサイトで正当な理由 (`by` または `proof ... end`) を含む `assert` (§20.4.2) を追加します。

### 20.4.2 インラインアサーション

```ebnf
assert_stmt ::= "assert" formula [ justification ] ";" ;
```

`assert P` は証明チェックポイントを挿入します。検証器は、現在の論理コンテキストと引用された補題またはインライン証明を使用して、そのプログラム ポイントで `P` を確立する必要があります。

**`return expr by ...` — インライン アサーションの省略表記。**
`return` が正当化を伴う場合、それは `expr` の `ensures` 事後条件を実行し、その後に `return` を実行する `assert` の糖衣構文です。

```mizar
  ensures result = a
do
  ...
  return b by th1;   :: sugar for: assert a = b by th1; return b;
  ...
```

アサートされた式は、返された式によって `result` が置換された `ensures` 句です。  完全な `proof ... end` ブロックの使用も許可されます。

```mizar
  return b
  proof
    ...
  end;
```

```mizar
definition
  let arr be FinSequence of Real;
  let lo, hi, target be Integer;

  algorithm binary_search(arr, lo, hi, target) -> Integer
  do
    var l := lo;
    var r := hi;
    while l <= r do
      invariant lo <= l & r <= hi;
      decreasing r - l + 1;
      const mid := (l + r) div 2;
      assert l <= mid & mid <= r by arithmetic;
      if arr.mid = target do
        return mid;
      else if arr.mid < target do
        l := mid + 1;
      else
        r := mid - 1;
      end;
    end;
    return -1;
  end;
end;
```

### 20.4.3 検証義務

|構築 |義務 |
|---|---|
|サイト `f(args)` に電話する | `requires` `f` の前提条件、`args` でインスタンス化 |
| `return expr` | `ensures` 事後条件、`result` が `expr` にバインドされる |
| `assert P` | `P` はそのプログラム ポイントで保持されます。
| `break` ループ内 |ループ `invariant` はブレークポイントで保持されます |
| `continue` ループ内 |ループ `invariant` は継続ポイントで保持されます。 `decreasing` メジャーは、現在の反復の開始と比較してすでに減少しています。
|可変状態の `for x in S` |ループ `invariant` は、結果が順序に依存しないことを証明するために、(`ghost var` 経由で) セットベースのfunctorに関して記述する必要があります。
| `for x in S` | `S` は `finite` |
| `the Element of S` | `S <> {}` |
| `decreasing e` (ループ) | `e` は `Nat` 値です。 `e` はループ反復ごとに厳密に減少します。
| `decreasing e` (ヘッダー) | `e` は `Nat` 値です。 `e` は再帰呼び出しごとに厳密に減少します。
| `terminating` |すべての `decreasing` 対策が有効です。アルゴリズムは、`requires` を満たすすべての入力に対して戻ります。

## 20.5 ループ検証

### 20.5.1 ループの不変条件 (`invariant`)

ループ不変式は次の条件を満たしている必要があります。
1. **最初の反復 (確立) の前**。
2. **後続のすべての反復のループ先頭** (メンテナンス)。
3. **通常の終了時** — ループ条件が false になったとき。
4. **すべての `break` ステートメント** — ブレークポイントで不変式が保持される必要があります。
5. **すべての `continue` ステートメント** — 継続点で不変式が保持される必要があります (ループは先頭から再入力されます)。

```mizar
var i := 1;
var s := 0;
while i <= n do
  invariant s = (i - 1) * i div 2;
  decreasing n - i + 1;
  s := s + i;
  i := i + 1;
end;
:: After loop: i = n + 1, s = n * (n + 1) div 2
```

複数の `invariant` 句は接続詞として解釈されます。

### 20.5.2 終了措置(`decreasing`)

```ebnf
(* loop termination — inside while body, via while_annotation *)
(* while_annotation ::= "invariant" formula | "decreasing" term_list ; *)

(* recursion termination — in algorithm header *)
(* algorithm_def ::= ... [ "decreasing" term_list ] algorithm_body ";" ; *)

term_list ::= term_expression { "," term_expression } ;
```

`decreasing` 注釈は、次の 2 つの異なる位置で使用されます。

- **アルゴリズム ヘッダー** (`algorithm_def` の `[ "decreasing" term_list ]`) — 再帰終了手段を宣言します。タプルは再帰呼び出しごとに減少する必要があります。
- **`while` 本体** (`while_annotation`) — ループ終了措置を宣言します。タプルは反復ごとに減少する必要があります。

`for` ループは `decreasing` を受け入れません**。有限セットの反復は有限性義務によって終了することが保証されています。

どちらの立場でも、検証器は同じ義務を生成します。

1. リスト内のすべての式は **`Nat` 値 ** (負ではない整数) です。
2. タプルは、ループの反復または再帰呼び出しごとに **辞書順に減少**します。

|フォーム |注文 |例 |
|---|---|---|
| `decreasing e` | `<` の `Nat` | |
| `decreasing e, f` | `Nat × Nat` の辞書編集用 `<` | |

十分に根拠のあるカスタムの順序付けを行うには、`decreasing` を使用する前に、ユーザー定義のfunctorを介してメジャーを `Nat` にマップします。

```mizar
:: User-defined measure mapping a tree to its depth (a Nat)
const d := tree_depth(t);
decreasing d;
```

非負性義務 (`e >= 0`) は定義サイトで履行されます。自動証明器がそれを確立できない場合は、`assert` を追加します。

```mizar
assert n - i >= 0 by arithmetic;
decreasing n - i;
```

### 20.5.3 構造的再帰測定 (`term_size`)

`match` を介して構文の部分項を再帰するアルゴリズムの場合、言語は組み込みのゴースト functor `term_size` を提供します。

```ebnf
term_size_expr ::= "term_size" "(" term_expression ")" ;
```

`term_size(t)` は、型 `Nat` の `ghost const` です。その具体的な値は未指定です。これは、1 つの公理ファミリーを満たす抽象的な十分に根拠のある尺度です。任意の二項演算子 `⊕` について、`match` ケースがパターン `p ⊕ q` を介して `t` から `p` および `q` をキャプチャする場合、検証器は自動的に仮説を追加します。

```
term_size(p) < term_size(t)
term_size(q) < term_size(t)
```

ケース本体に。これらの不等式についてユーザーによる証明は必要ありません。それらは構造的な一致そのものによって証明されます。

`term_size` は `ghost const` であるため、仕様コンテキスト (`decreasing`、`invariant`、`ensures`、`requires`、`assert`) でのみ現れる可能性があります。非ゴースト呼び出しに渡したり、非ゴースト変数に割り当てたりすることはできません。

```mizar
algorithm eval_at(t, v) -> Element of K
  ensures result = Eval(t, v)
  decreasing term_size(t)    :: ghost measure; verifier knows term_size(p) < term_size(t)
do
  var p, q as Element of PolyRing[K,x];
  match t do
    case p + q do
      :: hypotheses available: term_size(p) < term_size(t),  term_size(q) < term_size(t)
      return eval_at(p, v) + eval_at(q, v) by poly_eval_add;
    end;
    case p * q do
      return eval_at(p, v) * eval_at(q, v) by poly_eval_mul;
    end;
    otherwise
      return Eval(t, v);
    end;
  end;
end;
```

`term_size` は、唯一の組み込み AST レベルのゴースト functorです。 `match` がすでに利用可能にしているものを超える用語の構造は公開されません。将来のユースケースでより詳細な AST イントロスペクションが必要な場合 (例: ランタイム値としてノード数や深さを必要とするトラバーサル アルゴリズム)、その時点で追加のゴースト functorが導入される可能性があります。

## 20.6 スナップショットとクレーム

### 20.6.1 スナップショット

**スナップショット**は、特定のプログラム ポイントにおけるアルゴリズムの完全なローカル状態を記録します。これは仕様のみの構成です。スナップショットはコード抽出時に消去され (§20.10)、実行時のコストはかかりません。

```ebnf
snapshot_stmt ::= "snapshot" identifier ";" ;
```

スナップショットは、その時点で表示されるすべてのバインディング (`var`、`const`、`ghost var`、`ghost const`、およびアルゴリズム パラメーター) を単一の名前でキャプチャします。スナップショット名はアルゴリズム本体内で一意である必要があります。

### 20.6.2 クレームブロック

`claim` ブロックは、単一アルゴリズムのスナップショット状態に関する定理をグループ化し、`requires`/`ensures` コントラクトだけで述べられるものを超える内部検証プロパティを表現します。

```ebnf
claim_block ::= "claim" identifier "do"
                  { annotated_theorem_item }
                "end" ";" ;
annotated_theorem_item ::= { annotation } theorem_item ;
```

識別子は、スナップショットがスコープ内に取り込まれるアルゴリズムの名前を指定します。ブロック内では、スナップショット変数は `S.x` (スナップショット `S`、変数 `x`) としてアクセスされます。

**普遍的定量化。** `claim` ブロック内のすべての定理は、すべての有効な実行、つまり、アルゴリズムの `requires` 前提条件を満たすすべての呼び出しにわたって、暗黙的に普遍的に定量化されます。

**スナップショット 意味論。** スナップショットは、完全なローカル状態、つまりそのプログラム ポイントで表示されるすべての `var`、`const`、`ghost var`、および `ghost const` バインディングとアルゴリズム パラメーターをキャプチャします。

### 20.6.3 例

次のアルゴリズムは、`0` から `n` までカウントします。スナップショット `Init` は、ループ前の状態をキャプチャします。スナップショット `S` は、各反復の先頭の状態をキャプチャします。

```mizar
definition
  let n be Nat;

  algorithm count_up(n) -> Nat
    requires n >= 0
    ensures result = n
  do
    var x := 0;
    var i := 0;
    snapshot Init;
    while i < n do
      snapshot S;
      x := x + 1;
      i := i + 1;
    end;
    return x;
  end;
end;

claim count_up do
  :: state before the loop
  theorem InitState:
    Init.x = 0 & Init.i = 0;

  :: loop invariant: accumulator equals iteration counter at each step
  theorem Progress:
    S.x = S.i;
end;
```

`InitState` は、ループが開始される前に `x` と `i` が両方とも 0 であることを示します。 `Progress` は、各反復ステップでアキュムレータ `x` がカウンタ `i` と等しいことを示します。これは、`ensures result = n` だけではキャプチャできない内部プロパティです。

## 20.7 終了証明

### 20.7.1 終了措置の宣言

再帰的アルゴリズムの場合、`decreasing` はコントラクトと本文の間の **アルゴリズム ヘッダー**に配置されます。

```mizar
definition
  let m, n be Nat;

  terminating algorithm ackermann(m, n) -> Nat
    decreasing m, n
  do
    if m = 0 do
      return n + 1;
    end;
    if n = 0 do
      return ackermann(m - 1, 1);
    end;
    return ackermann(m - 1, ackermann(m, n - 1));
  end;
end;
```

検証器は、再帰呼び出しごとにタプル `(m, n)` が辞書順に減少していることをチェックします。

- `ackermann(m - 1, 1)` — 最初のコンポーネント `m - 1 < m`。 ✓
- `ackermann(m, n - 1)` — 最初のコンポーネントは等しい、2 番目の `n - 1 < n`。 ✓
- `ackermann(m - 1, ackermann(m, n - 1))` — 最初のコンポーネント `m - 1 < m`。 ✓

### 20.7.2 アルゴリズムの終了と`func`への昇格

`terminating` というプレフィックスが付いたアルゴリズムは、`requires` 前提条件を満たすすべての入力に対して終了することを宣言します。これにより、`by computation` 証明だけでなく、任意の形式的推論においても、`func` が期待されるあらゆる場所でアルゴリズムを使用できるようになります。

`algorithm_def` の完全な文法については、§20.1.1 を参照してください。

```mizar
definition
  let n be Nat;

  terminating algorithm factorial(n) -> Nat
    decreasing n
  do
    if n = 0 do
      return 1;
    end;
    return n * factorial(n - 1);
  end;
end;
```

`terminating` が宣言され検証されると、`factorial` をfunctorとして使用できるようになります。

```mizar
theorem FactorialZero: factorial(0) = 1;
theorem FactorialStep: factorial(n + 1) = (n + 1) * factorial(n);
```

> [！注記]
> `terminating` がなければ、アルゴリズムは計算可能な手続きとしてのみ存在します。具体的な値に対して `by computation` を使用して呼び出すことはできますが、普遍的に定量化したり、証明で第一級のfunctorとして使用したりすることはできません。

## 20.8 再帰的定義

### 20.8.1 直接再帰

アルゴリズムはそれ自体を再帰的に呼び出すことがあります。検証器は追加の注釈なしで再帰呼び出しを受け入れますが、終了を証明するために `decreasing` が必要です。 `decreasing` がない場合、アルゴリズムは部分的に定義されているものとして扱われます。

```mizar
definition
  let n be Nat;

  terminating algorithm fibonacci(n) -> Nat
    decreasing n
  do
    if n = 0 do return 0; end;
    if n = 1 do return 1; end;
    return fibonacci(n - 1) + fibonacci(n - 2);
  end;
end;
```

各再帰呼び出しサイトは、`requires` 前提条件が満たされていることの証明義務を生成します。

### 20.8.2 相互再帰

2 つ以上のアルゴリズムが相互に呼び出しを行う場合があります。これらは **同じ `definition` ブロック**内に配置されます。特別な宣言は必要ありません。検証器は相互再帰を自動的に検出します。

`boolean` は組み込みの真理値型です。その値は `true` および `false` です。

```mizar
definition
  let n be Nat;

  terminating algorithm is_even(n) -> boolean
    decreasing n
  do
    if n = 0 do return true; end;
    return is_odd(n - 1);
  end;

  terminating algorithm is_odd(n) -> boolean
    decreasing n
  do
    if n = 0 do return false; end;
    return is_even(n - 1);
  end;
end;
```

`decreasing` 対策は、通話サイクル全体にわたって集合的に終了を確立する必要があります。検証器は、無限の呼び出しシーケンスが発生しないことを確認します。

## 20.9 実行モデルと計算可能性

### 20.9.1 Mizar 仮想マシン (MVM)

**Mizar Virtual Machine (MVM)** は、検証中にアルゴリズム式を評価します。次の場合に呼び出されます。

- `by computation` 証明ステップが使用されます (§20.9.2)。
- アルゴリズムは、検証コンテキストで完全に具体的な (根拠) 引数を使用して呼び出されます。

MVM は **値による呼び出し** モデルを使用します。副作用はローカルの `var` 状態に限定されます。グローバルな可変状態はありません。

**インポートされた algorithm はパッケージ境界を越えて MVM 評価できません。** MVM は、**現在検証中のパッケージで定義された** algorithm だけを実行します。別パッケージからインポートされた algorithm は、その opaque インターフェース — 署名と契約のみ (§12.5、§12.6.1) — を通じて消費され、本体がインポート側の検証コンテキストに読み込まれることは決してありません。**その依存のソースがローカルに存在する場合でも同様です** (`path` または `git` 依存。ソースの利用可能性はビルド時の利便にすぎず、インポート境界 — これはパッケージの semver 境界でもあります、§23.2 — を緩めません)。したがって `by computation` はインポートされた algorithm の呼び出しを簡約できず、それに関する下流の証明は `ensures` 契約に依存し、具体的な計算には決して依存しません。これが、§23.2 のもとで public algorithm への本体のみの編集を下位互換 (PATCH) な変更に保つ理由です。インポート側は本体を観測できないため、本体の変更が下流の証明を壊すことはできません。(インポートした algorithm を `by computation` で実行できるようにする、明示的にバージョン管理された別個の *ランタイムアーティファクト* の公開は、本書では保留の機能でありスコープ外です。そのようなアーティファクトが公開された場合、その計算的内容は公開 API サーフェスの一部となり、それに影響する本体変更は MAJOR になります。)

### 20.9.2 `by computation` の証明ステップ

```ebnf
computation_proof  ::= "by" "computation" [ "(" computation_option { "," computation_option } ")" ] ;
computation_option ::= "steps"   ":" nat_literal
                     | "timeout" ":" nat_literal
                     | "nest"    ":" nat_literal ;
```

オプション:

|オプション |デフォルト |意味 |
|---|---|---|
| `steps` | `0` (無制限) | MVM 実行ステップの最大数 |
| `timeout` | `0` (無制限) |実時間の制限時間 (ミリ秒) |
| `nest` | `0` (無制限) |関数呼び出しスタックの最大深さ |

```mizar
theorem FactorialFiveByComputation: factorial(5) = 120 by computation;

theorem Fact10 : factorial(10) = 3628800
proof
  thus thesis by computation(steps: 100000);
end;
```

**`by computation` が証明できること:**

|ゴールフォーム | MVM の動作 |
|---|---|
| `f(v₁, …, vₙ) = c` (接地等価) | `f(v₁, …, vₙ)` を評価し、結果が `c` と等しいことを確認します。
| `P(v₁, …, vₙ)` (接地述語) | `P` を評価し、それが成立することを確認します。
|非地上フォーミュラ | **サポートされていません** — 古典的な証明が必要です |

`ex x st P(x)` 形式の存在目標の場合、推奨されるアプローチは、検索アルゴリズムを明示的に実装することです。

```mizar
terminating algorithm find_witness(S) -> Integer
  requires S <> {}
  ensures result in S & P(result)
do
  ...
end;
```

`ensures` 句は、特別な MVM 検索意味論を必要とせずに、証明システム内で存在証明を提供します。

### 20.9.3 計算可能性と実行時の障害

アルゴリズム定義は、非構築的な操作 (計算不可能な関数、抽象セットの選択など) を参照する場合があります。このような定義は数学的には有効ですが、MVM はそれらを実行できません。

MVM が実行不可能な操作に遭遇すると、**実行時検証エラー**が発生します。

|操作 |実行時のエラー状態 |
|---|---|
| `the Element of S` | MVM は `S` を列挙できません (たとえば、`S` は無限で、登録された列挙子がありません)。
| `algorithm` ボディなしで `func f(x)` を呼び出します | `f` には計算可能な実装がありません。
| `pred P(x)` または `func F(x)` template パラメータを呼び出します。具体的なインスタンス化には計算可能な実装がありません (例: `means` 定義のfunctor)。
|ゼロ除算 (`a / 0`) |ゼロの分母 |

実行時のエラーにより、`by computation` ステップは **失敗** しますが、アルゴリズムの定義は無効になりません。この定義は依然として数学的に意味があります。 MVM を介した実行可能性のみが制限されます。

> [！注記]
> `S` が数学的に有効な無限集合である `the Element of S` (例: `NAT`) は、定義レベルではエラーではありません。 MVM は単純にそれを実行できません。このアルゴリズムは、そのコントラクトを通じて古典的な証明で引き続き使用できます。

### 20.9.4 ガス制限

分岐アルゴリズムによって検証器がハングアップするのを防ぐために、MVM は **ガス制限**、つまり `by computation` 呼び出しごとの実行ステップの最大数を強制します。

```mizar
theorem HardComputation42: hard_computation(42) = 123456789
proof
  thus thesis by computation(steps: 1000000);
end;
```

いずれかのオプションを `0` に設定すると、その制限が無効になります (デフォルト)。オプションは組み合わせることができます:

```mizar
thus thesis by computation(steps: 500000, timeout: 2000, nest: 200);
```

## 20.10 コードの抽出

検証済みのアルゴリズムからのコード抽出は、コア言語仕様の一部ではなく、**別のtool**として提供されます。エクストラクターは `terminating` アルゴリズムを採用し、ターゲット言語 (Python、GMP を使用した C、Rust など) で実行可能コードを生成し、構築によって形式的に検証されたすべてのプロパティを保持します。

ターゲット言語、数値型マッピング (C の `Nat` → GMP `mpz_t` など)、ホスト library バインディングなどの構成は、インライン アノテーションではなくエクストラクター構成fileを介して提供されます。

詳細については、抽出toolのドキュメントを参照してください。

## 20.11 完全な構文 (EBNF)

```ebnf
(* ── Algorithm definition ─────────────────────────────── *)
algorithm_def ::= [ "terminating" ] "algorithm" identifier [ "[" schema_params "]" ] "(" [ identifier_list ] ")" [ "->" type_expression ]
                  [ "requires" formula ]
                  [ "ensures"  formula ]
                  [ "decreasing" term_list ]
                  algorithm_body ";" ;

identifier_list   ::= identifier { "," identifier } ;

algorithm_body    ::= "do" algo_statement_list "end" ;

(* ── Statement list ────────────────────────────────────── *)
algo_statement_list ::= { algo_statement } ;

algo_statement ::= var_decl
                 | const_decl
                 | ghost_var_decl
                 | ghost_const_decl
                 | assignment
                 | ghost_assignment
                 | if_stmt
                 | while_stmt
                 | for_range_stmt
                 | for_collection_stmt
                 | match_stmt
                 | return_stmt
                 | break_stmt
                 | continue_stmt
                 | assert_stmt
                 | snapshot_stmt
                 | standalone_diagnostic_annotation ;

(* ── Declarations and assignment ───────────────────────── *)
var_decl         ::= "var"         var_binding   { "," var_binding   } [ "as" type_expression [ justification ] ] ";" ;
ghost_var_decl   ::= "ghost" "var" var_binding   { "," var_binding   } [ "as" type_expression [ justification ] ] ";" ;
const_decl       ::= "const"       const_binding { "," const_binding } [ "as" type_expression [ justification ] ] ";" ;
ghost_const_decl ::= "ghost" "const" const_binding { "," const_binding } [ "as" type_expression [ justification ] ] ";" ;

var_binding   ::= identifier [ ":=" term ] ;
const_binding ::= identifier ":=" term ;

assignment       ::= lvalue ":=" term ";" ;
ghost_assignment ::= "ghost" lvalue ":=" term ";" ;
lvalue           ::= identifier
                   | lvalue "." identifier ;

justification ::= simple_justification | proof | computation_proof ;
(* simple_justification, proof, computation_proof, reasoning, references — defined in §15/§16/§20.9.2 *)

(* ── Control flow ──────────────────────────────────────── *)
if_stmt ::= "if" formula "do" algo_statement_list if_tail ;
if_tail ::= "end" ";"
          | "else" if_stmt
          | "else" algo_statement_list "end" ";" ;

while_stmt ::= "while" formula "do"
                 { while_annotation ";" }
                 algo_statement_list
               "end" ";" ;

while_annotation ::= "invariant" formula [ justification ]
                   | "decreasing" term_list [ justification ] ;

term_list ::= term_expression { "," term_expression } ;

for_range_stmt ::= "for" identifier "=" expr ( "to" | "downto" ) expr
                   [ "step" expr ]
                   "do" { for_annotation ";" } algo_statement_list "end" ";" ;

for_collection_stmt ::= "for" identifier "in" expr [ "processed" identifier ] "do"
                          { for_annotation ";" }
                          algo_statement_list
                        "end" ";" ;

for_annotation ::= "invariant" formula [ justification ] ;

match_stmt ::= "match" expr "do"
               match_case+
               ( "otherwise" algo_statement_list "end" ";"
               | exhaustiveness_proof )
               "end" ";" ;

match_case           ::= "case" term_pattern "do" algo_statement_list "end" ";" ;
term_pattern         ::= term_expression ;
exhaustiveness_proof ::= "exhaustive" [ justification ] ";" ;

return_stmt   ::= "return" [ expr [ justification ] ] ";" ;
break_stmt    ::= "break" ";" ;
continue_stmt ::= "continue" ";" ;
assert_stmt   ::= "assert" formula [ justification ] ";" ;
snapshot_stmt ::= "snapshot" identifier ";" ;

(* ── Snapshot and Claim ────────────────────────────────────── *)
claim_block ::= "claim" identifier "do"
                  { annotated_theorem_item }
                "end" ";" ;
annotated_theorem_item ::= { annotation } theorem_item ;

(* snapshot_access inside claim_block theorems:                  *)
(*   snapshot_name "." identifier                                 *)

(* ── Proof step ─────────────────────────────────────────────
   Used in proof blocks (outside algorithm bodies) as:
     "thus" formula "by" "computation" ...
   or in short-form theorems:
     theorem label_identifier ":" formula "by" "computation" ... ";" *)
computation_proof   ::= "by" "computation"
                        [ "(" computation_option { "," computation_option } ")" ] ;
computation_option  ::= "steps"   ":" nat_literal
                      | "timeout" ":" nat_literal
                      | "nest"    ":" nat_literal ;

(* ── Pick expression ─────────────────────────────────────────
   Used as a term on the right-hand side of const_decl, var_decl,
   assignment, ghost_var_decl, ghost_const_decl, or ghost_assignment
   inside an algorithm body:
     const x := the Element of S ;
     const p := the prime Number ;
     ghost witness := the Element of S ;                         *)
pick_expr ::= "the" type_expression ;
term_size_expr ::= "term_size" "(" term_expression ")" ;
```

## 20.12 拡張例

### 高速累乗 (二乗による累乗)

非自明な終了手段と `by computation` によって検証された数学的 `ensures` 句を使用した再帰を示します。

```mizar
definition
  let x be Nat;
  let n be Nat;

  terminating algorithm fast_power(x, n) -> Nat
    ensures result = x |^ n
    decreasing n
  do
    if n = 0 do
      return 1;
    end;
    if n mod 2 = 0 do
      const half := fast_power(x, n div 2);
      return half * half;
    end;
    return x * fast_power(x, n - 1);
  end;
end;

:: Verified by computation for concrete inputs
theorem FastPower2_10: fast_power(2, 10) = 1024 by computation;
theorem FastPower3_5:  fast_power(3, 5)  = 243  by computation;

:: Promoted to func via terminating; usable in general proofs
theorem fast_power_correct :
  for x, n being Nat holds fast_power(x, n) = x |^ n;
```

### 整数因数分解（試行部門）

```mizar
definition
  let n be Nat;

  algorithm trial_factor(n) -> Nat
    requires n >= 2
    ensures result >= 2 & result divides n
  do
    var d := 2;
    while d * d <= n do
      invariant d >= 2 & for k being Nat st 2 <= k & k < d holds not k divides n;
      assert n - d * d >= 0 by arithmetic;
      decreasing n - d * d + 1;
      if n mod d = 0 do
        return d;
      end;
      d := d + 1;
    end;
    return n;   :: n is prime
  end;
end;
```

### ユークリッド GCD アルゴリズム

このアルゴリズムは、標準libraryで定義されている数学functor `Gcd` と区別するために、`euclid_gcd` という名前が付けられています。 `ensures` 句とループ不変式は両方とも、アルゴリズム自体ではなく `Gcd` (数学的) を使用するため、循環性はありません。

```mizar
definition
  let a, b be Nat;

  terminating algorithm euclid_gcd(a, b) -> Nat
    requires a >= 1 & b >= 1
    ensures result = Gcd(a, b)
  do
    var x := a;
    var y := b;
    while y <> 0 do
      invariant x >= 1 & y >= 0 & Gcd(a, b) = Gcd(x, y);
      decreasing y;
      const r := x mod y;
      x := y;
      y := r;
    end;
    return x;
  end;
end;

theorem EuclidGcd12_8: euclid_gcd(12, 8) = 4 by computation;
theorem EuclidGcd100_75: euclid_gcd(100, 75) = 25 by computation;
```

### 記号微分

式 AST での `match`、`term_size` による構造再帰、および `return ... by` 短縮表現を示します。 `Expr[K, x]` は、フィールド `K` および変数 `x` に対する算術式タイプです。 `Diff(t, x)` は、支持理論で定義された数学的 (形式的導関数) functorです。

```mizar
definition
  let K be Field;
  let x be Variable;
  let t be Expr[K, x];

  terminating algorithm diff(t) -> Expr[K, x]
    ensures result = Diff(t, x)
    decreasing term_size(t)
  do
    var p, q as Expr[K, x];
    match t do
      case p + q do
        :: D(p + q) = D(p) + D(q)
        return diff(p) + diff(q) by diff_add;
      end;
      case p * q do
        :: D(p * q) = D(p) * q + p * D(q)
        return diff(p) * q + p * diff(q) by diff_mul;
      end;
      case x do
        :: D(x) = 1
        return 1 by diff_var;
      end;
      otherwise
        :: t is a constant (no occurrence of x); D(c) = 0
        return 0 by diff_const;
      end;
    end;
  end;
end;

:: Spot-check on concrete expressions
theorem DiffSquare: diff(x * x) = x + x by computation;           :: D(x^2) = 2x
theorem DiffSquarePlusX: diff(x * x + x) = x + x + 1 by computation;
```


## 20.13 論理エンコーディング

このセクションでは、アルゴリズム構造が Mizar kernel内の論理式にどのようにマップされるかを指定します。これは、`terminating` アルゴリズム用に生成された形式公理と各ステートメント形式の検証条件 (VC) スキーマによって、§20.4 ～ §20.7 の動作記述を補完します。

### 20.13.1 契約公理

すべてのアルゴリズム呼び出しサイトは `requires` から証明義務を生成し、すべての `return` は `ensures` から証明義務を生成します。以下の表に FOL エンコーディングを示します。囲んでいる `definition` ブロックの型パラメーターは普遍的に定量化され、型ガード `is_A(x)` として表示されます。

|構築 | FOL エンコーディング |
|---|---|
| `requires P` |各コールサイトでの証明義務: `Γ ⊢ P(a₁, …, aₙ)` |
| `ensures Q` で `return e` |証明義務: `Γ ⊢ Q[result/e]` |
| `ensures Q` (呼び出し側、部分的なアルゴリズム) |通話が終了することがわかっている場合にのみ使用できます。無条件の公理は出力されません。
| `assert φ` |証明義務: そのプログラム時点での `Γ ⊢ φ`。 `φ` はその後 `Γ` に追加されます |
| `f(a₁, …, aₙ)`に電話する | (1) `Γ ⊢ requires_f(a₁, …, aₙ)`; (2) `ensures_f(a₁, …, aₙ, f(a₁, …, aₙ))` を `Γ` に追加 |

複数の `requires` 句と `ensures` 句が結合されます。

partial algorithm について「停止すると分かっている」とは、その call site がその具体的な
呼び出しに対する検証済み停止性定理を引用しているか、文脈に持っていること、または callee が
検証済みの `terminating` declaration によって昇格済みであることを意味します。そのような証拠が
ない場合、caller は call の型事実だけを利用でき、`ensures` 句を仮定できません。

### 20.13.2 `terminating` のプロモーション公理

`terminating` が宣言され検証されると、アルゴリズムは `func` に昇格され、次の公理が Mizar kernelに出力されます。

|公理 | FOL スキーマ |
|---|---|
| **全体性** | `∀x₁,…,xₙ. (is_A₁(x₁) ∧ … ∧ P(x₁,…,xₙ) → is_T(f(x₁,…,xₙ)))` |
| **契約** | `∀x₁,…,xₙ. (is_A₁(x₁) ∧ … ∧ P(x₁,…,xₙ) → Q(x₁,…,xₙ, f(x₁,…,xₙ)))` |

`requires` が存在しない場合、`P` はデフォルトで `⊤` になります。functorは宣言された型のすべての入力の合計です。複数の `ensures` 句が 1 つの `Q` に結合されます。

**例** — `factorial`:

```
terminating algorithm factorial(n) -> Nat
  ensures result >= 1
  decreasing n
```

生成された公理:

|公理 |式 |
|---|---|
|全体性 | `∀n. (is_Nat(n) → is_Nat(factorial(n)))` |
|契約 | `∀n. (is_Nat(n) → factorial(n) ≥ 1)` |

これらの公理は、`terminating` が検証されるとすぐに証明システムで使用できるようになります。アルゴリズム本体は展開されていません。さらなるプロパティは明示的な定理 (`theorem FactorialZero: factorial(0) = 1 by computation;` など) によって確立されます。

### 20.13.3 検証条件スキーマ

検証器はアルゴリズム本体を走査し、標準的な Hoare 論理 スキーマによって VC を生成します。 `Γ` は、現在の論理コンテキスト (そのプログラム ポイントで蓄積されたファクト) を示します。

statement verification は `Normal`、`Return`、`Break(loop)`、`Continue(loop)`
の exit context を別々に返します。逐次合成は `Normal` からだけ継続します。
loop construct は自分に属する `Break` と `Continue` context を消費し、無関係な
exit はそのまま残します。loop の post-context は normal exit と break exit の
join です。`while C` の `¬C` のように通常終了だけで成り立つ fact は、break exit
には追加されません。

`assign(Γ, l, e)` は §20.1.4 の old-state assignment transformer を表します。
`e` を pre-state で評価し、`l` と alias し得る location を含む fact を forget
または rename し、alias しない fact を保存して、`l` の post-state equality を
追加します。`assign_field` は解決済み lvalue identity を通じた field update 用の
対応する transformer です。`V_after` は normal back edge または `continue` edge
の post-state で評価された loop measure、`V_before` は body 実行前の loop head
で評価された同じ measure です。range loop では `step` の default は `1` で、
positive `Nat` でなければなりません。`next(i)` は `to` loop では `i + step`、
`downto` loop では `i - step` です。

| Statement | 生成される VC | 後続コンテキスト |
|---|---|---|
| `var x := e` | — | `Γ ∪ { x = e, is_inferred_T(x) }` |
| `var x := e as T` | `Γ ⊢ is_T(e)` (自動または justification による) | `Γ ∪ { x = e, is_T(x) }` |
| `const x := e` | — | `Γ ∪ { x = e }` |
| `x := e` | — | `assign(Γ, x, e)` |
| `p.f := e` | `Γ ⊢ field_update_well_typed(p, f, e)` | `assign_field(Γ, p.f, e)`; `p.f` の alias は解決済み lvalue identity を通じてのみ更新される |
| `ghost x := e` | — (仕様のみ) | ghost state 上の `assign(Γ, x, e)`; Pick site は ghost-only |
| `return e` | `Γ ⊢ ensures[result/e]` | — (退出) |
| `return` (ボイド) | `Γ ⊢ ensures` | — (退出) |
| `assert φ` | `Γ ⊢ φ` | `Γ ∪ { φ }` |
| call `f(args)` | `Γ ⊢ requires_f(args)` | `Γ ∪ { ensures_f(args, result) }` |
| `if C do S₁ end` | `Γ ∪ {C}` の下の `S₁` を確認する。存在しない `else` branch は `Γ ∪ {¬C}` を寄与する | `S₁` の normal exit context と `Γ ∪ {¬C}` の join |
| `if C do S₁ else S₂ end` | `Γ ∪ {C}` の下の `S₁` を確認し、`Γ ∪ {¬C}` の下の `S₂` を検証する | exit context の join |
| `while C do invariant I; decreasing V; S end` | (1) `Γ ⊢ I` (設立); (2) `Γ ∪ {I, C}` の下で `S` を検証する。normal back edge と `continue` edge はすべて `I` を証明する; (3) `I ∧ C ⊢ is_Nat(V_before)`; (4) normal back edge と `continue` edge はすべて `V_after < V_before` を証明する。`break` edge はすべて `I` を証明する | normal exit の `Γ ∪ { I, ¬C }` と loop の break exit context の join。break exit は `¬C` を得ない |
| `break` ループ内 | 最内 loop について `Γ ⊢ I` | `Break(loop, Γ ∪ {I})` |
| `continue` ループ内 | `Γ ⊢ I`; decreasing measure が宣言されている場合は `Γ ⊢ V_after < V_before` | `Continue(loop, Γ ∪ {I})` |
| `for i = a (to | downto) b [step s] do invariant I; S end` | (1) `Γ ⊢ is_positive_Nat(s)`; (2) `Γ ⊢ I[i/a]`; (3) `Γ ∪ {I, in_range(i)}` の下で `S` を検証する。normal back edge はすべて `I[i/next(i)]` を証明する。`break` edge はすべて `I` を証明する | normal exit summary `I[i/i_exit] ∧ past_end(i_exit)` と loop の break exit context の join。`i_exit` は hidden loop metadata であり、source-visible variable ではない。`to` では `in_range(i)` は `a ≤ i ≤ b`、`past_end(i_exit)` は `i_exit > b`。`downto` では `in_range(i)` は `b ≤ i ≤ a`、`past_end(i_exit)` は `i_exit < b` |
| `for x in S processed V do invariant I; S_body end` | (1) `Γ ⊢ I[V/∅]`; (2) `I ∧ x ∈ S ∖ V ⊢ wp(S_body, I[V ↦ V ∪ {x}])` | `Γ ∪ { I ∧ V = S }` |
| `for x in S do S_body end` (`processed` なし) | `Γ ∪ { x ∈ S }` の下の `S_body` を検証する | `Γ` |
| `match e do case π₁ do S₁ end; … end` | `i` の場合: キャプチャ変数をバインドします。 `Γ ∪ { e = πᵢ, ¬(∃ cvars₁. e = π₁), …, ¬(∃ cvarsᵢ₋₁. e = πᵢ₋₁) }` の下の `Sᵢ` を確認します。 `otherwise` または `exhaustive` が存在することを確認します |出口コンテキストの結合 |

複数の `invariant` 句が結合されます。`invariant P; invariant Q;` は `invariant P ∧ Q;` として扱われます。

### 20.13.4 再帰的アルゴリズムのエンコーディング

再帰的 `terminating` アルゴリズムの終了は、**十分に根拠のある再帰引数**によって確立されます。基礎となる原則は次のとおりです。

> *`requires` を満たすすべての入力 `x` について、すべての再帰呼び出し `f(y)` が `measure(y) < measure(x)` (辞書編集的には `Nat`) を満たし、`measure` が `Nat` 値である場合、`f` は、`requires` を満たすすべての入力の合計になります。

これは、`Nat` (または `Nat × Nat × …`) の十分に根拠のある帰納スキーマに対応します。

```
∀x. (∀y. (measure(y) < measure(x) → P(y))) → P(x)
─────────────────────────────────────────────────────
∀x. P(x)
```

ここで、`P(x)` は、入力 `x` の全体性と契約性の積です。

**相互再帰**の場合、同じ原則が集合的に適用されます。検証器は、相互再帰グループを単一の帰納法として扱い、共有 `decreasing` 測定値がクロスアルゴリズム呼び出しごとに減少することを検証することによって、無限の呼び出しシーケンスが発生しないことを確認します。

すべての終了が確認されると、グループ内のアルゴリズムごとにプロモーション公理 (§20.13.2) が発行されます。

### 20.13.5 ゴースト変数の消去

ゴースト変数は検証中に論理コンテキストに存在しますが、ランタイム プログラムからは消去されます。消去は次の意味で**健全**です。

> *ゴースト変数の存在下で検証された非ゴーストコントラクト (`requires`、`ensures`、`assert`) は、消去されたステートメントでゴースト変数について言及されていない限り、消去されたプログラムにも保持されます。

|プロパティ |声明 |
|---|---|
| **契約の保存** | `S'` を、`ghost var`、`ghost const`、および `ghost` の割り当てをすべて削除した `S` とします。 `S` が契約 `P`/`Q` を満たし、`P`、`Q` がゴースト変数について言及していない場合、`S'` は `P`/`Q` も満たします。 |
| **不変射影** |ゴースト変数 (`S_visited` など) に言及するループ不変式は、型チェック中にのみ検証されます。実行時の義務は発生しません。 |
| **ゴーストの隔離** |検証器は、ゴースト変数が非ゴースト代入の右側に表示されないこと、および非ゴースト アルゴリズム呼び出しに渡されないことを静的にチェックします (§20.1.3)。これは消去が健全であるための重要な条件です。 |
