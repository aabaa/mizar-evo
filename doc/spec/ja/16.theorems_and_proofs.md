## 16.1 定理の役割とステータス

Mizar Evolution は、2 つの独立した軸に沿って宣言型論理ユニットを区別します。

1. **役割**: 項目の数学的または説明的な役割。
2. **ステータス**: 証拠と信頼に関して検証器がアイテムをどのように扱うか。

最初の表面的な言語では、意図的に役割を小さく設定しています。 `corollary` や `proposition` などの追加のロールは、エイリアスまたはドキュメント カテゴリとして後で導入される可能性がありますが、検証の意味論は変更されません。

**構文 (EBNF):**

```ebnf
theorem_item     ::= [ theorem_status ] theorem_role label_identifier ":" formula [ justification ] ";" ;
theorem_status   ::= "open" | "assumed" | "conditional" ;
theorem_role     ::= "theorem" | "lemma" ;
justification    ::= simple_justification | proof ;
```

**役割:**

|役割 |意味論 |
|---|---|
| `theorem` |理論の一部として意図された名前付きの数学的結果。
| `lemma` | `theorem` | と同じルールによって検証される、名前付きの補助結果。

**ステータス:**

|ステータス |正当化 |依存関係ルール |意味論 |
|---|---|---|---|
|無修正 |オプション (省略した場合は検証器が自動証明します) |清潔でなければなりません |完全に検証された提案 |
| `open` |オプション |正当な理由がある場合は、他の非クリーン項目に依存する可能性があります。未解決の提案、公開資料として記録 |
| `assumed` |許可されません |証明依存性なし |意図的に受け入れられた理論またはモデルの仮定 |
| `conditional` |必須 |クリーンでない依存関係が必要です |結果が少なくとも 1 つの `open`、`assumed`、または `conditional` 項目に依存する導出を検証しました。

依存関係に検証済みの未変更の定理項目と検証済みの定義または登録のみが含まれている場合、その依存関係は **クリーン** です。依存関係に `open`、`assumed`、または `conditional` の定理項目が含まれている場合、その依存関係は **非クリーン** です。

**例：**

```mizar
:: Theorem with proof
theorem Th1:
  for n being Nat holds n + 0 = n
proof
  let n be Nat;
  thus thesis by mml.number.natural.Nat_add_zero;
end;

:: Theorem without explicit justification (auto-proved by verifier)
theorem Trivial: 0 = 0;

:: Lemma with the same verification semantics as theorem
lemma AddZeroLeft:
  for n being Nat holds 0 + n = n
  by mml.number.natural.Nat_add_comm, Th1;

:: Open theorem: an unsettled proposition
open theorem Goldbach:
  for n being even Nat st n > 2
    ex p, q being Prime st n = p + q;

:: Assumed theorem: accepted as a model-level or theory-level assumption
assumed theorem RandomOracle:
  for H being HashFunction holds H is random_oracle_like;

:: Conditional theorem: proved relative to open or assumed items
conditional theorem Even_Goldbach:
  for n being even Nat st n > 4
    ex p, q being odd Prime st n = p + q
  by Goldbach, mml.number.prime.Odd_prime_except_2;
```

> [！注記]
> パラメーター化された推論template (`scheme`) は、template定理として構文的に表現されます。構文とアプリケーション パターンについては、[第 18 章 (テンプレート)、§18.8](./18.templates.md#188-パラメータ化された定理とスキーム) を参照してください。

> [！注記]
> ATP の統合、ソルバーの選択、証明ヒントの注釈 (`@proof_hint` など) については、[第21章 (source code annotation と ATP integration)](./21.source_code_annotation_and_atp.md) を参照してください。

## 16.2 定理宣言の構文

**構文 (EBNF):**

```ebnf
theorem_item     ::= [ theorem_status ] theorem_role label_identifier ":" formula [ justification ] ";" ;
theorem_status   ::= "open" | "assumed" | "conditional" ;
theorem_role     ::= "theorem" | "lemma" ;

justification ::= simple_justification | proof ;
proof         ::= "proof" reasoning "end" ;
reasoning     ::= { statement } ;
```

- **ラベル**: 必須。 `by` 正当化を介して他の証明から定理を参照できるようにします。
- **式**: 証明されるべき命題Φ。数式の構文については、「[第14章 (式)](./14.formulas.md)」を参照してください。
- **理由**: `by` 参照リスト、完全な `proof ... end` ブロック、またはアイテムのステータスが許可する場合は完全に省略されます。未変更のアイテムに対して省略された場合、検証器は自動証明を試みます。証明内のステートメント構文については、[第15章 (文)](./15.statements.md) を参照してください。

**例：**

```mizar
:: Theorem with a full proof
theorem Lagrange:
  for G being finite Group, H being Subgroup of G
    holds card H divides card G
proof
  let G be finite Group;
  let H be Subgroup of G;
  consider I being Left_Cosets of H by mml.group.Left_Cosets_def;
  card G = card H * card I by mml.group.Coset_partition;
  hence thesis by mml.number.natural.Nat_divides_def;
end;

:: Theorem with simple justification (derived from existing results)
theorem Cor1:
  for G being finite Group, H being Subgroup of G
    holds card H <= card G
  by Lagrange, mml.number.natural.Nat_divides_le;

:: Lemma with a simple justification
lemma SubgroupCardNonzero:
  for G being finite Group, H being Subgroup of G
    holds card H > 0
  by mml.algebra.group.Subgroup_nonempty;
```

**ステータスの伝播:**

変更されていない `theorem` または `lemma` は、直接的または推移的に、`open`、`assumed`、または `conditional` 項目を参照してはなりません。このルールに違反すると、検証器はエラーを生成します。

未確定の資料または想定されている資料と比較して証明された結果を記録するには、それを `conditional` として宣言します。検証器は引用された事実から依存関係ステータスを計算し、宣言されたステータスが実際の依存関係グラフと一致することを確認します。

```mizar
open theorem C1: for n being Nat holds P[n];

:: ERROR: unmodified theorem must not depend on open material
theorem Th2: P[42] by C1;
:: Verifier error: "Th2 depends on open theorem C1; declare it conditional"

:: OK: conditional theorem records the dependency
conditional theorem C2: P[42] by C1;
```

`assumed` 項目は証明なしで受け入れられ、モデルの仮定、外部仕様、計算上の仮定、および同様の前提を対象としています。 `open` とは異なり、証明を待つ未解決の定理としては提示されません。

## 16.3 証明スケルトンと自然演繹

Mizar は、証明される定理の論理構造が証明の形式を決定する宣言的証明フレームワークである **ヤシュコフスキー スタイルの自然演繹** を採用しています。 (戦術ベースのシステムのように) 目標を手続き的に変換するのではなく、証明の作成者は、式の構造を反映する一連の宣言を作成します。

### 16.3.1 スケルトンの抽出

`proof ... end` ブロックが開始されると、検証器は証明される式 Φ を分析し、**証明スケルトン** (Φ の最も外側の論理接続子によって決定される入れ子になったブロックのツリー) を抽出します。スケルトン内の各ノードは構造分解ルールに対応し、各リーフは正当化される必要がある**最終的な義務**です。

**例：**

定理を考えると:

```mizar
theorem Th2:
  for x being Real st x > 0 holds x^2 > 0
```

式の構造は `∀x:Real. (x > 0 → x² > 0)` です。スケルトンは次のとおりです。

```
Skeleton of: ∀x:Real. (x > 0 → x² > 0)
├─ [∀-intro]  let x be Real;
│  └─ [→-intro]  assume x > 0;
│     └─ [leaf]  ⊢ x² > 0        ← terminal obligation
```

対応する Mizar 証明は、この骨格を反映しています。

```mizar
proof
  let x be Real;                   :: ∀-intro: introduce x
  assume A1: x > 0;                :: →-intro: assume antecedent
  thus x^2 > 0                     :: terminal obligation: justify conclusion
    by A1, mml.number.real.Square_positive;
end;
```

証明は**宣言的**です。各ステートメントは、目標状態を変更するコマンドを発行するのではなく、式に関する構造的事実を宣言します。検証器は、一連の宣言が Φ の構造と一致すること、および各最終義務が適切に正当化されることをチェックします。

### 16.3.2 `thesis` キーワード

キーワード `thesis` は、**現在の最終義務** (証明スケルトンの現在の位置でまだ結論が出ていない式) に評価される動的定数です。

```mizar
theorem Th3:
  for n being Nat holds n >= 0
proof
  let n be Nat;
  thus thesis by mml.number.natural.Nat_nonneg;   :: thesis = (n >= 0)
end;
```

- `let x be T;` の後、`thesis` が普遍的に定量化された式の本体に更新されます。
- `assume φ;` の後、`thesis` は含意の結果に更新されます。
- 残りの義務が結合 φ ∧ ψ である場合、`thesis` は、`thus` が 1 つのコンポーネントを排出するまでの残りの結合全体を指します。
- `thesis` を使用すると、完全な式を再記述することなく、一般的な理由を記述することができます。

> [！注記]
> 数式構文の `thesis` 定数については、「[第14章 (式), §14.3.6](./14.formulas.md#1436-特殊定数)」を参照してください。

### 16.3.3 スケルトンの分解ルール

次のルールは、式内の各論理接続詞が対応する証明スケルトン要素をどのように決定するかを定義します。ステートメントが予想される数式の構造と一致しない場合、検証エラーが発生します。

**1.ユニバーサル定量化 (`let`)**

| | |
|---|---|
| **式** | ∀x:T. φ(x) |
| **スケルトン** | `let y be T;` — 新しい変数を導入します |
| **残りの義務** | φ(y) |

```mizar
:: Formula: for x being Real holds x + 0 = x
let x be Real;
:: Obligation becomes: x + 0 = x
```

**2.含意 (`assume`)**

| | |
|---|---|
| **式** | φ → ψ |
| **スケルトン** | `assume φ;` — 先行詞をローカル ファクトとして宣言します。
| **残りの義務** | ψ |

```mizar
:: Formula: x > 0 implies x^2 > 0
assume A1: x > 0;
:: Obligation becomes: x^2 > 0
```

**3.接続 — 順次放電 (`thus`)**

| | |
|---|---|
| **式** | φ ∧ ψ |
| **スケルトン** | 2 つの連続した `thus` ステートメント |
| **義務** |最初に φ、次に ψ |

```mizar
:: Formula: x > 0 & x < 10
thus x > 0 by A1;            :: discharges first conjunct
:: Obligation becomes: x < 10
thus x < 10 by A2;           :: discharges second conjunct
:: All obligations discharged
```

**4.直接結論 (`thus`)**

| | |
|---|---|
| **式** | φ (原子または非分解性) |
| **スケルトン** | `thus φ by J;` — 終了義務を正当化する |
| **結果** |義務の免除 |

**5.存在の定量化 (`take`)**

| | |
|---|---|
| **式** | ∃x:T. φ(x) |
| **スケルトン** | `take t;` — 証人用語を提供します |
| **残りの義務** | φ(t) |

```mizar
:: Formula: ex n being Nat st n > 100
take 101;
:: Obligation becomes: 101 > 100
thus thesis;
```

**6.等価性 (`iff`)**

| | |
|---|---|
| **式** | φ ↔ ψ |
| **スケルトン** | 2 つの部分証明: φ → ψ および ψ → φ |
| **義務** |各方向は独立して証明されました |

```mizar
:: Formula: x in A /\ B iff x in A & x in B
thus x in A /\ B implies x in A & x in B
proof
  assume x in A /\ B;
  hence thesis by mml.set.Intersection_def;
end;
thus x in A & x in B implies x in A /\ B
proof
  assume that A1: x in A and A2: x in B;
  hence thesis by A1, A2, mml.set.Intersection_def;
end;
```

**7.ケース分析 (`per cases`)**

| | |
|---|---|
| **式** | φ (証明に格分割が必要な場合は任意の形式) |
| **スケルトン** | `per cases by J;` の後に `case` サブブロックが続く |
| **義務** |各場合の仮定の下で φ を証明する |

```mizar
:: Formula: |x| >= 0
per cases;
case x >= 0;
  thus |x| = x & x >= 0;
end;
case x < 0;
  thus |x| = -x & -x > 0;
end;
```

> [！注記]
> `let`、`assume`、`thus`、`take`、および `per cases` の構文の詳細については、「[第15章 (文)](./15.statements.md)」を参照してください。

**複合スケルトンの例:**

```mizar
:: Formula: ∀x:Real. (x > 0 → ∃y:Real. (y > 0 ∧ y < x))
theorem Th5:
  for x being Real st x > 0
    ex y being Real st y > 0 & y < x
proof
  let x be Real;                    :: ∀-intro
  assume A1: x > 0;                 :: →-intro
  take y = x / 2;                   :: ∃-intro (witness)
  thus y > 0                        :: ∧-left obligation
    by A1, mml.number.real.Half_positive;
  thus y < x                        :: ∧-right obligation
    by A1, mml.number.real.Half_less;
end;
```

スケルトンツリー:

```
∀x:Real. (x > 0 → ∃y:Real. (y > 0 ∧ y < x))
├─ [∀-intro]  let x be Real;
│  └─ [→-intro]  assume x > 0;
│     └─ [∃-intro]  take x/2;
│        ├─ [leaf]  ⊢ x/2 > 0     (by Half_positive)
│        └─ [leaf]  ⊢ x/2 < x     (by Half_less)
```

## 16.4 証明の構造と範囲

### 16.4.1 証明ブロック

`proof` ブロックは、証明される式のスケルトン分解のための新しい論理スコープを作成します。

**構文 (EBNF):**

```ebnf
proof     ::= "proof" reasoning "end" ;
reasoning ::= { statement } ;
```

- **入力**: 前の `theorem`、`definition`、または正確性条件から式を継承します。
- **出力**: すべての最終義務は履行されなければなりません。 `end` で不正な `thesis` があると、「証明されていません」エラーがトリガーされます。

**例：**

```mizar
theorem Th4:
  for x being Real holds x * 0 = 0
proof
  let x be Real;
  thus x * 0 = x * (1 - 1)
            .= x * 1 - x * 1 by mml.number.real.Distributive
            .= 0;
end;
```

### 16.4.2 範囲と可視性

- **ラベル**: 証明内で定義されたラベル (例: `A1:`) は、その証明ブロックに対して **ローカル** です。外部からはアクセスできません。
- **変数**: `let`、`given`、または `consider` によって導入された変数は、それを囲んでいるブロックに対してローカルです。
- **同一スコープの一意性**: 同じスコープ内での重複ラベルおよび重複変数名は禁止されています (コンパイル エラー)。
- **内部スコープのシャドウイング**: 内部ブロック内の変数は、同じ名前の外部変数をシャドウすることができます。

> [！注記]
> スコープとシャドウイングのルールの詳細については、「[第4章 (変数と定数), §4.6](./04.variables_and_constants.md#46-スコープとシャドウイング)」を参照してください。

### 16.4.3 拡散推論 (入れ子になった証明)

複雑な証明には副引数が必要になることがよくあります。これらは、ローカル スコープの匿名定理として動作する拡散ステートメントによって処理されます。

**`now ... end`** *(非推奨)*: 収集された結論が参照可能な命題となるラベル付き推論ブロックを作成します。

```mizar
A1: now
  assume x > 0;
  thus x^2 > 0 by mml.number.real.Square_positive;
end;
:: A1 refers to: x > 0 implies x^2 > 0

thus thesis by A1;
```

**`hereby ... end`**: `thus now ... end` の糖衣構文。現在の義務の部分的な結論としてブロックの結果をアサートします。

```mizar
hereby
  assume x > 0;
  thus x^2 > 0 by mml.number.real.Square_positive;
end;
:: Equivalent to: thus (x > 0 implies x^2 > 0) proof ... end;
```

> [！注記]
> `now` および `hereby` の構文については、「[第15章 (文), §15.6](./15.statements.md#156-証明組織)」を参照してください。

## 16.5 証明の正当性 (`by`)

`by` 正当化は、Mizar 証明で証拠を引用するための中心的なメカニズムです。これは、自動証明探索を使用して、現在の目標 (`thesis`) がリストされた参考文献に従っていることを確認するように検証器に指示します。

**構文 (EBNF):**

```ebnf
simple_justification ::= [ "by" references ] ;
references           ::= reference { "," reference } ;
reference            ::= label_identifier [ template_args ]
                       | qualified_reference [ template_args ]
                       | grouped_reference
                       | bulk_reference ;
qualified_reference  ::= namespace_path "." identifier ;
grouped_reference    ::= namespace_path ".{" grouped_item { "," grouped_item } "}" ;
grouped_item         ::= identifier [ template_args ] ;
bulk_reference       ::= namespace_path ".*" ;
namespace_path       ::= identifier { "." identifier } ;
template_args        ::= "[" template_arg { "," template_arg } "]" ;
template_arg         ::= type_expression
                       | qua_arg
                       | defpred_identifier
                       | deffunc_identifier ;
qua_arg              ::= identifier { "qua" radix_type } ;
    (* identifier must be a variable whose bound mode inherits from each radix_type;
       see Chapter 18 §18.2.2 for coercion semantics *)
```

**意味論:**

検証器は `by ref₁, ref₂, ...` に遭遇すると、次のことを行います。

1. FOL 式のセットへの各参照を解決します (各参照の種類が解決される方法については、§16.5.1 ～ §16.5.3 を参照してください)。
2. 型理論のコンテキスト (変数の型、その時点でアクティブなクラスター登録) を追加します。
3. ATP を呼び出して、現在の `thesis` が解決されたファクトとコンテキストから導出可能であることを確認します。
4. ATP が設定された制限時間内に導出可能であることを確認した場合、目標は達成されます。それ以外の場合は、「証明されていない」エラーが報告されます。

**`by` を省略:**

`by` が完全に省略されている場合 (`thus φ;` など)、検証器は型コンテキストとアクティブなクラスター登録のみを使用して自動証明を試みます (ユーザーが指定したヒントは使用しません)。これは、単純なトートロジーや型理論的な義務に対しては成功しますが、自明ではない数学的ステップに対しては失敗します。

```mizar
thus 0 >= 0;   :: OK — discharged automatically from Nat_nonneg registration
```

> [！注記]
> 個々のステートメントに適用される `simple_justification` の完全な構文については、「[第15章 (文), §15.8](./15.statements.md#158-正当化フォーム)」を参照してください。

### 16.5.1 単純な引用

**単純な引用**では、一度に 1 つの参考文献に名前を付けます。検証器はそれを単一の FOL 式に解決します。

|種類 |構文 |引用されているもの |
|---|---|---|
|ローカルラベル | `A1` |同じ証明で以前に紹介された命題 (`assume A1: ...`、`A1: φ by ...` など) |
|moduleローカルの結果 | `Lagrange` |現在のmoduleで宣言された `theorem` または `lemma` |
|libraryリファレンス | `mml.algebra.group.Lagrange` | Mizar Math Library の定理 (完全修飾パス経由) |
|template のインスタンス化 | `NatInduction[P]` |特定の述語またはfunctorに適用されるパラメータ化された定理 (スキーマ) |

```mizar
:: Local label + library reference
theorem Cor1:
  for G being finite Group, H being Subgroup of G
    holds card H <= card G
  by Lagrange, mml.number.natural.Nat_divides_le;

:: Template instantiation
thus for n being Nat holds P(n)
  by NatInduction[P], Base, Step;
```

### 16.5.2 グループ化された引用 (`module_path.{th1, th2}`)

同じmoduleからの複数の定理が必要な場合、**グループ化された引用** 構文では共有パスの接頭辞を付けて名前を付け、繰り返しを避けます。

**構文 (EBNF):**

```ebnf
grouped_reference ::= namespace_path ".{" grouped_item { "," grouped_item } "}" ;
grouped_item      ::= identifier [ template_args ] ;
```

`{...}` 内の各項目には、独自の `template_args` が含まれる場合があります。検証器は、`module_path.{th1, th2[T], ...}` を単純な引用として明示的なセット `{module_path.th1, module_path.th2[T], ...}` に解決します。これは単に表記上の省略表現であり、検証動作はそれらを個別にリストすることと同じです。

```mizar
:: Without grouped citation
thus thesis by mml.number.natural.Nat_add_comm,
               mml.number.natural.Nat_add_assoc,
               mml.number.natural.Nat_add_zero;

:: Equivalent grouped citation
thus thesis by mml.number.natural.{Nat_add_comm, Nat_add_assoc, Nat_add_zero};

:: Template args on individual items within the group
thus thesis by mml.algebra.{PermProduct[R qua AddMagma], PermProduct[commutative MulMagma]};

:: Mixed: grouped + local label + individual reference
thus thesis by A1, mml.algebra.group.{Lagrange, Coset_partition}, mml.number.natural.Nat_divides_le;
```

### 16.5.3 一括引用 (`module_path.*`)

単一module内の結果のコレクションから目標が得られ、特定の項目がまだわかっていない場合、**一括引用** 構文 `module_path.*` には、名前付きmoduleのすべての公定理項目が正当化の候補として含まれます。

**構文 (EBNF):**

```ebnf
bulk_reference ::= namespace_path ".*" ;
```

検証器は、`module_path.*` を、そのmoduleによってエクスポートされたすべてのパブリック `theorem` および `lemma` 項目のセットに拡張し、それらを候補公理として ATP に渡します。ステータスの伝播は引き続き適用されます。未変更の項目は、`open`、`assumed`、または `conditional` マテリアルに依存する原因となる一括引用を使用して証明できない場合があります。

```mizar
:: All public theorem items in mml.number.natural are candidates
thus thesis by mml.number.natural.*;

:: Mix: a local label and bulk from one module
thus thesis by A1, mml.algebra.group.*;
```

**トレードオフ:**

| |単純/グループ化された引用 |一括引用 |
|---|---|---|
|検証速度 |高速 (ATP の公理が少ない) |遅い (ATP に関する多くの公理) |
|トレーサビリティ |明示的 — 読者は何が使用されたかを正確に確認できます。暗黙的 — ATP はmodule内の任意の定理を使用できます。
|推奨用途 |公開された証明、library品質のコード |証拠探索、ラピッドプロトタイピング |

公開されたコードまたはlibrary品質のコードの場合は、読者が検証器を実行せずに推論に従うことができるように、一括引用を単純な引用またはグループ化された引用に置き換えることを強くお勧めします。

## 16.6 型システムの証明義務

Mizar の定義 (`func`、`pred`、`mode`、`attr`、`cluster`) は、履行する必要がある暗黙の証明義務を生成します。これらは **正確性条件** と呼ばれます。

### 16.6.1 存在と独自性

`means` を使用したfunctor定義の場合:

```mizar
definition
  let x be A;
  func F(x) -> B means
    :: it is the result value
    P[it, x];
  existence
  proof
    :: Must prove: ex y being B st P[y, x]
    ...
  end;
  uniqueness
  proof
    :: Must prove: for y1, y2 being B st P[y1, x] & P[y2, x] holds y1 = y2
    ...
  end;
end;
```

|状態 | FOLの義務 |
|---|---|
| `existence` | `∀x. (is_A(x) → ∃y. (is_B(y) ∧ P(y, x)))` |
| `uniqueness` | `∀x. (is_A(x) → ∀y₁,y₂. (is_B(y₁) ∧ is_B(y₂) ∧ P(y₁,x) ∧ P(y₂,x) → y₁ = y₂))` |

> [！注記]
> `equals` スタイルのfunctor定義の場合、一意性は自明に満たされるため、省略できます。 [第 10 章 (functor)、§10.6](./10.functors.md#106-正確性の条件)を参照してください。

### 16.6.2 一貫性と互換性

functorまたは属性を再定義する場合は、論理的一貫性を証明する必要があります。

**属性の再定義:**

```mizar
definition
  let B be A;  :: B is a subtype of A
  redefine attr RefinedNameDef: B is Name means P'[it];
  coherence
  proof
    :: Must prove: for x being B holds (OriginalDef[x] iff P'[x])
    ...
  end;
end;
```

|状態 | FOLの義務 |
|---|---|
| `coherence` (属性) | `∀x. (is_B(x) → (OriginalDef(x) ↔ P'(x)))` |
| `coherence` (結果タイプ) | `∀x. is_B(F(x))` (戻り値の型を A から B に絞り込む場合) |

`coherence` 証明がいったん受理されると、その `redefine` は、無関係な overload ではなく、元定義を精密化するものとして扱われます。これにより検証器は、同じ元定義に対する複数の再定義が使用箇所で適用可能なとき、曖昧な overload を報告する代わりにそれらを統合できます（第 19 章）。この統合が信頼されるのは、`coherence` 証明がすでに受理済みの再定義に対してだけです。証明が保留中、または失敗した再定義は、単に対象から外されます。

> [！注記]
> 述語の正確性の条件については、「[第9章 (述語), §9.5](./09.predicates.md#95-正確性の条件)」を参照してください。

### 16.6.3 クラスター登録の義務

クラスターの登録により、型理論の一貫性を確保するための証明義務が生成されます。

**条件付きクラスター:**

```mizar
registration
  cluster empty -> finite for set;
  coherence
  proof
    :: Must prove: for x being set st x is empty holds x is finite
    let x be set;
    assume x is empty;
    thus x is finite by mml.set.Empty_is_finite;
  end;
end;
```

|登録タイプ | FOLの義務 |
|---|---|
|条件付き | `∀x. (is_T(x) ∧ is_A(x) → is_B(x))` |
|実存的 | `∃x. (is_T(x) ∧ is_A(x))` |
|関数 | `∀args. is_A(F(args))` |

> [！注記]
> クラスターの構文と登録の詳細については、「[第17章 (cluster と registration)](./17.clusters_and_registrations.md)」を参照してください。

### 16.6.4 一貫性

`otherwise` 句を持つ conditional definiens (default case を持つ partial
definition) を使う functor definition は、defining conditions が
**mutually consistent** であることを証明しなければなりません。guarded
`if` branch は conflicting values を割り当てる形で overlap してはならず、
`otherwise` branch はそれらの guard の否定と compatible でなければなりません。

**例：**

```mizar
definition
  let x be Real;
  func SignDef: sgn(x) -> Integer means
    it = 1 if x > 0,
    it = 0 if x = 0,
    it = -1 if x < 0
  otherwise it = 0;
  existence proof ... end;
  uniqueness proof ... end;
  consistency
  proof
    :: Must prove: the main condition and `otherwise` do not produce
    :: conflicting values for any input in their overlap
    let x be Real;
    assume A1: not (x > 0 or x = 0 or x < 0);
    thus contradiction by A1, mml.number.real.Real_trichotomy;
  end;
end;
```

|状態 | FOLの義務 |
|---|---|
| `consistency` | conditional branch と `otherwise` default が同じ入力に conflicting values を割り当てることはできません |

> [！注記]
> `consistency` は、`otherwise` が使用される場合にのみ必要です。 `otherwise` のない標準の `means` 定義では、この条件は必要ありません。

## 16.7 論理エンコーディングの詳細

このセクションでは、定理の宣言と証明構造が一階論理 (FOL) にどのようにマッピングされるかについて説明します。式のエンコードについては、[第14章 (式), §14.7](./14.formulas.md#147-論理エンコーディングの詳細) を参照してください。ステートメントのエンコーディングについては、[第15章 (文), §15.11](./15.statements.md#1511-論理エンコーディングの詳細) を参照してください。

### 16.7.1 定理の宣言

|Mizar source | FOL エンコーディング |
|---|---|
| `theorem Th: φ proof ... end;` |検証後に `φ` を公理セットに追加します。
| `lemma L: φ proof ... end;` |検証後に `φ` を公理セットに追加します。
| `open theorem C: φ;` | `φ` を利用可能な理論にオープンマテリアルとして追加し、項目を未解決として記録します。
| `assumed theorem A: φ;` | `φ` を明示的に信頼できる仮定として利用可能な理論に追加します。
| `conditional theorem T: φ by C;` |証明グラフから記録された依存関係ステータスとともに `φ` を利用可能な理論に追加します。
| `theorem Th: for x being T holds P(x) proof ... end;` | `∀x. (is_T(x) → P(x))` |

### 16.7.2 証明義務

|正確性の条件 | FOLの義務 |
|---|---|
| `existence` (functor `F(x) -> B means P[it, x]`) | |
| `uniqueness` (functor `F(x) -> B means P[it, x]`) | |
| `coherence` (属性の再定義) | `∀x. (is_B(x) → (Orig(x) ↔ New(x)))` |
| `coherence` (結果型の再定義) | `∀x. is_B(F(x))` |
| `coherence` (条件付きクラスター) | `∀x. (is_T(x) ∧ is_A(x) → is_B(x))` |
| `existence` (存在クラスタ) | `∃x. (is_T(x) ∧ is_A(x))` |
| `consistency` (`otherwise` の関数) |メイン条件とデフォルトは矛盾しません |

### 16.7.3 スケルトンエンコーディング

証明のスケルトンは、ヤシコウスキー式の自然演繹導出にマッピングされます。

|スケルトン要素 |自然控除ルール |
|---|---|
| `let x be T;` | ∀-導入: 型ガードを使用して新しい変数を導入します |
| `assume φ;` | →-導入: 先行件をローカルファクトとして宣言 |
| `thus φ by J;` |直接導出: Γ ⊢ φ (正当化 J を使用) |
| `take t;` | ∃-導入: 証人を提供してください `t` |
| `per cases; case φᵢ;` | ∨-消去: それぞれのケースに基づく義務を証明する |
| `given x such that φ;` | ∃-削除: 存在的なインスタンスを作成します。

**完全な例:**

```mizar
theorem Th5:
  for x being Real st x > 0
    ex y being Real st y > 0 & y < x
proof
  let x be Real;                    :: ∀-intro
  assume A1: x > 0;                 :: →-intro
  take y = x / 2;                   :: ∃-intro with witness
  thus y > 0 by A1, mml.number.real.Half_positive;
  thus y < x by A1, mml.number.real.Half_less;
end;
```

定理の FOL エンコード:

```
∀x. (is_Real(x) ∧ x > 0 → ∃y. (is_Real(y) ∧ y > 0 ∧ y < x))
```

導出 (ヤシコウスキー スタイル):

```
│ 1. let x : Real                          [∀-intro, type guard]
│ │ 2. assume x > 0                        [→-intro, assume A1]
│ │ 3. x/2 > 0                             [from 2, by Half_positive]
│ │ 4. x/2 < x                             [from 2, by Half_less]
│ │ 5. is_Real(x/2) ∧ x/2 > 0 ∧ x/2 < x   [∧-intro: 3, 4, type inference]
│ │ 6. ∃y. (is_Real(y) ∧ y > 0 ∧ y < x)    [∃-intro with witness x/2]
│ 7. x > 0 → ∃y. (...)                     [→-intro closing: 2–6]
8. ∀x. (is_Real(x) → x > 0 → ∃y. (...))   [∀-intro closing: 1–7]
```

## 16.8 完全な構文 (EBNF)

この章のすべての定理と証明に関連する構文の概要。

```ebnf
(* === Theorem Declarations === *)
theorem_item     ::= [ theorem_status ] theorem_role label_identifier ":" formula [ justification ] ";" ;
theorem_status   ::= "open" | "assumed" | "conditional" ;
theorem_role     ::= "theorem" | "lemma" ;
label_identifier ::= identifier ;

(* === Proof Blocks === *)
justification   ::= simple_justification | proof ;
proof           ::= "proof" reasoning "end" ;
reasoning       ::= { statement } ;

(* === Correctness Conditions === *)
correctness_conditions ::= { correctness_condition } ;
correctness_condition  ::= existence_block
                         | uniqueness_block
                         | coherence_block
                         | compatibility_block
                         | consistency_block ;
existence_block     ::= "existence" justification ";" ;
uniqueness_block    ::= "uniqueness" justification ";" ;
coherence_block     ::= "coherence" justification ";" ;
compatibility_block ::= "compatibility" justification ";" ;
consistency_block   ::= "consistency" justification ";" ;

(* === Justification === *)
(* For full statement syntax, see Chapter 15 (Statements) *)
(* For formula syntax, see Chapter 14 (Formulas) *)
simple_justification ::= [ "by" references ] ;
references           ::= reference { "," reference } ;
reference            ::= label_identifier [ template_args ]
                       | qualified_reference [ template_args ]
                       | grouped_reference
                       | bulk_reference ;
qualified_reference  ::= namespace_path "." identifier ;
grouped_reference    ::= namespace_path ".{" grouped_item { "," grouped_item } "}" ;
grouped_item         ::= identifier [ template_args ] ;
bulk_reference       ::= namespace_path ".*" ;
namespace_path       ::= identifier { "." identifier } ;
template_args        ::= "[" template_arg { "," template_arg } "]" ;
template_arg         ::= type_expression
                       | qua_arg
                       | defpred_identifier
                       | deffunc_identifier ;
qua_arg              ::= identifier { "qua" radix_type } ;
```

## 16.9 実装上の注意事項

- **検証順序**: 証明はトップダウンで検証されます。正当な理由のない `open` 項目と `assumed` 項目は、ステータスが記録された状態で利用可能な理論に追加されます。
- **ステータスの伝播**: 未変更の `theorem` または `lemma` は、`open`、`assumed`、または `conditional` 項目を参照してはなりません。 `conditional` 項目はそれらを参照することができ、その依存関係ステータスは推移的に計算されます。
- **自動証明**: 変更されていない項目の正当化が省略されている場合、検証器は式を自動的に証明しようとします (自明な真の命題、型理論的トートロジーなど)。 `conditional` アイテムには明示的な理由が必要です。 `assumed` 項目には正当な理由があってはなりません。
- **エラー処理**:
    - `proof` ブロックに不当な終了義務が含まれている場合、定理は無効としてマークされます。
    - 未変更の `theorem` または `lemma` から `open`、`assumed`、または `conditional` マテリアルを参照すると、検証器 エラーが発生します。
    - 計算された依存関係ステータスがクリーンなときに項目を `conditional` として宣言すると、修飾子が不要であるため、警告が発行されます。
    - 正当性が証明されていない条件があると、定義の登録がブロックされます。
- **スケルトン マッチング**: 検証器は、各証明ステートメントを式の予想されるスケルトン構造と照合します。マッチングはモジュロ定義拡張で実行されます。構文一致が失敗した場合、検証器は定義を展開しようとします。
- **論文追跡**: 検証器は、各スケルトン位置で現在の `thesis` (端末義務) を追跡し、デバッグ目的でそれを報告する必要があります (`show thesis` アノテーションまたは IDE 統合を介してアクセス可能)。
