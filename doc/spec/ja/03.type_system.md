## 3.1 概要

Mizar Evolution は、型なし集合理論を重ねた **ソフトタイピング システム** を採用しています。型は構文ガイドとして機能し、論理基盤の一部ではありません。すべての式は最終的にセットを表し、型チェックは一階論理の証明義務の生成として実装されます。

> フリーク・ヴィーダイク。 [Mizar のソフトタイプシステム](https://cs.ru.nl/F.Wiedijk/mizar/miztype.pdf) TPHOL 2007

**設計原則**:

* **数学的調整**: 型は数学的実践と一致します。
* **懸念事項の分離**: 型は可読性と検証をガイドしますが、論理コアについては消去されます。
* **効率**: 構文と推論メカニズムは、高速チェックのために最適化されています。
* **一貫性**: すべての型付け構造は、統一された構文と意味論に従います。

## 3.2 中心となる概念

|コンセプト |説明 | | で定義
| --- | --- | --- |
| **基数型** |型階層のベースにあるルート型: 組み込みの `object` および `set`、およびユーザー定義の構造体。 §3.4、第5章 |
| **属性** |既存の型を改良する述語 (パラメータ化される可能性がある) |第6章 |
| **モードの種類** | `mode` 宣言によって導入された名前付きタイプ。それぞれは、基数型 | に適用される属性チェーンに展開されます。第7章 |
| **構造** |名前付きフィールドとプロパティを含む複合基数型 |第5章 |
| **クラスター** |型推論と属性伝播のための登録メカニズム |第17章 |

基数型とモード型は 2 つの互いに素な構文カテゴリを形成し、どちらも型式の **先頭** (属性チェーンの後) に表示されます。モード型は意味的には `attribute_chain radix_type` の短縮形であり、型チェック中にその形式に展開されます (§Ch.7)。

## 3.3 型式

**型式**は、値を分類する方法を指定します。

**構文 (EBNF)**:

```ebnf
type_expression   ::= attribute_chain type_head ;
type_head         ::= radix_type | mode_type ;
attribute_chain   ::= { [ "non" ] attribute_ref } ;
attribute_ref     ::= [ param_prefix ] [ struct_name "." ] attribute_name
                      [ "(" argument_list ")" ] ;
param_prefix      ::= parameter "-" | "(" parameter_list ")" "-" ;
radix_type        ::= builtin_type | struct_name [ type_args ] ;
mode_type         ::= mode_name [ type_args ] ;
type_args         ::= ( "of" | "over" ) argument_list
                    | "[" type_arg_list "]" ;
type_arg_list     ::= type_arg { "," type_arg } ;
type_arg          ::= type_expression | qua_arg ;
qua_arg           ::= identifier { "qua" radix_type } ;
argument_list     ::= term_expression { "," term_expression } ;
builtin_type      ::= "object" | "set" ;

mode_name         ::= qualified_constructor_name ;
struct_name       ::= qualified_constructor_name ;
attribute_name    ::= qualified_constructor_name ;
```

* `mode_name`、`struct_name`、`attribute_name` のそれぞれは、裸の
  コンストラクタ名、または namespace で修飾されたコンストラクタ名
  (例: `std.algebra.Group`、`top.Space`、`mml.structures.1-sorted`) として
  記述できます。`qualified_constructor_name` プロダクションは **Ch.12 §12.7** で
  定義され、その末尾トークンは、Ch.7 (モード)、Ch.5 (構造体)、Ch.6 (属性) に
  よって登録される `constructor_name` (Ch.2 §2.5.2) です。
* `[ struct_name "." ] attribute_name` では、`struct_name` 自体が
  `qualified_constructor_name` であるため、`algebra.RingStr.commutative`
  のような形式が許容されます。すべての `.` の出現は第 2 章 2.5.3 に従い、
  スコープ内の変数によってシャドウされた namespace セグメントはセレクタ
  アクセスとして再解釈されます。
* `+`、`<=`、`\/`、`C*-algebra` のような演算子風の記号名は、type-head や
  属性名ではありません。通常の識別子より読みやすい場合は、`C-star-algebra` の
  ような識別子風のコンストラクタ綴りを使います。`param_prefix` の決定的な
  ハイフン分割順序は Ch.2 §2.5.1 で定義されます。type-head 位置ではこの分割を
  試みないため、`let S be 1-sorted;` は、その構造体名またはモード名が見えて
  いれば `1-sorted` を type head として解決します。
* `parameter`、`parameter_list`は、18章(template)で定義されたtemplateパラメータです。
* bracket 形式の `type_args` は、第18章の完全なtemplate instantiate構文です。`of` および `over` 形式は、宣言がその短縮形を提供する場合の通常の type argument の synonym として受け入れられます。

**例**:

```mizar
set                           :: Built-in radix-type
non empty set                 :: Attribute + radix-type
positive Integer              :: Attribute + mode
Element of G                  :: Dependent mode
associative unital Magma      :: Multiple attributes + mode
Polynomial over R             :: Parameterized structure
```

## 3.4 組み込み型

Mizar は 2 つの組み込み基数型を提供します。

|タイプ |説明 |
| --- | --- |
| `object` |ユニバーサルタイプ。 Mizar ユニバースのすべての値 (構造を含む)。 |
| `set` | `object` のサブタイプ。 ZFC の意味での数学的集合。 |

**タイプ階層**:

```
object
├── set
│   └── (user-defined modes based on set)
└── (user-defined structures)
```

* `struct` によって定義された構造は、`object` のサブタイプですが、`set` の**ではありません**。
* この区別により、抽象代数構造を一般集合から分離することができます。

## 3.5 サブタイプ化と拡張

`S` のメンバーシップが `T` のメンバーシップを意味する場合、`S` は `T` の **サブタイプ** です。

* **拡張**: 値をスーパータイプに自動的に変換します。
* **絞り込み**: サブタイプに変換 — 証明義務のある `reconsider` (Ch.15) が必要です。

**例**:

```mizar
:: Widening (automatic)
let n be Nat;
:: n can be used where Integer is expected

:: Narrowing (explicit)
let x be Integer such that x >= 0;
reconsider x as Nat by mml.number.real.Nat_is_nonneg_Real;  :: Proof required
```

**サブタイプ チェーン** (各型は右側の型のサブタイプです):

* `Prime`、次に `Integer`、その後 `Number`
* `non empty set`、次に `set`、その後 `object`

## 3.6 クラスター (概要)

クラスターは、自動型推論と伝播を可能にする登録メカニズムです。

|クラスターの種類 |目的 |例 |
| --- | --- | --- |
| **existential cluster** |型付き値の存在を証明する | `cluster empty finite set;` |
| **conditional cluster** |属性の影響を宣言する | `cluster empty -> finite set;` |
| **functional cluster** |functorの結果に型を割り当てる | `cluster f(x) -> positive for Real;` |

詳細な構文と例は、**第 17 章 (クラスターと登録)** に記載されています。

## 3.7 論理エンコーディングの詳細

このセクションでは、型が一階論理にどのようにマップされるかについて説明します。

### 3.7.1 述語としての型

各型 `T` は、単項述語 `is_T(x)` としてエンコードされます。

|Mizar source | FOL エンコーディング |
| --- | --- |
| `x is set` | `is_set(x)` |
| `x is non empty set` | `is_set(x) ∧ ¬ is_empty(x)` |
| `x is Element of G` | `is_Element_of(x, G)` |

(この章の他の場所で使用されている名前 `Real`、`Integer`、および類似の名前は、組み込みではなく MML で定義されたモード タイプです。それらの FOL エンコーディングは同じ `is_T` スキーマに従います。)

### 3.7.2 暗示としてのサブタイプ

「`S` は `T` のサブタイプです」は、普遍的な含意 `∀x. is_S(x) ⇒ is_T(x)` としてエンコードされます。

**例**: 「`Prime` は `Integer` のサブタイプです」は、`∀x. is_Prime(x) ⇒ is_Integer(x)` に対応します。

### 3.7.3 ATP の型消去

ATP (TPTP 形式) にエクスポートすると、型は消去されます。

1. 変数宣言は型なしになります: `let x be Real` → `x`
2. 型の仮定が仮説になる: `is_Real(x) ⇒ ...`
3. 属性チェーンが接続詞になる

## 3.8 完全な構文 (EBNF)

この章の型関連の構文の概要。

```ebnf
(* Type Expressions *)
type_expression   ::= attribute_chain type_head ;
type_head         ::= radix_type | mode_type ;
attribute_chain   ::= { [ "non" ] attribute_ref } ;
attribute_ref     ::= [ param_prefix ] [ struct_name "." ] attribute_name
                      [ "(" argument_list ")" ] ;
param_prefix      ::= parameter "-" | "(" parameter_list ")" "-" ;

radix_type        ::= builtin_type | struct_name [ type_args ] ;
mode_type         ::= mode_name [ type_args ] ;

type_args         ::= ( "of" | "over" ) argument_list
                    | "[" type_arg_list "]" ;
type_arg_list     ::= type_arg { "," type_arg } ;
type_arg          ::= type_expression | qua_arg ;
qua_arg           ::= identifier { "qua" radix_type } ;
argument_list     ::= term_expression { "," term_expression } ;

builtin_type      ::= "object" | "set" ;

attribute_name    ::= qualified_constructor_name ;  (* registered by Chapter 6 definitions *)
mode_name         ::= qualified_constructor_name ;  (* registered by Chapter 7 declarations *)
struct_name       ::= qualified_constructor_name ;  (* registered by Chapter 5 declarations *)

(* qualified_constructor_name: see Chapter 12 §12.7 *)
(* constructor_name: see Chapter 2 §2.5.2 *)
(* parameter, parameter_list: see Chapter 18 (Templates) *)
```
