この章では、Mizar Evolution 検証器の診断出力、つまりエラー メッセージの形式、エラー コード システム、フェーズごとのエラーの分類、および証明義務が失敗したときに検証器が提供するガイダンスについて説明します。設計目標は **実用的な診断** です。すべてのメッセージが、何が問題で、どこで、次に何を試すべきかを作成者に伝える必要があります。

## 22.1 エラーメッセージの設計原則

### 22.1.1 エラーコード体系(Eシリーズ)

検証器によって発行されるすべての診断には、**固有の安定したコード**が含まれています。コードはフェーズと重大度ごとにグループ化されています。

|プレフィックス |重大度 |フェーズ |
|---|---|---|
| `E` |エラー — 検証を続行できません |構文、型、証明、論理的一貫性 |
| `W` |警告 - 検証は続行されますが、何か不審な点があります |全フェーズ |
| `I` |情報 — 問題ありません。追加情報が必要です |ヒントを表示する (§21.7) |

コードの形式は `Xnnnn` で、`X` は重大度のプレフィックス、`nnnn` は 4 桁の 10 進数です。番号は永続的に割り当てられます。コードは、割り当てられた後に別の診断に再利用されることはありません。診断が検証 toolから削除されると、そのコードは廃止され、番号が再発行されることはありません。

各コードには、ドット区切りのカテゴリ表記の **意味名** もあります。セマンティック名はバージョン間で変更される場合があります (タイプミスを修正したり、明確さを改善するためなど)。数値コードは安定した識別子です。toolは数値コードを入力する必要があります。人間はどちらの形式でも使用できます。

```
error[E0101]: type mismatch (type.mismatch)
  --> src/mylib.miz:14:16
   |
14 |    var n := sqrt(2) as Integer;
   |             ^^^^^^^ inferred type: Real; expected: Integer
   |
help: use a narrowing cast with justification: `sqrt(2) as Integer by ...`
```

フェーズごとのコード範囲:

|範囲 |フェーズ |
|---|---|
| E0001–E0099 |語彙/構文エラー |
| E0100–E0199 |タイプエラー |
| E0200–E0299 |overloadの解決とtemplateのエラー |
| E0300–E0399 |証明および ATP エラー |
| E0400–E0499 |論理不一致エラー |
| E0500–E0599 |アルゴリズム検証エラー |
| W0001–W0099 |構造上の警告 |
| W0100–W0199 |証明および ATP に関する警告 |
| W0200–W0299 |アルゴリズムと契約に関する警告 |
| I0001–I0099 |情報表示 (§21.7) |

### 22.1.2 sourceスパンとコンテキストの表示

すべての診断は、**スパン**、つまりsource file内の半オープン範囲 `[start, end)` を使用してsource の場所を識別し、行番号と列番号 (どちらも 1 から始まる) で表されます。

表示形式は次のとおりです。

```
severity[CODE]: short description (semantic.name)
  --> path/to/file.miz:line:col
   |
LL | source text of the relevant line
   | ^^^^^^ annotation under the span
   |
   = note: additional context
```

ルール:
- `-->` 行は、ワークスペース ルート (§23.3) を基準としたfile パス、行番号、およびプライマリ スパンの **start** の列を示します。
- 最大 1 つの **プライマリ スパン**が `^` の下線付きで表示されます。セカンダリ スパン (競合するoverloadの定義サイトなど) は、別の `|` ブロック上に `-` の下線で表示されます。
- スパンが複数の行にまたがる場合、最初と最後の行が間に `...` を挟んで表示されます。
- すべての列番号は、バイト数ではなく Unicode スカラー値でカウントされます。

**セカンダリ スパン** 相互参照関連のsource の場所:

```
error[E0201]: ambiguous overload (resolve.ambiguous_symbol)
  --> src/algebra.miz:42:5
   |
42 |   a + b
   |   ^^^^^ two candidates match
   |
  --> src/ring.miz:7:3
   |
 7 |   func add(a, b: Element of R) -> Element of R;
   |   ---- candidate 1
   |
  --> src/nat.miz:11:3
   |
11 |   func add(a, b: Nat) -> Nat;
   |   ---- candidate 2
   |
help: use a qualified name or `qua` to disambiguate
```

### 22.1.3 修正提案と `help:` 注釈

検証器が修正の可能性を判断できる場合、診断に 1 つ以上の `help:` 行を追加します。修正提案は信頼度によって分類されます。

|キーワード |意味 |
|---|---|
| `help:` |信頼性の高い提案。それを適用するのがおそらく正しいでしょう。 |
| `note:` |特定の修正のないコンテキスト情報。 |
| `hint:` |信頼性の低い提案。いくつかの可能性のうちの 1 つ。 |

修正提案は、次のパターンに従っている場合、機械で読み取り可能になります。

```
help: replace `old text` with `new text`
help: insert `text` after line LL
help: remove the declaration at line LL
```

IDE コード アクション (§23.6.1) はこれらのパターンを解析して、ワンクリックで修正を提供します。修正を単純なテキスト操作として表現できない場合は、代わりに `help:` 行で自然言語の説明が表示されます。

```
error[E0102]: narrowing cast requires proof (type.narrowing_requires_proof)
  --> src/mylib.miz:7:15
   |
 7 |   var n := k as Nat;
   |                 ^^^ Integer cannot be narrowed to Nat without proof
   |
help: add a justification: `var n := k as Nat by non_negative_lemma;`
note: alternatively, supply a proof block:
      `var n := k as Nat
       proof
         k >= 0 by ...;
         thus k is Nat;
       end;`
```

## 22.2 構文エラー

構文エラーは、型チェックや証明検証が開始される前に、解析中に検出されます。 E0001 ～ E0099 の範囲のコードが含まれます。

### 22.2.1 不正なtokenと予期しないシンボル

**E0001 — `syntax.unexpected_token`**。parserは、現在のコンテキストで構文構造を開始または継続できないtokenを検出しました。

```
error[E0001]: unexpected token `:=` (syntax.unexpected_token)
  --> src/lib.miz:5:12
   |
 5 |   theorem := foo;
   |           ^^ expected an identifier after `theorem`
```

**E0002 — `syntax.malformed_literal`**。数値または文字列リテラルは、§2.3 の語彙規則に準拠していません。

```
error[E0002]: malformed numeric literal (syntax.malformed_literal)
  --> src/lib.miz:3:10
   |
 3 |   const x = 1_000_000_;
   |             ^^^^^^^^^^^ trailing underscore in integer literal
```

**E0003 — `syntax.unexpected_end_of_file`**。オープンコンストラクトがまだ保留されている間にsourcefileが終了しました。

```
error[E0003]: unexpected end of file (syntax.unexpected_end_of_file)
  --> src/lib.miz:20:1
   |
20 | (end of file)
   | ^ expected `end` to close `definition` opened at line 3
```

### 22.2.2 構造エラー (`end` の欠落、区切り文字の不一致)

**E0010 — `syntax.missing_end`**。 `definition`、`proof`、`do...end`、または同様のブロックが開かれましたが、閉じられていませんでした。

```
error[E0010]: missing `end` for `proof` block (syntax.missing_end)
  --> src/lib.miz:18:1
   |
 8 |   proof
   |   ----- `proof` opened here
   |
18 |   theorem next_theorem : ...;
   |   ^^^^^^^ expected `end` before new top-level item
```

**E0011 — `syntax.unmatched_delimiter`**。括弧、括弧、または `do` キーワードには、一致する終わりがありません。

```
error[E0011]: unmatched `(` (syntax.unmatched_delimiter)
  --> src/lib.miz:12:8
   |
12 |   func f(a, b -> Nat;
   |          ^ no matching `)`
```

**E0012 — `syntax.reserved_keyword_as_identifier`**。予約キーワード (§2.4) は、識別子が必要な場合に使用されます。

> [！注記]
> parserは、すべての構文エラーの後に **エラー回復** を適用します。parserは、同期ポイント (通常は `;`、`end`、またはトップレベルのキーワード) が見つかるまでtokenをスキップしようとし、その後、解析を再開します。これにより、1 回のコンパイル実行で、最初の構文エラーで停止するのではなく、複数の独立した構文エラーを報告できるようになります。

## 22.3 型エラー

型エラーは、型チェックと型推論中に検出されます (第 3 章と第 8 章)。 E0100 ～ E0199 の範囲のコードが含まれます。

### 22.3.1 型の不一致

**E0101 — `type.mismatch`**。型 `T` の式は、型 `U` が必要な場所に表示されますが、`T` は `U` のサブタイプではなく、強制パスは存在しません。 E0102 (ナローキャストは存在するが証拠が欠けている) とは異なり、このエラーは型がまったく関連していない場合に発生します。型をブリッジするキャストはありません。

```
error[E0101]: type mismatch (type.mismatch)
  --> src/lib.miz:9:7
   |
 9 |   var n as Plane := some_line;
   |            ^^^^^ ^^^^^^^^^^^ inferred type: Line; required: Plane; no coercion path registered
   |
note: `Line` and `Plane` are unrelated modes; no widening, narrowing, or coercion applies
```

**E0102 — `type.narrowing_requires_proof`**。縮小強制 (`Integer` から `Nat` など) が正当な理由なく書き込まれています。拡張強制 (例: `Nat` から `Integer`) は黙って受け入れられます。縮小強制には、`by` 位置揃えまたは `proof ... end` ブロック (§20.1.3) のいずれかが必要です。

**E0103 — `type.sethood.missing`**。フランケル集合内包表記 `{ f(x) where x is T : P(x) }` が使用されていますが、`sethood` は型 `T` に登録されていません (§21.7.6)。内包表記は型チェック時に拒否されます。

```
error[E0103]: `sethood` not registered for type `MyType` (type.sethood.missing)
  --> src/lib.miz:22:7
   |
22 |   const S = { f(x) where x is MyType : P(x) };
   |              ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ Fraenkel comprehension requires sethood
   |
help: add a `sethood` proof to the mode definition of `MyType`:
      `definition
         mode MyType is ...;
         sethood
         proof
           :: ex S being set st for x being MyType holds x in S
           ...
         end;
       end;`
```

### 22.3.2 型推論の失敗

**E0110 — `type.inference_conflict`**。式の異なる部分から派生した 2 つの制約には、互換性のない型が必要です。これは、同じ型パラメーターが 2 つの互換性のない型 (どちらも他方のサブタイプではない) で統合されている場合に、template/スキーマ推論で発生します。そのため、「間違った」式サイトが 1 つも存在しません (特定の不一致用語で起動する E0101 と対比してください)。

```
error[E0110]: type inference conflict (type.inference_conflict)
  --> src/lib.miz:8:11
   |
 8 |   const z = combine(some_list, some_nat);
   |             ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ type parameter `T` inferred as both `List of Nat` and `Nat`; no common supertype
   |
   = note: `T` = `List of Nat` (from argument 1 at col 19)
   = note: `T` = `Nat`         (from argument 2 at col 30)
   |
help: supply an explicit type for one or both arguments
```

> [！注記]
> E0110 は E0102 とは異なります。キャストは存在するが証明がない場合ではなく、推論エンジンがナローイング キャスト解決パスを持たずに矛盾する制約を導出した場合に起動されます。
>
> 省略されたtemplate引数が使用可能な引数の型から決定できない場合は、別個の汎用型推論の失敗としてではなく、§22.3.4 で E0203 (`template.argument_omitted_not_inferable`) として報告されます。

### 22.3.3 モードおよび属性の違反

**E0120 — `type.mode_mismatch`**。モード `M` の用語は、モード `N` が必要な場合に使用され、登録されたモード拡張は適用されません (§7.4)。

```
error[E0120]: mode mismatch (type.mode_mismatch)
  --> src/lib.miz:12:22
   |
12 |   const r = ordinal_add(3, omega);
   |                          ^ mode `Nat`; required: `Ordinal`; no registered widening from `Nat` to `Ordinal`
   |
help: register a mode widening: `registration cluster Nat -> Ordinal; end;`
note: alternatively, introduce an explicit coercion at the call site
```

**E0121 — `type.attribute_required`**。操作では、引数に特定の属性 (`non zero`、`finite` など) を渡す必要がありますが、その属性は指定された型に登録されていません。検証器は、このエラーを発行する前に、クラスター登録で属性を放電できるかどうかをチェックします。これは、クラスターの解決が決定的に失敗した場合にのみ発行されます。これは E0102 とは異なります。E0102 はサブ型の絞り込み (`Integer → Nat` など) に関するものですが、E0121 はそれ以外は正しい型の属性の欠落に関するものです。

```
error[E0121]: attribute `non zero` required (type.attribute_required)
  --> src/lib.miz:15:16
   |
15 |   const q = a / b;
   |                ^ denominator must be `non zero`
   |
note: `b` has type `Integer`; `Integer` does not carry `non zero` by any registered cluster
help: add a precondition: `requires b <> 0`
```

**E0122 — `type.attribute_contradiction`**。型式の属性の組み合わせは、クラスター登録システムによって拒否されます。これは、次の 2 つの異なる状況で発生します。

- **矛盾した属性。** 用語は属性 `A` と属性 `B` を同時に持つ必要があり、`A` と `B` は登録された不一致クラスター (§17.5) によって相互に排他的です。
- **存在クラスタがありません。** 結合タイプ (`finite odd Subset of NAT` など) には、その型のオブジェクトが存在することを証明する登録された存在クラスタ (§17.3) がありません。存在するクラスターがないと、検証器はその型が存在することを確認できず、その使用を拒否します。

```
error[E0122]: contradictory attributes (type.attribute_contradiction)
  --> src/lib.miz:7:7
   |
 7 |   var s as infinite finite set := ...;
   |            ^^^^^^^^ ^^^^^^ attributes `infinite` and `finite` are mutually exclusive
   |
   = note: registered inconsistency: `cluster infinite finite -> ;` (§17.5)
```

```
error[E0122]: no existential cluster for type (type.attribute_contradiction)
  --> src/lib.miz:12:13
   |
12 |   var s as finite odd Subset of NAT := ...;
   |            ^^^^^^^^^^^^^^^^^^^^^^^ no existential cluster registered for `finite odd Subset of NAT`
   |
help: register an existential cluster: `cluster finite odd for Subset of NAT;` with a proof of existence (§17.3)
```

> [！注記]
> 2 つのサブケースはどちらも、クラスター登録データベース (§17) が型式を承認できないことを示しています。存在クラスタが欠落しているケースは論理矛盾ではありません。それは登録されたlibrary の不完全さです。適切なクラスター登録を追加すると、用語自体を変更することなくエラーが解決されます。

### 22.3.4 過負荷解決の失敗

**E0201 — `resolve.ambiguous_symbol`**。overloadの解決 (第 19 章) は、指定された引数に対して同様に適切に型付けされた 2 つ以上の候補を見つけます。検証器は追加情報がなければそれらを選択することはできません。

```
error[E0201]: ambiguous overload for `f` (resolve.ambiguous_symbol)
  --> src/lib.miz:8:11
   |
 8 |   const z = f(1, 1);
   |             ^^^^^^^^ two equally-ranked candidates; argument types are both `Nat`
   |
  --> src/algebra.miz:3:3
   |
 3 |   func f(x: Integer, y: Real) -> Real;
   |   ---- candidate 1
   |
  --> src/algebra.miz:7:3
   |
 7 |   func f(x: Real, y: Integer) -> Real;
   |   ---- candidate 2
   |
help: supply explicit types to disambiguate: `f(1 as Integer, 1 as Real)`
```

**E0202 — `resolve.no_viable_overload`**。指定された引数の型の型チェックに生き残る候補はありません。これは、関数がインポートされていないか、引数の型が間違っていることを意味している可能性があります。

**E0203 — `template.argument_omitted_not_inferable`**。template (§18) はすべてのスキーマ パラメーターを指定せずに使用されており、欠落しているパラメーターはコンテキストから推測できません。

```
error[E0203]: template argument cannot be inferred (template.argument_omitted_not_inferable)
  --> src/lib.miz:12:3
   |
12 |   SomeTemplate(x);
   |   ^^^^^^^^^^^^ schema parameter `T` is not inferable from argument types
   |
help: supply the argument explicitly: `SomeTemplate[Nat](x)`
```

## 22.4 証明エラーと ATP 診断

証明エラーは、検証条件 (VC) の生成および ATP 相互作用中に検出されます (§21.8)。 E0300 ～ E0399 の範囲のコードが含まれます。 ATP 診断は**実用的**になるように設計されています。すべての失敗にはトリアージ分類 (§22.4.2) が含まれ、可能な場合は具体的な提案が含まれます。

### 22.4.1 証明義務違反

**E0301 — `proof.by.search_exhausted`**。 `by` ステップが失敗しました。検証器は証拠を見つけることなく検索戦略を使い果たしました。 ATP が呼び出されましたが、リソース バジェット内で証明書が返されませんでした。

```
error[E0301]: proof obligation not discharged (proof.by.search_exhausted)
  --> src/lib.miz:34:3
   |
34 |   thus n + 0 = n by Nat_add;
   |   ^^^^^^^^^^^^^^^^^^^^^^^^^^ could not prove: n + 0 = n
   |
note: goal: n + 0 = n
note: ATP: Vampire (timeout 10s), E (timeout 10s) — both exhausted
hint: try adding `add_0_right` to the `by` clause
```

**E0302 — `proof.by.missing_fact`**。検証器は、目標が証明可能である可能性が高いというbackend固有の証拠を持っていますが、1 つ以上の必要な補題が範囲外であるか、現在の `by` 句からは到達できません。

```
error[E0302]: likely missing lemma (proof.by.missing_fact)
  --> src/lib.miz:41:3
   |
41 |   thus result = Eval(t, v);
   |   ^^^^^^^^^^^^^^^^^^^^^^^^^ goal appears provable but requires additional facts
   |
note: goal: result = Eval(t, v)
note: candidate lemmas not in scope:
        poly_eval_add  (src/poly.miz:88)
        poly_eval_mul  (src/poly.miz:102)
help: add to `by` clause: `by poly_eval_add, poly_eval_mul`
```

**E0303 — `proof.obligation.open`**。証明ブロックは、1 つ以上のサブゴールがまだ開いたまま終了しました。それぞれの未解決の目標は、別個の添付メモとして報告されます。

```
error[E0303]: proof block has 2 open goals (proof.obligation.open)
  --> src/lib.miz:50:3
   |
50 |   end;
   |   ^^^ proof block closed with goals remaining
   |
note: open goal 1: n > 0
note: open goal 2: m divides n
```

### 22.4.2 失敗のトリアージ: 反例、補題の欠落、または計算制限

証明義務が失敗した場合、検証器は報告する前にその失敗を 3 つのカテゴリのいずれかに分類します。トリアージの結果によって、どの診断コードと提案が出力されるかが決まります。

|トリアージ結果 |状態 |主なアクション |
|---|---|---|
| **反例** |反例モデルが見つかりました (§22.4.3)。 |ステートメントを修正するか、欠落している前提条件を追加します。 |
| **補題がありません** |backendのフィードバックまたはリプレイヒューリスティックは、追加の明示的な事実が必要である可能性があることを示しています。 |不足している補題を `by` 句に追加します。 |
| **計算制限** | ATP がtimeoutしたか、公理バジェットに達しました。反例は見つかりませんでした。 |予算を増やすか、目標を分割するか、中間の `assert` ステップを追加します。 |

トリアージは、最初の ATP コールが失敗した後に実行されます。反例検索 (§22.4.3) によって時間予算内でモデルが返された場合、優先順位は `Counterexample` です。それ以外の場合、検証器はbackend固有のフィードバックとリプレイ ヒューリスティックを使用して、障害が `Missing lemma` として最もよく説明されるかどうかを判断します。そのような証拠が利用できない場合、トリアージは `Computation limit` に戻ります。

### 22.4.3 反例生成の統合

`by` ステップまたは `proof ... end` ブロック内の義務が失敗した場合、検証器はオプションで **反例ジェネレーター** (Nitpick や CVC5 の有限モデル ファインダーなど) を呼び出して、目標を偽るモデルを検索します。これは検証器構成によって制御されます。有限領域の量指定子を含む目標ではデフォルトで有効になり、大規模な代数目標では無効になります。

反例が見つかった場合:

- 診断コードは **E0310 — `proof.counterexample.found`** です。
- モデルは、具体的な値の割り当てのセットとして表示されます。
- 診断メッセージには、どの仮説または前提条件が欠落している可能性があるかが示されます。

> [！注記]
> 反例検索が失敗しても、目標が偽であることを意味するものではありません。これは、検索予算内で目標を反駁できなかったことを意味します。その後、トリアージは欠落補題または計算限界のカテゴリに分類されます。

本物の反例 (目標を偽るモデル) が見つかった場合、検証器は次のように出力します。

```
error[E0310]: statement is false — counterexample found (proof.counterexample.found)
  --> src/lib.miz:15:3
   |
15 |   thus n * 2 = n + n + 1;
   |   ^^^^^^^^^^^^^^^^^^^^^^^ falsified
   |
note: counterexample:
        n := 1
        n * 2 = 2
        n + n + 1 = 3
        2 ≠ 3
help: the statement is incorrect; check the formula
```

### 22.4.4 前提の選択と補題の提案

初期実装では、**前提条件の選択は完全に ATP backend**に委任されます。検証器は、独立したlibrary 全体のランキング パスを実行しません。これは、§21.4.1 および §21.8 の証明検索ポリシーと一致します。作成者は、`by` 節で補題を明示的に引用する責任がありますが、ATP は受け取った公理に内部的に優先順位を付けることができます。

したがって、このセクションでは、限られたベストエフォート型の意味で「補題提案」を使用します。

- **ATP 統合前提選択:** Vampire や E Prover などのbackendは、渡された公理セットに内部 SInE スタイルのフィルタリングまたはランキングを適用する場合があります。
- **検証器側の提案 (オプション、backend依存):** 実行の失敗または再試行の成功後にbackendが十分な情報を公開する場合、検証器は候補の補題を診断メモとして表示することがあります。これは出力層の便宜的なものであり、別個の証明検索フェーズではありません。

このようなbackend由来の候補が利用可能な場合、それらはsource の場所とともに `note:` 行として診断に添付されます。

```
note: candidate lemmas (by relevance score):
  1. add_0_right       (src/nat.miz:45)   — score 0.92
  2. add_comm          (src/nat.miz:51)   — score 0.87
  3. add_assoc         (src/nat.miz:58)   — score 0.71
```

検証器は、そのような再試行戦略が構成されている場合にのみ、拡張された明示的なファクト セットを使用して **再試行** をオプションで試行できます。再試行が成功すると、エラーは提案にアップグレードされます。

```
help: the goal is provable with the following addition:
      `thus n + 0 = n by add_0_right;`
```

再試行も失敗した場合、backendから派生した候補リストは `hint:` としてのみ表示され、backendが有用な前提情報を公開しない場合は完全に省略される可能性があります。

`@missing_facts` ビルド アーティファクト (§23.5.2) は、検証器がbackend feedbackから回復できた候補ファクトを記録します。したがって、ATP が使用可能な前提情報を公開しなかった義務については空になる可能性があります。

### 22.4.5 ATP timeoutと計算制限の処理

**E0320 — `proof.atp.timeout`**。すべての ATP backendには義務ごとの時間予算があります (デフォルト: backendごとに 30 秒、マニフェストで構成可能、§23.1)。証明書なしで予算が使い果たされると、義務はtimeoutとして報告されます。

```
error[E0320]: ATP timeout (proof.atp.timeout)
  --> src/lib.miz:88:3
   |
88 |   thus thesis;
   |   ^^^^^^^^^^^^ Vampire (30s), E (30s) — all backends timed out
   |
note: goal: for n being Nat holds Fib(n) + Fib(n+1) = Fib(n+2)
note: triage: computation limit — no counterexample found
hint: split into subgoals with `assert` statements
hint: increase the ATP budget: `@proof_hint(solver: vampire)` with longer timeout in mizar.pkg
hint: add intermediate lemmas to reduce the goal complexity
```

**E0321 — `proof.atp.axiom_budget_exceeded`**。義務に対して組み立てられた公理の数が、構成された `max_axioms` 制限 (§21.4.1、§21.8) を超えています。検証器は、必要な事実が予算の上限によって除外されている可能性があるため、証明の試みはもはや信頼できなくなるため、これを警告にサイレントにダウングレードするのではなく、失敗した義務のエラーとして報告します。

```
error[E0321]: axiom budget exceeded (proof.atp.axiom_budget_exceeded)
  --> src/lib.miz:44:3
   |
44 |   thus thesis by ...;
   |   ^^^^^^^^^^^^^^^^^^^ assembled axiom set has 312 items, exceeding `max_axioms: 128`
   |
help: cite specific lemmas in the `by` clause to reduce the axiom set
hint: add or raise `@proof_hint(max_axioms: 256)` on this step if the larger budget is intentional
hint: alternatively, increase the package-level `max_axioms` limit in `mizar.pkg`
```

### 22.4.6 既知の ATP 制限と正直な開示

検証器は、kernel検証なしで、ATP の失敗がステートメントが false であることを意味するわけでも、ATP の成功がステートメントが true であることを意味するわけでもありません。次の情報は診断出力で明示的に示されます。

- **timeoutは虚偽を意味するものではありません。** `proof.atp.timeout` が発行されると、診断には `note: a timeout does not mean the goal is false` が含まれます。作成者は、中間ステップで ATP を支援するように努めるべきです。
- **外部で証明された証明にはフラグが付けられます。** 証明器がkernel検証可能な証明書 (§21.7.5) を生成できない場合、証明にはビルド アーティファクト (§23.5.1) で `externally_attested` のマークが付けられ、`W0102` 警告が発行されます。
- **AC 推論の制限。** 義務に AC オペレーターが関与しているが、選択されたbackendがネイティブ AC 推論をサポートしていない場合、検証器は `hint: switch to a backend with AC support (vampire or e)` を発行します。
- **ATP の不完全性の開示。** 一般的な 一階論理は半分決定可能です。ATP に正しい公理が与えられたとしても、有効な目標は固定リソース予算内では証明できない可能性があります。反例が見つからず、ATP がtimeoutになると、検証器は単に失敗を報告するのではなく、timeoutが目標が false であることを示していないことを説明する `note:` を発行します。

## 22.5 論理矛盾

論理矛盾は、蓄積された公理、登録、または契約が矛盾していることを検証器が証明できる場合に検出されます。これらは最も重大なエラーであり、fileの数学的内容が不適切であることを示しています。 E0400 ～ E0499 の範囲のコードが含まれます。

### 22.5.1 矛盾する公理

**E0401 — `logic.contradictory_axioms`**。スコープ内の `axiom` 宣言の組み合わせにより、検証器 (または ATP) は `False` を導出できます。これは、少なくとも 1 つの公理が間違っているか、他の公理と矛盾していることを意味します。

```
error[E0401]: contradictory axioms detected (logic.contradictory_axioms)
  --> src/lib.miz:30:1
   |
30 |   axiom all_zero : for n being Nat holds n = 0;
   |   ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ this axiom, together with
   |
  --> src/lib.miz:25:1
   |
25 |   axiom succ_ne_zero : for n being Nat holds succ(n) <> 0;
   |   ---- ... makes the system inconsistent (1 = succ(0) <> 0 = 1)
   |
note: ATP derived `False` from these two axioms in 0.3s (Vampire)
```

矛盾した系ではどの定理も証明可能になるため、矛盾する公理は警告ではなくエラーです。検証器は、影響を受けるmoduleのさらなる証明チェックを停止します。

### 22.5.2 循環定義

**E0410 — `logic.circular_definition`**。 `func`、`pred`、`mode`、または `attr` 定義は、明示的な `algorithm` ブロック (終了証明による再帰をサポートします、§20.8) の外側でそれ自体を参照します。 `definition` レイヤーの直接循環定義は無条件に拒否されます。

```
error[E0410]: circular definition (logic.circular_definition)
  --> src/lib.miz:8:3
   |
 8 |   func f(n) -> Nat equals g(n) + 1;
   |   ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
   |
  --> src/lib.miz:12:3
   |
12 |   func g(n) -> Nat equals f(n) - 1;
   |   ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ `f` and `g` are mutually circular
   |
note: use `algorithm` with `decreasing` for recursive definitions (§20.8)
```

**E0411 — `logic.circular_cluster`**。クラスター登録により、属性継承グラフにサイクルが作成されます。属性 `A` は `B` を暗示し、`B` は `A` を暗示します。ここで、`A` と `B` は両方とも矛盾する要件を伴います。これは、許可されている良性の双方向登録とは異なります。

### 22.5.3 契約違反 (`requires`、`ensures`、`assert`)

**E0420 — `vc.postcondition.return`**。検証器は、`ensures` 節が `return` ステートメントで成立することを証明できません (§20.4.1)。

```
error[E0420]: postcondition not provable at return (vc.postcondition.return)
  --> src/lib.miz:18:5
   |
18 |     return -n;
   |     ^^^^^^^^^^ could not prove `result >= 0` when `n < 0`
   |
note: goal: -n >= 0
note: available: n < 0
note: ATP: Vampire — timeout; E — timeout
hint: the postcondition `ensures result >= 0` requires `n < 0 => -n >= 0`
hint: this holds over `Integer` but may not be registered; try `by neg_nonneg`
```

**E0421 — `vc.assert.failed`**。アルゴリズム本体内の `assert` ステートメント (§20.4.2) を証明できませんでした。

**E0422 — `vc.precondition.call_site`**。呼び出しサイトでは、検証器は呼び出し先の `requires` 句が満たされていることを証明できません。

```
error[E0422]: precondition not provable at call site (vc.precondition.call_site)
  --> src/lib.miz:35:15
   |
35 |   const q = divide(a, b);
   |                       ^ cannot prove `b <> 0`
   |
note: `divide` requires: b <> 0
note: available hypotheses do not establish this
help: add `requires b <> 0` to the enclosing algorithm, or add `assert b <> 0 by ...;` before the call
```

**E0423 — `vc.loop.establish`**。ループ不変条件 (§20.5.1) は、最初の反復の前に成立することを証明できません。

**E0424 — `vc.loop.maintain`**。ループの不変条件は、ある反復から次の反復まで維持されることを証明することはできません。

**E0425 — `vc.loop.decrease`**。終了尺度 (§20.5.2) は、反復ごとに厳密に減少することを証明することはできません。また、以下に制限されることも証明できません。

**E0426 — `vc.recursion.decrease`**。 `decreasing` を使用した再帰アルゴリズムの場合、再帰呼び出しサイトで終了尺度が減少することは証明できません。

### 22.5.4 クラスター登録の不一致

**E0430 — `logic.cluster.inconsistency`**。クラスターの登録により、libraryですでに利用可能な事実と矛盾する公理が導入されます。これは、たとえば、条件付きクラスター (§17.4) が `¬A` を満たすことがすでにわかっているオブジェクトの属性 `A` を導出する場合、または別のクラスター登録によって同じ矛盾が導出可能になる場合に発生する可能性があります。

```
error[E0430]: cluster registration creates inconsistency (logic.cluster.inconsistency)
  --> src/lib.miz:45:3
   |
45 |   cluster EmptyImpliesNonEmpty: empty -> non empty for set;
   |   ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ registration derives `non empty` from `empty`
   |
note: `{}` is an `empty set`
note: applying the registration yields: `{}` is `non empty`
note: contradiction: `{}` cannot be both `empty` and `non empty`
```

## 22.6 警告メッセージ

警告によって検証の完了が妨げられることはありません。これらは、意図的ではない、最適ではない、または非推奨である可能性が高いパターンを示します。すべての警告は、`@suppress(Wnnnn)` を使用して項目ごとに、またはパッケージ マニフェスト (§23.1) でグローバルに抑制できます。

### 22.6.1 未使用の変数と定義

**W0001 — `warn.unused_variable`**。アルゴリズム本体内の `var` または `const` バインディングは、その宣言後に読み取られることはありません。

```
warning[W0001]: unused variable `tmp` (warn.unused_variable)
  --> src/lib.miz:12:7
   |
12 |   var tmp := compute_something();
   |       ^^^ declared but never used
   |
help: remove the binding, or prefix with `_` to suppress: `var _tmp := ...`
```

**W0002 — `warn.unused_definition`**。module内で定義された `func`、`pred`、`mode`、または `attr` は、module内またはそれをインポートするmodule内で参照されることはありません。この警告は、fileごとではなく、パッケージ レベル (§23.4) でのみ発行されます。

**W0003 — `warn.unused_hypothesis`**。証明ブロック内の `assume` または `let` によって導入された仮説は、後続のステップでは参照されません。これは W0101 (冗長仮説、§22.6.2) とは異なります。

### 22.6.2 冗長な仮説

**W0101 — `warn.redundant_hypothesis`**。 `by` 句は、見つかった ATP 証明で使用されていないファクトを参照しています。事実は依然として正しいかもしれませんが、それを削除すると、より単純でより堅牢な証明が生成されます。

```
warning[W0101]: `Nat_add_comm` appears redundant in `by` clause (warn.redundant_hypothesis)
  --> src/lib.miz:22:22
   |
22 |   thus n = n by reflexivity, Nat_add_comm;
   |                              ^^^^^^^^^^^^ not used in the ATP proof
   |
help: remove `Nat_add_comm` from the `by` clause
```

**W0102 — `warn.externally_attested_proof`**。証明義務は ATP によって受け入れられましたが、kernel検証可能な証明書が生成されませんでした (§21.7.5)。その証拠は、ビルド アーティファクトの `externally_attested` です。

```
warning[W0102]: proof accepted without kernel certificate (warn.externally_attested_proof)
  --> src/lib.miz:55:3
   |
55 |   thus thesis;
   |   ^^^^^^^^^^^^ Z3 accepted the goal but produced no LFSC certificate
   |
note: this proof is marked `externally_attested` in the build artifact
note: the proof cannot be published to the package registry without kernel verification
hint: switch to `cvc5` or `vampire` for a kernel-verifiable certificate
```

**W0103 — `proof.citation.unused`**。証明ステップの明示的な `by` 節が、kernel に受理された `used_axioms` set に現れない補題を指定しています (§23.5.4、§23.9.1)。検証は成功しますが、その引用は削除または refine できます。

```
warning[W0103]: by clause contains unused citations (proof.citation.unused)
  --> src/lib.miz:35:7
   |
35 |   thus thesis by poly_eval_add, poly_eval_mul, poly_eval_const;
   |                                                ^^^^^^^^^^^^^^^ not used in proof certificate
   |
help: remove unused citation, or run `mizar refine` to rewrite automatically
```

### 22.6.3 廃止された機能

**W0010 — `warn.deprecated_syntax`**。構文構造、アノテーション、またはlibrary関数が非推奨としてマークされています。警告には、代替フォームと、非推奨のフォームが削除されるバージョンが含まれます。

```
warning[W0010]: `@infer_as` is deprecated (warn.deprecated_syntax)
  --> src/lib.miz:7:3
   |
 7 |   const z = @infer_as(0, Real);
   |             ^^^^^^^^^^^^^^^^^^^ deprecated since version 0.8; removed in version 1.0
   |
help: replace with `const z = 0 as Real;`
```

### 22.6.4 アルゴリズムにおける到達不能なコード

**W0201 — `warn.unreachable_code`**。アルゴリズム本体内のステートメントは、そこに到達するすべての制御フロー パスが、検証器が常に false であると静的に決定できる条件によって保護されているため、または以前の `return`、`break`、または `continue` が無条件に制御を移譲しているため、決して実行できません。

```
warning[W0201]: unreachable statement (warn.unreachable_code)
  --> src/lib.miz:19:5
   |
14 |     if n = 0 do
15 |       return 1;
16 |     end;
17 |     if n > 0 do
18 |       return n * factorial(n - 1);
19 |     end;
   |     ^^^^ this branch is unreachable: `n = 0` and `n > 0` together with `n: Nat` are exhaustive
20 |     return 0;
   |     ^^^^^^^^^ also unreachable
```

**W0202 — `warn.loop_may_not_terminate`**。アルゴリズムは `terminating` と宣言されていますが、その `decreasing` 尺度が十分に根拠のある条件を満たすかを静的に検証できないループが含まれています。検証器はユーザーの注釈に基づいて終了要求を受け入れましたが、それを独自に確認することはできませんでした。

### 22.6.5 弱化された事後条件 (`ensures` はより強力な形式であることが証明可能)

**W0210 — `warn.weakened_postcondition`**。検証器は、`ensures` 節が、記述されているより厳密に強い形式で証明可能であることを検出しました。これは、作成者がより強力な保証を意図していたか、編集中に事後条件が誤って単純化された可能性があることを示しています。

```
warning[W0210]: `ensures` clause weaker than provable (warn.weakened_postcondition)
  --> src/lib.miz:8:5
   |
 8 |   terminating algorithm double(n) -> Nat
 9 |     ensures result >= n
   |             ^^^^^^^^^^^ this is provable, but `result = 2 * n` is also provable
   |
note: the verifier can prove `result = 2 * n` which implies `result >= n`
hint: consider strengthening to `ensures result = 2 * n` for a more precise specification
```

この警告は、単に指定された事後条件が抽象的に弱い場合ではなく、検証器が **特定** より強力な式を識別できる場合にのみ発行されます。より強力な式は、前提条件選択ヒューリスティック (§22.4.4) を逆に実行して、ATP がアルゴリズム本体から証明できる等式を探すことによって取得されます。
