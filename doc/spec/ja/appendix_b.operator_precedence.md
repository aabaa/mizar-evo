## B.1 優先ドメイン

Mizar Evolution は 2 つの優先ドメインを使用します。

|ドメイン |適用対象 |優先source |
|---|---|---|
|用語表現 |ユーザー定義functor、前置/後置/中置記法、セレクター アクセス、`qua`、括弧内の用語 |演算子の宣言と固定期間の構文 |
|数式 |元素式、`not`、`&`、`or`、`implies`、`iff`、数量子 |言語定義の階層を修正 |

用語の優先順位は、数式レベルの接続詞が解析される前に解決されます。たとえば、`a + b = c + d` はアトミック式 `(a + b) = (c + d)` として解析され、そのアトミック式は式レベルの優先順位に加わります。

括弧は両方のドメインの優先順位をオーバーライドします。

## B.2 用語演算子

シンボリックfunctorは、スタンドアロンの演算子宣言を使用してメタデータの解析を宣言できます。

```mizar
infix_operator("+", left, 80);
infix_operator("*", left, 90);
infix_operator("^", right, 95);
prefix_operator("-", 85);
postfix_operator("!", 95);
```

|宣言 |意味 |
|---|---|
| `infix_operator("sym", left, N);` |二項中置演算子、優先順位左結合演算子 `N` |
| `infix_operator("sym", right, N);` | 2 項中置演算子、優先順位は右結合 `N` |
| `infix_operator("sym", none, N);` | 2 項中置演算子、優先順位は非結合です `N` |
| `prefix_operator("sym", N);` |優先順位の接頭演算子 `N` |
| `postfix_operator("sym", N);` |後置演算子が優先 `N` |

優先順位の値の範囲は `0` から `255` です。値が大きいほど、結合がより強くなります。宣言のないシンボリックfunctorのデフォルトの優先順位は、非結合の `64` です。

`=`、`<>`、`in` などの組み込み述語シンボルは、暗黙的にインポートされたコア moduleで宣言されます。ユーザー宣言によってこれらをオーバーライドすることはできません。

`qua` 型修飾演算子は、`infix_operator` によって宣言されるのではなく、言語によって固定されます。これは用語レベルの優先順位が最も低く、左結合であるため、`R qua AddGroup qua Magma` は `(R qua AddGroup) qua Magma` として解析されます。条件をより大きな用語式内のサブ用語にのみ適用する必要がある場合は、括弧を使用します。

例:

```mizar
a + b * c       :: a + (b * c)
a * b + c       :: (a * b) + c
a ^ b ^ c       :: a ^ (b ^ c), if ^ is right-associative
a %% b %% c     :: error, if %% is a non-associative term operator
```

## B.3 原子式境界

アトミック式は、用語解析と式解析の間の橋渡しとなります。parserはまず、述語、型アサーション、属性アサーション、等価、不等式、またはメンバーシップ テストで必要な用語式を消費します。結果として得られる原子式は、式parserの 1 つのオペランドになります。

|原子の形 |例 |用語の内部解析 |
|---|---|---|
|述語の適用 | `a divides b` |述語位置は用語式として解析されます。
|平等 / 不平等 | `a + b = c + d` |両側は用語式として解析されます。
|メンバーシップ | `x in dom f` |両側は用語式として解析されます。
|型アサーション | `x is Element of A` |件名は用語式として解析されます |
|属性アサーション | `n is positive even Integer` |件名は用語式として解析されます |

数式接続詞は用語内で結合しません。したがって、文法位置が明示的に式を予期しない限り、`f(x & y)` は無効です。同様に、`x + y & z` は記述どおり無効です。`x + y` は項であり、原子式ではないため、`&` の左側のオペランドは不完全です。作成者は、`x + y = 0 & z is Nat` などの完全な式を作成する必要があります。

Predicate-chain notation is resolved at the atomic-formula boundary, not as
term-operator associativity. For example, after predicate resolution,
`a < b < c` denotes the conjunction `a < b & b < c`; it is not rejected merely
because `<` would be non-associative as a term operator.

## B.4 数式演算子

数式演算子は固定階層を使用します。以下の表の優先順位の数値が低いほど、バインドがより強くなります。

|優先順位 |演算子フォーム |結合性 |例 |
|---|---|---|---|
| 1 |原子式 | - | `x is Nat`、`a divides b`、`x = y` |
| 2 | `not` |プレフィックス | `not x > 0` |
| 3 | `&` |左 | `a & b & c` は `(a & b) & c` として解析されます。
| 4 | `or` |左 | `a or b or c` は `(a or b) or c` として解析されます。
| 5 | `implies` |右 | `a implies b implies c` は `a implies (b implies c)` として解析されます。
| 6 | `iff` |非関連 | `a iff b iff c` はエラーです |
| 7 | `for`、`ex` |最も外側のバインディング | |

例:

```mizar
not x > 0 & y > 0
:: (not (x > 0)) & (y > 0)

a or b implies c
:: (a or b) implies c

for x being Nat holds x > 0 implies x >= 1
:: for x being Nat holds (x > 0 implies x >= 1)
```

## B.5 推奨される解析アルゴリズム

Pratt parserは、null と left の表記による優先クライミングとも呼ばれ、用語式と数式に推奨されます。アクティブなレキシコンはインポートに依存し、演算子の宣言はテーブル駆動であるため、言語によく適合します。

推奨される実装概要:

1. インポートを前処理し、表示される各演算子の固定性、優先順位、結合性、アリティ、および定義moduleを含むアクティブな辞書を構築します。
2. 第 2 章の最長一致ルールを使用してtoken化します。文字列リテラル認識は、文字列引数を必要とする文法の位置でのみ有効のままです。
3. 主な用語形式を解析します: 変数、数値、括弧で囲まれた用語、構造コンストラクター、集合式、および `the` 式。
4. 宣言された結合力を使用して、用語レベルの接頭辞および後置演算子を解析します。
5. 次の演算子の左結合力と現在の最小結合力を比較することにより、用語レベルの中置演算子を解析します。
6. `qua` を最も優先順位の低い用語レベルの型修飾として解析します。
7. 解析された用語オペランドの周囲の述語表記、等価性、メンバーシップ、型アサーション、または属性アサーションを解析することにより、アトミック式を完成させます。
8. 別の固定結合力テーブルを使用して、数式レベルの接頭語、中置語、および数量詞の形式を解析します。

中置項演算子の場合は、次の結合力規則を使用します。

|結合性 |左の結合力 |右の最小結合力 |
|---|---:|---:|
|左 | `N` | `N + 1` |
|右 | `N` | `N` |
|なし | `N` | `N + 1`、および同じオペレーターの連鎖チェック |

同じパターンが、固定式テーブルを使用する式接続詞にも適用されます。非結合演算子は、同じ演算子が括弧なしで連鎖されている場合に構文エラーを生成する必要があります。

parserは、優先順位を決定する際にoverloadの解決を実行しないでください。解析では、宣言された演算子のメタデータから構文の形状が選択されます。overload解決は、第 19 章に従ってセマンティック定義を選択します。インポートされた 2 つのmoduleが同じ表示ユーザー シンボルに異なる優先順位または結合性を割り当てる場合、これは §10.9 で指定されているリンク時の競合です。

## B.6 診断

優先順位関連の診断では、ローカルのsource レベルの説明を優先する必要があります。

|状況 |推奨診断 |
|---|---|
|連鎖非結合演算子 |演算子を報告し、かっこを追加することを提案します。
|インポートされた優先順位宣言の競合 |定義moduleと競合する宣言の両方を報告します。
|デフォルトの演算子の優先順位によって予期しないグループ化が発生する |デフォルトの優先順位 `64` を報告し、明示的な宣言を追加することを提案します。
|用語が必要な場所に数式接続詞が表示されます |予想される用語の境界を報告し、括弧を付けるか接続詞を用語の外に移動することを提案します |

診断では、parserの実際のグループ化が驚くべきものであるものの有効である場合に、そのグループ化が表示される必要があります。
