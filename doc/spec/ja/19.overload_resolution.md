## 19.1 概要

Mizar Evolution では、異なるパラメータ型を持つ同じシンボル名を複数定義できます。これは **overload** と呼ばれるメカニズムです。シンボルが使用される場合、検証器は **overload解決** を通じて、最も適切な定義を一意に選択します。

overload解決プロセスは 4 つの段階に従い、C++ overload解決をモデルとしていますが、Mizar のソフト型システムに適合しています。

1. **候補セットの構築** — スコープ内にある一致するシンボル名を持つすべての定義を収集します。
2. **実行可能な候補フィルタリング** - パラメータの型が引数の型と互換性がある候補を保持します (クラスタベースの属性クロージャまたは構造継承のアップキャスト後)。
3. **ベストマッチの選択** — 他のすべての候補よりも厳密に具体的な候補を選択します。
4. **あいまいさまたはエラー** — 一意の最適一致が存在しない場合は、エラーを報告します。

**主な制限事項:**

|ルール |理論的根拠 |
|---|---|
|戻り値の型はoverloadの解決に関与しません。戻り型のoverloadを許可すると、暗黙的な変換との解決不可能な相互作用が作成されます。
|引数の型のシグネチャが同じだが戻り値の型が異なる定義は、overloadではなく **定義の競合** エラーです。 |
| `redefine` は、通常の定義と同じoverload解決規則に従います。 |

## 19.2 暗黙的な型変換

### 19.2.1 クラスタベースの属性クロージャ

型 `T` の **属性クロージャー** は、すべてのアクティブな条件付きクラスター登録を固定点に適用することによって、T の宣言された属性から導出できる最大の属性セットです。

アクティブなクラスター登録を考慮すると、クロージャー演算子 `cl(T)` は次のように定義されます。

```text
cl₀(T)  = attributes explicitly declared on T
cl_{i+1}(T) = cl_i(T) ∪ { B | ∃ cluster A → B for T, A ⊆ cl_i(T) }
cl(T)   = fixpoint of the above sequence
```

属性セットは有限であり、各ステップは単調であるため、固定点は常に存在し、クラスターが適用される順序に関係なく一意です。これはクラスター システムの **confluence プロパティ**です。

```mizar
:: Given registrations:
::   cluster finite -> countable for set;
::   cluster finite -> compact for TopSpace;
::   cluster countable compact -> second_countable for TopSpace;

let S be finite TopSpace;
:: cl(S) = { finite, countable, compact, second_countable, ... }
```

クロージャは、各overload解決サイトで引数ごとに 1 回計算されます。結果は決定的です。

### 19.2.2 構造継承のアップキャスト

構造体型は、`inherit ... extends ...` チェーンを通じて暗黙的にその **祖先モード** のいずれかと見なすことができます。このアップキャストは常に有効 (拡張) であり、証明は必要ありません。継承宣言自体が構造的な埋め込みを保証します。

```mizar
:: Ring inherits from Group (via AddGroup) and from Monoid (via MulMonoid)
let R be commutative Ring;

:: Implicit upcast: R is usable wherever a Group is expected (via AddGroup path),
:: but only if the inheritance path is unambiguous.
:: When two paths exist, the upcast must be made explicit with qua (see §19.3.1).
```

アップキャストは実行可能な候補フィルタリングに参加します。`T` の構造が **一意** `inherit` チェーンを介して `M` から継承されている場合、型 `T` の引数は `M` 型のパラメータに対して実行可能です。複数のパスが存在する場合、引数は暗黙的に実行できません。明示的な `qua` が必要です。

|状況 |結果 |
|---|---|
| `T` は、正確に 1 つのパスを介して `M` から継承します。暗黙的なアップキャスト — アノテーションなしで実行可能 |
| `T` は 2 つ以上のパスを介して `M` から継承します。あいまいさ — パスを選択するには `qua` が必要です |
| `T` は `M` を継承しません。実行不可能 |

### 19.2.3 特異性の順序付け

次の両方が当てはまる場合、型 `T₁` は **少なくとも `T₂` (`T₁ ⊑ T₂` と表記) と同じくらい特定的です**。

1. **基数の互換性**: `T₁` の基数タイプ (モードまたは構造) は、`T₂` の基数タイプと同一かサブモードです。
2. **属性の包含**: `cl(T₁) ⊇ cl(T₂)` — `T₂` (クロージャ後) で必要なすべての属性は、`T₁` (クロージャ後) にも存在します。

`T₁ ⊑ T₂` および `T₂ ⋢ T₁` の場合、`T₁` は `T₂` (`T₁ ⊏ T₂` と書かれる) よりも**厳密にはより具体的**です。

この順序は **部分順序** (再帰的、反対称、推移的) です。 2 つの型は**比類のない**かもしれません。どちらも少なくとも他方ほど特異的ではありません。

```mizar
:: Specificity examples (with standard cluster registrations):
::   Field ⊏ commutative Ring ⊏ Ring      (strictly more specific left to right)
::   finite TopSpace ⊏ countable TopSpace  (finite implies countable by cluster)
::   countable TopSpace and compact TopSpace are incomparable
::     (neither implies the other in general)
```

**重要**: 特異性の順序は、生の属性宣言ではなく、**タイプ** (基数 + クロージャ後の属性) で定義されます。生の宣言は異なるがクロージャが同一である 2 つの引数型は、同様に固有であるとみなされます。

## 19.3 明示的な型変換

overloadの解決があいまいな場合、または検証器が意図した型を推測できない場合、プログラマは明示的な変換を提供できます。

### 19.3.1 `qua` 演算子

`qua` 演算子は、**証明を必要とせずに**、overload解決のために対象となる項の型を明示的に選択します。以下の 2 つの相補的なケースについて説明します。

1. **クラスターベースの拡張** — 複数の属性互換性のあるoverloadが存在する場合、属性の少ない祖先型を選択します。
2. **構造継承のアップキャスト** — 構造が複数のパスを介して構造を継承するときに、特定の祖先モードを選択します (ダイヤモンド継承)。

どちらの場合も、ユーザーが作成した証明は必要ありません。検証器は自動的に FOL 証明義務を発行し、クラスター登録公理と `inherit` チェーンを介してそれらを実行します。完全な FOL エンコーディングについては、[§13.8.7](./13.term_expression.md#1387-qua-演算子のエンコーディング) を参照してください。

```ebnf
(* unqualified_term_expression is defined in Chapter 13. *)
qua_expression ::= unqualified_term_expression "qua" type_expression { "qua" type_expression } ;
```

**クラスターベースの曖昧さ回避:**

```mizar
:: f is defined for both countable TopSpace and compact TopSpace
:: S is finite TopSpace — ambiguous (finite implies both countable and compact)
f(S qua countable TopSpace)   :: resolves to the countable overload
f(S qua compact TopSpace)     :: resolves to the compact overload
```

**継承パスの曖昧さ回避** (以前の `as` 強制を置き換えます):

```mizar
:: Ring inherits from Magma via both AddGroup and MulMonoid
let R be commutative Ring;
f(R qua AddMagma)    :: selects R's additive Magma structure
f(R qua MulMagma)    :: selects R's multiplicative Magma structure
```

**マルチレベル パスの `qua` のチェーン:**

```mizar
:: Traverse two levels of the inheritance chain explicitly
f(R qua AddGroup qua Magma)   :: AddGroup view, then its Magma ancestor
```

各ステップは、`inherit ... from ...` チェーンに沿った有効な拡幅区間である必要があります。ステップは左結合です。

> [！注記]
> 完全な `qua` 構文と意味論については、[第13章 (項式), §13.6](./13.term_expression.md#136-型式認定-qua) を参照してください。 `qua` は、証明不要 ワイドニング (クラスターまたはアップキャスト) のみを処理します。型を絞り込むには (たとえば、より具体的なモードで変数を再バインドする)、証明付きで `reconsider` を使用します (§19.3.2)。

### 19.3.2 `reconsider` ステートメント

`reconsider` ステートメントは、変数をより具体的な型に再バインドし、証明コンテキストでの後続の使用のためにその型を絞り込みます。

```mizar
let S be finite TopSpace;
reconsider S as countable TopSpace by mml.topology.finite_is_countable;
:: S now has type countable TopSpace in subsequent context
f(S)  :: unambiguously resolves to the countable overload
```

> [！注記]
> 完全な `reconsider` 構文については、「[第15章 (文), §15.5.1](./15.statements.md#1551-型の再検討-reconsider)」を参照してください。

## 19.4 過負荷解決アルゴリズム

### 19.4.1 候補セット

シンボル使用 `f(a₁, ..., aₙ)` の **候補セット**は、使用時点で **スコープ内**である `n` パラメーターを含む `f` のすべての定義で構成されます。範囲は次によって決定されます。

1. 現在のmodule内の定義 (宣言順)。
2. `import` ステートメントを介してインポート順序でインポートされた定義。
3. templateのインスタンス化サイトの場合: 囲む `definition` ブロック内で定義が表示されます。

外側のスコープからの定義は、同じ署名を持つ内側のスコープの定義によってシャドウされます。

### 19.4.2 有力な候補者

各位置 `i` について次の場合、候補定義 `f(p₁: T₁, ..., pₙ: Tₙ)` は引数 `(a₁, ..., aₙ)` に対して**実行可能**です。

```text
type_of(aᵢ) ⊑ Tᵢ
```

つまり、引数の型 (クラスター クロージャー後) は、少なくともパラメーターの型と同じくらい具体的です。

実行可能な候補が存在しない場合、検証器は **一致する定義がありません** エラーを報告します。

### 19.4.3 ベストマッチの選択

実行可能な候補の中で、**最も一致する**は候補 `f*` であり、他のすべての実行可能な候補 `g` については次のとおりです。

```text
parameter_type(f*) ⊑ parameter_type(g)    (for each argument position)
```

比較は位置ごとに実行されます。`f*` は、**すべての** 引数の位置で少なくとも `g` と同じくらい具体的である必要があります。

一意の最適一致が存在する場合、それが選択されます。

### 19.4.4 あいまいさエラー

**曖昧さ**は、2 つ以上の実行可能な候補が**比較不可能**である場合に発生します。つまり、どちらも同時にすべての引数の位置において他方よりも少なくとも同程度具体的ではありません。検証器は **曖昧なoverload** エラーを報告し、`qua` または `reconsider` による明示的な曖昧さの解消を必要とします。

```mizar
:: Two viable candidates, incomparable:
func f(x : countable TopSpace) -> Nat;
func f(x : compact TopSpace) -> Nat;

let S be finite TopSpace;
f(S);   :: ERROR: ambiguous — both countable and compact apply,
        ::        neither is more specific than the other
f(S qua countable TopSpace);   :: OK
```

library設計者に推奨される解決策は、より具体的な共通サブ型の定義を追加することです。

```mizar
:: Resolves the ambiguity for finite TopSpace callers
func f(x : finite TopSpace) -> Nat;
```

library設計者は合流証明を必要としません。意図された使用例に対して明確な定義を提供する責任はlibraryにあります。

## 19.5 シンボルの再定義 (`redefine`)

`redefine` ブロックは、より狭い引数の型に対して既存のシンボルのより具体的な定義を導入します。通常の定義と同様にoverloadの解決に参加します。

```mizar
:: Original definition
definition
  let G be Group;
  func order(G) -> Nat means ...;
end;

:: Redefinition for finite groups — more specific
definition
  let G be finite Group;
  redefine func order(G) -> Nat equals card G.carrier;
end;
```

**解決動作:**

- 引数の型が `finite Group` の場合、再定義された (より具体的な) 定義が選択されます。
- 引数の型が `Group` (`finite` なし) の場合、元の定義が選択されます。
- 再定義された定義には、元の定義よりも **厳密に具体的な**パラメータ タイプが必要です。それ以外の場合は、定義競合エラーです。

**一貫性の義務:** `redefine` は、新しい定義が重複ドメイン上の元の定義と**互換性がある**ことを証明する必要があります。これは、`coherence` の正確性条件として表されます。

```mizar
redefine func order(G) -> Nat equals card G.carrier;
coherence
proof
  :: Must prove: for G being finite Group holds
  ::   order(G) = card G.carrier
  ...
end;
```

> [！注記]
> 正当性条件の構文については、[第 16 章 (定理と証明)、§16.6](./16.theorems_and_proofs.md#166-型システムの証明義務) を参照してください。

## 19.6 高度な解決メカニズム

### 19.6.1 templateパラメータの推論

template 定義が使用される場合 ([第 18 章、§18.2.7](./18.templates.md#1827-templateパラメータの推論) を参照)、template引数 `[T]` は、引数の型から一意に推測できる場合は省略できます。

このプロセスには 2 つの異なるフェーズがあります。 **フェーズ A (template拡張、ステップ 1 ～ 3)** は、一致する可能性のある各template 定義を具体的な候補に変換します。 **フェーズ B (ステップ 4)** では、標準のoverload解決ルールを使用して、すべての具体的な候補の中から選択します。 2 つのフェーズは異なるマッチング ルールを使用するため、インターリーブしないでください (以下の制限を参照)。

**手順:**

1. **template パラメーターの推論** — 呼び出しサイトと名前が一致するスコープ内のtemplate 定義ごとに、`T` を引数の正確な型 (完全な属性型、変更されていない) にバインドします。次に、制約を確認します。`T` の基数モードが `M` またはモード階層内の `M` のサブモード (クラスター拡張なし) であれば、`T be M` は満たされます。制約が満たされる場合、template は候補を生成します。それ以外の場合はスキップされます。複数のtemplate はすべて、同じバインドされた `T` で成功する可能性があります。ステップ 4 で勝者を選択します。

2. **template のインスタンス化** — template 定義全体で推論された `T` を置き換えて、具体的な (非template) 定義を生成します。ステップ 1 で推論が失敗した場合、このtemplate は候補を提供しないため、スキップされます。

3. **template 派生としてタグ付けされた候補セットに追加** — インスタンス化された候補を、そのtemplateの起源をマークするフラグとともに記録します。続行する前に、スコープ内のすべてのtemplateに対してステップ 1 ～ 3 を繰り返します。

4. **template以外の優先順位で標準のoverload解決を実行します** - すべての候補を収集し、通常の `⊑` 順序を適用します。さらに、非template候補とtemplateから派生した候補が同点の場合 (`⊑` で同等に特異的)、非template候補が勝ちます。これにより、汎用templateが意図的な非template特殊化と競合するのを防ぎます。

**この順序が重要な理由:** ルールが異なるため、ステップ 4 が開始される前に *すべての* templateに対してステップ 1 ～ 3 が完了する必要があります。ステップ 1 では、クラスタの拡張を意図的に抑制して、推論された型を一意に保ちます (以下の制限を参照)。ステップ 4 では、クラスターの閉鎖を含む完全な `⊑` 順序付けを適用して実行可能性を確認し、非templateの優先順位ルールを適用して関係を解決します。 2 つのフェーズを混合すると、推論された型が一意でなくなり、予測不能になります。

template 推論により複数の有効なインスタンス化 (曖昧な推論) が生成された場合、検証 toolはoverloadの曖昧性とは別に、**曖昧なtemplate のインスタンス化** エラーを報告します。

次の例では、抽象モード階層 `B extends A` および `C extends B` を使用して、数学的領域の知識を導入せずに各ケースを説明します。

次の例では、抽象モード階層 `B extends A` および `C extends B` を使用します。

次の例では、明示的に宣言されたモード階層を想定しています。

```mizar
inherit B extends A;
inherit C extends B;
```

**ケース 1 — 基本的な推論と明示的なアノテーション:**

```mizar
definition
  let T be type extends A;
  let x be T;
  func f[T](x) -> T;
end;

let c be C;
f(c)      :: T = C (C ⊑ A in mode hierarchy) → OK
f[A](c)   :: explicit T = A; viability: C ⊑ A → OK
```

**ケース 2 — 最も具体的なtemplateが優先されます:**

複数のtemplateがステップ 1 で成功した場合、ステップ 4 では最も厳しい制約を持つtemplateが選択されます。

```mizar
definition
  let T be type extends A;
  let x be T;
  func f[T](x) -> T;   :: overload 1
end;

definition
  let T be type extends B;
  let x be T;
  func f[T](x) -> T;   :: overload 2
end;

let c be C;
f(c)
:: Step 1: T = C; C ⊑ A ✓ and C ⊑ B ✓ → both produce candidates
:: Step 4: overload 2 (constraint B) is stricter than overload 1 (constraint A) → overload 2 wins

let a be A;
f(a)
:: Step 1: T = A; A ⊑ A ✓ but A ⊑ B? No → only overload 1 produces a candidate
:: Step 4: overload 1 selected unconditionally
```

**ケース 3 — template以外の特殊化が優先されます:**

```mizar
definition
  let T be type extends A;
  let x be T;
  func f[T](x) -> T;   :: generic template
end;

definition
  let x be B;
  func f(x) -> B;      :: non-template specialization
end;

let b be B;
f(b)
:: Step 1: T = B, B ⊑ A ✓ → template candidate; non-template also matches
:: Step 4: non-template wins the tie — no ambiguity error

let c be C;
f(c)
:: Step 1: T = C, C ⊑ A ✓ → template candidate; non-template: C ⊑ B ✓ → also matches
:: Step 4: non-template (constraint B) is stricter than template (constraint A) → non-template wins
```

**ケース 4 — 複数のtemplate パラメーターがすべて一致する必要があります:**

```mizar
definition
  let T be type extends A;
  let x, y be T;
  func g[T](x, y) -> T;
end;

let b be B;
let a be A;

g(b, b)      :: T = B from both → agree → OK
g(b, a)      :: T = B from x, T = A from y → mismatch → inference fails
g[A](b, a)   :: explicit T = A; viability: B ⊑ A ✓ and A ⊑ A ✓ → OK
```

**重要な制限事項 - template 推論中にクラスター拡張は適用されません。**

`[T]` が省略された場合、`T` は引数の正確な型にバインドされ、モード階層のみを使用して制約がチェックされます (クラスター拡張は行われません)。 `[T]` が明示的に記述されている場合、推論はスキップされ、検証器は完全な `⊑` 順序付け (クラスター拡張を含む) の下で実行可能性をチェックします。

```mizar
:: P and Q are unrelated in the mode hierarchy.
:: Given: cluster Q -> P;    (Q is also P, established by cluster registration)

mode P;
mode Q;   :: Q does not extend P in the mode hierarchy

definition
  let T be type extends P;
  let x be T;
  func h[T](x) -> T;
end;

let q be Q;

h(q)      :: ERROR — T bound to Q; Q ⊑ P? Not in mode hierarchy → inference fails
h[P](q)   :: OK — T = P explicitly; viability: Q ⊑ P via cluster → OK
```

> [！注記]
> 推論中にクラスターの拡張を許可すると、推論された型が一意でなくなります。同じ引数がクラスター登録のチェーンを通じて多くの異なる制約を満たす可能性があり、予測できないインスタンス化の選択が生じる可能性があります。明示的な `[T]` アノテーションにより、意図したインスタンス化が明確になり、クラスター拡張コストが予測可能になります。

### 19.6.2 複数の継承パス

構造タイプが複数のパス (ダイヤモンド継承) を介して祖先モードから継承され、定義がその祖先によってパラメーター化される場合、意図したパスを選択するために `qua` を使用する必要があります。 `qua` が省略された場合、検証器はパスを自動的に推測せず、**曖昧な継承パス** エラーを報告します。

```mizar
:: Ring inherits from Magma via both AddGroup and MulMonoid
func PermProduct(M : commutative Magma) -> ...;

let R be commutative Ring;
PermProduct(R)                :: ERROR: ambiguous path to Magma
PermProduct(R qua AddMagma)   :: OK — uses additive structure
PermProduct(R qua MulMagma)   :: OK — uses multiplicative structure
```

これは、通常のoverloadのあいまいさとは異なります。ここでは、`PermProduct` の定義が 1 つだけ存在しますが、引数の型は複数の継承パスを介して解釈できます。

### 19.6.3 moduleのスコープと解決順序

overloadの解決では、使用時点で**スコープ内**にある定義のみが考慮されます。範囲は次によって決定されます。

1. **現在のmodule**: サイトを使用する前に宣言されたすべての定義。
2. **インポートされたmodule**: `import` 宣言にリストされているmoduleのすべての公開定義。
3. **シャドウイング**: 同じ名前と引数型の署名を持つ現在のmodule内の定義は、インポートされた定義をシャドウします。

module `import` によって既存の定義と名前の競合が発生した場合、その競合は特異性によって解決されます。インポートされた定義が厳密により具体的な場合、通常はoverloadの解決に参加します。異なるインポートされたmoduleからの 2 つの定義が比較できない場合、競合は **インポートの曖昧さ** エラーとして報告され、プログラマは参照を明示的に修飾する必要があります (`module_path.symbol_name`)。

> [！注記]
> moduleのインポート構文と可視性ルールについては、「[第12章 (module と namespace)](./12.modules_and_namespaces.md)」を参照してください。

### 19.6.4 `of`/`over` 貪欲な解析

`of`/`over` 構文は、`[]` 括弧形式の糖衣構文です ([第 18 章、§18.1](./18.templates.md#181-概要) を参照)。parserは **貪欲な (最長一致) 引数の消費**を使用します。`of`/`over` の後のカンマ区切りの引数をスキャンする場合、parserはモードの最も高いアリティの定義が受け入れるのと同じ数の引数を消費します。このルールは決定的であり、一般的なケースではプログラマによる曖昧さの解消を必要としません。

**貪欲なルール:** `M of A, B, ...` は、そのアリティに `M` の定義が存在する引数の最大数を消費します。最大アリティを超える引数は、それを囲んでいる式に残ります。

```mizar
:: Suppose both a 1-parameter and 2-parameter M exist:
mode M of X is ...;
mode M of X, Y is ...;

M of A, B      :: M[A, B]  — greedy; 2-parameter definition exists, both args consumed
M of A         :: M[A]     — only 1 arg present; 1-parameter definition used
```

**さらに多くの引数が続く場合に低位数形式を使用するには**、括弧を使用して消費を制限するか、`[]` 括弧形式を明示的に使用します。

```mizar
(M of A), B    :: M[A] as one value, B separately — parentheses stop greedy consumption
M[A], B        :: same effect with bracket form
```

**ネストされた `of` 式**は、各レベルで左から右に同じ貪欲ルールに従います。

```mizar
Function of Subset of X, Y
:: inner first: Subset of X, Y → Subset[X, Y]  (greedy, 2-param Subset)
:: outer:       Function of Subset[X, Y]        → Function[Subset[X, Y]]

Function of (Subset of X), Y
:: (Subset of X) is parenthesized → Subset[X]
:: Function of Subset[X], Y → Function[Subset[X], Y]  (2-param Function)
```

**推奨事項:** どのような定義が範囲内にあるかに関係なく、意図したアリティを明確に記述する必要がある場合は、`[]` 括弧形式を使用してください。

## 19.7 論理エンコーディングの詳細

overloadの解決は **コンパイル時** のメカニズムです。それ自体は FOL 公理を生成しません。その効果は、各使用サイトでどの定義の公理をインスタンス化するかを選択することです。

|overloadの解決結果 | FOL効果 |
|---|---|
|ユニークなベストマッチ `f*` が選択されました | `f*` の公理は、特定の引数の型でインスタンス化されます。
| `redefine` 定義が選択されました |再定義された公理 (より具体的な) が使用されます。 coherence は、元の | との一貫性を保証します。
|あいまいさエラー | FOL 用語は生成されません。検証が失敗する |

`redefine` の **一貫性条件** により、次の FOL 義務が生成されます。

```fol
∀args. (more_specific_type(args) → new_def(args) = original_def(args))
```

これにより、2 つの定義が共通領域で一致することが保証され、両方が公理化された場合でも健全性が維持されます。

## 19.8 完全な構文 (EBNF)

overloadの解決は、新しい構文形式には関連付けられていません。これは、関手、述語、モード、および属性の既存の使用に適用される解決手順です。関連する構文形式は、それぞれの章で定義されています。

```ebnf
(* Explicit disambiguation forms used when overload resolution is ambiguous *)

(* qua — widens apparent type (cluster or upcast) without proof; see Chapter 13 §13.6 *)
(* Covers both attribute-based disambiguation and inheritance path selection.      *)
(* unqualified_term_expression is defined in Chapter 13.                          *)
(* Steps are left-associative: R qua AddGroup qua Magma                           *)
qua_expression ::= unqualified_term_expression "qua" type_expression { "qua" type_expression } ;

(* redefine — introduces a more specific overload; coherence is required *)
redefine_item  ::= "redefine" ( func_def | pred_def | attr_def | mode_def ) ;

(* Specificity ordering (meta-notation, not surface syntax) *)
(* T₁ ⊑ T₂  iff  radix(T₁) ≤ radix(T₂)  ∧  cl(T₁) ⊇ cl(T₂) *)
(* cl(T)  =  fixpoint of conditional cluster applications on T's attributes *)
```
