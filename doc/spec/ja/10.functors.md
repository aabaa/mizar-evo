## 10.1 概要と構文

functorは、型付き入力を型付き出力にマップするシンボリック関数を導入します。各functorには、名前 (ラベル)、パターン、戻り値の型、および定義本体があります。

**構文 (EBNF)**:

```ebnf
func_def         ::= "func" label ":" func_pattern "->" type_expression
                     ( "means" formula_definiens | "equals" term_definiens ) ";" ;
label            ::= identifier ;
func_pattern     ::= [ loci ] functor_symbol [ loci ] ;
loci             ::= locus | "(" locus_list ")" ;
locus_list       ::= locus { "," locus } ;
locus            ::= identifier ;
functor_symbol   ::= identifier | symbolic_func ;
symbolic_func    ::= symbol_char+ ;
formula_definiens ::= formula
                    | formula_case { "," formula_case } [ "otherwise" formula ] ;
formula_case      ::= formula "if" formula ;
term_definiens    ::= term_expression
                    | term_case { "," term_case } [ "otherwise" term_expression ] ;
term_case         ::= term_expression "if" formula ;
type_expression  ::= attribute_chain radix_type ;
```

**例**：

```mizar
definition
  let X, Y be set;
  func UnionDef: X \/ Y -> set means
    for z being object holds z in it iff z in X or z in Y;
end;
```

functorは次のとおりです。

* **ユーザー定義**: `func` および `means` または `equals` で宣言
* **overload**: パラメーターの型によって曖昧さが解消されます。

主な特徴:
* `it` は、`means` 定義の戻り値を示します。
* 戻り値の型は `->` の後に指定されます
* functorシンボルには、空白以外の任意の文字シーケンスを使用できます
* 字句解析は最長一致ルールに従います

> [！注記]
> 証明 (`deffunc`) 内の **インラインfunctor定義** については、[第15章 (文), §15.2.3](./15.statements.md#1523-インライン関数定義-deffunc) を参照してください。

## 10.2 関数と述語

functorと述語は両方ともユーザー定義のシンボルですが、異なる目的を果たします。

|側面 |functor (`func`) |述語 (`pred`) |
| --- | --- | --- |
| **目的** |値を計算して返す |ブール関係を表現する |
| **戻り値の型** |任意のタイプ (`->` の後に指定) |常にブール値 (true/false) |
| **構文** | `x + y`、`f(x)`、`X \/ Y` | `x c= y`、`a divides b` |
| **定義キーワード** | `equals` または `means` | `means`のみ |
| **正確さ** | `means` の存在/一意性が必要です |存在/独自性は必要ありません |

**functorを使用する場合**:
- 値の計算: `x + y`、`card X`
- 建物の構造: `X \/ Y`、`[: X, Y :]`
- 数学的演算の定義

**述語を使用する場合**:
- 試験条件：`x c= y`
- 関係の定義: `a divides b`
- 数式でプロパティを表現する

**比較例**:

```mizar
:: As a functor (returns a value)
definition
  let x, y be Real;
  func SumDef: x + y -> Real equals ...;
end;

:: As a predicate (returns boolean)
definition
  let x, y be Real;
  pred LessOrEqualDef: x <= y means x < y or x = y;
end;

:: Functors are used in term expressions
set z = x + y;

:: Predicates are used in formulas
assume x <= y;
```

> [！注記]
> 詳細な述語構文と意味論は、[第9章 (述語)](./09.predicates.md) で定義されています。

## 10.3 定義スタイル: `equals` と `means`

Mizar は、2 つのスタイルのfunctor定義をサポートしています。

### 10.3.1 明示的な定義 (`equals`)

明示的な値を提供します。システムはfunctorが構築によって明確に定義されているとみなすため、追加の証明は必要ありません。

```mizar
definition
  let n be Nat;
  func DoubleDef: twice(n) -> Nat equals 2 * n;
end;

definition
  let X be set;
  func PowerSetDef: power_set(X) -> set equals
    { Y where Y is set : Y c= X };
end;            
```

### 10.3.2 暗黙的な定義 (`means`)

`it` をプレースホルダーとして使用して、結果の定義条件を指定します。このスタイルでは、ユーザーはそのような固有の結果が存在することを証明する必要があります。

```mizar
definition
  let x, y be Real;
  assume y <> 0;
  func DivDef: x / y -> Real means it * y = x;
  existence
  proof
    :: Goal (auto-generated): ex z being Real st z * y = x
    ...;
  end;
  uniqueness
  proof
    :: Goal (auto-generated): for z1, z2 being Real st
    ::   z1 * y = x & z2 * y = x holds z1 = z2
    ...;
  end;
end;

definition
  let G be Group;
  func IdentityDef: 1_G -> Element of G means
    for a being Element of G holds it * a = a & a * it = a;
  existence
  proof
    :: Goal: ex e being Element of G st
    ::   for a being Element of G holds e * a = a & a * e = a
    ...;
  end;
  uniqueness
  proof
    :: Goal: for e1, e2 being Element of G st
    ::   (for a being Element of G holds e1 * a = a & a * e1 = a) &
    ::   (for a being Element of G holds e2 * a = a & a * e2 = a)
    ::   holds e1 = e2
    ...;
  end;
end;
```

**各スタイルを使用する場合**:

|スタイル |次の場合に使用します... |例 |
| --- | --- | --- |
| `equals` |値は直接計算できます | `twice(n) equals 2 * n` |
| `means` |値はプロパティによって特徴付けられます。 `x / y means it * y = x` |

## 10.4 部分関数と`assume`

`assume` 句は定義範囲を制限します。

```mizar
definition
  let x, y be Real;
  assume y <> 0;
  func PartialDivDef: x / y -> Real means it * y = x;
  existence
  proof ... end;
  uniqueness
  proof ... end;
end;

definition
  let f be Function, x be object;
  assume x in dom f;
  func ApplyDef: f . x -> object equals f(x);
end;
```

**意味論**:
* 想定されたドメインの外では、結果は未定義です
* 部分functorの呼び出しには、前提条件が成立するという証明が必要です

> [！ヒント]
> 可能であれば、前提条件を引数の型にエンコードすることをお勧めします。たとえば、`assume y <> 0` を使用するのではなく、`y` を `non zero Real` にする必要があります。これにより、明確さが向上し、検証の負担が軽減されます。

## 10.5 依存する戻り値の型

戻り値の型は入力値に応じて異なります。

```mizar
definition
  let G be Group;
  func CenterDef: center G -> Subgroup of G means
    for a being Element of it holds
      for b being Element of G holds a * b = b * a;
end;

definition
  let X be set;
  func IdDef: id X -> Function of X, X equals
    { [x, x] where x is Element of X : x in X };
end;
```

複数の依存関係がサポートされています。

```mizar
definition
  let G be Group;
  let H, K be Subgroup of G;
  func SubgroupProdDef: H * K -> Subset of G equals
    { h * k where h is Element of H, k is Element of K };
end;
```

## 10.6 正確性の条件

述語とは異なり、functorには厳密な正確性要件があります。

|状態 |適用対象 |説明 |
| --- | --- | --- |
| **存在** | `means` のみ |少なくとも 1 つの値が条件 | を満たすことを証明します。
| **独自性** | `means` のみ |最大 1 つの値が条件 | を満たすことを証明します。
| **整形式** |両方 |定義する式/式は構文的に有効である必要があります。
| **型の正確性** |両方 |すべての用語は型が正しい必要があります |
| **定義** | `assume` |想定されたドメインの外では、functorは未定義です。

**構文**：

```ebnf
correctness_conditions ::= [ existence_block ] [ uniqueness_block ] ;
existence_block        ::= "existence" justification ";" ;
uniqueness_block       ::= "uniqueness" justification ";" ;
```

**証明付きの例**:

```mizar
definition
  let G be Group;
  func InverseDef: a" -> Element of G means it * a = 1_G & a * it = 1_G;
  existence
  proof
    :: Goal (auto-generated): ex b being Element of G st b * a = 1_G & a * b = 1_G
    consider b being Element of G such that
      A1: b * a = 1_G & a * b = 1_G by group_inverse_exists;
    take b;
    thus b * a = 1_G & a * b = 1_G by A1;
  end;
  uniqueness
  proof
    :: Goal (auto-generated): for b1, b2 being Element of G st
    ::   (b1 * a = 1_G & a * b1 = 1_G) & (b2 * a = 1_G & a * b2 = 1_G)
    ::   holds b1 = b2
    let b1, b2 be Element of G;
    assume A1: b1 * a = 1_G & a * b1 = 1_G;
    assume A2: b2 * a = 1_G & a * b2 = 1_G;
    b1 = b1 * 1_G by group_identity
       .= b1 * (a * b2) by A2
       .= (b1 * a) * b2 by group_assoc
       .= 1_G * b2 by A1
       .= b2 by group_identity;
    hence b1 = b2;
  end;
end;
```

> [！注記]
> `equals` によって定義されたfunctorは構築によって明確に定義されており、存在証明や一意性証明を必要としません。

### 10.6.1 functorのプロパティ

functorは、検証器が自動等式推論に使用する代数プロパティを宣言できます。各性質には証明義務が必要です。

**構文 (EBNF):**

```ebnf
func_property ::= ( "commutativity" | "idempotence"
                | "involutiveness" | "projectivity" ) justification ";" ;
```

**財産とFOLの義務:**

型 T の引数を持つ二項functor `F(a, b)` または単項functor `F(a)` の場合:

|プロパティ |アリティ | FOLの義務 |検証器の効果 |
|---|---|---|---|
| `commutativity` |バイナリ | `∀a,b:T. F(a,b) = F(b,a)` |一致する引数を自動的に並べ替えます |
| `idempotence` |バイナリ | `∀a:T. F(a,a) = a` | `F(x,x)` を `x` に簡略化します。
| `involutiveness` |単項 | `∀a:T. F(F(a)) = a` |二重申請を簡素化 |
| `projectivity` |単項 | `∀a:T. F(F(a)) = F(a)` |繰り返しの適用を簡素化 |

**例：**

```mizar
definition
  let X, Y be set;
  func UnionDef: X \/ Y -> set means
    for z being object holds z in it iff z in X or z in Y;
  existence proof ... end;
  uniqueness proof ... end;
  commutativity
  proof
    let A, B be set;
    thus A \/ B = B \/ A by mml.set.Union_commutative;
  end;
  idempotence
  proof
    let A be set;
    thus A \/ A = A by mml.set.Union_idempotent;
  end;
end;

definition
  let x be Real;
  func NegDef: - x -> Real equals 0 - x;
  involutiveness
  proof
    let a be Real;
    thus - (- a) = a by mml.number.real.Neg_involutive;
  end;
end;

definition
  let x be Real;
  func AbsDef: |.x.| -> Real means
    (x >= 0 implies it = x) & (x < 0 implies it = -x);
  existence proof ... end;
  uniqueness proof ... end;
  projectivity
  proof
    let a be Real;
    thus |.|.a.|.| = |.a.| by mml.number.real.Abs_projective;
  end;
end;
```

> [！注記]
> プロパティは、定義式と正確性条件の後に、`definition ... end;` ブロック内に配置されます。

## 10.7 functorの再定義

既存のfunctorは、より具体的な型を提供するように再定義できます。
- より狭い戻り値の型
- 簡略化された定義条件
- タイプ固有の最適化

**構文**：

```ebnf
redefine_func  ::= "redefine" "func" label ":" func_pattern "->" type_expression
                   ( "means" formula_definiens | "equals" term_definiens ) ";"
                   "coherence" [ "with" label ] justification ";" ;
```

**例**：

```mizar
definition
  let x be Real;
  func AbsGeneral: |.x.| -> Real means
    (x >= 0 implies it = x) & (x < 0 implies it = -x);
end;

definition
  let x be non negative Real;
  redefine func AbsNonNeg: |.x.| -> non negative Real equals x;
  coherence with AbsGeneral
  proof
    :: Goal (auto-generated): |.x.| = x & |.x.| is non negative Real
    :: i.e., the original definition satisfies the new equals clause,
    ::       and the result has the refined return type
    thus |.x.| = x by AbsGeneral;  :: x >= 0 implies |.x.| = x
    thus |.x.| is non negative Real;  :: x is non negative, so |.x.| = x is non negative
  end;
end;
```

再定義要件:

* `redefine func`を使用する
* 再定義された戻り値の型は、元の戻り値のサブタイプである必要があります。
* 新しいパラメータ タイプでの等価性を示す `coherence` 証明を提供します
* 以前の使用はすべてタイプセーフのままでなければなりません

再定義(redefine)は *同じ* functorを精密化します。元定義と同じ値を返し、ただより精密に記述するだけなので、競合する overload ではありません。このため、同じfunctorに対する複数の再定義が使用箇所で適用可能であっても、検証器はそれを曖昧とは扱いません。解決が元のfunctorに確定した後、検証器は、`coherence` 証明が受理済みである適用可能なすべての再定義の結果型を統合します（第 19 章ではこの統合の手順を *再定義の統合（refinement join）* と呼びます）。

- ある結果型が他の型以上に具体的であれば、呼び出し側にはその最も強い型が見えます。
- 結果型が同じ基底のモードまたは構造を持ち、属性が互いに矛盾しなければ、呼び出し側にはそのモードまたは構造が、すべての属性をまとめて担ったもの（有効なクラスタで閉じたもの）として見えます。
- 結果型が互いに非互換なモードや構造の上にある場合、または統合した属性が互いに矛盾する場合、検証器は incompatible redefinition（互換でない再定義）エラーを報告します。

この緩和は、まったく同じ元のfunctorに対する再定義にのみ適用されます。名前を共有するだけの真に異なる 2 つのfunctorは、依然として第 19 章の曖昧 overload 規則に従います。

## 10.8 表記スタイル

サポートされている表記法:

|表記法 |パターン |例 |
| --- | --- | --- |
| **中置** | `x symbol y` | `x + y`、`X \/ Y` |
| **プレフィックス** | `symbol x` | `- x`、`card X` |
| **後置** | `x symbol` | `n !`、`f "` |
| **周囲固定 (ブラケット)** | `left x right` | `[: X, Y :]`、`|. x .|` |
| **機能的** | `symbol ( args )` | `sin(x)`、`max(a, b)` |

**例**:

```mizar
:: Infix
func AddDef: x + y -> Real equals ...;
func UnionDef: X \/ Y -> set means ...;

:: Prefix
func NegDef: - x -> Real equals ...;
func CardDef: card(X) -> Cardinal means ...;

:: Postfix
func FactorialDef: n ! -> Nat equals Product(id(Seg n));
func InverseDef: f " -> Function means ...;

:: Circumfix (bracket notation)
func CartesianDef: [: X, Y :] -> set means ...;
func AbsDef: |. x .| -> Real means ...;

:: Functional (parenthesized)
func SinDef: sin(x) -> Real means ...;
func MaxDef: max(a, b) -> Real equals ...;
```

## 10.9 演算子の優先順位と結合性

演算子の優先順位と結合性は、スタンドアロンの宣言ステートメントを使用して宣言されます。

```mizar
infix_operator("+", left, 80);
infix_operator("*", left, 90);
infix_operator("^", right, 95);
infix_operator("<", none, 50);
prefix_operator("-", 85);
postfix_operator("!", 95);
```

**構文**：

|フォーム |結合性 |例 |
| --- | --- | --- |
| `infix_operator("sym", left, N);` |左連想 | `a + b + c` → `(a + b) + c` |
| `infix_operator("sym", right, N);` |右連想 | `a ^ b ^ c` → `a ^ (b ^ c)` |
| `infix_operator("sym", none, N);` |非関連 | `a < b < c` → 解析エラー |
| `prefix_operator("sym", N);` | — | `- (- x)` は優先順位によって解析されます |
| `postfix_operator("sym", N);` | — | `n !` は優先順位によって解析されます |

優先順位の値の範囲は 0 ～ 255 です。値が大きいほど、結合がより強くなります。宣言のないシンボルのデフォルトの優先順位は 64 (非結合) です。

**組み込みシンボル。** 基本シンボル (`=`、`<>`、`in`) は、従来の Mizar の `hidden.miz` と同様に、暗黙的にインポートされたコア moduleで宣言された優先順位を持ちます。これらの宣言はオーバーライドできません。組み込みシンボルのユーザー宣言はリンク時エラーです。

**競合処理** 同じユーザー定義シンボルに対して異なる優先順位を宣言する 2 つのmoduleにより、リンク時に競合エラーが発生します。パッケージ内のすべてのmoduleは、すべてのシンボルの優先順位に同意する必要があります。

## 10.10 シンボルの解決とインポート

ユーザー定義のfunctorは、その定義moduleをインポートすることで利用可能になります。

```mizar
import algebra.group as grp;
let G be set;
let x = grp.center(G);
```

名前間の競合は次のように解決されます。

* **レクスト中の最長一致ルール**
* **必要に応じて修飾名**

**例**：

```mizar
import geometry.vector;
import linear_algebra.vector as linalg;

:: Use qualified names to disambiguate
let v = geometry.vector.zero;
let w = linalg.zero;
```

> [！注記]
> functorの同義語については、「[第 11 章 (シンボル管理)、§11.1](./11.symbol_management.md#111-同義語と反意語)」を参照してください。

## 10.11 関数の 3 つの表現

Mizar-Evo は、関数を操作するための 3 つの異なるメカニズムを提供します。それぞれは言語の異なるレベルで動作し、スコープ、型指定、論理ステータスが異なります。正確さと読みやすさのためには、適切な表現を選択することが不可欠です。

### 10.11.1 概要

|代表 |宣言 |範囲 | FOLステータス | `is`-述語 |
|---|---|---|---|---|
|オブジェクトレベルのfunctor | `definition` ブロック内の `func` 定義 |グローバル |一階項 |サポートされている |
|ローカル用語の略語 |証明ブロックの`deffunc` |証明ローカル |用語の置換 |該当なし |
|スキーマfunctorパラメータ | `definition` ブロック内の `let F be func(T) -> S;` |templateスキーマ |メタレベルシンボル |サポートされていません |

### 10.11.2 オブジェクトレベルのfunctor

標準の `func` 定義 (10.1 ～ 10.10 章) では、**グローバル一階関数シンボル** が導入されています。定義されたfunctorは集合論的関数です。その出力は集合 (談話領域の要素) であり、その定義により FOL 公理が生成されます。

```mizar
:: A functor is a globally available, first-order term-forming operation
definition
  let G be Group;
  func neutral[G] -> Element of G means
    for a being Element of G holds it * a = a & a * it = a;
  existence proof ... end;
  uniqueness proof ... end;
end;

:: neutral[G] is a first-order term; it can appear in type expressions
let e be Element of G;
e = neutral[G];

:: Mode-based function parameters are also first-order:
:: Homomorphism[G,H] is defined as a mode (Chapter 7 / Chapter 18)
let phi be Homomorphism[G, H];
:: phi is a set element; 'is' predicates apply
assume phi is injective;
```

オブジェクトレベルのfunctorは以下をサポートします。
- `is` 述語推論 (例: `phi is injective`、`phi is bijective`)
- モード宣言 (例: `mode Homomorphism[G,H] is ...`)
- 他のモードやfunctorへのパラメーターとして使用
- 検証器による完全な FOL 推論

### 10.11.3 ローカル用語の略語 (`deffunc`)

`deffunc` 構造 (第 15 章、§15.2.3) では、**証明ローカル用語の略語**が導入されています。新しい FOL 関数シンボルは作成されません。検証中に定義されたパターンの出現を定義した式に置き換えるのは糖衣構文です。

```mizar
proof
  deffunc F(n) = 2 * n + 1;
  :: F(3) is immediately reduced to 2 * 3 + 1 = 7 by the verifier
  :: F is not available outside this proof block
  thus F(0) = 1;
end;
```

`deffunc` は、関数式が 1 つの証明内で繰り返し使用されるが、永続的な名前は必要ない場合に適しています。これは一階オブジェクトではなく**、FOL 公理を生成しません。

### 10.11.4 スキーマfunctorパラメータ

`definition` ブロック内の `let F be func(T) -> S;` 宣言 (第 18 章、§18.2.4) では、パラメーター化された定理スキーマで使用する **スキーマレベルの関数変数** が導入されています。これは、グローバル functorでもローカルの略語でもありません。templateによって生成された FOL スキーマ内の未解釈の関数シンボルです。

```mizar
definition
  let A be type;
  let F be func(A) -> A;   :: schema functor: NOT a set element
  let P be pred(A);
  theorem StepInvariant:
    for x being A st P(x) holds P(F(x));
end;
```

スキーマfunctorパラメーターは、オブジェクト理論の集合要素ではないため、`is` 述語推論をサポートしません**。これらはメタレベルで「この署名の任意の機能」を表します。

**結果**: 定理が関数の引数が代数的性質を満たすことを要求する場合 (例: 「群準同型である」)、代わりに **モードベースのパラメーター** を使用します。

```mizar
:: Correct: mode-based, phi is a first-order object, supports 'is'
definition
  let M be Monoid;
  let N be Monoid;
  let phi be Homomorphism[M, N];   :: mode, not func(...)
  theorem HomPreservesNeutral:
    phi(neutral[M]) = neutral[N]
  proof ... end;
end;

:: Avoid: schema functor + is-predicate (not supported)
:: let phi be func(Element of M) -> Element of N;
:: assume phi is group_homomorphism;   :: ERROR: is not applicable to schema functor
```

### 10.11.5 意思決定ガイド

|質問 |推薦 |
|---|---|
|関数には永続的なグローバル名が必要ですか? |オブジェクトレベルのfunctor (`func` 定義) |
|この関数は 1 つの証明内でのみ必要ですか? | `deffunc` ローカルの略語 |
|定理スキーマは、特定の型の **任意** 関数に対して有効ですか? |スキーマfunctorパラメーター (`let F be func(T) -> S;`) |
|関数の引数には代数的プロパティ (`is injective` など) が必要ですか? |モードベースのパラメータ (例: `let phi be Homomorphism[G,H];`) |
|関数は計算 (合成、制限など) の出力ですか? |オブジェクトレベルのfunctor |

> [！注記]
> `deffunc` の構文と意味論については、[第15章 (文), §15.2.3](./15.statements.md#1523-インライン関数定義-deffunc) を参照してください。スキーマfunctorパラメーターとtemplate定理については、「[第 18 章 (テンプレート)、§18.2.4](./18.templates.md#1824-functorパラメータ)」を参照してください。

## 10.12 論理エンコーディングの詳細

このセクションでは、functorが一階論理にどのようにマップされるかを説明します。

### 10.12.1 FOL 関数シンボルとしてのfunctor

各ユーザー定義functorは、FOL 関数シンボルとしてエンコードされます。

|Mizar source | FOL エンコーディング |
| --- | --- |
| `x + y` | `add(x, y)` |
| `card X` | `card(X)` |
| `[: X, Y :]` | `cartesian_product(X, Y)` |
| `f . x` | `apply(f, x)` |

### 10.12.2 方程式としての `equals` の定義

`equals` によって定義されたfunctorは、普遍的に定量化された方程式を生成します。

```
func twice(n) equals 2 * n
⇒  forall n (twice(n) = 2 * n)
```

より一般的には:

```
func F(x1, ..., xn) equals t
⇒  forall x1, ..., xn (F(x1, ..., xn) = t)
```

### 10.12.3 一意の存在としての `means` の定義

`means` によって定義されたfunctorは、一意の存在を主張する公理を生成します。

```
func x / y means it * y = x
⇒  forall x, y (
     exists! z (z * y = x) ∧
     div(x, y) * y = x
   )
```

`it` プレースホルダーは、生成された公理内の関数アプリケーションによって置き換えられます。

### 10.12.4 ガードとしての型付きパラメータ

パラメータの型制約は先行詞としてエンコードされます。

|Mizar source | FOL エンコーディング |
| --- | --- |
| `let x, y be Real; func x + y -> Real` | `forall x, y (is_Real(x) ∧ is_Real(y) → is_Real(add(x, y)))` |
| `let G be Group; func center(G) -> Subgroup of G` | `forall G (is_Group(G) → is_Subgroup(center(G), G))` |

### 10.12.5 ドメイン制限 (`assume`)

functorが `assume` を使用する場合、条件は追加の先行詞になります。

```
let x, y be Real;
assume y <> 0;
func x / y -> Real means ...

⇒  forall x, y (is_Real(x) ∧ is_Real(y) ∧ y ≠ 0 → is_Real(div(x, y)) ∧ div(x, y) * y = x)
```

### 10.12.6 戻り値の型のエンコーディング

戻り値の型により、結果の型アサーションが生成されます。

|Mizar source | FOL エンコーディング |
| --- | --- |
| `func f(x) -> Nat` | `forall x (...preconditions... → is_Nat(f(x)))` |
| `func g(X) -> non empty set` | `forall X (...preconditions... → is_set(g(X)) ∧ ¬is_empty(g(X)))` |

### 10.12.7 再定義エンコーディング

再定義により、制限された型の下で等価公理が生成されます。論理項としての中核は元のfunctorシンボルのままであり、再定義は、制限されたドメインについて検証済みの等式と、必要に応じてより強い結果型の事実を追加します。

```
redefine func |.x.| -> non negative Real equals x;
⇒  forall x (
     is_Real(x) ∧ is_non_negative(x)
     → abs(x) = x ∧ is_non_negative(abs(x))
   )
```

同じ呼び出し箇所で複数の再定義が有効である場合、検証器は §10.7 のとおりそれらを統合した結果型を公開できます。再定義本体どうしが一致することを別途証明する必要はありません。各本体は、適用される範囲で同じ元のfunctorと等しいことをすでに証明しているため、そこでは互いにも自動的に等しくなるからです。

## 10.13 完全な構文 (EBNF)

functor関連の構文の概要。

```ebnf
(* Functor Definition *)
func_def         ::= "func" label ":" func_pattern "->" type_expression
                     ( "means" formula_definiens | "equals" term_definiens ) ";"
                     [ correctness_conditions ] ;
label            ::= identifier ;
func_pattern     ::= [ loci ] functor_symbol [ loci ] ;
loci             ::= locus | "(" locus_list ")" ;
locus_list       ::= locus { "," locus } ;
locus            ::= identifier ;
functor_symbol   ::= identifier | symbolic_func ;
symbolic_func    ::= symbol_char+ ;
formula_definiens ::= formula
                    | formula_case { "," formula_case } [ "otherwise" formula ] ;
formula_case      ::= formula "if" formula ;
term_definiens    ::= term_expression
                    | term_case { "," term_case } [ "otherwise" term_expression ] ;
term_case         ::= term_expression "if" formula ;

(* Correctness Conditions *)
correctness_conditions ::= [ existence_block ] [ uniqueness_block ] ;
existence_block  ::= "existence" justification ";" ;
uniqueness_block ::= "uniqueness" justification ";" ;

(* Functor Properties *)
func_property    ::= ( "commutativity" | "idempotence"
                   | "involutiveness" | "projectivity" ) justification ";" ;

(* Functor Redefinition *)
redefine_func    ::= "redefine" "func" label ":" func_pattern "->" type_expression
                     ( "means" formula_definiens | "equals" term_definiens ) ";"
                     "coherence" [ "with" label ] justification ";" ;

(* Functor Application - see Chapter 13 *)
functor_application ::= [ functor_loci ] functor_symbol [ functor_loci ]
                      | left_bracket loci right_bracket ;
functor_loci       ::= locus | "(" loci ")" ;
loci               ::= locus { "," locus } ;

(* Operator Precedence Declaration *)
infix_operator_decl   ::= "infix_operator"   "(" string_literal "," infix_assoc "," nat_literal ")" ";" ;
prefix_operator_decl  ::= "prefix_operator"  "(" string_literal "," nat_literal ")" ";" ;
postfix_operator_decl ::= "postfix_operator" "(" string_literal "," nat_literal ")" ";" ;
infix_assoc           ::= "left" | "right" | "none" ;
```
