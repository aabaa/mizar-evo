## 15.1 概要

Mizar のステートメントは、数学的推論の構成要素を形成します。それらは、定理の証明や補助推論ブロックなど、さまざまな文脈で現れます。この章では、すべてのステートメント形式、その構文、意味論、および適切な使用法について説明します。

ステートメントは大きく次のように分類できます。
- **宣言ステートメント**: 変数、定数、または仮定を導入します。
- **正当化声明**: 裏付けとなる証拠とともに命題を主張する
- **構造ステートメント**: 証明フローと推論パターンを整理する
- **型関連のステートメント**: 型の変換と再解釈を処理します。

## 15.2 変数と定数の概要

### 15.2.1 一般化ステートメント (`let`)

`let` ステートメントは、オプションの制約を持つ普遍的に定量化された変数を導入します。

**構文 (EBNF):**

```ebnf
generalization              ::= "let" qualified_vars [ "such" conditions ] [ "by" references ] ";" ;
qualified_vars              ::= explicit_qualified_vars [ "," implicit_qualified_vars ]
                              | implicit_qualified_vars ;
explicit_qualified_vars     ::= qualified_segment { "," qualified_segment } ;
qualified_segment           ::= variable_list qualification ;
implicit_qualified_vars     ::= variable_list ;
variable_list               ::= identifier { "," identifier } ;
qualification               ::= ( "being" | "be" ) type_expression ;
conditions                  ::= "that" proposition { "and" proposition } ;
```

**例：**

```mizar
let x be Real;
let y, z be Real such that y < z;
let n be Nat, x be Real;
let G be Group;
let H be Subgroup of G;
let B be Basis[V] by FinDim;
let n be Nat such that n > 0 and n is prime;
```

**意味論:**
- 証明のコンテキスト: 任意だが固定された値を導入します (普遍的な一般化)
- 定義コンテキスト: 定義のパラメータを導入します。
- スコープ内のシャドウイングは許可されていますが、明確にするために使用は控えめにする必要があります ([Ch.4 §4.6](./04.variables_and_constants.md#46-スコープとシャドウイング) を参照)
- `such that` 句は、導入された変数に関する仮定を追加します。
- 末尾の `by references` 句は、use site で宣言型が導入する well-formedness constraint を放免するためのものです。任意の仮定を追加するものではありません。参照されたラベルまたは定理は、変数がスコープに入る前に生成された declaration constraint を証明しなければなりません。
- 型付きセグメントを複数並べることができ、同一セグメント内の変数はそのセグメントの型を共有します

> [！注記]
> 変数の宣言と有効範囲の規則については、「[第4章 (変数と定数)](./04.variables_and_constants.md)」を参照してください。
> 証明における `let` ステートメントは、証明された定理における全称数量化に対応します。 `for ... holds` パラレルについては、[第14章 (式), §14.4.1](./14.formulas.md#1441-ユニバーサル定量化-for) を参照してください。

### 15.2.2 定数定義(`set`)

`set` ステートメントでは、用語の略語が導入されています。

**構文 (EBNF):**

```ebnf
constant_definition ::= "set" equating_list ";" ;
equating_list       ::= equating { "," equating } ;
equating            ::= identifier "=" term_expression ;
```

**例：**

```mizar
set x = a + b;
set y = f.x;
set n = card X;
set dom_f = dom f;        :: abbreviation for complex subexpressions
set prod = [:A, B:];
set a = f.x, b = f.y;     :: multiple abbreviations
```

**意味論:**
- 検証中に capture を避ける置換で展開される構文上の略語を作成します
- 型は右辺の式から推測されます。
- ラベルは必要ありません (ラベルは定義ではなく命題に使用されます)
- 1 つの `set` statement に複数の assignment を書いた場合、左から右へ導入します
- 右辺の自由変数は、定義位置で解決済み binding ID によって捕捉されます。
  後で同じ表示名が shadowing されても abbreviation の意味は変わりません。

### 15.2.3 インライン関数定義 (`deffunc`)

証明内でローカル関数の略語を定義します。

**構文 (EBNF):**

```ebnf
inline_functor_definition ::= "deffunc" identifier "(" [ typed_params ] ")" "->" type_expression
                              "equals" term_expression ";" ;
typed_params              ::= typed_param { "," typed_param } ;
typed_param               ::= identifier ( "being" | "be" ) type_expression ;
```

**例：**

```mizar
proof
  deffunc F(n be Nat) -> Nat equals n * (n + 1);
  deffunc G(x be Real, y be Real) -> Real equals x^2 + y^2;
  ...
  F(3) = 12;              :: by definition expansion
  G(3, 4) = 25;           :: by definition expansion
end;
```

**意味論:**
- インラインfunctorは使用場所ごとに capture を避ける置換によって展開されます。
- 範囲は周囲の証明ブロックに限定されます - 外部からは見えません
- 戻り値の型は明示的に指定する必要があります
- 引数なしの定義が許可されます: `deffunc C() -> Nat equals 42;`
- 定義項の自由変数は定義位置で捕捉されます。
- 適用箇所は、展開前に形式パラメータの型 guard を満たさなければなりません。

> [！注記]
> インラインfunctor **application** の構文については、[第13章 (項式), §13.2.3](./13.term_expression.md#1323-インラインfunctorアプリケーション) を参照してください。

### 15.2.4 インライン述語定義 (`defpred`)

証明内のローカル述語の省略形を定義します。

**構文 (EBNF):**

```ebnf
inline_predicate_definition ::= "defpred" identifier "(" [ typed_params ] ")" "means" formula ";" ;
```

**例：**

```mizar
proof
  defpred P(n be Nat) means n is prime & n > 2;
  defpred Q(x be Real, y be Real) means x < y & y < x + 1;
  ...
  P(n) implies n is odd;
  for n being Nat st P(n) holds n is odd;  :: using defpred in quantified formulas
end;
```

**意味論:**
- インライン述語は使用箇所ごとに capture を避ける置換により展開されます。
- 範囲は周囲の証明ブロックに限定されます - 外部からは見えません
- 引数なしの定義が許可されます: `defpred Trivial() means 0 = 0;`
- 定義論理式の自由変数は定義位置で捕捉されます。
- 適用箇所は、展開前に形式パラメータの型 guard を満たさなければなりません。

> [！注記]
> インライン述語 **application** の構文については、[第14章 (式), §14.2.2](./14.formulas.md#1422-インライン述語アプリケーション) を参照してください。

## 15.3 仮定と主張

### 15.3.1 単一の仮定 (`assume`)

仮定として 1 つの命題を導入します。

**構文 (EBNF):**

```ebnf
single_assumption ::= "assume" proposition ";" ;
proposition       ::= [ label_identifier ":" ] formula ;
```

**例：**

```mizar
assume x > 0;
assume A1: G is abelian;
assume A2: for x being Element of G holds x * x = 1_G;
```

> [！注記]
> 命題で使用される式の構文については、「[第14章 (式)](./14.formulas.md)」を参照してください。

### 15.3.2 集団的仮定 (`assume that`)

複数の関連する仮定が導入され、それぞれに独自のラベルが付けられる場合があります。

**構文 (EBNF):**

```ebnf
collective_assumption ::= "assume" conditions ";" ;
conditions            ::= "that" proposition { "and" proposition } ;
```

**例：**

```mizar
assume that A1: x > 0
        and A2: y > 0
        and x < y;
```

> [！重要]
> ここでの `and` キーワードは、式レベルの論理積 `&` ではなく、**ステートメント レベルの条件区切り文字**です。 `and` で区切られた各命題は独自のラベルを持つことができ、後の `by` 正当化 (例: `by A1, A2`) で独立して参照できます。対照的に、`&` は、サブラベルなしで 2 つの式を 1 つの式に結合します。同様に、ステートメントの `such that` は、定量化された式 (Ch.14) の `st` とは異なります。どちらも条件を導入しますが、`such that` はラベルのサポートによりステートメント レベルで動作します。 `&` 演算子については、「[第14章 (式), §14.3.1](./14.formulas.md#1431-接続詞-)」を参照してください。

### 15.3.3 存在の仮定 (`given`)

プロパティを備えた存在的に定量化された変数を導入します。

**構文 (EBNF):**

```ebnf
existential_assumption ::= "given" qualified_vars [ "such" conditions ] ";" ;
```

**例：**

```mizar
given x being Real such that x^2 = 2;
given n being Nat, x being Real such that x = n;
given G being Group such that G is finite and card G = p;
given n being Nat such that A1: n > 0 and A2: n is prime;
```

**意味論:**
- 与えられた条件を満たす Skolem 定数を導入します
- 自然演繹における実存的インスタンス化に相当
- 型付きセグメントを複数並べることができ、同一セグメント内の変数はそのセグメントの型を共有します
- すでに仮説として存在が確定している場合に使用

### 15.3.4 選択ステートメント (`consider`)

必須の正当性を伴う条件を満たす変数を導入します。

**構文 (EBNF):**

```ebnf
choice_statement ::= "consider" qualified_vars "such" conditions simple_justification ";" ;
```

**例：**

```mizar
consider x being Real such that x > 0 and f.x = 0 by A1, Th5;
consider p being Element of NAT such that p is prime and p > n by Euclid;
consider x, y being Real such that x < y and y < z by A2;
```

> [！注記]
> `given` とは異なり、`consider` は存在を証明するために明示的な正当化 (`by` 節) を必要とします。

## 15.4 結論と導出

### 15.4.1 直接結論 (`thus`)

現在の証明の目標を前進させる命題を正当化して主張します。

**構文 (EBNF):**

```ebnf
conclusion ::= ( "thus" | "hence" ) proposition justification ";" ;
```

**例：**

```mizar
thus A3: x > 0 by A1, A2;
thus thesis by Lemma1, Def3;
thus x in A & x in B by A1, A2;  :: concluding a conjunction
```

### 15.4.2 逐次結論 (`hence`)

`then thus` と同等 — 正当化の一部として直前のステートメントを暗黙的に参照します。

**例：**

```mizar
assume A1: x > 0;
then x^2 > 0 by Square_positive;
hence x^2 + 1 > 1;  :: implicitly uses preceding "x^2 > 0"
```

### 15.4.3 ブロック結論 (`hereby`)

現在の証明の目標を正当化する結果となる推論ブロックを導入します。 `thus now ... end`と同等。

**構文 (EBNF):**

```ebnf
diffuse_conclusion ::= "hereby" reasoning "end" ";" ;
```

**例：**

```mizar
hereby
  assume x > 0;
  then x^2 > 0 by Th1;
  hence thesis;
end;
```

### 15.4.4 証人の紹介 (`take`)

証拠内の存在主張の証人を提供します。

**構文 (EBNF):**

```ebnf
exemplification ::= "take" example_list ";" ;
example_list    ::= example { "," example } ;
example         ::= term_expression | identifier "=" term_expression ;
```

**例：**

```mizar
:: Proving: ex n being Nat st n > 100
proof
  take 101;
  thus 101 > 100;
end;

:: With named witness
proof
  take k = 2 * n + 1;
  thus k is odd;
end;

:: Multiple witnesses
proof
  take x = a, y = b;
  ...;
end;
```

> [！注記]
> 複数の witness は左から右へ導入され、elaboration 後の existential goal の binder order と対応しなければなりません。

## 15.5 型管理

### 15.5.1 型の再検討 (`reconsider`)

用語のタイプビューを位置揃えで変更します。これは、値をより具体的な型として表示する **型の絞り込み** のメカニズムです。

**構文 (EBNF):**

```ebnf
type_changing_statement ::= "reconsider" type_change_list "as" type_expression simple_justification ";" ;
type_change_list        ::= reconsider_item { "," reconsider_item } ;
reconsider_item         ::= identifier | identifier "=" term_expression ;
```

**例：**

```mizar
:: Narrowing an existing variable
reconsider x as Element of NAT by A1;

:: Introducing a new constant with a narrowed type
reconsider f = F|A as Function of A, B by Th3;

:: Narrowing after establishing a property
assume A1: n >= 0;
reconsider n as Nat by A1, mml.number.integer.Nat_from_nonneg_Int;

:: Narrowing several values to the same type
reconsider x, y = f.x as Element of A by A2;
```

**意味論:**
- 証明義務を生成します。正当化は、ターゲット型 `T` の `is_T(x)` を証明する必要があります。
- `reconsider` statement 内のすべての item は同じ target type に対して検査されます
- 自動タイプ拡張 (サブタイプ → スーパータイプ) の場合、`reconsider` は必要ありません。

> [！注記]
> 縮小と拡大の関係については、[第3章 (型システム), §3.5](./03.type_system.md#35-サブタイプ化と拡張) を参照してください。
> 型推論の詳細については、「[第8章 (型推論)](./08.type_inference.md)」を参照してください。
> 用語削減 (`reduce`) は登録メカニズムであり、[第17章 (cluster と registration)](./17.clusters_and_registrations.md) で定義されています。

## 15.6 証明組織

### 15.6.1 ラベル付き推論ブロック (`now`)

> [！警告]
> `now ... end` は **非推奨です**。どのような命題が確立されているかを知る前に、読者はブロック全体を読む必要があるため、証明の意図を追跡することが難しくなります。明確に述べられた理論を備えた明示的な `thus <goal> by ...` ブロックまたは `proof ... end` ブロックを優先します。 `now` は、既存の Mizar 証明との互換性のために残されています。

後で参照できるように結論を導き出す、ラベル付きのサブ証明を作成します。

**構文 (EBNF):**

```ebnf
diffuse_statement ::= [ label_identifier ":" ] "now" reasoning "end" ";" ;
```

**例：**

```mizar
A1: now
  assume x > 0;
  then x^2 > 0 by Square_positive;
  thus x^2 + 1 > 1;
end;
:: A1 now refers to the derived conclusion "x > 0 implies x^2 + 1 > 1"

thus thesis by A1;
```

**意味論:**
- `now` ブロックは、すべての `thus` 結論を単一の結合に収集します。
- ラベルは、block skeleton を閉じて得られる式を参照します。
  `let` 導入は全称量化になり、assumption は先行詞になり、`thus` 結論は結論の conjunction になります。
- 後で `by` の位置揃えで参照できます
- block 内で `consider` または `given` によって導入された witness はローカル推論では使えますが、
  block から export されるラベル付き式に自由出現してはいけません。そのような漏出は scope error です。

### 15.6.2 ケース分析 (`case`)

事例による推論を構造化します。 `per cases` ブロック内で使用されます。

**構文 (EBNF):**

```ebnf
case_reasoning ::= [ "then" ] "per" "cases" simple_justification ";" ( case_list | suppose_list ) ;
case_list      ::= case_item { case_item } ;
case_item      ::= "case" ( proposition | conditions ) ";" reasoning "end" ";" ;
```

**例：**

```mizar
:: Prove a property for all natural numbers by splitting into two cases.
:: Nat_cases is a theorem asserting: for n being Nat holds n = 0 or n > 0
let n be Nat;
thus P[n]
proof
  per cases by mml.number.natural.Nat_cases;
  case C1: n = 0;
    thus thesis by C1, Base_case;
  end;
  case C2: n > 0;
    thus thesis by C2, Inductive_step;
  end;
end;
```

**意味論:**
- `per cases` の正当化は、ケースが網羅的であることを証明する必要があります (`φ₁ ∨ ... ∨ φₙ`)
- 各 `case` はその命題を前提として導入し、現在の目標を証明する必要があります
- `case` と `suppose` はどちらも `proposition` を受け入れるため、両方にラベルを付けることができます (例: `case C1: n = 0;`)。

### 15.6.3 仮定 (`suppose`)

代替ケース導入構文。 `case` の代替として `per cases` 内で使用されます。

**構文 (EBNF):**

```ebnf
suppose_list ::= suppose_item { suppose_item } ;
suppose_item ::= "suppose" ( proposition | conditions ) ";" reasoning "end" ";" ;
```

**例：**

```mizar
per cases;
suppose A1: x > 0;
  thus thesis by A1, Positive_case;
end;
suppose A2: x <= 0;
  thus thesis by A2, Non_positive_case;
end;
```

> [！注記]
> `case` と `suppose` は構文的に交換可能で、どちらも `proposition` (ラベルを含めることができる) または `conditions` を受け入れます。スタイルの好みに基づいて使用できます。

## 15.7 反復的等価性

等価推論ステップを中間の位置合わせで連鎖させ、段階的な等式導出を可能にします。

**構文 (EBNF):**

```ebnf
iterative_equality ::= [ label_identifier ":" ] term_expression "=" term_expression simple_justification
                       ".=" term_expression simple_justification
                       { ".=" term_expression simple_justification } ";" ;
```

**例：**

```mizar
A1: f.(x + y) = f.x + f.y by Additive
             .= g.x + f.y by A2
             .= g.x + g.y by A3;

:: Can be combined with the then modifier
assume A2: f = g;
then f.x + f.y = g.x + f.y by FuncEq
              .= g.x + g.y by A2;
```

**意味論:**
- `.=` の各ステップでは、独自の正当性を持つ 1 つの等式が導入されます。
- チェーンは推移性により `a = b`、`b = c`、`c = d` から全体的な等価性 `a = d` を導出します。
- ラベル (存在する場合) はチェーン全体 (つまり、全体的な等価性) を参照します。
- 割り当てではなく、正当な等価性の主張

## 15.8 正当化フォーム

Mizar のすべてのアサーションは、単純な参照リストまたは完全な証明ブロックによって正当化されなければなりません。

### 15.8.1 単純な位置揃え

**構文 (EBNF):**

```ebnf
simple_justification ::= [ "by" references ] ;
references           ::= reference { "," reference } ;
reference            ::= identifier [ template_args ]
                       | qualified_reference [ template_args ]
                       | grouped_reference
                       | bulk_reference ;
qualified_reference  ::= namespace_path "." identifier ;
grouped_reference    ::= namespace_path ".{" grouped_item { "," grouped_item } "}" ;
grouped_item         ::= identifier [ template_args ] ;
bulk_reference       ::= namespace_path ".*" ;
namespace_path       ::= identifier { "." identifier } ;
template_args        ::= "[" template_arg { "," template_arg } "]" ;
template_arg         ::= type_expression
                       | qua_arg
                       | defpred_identifier
                       | deffunc_identifier ;
qua_arg              ::= identifier { "qua" radix_type } ;
```

**例：**

```mizar
by A1, A2;
by mml.algebra.group.Lagrange_theorem;
by Math_induction[n, n^2 >= n];
by mml.number.natural.{Nat_add_comm, Nat_add_assoc};  :: grouped citation
by mml.number.natural.*;                               :: bulk citation
```

**参照タイプ:**
- **ローカル参照**: 現在の証明内のラベルと定理名 (例: `A1`、`Th5`)
- **修飾された参照**: namespace表記を使用した完全修飾パス (例: `mml.algebra.group.Th1`)
- **template のインスタンス化**: 明示的な引数を持つパラメーター化された定理 (例: `Induction[n, P]`)
- **グループ化された参照**: 共有プレフィックスを持つ同じmoduleからの複数の定理 (例: `mml.foo.{Th1, Th2}`)
- **一括参照**: moduleからのすべての公定理項目 (例: `mml.foo.*`)。トレードオフについては、[第 16 章 (定理と証明)、§16.5](./16.theorems_and_proofs.md#165-証明の正当性-by) を参照してください。

### 15.8.2 完全証明

**構文 (EBNF):**

```ebnf
justification ::= simple_justification | proof ;
proof         ::= "proof" reasoning "end" ;
```

## 15.9 ステートメント修飾子

### 15.9.1 シーケンシャル修飾子 (`then`)

`then` 修飾子は、直前のステートメントからの直接の派生を示します。これは、前のステートメントの結論を暗黙的に正当化に追加します。

**構文 (EBNF):**

```ebnf
statement ::= [ "then" ] linkable_statement | standalone_statement ;
standalone_statement ::= generalization
                       | constant_definition
                       | inline_functor_definition
                       | inline_predicate_definition
                       | assumption
                       | diffuse_statement
                       | diffuse_conclusion
                       | exemplification
                       | standalone_diagnostic_annotation ;
```

**例：**

```mizar
assume A1: x > 0;
then x^2 > 0 by Square_positive;
hence x^2 + 1 > 1;
```

> [！注記]
> `hence` は、`then thus` の糖衣構文です。前のステートメントから派生したケース分析のために、`then` 修飾子を `per cases` の前に置くこともできます。

## 15.10 スコープとコンテキストのルール

1. **変数のスコープ**: `let` によって導入された変数のスコープは、それを囲んでいるブロックに限定されます。
   - **同じスコープの再宣言は禁止されています**: 同じスコープ内で同じ変数名を 2 回宣言するとコンパイル エラーになります。
   - **内部スコープのシャドウイングが許可されています**: 内部ブロック内の変数 (`now ... end` など) は、同じ名前の外部変数をシャドウすることができます。内部宣言はそのブロック内で優先されます。
2. **ラベルのスコープ**: ラベルのスコープは、それを囲んでいる推論ブロックに限定されます。
   - **同じスコープの重複ラベルは禁止されています**: 同じスコープ内で同じラベル名を 2 回使用すると、コンパイル エラーになります。
   - **スコープ内のラベルのシャドウイングは禁止されています**: `by` 位置合わせにおけるあいまいな参照を避けるために、ラベルはすべての表示可能なスコープにわたって一意である必要があります。
3. **定義コンテキスト**: 定義ブロックでは、`let` によって定義パラメーターが導入されます。
4. **証明コンテキスト**: 証明ブロックでは、`let` は任意の固定値を導入します。
5. **インライン定義**: `deffunc` および `defpred` は、それを囲んでいる証明ブロックにスコープされており、その外部で参照することはできません。

> [！注記]
> スコープとシャドウイングのルールの詳細については、「[第4章 (変数と定数), §4.6](./04.variables_and_constants.md#46-スコープとシャドウイング)」を参照してください。

## 15.11 論理エンコーディングの詳細

このセクションでは、各ステートメント タイプが一階論理 (FOL) にどのようにマップされるかについて説明します。型のエンコード規則については、「[第3章 (型システム), §3.7](./03.type_system.md#37-論理エンコーディングの詳細)」を参照してください。式のエンコードについては、[第14章 (式), §14.7](./14.formulas.md#147-論理エンコーディングの詳細) を参照してください。

### 15.11.1 変数の導入 (`let`)

`let` ステートメントは、証明された定理における全称量化に対応します。

|Mizar source | FOL エンコーディング |
| --- | --- |
| `let x be T;` ... `thus φ;` | `∀x. (is_T(x) → φ)` |
| `let x be T by R;` ... `thus φ;` | side condition: `R ⊢ DeclOK(T)`; FOL body remains `∀x. (is_T(x) → φ)` |
| `let x be T such that A1: ψ;` ... `thus φ;` | `∀x. (is_T(x) ∧ ψ → φ)` |
| `let x, y be T;` ... `thus φ;` | `∀x, y. (is_T(x) ∧ is_T(y) → φ)` |

### 15.11.2 定数定義(`set`)

`set` ステートメントは、ローカルの略語を導入します。FOL 公理は生成されません。
後続の識別子の使用箇所は、定義時 closure として記録された項によって
capture を避ける置換で展開されます。

|Mizar source | FOL エンコーディング |
| --- | --- |
| `set c = t;` | capture を避ける置換: 後続の `c` の使用を、定義時 closure として記録された `t` で置き換える |

### 15.11.3 インライン定義 (`deffunc`、`defpred`)

インライン定義は、定義時 closure から capture を避ける置換によって展開されます。
ローカルな FOL 関数シンボルや述語シンボルは生成せず、証明コンテキストに
ローカル abbreviation 公理も追加しません。

|Mizar source | FOL エンコーディング |
| --- | --- |
| `deffunc F(x be T) -> U equals t;` | 展開規則: `is_T(a)` を検査した後、`F(a)` は `t[x/a]` へ低位化される |
| `defpred P(x be T) means φ;` | 展開規則: `is_T(a)` を検査した後、`P(a)` は `φ[x/a]` へ低位化される |

### 15.11.4 前提条件

仮定は、現在の証明コンテキストに前提条件を追加します。

|Mizar source | FOL エンコーディング |
| --- | --- |
| `assume φ;` | `φ` を前件として追加: 目標は `φ → ψ` になります |
| `assume that φ₁ and φ₂;` | `φ₁ ∧ φ₂` を前件として追加: 目標は `φ₁ ∧ φ₂ → ψ` になります |
| `given x be T such that ψ;` |実存インスタンス化: `∃x. (is_T(x) ∧ ψ)` → Skolem 定数 `c` と `is_T(c) ∧ ψ[x/c]` |
| `consider x, y be T such that ψ by J;` |両端揃え `∃x, y. (is_T(x) ∧ is_T(y) ∧ ψ)` → スコレム定数 |

### 15.11.5 結論

|Mizar source | FOL エンコーディング |
| --- | --- |
| `thus φ by J;` | 正当化 `J` を使用して `φ` を証明します |
| `hence φ;` | 前のステートメントと `J` を使用して `φ` を証明します |
| `take t, u;` | 実存 witness: goal `∃x:T, y:U. ψ` に対して witness 型義務 `is_T(t)` と `is_U(u)` を生成し、その後 `ψ[x/t, y/u]` をインスタンス化する |
| `take k = t;` | `take t;` と同じ witness 義務を生成し、残りの義務に対してローカル abbreviation `k = t` を導入する |

### 15.11.6 型管理

|Mizar source | FOL エンコーディング |
| --- | --- |
| `reconsider x as T by J;` |正当化 `J` を使用して `is_T(x)` を証明します。
| `reconsider y = t as T by J;` | `y = t` を導入し、`J` を使用して `is_T(t)` を証明します。
| `reconsider x, y = t as T by J;` | listed item がすべて `T` 型であることを証明し、equated item には name を導入します |

### 15.11.7 証明組織

|Mizar source | FOL エンコーディング |
| --- | --- |
| `A1: now assume φ; thus ψ; end;` | `A1` ≡ `φ → ψ` |
| `per cases by J; case φ₁; ... case φₙ;` | `J` は `φ₁ ∨ ... ∨ φₙ` を証明します。目標は各 `φᵢ` で証明される必要があります。
| `a = b by J1 .= c by J2;` | `a = b` (`J1` による) ∧ `b = c` (`J2` による) → `a = c` (推移性による) |

**複合例:**

```mizar
theorem Th1:
  for n being Nat holds n + 0 = n
```

FOL エンコーディング:

```
∀n. (is_Nat(n) → n + 0 = n)
```

**証明エンコーディングの例:**

```mizar
theorem Th2:
  for x being Real st x > 0 holds x^2 > 0
proof
  let x be Real;
  assume A1: x > 0;
  thus x^2 > 0 by A1, Square_positive;
end;
```

FOL エンコーディング:

```
∀x. (is_Real(x) → (x > 0 → x² > 0))
```

`thus` ステップでの証明義務は次のとおりです。

```
is_Real(x), x > 0  ⊢  x² > 0    (justified by A1, Square_positive)
```

## 15.12 完全な構文 (EBNF)

この章のすべてのステートメント関連構文の概要。

```ebnf
(* === Main Statement Categories === *)
statement             ::= [ "then" ] linkable_statement
                        | standalone_statement ;

linkable_statement    ::= compact_statement
                        | conclusion
                        | choice_statement
                        | type_changing_statement
                        | iterative_equality
                        | case_reasoning ;

standalone_statement  ::= generalization
                        | constant_definition
                        | inline_functor_definition
                        | inline_predicate_definition
                        | assumption
                        | diffuse_statement
                        | diffuse_conclusion
                        | exemplification
                        | standalone_diagnostic_annotation ;

(* === Variable and Constant Introduction === *)
generalization               ::= "let" qualified_vars [ "such" conditions ] [ "by" references ] ";" ;
constant_definition          ::= "set" equating_list ";" ;
equating_list                ::= equating { "," equating } ;
equating                     ::= identifier "=" term_expression ;
inline_functor_definition    ::= "deffunc" identifier "(" [ typed_params ] ")" "->" type_expression
                                 "equals" term_expression ";" ;
inline_predicate_definition  ::= "defpred" identifier "(" [ typed_params ] ")" "means" formula ";" ;
typed_params                 ::= typed_param { "," typed_param } ;
typed_param                  ::= identifier ( "being" | "be" ) type_expression ;

(* === Assumptions === *)
assumption                   ::= single_assumption
                               | collective_assumption
                               | existential_assumption ;
single_assumption            ::= "assume" proposition ";" ;
collective_assumption        ::= "assume" conditions ";" ;
existential_assumption       ::= "given" qualified_vars [ "such" conditions ] ";" ;
choice_statement             ::= "consider" qualified_vars "such" conditions
                                 simple_justification ";" ;

(* === Conclusions === *)
conclusion                   ::= ( "thus" | "hence" ) proposition justification ";" ;
diffuse_conclusion           ::= "hereby" reasoning "end" ";" ;
exemplification              ::= "take" example_list ";" ;
example_list                 ::= example { "," example } ;
example                      ::= term_expression | identifier "=" term_expression ;

(* === Type Management === *)
type_changing_statement      ::= "reconsider" type_change_list "as" type_expression
                                 simple_justification ";" ;
type_change_list             ::= reconsider_item { "," reconsider_item } ;
reconsider_item              ::= identifier | identifier "=" term_expression ;

(* === Structured Statements === *)
compact_statement            ::= proposition justification ";" ;
iterative_equality           ::= [ label_identifier ":" ] term_expression "=" term_expression
                                 simple_justification ".=" term_expression
                                 simple_justification
                                 { ".=" term_expression simple_justification } ";" ;
diffuse_statement            ::= [ label_identifier ":" ] "now" reasoning "end" ";" ;

(* === Case Analysis === *)
case_reasoning               ::= "per" "cases" simple_justification ";"
                                 ( case_list | suppose_list ) ;
case_list                    ::= case_item { case_item } ;
suppose_list                 ::= suppose_item { suppose_item } ;
case_item                    ::= "case" ( proposition | conditions ) ";" reasoning "end" ";" ;
suppose_item                 ::= "suppose" ( proposition | conditions ) ";" reasoning "end" ";" ;

(* === Supporting Rules === *)
label_identifier             ::= identifier ;
proposition                  ::= [ label_identifier ":" ] formula ;
conditions                   ::= "that" proposition { "and" proposition } ;
qualified_vars               ::= explicit_qualified_vars [ "," implicit_qualified_vars ]
                               | implicit_qualified_vars ;
explicit_qualified_vars      ::= qualified_segment { "," qualified_segment } ;
qualified_segment            ::= variable_list qualification ;
implicit_qualified_vars      ::= variable_list ;
variable_list                ::= identifier { "," identifier } ;
qualification                ::= ( "being" | "be" ) type_expression ;

(* === Justification === *)
justification                ::= simple_justification | proof ;
simple_justification         ::= [ "by" references ] ;
proof                        ::= "proof" reasoning "end" ;

references                   ::= reference { "," reference } ;
reference                    ::= identifier [ template_args ]
                               | qualified_reference [ template_args ]
                               | grouped_reference
                               | bulk_reference ;
qualified_reference          ::= namespace_path "." identifier ;
grouped_reference            ::= namespace_path ".{" grouped_item { "," grouped_item } "}" ;
grouped_item                 ::= identifier [ template_args ] ;
bulk_reference               ::= namespace_path ".*" ;
namespace_path               ::= identifier { "." identifier } ;
template_args                ::= "[" template_arg { "," template_arg } "]" ;
template_arg                 ::= type_expression
                               | qua_arg
                               | defpred_identifier
                               | deffunc_identifier ;
qua_arg                      ::= identifier { "qua" radix_type } ;
standalone_diagnostic_annotation ::= show_type_annotation | eval_annotation ;

(* For formula syntax, see Chapter 14 (Formulas) *)
(* For type_expression syntax, see Chapter 3 (Type System) *)
(* For term_expression syntax, see Chapter 13 (Term Expressions) *)
(* For show_type_annotation and eval_annotation, see Chapter 21 (Annotations). *)
```
