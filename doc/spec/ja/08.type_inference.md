## 8.1 基本的な型推論

型の推論は、継承、属性クラスター、および構造の互換性に基づいています。

```mizar
let G be Group;
reconsider M = G as Magma;  :: Group is inferred to be a Magma
```

推論エンジンは次のコンテキストで呼び出されます。

|コンテキスト |説明 |
| --- | --- |
| `let` 宣言 |モード定義から変数の型を推測します。
| `reconsider` |証明義務のある型変換を検証します。
| `consider` |存在的な選択のための型制約を検証します。
| `given` |存在仮定の型を紹介 |
| `take` |存在証明の証人の種類をチェックします |
|モードアプリケーション |パラメーター化された型を解決します。
|functor適用 |引数の型を検証し、戻り値の型を推測します。
|属性のテスト |クラスターを介して属性のメンバーシップを確認します。

## 8.2 型変換

暗黙的 (自動) と明示的 (ユーザー指定) の 2 種類の変換がサポートされています。

**構文 (EBNF)**:

```ebnf
qua_expression      ::= term_expression "qua" type_expression
                        { "qua" type_expression } ;
reconsider_stmt     ::= "reconsider" reconsider_item "as" type_expression
                        [ simple_justification ] ";" ;
reconsider_item     ::= identifier | identifier "=" term_expression ;
simple_justification ::= "by" references ;  (* can be omitted entirely *)
references          ::= reference { "," reference } ;
justification       ::= simple_justification | proof_block ;  (* see Chapter 15 *)
```

> [！注記]
> `reconsider` の使用状況の詳細については、**第 4 章 (変数と定数)** も参照してください。

### 8.2.1 暗黙的な変換 (拡張)

スーパータイプが予期される場所でサブタイプが使用される場合、検証器によって自動的に適用されます。

* **アップキャスト**: 例: `Group` → `Magma`、`Prime` → `Integer`
* **属性の影響**: クラスター登録経由
* **モード/構造ベースの互換性**: 構造サブタイプ

**例**：

```mizar
let n be Prime;
:: n can be used where Integer is expected (implicit widening)
let m be Integer;
reconsider m = n as Integer;  :: explicit, but no proof needed
```

### 8.2.2 明示的な変換

#### `qua` オペレーター (アップキャスト / ビュー)

`qua` 演算子は、**スーパータイプ** または **互換性のある構造タイプ** として値にアクセスします。追加の特性を主張せず、用語の制限されたビューを提供するだけであるため、正当化する必要はありません。

**特徴**:
- **正当な理由は必要ありません**: `qua` は暗黙的なアップキャストのみです
- **新しいバインディングは作成されません**: 元の用語には、タイプ制限されたビューを通じてアクセスされます
- **使用例**: 多重継承におけるフィールド/プロパティ アクセスの曖昧さをなくす

```mizar
let R be DoubleLoopStr;
(R qua AddLoopStr) is associative;  :: View R as AddLoopStr
(R qua MulLoopStr) is commutative;  :: View R as MulLoopStr
```

> [！注記]
> `qua` はダウンキャスト (より特定のタイプに絞り込む) には使用できません。そのためには、`reconsider` を使用してください。

#### `reconsider` ステートメント (ナローイング/ダウンキャスト)

より具体的な型に変換する場合に必要です。証明義務が発生します。

**`reconsider` ステートメント**: 異なる型の新しい変数を宣言するか、既存の変数を再解釈します (正当な理由が必要です)。

**フォーム 1: 新しいバインディング付き**:
```mizar
let x be Real;
assume x >= 0;
reconsider y = x as non negative Real by assumption;
```

**フォーム 2: 既存の変数を再解釈**:
```mizar
let n be Integer;
assume n >= 0;
reconsider n as Nat by A1;  :: Same variable, narrower type
```

**フォーム 3: 証明ブロック付き**:
```mizar
let f be Function;
reconsider g = f as Bijection
proof
  thus f is one-to-one & f is onto by ...;
end;
```

**属性付き型の絞り込み**:
```mizar
let S be set;
assume S <> {};
reconsider S as non empty set by assumption;

let G be Magma;
assume G is associative & G is unital & G is invertible;
reconsider H = G as Group by A1, A2, A3;
```

### 8.2.3 `qua` と `reconsider` の選択

|シナリオ |使用 |例 |
| --- | --- | --- |
|スーパー型のフィールド/プロパティにアクセスする | `qua` | `(R qua AddLoopStr).zero` |
|継承された属性の曖昧さをなくす | `qua` | `(R qua MulLoopStr) is commutative` |
|サブタイプに絞り込む (証明付き) | `reconsider` | `reconsider n as Prime by ...` |
|属性の追加 (証明付き) | `reconsider` | `reconsider S as non empty set by ...` |
|新しい型付きバインディングを作成する | `reconsider` | `reconsider y = x as Nat by ...` |

## 8.3 型チェックプロセス

型チェックのプロセスは次の順序で進行します。

1. **解析**: 型宣言と使用サイトを抽出します。
2. **解決**: モード定義と属性クラスターを適用します
3. **義務の生成**: コンバージョンの証明目標を作成する
4. **検証**: 推論または明示的な証明を使用した義務の履行
5. **変換を適用**: 暗黙的/明示的キャストを挿入します。

```
Declaration: let x be T
⇒  Assert: is_T(x)

Conversion: reconsider y = x as U
⇒  Goal: is_T(x) → is_U(x)
⇒  Proof required if not implied by clusters
```

## 8.4 論理エンコーディングの詳細

このセクションでは、型推論と変換が一階論理にどのようにマップされるかについて説明します。

### 8.4.1 型アサーション

変数宣言は型述語を生成します。

|Mizar source | FOL エンコーディング |
| --- | --- |
| `let x be Integer;` | `is_Integer(x)` |
| `let G be Group;` | `is_Group(G)` |

### 8.4.2 拡張（暗黙的な変換）

拡張はサブ型の公理に依存します。

```
mode Prime is prime Integer
⇒  forall x (is_Prime(x) implies is_Integer(x))
```

`Integer` が予期される場所で `Prime` が使用されると、その意味合いが自動的に適用されます。

### 8.4.3 ナローイング（明示的な変換）

絞り込むと証明義務が発生します。

|Mizar source | FOL エンコーディング |
| --- | --- |
| `reconsider y = x as Prime` | `Goal: is_Integer(x) ∧ is_prime(x)` |
| `(R qua AddLoopStr)` | `Assumption: is_AddLoopStr(R)` |

### 8.4.4 `qua` 演算子のエンコーディング

`qua` 演算子は、タイプ制限されたビューを作成します。 `x` という用語自体は、FOL エンコードでは変更されていません。 `qua` は type-dispatch のみに影響します。

**継承のアップキャスト:**

```
(x qua T).field
⇒  Validity obligation:  ⊢_FOL  inherit*(type_radix(x), T)
⇒  FOL result:           T_field(x)
```

**クラスターベースの拡大:**

```
f(x qua T)
⇒  Validity obligation:  ⊢_FOL  cl(A, type_radix(x))  for each A ∈ attributes(T)
⇒  Overload selected:    the overload of f whose parameter type matches T
```

どちらの場合も、有効性の義務は、`inherit` チェーンとクラスター登録の公理を使用して検証器によって自動的に免除されます。ユーザー証明は必要ありません。完全な FOL 公理セット (推移閉包、サイクル自由度、クラスター伝播、一貫性) については、[§13.8.7](./13.term_expression.md#1387-qua-演算子のエンコーディング) を参照してください。

## 8.5 完全な構文 (EBNF)

型推論と変換構文の概要。

> [！注記]
> 暗黙的な変換、overloadの解決、シンボルの再定義などの高度な型変換シナリオについては、**第 19 章 (overloadの解決)** を参照してください。

```ebnf
(* Explicit Conversion *)
qua_expression      ::= term_expression "qua" type_expression
                        { "qua" type_expression } ;
reconsider_stmt     ::= "reconsider" reconsider_item "as" type_expression
                        [ simple_justification ] ";" ;
reconsider_item     ::= identifier | identifier "=" term_expression ;

(* Justification - see Chapter 15 for full details *)
simple_justification ::= "by" references ;  (* can be omitted entirely *)
references          ::= reference { "," reference } ;
justification       ::= simple_justification | proof_block ;  (* see Chapter 15/18 *)

(* Type Expression - see Chapter 3 *)
type_expression     ::= attribute_chain radix_type ;
attribute_chain     ::= { [ "non" ] attribute_ref } ;
attribute_ref       ::= [ param_prefix ] [ struct_name "." ] attribute_name ;
param_prefix        ::= parameter "-" | "(" parameter_list ")" "-" ;
radix_type          ::= builtin_type | mode_name [ type_args ] | struct_name [ type_args ] ;
```
