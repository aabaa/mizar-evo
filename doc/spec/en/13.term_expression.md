## 13.1 Primary Expressions

Primary expressions represent atomic values, references, and basic constructs that form the building blocks of more complex terms.

**Syntax (EBNF)**:

```ebnf
primary_expression ::= variable_identifier
                     | "it"
                     | "(" term_expression ")"
                     | numeral
                     | bracket_functor_application ;
variable_identifier ::= identifier ;
numeral            ::= digit+ ;
bracket_functor_application ::= user_symbol term_list user_symbol
                              | "[" term_list "]" ;
term_list         ::= term_expression { "," term_expression } ;
```

### 13.1.1 Variable Identifiers

Variable identifiers are references to previously declared variables. Their usage follows the general scoping rules of the language.

See [Chapter 4 (Variables and Constants)](./04.variables_and_constants.md) for detailed rules on variable declaration, scope, shadowing, and reserved identifiers.

**Example**:

```mizar
let x be Nat;
let y be Nat;
x + y;
```

### 13.1.2 The `it` Keyword

`it` refers to the object currently being defined within a `func` or property implementation using the `means` style.

**Example**:

```mizar
definition
  let X, Y be set;
  func UnionDef: X \/ Y -> set means
    for z being object holds z in it iff z in X or z in Y;
end;
```

**Restrictions**:
- Only allowed inside a `func` definition with `means` or a `property ... means` implementation
- Refers exclusively to the return object or property value being defined
- Not available in `equals` definitions (the value is explicit)

### 13.1.3 Parenthesized Expressions

Used for grouping and precedence control.

**Example**:

```mizar
(a + b) * c           :: Override default precedence
(((a + b) * c) + d)   :: Arbitrary nesting
```

**Semantics**:
- Parentheses override default operator precedence
- Arbitrary nesting is supported
- Type of expression is preserved through parenthesization

### 13.1.4 Numerals

Numeric literals for natural numbers.

**Example**:

```mizar
0
42
1000
```

## 13.2 Functor Applications

Functor applications construct terms by applying user-defined functors to arguments.

**Syntax (EBNF)**:

```ebnf
functor_application ::= [ functor_loci ] functor_symbol [ functor_loci ] ;
functor_loci       ::= term_expression | "(" term_list ")" ;
inline_functor_application ::= inline_func_name "(" [ term_list ] ")" ;  (* mandatory parens *)
template_functor_application ::= functor_symbol template_args
                                 [ "(" [ term_list ] ")" ] ;
term_list         ::= term_expression { "," term_expression } ;
```

> [!NOTE]
> For symbolic functors, parentheses are **optional for single arguments** and **required for multiple arguments**. Inline functors always require parentheses.
> Template functor applications use the bracket argument list from Ch.16 §16.5
> and are parsed before bracket-functor notation after a template-capable name,
> as specified in Ch.2 §2.5.1.

### 13.2.1 Classification of Functors

| Type | Declaration | Scope | Example |
| --- | --- | --- | --- |
| **Symbolic** | `func Label: pattern -> type ...` | Module-level | `x + y`, `|.x.|` |
| **Inline** | `deffunc name(args) -> type equals ...` | Proof-local | `double(n)` |

### 13.2.2 Symbolic Functor Applications

Symbolic functors support multiple notation styles:

| Notation | Pattern | Example |
| --- | --- | --- |
| **Prefix** | `symbol x` or `symbol(x, y)` | `- x`, `max(a, b)` |
| **Infix** | `x symbol y` | `x + y`, `X \/ Y` |
| **Postfix** | `x symbol` | `n !`, `f "` |
| **Bracket** | `left x, y right` | `[: X, Y :]`, `|. x .|`, `[x, y]` |

> [!NOTE]
> For single arguments, parentheses are optional: `sin x` and `sin(x)` are equivalent.
> For multiple arguments, parentheses are required: `max(a, b)`.

```mizar
:: Prefix (single arg - parens optional)
- x
card X
sin x       :: equivalent to sin(x)
sin(x)

:: Prefix (multi-arg - parens required)
max(a, b)
min(x, y, z)

:: Infix
x + y
X \/ Y

:: Postfix
n !

:: Bracket
[: A, B :]
|. x .|
[a, b]
```

The reserved tokens `[` and `]` are admitted as a built-in bracket-functor
delimiter pair only where the parser is expecting a term primary. After a
template-capable name, the same tokens delimit template arguments
(`Module[R]`, `SomeTheorem[R]`), and the library annotation form starts with
the distinct compound token `@[`. Postfix indexing such as `x[y]` is not
defined.

### 13.2.3 Inline Functor Applications

Inline functors (defined with `deffunc` in proof blocks) are applied with mandatory parentheses.

> [!IMPORTANT]
> Inline functors **require parentheses** in both definition and application, even for zero or single arguments.

**Example**:

```mizar
proof
  deffunc double(n be Nat) -> Nat equals n + n;
  
  A1: double(3) = 6;
  ...;
end;
```

**Semantics**:
- Inline functors are expanded by substitution
- Not visible outside the defining proof block
- Useful for proof-local abbreviations

> [!NOTE]
> For `deffunc` definition syntax, see [Chapter 15 (Statements), §15.2.3](./15.statements.md#1523-inline-function-definition-deffunc).

## 13.3 Structure Expressions

Structure expressions allow construction and manipulation of structured data values based on user-defined `struct` types (see [Chapter 5](./05.structures.md)).

**Syntax (EBNF)**:

```ebnf
structure_expression ::= struct_constructor | selector_access ;
struct_constructor   ::= struct_name [ type_args ] "(" [ field_assignment_list ] ")" ;
field_assignment_list ::= field_assignment { "," field_assignment } ;
field_assignment     ::= field_name ":" term_expression ;
selector_access      ::= term_expression "." field_name [ "(" [ term_list ] ")" ] ;
```

### 13.3.1 Structure Constructor

A structure value is created by supplying field-value pairs.

**Example**:

```mizar
:: Define a Point structure with default values (from Ch.7)
struct Point where
  field x -> Real := 0;
  field y -> Real := 0;
end;

:: Construct with all fields specified
set p = Point(x: 3, y: 4);

:: Omit fields that have defaults
set origin = Point();             :: x = 0, y = 0
set on_x_axis = Point(x: 5);      :: x = 5, y = 0
```

**Default Value Syntax**:

Fields may declare default values using `:=` in the struct definition:

```ebnf
field_decl ::= "field" identifier "->" type_expression [ ":=" term_expression ] ";" ;
```

**Rules**:
- All fields without defaults must be specified in the constructor
- Fields with defaults may be omitted (the default value is used)
- Field names must match the structure's declaration
- Order of fields in the constructor is irrelevant
- Type of each field value must match the declared type

### 13.3.2 Selector Access

Fields of a structure are accessed using dot notation.

**Example**:

```mizar
set p = Point(x: 3, y: 4);
p.x + p.y = 7;  :: Field access

:: Chained access for nested structures
set line = Line(start: Point(x: 0, y: 0), end: Point(x: 1, y: 1));
line.start.x = 0;
line.end.y = 1;
```

If a selected field or property has a function value, the selector may be
followed immediately by an argument list:

```mizar
let M be Magma;
let x, y be Element of M.carrier;

M.binop(x, y)
```

This is parsed as selector access followed by application of the selected
function value, and is semantically equivalent to applying the value denoted by
`M.binop` to `x` and `y`. Type checking must verify that the selected field or
property has a function type whose domain accepts the supplied arguments.

> [!IMPORTANT]
> The traditional Mizar selector syntax `the x of p` is **deprecated**. Use dot notation `p.x` exclusively. This provides a more concise and universally familiar syntax while avoiding syntactic ambiguity with the definite description operator `the`.

### 13.3.3 Structure Update

Structure values are **immutable** in proof contexts. Updates produce new values via functional update syntax.

**Syntax (EBNF):**

```ebnf
functional_update ::= term_expression "with" "(" field_update_list ")" ;
field_update_list ::= field_update { "," field_update } ;
field_update      ::= selector ":=" term_expression ;
```

**Example (proof context — immutable):**

```mizar
set p = Point(x: 3, y: 4);
set q = p with (x := p.x + 1);   :: New point; p is unchanged
:: p.x = 3, q.x = 4
```

**Example (algorithm context — mutable with `var`):**

In algorithm blocks, structures declared with `var` support in-place field mutation.

```mizar
algorithm
  var p := Point(x: 3, y: 4);
  p.x := p.x + 1;               :: In-place update (p is var)
  :: p.x = 4

  const q := Point(x: 1, y: 2);  :: Immutable (no var)
  :: q.x := 5;                   :: ERROR: q is not var
end;
```

> [!NOTE]
> For structure definition syntax, see [Chapter 5 (Structures)](./05.structures.md).
> For algorithm blocks and `var` declarations, see [Chapter 20 (Algorithm and Verification)](./20.algorithm_and_verification.md).

## 13.4 Set Expressions

Set expressions construct sets by enumeration or comprehension.

**Syntax (EBNF)**:

```ebnf
set_expression    ::= set_enumeration | set_comprehension ;
set_enumeration   ::= "{" [ term_list ] "}" ;
set_comprehension ::= "{" term_expression "where" typed_var_list [ ":" formula ] "}" ;
term_list         ::= term_expression { "," term_expression } ;
typed_var_list    ::= typed_var { "," typed_var } ;
typed_var         ::= identifier "is" type_expression ;
```

### 13.4.1 Set Enumeration

Explicit listing of set elements.

**Example**:

```mizar
{}              :: empty set
{ a }           :: singleton
{ a, b, c }     :: finite set
```

### 13.4.2 Set Comprehension (Fraenkel Operator)

Constructs sets by specifying a mapping and filter over elements of a set.

> [!IMPORTANT]
> **Set-hood constraint**: Each generator variable must range over a domain that the verifier already knows is a set. This is satisfied either by an explicit element type such as `Element of S`, or by a mode `T` for which `sethood` has been proved (see [Chapter 7, §7.8.1](./07.modes.md#781-mode-property-sethood)). This ensures the result is a set rather than a proper class.
>
> ```mizar
> :: OK: x ranges over elements of a set
> { x where x is Element of NAT : x < 10 }
>
> :: OK if FiniteOrdinal has a proved sethood property
> { f where f is FiniteOrdinal : f is non empty }
>
> :: NG unless Group has a proved sethood property
> { G where G is Group : G is abelian }  :: REJECTED
> ```

**Example**:

```mizar
:: Even natural numbers
{ n where n is Element of NAT : n mod 2 = 0 }

:: Squares of naturals
{ n * n where n is Element of NAT : n < 10 }

:: All elements (condition omitted)
{ f.x where x is Element of dom f }
```

> [!NOTE]
> When the condition is omitted, all elements satisfying the generator type are included. Simply omit the `: formula` part.

### 13.4.3 Multiple Generators

**Example**:

```mizar
:: Cartesian product elements (no filter needed)
{ [x, y] where x is Element of A, y is Element of B }

:: Pairs satisfying a condition
{ [x, y] where x is Element of NAT, y is Element of NAT : x + y = 10 }
```

### 13.4.4 Nested Comprehensions

**Example**:

```mizar
:: Set of sets
{ { y where y is Element of NAT : y < x }
  where x is Element of NAT : x < 5 }
```

When an inner comprehension refers to an outer generator, the outer bound
variable is captured as a parameter of the inner generated core symbol. Capture
uses resolved binder identity, not the source display name, so later shadowing
cannot change the meaning of the nested comprehension.


## 13.5 The Choice Operator (`the`)

The `the` operator (definite description) selects an arbitrary element of a given type. It corresponds to Hilbert's ε-operator in logic.

**Syntax (EBNF):**

```ebnf
choice_expression ::= "the" type_expression ;
```

**Semantics:**

- `the T` returns an object of type `T`.
- **Existence Requirement**: The type `T` must be **non-empty**. This is guaranteed by the existential registration system (see [Chapter 17](./17.clusters_and_registrations.md)).
- **Determinism**: The choice is stable; `the T` always refers to the same arbitrary element within a given context, though no specific property other than `is T` is assumed.
- **Core Encoding**: Elaboration lowers `the T` to a generated choice term
  `choice_T(params)`, where `params` are the explicit free parameters of `T`
  after alpha-normalization. Repeated occurrences with the same normalized key
  in the same elaboration context denote the same core term. The elaboration
  context is the owning core item after template instantiation: a theorem or
  lemma proposition, theorem proof, definition body, registration proof, or
  algorithm specification formula. If a choice occurs in an exported definition
  body, the generated choice symbol is owned by that definition and reused
  whenever that definition is unfolded. Algorithm statement right-hand sides use
  the Pick lowering of
  [Chapter 20, §20.3](./20.algorithm_and_verification.md#203-non-determinism-pick)
  instead of sharing this stable choice symbol.
- **Obligation**: Each generated choice term creates a non-emptiness obligation
  `∃x. is_T(x, params)` and records the local type fact
  `is_T(choice_T(params), params)`.

**Example:**

```mizar
set n = the Nat;       :: An arbitrary natural number
set x = the Element of X;
```

> [!NOTE]
> This operator is essential for parameterized existential registrations, where one needs to construct a witness using an arbitrary element of a parameter type.

## 13.6 Type Qualification (`qua`)

The `qua` operator explicitly selects a specific type view of a term **without requiring a proof**. It is valid in two complementary situations, both statically verifiable:

1. **Structure inheritance upcast** — the target type is an ancestor mode reachable via the term's `inherit` chain.
2. **Cluster-based widening** — the target type has strictly fewer attributes than the term's type, and all of the target's attributes are derivable from the term's attributes via active cluster registrations.

In both cases the value itself is unchanged; only the type used for overload resolution, field access, and attribute checks is affected.

**Syntax (EBNF)**:

```ebnf
qua_expression ::= operator_expression "qua" type_expression { "qua" type_expression } ;
```

`qua` is a fixed term-level type-qualification operator, not a user-defined symbolic functor. It has the lowest precedence among term-level operators and chains left-to-right. Parentheses may be used to qualify only a subterm.

### 13.6.1 Structure Inheritance Upcast

When a structure inherits from an ancestor mode via an `inherit ... extends ...` chain, `qua` selects the intended ancestor view. This is most useful when multiple inheritance paths exist (diamond inheritance) and the verifier cannot resolve the path automatically.

```mizar
let R be DoubleLoopStr;

:: View R as AddLoopStr to access additive properties
(R qua AddLoopStr) is associative;

:: View R as MulLoopStr to access multiplicative properties
(R qua MulLoopStr) is commutative;
```

```mizar
:: Ring inherits from Magma via both AddGroup and MulMonoid
let R be commutative Ring;

f(R qua AddMagma)   :: selects R's additive Magma structure
f(R qua MulMagma)   :: selects R's multiplicative Magma structure
```

### 13.6.2 Cluster-Based Widening

When a term has more attributes than a parameter or overload requires, `qua` selects the intended less-attributed type. The verifier checks that the target attributes are a subset of those derivable from the term's type by cluster closure — no proof is generated.

```mizar
:: cluster registrations: finite → countable, finite → compact
let S be finite TopSpace;

:: f is overloaded for countable TopSpace and compact TopSpace
f(S qua countable TopSpace)   :: resolves to the countable overload
f(S qua compact TopSpace)     :: resolves to the compact overload
```

### 13.6.3 Chaining `qua`

`qua` steps can be chained left-to-right to traverse multiple levels of an inheritance hierarchy:

```mizar
f(R qua AddGroup qua Magma)
:: first selects R's AddGroup view, then that structure's Magma ancestor
```

Each step must be a valid widening (upcast or cluster). Chaining is evaluated left-to-right.

### 13.6.4 Restrictions

- The target type must be reachable by widening — either via an `inherit` chain (upcast) or via cluster closure (attribute subsetting). No proof is generated.
- Narrowing (casting to a more specific type) requires `reconsider` with a proof. `qua` cannot be used for narrowing.
- In template arguments `[...]`, `qua` uses the same precedence and associativity rules: `PermProduct[R qua AddMagma]` (see [Chapter 18, §18.2.2](./18.templates.md#1822-type-parameters)).
- When a target type contains term arguments, `qua` inside those arguments binds
  to the argument before any outer `qua` chain is continued. Thus
  `x qua Element of S qua Magma` parses as `x qua Element of (S qua Magma)`.
  To qualify the whole outer expression after `Element of S`, write
  `(x qua Element of S) qua Magma`.

> [!NOTE]
> For `reconsider` statement syntax, see [Chapter 8 (Type Inference)](./08.type_inference.md) and [Chapter 15 (Statements), §15.5.1](./15.statements.md#1551-type-reconsideration-reconsider).

## 13.7 Operator Precedence and Associativity

User-defined functors can have explicit precedence and associativity declarations. These are standalone statements, not annotations, because precedence affects parsing (see §10.9).

**Syntax (EBNF)**:

```ebnf
infix_operator_decl   ::= "infix_operator"   "(" string_literal "," infix_assoc "," nat_literal ")" ";" ;
prefix_operator_decl  ::= "prefix_operator"  "(" string_literal "," nat_literal ")" ";" ;
postfix_operator_decl ::= "postfix_operator" "(" string_literal "," nat_literal ")" ";" ;
infix_assoc           ::= "left" | "right" | "none" ;
```

### 13.7.1 Precedence Rules

Each symbolic functor has an associated precedence level (0–255). Higher values bind more tightly.

**Example**:

```mizar
infix_operator("+", left, 80);
infix_operator("*", left, 90);
infix_operator("^", right, 95);
```

With these declarations:
- `a + b * c` parses as `a + (b * c)` (higher precedence binds tighter)
- `a * b + c` parses as `(a * b) + c`
- `a + b = c + d` parses as `(a + b) = (c + d)` (`=` precedence is fixed in the core module)

### 13.7.2 Associativity

| Type | Grouping | Example |
| --- | --- | --- |
| `left_associative` | Left-to-right | `a + b + c` → `(a + b) + c` |
| `right_associative` | Right-to-left | `a ^ b ^ c` → `a ^ (b ^ c)` |
| `non_associative` | Requires parentheses | `a = b = c` → **error** |

> [!NOTE]
> `non_associative` only prohibits chaining the **same** operator (e.g., `a = b = c`). The `precedence_value` is still required because it determines how the operator interacts with **other** operators. For example, `=` at precedence 50 and `+` at precedence 80 ensures `a + b = c` parses as `(a + b) = c`.

### 13.7.3 Disambiguation Guidelines

- Use parentheses when precedence is unclear
- Use qualified names to disambiguate overloaded symbols
- Avoid relying on implicit grouping for complex expressions

## 13.8 Logic Encoding Details

This section describes how term expressions map to First-Order Logic.

### 13.8.1 Variable References

Variables are encoded as FOL variables with type guards:

| Mizar Source | FOL Encoding |
| --- | --- |
| `let x be Nat; x` | `x` (with context `is_Nat(x)`) |

### 13.8.2 The `it` Keyword

In `means` definitions, `it` is replaced by the function application:

```
func F(x) -> T means P(it)
⇒  forall x (P(F(x)))
```

This display is schematic. The full generated definition also includes parameter
type guards, result type obligations, and the existence/uniqueness obligations
for `means` definitions described in [Chapter 10, §10.12.3](./10.functors.md#10123-means-definitions-as-unique-existence)
and [§10.12.6](./10.functors.md#10126-return-type-encoding).

### 13.8.3 Functor Applications

Functor applications are encoded as FOL function applications:

| Mizar Source | FOL Encoding |
| --- | --- |
| `x + y` | `add(x, y)` |
| `- x` | `neg(x)` |
| `n !` | `factorial(n)` |
| `[: A, B :]` | `cartesian_product(A, B)` |

### 13.8.4 Structure Constructor Encoding

Structure constructors are encoded as FOL function applications:

| Mizar Source | FOL Encoding |
| --- | --- |
| `Point(x: 3, y: 4)` | `make_Point(3, 4)` |
| `Rect(w: a, h: b)` | `make_Rect(a, b)` |

### 13.8.5 Selector Access Encoding

Selectors are encoded as FOL functions:

| Mizar Source | FOL Encoding |
| --- | --- |
| `p.x` | `Point_x(p)` |
| `line.start.x` | `Point_x(Line_start(line))` |

### 13.8.6 Set Expression Encoding

**Enumeration**:

| Mizar Source | FOL Encoding |
| --- | --- |
| `{}` | `empty_set` |
| `{ a }` | `singleton(a)` |
| `{ a, b }` | `union(singleton(a), singleton(b))` |

**Comprehension** (encoded using the Replacement axiom schema):

```
{ f(x) where x is Element of S : P(x) }
⇒  Comp_H(S, params)
    with axiom ∀y. (y ∈ Comp_H(S, params)
      ↔ ∃x. x ∈ S ∧ P(x, params) ∧ y = f(x, params))
```

When the condition is omitted:

```
{ f(x) where x is Element of S }
⇒  Comp_H(S, params)
    with axiom ∀y. (y ∈ Comp_H(S, params)
      ↔ ∃x. x ∈ S ∧ y = f(x, params))
```

`Comp_H` is an internal generated set-valued core symbol. `H` is derived from
the alpha-normalized generator list, mapper, and optional predicate; `params`
are the free variables captured from the surrounding context. For a generator
`x is T` whose domain is accepted by a proved `sethood` property rather than an
explicit `Element of S`, the membership axiom uses the type guard `is_T(x)` and
the elaborator records the sethood witness used to justify that the result is a
set.

### 13.8.7 `qua` Operator Encoding

The `qua` operator generates a static type assertion. No new axiom is introduced; the verifier discharges the validity condition at compile time via the inheritance chain or cluster closure.

**Inheritance upcast** — selects the ancestor mode's field projection:

```
(x qua T).field
⇒  Validity (static): is_T reachable from type_of(x) via inherit chain
⇒  Result: T_field(x)
```

**Cluster-based widening** — selects the overload candidate matching the widened type:

```
f(x qua T)
⇒  Validity (static): cl(type_of(x)) ⊇ attributes(T)  (cluster closure subsumes T)
⇒  Overload resolution: f is resolved against parameter type T
```

In both cases the term `x` itself is unchanged in the FOL encoding; `qua` affects only type-dispatch at the meta level.

#### FOL Encoding of Validity Conditions

The two validity conditions above are not merely algorithmic compile-time checks; they can and should be encoded in FOL and discharged by an automated solver. This ensures that the static checks are themselves formally verifiable and free of inconsistency.

**Inherit chain — base and transitive closure:**

```
inherit(S, T)           -- direct inheritance: S inherits from T
inherit*(S, T)          -- transitive closure

Axioms:
  ∀S.     inherit*(S, S)                                        -- (Refl)
  ∀S T U. inherit*(S, T) ∧ inherit*(T, U) → inherit*(S, U)     -- (Trans)
  ∀S T.   inherit(S, T) → inherit*(S, T)                       -- (Base)
```

**Cycle freedom of the inherit chain (consistency check):**

```
  ∀S T.   inherit*(S, T) ∧ inherit*(T, S) → S = T              -- (AntiSym)
```

This must hold over the entire signature; a solver check at definition time catches circular inheritance before it can cause unsoundness.

**Validity condition for inheritance upcast (FOL discharge):**

```
(x qua T) via inheritance is valid
⇔  ⊢_FOL  inherit*(type_radix(x), T)
```

Where `type_radix(x)` is the radix mode of `x`'s type. The verifier emits this goal and an SMT/FOL solver discharges it from the `inherit` axioms declared in the signature.

**Cluster closure — base and propagation:**

```
has_attr(A, S)          -- attribute A is directly registered for mode S
cl(A, S)                -- A is in the cluster closure of S

Axioms:
  ∀A S.   has_attr(A, S) → cl(A, S)                            -- (ClBase)
  ∀A B S. cluster_implies(A, B) ∧ cl(A, S) → cl(B, S)          -- (ClProp)
```

Where `cluster_implies(A, B)` represents a registered cluster rule `A → B`.

**Consistency of cluster implications (no contradictory attributes):**

```
  ∀A S.   ¬(cl(A, S) ∧ cl(non_A, S))                           -- (ClConsist)
```

Here `non_A` denotes the negation of attribute `A`. This condition prevents a cluster chain from simultaneously deriving an attribute and its negation.

**Validity condition for cluster-based widening (FOL discharge):**

```
(x qua T) via cluster widening is valid
⇔  ⊢_FOL  ∀A ∈ attributes(T). cl(A, type_radix(x))
```

The verifier emits one proof obligation per attribute of the target type `T`, each discharged by the cluster closure axioms.

**Chained `qua` — FOL discharge per step:**

Chained expressions such as `x qua T1 qua T2` are left-associative:

```
(x qua T1 qua T2)
≡  ((x qua T1) qua T2)

Proof obligations emitted:
  Step 1:  validity of x qua T1, using either inheritance upcast
           or cluster-based widening
  Step 2:  validity of the T1-view qua T2, again using the
           applicable inheritance or cluster-widening rule
```

Each step is discharged independently; the overall chain is valid iff every step discharges. A chain may mix inheritance upcasts and cluster-based widening steps, but each adjacent edge is checked in order and source maps retain the evidence for the step that justified each view.

**Summary of solver proof obligations emitted per `qua` occurrence:**

| Case | Proof obligation |
| --- | --- |
| Inheritance upcast `x qua T` | `inherit*(type_radix(x), T)` |
| Cluster widening `x qua T` | `cl(A, type_radix(x))` for each `A ∈ attributes(T)` |
| Chained `x qua T1 qua T2` | obligations for each adjacent step independently; each step may be an inheritance upcast or cluster widening |
| Cycle check (once per `inherit` declaration) | `¬inherit+(T, T)` |
| Cluster consistency (once per cluster rule) | `¬(cl(A, S) ∧ cl(non_A, S))` |

### 13.8.8 Parenthesization

Parentheses do not affect FOL encoding; they only control parsing:

| Mizar Source | FOL Encoding |
| --- | --- |
| `(a + b) * c` | `mul(add(a, b), c)` |
| `a + (b * c)` | `add(a, mul(b, c))` |

## 13.9 Complete Syntax (EBNF)

Summary of all term expression syntax.

```ebnf
(* Term Expressions *)
term_expression      ::= operator_expression { "qua" type_expression } ;

operator_expression  ::= postfix_expression | functor_application ;
postfix_expression   ::= term_primary { term_postfix } ;

(* Primary Expressions *)
term_primary         ::= variable_identifier
                       | "it"
                       | numeral
                       | "(" term_expression ")"
                       | struct_constructor
                       | set_expression
                       | choice_expression
                       | inline_functor_application
                       | template_functor_application
                       | bracket_functor_application ;
variable_identifier ::= identifier ;
numeral            ::= digit+ ;

term_postfix        ::= "." field_name [ "(" [ term_list ] ")" ]
                      | "with" "(" field_update_list ")" ;

(* Functor Applications *)
functor_application ::= [ functor_loci ] functor_symbol [ functor_loci ] ;
functor_loci       ::= term_expression | "(" term_list ")" ;
inline_functor_application ::= inline_func_name "(" [ term_list ] ")" ;  (* mandatory parens *)
template_functor_application ::= functor_symbol template_args
                                 [ "(" [ term_list ] ")" ] ;
bracket_functor_application ::= user_symbol term_list user_symbol
                              | "[" term_list "]" ;
(* template_args is defined in Chapter 16 §16.5. *)
(* For deffunc definition syntax, see Chapter 15 (Statements) *)

(* Structure Expressions *)
structure_expression ::= struct_constructor | selector_access ;
struct_constructor   ::= struct_name [ type_args ] "(" [ field_assignment_list ] ")" ;
field_assignment_list ::= field_assignment { "," field_assignment } ;
field_assignment     ::= field_name ":" term_expression ;
selector_access      ::= term_expression "." field_name [ "(" [ term_list ] ")" ] ;
(* For field_decl with optional defaults, see Chapter 5 (Structures) *)

(* Set Expressions *)
set_expression       ::= set_enumeration | set_comprehension ;
set_enumeration      ::= "{" [ term_list ] "}" ;
set_comprehension    ::= "{" term_expression "where" typed_var_list [ ":" formula ] "}" ;
term_list            ::= term_expression { "," term_expression } ;
typed_var_list       ::= typed_var { "," typed_var } ;
typed_var            ::= identifier "is" type_expression ;  (* domain must be Element of <set> or have sethood *)

(* Choice Operator *)
choice_expression    ::= "the" type_expression ;

(* Type Qualification *)
qua_expression       ::= operator_expression "qua" type_expression { "qua" type_expression } ;

(* Operator Precedence Declarations *)
infix_operator_decl   ::= "infix_operator"   "(" string_literal "," infix_assoc "," nat_literal ")" ";" ;
prefix_operator_decl  ::= "prefix_operator"  "(" string_literal "," nat_literal ")" ";" ;
postfix_operator_decl ::= "postfix_operator" "(" string_literal "," nat_literal ")" ";" ;
infix_assoc           ::= "left" | "right" | "none" ;
```
