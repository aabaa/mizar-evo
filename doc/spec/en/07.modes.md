## 7.1 Definition and Purpose of Modes

Modes are user-defined type constructors built on top of radix-types such as `set` or `struct`. They allow:

* **Abstraction** of mathematical concepts as types
* **Refinement** of types using attributes
* **Composition** of complex types
* **Dependency** on values (i.e., dependent types)

Modes are a central mechanism for type classification in Mizar, offering an abstract and modular alternative to structures.

## 7.2 Syntax for Declaring and Using Modes

**Syntax (EBNF)**:

```ebnf
mode_def       ::= "mode" label ":" mode_def_name [ type_params ] "is" attribute_chain radix_type ";" ;
label          ::= identifier ;
mode_def_name  ::= constructor_name ;
type_params    ::= ( "of" | "over" ) type_parameter_list
                 | "[" type_parameter_list "]" ;
type_parameter_list ::= identifier { "," identifier } ;
attribute_chain ::= { [ "non" ] attribute_ref } ;
attribute_ref  ::= [ param_prefix ] [ struct_ref_name "." ] attribute_name
                   [ "(" argument_list ")" ] ;
param_prefix   ::= parameter "-" | "(" parameter_list ")" "-" ;
parameter_list ::= parameter { "," parameter } ;
parameter      ::= identifier | numeral ;
radix_type     ::= builtin_type
                 | mode_ref_name [ type_args ]
                 | struct_ref_name [ type_args ] ;
type_args      ::= ( "of" | "over" ) argument_list
                 | "[" type_arg_list "]" ;
type_arg_list  ::= type_arg { "," type_arg } ;
type_arg       ::= type_expression | qua_arg ;
qua_arg        ::= identifier { "qua" radix_type } ;
argument_list  ::= term_expression { "," term_expression } ;
mode_ref_name  ::= qualified_constructor_name ;
struct_ref_name ::= qualified_constructor_name ;
attribute_name ::= qualified_constructor_name ;
```

* `label`: Mandatory identifier for referencing this definition
* `mode_def_name`: Name of the mode being defined. It may be an ordinary
  identifier or a readable hyphenated constructor name, but not arbitrary
  operator-like symbolic notation.
* `type_params`: Optional parameters for dependent modes; bracket form is canonical, while `of` and `over` remain accepted shorthands
* `attribute_chain`: Zero or more attributes (possibly parameterized) refining the base type

### Mode Definition

```mizar
definition
  mode Label: <mode_name> is <attribute_chain> <type_expression>;
end;
```

Here, `<attribute_chain>` is a (possibly empty) sequence of predicates (possibly parameterized) that refine the base type. Attributes must be defined using the `attr` keyword (see Chapter 6).

Examples:

```mizar
definition
  mode IntegerDef: Integer is integer Number;
  mode PrimeDef: Prime is prime Integer;
  mode GroupDef: Group is non empty associative invertible unital Magma;
end;
```

### Mode Application

Modes are used in variable declarations, expressions, and type restrictions:

```mizar
let G be Group;
let n be Prime;
```

## 7.3 Type Inference with Modes

Mizar automatically infers type relations for variables declared with modes:

* Inference follows attribute clusters and structural inheritance
* Inference supports dependent modes and parameterized types

Examples:

```mizar
let n be Prime;  :: implies n is Integer and Number
let S be Subset of X;  :: implies S is set and S ⊆ X
```

## 7.4 Mode Implementation

Internally, each mode is interpreted as:

1. A predicate for membership (possibly with parameters for dependent modes)
2. A cluster registration linking attributes and base types

This supports a uniform and efficient type checking mechanism.

```mizar
definition
  mode Label: M is <attribute_chain> <type_expression>;
end;
```

is equivalent to defining a predicate and a cluster registration.

### 7.4.1 Property Implementation

A `property` declared in a struct body (see Chapter 5) is a **virtual slot** — it declares the name and type of a derived value, but provides no implementation. The implementation is supplied via a `property ... means` or `property ... equals` definition inside a `definition` block, parameterized over a mode:

Property implementations are the only source of property values. Default
structure constructors accept fields only and do not take property arguments
or emit property projection axioms (see
[Chapter 5, §5.5.1](./05.structures.md#551-default-constructor) and
[§5.8.4](./05.structures.md#584-constructor-axioms)).

Property implementations for the same struct property form one implementation
family. Several implementations may be defined for different mode domains, but
if two domains overlap, the implementations must be related by a `coherence`
correctness condition (§7.8.2). Domains overlap when their normalized mode
domains have a common instance that the verifier cannot statically rule out;
in particular, an implementation for a mode that refines or inherits from
another implementation's mode overlaps the broader one.

```mizar
:: means form: define by logical condition
definition
  let M be UnitalMagma;
  property M.unit means
    for x be Element of M.carrier holds
    M.binop(it, x) = x & M.binop(x, it) = x;
  existence proof ... end;
  uniqueness proof ... end;
end;

:: equals form: define by direct term
definition
  let S be FinSequenceStr;
  property S.length equals len S.coeffs;
end;
```

* `it` refers to the property value being defined (in `means` form).
* `means` requires `existence` and `uniqueness` proof obligations — see §7.8.2 for details.
* `equals` requires no `existence` or `uniqueness` proof obligation; those follow immediately from the term. A `coherence` obligation is still required when this implementation overlaps another implementation of the same property.

## 7.5 Relationship between Modes and Structures

Modes and structures serve complementary roles. Use **structures** when you need to define composite types with internal fields (e.g., algebraic structures like rings or groups). Use **modes** when you want to classify or restrict existing types by logical properties without adding internal structure (e.g., prime numbers, continuous functions).

| Aspect     | Modes                                | Structures                         |
| ---------- | ------------------------------------ | ---------------------------------- |
| Definition | Abstract (via attributes)            | Concrete (via fields/properties)   |
| Usage      | Conceptual classification            | Data modeling and component access |
| Conversion | `qua` operator for structural access | Field/property access is explicit  |

Example:

```mizar
definition
  struct MetricStruct where
    field carrier -> set;
    field distance -> Function of [:carrier, carrier:], REAL;
  end;

definition
  mode MetricSpaceDef: MetricSpace is reflective discerning symmetric triangle MetricStruct;
end;
```

## 7.6 Unified Mode Syntax

Mizar enforces the `is` syntax for mode definitions, replacing the legacy `means` form. This ensures:

* Simplified parsing and inference
* Uniform representation across the language
* Clear separation between types (modes) and predicates (logical properties)

Instead of:

```mizar
definition
  mode Prime means <definition>;
end;
```

Use:

```mizar
definition
  let n be Nat;
  attr PrimeAttr: n is prime means <definition>;
  mode PrimeDef: Prime is prime Integer;
end;
```

## 7.7 Dependent Modes

Modes can be parameterized using `of` or `over`:

```mizar
definition
  mode SubsetDef: Subset of X is X-subset_like set;
  mode MatrixDef: Matrix over R is R-matrix_like set;
end;

let A be Matrix over REAL;
```

These forms are syntactic sugar for template-style definitions. For example, `Matrix over REAL` is equivalent to `Matrix[REAL]` in other notations.

Multiple parameters are supported:

```mizar
definition
  mode FinDimVSDef: FinDimVectorSpace over F, n is n-dimensional vector_space_like structure over F;
end;

let V be FinDimVectorSpace over COMPLEX, 5;
```

> [!WARNING]
> All variables used in the mode body must be declared in `type_params`. The following is an error because `n` is not declared:

```mizar
definition
  :: ERROR: 'n' is not declared in type_params
  mode FinDimVSDef: FinDimVectorSpace over F
  is n-dimensional vector_space_like structure over F;
end;
```

## 7.8 Correctness Conditions

Mode definitions require inhabitation evidence for the full normalized right-hand
side type expression. For an unparameterized mode this means:

```text
∃x. is_RHS(x)
```

For a parameterized mode `T(params) is RHS(params)`, the obligation is
guarded over every valid parameter tuple:

```text
∀params. ParamGuard(params) → ∃x. is_RHS(x, params)
```

`ParamGuard` is the conjunction of the declared parameter type and attribute
guards after normalization. If the RHS has attributes, the evidence is a
visible active existential registration for the full normalized attributed type
(§17.3.4). If the RHS is unattributed, the verifier uses the base-shape
inhabitation table of §17.3.4 (`object`, `set`, accepted mode applications,
structure radixes, and schema type parameters inside template bodies).

Typical attributed modes use cluster registrations:

```mizar
registration
  cluster prime Integer;
  existence
  proof
    take 2; thus 2 is prime;
  end;
end;

definition
  mode PrimeDef2: Prime is prime Integer;
end;
```

This check is **mandatory**. The system raises a **hard error** at the point of
mode declaration if no inhabitation evidence is available for the RHS type
expression. This guarantees that all modes in use are non-empty, preserving the
soundness of the logic. Registration activation is item-ordered (§17.1), so a
later existential registration cannot repair an earlier mode declaration.

### 7.8.1 Mode Property: `sethood`

The `sethood` property asserts that all instances of a mode form a **set** (not a proper class). This is essential for set-theoretic foundations, ensuring that comprehension over the mode's instances produces a well-defined set.

**Syntax (EBNF):**

```ebnf
mode_property ::= "sethood" justification ";" ;
```

**FOL Obligation:**

| Property | FOL Obligation |
|---|---|
| `sethood` | `∃S. ∀x. (is_T(x) → x ∈ S)` |
| `sethood` for `T(params)` | `∀params. ParamGuard(params) → ∃S. ∀x. (is_T(x, params) → x ∈ S)` |

The witness set `S` may depend on the normalized parameter tuple. A use site
for `T(args)` receives sethood evidence only for that same resolved mode and
normalized argument vector; sethood for `T(a)` does not license comprehension
over `T(b)` unless the arguments normalize to the same tuple.

For a public/exported mode, an accepted `sethood` property is part of the
module interface as a semantic property fact and contributes to downstream
availability and semantic fingerprints. The concrete witness term for `S` is
not exported as a usable value. Private mode sethood remains private.

**Example:**

```mizar
definition
  mode FiniteOrdinalDef: FiniteOrdinal is ordinal finite set;
  sethood
  proof
    :: Must prove: ex S being set st
    ::   for x being FiniteOrdinal holds x in S
    take omega;
    let x be FiniteOrdinal;
    thus x in omega by mml.ordinal.Finite_ordinal_in_omega;
  end;
end;
```

**Verifier effect**: When `sethood` is proved for a mode T, the verifier permits Fraenkel expressions (set comprehensions) over elements of T. Without `sethood`, comprehension `{ f(x) where x is T : P(x) }` is rejected because the result might be a proper class.

> [!NOTE]
> For set comprehension restrictions, see [Chapter 13 (Term Expressions), §13.4.2](./13.term_expression.md#1342-set-comprehension-fraenkel-operator).

### 7.8.2 Property Implementation Correctness

A property implementation uses either `means` or `equals` (see §7.4.1). The two forms differ in their proof obligations:

**Syntax (EBNF)**:

```ebnf
property_impl    ::= "definition"
                       "let" identifier "be" mode_application ";"
                       ( property_means_impl | property_equals_impl )
                     "end" ";" ;

property_means_impl  ::= "property" identifier "." identifier "means" formula_definiens ";"
                         existence_block
                         uniqueness_block
                         [ coherence_block ] ;

property_equals_impl ::= "property" identifier "." identifier "equals" term_definiens ";"
                         [ coherence_block ] ;
formula_definiens ::= formula
                    | formula_case { "," formula_case } [ "otherwise" formula ] ;
formula_case      ::= formula "if" formula ;
term_definiens    ::= term_expression
                    | term_case { "," term_case } [ "otherwise" term_expression ] ;
term_case         ::= term_expression "if" formula ;

existence_block  ::= "existence" justification ";" ;
uniqueness_block ::= "uniqueness" justification ";" ;
coherence_block  ::= "coherence" justification ";" ;
```

| Form | Obligation | Reason |
|------|------------|--------|
| `means` | `existence` + `uniqueness` | The defining condition may not obviously yield a unique value |
| `equals` | no `existence`/`uniqueness` | The term directly supplies the value; existence and uniqueness are immediate |
| overlapping implementations | `coherence` | Same-property implementations that can apply to the same object must define the same value |

The `let M be <mode>` clause is the key to eliminating ad-hoc `assume` statements: the mode's attribute chain already guarantees the conditions needed for existence.

**Overlap coherence.** Let `I₁` and `I₂` be two visible implementations of
the same struct property `p`, with normalized mode domains `D₁` and `D₂`.
If their overlap `D₁ ∩ D₂` is not statically empty, the later implementation
must include a `coherence` block. One `coherence` block discharges all earlier
visible overlapping implementations of the same property.

Each implementation contributes a definitional relation `Defᵢ(x, v)` between
an object `x` in its mode domain and the property value `v`:

| Pairing | Relation used in the obligation |
|---|---|
| `means` / `means` | The two formula definientia, with `it` replaced by `v₁` and `v₂` respectively |
| `means` / `equals` | The `means` formula for one side; `v = term(x)` for the `equals` side |
| `equals` / `equals` | `v₁ = term₁(x)` and `v₂ = term₂(x)` |

The generated coherence obligation is:

```text
∀x, v₁, v₂.
  (is_D₁(x) ∧ is_D₂(x) ∧ Def₁(x, v₁) ∧ Def₂(x, v₂)) → v₁ = v₂
```

An overlapping implementation whose required `coherence` proof is missing,
pending, or rejected is not an accepted property implementation. The verifier
reports a property-coherence diagnostic rather than selecting one
implementation by source order.

**Example** — `UnitalMagma.unit`:

```mizar
:: struct UnitalMagmaStr declares: property unit -> Element of carrier  (virtual)
:: attr unital: MagmaStr is unital means ex e st (identity condition)
:: mode UnitalMagma is unital UnitalMagmaStr

definition
  let M be UnitalMagma;        :: M is unital is guaranteed by the mode
  property M.unit means
    for x be Element of M.carrier holds
    M.binop(it, x) = x & M.binop(x, it) = x;
  existence
  proof
    consider e being Element of M.carrier such that
    A1: for x holds M.binop(e, x) = x & M.binop(x, e) = x by UnitalDef;
    take e;
    thus for x holds M.binop(e, x) = x & M.binop(x, e) = x by A1;
  end;
  uniqueness
  proof
    let e1, e2 be Element of M.carrier;
    assume A1: for x holds M.binop(e1, x) = x & M.binop(x, e1) = x;
    assume A2: for x holds M.binop(e2, x) = x & M.binop(x, e2) = x;
    thus e1 = M.binop(e1, e2) by A2
           .= e2 by A1;
  end;
end;
```

## 7.9 Logic Encoding Details

This section describes how modes map to First-Order Logic.

### 7.9.1 Modes as Predicates

Each mode `M` is encoded as a membership predicate. A variable declaration `let x be M` generates:

| Mizar Source | FOL Encoding |
| --- | --- |
| `let x be Integer;` | `is_Integer(x)` |
| `let G be Group;` | `is_Group(G)` |
| `let S be Subset of X;` | `is_Subset(S, X)` |

### 7.9.2 Attribute Chain Expansion

Modes with attribute chains expand to conjunctions:

| Mizar Source | FOL Encoding |
| --- | --- |
| `mode Prime is prime Integer` | `∀x. is_Prime(x) ↔ (is_Integer(x) ∧ is_prime(x))` |
| `mode Group is non empty associative invertible unital Magma` | `∀x. is_Group(x) ↔ (is_Magma(x) ∧ ¬is_empty(x) ∧ is_associative(x) ∧ is_invertible(x) ∧ is_unital(x))` |

### 7.9.3 Dependent Mode Encoding

Dependent modes with parameters are encoded as multi-argument predicates:

| Mizar Source | FOL Encoding |
| --- | --- |
| `let S be Subset of X;` | `is_Subset(S, X)` |
| `let V be FinDimVectorSpace over F, n;` | `is_FinDimVectorSpace(V, F, n)` |

### 7.9.4 Subtype Relationship

Mode inheritance generates implication axioms:

```
mode Prime is prime Integer
⇒  forall x (is_Prime(x) implies is_Integer(x))
```

## 7.10 Complete Syntax (EBNF)

Summary of mode-related syntax.

```ebnf
(* Mode Definition *)
mode_def       ::= "mode" label ":" mode_def_name [ type_params ] "is" attribute_chain radix_type ";"
                   [ mode_property ] ;
label          ::= identifier ;
mode_def_name  ::= constructor_name ;
type_params    ::= ( "of" | "over" ) type_parameter_list
                 | "[" type_parameter_list "]" ;
type_parameter_list ::= identifier { "," identifier } ;
attribute_chain ::= { [ "non" ] attribute_ref } ;
attribute_ref  ::= [ param_prefix ] [ struct_ref_name "." ] attribute_name
                   [ "(" argument_list ")" ] ;
param_prefix   ::= parameter "-" | "(" parameter_list ")" "-" ;
parameter_list ::= parameter { "," parameter } ;
parameter      ::= identifier | numeral ;
radix_type     ::= builtin_type
                 | mode_ref_name [ type_args ]
                 | struct_ref_name [ type_args ] ;

(* Mode Property *)
mode_property  ::= "sethood" justification ";" ;

(* Property Implementation *)
property_impl        ::= "definition"
                           "let" identifier "be" mode_application ";"
                           ( property_means_impl | property_equals_impl )
                         "end" ";" ;

property_means_impl  ::= "property" identifier "." identifier "means" formula_definiens ";"
                         existence_block
                         uniqueness_block
                         [ coherence_block ] ;

property_equals_impl ::= "property" identifier "." identifier "equals" term_definiens ";"
                         [ coherence_block ] ;
formula_definiens ::= formula
                    | formula_case { "," formula_case } [ "otherwise" formula ] ;
formula_case      ::= formula "if" formula ;
term_definiens    ::= term_expression
                    | term_case { "," term_case } [ "otherwise" term_expression ] ;
term_case         ::= term_expression "if" formula ;
coherence_block   ::= "coherence" justification ";" ;

existence_block  ::= "existence" justification ";" ;
uniqueness_block ::= "uniqueness" justification ";" ;

(* Mode Application *)
mode_application ::= mode_name [ type_args ] ;
mode_name        ::= mode_ref_name ;
mode_ref_name    ::= qualified_constructor_name ;
struct_ref_name  ::= qualified_constructor_name ;
attribute_name   ::= qualified_constructor_name ;
type_args        ::= ( "of" | "over" ) argument_list
                   | "[" type_arg_list "]" ;
type_arg_list    ::= type_arg { "," type_arg } ;
type_arg         ::= type_expression | qua_arg ;
qua_arg          ::= identifier { "qua" radix_type } ;
argument_list    ::= term_expression { "," term_expression } ;
```
