This chapter specifies the package management system, the build pipeline, and the IDE integration layer of Mizar Evolution. A **package** is the unit of distribution and versioning: a self-contained collection of Mizar source files with a single manifest. The build system transforms source files into verified artifacts that can be consumed by downstream packages and by IDE tools.

## 23.1 Manifest File Format (`mizar.pkg`)

Every package must contain a file named `mizar.pkg` at its root directory. The manifest uses TOML syntax. All fields are described below; fields marked **required** must be present, all others are optional and take the indicated default when absent.

```toml
[package]
name    = "algebra"           # required; lowercase, hyphens allowed
version = "2.3.1"             # required; semantic version (§23.2)
authors = ["Alice <alice@example.com>"]
license = "MIT"
description = "Algebraic structures"
edition = "2025"              # language edition; defaults to current stable

[dependencies]
mml-core = "^1.0"
topology  = { version = "^0.9", features = ["metric"] }

[dev-dependencies]           # used only during verification of this package
test-utils = "^0.2"

[verifier]
max_axioms    = 128          # global axiom budget per obligation (§21.4.2)
atp_timeout   = 30           # seconds per ATP backend per obligation (§22.4.5)
default_solver = "auto"      # "auto" | "vampire" | "e" | "cvc5" | "z3"
require_kernel_certificates = true   # reject externally attested proofs in release builds

[build]
incremental = true           # reuse cached artifacts for unchanged files
cache_dir   = ".mizar-cache" # relative to package root
artifact_dir = "build"       # where *.mizir.json and logs are written
```

**`[package]`** fields:

| Field | Required | Default | Description |
|---|---|---|---|
| `name` | yes | — | Package identifier used in dependency declarations and import paths; must match `[a-z][a-z0-9-]*` |
| `version` | yes | — | Package version in `MAJOR.MINOR.PATCH` semantic versioning format (§23.2) |
| `authors` | no | `[]` | List of author strings, conventionally `"Name <email@example.com>"` |
| `license` | no | `""` | SPDX license identifier (e.g., `"MIT"`, `"Apache-2.0"`); empty string means no license declared |
| `description` | no | `""` | One-line human-readable description of the package |
| `edition` | no | — | Language edition; selects the parser and type rules. Editions may introduce incompatible syntax changes. Use the current stable edition when omitted |

**`[dependencies]`** entries specify packages required at verification time. Each entry is either a bare version string (`"^1.0"`) or a table with `version` and optional `features`. Features select optional subsets of a package's exported definitions.

**`[verifier]`** fields override the global defaults for ATP interaction:

| Field | Default | Description |
|---|---|---|
| `max_axioms` | `128` | Global axiom budget per obligation (§21.4.2) |
| `atp_timeout` | `30` | Seconds per ATP backend per obligation (§22.4.5) |
| `default_solver` | `"auto"` | ATP backend: `"auto"` \| `"vampire"` \| `"e"` \| `"cvc5"` \| `"z3"`; `"auto"` selects the fastest available backend at build time |
| `require_kernel_certificates` | `true` | When `true`, proofs marked `externally_attested` (§21.7.5, §22.6.2) are treated as build errors rather than warnings; recommended for packages intended for publication |

**`[build]`** fields control the incremental build system:

| Field | Default | Description |
|---|---|---|
| `incremental` | `true` | When `true`, each source file and its transitive dependency set are hashed; unchanged files reuse cached artifacts without re-verification |
| `cache_dir` | `".mizar-cache"` | Path to the build cache directory, relative to the package root |
| `artifact_dir` | `"build"` | Directory where `*.mizir.json` and logs are written, relative to the package root |

## 23.2 Versioning and Dependency Resolution

Package versions follow **semantic versioning** (SemVer 2.0.0). The bump a release requires is determined by the most severe change to the package's **public API surface** — the exported signature Σ and public theorem items of §12.6.1, never its proof bodies or private items:

- `MAJOR` — incremented for incompatible changes: removed or renamed definitions, changed type signatures, removed exports, strengthened preconditions, or any change that can invalidate a downstream package that previously verified against the old version.
- `MINOR` — incremented for backward-compatible additions: new definitions, new theorems, new registrations that do not invalidate existing proofs.
- `PATCH` — incremented for backward-compatible fixes: corrected proofs, corrected `ensures` clauses that were previously weaker, documentation updates.

#### Compatibility classification

The required bump is the **maximum** over every change in the diff between the released version and the candidate version. Each public API surface element carries two independent keys, and the classification depends on keeping them separate:

- an **identity** — its fully-qualified name / normalized origin id — used only to *pair* an element across the two versions;
- a **semantic fingerprint** (architecture §18, [18.dependency_fingerprint.md](../../design/architecture/en/18.dependency_fingerprint.md)) — the canonical semantic content compared as the element's *value* once it has been paired by identity.

This identity-versus-value split is what makes the table columns well-defined. An identity present on one side and absent on the other is a **removed/renamed** change (a rename is the disappearance of one identity plus the appearance of another); an identity present on both sides whose fingerprint value differs is a **signature or statement change**. Both are MAJOR for most elements, but they are classified by the kind of delta, not collapsed into remove-plus-add. Because the value is a canonical semantic fingerprint rather than source text, an overload-resolution shift (see the note below) is detected even when no identity was removed.

| Public API surface element | Removed / renamed | Signature or statement change | Backward-compatible addition | Body / proof only |
|---|---|---|---|---|
| definition / functor / mode / attribute | MAJOR | MAJOR | MINOR | — |
| exported algorithm (signature + contract) | MAJOR | MAJOR | MINOR (new algorithm) | PATCH (body only) |
| theorem statement | MAJOR | MAJOR | MINOR (new theorem) | PATCH |
| notation with parse effect | MAJOR | MAJOR | MINOR | — |
| cluster / registration | MAJOR | — | MINOR, but MAJOR when it shifts overload resolution (see note) | — |
| export visibility | MAJOR (export removed) | — | MINOR (new export) | — |

A `—` cell means the change kind does not apply to that element. Any change to an exported theorem **statement** (including weakening it to a strictly more provable form) is MAJOR, because a downstream proof may cite the theorem in its previously stated form; only changes confined to a theorem's **proof body** — or to an exported algorithm's implementation body, which importers cannot observe under `by computation` (§20.9.1) — are PATCH. Changes confined to proof or implementation bodies, comments, or non-syntactic source formatting never force a bump above PATCH — this mirrors the body-vs-signature fingerprint separation of architecture §18. As noted under §23.10, `mizar semver-check` gates only the MAJOR and MINOR verdicts; PATCH-versus-no-change is not derivable from the API surface and is left to the publish/registry step.

Direction-sensitive contract and edition changes are classified separately, because their effect depends on the direction of the change rather than on add/remove. The `requires` / `ensures` rules apply to the contracts of exported algorithms and functions (§20):

- Strengthening a `requires` precondition is **MAJOR** (an existing caller may no longer satisfy it); weakening a `requires` is MINOR.
- Weakening an `ensures` postcondition is **MAJOR** (a downstream proof may depend on the stronger guarantee); strengthening a previously weaker `ensures` is PATCH (as above).
- Raising the package `edition` (§23.1) is classified **MAJOR by default**, because an edition may introduce incompatible syntax and type rules. It is reported with the dedicated code **W0305** (§22.6.6), distinct from W0301, so the edition-driven verdict is reviewable: a maintainer who has confirmed the edition change is compatible may intentionally accept the MAJOR classification or override it, rather than the tool silently treating the increase as compatible.

> [!NOTE]
> **Cluster and overload registrations.** Adding a new conditional cluster, functional cluster, or overloaded definition is normally MINOR. However, when the new registration changes the outcome of **overload resolution** (§23.8.3 `PickMostSpecific`) for an expression that already type-checked in a downstream package — for example, by introducing a strictly more specific registration that supersedes the previously selected one, or by introducing an incomparable minimal element that causes a previously unambiguous resolution to become ambiguous — the change is observable to dependents and must be treated as **MAJOR**. Registrations that only extend the `UnionAll` firing set of attributes (without affecting overload resolution or creating new conflicts) remain MINOR.

The `mizar semver-check` tool (§23.10) computes the required bump from this classification and, optionally as a CI gate, reports when a candidate version's declared increment is smaller than required. The verdict is exact for structural changes but, for overload-resolution shifts, is only a best-effort local heuristic on the default single-package input (§23.10, *Gate semantics and completeness*).

**Version constraints** in dependency entries use the following syntax:

| Constraint | Meaning |
|---|---|
| `"1.2.3"` | Exactly version 1.2.3 |
| `"^1.2.3"` | Compatible with 1.2.3: `>= 1.2.3, < 2.0.0` |
| `"~1.2.3"` | Approximately 1.2.3: `>= 1.2.3, < 1.3.0` |
| `">= 1.0, < 2.0"` | Explicit range |

**Dependency resolution** uses the **PubGrub algorithm** — a conflict-driven backtracking resolver purpose-built for version constraint solving (the same algorithm used by Cargo's resolver and Dart's pub tool). The algorithm maintains a set of version terms (lower/upper bounds per package) and extends the current partial assignment one package at a time, always selecting the newest version compatible with the active terms. When a conflict is detected, it derives an **incompatibility** — a minimal set of terms that cannot all hold simultaneously — and backtracks to the earliest decision point that contributed to the conflict. This conflict-driven learning ensures that "newest compatible" is selected reliably, without relying on solver-internal heuristics. When no satisfying assignment exists, the accumulated incompatibilities form a human-readable conflict explanation that the build system surfaces as a dependency conflict error.

**Diamond dependencies** (two packages depending on different versions of a shared package) are resolved by unification when the version ranges overlap, and rejected with an error when they do not. Mizar packages do not support multiple simultaneous versions of the same package within one workspace.

**Reproducibility.** The resolved version assignment is written to a **lock file** (`mizar.lock`) at the workspace root. Subsequent builds use the lock file's exact versions rather than re-running the resolver. The lock file is committed to version control for applications and excluded for libraries (so downstream packages can resolve dependencies freely).

## 23.3 Workspace Layout

A standard single-package workspace has the following layout:

```
mypackage/
├── mizar.pkg          ← package manifest
├── mizar.lock         ← resolved dependency lock file
├── src/
│   ├── lib.miz        ← top-level source file (exports)
│   └── ...            ← additional source files
├── build/
│   ├── lib.mizir.json ← resolved metadata artifact (§23.5.1)
│   ├── lib.atp-log    ← ATP interaction log (§23.5.3)
│   ├── missing_facts.json ← premise candidates (§23.5.2)
│   └── proof-witnesses/
│       └── lib/
│           └── vc-42.json ← per-obligation proof witness (§23.5.1)
└── .mizar-cache/
    ├── cluster-db/    ← global cluster and registration indexes (§23.7.7)
    └── ...
```

A **multi-package workspace** groups several related packages under a common root:

```
workspace/
├── mizar.workspace    ← workspace manifest (lists member packages)
├── mizar.lock
├── algebra/
│   ├── mizar.pkg
│   └── src/
├── topology/
│   ├── mizar.pkg
│   └── src/
└── .mizar-cache/      ← shared cache across all members
```

The workspace manifest (`mizar.workspace`) lists member packages:

```toml
[workspace]
members = ["algebra", "topology"]
```

Member packages share the workspace lock file and the build cache. Cross-member imports are resolved through the workspace dependency graph rather than the public package registry.

**Import paths.** Within a package, source files are identified by their path relative to `src/`. A file `src/groups/basic.miz` is imported as:

```mizar
import groups.basic;
```

Cross-package imports use the package name as a prefix:

```mizar
import algebra.groups.basic;
```

## 23.4 Build Lifecycle and Reproducibility

The build system executes the following phases in order for each source file. Phases are parallelised across independent files; files with import dependencies are processed in topological order.

| Phase | Input | Output | Failure mode |
|---|---|---|---|
| **Parse** | `.miz` source | AST | Syntax errors (E0001–E0099, §22.2) |
| **Name resolution** | AST + imported symbols | Resolved AST | Undefined name errors |
| **Type-check** | Resolved AST | Typed AST + VC list | Type errors (E0100–E0299, §22.3) |
| **VC generation** | Typed AST | Proof obligations | Logical inconsistency errors (E0400–E0499, §22.5) |
| **Pre-ATP** | Proof obligations | Discharged / open obligations | — (built-in engine, §21.7) |
| **ATP dispatch** | Open obligations | Proof certificates or failures | Proof errors (E0300–E0399, §22.4) |
| **Kernel check** | Proof certificates | `kernel_verified` / `externally_attested` | Certificate rejection (E0350–E0359, §22.4) |
| **Artifact emit** | Fully processed AST + certificates | `*.mizir.json`, logs | — |

**Cluster and overload resolution.** Cluster resolution (§23.7) and overload resolution via the registration subsumption DAG (§23.8) are performed **during the Type-check phase**: once the AST is name-resolved, the type-checker consults the import-filtered cluster indexes and DAG to derive attribute sets, select overload winners, and insert `qua` coercions. The resolution trace emitted here (§23.7.7 `resolution-trace/`) feeds later phases and the LSP server.

**Incremental builds.** Each phase's output is cached with a hash of its inputs. When a source file changes, only the affected file and its reverse-dependency cone are re-processed. The cluster-db (§23.7.7) is updated incrementally: only registrations in changed files are reindexed.

**Reproducibility.** A build is **reproducible** when, given identical source files, lock file, and toolchain version, it produces bit-for-bit identical artifacts. The build system achieves this by:

- Fixing the dependency versions through the lock file.
- Seeding ATP calls with a deterministic random seed (configurable in `[verifier]`).
- Sorting all sets and maps in artifact output by a canonical key order.

When `require_kernel_certificates = true`, reproducibility also requires that ATP backends produce deterministic proof certificates. Backends that cannot guarantee certificate determinism (e.g., Z3 in some configurations) produce `externally_attested` proofs, which are rejected in this mode.

## 23.5 Verifier Artifacts and Build Output

The build system writes three kinds of artifacts to the `artifact_dir` for each verified source file. These artifacts are the primary interface between the verifier and downstream tools: IDEs, AI assistants, documentation generators, and dependent packages.

### 23.5.1 `*.mizir.json` Artifact Format

For each source file `src/foo.miz`, the build system writes `build/foo.mizir.json`. This file records the verifier's resolved view of the source: inferred types, resolved symbols, inserted coercions, active proof obligations, and their proof status. It is the primary input for LSP hover responses (§23.6.1) and AI tool integrations.

Top-level structure:

```json
{
  "schema_version": "1.0",
  "source_file":    "src/foo.miz",
  "source_hash":    "sha256:a3f4...",
  "verified_at":    "2025-09-01T12:34:56Z",
  "expressions":    [ ... ],
  "obligations":    [ ... ],
  "exports":        [ ... ]
}
```

`verified_at` is local, user-visible metadata for humans and IDEs inspecting the build directory. It is excluded from canonical artifact content hashes and from publication reproducibility checks; two artifacts that differ only in `verified_at` are equivalent for dependency resolution and package publication.

**`expressions` array.** Each entry covers one expression node in the typed AST:

```json
{
  "expr_id":          "e124",
  "source_range":     "20:14-20:23",
  "kind":             "functor_application",
  "inferred_type":    "Element of PolyRing[K, x]",
  "resolved_symbol":  "algebra.poly.add",
  "template_args":    ["K", "x"],
  "inserted_coercions": ["PolyRing[K, x] qua AddStr"],
  "active_thesis":    "p + q = q + p",
  "overload_resolution": {
    "candidate_count":    3,
    "winning_candidate":  "algebra.poly.add#Def3",
    "selection_reason":   "unique_match"
  }
}
```

| Field | Content |
|---|---|
| `expr_id` | Stable identifier within the file (opaque string, unique per file version) |
| `source_range` | `"line:col-line:col"` (1-based, Unicode scalar values) |
| `kind` | AST node kind: `functor_application`, `predicate`, `type_ascription`, `variable`, etc. |
| `inferred_type` | The type assigned by the type-checker, in Mizar surface syntax |
| `resolved_symbol` | Fully-qualified name of the resolved functor, predicate, or mode |
| `template_args` | Concrete arguments substituted for schema parameters. Covers three distinct binding sources, disambiguated by a sibling `template_args_kind` field: functor argument bindings (`?X₁, …, ?Xₙ` of §23.7.3), struct parameter bindings (`?P` of a parameterized struct type `S₀[?P]`, §23.7.1), and template attribute schema bindings (`?X` of a template attribute, §23.7.1). Each entry is an object `{ "var": "?X", "value": "…", "source": "functor_arg"|"struct_param"|"attr_schema" }` |
| `inserted_coercions` | `qua` coercions inserted implicitly by the type-checker |
| `active_thesis` | The current proof obligation at this expression's position, if inside a proof |
| `overload_resolution` | Present when the expression involves an overloaded symbol; `null` for non-overloaded expressions |
| `overload_resolution.candidate_count` | Number of overload candidates considered during resolution |
| `overload_resolution.winning_candidate` | Fully-qualified name of the selected definition followed by `#` and the label (identifier) assigned to that specific overloaded definition at its declaration site (e.g., `algebra.poly.add#Def3`) |
| `overload_resolution.selection_reason` | How the winner was selected, using the vocabulary of the registration subsumption DAG (§23.8.3): `"unique_match"` (only one registration's argument guard matched), `"most_specific_in_dag"` (multiple registrations fired and the DAG identified a unique minimal element under `PickMostSpecific`), or `"explicit_qua"` (the author wrote an explicit `qua` coercion that forced the choice) |

The full overload resolution trace — all candidates considered and their scoring details — is available via `explain-overload <expr-id>` (§23.7.8). The `overload_resolution` field here records only the summary needed for hover display and citation tools.

**`obligations` array.** Each entry covers one verification condition:

```json
{
  "obligation_id":  "vc-42",
  "kind":           "vc.loop.maintain",
  "source_range":   "35:3-35:30",
  "goal":           "s + i = i * (i + 1) div 2",
  "hypotheses":     ["i <= n", "s = i * (i - 1) div 2"],
  "proof_status":   "kernel_verified",
  "certificate":    "tstp",
  "prover":         "vampire",
  "elapsed_ms":     412,
  "used_axioms":    ["mml.nat.Nat_triangle_step", "mml.nat.Nat_add_comm"],
  "proof_witness":  {
    "file":          "build/proof-witnesses/foo/vc-42.json",
    "hash":          "sha256:c4d5...",
    "kernel_checked": true
  }
}
```

> [!NOTE]
> The `proof_witness` field format is **tentative and under consideration** — a candidate for extraction into a separate implementation specification covering kernel replay, per-axiom instantiation scoping, higher-order binding representation, and term sharing. The full instantiation data is stored in a separate file (referenced by `proof_witness.file`) rather than inline, because multiple axioms used in the same proof may independently bind the same variable names, making a single flat substitution map ambiguous. The external file stores per-axiom bindings so each axiom's variable scope is independent. The schema shown below is illustrative; the authoritative format will be defined in that separate specification.

The referenced witness file (`build/proof-witnesses/<source_basename>/vc-<id>.json`) has the following structure:

```json
{
  "schema_version": "1.0",
  "obligation_id":  "vc-42",
  "kernel_checked": true,
  "steps": [
    {
      "axiom":         "mml.nat.Nat_triangle_step",
      "instantiation": { "?n": "i" }
    },
    {
      "axiom":         "mml.nat.Nat_add_comm",
      "instantiation": { "?a": "s", "?b": "i" }
    }
  ]
}
```

Each entry in `steps` records one axiom application. The `instantiation` map is scoped to that axiom's quantified variables, so the same variable name in two different axioms does not conflict. The exact representation — in particular how to handle higher-order instantiations and term sharing — is not yet finalised and may change in future schema versions.

`proof_status` is one of:

| Value | Meaning |
|---|---|
| `"kernel_verified"` | Certificate produced and accepted by the minimum kernel |
| `"externally_attested"` | ATP accepted the goal; no kernel-verifiable certificate produced |
| `"discharged_builtin"` | Discharged by the pre-ATP built-in engine (§21.7); no ATP call |
| `"open"` | Not yet proved; an error or timeout was reported |

The `used_axioms` field lists the fully-qualified names of axioms that appear in the accepted proof certificate. It is populated only when `proof_status` is `"kernel_verified"` or `"discharged_builtin"`. This field is the primary input for citation refinement (§23.5.4, §23.8).

The `proof_witness` field is a reference to an external witness file. `proof_witness.file` is the path to the file (relative to the package root), and `proof_witness.hash` is its SHA-256 content hash for integrity checking. The `proof_witness.kernel_checked` flag is `true` when the minimum kernel has successfully replayed the instantiated certificate; `false` when the certificate was produced by the ATP but kernel replay has not yet been performed (e.g., during an incremental build pass before the kernel phase completes). The witness file itself stores per-axiom instantiation substitutions; see the NOTE above for its format.

**`exports` array.** Each entry records one item exported from the module (theorem item, definition, registration, or exported algorithm) with its fully-qualified name and source range. For an exported algorithm the entry records its signature and `requires` / `ensures` contracts but not its implementation body (§12.5, §20.9.1). This array is used by the dependency resolver, by IDE navigation features, and by `mizar semver-check` (§23.10) to read the exported algorithm surface of a published baseline.

### 23.5.2 `@missing_facts` Output (Verifier-generated Premise Candidates)

For every proof obligation with status `"open"`, the verifier runs premise selection (§22.4.4) and records the ranked candidate list in `build/missing_facts.json`. This file is the primary input for AI-assisted `by`-clause completion.

```json
{
  "schema_version": "1.0",
  "source_file":    "src/foo.miz",
  "missing_facts":  [
    {
      "obligation_id": "vc-42",
      "kind":          "vc.loop.maintain",
      "source_range":  "35:3-35:30",
      "goal":          "s + i = i * (i + 1) div 2",
      "triage":        "missing_lemma",
      "candidates": [
        {
          "rank":         1,
          "score":        0.94,
          "name":         "Nat_triangle_step",
          "fqn":          "mml.nat.Nat_triangle_step",
          "source":       "mml/nat.miz:213",
          "conclusion":   "for n being Nat holds n * (n+1) div 2 + (n+1) = (n+1) * (n+2) div 2"
        },
        {
          "rank":         2,
          "score":        0.81,
          "name":         "Nat_div_add",
          "fqn":          "mml.nat.Nat_div_add",
          "source":       "mml/nat.miz:198",
          "conclusion":   "for a, b being Nat holds (a + b) div 2 = a div 2 + b div 2 when 2 divides a+b"
        }
      ],
      "retry_succeeded": false
    }
  ]
}
```

`triage` is one of `"counterexample"`, `"missing_lemma"`, or `"computation_limit"` (§22.4.2).

`retry_succeeded` is `true` when the verifier re-ran the ATP with the top-N candidates added to the axiom set and succeeded. In that case, the first entry in `candidates` is the lemma that resolved the obligation, and the IDE presents a one-click fix.

> [!NOTE]
> The `missing_facts.json` file is regenerated on every build pass that produces open obligations. It is not part of the package's published artifacts; it is a development-time tool. Publishing a package with open obligations (status `"open"`) is rejected by the package registry.

### 23.5.3 ATP Interaction Summary Logs

For each source file, the build system writes `build/foo.atp-log` recording all ATP interactions for the current build. The log is line-oriented JSON (one JSON object per line) for efficient streaming and grep-based filtering.

Each line records one ATP call:

```json
{"obligation_id":"vc-42","prover":"vampire","version":"4.8","elapsed_ms":412,"status":"proved","certificate_format":"tstp","axiom_count":37,"goal_size_literals":5}
{"obligation_id":"vc-43","prover":"vampire","version":"4.8","elapsed_ms":30000,"status":"timeout","axiom_count":128,"goal_size_literals":22}
{"obligation_id":"vc-43","prover":"e","version":"3.1","elapsed_ms":30000,"status":"timeout","axiom_count":128,"goal_size_literals":22}
```

Fields:

| Field | Content |
|---|---|
| `obligation_id` | Matches the `obligation_id` in `*.mizir.json` |
| `prover` | Backend name: `"vampire"`, `"e"`, `"cvc5"`, `"z3"` |
| `version` | Prover version string at build time |
| `elapsed_ms` | Wall-clock time for this ATP call |
| `status` | `"proved"`, `"timeout"`, `"counterexample"`, `"error"` |
| `certificate_format` | `"tstp"`, `"lfsc"`, `"alethe"`, `"none"` |
| `axiom_count` | Number of axioms sent to the prover |
| `goal_size_literals` | Number of literals in the negated conjecture |

The log is not required to be complete: when `incremental = true`, calls for cached obligations are not re-emitted. The log is intended for performance analysis, ATP tuning, and post-mortem diagnosis of verification failures. It is not a formal record of proof validity; that role belongs to the certificates in `*.mizir.json`.

### 23.5.4 Used-Axiom Recording

When the minimum kernel accepts a proof certificate, it extracts the set of axioms that appear in the certificate and records them in the `used_axioms` field of the corresponding obligation entry (§23.5.1). This set is always a subset of the axioms submitted to the ATP; it may be strictly smaller when the ATP included redundant premises in its search but did not use all of them in the final proof.

The verifier compares `used_axioms` against the explicit citations in the `by` clause of the corresponding proof step. If the `by` clause names lemmas that do not appear in `used_axioms`, the verifier emits warning `W0103` (§23.9.1).

Bulk citations (`module.*`) are never flagged as individually unused. Their purpose is to delegate lemma selection to the ATP; requiring them to be minimal would defeat that purpose. After a successful bulk citation, the `used_axioms` field records the specific lemmas that were actually used, enabling `mizar refine` (§23.9.2) to replace the bulk citation with a precise grouped citation.

> [!NOTE]
> `used_axioms` is empty when `proof_status` is `"externally_attested"` — no kernel-parseable certificate is available in that case. Refinement tools silently skip obligations without a populated `used_axioms`.

## 23.6 LSP Integration and IDE Support

The Mizar Evolution Language Server Protocol (LSP) server reads the build artifacts (§23.5) and the cluster-db (§23.7.7) to provide interactive IDE features. The server runs as a background process and responds to LSP requests from the editor without re-running the full build.

### 23.6.1 Hover, Code Actions, and Diagnostics

**Hover.** When the user hovers over an expression, the LSP server looks up the corresponding `expr_id` in `*.mizir.json` and returns:

- The inferred type (`inferred_type` field).
- The resolved symbol FQN (`resolved_symbol` field).
- Template arguments (`template_args`), if any.
- Inserted coercions (`inserted_coercions`), formatted as a human-readable note.
- The active thesis at that position (`active_thesis`), if inside a proof.

Example hover response (rendered by the IDE):

```
Type:     Element of PolyRing[K, x]
Symbol:   algebra.poly.add
          (resolved from `+` via PolyRing[K, x] → RingStr → AddStr)
Coercion: PolyRing[K, x] qua AddStr  (inserted implicitly)
Thesis:   p + q = q + p
```

**Diagnostics.** All `error[Ennnn]` and `warning[Wnnnn]` messages produced during the last build are surfaced as LSP diagnostics with their source ranges. Diagnostics are updated incrementally after each save: only the files affected by the change are re-verified.

Info-level diagnostics (`I`-series, §21.7) from `@show_thesis`, `@show_resolution`, and `@show_type` are surfaced as LSP `Hint` severity messages and appear as subtle inline annotations rather than underlines.

**Code actions.** The LSP server parses the `help:` suggestions in diagnostic messages (§22.1.3) and converts them into LSP code actions. One-click fixes are offered for:

- Adding a missing `by` clause lemma (from `missing_facts.json`).
- Inserting a `qua` coercion to resolve ambiguity.
- Annotating a binding with an inferred type (`as T`).
- Inserting a `@proof_hint` annotation.
- Adding a theorem to the `by` clause of an open obligation.

**Show Verification Conditions code action.** When the cursor is on a loop or algorithm contract, the LSP server offers a "Show Verification Conditions" code action that displays the list of open VCs with their proof status, sourced from the `obligations` array in `*.mizir.json`.

## 23.7 Cluster Resolution Graph

Cluster resolution — the propagation of attributes through conditional cluster registrations, functional cluster registrations, and struct inheritance — is one of the most opaque aspects of the Mizar type system for both human authors and AI tools. The verifier maintains two cluster-resolution indexes plus the shared registration subsumption DAG (§23.8). Existential clusters (§17.3) establish type inhabitation but do not propagate attributes and do not appear in any graph.

| Index / graph | Source | Role |
|---|---|---|
| **Conditional cluster graph** | Conditional cluster registrations (§17.4) | Attribute propagation rules over types |
| **Struct inheritance graph** | `extends` declarations (§5) | Struct subtype relationships |
| **Registration subsumption DAG** | Functional clusters (§17.5) and overload registrations | Guard subsumption and firing-set selection (§23.8) |

The BFS resolution algorithm (§23.7.4) consults the conditional cluster graph and struct inheritance graph to close attributes over a known result type. Functional clusters provide the initial attribute set for functor applications through the registration subsumption DAG (§23.7.3, §23.8), using `UnionAll` rather than a separate functor-cluster graph.

### 23.7.1 Conditional Cluster Graph

The conditional cluster graph captures attribute propagation rules over types. It stores only registered conditional cluster rules and does not encode struct inheritance.

The graph is **indexed by struct type**: there is one directed hypergraph per struct type `S`, written `G_S`. Because each BFS run operates in the context of a single concrete struct type (§23.7.4), every vertex within `G_S` is a plain attribute name — no struct type qualifier is needed.

**Vertices** of `G_S`:
- Plain attribute names applicable to `S` (e.g., `finite`, `non empty`, `complete`)
- Template attribute patterns — attribute names carrying a schema argument (e.g., `element_like[?X]`, `subset_like[?X]`). The schema argument is bound at BFS match time against concrete facts in the fact set (§23.7.4). Template attributes arise from cluster registrations that use a parameterized attribute name (§17.4.4).

Modes are not vertices in the graph. A mode defined as `mode M is A₁ A₂ … Aₙ T` (§7.2) is expanded at graph-build time into its underlying struct type `T` (recursively, until a struct type is reached). Conditional cluster rules whose `for` clause names a mode are rewritten using the expanded struct type and plain attribute names before the graph is populated.

**Non-struct mode targets.** A mode chain may terminate at a **non-struct primitive** — notably `set` (the universe of all sets) or an abstract ancestor that is not declared as a `struct` (§5). In that case the rule contributes to the per-kind graph `G_set` (or the corresponding primitive graph). The primitive graphs are treated uniformly with struct graphs throughout the BFS algorithm (§23.7.4); the only practical difference is that they have no `extends` edges in the struct inheritance graph (§23.7.2) and therefore participate only through their own rule set. Modes that expand to a type variable or to a term whose head is not a registered struct or primitive are rejected at registration time with an error, since BFS requires a statically-known target kind.

> [!NOTE]
> **Parameterized struct types** (e.g., `Subgroup of G` with `let G be Group`) are represented by a **single template-indexed graph** `G_{Subgroup[?P]}` whose rules carry a `when: ?P is P` guard naming the parameter constraint of the registration. A registration `let G be Group; cluster cyclic -> abelian for Subgroup of G` contributes `{ cyclic } → { abelian }   when: ?P is Group`, while `let G be abelian Group; cluster some_attr -> another_attr for Subgroup of G` contributes `{ some_attr } → { another_attr }   when: ?P is abelian Group`. Both rules live in the same `G_{Subgroup[?P]}` graph.
>
> Cross-parameter propagation — ensuring that rules registered for `Subgroup[Group]` also apply when the parameter is `cyclic Group` — is handled by the BFS algorithm (§23.7.4) through **dynamic parameter matching**, using the same mechanism as template attributes. When BFS processes a concrete `H be Subgroup of G` with `G be cyclic Group`, it binds `?P := cyclic Group` and checks each rule's `when` clause by **upcast only**: the struct inheritance chain plus the attributes **declared on** the parameter. `cyclic Group is Group` holds (same base, declared-attribute superset), so rules guarded by `?P is Group` fire. `cyclic Group is abelian Group` does **not** hold through this guard — even if `cluster cyclic -> abelian for Group` is registered — because cluster propagation is deliberately not applied to parameter types (see §23.7.4 step 4a for the rationale). No auxiliary inheritance edges are generated.

**Edges:** Each conditional cluster registration (§17.4) contributes one hyperedge to the graph for its stated struct type:

| Edge kind | Meaning |
|---|---|
| `conditional_cluster` | Attribute propagation from a conditional cluster registration (§17.4) |

Clusters with multiple antecedent attributes (e.g., `finite ∧ non empty → complete for LatticeStr`) are stored as **hyperedges**: the source set contains the plain attribute names of all antecedents. The hyperedge fires only when all source vertices are present in the current fact set. Rules with multiple consequent attributes are stored as-is in the graph, preserving the original registration structure. The BFS expands each consequent individually when a rule fires.

Typical fragments of per-struct conditional cluster graphs:

```
(* G_set — rules for struct type `set` *)

(* Single-antecedent: degenerate hyperedge *)
{ empty }
  --conditional_cluster(mml.sets.cluster_empty_finite)-->    { finite }
  --conditional_cluster(mml.sets.cluster_empty_countable)--> { countable }

{ finite }
  --conditional_cluster(mml.sets.cluster_finite_countable)--> { countable }

(* Template attribute: ?X is a schema variable bound at BFS match time.
   Fires when the fact set contains `element_like[v]` for some concrete v
   that satisfies the parameter constraint `v is finite set`. *)
{ element_like[?X] }
  --conditional_cluster(mml.sets.cluster_element_of_finite,
                        when: ?X is finite set)--> { finite }

(* Template attribute in both antecedent and consequent.
   Fires when the fact set contains `element_like[v]` for some `v is set`,
   and adds `element_like[[:v, v:]]` to the fact set. *)
{ element_like[?X], element_like[?Y] }
  --conditional_cluster(mml.sets.cluster_pair_element,
                        when: ?X is set, ?Y is set)--> { element_like[[:?X, ?Y:]] }

(* G_LatticeStr — rules for struct type `LatticeStr` *)

(* Multi-antecedent: proper hyperedge — fires only when ALL sources are in the fact set *)
{ finite, non empty }
  --conditional_cluster(mml.lattice.FiniteNonEmptyComplete)--> { complete }

(* G_RingStr — rules for struct type `RingStr` *)
{ skew_field }
  --conditional_cluster(mml.ring.skew_field_is_domain)--> { domain_like }

{ domain_like }
  --conditional_cluster(mml.ring.domain_is_non_degenerate)--> { non_degenerate }

(* G_{Subgroup[?P]} — template-indexed graph for parameterized `Subgroup of G`.
   Each rule carries a `when: ?P is P` guard naming its parameter constraint.
   At BFS time, ?P is bound to the concrete parameter type and guards are checked. *)
{ cyclic }
  --conditional_cluster(mml.group.cyclic_subgroup_abelian,
                        when: ?P is Group)--> { abelian }

{ normal }
  --conditional_cluster(mml.group.normal_subgroup_of_solvable_is_solvable,
                        when: ?P is solvable Group)--> { solvable }
```

The conditional cluster graph is stored as a global registration index, not physically rebuilt for every import closure. Each rule records the package or source artifact that introduced it. For a source file with import closure `I`, `G_S[I]` is the view obtained by filtering `G_S` to rules whose origin is visible in `I` (or by following precomputed visibility references). Two source files with different import sets may therefore see different rule views and reach different attribute conclusions for the same expression, without duplicating the graph per import environment (§23.7.7).

### 23.7.2 Struct Inheritance Graph

The struct inheritance graph encodes subtype relationships between struct types declared via `extends`. It is a directed acyclic graph (DAG) separate from the conditional cluster graph.

**Vertices:** Struct types. Parameterized instantiations are **not** vertices here: parameter variation is handled inside the template-indexed conditional cluster graph `G_{S[?P]}` via dynamic BFS matching (§23.7.1, §23.7.4), not by auxiliary edges in this graph.

**Edges — `extends` declarations (§5):** Each `extends` declaration contributes one directed edge from sub-struct to super-struct:

```
RingStr        --inheritance--> AddStr
RingStr        --inheritance--> MulStr
```

The struct inheritance graph itself does not encode attribute facts — edges carry no attribute payload. Its sole role is to select which per-struct cluster rule graphs contribute to the BFS rule set (§23.7.4 step 2). Attribute propagation from parent-type cluster rules to child-type expressions does occur, but uniformly via the rule-union mechanism — not through any special "inheritance transfer" pass.

The `qua` keyword in source code (e.g., `x qua AddStr`) selects which inheritance path to follow when multiple paths exist (disambiguation); it does not add edges to this graph.

The struct inheritance graph is stored in the `cluster-db` (§23.7.7) alongside the conditional cluster rule index.

### 23.7.3 Functional Cluster Registrations

Functional cluster registrations assign initial attributes to the result of a specific functor application. They are stored as registration nodes in the shared registration subsumption DAG (§23.8), not as a separate functor-cluster graph. A functional cluster registration (§17.5) has the general form:

```
let x₁ be P₁, …, xₙ be Pₙ;
cluster Label: F(x₁, …, xₙ) -> A₁ A₂ … for T;
```

and declares that when the argument constraints `x₁ is P₁, …, xₙ is Pₙ` are satisfied, the expression `F(x₁, …, xₙ)` has struct type `T` with attributes `A₁, A₂, …`. The argument constraints are **guards** on the registration, handled analogously to the `when` guards of conditional clusters (§23.7.1).

Each functional cluster contributes a `RegistrationNode` (§23.8.1):

```
RegistrationNode {
  symbol            = F,
  argument_guard    = [when: ?X₁ is P₁, …, ?Xₙ is Pₙ],
  consequent        = { result_type = T, attributes = { A₁, A₂, … } },
  resolution_policy = UnionAll,
  assume_clause     = None
}
```

Registrations without argument constraints (the unconditional case) carry an empty `when` clause and always fire.

A typical fragment of functional cluster registration nodes:

```
(* Single guard without attribute refinement — applies to any Ring argument *)
mml.poly.PolyRing_is_Ring:
  symbol=PolyRing
  guard=[?R is Ring]
  consequent={ result_type=RingStr,
               attributes={ non empty, associative, abelian } }
  policy=UnionAll

(* Multiple conditional registrations for the same functor — all may fire *)
mml.sets.CartesianNonEmpty:
  symbol=Cartesian
  guard=[?X is non empty set, ?Y is non empty set]
  consequent={ result_type=set, attributes={ non empty } }
  policy=UnionAll

mml.sets.CartesianCountable:
  symbol=Cartesian
  guard=[?X is countable set, ?Y is countable set]
  consequent={ result_type=set, attributes={ countable } }
  policy=UnionAll

mml.sets.CartesianFinite:
  symbol=Cartesian
  guard=[?X is finite set, ?Y is finite set]
  consequent={ result_type=set, attributes={ finite } }
  policy=UnionAll
```

**Guard evaluation.** Unlike the parameter guards of §23.7.4 step 4a (which are evaluated by upcast only), functional cluster guards **do consult the BFS-derived attribute set of each argument**. This matches current Mizar behavior: when checking whether `?X is finite set` holds for a concrete argument `A`, the verifier uses `A`'s fully propagated attribute set (e.g., `A` inherited `finite` via a conditional cluster such as `empty → finite`). The distinction is justified by the different roles of the two guards:

- **Parameter guard (§23.7.4 step 4a)** selects which rule view applies for a *type declaration* (`let G be abelian Group`) — a declarative act where only declared attributes are considered in scope.
- **Functional cluster guard (here)** applies to argument *expressions* at a functor application site — evaluating their actual (BFS-closed) attribute content is appropriate because those arguments already carry derived facts in the ambient fact set.

**Interaction with the other graphs.** When the verifier infers the type of an expression `e = F(a₁, …, aₙ)`:

1. Compute the BFS-closed attribute set `F_i` for each argument `aᵢ` (recursive application of §23.7.4).
2. Enumerate every visible functional-cluster registration node for symbol `F` and test its guard against `F₁, …, Fₙ`. All registrations whose guard holds fire simultaneously under the `UnionAll` policy. The registration subsumption DAG (§23.8) accelerates this enumeration: once the most-specific firing registration is identified, all its active ancestors are known to fire without re-evaluating their guards.
3. Let `A₀ := ∪ { consequent.attributes of each firing registration }`. The result struct type `T` is taken from the firing registrations. All firing registrations for the same functor must agree on `T`; this consistency is checked for the current import closure and reported as a build error if violated, since two registrations with different result types only conflict once both become visible together.
4. Seed BFS (§23.7.4) with `S = T` and `A₀` for further propagation via conditional clusters on `T`.

The same DAG is also used for overload resolution with the `PickMostSpecific` policy (§23.8.3). Thus the build system maintains one guard-subsumption structure for both functional cluster firing and overloaded functor/mode/predicate/attribute selection.

### 23.7.4 BFS Resolution Algorithm

The BFS resolution algorithm determines the complete attribute set of an expression whose result struct type `S` is already known, given an initial attribute set `A₀`. For a functor application, `S` and `A₀` are obtained first by firing visible functional-cluster registration nodes through the registration subsumption DAG (§23.7.3, §23.8). BFS itself does not resolve functor overloads and does not traverse functional-cluster registrations; it only closes `A₀` under visible conditional clusters and struct inheritance.

**Inputs:**
- Concrete struct type `S` (possibly a parameterized instantiation `S₀[P₀]` — e.g., `Subgroup[abelian Group]` derived from `H be Subgroup of G` with `G be abelian Group`)
- Initial attribute set `A₀` (plain or instantiated attribute names; supplied by functional cluster registrations when applicable, or empty otherwise)
- Import closure `I` — the set of packages transitively imported by the source file being verified
- Global conditional cluster rule index `G`, with each rule tagged by its origin package/source artifact (§23.7.1)
- Global struct inheritance graph `H`, with each edge tagged by its origin package/source artifact (§23.7.2)
- Visibility predicate `visible_I(origin)`, derived from the import closure and used to filter both indexes

**Algorithm:**

1. **Enumerate applicable super-structs.** Traverse only visible inheritance edges:
   ```
   H[I]      := { e ∈ H | visible_I(e.origin) }
   Supers(S) := { T | S →* T in H[I] }
   ```
   For a parameterized `S = S₀[P₀]`, keep the global parameter binding `?P := P₀`. Parameterized instantiations are not expanded into separate inheritance vertices.

2. **Collect a visible rule view.** Build a local rule view by selecting rules from the global conditional cluster index whose origin is visible and whose target kind is applicable to `S`:
   ```
   Sources(S) := { base(S) } ∪ Supers(S)
   if S = S₀[P₀], also include the template source S₀[?P]

   Rules :=
     { r ∈ G
       | visible_I(r.origin)
       ∧ r.target ∈ Sources(S) }
   ```
   This is a filtered view, not a rebuilt graph. Implementations may materialize it for the current source file, or may keep references into the global index and test `visible_I` lazily. Each selected rule has the form `{ B₁, …, Bₙ } → { C₁, …, Cₘ }` with plain or template attributes such as `element_like[?X]`, optionally carrying a parameter guard such as `when: ?P is Group`.

3. **Initialise fact set.** `F := A₀`.

4. **BFS loop** (runs until `F` is stable):

   a. **Prepare fixed parameter guards.** For each rule carrying a `when: ?P is P` guard from a template source `S₀[?P]`:
      - Bind `?P := P₀` using the concrete parameter of `S = S₀[P₀]`.
      - Evaluate the guard once for this BFS run by **upcast only**: `P₀ is P` holds iff `H[I]` has a path `base(P₀) →* base(P)` and the attribute set **declared on** `P₀` is a superset of the attribute set declared on `P`. Cluster propagation is **not** applied to the parameter — only directly declared attributes and modes participate.
      - If the guard fails, mark the rule inactive for this run. If it holds, substitute `?P` in the rule consequents and keep the rule active.

      For example, `abelian Group is Group` holds (attribute superset, same base struct); `cyclic Group is abelian Group` does **not** hold through this guard even if `cluster cyclic -> abelian for Group` is visible, because `abelian` is not among `cyclic Group`'s declared attributes.
      - **Rationale:** a parameter type is a declaration of intent by the author, not a fact to be inferred. Running BFS recursively on the parameter would make guard evaluation expensive, doubly dependent on the import closure, and would blur the separation between "which rule set applies" and "what attributes are derived". Authors who want the wider match should either widen the parameter declaration (e.g., `let G be abelian Group`) or register the rule at the weaker level (e.g., `for Subgroup[?P] when: ?P is Group`).

   b. **Plain active rules.** For each active rule `{ B₁, …, Bₙ } → { C₁, …, Cₘ }` with no fact-level schema variables:
      - If all antecedents `B₁, …, Bₙ` are present in `F`, add each consequent `C₁, …, Cₘ` to `F` and enqueue any rules whose new antecedent was just satisfied.

   c. **Template attribute active rules.** For each active rule `{ B₁[?X], …, Bₙ[?X] } → { C₁[?X], …, Cₘ[?X] }` containing a fact-level schema variable `?X`:
      - For each candidate binding `?X := v` obtained by matching any fact of the form `Bᵢ[v]` in `F`:
        - Verify the parameter constraint `v is P` (where `let ?X be P` is declared in the registration) **against the BFS-closed attribute set of `v`** — parallel to the functional cluster guard evaluation (§23.7.3), not the upcast-only policy of step 4a. `v` here is a concrete expression bound at match time, so derived attributes of `v` are in scope, consistent with the ambient fact set.
        - If all antecedents `B₁[v], …, Bₙ[v]` are present in `F` with the same binding:
          - Add each instantiated consequent `C₁[?X := v], …, Cₘ[?X := v]` to `F`.
          - Enqueue rules whose new antecedent was just satisfied.
      - Multiple schema variables (e.g., `?X` and `?Y`) are bound independently; all must be consistently satisfied before the rule fires.
      - Rules may combine a fixed `?P` parameter guard (step a, upcast-only) with fact-level schema variables (step c, BFS-closed); both matching conditions must hold under their respective policies.

   d. Use per-rule antecedent counters (with binding-aware tracking for template rules) to detect firing efficiently without re-scanning all rules on each iteration. Parameter guard results (step a) are evaluated once per BFS run since `?P` is fixed by the concrete input type `S`.

5. **Result.** The final fact set `F` contains all attribute names — plain and template-instantiated — that hold for the expression of struct type `S` **under the current import closure `I`**.

**Key properties:**
- **Import-sensitive.** The same expression may yield a larger `F` in a file that imports additional cluster-providing packages. Adding an import can only grow `F` (monotone with respect to imports); removing an import may shrink it, potentially causing type errors.
- All vertices in `F` are plain or instantiated attribute names (schema variables are eliminated at match time); the struct type `S` is implicit context. Inheritance only determines which conditional rule views contribute to `Rules`.
- Monotone over iterations: attributes are only added, never removed. Termination relies on the registered clusters being designed to terminate, which is enforced by the coherence proof requirement (§17.4).
- Attribute stripping (e.g., treating `finite DoubleMagmaStr` as `DoubleMagmaStr`) is implicit type subsumption: the fact set contains more attributes than required, which always satisfies a weaker requirement.

### 23.7.5 Local Subgraph Extraction (Path View, Neighborhood View)

AI tools and IDE features receive **local subgraphs**, not the full global graph. Two extraction modes are defined.

**Path view.** Returns the minimal sequence of edges that the verifier actually traversed to resolve a specific query. This is the primary response for "why does `T` have attribute `A`?" queries.

```json
{
  "query":  "S is countable",
  "status": "resolved",
  "path": [
    {
      "kind":   "conditional_cluster",
      "from":   "empty set",
      "to":     "finite set",
      "source": "mml.sets.cluster_empty_finite",
      "location": "mml/sets.miz:47"
    },
    {
      "kind":   "conditional_cluster",
      "from":   "finite set",
      "to":     "countable set",
      "source": "mml.sets.cluster_finite_countable",
      "location": "mml/sets.miz:112"
    }
  ]
}
```

Each path entry's `kind` field takes one of the following values, corresponding to the graph edges and registration-firing steps defined in §23.7.1–§23.7.3 and §23.8:

| `kind` | Source section | Meaning |
|---|---|---|
| `conditional_cluster` | §23.7.1 | Attribute propagation within a per-struct conditional cluster graph |
| `inheritance` | §23.7.2 | Struct subtyping via `extends` |
| `functional_cluster` | §23.7.3 | Initial attribute set from a functor application |
| `subsumption` | §23.8 | Firing-set inclusion between registrations for the same symbol |

**Neighborhood view.** Returns the immediate neighbors of a vertex in the graph, together with competing candidates and the reason one was selected (or why resolution is ambiguous). This is the primary response for "why is this ambiguous?" queries.

```json
{
  "query":  "PermProduct(R)",
  "status": "ambiguous",
  "vertex": "RingStr",
  "candidates": [
    {
      "target": "AddStr",
      "edge_kind": "inheritance",
      "path_length": 2,
      "reason": "RingStr → AddStr → MagmaStr (additive path)"
    },
    {
      "target": "MulStr",
      "edge_kind": "inheritance",
      "path_length": 2,
      "reason": "RingStr → MulStr → MagmaStr (multiplicative path)"
    }
  ],
  "suggested_fix": "insert `qua` annotation in source to disambiguate: `PermProduct(R qua MulStr)`"
}
```

The extraction depth is bounded: by default, path view returns at most 8 edges, and neighborhood view returns at most 16 candidates. These limits can be increased via the explanation API (§23.7.8).

### 23.7.6 Explanation Extractor (Success, Ambiguity, Failure)

The **explanation extractor** is the layer between the global graph and the local subgraph responses. It takes a resolution result from the verifier and constructs a human- and machine-readable explanation.

Three outcome classes are handled:

**Success.** The verifier found a unique resolution path. The extractor returns the path view (§23.7.5) showing each step, its source registration, and the location in the library where the registration is defined.

**Ambiguity.** Two or more resolution paths of equal length and priority were found. The extractor returns the neighborhood view with all competing candidates. The `suggested_fix` field proposes a concrete disambiguation (typically a `qua` coercion or a qualified name).

**Failure.** No resolution path exists. The extractor classifies the failure:

| Failure kind | Meaning | Suggested fix |
|---|---|---|
| `unreachable` | No path connects the source vertex to the required target | Register a missing cluster or add a `qua` coercion |
| `attribute_not_registered` | The required attribute has no cluster that fires for this type | Add a cluster registration for the type |
| `schema_var_unbound` | A schema variable (parameter `?P`, fact-level `?X`, or functor argument) could not be bound to a concrete value | Supply the argument explicitly, or widen the matching rule's guard |
| `qua_required` | A `qua` coercion is needed but was not written | Insert `qua Interface` at the relevant site |
| `sethood_missing` | Fraenkel comprehension requires `sethood` (§21.7.6) | Register `sethood` for the type |

For AI tools, failure explanations are more actionable than success traces: they identify the missing registration or coercion that would make the code type-check.

### 23.7.7 Storage Format (`cluster-db`, `resolution-trace`, `diagnostic-explanation`)

The cluster resolution infrastructure uses three storage layers in the build cache:

**`cluster-db/`.** The cluster and registration indexes, stored as static build-cache data with per-registration visibility metadata. The cache is not duplicated for every import closure: the verifier derives an import-scoped view by filtering rules, declarations, and DAG edges whose origins are visible from the source file's import closure. When the set of imported packages changes, only the visibility view is recomputed; when source files within the current package change, only the affected registrations and their derived index entries are updated.

The cluster-db is a directory containing:
- `graph.json` — the full vertex and edge list for the conditional cluster graph and struct inheritance graph in a compact adjacency format, with origin identifiers for import filtering.
- `subsumption-dag.json` — the registration subsumption DAG skeleton (§23.8): nodes keyed by `(symbol, guard)`, edges from stronger to weaker with their conditional-cluster preconditions recorded so the DAG can be filtered against any import closure at query time.
- `attr-index.json` — inverted index from attribute name to all registrations that produce it.
- `type-index.json` — inverted index from type name to all registrations triggered by it.
- `reduction-index.json` — term-head index from resolved `LHS` heads to reduction registrations, with each rule's FQN, type/attribute guard summary, `such` side-condition summary, simplification-order metadata, and visibility origin.

**`resolution-trace/`.** One file per verified source file (`foo.resolution-trace.json`), recording the cluster-resolution paths and reduction applications actually used during type-checking of that file. The recorded cluster paths are the inputs to the path-view extraction (§23.7.5). The recorded reduction steps are used for kernel replay and strategy audit (§17.6.4), not for graph path-view extraction. The file is regenerated on every verification pass for the affected file.

The resolution trace also serves as the **proof certificate for minimum-kernel verification of automatic registration effects**. Each trace step is tagged as either a cluster step or a reduction step. Cluster steps form the ordered sequence of rule applications actually taken by the BFS traversal; because each referenced rule has already been validated by a `coherence` proof (§17.4, §17.5), the kernel verifies the chain by checking that the referenced cluster registration exists, that all antecedents are present in the current fact set, and then adding the consequent.

For a reduction step, the trace records two layers of evidence. The kernel-replay layer records the reduction rule FQN, the source redex, the instantiated target term, the pattern substitution, and the discharged type/attribute and `such` guards. Because the rule has already been validated by a `reducibility` proof (§17.6), the minimum kernel verifies only the local rewrite evidence: the recorded redex matches the rule's `LHS`, the recorded target is the corresponding `RHS` instance, and every recorded guard has stable evidence, either as a fact/trace reference or as an explicit citation. The strategy-audit layer records the enclosing term fingerprint before the step, the redex path inside that term, the active reduction rule-view fingerprint, and the selected rule's specificity/FQN selection key. This data is checked by the deterministic normalizer before the trace is emitted and may be recomputed by artifact validators; the minimum kernel does not perform rule search or reselect reductions. Replay remains a linear-time check requiring no external solver.

**`diagnostic-explanation/`.** One file per source file (`foo.diagnostic-explanation.json`), recording the explanation extractor's output for every ambiguity and failure encountered during type-checking. This file is the primary input for IDE hover and code-action responses related to resolution errors.

All three files use UTF-8 encoded JSON. Their schemas are versioned with a `schema_version` field at the top level; tools must check this field and refuse to read files with an unsupported schema version.

### 23.7.8 Explanation API

The explanation API is exposed through three interfaces: the LSP server (for IDE use), the CLI (`mizar explain`), and directly from `*.mizir.json` files (for AI tool use).

The five query types are:

**`explain-type <expr-id>`**. Returns the full type inference trace for the expression with the given `expr_id`. Includes the path view of any cluster-driven type widening that contributed to the inferred type.

CLI: `mizar explain type --file src/foo.miz --expr e124`

LSP: triggered by hovering over an expression; the result populates the hover response.

**`explain-attribute <type> <attribute>`**. Returns the resolution path (or failure explanation) for the query "`<type>` has attribute `<attribute>`". Accepts Mizar surface syntax for both arguments.

CLI: `mizar explain attribute "finite set" "countable"`

Response (success):

```json
{
  "query":  "finite set is countable",
  "status": "resolved",
  "path": [
    { "kind": "conditional_cluster", "from": "finite set", "to": "countable set",
      "source": "mml.sets.cluster_finite_countable", "location": "mml/sets.miz:112" }
  ]
}
```

**`explain-overload <expr-id>`**. Returns the overload resolution result for the expression: the winning candidate, all candidates considered, and the scoring details that determined the winner. Equivalent to the output of `@show_resolution` (§21.5.2) but in structured JSON.

**`explain-qua <expr-id>`**. Returns the explanation for a `qua` coercion: which view was selected, why it was needed, and what alternatives exist. For implicitly-inserted coercions (recorded in `inserted_coercions` in `*.mizir.json`), this query explains why the insertion was necessary.

**`explain-failure <diagnostic-id>`**. Returns the full diagnostic explanation for a specific error or warning, identified by its `CODE` (e.g., `E0201`) and source range. The response includes the explanation extractor's classification (§23.7.6), the competing candidates or missing registrations, and the suggested fix. This is the programmatic equivalent of the `help:` lines in diagnostic output (§22.1.3).

CLI: `mizar explain failure --code E0201 --file src/foo.miz --line 42`

All API responses are JSON objects with a `schema_version` field. The CLI renders them as human-readable text by default; pass `--json` to receive the raw JSON. The LSP server always uses the JSON form internally and renders it for display using the IDE's hover and code-action formatting.

> [!NOTE]
> The explanation API is designed for incremental queries, not bulk dumps. AI tools should request explanations for specific expressions or failures rather than requesting the full cluster-db or all resolution traces at once. The bounded extraction depth (§23.7.5) enforces this: increasing the depth limit for a single query is supported, but the limit exists to prevent AI context windows from being overwhelmed by graph data for large library environments.

> [!NOTE]
> The contract for how an AI *agent* operates on a project is a tooling contract rather than a language feature, and is described in the architecture design document [doc/design/architecture/en/21.ai_agent_interface.md](../../design/architecture/en/21.ai_agent_interface.md), not in this specification. That document normatively defines only the safe edit classes (Green/Yellow/Red) and the authorization-scope vocabulary; the context and patch protocol is outlined there as a deferred, non-normative framework whose wire formats are not yet frozen.

### 23.7.9 Memory Design Principles

The storage and query design described in §23.5 and §23.7 follows a single, deliberate principle: **keep interfaces and indexes resident; load proof bodies, traces, and detailed AI-facing data lazily.** This is the build-system and tooling counterpart of the separate-compilation memory model (§12.6.3), and it is what lets the verifier and the language server scale to large library environments without their memory growing with the size of the library.

The principle is realized by the following design rules, each already specified in this chapter:

| Rule | Mechanism | Section |
|---|---|---|
| Do not duplicate the cluster/registration graph per import closure | A single global `cluster-db` from which an import-scoped view is derived by filtering on visibility origin | §23.7.7 |
| Do not load whole registration data sets into memory | On-disk inverted indexes (`attr-index`, `type-index`, `reduction-index`) and a compact adjacency format for the graph | §23.7.7 |
| Do not hold resolution/reduction traces resident | One `resolution-trace` artifact per source file, regenerated per pass and read on demand | §23.7.7 |
| Do not inline proof witnesses | External witness files referenced from `*.mizir.json` by path and hash | §23.5.1 |
| Do not bulk-dump graph data to AI tools | Bounded, incremental explanation queries with a default extraction-depth limit | §23.7.5, §23.7.8 |
| Recompute only what changed | Incremental cluster-db update and reverse-dependency-cone re-processing | §23.4 |

The LSP server (§23.6) consumes these artifacts the same way: it reads `*.mizir.json`, the `cluster-db` view, and the per-file traces rather than reconstructing global state in memory, and it answers hover, code-action, and explanation requests from indexed data plus on-demand lookups.

> [!NOTE]
> These rules express a qualitative memory model, not performance guarantees. Concrete memory budgets and benchmark metrics (e.g., peak RSS per file, per import-closure size, or LSP idle memory) belong to an evaluation methodology and are intentionally out of scope for this specification.

## 23.8 Registration Subsumption DAG

Multiple registrations for the same symbol commonly form a subsumption order on their argument guards — e.g., for the functor `[:_, _:]`, a registration guarded by `{ ?X, ?Y : finite set }` is strictly stronger than one guarded by `{ ?X, ?Y : countable set }`, because the conditional cluster `finite → countable` (§23.7.1) makes every `finite set` also `countable set`. The same phenomenon occurs for overloaded **modes**, **predicates**, and **attributes**, not only functors.

This chapter defines the **registration subsumption DAG** as a shared substrate serving both cluster firing (§23.7.3, `UnionAll` policy) and overload resolution (`PickMostSpecific` policy). Its purpose is twofold:

- **Acceleration.** Once the most-specific firing registration is identified, all its ancestors are known to fire (for `UnionAll`) without re-evaluating their guards.
- **Unification.** Overload resolution for functors, modes, predicates, and attributes reuses the same subsumption machinery, with only the resolution policy and the presence of an `assume` clause distinguishing consumers.

The language-level semantics of each overload consumer is specified in its own chapter (attributes §6, modes §7, functors §10, predicates §9); this chapter defines only the shared resolution infrastructure.

### 23.8.1 Abstract Registration Node

A registration node carries:

```
RegistrationNode {
  symbol              : functor | mode | predicate | attribute,
  argument_guard      : [when: ?X₁ is P₁, …, ?Xₙ is Pₙ],
  consequent          : <consumer-specific payload>,   (* cluster: attribute set;
                                                          overload: definition body *)
  resolution_policy   : UnionAll | PickMostSpecific,
  assume_clause       : Option<FOL formula>            (* overload only; None for clusters *)
}
```

- **`symbol`** identifies the overloaded entity whose registrations form one subsumption lattice.
- **`argument_guard`** uses the same `when: ?X is P` form as conditional cluster guards (§23.7.1), so the entailment machinery is shared.
- **`consequent`** is opaque to the DAG; each consumer chapter defines its shape.
- **`resolution_policy`** determines how the firing set is consumed (§23.8.3).
- **`assume_clause`** is attached to overload nodes but does not participate in subsumption (§23.8.4).

### 23.8.2 Subsumption Edges and Import-Scoped Instantiation

**Edge condition.** An edge `R_strong → R_weak` is added when `R_strong.argument_guard` entails `R_weak.argument_guard`. The entailment is witnessed by a (possibly empty) set of conditional cluster rules drawn from the cluster graph (§23.7.1); these are recorded as the edge's **precondition**:

```
R_finite_finite  --subsumption(via mml.sets.finite_is_countable on {?X, ?Y})--> R_countable_countable
```

The `on {?X, ?Y}` annotation lists the schema variables to which the cited conditional cluster rule is applied independently. When a precondition requires multiple distinct cluster rules, they are listed as a set: `via { rule_a on {?X}, rule_b on {?Y} }`.

**Import-scoped instantiation.** Because the precondition cites conditional cluster rules, the DAG is parameterized by the import closure `I`: an edge is active only if every rule it cites is in `I`. The full DAG skeleton (edges + preconditions) is built once and stored in `cluster-db` (§23.7.7); at query time it is filtered against `I` to obtain the active DAG.

When an edge `R_strong → R_weak` is inactive under the current `I` (because the conditional cluster rule proving `R_strong.guard ⇒ R_weak.guard` is not imported), traversal from `R_strong` cannot automatically imply that `R_weak` fires. In that case the `R_weak` registration must have its guard evaluated independently against the concrete arguments — the same work that would have been done without a DAG. The DAG is therefore a performance accelerator, not a correctness dependency: enumeration over all registrations followed by independent guard checks remains a valid (if slower) implementation, and results are identical.

**Acyclicity.** The subsumption relation is a partial order whose edges are always oriented from strictly stronger to strictly weaker guards (equal-guard registrations for the same symbol are an error detected at load time, not collapsed into edges). The resulting graph is therefore a DAG by construction.

### 23.8.3 Resolution Policies

Consumers differ in how they consume a firing set:

| Consumer | `resolution_policy` | Firing behavior |
|---|---|---|
| Functional cluster (§17.5) | `UnionAll` | All firing registrations contribute; consequents (attribute sets) are unioned. |
| Overloaded functor / mode / predicate / attribute | `PickMostSpecific` | Select the unique minimal element in the active DAG; ambiguity (incomparable minimal elements) is a hard error. |

For `UnionAll`, the DAG enables a single traversal: fire the most-specific matching registration, then walk to ancestors through active edges; all guard tests on ancestors are skipped.

For `PickMostSpecific`, the DAG directly provides the selection: the most-specific firing registration is the one with no firing descendants along active edges. Ambiguity — two or more minimal firing registrations that are incomparable under the active DAG — is reported as a resolution error that the author must resolve by explicit disambiguation (e.g., `qua`) or by importing the package that provides the subsumption edge.

### 23.8.4 Treatment of `assume` Clauses

Overload registrations may carry an `assume P(x)` clause. These clauses do **not** participate in subsumption edges, for three reasons:

1. `assume` clauses are arbitrary FOL formulas; logical implication between them is not decidable, so they cannot provide stable edge conditions.
2. Current Mizar's overload resolution resolves by argument guards first and discharges `assume` as a separate proof obligation at the use site. We preserve this two-stage behavior.
3. If `assume` participated in the subsumption order, "which registration is most specific" would depend on proof discharge, making type inference dependent on theorem proving — a non-starter for decidable resolution.

The `assume` clause is therefore carried on the selected `RegistrationNode` and discharged at the use site after resolution completes. Failure to discharge is a proof error, not a resolution error — the resolution outcome is fixed as soon as the argument guards are matched.

## 23.9 Citation Refinement

Citation refinement is the process of replacing verbose or bulk `by` citations with the minimal set of lemmas that the kernel actually used. The verifier records used axioms in proof artifacts (§23.5.4) and supports refinement through a warning and two purpose-built CLI tools.

### 23.9.1 Verifier Warning (W0103)

When the explicit `by` clause of a proof step names lemmas not present in `used_axioms` (§23.5.4), the verifier emits:

```
warning[W0103]: by clause contains unused citations (proof.citation.unused)
  --> src/lib.miz:35:7
   |
35 |   thus thesis by poly_eval_add, poly_eval_mul, poly_eval_const;
   |                                                ^^^^^^^^^^^^^^^ not used in proof certificate
   |
help: remove unused citation, or run `mizar refine` to rewrite automatically
```

This warning is informational: verification succeeds regardless. Its purpose is to surface redundant citations so that authors can tighten proofs incrementally.

Bulk citations (`module.*`) are exempt from W0103. The intent of a bulk citation is to delegate lemma selection to the ATP; requiring it to be minimal would defeat its purpose. Use `mizar refine` to convert a successful bulk citation to a precise explicit form.

### 23.9.2 `mizar refine` — Fast Refinement from Proof Certificates

`mizar refine` rewrites `by` citations in source files using the `used_axioms` sets recorded in `*.mizir.json`. It does not re-run the ATP or the verifier; it reads the existing build artifacts and performs a purely syntactic rewrite of the source.

```
mizar refine [--file <path>] [--dry-run] [--all]
```

| Option | Effect |
|---|---|
| `--file <path>` | Refine citations in the given source file only |
| `--dry-run` | Print proposed rewrites without modifying files |
| `--all` | Refine all source files in the package |

**Behavior.** For each proof step with a `by` clause:

1. Look up the corresponding obligation entry in `*.mizir.json`.
2. If `proof_status` is `"kernel_verified"` and `used_axioms` is non-empty, replace the citation list with the names in `used_axioms`, sorted canonically.
3. Bulk citations (`module.*`) are replaced with grouped citations (`module.{name1, name2, ...}`) when all used axioms come from the same module, or with flat explicit citations otherwise.
4. Steps where `proof_status` is `"externally_attested"` or `"open"` are left unchanged.

After running `mizar refine`, a subsequent build should succeed with all obligations at `kernel_verified` status. If it fails, the tool reports the discrepancy and restores the original citations.

> [!NOTE]
> `mizar refine` is idempotent: running it twice produces the same result as running it once. Its output is safe to commit directly.

### 23.9.3 `mizar minimize` — Brute-Force Minimization

`mizar minimize` goes further than `mizar refine` by iteratively attempting to remove individual citations from each `by` clause and re-verifying. It finds the minimal subset of citations that still allows the ATP to close the goal, independent of what is recorded in the proof certificate.

```
mizar minimize [--file <path>] [--strategy <greedy|delta>] [--timeout <sec>]
```

| Option | Effect |
|---|---|
| `--file <path>` | Minimize citations in the given source file only |
| `--strategy greedy` | Remove one citation at a time, keeping removals that still verify (default) |
| `--strategy delta` | Delta-debugging: binary-search over citation subsets (fewer ATP calls on average for large citation sets) |
| `--timeout <sec>` | Per-attempt ATP timeout; defaults to the package `atp_timeout` setting |

**When to prefer `mizar minimize` over `mizar refine`.**

- When `proof_status` is `"externally_attested"`: no certificate is available, so `mizar refine` cannot act, but `mizar minimize` can still find a smaller citation set by re-running the ATP with subsets.
- When a proof was written manually with a large explicit `by` list: `mizar refine` only removes lemmas absent from the certificate; `mizar minimize` may find that the ATP succeeds with a strictly smaller subset by discovering an alternative proof path.
- When maximally compact proofs are required, for example before publishing a package to the formal library.

`mizar minimize` is significantly slower than `mizar refine` because it requires multiple ATP calls per proof step. It is intended as an occasional cleanup tool rather than a step in the normal build cycle.

## 23.10 API Compatibility Checking (`mizar semver-check`)

`mizar semver-check` compares the public API surface of the working tree (or a candidate version) against a released **baseline** and reports the version bump the change requires under the compatibility classification of §23.2. It is the toolchain's version linter: it answers "does this change require a MAJOR or MINOR bump?" and, in CI, fails when the manifest's declared version increment is smaller than required. PATCH-level changes are out of its scope (see *Scope* below). The reported bump is the smallest one required by the changes the tool can detect **from its inputs**; for most change classes that detection is exact, but for overload-resolution shifts it is incomplete on the default single-package input (see *Gate semantics and completeness* below).

```
mizar semver-check [--baseline <git-ref|version>] [--check] [--json]
```

| Option | Effect |
|---|---|
| `--baseline <git-ref|version>` | Compare against the given git ref or published package version. Defaults to the latest published version of the package, or to the merge-base on the default branch when no published version exists |
| `--check` | Exit non-zero when the manifest version bump is smaller than the required bump (the CI gate). Without it, the tool is advisory and always exits zero on a successful comparison |
| `--json` | Emit the machine-readable report instead of rendered text (the schema carries a top-level `schema_version`, per §23.5) |
| `--rdeps <path>` *(deferred)* | Supply recorded reverse-dependency call-site traces so the overload-resolution-shift verdict (W0303) becomes exact instead of a best-effort heuristic (see *Gate semantics and completeness*). The trace input format is **not yet frozen**; until it is, only the default single-package check is available |

**Comparison is semantic, not textual.** The tool does not diff source text. For each side it loads the public API surface — definitions, exported algorithms and their `requires` / `ensures` contracts, theorem statements, notations with parse effect, clusters, registrations, modes, attributes, and export visibility (architecture §18) — and **pairs elements by stable identity** (fully-qualified name / normalized origin id), then compares the **semantic fingerprint** of each paired element as its value. This identity-versus-value split is what lets it report a rename as a removed identity plus an added identity (MAJOR) while reporting a same-identity fingerprint difference as a signature/statement change (also MAJOR, but not a remove-plus-add). Because the compared value is a canonical fingerprint, proof-body-only and comment-only edits are invisible to the comparison. The git or version diff is used only to locate the baseline; a purely textual diff is insufficient and the tool does not rely on one.

**Overload-resolution shifts are detected by a best-effort local heuristic.** The §23.2 note classifies a registration as MAJOR when it changes overload resolution (§23.8.3 `PickMostSpecific`) for an expression a *dependent* already type-checked. The tool cannot see dependents — its only inputs are the baseline and candidate of *this* package — so it does not attempt to confirm that some downstream call site is affected. Instead it applies a best-effort heuristic over the package's own surface, flagging as a potential shift (W0303, MAJOR):

- any added or changed registration that is strictly more specific than, or incomparable-minimal with, an existing registration in the same resolution domain; and
- any added or changed conditional cluster, attribute, or mode that alters the **attribute closure** (§19.2.1). A conditional cluster registers no new candidate, yet through the closure it feeds functional-cluster guard firing (§23.7.3) and subsumption-DAG edge preconditions (§23.8.2), so it can change the viability, specificity, or ambiguity of *existing* candidates in other resolution domains.

Because the precise outcome depends on which expressions dependents actually wrote, this single-package heuristic **both over-reports (false positives) and may miss a shift that manifests only downstream (false negatives)** — it is therefore neither sound nor complete: it is intended to surface the local causes of a shift, not to certify their presence or absence. A reverse-dependency corpus or recorded downstream call-site traces, when available, both suppress false positives and let the tool confirm an actual shift; they are not part of the default input.

**Behavior.**

1. Resolve the baseline and the candidate, building (or loading cached) fingerprints for each side's public API surface and pairing elements by identity.
2. Classify every added, removed, renamed, or changed surface element under §23.2 — applying the best-effort registration/cluster heuristic above — and take the **maximum** required bump across all deltas. An edition increase contributes a MAJOR to this maximum and is additionally reported as W0305.
3. Read the declared version in `mizar.pkg` for each side and compute whether the declared increment meets the required bump.
4. Report each contributing change with its element, classification, and the diagnostic code of §22.6.6, plus the overall required bump and the smallest acceptable next version.

When the declared bump is sufficient, the tool reports success (and, under `--check`, exits zero). When it is insufficient, it emits **W0304** (`compat.version_bump_insufficient`) naming the required bump and the smallest acceptable version; under `--check` this is a non-zero exit suitable for a release gate.

**Scope: MAJOR and MINOR only.** Because the verdict is derived from the public API surface, the tool determines only whether a **MAJOR or MINOR** bump is required. It cannot distinguish a PATCH-worthy change (a proof-body or documentation fix) from no change at all, since both are invisible to the API fingerprints. PATCH bumps are therefore not gated here; ensuring that a re-publish carries *some* new version — duplicate-version rejection — is enforced by the registry and the `mizar publish` step, not by `semver-check`. When the public API surface is unchanged, `semver-check` reports "no MAJOR/MINOR bump required" and exits zero under `--check` regardless of the declared PATCH increment.

**Gate semantics and completeness.** The two change classes carry different guarantees, and `--check` must not be read as a uniform proof of compatibility:

- **Structural changes** — added, removed, renamed, or signature/statement/contract-changed elements, export visibility, and edition increases — are **detection-complete** from the baseline and candidate alone. For these classes the reported bump is the exact minimum, and a passing `--check` certifies that the declared bump covers them.
- **Overload-resolution shifts** (W0303) are **not** decided completely on the default single-package input. The local heuristic above over-reports (false positives), and — because the precise outcome depends on expressions only dependents wrote — it may also miss a shift that manifests only downstream (false negatives). A passing `--check` therefore means "no overload-resolution break was *detected* from this package's surface," not "none exists." A complete verdict for this class requires **reverse-dependency traces** (recorded downstream call sites), supplied via the deferred `--rdeps` input above; when those are supplied, the W0303 verdict becomes exact and the gate guarantee extends to overload-resolution shifts. Until that input format is frozen, the gate covers only the structural classes.

In short: `--check` success is an exact compatibility guarantee for structural changes and only a best-effort signal for overload-resolution shifts unless reverse-dependency traces are provided.

> [!NOTE]
> `mizar semver-check` is a linter, not an authority on what *should* change: it reports the bump a change **requires**, never proposing to weaken an API to avoid a MAJOR bump. It pairs naturally with the AI-agent edit classes of the architecture's agent-interface design ([21.ai_agent_interface.md](../../design/architecture/en/21.ai_agent_interface.md)): a change this tool classifies as MAJOR overlaps with the Red edit class, since both mark changes that alter what dependents observe.
