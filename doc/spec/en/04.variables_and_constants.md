## 4.1 Variables and Logic

In Mizar, variables serve three distinct roles depending on their context:

1. **Free Variables**: Introduced by `let`. They act as fixed arbitrary constants within a proof block (Skolem constants).
2. **Bound Variables**: Introduced by quantifiers (`for`, `ex`). Their scope is limited to the formula or term.
3. **Local Constants**: Introduced by `set`, named `reconsider`, named `take`, `deffunc`, or `defpred`. They are names bound to specific values, terms, or local schemas (abbreviations).

**Soft Type System**:
Mizar variables are typed. Logically, a declaration `let x be T` corresponds to treating `x` as an untyped variable in FOL, constrained by a unary predicate `T(x)`.

## 4.2 Variable Declarations (`let`)

The `let` statement introduces a **Free Variable** into the current environment.

**Syntax (EBNF)**:

```ebnf
let_decl        ::= "let" qualified_vars [ "such" conditions ] [ "by" references ] ";" ;
qualified_vars  ::= explicit_qualified_vars [ "," implicit_qualified_vars ]
                  | implicit_qualified_vars ;
explicit_qualified_vars ::= qualified_segment { "," qualified_segment } ;
qualified_segment ::= identifier_list ( "being" | "be" ) type_expression ;
implicit_qualified_vars ::= identifier_list ;
identifier_list ::= identifier { "," identifier } ;
conditions      ::= "that" proposition { "and" proposition } ;
proposition     ::= [ label_identifier ":" ] formula ;
```

* **Semantics**: Introduces new arbitrary elements.
* **Type Assumption**: If `be T` is specified, the logical context is augmented with the assumption `x is T`.
* **Such That Clause**: The optional `such that` clause adds one or more assumptions (hypotheses) for the declared variables. Each condition may have its own label for later reference.
* **Constraint Discharge**: A trailing `by references` clause may discharge well-formedness constraints introduced by constrained modes at the use site. It is not a general justification for adding assumptions. For the statement-level rule, see [Chapter 15 (Statements), §15.2.1](./15.statements.md#1521-generalization-statement-let).

> [!NOTE]
> For the full statement-level syntax of `let` (generalization), see [Chapter 15 (Statements), §15.2.1](./15.statements.md#1521-generalization-statement-let).

**Examples**:

```mizar
let x be Real;                      :: x is an arbitrary Real
let n, m be Nat;                    :: n and m are Nats
let G be Group;
let a be positive Real;             :: Attributes can be included
let B be Basis[V] by FinDim;        :: FinDim discharges the Basis[V] constraint
let x be Real such that x > 0;      :: x is Real and assumed to be > 0
let p be Prime such that p > 100;   :: p is Prime and assumed > 100
```

## 4.3 Reserved Variables (`reserve`)

The `reserve` directive defines a **default type** for specific identifiers. It is **not** a variable declaration itself but a macro-like hint for the parser.

`reserve` declarations are permitted only as top-level module declarations. They are not permitted inside `definition`, `registration`, `proof`, `now`, `algorithm`, or other nested blocks. A block-local `reserve`-shaped statement is a syntax error; block-local variables must be introduced with `let`, quantified binders, or the local-constant forms defined below.

**Syntax (EBNF)**:

```ebnf
reserve_decl    ::= "reserve" reserve_segment ";" ;
reserve_segment ::= identifier_list "for" type_expression ;
```

**Semantics**:
If `x` is reserved for `Real`:

1. `let x;` is expanded to `let x be Real;`.
2. `for x holds P(x)` is expanded to `for x being Real holds P(x)`.
3. If a reserved identifier occurs free in a theorem declaration (`theorem`,
   `lemma`, any future theorem-role alias, including theorem items inside a
   `definition` or `claim` block) or in a scheme declaration, and no local
   binding shadows it, that theorem or scheme proposition is implicitly
   universally closed over the identifier with the reserved type.
4. Explicit types override the reservation: `let x be Nat;` is valid (shadows the reservation).

Implicit universal closure is not applied to proof-local propositions such as
compact statements, `assume`, `case`, or `suppose` items. A free reserved
identifier in those contexts must already be introduced by an enclosing binder
or local declaration; otherwise it is an unbound-variable error.

For `claim` blocks, this reserved-identifier closure is applied inside the
claim theorem before the theorem is interpreted as ranging over all valid
executions of the named algorithm.

When several reserved identifiers are implicitly closed, the quantifier order is
the dependency order of their reserved types: if the reserved type of `X`
mentions reserved identifier `n`, then `n` is quantified before `X`. Independent
reserved identifiers are ordered by first source occurrence in the proposition.
A cyclic dependency among reserved types is a compile-time error.

**Example**:

```mizar
reserve i, j for Nat;
reserve x for Real;

let i;      :: Equivalent to: let i be Nat;
let x;      :: Equivalent to: let x be Real;
let i be Integer; :: Valid override.

```

## 4.4 Local Constants (`set`, `reconsider`, `take`, `deffunc`, `defpred`)

Local constants assign a name to a specific term.

### 4.4.1 Set Statement (`set`)

Assigns a name to a term expression. This is purely a syntactic abbreviation (macro) or an equality axiom.

**Syntax (EBNF)**:

```ebnf
set_decl      ::= "set" equating_list ";" ;
equating_list ::= equating { "," equating } ;
equating      ::= identifier "=" term_expression ;

```

**Example**:

```mizar
set zero = 0;
set X = {1, 2, 3};
set f = id REAL;
set a = f.x, b = f.y;

```

* **Logic**: Introduces each identifier and adds the corresponding axiom `identifier = term_expression`.

### 4.4.2 Reconsider Statement (`reconsider`)

Defines a new constant from an existing term, but with a **new type**. This generates a proof obligation (Correctness Condition).

**Syntax (EBNF)**:

```ebnf
reconsider_decl ::= "reconsider" type_change_list "as" type_expression
                    [ simple_justification ] ";" ;
type_change_list ::= reconsider_item { "," reconsider_item } ;
reconsider_item  ::= identifier
                   | identifier "=" term_expression ;
```

There are two forms:

1. **New Constant Form**: `reconsider P = R as T;` — Creates a new constant `P` equal to `R` with type `T`.
2. **Type Cast Form**: `reconsider x as T;` — Casts an existing variable `x` to type `T` without creating a new name.

**Examples**:

```mizar
set R = { x where x is Real : x > 0 };
reconsider P = R as non empty set;  :: New constant P, must prove R is non empty

let n be Nat;
reconsider n as Integer;            :: Type cast: n is now treated as Integer
reconsider x, y = f.x as Element of A;
```

* **Semantics**: Both forms generate a proof obligation (Correctness Condition) that the term has the specified type.

### 4.4.3 Named Witness and Local Definition Forms

The following forms introduce local identifier bindings. They may affect
identifier-shaped token disambiguation in their lexical scope, but they do not
remove imported user-symbol candidates from the lexer dictionary:

```ebnf
take_decl       ::= "take" example_list ";" ;
example_list    ::= example { "," example } ;
example         ::= term_expression | identifier "=" term_expression ;
deffunc_decl    ::= "deffunc" identifier "(" [ typed_params ] ")"
                    "->" type_expression "equals" term_expression ";" ;
defpred_decl    ::= "defpred" identifier "(" [ typed_params ] ")"
                    "means" formula ";" ;
typed_params    ::= typed_param { "," typed_param } ;
typed_param     ::= identifier ( "being" | "be" ) type_expression ;
```

The unnamed witness form `take term_expression;` does not introduce a local
name. Similarly, `reconsider identifier as type_expression;` changes the type
view of an existing local name without introducing a new one.

## 4.5 Quantified Variables

Variables bound by universal (`for`) or existential (`ex`) quantifiers.

**Syntax (EBNF)**:

```ebnf
quantified_segment ::= identifier_list [ ( "being" | "be" ) type_expression ] ;

```

* **Note**: If `being T` is omitted, the `reserve` type is used. If no reservation exists, it is a syntax error (unless context allows untyped, e.g., Set Theory).

**Examples**:

```mizar
for x being Real holds x^2 >= 0;
ex n being Nat st n > 0;
```

## 4.6 Scoping and Shadowing

Mizar employs **Lexical Scoping** with block structures.

### 4.6.1 Scoping Levels

1. **Module Level**: top-level `reserve` directives apply to the whole file after the directive point (unless shadowed). They cannot be declared inside nested blocks.
2. **Theorem/Definition Block**: `let` variables and local constants are valid until the `end` of the block (or `thesis` completion).
3. **Local Block (`now ... end`, proof `case` / `suppose`, and algorithm `do`)**: Variables declared inside are local to the block.
4. **Statement Level**: Quantifier binders (`for`, `ex`) and `given` binders are limited to the statement/formula range that introduces them.

### 4.6.2 Shadowing Rules

* **Allowed**: A variable in an inner scope can have the same name as one in an outer scope. The inner declaration shadows the outer one.
* **Forbidden**: Redeclaring the same variable name within the **same** scope is a syntax error.

```mizar
let x be Real;
now
  let x be Nat;  :: Valid shadowing. Inside this block, x is Nat.
end;
:: Here x is Real again.

let x be Integer; :: Error: Redeclaration in same scope.

```

## 4.7 Logic Encoding Details

This section describes how variables map to ATP inputs (e.g., TPTP format).

### 4.7.1 Type Guards (Relativization)

Since First-Order Logic is typically untyped, Mizar types are encoded as **Type Guard Predicates**.

* **Source**: `let x be Element of G;`
* **Encoding**:
* Variable `x` is treated as a generic variable.
* A hypothesis `is_Element_of(x, G)` is added to the Antecedent.



### 4.7.2 Attributes as Predicates

Adjectives (Attributes) are flattened into conjunctions.

* **Source**: `let x be positive Real;`
* **Encoding**:
  * Hypothesis: `x is Real ∧ x is positive`

### 4.7.3 Constants and Skolemization

* **Global Constants** (`func` with 0 args): Encoded as function symbols with arity 0.
* **Local Constants** (`set x = t`): Encoded as an equality axiom `x = t` in the local scope.

## 4.8 Complete Syntax (EBNF)

Summary of variable-related syntax.

```ebnf
variable_directive ::= reserve_decl
                     | let_decl
                     | set_decl
                     | reconsider_decl ;

(* Reservation *)
reserve_decl       ::= "reserve" reserve_segment ";" ;
reserve_segment    ::= identifier_list "for" type_expression ;

(* Free Variable Declaration *)
let_decl           ::= "let" qualified_vars [ "such" conditions ] [ "by" references ] ";" ;
qualified_vars     ::= explicit_qualified_vars [ "," implicit_qualified_vars ]
                     | implicit_qualified_vars ;
explicit_qualified_vars ::= qualified_segment { "," qualified_segment } ;
qualified_segment  ::= identifier_list ( "being" | "be" ) type_expression ;
implicit_qualified_vars ::= identifier_list ;

(* Local Constant Declaration *)
set_decl           ::= "set" equating_list ";" ;
equating_list      ::= equating { "," equating } ;
equating           ::= identifier "=" term_expression ;
reconsider_decl    ::= "reconsider" type_change_list "as" type_expression
                       [ simple_justification ] ";" ;
type_change_list   ::= reconsider_item { "," reconsider_item } ;
reconsider_item    ::= identifier
                     | identifier "=" term_expression ;

(* Quantifier Binding (used in formulas) *)
quantified_segment ::= identifier_list [ ( "being" | "be" ) type_expression ] ;

identifier_list    ::= identifier { "," identifier } ;
```
