## 5.1 Concept and Purpose

Structures are radix-types used to define composite types with named fields and properties. They serve three purposes:

1. **Data encapsulation**: grouping logically related components
2. **Type formation**: creating new types via structural definitions
3. **Mathematical modeling**: representing algebraic and analytical structures

## 5.2 Syntax

**Syntax (EBNF)**:

```ebnf
struct_def    ::= "struct" struct_def_name [ type_params ] "where"
                  struct_member { struct_member } "end" ";" ;
struct_member ::= field_decl | property_decl ;
field_decl    ::= "field" identifier "->" type_expression
                  [ ":=" term_expression ] ";" ;
property_decl ::= "property" identifier "->" type_expression ";" ;
type_params   ::= ( "of" | "over" ) type_parameter_list
                | "[" type_parameter_list "]" ;
type_parameter_list ::= identifier { "," identifier } ;
struct_def_name ::= constructor_name ;
```

* `field`: represents intrinsic components (stored data)
* `property`: represents uniquely determined derived values; their defining
  condition and value source are established externally via property
  implementations (see Chapter 7). A property is not stored data and is not a
  default-constructor argument.
* `struct_def_name` may be an ordinary identifier or a readable hyphenated
  constructor name such as `1-sorted`, `2-sorted`, `R-module`, or
  `C-star-algebra`. Operator-like punctuation is not admitted in structure
  names. Field names, property names, variables, labels, and selectors remain
  ordinary identifiers.
* The bracket form `Name[T]` is the canonical template parameter spelling; `of` and `over` forms are accepted shorthands when the declaration provides them.

**Example**:

```mizar
definition
  struct UnitalMagmaStr where
    field carrier -> set;
    field binop -> BinOp of carrier;
    property unit -> Element of carrier;
  end;
end;
```

## 5.3 Inheritance

Structures may inherit from others via `inherit`. When inheriting, fields and properties can be specialized (narrowed) to more specific types.

### 5.3.1 Syntax (EBNF)

```ebnf
inherit_def     ::= "inherit" inherit_child "extends" parent_type
                    [ "where" inherit_member { inherit_member }
                      [ coherence_block ]
                      "end" ] ";" ;

inherit_child   ::= struct_name [ type_args ] ;
parent_type     ::= struct_name [ type_args ] | "set" ;

inherit_member  ::= field_redef | property_redef ;

(* name may differ from parent (renaming); type is optional if unchanged from parent *)
field_redef     ::= "field" identifier [ "->" type_expression ] "from" (identifier | "it") ";" ;
property_redef  ::= "property" identifier [ "->" type_expression ] "from" identifier ";" ;

coherence_block ::= "coherence" justification ";" ;
```

* **Shorthand form** (`inherit X extends Y;`): all fields and properties of `Y` are automatically mapped by name with identical types. No `where` block is needed.
* **Explicit form** (`inherit X extends Y where ... end;`): required when any field or property is **renamed** (child name differs from parent name), **type-specialized** (narrowed), or a **coherence** proof is needed.
* All base fields/properties must be covered in either form.

> [!NOTE]
> Each `inherit` statement specifies exactly **one parent**. For multiple inheritance, use multiple `inherit` definitions. This avoids field/property name conflicts within a single statement.

**Example** — renaming:

```mizar
definition
  inherit Derived extends Base where
    field derived_field from base_field;      :: renamed
    property derived_prop from base_prop;     :: renamed
  end;
end;
```

**Set inheritance** is allowed:

```mizar
definition
  inherit OneSorted extends set where
    field carrier from it;
  end;
end;
```

### 5.3.2 Coherence (Type Specialization)

When a **field or property** is inherited with a narrower type, a `coherence` proof is required to verify that the value actually belongs to the new type.

**Example**:
`PolyStr` defines a property `degree` returning `Nat`. `NonConstPolyStr` specializes it to `non zero Nat`, requiring a coherence proof:

```mizar
definition
  struct PolyStr where
    field coeffs -> FinSequence of Int;
    property degree -> Nat;
  end;
end;

definition
  struct NonConstPolyStr where
    property degree -> non zero Nat;
  end;

  inherit NonConstPolyStr extends PolyStr where
    field coeffs from coeffs;

    :: Property narrowing: proof required
    property degree -> non zero Nat from degree;
    coherence
    proof
      :: non zero Nat ⊆ Nat (upcast is valid)
      thus for x be non zero Nat holds x is Nat;
    end;
  end;
end;
```

## 5.4 Multiple and Diamond Inheritance

Multiple inheritance is allowed if structure names are distinct. Diamond inheritance is supported: when multiple parents share a field of the **same name and type**, the verifier automatically confirms consistency by tracing the `from` chains — no explicit proof is required. When the inherited types differ, a `coherence` proof is required to verify the type inclusion (to assure safe upcast).

**Example** — same type, proof not required:

```mizar
definition
  struct ZeroStr where
    field carrier -> set;
    property zero -> Element of carrier;
  end;

  struct OneStr where
    field carrier -> set;
    property one -> Element of carrier;
  end;

  struct ZeroOneStr where
    field carrier -> set;
    property zero -> Element of carrier;
    property one -> Element of carrier;
  end;

  :: shorthand: all fields/properties map by name with identical types
  inherit ZeroOneStr extends ZeroStr;
  inherit ZeroOneStr extends OneStr;
end;
```

**Example** — different types, `coherence` required:

A ring combines an additive structure (`AddStr`, `carrier -> non empty set`; the zero element guarantees non-emptiness) and a multiplicative structure (`MulStr`, `carrier -> set`). When `RingStr` inherits from both, the carrier types differ and a coherence proof is required:

```mizar
definition
  struct AddStr where
    field carrier -> non empty set;
    field add -> BinOp of carrier;
    property zero -> Element of carrier;
  end;

  struct MulStr where
    field carrier -> set;
    field mul -> BinOp of carrier;
  end;

  struct RingStr where
    field carrier -> non empty set;
    field add -> BinOp of carrier;
    field mul -> BinOp of carrier;
    property zero -> Element of carrier;
  end;

  :: shorthand: all fields/properties (carrier, add, zero) map by name
  inherit RingStr extends AddStr;

  :: RingStr.carrier : non empty set, MulStr.carrier : set
  :: non empty set ⊆ set: coherence required
  inherit RingStr extends MulStr where
    field carrier from carrier;
    field mul from mul;
    coherence
    proof
      :: non empty set is a subtype of set (upcast is valid)
      thus for x being non empty set holds x is set;
    end;
  end;
end;
```

## 5.5 Constructors

Structures can be instantiated using the default constructor or converting constructors.

### 5.5.1 Default Constructor

**Syntax (EBNF)**:

```ebnf
constructor_call ::= struct_name [ type_args ] "(" [ field_arg { "," field_arg } ] ")" ;
field_arg        ::= identifier ":" term_expression ;
```

Field arguments may be given in any order, but every argument name must be a
`field` declared by the structure after inheritance and defaults are
accounted for. A declared `property` is a selector name, not a constructor
argument; its value is supplied only by the applicable property implementation
of [Chapter 7, §7.4.1](./07.modes.md#741-property-implementation).

```mizar
set x = UnitalMagmaStr(carrier: A, binop: op);
:: rejected: unit is a property, not a field argument
set bad = UnitalMagmaStr(carrier: A, binop: op, unit: e);
```

### 5.5.2 Converting Constructors

Allow implicit type coercion between compatible struct types:

```mizar
let G be Group;
reconsider x = G as OneStr;  :: view G as OneStr (carrier and one come from G)
```

## 5.6 Dependent Structures

Structures can be parameterized using the canonical bracket form, or by the
`over` and `of` shorthands when those synonyms are declared.

**Example**:

```mizar
definition
  let R be Ring;
  struct Polynomial[R] where
    field coeffs -> FinSequence of R.carrier;
    property degree -> Nat;
  end;
end;
```

* Multiple parameters: `struct Name[T, U]`
* Usage: `let p be Polynomial[REAL];`
* Shorthand usage when declared: `let p be Polynomial over REAL;`

## 5.7 Field Access (Selectors)

Fields and properties are accessed using dot notation.

**Syntax (EBNF)**:

```ebnf
field_access ::= term_expression "." identifier [ "(" [ term_list ] ")" ] ;
term_list    ::= term_expression { "," term_expression } ;
```

**Examples**:

```mizar
let G be Group;
set c = G.carrier;        :: Access field
set u = G.unit;           :: Access property
set f = G.binop;          :: Access function-valued field
set xy = G.binop(x, y);   :: Apply function-valued field

:: Chained access (nested structures)
let S be Module over R;
set base_carrier = S.ring.carrier;  :: Access parent structure's field
```

## 5.8 Logic Encoding Details

This section describes how structures map to First-Order Logic. The encoding relies on flattening structure components into global predicates and functions.

### 5.8.1 Structures as Predicates

Each structure type `S` is encoded as a unary predicate `is_S(x)`.

| Mizar Source | FOL Encoding |
| --- | --- |
| `x is UnitalMagmaStr` | `is_UnitalMagmaStr(x)` |

### 5.8.2 Fields as Functions and Type Guards

Each `field` is encoded as a global function symbol. Additionally, a **Type Guard Axiom** ensures that the selector returns a value of the correct type when applied to a valid structure.

* **Symbol**: `carrier(x)`
* **Axiom**:
```
forall x (is_UnitalMagmaStr(x) implies is_set(carrier(x)))
```

> [!NOTE]
> Properties are also encoded as function symbols with the same Type Guard pattern.

### 5.8.3 Inheritance as Implication and Reduct Functions

The encoding of an `inherit` edge depends on its field mapping and on how many
inherit paths lead from the derived structure to the ancestor.

**Identity-mapped, unique-path edges** — when every field/property maps to a
parent member of the **same name** and the ancestor is reachable by exactly
one path, the edge is encoded as a universal implication (subsumption); the
selector equalities are trivial because the selectors coincide:

```
inherit Derived extends Base        (all members map by name; unique path)
⇔  forall x (is_Derived(x) implies is_Base(x))
```

**Renamed or multi-path edges** — when the edge renames any member
(`field d_field from b_field` with `d_field ≠ b_field`), or the ancestor is
reachable from the derived structure via more than one inherit path, the edge
is encoded with an explicit **reduct function** `view_{D→B}` instead of
subsumption:

```
inherit Derived extends Base where field d_field from b_field
⇔  forall x (is_Derived(x) implies is_Base(view_D_B(x)))
   forall x (is_Derived(x) implies b_field(view_D_B(x)) = d_field(x))
```

No `is_Derived(x) → is_Base(x)` implication is emitted for such edges. Chained
upcasts compose the per-edge reduct functions; distinct paths to a shared
ancestor produce distinct reduct terms.

> [!IMPORTANT]
> Emitting subsumption plus global selector equalities for renamed edges is
> **unsound**: with a diamond such as Ring → AddGroup → Magma
> (`field binop from add`) and Ring → MulMonoid → Magma
> (`field binop from mul`), the equalities would collapse into
> `add(x) = mul(x)` for every Ring. The reduct encoding keeps the two Magma
> views of a Ring distinct terms (`view_add(R)` vs `view_mul(R)`), which is
> also what makes attribute facts view-specific (see
> [Chapter 13, §13.8.7](./13.term_expression.md#1387-qua-operator-encoding)
> and [Chapter 18, §18.10.2](./18.templates.md#18102-type-parameter-encoding)).

### 5.8.4 Constructor Axioms

The default constructor is encoded as an aggregator function generating two
types of axioms: **Projection** (field values only) and **Typing** (result
type).

Given `set x = UnitalMagmaStr(carrier: A, binop: op)` encoded as term
`Agg_UnitalMagmaStr(A, op)`:

> Arguments of the aggregator follow the field declaration order in the struct definition.

1. **Projection Axioms**:
```
carrier(Agg_UnitalMagmaStr(A, op)) = A
binop(Agg_UnitalMagmaStr(A, op)) = op
```

2. **Typing Axiom**:
```
is_UnitalMagmaStr(Agg_UnitalMagmaStr(A, op))
exact_UnitalMagmaStr(Agg_UnitalMagmaStr(A, op))
```

No projection axiom is generated for a `property` member. Property selectors
are function symbols for type-guarding and use-site selection, but their
values come only from property implementations described in
[Chapter 7, §7.4.1](./07.modes.md#741-property-implementation).

The auxiliary predicate `exact_S` holds precisely for values built by `S`'s
aggregator (including reduct values produced by the `view` functions of
§5.8.3 targeting `S`), with `forall x (exact_S(x) implies is_S(x))`.

### 5.8.5 Extensionality

To capture the "Many-Sorted Set" nature of structures, an extensionality axiom dictates that identity is determined solely by fields. The axiom is stated over **exact** instances only.

For a structure `S` with fields `f1`, ..., `fn`:

```
forall x, y (
  (exact_S(x) & exact_S(y))
  implies
  ((f1(x) = f1(y) & ... & fn(x) = fn(y)) iff x = y)
)
```

> [!IMPORTANT]
> Stating extensionality over `is_S` at large is **unsound** in the presence
> of inheritance: a descendant satisfying `is_S` by subsumption carries more
> members than `S`, so two descendants agreeing on `S`'s fields but differing
> elsewhere (e.g. two `ZeroOneStr` values sharing `carrier` and `zero` but
> with different `one`) would be forced equal. The `exact_S` guard restricts
> the axiom to genuine `S`-aggregates, including the reduct views of §5.8.3,
> for which field-determined identity is the intended semantics.

## 5.9 Complete Syntax (EBNF)

Summary of structure-related syntax.

```ebnf
(* Structure Definition *)
struct_def     ::= "struct" struct_def_name [ type_params ] "where"
                   struct_member { struct_member } "end" ";" ;
struct_member  ::= field_decl | property_decl ;
field_decl     ::= "field" identifier "->" type_expression
                   [ ":=" term_expression ] ";" ;
property_decl  ::= "property" identifier "->" type_expression ";" ;
type_params    ::= ( "of" | "over" ) type_parameter_list
                 | "[" type_parameter_list "]" ;
type_parameter_list ::= identifier { "," identifier } ;

(* Inheritance *)
inherit_def    ::= "inherit" inherit_child "extends" parent_type
                   [ "where" inherit_member { inherit_member }
                     [ coherence_block ]
                     "end" ] ";" ;
inherit_child  ::= struct_name [ type_args ] ;
parent_type    ::= struct_name [ type_args ] | "set" ;

inherit_member ::= field_redef | property_redef ;
field_redef    ::= "field" identifier [ "->" type_expression ] "from" (identifier | "it") ";" ;
property_redef ::= "property" identifier [ "->" type_expression ] "from" identifier ";" ;

(* Coherence for Type Specialization *)
coherence_block ::= "coherence" justification ";" ;

(* Constructor Call *)
constructor_call ::= struct_name [ type_args ] "(" [ field_arg { "," field_arg } ] ")" ;
field_arg        ::= identifier ":" term_expression ;

(* Field Access *)
field_access   ::= term_expression "." identifier [ "(" [ term_list ] ")" ] ;
term_list      ::= term_expression { "," term_expression } ;

(* Common *)
struct_def_name ::= constructor_name ;
struct_name     ::= qualified_constructor_name ;
```
