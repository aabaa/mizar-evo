## 19.1 Overview

Mizar Evolution allows multiple definitions of the same symbol name with different parameter types — a mechanism called **overloading**. When a symbol is used, the verifier selects the uniquely most applicable definition through **overload resolution**.

The overload resolution process follows four stages, modelled on C++ overload resolution but adapted to Mizar's soft type system:

1. **Candidate set construction** — collect all definitions and redefinitions with the matching symbol name that are in scope.
2. **Viable candidate filtering** — retain candidates whose parameter types are compatible with the argument types (after cluster-based attribute closure or structure inheritance upcasting).
3. **Best match selection** — apply the allowed tie-breakers and select the unique maximal ordinary definition root.
4. **Ambiguity or error** — if no viable candidate exists, or if the maximal root is not unique, report an error.

**Key restrictions:**

| Rule | Rationale |
|---|---|
| Return type does not participate in overload resolution | Allowing return-type overloading creates unresolvable interactions with implicit conversions |
| Ordinary definitions with identical argument-type signatures are a **definition conflict** error, not an overload, regardless of return type | Same-argument ordinary declarations do not become distinct overload alternatives; if the return types differ, the conflict is also observable through the result contract. Template-derived candidates are compared after expansion by the rules of §19.6.1 |
| Overloading must resolve to a single definition family (§19.4.1) | Unrelated definitions may denote different functions, predicates, or attribute meanings; matching return types is not enough to prove they mean the same thing |
| A `redefine` sharpens an existing definition; it is not an unrelated competing overload | Its `coherence` proof guarantees that it agrees with the original wherever the two overlap, so several redefinitions of one definition are combined rather than treated as a conflict (§19.5.1) |

## 19.2 Implicit Type Conversions

### 19.2.1 Cluster-Based Attribute Closure

The **attribute closure** of a type `T` is the maximal attribute set derivable from T's declared attributes by applying all active conditional cluster registrations to a fixpoint.

Given the active cluster registrations, the closure operator `cl(T)` is defined as:

```text
cl₀(T)  = attributes explicitly declared on T
cl_{i+1}(T) = cl_i(T) ∪ { B | ∃ cluster A → B for T, A ⊆ cl_i(T) }
cl(T)   = fixpoint of the above sequence
```

Since the attribute set is finite and each step is monotone, the fixpoint always exists and is unique — independent of the order in which clusters are applied. This is the **confluence property** of the cluster system.

```mizar
:: Given registrations:
::   cluster finite -> countable for set;
::   cluster finite -> compact for TopSpace;
::   cluster countable compact -> second_countable for TopSpace;

let S be finite TopSpace;
:: cl(S) = { finite, countable, compact, second_countable, ... }
```

The closure is computed once per argument at each overload resolution site; the result is deterministic.

### 19.2.2 Structure Inheritance Upcasting

A structure type can be implicitly viewed as any of its **ancestor modes** through the `inherit ... extends ...` chain. This upcasting is always valid (widening) and requires no proof — the inheritance declaration itself guarantees the structural embedding.

```mizar
:: Ring inherits from Group (via AddGroup) and from Monoid (via MulMonoid)
let R be commutative Ring;

:: Implicit upcast: R is usable wherever a Group is expected (via AddGroup path),
:: but only if the inheritance path is unambiguous.
:: When two paths exist, the upcast must be made explicit with qua (see §19.3.1).
```

Upcasting participates in viable candidate filtering: an argument of type `T` is viable for a parameter of type `M` if `T`'s structure inherits from `M` via a **unique** `inherit` chain. If multiple paths exist, the argument is not implicitly viable — explicit `qua` is required.

Here "unique" means a single syntactic path through resolved `inherit`
declarations, not merely a semantically unique member embedding. Even if the
member roots and types along two diamond paths are coherent, the paths remain
distinct reduct/view paths (§5.8.3, §13.8.7) and the verifier does not choose
between them implicitly.

| Situation | Outcome |
|---|---|
| `T` inherits from `M` via exactly one syntactic path | Implicit upcast — viable without annotation |
| `T` inherits from `M` via two or more syntactic paths | Ambiguous — `qua` required to select path |
| `T` does not inherit from `M` | Not viable |

### 19.2.3 Specificity Ordering

A type `T₁` is **at least as specific as** `T₂` (written `T₁ ⊑ T₂`) when both of the following hold:

1. **Radix compatibility**: `T₁`'s radix type (mode or structure) is identical to or a submode of `T₂`'s radix type.
2. **Attribute subsumption**: `cl(T₁) ⊇ cl(T₂)` — every attribute required by `T₂` (after closure) is also present in `T₁` (after closure).

`T₁` is **strictly more specific** than `T₂` (written `T₁ ⊏ T₂`) if `T₁ ⊑ T₂` and `T₂ ⋢ T₁`.

This relation is a **preorder** on concrete normalized type expressions:
reflexive and transitive, but not antisymmetric on raw expressions. Two
syntactically distinct types whose radix and attribute closures are mutually
subsuming are equivalent for specificity (`T₁ ⊑ T₂` and `T₂ ⊑ T₁`).
Antisymmetry holds only after quotienting types by this closure-equivalence.
Two types may also be **incomparable** — neither is at least as specific as the
other.

```mizar
:: Specificity examples (with standard cluster registrations):
::   Field ⊏ commutative Ring ⊏ Ring      (strictly more specific left to right)
::   finite TopSpace ⊏ countable TopSpace  (finite implies countable by cluster)
::   countable TopSpace and compact TopSpace are incomparable
::     (neither implies the other in general)
```

**Important**: the specificity ordering is defined on **types** (radix + attributes after closure), not on the raw attribute declarations. Two argument types with different raw declarations but identical closures are considered equally specific.

## 19.3 Explicit Type Conversions

When overload resolution would be ambiguous, or when the verifier cannot infer the intended type, the programmer can supply an explicit conversion.

### 19.3.1 The `qua` Operator

The `qua` operator explicitly selects the intended type of a term for overload resolution **without requiring a proof**. It covers two complementary cases:

1. **Cluster-based widening** — selects a less-attributed ancestor type when multiple attribute-compatible overloads exist.
2. **Structure inheritance upcasting** — selects a specific ancestor mode when a structure inherits from it via multiple paths (diamond inheritance).

Both cases require no user-written proof: the verifier automatically emits FOL proof obligations and discharges them via the cluster registration axioms and the `inherit` chain. See [§13.8.7](./13.term_expression.md#1387-qua-operator-encoding) for the full FOL encoding.

```ebnf
(* unqualified_term_expression is defined in Chapter 13. *)
qua_expression ::= unqualified_term_expression "qua" type_expression { "qua" type_expression } ;
```

**Cluster-based disambiguation:**

```mizar
:: f is defined for both countable TopSpace and compact TopSpace
:: S is finite TopSpace — ambiguous (finite implies both countable and compact)
f(S qua countable TopSpace)   :: resolves to the countable overload
f(S qua compact TopSpace)     :: resolves to the compact overload
```

**Inheritance path disambiguation** (replaces the former `as` coercion):

```mizar
:: Ring inherits from Magma via both AddGroup and MulMonoid
let R be commutative Ring;
f(R qua AddMagma)    :: selects R's additive Magma structure
f(R qua MulMagma)    :: selects R's multiplicative Magma structure
```

**Chaining `qua` for multi-level paths:**

```mizar
:: Traverse two levels of the inheritance chain explicitly
f(R qua AddGroup qua Magma)   :: AddGroup view, then its Magma ancestor
```

Each step must be a valid widening along an `inherit ... extends ...` chain. Steps are left-associative. The chain names the intended syntactic path one edge at a time; if an adjacent widening still has multiple possible inherit paths, it remains ambiguous until the source selects a path explicitly.

> [!NOTE]
> For full `qua` syntax and semantics, see [Chapter 13 (Term Expressions), §13.6](./13.term_expression.md#136-type-qualification-qua). `qua` handles only proof-free widening (cluster or upcast). To narrow a type (e.g., re-bind a variable at a more specific mode), use `reconsider` with a proof (§19.3.2).

### 19.3.2 The `reconsider` Statement

The `reconsider` statement re-binds a variable at a more specific type, narrowing its type for subsequent uses in the proof context.

```mizar
let S be finite TopSpace;
reconsider S as countable TopSpace by mml.topology.finite_is_countable;
:: S now has type countable TopSpace in subsequent context
f(S)  :: unambiguously resolves to the countable overload
```

> [!NOTE]
> For full `reconsider` syntax, see [Chapter 15 (Statements), §15.5.1](./15.statements.md#1551-type-reconsideration-reconsider).

## 19.4 Overload Resolution Algorithm

### 19.4.1 Candidate Set

The **candidate set** for a symbol use `f(a₁, ..., aₙ)` consists of all ordinary definitions and `redefine` declarations of `f` with `n` parameters that are **in scope** at the point of use. Scope is determined by:

1. Definitions and redefinitions in the current module, in declaration order.
2. Definitions and redefinitions imported via `import` statements, in import order.
3. For template instantiation sites: definitions and redefinitions visible within the enclosing `definition` block.

Lexing may preserve several same-spelling candidates at a use site. The
resolver then filters those candidates by scope, declaration order, visibility,
arity, and type signature. A declaration from an inner semantic scope can make
an outer candidate non-viable for the resolved use, but the lexer does not
implement that shadowing by deleting candidates.

**Definition families and roots.** A symbol name can be introduced by several *unrelated* definitions, and each of those definitions can in turn be sharpened by one or more `redefine` declarations. These are two very different situations, and the resolver keeps them apart. An original definition together with all the `redefine` declarations that sharpen it forms one **definition family**; the original definition is the **root** of the family. Each candidate is assigned to a family as follows:

- an ordinary definition starts a new family and is its own root;
- a `redefine` declaration joins an existing family — the one named by its `coherence with` clause, or, when that clause is omitted, the unique earlier definition whose signature it sharpens;
- a `redefine` of a `redefine` stays in the same family as the original root.

Only roots — the mutually unrelated definitions — compete during overload resolution. The `redefine` declarations inside one family never compete with each other; they are combined afterwards by the rules of §19.5.

### 19.4.2 Viable Candidates

A candidate definition `f(p₁: T₁, ..., pₙ: Tₙ)` is **viable** for arguments `(a₁, ..., aₙ)` if, for each position `i`:

```text
type_of(aᵢ) ⊑ Tᵢ
```

That is, the argument's type (after cluster closure) is at least as specific as the parameter's type.

If no viable candidate exists, the verifier reports a **no matching definition** error.

### 19.4.3 Best Match Selection

Overload resolution compares **roots** only — never the individual `redefine`
declarations inside a family. For any two viable roots, root `f*` is at least
as specific as root `g` when it is at least as specific in every argument
position:

```text
parameter_type(f*) ⊑ parameter_type(g)    (for each argument position)
```

Comparison is performed position-wise: `f*` must be at least as specific as `g` in **every** argument position. A family can become viable through one of its redefinitions (which may apply at a narrower type than the root's original signature), but it still counts as a *single* competitor — its redefinitions never split it into rival roots.

After applying only the allowed tie-breakers defined by this chapter, collect
the maximal viable ordinary roots: those for which no other viable ordinary
root is strictly more specific. If this set contains exactly one root, that
root is selected. The verifier then combines the applicable redefinitions
inside that family, using the rules of §19.5.

### 19.4.4 Ambiguity Errors

**Ambiguity** occurs when at least one viable ordinary root exists but, after
the allowed tie-breakers are applied, the maximal-root set contains two or more
distinct ordinary roots. This includes both incomparable roots and distinct
roots whose parameter vectors are equivalent under mutual `⊑`. The verifier
reports an **ambiguous overload** error and requires explicit disambiguation via
`qua` or `reconsider`.

```mizar
:: Two viable candidates, incomparable:
func f(x : countable TopSpace) -> Nat;
func f(x : compact TopSpace) -> Nat;

let S be finite TopSpace;
f(S);   :: ERROR: ambiguous — both countable and compact apply,
        ::        neither is more specific than the other
f(S qua countable TopSpace);   :: OK
```

The recommended resolution for library designers is to add a definition for the more specific common subtype:

```mizar
:: Resolves the ambiguity for finite TopSpace callers
func f(x : finite TopSpace) -> Nat;
```

No confluence proof is required from library designers; the responsibility lies with the library to provide unambiguous definitions for intended use cases.

This ambiguity rule applies only to **roots** — genuinely independent definitions. It is *never* triggered by several `redefine` declarations of the **same** family, because those declarations are guaranteed to agree (see §19.5); they are combined instead. Two declarations that belong to **different** families always compete as ordinary overloads, even if they happen to return compatible types.

## 19.5 Symbol Redefinition (`redefine`)

A `redefine` block introduces a more specific definition of an existing symbol for a narrower argument type. It is a **coherent refinement** of an existing definition root, not a new ordinary overload root.

```mizar
:: Original definition
definition
  let G be Group;
  func order(G) -> Nat means ...;
end;

:: Redefinition for finite groups — more specific
definition
  let G be finite Group;
  redefine func order(G) -> Nat equals card G.carrier;
end;
```

**Resolution behaviour:**

- When the argument type is `finite Group`, the `order` root is selected and the finite-group refinement is active.
- When the argument type is `Group` (without `finite`), the original definition is selected.
- The redefined definition must have parameter types that are **strictly more specific** than the original; otherwise it is a definition conflict error.

**Coherence obligation:** A `redefine` must prove that the new definition is **compatible** with the original on the overlap domain. This is expressed as a `coherence` correctness condition:

```mizar
redefine func order(G) -> Nat equals card G.carrier;
coherence
proof
  :: Must prove: for G being finite Group holds
  ::   order(G) = card G.carrier
  ...
end;
```

> [!NOTE]
> For correctness condition syntax, see [Chapter 16 (Theorems and Proofs), §16.6](./16.theorems_and_proofs.md#166-type-system-proof-obligations).

### 19.5.1 Combining Several Applicable Redefinitions

**Why this is not an ambiguity.** Once a symbol has been sharpened by several `redefine` declarations, more than one of them can apply to the same argument at the same time. For instance, a single space may be both `countable` and `compact`, and there may be a redefinition registered for each. With unrelated overloads this would be a conflict, but here it is not: every `redefine` had to prove a **coherence** condition stating that it agrees with the family's root wherever it applies. Two redefinitions of the same root therefore agree with the root — and hence with each other — on every argument they share. There is no disagreement about *which value* (or *which truth*) the symbol denotes; at most they describe the result with differing precision. This section explains how the verifier combines such redefinitions instead of rejecting them.

**The active set.** Once overload resolution has chosen a single root (§19.4.3), the verifier collects every `redefine` of that family that applies to the call and whose `coherence` proof has been accepted. Call this set of **active redefinitions** `R`.

- If `R` is empty, the symbol behaves exactly as its root definition.
- If `R` has one element, that redefinition is in force.
- If `R` has several elements, all of them are in force, provided their results can be combined as described below.

**Functors.** Because coherence guarantees that the active redefinitions all produce the *same value*, they can differ only in the result *type* they advertise. The verifier combines those types as follows:

1. If one result type is at least as specific as all the others, that strongest type is the one callers see.
2. If the result types share the same underlying mode or structure and their attributes are mutually consistent, callers see that mode or structure carrying the union of all the attributes (closed under the active clusters of Chapter 17).
3. If the result types are built on incompatible modes or structures, or the combined attributes contradict one another, the verifier reports an **incompatible redefinition** error (`E0204`).

This is sound because each active redefinition has already proved that it equals the root on the arguments it covers. Using any single redefining body, or pooling the result-type facts of all of them, therefore describes the very same value as the root.

**Predicates and attributes.** Here coherence proves *logical equivalence* with the root, not an equation between values. Several active redefinitions of one root can therefore never disagree about whether the predicate holds or whether the attribute applies — choosing one rather than another can never change a truth value. The only thing they can clash over is bookkeeping: for example, two visible attribute redefinitions that claim the attribute belongs to different structures. Such a clash is reported as an incompatible redefinition or a cluster conflict — never as an ambiguous overload.

`assume` clauses attached to a definition or redefinition are checked only after the root and its active redefinitions are known. They take no part in the specificity ordering or in deciding whether redefinitions can be combined.

```mizar
:: Both refinements are coherent refinements of the same root.
definition
  let x be countable TopSpace;
  redefine func size(x) -> countable Cardinal equals ...;
  coherence with size_general proof ... end;
end;

definition
  let x be compact TopSpace;
  redefine func size(x) -> compact Cardinal equals ...;
  coherence with size_general proof ... end;
end;

let S be finite TopSpace;
size(S);  :: OK if both refinements are active and their result facts join.
```

This is different from ordinary overload:

```mizar
func f(x : countable TopSpace) -> Nat;
func f(x : compact TopSpace) -> Nat;

let S be finite TopSpace;
f(S);     :: ERROR: distinct ordinary roots remain ambiguous.
```

## 19.6 Advanced Resolution Mechanisms

### 19.6.1 Template Parameter Inference

When a template definition is used (see [Chapter 18, §18.2.7](./18.templates.md#1827-template-parameter-inference)), the template arguments `[T]` may be omitted if they can be uniquely inferred from the argument types.

The process has two distinct phases. **Phase A (template expansion, steps 1–3)** converts each template definition that could match into a concrete candidate. **Phase B (step 4)** selects among all concrete candidates using the standard overload resolution rules. The two phases must not be interleaved because they use different matching rules (see the restriction below).

**Steps:**

1. **Infer template parameters** — For each template definition in scope whose name matches the call site, infer omitted `func`/`pred` template parameters by the declared-type rule in [Chapter 18, §18.2.7](./18.templates.md#1827-template-parameter-inference): use the mode-unfolded declared argument types only, with no widening-lattice search, no cluster expansion, and no inferred `qua` views. If §18.2.7 reports ambiguous template instantiation, the call is rejected before Phase B rather than silently dropping the template candidate. Otherwise, check the template constraint against the inferred declared type using the mode hierarchy only. If the constraint is satisfied, the template produces a candidate; otherwise it is skipped. Multiple templates may all succeed with the same inferred `T`; step 4 selects the best ordinary root and then applies refinement joining when applicable.

2. **Instantiate the template** — Substitute the inferred `T` throughout the template definition to produce a concrete (non-template) definition. Templates whose constraints were not satisfied in step 1 contribute no candidate and are skipped; ambiguous inference has already rejected the call site.

3. **Add to the candidate set, tagged as template-derived** — Record the instantiated candidate together with a flag marking its template origin. Steps 1–3 are repeated for every template in scope before proceeding.

4. **Run standard overload resolution with non-template priority** — Collect all candidates and apply the normal `⊑` ordering. Additionally: if a non-template candidate and a template-derived candidate are otherwise tied (equally specific under `⊑`), the non-template candidate wins. This prevents a generic template from conflicting with an intentional non-template specialization.

**Why this order matters:** Steps 1–3 must finish for *all* templates before step 4 begins, because the rules differ. Step 1 deliberately suppresses cluster expansion to keep inferred types unique (see the restriction below). Step 4 applies full `⊑` ordering — including cluster closure — to check viability, and applies the non-template priority rule to resolve ties. Mixing the two phases would make inferred types non-unique and unpredictable.

If template inference yields multiple valid instantiations (ambiguous inference), the verifier reports an **ambiguous template instantiation** error, separate from overload ambiguity.

The following examples assume an explicitly declared mode hierarchy:

```mizar
inherit B extends A;
inherit C extends B;
```

**Case 1 — basic inference and explicit annotation:**

```mizar
definition
  let T be type extends A;
  let x be T;
  func f[T](x) -> T;
end;

let c be C;
f(c)      :: T = C (C ⊑ A in mode hierarchy) → OK
f[A](c)   :: explicit T = A; viability: C ⊑ A → OK
```

**Case 2 — equivalent template-derived roots are ambiguous:**

When multiple templates succeed in step 1, their declared constraint strictness
is not a tie-breaker. Step 4 compares their instantiated concrete signatures.
If those signatures are equivalent, the distinct roots remain ambiguous.

```mizar
definition
  let T be type extends A;
  let x be T;
  func f[T](x) -> T;   :: overload 1
end;

definition
  let T be type extends B;
  let x be T;
  func f[T](x) -> T;   :: overload 2
end;

let c be C;
f(c)
:: Step 1: T = C; C ⊑ A ✓ and C ⊑ B ✓ → both produce candidates
:: Step 4: both candidates have concrete parameter type C
::         → distinct equivalent roots, ambiguous

let a be A;
f(a)
:: Step 1: T = A; A ⊑ A ✓ but A ⊑ B? No → only overload 1 produces a candidate
:: Step 4: overload 1 selected unconditionally
```

**Case 3 — non-template specialization takes priority:**

```mizar
definition
  let T be type extends A;
  let x be T;
  func f[T](x) -> T;   :: generic template
end;

definition
  let x be B;
  func f(x) -> B;      :: non-template specialization
end;

let b be B;
f(b)
:: Step 1: T = B, B ⊑ A ✓ → template candidate; non-template also matches
:: Step 4: non-template wins the tie — no ambiguity error

let c be C;
f(c)
:: Step 1: T = C, C ⊑ A ✓ → template candidate; non-template: C ⊑ B ✓ → also matches
:: Step 4: template parameter C is strictly more specific than non-template parameter B
::         → template-derived candidate wins
```

**Case 4 — multiple template parameters must all agree:**

```mizar
definition
  let T be type extends A;
  let x, y be T;
  func g[T](x, y) -> T;
end;

let b be B;
let a be A;

g(b, b)      :: T = B from both → agree → OK
g(b, a)      :: T = B from x, T = A from y → mismatch → ambiguous template instantiation
g[A](b, a)   :: explicit T = A; viability: B ⊑ A ✓ and A ⊑ A ✓ → OK
```

**Important restriction — cluster expansion does not apply during template inference.**

When `[T]` is omitted, `T` is inferred from mode-unfolded declared argument
types according to §18.2.7, and the constraint is checked using the mode
hierarchy only (no cluster expansion). When `[T]` is written explicitly,
inference is skipped and the verifier checks viability under the full `⊑`
ordering (including cluster expansion).

```mizar
:: P and Q are unrelated in the mode hierarchy.
:: Given: cluster Q -> P;    (Q is also P, established by cluster registration)

mode P;
mode Q;   :: Q does not extend P in the mode hierarchy

definition
  let T be type extends P;
  let x be T;
  func h[T](x) -> T;
end;

let q be Q;

h(q)      :: ERROR — T bound to Q; Q ⊑ P? Not in mode hierarchy → inference fails
h[P](q)   :: OK — T = P explicitly; viability: Q ⊑ P via cluster → OK
```

> [!NOTE]
> Allowing cluster expansion during inference would make the inferred type non-unique: the same argument could satisfy many different constraints via chains of cluster registrations, leading to unpredictable instantiation choices. Explicit `[T]` annotation makes the intended instantiation unambiguous and keeps cluster-expansion cost predictable.

### 19.6.2 Multiple Inheritance Paths

When a structure type inherits from an ancestor mode via multiple paths (diamond inheritance), and a definition is parameterized by that ancestor, `qua` must be used to select the intended path. The verifier does not infer the path automatically and reports an **ambiguous inheritance path** error if `qua` is omitted.

Path identity is syntactic over resolved `inherit` declarations. Coherent
member joins do not collapse paths for overload resolution: two paths that
lower to different reduct/view terms are ambiguous even when they cover the
same root declarations with compatible types.

```mizar
:: Ring inherits from Magma via both AddGroup and MulMonoid
func PermProduct(M : commutative Magma) -> ...;

let R be commutative Ring;
PermProduct(R)                :: ERROR: ambiguous path to Magma
PermProduct(R qua AddMagma)   :: OK — uses additive structure
PermProduct(R qua MulMagma)   :: OK — uses multiplicative structure
```

This is distinct from ordinary overload ambiguity: here only one definition of `PermProduct` exists, but the argument's type can be interpreted via multiple inheritance paths.

### 19.6.3 Module Scope and Resolution Order

Overload resolution considers only definitions that are **in scope** at the point of use. The scope is determined by:

1. **Current module**: all definitions declared before the use site.
2. **Imported modules**: all public definitions of modules listed in `import` declarations.
3. **Resolver precedence**: a definition in the current module with the same
   name and argument-type signature is preferred over an imported definition
   for semantic resolution, even though lexing may retain both candidates.

When a module `import` introduces a name conflict with an existing definition, the conflict is resolved by specificity. If the imported definition is strictly more specific, it participates in overload resolution normally. If two definitions from different imported modules are incomparable, the conflict is reported as a **import ambiguity** error and the programmer must qualify the reference explicitly (`module_path.symbol_name`).

> [!NOTE]
> For module import syntax and visibility rules, see [Chapter 12 (Modules and Namespaces)](./12.modules_and_namespaces.md).

### 19.6.4 `of`/`over` Greedy Parsing

The `of`/`over` syntax is syntactic sugar for the `[]` bracket form (see [Chapter 18, §18.1](./18.templates.md#181-overview)). The parser uses **greedy (longest-match) argument consumption**: when scanning comma-separated arguments after `of`/`over`, the parser consumes as many arguments as the highest-arity definition of the mode accepts. This rule is deterministic and requires no disambiguation from the programmer in the common case.

**Greedy rule:** `M of A, B, ...` consumes the maximum number of arguments for which a definition of `M` exists at that arity. Arguments beyond the maximum arity remain for the enclosing expression.

```mizar
:: Suppose both a 1-parameter and 2-parameter M exist:
mode M1Def: M of X is ...;
mode M2Def: M of X, Y is ...;

M of A, B      :: M[A, B]  — greedy; 2-parameter definition exists, both args consumed
M of A         :: M[A]     — only 1 arg present; 1-parameter definition used
```

**To use a lower-arity form when more arguments follow**, limit consumption with parentheses or use the `[]` bracket form explicitly:

```mizar
(M of A), B    :: M[A] as one value, B separately — parentheses stop greedy consumption
M[A], B        :: same effect with bracket form
```

**Nested `of` expressions** follow the same greedy rule at each level, left-to-right:

```mizar
Function of Subset of X, Y
:: inner first: Subset of X, Y → Subset[X, Y]  (greedy, 2-param Subset)
:: outer:       Function of Subset[X, Y]        → Function[Subset[X, Y]]

Function of (Subset of X), Y
:: (Subset of X) is parenthesized → Subset[X]
:: Function of Subset[X], Y → Function[Subset[X], Y]  (2-param Function)
```

**Recommendation:** Use the `[]` bracket form when the intended arity must be stated unambiguously regardless of what definitions are in scope.

## 19.7 Logic Encoding Details

Overload resolution is a **compile-time** mechanism; it produces no FOL axioms of its own. Its effect is to select which definition's axiom is instantiated at each use site.

| Overload Resolution Result | FOL Effect |
|---|---|
| Unique root `f*` selected | The root's axiom fixes the meaning of the symbol at the call site |
| One or more redefinitions active | The active redefinitions' axioms and their combined result-type facts may be used; coherence keeps them consistent with the root, and so with each other, wherever they overlap |
| Ambiguity error | No FOL term generated; verification fails |

The **coherence condition** for `redefine` generates the following FOL obligation:

```fol
∀args. (more_specific_type(args) → new_def(args) = original_def(args))
```

This ensures that the two definitions agree on their common domain, preserving soundness even when the root and several refinements are axiomatised.

## 19.8 Complete Syntax (EBNF)

Overload resolution is not associated with new syntactic forms — it is a resolution procedure applied to existing uses of functors, predicates, modes, and attributes. The relevant syntax forms are defined in their respective chapters.

```ebnf
(* Explicit disambiguation forms used when overload resolution is ambiguous *)

(* qua — widens apparent type (cluster or upcast) without proof; see Chapter 13 §13.6 *)
(* Covers both attribute-based disambiguation and inheritance path selection.      *)
(* unqualified_term_expression is defined in Chapter 13.                          *)
(* Steps are left-associative: R qua AddGroup qua Magma                           *)
qua_expression ::= unqualified_term_expression "qua" type_expression { "qua" type_expression } ;

(* redefine — introduces a more specific coherent refinement; coherence is required *)
redefine_item  ::= "redefine" ( func_def | pred_def | attr_def ) ;

(* Specificity ordering (meta-notation, not surface syntax) *)
(* T₁ ⊑ T₂  iff  radix(T₁) ≤ radix(T₂)  ∧  cl(T₁) ⊇ cl(T₂) *)
(* cl(T)  =  fixpoint of conditional cluster applications on T's attributes *)
```
