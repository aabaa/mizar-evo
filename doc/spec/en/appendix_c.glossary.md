## C.1 Core Language

| Term | Meaning | See |
|---|---|---|
| Active lexicon | The set of reserved words, reserved symbols, and imported user-defined symbols available to the lexer at a source location. | Ch.2, Ch.11 |
| Annotation | A source-level `@` marker that supplies metadata, proof-search hints, display requests, or documentation information without changing logical meaning. | Ch.2, Ch.21, Ch.24 |
| Definition block | A `definition ... end` region that introduces attributes, modes, predicates, functors, registrations, algorithms, or related items. | Ch.6-Ch.10, Ch.17, Ch.20 |
| Identifier | An alphanumeric source name following the identifier grammar and not equal to a reserved word. | Ch.2 |
| Import closure | The complete set of modules and packages visible through direct and transitive imports. | Ch.11, Ch.12, Ch.23 |
| Label | A named reference attached to a theorem, definition, proof step, or other item so that later code can cite it. | Ch.10, Ch.16 |
| Module | The logical unit defined by one `.miz` source file. Its namespace is derived from the package root and source path. | Ch.12, Ch.23 |
| Namespace | A dotted path that qualifies modules and exported symbols to avoid ambiguity. | Ch.12 |
| Package | A build and distribution unit with a `mizar.pkg` manifest, source tree, dependencies, and generated artifacts. | Ch.23 |
| Reserved word | A fixed keyword of the language that cannot be used as an identifier or user symbol. | Ch.2 |
| Symbol | A name used for a predicate, functor, mode, attribute, or operator-like notation. Symbols may be identifier-shaped or punctuation-shaped. | Ch.2, Ch.11 |
| User-defined symbol | A non-reserved symbol introduced by a definition and made visible through scope or import. | Ch.2, Ch.9-Ch.11 |

## C.2 Types and Definitions

| Term | Meaning | See |
|---|---|---|
| Attribute | A unary property used to refine a type expression, such as `finite`, `empty`, or `associative`. | Ch.6 |
| Attribute chain | A sequence of attributes attached to a radix type or mode type; semantically it is interpreted conjunctively. | Ch.3, Ch.6 |
| Built-in type | A primitive type supplied by the core language, currently including `object` and `set`. | Ch.3 |
| Dependent mode | A mode whose type expression depends on one or more term parameters. | Ch.7, Ch.18 |
| Functor | A term-forming definition that maps arguments to an object-level value. | Ch.10 |
| Mode | A named type abbreviation or refinement, usually defined as an attribute chain over a radix type. | Ch.7 |
| Predicate | A formula-forming definition expressing a property or relation, such as `x in X` or `a divides b`. | Ch.9 |
| Radix type | The structural or primitive head of a type expression, before attributes are applied. | Ch.3 |
| Redefinition | A more specific replacement or refinement of an existing definition, subject to compatibility or coherence requirements. | Ch.6, Ch.9, Ch.10, Ch.19 |
| Structure | A composite type declared with fields, inherited components, and optional properties. | Ch.5 |
| Type expression | A syntactic type description formed from attributes and a radix or mode head. | Ch.3 |
| Type head | The radix type or mode type at the core of a type expression. | Ch.3 |

## C.3 Terms, Formulas, and Proofs

| Term | Meaning | See |
|---|---|---|
| Atomic formula | A formula that has not yet been combined with logical connectives; examples include predicate applications, equality, membership, type assertions, and attribute assertions. | Ch.14 |
| Citation | A reference in a proof justification, usually naming labels or imported theorem paths after `by`. | Ch.16 |
| Formula | A logical proposition built from atomic formulas, connectives, and quantifiers. | Ch.14 |
| Lemma | A theorem-like proved proposition intended mainly for reuse in later proofs. | Ch.16 |
| Proof obligation | A logical condition generated by the verifier that must be discharged for a definition, statement, algorithm, or proof step to be accepted. | Ch.16, Ch.20, Ch.22 |
| Proof step | A statement inside a proof that advances the current thesis or establishes an intermediate labelled fact. | Ch.16 |
| `qua` | A term-level type qualification operator that selects a widened type view for overload resolution, field access, or attribute checks. It is proof-free and left-associative. | Ch.13, Ch.19 |
| Selector access | Field access using dot notation, such as `p.x` or `line.end.y`. | Ch.5, Ch.13 |
| Term expression | An expression denoting an object-level value, such as a variable, functor application, structure value, set expression, or choice expression. | Ch.13 |
| Theorem | A named proposition declared with a theorem role. Ordinary theorems require discharged proof obligations; `open`, `assumed`, and `conditional` theorem items follow the status rules of Chapter 16. | Ch.16 |
| Thesis | The current goal formula that remains to be proved at a proof point. | Ch.16 |

## C.4 Inference and Resolution

| Term | Meaning | See |
|---|---|---|
| Cluster | A registration rule used by type inference to derive existence or attribute information. | Ch.17 |
| Cluster closure | The fixed point of applying visible conditional cluster rules to the known attributes of a type. | Ch.17, Ch.19, Ch.23 |
| Conditional cluster | A cluster rule that derives one or more attributes from other attributes for a type. | Ch.17 |
| Existential cluster | A registration proving that a type with certain attributes is inhabited. | Ch.17 |
| Functorial cluster | A registration describing attributes of the result of a functor application. | Ch.17 |
| Overload resolution | The process that selects one visible definition for an overloaded symbol based on argument types and specificity. | Ch.19 |
| Registration | A declaration that adds information to the verifier's inference database, especially for clusters, reductions, and type propagation. | Ch.17 |
| Refinement join | The step that, after overload resolution has chosen one original definition, combines the compatible results of the several `redefine` declarations that all sharpen it. | Ch.19, Ch.23 |
| Specificity | The ordering used to decide whether one candidate type or overload is more precise than another. | Ch.19 |
| Type inference | The verifier process that derives types and attributes from declarations, clusters, functor applications, and local context. | Ch.8, Ch.17, Ch.19 |
| Widening | Viewing a term at a less specific type, either through structure inheritance or by dropping attributes justified by cluster closure. | Ch.13, Ch.19 |

## C.5 Algorithms and Verification

| Term | Meaning | See |
|---|---|---|
| Algorithm | An executable or specification-oriented procedure with contracts, statements, and verification conditions. | Ch.20 |
| Assertion | A formula stated inside code or proof that must be verified before it may be used as a fact. | Ch.15, Ch.20 |
| Code extraction | The separate tooling step that turns verified terminating algorithms into executable target-language code. | Ch.20 |
| Contract | A precondition, postcondition, or assertion associated with an algorithm, usually written with `requires`, `ensures`, or `assert`. | Ch.20 |
| Decreasing measure | A natural-valued or tuple-valued measure that must decrease to prove termination of recursion or loops. | Ch.20 |
| Ghost variable | A specification-only binding available to verification but erased from extracted runtime code. | Ch.20 |
| Invariant | A formula that must hold at loop entry, after each iteration, and at specified control-flow exits. | Ch.20 |
| Match | An algorithmic pattern-matching construct over term structure. Matching is syntactic, not semantic. | Ch.20 |
| MVM | The Mizar Virtual Machine used to evaluate computable algorithm expressions during verification. | Ch.20 |
| Pick | The algorithmic interpretation of `the` expressions that choose an arbitrary element from a non-empty type or set. | Ch.13, Ch.20 |
| Snapshot | A specification-only record of local algorithm state at a program point. | Ch.20 |
| Terminating algorithm | An algorithm whose termination has been proved and which may be promoted for proof-level use as a functor. | Ch.20 |
| Verification condition | A proof obligation generated from contracts, loop rules, termination rules, calls, or return statements. | Ch.20 |

## C.6 Packages, Tooling, and Output

| Term | Meaning | See |
|---|---|---|
| ATP | An automated theorem prover backend used to discharge proof obligations or verification conditions. | Ch.21 |
| Build artifact | A generated file such as `*.mizir.json`, proof traces, diagnostic explanations, or cluster indexes. | Ch.23 |
| Certificate | Evidence returned by an ATP or verifier component and checked by the minimum kernel when required. | Ch.21, Ch.23 |
| Diagnostic | A structured error, warning, note, hint, or informational message emitted by the verifier or tooling. | Ch.22 |
| Documentation comment | A `:::` comment block extracted by `mizar doc` as user-facing API documentation. | Ch.24 |
| LSP | The Language Server Protocol interface used for hover, diagnostics, code actions, and editor integration. | Ch.23 |
| Minimum kernel | The small trusted checker responsible for validating proof certificates and critical logical artifacts. | Ch.21 |
| Proof trace | A recorded derivation or replay sequence used to justify an inference result, certificate, or cluster-resolution step. | Ch.21, Ch.23 |
| Statement / item annotation | A source-level `@name` or `@name(...)` annotation attached to the immediately following statement or item, such as `@proof_hint`, `@show_type`, `@eval`, `@latex`, or `@suppress`. This is distinct from documentation tags and bracket-form library annotations. | Ch.2, Ch.21 |
| Verifier | The implementation component that parses, type-checks, resolves symbols, generates obligations, dispatches provers, and accepts or rejects source files. | Ch.8, Ch.16, Ch.20-Ch.23 |
