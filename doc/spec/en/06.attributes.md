## 6.1 Definition and Purpose

Attributes are predicates (possibly parameterized) used to refine types by specifying additional logical properties. They serve the following purposes:

1. **Subtype Refinement**: Define subtypes of existing types based on logical conditions.
2. **Property Annotation**: Attach properties to types in a modular and reusable way.
3. **Type Inference**: Participate in the clustering mechanism for automatic type derivation.

## 6.2 Syntax

**Syntax (EBNF)**:

```ebnf
attr_def       ::= "attr" label ":" subject "is" attr_pattern "means" formula_definiens ";" ;
label          ::= identifier ;
subject        ::= identifier ;
attr_pattern   ::= [ param_prefix ] attribute_name ;
param_prefix   ::= parameter "-"
                 | "(" parameter_list ")" "-" ;
parameter_list ::= parameter { "," parameter } ;
parameter      ::= identifier | numeral ;
attribute_name ::= constructor_name ;
formula_definiens ::= formula
                    | formula_case { "," formula_case } [ "otherwise" formula ] ;
formula_case      ::= formula "if" formula ;
```

* `label`: Mandatory identifier for referencing this definition
* `subject`: The variable being tested (e.g., `T`, `n`, `V`)
* `param_prefix`: Optional prefix for parameterized attributes (e.g., `n-`, `(m,n)-`)
* `attribute_name`: An ordinary identifier or readable hyphenated constructor
  name. Attribute names do not admit arbitrary operator-like symbolic
  spellings.

`attr_pattern` is the only declaration form for attributes. It declares
parameter slots through `param_prefix`; there is no declaration syntax of the
form `attr_name(args)`. In ordinary type expressions and `is` assertions, the
Chapter 3 `attribute_ref` form `attribute_name(args)` may be used as a
positional application of a parameterized attribute with the same arity. The
arguments are term expressions resolved at the use site and are stored in the
attribute instance key together with the resolved attribute symbol. When every
argument is an identifier or numeral tuple accepted by `param_prefix`, the
argument-list spelling denotes the same instance as the prefix spelling. An
unparameterized attribute must not be used with an argument list. Cluster
registrations do not use `attribute_ref`; they use the restricted Chapter 17
`adjective` grammar, which admits no argument list.

**Example (Simple)**:

```mizar
definition
  let i be Integer;
  attr EvenDef: i is even means 2 divides i;
end;
```

**Example (Parameterized)**:

```mizar
definition
  let n be Nat;
  let V be VectorSpace;
  attr NDimDef: V is n-dimensional means
    ex B being Basis of V st card B = n;
end;
```

**Example (Multiple Parameters)**:

```mizar
definition
  let m, n be Nat;
  let A be Matrix;
  attr MNSizeDef: A is (m,n)-size means
    rows A = m & cols A = n;
end;
```

## 6.3 Usage

### 6.3.1 Type Qualification

Attributes refine types in variable declarations:

```mizar
let G be associative unital Magma;
let n be positive even Integer;
let V be n-dimensional VectorSpace;  :: Parameterized attribute
let A be (n,n)-size Matrix;          :: Multiple parameters
```

### 6.3.2 Property Testing

Attributes can be tested in formulas:

```mizar
assume M is commutative;
if n is odd then ...;
```

### 6.3.3 Negation and Combinations

The `non` keyword negates attributes:

```mizar
let S be non empty finite set;
```

### 6.3.4 Type-Qualified Attributes

Used to disambiguate when the same attribute name exists in multiple structures:

```mizar
R is AddLoopStr.associative;
R is MulLoopStr.commutative;
```

Alternatively, use the `qua` operator:

```mizar
(R qua AddLoopStr) is associative;
(R qua MulLoopStr) is commutative;
```

## 6.4 Clustering (Overview)

Clustering enables automatic propagation of attributes.

| Cluster Type | Purpose |
| --- | --- |
| **Existential** | Proves existence of typed values with attributes |
| **Conditional** | Declares implication between attributes |
| **Functor** | Assigns attributes to functor results |

**Example (Existential)**:

```mizar
registration
  cluster empty finite set;
  existence
  proof ... end;
end;
```

**Example (Conditional)**:

```mizar
registration
  cluster empty -> finite set;
  coherence
  proof ... end;
end;
```

> [!NOTE]
> Detailed cluster syntax and semantics are defined in **Chapter 17 (Clusters and Registrations)**.

## 6.5 Attribute Inheritance

Attributes defined for a parent structure apply to derived structures.

```mizar
definition
  inherit LoopStr extends Magma;
end;
```

A `LoopStr` inherits all attributes defined for `Magma`.

## 6.6 Disambiguation

When multiple inheritance paths define the same attribute name, use qualified syntax:

```mizar
R is AddLoopStr.associative;
R is MulLoopStr.associative;
```

Or define explicit disambiguated attributes:

```mizar
definition
  let R be DoubleLoopStr;
  attr AddAssocDef: R is add_associative means R is AddLoopStr.associative;
end;
```

## 6.7 Redefinition

To redefine an attribute for a different definition:

* Original definition
```mizar
definition
  let n be Integer;
  attr integer_even: n is even means
  ex k be Integer st n = 2*k;
end;
```

* A redefinition
```mizar
definition
  let n be Nat;
  redefine attr nat_even: n is even means ex k be Nat st n = 2*k;
  coherence with integer_even
  proof
    :: for Nat, divisibility and multiplication-based definitions coincide
    ...;
  end;
end;
```

**Requirements**:
* Logical equivalence with the original definition
* Coherence proof to justify redefinition
* (Optional) Labels to reference specific parent definitions

A redefinition sharpens the *same* attribute as the original; it is not a rival definition. So when several redefinitions of one original attribute apply at the same place, the verifier does not call this ambiguous. Each redefinition has proved, through its coherence condition, that it is logically equivalent to the original wherever it applies — so they all agree on whether the attribute holds.

The applicable redefinitions must, however, still be consistent in how they describe the attribute. In particular, they must not disagree about which structure the attribute belongs to, and the attributes they propagate through clusters must not contradict one another. Such a clash is reported as an incompatible redefinition or a cluster conflict — not as an ambiguous overload.

## 6.8 Logic Encoding Details

This section describes how attributes map to First-Order Logic.

### 6.8.1 Attributes as Predicates

Each attribute `A` applied to type `T` is encoded as a predicate.

| Mizar Source | FOL Encoding |
| --- | --- |
| `x is even` | `is_even(x)` |
| `x is positive` | `is_positive(x)` |
| `V is n-dimensional` | `is_n_dimensional(V, n)` |

### 6.8.2 Negation Encoding

The `non` keyword generates negated predicates:

| Mizar Source | FOL Encoding |
| --- | --- |
| `x is non empty` | `¬is_empty(x)` |
| `V is non n-dimensional` | `¬is_n_dimensional(V, n)` |

### 6.8.3 Attribute Chains as Conjunctions

Multiple attributes are encoded as conjunctions:

| Mizar Source | FOL Encoding |
| --- | --- |
| `x is positive even Integer` | `is_Integer(x) ∧ is_positive(x) ∧ is_even(x)` |

### 6.8.4 Clustering as Axioms

Cluster registrations generate FOL axioms:

**Conditional Cluster**:
```
cluster empty -> finite set
⇔  forall x (is_set(x) & is_empty(x) implies is_finite(x))
```

## 6.9 Complete Syntax (EBNF)

Summary of attribute-related syntax.

```ebnf
(* Attribute Definition *)
attr_def       ::= "attr" label ":" subject "is" attr_pattern "means" formula_definiens ";" ;
label          ::= identifier ;
subject        ::= identifier ;
attr_pattern   ::= [ param_prefix ] attribute_name ;
param_prefix   ::= parameter "-"
                 | "(" parameter_list ")" "-" ;
parameter_list ::= parameter { "," parameter } ;
parameter      ::= identifier | numeral ;
attribute_name ::= constructor_name ;

(* Attribute Redefinition *)
redefine_attr  ::= "redefine" "attr" label ":" subject "is" attr_pattern "means" formula_definiens ";"
                   "coherence" [ "with" label ] justification ";" ;
formula_definiens ::= formula
                    | formula_case { "," formula_case } [ "otherwise" formula ] ;
formula_case      ::= formula "if" formula ;

(* Attribute Usage in Types *)
attribute_chain ::= { [ "non" ] attribute_ref } ;
attribute_ref   ::= [ param_prefix ] [ struct_name "." ] attribute_name [ "(" argument_list ")" ];
```

The argument-list part of `attribute_ref` is the use-site application form
described in §6.2. Cluster registration adjectives use the restricted syntax
in Chapter 17.
