# crate plan: mizar-resolve

> 正本は英語です。英語版: [../en/00.crate_plan.md](../en/00.crate_plan.md)。

## 責務

所有する挙動:

- `SurfaceAst` と phase 0 の module-index 入力を消費し、`ResolvedAst` と
  `SymbolEnv` を生成する。
- module path、import、export、namespace、label、安定した symbol identity を
  解決する。
- type checking を行わずに declaration signature と依存側に見える module
  summary を収集する。
- 未解決または曖昧な参照を明示的に表現し、recoverability を保つ。
- output ordering、id、debug rendering、diagnostics を決定的に保つ。

対象外:

- syntax ownership、parser recovery、AST vocabulary の変更。
- type inference、overload winner selection、cluster firing、coercion insertion。
- proof validity、VC generation、ATP interaction、kernel acceptance、artifact emission。
- `mizar-build` と `mizar-session` が所有する package manifest parsing、
  lockfile resolution、source loading、module discovery。
- driver lifecycle、salsa/query integration、phase registry ownership。

## 開始可能性

`mizar-resolve` は [todo.md](./todo.md) の task 1〜7 から開始できる。
`mizar-build` wave A は完了済みであるため、resolver 側の module-index seam は
[module_index.md](../../mizar-build/ja/module_index.md) の build-side provider を
相手に仕様化できる。parser/syntax surface は `mizar-parser` task 45 まで最新であり、
task 46 は deferred なので resolver foundation をブロックしない。

完全な user-facing resolver diagnostics は、[architecture 03](../../architecture/ja/03.module_and_symbol_resolution.md)
と [spec chapter 22](../../../spec/ja/22.error_handling_and_diagnostics.md) に残る
diagnostic-code ownership gap が解決または明示的に deferred されるまでは開始しない。

observed behavior:

- `mizar-build` は session-local source id を割り当てずに deterministic `BuildPlan` と
  `ModuleIndex` construction を提供する。
- `mizar-session` は source identity、normalized path、snapshot identity を所有する。
- `mizar-frontend` は lexical environment と parser orchestration seam を提供する。
  semantic import validation は resolver-owned のままである。
- executable semantic import pass は R-010 の source-shaped candidate seam まで
  存在する。alias binding、relative-prefix interpretation、namespace/package
  binding、unresolved recovery、canonical graph construction は unit-test candidate 上で
  存在する。
- executable declaration pass は R-011 の source-shaped collector slice まで存在する。
  表現済み declaration-like `SurfaceAst` item、visibility wrapper、transparent annotation
  wrapper、recovered-shell state、export projection shell は semantic validation なしで
  inventory される。
- `names.md` は R-013〜R-016 の name-resolution contract として、scope、
  namespace-before-symbol lookup、visibility/shadowing、ambiguity、dot-chain
  finalization を仕様化する。executable names pass は R-013 namespace lookup slice まで
  存在する。import alias、reserved namespace root、package-name binding、
  current-package fallback、recovered namespace path、unresolved import-alias
  dependency、ambiguous alias candidate target、provider-error classification、
  earliest failing segment range が表現される。executable names pass は R-014
  symbol-name slice まで存在する: preliminary symbol projection、
  declaration-point filtering、qualified / unqualified visibility、
  current-module shadowing、overload-group placeholder collapse、enabled builtin
  fallback、deterministic `NameRefTable` outcome が selector type checking や overload
  winner selection なしで表現される。executable names pass は R-015 internal diagnostic
  slice まで存在する: deterministic `NameDiagnosticRootId` allocation、primary /
  cascade record、unresolved import-alias root、namespace/name dependent record、stable
  candidate payload、internal record ordering が public diagnostic code なしで表現される。
  executable names pass は R-016 dot-chain finalization slice まで存在する:
  `LocalTermScope`、`LocalTermBinding`、`DotChainCandidate`、`DotChainFinalizer` が
  dotted chain を checker-deferred selector または namespace-qualified symbol reference として
  分類し、selector validation は行わない。direct import/export semantic validation、label、
  full symbol extraction、signature は
  reverse engineering ではなく spec、test、design に従う。

## exit criteria

Hard gates:

- active milestone に blocking/high specification inconsistency が残っていない。
- source implementation 前に、全 public API と約束された挙動に design coverage がある。
- source behavior が `doc/spec`、既存 test、active design spec の外に残っていない。
- `.miz` test と expectation を implementation に合わせる目的だけで変更していない。
- resolver が parser、checker、proof、build、driver、artifact、session の責務を所有していない。
- diagnostics は明示的な spec/code-space decision に裏付けられているか、milestone 内の
  crate-local/internal として文書化されている。
- 残る risk は deferred、out of scope、human-owned として分類されている。

Verification commands:

```text
cargo fmt --check
cargo test -p mizar-resolve
cargo clippy -p mizar-resolve --all-targets --all-features -- -D warnings
```

boundary work では、必要な subset も実行する:

```text
cargo test -p mizar-build
cargo test -p mizar-session
cargo test -p mizar-syntax
cargo test -p mizar-frontend
cargo test -p mizar-test
```

Review expectations:

- active specification に対する review-only agent。
- implementation task を完了扱いにする前の test sufficiency review。
- 未解決 finding なしの implementation review。
- source/documentation consistency review。
- crate-wide completion には hard gate と 90/100 以上の read-only quality score が必要。
