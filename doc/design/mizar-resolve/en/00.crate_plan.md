# Crate Plan: mizar-resolve

> Canonical language: English. Japanese companion: [../ja/00.crate_plan.md](../ja/00.crate_plan.md).

## Responsibility

Owned behavior:

- consume `SurfaceAst` and phase-0 module-index input to produce `ResolvedAst`
  and `SymbolEnv`;
- resolve module paths, imports, exports, namespaces, labels, and stable symbol
  identities;
- collect declaration signatures and dependency-facing module summaries without
  type checking;
- preserve recoverability by representing unresolved and ambiguous references
  explicitly;
- keep output ordering, ids, debug renderings, and diagnostics deterministic.

Out of scope:

- syntax ownership, parser recovery, or AST vocabulary changes;
- type inference, overload winner selection, cluster firing, or coercion
  insertion;
- proof validity, VC generation, ATP interaction, kernel acceptance, and
  artifact emission;
- package manifest parsing, lockfile resolution, source loading, and module
  discovery owned by `mizar-build` and `mizar-session`;
- driver lifecycle, salsa/query integration, and phase registry ownership.

## Readiness

`mizar-resolve` can start with tasks 1-7 from
[todo.md](./todo.md). `mizar-build` wave A is complete, so the resolver-side
module-index seam can be specified against the build-side provider from
[module_index.md](../../mizar-build/en/module_index.md). The parser/syntax
surface is current through `mizar-parser` task 45; task 46 is deferred and does
not block the resolver foundation.

The crate is not ready for full user-facing resolver diagnostics until the
diagnostic-code ownership gap in [architecture 03](../../architecture/en/03.module_and_symbol_resolution.md)
and [spec chapter 22](../../../spec/en/22.error_handling_and_diagnostics.md)
is resolved or explicitly deferred by the relevant task.

## Exit Criteria

Hard gates:

- no blocking/high specification inconsistency remains for the active milestone;
- every public API and promised behavior has design coverage before source
  implementation;
- no source behavior remains absent from `doc/spec`, existing tests, or the
  active design spec;
- `.miz` tests and expectations are not changed merely to match implementation;
- resolver does not own parser, checker, proof, build, driver, artifact, or
  session responsibilities;
- diagnostics are either backed by an explicit spec/code-space decision or
  documented as crate-local/internal for the milestone;
- remaining risks are classified as deferred, out of scope, or human-owned.

Verification commands:

```text
cargo fmt --check
cargo test -p mizar-resolve
cargo clippy -p mizar-resolve --all-targets --all-features -- -D warnings
```

For boundary work, also run the relevant subset:

```text
cargo test -p mizar-build
cargo test -p mizar-session
cargo test -p mizar-syntax
cargo test -p mizar-frontend
cargo test -p mizar-test
```

Review expectations:

- review-only agent for the active specification;
- test sufficiency review before marking implementation tasks complete;
- implementation review with no unresolved findings;
- source/documentation consistency review;
- crate-wide completion requires hard gates plus a read-only quality score of
  at least 90/100.
