# Crate Plan: mizar-resolve

> Canonical language: English. Japanese companion: [../ja/00.crate_plan.md](../ja/00.crate_plan.md).

## Responsibility

Owned behavior:

- consume `SurfaceAst` and phase-0 module-index input to produce `ResolvedAst`
  and `SymbolEnv`;
- resolve module paths, imports, exports, namespaces, labels, and stable symbol
  identities;
- collect declaration signatures and dependency-facing module summaries without
  type checking;
- preserve recoverability by representing unresolved and ambiguous references
  explicitly;
- keep output ordering, ids, debug renderings, and diagnostics deterministic.

Out of scope:

- syntax ownership, parser recovery, or AST vocabulary changes;
- type inference, overload winner selection, cluster firing, or coercion
  insertion;
- proof validity, VC generation, ATP interaction, kernel acceptance, and
  artifact emission;
- package manifest parsing, lockfile resolution, source loading, and module
  discovery owned by `mizar-build` and `mizar-session`;
- driver lifecycle, salsa/query integration, and phase registry ownership.

## Readiness

`mizar-resolve` can start with tasks 1-7 from
[todo.md](./todo.md). `mizar-build` wave A is complete, so the resolver-side
module-index seam can be specified against the build-side provider from
[module_index.md](../../mizar-build/en/module_index.md). The parser/syntax
surface is current through `mizar-parser` task 45; task 46 is deferred and does
not block the resolver foundation.

The crate is not ready for full user-facing resolver diagnostics until the
diagnostic-code ownership gap in [architecture 03](../../architecture/en/03.module_and_symbol_resolution.md)
and [spec chapter 22](../../../spec/en/22.error_handling_and_diagnostics.md)
is resolved or explicitly deferred by the relevant task.

Observed behavior:

- `mizar-build` now provides deterministic `BuildPlan` and `ModuleIndex`
  construction without allocating session-local source ids.
- `mizar-session` owns source identity, normalized paths, and snapshot identity.
- `mizar-frontend` provides lexical environment and parser orchestration seams;
  semantic import validation remains resolver-owned.
- The executable semantic import pass is present up to the R-010
  source-shaped candidate seam: alias binding, relative-prefix interpretation,
  namespace/package binding, unresolved recovery, and canonical graph
  construction exist over unit-test candidates.
- The executable declaration pass is present up to the R-011 source-shaped
  collector slice: represented declaration-like `SurfaceAst` items, visibility
  wrappers, transparent annotation wrappers, recovered-shell state, and export
  projection shells are inventoried without semantic validation.
- `names.md` specifies the R-013 through R-016 name-resolution contract for
  scope, namespace-before-symbol lookup, visibility/shadowing, ambiguity, and
  dot-chain finalization. The executable names pass is present up to the R-013
  namespace lookup slice: import aliases, reserved namespace roots,
  package-name bindings, current-package fallback, recovered namespace paths,
  unresolved import-alias dependencies, ambiguous alias candidate targets,
  provider-error classification, and earliest failing segment ranges are
  represented. The executable names pass is also present up to the R-014
  symbol-name slice: preliminary symbol projections, declaration-point
  filtering, qualified/unqualified visibility, current-module shadowing,
  overload-group placeholder collapse, enabled builtin fallback, and
  deterministic `NameRefTable` outcomes are represented without selector type
  checking or overload winner selection. The executable names pass is also
  present up to the R-015 internal diagnostic slice: deterministic
  `NameDiagnosticRootId` allocation, primary/cascade records, unresolved
  import-alias roots, namespace/name dependent records, stable candidate
  payloads, and internal record ordering are represented without public
  diagnostic codes. The executable names pass is also present up to the R-016
  dot-chain finalization slice: `LocalTermScope`, `LocalTermBinding`,
  `DotChainCandidate`, and `DotChainFinalizer` classify dotted chains as
  checker-deferred selectors or namespace-qualified symbol references without
  selector validation. The R-017 label contract is specified in `labels.md`,
  and R-018 implements `LabelResolver`, `LabelScopePath`, `LabelProjection`,
  `LabelReferenceCandidate`, deterministic `LabelIndex` / `LabelRefTable`
  output, and crate-local/internal label conflict diagnostics. R-019 adds the
  `symbols.md` signature-collection contract, including algorithm signatures,
  exported summaries, and lexical-summary projections. R-020 adds
  `SymbolCollector`, `SymbolDeclarationProjection`, opaque `SignatureShell`
  population, duplicate/conflict diagnostics, overload grouping, registration
  indexing, and contribution tracking. R-021 adds
  `SignatureProjectionExtractor`, parser-owned opaque signature payloads,
  template-role preservation in owning declarations, and
  `ModuleLexicalSummaryIndex` seeds for exported lexer-visible spellings.
  R-022 adds a resolver-local recovered-syntax policy: recovered namespace and
  name/dot-chain references are retained without internal name diagnostic
  roots, recovered label projections do not emit conflict diagnostics, and
  recovered symbol projections retain `RecoveredShell` metadata without
  duplicate or illegal-overload diagnostics.
  Direct import/export semantic validation, artifact-backed summaries, grouped
  shared-prefix lowering, and corpus-level label/symbol lowering must still be
  driven from spec, tests, and design rather than reverse engineering.

## Exit Criteria

Hard gates:

- no blocking/high specification inconsistency remains for the active milestone;
- every public API and promised behavior has design coverage before source
  implementation;
- no source behavior remains absent from `doc/spec`, existing tests, or the
  active design spec;
- `.miz` tests and expectations are not changed merely to match implementation;
- resolver does not own parser, checker, proof, build, driver, artifact, or
  session responsibilities;
- diagnostics are either backed by an explicit spec/code-space decision or
  documented as crate-local/internal for the milestone;
- remaining risks are classified as deferred, out of scope, or human-owned.

Verification commands:

```text
cargo fmt --check
cargo test -p mizar-resolve
cargo clippy -p mizar-resolve --all-targets --all-features -- -D warnings
```

For boundary work, also run the relevant subset:

```text
cargo test -p mizar-build
cargo test -p mizar-session
cargo test -p mizar-syntax
cargo test -p mizar-frontend
cargo test -p mizar-test
```

Review expectations:

- review-only agent for the active specification;
- test sufficiency review before marking implementation tasks complete;
- implementation review with no unresolved findings;
- source/documentation consistency review;
- crate-wide completion requires hard gates plus a read-only quality score of
  at least 90/100.
