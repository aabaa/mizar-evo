# Crate Plan: mizar-frontend

> Canonical language: English. Japanese companion: [../ja/00.crate_plan.md](../ja/00.crate_plan.md).

This plan records the completed `mizar-frontend` crate against the autonomous
crate-development protocol. It is retrospective because the crate was
implemented before `doc/design/autonomous_crate_development.md` became the
standard crate-wide workflow.

## Responsibility

Owned behavior:

- source-to-syntax orchestration for the phase 1-3 frontend pipeline;
- projection of `mizar-session` loaded sources into frontend `SourceUnit`s;
- frontend-owned span bridging from lexer byte ranges to session `SourceRange`
  values;
- preprocessing coordination, mapped comments/doc comments, lexical text maps,
  and mapped shallow import stubs;
- active lexical environment requests, provider provenance validation,
  canonical import handling, conflict retry, and frontend diagnostics around
  dependency lexical summaries;
- parser-assisted lexing plans, tokenization invocation, mapped lexing
  diagnostics, and `TokenStream` output;
- parser seam invocation, parser input assembly, syntax diagnostic passthrough,
  and optional `SurfaceAst` handoff;
- layered frontend content cache keys for `SourceUnit`, `PreprocessedSource`,
  `ActiveLexicalEnvironment`, `TokenStream`, and `SurfaceAst`;
- deterministic frontend diagnostic merging and hard-error boundaries.

Out of scope:

- source identity, source hashes, source snapshots, and source retention
  (`mizar-session`);
- raw scanning, comment stripping rules, reserved tables, lexical environment
  assembly rules, scope-skeleton rules, and token disambiguation rules
  (`mizar-lexer`);
- `SurfaceAst` node definitions and syntax-owned trivia structures
  (`mizar-syntax`);
- grammar implementation, Pratt parsing, parser recovery decisions, and parser
  fuzz ownership (`mizar-parser`);
- module graph construction, import legality beyond shallow stubs, semantic name
  resolution, type checking, overload resolution, cluster registration, proof
  checking, VC generation, cache storage, artifact publication, and LSP display
  rendering.

`mizar-frontend` is a boundary crate. It coordinates language-facing source,
lexical, and parser outputs, but it must not convert that coordination into
semantic facts owned by later phases.

## Relevant Tests

The executable contract for `mizar-frontend` is mostly Rust coverage because the
crate coordinates already-owned session, lexer, syntax, and parser contracts.
Complete-source `.miz` tests are used where the current parser seam can expose
frontend-reachable parser behavior.

For this crate, the autonomous protocol gate "no source behavior remains that is
absent from `doc/spec` and tests" is interpreted as follows:

- language behavior must be represented in `doc/spec`, `.miz` tests, or
  traceability metadata before frontend source changes;
- frontend orchestration behavior that is not independently visible as a
  complete `.miz` language contract must be represented in
  `doc/design/mizar-frontend/en/` and Rust tests;
- future grammar growth must add parser/frontend `.miz` or traceability coverage
  through the owning parser task before the frontend treats that behavior as
  complete;
- currently unproducible non-exhaustive fallback variants remain deferred until
  their owning upstream producer exposes a concrete variant.

  preprocessing, active lexical environment construction, parser-assisted
  lexing, parser seam invocation, cache-key computation, and diagnostic merging.

  `FrontendDiagnostic`s when the pipeline can continue; source-load, span-bridge,

  constructible.

### Task MF-001: Source Identity Projection And Span Bridging

Purpose: bridge `mizar-session` source identity/maps into frontend source units
and convert lexer byte spans into session source ranges.

Spec refs:

- [source.md](./source.md)

- `crates/mizar-frontend/src/source.rs`

Completion condition:

- source loading metadata is projected without recomputation;

Forbidden behavior:

- owning source identity, hashing, snapshots, or retention;
- silently changing coordinate spaces;
- fabricating source ranges for source-load failures.

## Exit Criteria

Hard gates:

- no blocking/high specification inconsistency remains in `mizar-frontend`
  scope;
- no known frontend source behavior remains absent from design/spec/test
  evidence;
- milestone-owned language-facing items have active `.miz` coverage where
  complete-source behavior is frontend-reachable, or explicit deferred reasons;
- no test expectation is changed merely to match current implementation;
- English/Japanese design docs, README indexes, source correspondence evidence,
  and source behavior are synchronized;
- source/session, lexer, syntax, parser, resolver, cache-storage, and diagnostic
  rendering boundaries are respected;
- relevant verification commands pass, or skipped commands are justified;
- residual risks are classified as deferred, out of scope, or human-owned.

Verification commands:

```sh
cargo fmt --check
cargo clippy --all-targets --all-features -- -D warnings
cargo test
cargo run -p mizar-test -- plan --tests-root tests --manifest tests/coverage/spec_trace.toml
```

Review expectations:

- review the crate plan and exit report as retrospective protocol evidence;
- review `source_spec_correspondence.md` for source/design/test mapping;
- review `bilingual_documentation_synchronization.md` and the Japanese
  companions for synchronization;
- treat future parser grammar growth as parser-owned unless a bounded frontend
  follow-up is explicitly opened.
