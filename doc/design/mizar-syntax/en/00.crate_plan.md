# Crate Plan: mizar-syntax

> Canonical language: English. Japanese companion:
> [../ja/00.crate_plan.md](../ja/00.crate_plan.md).

Status: initial autonomous crate-development kickoff plan.

This plan is the crate-scope entry point required by
[the autonomous crate-development protocol](../../autonomous_crate_development.md)
before further autonomous `mizar-syntax` implementation work. It records the
current boundary, the specification and test inputs, known gaps and drift, the
remaining task split, and the exit gates for completing crate-wide syntax work.

## Responsibility

Owned behavior:

- define the source-shaped `SurfaceAst` boundary produced by `mizar-parser`;
- own rowan-backed syntax storage, deterministic syntax snapshots, typed
  compatibility views, and syntax-kind vocabulary;
- own syntax-level diagnostics and recovery node vocabulary;
- own syntax-facing trivia side tables for comment ranges, doc-comment
  attachment hints, skipped-token ranges, and whitespace hints;
- provide parser-facing builder/accessor APIs without exposing raw storage as a
  semantic identity contract.

Out of scope:

- lexing, token disambiguation, parser-assisted lexing plans, and raw comment
  extraction, which belong to `mizar-frontend` / `mizar-lexer`;
- grammar parsing and synchronization producers, which belong to `mizar-parser`;
- name resolution, type checking, overload resolution, cluster facts, proof
  obligations, theorem validity, algorithm VC generation, and package/build
  semantics;
- `salsa` query integration and cache orchestration;
- stable external artifact schemas.

Boundary rule: `mizar-syntax` may represent surface syntax and recovery facts,
but it must not encode resolved symbol ids, inferred types, semantic categories
that require the active environment, or proof/VC obligations.

## Task Decomposition

### Task S-009: Module, item, and shared path nodes

Purpose: represent shared qualified-symbol/module-path surface nodes, module
file shape, item lists, import/export items, and visibility syntax.

Spec refs: [12.modules_and_namespaces.md](../../../spec/en/12.modules_and_namespaces.md),
[appendix A](../../../spec/en/appendix_a.grammar_summary.md).

Tests: add parser task 4 unit coverage for path helpers, then activate module
prelude ordering, import path, misplaced import, and missing item/end rows from
the parse-only seed when the parser can produce them; add syntax snapshots after
node roles are specified.

Affected design files: [ast.md](./ast.md), [trivia.md](./trivia.md) for
doc-comment item attachment, [recovery.md](./recovery.md) for item-level
skipped or missing nodes.

Affected source files: `src/ast.rs`, `src/trivia.rs`, `src/recovery.rs` as
needed for item recovery.

Completion condition: shared path, module/item node kinds, child roles,
accessors, ranges, snapshots, corpus cases, and traceability entries are present
for parser tasks 4-7.

Forbidden behavior: import resolution, visibility semantics, symbol identity,
or item validity checking.

## Exit Criteria

Hard gates:

- no blocking/high specification inconsistency remains for milestone-owned
  syntax behavior;
- no language-facing source behavior remains that is absent from
  `doc/spec/en/`, `.miz` tests, or traceability metadata;
- no implementation-facing API behavior remains that is absent from
  `doc/design/mizar-syntax`, Rust tests, or explicit deferred notes;
- milestone-owned specification items have existing or test-first `.miz`
  coverage, or an explicit deferred reason;
- existing `.miz` tests and `.expect.toml` files are not rebaselined merely to
  match current implementation behavior;
- `doc/design/mizar-syntax` and `crates/mizar-syntax` agree within the target
  scope;
- parser, lexer, resolver, type, proof, and VC responsibility boundaries are
  respected;
- required verification commands pass, or every unrun command is explicitly
  justified;
- remaining risks are classified as deferred, out of scope, or human-owned;
- a read-only review assigns a valid crate quality score of at least 90/100
  after hard gates pass.

Verification commands:

```text
cargo fmt --check
cargo clippy --all-targets --all-features -- -D warnings
cargo test
cargo run -p mizar-test -- plan --tests-root tests --manifest tests/coverage/spec_trace.toml
```

Narrower commands may run first for individual increments:

```text
cargo test -p mizar-syntax
cargo clippy -p mizar-syntax --all-targets --all-features -- -D warnings
cargo test -p mizar-parser
cargo test -p mizar-frontend
```

Review expectations:

- Review the implementation specification before each increment.
- Review test sufficiency against this crate plan and the owning spec refs.
- Review the full implementation for bugs, boundary violations, missing edge
  cases, and drift.
- Review source/documentation consistency before committing.
- Produce a Crate Exit Report when crate-wide autonomous work is complete.
