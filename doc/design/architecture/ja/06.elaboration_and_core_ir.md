## 設計判断

### elaboration はソースの形を保つ最後の境界である

`ResolvedTypedAst` はまだ利用者のソースに近いものです。記法・挿入されたビュー・オーバーロードのメタデータ・回復可能な失敗箇所を保持します。`CoreIr` は意味論の消費者のために正規化された表現です。

elaboration が所有するもの:

- 解決された項と論理式を正準的な中核ノードへ低位化する
- 利用者向けのシンボルを正準的な `SymbolId` 参照で置き換える
- 挿入された `qua` ビューと型の仮定を明示する
- 意味論的な同一性を持たない構文糖を展開する
- 定義・定理・registration・アルゴリズムの境界を記録する
- 証明スケルトンの構造を中核の証明ノードへ抽出する
- `ResolvedTypedAst` へのソースマップを生成する

elaboration が所有しないもの:

- 名前・型・registration・オーバーロードの解決
- 証明探索や ATP ディスパッチ
- アルゴリズム本体の制御フローグラフの構築
- 最終的な VC 生成
- アーティファクトスキーマの出力

`ResolvedTypedAst` のノードが未解決またはブロックされている場合、elaboration は明示的なエラーノードを保持するか、その項目をスキップしなければなりません。中核項を捏造してはなりません。

### Core IR はソフト型を意図的に消去する

Mizar のソフト型は検査を導きますが、論理的な基礎の一部ではありません。そのため `CoreIr` は、型注釈を明示的な論理述語と仮定へ消去します。

例:

- `let x be T` という変数宣言は、`x` の中核束縛と、ローカルな事実 `is_T(x)` へ低位化される。
- 型主張 `x is T` は、正規化された型の述語論理式へ低位化される。
- `qua` ビューは、同じ基礎項と明示的なビューのメタデータ、および後の説明に必要な確立済みの型事実へ低位化される。
- `reconsider` の束縛は、新たな束縛または縮小された束縛と、関連する義務シードへの参照へ低位化される。

消去は明示的でなければなりません。後続のフェーズが、どの論理述語が利用可能かを知るためにソースの型構文を調べる必要があってはなりません。

### 定義は安定した展開境界とともに低位化される

elaboration は、定義を明示的な展開ポリシーを持つ中核項目として記録します。すべての定義を先行的にインライン化することはしません。

定義レコードは次を含みます。

- 正準的な定義シンボル
- パラメータの束縛子と型述語の仮定
- 存在する場合の定義本体（definiens）の論理式または項
- existence・uniqueness・coherence・compatibility・reducibility の各義務シードへの参照
- 定義が計算や簡約に参加できる場合の書き換えまたは簡約のメタデータ
- ソース範囲とドキュメントの結び付け ID

後続のフェーズは、証明コンテキスト・ATP エンコードプロファイル・カーネルポリシー・簡約メタデータに基づいて、定義を展開するかどうかを選びます。これにより elaboration を決定的に保ち、インライン化の順序による偶発的な証明強度の変化を防ぎます。

### テンプレートはすでに具体化されている

テンプレート推論とオーバーロード選択は、elaboration より前に完了しています。`CoreIr` は、通常の使用箇所に未解決のテンプレートパラメータを含むべきではありません。

elaboration はテンプレートの由来のメタデータを記録してよいです。

- ソースのテンプレートシンボル
- 具体的な実体化引数
- 生成された具体シンボルまたは実体化キー
- 明示的または推論されたテンプレート引数のソース範囲

このメタデータは診断・再現性・アーティファクト出力を支えますが、中核の論理ノードは具体的なシンボルと具体的な型を参照します。

### 証明スケルトンは中核の証明木になる

表層の証明構文は、`thesis`・構造化された自然演繹のステップ・ラベル・`by` 参照・入れ子の証明ブロックなど、多くの便宜的な形を含みます。elaboration はこれらを中核の証明木へ低位化します。

中核の証明木は次を記録します。

- 定理または補題の命題
- 各証明地点における現在の thesis
- 導入された変数と仮定
- 文のラベルと引用参照
- 末端の証明ゴールと、そのローカルな前提集合
- `open`・`assumed`・`conditional` の各項目の状態依存

`thesis` は魔法の識別子としては保持されません。証明スケルトンから計算された現在の中核論理式で置き換えられます。

### アルゴリズムは中核項目へ低位化されるが、まだ CFG にはしない

アルゴリズム定義は、論理的な検証と実行可能なコード抽出という、2 つの後続の関心事にまたがります。フェーズ 9 はアルゴリズム項目を `CoreIr` に記録しますが、完全な `ControlFlowIr` は構築しません。

elaboration が所有するもの:

- アルゴリズムの同一性とパラメータの束縛子
- 解決された返り値型と結果シンボル
- 事前条件・事後条件・不変条件・assertion・停止性測度を中核の論理式または項として
- 束縛と式の ghost／ランタイムの分類
- 解決済みシンボルを持つ、ソースの形を保った文の木

フェーズ 10 が所有するもの:

- 基本ブロックの構築
- 変更と代入のフロー
- ループ不変条件の配置
- 到達可能性と代入前使用の検査
- コード抽出に向けたランタイム／ghost の分離

これにより、純粋な論理的低位化と制御フロー解析を分離します。

アルゴリズムの `requires`・`ensures`・`assert`・`invariant`・`decreasing` の各句は、フェーズ 9 で中核の論理式または項へ低位化されます。それらの VC の配置・経路条件・ループのカットポイント・停止性検査の構造は、後続のフェーズ 10 と 11 が決定します。

### ソースマップは必須であり、デバッグ用のおまけではない

診断・義務・メタデータのエントリを生成し得るすべての中核項目と中核式は、ソースへ戻る対応付けを持たなければなりません。

ソースマップは次を保持すべきです。

- `ResolvedTypedAst` 内のソース範囲
- 由来となるシンボルまたはラベル
- 挿入されたビューの来歴
- 低位化された型述語の来歴
- テンプレート実体化の来歴
- 証明スケルトン上の位置

後続のフェーズは、中核の構造から位置を再構築するのではなく、これらの対応付けを診断とアーティファクトに用いなければなりません。

### Core IR はバックエンド中立である

`CoreIr` は TPTP・SMT-LIB・カーネルの証明書言語ではありません。正準的な内部の論理表現です。

バックエンド固有の関心事は先送りされます。

- 前提選択と VC スライシングはフェーズ 11-12
- ATP のシンボル命名とエンコードはフェーズ 13
- 証明書の形式と再生はフェーズ 14
- 安定した外部メタデータの射影はフェーズ 15

## elaboration パイプライン

### ステップ 1: 中核コンテキストの準備

入力:

- `ResolvedTypedAst`
- `SymbolEnv`
- `TypeFactTable`
- `ResolutionTrace`
- 依存サマリ

出力:

- `CoreContext`
- 中核シンボルテーブル
- elaboration のワークリスト

責務:

- 必要な overload resolution result が存在することを検証する。
- 参照に必要な依存の中核サマリを読み込む。
- 現在のモジュールの項目に安定した `CoreItemId` を割り当てる。
- ソースマップビルダを初期化する。
- 項目レベルの elaboration の依存関係を収集する。

### ステップ 2: 型と事実の低位化

入力:

- 正規化された型
- 型事実
- 挿入されたビュー
- 前段の意味論フェーズからの義務シード

出力:

- 中核の型述語
- 中核の仮定レコード
- ビューの説明
- 義務シードへの参照

責務:

- 型表現を述語適用へ低位化する。
- 属性の連鎖を述語の連言へ低位化する。
- 正と負の属性の極性を記録する。
- `ResolutionTrace` 由来の来歴を付与する。
- 後続の VC 生成のために義務シード ID を保持する。

### ステップ 3: 項と論理式の低位化

入力:

- 解決された項と論理式のノード
- オーバーロード解決テーブル
- 挿入されたビューと型事実

出力:

- `CoreTerm`
- `CoreFormula`
- 式のソースマップ

責務:

- 変数・定数・functor 適用・セレクタ・集約・集合式を低位化する。
- 述語適用・等号・型主張・属性・結合子・量化子を低位化する。
- オーバーロード箇所を選択された `SymbolId` で置き換える。
- `qua` をビューのメタデータと基礎項で置き換える。
- 未解決の失敗箇所を明示的な elaboration エラーとして保持する。

### ステップ 4: 宣言と定義の低位化

入力:

- 解決された宣言とシグネチャ
- 低位化された項と論理式
- 保留中の義務シードのメタデータ

出力:

- `CoreItem`
- `CoreDefinitionTable`
- 義務シードテーブル

責務:

- 構造体・mode・属性・述語・functor・定理・scheme・registration・reduction を低位化する。
- 定義の展開ポリシーを記録する。
- 正しさの義務シードへの参照を記録する。
- アーティファクトが必要とする可視性とエクスポートのメタデータを保持する。
- 意味論的な前提が失敗した項目を却下またはスキップする。

### ステップ 5: 証明の低位化

入力:

- 定理と証明のノード
- 証明ラベルと引用
- 低位化された論理式

出力:

- `CoreProofTable`
- 中核の証明木
- 末端の証明ゴールのシード

責務:

- 証明スケルトンを計算する。
- `thesis` を現在の中核論理式で置き換える。
- 仮定と導入された変数を低位化する。
- 引用された前提への参照を付与する。
- VC 生成のために末端の証明ゴールのシードを生成する。
- `open`・`assumed`・`conditional` の各項目の状態依存を保持する。

### ステップ 6: アルゴリズムシェルの低位化

入力:

- 解決されたアルゴリズム定義
- 低位化された契約の論理式と項
- ghost／ランタイムのメタデータ

出力:

- 中核のアルゴリズム項目
- フェーズ 10 向けの文木のシェル
- 契約と停止性のメタデータ

責務:

- アルゴリズムのパラメータと結果の束縛子を記録する。
- `requires`・`ensures`・`assert`・`invariant`・`decreasing` の各句を低位化する。
- 文の順序と解決済みの式メタデータを保持する。
- ghost とランタイムの束縛を区別する。
- CFG の構築をフェーズ 10 に先送りする。

## インターフェイス定義

### Core IR

```rust
struct CoreIr {
    module_id: ModuleId,
    items: Vec<CoreItem>,
    definitions: CoreDefinitionTable,
    proofs: CoreProofTable,
    algorithms: CoreAlgorithmTable,
    obligation_seeds: ObligationSeedTable,
    source_map: CoreSourceMap,
}
```

`CoreIr` はコンパイラ内部のものです。安定した外部アーティファクトは、後続で出力される射影です。

### 中核項目

```rust
struct CoreItem {
    id: CoreItemId,
    symbol: SymbolId,
    kind: CoreItemKind,
    visibility: Visibility,
    dependencies: Vec<CoreItemId>,
    source: CoreSourceRef,
}

enum CoreItemKind {
    Structure(CoreStructure),
    Mode(CoreMode),
    Attribute(CoreAttribute),
    Predicate(CorePredicate),
    Functor(CoreFunctor),
    Theorem(CoreTheorem),
    Registration(CoreRegistration),
    Reduction(CoreReduction),
    Algorithm(CoreAlgorithm),
}
```

### 中核の項と論理式

```rust
enum CoreTerm {
    Var(CoreVarId),
    Const(SymbolId),
    Apply { functor: SymbolId, args: Vec<CoreTermId> },
    Select { selector: SymbolId, base: CoreTermId },
    SetEnum(Vec<CoreTermId>),
    Choice { binder: CoreBinder, predicate: CoreFormulaId },
    Error(ElaborationErrorId),
}

enum CoreFormula {
    True,
    False,
    Atom { predicate: SymbolId, args: Vec<CoreTermId> },
    TypePred { subject: CoreTermId, ty: CoreTypePredicate },
    Not(CoreFormulaId),
    And(Vec<CoreFormulaId>),
    Or(Vec<CoreFormulaId>),
    Implies { premise: CoreFormulaId, conclusion: CoreFormulaId },
    Iff { left: CoreFormulaId, right: CoreFormulaId },
    Forall { binders: Vec<CoreBinder>, body: CoreFormulaId },
    Exists { binders: Vec<CoreBinder>, body: CoreFormulaId },
    Error(ElaborationErrorId),
}
```

中核ノードは正準的なシンボルと明示的な束縛子を用います。表層の優先順位・括弧・記法の形は表現しません。

### 中核の定義

```rust
struct CoreDefinition {
    symbol: SymbolId,
    params: Vec<CoreBinder>,
    body: DefinitionBody,
    expansion: ExpansionPolicy,
    obligations: Vec<ObligationSeedId>,
}

enum ExpansionPolicy {
    Opaque,
    Transparent,
    Reducible { reduction: RegistrationId },
    Computable { algorithm: SymbolId },
}
```

展開ポリシーは意味論的なメタデータです。elaboration 中の先行的なインライン化を強制するものではありません。

### 中核の証明

```rust
struct CoreProof {
    theorem: SymbolId,
    proposition: CoreFormulaId,
    root: CoreProofNodeId,
    status: TheoremStatus,
}

enum CoreProofNode {
    IntroduceBinder { binder: CoreBinder, child: CoreProofNodeId },
    Assume { label: Option<LabelRef>, formula: CoreFormulaId, child: CoreProofNodeId },
    Step { label: Option<LabelRef>, formula: CoreFormulaId, justification: CoreJustification },
    Branch(Vec<CoreProofNodeId>),
    TerminalGoal(ObligationSeedId),
    Error(ElaborationErrorId),
}
```

`ObligationSeedId` は `VcId` ではありません。VC 生成より前の、ソースに対応付けられた証明ゴールまたは正しさの条件を識別します。フェーズ 11 が、これらのシードと `CoreIr`・`ControlFlowIr` から具体的な `VcId` を生成します。

### ソースマップ

```rust
struct CoreSourceMap {
    item_sources: Map<CoreItemId, CoreSourceRef>,
    term_sources: Map<CoreTermId, CoreSourceRef>,
    formula_sources: Map<CoreFormulaId, CoreSourceRef>,
    obligation_sources: Map<ObligationSeedId, CoreSourceRef>,
}

struct CoreSourceRef {
    source_id: SourceId,
    resolved_node: AstNodeId,
    range: SourceRange,
    provenance: Vec<CoreProvenance>,
}
```

## 診断

elaboration の診断は、表層意味論と中核論理の境界に関する診断です。

| 診断クラス | 例 |
|---|---|
| 未解決の意味論入力 | オーバーロード箇所に selected root がない、型の前提が失敗した |
| 未対応の低位化 | 表層の構文要素にまだ中核表現がない |
| 不正な証明スケルトン | 証明本体を定理の構造に対応付けられない |
| 不正な消去 | 型またはビューのメタデータを述語へ変換できない |
| アルゴリズムシェル | ghost／ランタイムのメタデータの欠落、契約の低位化の失敗 |
| ソースマッピング | 中核ノードをソース範囲に対応付けられない |

診断には次を含めるべきです。

- `ResolvedTypedAst` 由来の主たるソース位置
- 挿入された `qua`・selected overload root・active refinement・型事実といった関連する意味論的な来歴
- 利用可能なら、影響を受けた中核項目の ID
- LSP とアーティファクト用の安定した診断コード

## 増分処理と再現性

elaboration のキャッシュキーには次を含めます。

- `ResolvedTypedAst` のハッシュ
- 中核の消去が用いる `TypeFactTable` と `ResolutionTrace` のハッシュ
- 依存サマリのハッシュ
- elaboration のスキーマバージョン
- 言語エディション

変更は次を無効化します。

- 変更された項目の `CoreItem`
- 依存する証明スケルトンと定義本体
- 契約または本体が変わったアルゴリズムシェル
- 変更された中核ノードを消費する VC 生成・ATP 変換・アーティファクト出力

順序は決定的でなければなりません。

- 中核項目は、ソース順の診断の後に正準的な項目順に従う。
- 束縛子は各項目内でソース順を保持する。
- 型消去が生成する連言は、安定した述語順を用いる。
- ソースマップの来歴リストは、フェーズとソース範囲でソートする。
- スキップ／エラーのノードは AST 走査順でシリアライズする。

## 検討した代替案

### 検証条件へ直接 elaboration する

`CoreIr` を省くと表現が 1 つ減りますが、VC 生成・カーネル検査・ドキュメントメタデータ・アルゴリズム準備は、それぞれ異なる射影を必要とします。`CoreIr` は共有された意味論的な基盤を提供します。

### 中核に表層構文を残す

記法やソースの糖を中核に残すとソースマッピングは単純になりますが、後続のすべてのフェーズが構文解析器レベルの構文要素を理解しなければなりません。採用した設計では、ソースマップを保持しつつ中核の意味論を正規化します。

### 先行的な定義のインライン化

elaboration 中にすべての定義をインライン化すると、後の一部の項は単純になりますが、証明探索の形を変え、問題サイズを爆発させ得ます。採用した設計では展開ポリシーを記録し、後続のフェーズが制御された展開を選択します。

### elaboration 中にアルゴリズムの CFG を構築する

`ControlFlowIr` を即座に構築すると、論理的な低位化とプログラム解析が結合します。採用した設計では、フェーズ 9 でアルゴリズムシェルを低位化し、CFG の構築をフェーズ 10 に委ねます。

## 採用方針

Mizar Evo は、専用の elaboration フェーズで `ResolvedTypedAst` をバックエンド中立な `CoreIr` へ低位化します。

elaborator は、ソフト型を明示的な述語へ消去し、定義の展開境界を記録し、項と論理式を正準的なノードへ低位化し、証明構造を中核の証明木へ変換し、後続の制御フロー準備のためにアルゴリズムシェルを記録します。診断・VC 生成・ATP 変換・カーネル検査・アーティファクトが同じ意味論的な出所を参照できるよう、ソースマップと来歴を保持します。

## 制約と前提

- elaboration は、オーバーロード解決が `ResolvedTypedAst` を生成した後にのみ開始する。
- 中核ノードは、ソースの綴りではなく正準的なシンボル同一性を用いなければならない。
- ソフト型注釈は、明示的な型述語と仮定を通じてのみ消去される。
- elaboration は証明探索や registration の有効化を行ってはならない。
- 失敗した意味論箇所は、明示的なエラーノードまたはスキップされた項目のまま残す。妥当な中核項にしてはならない。
- アルゴリズムの制御フローグラフの構築は、フェーズ 9 ではなくフェーズ 10 に属する。
- `CoreIr` はバックエンド中立であり、ATP 固有のエンコード判断を含んではならない。
- フェーズ 9 は義務シードだけを作る。具体的な `VcId` はフェーズ 11 が割り当てる。
