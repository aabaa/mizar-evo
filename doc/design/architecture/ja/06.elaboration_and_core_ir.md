# Architecture: Elaboration and Core IR

> Canonical language: English. English canonical version: [../en/06.elaboration_and_core_ir.md](../en/06.elaboration_and_core_ir.md).

## 目的

この文書は、Mizar Evo が fully resolved surface semantics を、後続 verification phases が使う prover- and kernel-oriented logical representation である `CoreIr` へ lower する方法を定義する。

これは [00.pipeline_overview.md](./00.pipeline_overview.md) の phase 9 を詳細化する。特に、diagnostics and LSP metadata のため source shape を保持する `ResolvedTypedAst` と、VC generation and kernel checking のため terms、formulas、definitions、proof structure、semantic annotations を normalize する `CoreIr` の境界を定義する。

## Design Decisions

### Elaboration Is the Last Source-Shaped Boundary

`ResolvedTypedAst` はまだ user source に近い。Notation、inserted views、overload metadata、recoverable failed sites を保持する。`CoreIr` は semantic consumers のために normalize された表現である。

Elaboration は次を所有する。

- resolved terms and formulas を canonical core nodes へ lower する
- user-facing symbols を canonical `SymbolId` references で置き換える
- inserted `qua` views and type assumptions を明示する
- semantic identity を持たない syntactic sugar を expand する
- definition、theorem、registration、algorithm boundaries を記録する
- proof skeleton structure を core proof nodes に extract する
- `ResolvedTypedAst` への source maps を生成する

Elaboration は次を所有しない。

- name、type、registration、overload resolution
- proof search or ATP dispatch
- algorithm bodies の control-flow graph construction
- final VC generation
- artifact schema emission

`ResolvedTypedAst` node が unresolved または blocked の場合、elaboration は explicit error node を保持するか affected item を skip しなければならない。Core term を捏造してはならない。

### Core IR Erases Soft Types Deliberately

Mizar soft types は checking を導くが logical foundation の一部ではない。そのため `CoreIr` は type annotations を explicit logical predicates and assumptions に erase する。

Examples:

- `let x be T` という variable declaration は、`x` の core binding と local fact `is_T(x)` に lower される
- type assertion `x is T` は normalized type の predicate formula に lower される
- `qua` view は同じ underlying term と explicit view metadata、および later explanation に必要な established type facts に lower される
- `reconsider` binding は new or narrowed binding と associated initial obligation reference に lower される

Erasure は explicit でなければならない。Later phases は、どの logical predicates が利用可能かを知るために source type syntax を inspect する必要があってはならない。

### Definitions Are Lowered with Stable Expansion Boundaries

Elaboration は definitions を explicit expansion policy を持つ core items として記録する。すべての definition を eager に inline しない。

Definition records は次を含む。

- canonical defining symbol
- parameter binders and type predicate assumptions
- definiens formula or term when present
- existence、uniqueness、coherence、compatibility、reducibility obligation references
- definition が computation or simplification に参加できる場合の rewrite or reduction metadata
- source ranges and documentation attachment ids

Later phases は proof context、ATP encoding profile、kernel policy、reduction metadata に基づいて definition を unfold するか選ぶ。これにより elaboration を deterministic に保ち、inlining order による accidental proof-strength changes を防ぐ。

### Templates Are Already Concrete

Template inference and overload selection は elaboration 前に完了している。`CoreIr` は ordinary use sites に unresolved template parameters を含むべきではない。

Elaboration は template origin metadata を記録してよい。

- source template symbol
- concrete instantiation arguments
- generated concrete symbol or instantiation key
- explicit or inferred template arguments の source range

この metadata は diagnostics、reproducibility、artifact output を支えるが、core logical nodes は concrete symbols and concrete types を参照する。

### Proof Skeletons Become Core Proof Trees

Surface proof syntax は `thesis`、structured natural-deduction steps、labels、`by` references、nested proof blocks など多くの convenience forms を含む。Elaboration はこれらを core proof tree に lower する。

Core proof tree は次を記録する。

- theorem or lemma proposition
- each proof point の current thesis
- introduced variables and assumptions
- statement labels and citation references
- terminal proof obligations and their local premise sets
- `open`、`assumed`、`conditional` items の status dependencies

`thesis` は magic identifier として保持されない。Proof skeleton から計算された current core formula で置き換えられる。

### Algorithms Are Lowered to Core Items, Not Yet CFG

Algorithm definitions は logical verification と executable extraction という 2 つの後続 concern にまたがる。Phase 9 は algorithm items を `CoreIr` に記録するが、full `ControlFlowIr` は構築しない。

Elaboration は次を所有する。

- algorithm identity and parameter binders
- resolved return type and result symbol
- preconditions、postconditions、invariants、assertions、termination measures as core formulas or terms
- bindings and expressions の ghost/runtime classification
- resolved symbols を持つ source-shaped statement tree

Phase 10 は次を所有する。

- basic-block construction
- mutation and assignment flow
- loop invariant placement
- reachability and use-before-assignment checks
- extraction-oriented runtime/ghost separation

これにより pure logical lowering と control-flow analysis を分離する。

### Source Maps Are Required, Not Debug Extras

Diagnostic、obligation、metadata entry を生成し得るすべての core item and core expression は source に戻る mapping を持たなければならない。

Source maps は次を保持するべきである。

- `ResolvedTypedAst` 内の source range
- originating symbol or label
- inserted view provenance
- lowered type predicate provenance
- template instantiation provenance
- proof skeleton location

Later phases は core structure から location を再構築するのではなく、これらの maps を diagnostics and artifacts に使わなければならない。

### Core IR Is Backend-Neutral

`CoreIr` は TPTP、SMT-LIB、kernel certificate language ではない。Canonical internal logical representation である。

Backend-specific concerns は defer される。

- premise selection and VC slicing は phase 11-12
- ATP symbol naming and encoding は phase 13
- certificate format and replay は phase 14
- stable external metadata projection は phase 15

## Elaboration Pipeline

### Step 1: Prepare Core Context

Input:

- `ResolvedTypedAst`
- `SymbolEnv`
- `TypeFactTable`
- `ResolutionTrace`
- dependency summaries

Output:

- `CoreContext`
- core symbol table
- elaboration worklist

Responsibilities:

- required overload winners が存在することを validate する
- references に必要な dependency core summaries を読み込む
- current-module items に stable `CoreItemId`s を割り当てる
- source-map builders を initialize する
- item-level elaboration dependencies を collect する

### Step 2: Lower Types and Facts

Input:

- normalized types
- type facts
- inserted views
- initial obligations

Output:

- core type predicates
- core assumption records
- view explanations
- obligation references

Responsibilities:

- type expressions を predicate applications に lower する
- attribute chains を predicate conjunctions に lower する
- positive and negative attribute polarity を記録する
- `ResolutionTrace` 由来の provenance を付与する
- later VC generation のため initial obligation ids を保持する

### Step 3: Lower Terms and Formulas

Input:

- resolved term and formula nodes
- overload resolution table
- inserted views and type facts

Output:

- `CoreTerm`s
- `CoreFormula`s
- expression source maps

Responsibilities:

- variables、constants、functor applications、selectors、aggregates、set expressions を lower する
- predicate applications、equality、type assertions、attributes、connectives、quantifiers を lower する
- overload sites を selected `SymbolId`s で置き換える
- `qua` を view metadata and underlying term に置き換える
- unresolved failed sites を explicit elaboration errors として保持する

### Step 4: Lower Declarations and Definitions

Input:

- resolved declarations and signatures
- lowered terms and formulas
- pending obligation metadata

Output:

- `CoreItem`s
- `CoreDefinitionTable`
- initial obligation table

Responsibilities:

- structures、modes、attributes、predicates、functors、theorems、schemes、registrations、reductions を lower する
- definition expansion policy を記録する
- correctness obligation references を記録する
- artifacts が必要とする visibility and export metadata を保持する
- semantic prerequisites が failed した items を reject or skip する

### Step 5: Lower Proofs

Input:

- theorem and proof nodes
- proof labels and citations
- lowered formulas

Output:

- `CoreProofTable`
- core proof trees
- terminal obligation seeds

Responsibilities:

- proof skeletons を計算する
- `thesis` を current core formula で置き換える
- assumptions and introduced variables を lower する
- cited premise references を付与する
- VC generation のため terminal obligation seeds を生成する
- `open`、`assumed`、`conditional` items の status dependencies を保持する

### Step 6: Lower Algorithm Shells

Input:

- resolved algorithm definitions
- lowered contract formulas and terms
- ghost/runtime metadata

Output:

- core algorithm items
- phase 10 向け statement-tree shells
- contract and termination metadata

Responsibilities:

- algorithm parameter and result binders を記録する
- `requires`、`ensures`、`assert`、`invariant`、`decreasing` clauses を lower する
- statement order and resolved expression metadata を保持する
- ghost and runtime bindings を区別する
- CFG construction を phase 10 に defer する

## Interface Definitions

### Core IR

```rust
struct CoreIr {
    module_id: ModuleId,
    items: Vec<CoreItem>,
    definitions: CoreDefinitionTable,
    proofs: CoreProofTable,
    algorithms: CoreAlgorithmTable,
    obligations: InitialObligationTable,
    source_map: CoreSourceMap,
}
```

`CoreIr` は compiler-internal である。Stable external artifacts は後続で emit される projections である。

### Core Items

```rust
struct CoreItem {
    id: CoreItemId,
    symbol: SymbolId,
    kind: CoreItemKind,
    visibility: Visibility,
    dependencies: Vec<CoreItemId>,
    source: CoreSourceRef,
}

enum CoreItemKind {
    Structure(CoreStructure),
    Mode(CoreMode),
    Attribute(CoreAttribute),
    Predicate(CorePredicate),
    Functor(CoreFunctor),
    Theorem(CoreTheorem),
    Registration(CoreRegistration),
    Reduction(CoreReduction),
    Algorithm(CoreAlgorithm),
}
```

### Core Terms and Formulas

```rust
enum CoreTerm {
    Var(CoreVarId),
    Const(SymbolId),
    Apply { functor: SymbolId, args: Vec<CoreTermId> },
    Select { selector: SymbolId, base: CoreTermId },
    SetEnum(Vec<CoreTermId>),
    Choice { binder: CoreBinder, predicate: CoreFormulaId },
    Error(ElaborationErrorId),
}

enum CoreFormula {
    True,
    False,
    Atom { predicate: SymbolId, args: Vec<CoreTermId> },
    TypePred { subject: CoreTermId, ty: CoreTypePredicate },
    Not(CoreFormulaId),
    And(Vec<CoreFormulaId>),
    Or(Vec<CoreFormulaId>),
    Implies { premise: CoreFormulaId, conclusion: CoreFormulaId },
    Iff { left: CoreFormulaId, right: CoreFormulaId },
    Forall { binders: Vec<CoreBinder>, body: CoreFormulaId },
    Exists { binders: Vec<CoreBinder>, body: CoreFormulaId },
    Error(ElaborationErrorId),
}
```

Core nodes は canonical symbols and explicit binders を使う。Surface precedence、parentheses、notation forms は表現しない。

### Core Definition

```rust
struct CoreDefinition {
    symbol: SymbolId,
    params: Vec<CoreBinder>,
    body: DefinitionBody,
    expansion: ExpansionPolicy,
    obligations: Vec<InitialObligationId>,
}

enum ExpansionPolicy {
    Opaque,
    Transparent,
    Reducible { reduction: RegistrationId },
    Computable { algorithm: SymbolId },
}
```

Expansion policy は semantic metadata である。Elaboration 中の eager inlining を強制しない。

### Core Proof

```rust
struct CoreProof {
    theorem: SymbolId,
    proposition: CoreFormulaId,
    root: CoreProofNodeId,
    status: TheoremStatus,
}

enum CoreProofNode {
    IntroduceBinder { binder: CoreBinder, child: CoreProofNodeId },
    Assume { label: Option<LabelRef>, formula: CoreFormulaId, child: CoreProofNodeId },
    Step { label: Option<LabelRef>, formula: CoreFormulaId, justification: CoreJustification },
    Branch(Vec<CoreProofNodeId>),
    TerminalObligation(InitialObligationId),
    Error(ElaborationErrorId),
}
```

### Source Map

```rust
struct CoreSourceMap {
    item_sources: Map<CoreItemId, CoreSourceRef>,
    term_sources: Map<CoreTermId, CoreSourceRef>,
    formula_sources: Map<CoreFormulaId, CoreSourceRef>,
    obligation_sources: Map<InitialObligationId, CoreSourceRef>,
}

struct CoreSourceRef {
    source_id: SourceId,
    resolved_node: AstNodeId,
    range: SourceRange,
    provenance: Vec<CoreProvenance>,
}
```

## Diagnostics

Elaboration diagnostics は surface semantics and core logic の boundary diagnostics である。

| Diagnostic Class | Example |
|---|---|
| unresolved semantic input | overload site has no winner、type prerequisite failed |
| unsupported lowering | surface construct has no core representation yet |
| malformed proof skeleton | proof body cannot be matched to theorem structure |
| invalid erasure | type or view metadata cannot be converted to predicates |
| algorithm shell | ghost/runtime metadata missing、contract lowering failed |
| source mapping | core node cannot be mapped to a source range |

Diagnostics は次を含むべきである。

- `ResolvedTypedAst` 由来の primary source span
- inserted `qua`、overload winner、type fact などの relevant semantic provenance
- 利用可能なら affected core item id
- LSP and artifacts 用の stable diagnostic code

## Incrementality and Reproducibility

Elaboration cache keys は次を含む。

- `ResolvedTypedAst` hash
- core erasure が使う `TypeFactTable` and `ResolutionTrace` hashes
- dependency summary hashes
- elaboration schema version
- language edition

変更は次を invalidate する。

- changed item の `CoreItem`
- dependent proof skeletons and definition bodies
- contracts or bodies が変わった algorithm shells
- changed core nodes を consume する VC generation、ATP translation、artifact outputs

Ordering は deterministic でなければならない。

- core items は source-order diagnostics 後に canonical item order に従う
- binders は各 item 内で source order を保持する
- type erasure が生成する conjunctions は stable predicate ordering を使う
- source-map provenance lists は phase and source range で sort する
- skipped/error nodes は AST traversal order で serialize する

## Alternatives Considered

### Elaborate Directly to Verification Conditions

`CoreIr` を skip すると representation は 1 つ減るが、VC generation、kernel checking、documentation metadata、algorithm preparation は異なる projections を必要とする。`CoreIr` は共有 semantic base を提供する。

### Preserve Surface Syntax in Core

Notation and source sugar を core に残すと source mapping は単純になるが、後続 phase すべてが parser-level constructs を理解しなければならない。採用案では source maps を保持しつつ core semantics を normalize する。

### Eager Definition Inlining

Elaboration 中にすべての definitions を inline すると一部の later terms は単純になるが、proof search shape を変え、problem size を爆発させ得る。採用案では expansion policy を記録し、後続 phases が controlled unfolding を選択する。

### Build Algorithm CFG During Elaboration

`ControlFlowIr` を即座に構築すると logical lowering と program analysis が結合する。採用案では phase 9 で algorithm shells を lower し、CFG construction は phase 10 に委ねる。

## Adopted Approach

Mizar Evo は dedicated elaboration phase で `ResolvedTypedAst` を backend-neutral `CoreIr` に lower する。

Elaborator は soft types を explicit predicates に erase し、definition expansion boundaries を記録し、terms and formulas を canonical nodes に lower し、proof structure を core proof trees に変換し、later control-flow preparation のため algorithm shells を記録する。Diagnostics、VC generation、ATP translation、kernel checking、artifacts が同じ semantic source を参照できるよう、source maps and provenance を保持する。

## Constraints and Assumptions

- Elaboration は overload resolution が `ResolvedTypedAst` を生成した後にだけ開始する。
- Core nodes は source spellings ではなく canonical symbol identities を使わなければならない。
- Soft type annotations は explicit type predicates and assumptions を通じてのみ erase される。
- Elaboration は proof search や registration activation を行ってはならない。
- Failed semantic sites は explicit error nodes or skipped items のまま残す。valid core terms にしてはならない。
- Algorithm control-flow graph construction は phase 9 ではなく phase 10 に属する。
- `CoreIr` は backend-neutral であり、ATP-specific encoding decisions を含んではならない。
