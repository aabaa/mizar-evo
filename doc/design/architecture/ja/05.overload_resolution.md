## 設計判断

### Overload Resolution Finalizes Surface Semantics

Earlier phases は、symbol use が overload set を参照することを知っていてよい。また type checking は arity、kind、mandatory typing constraints により不可能な candidates を除外してよい。Phase 8 は call site で使われる final definition を選択できる最初の phase である。

Overload resolution は次を所有する。

- template-derived candidates の expansion
- enriched type facts を使った viability filtering
- per-site specificity comparisons の構築
- unique overload winner の選択
- overload、template、inheritance-path ambiguity の検出
- selected candidates が必要とする final `qua` views の insertion
- `ResolvedTypedAst` の構築

これは次を所有しない。

- name lookup or import visibility
- type expression normalization
- cluster or registration closure
- `reconsider` または registration correctness の proof discharge
- selected definitions の core logic への lowering

これにより phase 8 を deterministic かつ source-shaped に保ち、elaboration に各 resolved use site の単一 definition identity を渡せる。

### Candidate Construction Preserves Provenance

Resolver and checker は resolved surface reference で key された candidate groups を提供する。Phase 8 はそれらを concrete overload candidates に変換する。

各 candidate は次を記録する。

- defining `SymbolId`
- declaration kind and arity
- parameter `NormalizedType`s
- overload selection には使わない return type metadata
- source and import provenance
- template instantiation 由来かどうか
- arguments から parameters への required implicit views
- candidate が drop された理由を説明する pending diagnostics

Return type は overload selection に参加しない。同じ argument-type signatures で return types だけが異なる 2 つの definitions は overload alternatives ではなく declaration conflicts である。それらの conflicts は signature collection 中に検出される。Phase 8 は use site で pre-existing diagnostic を surface してよいが、conflicting declarations を overload candidates として再分類してはならない。

Phase 8 に渡される candidate groups は、source-order and scope visibility をすでに尊重していなければならない。Current-module unqualified references では、use site より前で visible な declarations だけが含まれる。Phase 8 は diagnostics のため candidate provenance を validate してよいが、`SymbolEnv` 全体を scan して candidate set を広げない。

### Template Expansion Precedes Ordinary Candidate Ordering

Template handling には 2 つの distinct phases がある。

1. Template parameter inference または explicit template argument checking が concrete candidates を生成する。
2. Standard overload ordering が concrete candidates と ordinary non-template candidates をまとめて処理する。

この 2 steps は interleave してはならない。Template inference は omitted template parameters を exact argument types と mode-hierarchy constraints から bind する。Cluster expansion は使わない。Cluster-derived alternatives は inferred parameters を non-unique にし得るためである。Template candidates が concrete になった後、ordinary viability and specificity は full type facts and cluster closure を使ってよい。

Template-derived candidate と non-template candidate が otherwise tied の場合、non-template candidate が勝つ。これにより libraries は不要な ambiguity を作らず explicit specializations を提供できる。

### Viability Uses Type Facts, Not New Inference

Candidate は、各 argument が対応する parameter type に proof-free widening で view できるとき viable である。

- identical normalized type
- unique inheritance path による mode or structure upcast
- enriched type facts and registration closure により justified される attribute weakening
- source が与えた explicit `qua` view

Phase 8 は phase 6-7 が生成した `TypeFactTable`、`ResolutionTrace`、coercion candidates を参照してよい。新しい cluster facts を derive したり registrations を activate したりしてはならない。Viability check に必要な facts が存在しない場合、type environment を拡張するのではなく candidate を reject または diagnostic 付きで blocked にする。

### Specificity Is a Partial Order

Best candidate は、その parameter vector が他のすべての viable candidates 以上に specific である一意の viable candidate である。

Phase 8 はこれを per-site specificity graph として表現する。

- nodes は viable concrete candidates
- edge `A -> B` は `A` が `B` 以上に specific であることを意味する
- strictness は `A -> B` が存在し `B -> A` が存在しない場合に導出される
- tied candidates は、仕様化された tie-breaker が適用されない限り equivalent のまま残る

Selected candidate は、許可された tie-breakers 適用後の unique maximal element でなければならない。その candidate が存在しない場合、site は ambiguity diagnostic 付きで unresolved のまま残る。

許可される tie-breakers は意図的に狭い。

- parameter vectors が otherwise equivalent のとき、non-template candidate は template-derived candidate に勝つ
- source-local declaration は、argument-type signature が同一で shadowing が name/signature collection によりすでに受理されている場合にだけ imported declaration を shadow する
- return type を使う tie-breaker は存在しない

### `qua` Is the Only Implicit View Made Explicit in Output

Source `qua` expressions は保持される。さらに phase 8 は、selected candidate が後続 phases に明示的に見せる必要のある proof-free widening view を要求する場合、implicit `qua` views を `ResolvedTypedAst` に挿入してよい。

Inserted views は次に対して許可される。

- unique structure inheritance upcasts
- すでに記録された type facts による attribute weakening
- concrete candidate が選択された後の template explicit-argument viability

Inserted views は narrowing には使えない。Narrowing は `reconsider` に属し、type checking により proof obligation または accepted local fact をすでに持っていなければならない。

各 inserted view は次を記録する。

- source argument expression
- target parameter type
- inheritance path または type-fact explanation
- source-written か compiler-inserted か
- `@show_resolution` and LSP metadata 用の diagnostic provenance

### Multiple Inheritance Paths Are Ambiguity, Not Search

Argument が parameter type へ複数の inheritance paths で到達できる場合、phase 8 は heuristic に path を選ばない。Source が `qua` で path を選択していない限り、その candidate は ambiguous inheritance path により blocked される。

これは overload ambiguity とは異なる。単一 candidate でも、その required argument view が ambiguous なため blocked され得る。

### Redefinitions Are Ordinary Candidates with Extra Checks

`redefine` declaration は ordinary definition と同じように candidate を提供する。ただし signature collection and type checking は original definition との関係を記録していなければならない。

Phase 8 は redefinitions にも同じ specificity ordering を使う。Redefine coherence は証明しない。Coherence は initial obligation として表現され、後続 proof phases が扱う。

### Output Must Preserve Failed Sites

`ResolvedTypedAst` は recoverable errors のために explicit unresolved overload nodes を含むべきである。Elaboration は fabricated winners を受け取るのではなく、それらの nodes を skip または degrade しなければならない。

これにより invalid selections を core logic に入れず、hover、candidate lists、`@show_resolution`、`qua` 追加 code actions などの LSP features を提供できる。

## 解決パイプライン

### Step 1: Collect Candidate Sites

Input:

- `TypedAst`
- name and type checking からの candidate groups
- `SymbolEnv` からの source-order-filtered visible symbol signatures
- enriched type facts and `ResolutionTrace`

Output:

- `OverloadSite`s
- initial candidate lists
- site-level expected type metadata

Responsibilities:

- overloaded functor、predicate、attribute、mode、selector、template uses を見つける
- source spans and AST node ids を付与する
- qualified-name and import provenance を保持する
- current-module candidates が use-site visibility で filtering 済みであることを enforce する
- name or type prerequisites が unrecoverably failed した sites を skip する

### Step 2: Expand Templates

Input:

- overload sites
- template definitions and explicit template arguments
- `TypedAst` からの exact argument types

Output:

- concrete template-derived candidates
- template inference diagnostics

Responsibilities:

- omitted template parameters を exact argument types から infer する
- inference 中は mode hierarchy のみで template constraints を検査する
- successful templates を concrete candidate signatures に instantiate する
- ambiguous or inconsistent template inference を reject する
- later tie-breaking のため candidates を template-derived として mark する

### Step 3: Filter Viable Candidates

Input:

- concrete candidates
- argument types
- enriched type facts
- coercion candidate table
- source-written `qua` views

Output:

- viable candidates
- rejected candidate explanations
- blocked candidate explanations

Responsibilities:

- arity and declaration kind を検査する
- argument-to-parameter compatibility を検査する
- missing facts and invalid narrowing を検出する
- ambiguous inheritance paths を検出する
- viable candidates の required view plans を計算する

### Step 4: Build the Specificity Graph

Input:

- viable candidates
- normalized parameter types
- type closure summaries

Output:

- `SpecificityGraph`
- equivalence classes
- ambiguous sites 用の comparison diagnostics

Responsibilities:

- candidates を position-wise に比較する
- at-least-as-specific edges を計算する
- incomparable candidates を検出する
- comparison が成功または失敗した理由を記録する
- type comparison 後に allowed tie-breakers を適用する

### Step 5: Select Winners and Insert Views

Input:

- specificity graph
- viable candidates
- view plans

Output:

- `OverloadResolutionTable`
- `CoercionInsertionTable`
- overload diagnostics

Responsibilities:

- unique maximal candidate を選択する
- no-match or ambiguity errors を報告する
- selected candidates のため compiler-generated `qua` views を挿入する
- source-written `qua` nodes を保持する
- diagnostics and LSP metadata のため candidate lists を記録する

### Step 6: Build `ResolvedTypedAst`

Input:

- `TypedAst`
- selected overload winners
- inserted views
- enriched type facts and resolution traces

Output:

- `ResolvedTypedAst`
- final overload metadata
- recoverable overload diagnostics

Responsibilities:

- candidate-group nodes を resolved-use nodes に rewrite する
- final definition `SymbolId`s を付与する
- final argument view metadata を付与する
- recoverable failures のため unresolved overload nodes を保持する
- elaboration and artifacts のため stable metadata を生成する

## インターフェイス定義

### Overload Site

```rust
struct OverloadSite {
    id: OverloadSiteId,
    node: AstNodeId,
    kind: OverloadSiteKind,
    name_ref: SymbolRef,
    source_range: SourceRange,
    arguments: Vec<ExprId>,
    expected: Option<ExpectedType>,
}
```

`OverloadSite` は final candidate selection が必要になり得る source-shaped use である。

### Candidate

```rust
struct OverloadCandidate {
    id: CandidateId,
    symbol: SymbolId,
    origin: CandidateOrigin,
    params: Vec<NormalizedType>,
    result: Option<NormalizedType>,
    template: Option<TemplateInstantiation>,
    provenance: CandidateProvenance,
}

enum CandidateOrigin {
    Ordinary,
    Redefinition { original: SymbolId },
    TemplateDerived { template: SymbolId },
}
```

`result` は metadata and later elaboration のために記録される。Winner selection には参加しない。

### Viability

```rust
struct ViableCandidate {
    candidate: CandidateId,
    argument_views: Vec<ArgumentView>,
}

enum CandidateRejection {
    ArityMismatch,
    KindMismatch,
    TypeMismatch { argument: ExprId, expected: NormalizedType, actual: NormalizedType },
    AmbiguousInheritancePath { argument: ExprId, target: NormalizedType, paths: Vec<InheritancePath> },
    MissingTypeFact { argument: ExprId, required: TypePredicate },
}
```

Blocked candidates は win できなくても diagnostics のために保持する。

### Specificity Graph

```rust
struct SpecificityGraph {
    site: OverloadSiteId,
    candidates: Vec<CandidateId>,
    edges: Vec<SpecificityEdge>,
    tie_breaks: Vec<TieBreak>,
}

struct SpecificityEdge {
    more_specific: CandidateId,
    less_specific: CandidateId,
    reasons: Vec<SpecificityReason>,
}
```

Graph は overload site ごとに作られる。Declarations 全体に対する global ordering ではない。

### Resolution Result

```rust
enum OverloadResolution {
    Resolved {
        site: OverloadSiteId,
        winner: CandidateId,
        inserted_views: Vec<InsertedViewId>,
    },
    NoMatch {
        site: OverloadSiteId,
        rejected: Vec<CandidateRejection>,
    },
    Ambiguous {
        site: OverloadSiteId,
        candidates: Vec<CandidateId>,
        graph: SpecificityGraph,
    },
    Blocked {
        site: OverloadSiteId,
        reason: BlockedReason,
        candidates: Vec<CandidateId>,
    },
}

enum BlockedReason {
    TemplateInferenceFailed,
    AmbiguousTemplateInference,
    AmbiguousInheritancePath,
    PreExistingDeclarationConflict,
}
```

`NoMatch` は candidates が検討されたが viable なものが無い場合に使う。`Blocked` は prerequisite choice 自体が ambiguous または invalid で、phase 8 が ordinary winner selection を安全に実行できない場合に使う。

### Inserted View

```rust
struct InsertedView {
    id: InsertedViewId,
    expression: ExprId,
    target: NormalizedType,
    reason: ViewReason,
    source_written: bool,
}

enum ViewReason {
    SourceQua(SourceRange),
    UniqueInheritancePath(InheritancePath),
    AttributeWeakening(Vec<TypeFactId>),
    TemplateExplicitArgument,
}
```

Inserted views は `ResolvedTypedAst` 上の semantic metadata であり、source edits ではない。

## 診断

Overload diagnostics は typed source syntax 上の semantic diagnostics である。

| Diagnostic Class | Example |
|---|---|
| no match | no candidate accepts the argument types |
| ambiguity | two viable candidates are incomparable |
| template inference | omitted template parameter cannot be inferred uniquely |
| inheritance path | multiple implicit paths to parameter type |
| invalid `qua` | source `qua` is not a proof-free widening |
| redefine | redefinition is not strictly more specific than original |
| declaration conflict | pre-existing same-signature conflict prevents candidate construction |

Diagnostics は次を含むべきである。

- use site の primary span
- declaration spans and import provenance を持つ candidate list
- expected and actual type summaries
- specificity comparison explanation
- 1 つの candidate が unique になる場合の suggested explicit `qua` target
- machine-readable output のための stable candidate ordering

## 増分処理と再現性

Overload cache keys は次を含む。

- `TypedAst` hash
- `TypeFactTable` and `ResolutionTrace` hashes
- visible signature and overload-set hashes
- template definition hashes
- checker and overload schema versions

変更は次を invalidate する。

- argument types or type facts が変わった overload sites
- changed overload sets を参照する sites
- template definitions or constraints が変わった sites
- changed `ResolvedTypedAst` nodes を consume する elaboration、VC、artifact outputs

Ordering は deterministic でなければならない。

- candidates は canonical `SymbolId`、template instantiation key、import provenance の順で sort する
- rejected candidate diagnostics は同じ candidate order を使う
- specificity graph edges は candidate keys で serialize する
- ambiguous candidate sets は stable order で報告する
- inserted views は AST traversal order で emit する

## 検討した代替案

### Checker-Owned Winner Selection

Type checking 中に winners を選択すると IR boundary は減るが、type fact derivation と candidate ordering が混ざる。採用案では checker が facts を計算し、final choice は phase 8 に委ねる。

### Return-Type-Based Overloading

Expected return type を使うと一部の calls は解決できるが、coercions and template inference との circular interactions を生む。採用案では return type を metadata and elaboration のために記録するが、selection からは除外する。

### Global Specificity DAG

すべての overload declarations に対する global DAG は再利用しやすそうに見えるが、specificity は instantiated templates、imported visibility、source-written `qua`、local type facts に依存し得る。採用案では per-site graphs を構築し、必要に応じて comparison summaries を cache する。

### Heuristic Inheritance Path Selection

Shortest path や first path を選べば一部の errors は避けられるが、declaration order に semantic choices を隠してしまう。採用案では複数 paths が利用可能な場合、explicit `qua` を要求する。

## 採用方針

Mizar Evo は type and registration resolution 後の専用 phase として overload resolution を実行する。

Phase 8 は templates を expand し、既存の type facts を使って viable candidates を filter し、per-site specificity graph を構築し、狭い tie-breakers を適用し、可能なら unique winner を選択し、inserted `qua` views を記録する。Output は elaboration and LSP metadata が consume する final source-shaped semantic representation、`ResolvedTypedAst` である。

## 制約と前提

- Phase 8 は type checking and registration closure が enriched type facts を生成した後にだけ開始する。
- Phase 8 は `ResolutionTrace` を読んでよいが、新しい registration-derived facts を追加してはならない。
- Return type は overload winner selection に参加しない。
- Explicit `qua` は proof-free widening だけを disambiguate できる。Narrowing には `reconsider` が必要である。
- Multiple inheritance paths は explicit `qua` を必要とする。
- Failed overload sites は `ResolvedTypedAst` 内に explicit に残り、valid core terms として elaborate されてはならない。
- Phase 8 に入る candidate groups は lexical scope and source-order visibility で filtering 済みでなければならない。
- Same-signature declaration conflicts は signature collection が所有する。ただし phase 8 は影響を受ける use sites にその diagnostics を attach してよい。
- Candidate and diagnostic ordering は reproducible でなければならない。
