これは [00.pipeline_overview.md](./00.pipeline_overview.md) のフェーズ 8 を詳細化します。とくに、候補集合を持つ型付き表層構文と、最終的な overload root、アクティブな refinement set、および挿入された型強制ビューを持つ `ResolvedTypedAst` との境界を定義します。

## 設計判断

### オーバーロード解決は表層意味論を確定させる

前段のフェーズは、シンボルの使用がオーバーロード集合を参照することを知っていてよいです。また型検査は、アリティ・種別・必須の型付け制約によって不可能な候補を除外してよいです。フェーズ 8 は、呼び出し箇所で用いられる最終的な定義を選択できる最初のフェーズです。

オーバーロード解決が所有するもの:

- テンプレート由来の候補の展開
- 拡充された型事実を用いた適用可能性の絞り込み
- 箇所ごとの具体性の比較の構築
- 一意な ordinary overload root の選択
- その root に接続された互換な coherent redefinition の join
- オーバーロード・テンプレート・継承経路の曖昧性の検出
- 選択された候補が必要とする最終的な `qua` ビューの挿入
- `ResolvedTypedAst` の構築

オーバーロード解決が所有しないもの:

- 名前の参照やインポートの可視性
- 型表現の正規化
- cluster や registration の閉包
- `reconsider` または registration の正しさの証明の解消
- 選択された定義の中核論理への低位化

これにより、フェーズ 8 を決定的かつソースの形を保ったものにし、各解決済み使用箇所について単一の semantic root identity と、必要な場合には明示的な active-refinement set を elaboration へ渡せます。

### 候補の構築は来歴を保つ

リゾルバとチェッカは、解決済みの表層参照でキー化された候補グループを提供します。フェーズ 8 は、それらを具体的なオーバーロード候補に変換します。

各候補は次を記録します。

- 定義元の `SymbolId`
- ordinary definition root の `SymbolId`
- 宣言の種別とアリティ
- パラメータの `NormalizedType`
- オーバーロード選択には用いない返り値型のメタデータ
- ソースとインポートの来歴
- テンプレート実体化に由来するかどうか
- coherence 義務が受理された coherent redefinition に由来するかどうか
- 引数からパラメータへの、必要となる暗黙ビュー
- その候補が外された理由を説明する保留中の診断

返り値型はオーバーロード選択に参加しません。同じ引数型シグネチャを持つ 2 つの ordinary 定義は、返り値型に関わらず、オーバーロードの選択肢ではなく宣言の衝突です。それらの衝突はシグネチャ収集の際に検出されます。フェーズ 8 は使用箇所で既存の診断を表示してよいですが、衝突する宣言をオーバーロード候補として再分類してはなりません。

フェーズ 8 に渡される候補グループは、ソース順とスコープの可視性をすでに尊重していなければなりません。現在のモジュールの修飾されていない参照では、使用箇所より前で可視な宣言だけが含まれます。フェーズ 8 は診断のために候補の来歴を検証してよいですが、`SymbolEnv` 全体を走査して候補集合を広げることはありません。

### テンプレート展開は通常の候補の順序付けより先に行う

テンプレートの取り扱いには 2 つの異なるフェーズがあります。

1. テンプレートパラメータの推論、または明示的なテンプレート引数の検査が、具体的な候補を生成する。
2. 標準的なオーバーロードの順序付けが、具体化された候補と通常の非テンプレート候補をまとめて処理する。

この 2 つのステップを交互に行ってはなりません。テンプレート推論は、省略されたテンプレートパラメータを、厳密な引数型と mode 階層の制約から束縛します。cluster 展開は用いません。cluster 由来の選択肢が、推論されたパラメータを一意でなくし得るためです。テンプレート候補が具体化された後、通常の適用可能性と具体性の判定は、完全な型事実と cluster 閉包を用いてよいです。

テンプレート由来の候補と非テンプレートの候補がそれ以外の点で同点の場合、非テンプレートの候補が勝ちます。これにより、ライブラリは不要な曖昧性を生むことなく明示的な特殊化を提供できます。

### 適用可能性は型事実を用い、新たな推論は行わない

候補は、各引数が対応するパラメータ型へ証明なしの拡大で見なせるとき、適用可能です。

- 同一の正規化された型
- 一意な継承経路による mode または構造体のアップキャスト
- 拡充された型事実と registration 閉包によって正当化される属性の弱化
- ソースが与えた明示的な `qua` ビュー

フェーズ 8 は、フェーズ 6-7 が生成した `TypeFactTable`・`ResolutionTrace`・型強制候補を参照してよいです。新しい cluster 事実を導出したり registration を有効化したりしてはなりません。適用可能性の検査に必要な事実が存在しない場合は、型環境を拡張するのではなく、候補を却下するか、診断付きでブロックします。

### 具体性は前順序である

選択される ordinary root は、許可されたタイブレーカ後の一意な極大 viable
root です。root が極大であるとは、他の viable ordinary root が厳密により
特異ではないことを意味します。同じ root に接続された redefinition は、
競合する ordinary root ではありません。

フェーズ 8 はこれを、箇所ごとの具体性グラフとして表現します。

- ノードは ordinary root ごとにグループ化された、適用可能な具体化された候補。
- 辺 `A -> B` は、root `A` が root `B` と同等以上に具体的であることを意味する。
- 厳密性は、`A -> B` が存在し `B -> A` が存在しない場合に導かれる。
- 同点の候補は、仕様化されたタイブレーカが適用されない限り、同等のまま残る。

選択される ordinary root は、許可されたタイブレーカを適用した後の一意な極大要素でなければなりません。viable candidate がなければその箇所は `NoMatch` のままです。比較不能または相互具体性による同値のため、相異なる root が 2 つ以上極大のままなら、その箇所は曖昧性の診断を伴って未解決のまま残ります。root が選択された後、フェーズ 8 はアクティブな same-root redefinition set を計算し、互換な result facts を join します。

許可されるタイブレーカは意図的に狭くしています。

- パラメータベクトルがそれ以外の点で同等のとき、非テンプレートの候補がテンプレート由来の候補に勝つ。
- ソースローカルな宣言は、引数型シグネチャが同一であり、かつそのシャドーイングが名前／シグネチャ収集によってすでに受理されている場合にのみ、インポートされた宣言をシャドーイングする。
- template 宣言制約の厳しさは、template 展開後の tie-breaker ではない。
- 返り値型を用いるタイブレーカは存在しない。

Same-root redefinition はタイブレーカではありません。`RefinementJoin` により、選択された root の適用可能で受理済みの refinement は、payload が互いに join できる場合すべてアクティブになります。

### `qua` は出力で明示化される唯一の暗黙ビューである

ソースの `qua` 式は保持されます。さらにフェーズ 8 は、選択された候補が後続フェーズに明示的に見せる必要のある、証明なしの拡大ビューを要求する場合、暗黙の `qua` ビューを `ResolvedTypedAst` に挿入してよいです。

挿入されるビューは、次に対して許可されます。

- 一意な構造体継承のアップキャスト
- すでに記録された型事実による属性の弱化
- 具体的な候補が選択された後の、テンプレートの明示引数の適用可能性

挿入されるビューは縮小には用いられません。縮小は `reconsider` に属し、型検査によってすでに証明義務または受理済みのローカルな事実を持っていなければなりません。

挿入される各ビューは次を記録します。

- ソースの引数式
- 対象のパラメータ型
- 継承経路または型事実による説明
- ソースに書かれたものか、コンパイラが挿入したものか
- `@show_resolution` および LSP メタデータ用の診断の来歴

### 複数の継承経路は曖昧性であり、探索ではない

引数がパラメータ型へ複数の継承経路で到達できる場合、フェーズ 8 はヒューリスティックに経路を選びません。ソースが `qua` で経路を選択していない限り、その候補は継承経路の曖昧性によってブロックされます。

これはオーバーロードの曖昧性とは異なります。単一の候補であっても、その必要とする引数ビューが曖昧であるためにブロックされ得ます。

### 再定義は refinement group を形成する

`redefine` 宣言は、それが refine する ordinary definition root でタグ付けされた候補を提供します。シグネチャ収集と型検査は、元の定義との関係を記録していなければならず、受理済み coherence 義務だけが解決に参加できます。

フェーズ 8 は、`redefine` を無関係な ordinary overload root として扱いません。まず通常の具体性順序により一意な ordinary root を選択し、その後、その root に接続された適用可能で受理済みの redefinition を集めます。

- active refinement がない場合、root の通常の result metadata が公開される。
- active refinement が 1 つの場合、その refinement の metadata が公開される。
- active refinement が複数ある場合、result facts が互換であれば join される。
- joined facts が互換しない場合、source order にフォールバックせず refinement 診断を生成する。

フェーズ 8 は redefine coherence を証明しません。coherence は初期義務として表現され、後続の証明フェーズが扱います。pending または failed redefinition は `RefinementJoin` からは見えません。

### 出力は失敗した箇所を保たなければならない

`ResolvedTypedAst` は、回復可能なエラーのために、明示的な未解決オーバーロードノードを含むべきです。elaboration は、捏造された解決結果を受け取るのではなく、それらのノードをスキップするか穏当に劣化させなければなりません。

これにより、不正な選択を中核論理に入れることなく、ホバー・候補一覧・`@show_resolution`・`qua` を追加するコードアクションといった LSP 機能を提供できます。

## 解決パイプライン

### ステップ 1: 候補箇所の収集

入力:

- `TypedAst`
- 名前解決と型検査からの候補グループ
- `SymbolEnv` 由来の、ソース順で絞り込まれた可視なシンボルシグネチャ
- 拡充された型事実と `ResolutionTrace`

出力:

- `OverloadSite`
- 初期の候補一覧
- 箇所レベルの期待される型のメタデータ

責務:

- オーバーロードされた functor・述語・属性・mode・セレクタ・テンプレートの使用を見つける。
- ソース範囲と AST ノード ID を付与する。
- 修飾名とインポートの来歴を保持する。
- 現在のモジュールの候補が使用箇所の可視性で絞り込み済みであることを保証する。
- 名前や型の前提が回復不能に失敗した箇所をスキップする。

### ステップ 2: テンプレートの展開

入力:

- オーバーロード箇所
- テンプレート定義と明示的なテンプレート引数
- `TypedAst` 由来の厳密な引数型

出力:

- 具体化されたテンプレート由来の候補
- テンプレート推論の診断

責務:

- 省略されたテンプレートパラメータを厳密な引数型から推論する。
- 推論中はテンプレート制約を mode 階層のみで検査する。
- 成功したテンプレートを具体的な候補シグネチャへ実体化する。
- 曖昧または矛盾するテンプレート推論を却下する。
- 後のタイブレークのために、候補をテンプレート由来として印付けする。

### ステップ 3: 適用可能な候補の絞り込み

入力:

- 具体化された候補
- 引数型
- 拡充された型事実
- 型強制候補のテーブル
- ソースに書かれた `qua` ビュー

出力:

- 適用可能な候補
- 却下された候補の説明
- ブロックされた候補の説明

責務:

- アリティと宣言の種別を検査する。
- 引数からパラメータへの互換性を検査する。
- 欠けている事実と不正な縮小を検出する。
- 曖昧な継承経路を検出する。
- 適用可能な候補について、必要なビュー計画を算出する。

### ステップ 4: 具体性グラフの構築

入力:

- 適用可能な候補
- 正規化されたパラメータ型
- 型閉包のサマリ

出力:

- `SpecificityGraph`
- 同値類
- 曖昧な箇所のための比較診断

責務:

- 候補を位置ごとに比較する。
- 「同等以上に具体的」の辺を算出する。
- 比較不能な候補を検出する。
- 比較が成功または失敗した理由を記録する。
- 型の比較の後に、許可されたタイブレーカを適用する。

### ステップ 5: root の選択、refinement の join、およびビューの挿入

入力:

- 具体性グラフ
- 適用可能な候補
- ビュー計画

出力:

- `OverloadResolutionTable`
- `CoercionInsertionTable`
- オーバーロードの診断

責務:

- 一意な極大 ordinary root を選択する。
- 互換な same-root redefinition を join する。
- 一致なし、または曖昧性のエラーを報告する。
- 選択された候補のためにコンパイラ生成の `qua` ビューを挿入する。
- ソースに書かれた `qua` ノードを保持する。
- 診断と LSP メタデータのために候補一覧を記録する。

### ステップ 6: `ResolvedTypedAst` の構築

入力:

- `TypedAst`
- 選択された overload root と active refinement set
- 挿入されたビュー
- 拡充された型事実と解決トレース

出力:

- `ResolvedTypedAst`
- 最終的なオーバーロードのメタデータ
- 回復可能なオーバーロードの診断

責務:

- 候補グループのノードを解決済み使用のノードへ書き換える。
- 最終的な定義の `SymbolId` を付与する。
- 最終的な引数ビューのメタデータを付与する。
- 回復可能な失敗のために未解決オーバーロードノードを保持する。
- elaboration とアーティファクトのために安定したメタデータを生成する。

## インターフェイス定義

### オーバーロード箇所

```rust
struct OverloadSite {
    id: OverloadSiteId,
    node: AstNodeId,
    kind: OverloadSiteKind,
    name_ref: SymbolRef,
    source_range: SourceRange,
    arguments: Vec<ExprId>,
    expected: Option<ExpectedType>,
}
```

`OverloadSite` は、最終的な候補選択が必要になり得る、ソースの形を保った使用箇所です。

### 候補

```rust
struct OverloadCandidate {
    id: CandidateId,
    symbol: SymbolId,
    origin: CandidateOrigin,
    params: Vec<NormalizedType>,
    result: Option<NormalizedType>,
    root: SymbolId,
    coherence: Option<CoherenceStatus>,
    template: Option<TemplateInstantiation>,
    provenance: CandidateProvenance,
}

enum CandidateOrigin {
    Ordinary,
    Redefinition { root: SymbolId, original: SymbolId },
    TemplateDerived { template: SymbolId },
}

enum CoherenceStatus {
    Accepted,
    Pending,
    Rejected,
}
```

`result` はメタデータ、refinement joining、および後の elaboration のために記録されます。ordinary root の選択には参加しません。
`coherence = Some(Accepted)` を持つ候補だけが active redefinition として参加できます。pending または rejected redefinition は診断には残ってよいですが、`RefinementJoin` からは見えません。

### 適用可能性

```rust
struct ViableCandidate {
    candidate: CandidateId,
    argument_views: Vec<ArgumentView>,
}

enum CandidateRejection {
    ArityMismatch,
    KindMismatch,
    TypeMismatch { argument: ExprId, expected: NormalizedType, actual: NormalizedType },
    AmbiguousInheritancePath { argument: ExprId, target: NormalizedType, paths: Vec<InheritancePath> },
    MissingTypeFact { argument: ExprId, required: TypePredicate },
}
```

ブロックされた候補は、勝てなくても診断のために保持します。

### 具体性グラフ

```rust
struct SpecificityGraph {
    site: OverloadSiteId,
    candidates: Vec<CandidateId>,
    edges: Vec<SpecificityEdge>,
    tie_breaks: Vec<TieBreak>,
}

struct SpecificityEdge {
    more_specific: CandidateId,
    less_specific: CandidateId,
    reasons: Vec<SpecificityReason>,
}
```

グラフはオーバーロード箇所ごとに作られます。宣言全体に対する大域的な順序ではありません。

### 解決結果

```rust
enum OverloadResolution {
    Resolved {
        site: OverloadSiteId,
        root: CandidateId,
        active_refinements: Vec<CandidateId>,
        exposed_result: Option<ExposedResultPayload>,
        inserted_views: Vec<InsertedViewId>,
    },
    NoMatch {
        site: OverloadSiteId,
        rejected: Vec<CandidateRejection>,
    },
    Ambiguous {
        site: OverloadSiteId,
        candidates: Vec<CandidateId>,
        graph: SpecificityGraph,
    },
    IncompatibleRefinementJoin {
        site: OverloadSiteId,
        root: CandidateId,
        refinements: Vec<CandidateId>,
        reason: RefinementJoinFailure,
    },
    Blocked {
        site: OverloadSiteId,
        reason: BlockedReason,
        candidates: Vec<CandidateId>,
    },
}

enum BlockedReason {
    TemplateInferenceFailed,
    AmbiguousTemplateInference,
    AmbiguousInheritancePath,
    PreExistingDeclarationConflict,
}

enum RefinementJoinFailure {
    IncompatibleResultRadix,
    ContradictoryResultAttributes,
    NoUniqueJoinedType,
    IncompatibleMetadata,
}
```

`NoMatch` は、候補は検討されたものの適用可能なものがない場合に用います。`IncompatibleRefinementJoin` は、ordinary root は一意だが same-root active redefinition が公開する result facts を join できない場合に用います。`Blocked` は、前提となる選択そのものが曖昧または不正で、フェーズ 8 が通常の root 選択を安全に実行できない場合に用います。

`ExposedResultPayload` は、active refinement の適用後に公開される結果型および metadata facts を記録します。functor では通常 joined `NormalizedType` を含みます。predicate と attribute では、真理値を保つ metadata または attribute facts だけを含む場合があります。

### 挿入されたビュー

```rust
struct InsertedView {
    id: InsertedViewId,
    expression: ExprId,
    target: NormalizedType,
    reason: ViewReason,
    source_written: bool,
}

enum ViewReason {
    SourceQua(SourceRange),
    UniqueInheritancePath(InheritancePath),
    AttributeWeakening(Vec<TypeFactId>),
    TemplateExplicitArgument,
}
```

挿入されたビューは `ResolvedTypedAst` 上の意味論的メタデータであり、ソースの編集ではありません。

## 診断

オーバーロードの診断は、型付きのソース構文に対する意味論的な診断です。

| 診断クラス | 例 |
|---|---|
| 一致なし | どの候補も引数型を受け付けない |
| 曖昧性 | 許可されたタイブレーカ後も 2 つ以上の相異なる viable ordinary root が極大のまま残る |
| incompatible refinement join | same-root redefinition が互換しない joined result facts を公開する |
| テンプレート推論 | 省略されたテンプレートパラメータを一意に推論できない |
| 継承経路 | パラメータ型への暗黙経路が複数ある |
| 不正な `qua` | ソースの `qua` が証明なしの拡大ではない |
| redefine | 再定義が元の定義より厳密に具体的でない |
| 宣言の衝突 | 既存の同一シグネチャの衝突により候補を構築できない |

診断には次を含めるべきです。

- 使用箇所における主たる位置情報
- 宣言の位置情報とインポートの来歴を持つ候補一覧
- 期待される型と実際の型の要約
- 具体性の比較の説明
- 関連する場合は active redefinition set と joined result facts
- 1 つの候補が一意になる場合の、提案する明示的な `qua` の対象
- 機械可読な出力のための安定した候補の順序

## 増分処理と再現性

オーバーロードのキャッシュキーには次を含めます。

- `TypedAst` のハッシュ
- `TypeFactTable` と `ResolutionTrace` のハッシュ
- 可視なシグネチャとオーバーロード集合のハッシュ
- テンプレート定義のハッシュ
- チェッカとオーバーロードのスキーマバージョン

変更は次を無効化します。

- 引数型または型事実が変わったオーバーロード箇所
- 変更されたオーバーロード集合を参照する箇所
- テンプレート定義または制約が変わった箇所
- 変更された `ResolvedTypedAst` ノードを消費する elaboration・VC・アーティファクト出力

順序は決定的でなければなりません。

- 候補は、正準的な `SymbolId`・テンプレート実体化キー・インポートの来歴の順でソートする。
- 却下された候補の診断は、同じ候補順を用いる。
- 具体性グラフの辺は候補キーでシリアライズする。
- 曖昧な候補集合は安定した順序で報告する。
- 挿入されたビューは AST 走査順で出力する。

## 検討した代替案

### チェッカが所有する root 選択

型検査中に ordinary root を選択すると IR の境界は減りますが、型事実の導出と候補の順序付けおよび refinement joining が混ざります。採用した設計では、チェッカが事実を計算し、最終的な root 選択と active-refinement 計算をフェーズ 8 に委ねます。

### 返り値型に基づくオーバーロード

期待される返り値型を用いると一部の呼び出しは解決できますが、型強制やテンプレート推論との循環的な相互作用を生みます。採用した設計では、返り値型をメタデータと elaboration のために記録しますが、選択からは除外します。

### 大域的な具体性 DAG

すべてのオーバーロード宣言に対する大域的な DAG は再利用しやすそうに見えますが、具体性は実体化されたテンプレート・インポートの可視性・ソースに書かれた `qua`・ローカルな型事実に依存し得ます。採用した設計では、箇所ごとのグラフを構築し、必要に応じて比較のサマリをキャッシュします。

### ヒューリスティックな継承経路の選択

最短経路や最初の経路を選べば一部のエラーは避けられますが、意味論的な選択を宣言順に隠してしまいます。採用した設計では、複数の経路が利用可能な場合に明示的な `qua` を要求します。

## 採用方針

Mizar Evo は、型と registration の解決の後の専用フェーズとして、オーバーロード解決を実行します。

フェーズ 8 はテンプレートを展開し、既存の型事実を用いて適用可能な候補を絞り込み、箇所ごとの具体性グラフを構築し、狭いタイブレーカを適用し、可能なら一意な ordinary root を選択し、互換な same-root redefinition を join して、挿入された `qua` ビューを記録します。出力は、elaboration と LSP メタデータが消費する、最終的な、ソースの形を保った意味論的表現である `ResolvedTypedAst` です。

## 制約と前提

- フェーズ 8 は、型検査と registration 閉包が拡充された型事実を生成した後にのみ開始する。
- フェーズ 8 は `ResolutionTrace` を読んでよいが、新しい registration 由来の事実を追加してはならない。
- 返り値型は ordinary overload root 選択に参加しない。same-root refinement join の互換性にのみ参加する。
- 明示的な `qua` は証明なしの拡大だけを曖昧性解消できる。縮小には `reconsider` が必要である。
- 複数の継承経路は明示的な `qua` を必要とする。
- 失敗したオーバーロード箇所は `ResolvedTypedAst` 内に明示的に残り、妥当な中核項として elaboration されてはならない。
- フェーズ 8 に入る候補グループは、字句スコープとソース順の可視性で絞り込み済みでなければならない。
- 同一シグネチャの宣言の衝突はシグネチャ収集が所有する。ただしフェーズ 8 は、影響を受ける使用箇所にその診断を結び付けてよい。
- 候補と診断の順序は再現可能でなければならない。
