# アーキテクチャ: AI エージェントインターフェイスと文脈プロトコル

> 正本は英語です。英語版: [../en/21.ai_agent_interface.md](../en/21.ai_agent_interface.md)。

## 目的

この文書は、AI エージェントが Mizar Evo プロジェクトを**安全に操作する**ための **tooling contract** を定義します。すなわち、エージェントがどの編集を提案してよいか、それを実行するにはどの権限が必要か、そしてエージェントがどのように文脈を読み、パッチを提案し、検証器の結果を受け取るかを定めます。

これは意図的に**言語仕様の章ではありません**。Mizar Evo の言語、表層構文、FQN、カーネルの健全性は別の場所で定義されており、ここに書かれる内容によって変わることはありません。この文書が記述するのは検証器の*周囲*の層であり、「AI が Mizar Evo のアーティファクトを読める」（passive AI friendliness）を、「AI が明示的な安全制御のもとで Mizar Evo プロジェクトを検索・説明・編集・再検証できる」（active AI operability）へと引き上げるものです。

この文書が規範的に定めるのは、下層の wire schema にも成熟した検証器にも依存しない 2 つのポリシー項目のみです。それ以外はすべて非規範のロードマップとして枠組みだけを記述し、wire 形式は、それが包むアーティファクトが安定したのちに凍結できるようにします。

## スコープと規範性

この文書は規範的なポリシーと非規範のロードマップを混在させています。区別は意図的であり、節ごとに明示します。

| 節 | 状態 |
|---|---|
| 信頼境界と安全編集クラス | **規範** |
| 認可スコープ | **スコープ語彙とその編集クラスへの対応づけは規範**。transport／OAuth 束縛は非規範 |
| 設計原則 | 非規範の枠組み |
| 枠組みとロードマップ | 非規範。各項目はブロッキング依存を明示 |

規範化する 2 項目——Green/Yellow/Red の編集クラスと、認可スコープ語彙——はポリシーです。これらは patch 検証 API が存在する前から意味を持ちます。なぜなら、実装上どう配線されるかにかかわらず、エージェントが何を*試みてよいか*を制約するからです。これらは [08.reasoning_boundary.md](./08.reasoning_boundary.md) と、historical name として `require_kernel_certificates` と呼ばれる kernel-evidence release policy を、エージェント向けの表面へ自然に延長したものです。

それ以外——context budget モデル、resource URI スキーム、theorem/symbol index、obligation 文脈抽出、patch wire schema、patch 検証 API、proof-repair 状態機械、MCP 束縛、dataset export、ベンチマーク——は枠組みとしてのみ記述します。**ここでは wire schema を一切凍結しません。** wire 形式は、それが包む下層アーティファクト（`*.mizir.json`、kernel evidence、`ObligationAnchor`、proof witness）自体が安定したのちにのみ凍結します。patch verification API は [22.incremental_verification_contract.md](./22.incremental_verification_contract.md) が定義する `ObligationAnchor` 契約に依存します。

## 設計原則

以下の原則は、続く規範的な節のための非規範の枠組みです。

- **正しさは検証器・ATP・カーネルに残す。** エージェントは検索・説明・計画・編集を行うが、証明の妥当性を自称しない。これが [08.reasoning_boundary.md](./08.reasoning_boundary.md) の核心です。
- **編集は小さく検証可能に。自由文の全面書き換えにしない。** エージェントの操作単位は単一の有界な編集（`by` への citation 追加、`qua` 挿入、`assert` 追加、`import` 追加）であり、各編集は検証器が独立に受理または棄却します。
- **ライブラリ全体ではなく最小限の文脈を読む。** エージェントには、[README.md](./README.md) の常駐集合メモリモデルに整合する、有界で obligation 単位の射影を渡します。
- **apply の前に dry-run。** パッチはソースへ適用する前に必ず検証（dry-run）します。
- **権限は明示的かつ最小権限。** 信頼境界を動かしうる能力は、通常のエージェントには決して付与しません。

## 信頼境界と安全編集クラス

**状態: 規範。**

AI エージェントが提案しうるすべての編集を、3 つの安全クラスのいずれかに分類します。クラスは*ポリシー*を支配します（自動提案してよいか、人間レビューが必須か、エージェント単独で実行してよいか）。これは検証を**置き換えるものではありません**。Green や Yellow の編集も、検証器が受理して初めて受理され、その受理は依然として [08.reasoning_boundary.md](./08.reasoning_boundary.md) のカーネル規律に従います。

### Green — 自動提案してよい

意味を変えにくく、その効果を検証器が完全に検査する編集:

- `by` 句に lemma を追加する;
- `by` 句から未使用の citation を削除する;
- overload を解決する `qua` coercion を挿入する;
- 情報アノテーション（`@show_type`、`@show_resolution`、`@show_thesis`、`@proof_hint`）を挿入する;
- コメント・doc comment を追加する;
- `mizar refine` を適用して citation を精密化する。

### Yellow — 提案してよいが、検証成功後も人間確認が望ましい

検証器が検査できるが、証明構造・探索空間・ライブラリ設計を変える編集:

- `assert` を追加して goal をステップに分割する;
- local lemma を追加する;
- `import` を追加する;
- registration を追加する;
- cluster を追加する;
- algorithm invariant を追加する。

これらは検証が通れば論理的には健全ですが、設計上の影響（可視シンボルの増加、探索空間の変化、ライブラリ形状）があるため、人間が確認すべきです。

### Red — AI エージェント単独では禁止

信頼境界を動かす・弱める・迂回する編集:

- axiom を追加する;
- theorem statement を弱める;
- definition を変更する;
- `requires`／`ensures` 契約を弱める;
- `proof_status` を手で編集する;
- `proof_witness` を直接編集する;
- カーネル設定を緩める;
- `require_kernel_certificates = false`、または kernel evidence acceptance を迂回する同等の switch を設定する。

AI エージェントはカーネルの*外側*では何を提案してもよいですが、カーネルが検査する対象や、稼働中の検証器ポリシーが信頼する対象を決して弱めてはなりません。Red の操作は、goal・axiom・definition・受理ポリシーを変えることで goal を「証明」してしまえる操作にちょうど一致します。これらは後述の `unsafe-edit` スコープを必要とし、それは通常のエージェントには付与されず、自動提案の対象から完全に外れます。

最も頻度の高いエージェントの失敗様式——goal を弱める、theorem statement を変える、都合のよい axiom を足す、シンボルをこっそり再定義する——はすべて Red であり、この分類は、通常のエージェントにとってそれらを表現不能にするために存在します。

> [！注記]
> **セマンティックバージョニングとの関係。** Red 編集クラスと MAJOR バージョンバンプ（仕様 §23.2、§23.10 の `mizar semver-check` が報告）は、別個の軸であり、重なりはあっても一致しません。Red はエージェントが何を試みてよいかへの*認可*ゲートであり、MAJOR は公開された変更が何を必要とするかへの*互換性*判定です。そして互換性判定は **方向依存** です。`requires` の強化や `ensures` の弱化は MAJOR ですが、`requires` の弱化は MINOR にすぎません（§23.2）。両軸は双方向に乖離します。一部の Red 編集——theorem statement の変更、definition の変更——は確実に MAJOR です。しかし registration や cluster の追加のような Yellow 編集も、overload 解決をシフトさせる場合は MAJOR になりえます（§23.2 注記）。逆にエクスポートの削除は Red ではないのに MAJOR です。したがって編集クラスはバージョン判定を**決定しません**。エージェント（またはそのレビューア）は、Green/Yellow/Red のラベルからバンプを推測するのではなく、変更を分類するために `mizar semver-check` を実行しなければなりません。

## 認可スコープ

**状態: 語彙とその編集クラスへの対応づけは規範。transport および OAuth 束縛は非規範であり遅延（「枠組みとロードマップ」参照）。**

エージェントの能力は、固定されたスコープ語彙で命名します。デプロイメントはこれらのスコープの部分集合をエージェントに付与し、スコープは transport によらずエージェントの可能操作を制限します。

| スコープ | 付与するもの | 到達しうる最上位編集クラス |
|---|---|---|
| `read` | source・artifact・theorem・obligation の読み取り | なし（読み取り専用） |
| `explain` | explanation API（`explain-type`、`explain-overload`、`explain-failure` 等）の呼び出し | なし（読み取り専用） |
| `verify` | 適用せずに patch 検証（dry-run）を実行 | なし（ソースを変更しない） |
| `write` | 検証済み patch をソースへ適用 | Green、および人間レビュー付きの Yellow |
| `refine` | citation の refine／minimize を適用 | Green |
| `publish` | パッケージを publish | 直接にはなし。gated な操作 |
| `unsafe-edit` | axiom・definition・theorem statement・契約の編集 | Red |

規則:

- `read`・`explain`・`verify` はソースに対して読み取り専用であり、プロジェクト状態を変更しません。
- `write` は Green 編集を適用でき、Yellow 編集も適用できますが、Yellow は上記の編集クラスポリシーに従い人間レビューの対象のままです。`write` は Red には決して到達しません。
- `unsafe-edit` は Red クラスに到達する唯一のスコープです。**通常の AI エージェントには付与しません。**
- `publish` は人間承認を必須とし、エージェントが自動で行使することはありません。
- 既定の「AI-safe」エージェントプロファイルは、多くとも `read, explain, verify, refine` を付与し、ソース変更を意図する場合に `write` を加えます。`unsafe-edit` と `publish` は拒否します。

この語彙は、リモート MCP サーバの OAuth スコープや、ローカルの `--safe-mode` allow/deny リストへ対応づけることを意図していますが、それらの束縛は遅延します。ここで規範なのは、上記の名前と、それらの編集クラスへの関係のみです。

## 枠組みとロードマップ（非規範）

以下の構成要素は AI エージェントインターフェイスを完成させますが、本改訂では**規範的には**定めません。各項目は、語彙を固定し、その wire 形式を凍結する前に解消すべき依存を記録するのに足るだけ素描します。

これらすべてに共通する実装上の現実があります。検証器とそのアーティファクト生成はまだ成熟途上であり、patch の dry-run も theorem index も、それが包む検証器とアーティファクトが存在するまで実体を持てません。これが、wire-schema・MCP 束縛・ベンチマークの項目を最後に並べる理由です——概念的な慎重さだけでなく、それらが包む下層アーティファクトがまだ動いていないからです。

- **Context Budget モデル。** エージェントはライブラリ全体ではなく、トークン有界の射影を受け取る。要求は budget を持ち、応答は推定値を返す。（`doc/spec/en/23` §23.7.5 にある有界 local subgraph 抽出の延長。）
- **AI Resource URI スキーム。** 読み取り専用の文脈を URI で参照可能にする（例: `mizar://obligation/...`、`mizar://theorem/...`、`mizar://diagnostic/...`）。MCP Resources に対応。
- **Theorem および Symbol Index アーティファクト。** 検索向けの `theorem-index.jsonl`／`symbol-index.jsonl`。theorem／definition／registration ごとに次を出力する: `fqn`・`label`・`module`・`source_range`、複数形の `statement`——`surface`（人間が書いた Mizar 表記）・`canonical`（FQN 展開後の安定表記）・`normalized`（交換・結合・binder 名などを正規化した検索用表記）・`fol`/TPTP（symbolic／ATP 検索用の論理形式）、`symbols`（`head_symbols`・`all_symbols`・`types`）、`dependencies`（`used_axioms`・`imports`）、`search_text`（doc・keywords・latex・自然言語検索用に合成した embedding text）、`trust`（`proof_status`・package・version）。statement を複数形で持つことが、表面記法・synonym・overload・module alias にエージェントが振り回されないための鍵となる。**依存:** 充実フィールド（`head_symbols`、normalized/FOL statement、embedding text、合成 `search_text`）は現状の `*.mizir.json` には*ない*——直接得られるのは `exports` の FQN・`source_range` と obligations の `used_axioms` まで（§23.5.1）。残りは source comments／documentation extraction からの抽出、または新規抽出経路を要する。薄い初版は既存フィールド（FQN、`source_range`、`used_axioms`）のみから構築し、FOL／normalized／embedding 系は後段。
- **Theorem および Symbol 検索。** index 上の二系統の補完的検索。形式数学では自然言語的な近さと論理的な関連性がしばしば乖離するため。*Symbolic search* は goal の head symbol、goal に現れる FQN／型／attribute、conclusion pattern、rewrite direction、hypothesis shape をキーにする（例: `mizar search theorem --symbols std.nat.add,std.nat.div --conclusion-head "=" --type std.nat.Nat --json`）。*Semantic／自然言語検索* は合成した embedding text（source comment そのものではなく合成説明文）をキーにし、「名前は知らないが意味は説明できる」補題を探せるようにする。両者は代替関係になく、エージェントは両方を組み合わせられるべき。
- **Import Suggestion API。** Yellow の「import を追加」編集とは別の検索／提案 surface: `mizar search import --symbol Group`／`--theorem add_comm`、`mizar suggest import --for-obligation <obligation-anchor>`。obligation handle は抽象的で安定した anchor であり、生の snapshot-local `VcId` ではありません。各候補は `module`・`fqn`・`already_visible`・追加すべき `import_statement`、加えて `risk` と `added_visible_symbols_count` を返し、エージェントが過度に広い import で active lexical environment と探索空間を肥大させないようにする。prelude 限定の import モデル（`doc/spec/en/12`）に整合。
- **Obligation 文脈抽出。** `mizar context obligation` が obligation 単位の goal・hypotheses・candidate facts・diagnostics・`allowed_patch_kinds` を束ねる。**依存:** これは主に既存の有界抽出にトークン予算を加えた表面整備であり、`allowed_patch_kinds` フィールドこそ、上記の編集クラスが obligation 単位の制約になる箇所。
- **診断 Recovery Action と修復フロー。** エージェント向けの経路は `diagnostic → explanation → recovery_action → patch proposal → verify.patch`。`recovery_action` は [12.diagnostics_and_lsp.md](./12.diagnostics_and_lsp.md) の `FixSuggestion`／コードアクションモデルの構造化拡張であり（内部設計 `03.diagnostics_model_and_lsp_bridge` も参照）、各々が patch の `kind`（例: `insert_qua`・`qualify_symbol`）、対象 `range`、`replacement_template` および／または候補リストを持つ。これにより診断の machine explanation が、既知の編集クラスに属する提案可能な patch へ直接対応づく。**依存:** 具体スキーマはここでは非規範であり、patch wire schema とともに凍結する。
- **Patch Proposal 形式。** 対象 obligation と source hash に対する単一の有界編集を記述する JSON patch。**依存:** [22.incremental_verification_contract.md](./22.incremental_verification_contract.md) の `ObligationAnchor` 契約を要する。`VcId` は 1 つの `BuildSnapshot` 内でだけ決定的であり、編集をまたぐ patch target identity にはできない。
- **Patch 検証 API。** 受理／棄却と changed obligations・new diagnostics を返す dry-run。`verify` スコープでのみ到達可能。下位 artifact、cross-edit obligation anchor、proof witness format が安定するまで patch wire schema は凍結しない。
- **Proof Repair 状態機械。** `OPEN_OBLIGATION → CANDIDATE_FACTS → PATCH_DRY_RUN → {ACCEPTED, REJECTED} → KERNEL_VERIFIED → REFINED → DONE`。各状態の許可／禁止操作を編集クラスに整合させる。
- **MCP 束縛。** 上記を MCP の Resources（読み取り専用文脈）・Tools（search/explain/verify/apply/refine）・Prompts（人間が選ぶテンプレート）へ対応づける。**遅延。**
- **認可束縛（OAuth／`--safe-mode`）。** スコープ語彙の transport 実装。リモートサーバ向けの OAuth 2.1 protected-resource スコープと、ローカル stdio 向けの明示的 allow/deny `--safe-mode`。**遅延。** いまは語彙のみが規範。
- **Proof-Witness 由来データ。** エージェント学習データに供する、per-axiom な instantiation トレース。**遅延:** proof-witness 形式はまだ tentative（`doc/spec/en/23` §23.5、および [15.kernel_certificate_format.md](./15.kernel_certificate_format.md)）。per-axiom instantiation scoping は既に確立されており、各 axiom の変数スコープは独立している。その scoping を `binder_id` ベースで表現することは、[16.substitution_and_binding.md](./16.substitution_and_binding.md) に整合する*将来方針*であって、確定した形式ではない。
- **Dataset Export 形式。** `goal → used_axioms`、`failed_goal → successful_patch`、`overload_error → qua_fix` 等。corpus と witness の安定まで**遅延**。
- **ベンチマークと指標。** proof-repair・overload-repair・context-budget の各ベンチマーク。**遅延。** 具体的な指標は規範仕様ではなく評価戦略（[20.test_strategy.md](./20.test_strategy.md) の「AI Agent Evaluation」）に置く。

## インターフェイス定義

ここで固定する規範の表面は、編集クラス表とスコープ→編集クラス対応のみです。

### 安全編集クラス

```text
Green  : 自動提案可能、検証器が検査
Yellow : 提案可能。検証成功時も人間レビュー
Red    : 単独のエージェントには禁止。unsafe-edit スコープを要する
```

### スコープ → 編集クラス対応

```text
read        -> 読み取り専用
explain     -> 読み取り専用
verify      -> dry-run のみ（ソース変更なし）
write       -> Green。Yellow は人間レビュー付き
refine      -> Green（citation refine/minimize）
publish     -> gated、人間承認
unsafe-edit -> Red（通常のエージェントには付与しない）
```

その他の wire レベルのインターフェイス（context、patch、検証応答、URI、index）は上の非規範ロードマップで記述し、意図的に未凍結のままとします。

## 制約と前提

- AI エージェントは、カーネルが検査する対象や、稼働中の検証器ポリシーが信頼する対象を決して弱めてはならない。Red 編集は `unsafe-edit` なしには到達不能。
- 安全編集クラスとスコープ語彙はポリシーであり、検証器が存在しなくても機能する。これらは*意図*を制約し、検証は*受理*を制約する。
- 本改訂では wire schema を一切凍結しない。patch・context・index・witness の各形式は、その下層アーティファクト、`ObligationAnchor`、proof witness format が安定したのちにのみ凍結する。
- patch プロトコルは cross-edit 安定 obligation anchor を前提とする。その anchor は patch wire schema の凍結より前に規定しなければならない。
- エージェント向け文脈は有界かつ obligation 単位であり、横断的メモリモデルに整合する。ライブラリ・cluster-db・全トレースの一括ダンプは提供しない。
- 具体的な予算やベンチマーク数値はここでは対象外であり、評価戦略に属する。
