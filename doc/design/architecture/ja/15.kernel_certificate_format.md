## 目的

この文書は、信頼されるカーネルが消費する最終的な証明書の形式を定義します。

これはパイプラインのフェーズ 13-14 を詳細化します。ATP バックエンド・決定的な ATP 前の解消・Mizar 側の証明再構成は根拠を生成してよいですが、カーネルが受理するのは、正規化され再生可能な証明書だけです。

## 設計判断

### 証明書のトップレベル

証明書は正準的かつバージョン管理されたオブジェクトです。

```text
Certificate
  version
  target_vc
  kernel_profile
  imported_axioms
  imported_theorems
  generated_clauses
  substitutions
  resolution_trace
  derived_facts
  final_goal
```

`target_vc` は証明書を 1 つの正準的な検証条件に束縛します。`kernel_profile` は、受理する証明書スキーマ・節のエンコード・ハッシュアルゴリズム・有効な組み込みチェッカを記録します。

### 信頼の範囲

カーネルが検査するのは、次だけです。

- 証明書のパースとスキーマ検証
- 安定した ID とコンテンツハッシュによる、インポートされた事実の利用可能性
- SAT 節の整形性
- 置換・α 変換・自由変数の副条件
- 解決トレースの再生
- 検査済みの節から最終ゴールが導出されること

カーネルは、証明探索・前提選択・オーバーロード解決・cluster 展開・型強制の挿入・大域的な可変コンパイラ状態の参照を行ってはなりません。

### 節の表現

節のシリアライズは正準的でなければなりません。

- 変数は、正規化後の正準的な変数 ID で順序付ける。
- リテラルは、極性・述語／functor シンボル ID・アリティ・正規化された引数のエンコードの順で順序付ける。
- 重複するリテラルは、証明書の正規化中に除去する。
- トートロジーの節は、却下するか、カーネルプロファイルに従って明示的な `tautology` マーカーで表す。
- 安定したハッシュは、正準的なバイトエンコードとスキーマバージョンから計算する。

map／set の反復順序が、節の ID や節のハッシュに影響してはなりません。

### 解決トレース

各解決ステップは次を記録します。

```text
ResolutionStep
  step_id
  parent_clause_a
  parent_clause_b
  pivot_literal
  generated_clause
```

カーネルは両方の親を参照し、pivot が反対の極性で出現することを検査し、親同士を解決し、結果を正準化して `generated_clause` と比較します。

親への参照は、インポートされた節・生成された節・それより前の解決ステップだけを指します。前方参照は不正な証明書です。

### 置換の規則

置換は証明書内の明示的なエントリです。

```text
SubstitutionStep
  source_term
  target_term
  binder_context
  freshness_witnesses
  free_variable_constraints
```

カーネルは、[16.substitution_and_binding.md](./16.substitution_and_binding.md) で定義したとおり、捕獲回避・α 変換・自由変数制約・束縛子の正規化を検証します。

### インポートされた事実

インポートされた公理と定理は、次で参照します。

- パッケージ ID
- モジュールパス
- エクスポートされた項目 ID
- ステートメントのフィンガープリント
- ポリシーが要求する受理済みの証明状態

インポートされた定理が利用不可、フィンガープリントが不一致、または現在のカーネルプロファイルが許さない弱いポリシーでしか受理されていない場合、その証明書は無効です。

### カーネルによる棄却の意味論

棄却の分類は安定した API 値です。

| 分類 | 意味 |
|---|---|
| `malformed_certificate` | スキーマ・バージョン・参照・正準エンコードの失敗 |
| `invalid_substitution` | 捕獲・α 変換・新鮮性・自由変数の規則の失敗 |
| `invalid_sat_proof` | 節または解決の再生の失敗 |
| `unresolved_symbol` | 参照されたシンボル・定理・公理・VC が利用不可 |
| `timeout` | 検査が決定的な資源予算を超過 |
| `resource_exhaustion` | メモリ・節数・トレース長・再帰上限の超過 |

`timeout` と `resource_exhaustion` は証明の受理ではありません。義務は未検証のまま残ります。

## インターフェイス定義

```rust
struct Certificate {
    version: CertificateVersion,
    target_vc: VcFingerprint,
    kernel_profile: KernelProfile,
    imported_axioms: Vec<ImportedFactRef>,
    imported_theorems: Vec<ImportedFactRef>,
    generated_clauses: Vec<Clause>,
    substitutions: Vec<SubstitutionStep>,
    resolution_trace: Vec<ResolutionStep>,
    derived_facts: Vec<DerivedFact>,
    final_goal: GoalRef,
}
```

## 制約と前提

- 証明書の検査は、同一の証明書バイト列と依存アーティファクトに対して決定的でなければならない。
- 証明書のパースは信頼境界の一部であり、小さく、全域的（total）で、資源が制限されていなければならない。
- バックエンド固有の証明形式は、カーネル受理の前にこの形式へ変換する。
- 証明書のハッシュは正準的な内容だけから計算し、ファイルパス・タイムスタンプ・バックエンドのランタイムログを含めない。
