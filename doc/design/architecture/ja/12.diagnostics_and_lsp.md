# アーキテクチャ: 診断と LSP

> 正本は英語です。英語版: [../en/12.diagnostics_and_lsp.md](../en/12.diagnostics_and_lsp.md)。

## 目的

この文書は、Mizar Evo における診断・ソースメタデータ・説明データ・LSP 応答の流れを定義します。

診断はすべてのパイプラインフェーズで生成されますが、フェーズ 15 でツール向けの安定したアーティファクトデータになります。LSP サーバはそれらのアーティファクトと小さなクエリサービスを読み、コンパイラ内部の IR レイアウトに依存せずにエディタフィードバックを提供します。

## 設計判断

### 診断は構造化されたフェーズ出力である

すべてのパイプラインフェーズは、整形済みの文字列ではなく、構造化された `Diagnostic` レコードを出力します。

CLI の出力フォーマッタ・アーティファクトエミッタ・LSP サーバは、同じ正準的なレコードを読みます。これにより、CLI が説明するエラーと、IDE やアーティファクトメタデータが見るエラーがずれることを防ぎます。

### 診断の同一性

各診断は、2 種類の識別子を持ちます。

| 識別子 | スコープ | 目的 |
|---|---|---|
| `DiagnosticCode` | リリースをまたいで安定 | 人間向け・ツール向けの分類。例: `E0201` |
| `DiagnosticId` | 1 つのビルドスナップショット | LSP の診断を説明クエリやコードアクションへ結び付ける |

`DiagnosticCode` は診断レジストリによって恒久的に割り当てられます。`DiagnosticId` はビルドスナップショット内で決定的であり、ソースファイル・コード・主たる位置・フェーズ・位置ごとの安定した索引から導出されます。ツールは、無関係なビルドスナップショットをまたいで `DiagnosticId` を保持してはなりません。

### ソース位置は一元化されている

診断は、その場限りの行・列の計算を保存しません。`SourceUnit` の行マップに基づく `SourceRange` を参照します。

ルール:

- 主たる位置は、利用者が最初に編集すべき箇所である。
- 副次的な位置は、定義・候補・生成された義務・インポートされた事実を示す。
- 生成された義務は、診断がソースレベルの構文要素を指せるよう、`GeneratedFrom` の来歴を持つ。
- LSP の UTF-16 位置は、ソーステキストのスナップショットから、LSP の境界でのみ計算される。

これにより、コンパイラの診断はエディタプロトコルのエンコードの詳細から独立します。

### 診断は部分的な失敗を生き延びる

コンパイラは、致命的なフェーズの失敗の後に意味論的な事実を捏造せず、可能な限り多くの独立した診断を報告すべきです。

フェーズ出力は次を区別します。

- `fatal`: 下流の意味論フェーズはこのフェーズの出力を信頼できない。
- `blocking`: この項目または義務は進められないが、他の項目は続行できる。
- `recoverable`: 後続のフェーズは劣化したメタデータで続行できる。

たとえば、構文解析器のリカバリは回復された構文の上での名前解決を可能にする場合がありますが、インポートグラフの失敗は、影響を受けたモジュールとその逆依存の意味論検査をブロックします。

### 説明データは遅延的かつ上限つきである

型推論・cluster 解決・オーバーロード解決・証明失敗の豊富な説明は、通常の診断よりかなり大きくなり得ます。

既定の診断レコードは、コンパクトな要約と参照を保存します。

- 説明の種別
- 関連する `expr_id`・`vc_id`・`diagnostic_id`
- 説明アーティファクトが出力された場合のパス
- CLI とエディタ表示用の上限つきプレビューテキスト

LSP サーバと `mizar explain` は、詳細な説明を必要に応じて解決します。すべてのトレースの一括抽出は避けます。

### LSP は安定したスナップショットを読む

LSP サーバは、可変なコンパイラの状態ではなく、整合的なビルドスナップショットを読みます。

スナップショットは次から構成されます。

- アーティファクトマニフェスト
- `*.mizir.json` ファイル
- 診断の説明アーティファクト
- 存在する場合の `missing_facts.json`
- 説明クエリが必要とする場合の `cluster-db/` およびインポートスコープのビュー
- 開いているエディタバッファのソーステキストのスナップショット

新しいビルドが完了すると、サーバは以前のスナップショットから新しいスナップショットへアトミックに切り替えます。処理中のリクエストは、古いスナップショットか新しいスナップショットのいずれかを用い、両者を混在させません。

### 開いているバッファはディスク上のソースを上書きする

開いているエディタバッファについては、LSP リクエストはエディタが提供するテキストを現在のソーススナップショットとして用います。ビルドアーティファクトは、依然としてディスク上の最後の検証パスを反映している場合があります。

そのためサーバは、応答に鮮度を付与します。

| 鮮度 | 意味 |
|---|---|
| `current` | 応答は現在の開いているバッファのテキストから計算された |
| `stale_artifact` | 応答は最後に検証されたアーティファクトに由来し、未保存のテキストと一致しない場合がある |
| `pending` | 検証がスケジュールされている、または実行中である |

古いアーティファクト由来の診断は表示し続けてもよいですが、対象範囲が現在のバッファのテキストと一致しない場合、サーバは破壊的なコードアクションを提供すべきではありません。

## 診断パイプライン

### 生成者

各フェーズは、フェーズ固有のメタデータを伴う診断を提供します。

| フェーズ領域 | 典型的な診断 | 追加メタデータ |
|---|---|---|
| フロントエンド | 不正なトークン、構文リカバリ、`end` の欠落 | 期待されるトークン、リカバリ地点 |
| リゾルバ | 未解決の名前、曖昧なインポート、重複シンボル | 候補シンボル、インポートパス |
| チェッカ | 型不一致、sethood の失敗、cluster の失敗 | 推論された型／期待される型、解決トレースの参照 |
| オーバーロード | 曖昧なオーバーロード、`qua` の欠落 | 候補、選択されたポリシー、説明の参照 |
| VC／証明 | open な義務、契約の失敗 | `vc_id`、ゴールの要約、前提の参照 |
| ATP／カーネル | タイムアウト、バックエンドエラー、証明書の棄却 | バックエンドプロファイル、証明書の状態、witness の参照 |
| アーティファクト／ビルド | 古いアーティファクト、キャッシュ／スキーマの不一致 | アーティファクトパス、スキーマバージョン、キャッシュキーの要約 |

後続のフェーズが出力する診断は前段のメタデータを参照してよいですが、単独のレコードとして表現できなければなりません。

### 集約

診断は、ソースファイルごと、ビルドスナップショットごとに集約されます。

集約器は次を行います。

- パスをワークスペース相対のパスへ正規化する。
- 決定的な `DiagnosticId` を割り当てる。
- 複数の経路で生成された同一の診断を重複排除する。
- ソース範囲・重大度・フェーズ順・コードで診断をソートする。
- ビルドの鮮度とフェーズの状態を付与する。
- 利用者向けの診断と、コンパイラのデバッグ用の注記を分離する。

重複排除は、主たる位置や実行可能な修正が異なる診断を統合してはなりません。

### 重大度の対応付け

コンパイラの重大度は、CLI と LSP の重大度に次のように対応付けられます。

| コンパイラの重大度 | CLI | LSP |
|---|---|---|
| `Error` | `error[...]` | `Error` |
| `Warning` | `warning[...]` | `Warning` |
| `Info` | `info[...]` | `Information` |
| `Hint` | `hint[...]` またはインラインの注記 | `Hint` |

`@show_type` のようなアノテーションが要求する `I` 系列の診断は、インライン表示のヒントか明示的な出力メッセージかに応じて、LSP の `Hint` または `Information` になります。

### 修正の提案

コンパイラが安全に計算できる場合、修正の提案は構造化された編集として保存されます。

```rust
struct FixSuggestion {
    title: String,
    confidence: FixConfidence,
    edits: Vec<TextEdit>,
    command: Option<DiagnosticCommand>,
    applicability: Applicability,
}
```

CLI 出力の `help:` テキストはこの構造から導出されます。LSP サーバは同じ構造をコードアクションへ変換します。

現在のバッファ内容に依存する提案は、編集を適用する前に対象テキストがまだ一致することを検証しなければなりません。

## LSP モデル

### 責務

LSP サーバは、エディタプロトコルに関わる関心事を所有します。

責務:

- ドキュメントの同期と開いているバッファのスナップショット
- ソース範囲から LSP 位置への変換
- 診断の公開
- ホバー・定義へ移動・参照・ドキュメントシンボル・セマンティックトークンの提供
- コードアクションと説明リクエストの提供
- 検証のスケジューリング、またはビルドサービスへのスケジューリング要求

責務でないもの:

- 証明結果の受理
- 型検査や ATP 推論の再実装
- 生のコンパイラ IR の解釈
- ビルドアーティファクトの直接の変更

### リクエストの出所

| LSP 機能 | 主なデータソース |
|---|---|
| 診断 | 最新のビルドスナップショットの集約された `Diagnostic` レコード |
| ホバー | `ExpressionMetadata`、シンボルサマリ、アクティブな thesis のメタデータ |
| 定義へ移動 | 解決されたシンボルメタデータとエクスポートサマリ |
| 参照 | アーティファクトとソーススナップショットから構築したシンボル索引 |
| ドキュメントシンボル | 現在のバッファの構文解析されたアウトライン。鮮度があればアーティファクトで拡充 |
| コードアクション | `FixSuggestion`、`missing_facts.json`、説明の参照 |
| セマンティックトークン | 現在のバッファの字句／構文解析の出力。鮮度があれば解決済みのシンボルクラス |
| インレイヒント | 推論された型、挿入された型強制、アクティブな thesis、証明状態 |
| explain コマンド | 説明アーティファクトと上限つきのクエリサービス |

鮮度のある意味論アーティファクトが利用できない場合、サーバは構文のみの機能へ穏当に劣化すべきです。

### 増分更新の流れ

1. エディタがドキュメントの変更を送る。
2. LSP は開いているバッファのスナップショットを更新し、可能なら軽量な構文診断を公開する。
3. ビルドスケジューラが依存グラフを用いて影響を受けるファイルを算出する。
4. コンパイラの各フェーズが実行され、完了したビルドスナップショットのアーティファクトを出力する。
5. LSP は新しいスナップショットへ切り替え、影響を受けたファイルの診断を再公開する。
6. 以前のスナップショットの古いコードアクションと説明ハンドルは無効化される。

サーバは急速な編集をデバウンスしてよいですが、バージョンを確認せずに、古いバージョン向けの範囲で新しいドキュメントバージョンに診断を公開してはなりません。

### コードアクション

コードアクションは 3 種類に分かれます。

| 分類 | 例 | 安全のルール |
|---|---|---|
| ローカルなテキスト編集 | `qua` の挿入、欠けているアノテーションの追加、誤字の置換 | 範囲／テキストの一致を要求する |
| アーティファクト補助の編集 | `missing_facts` からの補題追加、引用の精緻化 | 一致するアーティファクトとソースハッシュを要求する |
| コマンド | `mizar explain` の実行、VC パネルの表示、検証の再実行 | コマンドが後で検証しない限りソース編集はしない |

サーバは、広範な書き換えより小さく明示的な編集を優先すべきです。証明の提案では、バックエンド由来の候補とカーネル検証済みの `used_axioms` を区別しなければなりません。

## インターフェイス定義

### 診断レコード

```rust
struct Diagnostic {
    id: DiagnosticId,
    code: DiagnosticCode,
    semantic_name: String,
    severity: DiagnosticSeverity,
    phase: PipelinePhase,
    message: String,
    primary_span: SourceRange,
    secondary_spans: Vec<DiagnosticSpan>,
    notes: Vec<DiagnosticNote>,
    fixes: Vec<FixSuggestion>,
    explanation: Option<ExplanationRef>,
    related_ids: Vec<DiagnosticId>,
}
```

`message` は表示用に簡潔で十分に安定していますが、ツールはメッセージのテキストではなく `code` をキーにすべきです。

### LSP スナップショット

```rust
struct LspSnapshot {
    snapshot_id: SnapshotId,
    artifact_manifest: ArtifactManifestRef,
    source_versions: Vec<SourceVersion>,
    diagnostics: DiagnosticIndex,
    expression_index: ExpressionIndex,
    symbol_index: SymbolIndex,
    explanation_index: ExplanationIndex,
    freshness: SnapshotFreshness,
}
```

スナップショットは、リクエストハンドラに公開された後はイミュータブルです。

### 説明への参照

```rust
struct ExplanationRef {
    kind: ExplanationKind,
    diagnostic_id: Option<DiagnosticId>,
    expr_id: Option<ExprId>,
    vc_id: Option<VcId>,
    artifact_path: Option<PathBuf>,
    summary_hash: Hash,
}
```

説明への参照はハンドルであり、埋め込まれたトレースではありません。大きなトレースは、専用のアーティファクトまたはクエリサービスに残します。

## 検討した代替案

### 各フェーズで診断を整形する

これは初期の CLI 出力には簡単ですが、CLI・アーティファクト・LSP でロジックが重複します。構造化されたレコードにより、唯一の真実の出所を保てます。

### LSP がコンパイラのロジックをプロセス内で再実行する

チェッカのロジックを LSP プロセス内で直接実行すると遅延は下がるかもしれませんが、バッチビルドとの乖離と信頼境界の複雑化を招きます。採用した設計では、LSP は統制と安定したスナップショットの読み取りに徹します。

### すべての診断に完全な説明トレースを格納する

各診断は自己完結しますが、アーティファクトが大きくなり、エディタの遅延も悪化します。採用した設計では、コンパクトな参照を保存し、詳細な説明は遅延的に取得します。

### 開いているバッファをビルドアーティファクトとして扱う

未保存のエディタバッファを通常のアーティファクトディレクトリに書くと参照は単純になりますが、再現可能なビルド出力を汚します。採用した設計では、開いているバッファのスナップショットを LSP のメモリに保持し、アーティファクトの鮮度を明示します。

## 採用方針

Mizar Evo は、構造化された診断を正準的なフェーズ出力として用い、コンパクトな診断メタデータをアーティファクトへ出力し、エディタ機能をイミュータブルな LSP スナップショットから提供します。豊富な説明は遅延的に参照され、ソース編集は範囲とテキストの検証の後にのみ適用されます。

この設計により、バッチ検証器を意味論的真実の出所として保ちながら、CLI 出力・アーティファクト消費者・IDE 機能の整合性を維持します。

## 制約と前提

- 診断の数値コードは安定しており、再利用されない。
- 診断メッセージはバージョン間で改善され得る。ツールはメッセージのテキストをキーにしてはならない。
- LSP の範囲変換は、そのリクエストの厳密なソーススナップショットを用いて、エディタの境界で行う。
- ビルドアーティファクトは、検証済みメタデータの意味論的な真実の出所であり続ける。
- 開いているバッファの機能はアーティファクトより鮮度が高くなり得るが、その鮮度を印付けしなければならない。
- コードアクションは、編集を適用する前に対象範囲を検証しなければならない。
- 説明 API は上限つきかつクエリ指向であり、コンパイラ状態の一括ダンプではない。
- LSP は、アーティファクトが欠落・古い・未対応スキーマ由来の場合でも穏当に劣化しなければならない。
