# Architecture: Diagnostics and LSP

> Canonical language: English. English canonical version: [../en/12.diagnostics_and_lsp.md](../en/12.diagnostics_and_lsp.md).

## 目的

この文書は、Mizar Evo における diagnostics、source metadata、explanation data、LSP responses の流れを定義する。

Diagnostics はすべての pipeline phase で生成されるが、phase 15 で tools 向けの stable artifact data になる。LSP server はそれらの artifacts と小さな query services を読み、compiler-internal IR layouts に依存せず editor feedback を提供する。

## Design Decisions

### Diagnostics Are Structured Phase Outputs

すべての pipeline phase は formatted strings ではなく structured `Diagnostic` records を emit する。

CLI output formatter、artifact emitter、LSP server は同じ canonical record を読む。これにより、CLI が説明する error と IDE や artifact metadata が見る error がずれることを防ぐ。

### Diagnostic Identity

各 diagnostic は二種類の identifier を持つ。

| Identifier | Scope | Purpose |
|---|---|---|
| `DiagnosticCode` | stable across releases | human-facing and tool-facing classification, e.g. `E0201` |
| `DiagnosticId` | one build snapshot | LSP diagnostics を explanation queries and code actions へ link する |

`DiagnosticCode` は diagnostic registry により永久に割り当てられる。`DiagnosticId` は build snapshot 内で deterministic であり、source file、code、primary span、phase、stable per-span index から導出される。Tools は unrelated build snapshots をまたいで `DiagnosticId` を persist してはならない。

### Source Spans Are Centralized

Diagnostics は ad hoc な line/column calculation を保存しない。`SourceUnit` line map に基づく `SourceRange` を参照する。

Rules:

- primary span は user が最初に edit すべき location である
- secondary spans は definitions、candidates、generated obligations、imported facts を示す
- generated obligations は diagnostics が source-level constructs を指せるよう `GeneratedFrom` provenance を持つ
- LSP UTF-16 positions は source text snapshot から LSP boundary でのみ計算される

これにより compiler diagnostics は editor protocol encoding details から独立する。

### Diagnostics Survive Partial Failure

Compiler は fatal phase failure の後に semantic facts を捏造せず、可能な限り多くの independent diagnostics を報告するべきである。

Phase outputs は次を区別する。

- `fatal`: downstream semantic phases は phase output を trust できない
- `blocking`: この item または obligation は進めないが、他の item は続行できる
- `recoverable`: later phases は degraded metadata で続行できる

たとえば parser recovery は recovered syntax 上で name resolution を可能にする場合があるが、import graph failure は affected module とその reverse dependencies の semantic checking を block する。

### Explanation Data Is Lazy and Bounded

Type inference、cluster resolution、overload resolution、proof failure の rich explanations は通常の diagnostics よりかなり大きくなり得る。

Default diagnostic record は compact summary and references を保存する。

- explanation kind
- relevant `expr_id`, `vc_id`, or `diagnostic_id`
- explanation artifact が emit された場合の path
- CLI and editor display 用の bounded preview text

LSP server と `mizar explain` は詳細 explanation を on demand で解決する。すべての trace の bulk extraction は避ける。

### LSP Reads Stable Snapshots

LSP server は mutable compiler state ではなく consistent build snapshot を読む。

Snapshot は次から構成される。

- artifact manifest
- `*.mizir.json` files
- diagnostic explanation artifacts
- 存在する場合 `missing_facts.json`
- explanation queries が必要とする場合の `cluster-db/` and import-scoped views
- open editor buffers の source text snapshots

New build が完了すると、server は previous snapshot から new snapshot へ atomic に切り替える。In-flight requests は old snapshot か new snapshot のどちらかを使い、混在させない。

### Open Buffers Override Disk Source

Open editor buffers について、LSP requests は editor-provided text を current source snapshot として使う。Build artifacts は still last completed verification pass on disk を反映している場合がある。

そのため server は responses に freshness を付与する。

| Freshness | Meaning |
|---|---|
| `current` | response was computed from the current open-buffer text |
| `stale_artifact` | response comes from the last verified artifact and may not match unsaved text |
| `pending` | verification is scheduled or running |

Stale artifact 由来の diagnostics は表示し続けてもよいが、target range が current buffer text と一致しない場合、server は destructive code actions を提供すべきではない。

## Diagnostic Pipeline

### Producers

各 phase は phase-specific metadata を伴う diagnostics を提供する。

| Phase Area | Typical Diagnostics | Extra Metadata |
|---|---|---|
| frontend | malformed tokens, syntax recovery, missing `end` | expected tokens, recovery point |
| resolver | unresolved names, ambiguous imports, duplicate symbols | candidate symbols, import path |
| checker | type mismatch, sethood failure, cluster failure | inferred/expected types, resolution trace ref |
| overload | ambiguous overload, missing `qua` | candidates, selected policy, explanation ref |
| VC / proof | open obligation, contract failure | `vc_id`, goal summary, premise refs |
| ATP / kernel | timeout, backend error, certificate rejection | backend profile, certificate status, witness ref |
| artifact / build | stale artifact, cache/schema mismatch | artifact path, schema version, cache key summary |

Later phases が emit する diagnostics は earlier metadata を参照してよいが、standalone records として表現できなければならない。

### Aggregation

Diagnostics は source file ごと、build snapshot ごとに aggregate される。

Aggregator は次を行う。

- paths を workspace-relative paths へ normalize する
- deterministic `DiagnosticId`s を割り当てる
- 複数経路で生成された identical diagnostics を deduplicate する
- source range、severity、phase order、code で diagnostics を sort する
- build freshness and phase status を attach する
- user-facing diagnostics と compiler debug notes を分離する

Deduplication は primary spans や actionable fixes が異なる diagnostics を merge してはならない。

### Severity Mapping

Compiler severity は CLI and LSP severities に次のように map される。

| Compiler Severity | CLI | LSP |
|---|---|---|
| `Error` | `error[...]` | `Error` |
| `Warning` | `warning[...]` | `Warning` |
| `Info` | `info[...]` | `Information` |
| `Hint` | `hint[...]` or inline note | `Hint` |

`@show_type` のような annotations が要求する `I`-series diagnostics は、inline display hints か explicit output messages かに応じて LSP `Hint` または `Information` になる。

### Fix Suggestions

Compiler が安全に計算できる場合、fix suggestions は structured edits として保存される。

```rust
struct FixSuggestion {
    title: String,
    confidence: FixConfidence,
    edits: Vec<TextEdit>,
    command: Option<DiagnosticCommand>,
    applicability: Applicability,
}
```

CLI output の `help:` text はこの structure から導出される。LSP server は同じ structure を code actions に変換する。

Current buffer contents に依存する suggestions は、edits を適用する前に target text がまだ一致することを検証しなければならない。

## LSP Model

### Responsibilities

LSP server は editor protocol concerns を所有する。

責務:

- document synchronization and open-buffer snapshots
- source ranges から LSP positions への変換
- diagnostics publication
- hover, go-to-definition, references, document symbols, semantic tokens の提供
- code actions and explanation requests の提供
- verification の scheduling または build service への scheduling request

責務ではないもの:

- proof results の acceptance
- type checking or ATP reasoning の再実装
- raw compiler IR の解釈
- build artifacts の直接 mutation

### Request Sources

| LSP Feature | Primary Data Source |
|---|---|
| diagnostics | latest build snapshot の aggregated `Diagnostic` records |
| hover | `ExpressionMetadata`, symbol summaries, active thesis metadata |
| go-to-definition | resolved symbol metadata and export summaries |
| references | artifacts and source snapshots から構築した symbol index |
| document symbols | current buffer の parsed outline、fresh な場合 artifacts で enrichment |
| code actions | `FixSuggestion`, `missing_facts.json`, explanation refs |
| semantic tokens | current buffer の lexer/parser output、fresh な場合 resolved symbol classes |
| inlay hints | inferred types, inserted coercions, active thesis, proof status |
| explain commands | explanation artifacts and bounded query services |

Fresh semantic artifacts が利用できない場合、server は syntax-only features へ graceful に degrade するべきである。

### Incremental Update Flow

1. Editor が document change を送る。
2. LSP は open-buffer snapshot を更新し、可能なら lightweight syntax diagnostics を publish する。
3. Build scheduler が dependency graph を使って affected files を計算する。
4. Compiler phases が実行され、completed build snapshot の artifacts を emit する。
5. LSP は new snapshot へ切り替え、affected files の diagnostics を republish する。
6. Previous snapshot の stale code actions and explanation handles は invalidated される。

Server は rapid edits を debounce してよいが、version を確認せずに older version 用の ranges で newer document version に diagnostics を publish してはならない。

### Code Actions

Code actions は三種類に分かれる。

| Class | Examples | Safety Rule |
|---|---|---|
| local text edits | insert `qua`, add missing annotation, replace typo | require range/text match |
| artifact-assisted edits | add lemma from `missing_facts`, refine citation | require matching artifact and source hash |
| commands | run `mizar explain`, show VC panel, re-run verification | no source edit unless command later validates |

Server は broad rewrites より small explicit edits を優先するべきである。Proof suggestions では backend-derived candidates と kernel-verified `used_axioms` を区別しなければならない。

## Interface Definitions

### Diagnostic Record

```rust
struct Diagnostic {
    id: DiagnosticId,
    code: DiagnosticCode,
    semantic_name: String,
    severity: DiagnosticSeverity,
    phase: PipelinePhase,
    message: String,
    primary_span: SourceRange,
    secondary_spans: Vec<DiagnosticSpan>,
    notes: Vec<DiagnosticNote>,
    fixes: Vec<FixSuggestion>,
    explanation: Option<ExplanationRef>,
    related_ids: Vec<DiagnosticId>,
}
```

`message` は display 用に簡潔で十分 stable だが、tools は message text ではなく `code` を key にするべきである。

### LSP Snapshot

```rust
struct LspSnapshot {
    snapshot_id: SnapshotId,
    artifact_manifest: ArtifactManifestRef,
    source_versions: Vec<SourceVersion>,
    diagnostics: DiagnosticIndex,
    expression_index: ExpressionIndex,
    symbol_index: SymbolIndex,
    explanation_index: ExplanationIndex,
    freshness: SnapshotFreshness,
}
```

Snapshot は request handlers に publish された後 immutable である。

### Explanation Reference

```rust
struct ExplanationRef {
    kind: ExplanationKind,
    diagnostic_id: Option<DiagnosticId>,
    expr_id: Option<ExprId>,
    vc_id: Option<VcId>,
    artifact_path: Option<PathBuf>,
    summary_hash: Hash,
}
```

Explanation references は handles であり、embedded traces ではない。Large traces は dedicated artifacts または query services に残す。

## Alternatives Considered

### Formatting Diagnostics at Each Phase

これは初期 CLI output には簡単だが、CLI、artifacts、LSP で logic が重複する。Structured records により single source of truth を保つ。

### LSP Re-runs Compiler Logic In-Process

Checker logic を LSP process 内で直接実行すると latency は下がるかもしれないが、batch builds との divergence と trust boundaries の複雑化を招く。採用案では LSP は orchestration と stable snapshots の読み取りに徹する。

### Store Full Explanation Traces in Every Diagnostic

各 diagnostic は self-contained になるが、artifacts が大きくなり editor latency も悪化する。採用案では compact references を保存し、detailed explanations は lazy に取得する。

### Treat Open Buffers as Build Artifacts

Unsaved editor buffers を normal artifact directory に書くと lookup は単純になるが、reproducible build output を汚す。採用案では open-buffer snapshots を LSP memory に保持し、artifact freshness を明示する。

## Adopted Approach

Mizar Evo は structured diagnostics を canonical phase outputs として使い、compact diagnostic metadata を artifacts に emit し、editor features を immutable LSP snapshots から提供する。Rich explanations は lazy に参照され、source edits は range and text validation 後にのみ適用される。

この設計により、batch verifier を semantic truth の source として保ちながら、CLI output、artifact consumers、IDE features の整合性を維持する。

## Constraints and Assumptions

- Diagnostic numeric codes は stable で reuse されない。
- Diagnostic messages は version 間で改善され得る。Tools は message text を key にしてはならない。
- LSP range conversion は、その request の exact source snapshot を使って editor boundary で行う。
- Build artifacts は verified metadata の semantic source of truth であり続ける。
- Open-buffer features は artifacts より fresh になり得るが、その freshness を mark しなければならない。
- Code actions は edits 適用前に target ranges を validate しなければならない。
- Explanation APIs は bounded and query-oriented であり、compiler state の bulk dump ではない。
- LSP は artifacts が missing、stale、unsupported schema 由来の場合でも graceful に degrade しなければならない。
