# アーキテクチャ: ソースとフロントエンド

> 正本は英語です。英語版: [../en/02.source_and_frontend.md](../en/02.source_and_frontend.md)。

## 目的

この文書では、Mizar Evo のフロントエンドが `.miz` ソースファイルを読み込み、前処理・字句解析・構文解析・構文出力を統制して、後続のモジュール解決と名前解決へ渡すまでの流れを定義します。

これは [00.pipeline_overview.md](./00.pipeline_overview.md) のフェーズ 1-3 を詳細化する文書です。とくに、文脈依存の字句解析、アクティブな字句環境、ユーザー定義シンボル、ドットの扱い、字句解析器が完全に切り出す文字列リテラル、ドキュメントコメント、アノテーションの結び付けについて、その境界を明確にします。

## 設計判断

### フロントエンドは意味論ではなく構文を生成する

フロントエンドは、意味論的な名前解決や型検査を行ってはなりません。

フロントエンドが行ってよいこと:

- UTF-8 とコード領域の ASCII 制約を検証する。
- コメントとドキュメントコメントを識別する。
- アクティブな字句環境を要求できる程度に、インポート宣言を事前走査する。
- 予約語・予約記号・アクティブなユーザーシンボルを用いてトークン化する。
- ソースの形を保った AST へ構文解析する。
- 構文レベルのエラーリカバリを行う。

フロントエンドが行ってはならないこと:

- スコープの知識を要する場面で、ドット付きパスが名前空間を指すのか、選択された項を指すのかを決める。
- オーバーロードの勝者を選ぶ。
- 式の型を推論する。
- cluster registration を発火する。
- 証明義務を生成する。

### 構文とパーサーは独立した crate 境界にする

実装では、構文データ構造とパーサーロジックを、統制を主責務とする `mizar-frontend` crate の外に置くべきです。

| Crate | 責務 |
|---|---|
| `mizar-syntax` | `SurfaceAst`、構文ノード種別、arena、trivia、回復マーカー、パーサー向け構文診断を所有する。 |
| `mizar-parser` | `TokenStream` を消費し、項／論理式の Pratt parser を含む文法を適用し、アノテーションとドキュメントコメントを結び付け、`SurfaceAst` を出力する。 |
| `mizar-frontend` | ソース読み込み、ソースマップ構築、前処理、アクティブ字句環境構築、字句解析、parser seam 呼び出し、統合された `FrontendOutput` を統制する。 |

この分割により、LSP、フォーマッタ、診断、リゾルバ、テストは、パーサー内部に依存せずに構文構造へ依存できます。また、パーサーの回復処理と文法の進化が、ソース読み込み／統制レイヤを肥大化させることも避けられます。
`mizar-syntax` / `mizar-parser` が存在するまでは、フロントエンドは `ast = None` を返す stub parser seam を使い、parser への hard dependency なしで source → tokens pipeline を実行できるようにします。

### インポート事前走査は浅く行う

インポートされたモジュールはアクティブな字句環境を拡張するため、フロントエンドは最終的なトークン化より前にインポート情報を必要とします。ただし、完全なインポート解決はモジュールリゾルバの責務です。

そのため前処理では、浅いインポート事前走査だけを行います。

- 制限された字句モードを用いて、トップレベルの `import` 形式を認識する。
- 生のドット付きモジュールパス、alias、全体 span、分岐インポートの `path.source_segments` を抽出する。
- 語彙の読み込みを妨げる、不正なインポート構文だけを報告する。
- パッケージ／モジュールの存在・可視性・エクスポートの検査は後続フェーズに委譲する。

インポート事前走査は生の字句解析器の出力を消費します。生のスキャン自体はインポートを解釈せず、アクティブな字句環境も構築しません。strict raw scan が完全な raw stream を生成できない場合、前処理はインポート事前走査診断を記録し、部分的な raw text から import を推測せず import stub を返しません。

ビルドプランナ／モジュールリゾルバは、これらの `ImportStub` から解決済みインポートと軽量なモジュール字句サマリを用意します。字句解析器はそれらから `ActiveLexicalEnvironment` を構築し、そのインポートから見えるシンボルを使って最終的な曖昧性解消を進めます。

### 字句解析は生のスキャンを先に行い、その後で文脈に応じて確定する

字句解析器は、概念的に生のスキャンと最終的なトークン曖昧性解消に分かれます。

生のスキャナはソース位置を保持し、`LexemeRun` のような粗い単位を出力します。これらの生の単位は最終的な言語トークンではなく、後段で分割され得ます。

曖昧性解消器は、次のものに対して最長一致のルールを適用します。

1. アクティブな字句環境に由来するユーザー定義シンボル
2. 予約された特殊記号
3. 予約語
4. 識別子と数値リテラルのルール
5. 構文解析器の期待と、言語が要求する場合のスコープ付き識別子の上書きルール

ある綴りが識別子の構文とアクティブなユーザーシンボルの両方に一致する場合、最終的な分類には構文解析器の位置とスコープ情報が必要になることがあります。スコープ付きの識別子束縛は、言語規則に従ってシンボルを上書きし得ます。この判断は生のスキャンではなく、トークン曖昧性解消の責務です。

字句的な曖昧性解消に用いるスコープ情報は、生の字句解析器の出力に対する専用のスコープスケルトン事前走査から供給されます。曖昧性解消器はこの読み取り専用ビューを消費するだけで、自分でスコープを構築することはありません。

字句解析器は、識別子が定義済みかどうか、シンボルが項／論理式の文脈で適法に適用できるかどうか、どのオーバーロードが選ばれるかを決めません。それらはリゾルバ／チェッカの関心事です。

### ドットの扱いは字句解析器・構文解析器・リゾルバに分担される

`.` は意図的に一箇所では解決しません。

| 役割 | 層 | ルール |
|---|---|---|
| 複合予約トークン `.{`, `.*`, `.=`, `...` | 字句解析器 | 単一の予約トークンとして認識する |
| `.` を含むユーザー定義シンボル | 曖昧性解消器 | アクティブな字句環境・構文解析器の文脈・上書きルール・最長一致により認識する |
| セレクタのアクセス／更新 | 構文解析器 + リゾルバ | 項の文脈でセレクタ連鎖の候補として構文解析する |
| 名前空間の区切り | 構文解析器 + リゾルバ | インポート・引用・アノテーション・修飾名の文脈で構文解析する |
| 名前空間パスに対する変数のシャドーイング | リゾルバ | ローカル変数が名前空間の構成要素に勝つ |

意味論的なスコープが必要な場面では、フロントエンドは `DottedPathOrSelection` のような、表層の形を保った曖昧な表現を生成してよいです。リゾルバが後で名前空間かセレクタかを確定します。

### 文字列リテラルは字句解析器が完全にトークン化する

`"` と `'` はユーザーシンボルに使える文字であるため、字句解析器はこれらを一律に文字列区切りとして扱えません。

それでもフロントエンドは、構文解析より前に完全な `TokenStream` を生成することを目標とします。現在の字句解析器 API は、この依存関係を限定的だが uniform な `ParserLexContext` として表現します。フロントエンドと構文解析器の統合は、任意の構文解析器の状態を字句解析器に公開することなく、後で文法から導かれる位置別の「文字列が必要な文脈」を渡してよいです。

文字列リテラルは、文字列リテラルを明示的に要求する文法上の位置でのみ認識されます。たとえば次のような位置です。

- `@latex("...")` のようなアノテーション引数
- `infix_operator("+", left, 80)` のような演算子宣言の引数
- 将来、文字列値の引数として登録される文法上の位置

これらの位置以外では、引用符は、アクティブな字句環境と最長一致のルールに従う通常の記号文字のままです。

構文解析器は、すでにトークン化された `StringLiteral` トークンを消費します。構文解析器が字句解析器のカーソルを直接動かすことはありません。parser-assisted lexing contract が確定するまでは、フロントエンド実装テストでは general/stub lexing と bounded uniform `StringRequired` lexing だけを前提にし、注釈／演算子位置の文字列リテラルテストは gate します。

### コメントとドキュメントコメントはソースのメタデータである

コメントは字句的な入力から除去されますが、破棄されるわけではありません。

- 通常のコメントは、整形／デバッグ用ツールのためにのみ保持する。
- ドキュメントコメントは、可能な場合に直後のドキュメント化可能な項目へ結び付ける。
- ドキュメントコメントの結び付けは構文的なものにとどまり、対象がドキュメント化可能でない場合は後で却下され得る。
- 構造化されたドキュメントタグは、タグ名・生の引数・ソース位置を保持できる程度に構文解析する。

### アノテーションは構文解析器が所有する構文である

アノテーションは表層構文の一部であり、ソース位置と生の引数を保持しなければなりません。

前処理はアノテーションを別のメタデータ経路に集めません。アノテーションのトークンは `lexical_text` に残り、`TokenStream` に出力され、構文解析器によって `SurfaceAst` のアノテーションノードへ構文解析されます。

構文解析器はアノテーションの構文を検証しますが、意味論的な効果は先送りされます。

- `@[...]` ライブラリアノテーションは、生のライブラリラベルとして項目に結び付く。
- `@latex`・`@proof_hint`・`@show_*`・`@eval` は、アノテーションノードとして表現する。
- アノテーションレジストリの検証は、構文に関しては構文解析器で行ってもよいが、意味は後続フェーズが解釈する。
- アノテーションは中核言語の意味論を変更してはならない。

## フロントエンドパイプライン

### ステップ 1: SourceUnit の読み込み

入力:

- パッケージ／ワークスペースの `BuildPlan`
- `.miz` ファイルパス
- 期待されるパッケージルートと `src/` ルート

出力:

- `SourceUnit`

責務:

- ソースのバイト列を読む。
- UTF-8 を検証する。
- 読み込み済みソーステキストを作る前に、パッケージが記述したソースの先頭にある UTF-8 BOM を 1 つ取り除く。
- `source_hash` を計算する。
- ファイルパスからモジュールパスを導出する。
- Unicode スカラー値による列のルールを用いて `LineMap` を構築する。
- 読み込みによってオフセットが変わった場合、読み込み済みテキストのオフセットから元のファイルのバイトオフセット、またはエディタが与えたテキストのオフセットへの `LoadingMap` を保持する。
- ファイルレベルの診断を出力する。

失敗の例:

- 読み取れないソースファイル
- 不正な UTF-8
- パッケージの `src/` ルート外のファイル
- 不正な `.miz` モジュールファイル名

### ステップ 2: ソースの前処理

入力:

- `SourceUnit`

出力:

- `PreprocessedSource`

責務:

- コメントには Unicode を許しつつ、コード領域の ASCII を検証する。注釈文字列引数内の Unicode は、ASCII 前提診断より前に parser-assisted lexing が文字列必須 span を識別できるようになってから受理する。
- 通常のコメントを識別し、字句的な入力から除去する。
- ドキュメントコメントをソース位置付きで保持する。
- 構文解析器に所有させるため、アノテーション構文を字句的な入力に残す。
- トップレベルのインポートを浅く走査する。
- 字句テキストからソース範囲への正確なマッピングを保持する。

現在の `mizar-lexer` クレートは、コメント除去・字句的前提条件の診断・モジュールソース命名・生スキャン・インポート事前走査のヘルパー API を提供します。ファイル入出力・UTF-8 検証・行マップ・全体の統制は、引き続きフロントエンドの責務です。

失敗の例:

- 閉じられていないブロックコメント
- コード領域内の不正な非 ASCII 文字
- アクティブな字句環境の構築を妨げるインポート事前走査の失敗

### ステップ 3: ActiveLexicalEnvironment の構築

入力:

- 前処理由来の `ImportStub`
- ビルドプランから利用可能なパッケージ／モジュール索引
- 構築済みの依存シンボルのエクスポート

出力:

- `ActiveLexicalEnvironment`

責務:

- インポートされたモジュールがエクスポートするユーザー定義の記号名を集める。
- 組み込みの予約記号とキーワードテーブルを含める。
- 決定的 fingerprint と provenance のために、インポート順序を記録する。
- 同綴りのインポート済みユーザーシンボル衝突は、import-order tie-break で解決せず、lexer/spec contract に従って拒否または診断する。
- 診断のためにシンボルの来歴を記録する。

このステップはフロントエンドに隣接するサービスです。フロントエンド由来の浅いインポートを使いますが、完全なインポートの適法性はモジュール解決の一部であり続けます。

### ステップ 4: 字句解析

入力:

- `PreprocessedSource`
- `ActiveLexicalEnvironment`
- 最終的な曖昧性解消が必要とする場合の、構文解析器の期待とスコープ付き束縛のビュー

出力:

- `TokenStream`

責務:

- 早すぎる分割をせずに、生の `LexemeRun` の範囲を保持する。
- 予約語と予約された特殊記号を出力する。
- 文脈依存の最長一致のもとで、ユーザー定義シンボルを出力する。
- 識別子と数値リテラルを出力する。
- 文字列が必要な位置で `StringLiteral` トークンを出力する。
- ソース位置と元の綴りを保持する。
- 回復可能なトークンを失わずに、字句診断を提示する。

失敗の例:

- 未知または不正なトークン列
- 不正な数値リテラルの形式
- 文字列が必要な位置における不正な文字列リテラル

### ステップ 5: 構文解析

入力:

- `TokenStream`
- アクティブな字句環境から導かれるパーサー入力。例: 演算子の fixity、文字列リテラルを要求する文法文脈

出力:

- `SurfaceAst`

責務:

- モジュール・定義・registration・文・項・論理式・定理・証明・アルゴリズムを構文解析する。
- 項と論理式の式構文には、Pratt parser または同等のテーブル駆動の優先順位 parser を用いる。
- アノテーションの引数リストを構文解析し、アノテーションを構文ノードに結び付ける。
- ドキュメントコメントを近くのドキュメント化可能な項目に結び付ける。
- ソース順とソース範囲を保持する。
- `;`・`end`・トップレベル項目のキーワードなどの同期点で、構文エラーから回復する。

失敗の例:

- 予期しないトークン
- 対応の取れない区切り記号
- `end` の欠落
- 期待される文字列リテラルトークンの欠落

### PreprocessedSource

```rust

    comments: Vec<Comment>,
    doc_comments: Vec<DocComment>,
    import_stubs: Vec<ImportStub>,
    source_map: LexicalSourceMap,

```

### ImportStub

```rust
struct ImportStub {
    path: ImportStubPath,
    alias: Option<ImportStubAlias>,
    span: SourceRange,
}

struct ImportStubPath {
    spelling: String,
    relative: Option<ImportStubRelativePrefix>,
    components: Vec<String>,
    source_segments: Vec<SourceRange>,
    span: SourceRange,
}

enum ImportStubRelativePrefix {
    Current,
    Parent,
}

struct ImportStubAlias {
    spelling: String,
    span: SourceRange,
}
```

`ImportStub` は lexer のインポート事前走査 stub を frontend が写像した対応物です。`./` と `../` の相対 prefix の区別を含め、lexer の `RawModulePath` / `RawModuleAlias` の形を写しますが、すべての範囲は session `SourceRange` です。解決済みインポートではなく、import-kind field を持ちません。語彙のエントリを要求し、語彙の読み込みが失敗した場合に良質な診断を出すには十分な情報です。

### ActiveLexicalEnvironment

```rust
struct ActiveLexicalEnvironment {
    reserved_words: ReservedWordTable,
    reserved_symbols: ReservedSymbolTable,
    user_symbols: UserSymbolIndex,
    fingerprint: LexicalEnvironmentFingerprint,
}

struct UserSymbolCandidate {
    spelling: SymbolText,
    symbol_id: SymbolId,
    source_module: ModuleId,
    imported_module: ModuleId,
    import_ordinal: usize,
    export_rank: ExportRank,
}
```

`ActiveLexicalEnvironment` は、解決済みインポートと軽量な `ModuleLexicalSummary` から構築されます。これは依然として字句解析器に閉じた構造であり、字句的な形と来歴を記録するだけで、完全なモジュール IR や意味論的な適用可能性は保持しません。

### TokenStream

```rust
struct TokenStream {
    source_id: SourceId,
    tokens: Vec<Token>,
    diagnostics: Vec<LexingDiagnostic>,
}

struct Token {
    kind: TokenKind,
    span: SourceRange,
    text: InternedText,
}

enum TokenKind {
    Identifier,
    ReservedWord,
    ReservedSymbol,
    Numeral,
    LexemeRun,
    UserSymbol,
    StringLiteral,
    ErrorRecovery,
}
```

`StringLiteral` は、lexer context が文字列必須位置を認識した箇所でのみ現れます。現在の lexer context は uniform であり、正確な文法位置認識は parser-assisted lexing contract に gate されます。
`LexingDiagnostic` は raw-scan、scope-skeleton、曖昧性解消器の診断を、frontend が写像した primary session `SourceRange` と、source map が composite または degraded の場合の secondary source anchor とともに包みます。

### SurfaceAst

`mizar-syntax` が所有する。

```rust
struct SurfaceAst {
    source_id: SourceId,
    module: SurfaceModule,
    nodes: AstArena<SurfaceNode>,
    trivia: TriviaMap,
    diagnostics: Vec<Diagnostic>,
}
```

### FrontendOutput

`mizar-frontend` が、字句解析器と設定済み parser seam を呼び出した後に生成する。

```rust
struct FrontendOutput {
    source: SourceUnit,
    preprocessed: PreprocessedSource,
    tokens: TokenStream,
    ast: Option<SurfaceAst>,
    diagnostics: Vec<Diagnostic>,
}
```

実 parser seam では、`ast = None` は構文解析が後続フェーズで使えるだけの構造へ回復できなかったことを意味します。stub parser seam では期待される placeholder 結果です。字句診断はそれでも返され、構文診断は実 parser seam が動く場合だけ存在します。

## エラーリカバリ

フロントエンドのリカバリは、意味論的な事実を捏造することなく、複数の有用な診断を報告することを目指します。

字句解析器のリカバリ:

- 可能な場合、不正な範囲に `TokenKind::ErrorRecovery` を出力する。
- 空白・予約区切り記号・行境界で再開する。
- 診断のために不正なソース範囲を保持する。

構文解析器のリカバリ:

- `;`・`end`・`definition`・`registration`・`theorem`・`lemma`・`proof`・`algorithm`・EOF で同期する。
- 構文要素が欠けている箇所に明示的なエラーノードを作る。
- 偽の識別子や偽の解決済みシンボルを作らない。
- 結び付けに失敗しても、ドキュメントコメントを元のソース位置の付近に保持する。
- 不正なアノテーションは、可能な場合に構文レベルのエラーノードとして保持する。

回復された AST ノードは、後続フェーズが明示的にスキップしたり穏当に劣化させたりできるよう、印を付けなければなりません。

## 診断

フロントエンドの診断は、[doc/spec/en/22.error_handling_and_diagnostics.md](../../../spec/en/22.error_handling_and_diagnostics.md) の構文／字句の範囲を用います。

| 診断クラス | フェーズ | 例 |
|---|---|---|
| 字句的前提条件 | ソース読み込み／前処理 | 不正な UTF-8、非 ASCII のコード文字 |
| コメント構造 | 前処理 | 閉じられていないブロックコメント |
| インポート事前走査 | 前処理 | 不正な import prelude、import 用 raw scan の失敗 |
| 字句環境 | アクティブ字句環境 | 未解決 import、欠落した字句サマリ、ユーザーシンボル import 衝突 |
| スコープスケルトン | 字句解析の曖昧性解消前 | 不正な binder list、対応しない lexical block end |
| トークン化 | 字句解析 | 不正なリテラル、未知のトークン |
| 構文 | 構文解析 | 予期しないトークン、`end` の欠落、対応の取れない区切り記号 |
| アノテーション構文 | 構文解析 | 不正なアノテーション引数リスト |

フロントエンドの診断には、次を含めなければなりません。

- 安定した診断コード
- 主たるソース位置
- 有用な場合の副次的な source anchor/span。例: 閉じ忘れに対する開き区切り記号、source mapping 由来の removed-comment / generated anchor
- 後続の診断が影響を受け得る場合のリカバリに関する注記

フロントエンドの診断は、「未定義のシンボル」や「曖昧なオーバーロード」といった意味論的な事実を主張してはなりません。それらは後続フェーズの責務です。

## 増分処理

フロントエンドのキャッシュキーは層をなしています。

| 出力 | キャッシュキー |
|---|---|
| `SourceUnit` | 正規パス, ソースのバイト列, エディション, origin |
| `PreprocessedSource` | `source_hash`, フロントエンドのバージョン |
| `ActiveLexicalEnvironment` | 解決済みインポート, 依存の字句サマリのフィンガープリント, インポート順序 |
| `TokenStream` | 前処理済みのハッシュ, アクティブな字句環境のフィンガープリント, parser lexing context / parser-assisted lexing plan |
| `SurfaceAst` | トークンストリームのハッシュ, 構文解析器のバージョン, エディション |

重要な無効化のルール:

- コメントのみの編集は `PreprocessedSource` とドキュメントメタデータを無効化するが、字句テキストが変わっていなければ意味論的な出力を再利用できる可能性がある。
- インポートの編集は、アクティブな字句環境・トークンストリーム・AST、およびそのソースファイルのすべての意味論層を無効化する。
- 依存シンボルのエクスポートの編集は、ローカルのソースファイルが変わっていなくてもトークン化を無効化し得る。
- parser lexing context または parser-assisted lexing plan の変更は、ソースと import が不変でも `TokenStream` を無効化する。
- 構文解析器のバージョンと言語エディションは、AST のキャッシュキーの一部である。

## 検討した代替案

### 代替案 1: インポート解決より前の純粋な字句解析

インポートされたユーザーシンボルを考慮せずにすべてのソースファイルを字句解析し、その後、構文解析や名前解決の際にトークンを再解釈する案です。

これは字句解析を単純にしますが、識別子の形をしたユーザーシンボルや最長一致のルールと衝突します。また、後続フェーズにトークン列の分割や結合を強いることになり、ソース位置や構文解析器のリカバリを脆弱にします。

### 代替案 2: 字句解析より前の完全なモジュール解決

各ファイルを字句解析する前に、インポートを完全に解決する案です。

これは字句解析器に正確なシンボル同一性を与えますが、循環を生みます。モジュールの内容を理解するには構文解析が必要であり、一方で字句解析にはインポートされたシンボルが必要だからです。採用した浅い事前走査は、この循環を断ち切りつつ、アクティブな字句環境を構築するのに十分な情報を保ちます。

### 代替案 3: 字句解析器がすべてのドットの役割を決める

`.` をセレクタ・名前空間区切り・ユーザー functor として、字句解析器が直接分類する案です。

これは、変数のシャドーイングや項の文脈が正しい解釈を決める場合に破綻します。採用した分担は、複合トークンとアクティブなユーザーシンボルを字句解析器に残し、セレクタか名前空間かの判断を構文解析器とリゾルバに委ねます。

### 代替案 4: 引用符を一律に文字列区切りとして扱う

`'...'` と `"..."` を常に文字列リテラルとして字句解析する案です。

これは、後置の逆元記法のようなユーザーシンボルと衝突します。採用した文脈依存の字句解析器は、構文解析より前に完全な `TokenStream` を生成しつつ、文字列リテラルをそれを必要とする文法上の位置だけで認識します。

## 採用方針

段階的なフロントエンドを用います。

1. ソースを読み込み、ソースマッピングを構築する。
2. コメント・ドキュメントコメント・浅いインポートを前処理する。
3. 浅いインポートと依存の字句サマリから、アクティブな字句環境を構築する。
4. lexeme run を生スキャンし、必要に応じて最長一致・アクティブなシンボル・構文解析器の期待・スコープ付き識別子の上書きルールで曖昧性を解消する。
5. 明示的な回復ノードを持つ、ソースの形を保った `SurfaceAst` へ構文解析する。

この設計は、モジュール解決や型検査に属する意味論的な確定を避けつつ、フロントエンドの出力をソースに忠実に保ちます。

## 制約と前提

- ソースファイルは UTF-8 で、拡張子は `.miz` である。
- コード領域は ASCII であり、コメントは Unicode を含んでもよい。注釈文字列引数内の Unicode は parser-assisted lexing contract に gate される。
- インポート事前走査は意図的に不完全であり、モジュール解決を置き換えることはできない。
- アクティブな字句環境はトークン境界に影響し得るため、トークン化は依存の字句サマリのフィンガープリントに依存する。
- 字句解析器は複合予約トークンとアクティブなユーザーシンボルを認識するが、セレクタか名前空間かといった意味論的な役割は認識しない。
- 字句解析器は、parser lexing context が文字列を要求するときだけ、完全な `StringLiteral` トークンを出力する。正確な文法位置認識は parser-assisted lexing contract に gate される。
- アノテーションの構文と結び付けは構文解析器が所有する。前処理はアノテーションを別のメタデータ経路として集めない。
- `SurfaceAst` はソースの形を保っており、回復ノードを含み得る。後続の意味論フェーズは、これらを許容するか明示的に却下しなければならない。
- `mizar-frontend` は構文解析を統制するが、`SurfaceAst` のノード定義や文法実装は所有しない。
- フロントエンドのアーティファクトはコンパイラ内部のデータであり、安定した公開ビルドアーティファクトではない。
