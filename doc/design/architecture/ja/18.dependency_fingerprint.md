## 目的

この文書は stable incremental rebuild のための dependency fingerprints と dependency slices を定義する。

目的は、変更に影響される semantic work だけを rebuild しつつ、deterministic artifacts と proof acceptance を保つことである。

## 設計判断

### Fingerprint Targets

build system は少なくとも次を fingerprint する。

- theorem statements と accepted proof status
- definitions と definitional expansion boundaries
- notation declarations と parse effects
- clusters
- registrations
- modes
- attributes

proof body は theorem statement とは別に fingerprint する。これにより、accepted theorem statement が変わらず proof implementation だけが変わった場合、downstream modules を rebuild しない。

### Dependency Slice

各 theorem と verification condition は dependency slice を保存する。

```text
DependencySlice
  used_theorems
  used_definitions
  used_clusters
  used_notations
  used_modes
  used_attributes
```

slice は parse、resolve、type-check、elaborate、VC generation、ATP translation、certificate acceptance に使われたものを正確に記録する。

### Stable Fingerprint Inputs

fingerprint は次を含める。

- canonical semantic content
- schema version
- language edition
- dependency-facing policy fields
- normalized origin id
- visibility が依存する場合の accepted proof status

fingerprint は次を含めない。

- comments
- syntax-sensitive notation 以外の source formatting
- timestamps
- absolute local paths
- worker ids
- backend runtime duration

### Rebuild Triggers

この table は downstream semantic rebuild と local artifact refresh を区別する。proof-body-only edit は local proof witnesses や implementation hashes の rebuild を必要とする場合があるが、exported theorem statement と accepted proof status が変わらない限り downstream modules を invalidate してはならない。

| Change | Downstream Semantic Rebuild | Local Artifact Refresh |
|---|---|---|
| comment only | no | no |
| theorem proof body only | no | yes, when proof witnesses or implementation hashes change |
| theorem statement | yes | yes |
| cluster semantics | yes | yes |
| notation parse result | yes | yes |
| registration coherence proof status | yes for importers that can see the registration | yes |
| diagnostic wording only | no | yes, for diagnostic snapshots only |

### Cache Validation

cache hit は、dependency slice 内のすべての fingerprint が一致し、active verifier policy が recorded policy と compatible な場合にのみ valid である。

slice が missing、または unsupported schema で生成された場合、build は affected phase を conservatively re-run しなければならない。

## インターフェイス定義

```rust
struct DependencySlice {
    owner: SemanticOwnerId,
    used_theorems: Vec<TheoremFingerprint>,
    used_definitions: Vec<DefinitionFingerprint>,
    used_clusters: Vec<ClusterFingerprint>,
    used_notations: Vec<NotationFingerprint>,
    used_modes: Vec<ModeFingerprint>,
    used_attributes: Vec<AttributeFingerprint>,
}
```

## 制約と前提

- fingerprint computation は canonical ordering と canonical serialization を使う。
- incremental rebuild は optimization であり、full clean build が reference behavior である。
- missing dependency data を "no dependency" と解釈してはならない。
- dependency slices は deterministic build testing の regression snapshots に含める。
