# アーキテクチャ: 依存フィンガープリント

> 正本は英語です。英語版: [../en/18.dependency_fingerprint.md](../en/18.dependency_fingerprint.md)。

## 目的

この文書は、安定した増分再ビルドのための依存フィンガープリントと依存スライスを定義します。

目的は、変更に影響される意味論的な作業だけを再ビルドしつつ、決定的なアーティファクトと証明受理を保つことです。

## 設計判断

### フィンガープリントの対象

ビルドシステムは、少なくとも次をフィンガープリントします。

- 定理ステートメントと受理済みの証明状態
- 定義と定義的な展開境界
- エクスポートされた algorithm の署名とその `requires` / `ensures` 契約（algorithm 本体とは別に。定理ステートメント／証明本体の分離と同様）
- 記法宣言と構文解析への効果
- cluster
- registration
- mode
- 属性

証明本体は、定理ステートメントとは別にフィンガープリントします。これにより、受理済みの定理ステートメントが変わらず、証明の実装だけが変わった場合に、下流モジュールを再ビルドしません。

### 依存スライス

各定理と検証条件は、依存スライスを保存します。

```text
DependencySlice
  used_theorems
  used_definitions
  used_clusters
  used_notations
  used_modes
  used_attributes
```

スライスは、構文解析・解決・型検査・elaboration・VC 生成・ATP 変換・証明書受理に使われたものを正確に記録します。

### 安定したフィンガープリント入力

フィンガープリントには、次を含めます。

- 正準的な意味論の内容
- スキーマバージョン
- 言語エディション
- 依存向けのポリシーフィールド
- 正規化された由来 ID
- 可視性が依存する場合の受理済みの証明状態

フィンガープリントには、次を含めません。

- コメント
- 構文に影響する記法以外のソースの整形
- タイムスタンプ
- ローカルの絶対パス
- ワーカー ID
- バックエンドのランタイム実行時間

### 再ビルドのトリガ

この表は、下流の意味論的な再ビルドと、ローカルなアーティファクトの更新を区別します。証明本体のみの編集は、ローカルな証明 witness や実装ハッシュの再ビルドを必要とする場合がありますが、エクスポートされた定理ステートメントと受理済みの証明状態が変わらない限り、下流モジュールを無効化してはなりません。

| 変更 | 下流の意味論的な再ビルド | ローカルなアーティファクトの更新 |
|---|---|---|
| コメントのみ | no | no |
| 定理の証明本体のみ | no | yes（証明 witness または実装ハッシュが変わった場合） |
| 定理ステートメント | yes | yes |
| cluster の意味論 | yes | yes |
| 記法の構文解析結果 | yes | yes |
| registration の coherence 証明状態 | その registration を見られるインポート側について yes | yes |
| 診断の文言のみ | no | yes（診断スナップショットのみ） |

### API 互換性差分

同じ公開 API フィンガープリントが `mizar semver-check` linter（仕様 §23.10）を駆動します。この linter は、仕様 §23.2 のセマンティックバージョニング規則に対して変更を分類します。linter は **サーフェス要素ごとに 2 つの別個のキー** を用います。**正規化された由来 ID** は、ベースラインと候補の間で要素を対応づける *アイデンティティ* であり、**フィンガープリント** は、アイデンティティで対応づけられた後に比較される *値* です。要素ごとの差分すべてにわたる最大値として、必要なバージョンバンプを取ります。この判定は、**構造的差分**（追加・削除・リネーム・署名/文/契約の変更・エクスポート可視性・edition）については厳密ですが、overload 解決のシフトについては入力依存の best-effort なヒューリスティックにすぎません（後述および仕様 §23.10）。アイデンティティと値を分離して保つことが構造的判定を曖昧でなくします。欠落／新規のアイデンティティは削除/リネームであり、同一アイデンティティ／異なる値の対は署名または文の変更です。どちらも通常は MAJOR ですが、両者が混同されることはありません。

ソーステキストの diff ではなくフィンガープリントを再利用することで、linter は 3 つの性質を無償で得ます。リネームは編集ではなく「ID の削除 + ID の追加」になるため正しく MAJOR になります。証明本体のみ・algorithm 本体のみ・コメントのみの編集は、上記の署名／契約フィンガープリント入力から既に除外されているため、決してバンプを強制しません。overload 解決のシフト（仕様 §23.8.3）は、シンボルがテキスト上削除されていなくても、変化したフィンガープリント — registration の、あるいは（属性閉包を通じて解決をシフトさせる、§19.2.1）条件付き cluster・attribute・mode の — として可視になります。

**overload 解決のシフトは best-effort な局所ヒューリスティックで検出される。** 変更が実際に解決をシフトさせるかどうかは下流の呼び出し箇所に依存しますが、それは単一パッケージのフィンガープリント差分の入力ではありません。そこで linter は本パッケージ自身のベースラインと候補のサーフェスに best-effort なヒューリスティックを適用し（仕様 §23.10、W0303）、次のすべてを MAJOR としてフラグします。(a) 同じ解決ドメイン内の既存 ordinary root に対して厳密により具体的、または比較不能で極小である追加・変更された ordinary overload registration、(b) root の joined exposed result facts を変更する、または incompatible refinement join を導入し得る追加・変更された coherent redefinition、(c) 属性閉包を変える追加・変更された条件付き cluster・attribute・mode。閉包は functional cluster の guard 発火、`RefinementJoin` の result facts、および subsumption DAG のエッジ前提条件に影響する（仕様 §19.2.1、§23.7.3、§23.8.2）ため、cluster の変更は新しい候補を登録せずに既存候補の viability や specificity をシフトさせうるからです。正確な結果は依存側しか見られない式に依存するため、この単一パッケージのヒューリスティックは過剰報告 (誤検出) もすれば下流でのみ顕在化するシフトを見逃すこと (検出漏れ) もあり、**健全でも完全でもありません**。reverse-dependency コーパスや記録された呼び出し箇所トレースが利用可能であれば、判定は厳密になります。

**再ビルドのトリガとセマンティックバージョン分類は、関連するが別個の軸である。** 下流の意味論的な再ビルドは「依存側は再コンパイルが必要か?」に答え、セマンティックバージョンのバンプは「依存側のバージョン制約は依然として満たせるか?」に答えます。両者は双方向に乖離します。

| 変更 | 下流の再ビルド | 必要なバンプ |
|---|---|---|
| 新規エクスポート定理または定義 | yes（新しいフィンガープリントが可視） | MINOR — 追加は依存側を壊さない |
| エクスポートの削除 | yes | MAJOR |
| `ensures` の弱化 / `requires` の強化 | yes | MAJOR |
| 定理の証明本体のみ | no | PATCH |

MINOR の追加でも依存スライスが変わるため下流の再ビルドは起きますが、非破壊です。逆にエクスポートの削除は、再ビルドであると同時に MAJOR の破壊です。したがって linter は仕様 §23.2 のもとで差分を直接分類し、再ビルドのトリガ表を互換性の判定としては再利用しません。

### キャッシュの検証

キャッシュヒットが有効なのは、依存スライス内のすべてのフィンガープリントが一致し、有効な検証器ポリシーが記録されたポリシーと互換である場合だけです。

スライスが欠落している、または未対応のスキーマで生成された場合、ビルドは影響を受けるフェーズを保守的に再実行しなければなりません。

## インターフェイス定義

```rust
struct DependencySlice {
    owner: SemanticOwnerId,
    used_theorems: Vec<TheoremFingerprint>,
    used_definitions: Vec<DefinitionFingerprint>,
    used_clusters: Vec<ClusterFingerprint>,
    used_notations: Vec<NotationFingerprint>,
    used_modes: Vec<ModeFingerprint>,
    used_attributes: Vec<AttributeFingerprint>,
}
```

## 制約と前提

- フィンガープリントの計算は、正準的な順序と正準的なシリアライズを用いる。
- 増分再ビルドは最適化であり、完全なクリーンビルドが基準となる挙動である。
- 欠落した依存データを「依存なし」と解釈してはならない。
- 依存スライスは、決定的ビルドのテストにおける回帰スナップショットに含める。
