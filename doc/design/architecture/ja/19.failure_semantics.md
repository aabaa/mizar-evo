stable failure semantics は reproducible build、reliable LSP behavior、unsoundness prevention を重視する regression tests に必要である。

## 設計判断

### Failure Categories

implementation は少なくとも次を区別しなければならない。

- `parse_error`
- `resolve_error`
- `type_error`
- `overload_ambiguity`
- `cluster_loop`
- `atp_timeout`
- `certificate_rejection`
- `kernel_rejection`

これらの category は machine-readable であり、wording change に対して stable である。

kernel / certificate failures は 2 層の classification を使う。`FailureCategory` は build propagation と diagnostic ordering に使う phase-level class を記録する。`malformed_certificate`、`invalid_substitution`、`invalid_sat_proof` のようなより具体的な rejection reason は、failure record の stable structured detail として保存する。

### Phase Ownership

各 failure は、それを sound に検出できる最も早い phase が owner になる。

例:

- invalid syntax は parser failure であり type failure ではない。
- unresolved symbol は resolver failure であり ATP failure ではない。
- ambiguous overload は overload-resolution failure であり kernel failure ではない。
- malformed certificate は certificate rejection である。
- invalid resolution replay は kernel rejection である。

### Deterministic Error Ordering

errors は canonical order で emit する。

1. package id と module path による source file
2. source range
3. phase order
4. severity
5. diagnostic code
6. stable detail key

parallel verification completion order が diagnostic order に影響してはならない。

### Failure Propagation

failure は dependent work だけを block する。

- parse errors は同一 file の semantic phases を block する。
- interface failures は failed interface を必要とする importers を block する。
- failed VC は同一 module 内の independent VC failures を隠してはならない。
- ATP timeout は、別の accepted certificate がない限り obligation を open のままにする。
- certificate / kernel rejection は release policy 下で warning にならない。

blocked work は explicit に blocked として表現し、silent に omit してはならない。

### Soundness Priority

classification が曖昧な場合、implementation は unsound acceptance を防ぎ、最も precise な rejection boundary を保つ category を選ぶ。

failure を success に変えるために fallback inference、hidden coercion、implicit cluster insertion、unchecked ATP result を使ってはならない。

## インターフェイス定義

```rust
enum FailureCategory {
    ParseError,
    ResolveError,
    TypeError,
    OverloadAmbiguity,
    ClusterLoop,
    AtpTimeout,
    CertificateRejection,
    KernelRejection,
}

struct FailureRecord {
    category: FailureCategory,
    phase: PipelinePhase,
    owner: SemanticOwnerId,
    source_range: Option<SourceRange>,
    diagnostic_code: DiagnosticCode,
    stable_detail_key: String,
    rejection_reason: Option<RejectionReason>,
}
```

## 制約と前提

- diagnostic text は改善してよいが、category、phase ownership、stable code の変更には compatibility review が必要である。
- timeout と resource exhaustion は non-acceptance outcome である。
- failure snapshots は regression testing に含める。
- theorem proving success は earlier deterministic failure を無視することに依存してはならない。
