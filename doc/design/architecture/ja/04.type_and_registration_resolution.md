## 設計判断

### 型検査はソフトな型事実を所有し、論理的な証明探索は所有しない

チェッカは、解決済みの構文を、ソフト型を持つ Mizar の表層意味論として解釈します。項または論理式が、現在可視なシグネチャのもとで整形式かどうかを判定し、型に関する主張に根拠付けが必要な場合は証明義務を生成します。

チェッカが所有するもの:

- 正規化された型表現と型ヘッド
- ローカル変数・定数・パラメータの型付けコンテキスト
- 推論された項と論理式の型
- 拡大・縮小の検査
- 型表現や Fraenkel 形式の構造が必要とする sethood の検査
- ローカルな証明コンテキストに追加される、型から導かれる仮定
- ordinary overload root 選択より前に有効な型強制の候補
- 型と registration の診断

チェッカが所有しないもの:

- 複数の型付き候補が残る場合の ordinary overload root と active refinement の選択
- オーバーロードの曖昧性解消のための最終的な `qua` ノードの挿入
- 表層構文から中核論理への elaboration
- 自明でない証明義務の ATP 探索
- kernel evidence の受理

これにより、フェーズ 6 と 7 を決定的かつ Mizar 意味論に局所化し、オーバーロード解決および証明処理との境界を明確に保ちます。

### 型は正規化された述語として表現される

Mizar の型は、型を持たない論理対象の上のソフトな分類です。そのためチェッカは、型を正規化された radix と、極性を意識した属性集合として扱います。

```rust
struct NormalizedType {
    radix: RadixTypeRef,
    args: Vec<TermId>,
    attrs: AttributeSet,
}

struct AttributeSet {
    positive: SortedSet<AttributeInstance>,
    negative: SortedSet<AttributeInstance>,
}
```

mode 型は、型の正規化の際に、定義元の radix と属性の連鎖へ展開されます。ただし診断とメタデータのために、ソース側の mode の綴りは保持します。

正規化は決定的でなければなりません。

- mode の展開は正準的な `SymbolId` 順に従う。
- 重複する属性は 1 つのインスタンスに統合する。
- 矛盾する属性は黙って削除せず、整合性に関する診断として記録する。
- パラメータ付きの属性は、解決された属性シンボルと正規化された引数リストでキー化する。
- ソースの綴りは、意味論的な同一性とは別に保存する。

### ローカルな型コンテキストは層をなす

チェッカは、1 つの可変な大域環境ではなく、層をなしたコンテキストを用います。

| 層 | 含むもの | 寿命 |
|---|---|---|
| モジュールコンテキスト | インポートされたシグネチャ、エクスポートされた registration、組み込み | モジュール全体 |
| 宣言コンテキスト | 宣言のパラメータ、定義ローカルな仮定 | 現在の項目 |
| 証明コンテキスト | thesis、仮定、証明ラベル、ローカルな事実 | 現在の証明ブロック |
| ブロックコンテキスト | `let`・`given`・`consider`・文ローカルな束縛 | 字句ブロック |
| 式コンテキスト | 期待される型、期待される論理式の形、型強制モード | 現在の式 |

入れ子のコンテキストは外側の事実を読めますが、自分の層だけに書き込みます。ブロックを抜けると、定理の結論や明示的に導入された束縛など、外へ持ち出してよい事実だけが凍結されます。

### cluster 解決とリダクション正規化は再生可能である

cluster 解決は、アクティブな cluster registration を不動点に達するまで繰り返し適用し、型事実の閉包を計算します。reduction registration は、§17.6.4 に従う決定的な項正規化として別に扱います。これは項を書き換えて証拠を記録しますが、型事実飽和の尺度には参加しません。

cluster の事実解決は単調です。

- アクティブな cluster registration の適用は導出された事実を追加してよい。
- 既存の事実を削除してはならない。
- ソース走査のたまたまの順序に依存してはならない。
- 正準的な事実キーと飽和の上限によって停止する。

リダクション正規化は、受理されたすべてのリダクションが §17.6.4 の簡約順序を減少させるため停止します。リダクションのステップ制限は存在しません。飽和の上限は cluster の事実閉包にだけ適用します。

両方の機構は再生可能です。

- 導出された cluster 事実は、用いた registration ラベルと入力事実を記録する。
- 各リダクションステップは、規則 FQN、source redex、target term、置換、解消済みガード、戦略監査キーを記録する。
- トレースは正準的な `SymbolId`・`RegistrationId`・FQN・ソース来歴を参照する。
- トレースの順序は決定的である。
- カーネルと診断は、ヒューリスティックな探索を再実行せずに導出を再生・点検できる。

### registration データベースは保留中と有効化済みの registration を分離する

registration 宣言は、それ自身の整形性と正しさの義務が、設定された検証器ポリシーによって
受理された後にのみ有効になります。その受理は後続の VC、proof、kernel、artifact phase から
非同期に到着してよいです。フェーズ 7 は宣言を型検査して保留中の registration レコードを
作成できますが、後続の VC／ATP／カーネルのパイプラインで生成・検証される義務がすでに
成功したとは仮定できません。

チェッカはまず、解決された宣言から `PendingRegistration` を構築します。その後、次を検証します。

- 参照される属性・mode・functor・構造体が、互換性のあるシンボルに解決される。
- パラメータが整合的に型付けされている。
- registration のパターンが整形式である。
- 必要な `existence`・`coherence`・`reducibility` の義務を初期義務として出力できる。
- 保留中レコードが、義務の受理後に有効化できる安定した同一性と来歴を持つ。

自動推論に用いられ、下流モジュールへエクスポートされるアクティブな `RegistrationIndex` に入るのは、有効化済みの registration だけです。不正または未検証の registration は、診断やローカルなメタデータに現れてよいですが、自動的な型事実に寄与してはなりません。

これは宣言するモジュールとインポートする側の双方に適用されます。accepted proof または
artifact input を欠く checker-only pass では、ローカルな registration は、ソース順で先に
現れるというだけでは後続箇所で使われません。この状態は暫定的な pending result であり、
完了した verification pass なら受理する後続 use を最終的に拒否する言語規則ではありません。
その義務が受理された後、完了済み pass または後続の増分 pass は、後続 source item に対してだけ
それを有効化済み registration サマリに含めなければなりません。以前の item は retroactive には
正当化されません。

### existential registration は属性付き型の使用を制御する

属性付きの型は、その型の値を導入するコンテキストで使う前に、値を持つ（inhabited である）ことが分かっていなければなりません。

チェッカは、次の場面で existential registration を参照します。

- `let x be A T`
- 定義本体（definiens）が属性を含む mode 宣言
- 属性を持つ functor の返り値型
- 属性付き型の witness を主張する `consider`・`given`・`take` のコンテキスト

存在の根拠が欠けている場合は型エラーです。チェッカは、後続の診断を有用に保つために劣化した型事実で処理を続けてよいですが、下流のアーティファクトはその値を完全に検証済みとして扱ってはなりません。

### reduction registration は来歴を伴う意味論的な書き換えである

reduction registration は項を簡約できますが、構文解析時の書き換えではなく、来歴を消してはなりません。

簡約の結果は次を記録します。

- 元の項
- 簡約後の項
- 用いた registration
- 解消した副条件
- 簡約が適用されたフェーズ

後続のフェーズは、等号推論や計算ベースの解消に簡約後の項を用いてよいです。一方で、診断と LSP メタデータは元のソース式を指せます。

### オーバーロード候補の絞り込みは許されるが、root 選択は先送りされる

型検査は、候補のアリティ・種別・必須の型制約が型付けされた引数と一致しない場合、その候補を不可能なものとして破棄してよいです。また、残った各候補に型事実を付与してよいです。

ただし、次に依存する場合は最終的な ordinary overload root を選んではなりません。

- 最も具体的な型の比較
- 挿入された `qua` のビュー
- 候補の subsumption 後の、registration から導かれる優先順位
- 利用者向けの曖昧性ポリシー

これらの判断はフェーズ 8 に属し、`05.overload_resolution.md` で定義されます。

### 診断は部分的な型付けを保つ

型検査と registration 解決は、回復可能なエラーの後も処理を継続すべきです。型エラーを持つノードには、成功した型を捏造するのではなく、`TypeStatus::Error` または `TypeStatus::Unknown` の印を付けます。

後続のフェーズは、次を区別しなければなりません。

- 整合的に型付けされたノード
- 回復のための仮定のもとで型付けされたノード
- 型が不明なノード
- 前提となる解決の失敗によりスキップされたノード

これにより、不正な事実を証明やアーティファクトに漏らすことなく、有用なエディタフィードバックを提供できます。

## 解決パイプライン

### ステップ 1: 型検査コンテキストの構築

入力:

- `ResolvedAst`
- `SymbolEnv`
- 依存の `ModuleSummary` アーティファクト
- 依存からの有効化済み registration サマリ

出力:

- モジュールレベルの `TypeContext`
- アクティブな `RegistrationIndex`
- 初期の `TypeFactTable`

責務:

- 組み込みの型ヘッドと組み込みの属性を読み込む。
- 可視な mode・構造体・属性・functor・述語・registration を読み込む。
- トリガキーによって registration を索引化する。
- 宣言とインポートされたシグネチャからローカルな事実を初期化する。
- スキーマ非互換の registration アーティファクトを却下する。

### ステップ 2: 型表現の正規化

入力:

- 解決された型構文
- `SymbolEnv`
- ローカルなパラメータコンテキスト

出力:

- `NormalizedType`
- mode 展開のレコード
- 型表現の診断

責務:

- radix ヘッドを組み込み・mode・構造体に解決する。
- mode を radix と属性へ展開する。
- 属性の極性とパラメータを正規化する。
- 型引数のアリティと種別を検査する。
- 診断のためにソースの綴りを保持する。

### ステップ 3: 宣言とローカルな束縛の検査

入力:

- `ResolvedAst`
- 正規化された型表現
- モジュールコンテキストと宣言コンテキスト

出力:

- 型付けされた宣言シェル
- ローカルな束縛の事実
- 宣言レベルの義務

責務:

- `let`・`given`・`consider`・`reconsider`・定義のパラメータを型検査する。
- 型に関する仮定を正しいコンテキスト層に導入する。
- 属性付きの宣言が存在の根拠を持つことを検査する。
- `reconsider` の縮小義務を出力する。
- 束縛に、ソース位置付きの型情報を付与する。

### ステップ 4: 項と論理式の推論

入力:

- 解決された項と論理式
- ローカルな型付けコンテキスト
- `SymbolEnv` 由来の候補シグネチャ

出力:

- `TypedAst`
- `TypeFactTable`
- 型強制候補のテーブル
- 型の診断

責務:

- 変数・定数・セレクタ・functor 適用・集約の型を推論する。
- 述語と論理式の整形性を検査する。
- `qua` 式を、縮小証明ではなくビューとして検証する。
- 曖昧なオーバーロード集合を、型付けされた候補グループとして保持する。
- フェーズ 8 向けに期待される型の制約を記録する。

### ステップ 5: registration 閉包の解決

入力:

- 現在の `TypeFactTable`
- アクティブな `RegistrationIndex`
- 型付けされた項と正規化された型

出力:

- 拡充された `TypeFactTable`
- `ResolutionTrace`
- registration の診断

責務:

- existential・conditional・functorial registration を属性事実に適用する。
- reduction registration を §17.6.4 に従う決定的な項正規化として適用する。
- cluster 属性事実を決定的な不動点まで飽和させる。
- 導出された事実ごとに入力事実と registration ラベルを記録し、各リダクションステップに局所書き換え証拠を記録する。
- cluster の循環・矛盾・飽和上限、およびリダクションの簡約順序違反・ガード失敗を検出する。
- アクティブな `RegistrationIndex` に存在しない registration を無視する。

### ステップ 6: 保留中の registration 宣言の検証

入力:

- 型付けされた registration 宣言
- 証明義務になるべき正しさの条件
- ソースの来歴と registration ラベル

出力:

- 保留中のローカル registration
- 却下された保留中 registration
- 初期の registration 義務

責務:

- パターンの整形性を検証する。
- registration ラベルを `RegistrationId` に束縛する。
- 初期の `existence`・`coherence`・`reducibility` の義務を生成する。
- 保留中 registration ごとに、初期義務との依存関係を記録する。
- 有効化とエクスポートは、検証器ポリシーが義務を受理した後のアーティファクトフェーズに委ねる。

## インターフェイス定義

### 型環境

```rust
struct TypeContext {
    module: ModuleId,
    symbols: SymbolEnv,
    registrations: RegistrationIndex,
    facts: TypeFactTable,
    scopes: Vec<TypeScope>,
}

struct TypeScope {
    kind: TypeScopeKind,
    bindings: BindingTypeTable,
    assumptions: Vec<TypeFactId>,
}
```

`TypeContext` はチェッカが所有し、直接シリアライズはされません。アーティファクトには、推論された型・事実・有効化済み registration の安定した射影が入ります。

### 型事実

```rust
struct TypeFact {
    id: TypeFactId,
    subject: TermId,
    predicate: TypePredicate,
    polarity: Polarity,
    provenance: FactProvenance,
}

enum FactProvenance {
    Declared(SourceRange),
    Assumed(LabelRef),
    Inferred(ResolutionStepId),
    Obligation(InitialObligationId),
    Builtin(BuiltinRuleId),
}
```

型事実は、型検査・registration 解決・診断・VC 生成に共通の単位です。

### registration 索引

この checker-side active `RegistrationIndex` は、整形性と義務が受理された
registration だけを含む。`doc/design/mizar-resolve/ja/env.md` で説明する
resolver-side pre-checker registration declaration index とは別物である。

```rust
struct RegistrationIndex {
    existential: TriggerMap<TypeHeadKey, Vec<RegistrationId>>,
    conditional: TriggerMap<AttributeKey, Vec<RegistrationId>>,
    functorial: TriggerMap<FunctorKey, Vec<RegistrationId>>,
    reduction: TriggerMap<TermHeadKey, Vec<RegistrationId>>,
}

struct RegistrationDef {
    id: RegistrationId,
    label: LabelRef,
    kind: RegistrationKind,
    pattern: RegistrationPattern,
    parameters: Vec<TypedParameter>,
    correctness: CorrectnessCondition,
    source_range: SourceRange,
    origin: ModuleId,
}
```

トリガマップは最適化のための索引であり、意味論を定義しません。意味論は、解決された `RegistrationDef` レコードによって定義されます。

アクティブな `RegistrationIndex` には、依存の registration と以前に検証済みのローカル registration だけが含まれます。新たに宣言されたローカル registration は、その初期義務が `VcId` に対応付けられ、解消され、受理されるまで、`PendingRegistration` として別に表現されます。

### 解決トレース

具体的な `ResolutionTrace` アーティファクトスキーマは [17.cluster_trace_format.md](./17.cluster_trace_format.md) で定義します。フェーズ 7 は、そのスキーマに、型事実を導出する cluster step と、自動的な項書き換えを表す reduction step を格納します。

このフェーズでは、cluster step は適用された registration、入力型事実、導出された事実、ソース来歴を保持します。reduction step は、適用された registration と規則 FQN、包含項 fingerprint、redex path、source redex、target term、置換、解消済みガード証拠、有効な rule-view fingerprint、selection key、ソース来歴を保持します。これらの値は、トレース形式文書の正準フィールド名でシリアライズします。

`ResolutionTrace` は、自動的な型付けとリダクション正規化の再生可能な説明です。診断・カーネルの再生・アーティファクト・`@show_resolution` メタデータが消費します。

### 型付き AST

```rust
struct TypedAst {
    source_id: SourceId,
    module_id: ModuleId,
    nodes: AstArena<TypedNode>,
    types: TypeTable,
    facts: TypeFactTable,
    coercions: CoercionCandidateTable,
    errors: TypeErrorTable,
}
```

`TypedAst` はソースの形を保ちます。registration とオーバーロードの解決に十分な情報を含みますが、overload root と active refinement が未確定の場合があるため、まだ最終的な表層意味論ではありません。

## 診断

型と registration の診断は、解決済みのソース構文に対する意味論的な診断です。

| 診断クラス | 例 |
|---|---|
| 型表現 | 未知の型ヘッド、型引数の個数の誤り、矛盾する属性 |
| 束縛 | 宣言された型が値を持たない、ローカル束縛の重複、ローカルな事実の不正な持ち出し |
| 項の型付け | セレクタが利用不可、functor 引数の型不一致、論理式が期待される |
| 変換 | 不正な `qua`、証明を伴わない縮小、失敗した `reconsider` 義務 |
| registration | 不正な cluster パターン、正しさの証明の欠落、無効な registration の使用 |
| 解決 | cluster の循環、飽和上限、矛盾する導出属性 |
| 簡約 | 簡約順序違反、解消されない副条件 |

診断には次を含めるべきです。

- 使用箇所における主たるソース位置
- 関連する宣言・registration・導出された事実の副次的な位置情報
- 機械可読な、期待される型と実際の型の要約
- 推論された事実が診断の原因である場合の、解決トレースへの参照
- 型または registration の範囲の安定した診断コード

## 増分処理と再現性

型と registration のキャッシュキーには次を含めます。

- `ResolvedAst` のハッシュ
- `SymbolEnv` のハッシュ
- 可視な依存 registration サマリのハッシュ
- 有効な言語エディション
- チェッカのスキーマバージョン
- registration の有効化に影響する検証器設定

変更は次を無効化します。

- 変更されたモジュールの `TypedAst`・`TypeFactTable`・`ResolutionTrace`
- 変更された有効化済み registration をインポートする下流モジュール
- 変更されたエクスポート型シグネチャに依存する下流モジュール
- 影響を受けた型事実を消費するオーバーロード・elaboration・VC・アーティファクトの各フェーズ

順序は決定的でなければなりません。

- registration のトリガは正準的な `RegistrationId` でソートする。
- 不動点のワークリストは正準的な事実キーを用いる。
- トレースのステップは、安定したタイブレークを伴う導出順でシリアライズする。
- 同じ範囲の診断は、診断コードと安定した型キーで順序付ける。

## 検討した代替案

### 型問い合わせのたびに cluster をインライン解決する

問い合わせのたびに cluster を遅延解決すると前倒しの作業は減りますが、診断・再生・増分キャッシュキーが扱いにくくなります。採用した設計では、明示的な閉包を計算してトレースを記録します。

### registration を即座に信頼された宣言として扱う

即時の有効化は単純ですが、正しさの義務が受理される前に、不正な registration が後続の検査に影響してしまいます。採用した設計では、保留中の registration と有効化済みの registration を分離します。

### 型・registration・オーバーロードの解決を統合する

これらのフェーズを統合すると中間データ構造は減りますが、事実の導出と候補の選択の違いが曖昧になります。採用した設計では、型と registration のフェーズが事実を計算し、最終的なオーバーロードの選択を専用フェーズに委ねます。

## 採用方針

Mizar Evo は、フェーズ 6 と 7 に対して 2 部構成のチェッカ境界を採用します。

型検査は型表現を正規化し、ローカルな型コンテキストを構築し、項と論理式を検査して、`TypedAst` と型事実を生成します。続いて registration 解決が、有効化済みの cluster registration で cluster 事実の閉包を計算し、有効化済みの reduction registration を決定的な項正規化として適用して、その両方を再生可能な `ResolutionTrace` に記録します。

その結果として、型付けされた構文・明示的な事実・追跡可能な自動導出という、決定的な Mizar 側の意味論的情報が得られます。オーバーロード解決・elaboration・証明義務の生成・ATP ディスパッチ・kernel evidence 検査は、型や registration の推論を再実行せずにこれらの出力を消費します。

## 制約と前提

- 型検査は、モジュール／名前解決とシグネチャ収集が使用可能な `SymbolEnv` を生成した後にのみ開始する。
- 型システムはソフトである。すべての型事実は、論理述語または registration から導かれた事実として説明可能でなければならない。
- 依存からの有効化済み registration は、互換性のあるスキーマバージョンを持つ検証済みアーティファクトの射影としてのみ信頼する。
- 未検証のローカル registration は、自動推論に影響してはならない。
- ローカル registration の有効化は、設定された検証器ポリシーによって証明義務が受理されるまで遅延される。
- フェーズ 6-7 の構造は `InitialObligationId` を参照してよい。`VcId` は後続の VC 生成で割り当てられる。
- registration 解決は決定的に停止するか、上限つき飽和の診断を出力しなければならない。
- チェッカは回復可能なエラーの後に部分的な型付け出力を生成してよいが、不正な事実を検証済みメタデータとしてエクスポートしてはならない。
- ATP バックエンドは、型検査・registration 解決・オーバーロード解決を実行しない。
