## 設計判断

### Type Checking Owns Soft Type Facts, Not Logical Proof Search

Checker は resolved syntax を soft-typed な Mizar surface semantics として解釈する。Term または formula が現在 visible な signatures のもとで well-formed かを判定し、type claim に justification が必要な場合は proof obligations を生成する。

Checker は次を所有する。

- normalized type expressions and type heads
- local variable、constant、parameter typing contexts
- inferred term and formula types
- widening and narrowing checks
- type expressions と Fraenkel-style constructs が必要とする sethood checks
- local proof contexts に追加される type-derived assumptions
- overload winner selection 前に有効な coercion candidates
- type and registration diagnostics

Checker は次を所有しない。

- 複数 typed candidates が残る場合の overload winner selection
- overload disambiguation のための final `qua` node insertion
- surface syntax から core logic への elaboration
- non-trivial proof obligations の ATP search
- proof certificate acceptance

これにより phase 6 と 7 を deterministic かつ Mizar semantics に局所化し、overload resolution と proof processing の境界を明確に保つ。

### Types Are Represented as Normalized Predicates

Mizar の types は untyped logical objects 上の soft classifications である。そのため checker は type を normalized radix と polarity-aware attribute set として扱う。

```rust
struct NormalizedType {
    radix: RadixTypeRef,
    args: Vec<TermId>,
    attrs: AttributeSet,
}

struct AttributeSet {
    positive: SortedSet<AttributeInstance>,
    negative: SortedSet<AttributeInstance>,
}
```

Mode types は type normalization 中に defining radix と attribute chain へ unfold される。ただし diagnostics と metadata のため、source-facing mode spelling は保持する。

Normalization は deterministic でなければならない。

- mode unfolding は canonical `SymbolId` order に従う
- duplicate attributes は 1 つの instance に collapse する
- contradictory attributes は黙って削除せず consistency diagnostics として記録する
- parameterized attributes は resolved attribute symbol と normalized argument list で key 化する
- source spelling は semantic identity とは別に保存する

### Local Type Contexts Are Layered

Checker は 1 つの mutable global environment ではなく layered contexts を使う。

| Layer | Contains | Lifetime |
|---|---|---|
| module context | imported signatures、exported registrations、built-ins | entire module |
| declaration context | declaration parameters、definition-local assumptions | current item |
| proof context | thesis、assumptions、proof labels、local facts | current proof block |
| block context | `let`、`given`、`consider`、statement-local bindings | lexical block |
| expression context | expected type、expected formula shape、coercion mode | current expression |

Nested contexts は outer facts を読めるが、自分の layer だけに書き込む。Block から出ると、theorem conclusions や明示的に導入された bindings など、外へ逃がしてよい facts だけが freeze される。

### Registration Resolution Is Monotone and Replayable

Cluster and registration resolution は active registrations を fixed point に達するまで繰り返し適用し、type facts の closure を計算する。

Resolution は monotone である。

- registration application は derived facts を追加してよい
- existing facts を削除してはならない
- source traversal の偶然に依存してはならない
- canonical keys と saturation limits により termination する

Resolution は replayable である。

- derived fact は、使った registration label と input facts を記録する
- traces は canonical `SymbolId`、`RegistrationId`、source ranges を参照する
- trace ordering は deterministic である
- kernel と diagnostics は heuristic search を再実行せずに derivation を replay または inspect できる

### Registration Databases Separate Pending and Activated Registrations

Registration declaration は、それ自身の well-formedness と correctness obligations が configured verifier policy により受理された後にのみ active になる。Phase 7 は declaration を type-check し pending registration records を作成できるが、後続の VC/ATP/kernel pipeline で生成・検証される obligations がすでに成功したとは仮定できない。

Checker はまず resolved declaration から `PendingRegistration` を構築する。その後、次を検証する。

- referenced attributes、modes、functors、structures が compatible symbols に解決される
- parameters が well-typed である
- registration pattern が well-formed である
- required `existence`、`coherence`、`reducibility` obligations を initial obligations として emit できる
- pending record が obligations 受理後に activate できる stable identity と provenance を持つ

Activated registrations だけが automatic inference に使われ downstream modules へ export される active `RegistrationIndex` に入る。Invalid または unverified な registrations は diagnostics と local metadata に現れてよいが、自動 type facts に寄与してはならない。

これは declaring module と importers の両方に適用される。Local registration は source order で先に現れるだけでは、同じ unchecked pass の後続箇所で使われない。その obligations が受理された後、後続の incremental pass はそれを activated registration summary に含めてよい。

### Existential Registrations Gate Attributed Type Use

Attributed types は、その type の値を導入する context で使う前に inhabited であることが知られていなければならない。

Checker は次で existential registrations を参照する。

- `let x be A T`
- definiens が attributes を含む mode declarations
- attributes を持つ functor return types
- attributed types の witnesses を主張する `consider`、`given`、`take` contexts

Existence evidence が欠ける場合は type error である。Checker は後続 diagnostics を有用に保つため degraded type fact で処理を続けてよいが、downstream artifact はその値を fully verified として扱ってはならない。

### Reduction Registrations Are Semantic Rewrites with Provenance

Reduction registrations は terms を simplify できるが、parser rewrites ではなく provenance を消してはならない。

Reduction result は次を記録する。

- source term
- reduced term
- 使用した registration
- discharge された side conditions
- reduction が適用された phase

Later phases は equality reasoning と computation-based discharge に reduced terms を使ってよい。一方 diagnostics と LSP metadata は original source expression を指せる。

### Overload Candidate Filtering Is Allowed, Winner Selection Is Deferred

Type checking は、candidate の arity、kind、mandatory type constraints が typed arguments と一致しない場合、impossible candidates を破棄してよい。また、残る各 candidate に type facts を付与してよい。

ただし、次に依存する場合は final overload winner を選んではならない。

- most-specific type comparison
- inserted `qua` views
- candidate subsumption 後の registration-derived preference
- user-facing ambiguity policy

これらの判断は phase 8 に属し、`05.overload_resolution.md` で定義される。

### Diagnostics Preserve Partial Typing

Type checking and registration resolution は recoverable errors の後も処理を継続するべきである。Type error を持つ node には、成功した type を捏造するのではなく `TypeStatus::Error` または `TypeStatus::Unknown` marker を付与する。

Later phases は次を区別しなければならない。

- well-typed nodes
- recoverable assumptions のもとで typed された nodes
- unknown type の nodes
- prerequisite resolution failure により skipped された nodes

これにより invalid facts を proofs や artifacts に漏らさず、有用な editor feedback を提供できる。

## 解決パイプライン

### Step 1: Build the Type Checking Context

Input:

- `ResolvedAst`
- `SymbolEnv`
- dependency `ModuleSummary` artifacts
- dependencies からの activated registration summaries

Output:

- module-level `TypeContext`
- active `RegistrationIndex`
- initial `TypeFactTable`

Responsibilities:

- built-in type heads と built-in attributes を読み込む
- visible modes、structures、attributes、functors、predicates、registrations を読み込む
- trigger keys により registrations を index する
- declarations と imported signatures から local facts を seed する
- schema-incompatible registration artifacts を拒否する

### Step 2: Normalize Type Expressions

Input:

- resolved type syntax
- `SymbolEnv`
- local parameter context

Output:

- `NormalizedType`
- mode unfolding records
- type-expression diagnostics

Responsibilities:

- radix heads を built-ins、modes、structures に解決する
- modes を radix plus attributes に unfold する
- attribute polarity と parameters を normalize する
- type argument arity and kind を検査する
- diagnostics のため source spelling を保持する

### Step 3: Check Declarations and Local Bindings

Input:

- `ResolvedAst`
- normalized type expressions
- module and declaration contexts

Output:

- typed declaration shells
- local binding facts
- declaration-level obligations

Responsibilities:

- `let`、`given`、`consider`、`reconsider`、definition parameters を type-check する
- type assumptions を正しい context layer に導入する
- attributed declarations が existence evidence を持つことを検査する
- `reconsider` の narrowing obligations を emit する
- bindings に source-spanned type information を付与する

### Step 4: Infer Terms and Formulas

Input:

- resolved terms and formulas
- local typing context
- `SymbolEnv` からの candidate signatures

Output:

- `TypedAst`
- `TypeFactTable`
- coercion candidate table
- type diagnostics

Responsibilities:

- variables、constants、selectors、functor applications、aggregates の types を infer する
- predicate and formula well-formedness を検査する
- `qua` expressions を narrowing proofs ではなく views として validate する
- ambiguous overload sets を typed candidate groups として保持する
- phase 8 向けに expected-type constraints を記録する

### Step 5: Resolve Registration Closure

Input:

- current `TypeFactTable`
- active `RegistrationIndex`
- typed terms and normalized types

Output:

- enriched `TypeFactTable`
- `ResolutionTrace`
- registration diagnostics

Responsibilities:

- existential、conditional、functorial、reduction registrations を適用する
- attribute facts を deterministic fixed point まで saturate する
- derived fact ごとに input facts と registration label を記録する
- cycles、contradictions、saturation limits を検出する
- active `RegistrationIndex` に存在しない registrations を無視する

### Step 6: Validate Pending Registration Declarations

Input:

- typed registration declarations
- proof obligations になるべき correctness conditions
- source provenance and registration labels

Output:

- pending local registrations
- rejected pending registrations
- initial registration obligations

Responsibilities:

- pattern well-formedness を検証する
- registration labels を `RegistrationId`s に bind する
- initial `existence`、`coherence`、`reducibility` obligations を生成する
- pending registration ごとに initial obligations との dependency を記録する
- activation and export は verifier policy が obligations を受理した後の artifact phase に委ねる

## インターフェイス定義

### Type Environment

```rust
struct TypeContext {
    module: ModuleId,
    symbols: SymbolEnv,
    registrations: RegistrationIndex,
    facts: TypeFactTable,
    scopes: Vec<TypeScope>,
}

struct TypeScope {
    kind: TypeScopeKind,
    bindings: BindingTypeTable,
    assumptions: Vec<TypeFactId>,
}
```

`TypeContext` は checker-owned であり、直接 serialize されない。Artifacts には inferred types、facts、activated registrations の stable projections が入る。

### Type Facts

```rust
struct TypeFact {
    id: TypeFactId,
    subject: TermId,
    predicate: TypePredicate,
    polarity: Polarity,
    provenance: FactProvenance,
}

enum FactProvenance {
    Declared(SourceRange),
    Assumed(LabelRef),
    Inferred(ResolutionStepId),
    Obligation(InitialObligationId),
    Builtin(BuiltinRuleId),
}
```

Type facts は type checking、registration resolution、diagnostics、VC generation の共通単位である。

### Registration Index

```rust
struct RegistrationIndex {
    existential: TriggerMap<TypeHeadKey, Vec<RegistrationId>>,
    conditional: TriggerMap<AttributeKey, Vec<RegistrationId>>,
    functorial: TriggerMap<FunctorKey, Vec<RegistrationId>>,
    reduction: TriggerMap<TermHeadKey, Vec<RegistrationId>>,
}

struct RegistrationDef {
    id: RegistrationId,
    label: LabelRef,
    kind: RegistrationKind,
    pattern: RegistrationPattern,
    parameters: Vec<TypedParameter>,
    correctness: CorrectnessCondition,
    source_range: SourceRange,
    origin: ModuleId,
}
```

Trigger maps は optimization indexes であり、semantics を定義しない。Semantics は resolved `RegistrationDef` records により定義される。

Active `RegistrationIndex` は dependency registrations と previously verified local registrations だけを含む。Newly declared local registrations は、それらの initial obligations が `VcId`s に写像され、discharge され、受理されるまで `PendingRegistration`s として別に表現される。

### Resolution Trace

```rust
struct ResolutionTrace {
    module: ModuleId,
    steps: Vec<ResolutionStep>,
}

struct ResolutionStep {
    id: ResolutionStepId,
    registration: RegistrationId,
    inputs: Vec<TypeFactId>,
    output: ResolutionOutput,
    source_range: SourceRange,
}

enum ResolutionOutput {
    Fact(TypeFactId),
    Reduction { from: TermId, to: TermId },
    NoChange,
}
```

`ResolutionTrace` は automatic typing の replayable explanation である。Diagnostics、kernel replay、artifacts、`@show_resolution` metadata が consume する。

### Typed AST

```rust
struct TypedAst {
    source_id: SourceId,
    module_id: ModuleId,
    nodes: AstArena<TypedNode>,
    types: TypeTable,
    facts: TypeFactTable,
    coercions: CoercionCandidateTable,
    errors: TypeErrorTable,
}
```

`TypedAst` は source shape を保持する。Registration and overload resolution に十分な情報を含むが、overload winners が未確定の場合があるため、まだ final surface semantics ではない。

## 診断

Type and registration diagnostics は resolved source syntax 上の semantic diagnostics である。

| Diagnostic Class | Example |
|---|---|
| type expression | unknown type head、wrong type argument count、contradictory attributes |
| binding | declared type is not inhabited、duplicate local binding、illegal escape of local fact |
| term typing | selector unavailable、functor argument type mismatch、formula expected |
| conversion | invalid `qua`、narrowing without proof、failed `reconsider` obligation |
| registration | malformed cluster pattern、missing correctness proof、inactive registration used |
| resolution | cluster cycle、saturation limit、contradictory derived attributes |
| reduction | non-terminating reduction pattern、side condition not discharged |

Diagnostics は次を含むべきである。

- use site の primary source span
- 関連する declaration、registration、derived fact の secondary spans
- machine-readable expected and actual type summaries
- inferred fact が diagnostic の原因である場合の resolution trace references
- type または registration ranges の stable diagnostic codes

## 増分処理と再現性

Type and registration cache keys は次を含む。

- `ResolvedAst` hash
- `SymbolEnv` hash
- visible dependency registration summary hashes
- active language edition
- checker schema version
- registration activation に影響する verifier settings

変更は次を invalidate する。

- changed module の `TypedAst`、`TypeFactTable`、`ResolutionTrace`
- changed activated registrations を import する downstream modules
- changed exported type signatures に依存する downstream modules
- affected typed facts を consume する overload、elaboration、VC、artifact phases

Ordering は deterministic でなければならない。

- registration triggers は canonical `RegistrationId` で sort する
- fixed-point worklists は canonical fact keys を使う
- trace steps は stable tie-breaking 付きの derivation order で serialize する
- 同じ span の diagnostics は diagnostic code と stable type key で order する

## 検討した代替案

### Inline Cluster Resolution During Every Type Query

各 query で clusters を lazy に解決すると upfront work は減るが、diagnostics、replay、incremental cache keys が扱いにくくなる。採用案では explicit closures を計算し traces を記録する。

### Treat Registrations as Trusted Declarations Immediately

即時 activation は単純だが、correctness obligations が受理される前に invalid registrations が後続 checking へ影響する。採用案では pending registrations と activated registrations を分離する。

### Combine Type, Registration, and Overload Resolution

これらの phases を結合すると intermediate data structures は減るが、fact derivation と candidate choice の違いが曖昧になる。採用案では type and registration phases が facts を計算し、final overload selection は専用 phase に委ねる。

## 採用方針

Mizar Evo は phase 6 と 7 に対して two-part checker boundary を採用する。

Type checking は type expressions を normalize し、local type contexts を構築し、terms and formulas を検査し、`TypedAst` と type facts を生成する。その後 registration resolution が activated cluster and reduction registrations で facts を saturate し、replayable `ResolutionTrace` を記録する。

結果として、typed syntax、explicit facts、traceable automatic derivations という deterministic な Mizar-side semantic information が得られる。Overload resolution、elaboration、proof obligation generation、ATP dispatch、kernel certificate checking は、type or registration inference を再実行せずにこれらの outputs を consume する。

## 制約と前提

- Type checking は module/name resolution と signature collection が usable `SymbolEnv` を生成した後にだけ開始する。
- Type system は soft である。すべての type facts は logical predicates または registration-derived facts として説明可能でなければならない。
- Dependencies からの activated registrations は、compatible schema versions を持つ verified artifact projections としてのみ trust する。
- Unverified local registrations は automatic inference に影響してはならない。
- Local registration activation は、configured verifier policy により proof obligations が受理されるまで遅延される。
- Phase 6-7 structures は `InitialObligationId`s を参照してよい。`VcId`s は後続の VC generation で割り当てられる。
- Registration resolution は deterministic に terminate するか、bounded saturation diagnostic を emit しなければならない。
- Checker は recoverable errors 後に partial typed output を生成してよいが、invalid facts を verified metadata として export してはならない。
- ATP backends は type checking、registration resolution、overload resolution を実行しない。
