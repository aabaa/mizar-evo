# Architecture: Artifact and Incremental Build

> Canonical language: English. English canonical version: [../en/11.artifact_and_incremental_build.md](../en/11.artifact_and_incremental_build.md).

## 目的

この文書は、Mizar Evo が verified artifact を出力し、incremental build cache を更新する方法を定義する。

これは pipeline phase 15 を詳細化する。前段の phase は verified semantic data、diagnostics、ATP logs、kernel-checked proof status を生成する。Phase 15 はそれらの internal result を stable な on-disk artifact に投影し、reproducibility に必要な provenance を記録し、後続 build と IDE workflow が使う cache entries を更新する。

## Design Decisions

### Artifact Classes

Phase 15 は三種類の data を書き出す。それぞれ compatibility と retention rule が異なる。

| Class | Examples | Consumer | Stability |
|---|---|---|---|
| Published artifacts | `*.mizir.json`, exported module summaries, proof witness references | downstream packages, docs, LSP, AI tooling | versioned stable schema |
| Development artifacts | `missing_facts.json`, `*.atp-log`, diagnostic explanations, debug dumps | local developer workflows | structured なものは versioned、publication input ではない |
| Build cache entries | phase cache records, `cluster-db/`, import-scoped views, resolution traces | compiler and LSP server | internal schema、toolchain/cache version で invalidated |

Package compatibility を定義するのは published artifacts だけである。Development artifacts は release build で存在しなくてもよく、cache entries はいつ削除されても package semantics を変えない。

### VerifiedArtifact Is a Projection

`*.mizir.json` は verified phase outputs から emit されるが、compiler state の serialize ではない。

含むもの:

- source identity and hashes
- public exports and dependency-facing signatures
- IDE and AI tools が必要とする expression metadata
- verification obligations and accepted proof status
- proof witness references and content hashes
- completed build pass で出た diagnostics

含まないもの:

- raw AST arenas and internal IR layouts
- full ATP stdout/stderr streams
- kernel-internal proof state
- non-deterministic build-session identifiers
- local diagnostics 用に明示要求された場合を除く absolute paths

Emitter は複数の internal input から artifact を計算してよいが、downstream tool はそれらの internal input に直接依存してはならない。

### Publication Boundary

Package は active release policy のもとで proof obligations が受理された published artifacts からのみ publish できる。

Publication check は次を reject する。

- `proof_status = "open"` の obligation
- `require_kernel_certificates = true` のときの externally attested proofs
- publishing tool が support しない schema version の artifacts
- 記録された source hash が source file と一致しない artifacts
- content hash が referenced file と一致しない proof witness references

`missing_facts.json` のような development artifact は published package interface の一部にならない。

### Cache Is an Optimization, Not Authority

Incremental cache entries は後続 build の work を skip するために使えるが、それ自体が correctness を確立するわけではない。Cached proof result を reuse できるのは、accepted result を変え得るすべての input が cache key に含まれている場合だけである。

Cache keys は次を含む。

- source hash and normalized source path
- package id、module path、language edition、lockfile hash
- toolchain version and relevant phase schema versions
- verifier configuration, including certificate policy and computation limits
- dependency artifact hashes and imported symbol summaries
- file から visible な registration and cluster index hashes
- 必要に応じて VC、ATP problem、backend profile、certificate hashes

Key component が欠けている場合、または unsupported toolchain/cache schema で生成された場合、その entry は miss として扱う。

### Deterministic Artifact Output

Artifacts は、同一 source、lockfile、toolchain、verifier settings、accepted certificates に対して reproducible でなければならない。

Phase 15 は次によってこれを保証する。

- serialization 前に maps and sets を canonical keys で sort する
- deterministic source ranges と stable per-file ordering から artifact IDs を割り当てる
- `verified_at` のような user-visible timestamps は、canonical content hashes and publication equivalence checks から除外される local artifact metadata としてのみ保存する
- paths は workspace root または package root からの relative path として記録する
- referenced witness and log files は content hash で記録する
- canonical formatter で JSON を serialize する

ATP runtime variability は provenance として記録されるが、stable artifact entries の order や name を変えてはならない。

### Atomic Writes

Phase 15 は atomic replace protocol で artifacts を書く。

1. Target directory 内の temporary file に artifact を serialize する。
2. File contents を flush する。
3. Temporary file を previous artifact の上に rename する。
4. Per-package artifact manifest を更新する。
5. Platform が support する場合、containing directory を flush する。

Reader は previous complete artifact または new complete artifact のどちらかを見る。Partially-written JSON files は build error であり、黙って受け入れてはならない。

## Incremental Build Model

### Build Graph

Build system は少なくとも次の node classes を持つ dependency graph を管理する。

| Node | Invalidated By |
|---|---|
| `SourceUnit` | source content or source path change |
| `ModuleSummary` | exports, imports, or public declaration signature change |
| `RegistrationIndexContribution` | registration declaration or accepted coherence proof change |
| `ResolvedTypedArtifactInput` | source, imports, symbol summaries, type rules, registrations |
| `VcSet` | core IR, control-flow IR, type facts, resolution traces, verifier policy |
| `AtpProblemRecord` | VC, premise set, encoding profile, backend profile |
| `KernelProofRecord` | certificate, kernel version, trusted environment, dependency proof refs |
| `VerifiedArtifact` | any published projection input |

Invalidation は reverse dependencies に流れる。Source edit は edited file と、その imported public interface、registration view、proof context に依存するすべての file を recheck する。Proof-body-only edit は、exported signatures と accepted theorem statements が変わらない場合、downstream modules の invalidation を避けられる。

### Interface Hashes and Implementation Hashes

各 verified module は二つの重要な hash を持つ。

```rust
struct ModuleArtifactHashes {
    interface_hash: Hash,
    implementation_hash: Hash,
}
```

`interface_hash` は importers から visible な data を cover する。

- exported definitions, theorem statements, modes, predicates, functors, attributes, structs, and registrations
- exported theorems and activated registrations の proof acceptance status
- dependency-facing notation and overload summaries
- importer interpretation に影響する schema versions

`implementation_hash` は source file の full verified artifact を cover し、local expression metadata、local obligations、diagnostics、proof witness references を含む。

Downstream semantic phases は主に `interface_hash` に依存する。LSP、documentation、local rebuild decisions は `implementation_hash` にも依存してよい。

### Cache Reuse Rules

Cached phase output を reuse できるのは次の場合だけである。

- cache key が requested key と完全に一致する
- referenced dependency artifact がすべて存在し、recorded hash と一致する
- producing toolchain and schema version が compatible である
- active verifier policy が entry に記録された policy と同等以上に strict である
- reused proof records が accepted proof witnesses または deterministic built-in discharge records を指している

Policy strictness は認識済み policy fields に対してのみ monotonic である。たとえば、`require_kernel_certificates = true` で生成された entry は同じ policy で reuse できるが、externally attested proof を policy bit の変更だけで kernel-verified に upgrade することはできない。

### Registration and Cluster Cache Updates

`cluster-db/` cache は parsed declarations から直接ではなく、verified registration contributions から更新される。Coherence proof が受理されていない registration は importers から visible ではなく、`interface_hash` の一部でもない。

Changed source files に対して、phase 15 は次を更新する。

- changed module の registration contribution records
- origins が changed module である derived graph and index entries
- changed origins を含む import-scoped visibility views
- affected files の resolution traces

Cache は origin metadata を保存するため、registration が delete または rename されたときに stale contributions を remove できる。

## Interface Definitions

### Artifact Manifest

各 package artifact directory は emitted artifacts を index する manifest を持つ。

```rust
struct ArtifactManifest {
    schema_version: String,
    package_id: PackageId,
    package_version: Version,
    lockfile_hash: Hash,
    toolchain: ToolchainInfo,
    verifier_config_hash: Hash,
    modules: Vec<ModuleArtifactEntry>,
}

struct ModuleArtifactEntry {
    module_path: ModulePath,
    source_file: PathBuf,
    source_hash: Hash,
    artifact_file: PathBuf,
    artifact_hash: Hash,
    interface_hash: Hash,
    implementation_hash: Hash,
    proof_witnesses: Vec<ProofWitnessEntry>,
    diagnostics_hash: Option<Hash>,
}
```

Manifest は downstream tools が最初に読む file である。これにより arbitrary build output を scan せずに module artifacts を locate できる。

### Verified Artifact Schema

`VerifiedArtifact` の architectural shape は次の通りである。

```rust
struct VerifiedArtifact {
    schema_version: String,
    source_file: PathBuf,
    source_hash: Hash,
    verified_at: Option<Timestamp>,
    module_path: ModulePath,
    interface_hash: Hash,
    implementation_hash: Hash,
    exports: Vec<ExportedItem>,
    expressions: Vec<ExpressionMetadata>,
    obligations: Vec<ObligationMetadata>,
    proof_witnesses: Vec<ProofWitnessRef>,
    diagnostics: Vec<Diagnostic>,
    provenance: BuildProvenance,
}

struct BuildProvenance {
    toolchain: ToolchainInfo,
    language_edition: Edition,
    lockfile_hash: Hash,
    verifier_config_hash: Hash,
    dependency_artifact_hashes: Vec<DependencyArtifactHash>,
    cache_key: Hash,
}
```

`verified_at` は user-visible local metadata である。IDE や `build/` を見る人間には有用だが、`interface_hash`、`implementation_hash`、canonical reproducibility comparisons には含めない。

`BuildProvenance` は reproducibility and debugging の一部である。ただし referenced artifacts and proof witnesses の check の代替ではない。

### Cache Record

```rust
struct CacheRecord<T> {
    cache_schema_version: String,
    key: CacheKey,
    produced_by: ToolchainInfo,
    dependencies: Vec<CacheDependency>,
    output_hash: Hash,
    output: T,
}

struct CacheKey {
    phase: PipelinePhase,
    source_hash: Option<Hash>,
    input_hashes: Vec<Hash>,
    config_hash: Hash,
    dependency_hashes: Vec<Hash>,
    schema_versions: Vec<SchemaVersion>,
}
```

Cache records は internal である。Cache schema version と compatibility rules が明示されていれば binary encoding を使ってよい。

## Alternatives Considered

### Raw IR Artifacts

Internal IRs を serialize すると初期 tooling は簡単になるが、implementation details を freeze し、schema compatibility を管理しにくくする。採用案では stable projections を emit し、raw IR dumps は optional debug artifacts に留める。

### Trusting Cached Proof Results Without Witness Hashes

これは incremental builds を速くするが、stale または corrupted cache entry が invalid proof status を黙って維持できてしまう。採用案では proof-related cache keys に proof witness または deterministic discharge hash を要求する。

### Rebuilding Whole Packages on Any Change

Whole-package rebuilds は単純で fallback としては許容できるが、IDE feedback が遅すぎる。採用案では interface hash と implementation hash を分離し、proof-body edits が downstream modules を不要に invalidate しないようにする。

### Content-Addressed Store Only

Pure content-addressed store は deduplication には魅力的だが、LSP、docs、logs、inspection には予測可能な `build/` paths が必要である。採用案では cache blobs を content-addressable に保存してもよいが、stable package-relative artifact paths を emit する。

## Adopted Approach

Mizar Evo は stable で versioned な `VerifiedArtifact` projections を emit し、incremental cache entries は internal optimization data として保持する。Reuse は complete cache keys、dependency artifact hashes、proof witness hashes を通じてのみ許可される。Artifact output は deterministic、package-relative、atomic に書き込まれる。

この設計は三つの関心を分離する。

1. published artifacts は downstream tools and packages の external API を定義する。
2. development artifacts は local diagnosis and proof repair を支援する。
3. cache records は future builds を高速化するが、trusted proof authority にはならない。

## Constraints and Assumptions

- Published artifacts は、後続 schema が別 format を明示しない限り UTF-8 JSON である。
- Published artifacts は compiler-internal cache records を load せずに読めなければならない。
- Portable artifacts 内の paths は package-relative または workspace-relative である。
- `verified_at` のような local artifact metadata は user-visible でよいが、canonical artifact hashes から除外しなければならない。
- Cache entries は source-level semantics を変えずに削除できる。
- Artifact schemas は明示的な `schema_version` fields and compatibility checks を持つ。
- Reused proof results は accepted proof witnesses または deterministic built-in discharge records に結びついていなければならない。
- Incremental builds は初期実装でより粗い invalidation を選んでもよいが、clean build と同じ externally visible results を保たなければならない。
- Phase 15 中の build failures は、reader が complete output と誤認する partially-written artifacts を残してはならない。
