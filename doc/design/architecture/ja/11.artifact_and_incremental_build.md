# アーキテクチャ: アーティファクトと増分ビルド

> 正本は英語です。英語版: [../en/11.artifact_and_incremental_build.md](../en/11.artifact_and_incremental_build.md)。

## 目的

この文書は、Mizar Evo が検証済みアーティファクトを出力し、増分ビルドのキャッシュを更新する方法を定義します。

これはパイプラインのフェーズ 15 を詳細化します。前段のフェーズは、検証済みの意味論データ・診断・ATP ログ・カーネル検査済みの証明状態を生成します。フェーズ 15 は、それらの内部結果を安定したディスク上のアーティファクトへ射影し、再現性に必要な来歴を記録し、後続のビルドや IDE のワークフローが用いるキャッシュエントリを更新します。

## 設計判断

### アーティファクトの分類

フェーズ 15 は 3 種類のデータを書き出します。それぞれ互換性と保持のルールが異なります。

| 分類 | 例 | 消費先 | 安定性 |
|---|---|---|---|
| 公開アーティファクト | `*.mizir.json`, エクスポートされたモジュールサマリ, 証明 witness への参照 | 下流パッケージ, ドキュメント, LSP, AI ツール | バージョン管理された安定スキーマ |
| 開発用アーティファクト | `missing_facts.json`, `*.atp-log`, 診断の説明, デバッグダンプ | ローカルの開発ワークフロー | 構造化されたものはバージョン管理されるが、公開の入力ではない |
| ビルドキャッシュエントリ | フェーズのキャッシュレコード, `cluster-db/`, インポートスコープのビュー, 解決トレース | コンパイラと LSP サーバ | 内部スキーマ。ツールチェイン／キャッシュバージョンで無効化される |

パッケージの互換性を定義するのは公開アーティファクトだけです。開発用アーティファクトはリリースビルドに存在しなくてもよく、キャッシュエントリはいつ削除されてもパッケージの意味論を変えません。

### VerifiedArtifact は射影である

`*.mizir.json` は検証済みのフェーズ出力から出力されますが、コンパイラの状態のシリアライズではありません。

含むもの:

- ソースの同一性とハッシュ
- 公開エクスポートと依存向けのシグネチャ
- IDE と AI ツールが必要とする式メタデータ
- 検証義務と受理された証明状態
- 証明 witness への参照とコンテンツハッシュ
- 完了したビルドパスで出た診断

含まないもの:

- 生の AST アリーナと内部 IR のレイアウト
- ATP の完全な標準出力／標準エラーストリーム
- カーネル内部の証明状態
- 非決定的なビルドセッション識別子
- ローカルな診断用に明示的に要求された場合を除く絶対パス

エミッタは複数の内部入力からアーティファクトを計算してよいですが、下流ツールはそれらの内部入力に直接依存してはなりません。

### 公開の境界

パッケージは、有効なリリースポリシーのもとで証明義務が受理された公開アーティファクトからのみ公開できます。

公開の検査は次を却下します。

- `proof_status = "open"` の義務
- `require_kernel_certificates = true` のときの外部証跡の証明
- 公開ツールが対応しないスキーマバージョンのアーティファクト
- 記録されたソースハッシュがソースファイルと一致しないアーティファクト
- コンテンツハッシュが参照先ファイルと一致しない証明 witness 参照

`missing_facts.json` のような開発用アーティファクトは、公開されるパッケージインターフェイスの一部にはなりません。

### キャッシュは最適化であり、権威ではない

増分キャッシュのエントリは後続ビルドの作業を省くために使えますが、それ自体が正しさを確立するわけではありません。キャッシュされた証明結果を再利用できるのは、受理結果を変え得るすべての入力がキャッシュキーに含まれている場合だけです。

キャッシュキーは次を含みます。

- ソースハッシュと正規化されたソースパス
- パッケージ ID、モジュールパス、言語エディション、ロックファイルハッシュ
- ツールチェインバージョンと関連するフェーズのスキーマバージョン
- 証明書ポリシーと計算上限を含む検証器の設定
- 依存アーティファクトのハッシュとインポートされたシンボルサマリ
- ファイルから可視な registration および cluster の索引のハッシュ
- 必要に応じて、VC・ATP 問題・バックエンドプロファイル・証明書のハッシュ

キーの構成要素が欠けている場合、または未対応のツールチェイン／キャッシュスキーマで生成された場合、そのエントリはミスとして扱います。

### 決定的なアーティファクト出力

アーティファクトは、同一のソース・ロックファイル・ツールチェイン・検証器設定・受理済み証明書に対して再現可能でなければなりません。

フェーズ 15 は次によってこれを保証します。

- シリアライズの前に map と set を正準的なキーでソートする。
- 決定的なソース範囲と安定したファイルごとの順序からアーティファクト ID を割り当てる。
- `verified_at` のような利用者に見えるタイムスタンプは、正準的なコンテンツハッシュや公開の同一性検査から除外されるローカルなアーティファクトメタデータとしてのみ保存する。
- パスはワークスペースルートまたはパッケージルートからの相対パスとして記録する。
- 参照される witness とログのファイルはコンテンツハッシュで記録する。
- 正準的なフォーマッタで JSON をシリアライズする。

ATP のランタイムの変動は来歴として記録されますが、安定したアーティファクトエントリの順序や名前を変えてはなりません。

### アトミックな書き込み

フェーズ 15 は、アトミックな置換プロトコルでアーティファクトを書きます。

1. 対象ディレクトリ内の一時ファイルにアーティファクトをシリアライズする。
2. ファイル内容をフラッシュする。
3. 一時ファイルを以前のアーティファクトの上にリネームする。
4. パッケージごとのアーティファクトマニフェストを更新する。
5. プラットフォームが対応する場合、格納先ディレクトリをフラッシュする。

読み手は、以前の完全なアーティファクトか、新しい完全なアーティファクトのいずれかを見ます。途中まで書かれた JSON ファイルはビルドエラーであり、黙って受け入れてはなりません。

## 増分ビルドモデル

### ビルドグラフ

ビルドシステムは、少なくとも次のノードクラスを持つ依存グラフを管理します。

| ノード | 無効化のきっかけ |
|---|---|
| `SourceUnit` | ソース内容またはソースパスの変更 |
| `ModuleSummary` | エクスポート・インポート・公開宣言シグネチャの変更 |
| `RegistrationIndexContribution` | registration 宣言または受理済み coherence 証明の変更 |
| `ResolvedTypedArtifactInput` | ソース, インポート, シンボルサマリ, 型規則, registration |
| `VcSet` | 中核 IR, 制御フロー IR, 型事実, 解決トレース, 検証器ポリシー |
| `AtpProblemRecord` | VC, 前提集合, エンコードプロファイル, バックエンドプロファイル |
| `KernelProofRecord` | 証明書, カーネルバージョン, 信頼された環境, 依存の証明参照 |
| `VerifiedArtifact` | 公開射影のいずれかの入力 |

無効化は逆依存方向へ伝播します。ソース編集は、編集されたファイルと、その公開インターフェイス・registration ビュー・証明コンテキストに依存するすべてのファイルを再検査します。証明本体のみの編集は、エクスポートシグネチャと受理済みの定理ステートメントが変わらない場合、下流モジュールの無効化を避けられます。

### インターフェイスハッシュと実装ハッシュ

検証済みの各モジュールは、2 つの重要なハッシュを持ちます。

```rust
struct ModuleArtifactHashes {
    interface_hash: Hash,
    implementation_hash: Hash,
}
```

`interface_hash` は、インポートする側から可視なデータを対象とします。

- エクスポートされた定義・定理ステートメント・mode・述語・functor・属性・構造体・registration
- エクスポートされた定理と有効化済み registration の証明受理状態
- 依存向けの記法とオーバーロードのサマリ
- インポート側の解釈に影響するスキーマバージョン

`implementation_hash` は、ソースファイルの完全な検証済みアーティファクトを対象とし、ローカルな式メタデータ・ローカルな義務・診断・証明 witness 参照を含みます。

下流の意味論フェーズは主に `interface_hash` に依存します。LSP・ドキュメント・ローカルな再ビルドの判断は `implementation_hash` にも依存してよいです。

### キャッシュ再利用のルール

キャッシュされたフェーズ出力を再利用できるのは、次の場合だけです。

- キャッシュキーが要求されたキーと完全に一致する。
- 参照される依存アーティファクトがすべて存在し、記録されたハッシュと一致する。
- 生成元のツールチェインとスキーマバージョンが互換である。
- 有効な検証器ポリシーが、エントリに記録されたポリシーと同等以上に厳格である。
- 再利用される証明レコードが、受理済みの証明 witness または決定的な組み込み解消のレコードを指している。

ポリシーの厳格さは、認識済みのポリシーフィールドに対してのみ単調です。たとえば、`require_kernel_certificates = true` で生成されたエントリは同じポリシーで再利用できますが、外部証跡の証明をポリシービットの変更だけでカーネル検証済みに格上げすることはできません。

### registration と cluster のキャッシュ更新

`cluster-db/` のキャッシュは、構文解析された宣言から直接ではなく、検証済みの registration 寄与から更新されます。coherence 証明が受理されていない registration は、インポートする側から可視ではなく、`interface_hash` の一部でもありません。

変更されたソースファイルに対して、フェーズ 15 は次を更新します。

- 変更されたモジュールの registration 寄与レコード
- 由来が変更されたモジュールである、導出されたグラフと索引のエントリ
- 変更された由来を含むインポートスコープの可視性ビュー
- 影響を受けたファイルの解決トレース

キャッシュは由来のメタデータを保存するため、registration が削除またはリネームされたときに古い寄与を取り除けます。

## インターフェイス定義

### アーティファクトマニフェスト

各パッケージのアーティファクトディレクトリは、出力されたアーティファクトを索引するマニフェストを持ちます。

```rust
struct ArtifactManifest {
    schema_version: String,
    package_id: PackageId,
    package_version: Version,
    lockfile_hash: Hash,
    toolchain: ToolchainInfo,
    verifier_config_hash: Hash,
    modules: Vec<ModuleArtifactEntry>,
}

struct ModuleArtifactEntry {
    module_path: ModulePath,
    source_file: PathBuf,
    source_hash: Hash,
    artifact_file: PathBuf,
    artifact_hash: Hash,
    interface_hash: Hash,
    implementation_hash: Hash,
    proof_witnesses: Vec<ProofWitnessEntry>,
    diagnostics_hash: Option<Hash>,
}
```

マニフェストは、下流ツールが最初に読むファイルです。これにより、任意のビルド出力を走査することなくモジュールのアーティファクトを特定できます。

### VerifiedArtifact のスキーマ

`VerifiedArtifact` のアーキテクチャ上の形は次のとおりです。

```rust
struct VerifiedArtifact {
    schema_version: String,
    source_file: PathBuf,
    source_hash: Hash,
    verified_at: Option<Timestamp>,
    module_path: ModulePath,
    interface_hash: Hash,
    implementation_hash: Hash,
    exports: Vec<ExportedItem>,
    expressions: Vec<ExpressionMetadata>,
    obligations: Vec<ObligationMetadata>,
    proof_witnesses: Vec<ProofWitnessRef>,
    diagnostics: Vec<Diagnostic>,
    provenance: BuildProvenance,
}

struct BuildProvenance {
    toolchain: ToolchainInfo,
    language_edition: Edition,
    lockfile_hash: Hash,
    verifier_config_hash: Hash,
    dependency_artifact_hashes: Vec<DependencyArtifactHash>,
    cache_key: Hash,
}
```

`verified_at` は利用者に見えるローカルなメタデータです。IDE や `build/` を見る人間には有用ですが、`interface_hash`・`implementation_hash`・正準的な再現性の比較には含めません。

`BuildProvenance` は再現性とデバッグの一部です。ただし、参照されるアーティファクトと証明 witness の検査の代わりにはなりません。

### キャッシュレコード

```rust
struct CacheRecord<T> {
    cache_schema_version: String,
    key: CacheKey,
    produced_by: ToolchainInfo,
    dependencies: Vec<CacheDependency>,
    output_hash: Hash,
    output: T,
}

struct CacheKey {
    phase: PipelinePhase,
    source_hash: Option<Hash>,
    input_hashes: Vec<Hash>,
    config_hash: Hash,
    dependency_hashes: Vec<Hash>,
    schema_versions: Vec<SchemaVersion>,
}
```

キャッシュレコードは内部のものです。キャッシュのスキーマバージョンと互換性のルールが明示されていれば、バイナリエンコードを用いてよいです。

## 検討した代替案

### 生 IR のアーティファクト

内部 IR をシリアライズすると初期のツール作りは簡単になりますが、実装の詳細を固定してしまい、スキーマの互換性を管理しにくくします。採用した設計では、安定した射影を出力し、生 IR のダンプは任意のデバッグアーティファクトに留めます。

### witness ハッシュなしにキャッシュされた証明結果を信頼する

これは増分ビルドを速くしますが、古い、または壊れたキャッシュエントリが不正な証明状態を黙って維持できてしまいます。採用した設計では、証明関連のキャッシュキーに証明 witness または決定的な解消のハッシュを要求します。

### 変更があるたびにパッケージ全体を再ビルドする

パッケージ全体の再ビルドは単純で、フォールバックとしては許容できますが、IDE のフィードバックが遅すぎます。採用した設計では、インターフェイスハッシュと実装ハッシュを分離し、証明本体の編集が下流モジュールを不要に無効化しないようにします。

### コンテンツアドレスストアのみ

純粋なコンテンツアドレスストアは重複排除には魅力的ですが、LSP・ドキュメント・ログ・点検には予測可能な `build/` パスが必要です。採用した設計では、キャッシュの blob をコンテンツアドレス可能に保存してもよいですが、安定したパッケージ相対のアーティファクトパスを出力します。

## 採用方針

Mizar Evo は、安定したバージョン管理された `VerifiedArtifact` の射影を出力し、増分キャッシュのエントリは内部的な最適化データとして保持します。再利用は、完全なキャッシュキー・依存アーティファクトのハッシュ・証明 witness のハッシュを通じてのみ許可されます。アーティファクト出力は決定的・パッケージ相対・アトミックに書き込まれます。

この設計は 3 つの関心を分離します。

1. 公開アーティファクトは、下流ツールおよびパッケージの外部 API を定義する。
2. 開発用アーティファクトは、ローカルな診断と証明修復を支援する。
3. キャッシュレコードは、将来のビルドを高速化するが、信頼される証明の権威にはならない。

## 制約と前提

- 公開アーティファクトは、後続のスキーマが別形式を明示しない限り UTF-8 JSON である。
- 公開アーティファクトは、コンパイラ内部のキャッシュレコードを読み込まずに読めなければならない。
- 可搬なアーティファクト内のパスは、パッケージ相対またはワークスペース相対である。
- `verified_at` のようなローカルなアーティファクトメタデータは利用者に見えてよいが、正準的なアーティファクトハッシュから除外しなければならない。
- キャッシュエントリは、ソースレベルの意味論を変えずに削除できる。
- アーティファクトスキーマは、明示的な `schema_version` フィールドと互換性検査を持つ。
- 再利用される証明結果は、受理済みの証明 witness または決定的な組み込み解消のレコードに結び付いていなければならない。
- 増分ビルドは初期実装でより粗い無効化を選んでもよいが、クリーンビルドと同じ外部から観測できる結果を保たなければならない。
- フェーズ 15 中のビルド失敗は、読み手が完全な出力と誤認するような、途中まで書かれたアーティファクトを残してはならない。
