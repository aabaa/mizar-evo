# アーキテクチャ: ドキュメント生成とコード抽出

> 正本は英語です。英語版: [../en/13.documentation_and_extraction.md](../en/13.documentation_and_extraction.md)。

## 目的

この文書は、Mizar Evo が検証済みアーティファクトから、ドキュメントと抽出されたランタイムコードを生成する方法を定義します。

これはパイプラインのフェーズ 16 を詳細化します。ドキュメント生成とコード抽出は、検証済みの意味論メタデータの下流の消費者です。これらは証明の妥当性を確立せず、検証結果を変更せず、対応する `VerifiedArtifact` に照らして実装固有のキャッシュが検証済みでない限り、コンパイラ内部の IR を直接読みません。

## 設計判断

### フェーズ 16 は消費者フェーズである

ドキュメント生成と抽出は、検証済みの出力を消費します。

これらはドキュメントや抽出の診断で失敗してよいですが、次をしてはなりません。

- 名前・型・registration・オーバーロード・証明の判断を再解釈する。
- 証明書を受理または棄却する。
- `*.mizir.json` や証明 witness を書き換える。
- 下流パッケージを生のコンパイラ IR に依存させる。

フェーズ 16 がキャッシュされた `ControlFlowIr` のような実装固有のデータを必要とする場合、キャッシュキーがアーティファクトマニフェスト・ソースハッシュ・ツールチェイン・スキーマバージョンと一致することを検証しなければなりません。一致しない場合は、安定したアーティファクトから必要な射影を再構築するか、キャッシュミスとして報告します。

### ドキュメントは公開の意味論メタデータを用いる

生成されるドキュメントは、ソースファイルのベストエフォートな構文解析ではなく、公開された検証済みメタデータに基づきます。

主なドキュメント入力:

- アーティファクトマニフェスト
- `VerifiedArtifact.exports`
- 前処理から得られる、ソースに対応付けられたドキュメントコメント
- 解決された FQN とラベル
- 整形されたシグネチャ・定理ステートメント・アルゴリズム契約・ソースリンク
- `@latex` アノテーションと既定の論理式整形のメタデータ
- 名前・バージョン・著者・エディションといったパッケージマニフェストのメタデータ

ソースファイルは、ドキュメントコメントのテキストとソース断片を回収するために読みますが、意味論的な判断はアーティファクトから来ます。

### 抽出は検証済みの実行可能な部分集合を用いる

コード抽出は、ランタイムの挙動が実行可能であり、必要な義務が有効な抽出ポリシーのもとで受理されたアルゴリズムにのみ適用されます。

アルゴリズムが抽出可能である条件:

- ソースアーティファクトと依存アーティファクトがマニフェストと一致する。
- 必要な契約と停止性の義務がすべて受理されている。
- ghost 宣言・ghost 代入・スナップショット・証明専用の assertion・アノテーションを消去できる。
- すべてのランタイム演算が、選択された対象への抽出の低位化を持つ。
- 非決定的な `Pick` 式が、選択された対象で実行可能である、または却下される。
- 外部証跡の証明がポリシーで許可されている、または存在しない。

抽出は追加の性質を証明しません。フェーズ 14 の証明受理とフェーズ 15 のアーティファクト整合性を信頼します。

### ドキュメントと抽出はシンボル同一性を共有する

どちらの出力も、アーティファクト由来の同じ `ModulePath`・FQN・`ItemId`・ソース範囲・エクスポートサマリを用います。

これにより次を保証します。

- ドキュメントのアンカーがエクスポートされたラベルと一致する。
- 抽出されたシンボル名をソース項目へ遡れる。
- ソースリンクと診断が、LSP および CLI 出力と同じ範囲を指す。
- 生成されたドキュメントと抽出されたコードが、公開項目について一致する。

対象言語の名前は適法性のためにマングルしてよいですが、その対応を記録しなければなりません。

### 出力は決定的である

同一のソース・ロックファイル・検証済みアーティファクト・抽出対象・ドキュメントオプションに対して、フェーズ 16 の出力は決定的でなければなりません。

ルール:

- モジュールと項目は、正準的なモジュール／FQN 順で出力する。
- 検索索引と相互参照索引は、安定したキーでソートする。
- 生成されるファイル名は、モジュールパスと安定したラベルから導出する。
- 対象言語の名前は、決定的な名前マングリングで生成する。
- タイムスタンプは、再現可能な出力から省くか、出力ハッシュから除外されるローカルメタデータとして隔離する。
- 診断は [12.diagnostics_and_lsp.md](./12.diagnostics_and_lsp.md) と同じ規則で順序付ける。

### ドキュメントは証明妥当性の公開ゲートではない

公開プロファイルがドキュメント品質の検査を明示的に要求しない限り、ドキュメントの警告は検証済みパッケージを無効にすべきではありません。

例:

- 未解決のドキュメントリンク: 既定では警告
- 不正な Markdown: 劣化した描画を伴う警告
- 不正な LaTeX: 警告、または描画時のマーカー
- ドキュメントコメントの欠落: 任意の lint
- ソースリンクが利用不可: ソースがパッケージされていない場合は警告

証明の妥当性はフェーズ 0-15 によって管理されます。

## ドキュメント生成パイプライン

### ステップ 1: ドキュメント入力の読み込み

入力:

- パッケージのアーティファクトマニフェスト
- 検証済みのモジュールアーティファクト
- ソースファイルまたはソースバンドル
- パッケージマニフェストのメタデータ
- ドキュメントの設定

出力:

- `DocBuildPlan`
- モジュールごとの `DocInput`
- 欠落または古いアーティファクトに対する診断

ローダは、描画の前にアーティファクトのスキーマバージョン・ソースハッシュ・パッケージ相対パスを検証します。

### ステップ 2: ドキュメントコメントの結び付け

ドキュメントコメントは、検証器が生成したモジュールおよび項目の同一性に結び付けられます。

結び付けのルール:

- モジュールレベルのコメントはモジュールページに結び付ける。
- 項目のコメントは、ソース範囲と項目の同一性により結び付ける。
- 非エクスポート項目に結び付いたコメントは、プライベートドキュメント出力が要求されない限り、公開ドキュメントから省いてよい。
- 不正または未検証の項目のコメントは、検証済み API ドキュメントとして描画しない。

ドキュメント生成器は、コメント内の診断が該当する `:::` 行を指せるよう、Markdown のソース範囲を保持すべきです。

### ステップ 3: リンクの解決と数式の描画

相互参照は、アーティファクト由来のエクスポートされたラベルと FQN で解決します。

描画器は次を行います。

- インラインのコード参照は、可視なラベルまたは FQN に完全一致する場合だけ解決する。
- 明示的な Markdown リンクは、仕様で定義された参照形式で解決する。
- 利用可能な場合、`used_axioms` と引用メタデータから自動的な「Used by」索引を構築する。
- シンボルの描画に `@latex` アノテーションを適用する。
- アノテーションがない場合は、既定の構文から LaTeX への描画にフォールバックする。

未解決または曖昧なドキュメントリンクは診断を生成しますが、安全にテキストとして描画します。

### ステップ 4: 静的サイトの出力

エミッタは次を書き出します。

- パッケージ索引
- モジュールページ
- 項目ページ
- 検索索引
- 相互参照索引
- 静的アセット
- 任意の機械可読なドキュメント索引

出力の書き込みは、フェーズ 15 と同じアトミックな書き込みの原則に従います。読み手が、途中まで書かれたページや索引を観測してはなりません。

## 抽出パイプライン

### ステップ 1: 抽出可能な項目の選択

抽出器は、エクスポートされたアルゴリズム・明示的な抽出設定・CLI で選択された項目から開始します。

各候補を分類します。

| 状態 | 意味 |
|---|---|
| `Extractable` | 検証・計算可能性・対象の制約がすべて満たされている |
| `SpecOnly` | 数学的には妥当だが、選択された対象では実行可能でない |
| `BlockedByProof` | 必要な義務が open、またはポリシーで許可されていない |
| `BlockedByTarget` | 対象がランタイム構文要素の低位化を持たない |
| `BlockedByDependency` | 依存アーティファクトまたはランタイム対応が欠けている |

`Extractable` の項目だけを出力します。

### ステップ 2: ランタイム IR の構築

ランタイム IR は、検証済みのアルゴリズムデータの射影です。

除去するもの:

- ghost の宣言と代入
- ランタイム検査ポリシーを持たない assertion
- 証明ヒントとソースアノテーション
- 仕様専用のスナップショットと claim ブロック
- 定理／証明専用の構造

保持するもの:

- ランタイムのパラメータと返り値
- 可変変数と代入
- 制御フロー構造
- 抽出可能なアルゴリズムへの呼び出し
- 選択されたポリシーが要求するランタイム検査
- 診断とデバッグ出力用のソースマップ

### ステップ 3: 対象コードへの低位化

各対象バックエンドは、構文の出力とランタイムライブラリの束縛を所有します。

共通の低位化の責務:

- 名前マングリングと逆方向のソースマップ出力
- 型表現の対応付け
- 関数／手続きのシグネチャ出力
- 制御フローの低位化
- 許可された動的検査のランタイム失敗処理
- 生成されたファイルを、要求された出力ディレクトリにパッケージする

バックエンドは、未対応の構文要素に対して部分的なコードを生成せず、明示的に却下しなければなりません。

### ステップ 4: 抽出マニフェストの出力

抽出は、生成されたコードと検証済みソースを結び付けるマニフェストを書きます。

```rust
struct ExtractionManifest {
    schema_version: String,
    target: ExtractionTarget,
    package_id: PackageId,
    artifact_manifest_hash: Hash,
    verifier_config_hash: Hash,
    items: Vec<ExtractedItem>,
    diagnostics: Vec<Diagnostic>,
}

struct ExtractedItem {
    source_item: SymbolId,
    source_range: SourceRange,
    generated_name: String,
    generated_files: Vec<PathBuf>,
    source_artifact_hash: Hash,
    target_options_hash: Hash,
}
```

マニフェストは証明の根拠ではありません。生成されたコードの追跡可能性のメタデータです。

## 検討した代替案

### ソースから直接ドキュメントを生成する

ソースだけを構文解析すればドキュメント生成は検証から独立しますが、解決されたラベル・オーバーロードの判断・整形された意味論的シグネチャ・証明状態を失います。採用した設計では、コメントと断片にはソースを読み、意味論にはアーティファクトを用います。

### ControlFlowIr から直接抽出する

`ControlFlowIr` は実行可能な構造をすでに含むため魅力的ですが、検証固有の事実や ghost メタデータも含みます。採用した設計では、対象への低位化の前に、より小さな `RuntimeIr` を射影します。

### ドキュメント生成を検証フェーズとして扱う

すべてのドキュメント警告でビルドを失敗させるとドキュメント品質は上がりますが、証明の妥当性と表示の品質が混ざってしまいます。採用した設計では、ドキュメントの診断を分離し、より厳格な公開プロファイルを許します。

### まず 1 つの対象バックエンド、その後一般化する

1 つの対象をハードコードすれば初期実装は速いですが、対象の前提がランタイム IR に漏れます。採用した設計では、最初の実装が 1 つのバックエンドだけに対応する場合でも、対象中立な抽出の境界を保ちます。

## 採用方針

Mizar Evo は、ドキュメント生成とコード抽出を、検証済みアーティファクトの決定的な消費者として扱います。ドキュメントは公開の意味論メタデータとソースのドキュメントコメントを描画します。抽出は、検証済みの実行可能なアルゴリズムを対象中立な `RuntimeIr` へ射影し、証明専用の状態を消去して、対象固有のバックエンドで低位化します。

どちらの出力もソースアーティファクトへの追跡可能性のメタデータを持ちますが、信頼される証明の根拠ではありません。

## 制約と前提

- ドキュメントと抽出は、意味論的な出力のために検証済みアーティファクトを必要とする。
- ドキュメントは表示エラーに対して穏当に劣化してよい。抽出は未対応のランタイム構文要素を却下しなければならない。
- ghost 状態と証明専用のアノテーションは、抽出されたランタイムコードに現れてはならない。
- 抽出されたコードは、ソース項目とアーティファクトハッシュへ遡れなければならない。
- 対象バックエンドは、同一の `RuntimeIr` と対象オプションに対して決定的でなければならない。
- フェーズ 16 の出力は、証明の妥当性を変えずに削除・再生成できる。
- 公開プロファイルは、通常のローカルビルドより厳格なドキュメントまたは抽出の要件を課してよい。
