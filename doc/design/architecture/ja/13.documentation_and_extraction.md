# アーキテクチャ: ドキュメント生成と抽出

> 正本は英語です。英語版: [../en/13.documentation_and_extraction.md](../en/13.documentation_and_extraction.md)。

## 目的

この文書は、Mizar Evo が verified artifacts から documentation と extracted runtime code を生成する方法を定義する。

これは pipeline phase 16 を詳細化する。Documentation generation と code extraction は verified semantic metadata の downstream consumers である。これらは proof validity を確立せず、verification results を変更せず、対応する `VerifiedArtifact` に対して implementation-specific cache が検証済みでない限り compiler-internal IR を直接読まない。

## 設計判断

### Phase 16 Is a Consumer Phase

Documentation and extraction は verified outputs を consume する。

これらは documentation or extraction diagnostics で失敗してよいが、次をしてはならない。

- name, type, registration, overload, proof decisions を再解釈する
- proof certificates を accept または reject する
- `*.mizir.json` や proof witnesses を mutate する
- downstream packages を raw compiler IR に依存させる

Phase 16 が cached `ControlFlowIr` のような implementation-specific data を必要とする場合、cache key が artifact manifest、source hash、toolchain、schema versions と一致することを検証しなければならない。一致しない場合は stable artifacts から必要な projection を再構築するか、cache miss として報告する。

### Documentation Uses Public Semantic Metadata

Generated documentation は source files の best-effort parse ではなく、public verified metadata に基づく。

Primary documentation inputs:

- artifact manifest
- `VerifiedArtifact.exports`
- preprocessing から得られる source-mapped doc comments
- resolved FQNs and labels
- rendered signatures, theorem statements, algorithm contracts, source links
- `@latex` annotations and default formula rendering metadata
- package manifest metadata such as name, version, authors, edition

Source files は documentation comment text and source snippets を回収するために読むが、semantic decisions は artifacts から来る。

### Extraction Uses Verified Executable Subset

Code extraction は、runtime behavior が executable で、必要な obligations が active extraction policy のもとで受理された algorithms にのみ適用される。

Algorithm が extractable である条件:

- source artifact and dependency artifacts が manifest と一致する
- required contract and termination obligations がすべて accepted である
- ghost declarations, ghost assignments, snapshots, proof-only assertions, annotations を erase できる
- すべての runtime operation が selected target への extraction lowering を持つ
- non-deterministic `Pick` expressions が selected target で executable である、または reject される
- externally attested proofs が policy で許可されている、または存在しない

Extraction は追加の property を証明しない。Phase 14 の proof acceptance と phase 15 の artifact integrity を信頼する。

### Documentation and Extraction Share Symbol Identity

両方の outputs は artifacts 由来の同じ `ModulePath`、FQN、`ItemId`、source ranges、export summaries を使う。

これにより次を保証する。

- documentation anchors が exported labels と一致する
- extracted symbol names を source items へ trace できる
- source links and diagnostics が LSP and CLI output と同じ ranges を指す
- generated docs and extracted code が public items について一致する

Target-language names は合法性のため mangle してよいが、その mapping を記録しなければならない。

### Outputs Are Deterministic

同一 source、lockfile、verified artifacts、extraction target、documentation options に対して、phase 16 output は deterministic でなければならない。

Rules:

- modules and items は canonical module/FQN order で emit する
- search indexes and cross-reference indexes は stable keys で sort する
- generated filenames は module paths and stable labels から導出する
- target-language names は deterministic name mangling で生成する
- timestamps は reproducible outputs から省くか、output hashes から除外される local metadata として隔離する
- diagnostics は [12.diagnostics_and_lsp.md](./12.diagnostics_and_lsp.md) と同じ規則で order する

### Documentation Is Not a Publication Gate for Proof Validity

Publication profile が documentation quality checks を明示的に要求しない限り、documentation warnings は verified package を invalid にすべきではない。

Examples:

- unresolved doc link: default では warning
- malformed Markdown: degraded rendering を伴う warning
- invalid LaTeX: warning または rendering-time marker
- missing doc comment: optional lint
- source link unavailable: source が package されていない場合 warning

Proof validity は phases 0-15 によって管理される。

## ドキュメント生成パイプライン

### Step 1: Load Documentation Inputs

Input:

- package artifact manifest
- verified module artifacts
- source files or source bundles
- package manifest metadata
- documentation configuration

Output:

- `DocBuildPlan`
- per-module `DocInput`
- missing or stale artifacts に対する diagnostics

Loader は rendering 前に artifact schema versions、source hashes、package-relative paths を検証する。

### Step 2: Attach Doc Comments

Doc comments は verifier が生成した module and item identities に attach される。

Attachment rules:

- module-level comments は module page に attach する
- item comments は source range and item identity により attach する
- non-exported items に attach された comments は private-doc output が要求されない限り public docs から omit してよい
- invalid or unverified items の comments は verified API documentation として render しない

Doc generator は comments 内の diagnostics が該当する `:::` line を指せるよう Markdown source ranges を保持するべきである。

### Step 3: Resolve Links and Render Math

Cross-references は artifacts 由来の exported labels and FQNs で解決する。

Renderer は次を行う。

- inline code references は visible label or FQN に完全一致する場合だけ resolve する
- explicit Markdown links は spec-defined reference forms で resolve する
- 利用可能な場合、`used_axioms` and citation metadata から automatic "Used by" indexes を構築する
- symbol rendering に `@latex` annotations を適用する
- annotation がない場合は default syntax-to-LaTeX rendering に fallback する

Unresolved or ambiguous documentation links は diagnostics を生成するが、安全に text として render する。

### Step 4: Emit Static Site

Emitter は次を書き出す。

- package index
- module pages
- item pages
- search index
- cross-reference index
- static assets
- optional machine-readable documentation index

Output writes は phase 15 と同じ atomic-write principles に従う。Readers が partially-written pages or indexes を観測してはならない。

## 抽出パイプライン

### Step 1: Select Extractable Items

Extractor は exported algorithms、explicit extraction configuration、または CLI-selected items から開始する。

各 candidate を分類する。

| Status | Meaning |
|---|---|
| `Extractable` | all verification, computability, and target constraints are satisfied |
| `SpecOnly` | mathematically valid but not executable for the selected target |
| `BlockedByProof` | required obligations are open or disallowed by policy |
| `BlockedByTarget` | target lacks a lowering for a runtime construct |
| `BlockedByDependency` | dependency artifact or runtime mapping is missing |

`Extractable` items だけを emit する。

### Step 2: Build Runtime IR

Runtime IR は verified algorithm data の projection である。

Remove:

- ghost declarations and assignments
- runtime checking policy を持たない assertions
- proof hints and source annotations
- specification-only snapshots and claim blocks
- theorem/proof-only structures

Preserve:

- runtime parameters and return values
- mutable variables and assignments
- control-flow structure
- extractable algorithms への calls
- selected policy が要求する runtime checks
- diagnostics and debug output 用の source maps

### Step 3: Lower to Target Code

各 target backend は syntax emission and runtime-library bindings を所有する。

Common lowering responsibilities:

- name mangling and reverse source map emission
- type representation mapping
- function/procedure signature emission
- control-flow lowering
- permitted dynamic checks の runtime failure handling
- generated files を requested output directory に package する

Backends は unsupported constructs に対して partial code を生成せず、明示的に reject しなければならない。

### Step 4: Emit Extraction Manifest

Extraction は generated code と verified sources を接続する manifest を書く。

```rust
struct ExtractionManifest {
    schema_version: String,
    target: ExtractionTarget,
    package_id: PackageId,
    artifact_manifest_hash: Hash,
    verifier_config_hash: Hash,
    items: Vec<ExtractedItem>,
    diagnostics: Vec<Diagnostic>,
}

struct ExtractedItem {
    source_item: SymbolId,
    source_range: SourceRange,
    generated_name: String,
    generated_files: Vec<PathBuf>,
    source_artifact_hash: Hash,
    target_options_hash: Hash,
}
```

Manifest は proof evidence ではない。Generated code の traceability metadata である。

## 検討した代替案

### Generate Documentation Directly from Source

Source だけを parse すれば documentation generation は verification から独立するが、resolved labels、overload decisions、rendered semantic signatures、proof status を失う。採用案では comments and snippets には source を読み、semantics には artifacts を使う。

### Extract Directly from ControlFlowIr

`ControlFlowIr` は executable structure をすでに含むため魅力的だが、verification-specific facts and ghost metadata も含む。採用案では target lowering の前により小さい `RuntimeIr` を project する。

### Treat Documentation Generation as a Verification Phase

すべての doc warning で build を失敗させると documentation quality は上がるが、proof validity と presentation quality が混ざってしまう。採用案では documentation diagnostics を分離し、より strict な publication profiles を許す。

### One Target Backend First, Then Generalize

一つの target を hard-code すれば初期実装は速いが、target assumptions が Runtime IR に漏れる。採用案では、最初の実装が一つの backend だけを support する場合でも target-neutral extraction boundaries を保つ。

## 採用方針

Mizar Evo は documentation generation and code extraction を verified artifacts の deterministic consumers として扱う。Documentation は public semantic metadata と source doc comments を render する。Extraction は verified executable algorithms を target-neutral `RuntimeIr` に project し、proof-only state を erase して、target-specific backends で lower する。

どちらの output も source artifacts への traceability metadata を持つが、trusted proof evidence ではない。

## 制約と前提

- Documentation and extraction は semantic output のため verified artifacts を必要とする。
- Documentation は presentation errors に対して graceful に degrade してよい。Extraction は unsupported runtime constructs を reject しなければならない。
- Ghost state and proof-only annotations は extracted runtime code に現れてはならない。
- Extracted code は source items and artifact hashes へ trace できなければならない。
- Target backends は同一 `RuntimeIr` and target options に対して deterministic でなければならない。
- Phase 16 outputs は proof validity を変えずに削除 and regenerate できる。
- Publication profiles は normal local builds より strict な documentation or extraction requirements を課してよい。
