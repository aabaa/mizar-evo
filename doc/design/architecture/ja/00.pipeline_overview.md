## Design Decisions

### 主要 architecture boundary としての pipeline

| Phase | Input | Output | Main Responsibility | Failure Mode |
|---|---|---|---|---|
| 0. Package / Workspace Resolution | `mizar.pkg`, `mizar.workspace`, `mizar.lock` | `BuildPlan` | package graph、dependency versions、edition、verifier settings を解決する | dependency / manifest errors |
| 1. Source Loading / Preprocessing | `.miz` source files, `BuildPlan` | `SourceUnit`, `PreprocessedSource` | source を検証し、comments/doc comments を分離し、imports を pre-scan する | lexical precondition errors |
| 2. Lexing | `PreprocessedSource`, active lexicon seed | `TokenStream` | reserved words、reserved symbols、user symbols、numerals、contextual string literals を token 化する | lexical errors |
| 3. Parsing | `TokenStream` | `SurfaceAst` | declarations、statements、terms、formulas、proofs、algorithms、annotations を parse する | syntax errors |
| 4. Module / Name Resolution | `SurfaceAst`, import graph, symbol indexes | `ResolvedAst` | namespaces、imports/exports、labels、qualified names、selector ambiguity を解決する | unresolved / ambiguous names |
| 5. Signature Collection | `ResolvedAst` | `SymbolEnv` | structs、modes、attributes、predicates、functors、theorems、registrations、templates を登録する | duplicate / illegal declarations |
| 6. Type Checking / Type Inference | `ResolvedAst`, `SymbolEnv` | `TypedAst` | soft types、coercions、sethood、narrowing obligations を check する | type errors |
| 7. Cluster / Registration Resolution | `TypedAst`, registration indexes | `ResolutionTrace`, enriched type facts | attributes を伝播し、replayable registration derivations を記録する | inconsistent / missing registrations |
| 8. Overload Resolution | enriched typed AST, candidate sets | `ResolvedTypedAst` | overload winner を選び、`qua` coercions を挿入する | overload errors |
| 9. Elaboration / Desugaring | `ResolvedTypedAst` | `CoreIr` | surface syntax を logical core representation へ lower する | elaboration errors |
| 10. Algorithm Verification Preparation | `CoreIr` | `ControlFlowIr` | contracts、invariants、ghost rules、termination measures を準備する | algorithm verification errors |
| 11. VC Generation | `CoreIr`, `ControlFlowIr` | `VcIr` | theorems、proof steps、assertions、contracts、registrations から proof obligations を生成する | logical well-formedness errors |
| 12. Pre-ATP Discharge | `VcIr`, local facts | discharged/open VCs | ATP 前に deterministic または computation-based obligations を解く | computation limit / unsupported forms |
| 13. ATP Translation / Dispatch | open VCs, cited premises, proof hints | `AtpProblem`, ATP results, certificates | problems を encode し backend provers を実行する | timeout / unknown / backend errors |
| 14. Kernel Certificate Check | proof certificates, kernel context | verified proof status | proof evidence を独立検証する | certificate rejection |
| 15. Artifact Emit / Cache Update | verified AST/IR, metadata, logs | `VerifiedArtifact`, logs, witnesses, cache | reproducible artifacts を出力し incremental cache を更新する | artifact I/O errors |
| 16. Documentation / Extraction | verified artifacts, doc comments | rendered docs, extracted code | documentation と extracted runtime code を生成する | documentation / extraction errors |

### Ordering Rules

- Language semantics は ATP dispatch 前に解決されなければならない。
- Name resolution、type inference、cluster resolution、overload resolution、template instantiation は Mizar 側の責務である。
- ATP backend は well-typed、name-resolved、elaborated obligations だけを受け取る。
- ATP result は、policy が externally attested evidence を明示的に許す場合を除き、kernel certificate check 後にのみ受理される。

### IR Layering

| IR | Produced By | Consumed By | Role |
|---|---|---|---|
| `SourceUnit` | Source Loading | Preprocessing, diagnostics | file path、package、source text、line map |
| `PreprocessedSource` | Preprocessing | Lexing | lexical text、comments、doc comments、import stubs |
| `TokenStream` | Lexing | Parsing | source-spanned token sequence |
| `SurfaceAst` | Parsing | Name Resolution | source-shaped syntax tree |
| `ResolvedAst` | Name Resolution | Signature Collection, Type Checking | resolved names、labels、imports、namespace references を持つ AST |
| `SymbolEnv` | Signature Collection | Resolver and checker phases | visible symbol と signature environment |
| `TypedAst` | Type Checking | Registration and overload phases | inferred types、coercion candidates、type facts |
| `ResolutionTrace` | Registration Resolution | Kernel, diagnostics, artifacts | replayable cluster/registration derivation trace |
| `ResolvedTypedAst` | Cluster / Overload Resolution | Elaboration, LSP metadata | final surface semantics |
| `CoreIr` | Elaboration | VC Generation, Kernel | desugared logical representation |
| `ControlFlowIr` | Algorithm Preparation | VC Generation, Extraction | algorithm verification/extraction representation |
| `VcIr` | VC Generation | Pre-ATP, ATP Translation | prover-independent verification condition |
| `AtpProblem` | ATP Translation | ATP Backends | backend-neutral prover problem |
| `ProofCertificate` | ATP Backends | Kernel Certificate Check | backend proof evidence |
| `VerifiedArtifact` | Artifact Emit | LSP, docs, downstream packages | stable external metadata projection |

詳細な IR ownership は [01.ir_layers.md](./01.ir_layers.md) で定義する。

### Diagnostics as Phase Outputs

Diagnostics は incidental side effects ではなく、構造化された outputs である。

- Diagnostic codes は [doc/spec/en/22.error_handling_and_diagnostics.md](../../../spec/en/22.error_handling_and_diagnostics.md) に従う。
- Source spans は `SourceUnit` line maps に基づく。
- Name、type、proof diagnostics は、可能な場合 secondary spans と machine-readable details を含める。
- `@show_type`, `@show_resolution`, `@show_thesis`, `@eval` のような informational annotations は diagnostics と metadata artifacts の両方へ流れる。

### Incrementality and Reproducibility

- Cache keys は source hashes、dependency hashes、toolchain version、language edition、verifier settings を含む。
- Import dependency graphs により reverse-dependency invalidation を決定する。
- Cluster and registration indexes は変更された source contribution ごとに更新される。
- Artifact output は reproducibility のため canonical ordering を用いる。
- ATP nondeterminism は seeds、timeout settings、backend metadata、certificate metadata、externally attested status として記録される。

## Alternatives Considered

### Frontend / Middle-End / Backend Only

伝統的な compiler の三分割は Mizar Evo には粗すぎる。cluster resolution、overload resolution、ATP dispatch、certificate checking、artifact emission は、それぞれ異なる trust boundary と cache boundary を持つためである。

### Existing Build Lifecycle Table Only

言語仕様の lifecycle は user 向けには有用だが、implementation design には type checking と proof processing の内部により精密な boundary が必要である。

### Fully Demand-Driven Pipeline

Demand-driven analysis は LSP workflows には魅力的だが、batch pipeline の方が correctness と reproducibility の baseline を明確にできる。Demand-driven execution はこの phase boundary の内部で後から追加できる。

## Adopted Approach

Mizar Evo は明示的な multi-phase pipeline を採用する。これにより semantic ownership、trust boundaries、diagnostics、cache invalidation、artifact generation を推論しやすくする。

## Interface Definitions

```rust
struct BuildPlan {
    workspace_root: PathBuf,
    packages: Vec<PackagePlan>,
    dependency_graph: DependencyGraph,
    lockfile: Lockfile,
    toolchain: ToolchainInfo,
    verifier_config: VerifierConfig,
    build_config: BuildConfig,
}

struct PhaseResult<T> {
    output: Option<T>,
    diagnostics: Vec<Diagnostic>,
    artifacts: Vec<IntermediateArtifact>,
}

struct VerificationCondition {
    id: VcId,
    source_range: SourceRange,
    kind: VcKind,
    local_context: LocalContext,
    premises: Vec<PremiseRef>,
    goal: Formula,
    proof_hint: Option<ProofHint>,
}

struct VerifiedArtifact {
    schema_version: String,
    source_file: PathBuf,
    source_hash: Hash,
    exports: Vec<ExportedItem>,
    expressions: Vec<ExpressionMetadata>,
    obligations: Vec<ObligationMetadata>,
    proof_witnesses: Vec<ProofWitnessRef>,
    diagnostics: Vec<Diagnostic>,
}
```

## Constraints and Assumptions

- `.miz` source file は module unit である。
- Code regions は ASCII であり、comments と annotations は Unicode を含んでもよい。
- Import resolution は active lexicon に影響するため、final tokenization 前に利用可能でなければならない。
- Type checking、cluster resolution、overload resolution は Mizar 側の責務である。
- ATP backends は well-typed、name-resolved、elaborated obligations だけを受け取る。
- Artifacts は、同一 source、lockfile、toolchain、verifier settings、accepted certificates に対して reproducible でなければならない。
