# アーキテクチャ: パイプライン概要

> 正本は英語です。英語版: [../en/00.pipeline_overview.md](../en/00.pipeline_overview.md)。

## 目的

この文書は、`.miz` ソースファイルから検証済みアーティファクト・ドキュメント・抽出コードに至るまでの、Mizar Evo のコンパイラ／検証器のエンドツーエンドのパイプラインを定義します。

これはアーキテクチャディレクトリの親文書です。より範囲を絞った各アーキテクチャ文書が、個々のフェーズや横断的な境界を詳細化します。

## 設計判断

### 主要なアーキテクチャ境界としてのパイプライン

| フェーズ | 入力 | 出力 | 主な責務 | 失敗モード |
|---|---|---|---|---|
| 0. パッケージ／ワークスペース解決 | `mizar.pkg`, `mizar.workspace`, `mizar.lock` | `BuildPlan` | パッケージグラフ、依存バージョン、エディション、検証器設定を解決する | 依存／マニフェストのエラー |
| 1. ソース読み込み／前処理 | `.miz` ソースファイル, `BuildPlan` | `SourceUnit`, `PreprocessedSource` | ソースを検証し、コメントとドキュメントコメントを分離し、インポートを事前走査する | 字句の前提条件エラー |
| 2. 字句解析 | `PreprocessedSource`, アクティブな字句環境 | `TokenStream` | 予約語・予約記号・ユーザーシンボル・数値リテラル・文脈依存の文字列リテラルをトークン化する | 字句エラー |
| 3. 構文解析 | `TokenStream` | `SurfaceAst` | 宣言・文・項・論理式・証明・アルゴリズム・アノテーションを構文解析する | 構文エラー |
| 4. モジュール／名前解決 | `SurfaceAst`, インポートグラフ, シンボル索引 | `ResolvedAst` | 名前空間・インポート／エクスポート・ラベル・修飾名・セレクタの曖昧性を解決する | 未解決／曖昧な名前 |
| 5. シグネチャ収集 | `ResolvedAst` | `SymbolEnv` | 構造体・mode・属性・述語・functor・定理・registration・テンプレートを登録する | 重複／不正な宣言 |
| 6. 型検査／型推論 | `ResolvedAst`, `SymbolEnv` | `TypedAst` | ソフト型・型強制・sethood・narrowing 義務を検査する | 型エラー |
| 7. cluster／registration 解決 | `TypedAst`, registration 索引 | `ResolutionTrace`, 拡充された型事実 | 属性を伝播し、再生可能な cluster 展開とリダクション適用を記録する | 矛盾／欠落した registration |
| 8. オーバーロード解決 | 拡充された型付き AST, 候補集合 | `ResolvedTypedAst` | ordinary overload root を選択し、coherent refinement を join し、`qua` 型強制を挿入する | オーバーロードエラー |
| 9. elaboration／脱糖 | `ResolvedTypedAst` | `CoreIr` | 表層構文を中核論理表現へ低位化する | elaboration エラー |
| 10. アルゴリズム検証の準備 | `CoreIr` | `ControlFlowIr` | 契約・不変条件・ghost 規則・停止性測度を準備する | アルゴリズム検証エラー |
| 11. VC 生成 | `CoreIr`, `ControlFlowIr` | `VcIr` | 定理・証明ステップ・assertion・契約・registration から証明義務を生成する | 論理的整合性エラー |
| 12. ATP 前の解消 | `VcIr`, ローカルな事実 | 解消済み／未解消の VC | ATP に渡す前に、決定的または計算ベースの義務を解く | 計算上限／未対応の形式 |
| 13. ATP 変換／ディスパッチ | 未解消の VC, 引用された前提, 証明ヒント | `AtpProblem`, ATP 結果, 候補 kernel evidence | 問題をエンコードし、バックエンドの証明器を実行する | タイムアウト／unknown／バックエンドエラー |
| 14. Kernel evidence 検査 | formula/substitution evidence, カーネルコンテキスト | kernel 検証済みの証明ステータス | evidence から instantiated formula と SAT を導出し、反証を検査する | evidence の棄却 |
| 15. アーティファクト出力／キャッシュ更新 | 検証済みの AST/IR, メタデータ, ログ | `VerifiedArtifact`, ログ, witness, キャッシュ | 再現可能なアーティファクトを出力し、増分キャッシュを更新する | アーティファクトの入出力エラー |
| 16. ドキュメント／コード抽出 | 検証済みアーティファクト, ドキュメントコメント | 整形済みドキュメント, 抽出コード | ドキュメントと抽出されたランタイムコードを生成する | ドキュメント／抽出エラー |

### 順序のルール

- 言語の意味論は、ATP へのディスパッチより前に解決されていなければならない。
- 名前解決・型推論・cluster 解決・オーバーロード解決・テンプレート実体化は、いずれも Mizar 側の責務である。
- ATP バックエンドは、型付け・名前解決・elaboration を済ませた義務だけを受け取る。
- ATP の結果は、ポリシーが外部証跡（externally attested evidence）を明示的に許す場合を除き、kernel evidence 検査を経てはじめて受理される。

### Evo2 の正しさに関する原則

evo2 の実装は、次の 4 つの横断的な原則に従います。これらは単なる実装上の好みより強い拘束力を持ちます。モジュール単位の設計がこれらと衝突する場合、その設計は却下するか、アーキテクチャのレベルで明示的に改訂しなければなりません。

#### 小さなカーネルの原則（Small Kernel Principle）

信頼されるカーネルは、根拠（evidence）の検証だけを行います。通常の proof policy でカーネルが信頼してよいのは次のものに限られます。

- formula/substitution evidence のパースと構造的妥当性検証
- provenance、imported fact、target、goal binding の検査
- 置換、α 変換、自由変数条件の検査
- 検査済み evidence から kernel が導出する deterministic SAT encoding
- kernel が構築した SAT problem に対する trusted in-process Rust SAT refutation checking

backend proof method、resolution trace、SMT proof object、backend log、backend-reported
success は trusted acceptance material ではありません。legacy certificate parsing と
clause-resolution trace replay は、後続のアーキテクチャ改訂で normal proof-policy
input として明示的に再導入されない限り、migration/audit-only surface です。

カーネルは、証明探索・ヒューリスティックな選択・オーバーロード解決・cluster 探索・ATP 探索・暗黙の型強制の挿入・フォールバック推論を行ってはなりません。探索は前段のフェーズまたは外部ツールが担い、フェーズ 14 は kernel 自身が導出した SAT problem を持つ、kernel-checkable evidence だけを受理します。

#### イミュータブル IR の原則（Immutable IR Principle）

各 IR 層はイミュータブルなスナップショットです。後続のフェーズは前段のスナップショットを消費し、既存の出力を書き換えるのではなく、新しいスナップショットを生成します。

```text
SourceText
  -> SurfaceAst
  -> ResolvedAst
  -> TypedAst
  -> CoreIr
  -> VcIr
  -> AtpProblem
  -> KernelEvidence
  -> VerifiedArtifact
```

各遷移は、同一の入力・ツールチェインバージョン・言語エディション・依存アーティファクト・検証器設定に対して決定的でなければなりません。

#### 決定的ビルドの原則（Deterministic Build Principle）

同一の入力は、同一のアーティファクト・ハッシュ・証明トレース・診断・診断の順序を生成しなければなりません。並列検証はレイテンシを変えてもよいですが、外部から観測できる結果を変えてはなりません。

これは、map／set の反復順序、VC の順序、前提選択の順序、cluster の走査、オーバーロード候補の順序、kernel evidence のシリアライズ、アーティファクト出力、エラー報告のすべてに適用されます。

#### 局所検証の原則（Local Verification Principle）

article（記事）の最終的な検証アーティファクトは、その article を受理する際に用いた事実に関して自己完結していなければなりません。アーティファクトは、直接またはコンテンツアドレスによる参照として、次を記録します。

- 受理された義務で用いた、インポートされた定理とインポートされた公理
- 展開された cluster／registration の連鎖
- 解決されたオーバーロードの選択と挿入された型強制
- 生成された制約と検証条件
- SAT 問題、またはその他のカーネルが検査できるエンコード
- kernel evidence と受理メタデータ

下流のビルドは、隠れたコンパイラのプロセス状態を介してではなく、検証済みのフィンガープリントと依存スライスを介してのみ、アーティファクトを再利用できます。

### IR の層構成

| IR | 生成元 | 消費先 | 役割 |
|---|---|---|---|
| `SourceUnit` | ソース読み込み | 前処理, 診断 | ファイルパス、パッケージ、ソーステキスト、行マップ |
| `PreprocessedSource` | 前処理 | 字句解析 | 字句テキスト、コメント、ドキュメントコメント、インポートスタブ |
| `TokenStream` | 字句解析 | 構文解析 | ソース位置付きのトークン列 |
| `SurfaceAst` | 構文解析 | 名前解決 | ソースの形を保った構文木 |
| `ResolvedAst` | 名前解決 | シグネチャ収集, 型検査 | 名前・ラベル・インポート・名前空間参照を解決した AST |
| `SymbolEnv` | シグネチャ収集 | リゾルバおよびチェッカの各フェーズ | 可視なシンボルとシグネチャの環境 |
| `TypedAst` | 型検査 | registration・オーバーロードの各フェーズ | 推論された型、型強制の候補、型事実 |
| `ResolutionTrace` | registration 解決 | カーネル, 診断, アーティファクト | 再生可能な cluster 展開とリダクション適用のトレース |
| `ResolvedTypedAst` | cluster／オーバーロード解決 | elaboration, LSP メタデータ | 最終的な表層意味論 |
| `CoreIr` | elaboration | VC 生成, カーネル | 脱糖された論理表現 |
| `ControlFlowIr` | アルゴリズム準備 | VC 生成, コード抽出 | アルゴリズムの検証／抽出用表現 |
| `VcIr` | VC 生成 | ATP 前処理, ATP 変換 | 証明器に依存しない検証条件 |
| `AtpProblem` | ATP 変換 | ATP バックエンド | バックエンド中立な証明器問題 |
| `KernelEvidence` | ATP / VC evidence producer | Kernel evidence 検査 | formula/substitution evidence candidate |
| `VerifiedArtifact` | アーティファクト出力 | LSP, ドキュメント, 下流パッケージ | 安定した外部メタデータの射影 |

IR の所有関係の詳細は [01.ir_layers.md](./01.ir_layers.md) で定義します。

### フェーズの出力としての診断

診断は付随的な副作用ではなく、構造化された出力です。

- 診断コードは [doc/spec/en/22.error_handling_and_diagnostics.md](../../../spec/en/22.error_handling_and_diagnostics.md) に従います。
- ソース位置は `SourceUnit` の行マップに基づきます。
- 名前・型・証明に関する診断は、可能な場合に副次的な位置情報と機械可読な詳細を含めます。
- `@show_type`・`@show_resolution`・`@show_thesis`・`@eval` といった情報提供用のアノテーションは、診断とメタデータアーティファクトの両方に供給されます。

### 増分性と再現性

- キャッシュキーには、ソースのハッシュ、依存のハッシュ、ツールチェインバージョン、言語エディション、検証器設定を含めます。
- インポートの依存グラフによって、逆依存方向の無効化を決定します。
- cluster と registration の索引は、変更されたソースの寄与ごとに更新されます。
- アーティファクト出力は、再現性のために正準的な順序を用います。
- ATP の非決定性は、シード・タイムアウト設定・バックエンドメタデータ・候補 evidence メタデータ・外部証跡ステータスとして記録されます。

## 検討した代替案

### フロントエンド／ミドルエンド／バックエンドの 3 分割のみ

伝統的なコンパイラの 3 分割は、Mizar Evo には粗すぎます。cluster 解決・オーバーロード解決・ATP ディスパッチ・kernel evidence 検査・アーティファクト出力は、それぞれ異なる信頼境界とキャッシュ境界を持つためです。

### 既存のビルドライフサイクル表のみ

言語仕様にあるライフサイクルは利用者には有用ですが、実装の設計には、型検査と証明処理の内部にもっと精密な境界が必要です。

### 完全な要求駆動パイプライン

要求駆動の解析は LSP のワークフローには魅力的ですが、バッチ型のパイプラインのほうが正しさと再現性の基準を明確にできます。要求駆動の実行は、これらのフェーズ境界の内側で後から追加できます。

## 採用方針

Mizar Evo は明示的な多フェーズパイプラインを採用します。これにより、意味論上の所有関係・信頼境界・診断・キャッシュ無効化・アーティファクト生成について見通しよく考えられるようになります。

## 制約と前提

- `.miz` ソースファイルがモジュールの単位です。
- コード領域は ASCII であり、コメントとアノテーションは Unicode を含んでもかまいません。
- インポート解決はアクティブな字句環境に影響するため、最終的なトークン化より前に利用可能でなければなりません。
- 型検査・cluster 解決・オーバーロード解決は、いずれも Mizar 側の責務です。
- ATP バックエンドは、型付け・名前解決・elaboration を済ませた義務だけを受け取ります。
- アーティファクトは、同一のソース・ロックファイル・ツールチェイン・検証器設定・受理済み kernel evidence に対して再現可能でなければなりません。
