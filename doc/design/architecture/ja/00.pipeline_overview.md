# Architecture: Pipeline Overview

> Canonical language: English. English canonical version: [../en/00.pipeline_overview.md](../en/00.pipeline_overview.md).

## 目的

Mizar Evo の言語処理系が、`.miz` ソースを読み込んでから検証済み成果物を出力するまでの全体パイプラインを定義する。
この文書は architecture 文書群の親文書であり、各フェーズの入力、出力、責務、失敗モードを揃えるための背骨になる。

## フェーズ概要

| Phase | Output | Responsibility |
|---|---|---|
| 0. Package / Workspace Resolution | `BuildPlan` | manifest, lockfile, dependency graph, verifier settings を確定する |
| 1. Source Loading / Preprocessing | `SourceUnit`, `PreprocessedSource` | source validation, comment/doc-comment separation, import pre-scan |
| 2. Lexing | `TokenStream` | reserved words, reserved symbols, user symbols, numerals, contextual string literals を token 化する |
| 3. Parsing | `SurfaceAst` | source-shaped AST と syntax diagnostics を生成する |
| 4. Module / Name Resolution | `ResolvedAst` | namespace, import/export, labels, qualified names を解決する |
| 5. Signature Collection | `SymbolEnv` | struct/mode/attr/pred/func/theorem/registration/template を登録する |
| 6. Type Checking / Type Inference | `TypedAst` | soft type checking, coercion, sethood, narrowing obligations を扱う |
| 7. Cluster / Registration Resolution | `ResolutionTrace`, enriched typed facts | attribute propagation と registration firing を記録する |
| 8. Overload Resolution | `ResolvedTypedAst` | overload winner と inserted `qua` を確定する |
| 9. Elaboration / Desugaring | `CoreIr` | surface syntax を論理寄りの core 表現へ下げる |
| 10. Algorithm Verification Preparation | `ControlFlowIr` | contracts, invariants, ghost rules, termination measures を整理する |
| 11. VC Generation | `VcIr` | theorem/proof/assert/contract 由来の verification condition を生成する |
| 12. Pre-ATP Discharge | discharged/open VCs | Mizar 側で閉じられる obligation を処理する |
| 13. ATP Translation / Dispatch | `AtpProblem`, certificates | open VC を prover problem へ変換し ATP を実行する |
| 14. Kernel Certificate Check | verified proof status | ATP certificate を独立検証する |
| 15. Artifact Emit / Cache Update | `VerifiedArtifact` | metadata, logs, witnesses, cache を reproducible に出力する |
| 16. Documentation / Extraction | docs, extracted code | documentation generation と executable extraction を行う |

## 基本方針

- ATP を呼ぶ前に、名前、型、registration、overload、template は Mizar 側で確定する。
- ATP は型推論や overload resolution を担当しない。
- ATP の結果は kernel certificate check を通るまで信頼しない。
- diagnostics は副作用ではなく、各フェーズの構造化された出力として扱う。
- incremental build のため、各フェーズの入力 hash と出力境界を明示する。
