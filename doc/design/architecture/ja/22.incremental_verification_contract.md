# アーキテクチャ: 増分検証契約

> 正本は英語です。英語版: [../en/22.incremental_verification_contract.md](../en/22.incremental_verification_contract.md)。

## 目的

この文書は、部分編集に対する増分検証、キャッシュ再利用、証明／VC 再利用、並列検証互換性の健全性契約を定義します。

増分検証と並列検証は、同じクリーンビルド意味論の最適化です。ソースレベルの意味、証明受理、公開 artifact、interface hash、依存側に見える summary、正準的な診断を変えてはなりません。

この文書はアーキテクチャ層の契約です。フェーズをまたぐ正しさの条件と意味的なデータ形状を固定しますが、Rust 型名、field layout、JSON schema、MCP resource、LSP payload、proof witness の wire format、cache database layout は固定しません。

## 規範性

この文書は、増分検証の健全性、cache reuse の前提条件、編集をまたぐ obligation identity、snapshot freshness、並列互換 scheduling について規範的です。

最終的な crate API、Rust 表現、JSON schema、MCP/LSP schema、proof witness 外部形式、cache storage layout については非規範的です。下の疑似コードは概念名であり、最終的な実装型ではありません。

## 設計判断

### クリーンビルド同値性

増分検証は最適化です。

受理された任意の増分結果について、すべての内部 cache を削除し、同じ source snapshot、依存 artifact、lockfile、toolchain、language edition、verifier policy、受理済み kernel evidence で clean build を実行したとき、外部から見える proof acceptance、公開 artifact、interface hash、依存側に見える summary、正準的な診断は同じでなければなりません。

timing、worker id、一時 path、backend runtime duration、cache lookup timing、scheduling priority のような非正準の local metadata は、受理や公開 artifact identity に影響してはなりません。

実装が cached result を現在の `BuildSnapshot` に適用できると証明できない場合、その cache lookup は miss として扱わなければなりません。

### Cache-Miss Fallback

Cache は proof authority ではありません。

cache hit が有効なのは、関連する key、dependency hash、dependency slice、schema version、toolchain 情報、verifier policy、proof witness または deterministic discharge hash がすべて検証された後だけです。

cache data の欠落、未対応 schema、不完全な dependency slice、非互換の verifier policy、不明な toolchain 互換性、proof witness / deterministic discharge mismatch は cache miss を強制します。

cache miss は性能だけを変えなければなりません。ソースレベルの意味、証明受理、interface hash、依存側に見える summary、公開 artifact の順序、正準的な診断を変えてはなりません。

### Snapshot-Scoped Results

すべての増分検証結果は、ただ 1 つの `BuildSnapshotId` に属します。

古い snapshot のために生成された診断、proof status、phase output、cache record、artifact draft は、現在の結果として公開してはなりません。

古い結果を再利用してよいのは、新しい `BuildSnapshot` に対する通常の cache validation を通過した場合だけです。

開いている editor buffer は source snapshot であり、公開 artifact ではありません。LSP build や dry-run の入力にはなりますが、artifact publication boundary が受理するまでは依存 artifact ではありません。

### Identity Model: VcId vs ObligationAnchor

`ObligationAnchor` は、verification obligation の編集をまたぐ best-effort stable identity です。

これは `VcId` とは別物です。

`VcId` は 1 つの `BuildSnapshot` 内で決定的であり、正準的な順序、artifact order、diagnostic order、result collation に使われます。

`ObligationAnchor` は、編集をまたぐ tracking、proof repair、LSP obligation tracking、AI patch dry-run、obligation 単位の cache reuse candidate に使われます。

`ObligationAnchor` は proof evidence ではなく、kernel から信頼されません。

安全ルール:

- 一致する `VcId` だけでは、編集をまたぐ proof reuse には決して十分ではない。
- 一致する `SourceRange` だけでは、編集をまたぐ proof reuse には決して十分ではない。
- 一致する `SurfaceNodeId` や rowan node identity だけでは、編集をまたぐ proof reuse には決して十分ではない。
- 一致する arena id、syntax-node order、parser node order だけでは、編集をまたぐ proof reuse には決して十分ではない。
- 一致する `ObligationAnchor` だけでも十分ではない。それは最初の candidate match にすぎない。

編集をまたぐ proof result reuse には、さらに次のすべてが必要です。

- 一致する `ObligationAnchor`;
- 一致する canonical VC fingerprint;
- 一致する canonical local context fingerprint;
- 一致する dependency slice fingerprint;
- 互換 verifier policy;
- 一致する proof witness hash または deterministic discharge hash。

概念的な形:

```rust
struct ObligationAnchor {
    owner: OwnerOriginId,
    kind: ObligationKind,
    local_path: LocalProofOrProgramPath,
    label_hint: Option<LabelId>,
    source_shape_hash: Hash,
    canonical_goal_hash: Hash,
    canonical_context_hash: Hash,
    generation_schema_version: SchemaVersion,
}
```

| Field | 意味 |
|---|---|
| `owner` | theorem、registration、algorithm、proof block owner などの正規化された public または module-local owner identity |
| `kind` | theorem proof obligation、registration coherence、reduction reducibility、algorithm assertion、loop invariant、termination obligation など |
| `local_path` | proof step path、case/suppose branch path、algorithm block path、generated obligation site など、proof/program 構造を通る source-shaped path |
| `label_hint` | 利用可能な場合の user-written label。便利だが必須ではない |
| `source_shape_hash` | obligation を生成した局所的な構文形状の hash |
| `canonical_goal_hash` | 正規化された goal formula の canonical hash |
| `canonical_context_hash` | local assumption、visible premise、関連 type fact、active registration view、その他 obligation に影響する context の canonical hash |
| `generation_schema_version` | VC generation scheme の version |

正確な Rust 型は異なってよいです。契約の本質は、snapshot 内の正準順序 (`VcId`) と、編集をまたぐ candidate identity (`ObligationAnchor`) を分離することです。

編集後に anchor を十分な確信で対応付けられない場合、その obligation は新規扱いとなり、以前の proof result を再利用してはなりません。

### Dependency Slice 完全性

cache から再利用される phase output は、その phase の完全な dependency slice または dependency footprint を持たなければなりません。その完全性を確立できない場合、その output は `uncacheable` として印を付け、cache lookup は miss しなければなりません。

dependency data の欠落を「依存なし」と解釈してはなりません。

theorem/VC reuse に使う dependency slice は、その theorem または VC に影響した phase slice の合成です。初期実装では理想より粗い slice を使ってよいですが、dependency を省略してはなりません。false-positive rebuild は許容されますが、false-negative reuse は soundness bug です。

概念的な phase slice:

| Slice | 必要な footprint |
|---|---|
| `LexicalParseSlice` | source hash; lexical hash; import pre-scan hash; imported lexical summary fingerprint; active lexical environment fingerprint; parser lexing plan hash; parser grammar/schema version; source-position-aware operator view hash; notation parse-effect fingerprint |
| `ResolveSlice` | imported module summary; import/export edge fingerprint; resolved label; resolved symbol; namespace/visibility decision; selected module identity |
| `TypeRegistrationSlice` | 参照した mode/attribute; 参照した registration; 使用した cluster fact; 適用した reduction rule; active registration view hash; `ResolutionTrace` hash |
| `OverloadSlice` | overload domain view hash; 適切な粒度での considered candidate; selected root fingerprint; coherent refinement metadata; 挿入された `qua` evidence |
| `ElaborationSlice` | unfold された definition; opacity/transparency policy; generated obligation seed; source-to-core provenance; normalized owner id |
| `VcSlice` | `ObligationAnchor`; 現在の `BuildSnapshot` 内の `VcId`; obligation seed hash; canonical local context hash; canonical goal hash; premise fingerprint; proof hint と computation policy |
| `AtpKernelSlice` | ATP problem hash; premise-selection hash; backend profile; encoding profile; kernel evidence hash; imported fact fingerprint; accepted proof witness hash または deterministic discharge hash |

正確な field 名は crate-level decision です。重要なのは具体的な record layout ではなく完全性です。

### Frontend Lexical/Parse Footprints

Frontend cache footprint は semantic `DependencySlice` ではありません。これは theorem、definition、registration、proof fact の利用ではなく、tokenization/parsing に影響する入力を記録します。

`TokenStream` reuse には、source lexical hash、active lexical environment fingerprint、imported lexical-summary fingerprint、parser lexing plan / filter hash、lexer/schema version の一致が必要です。bundle/source-level reuse では、token と AST entry を検討する前に互換な language edition が必要です。

`SurfaceAst` reuse には、`TokenStream` content hash、parser schema version、parser inputs hash、source-position-aware operator view hash、language edition の一致が必要です。

notation declaration、synonym/antonym parse effect、predicate/functor operator metadata、または該当 source position で見える active operator view が変わった場合、cached `SurfaceAst` を再利用してはなりません。

### Registration, Cluster, and Reduction Activation

parse 済み registration は active fact ではありません。

importer-visible な `RegistrationIndex`、`cluster-db`、reduction index、`ResolutionTrace`、`interface_hash`、downstream summary に入ってよいのは、受理済み registration contribution だけです。

未受理の coherence、existence、compatibility、reducibility proof は、対応する registration を automated reasoning に見せてはなりません。

次のいずれかが変わった場合、その origin を見えるすべての import-scoped registration/cluster/reduction view を無効化します。

- registration label;
- target pattern;
- guard;
- declared cluster または reduction contribution;
- coherence proof status;
- reducibility proof status;
- accepted witness;
- origin identity;
- export visibility.

registration の削除または rename では、依存 cache hit を受理する前に stale origin contribution を削除しなければなりません。

derived cluster または reduction fact を使うすべての VC/cache key には、`ResolutionTrace` hash が参加しなければなりません。

### Proof Witness and Deterministic Discharge Reuse

proof 関連 cache record を再利用してよいのは、現在の verifier policy に対して有効な accepted proof witness または deterministic built-in discharge record を参照している場合だけです。

proof witness hash mismatch は cache miss です。

deterministic discharge hash mismatch は cache miss です。

外部認証された証拠は、cache reuse によって kernel-verified status へ昇格してはなりません。

backend success result は、active policy と kernel boundary が受理しない限り proof acceptance ではありません。

### Parallel Compatibility

増分検証と並列検証は同じ correctness contract を共有します。

同じ `BuildSnapshot` と verifier policy について、次のすべては、外部から見える proof acceptance、公開 artifact、interface hash、依存側に見える summary、正準的な診断で一致しなければなりません。

- clean sequential build;
- clean parallel build;
- incremental sequential build;
- incremental parallel build.

worker completion order、cache lookup timing、cache hit/miss timing、ATP runtime duration、cancellation timing、scheduling priority は semantic result に影響してはなりません。

scheduler task graph は、incremental invalidation と parallel readiness の共通基盤です。task は clean-build equivalence に必要な dependency edge がすべて満たされた場合にのみ並列実行してよいです。

task dependency edge は correctness edge であり、単なる scheduling hint ではありません。性能のために dependency edge を削る、または弱めることは soundness bug です。

初期実装では task granularity が粗くてもよいですが、dependency edge の欠落は許されません。

`VcId` は 1 つの `BuildSnapshot` 内の正準順序を決めます。並列 worker は任意の順序で VC を完了してよいですが、result collation、artifact order、diagnostic order は正準順序を使わなければなりません。

`ObligationAnchor` は編集をまたぐ tracking key であり、`VcId` ordering を置き換えてはなりません。

artifact commit は並列実行の serialization boundary です。公開 artifact を worker completion order で commit してはなりません。

portfolio winner selection は active proof policy の下で決定的でなければなりません。「最初に完了した backend」を semantic winner rule にしてはなりません。複数の受理可能な evidence candidate がある場合、canonical backend priority、kernel-evidence strength、決定的 tie-breaker により選択します。

runtime duration は provenance として記録してよいですが、canonical proof identity に参加してはなりません。

### Edit-Class Invalidation Rules

| Change kind | Downstream semantic rebuild | Local rebuild / artifact refresh |
|---|---:|---:|
| syntax-sensitive notation の外側の whitespace/comment | No | 必要なら diagnostics/docs のみ |
| theorem proof body only | statement と accepted proof status が不変なら No | Yes |
| theorem statement | Yes | Yes |
| theorem accepted/open/failing status change | その theorem を見える／使える importer では Yes | Yes |
| definition body または definitional boundary | Yes | Yes |
| algorithm body only | exported signature と requires/ensures contract が不変なら No | Yes |
| algorithm requires/ensures contract | Yes | Yes |
| notation/operator metadata/parse effect | active lexical/operator view から見える file では Yes | Yes |
| registration declaration | その registration を見える importer では Yes | Yes |
| registration coherence/reducibility proof status | その registration を見える importer では Yes | Yes |
| cluster/reduction contribution | Yes | Yes |
| diagnostic wording only | No | diagnostic artifact refresh only |

theorem proof-body-only edit と registration proof-body edit は同等ではありません。registration coherence/reducibility proof status は active reasoning fact を変える可能性があるため、downstream semantic work を無効化し得ます。

semantic-versioning classification は cache-reuse predicate ではありません。

overload-resolution shift heuristic を含む semver heuristic は、関連する dependency fingerprint と slice が完全に一致しない限り、resolver、checker、VC、ATP、kernel、artifact work を skip するために使ってはなりません。

### 検討した代替案

#### Source Range や Parser Node Identity による再利用

これは高速ですが unsound です。source range、rowan node、compatibility node id、arena id は source-shape handle であり proof identity ではありません。編集は range を保ったまま semantics を変えたり、obligation を保ったまま range をずらしたりできます。

#### VcId を編集横断で安定とみなす

`VcId` は 1 snapshot 内の決定的順序には適していますが、obligation の挿入や削除で後続 VC が renumber されます。唯一の編集横断 key にすると、stale proof を再利用するか、安全な再利用機会を逃します。

#### Cache Hit を Proof Authority として信頼する

これは build を速くしますが、proof authority を verifier/kernel boundary の外に移してしまいます。採用方針では、reuse 前にすべての関連入力、dependency slice、policy、witness/discharge hash を検証します。

#### 最初に完了した Backend を採用する

latency は改善しますが、proof identity が runtime scheduling に依存します。採用方針では runtime を provenance として記録し、evidence は policy の下で決定的に選択します。

### 採用方針

Mizar Evo は clean-build behavior を参照意味論とします。増分検証と並列実行は、同じ `BuildSnapshot` と verifier policy に対して外部から見える結果を再現する場合に限り許されます。

採用する identity model は次を分離します。

- `VcId`: snapshot 内の決定的 ordering / collation key。
- `ObligationAnchor`: 編集をまたぐ tracking / reuse-candidate key。
- canonical VC/context/dependency/policy/witness fingerprint: proof reuse validation に必須の key。

cache record は最適化データです。検証後に build を加速してよいだけであり、削除しても source-level semantics は変わりません。

## インターフェイス定義

次の疑似コードは概念的なものです。最終的な Rust 名、field layout、serialization、storage decision は crate-level design に属します。

```rust
struct IncrementalReuseCandidate {
    snapshot: BuildSnapshotId,
    vc_id: VcId,
    anchor: ObligationAnchor,
    vc_fingerprint: Hash,
    local_context_fingerprint: Hash,
    dependency_slice_fingerprint: Hash,
    verifier_policy_fingerprint: Hash,
    proof_evidence: ProofEvidenceFingerprint,
}

enum ProofEvidenceFingerprint {
    AcceptedWitness(Hash),
    DeterministicDischarge(Hash),
}

struct PhaseDependencyFootprint {
    phase: PipelinePhase,
    schema_version: SchemaVersion,
    completeness: FootprintCompleteness,
    hashes: Vec<NamedHash>,
}

enum FootprintCompleteness {
    Complete,
    IncompleteUncacheable,
}
```

validation predicate:

```text
Reusable(candidate, current_snapshot) iff
  candidate.anchor が現在の obligation と十分な確信で一致し
  and candidate.vc_fingerprint == current canonical VC fingerprint
  and candidate.local_context_fingerprint == current local context fingerprint
  and candidate.dependency_slice_fingerprint == current composed dependency slice fingerprint
  and candidate.verifier_policy_fingerprint が active policy と互換で
  and candidate.proof_evidence が accepted witness または deterministic discharge record と一致し
  and 参照されたすべての dependency artifact と schema が利用可能かつ互換である。
```

それ以外はすべて cache miss です。

## 関連モジュール

想定される module-level owner:

- `mizar-session` - `BuildSnapshotId`, `SourceId`, source hash, source range coordinate system を所有する。`ObligationAnchor` は所有しない。
- `mizar-lexer` - tokenization input と lexical fingerprint を所有する。
- `mizar-syntax` - source-shaped syntax と anchor ingredient を所有するが semantic identity は所有しない。
- `mizar-parser` - syntactic parse shape と source-position-aware parser operator view を所有する。
- `mizar-frontend` - `LexicalParseSlice` のような frontend cache footprint を所有する。
- `mizar-resolve` - normalized semantic owner/origin id、import/export、label、symbol identity を所有する。
- `mizar-core` - source-to-core provenance と obligation seed を所有する。
- `mizar-vc` - snapshot 内の `VcId` assignment、`ObligationAnchor` construction、canonical VC fingerprint、VC-local slice assembly を所有する。
- `mizar-cache` - cache lookup、proof reuse validation、dependency-slice validation、policy compatibility check を所有する。
- `mizar-build` - task graph construction、invalidation scheduling、reverse dependency、build/session orchestration を所有する。
- `mizar-artifact` - stable published projection、proof witness reference、interface/implementation hash、manifest publication を所有する。
