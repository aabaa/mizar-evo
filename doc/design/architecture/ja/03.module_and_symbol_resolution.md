## 設計判断

### リゾルバは最初の意味論的所有者である

フロントエンドはソースの形を保った構文を生成します。リゾルバは、安定した意味論的同一性を付与することを許される最初のフェーズです。

リゾルバが所有するもの:

- 構文解析されたモジュールパスから正準的な `ModuleId` への対応付け
- インポート・別名・エクスポート・可視性・循環の検証
- 名前空間で修飾された名前の解決
- ラベルとローカル参照の解決
- 宣言された項目への安定した `SymbolId` の割り当て
- 型検査と下流ファイルが用いる `SymbolEnv` の構築

リゾルバが所有しないもの:

- overload resolution result の選択
- 式の型推論
- cluster registration の発火
- 構文から中核論理への低位化
- 証明の妥当性の判断

これにより、型検査より前に意味論的同一性を利用可能にしつつ、リゾルバがチェッカ固有の事実に依存しないようにします。

### パッケージ計画とモジュール解決は別個だが結合している

パッケージ計画は、パッケージのバージョン・ルート・エディション・検証器設定を解決します。モジュール解決はその計画を用いて、ソースファイルとインポートパスをモジュールへ対応付けます。

パッケージプランナは、パッケージレベルのビューを生成します。

- パッケージ ID とパッケージ名
- 物理的なソースルート
- ロックファイル由来の依存バージョン
- 有効な言語エディションと検証器設定
- パッケージの可視性とレジストリ／ソースのメタデータ

モジュールリゾルバは、モジュールレベルのビューを生成します。

- 各ソースファイルの正準的な `ModuleId`
- パッケージの名前空間とファイルパスから導出される `ModulePath`
- インポート辺とエクスポート辺
- トポロジカルなビルド順序
- 辺の位置情報を伴う循環の診断

同じ `BuildPlan` は、バッチ検証・LSP セッション・ドキュメントビルドで共有できます。リゾルバは、構文解析された AST から見つけたソースごとのインポート辺によって、これを詳細化してよいです。

### インポート解決は 2 パスで行う

字句解析は完全な意味論的解決より前にアクティブな字句環境を必要とするため、インポート解決は意図的に分割されています。

| パス | 入力 | 出力 | 許される判断 |
|---|---|---|---|
| インポート事前走査 | `PreprocessedSource` | `ImportStub` | 候補となるエクスポートシンボルを読み込むのに十分な、生のインポート構文 |
| 意味論的なインポート／エクスポート解決 | `SurfaceAst`, `BuildPlan`, モジュール索引 | `ResolvedImport`, `ResolvedExport` | モジュールの存在、別名の束縛、可視性、再エクスポートの適法性、衝突の診断 |

アクティブな字句環境は、インポートが完全に検証される前に、候補インポート由来の記号名を含んでよいです。完全な検証は意味論的なインポート解決で行われ、不正なインポートは、たとえトークン化が成功していても、後続のシンボル使用を未解決にします。

意味論的に不正なインポート由来の語彙エントリにトークンの分類が依存する場合、そのトークンには語彙の来歴を印として付けなければなりません。バッチ検証はそれらのトークンを汚染されたもの（tainted）として扱い、それに依存する下流の意味論的確定を抑制します。LSP のリカバリは、ナビゲーションや後続の診断のために、汚染されたトークン化をなお利用してよいです。モジュールが見つからないインポートは、読み込むべき候補モジュールサマリが存在しないため、語彙エントリを一切提供しません。

### 名前空間はシンボルより先に解決する

修飾された参照は、2 つの層で解決します。

1. 先頭のパスを、モジュール名前空間またはインポート別名として解決する。
2. 末尾の名前または記号の綴りを、その名前空間のエクスポートシンボルテーブル内で解決する。

修飾されていない参照では、リゾルバは次の順序でスコープを探索します。

1. ローカルな証明またはブロックの束縛
2. 現在の定義または定理のパラメータ
3. 現在のモジュールのプライベートおよびパブリックな宣言
4. 明示的にインポートされたシンボル
5. インポートされたファサードモジュールを通じて可視になる、再エクスポートされたシンボル
6. エディションが暗黙のプレリュードを有効化している場合の、組み込みプレリュードのシンボル

ローカルな束縛は名前空間の構成要素をシャドーイングします。`G` がローカル変数であれば、`G.x` は名前空間パス `G.x` ではなく、セレクタアクセスとして構文解析・解決されます。セレクタアクセスの解決に型情報が必要な場合、リゾルバは推測せず、チェッカ向けに `DottedTerm` ノードを記録します。

### シンボル同一性は安定かつ完全修飾である

宣言された項目はすべて、安定した `SymbolId` と正準的な完全修飾名を受け取ります。

FQN は次から導出されます。

- パッケージ同一性と名前空間ルート
- モジュールパス
- 宣言ローカルな項目名または記号の綴り
- 複数の宣言が同じ表層の綴りを共有する場合の、オーバーロードスロットまたは宣言ラベル
- 同義語や反義語のような関係宣言のための、宣言ローカルな同一性

FQN は同一性とアーティファクトのためのものです。利用者向けの綴りはソースの形を保ち、別名・インポート名・同義語・記号記法を用いてよいです。

### シグネチャ収集は宣言のパスであり、型検査ではない

シグネチャ収集は、後続の宣言が参照できるよう十分早く宣言を登録しますが、その完全な意味論を証明したり推論したりはしません。

シグネチャ収集が記録するもの:

- 構造体・mode・属性・述語・functor・定理・補題・scheme・registration・テンプレート・同義語・反義語
- 宣言のパラメータと構文上のアリティ
- 可視性とエクスポート状態
- 定義しているソース範囲とドキュメントコメントの結び付け
- 下流のモジュール字句サマリへ提供する記号の綴り
- 型検査に必要な、種別ごとの暫定的なシグネチャデータ

シグネチャ収集が却下するもの:

- オーバーロード集合を形成できない重複宣言
- 不正なプライベート項目の再エクスポート
- 名前レベルで不正な同義語／反義語の参照
- アクティブなエディションで表現できない名前を持つ宣言

シグネチャ収集が後続フェーズに委ねるもの:

- シグネチャの型としての正しさ
- 属性の整合性
- overload resolution result の選択
- 定義・registration・定理の証明義務

### エクスポートは明示的なインターフェイスの射影である

モジュールのパブリックインターフェイスは、その宣言と再エクスポート文の射影です。

リゾルバは、関連する 2 つのテーブルを構築します。

- `local_symbols`: プライベート項目を含む、現在のモジュール内で可視なすべての宣言
- `exported_symbols`: インポートする側から可視な、パブリックな宣言と適法な再エクスポート

プライベートシンボルは `exported_symbols` には決して現れません。インポートしたモジュールを再エクスポートしてもプライベート項目はコピーされず、診断とアーティファクトが元の定義モジュールを指せるよう、来歴を記録します。

### ラベル解決は項目解決とは別にスコープ化される

ラベルは通常のシンボルではありません。リゾルバは定理／証明のラベルを専用のラベルスコープに保持します。

| ラベルの種別 | スコープ | 消費先 |
|---|---|---|
| 定理／補題ラベル | パブリックならモジュールのエクスポート面、プライベートならモジュールローカル | 引用、アーティファクト、ATP 前提選択 |
| 定義ラベル | 定義している項目と生成された義務 | チェッカ、VC 生成 |
| 証明ステップラベル | 証明規則に従う、現在の証明ブロックとその入れ子の子ブロック | 証明の根拠付け |
| registration ラベル | registration データベースと解決トレース | チェッカ、カーネルの再生、診断 |

引用が修飾されたモジュールパスを用いる場合、モジュール部分は名前空間のルールで解決され、ラベル部分はそのモジュールのエクスポートラベルテーブルで解決されます。

### リゾルバの出力は回復可能性を保たなければならない

リゾルバは、回復可能な名前エラーの後も処理を継続し、エディタが有用な診断とナビゲーションを提供し続けられるようにすべきです。

未解決または曖昧なノードは、可能な場合に診断と候補情報を伴って、`ResolvedAst` 内に明示的に表現されます。後続フェーズは、捏造されたシンボル同一性を受け取るのではなく、これらのノードをスキップするか穏当に劣化させなければなりません。

## 解決パイプライン

### ステップ 1: パッケージ索引とモジュール索引の構築

入力:

- `mizar.pkg`, `mizar.workspace`, `mizar.lock`
- パッケージのソースルート
- 以前にビルドした依存アーティファクト

出力:

- `BuildPlan`
- `PackageIndex`
- 初期の `ModuleIndex`

責務:

- `PackageId` を割り当てる。
- 名前空間ルートとパッケージ名を、ソースルートまたはアーティファクトルートへ対応付ける。
- 現在のワークスペースのソースファイルを発見する。
- 依存からエクスポートされたモジュールサマリを読み込む。
- 正準的なモジュールパスの重複を却下する。

### ステップ 2: インポートグラフの解決

入力:

- `SurfaceAst` 由来の、構文解析されたモジュールディレクティブ
- パッケージ索引とモジュール索引
- 依存モジュールサマリ

出力:

- `ImportGraph`
- モジュールごとの `ResolvedImport` と `ResolvedExport`
- トポロジカルなモジュール順序

責務:

- 絶対および相対のモジュールパスを解決する。
- 別名を束縛する。
- インポートの可視性を検証する。
- エクスポート対象とプライベート項目の制限を検証する。
- 循環を検出する。
- インポート辺とエクスポート辺をソース位置付きで記録する。
- ビルドとアーティファクトのために決定的な順序を生成する。

### ステップ 3: ローカルな宣言シェルの構築

入力:

- `SurfaceAst`
- 現在のモジュールの同一性
- 解決済みインポート

出力:

- ローカルな宣言シェル
- 暫定的な `SymbolId`
- ローカルなラベルスコープ

責務:

- トップレベルおよび定義ブロックの宣言に同一性を割り当てる。
- ラベルを登録する。
- 項目の種別と可視性を分類する。
- モジュールのエクスポート面に寄与する記号の綴りを記録する。
- 重複ラベルと不正な宣言を検出する。

宣言シェルは、言語が許す範囲で前方参照を可能にしますが、名前レベルの情報だけを含みます。

### ステップ 4: モジュール内の名前解決

入力:

- `SurfaceAst`
- 宣言シェル
- インポートされたエクスポートシンボル
- ラベルスコープ

出力:

- `ResolvedAst`
- リゾルバの診断

責務:

- 名前空間パス・別名・項目名・ラベルを解決する。
- 解決された項目参照に `SymbolRef` を付与する。
- 引用に `LabelRef` を付与する。
- チェッカが所有するセレクタ判断のために、曖昧なドット付き構文を保持する。
- 未解決および曖昧な参照を明示的に記録する。

### ステップ 5: シグネチャ環境の収集

入力:

- `ResolvedAst`
- ローカルな宣言シェル
- インポートされたモジュールサマリ

出力:

- `SymbolEnv`
- 下流モジュール向けの `ModuleSummary`
- 下流のアクティブな字句環境のためのモジュール字句サマリ

責務:

- チェッカ向けに可視なシンボルテーブルを構築する。
- インポートする側向けにエクスポートモジュールサマリを構築する。
- 同義語と反義語のリンクを名前レベルで記録する。
- 宣言の依存辺を記録する。
- キャッシュキーとアーティファクト出力のために安定した順序を生成する。

## インターフェイス定義

### モジュール同一性

```rust
struct ModuleId {
    package: PackageId,
    path: ModulePath,
}

struct ModulePath {
    components: Vec<Ident>,
}
```

`ModuleId` は正準的です。インポート別名は `ModuleId` を指すローカル名であり、正準的な同一性の一部になることは決してありません。

### ResolvedImport

```rust
struct ResolvedImport {
    source_range: SourceRange,
    module: ModuleId,
    alias: Option<Ident>,
}
```

`ResolvedImport` は `import` ディレクティブのみを表します。これは依存モジュールを現在のモジュールの名前空間に束縛し、必要に応じてローカル別名を導入します。

### ResolvedExport

```rust
struct ResolvedExport {
    source_range: SourceRange,
    target: ExportTarget,
}

enum ExportTarget {
    Module(ModuleId),
    ImportAlias(Ident, ModuleId),
    Symbol(SymbolId),
}
```

`ResolvedExport` は `export` ディレクティブを表します。別名をエクスポートする場合は、ローカル別名が変わってもアーティファクトが安定であり続けるよう、別名の綴りと正準的な対象モジュールの両方を記録します。

### シンボル同一性

```rust
struct SymbolId {
    module: ModuleId,
    local: LocalSymbolId,
}

struct SymbolDef {
    id: SymbolId,
    fqn: FullyQualifiedName,
    kind: SymbolKind,
    visibility: Visibility,
    primary_spelling: SymbolText,
    source_range: SourceRange,
    doc: Option<DocCommentId>,
}
```

`primary_spelling` は診断に用いる宣言の綴りです。同義語と反義語は、別の関係レコードを通じて対象の `SymbolId` を指します。

### ResolvedAst の参照

```rust
enum NameResolution {
    Resolved(SymbolRef),
    ResolvedBuiltin(BuiltinRef),
    Ambiguous(AmbiguousNameRef),
    Unresolved(UnresolvedNameRef),
    DeferredSelector(DeferredSelectorRef),
}

enum LabelResolution {
    Resolved(LabelRef),
    Ambiguous(AmbiguousLabelRef),
    Unresolved(UnresolvedLabelRef),
}

struct SymbolRef {
    symbol: SymbolId,
    source_range: SourceRange,
    import_path: Option<ImportPathId>,
}
```

name reference と label reference は
`doc/design/mizar-resolve/ja/resolved_ast.md` の別々の table に格納する。同文書が
payload shape の正本である。`DeferredSelector` は、構文解析器がドット付き構文を
生成し、リゾルバが項のベースは識別できるものの、型情報なしにはセレクタを
決定できない場合に用います。

### シンボル環境

```rust
struct SymbolEnv {
    module_id: ModuleId,
    imports: ResolvedImportIndex,
    exports: ResolvedExportIndex,
    symbols: SymbolIndex,
    labels: LabelIndex,
    definitions: DefinitionIndex,
    overloads: OverloadIndex,
    registrations: RegistrationIndex,
    namespace_graph: NamespaceGraph,
    declaration_dependencies: DeclarationDependencyIndex,
    contributions: SourceContributionIndex,
    module_summaries: ModuleSummaryIndex,
}
```

`SymbolEnv` は resolver/checker 境界である。詳細な index family と
contribution-tracking invariant は `doc/design/mizar-resolve/ja/env.md` で定義する。
visible symbol と label の projection は、現在の module の local entry に加え、
import と正当な re-export により visible になるすべての entry を含む。
`module_summaries` は、`ModuleId` を key とする、読み込み済み依存 summary の
in-memory index である。これらの field を合わせれば、依存 source file を再読込せずに
type checking できるだけの情報を持つべきである。

### モジュールサマリ

```rust
struct ModuleSummary {
    module: ModuleId,
    source_hash: Hash,
    exported_symbols: Vec<SymbolDef>,
    exported_labels: Vec<LabelDef>,
    lexical_summary: ModuleLexicalSummary,
    reexports: Vec<ModuleId>,
    schema_version: String,
}
```

`ModuleSummary` は、下流モジュールと増分ビルドが用いる、依存向けの射影です。

## 診断

リゾルバの診断は、ソース構文に対する意味論的な診断です。

| 診断クラス | 例 |
|---|---|
| モジュール解決 | モジュールが見つからない、相対インポートがパッケージルートを逸脱する |
| インポートグラフ | インポートの循環、別名の重複 |
| 可視性 | プライベートシンボルのインポートまたは再エクスポート |
| シンボル参照 | 未定義のシンボル、修飾されていない曖昧な名前 |
| 名前空間参照 | 未知の名前空間セグメント、別名の衝突 |
| ラベル参照 | 重複ラベル、未解決の引用ラベル |
| 宣言収集 | 重複宣言、不正なオーバーロードのグループ化 |

診断には次を含めるべきです。

- 使用箇所における主たる位置情報
- 宣言・インポート・衝突する候補に対する副次的な位置情報
- 曖昧な名前に対する機械可読な候補一覧
- 外部の診断仕様にあるリゾルバ用の範囲に属する、安定した診断コード

現在の診断仕様は、構文・型・オーバーロード／テンプレート・証明・論理的矛盾の各範囲を定義していますが、名前／モジュールリゾルバ専用の範囲はまだ割り当てていません。実装より前に、[doc/spec/en/22.error_handling_and_diagnostics.md](../../../spec/en/22.error_handling_and_diagnostics.md) でリゾルバ用のコードを予約するか、モジュール／名前の診断を既存のどの範囲に置くかを明示しなければなりません。

## 増分処理と再現性

リゾルバのキャッシュキーには次を含めます。

- ソースハッシュ
- 構文解析されたインポートディレクティブ
- 依存の `ModuleSummary` ハッシュ
- パッケージのロックファイルハッシュ
- 言語エディション
- リゾルバのスキーマバージョン

変更は次を無効化します。

- 変更されたモジュールの `ResolvedAst` と `SymbolEnv`
- 変更されたエクスポートシンボル・エクスポートラベル・再エクスポート辺・モジュール字句サマリのエントリをインポートする下流モジュール
- 変更された別名または名前空間ファサードに依存する下流モジュール
- 変更された `SymbolEnv` を消費する型／チェッカの各フェーズ

順序は決定的でなければなりません。

- インポートされたモジュールは、ソース順の衝突検査の後に、正準的な `ModuleId` でソートする。
- エクスポートシンボルテーブルは FQN でシリアライズする。
- 曖昧な候補一覧は、正準的な FQN とソース範囲でソートする。
- ソース範囲が等しい診断は、診断コードと安定した候補キーで順序付ける。

## 検討した代替案

### フロントエンドとリゾルバの単一統合

構文解析と名前解決を統合すると初期実装は単純になりますが、アクティブな字句環境のブートストラップ、構文リカバリ、LSP の部分構文解析が難しくなります。構文解析器を意味論を含まないものに保つことで、明確な構文境界を維持できます。

### チェッカが所有する名前解決

型検査に名前解決を担わせると、セレクタやオーバーロードの場面が局所的に感じられますが、シンボル同一性と型推論が混ざってしまいます。採用した設計は、チェッカに安定した `SymbolEnv` を与えつつ、セレクタ・型強制・overload root・active refinement についてはチェッカが所有する判断を残します。

### 完全に展開されたインポート環境

すべてのインポートシンボルを 1 つの環境に平坦化するのは単純ですが、来歴が失われ、診断・再エクスポート・増分無効化が粗くなります。採用した設計は、`ModuleSummary`・`ResolvedImport`・`import_path` の参照を通じて来歴を保持します。

## 採用方針

Mizar Evo は 2 段階のリゾルバを採用します。

パッケージ解決とモジュール解決が、正準的なモジュール同一性とインポートグラフの順序を確立します。続いて名前解決とシンボル解決が `SurfaceAst` を `ResolvedAst` へ変換し、`SymbolEnv` / `ModuleSummary` の射影を構築します。リゾルバは意味論的同一性を記録しますが、型検査や証明上の推論は行いません。

これにより、後続フェーズに安定した入力を与え、依存アーティファクトをコンパクトに保ち、診断・LSP ナビゲーション・再現可能なビルドのための来歴を十分に保持できます。

## 制約と前提

- `.miz` ファイルがモジュール同一性の単位である。
- インポートの循環は却下される。
- 正準的なモジュール同一性はローカル別名に依存しない。
- リゾルバの出力は、回復可能な診断の後に、明示的な未解決ノードや曖昧なノードを含んでよい。
- プライベートシンボルは、定義しているモジュールの内部でのみ可視である。
- 依存ソースファイルは、その `ModuleSummary` アーティファクトが利用可能でスキーマ互換であれば、再読込する必要はない。
- アクティブな字句環境の構築は候補インポートサマリを用いてよいが、それらのインポートが適法かどうかを検証するのは意味論的なインポート解決だけである。
