## Design Decisions

### Supported Backends

| Backend | Type | Input Format | Certificate Format | Priority |
|---|---|---|---|---|
| Vampire | ATP | TPTP | TSTP | Primary |
| E | ATP | TPTP | TSTP | Primary |
| CVC5 | SMT | SMT-LIB | LFSC / Alethe | Primary for theory-heavy goals |
| Z3 | SMT | SMT-LIB | proof log / externally attested | Optional |

Kernel-verifiable certificates を生成できない backend は、development profiles では diagnostics または externally attested evidence のために実行してよい。しかし `require_kernel_certificates` または同等の policy が有効な release build では winner になれない。`solver: auto` では、externally attested backend が先に返った場合でも、kernel-verifiable evidence を優先する。

### Process Model

Backends は child processes として起動する。

- Input は stdin または temporary problem file 経由で渡す。
- Stdout と stderr は logs のために capture する。
- Timeout は obligation ごとに設定する。default は `mizar.pkg` の `[verifier].atp_timeout` である。
- Crash した backend は backend error として報告され、verifier を crash させない。
- Process exit status、command-line flags、working directory policy、temporary file policy、stdout hash、stderr hash は reproducibility and post-mortem diagnostics のために記録する。

### Portfolio Execution

selected solver が `auto` の場合、Mizar Evo は複数 backend を parallel に起動してよい。Phase 13 は backend から candidate evidence を収集し、phase 14 がその evidence を kernel が受理するかを決める。Portfolio winner は active policy のもとで evidence が受理された最初の candidate であり、kernel-verified certificate は externally attested result より優先される。残りの process は、winning candidate が選ばれた後、または残りの candidate が acceptable result を生成できないと policy が判断した後に terminate される。

Reproducibility のため、portfolio run は次を記録しなければならない。

- backend names and versions
- concrete input hash
- selected backend profile and command-line flags
- timeout and resource limits
- random seed when applicable
- exit status, stdout hash, and stderr hash
- certificate status

### Result Classification

```rust
enum ATPResult {
    Proved {
        certificate: ProofCertificate,
        observed_result: BackendResultKind,
    },
    Counterexample(CounterexampleEvidence),
    Timeout,
    Unknown,
    Error(ATPError),
}
```

`Proved` は backend result が `EncodedProblem.expected_result` と一致し、backend が proof evidence を生成したことを意味する。kernel が proof を受理したことは意味しない。

現在の ATP interface では、期待される successful result は `Unsat` である。SMT-LIB backends は `premises and not goal` に対して `unsat` を報告しなければならず、TPTP backends は selected profile に対応する refutation または theorem status を報告しなければならない。`sat`、`counter-satisfiable`、`unknown`、timeout、proof payload のない process success、parse failures は決して `Proved` に分類されない。

`Counterexample` は diagnostic evidence であり、proof acceptance ではない。Backend model が encoding profile と互換であり、`AtpProvenance` を通じて source-level terms へ map できる場合に限り failure triage に使ってよい。それ以外の場合、その result は `Unknown` または `Error` として扱う。

## Alternatives Considered

1. **Library linking**: process overhead は低いが、licensing と version isolation が難しい。
2. **Long-running prover daemon**: startup cost は低いが、resource management が複雑になる。
3. **Child process execution**: isolation と version management が単純で、overhead も許容範囲である。

## Adopted Approach

Mizar Evo は child process execution と optional portfolio parallelism を採用する。

## Interface Definitions

```rust
trait ATPBackend {
    fn name(&self) -> &str;
    fn solve(&self, problem: EncodedProblem, timeout: Duration) -> Result<ATPResult, ATPError>;
}
```

`EncodedProblem` は concrete backend input に加えて、execution 後に必要な `AtpProblem` metadata を持つ。

- expected backend result and logic profile
- concrete input hash and selected backend profile
- symbol map, formula labels, and `AtpProvenance`
- command-line flags, resource limits, and random seed
- stdout/stderr、certificates、artifacts を接続するための log metadata

Certificate parsing は formula labels と backend symbol names を保持し、phase 14 が original `AtpProvenance` に対して certificate を validate できるようにしなければならない。

## Constraints and Assumptions

- Backend binaries は `PATH` 上にあるか、明示的に設定されることを期待する。
- Backend versions は verifier artifacts と ATP logs に記録する。
- Backend crashes は graceful に扱う。
- Kernel certificate checking が trusted acceptance boundary である。
- `Proved` は observed backend result が `EncodedProblem.expected_result` と一致し、proof evidence が存在する場合にのみ emit される。
- Externally attested evidence は policy-controlled であり、kernel-verified proof status とは区別されたままでなければならない。
- Counterexample results は diagnostics を支援するが、trusted proof status を確立しない。
