## 設計判断

### 対応するバックエンド

| バックエンド | 種別 | 入力形式 | 証明書形式 | 優先度 |
|---|---|---|---|---|
| Vampire | ATP | TPTP | TSTP | 第一 |
| E | ATP | TPTP | TSTP | 第一 |
| CVC5 | SMT | SMT-LIB | LFSC / Alethe | 理論寄りのゴールで第一 |
| Z3 | SMT | SMT-LIB | 証明ログ／外部証跡 | 任意 |

カーネルが検証可能な証明書を生成できないバックエンドは、開発用プロファイルでは診断や外部証跡の根拠のために実行してよいです。ただし、`require_kernel_certificates` または同等のポリシーが有効なリリースビルドでは勝者になれません。`solver: auto` では、外部証跡のバックエンドが先に返った場合でも、カーネルが検証可能な根拠を優先します。

### プロセスモデル

バックエンドは子プロセスとして起動します。

- 入力は標準入力または一時的な問題ファイル経由で渡す。
- 標準出力と標準エラーはログのために取り込む。
- タイムアウトは義務ごとに設定する。既定は `mizar.pkg` の `[verifier].atp_timeout` である。
- クラッシュしたバックエンドはバックエンドエラーとして報告され、検証器をクラッシュさせない。
- プロセスの終了ステータス・コマンドラインフラグ・作業ディレクトリのポリシー・一時ファイルのポリシー・標準出力のハッシュ・標準エラーのハッシュは、再現性と事後の診断のために記録する。

### ポートフォリオ実行

選択されたソルバが `auto` の場合、Mizar Evo は複数のバックエンドを並列に起動してよいです。フェーズ 13 はバックエンドから候補となる根拠を収集し、フェーズ 14 がその根拠をカーネルが受理するかを決めます。ポートフォリオの勝者は、受理可能な根拠の中から決定的なポリシー順位によって選ばれ、カーネル検証済みの証明書は外部証跡の結果より優先されます。生の完了順は provenance として記録してよいですが、canonical winner を選んではなりません。残りのプロセスは、選ばれた勝者がポリシー上確定した後、または残りの候補が受理可能な結果を生成できないとポリシーが判断した後に終了されます。

再現性のために、ポートフォリオ実行は次を記録しなければなりません。

- バックエンドの名前とバージョン
- 具体的な入力のハッシュ
- 選択されたバックエンドプロファイルとコマンドラインフラグ
- タイムアウトと資源上限
- 該当する場合の乱数シード
- 終了ステータス、標準出力のハッシュ、標準エラーのハッシュ
- 証明書の状態

### 結果の分類

```rust
enum ATPResult {
    Proved {
        certificate: ProofCertificate,
        observed_result: BackendResultKind,
    },
    Counterexample(CounterexampleEvidence),
    Timeout,
    Unknown,
    Error(ATPError),
}
```

`Proved` は、バックエンドの結果が `EncodedProblem.expected_result` と一致し、バックエンドが証明の根拠を生成したことを意味します。カーネルが証明を受理したことを意味するものではありません。

現在の ATP インターフェイスでは、期待される成功結果は `Unsat` です。SMT-LIB バックエンドは `premises and not goal` に対して `unsat` を報告しなければならず、TPTP バックエンドは選択されたプロファイルに対応する反証または定理の状態を報告しなければなりません。`sat`・`counter-satisfiable`・`unknown`・タイムアウト・証明ペイロードのないプロセス成功・パース失敗は、決して `Proved` には分類されません。

`Counterexample` は診断のための根拠であり、証明の受理ではありません。バックエンドのモデルがエンコードプロファイルと互換で、`AtpProvenance` を通じてソースレベルの項へ対応付けできる場合に限り、失敗のトリアージに使ってよいです。それ以外の場合、その結果は `Unknown` または `Error` として扱います。

## 検討した代替案

1. **ライブラリリンク**: プロセスのオーバーヘッドは低いが、ライセンスとバージョンの分離が難しい。
2. **常駐型の証明器デーモン**: 起動コストは低いが、資源管理が複雑になる。
3. **子プロセス実行**: 分離とバージョン管理が単純で、オーバーヘッドも許容範囲である。

## 採用方針

Mizar Evo は子プロセス実行と、任意のポートフォリオ並列実行を採用します。

## インターフェイス定義

```rust
trait ATPBackend {
    fn name(&self) -> &str;
    fn solve(&self, problem: EncodedProblem, timeout: Duration) -> Result<ATPResult, ATPError>;
}
```

`EncodedProblem` は、具体的なバックエンド入力に加えて、実行後に必要となる `AtpProblem` のメタデータを持ちます。

- 期待されるバックエンド結果と論理プロファイル
- 具体的な入力のハッシュと選択されたバックエンドプロファイル
- シンボル対応、論理式ラベル、`AtpProvenance`
- コマンドラインフラグ、資源上限、乱数シード
- 標準出力／標準エラー・証明書・アーティファクトを結び付けるためのログメタデータ

証明書のパースは論理式ラベルとバックエンドのシンボル名を保持し、フェーズ 14 が元の `AtpProvenance` に照らして証明書を検証できるようにしなければなりません。

## 制約と前提

- バックエンドのバイナリは `PATH` 上にあるか、明示的に設定されることを期待する。
- バックエンドのバージョンは、検証器のアーティファクトと ATP ログに記録する。
- バックエンドのクラッシュは穏当に扱う。
- カーネル証明書検査が、信頼される受理の境界である。
- `Proved` は、観測されたバックエンド結果が `EncodedProblem.expected_result` と一致し、証明の根拠が存在する場合にのみ出力される。
- 外部証跡の根拠はポリシーで制御され、カーネル検証済みの証明状態とは区別されたままでなければならない。
- 反例の結果は診断を支援するが、信頼される証明状態を確立しない。
