## Design Decisions

### Supported Formats

| Format | Target Backends | Primary Use |
|---|---|---|
| TPTP FOF/TFF | Vampire, E | First-order reasoning、equational reasoning、大きな premise sets |
| SMT-LIB 2 | CVC5, Z3 | Arithmetic、SMT theories、equality と theory reasoning の混在 |

### Backend-Neutral Problem Layer

`AtpProblem` は concrete TPTP text や SMT-LIB text ではない。`VcIr` から生成される backend-neutral representation である。

```rust
struct AtpProblem {
    problem_id: AtpProblemId,
    vc_id: VcId,
    logic_profile: LogicProfile,
    expected_result: ExpectedBackendResult,
    declarations: Vec<AtpDeclaration>,
    axioms: Vec<AtpFormula>,
    conjecture: AtpFormula,
    type_context: AtpTypeContext,
    properties: Vec<EncodedProperty>,
    symbol_map: AtpSymbolMap,
    provenance: Vec<AtpProvenance>,
}

enum ExpectedBackendResult {
    Unsat,
}

struct AtpProvenance {
    encoded_id: AtpFormulaId,
    source: AtpSourceRef,
}

enum AtpSourceRef {
    Premise(PremiseRef),
    LocalHypothesis(CoreFormulaId),
    TypeFact(TypeFactRef),
    Property(PropertyRef),
    RegistrationTrace(ResolutionTraceRef),
    DependencyArtifact(ArtifactProofRef),
}
```

異なる backend profile が異なる encoding を必要とする場合、1 つの `VcIr` から複数の `AtpProblem` が生成されることがある。

### Encoding Strategy

- Soft types は、その backend profile に対して lossless に map できる場合に限り backend sorts として encode してよい。sort で表現されない attribute、mode、subtype、coercion、intersection-like facts は guard predicates または explicit axioms として残さなければならない。
- Mizar functors、predicates、modes、attributes は backend-safe symbol names へ map する。`symbol_map` は backend names から canonical Mizar `SymbolId` または generated binder IDs への対応を記録する。
- Premises は local hypotheses、明示的に cited された `by` references、Mizar 側で resolved された facts から得る。
- commutativity、symmetry、reflexivity、idempotence などの properties は、backend-native declarations または axioms として encode する。
- translation が導入するすべての encoded declaration、axiom、native property declaration、guard は `AtpProvenance` を持たなければならない。これにより kernel certificate checking は backend evidence を利用可能な Mizar-side facts へ map できる。
- Backend-reported used-axiom metadata は advisory である。Trusted `used_axioms` は kernel certificate checking 中に抽出または validate され、その後 artifact に記録される。

### Validity Checking Polarity

ATP problems は proof obligations を validity checks として encode する。

- TPTP refutation backends は premises を axioms として受け取り、selected TPTP profile に応じて goal を conjecture または negated conjecture として受け取る。successful proof は premises が goal を entail することを示さなければならない。
- SMT-LIB backends は premises と goal の negation を assert し、`unsat` を期待する。goal を assert して `sat` を proof として受理してはならない。
- `expected_result` はこの contract を明示的に記録する。現在の interface で trusted successful result とみなせるのは `Unsat` だけであり、backend-specific な `sat`、`unknown`、timeout、error results は VC を証明しない。

### Property Encoding

| Property | Formula Shape | TPTP Strategy | SMT-LIB Strategy |
|---|---|---|---|
| `commutativity` | `F(a, b) = F(b, a)` | native AC support が利用可能なら使い、それ以外では axiom | quantified axiom |
| `symmetry` | `P(a, b) -> P(b, a)` | implication axiom | quantified axiom |
| `reflexivity` | `P(a, a)` | universal axiom | quantified axiom |
| `idempotence` | `F(a, a) = a` | equality axiom | equality axiom |
| `involutiveness` | `F(F(a)) = a` | equality axiom | equality axiom |
| `projectivity` | `F(F(a)) = F(a)` | equality axiom | equality axiom |
| `asymmetry` | `P(a, b) -> not P(b, a)` | implication plus negation axiom | implication axiom |
| `connectedness` | `a != b -> P(a, b) or P(b, a)` | implication axiom | implication axiom |
| `irreflexivity` | `not P(a, a)` | negated reflexive axiom | negated reflexive axiom |

各 encoded property は `PropertyRef` と、それが適用される defining symbol を記録する。Axiom ではなく backend-native declaration を使う場合でも、その declaration には `encoded_id` と provenance entry を与え、certificates と used-property reporting が参照できるようにする。

## Alternatives Considered

1. **TPTP only**: 実装は単純だが、SMT arithmetic と theory support を失う。
2. **SMT-LIB only**: theory support は強いが、general FOL や大きな equational search には適合しにくい。
3. **Dual format support**: 実装量は増えるが、obligation の形に応じて backend を選べる。

## Adopted Approach

Mizar Evo は backend-neutral `AtpProblem` layer を使い、TPTP と SMT-LIB の concrete encoders の両方を support する。

## Interface Definitions

Concrete encoders は `AtpProblem` を消費して backend input を生成する。

- TPTP encoder: `symbol_map` から導いた stable names と provenance-preserving formula labels を使って `fof(...)` / `tff(...)` declarations を emit する。
- SMT-LIB encoder: `(declare-...)`、premise `(assert ...)` forms、`(assert (not <goal>))`、`(check-sat)` を emit し、certificates と logs から provenance を recover するために十分な labels または comments を付ける。

## Constraints and Assumptions

- Encoding は reversible である必要はない。
- Certificate validation は kernel が実行し、encoder output を信頼して行うものではない。
- Backend-specific extensions は、selected backend profile がそれを記録する場合にのみ使用できる。
- Sort encoding は VC の正当化に必要な soft-type facts を消してはならない。
- Backend-visible formula または native property declaration はすべて `AtpProvenance` で trace 可能でなければならない。
- Backend-reported used axioms は、kernel checking が validate するまで artifact `used_axioms` にはならない。
