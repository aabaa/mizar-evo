## 設計判断

### 対応する形式

| 形式 | 対象バックエンド | 主な用途 |
|---|---|---|
| TPTP FOF/TFF | Vampire, E | 一階推論、等式推論、大きな前提集合 |
| SMT-LIB 2 | CVC5, Z3 | 算術、SMT 理論、等式と理論推論の混在 |

### バックエンド中立な問題層

`AtpProblem` は具体的な TPTP テキストや SMT-LIB テキストではありません。`VcIr` から生成される、バックエンド中立な表現です。

```rust
struct AtpProblem {
    problem_id: AtpProblemId,
    vc_id: VcId,
    logic_profile: LogicProfile,
    expected_result: ExpectedBackendResult,
    declarations: Vec<AtpDeclaration>,
    axioms: Vec<AtpFormula>,
    conjecture: AtpFormula,
    type_context: AtpTypeContext,
    properties: Vec<EncodedProperty>,
    symbol_map: AtpSymbolMap,
    provenance: Vec<AtpProvenance>,
}

enum ExpectedBackendResult {
    Unsat,
}

struct AtpProvenance {
    encoded_id: AtpFormulaId,
    source: AtpSourceRef,
}

enum AtpSourceRef {
    Premise(PremiseRef),
    LocalHypothesis(CoreFormulaId),
    TypeFact(TypeFactRef),
    Property(PropertyRef),
    ResolutionTrace(ResolutionTraceRef),
    DependencyArtifact(ArtifactProofRef),
}
```

異なるバックエンドプロファイルが異なるエンコードを必要とする場合、1 つの `VcIr` から複数の `AtpProblem` が生成されることがあります。

### エンコード戦略

- ソフト型は、対象のバックエンドプロファイルに対して損失なく対応付けできる場合に限り、バックエンドのソートとしてエンコードしてよい。ソートで表現されない属性・mode・部分型・型強制・交差に類する事実は、ガード述語または明示的な公理として残さなければならない。
- Mizar の functor・述語・mode・属性は、バックエンドで安全なシンボル名へ対応付ける。`symbol_map` は、バックエンド名から正準的な Mizar の `SymbolId` または生成された束縛子 ID への対応を記録する。
- 前提は、ローカルな仮説・明示的に引用された `by` 参照・Mizar 側で解決された事実から得る。
- 可換性・対称性・反射性・冪等性などの性質は、バックエンドネイティブな宣言または公理としてエンコードする。
- 変換が導入するすべてのエンコード済み宣言・公理・ネイティブな性質宣言・ガードは、`AtpProvenance` を持たなければならない。これにより証明書検査は、バックエンドの根拠を利用可能な Mizar 側の事実へ対応付けできる。
- バックエンドが報告する使用公理のメタデータは参考情報である。信頼される `used_axioms` は、証明書検査の際に抽出または検証され、その後アーティファクトに記録される。

### 妥当性検査の極性

ATP 問題は、証明義務を妥当性検査としてエンコードします。

- TPTP の反証バックエンドは、前提を公理として受け取り、選択された TPTP プロファイルに応じてゴールを conjecture または否定された conjecture として受け取る。証明の成功は、前提がゴールを含意することを示さなければならない。
- SMT-LIB バックエンドは、前提とゴールの否定を assert し、`unsat` を期待する。ゴールを assert して `sat` を証明として受理してはならない。
- `expected_result` はこの契約を明示的に記録する。現在のインターフェイスで信頼できる成功結果とみなせるのは `Unsat` だけであり、バックエンド固有の `sat`・`unknown`・タイムアウト・エラーの結果は VC を証明しない。

### 性質のエンコード

| 性質 | 論理式の形 | TPTP 戦略 | SMT-LIB 戦略 |
|---|---|---|---|
| `commutativity` | `F(a, b) = F(b, a)` | ネイティブな AC 対応が利用可能なら使い、それ以外は公理 | 量化された公理 |
| `symmetry` | `P(a, b) -> P(b, a)` | 含意の公理 | 量化された公理 |
| `reflexivity` | `P(a, a)` | 全称の公理 | 量化された公理 |
| `idempotence` | `F(a, a) = a` | 等式の公理 | 等式の公理 |
| `involutiveness` | `F(F(a)) = a` | 等式の公理 | 等式の公理 |
| `projectivity` | `F(F(a)) = F(a)` | 等式の公理 | 等式の公理 |
| `asymmetry` | `P(a, b) -> not P(b, a)` | 含意と否定の公理 | 含意の公理 |
| `connectedness` | `a != b -> P(a, b) or P(b, a)` | 含意の公理 | 含意の公理 |
| `irreflexivity` | `not P(a, a)` | 否定された反射の公理 | 否定された反射の公理 |

エンコードされた各性質は、`PropertyRef` と、それが適用される定義シンボルを記録します。公理ではなくバックエンドネイティブな宣言を用いる場合でも、その宣言には `encoded_id` と来歴のエントリを与え、証明書と使用性質の報告が参照できるようにします。

## 検討した代替案

1. **TPTP のみ**: 実装は単純だが、SMT の算術と理論対応を失う。
2. **SMT-LIB のみ**: 理論対応は強いが、一般的な一階論理や大きな等式探索には適合しにくい。
3. **両形式への対応**: 実装量は増えるが、義務の形に応じてバックエンドを選べる。

## 採用方針

Mizar Evo はバックエンド中立な `AtpProblem` 層を用い、TPTP と SMT-LIB の両方の具体的なエンコーダに対応します。

## インターフェイス定義

具体的なエンコーダは `AtpProblem` を消費して、バックエンドの入力を生成します。

- TPTP エンコーダ: `symbol_map` から導いた安定した名前と、来歴を保持する論理式ラベルを用いて、`fof(...)` / `tff(...)` の宣言を出力する。
- SMT-LIB エンコーダ: `(declare-...)`、前提の `(assert ...)` 形式、`(assert (not <goal>))`、`(check-sat)` を出力し、証明書とログから来歴を復元するのに十分なラベルまたはコメントを付ける。

## 制約と前提

- エンコードは可逆である必要はない。
- 証明書の検証はカーネルが実行するものであり、エンコーダの出力を信頼して行うものではない。
- バックエンド固有の拡張は、選択されたバックエンドプロファイルがそれを記録する場合にのみ使用できる。
- ソートのエンコードは、VC の正当化に必要なソフト型の事実を消してはならない。
- バックエンドから見えるすべての論理式またはネイティブな性質宣言は、`AtpProvenance` で追跡可能でなければならない。
- バックエンドが報告した使用公理は、カーネルの検査が検証するまで、アーティファクトの `used_axioms` にはならない。
