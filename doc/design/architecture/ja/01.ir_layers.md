# アーキテクチャ: IR レイヤー

> 正本は英語です。英語版: [../en/01.ir_layers.md](../en/01.ir_layers.md)。

## 目的

この文書は、Mizar Evo の処理パイプラインで用いる中間表現（IR）の各層を定義します。

各 IR がどのフェーズで生成され、何を所有し、何を意図的に所有しないのかを明確にします。狙いは、フロントエンド・リゾルバ・チェッカ・VC ジェネレータ・ATP トランスレータ・カーネル・アーティファクトエミッタが互いの内部に過度に依存しないようにしつつ、増分ビルドと診断の境界を安定させることにあります。

## 設計判断

### IR はイミュータブルなスナップショットである

各 IR は、それを生成するフェーズが完了した時点で、イミュータブルなスナップショットとして扱います。

- 後続のフェーズは、前段の IR を直接書き換えません。
- 追加情報は、新しい IR または副テーブルとして生成します。
- ノードの同一性は、必要に応じて安定した `NodeId`・`ExprId`・`ItemId` で追跡します。
- 診断は、各 IR のソースマッピングを通じてソース位置へ遡れます。

これにより、キャッシュキー・増分無効化・LSP メタデータ・デバッグダンプの設計が容易になります。

### 表層の保存は elaboration で終わる

`SurfaceAst`・`ResolvedAst`・`TypedAst`・`ResolvedTypedAst` は、ソース構文の形をできる限り保ちます。
`CoreIr` 以降の層は、証明・検証・カーネル検査のために正規化された表現です。

| 層 | 表層の形 | 意味論の完全さ | 主な消費先 |
|---|---|---|---|
| `SourceUnit` / `TokenStream` | 正確／字句的 | なし | 字句解析器, 構文解析器, 診断 |
| `SurfaceAst` | 高い | 低い | リゾルバ |
| `ResolvedAst` | 高い | 中程度 | チェッカ |
| `TypedAst` | 高い | 高いが、オーバーロードが未確定の場合あり | registration／オーバーロード |
| `ResolvedTypedAst` | 高い | 表層の意味論として完全 | elaborator, LSP |
| `CoreIr` | 低い | 論理的意味論として完全 | VC ジェネレータ, カーネル |
| `VcIr` / `AtpProblem` | なし | 証明義務の意味論 | 証明器パイプライン |
| `VerifiedArtifact` | 射影済み | 検証済みの公開メタデータ | 下流ツール |

### 名前・型・証明の事実は寿命が異なる

名前解決・型推論・証明義務はそれぞれ寿命が異なるため、すべてを同じ AST ノードに押し込むべきではありません。

- 名前解決の結果は `ResolvedAst` と `SymbolEnv` に保持します。
- 推論された型・型強制の候補・型事実は `TypedAst` と `TypeFactTable` に保持します。
- cluster の導出とリダクション適用は、自動 registration 効果ごとの局所証拠をカーネルが再生できるよう `ResolutionTrace` に保持します。
- オーバーロードの勝者と挿入された `qua` 型強制は `ResolvedTypedAst` で確定します。
- 証明義務は AST ノードの内部に隠さず、`VcIr` へ抽出します。

### 安定した外部アーティファクトは射影であり、生の IR ダンプではない

`VerifiedArtifact` は生の IR ダンプではありません。下流パッケージ・LSP・ドキュメント・AI ツールが必要とする情報だけを射影した、安定したスキーマです。

- 生の IR はコンパイラ内部のものであり、バージョン間で変わり得ます。
- アーティファクトのスキーマは `schema_version` と明示的な互換性ポリシーを持ちます。
- ソース範囲・解決済みシンボル・推論された型・義務の状態・証明 witness への参照は、アーティファクトに含めます。
- ソルバ内部のログやカーネル内部の証明状態は、必要に応じて別のアーティファクトから参照します。

## IR の概要

| IR | 生成元 | 消費先 | キャッシュ単位 | コンパイラ外で安定 |
|---|---|---|---|---|
| `SourceUnit` | ソース読み込み | すべての診断, 前処理 | ソースファイル | no |
| `PreprocessedSource` | 前処理 | 字句解析 | ソースファイル + インポート事前走査 | no |
| `TokenStream` | 字句解析 | 構文解析 | ソースファイル + アクティブな字句環境 | no |
| `SurfaceAst` | 構文解析 | 名前解決 | ソースファイル | no |
| `ResolvedAst` | 名前解決 | シグネチャ収集, 型検査 | ソースファイル + インポートグラフ | no |
| `SymbolEnv` | シグネチャ収集 | 名前解決, 型検査, 下流ファイル | パッケージ／モジュールグラフ | no |
| `TypedAst` | 型検査 | registration, オーバーロード, LSP 下書きメタデータ | ソースファイル + シンボル環境 | no |
| `ResolutionTrace` | registration 解決 | カーネル, 診断, アーティファクト | ソースファイル + registration 索引 | 部分的 |
| `ResolvedTypedAst` | cluster／オーバーロード解決 | elaboration, LSP メタデータ | ソースファイル + チェッカ入力 | no |
| `CoreIr` | elaboration | VC 生成, カーネル | ソースファイル + 型付き AST | no |
| `ControlFlowIr` | アルゴリズム準備 | VC 生成, コード抽出 | アルゴリズム項目 | no |
| `VcIr` | VC 生成 | ATP 前処理, ATP 変換 | 義務 | no |
| `AtpProblem` | ATP 変換 | ATP バックエンド | 義務 + バックエンドプロファイル | no |
| `ProofCertificate` | ATP バックエンド | カーネル証明書検査 | 義務 + バックエンド結果 | バックエンド依存 |
| `VerifiedArtifact` | アーティファクト出力 | LSP, ドキュメント, 下流パッケージ | ソースファイル | yes |

## 各層の定義

### SourceUnit

`SourceUnit` は 1 つのソースファイルの正準的なビューです。

```rust
struct SourceUnit {
    source_id: SourceId,
    package_id: PackageId,
    module_path: ModulePath,
    file_path: PathBuf,
    source_text: Arc<str>,
    source_hash: Hash,
    line_map: LineMap,
}
```

所有するもの:

- 元のソーステキスト
- ファイル／モジュールの同一性
- ソースハッシュ
- 診断のための行・列マッピング

所有しないもの:

- 構文解析済みの構文
- ドキュメントとして分類されたコメント
- 意味論的情報

補足:

- 以降のソース範囲はすべて `SourceId` を参照する。
- ソーステキストは読み込み後はイミュータブルである。
- コード領域の ASCII 検証に関する診断は、ここに紐づく。

### PreprocessedSource

`PreprocessedSource` は、字句的な入力をコメントおよびソースレベルのメタデータから分離します。

```rust
struct PreprocessedSource {
    source_id: SourceId,
    lexical_text: LexicalText,
    lexical_hash: Hash,
    comments: Vec<Comment>,
    doc_comments: Vec<DocComment>,
    import_stubs: Vec<ImportStub>,
}
```

所有するもの:

- コメントを除去した字句テキスト、または同等のテキストマッピング
- 下流のトークン／構文再利用キーに使う字句テキストハッシュ
- ドキュメントコメントのブロック
- アクティブな字句環境を構築するのに十分なインポート事前走査の結果

所有しないもの:

- 完全に解決されたインポート
- 最終的なアクティブ字句環境に依存するトークン分類
- ドキュメントコメントの AST への結び付け
- アノテーション構文の所有

補足:

- 前処理は、診断に十分な精度でソースマッピングを保たなければならない。
- アノテーションは字句テキストに残り、構文解析器の構文が所有する。
- インポート事前走査は意図的に浅く、意味論的な名前解決は行えない。

### TokenStream

`TokenStream` は、アクティブな字句環境を適用した後の構文解析器への入力です。

```rust
struct TokenStream {
    source_id: SourceId,
    parser_context: ParserLexContext,
    tokens: Vec<Token>,
    scope_view: ScopeView,
    diagnostics: Vec<LexingDiagnostic>,
}

struct Token {
    kind: TokenKind,
    span: SourceRange,
    text: InternedText,
}
```

所有するもの:

- トークンの種別
- トークンのソース範囲
- インターン化された綴り

所有しないもの:

- 狭い `ParserLexContext` を超える任意の parser state
- `.` の名前空間としての解釈
- 文字列が必要な位置以外での文字列リテラルの解釈
- シンボル解決

補足:

- ユーザー定義の記号名は、アクティブな字句環境に照らして認識される。
- frontend foundation では `InternedText` は local `Arc<str>` spelling handle である。後で shared string interner に置き換えても、トークン所有境界は変わらない。
- `.` の曖昧な役割は、字句仕様に従って構文解析器／名前リゾルバに委ねる。

### SurfaceAst

`SurfaceAst` は、構文解析されたソース構造を保持します。

```rust
struct SurfaceAst {
    source_id: SourceId,
    nodes: Vec<SurfaceNode>,
    root: Option<SurfaceNodeId>,
    token_nodes: Vec<SurfaceNodeId>,
    expression_root: Option<SurfaceNodeId>,
}
```

task 11 の syntax boundary は最小限であり、所有するものは次の範囲に限られる。

- ソース順のトークンノード
- トークンノードと infix ノードのソース範囲
- 任意の root node id と expression root node id
- 回復済みトークンノードの recovered flag

完全な module 構造、trivia map、より広い recovered syntax tree は parser/syntax の後続作業で扱う。その将来の syntax layer が所有するもの:

- ソース順の宣言と項目
- 項、論理式、文、証明ブロック、アルゴリズム本体
- 近接する項目に結び付いた構文上のアノテーション
- 構文解析の回復マーカー

所有しないもの:

- 解決済みのシンボル同一性
- 推論された型
- オーバーロードの勝者
- cluster の事実

補足:

- この層の診断は構文寄りである。
- ソースへの忠実さは、ドキュメントとコードアクションのために重要である。
- 構文ノードは意味論的な妥当性を前提としてはならない。

### ResolvedAst

`ResolvedAst` は、表層の形を保ちながら名前とモジュールの解決結果を記録します。

```rust
struct ResolvedAst {
    source_id: SourceId,
    module_id: ModuleId,
    nodes: AstArena<ResolvedNode>,
    name_refs: NameRefTable,
    label_refs: LabelRefTable,
    imports: ResolvedImports,
}
```

所有するもの:

- モジュールの同一性
- インポート／エクスポートの解決
- 名前空間で修飾された名前の解決
- ラベル参照の解決
- 名前の文脈だけで足りる、セレクタか名前空間かの判断

所有しないもの:

- 型に基づくオーバーロードの勝者
- 式の推論された型
- cluster から導かれる属性
- 生成された証明義務

補足:

- 未解決または曖昧な参照は、診断のために明示的に表現する。
- シンボルが解決された場合、下流のフェーズは生の文字列ではなく `SymbolId` を用いるべきである。

### SymbolEnv

`SymbolEnv` は、あるモジュールから見える、索引化されたシグネチャ環境です。

```rust
struct SymbolEnv {
    module_id: ModuleId,
    visible_symbols: SymbolIndex,
    definitions: DefinitionIndex,
    overload_sets: OverloadIndex,
    registration_index: RegistrationIndex,
    namespace_graph: NamespaceGraph,
}
```

所有するもの:

- エクスポートおよびインポートされたシンボル
- 宣言のシグネチャ
- オーバーロード候補のグループ
- 発火前の registration 宣言
- 名前空間の可視性グラフ

所有しないもの:

- 任意の式に対する型推論の結果
- 特定の項に対する registration 解決の結果
- 証明の状態

補足:

- `SymbolEnv` は、同じ依存閉包をインポートする複数のファイルで共有できる。
- 増分無効化は、各シンボルがどのソース単位から来たかを追跡すべきである。

### TypedAst

`TypedAst` は、最終的な registration とオーバーロードの確定より前の型検査結果を付加します。

```rust
struct TypedAst {
    source_id: SourceId,
    module_id: ModuleId,
    nodes: AstArena<TypedNode>,
    expr_types: TypeTable,
    type_facts: TypeFactTable,
    coercion_candidates: CoercionTable,
    initial_obligations: Vec<InitialObligation>,
}
```

所有するもの:

- 推論された型表現
- 局所的な型コンテキスト
- 既知の型事実
- 拡大／縮小／`qua` の型強制候補
- sethood 検査と narrowing 義務

所有しないもの:

- オーバーロードされた各適用に対する最終的な勝者
- 完全な cluster 閉包
- 証明器パイプライン向けの最終的な証明義務

補足:

- ソフト型は意味論的なメタデータであり、論理中核を置き換えるものではない。
- 型エラーの場合でも、可能な限り部分的に推論された情報を提示すべきである。

### ResolutionTrace

`ResolutionTrace` は、cluster 導出とリダクション適用を記録します。正準的なアーティファクトスキーマは [17.cluster_trace_format.md](./17.cluster_trace_format.md) で定義し、この IR レイヤーの見方は所有関係だけを述べます。

所有するもの:

- 順序付けられた規則の適用
- 各適用で用いた前件の事実
- 導出された属性の事実
- リダクションの redex path、置換、ガード証拠、選択規則の監査キー
- トレースが診断に関連付けられる場合の、失敗や曖昧性の説明への参照

所有しないもの:

- 定理証明の探索
- ATP の前提選択
- 任意の一階推論

補足:

- トレースの再生は、トレースのサイズに対して線形または準線形でなければならない。
- 成功したトレースは、説明とカーネル検査のためのアーティファクトとして出力できる。
- 失敗した解決試行は、診断と LSP の説明 API に供給される。

### ResolvedTypedAst

`ResolvedTypedAst` は、最終的な、表層の形を保った意味論的 AST です。

```rust
struct ResolvedTypedAst {
    source_id: SourceId,
    module_id: ModuleId,
    nodes: AstArena<ResolvedTypedNode>,
    expr_metadata: ExpressionMetadataTable,
    resolved_overloads: OverloadResolutionTable,
    inserted_coercions: CoercionInsertionTable,
    cluster_facts: ClusterFactTable,
}
```

所有するもの:

- 最終的に推論された型
- 最終的なオーバーロードの勝者
- 挿入された `qua` 型強制
- 導出された cluster 属性
- LSP とアーティファクトが用いる式メタデータ

所有しないもの:

- 低位化された論理節
- VC 固有の局所的な証明コンテキスト
- ATP エンコード

補足:

- これはホバー表示や `@show_resolution` のデータの主な供給源である。
- elaboration は、リゾルバ／チェッカのロジックを再実行せず、この層を消費すべきである。

### CoreIr

`CoreIr` は、正規化された論理表現です。

```rust
struct CoreIr {
    module_id: ModuleId,
    items: Vec<CoreItem>,
    definitions: CoreDefinitionTable,
    theorem_bodies: CoreProofTable,
    source_map: CoreSourceMap,
}
```

所有するもの:

- 脱糖された項と論理式
- 実体化されたテンプレート
- 正規化された定義境界
- カーネル向け表現における証明構造
- `ResolvedTypedAst` へのソースマップ

所有しないもの:

- 構文解析器の trivia
- 未解決の名前
- 表層のみの構文上の詳細
- バックエンド固有の ATP エンコード

補足:

- `CoreIr` は、カーネルと VC 生成が扱える程度に小さく保つべきである。
- ソフト型注釈の消去（erasure）の判断は、偶発的ではなく明示的でなければならない。

### ControlFlowIr

`ControlFlowIr` は、検証と抽出のための実行可能なアルゴリズム本体を表します。

```rust
struct ControlFlowIr {
    algorithm_id: AlgorithmId,
    blocks: Vec<BasicBlock>,
    locals: LocalTable,
    contracts: ContractSet,
    ghost_effects: GhostEffectTable,
    termination: Option<TerminationMeasure>,
}
```

所有するもの:

- 制御フローグラフ
- ローカル変数の可変／不変の束縛情報
- ghost 変数の追跡
- 事前条件、事後条件、assertion、不変条件
- 停止性測度

所有しないもの:

- 証明探索の結果
- 抽出された対象言語のコード
- 定理レベルの大域的な証明コンテキスト

補足:

- ランタイムの作用と ghost の作用は、抽出より前に区別されなければならない。
- 代入前使用や到達不能コードの診断は、ここに自然に紐づく。

### VcIr

`VcIr` は、検証条件の証明器非依存な表現です。

```rust
struct VcIr {
    vc_id: VcId,
    source_range: SourceRange,
    kind: VcKind,
    local_context: LocalContext,
    premises: Vec<PremiseRef>,
    goal: CoreFormula,
    proof_hint: Option<ProofHint>,
}
```

所有するもの:

- 局所的な仮定
- 引用された前提
- 目標となる論理式
- VC の種別とソース位置
- 証明ヒントの注釈

所有しないもの:

- 具体的な TPTP／SMT-LIB のテキスト
- 抽象的なヒントを超えるバックエンドプロセスの設定
- 証明書

補足:

- `VcIr` は、Mizar 側の義務生成と証明器側の変換との境界である。
- 前提への参照は、ATP 変換がエンコードを選ぶまでは記号的なまま保つべきである。

### AtpProblem

`AtpProblem` は、バックエンド中立な証明器問題です。

```rust
struct AtpProblem {
    problem_id: AtpProblemId,
    vc_id: VcId,
    logic_profile: LogicProfile,
    expected_result: ExpectedBackendResult,
    declarations: Vec<AtpDeclaration>,
    axioms: Vec<AtpFormula>,
    conjecture: AtpFormula,
    type_context: AtpTypeContext,
    properties: Vec<EncodedProperty>,
    symbol_map: AtpSymbolMap,
    provenance: Vec<AtpProvenance>,
}
```

所有するもの:

- エンコードされた宣言
- 選択された公理
- 予想（conjecture）
- 証明器のエンコードが必要とする型コンテキスト
- 具体構文を出力する前の、プロパティのエンコード判断
- バックエンドのシンボル対応と論理式の来歴
- 妥当性検査のための、期待されるバックエンド結果

所有しないもの:

- 外部プロセスの状態
- 証明書
- 最終的なカーネルの証明状態

補足:

- 1 つの `VcIr` から、バックエンドごとに複数の `AtpProblem` が生成され得る。
- バックエンド固有の具体構文は、プロトコルエンコーダがこの層から出力する。

### ProofCertificate

`ProofCertificate` は、バックエンドの証明根拠をラップします。

```rust
struct ProofCertificate {
    vc_id: VcId,
    backend: BackendId,
    format: CertificateFormat,
    payload: CertificatePayload,
    used_axioms: Vec<PremiseRef>,
    backend_metadata: BackendMetadata,
}
```

所有するもの:

- バックエンドの同一性とバージョンのメタデータ
- 証明書の形式タグ
- 生またはパース済みの証明ペイロード
- 利用可能な場合の、使用公理の一覧
- ソルバの時間・資源メタデータ

所有しないもの:

- 信頼された証明状態
- カーネルの再生結果
- それ自体でのソースレベルの診断

補足:

- 証明書は根拠であって、受理ではない。
- 信頼された結果を生むのはカーネルの証明書検査である。

### VerifiedArtifact

`VerifiedArtifact` は、検証済みソースファイルの安定した外部射影です。

```rust
struct VerifiedArtifact {
    schema_version: String,
    source_file: PathBuf,
    source_hash: Hash,
    module_id: ModuleId,
    exports: Vec<ExportedItem>,
    expressions: Vec<ExpressionMetadata>,
    obligations: Vec<ObligationMetadata>,
    proof_witnesses: Vec<ProofWitnessRef>,
    diagnostics: Vec<Diagnostic>,
}
```

所有するもの:

- 公開エクスポート
- IDE／AI ツール向けの式メタデータ
- 検証義務の状態
- 証明 witness と ATP ログへの参照
- ビルド中に出力された診断

所有しないもの:

- 生の AST アリーナ
- 生のソーステキスト
- ATP プロセスの完全な標準出力／標準エラー
- カーネル内部の証明状態

補足:

- 下流パッケージは、生のコンパイラ IR ではなく、エクスポートされた宣言と検証済みの状態に依存すべきである。
- アーティファクトのスキーマ変更には、明示的な `schema_version` の取り扱いが必要である。

## 層をまたぐ同一性

### 識別子

IR ノードには安定した同一性が必要ですが、すべての層が同じノードの形を持つと仮定してはなりません。

| 識別子 | スコープ | 目的 |
|---|---|---|
| `SourceId` | ビルドセッション | ソースファイルの参照 |
| `ModuleId` | パッケージグラフ | モジュールの同一性 |
| `ItemId` | モジュール | トップレベルの定義／定理／registration の同一性 |
| `NodeId` | AST アリーナ | 1 つの AST 層内での構文ノード追跡 |
| `ExprId` | ソースファイル | 式メタデータと LSP ホバー |
| `FactId` | チェッカコンテキスト | 型／cluster 事実の追跡 |
| `VcId` | ソースファイルまたはビルド | 検証条件の同一性 |

ID は、同一のソース・依存グラフ・ツールチェインに対して決定的でなければなりません。ソース編集によって完全な安定性が不可能な場合、ID は予測可能な形で劣化すべきであり、アーティファクトは内部のアリーナ索引を安定 API として公開してはなりません。

### ソースマッピング

構文解析後のすべての層は、ソースへ遡れなければなりません。

- `SourceRange` は常に `SourceId` を参照する。
- `CoreIr` と `VcIr` の低位化されたノードは、複数のソース範囲に対応してよい。
- 合成ノードは、その生成の原因となったソース構造を参照する `GeneratedFrom` マーカーを持たなければならない。
- 内部エラーが低位化された IR で見つかった場合でも、診断は最も利用者に近いソース範囲を優先すべきである。

## 増分処理の境界

IR の各層は、キャッシュと無効化の単位を定義します。

| 変更 | 無効化されるもの |
|---|---|
| コメント外の空白のみのソース編集 | `SourceUnit`。場合により診断の表示のみ |
| ドキュメントコメントの編集 | ドキュメントアーティファクト、`VerifiedArtifact` のドキュメント射影 |
| インポートの編集 | アクティブな字句環境、`TokenStream`、`SurfaceAst`、当該ファイルと逆依存のすべての意味論層 |
| シンボル宣言の編集 | `SymbolEnv`、それに依存する `ResolvedAst` 以降の層 |
| registration の編集 | `RegistrationIndex`、`ResolutionTrace`、`ResolvedTypedAst`、導出事実に依存する VC |
| 定理本体の証明の編集 | 証明に関わる `CoreIr`、`VcIr`、証明書、アーティファクトの状態 |
| アルゴリズム本体の編集 | `ControlFlowIr`、関連する VC、抽出出力 |

ビルドシステムは当初はより粗い無効化を選んでもかまいませんが、IR の所有モデルが後でより細かい無効化を妨げてはなりません。

## 診断とデバッグ容易性

各 IR 層は、コンパイラ開発のための人間可読なデバッグダンプをサポートすべきです。

- ダンプは安定したアーティファクトではない。
- ダンプには ID・ソース範囲・主要な意味論フィールドを含めるべきである。
- ダンプは、明示的に要求されない限り絶対パスを埋め込むべきではない。
- 診断は、デバッグダンプのフィールド名ではなく、安定した診断コードを参照すべきである。

LSP や AI ツール向けには、生のデバッグダンプではなく、`VerifiedArtifact` と専用の説明用アーティファクトを用います。

## 検討した代替案

### 代替案 1: 単一の注釈付き AST

1 つの AST を全フェーズを通じて in-place に書き換える案です。

最初は単純ですが、構文解析器・リゾルバ・チェッカ・ATP・アーティファクトエミッタを結合してしまいます。また、後段の各フェーズが、本来は前段のフェーズに閉じておくべきフィールドに偶然依存できてしまうため、キャッシュ無効化や部分的な診断が難しくなります。

### 代替案 2: 公開アーティファクトとしての生 IR ダンプ

コンパイラが内部 IR をそのまま `*.mizir.json` として出力する案です。

初期実装は容易ですが、内部の設計判断を外部 API として固定してしまいます。射影された `VerifiedArtifact` は、内部 IR の進化を許しつつ外部ツールを安定させます。

### 代替案 3: 唯一の意味論 IR としてのカーネル IR

フロントエンドが、構文解析済みの AST から直接カーネル向けの論理式へ elaboration する案です。

信頼される中核は明確になりますが、診断・オーバーロードの説明・LSP ホバー・ドキュメント・対話的な証明開発に必要な、表層の形を保った意味論的情報を失います。Mizar Evo には、表層の形を保った意味論 IR と、より小さな論理中核の両方が必要です。

## 採用方針

層構成を持つ IR パイプラインを用います。

1. `ResolvedTypedAst` まで表層の形を保持する。
2. 名前・型・registration・オーバーロードの意味論が完了した後にのみ `CoreIr` へ低位化する。
3. 証明義務を `VcIr` へ抽出する。
4. `VcIr` をバックエンド中立な `AtpProblem` へ変換し、その後に具体的な証明器プロトコルへ変換する。
5. 証明結果は、カーネルが検査した証明書を通じてのみ受理する。
6. 安定した外部メタデータは、生のコンパイラ IR ではなく `VerifiedArtifact` を通じて出力する。

この設計は、カーネル周りの信頼境界とアーティファクト周りの API 境界を守りつつ、コンパイラをデバッグしやすく保ちます。

## インターフェイス定義

### 共通のフェーズ出力

すべての IR を生成するフェーズは、共通の結果の形を用いるべきです。

```rust
struct IrPhaseOutput<T> {
    ir: Option<T>,
    diagnostics: Vec<Diagnostic>,
    dependencies: Vec<IrDependency>,
    debug_artifacts: Vec<DebugArtifactRef>,
}
```

`ir = None` は、そのフェーズが下流フェーズで使える IR を生成できないことを意味します。警告や情報提供の診断は `Some(ir)` と共存できます。

### ソース範囲付きの値

診断に現れ得る意味論的な値は、ソースマッピングを保持すべきです。

```rust
struct Spanned<T> {
    value: T,
    span: SourceRange,
}
```

合成された値は、明示的な生成マーカーを用いるべきです。

```rust
enum Origin {
    Source(SourceRange),
    GeneratedFrom(SourceRange, GeneratedKind),
}
```

### メタデータの射影

アーティファクトエミッタは、内部のメタデータを安定したレコードへ射影します。

```rust
struct ExpressionMetadata {
    expr_id: ExprId,
    source_range: SourceRange,
    kind: ExpressionKind,
    inferred_type: SurfaceTypeRepr,
    resolved_symbol: Option<SymbolId>,
    inserted_coercions: Vec<SurfaceCoercionRepr>,
    active_thesis: Option<SurfaceFormulaRepr>,
    overload_resolution: Option<OverloadMetadata>,
}
```

## 制約と前提

- この文書の IR 名はアーキテクチャ上の名前である。モジュール単位の設計が正当化する場合、最終的な Rust の型名は異なってよい。
- すべての診断は、低位化された IR で見つかった場合でも `SourceUnit` まで遡れなければならない。
- 内部 IR は安定した外部 API ではない。
- `VerifiedArtifact` は、下流パッケージ・LSP・ドキュメント・AI ツールにとっての主要な安定 API である。
- `CoreIr` は、VC 生成とカーネル検査に十分なほど小さく明示的でなければならない。
- `AtpProblem` と `ProofCertificate` は、カーネルの証明書検査が成功するまで、信頼された受理の境界の外側にある。
- 決定的な ID とアーティファクトの射影は、同一のソース・ロックファイル・ツールチェイン・検証器設定のもとで再現可能でなければならない。
