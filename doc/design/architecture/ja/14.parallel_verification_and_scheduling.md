# Architecture: Parallel Verification and Scheduling

> Canonical language: English. English canonical version: [../en/14.parallel_verification_and_scheduling.md](../en/14.parallel_verification_and_scheduling.md).

## 目的

この文書は、Mizar Evo が packages、modules、phases、verification conditions、ATP backends、artifact updates にまたがる verification work を scheduling する方法を定義する。

これは pipeline phases 0 and 10-15 を詳細化する。目的は、parallel verification と responsive IDE workflows を可能にしつつ、semantic results、diagnostic ordering、proof acceptance、artifact reproducibility を変えないことである。

## Design Decisions

### Scheduling Is Separate From Semantics

Scheduler は work がいつ実行されるかを決めるが、work の意味は決めない。

Scheduler がしてよいこと:

- independent modules and VCs の parallel execution order を選ぶ
- keys が一致する cache entries を reuse する
- obsolete work を cancel する
- resource budgets を enforce する
- interactive files を background work より優先する

してはならないこと:

- phase semantics を変更する
- emitted artifacts の source-visible diagnostics を並べ替える
- policy validation なしに最初の nondeterministic ATP result を accept する
- cache hits を trusted proof evidence にする
- partially-written artifacts を expose する

すべての semantic outputs は worker tasks 完了後、canonical ordering に従って sort and commit される。

### BuildPlan Produces the Initial Task Graph

Phase 0 は workspace を解決し `BuildPlan` を生成する。Scheduler はその plan を task graph に変換する。

Task graph nodes:

| Node Kind | Unit | Main Dependencies |
|---|---|---|
| `PackageResolve` | workspace/package | manifests, lockfile, registry metadata |
| `SourceLoad` | source file | package plan |
| `Frontend` | source file | source load, active lexicon seed |
| `ModuleResolve` | source file | frontend, imports, dependency summaries |
| `CheckAndElaborate` | source file | resolved AST, symbol env, registration view |
| `VcGenerate` | source file | `CoreIr`, `ControlFlowIr`, resolution traces |
| `VcDischarge` | VC | `VcIr`, deterministic facts |
| `AtpSolve` | VC portfolio | ATP problem, prover policy, obligation budget |
| `BackendRun` | VC/backend profile | translated problem, backend availability, backend budget |
| `KernelCheck` | proof candidate | certificate, kernel context, dependency proof refs |
| `ArtifactCommit` | source file/package | verified metadata, logs, witnesses |

Scheduling level の graph は acyclic である。Mutually recursive algorithms のような cyclic language constructs は、VC generation 前に grouped semantic nodes として表現される。

### Module Dependencies Gate Semantic Phases

Module は imports がすべて完了する前に lex and parse されてよいが、semantic phases は relevant dependency summaries を必要とする。

Rules:

- frontend work はすべての changed files に対して即座に実行してよい
- final name resolution は imported module summaries and active lexicon data を待つ
- type checking は visible symbol summaries and registration indexes を待つ
- VC generation は completed elaboration and algorithm preparation を待つ
- artifact commit はその source file の proof statuses and diagnostics を待つ

これにより fast syntax feedback を可能にしつつ semantic correctness を保つ。

### VCs Are Fine-Grained Tasks

Canonical `VcIr` generation 後、contexts が complete な obligations は独立に schedule してよい。

Scheduler は次を parallelize してよい。

- VCs across deterministic pre-ATP discharge
- open VCs across ATP translation
- one VC 内の backend portfolio runs
- independent certificates の kernel checking

ただし generated `VcId`s、obligation order、artifact order、diagnostic order は completion order ではなく VC generation により固定される。

### Resource Budgets Are Hierarchical

Verification は hierarchical resource controls を使う。

| Budget | Applies To | Examples |
|---|---|---|
| workspace budget | whole build | max worker threads, memory ceiling |
| package budget | package | dependency build concurrency |
| module budget | source file | max active VCs from one module |
| obligation budget | VC | ATP timeout, axiom budget, proof search policy |
| backend budget | child process | wall time, memory, stdout/stderr capture |

Portfolio execution は VC budget と global backend worker pool の両方を消費する。これにより一つの難しい obligation が build 全体を starvation させない。

### Cancellation Is Cooperative and Versioned

Cancellation は build versions により駆動される。

Scheduled task はすべて `BuildSessionId` と source/dependency snapshot に属する。より新しい edit or build request がその snapshot を supersede した場合、scheduler は obsolete work を cancel する。

Cancellation rules:

- pure in-process tasks は phase boundaries and safe checkpoints で cancellation tokens を確認する
- child ATP processes は backend manager が terminate する
- completed obsolete results は、その cache keys が new snapshot にも一致しない限り discard する
- artifact commit は atomic write transaction 開始後には cancellable ではない
- cancelled work からの diagnostics は current diagnostics として publish しない

LSP は [12.diagnostics_and_lsp.md](./12.diagnostics_and_lsp.md) に従い、newer snapshot が replacement diagnostics を publish するまで stale diagnostics を表示し続けてよい。

### Deterministic Result Ordering

Parallel execution は clean sequential build と同じ externally visible result order を生成しなければならない。

Canonical ordering:

- packages by resolved dependency topological order, then package id
- modules by package id and module path
- items by source order and stable item id
- VCs by canonical `VcId` order
- backend attempts by configured backend priority, then backend profile id
- diagnostics by source range, severity, phase order, and diagnostic code
- artifacts by module path and manifest order

Completion order は emitted order として決して使わない。

### Failure Propagation Is Bounded

Failure は dependent work を block するが、必ずしも build 全体を止めない。

Examples:

- syntax failure はその file の semantic phases を block するが、他の files の frontend work は block しない
- module interface failure はその summary を必要とする importers を block する
- 一つの failed VC はその obligation を open or failed にするが、independent VCs の reporting を妨げない
- artifact I/O failure は affected output の commit を block するが、memory 内で計算済みの proof results を retroactively invalidate しない

Scheduler は blocked tasks を明示的に記録し、diagnostics が "failed" と "dependency failed のため not run" を区別できるようにする。

## Scheduling Model

### Task State

```rust
enum TaskState {
    Pending,
    Ready,
    Running,
    Completed,
    Failed,
    Blocked,
    Cancelled,
    CacheHit,
}
```

`CacheHit` は scheduling 上 completed state だが、reused record は [11.artifact_and_incremental_build.md](./11.artifact_and_incremental_build.md) の cache validation rules を満たさなければならない。

### Task Graph

```rust
struct VerificationTask {
    id: TaskId,
    kind: TaskKind,
    snapshot: BuildSnapshotId,
    inputs: Vec<TaskInput>,
    dependencies: Vec<TaskId>,
    priority: TaskPriority,
    cache_key: Option<CacheKey>,
    resource_class: ResourceClass,
}

struct TaskResult {
    task_id: TaskId,
    state: TaskState,
    output_refs: Vec<OutputRef>,
    diagnostics: Vec<Diagnostic>,
    timing: TimingSummary,
}
```

Tasks は immutable output references を通じて通信する。互いの IR を mutate しない。

### Priority Policy

Default scheduler は次を優先する。

1. edited/open files を unblock するために必要な tasks
2. 多くの downstream files に必要な dependency summaries
3. expensive ATP tasks より cheap deterministic tasks
4. cached dependencies ready な tasks
5. background documentation/extraction work

Priority は latency に影響するが、observable semantics には影響しない。

### Work Queues

Work classes ごとに separate queues を使ってよい。

| Queue | Work |
|---|---|
| CPU local | parsing, resolving, type checking, VC generation |
| deterministic proof | pre-ATP discharge, computation with gas limits |
| ATP process | external backend child processes |
| kernel | certificate replay/checking |
| I/O commit | artifact and manifest writes |

Queue separation により、unbounded ATP processes や artifact I/O が local semantic analysis を starvation させるのを防ぐ。

## Incremental and LSP Workflows

### Batch Build

Batch mode では、scheduler は task graph 全体の完了を試みる。

Independent diagnostics を生成できる場合、errors 後も継続してよい。Final exit status は non-obsolete ready work がすべて finish または blocked になった後に決定される。

### Watch / LSP Build

Watch or LSP mode:

- 各 edit は new source snapshot を作る
- stale tasks は cancelled または deprioritized される
- syntax and local semantic feedback が優先される
- background ATP and kernel tasks は、その cache keys が newest snapshot に対して valid な場合のみ継続してよい
- result publication は immutable LSP snapshots を通じて行う

User は newer source version の diagnostics と incompatible older version の artifacts が混在したものを見るべきではない。

### Cache-Aware Scheduling

Task を schedule する前に、scheduler は cache layer に valid entry が存在するか問い合わせる。

Cache hits:

- task execution を skip する
- output references を持つ task results を生成する
- ordinary completed results と同じように sort される
- downstream tasks を即座に trigger してよい

Cache miss は ordinary ready task である。Cache lookup failure は explicit mode で cache が required でない限り diagnostic ではなく miss として扱う。

## ATP and Kernel Scheduling

### Portfolio Runs

`AtpSolve` task は一つの VC portfolio の scheduling subgraph である。その child `BackendRun` tasks が個別の backend profiles を実行する。

```rust
struct PortfolioTask {
    vc_id: VcId,
    candidates: Vec<BackendRunTask>,
    policy: ProverPolicy,
}
```

Backend candidates は parallel に実行してよい。Portfolio task が complete する条件:

- backend candidate が active policy の下で即座に受理可能な evidence を生成した
- backend candidate が、successful `KernelCheck` 後に active policy が受理できる evidence を生成した
- すべての candidate が timeout/unknown/error を返した
- cancellation が VC snapshot を supersede した

Backend runs は candidate evidence の収集だけを行う。Policy が kernel validation を要求する場合、scheduler は eligible certificates に対して `KernelCheck` tasks を生成し、composed backend-plus-kernel subgraph が acceptable winner を選ぶか、acceptable winner が残っていないと判定するまで portfolio を complete としない。

Accepted candidate は raw completion order ではなく policy により選ばれる。たとえば release policy が certificates を要求する場合、後から返った kernel-verifiable certificate が先に返った externally attested result より優先されてよい。

### Kernel Check Ordering

Kernel checking は independent certificates に対して parallel に実行してよい。各 check の kernel context は immutable で、dependency artifact hashes により content-addressed である。

Kernel results は `VcId` order で artifacts に commit される。一つの VC の certificate rejection は他の VCs の accepted results を erase しない。

## Artifact Commit

Artifact commit は parallel internal execution と stable external output の serialization boundary である。

Rules:

- 一つの module artifact には一度に一つの commit transaction だけが write する
- package manifest updates は module artifacts が ready になった後 serialized される
- committed artifact content は canonically sort される
- obsolete build sessions は newer session の artifacts を overwrite できない
- commit records は post-mortem debugging に十分な task graph provenance を含む

Commit が失敗した場合、scheduler は artifact I/O diagnostics を報告し、previous complete artifacts を intact に残す。

### Build Scheduler


```rust
struct BuildRequest {

    Batch,
    Check,

```

### Build Events

```rust
enum BuildEvent {
    TaskStarted(TaskId),
```

Build events は progress reporting and LSP integration 用である。Artifacts で使う canonical ordering ではない。

## Alternatives Considered

### Run Whole Pipeline Sequentially Per File

これは理解しやすいが、parallelism を浪費し、independent modules の feedback を遅らせる。採用案では phase dependencies を明示しつつ、independent work を concurrently に実行できる。

### Fully Demand-Driven Query Engine

Query engine は優れた incremental behavior を提供できるが、reproducibility and artifact commit boundaries が見えにくくなる。採用案では query-style internals を使ってもよいが、外部には explicit task graph and deterministic commit model を見せる。

### Let ATP Portfolio Completion Order Decide Winners

これは latency を最小にするが、proof status が machine load and process timing に依存する。採用案では completion timing を記録するが、winners は policy と、必要な場合は separate kernel acceptance により選ぶ。

### Cancel by Killing Worker Threads

Hard cancellation は shared state corruption and partial outputs の危険がある。採用案では in-process work には cooperative cancellation を使い、external backends にのみ process termination を使う。

## Adopted Approach

Mizar Evo は immutable task outputs、hierarchical resource budgets、cooperative cancellation、cache-aware execution、deterministic result ordering を備えた explicit task graph scheduler を採用する。

Parallelism は dependencies が許す場所で認めるが、artifact emission、diagnostics、VC order、proof acceptance は worker completion order ではなく canonical IDs and policy により決定される。

## Constraints and Assumptions

- Parallel scheduling must not change verified semantics or emitted artifact ordering.
- All task outputs are immutable once published to dependent tasks.
- Completion order is never used as semantic or artifact order.
- Cancellation must not leave partially-written artifacts.
- External ATP processes must be resource-limited and terminable.
- Cache hits must satisfy the same validation rules as in a clean build.
- Scheduler は初期実装でより粗い task granularity を選んでもよいが、architectural dependency boundaries を保たなければならない。
