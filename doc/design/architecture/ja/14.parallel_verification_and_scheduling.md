# アーキテクチャ: 並列検証とスケジューリング

> 正本は英語です。英語版: [../en/14.parallel_verification_and_scheduling.md](../en/14.parallel_verification_and_scheduling.md)。

## 目的

この文書は、Mizar Evo がパッケージ・モジュール・フェーズ・検証条件・ATP バックエンド・アーティファクト更新にまたがる検証作業をスケジューリングする方法を定義します。

これはパイプラインのフェーズ 0 および 10-15 を詳細化します。目的は、並列検証と応答性の高い IDE のワークフローを可能にしつつ、意味論的結果・診断の順序・証明受理・アーティファクトの再現性を変えないことです。

## 設計判断

### スケジューリングは意味論から分離される

スケジューラは作業がいつ実行されるかを決めますが、作業の意味は決めません。

スケジューラがしてよいこと:

- 独立したモジュールと VC の並列実行順序を選ぶ。
- キーが一致するキャッシュエントリを再利用する。
- 不要になった作業をキャンセルする。
- 資源予算を強制する。
- 対話的なファイルをバックグラウンド作業より優先する。

してはならないこと:

- フェーズの意味論を変更する。
- 出力されるアーティファクトの、ソースに見える診断を並べ替える。
- ポリシー検証なしに最初の非決定的な ATP 結果を受理する。
- キャッシュヒットを信頼される証明の根拠にする。
- 途中まで書かれたアーティファクトを露出する。

すべての意味論的出力は、ワーカータスクの完了後に、正準的な順序に従ってソートされ確定されます。

### BuildPlan が初期のタスクグラフを生成する

フェーズ 0 はワークスペースを解決し `BuildPlan` を生成します。スケジューラはその計画をタスクグラフに変換します。

タスクグラフのノード:

| ノードの種別 | 単位 | 主な依存 |
|---|---|---|
| `PackageResolve` | ワークスペース／パッケージ | マニフェスト, ロックファイル, レジストリメタデータ |
| `SourceLoad` | ソースファイル | パッケージ計画 |
| `Frontend` | ソースファイル | ソース読み込み, アクティブな字句環境 |
| `ModuleResolve` | ソースファイル | フロントエンド, インポート, 依存サマリ |
| `CheckAndElaborate` | ソースファイル | 解決済み AST, シンボル環境, registration ビュー |
| `VcGenerate` | ソースファイル | `CoreIr`, `ControlFlowIr`, 解決トレース |
| `VcDischarge` | VC | `VcIr`, 決定的な事実 |
| `AtpSolve` | VC ポートフォリオ | ATP 問題, 証明器ポリシー, 義務予算 |
| `BackendRun` | VC／バックエンドプロファイル | 変換された問題, バックエンドの利用可否, バックエンド予算 |
| `KernelCheck` | 証明候補 | 証明書, カーネルコンテキスト, 依存の証明参照 |
| `ArtifactCommit` | ソースファイル／パッケージ | 検証済みメタデータ, ログ, witness |

スケジューリングレベルのグラフは非循環です。相互再帰のアルゴリズムのような循環的な言語構文要素は、VC 生成より前にグループ化された意味論ノードとして表現されます。

task graph は parallel scheduling と incremental invalidation の両方に使われる共通基盤です。task dependency edge は correctness edge であり、単なる scheduling hint ではありません。task は clean-build equivalence に必要な edge がすべて満たされた場合にのみ並列実行してよく、性能のために edge を削る、または弱めることは soundness bug です。

### モジュールの依存は意味論フェーズを制御する

モジュールは、インポートがすべて完了する前に字句解析・構文解析されてよいですが、意味論フェーズは関連する依存サマリを必要とします。

ルール:

- フロントエンドの作業は、変更されたすべてのファイルに対して即座に実行してよい。
- 最終的な名前解決は、インポートされたモジュールサマリとアクティブな字句環境のデータを待つ。
- 型検査は、可視なシンボルサマリと registration 索引を待つ。
- VC 生成は、完了した elaboration とアルゴリズム準備を待つ。
- アーティファクトのコミットは、そのソースファイルの証明状態と診断を待つ。

これにより、素早い構文フィードバックを可能にしつつ意味論的な正しさを保ちます。

### VC は粒度の細かいタスクである

正準的な `VcIr` の生成後、コンテキストが完全な義務は独立にスケジュールしてよいです。

スケジューラは次を並列化してよいです。

- 決定的な ATP 前の解消をまたいだ VC
- ATP 変換をまたいだ open な VC
- 1 つの VC 内のバックエンドポートフォリオの実行
- 独立した証明書のカーネル検査

ただし、生成される `VcId`・義務の順序・アーティファクトの順序・診断の順序は、完了順ではなく VC 生成によって固定されます。

`VcId` は snapshot 内の正準 ordering key です。`ObligationAnchor` は編集をまたぐ reuse candidate の tracking key であり、`VcId` ordering を置き換えてはなりません。

### 資源予算は階層的である

検証は階層的な資源制御を用います。

| 予算 | 適用先 | 例 |
|---|---|---|
| ワークスペース予算 | ビルド全体 | 最大ワーカースレッド数, メモリ上限 |
| パッケージ予算 | パッケージ | 依存ビルドの並行度 |
| モジュール予算 | ソースファイル | 1 モジュールからの最大アクティブ VC 数 |
| 義務予算 | VC | ATP タイムアウト, 公理予算, 証明探索ポリシー |
| バックエンド予算 | 子プロセス | 実時間, メモリ, 標準出力／標準エラーの取り込み |

ポートフォリオ実行は VC 予算と大域的なバックエンドワーカープールの両方を消費します。これにより、1 つの難しい義務がビルド全体を枯渇させないようにします。

### キャンセルは協調的かつバージョン管理される

キャンセルはビルドバージョンによって駆動されます。

スケジュールされたタスクはすべて、`BuildSessionId` とソース／依存のスナップショットに属します。より新しい編集またはビルド要求がそのスナップショットを置き換えた場合、スケジューラは不要になった作業をキャンセルします。

キャンセルのルール:

- 純粋なプロセス内タスクは、フェーズの境界と安全なチェックポイントでキャンセルトークンを確認する。
- 子の ATP プロセスはバックエンドマネージャが終了させる。
- 完了した不要な結果は、そのキャッシュキーが新しいスナップショットにも一致しない限り破棄する。
- アーティファクトのコミットは、アトミックな書き込みトランザクションの開始後はキャンセルできない。
- キャンセルされた作業からの診断は、現在の診断として公開しない。

LSP は [12.diagnostics_and_lsp.md](./12.diagnostics_and_lsp.md) に従い、より新しいスナップショットが置き換えの診断を公開するまで、古い診断を表示し続けてよいです。

### 決定的な結果の順序

並列実行は、クリーンな逐次ビルドと同じ、外部から観測できる結果順序を生成しなければなりません。

正準的な順序:

- パッケージは、解決された依存のトポロジカル順、次にパッケージ ID
- モジュールは、パッケージ ID とモジュールパス
- 項目は、ソース順と安定した項目 ID
- VC は、正準的な `VcId` 順
- バックエンドの試行は、設定されたバックエンド優先度、次にバックエンドプロファイル ID
- 診断は、ソース範囲・重大度・フェーズ順・診断コード
- アーティファクトは、モジュールパスとマニフェスト順

完了順を出力順として用いることは決してありません。

### 失敗の伝播は範囲が限定される

失敗は依存する作業をブロックしますが、必ずしもビルド全体を止めません。

例:

- 構文の失敗はそのファイルの意味論フェーズをブロックするが、他のファイルのフロントエンド作業はブロックしない。
- モジュールインターフェイスの失敗は、そのサマリを必要とするインポート側をブロックする。
- 1 つの失敗した VC は、その義務を open または失敗にするが、独立した VC の報告を妨げない。
- アーティファクトの入出力の失敗は、影響を受けた出力のコミットをブロックするが、メモリ内で計算済みの証明結果を遡って無効化しない。

スケジューラはブロックされたタスクを明示的に記録し、診断が「失敗した」と「依存の失敗により実行されなかった」を区別できるようにします。

## スケジューリングモデル

### タスクの状態

```rust
enum TaskState {
    Pending,
    Ready,
    Running,
    Completed,
    Failed,
    Blocked,
    Cancelled,
    CacheHit,
}
```

`CacheHit` はスケジューリング上は完了状態ですが、再利用されるレコードは [11.artifact_and_incremental_build.md](./11.artifact_and_incremental_build.md) のキャッシュ検証ルールを満たさなければなりません。

`CacheHit` は validation 後の task state であり、proof evidence ではありません。

### タスクグラフ

```rust
struct VerificationTask {
    id: TaskId,
    kind: TaskKind,
    snapshot: BuildSnapshotId,
    inputs: Vec<TaskInput>,
    dependencies: Vec<TaskId>,
    priority: TaskPriority,
    cache_key: Option<CacheKey>,
    resource_class: ResourceClass,
}

struct TaskResult {
    task_id: TaskId,
    state: TaskState,
    output_refs: Vec<OutputRef>,
    diagnostics: Vec<Diagnostic>,
    timing: TimingSummary,
}
```

タスクはイミュータブルな出力参照を通じて通信します。互いの IR を書き換えません。

### 優先度ポリシー

既定のスケジューラは次を優先します。

1. 編集中／開いているファイルのブロックを解くために必要なタスク
2. 多くの下流ファイルに必要な依存サマリ
3. 高コストな ATP タスクより、低コストな決定的タスク
4. キャッシュされた依存が揃っているタスク
5. バックグラウンドのドキュメント／抽出作業

優先度はレイテンシに影響しますが、観測可能な意味論には影響しません。

### 作業キュー

作業の種類ごとに別々のキューを用いてよいです。

| キュー | 作業 |
|---|---|
| CPU ローカル | 構文解析, 名前解決, 型検査, VC 生成 |
| 決定的な証明 | ATP 前の解消, gas 上限つきの計算 |
| ATP プロセス | 外部バックエンドの子プロセス |
| カーネル | 証明書の再生／検査 |
| 入出力コミット | アーティファクトとマニフェストの書き込み |

キューの分離により、上限のない ATP プロセスやアーティファクトの入出力が、ローカルな意味論解析を枯渇させるのを防ぎます。

## 増分処理と LSP ワークフロー

### バッチビルド

バッチモードでは、スケジューラはタスクグラフ全体の完了を試みます。

独立した診断を生成できる場合は、エラーの後も継続してよいです。最終的な終了ステータスは、不要でない実行可能な作業がすべて完了またはブロックされた後に決定されます。

### ウォッチ／LSP ビルド

ウォッチまたは LSP モード:

- 各編集が新しいソーススナップショットを作る。
- 古いタスクはキャンセルまたは優先度を下げる。
- 構文とローカルな意味論のフィードバックが優先される。
- バックグラウンドの ATP およびカーネルのタスクは、そのキャッシュキーが最新のスナップショットに対して有効な場合のみ継続してよい。
- 結果の公開はイミュータブルな LSP スナップショットを通じて行う。

利用者は、新しいソースバージョンの診断と、互換性のない古いバージョンのアーティファクトが混在したものを見るべきではありません。

### キャッシュを意識したスケジューリング

タスクをスケジュールする前に、スケジューラはキャッシュ層に有効なエントリが存在するかを問い合わせます。

キャッシュヒット:

- タスクの実行を省く。
- 出力参照を持つタスク結果を生成する。
- 通常の完了結果と同じようにソートされる。
- 下流のタスクを即座にトリガしてよい。

キャッシュミスは通常の実行可能タスクです。キャッシュ参照の失敗は、明示モードでキャッシュが必須でない限り、診断ではなくミスとして扱います。

cache hit/miss timing は proof status、diagnostics、artifact order、selected ATP evidence、cache publication に影響してはなりません。

## ATP とカーネルのスケジューリング

### ポートフォリオ実行

`AtpSolve` タスクは、1 つの VC ポートフォリオのスケジューリング部分グラフです。その子の `BackendRun` タスクが個々のバックエンドプロファイルを実行します。

```rust
struct PortfolioTask {
    vc_id: VcId,
    candidates: Vec<BackendRunTask>,
    policy: ProverPolicy,
}
```

バックエンド候補は並列に実行してよいです。ポートフォリオタスクが完了する条件:

- 決定的なポリシー選択により受理可能な勝者があり、同じポリシーのもとで保留中の候補がそれを覆せない。
- すべての候補がタイムアウト／unknown／エラーを返したか、必要な `KernelCheck` に失敗した。
- キャンセルが VC スナップショットを置き換えた。

バックエンドの実行は、候補となる根拠の収集だけを行います。ポリシーがカーネル検証を要求する場合、スケジューラは対象となる証明書に対して `KernelCheck` タスクを生成し、合成されたバックエンドとカーネルの部分グラフが受理可能な勝者を選ぶか、受理可能な勝者が残っていないと判定するまで、ポートフォリオを完了としません。

受理される候補は、生の完了順ではなくポリシーによって決定的に選ばれます。たとえばリリースポリシーが証明書を要求する場合、後から返ったカーネル検証可能な証明書が、先に返った外部証跡の結果より優先されてよいです。複数の受理可能な certificate がある場合、canonical backend priority、certificate strength、決定的 tie-breaker によって選択し、runtime duration は provenance として記録してよいですが canonical proof identity には含めません。

### カーネル検査の順序

カーネル検査は、独立した証明書に対して並列に実行してよいです。各検査のカーネルコンテキストはイミュータブルで、依存アーティファクトのハッシュによってコンテンツアドレス化されています。

カーネルの結果は `VcId` 順でアーティファクトにコミットされます。1 つの VC の証明書棄却は、他の VC の受理結果を消去しません。

## アーティファクトのコミット

アーティファクトのコミットは、並列な内部実行と安定した外部出力の間のシリアライズ境界です。

ルール:

- 1 つのモジュールアーティファクトには、一度に 1 つのコミットトランザクションだけが書き込む。
- パッケージマニフェストの更新は、モジュールアーティファクトが揃った後に直列化される。
- コミットされたアーティファクトの内容は正準的にソートされる。
- 不要になったビルドセッションは、新しいセッションのアーティファクトを上書きできない。
- コミットレコードは、事後デバッグに十分なタスクグラフの来歴を含む。
- worker completion order は proof status、diagnostics、artifact order、selected ATP evidence、cache publication を決めてはならない。

コミットが失敗した場合、スケジューラはアーティファクトの入出力診断を報告し、以前の完全なアーティファクトをそのまま残します。

## 検討した代替案

### ファイルごとにパイプライン全体を逐次実行する

これは理解しやすいですが、並列性を浪費し、独立したモジュールのフィードバックを遅らせます。採用した設計では、フェーズの依存を明示しつつ、独立した作業を並行して実行できます。

### 完全な要求駆動のクエリエンジン

クエリエンジンは優れた増分挙動を提供できますが、再現性とアーティファクトコミットの境界が見えにくくなります。採用した設計では、クエリ風の内部実装を用いてもよいですが、外部には明示的なタスクグラフと決定的なコミットモデルを見せます。

### ATP ポートフォリオの完了順で勝者を決める

これはレイテンシを最小にしますが、証明状態がマシン負荷とプロセスのタイミングに依存してしまいます。採用した設計では、完了タイミングは記録しますが、勝者はポリシーと、必要な場合は別途のカーネル受理によって選びます。

### ワーカースレッドを kill してキャンセルする

ハードなキャンセルは、共有状態の破損と部分的な出力の危険があります。採用した設計では、プロセス内の作業には協調的なキャンセルを用い、外部バックエンドにのみプロセスの終了を用います。

## 採用方針

Mizar Evo は、イミュータブルなタスク出力・階層的な資源予算・協調的なキャンセル・キャッシュを意識した実行・決定的な結果順序を備えた、明示的なタスクグラフスケジューラを採用します。

並列性は依存が許す箇所で認めますが、アーティファクト出力・診断・VC 順・証明受理は、ワーカーの完了順ではなく、正準的な ID とポリシーによって決定されます。

## 制約と前提

- 並列スケジューリングは、検証済みの意味論や出力されるアーティファクトの順序を変えてはならない。
- すべてのタスク出力は、依存タスクに公開された後はイミュータブルである。
- 完了順を意味論やアーティファクトの順序として用いることは決してない。
- キャンセルは、途中まで書かれたアーティファクトを残してはならない。
- 外部の ATP プロセスは、資源が制限され、終了可能でなければならない。
- キャッシュヒットは、クリーンビルドと同じ検証ルールを満たさなければならない。
- 同じ `BuildSnapshot` と verifier policy に対する incremental sequential、incremental parallel、clean sequential、clean parallel build は、proof acceptance、published artifact、interface hash、dependency-facing summary、canonical diagnostics で一致しなければならない。
- スケジューラは初期実装でより粗いタスク粒度を選んでもよいが、アーキテクチャ上の依存境界を保たなければならない。
