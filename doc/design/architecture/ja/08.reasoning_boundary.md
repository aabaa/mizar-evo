# アーキテクチャ: 推論境界

> 正本は英語です。英語版: [../en/08.reasoning_boundary.md](../en/08.reasoning_boundary.md)。

## 目的

この文書は、Mizar Evo における推論責務の境界を定義します。すなわち、どの検査を Mizar 側の検証器が実行し、どの義務を外部の ATP／SMT バックエンドに委譲し、どの結果を信頼されるカーネルが受理しなければならないかを定めます。

## 設計判断

### Mizar 側の推論

Mizar 側の検証器は、決定的な意味論処理を所有します。

- 型検査、部分型検査、型強制の検査
- Fraenkel 式の sethood 検査
- cluster と registration の解決
- オーバーロード解決
- 表層構文から中核論理 IR への elaboration
- 定義的な展開の境界
- 自明な VC の解消、計算ベースの解消、型由来の事実

これらの責務は ATP 変換より前に完了していなければなりません。これらが VC を閉じる場合は、決定的な解消の説明・registration の `ResolutionTrace`・上限つきの計算トレースといった再生可能な根拠を生成しなければなりません。Mizar 側の検証器が解消済みと印を付けただけでは受理されません。フェーズ 14 は根拠を再生するか、明示的なポリシー状態として分類しなければなりません。

### ATP 側の推論

外部の証明器は、探索を多く要する論理的推論に用います。

- `by` 根拠付けステップの一階推論
- 引用された前提とローカルな仮説に基づく等式推論
- 可換性・対称性・反射性などの性質に基づく推論
- バックエンドが対応する場合の反証スタイルの証明探索

ATP バックエンドは、型推論・オーバーロード解決・cluster 解決・名前空間解決を実行しません。

ATP バックエンドに渡される性質は、バックエンドが作り出すものではありません。それらは、検査済みの定義・検証済みの registration・ローカルな仮説・引用された前提・カーネルが受理した依存アーティファクトといった、すでに解決済みの Mizar 側の事実に由来しなければなりません。ATP 変換はそれらの性質を宣言または公理としてエンコードしてよいですが、「使われた性質はすべて VC のコンテキストで利用可能だった」と証明書検査が確認できるメタデータを保持しなければなりません。

### カーネルの責務

カーネルは、独立に検査可能な根拠だけを受理します。

- ATP の結果は根拠であって、信頼された証明状態ではない。
- 証明書または再生可能な witness は、受理より前に検査されなければならない。
- ATP 前の解消の根拠は、ATP の根拠と同じ受理規律で検査されなければならない。
- `externally_attested` の証明は、ポリシーで制御された例外として記録してもよいが、カーネル検証済みの証明と同等ではない。

ポリシー状態は検証済みの証明状態とは別物です。`externally_attested`・`PolicyOpen`・`AssumedByPolicy` は、有効な検証器ポリシーが許可する場合にのみアーティファクトに現れてよいです。`require_kernel_certificates` または同等のリリースポリシーが有効な場合、そのような状態はカーネル検証済みの結果に昇格されず、エラーとして報告されます。

## 検討した代替案

1. **すべての推論を Mizar 内部で行う**: 信頼の説明は単純になるが、自動化が弱く、検証器が大きくなる。
2. **すべての推論を ATP に委譲する**: 探索は強力だが、言語固有の意味論と信頼の制御が弱くなる。
3. **ハイブリッドな境界**: Mizar が意味論処理を所有し、ATP が論理的探索を所有し、カーネルが受理を所有する。

## 採用方針

Mizar Evo はハイブリッドな境界を採用します。これにより、信頼される基盤を小さく保ち、言語固有の検査を決定的に保ちつつ、ATP／SMT バックエンドが強みを持つ領域を活用します。

## インターフェイス定義

```text
型付け・elaboration 済みの Mizar コンテキスト
  -> ローカルな VC + 引用された前提
  -> ATP 前の解消の根拠、または NeedsAtp
  -> NeedsAtp の場合、ATP 変換とバックエンドの証明書／witness
  -> カーネル証明書検査
  -> kernel_verified, discharged_builtin, externally_attested, open, またはポリシー状態
```

信頼される証明結果は、`kernel_verified` と、カーネルが検査した決定的な解消の状態だけです。その他の状態はビルドポリシーの結果であり、診断・アーティファクト・下流の依存サマリにおいて区別可能なまま保持されなければなりません。

## 制約と前提

- カーネルは ATP の出力を盲目的に信頼してはならない。
- カーネルは Mizar 側の解消の出力を盲目的に信頼してはならない。
- ATP が利用できなくても、構文解析・名前解決・型検査・cluster 解決が妨げられてはならない。
- バックエンドが成功を報告したとしても、証明書検証の失敗は証明エラーである。
- ATP の利用のためにエンコードされるすべての性質は、解決済みかつ利用可能な Mizar 側の事実へ遡れなければならない。
