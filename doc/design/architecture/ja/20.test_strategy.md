# アーキテクチャ: テスト戦略

> 正本は英語です。英語版: [../en/20.test_strategy.md](../en/20.test_strategy.md)。

## 目的

この文書は、evo2 の回帰テスト戦略を定義します。

中心となるテストの目標は次のとおりです。

```text
Reject what must not pass
before
Accept everything that should pass
```

健全性のバグは構文解析の抜けより危険です。そのため、カーネルに隣接するテストは、失敗する入力と不正な入力を重視します。

## 設計判断

### テストの構成比

全体の推奨比率:

| テストの種類 | 比率 |
|---|---:|
| 成功（pass） | 40% |
| 失敗（fail） | 60% |

カーネルに隣接するテストの推奨比率:

| テストの種類 | 比率 |
|---|---:|
| 成功（pass） | 20% |
| 失敗（fail） | 80% |

### 規模の目標

| 段階 | 目標 |
|---|---|
| evo1 プロトタイプ | 5,000〜20,000 LOC、100〜300 個の `.miz` ファイル |
| evo2 アルファ | 30,000〜100,000 LOC、300〜1,000 個の `.miz` ファイル |
| evo2 ベータ／リリース候補 | 100,000〜300,000 LOC、1,000〜5,000 個の `.miz` ファイル |
| 成熟したエコシステム | 500,000〜1,000,000 LOC 相当 |

成熟したエコシステムの目標には、手書きのテスト・移行した例・生成されたテスト・ファズコーパスのエントリ・プロパティベースのテスト・証明書スナップショット・アーティファクトスナップショットを含めます。

### 性能とメモリの評価

性能とメモリの評価は、上記の健全性優先の回帰戦略とは**別トラック**です。健全性の回帰はリリースをブロックします（「制約と前提」を参照）が、性能とメモリの結果は計測・追跡するものの、それ単独では正しさのゲートにはしません。

このトラックは、定性的な常駐集合メモリモデル（spec [§12.6.3](../../../spec/ja/12.modules_and_namespaces.md#1263-メモリモデル)、[§23.7.9](../../../spec/ja/23.package_management_and_build_system.md#2379-メモリ設計原則)）が意図的に省いている**具体的な指標**の置き場です。候補となる計測には次が含まれます。

- 検証 file ごとのピーク RSS と、import closure サイズに対するその増加。
- 定理 statement 1 万件あたり・registration 1 千件あたりのメモリ（index の密度）。
- LSP のアイドル時メモリと、hover／go-to-definition／説明クエリごとのメモリ増分。
- proof witness を外部 file から必要時に読み込むメモリコストと、witness データを `*.mizir.json` に inline する場合の比較。
- `std.all` 的な広い import と最小限の import の比較。

これらのベンチマークはメモリモデルを経験的に検証します。すなわち、ピークメモリが active な source file・その import インターフェイス・import でフィルタした index・active な module の検証条件に応じて増減し、import した証明本体・private な補題・import closure 外の登録データとは独立であることを示すべきです。仕様はこれを定性的にのみ述べており、数値的な予算と閾値はここに置かれ、非規範的です。

### AI エージェントの評価

AI エージェント操作性（[21.ai_agent_interface.md](./21.ai_agent_interface.md)）には、上記の健全性ゲートとは別の評価トラックが必要であり、corpus と patch/verify プロトコルが存在するまで**遅延**します。性能トラックと同様、これらは測定・追跡する指標であって正しさのゲートではなく、ここでの数値は非規範です。候補ベンチマーク:

- **Proof-repair ベンチマーク。** 通る証明から `by` 句の citation を意図的に抜く（`thus thesis by A, B, C;` → `thus thesis by A;`、`B, C` の再追加を期待）。top-1／top-5 の修復成功率、`verify.patch` の平均往復回数、新規エラー発生率、修復後の `used_axioms` の最小性を測る。
- **Overload-repair ベンチマーク。** 必要な `qua` を意図的に抜く（`Product(R)`、期待は `Product(R qua MulStr)`）。正しい view の選択率、不要 `qua` の挿入率、説明の正確性を測る。
- **Context-budget ベンチマーク。** 大規模 module に対し context 予算を固定（8k／16k／32k tokens）し、修復成功、関連 theorem の retrieval recall、無関係 theorem の混入を測る。

これにより AI 親和性が「雰囲気」ではなく、継続的に改善可能な engineering target になります。

### 必須の失敗系・健全性テスト

次の定理は決して成功してはなりません。

```mizar
theorem
  1 = 0;
```

cluster の失敗系テストには、無限連鎖・循環する registration・意図しない型強制・隠れた推移的展開を含めます。

置換の失敗系テストには、変数の捕獲・束縛子の衝突・不正な置換・α 変換の失敗を含めます。

オーバーロード曖昧性のテストには、曖昧な記法・隠れた型強制・不安定な解決順序を含めます。成功系テストは、適用可能な coherent same-root redefinition が曖昧性なしに join されるケースも対象にしなければなりません。これにより、fail suite が妥当な `RefinementJoin` ケースを誤って拒否しないようにします。

証明書のテストには、不正な証明書構造・不正な置換・不正な SAT 証明・未解決のシンボル・タイムアウト・資源枯渇の各ケースを含めます。

### スナップショットテスト

スナップショットテストは次を対象とします。

- `SurfaceAst`
- `TypedAst`
- `CoreIr`
- `VcIr`
- SAT 節
- `ProofCertificate`
- `VerifiedArtifact`
- 依存スライスとフィンガープリント
- 失敗レコード

スナップショットのハッシュは決定的でなければなりません。

### プロパティベースのテスト

プロパティベースのテストは次を生成します。

- ランダムな定理
- ランダムな置換
- ランダムな束縛子の構造
- ランダムな cluster グラフ
- 不正な証明書
- インポートグラフと依存スライス

健全性のバグや非決定性を露出した生成ケースは最小化し、回帰テストとしてコミットします。

### ファジング

継続的なファジングは次を対象とします。

- 字句解析の前処理と生スキャナの入力
- 構文解析器の入力
- 証明書のバイト列
- 置換の根拠
- オーバーロード候補集合
- インポートグラフ

ファズの失敗は [19.failure_semantics.md](./19.failure_semantics.md) に従って分類します。クラッシュ・パニック・非決定的な受理・不健全な成功はリリースブロッカーです。

### Codex 実装のガードレール

AI 支援による実装は、次を導入してはなりません。

- カーネルのヒューリスティック
- 暗黙の cluster 挿入
- 証明書検査なしの ATP 成功の信頼
- 非決定的なハッシュ生成
- 順序付けされていない反復への依存
- 隠れた大域的な可変状態
- トレースの根拠を伴わないフォールバック推論

## インターフェイス定義

```text
test case
  -> expected phase
  -> expected pass/fail
  -> expected failure category, if fail
  -> expected snapshot hashes, when applicable
```

## 制約と前提

- クリーンな逐次ビルドを、決定的なテスト期待値の基準とする。
- 並列テスト実行は、同一の入力に対してバイト単位で同一のスナップショットを生成しなければならない。
- 健全性の回帰は、通常の成功系カバレッジが改善していてもリリースをブロックする。
- 生成／ファズのテストは、修正済みの健全性バグごとに最小化された再現ケースを保持する。
