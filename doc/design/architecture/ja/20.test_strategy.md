# アーキテクチャ: テスト戦略

> 正本は英語です。英語版: [../en/20.test_strategy.md](../en/20.test_strategy.md)。

## 目的

この文書は evo2 の regression strategy を定義する。

中心となる testing goal は次である。

```text
Reject what must not pass
before
Accept everything that should pass
```

soundness bug は parser gap より危険である。そのため kernel-adjacent tests は failing input と malformed input を重視する。

## 設計判断

### Test Mix

全体の推奨 ratio:

| Test Kind | Ratio |
|---|---:|
| pass | 40% |
| fail | 60% |

kernel-adjacent tests の推奨 ratio:

| Test Kind | Ratio |
|---|---:|
| pass | 20% |
| fail | 80% |

### Scale Targets

| Stage | Target |
|---|---|
| evo1 prototype | 5,000-20,000 LOC; 100-300 `.miz` files |
| evo2 alpha | 30,000-100,000 LOC; 300-1,000 `.miz` files |
| evo2 beta / release candidate | 100,000-300,000 LOC; 1,000-5,000 `.miz` files |
| mature ecosystem | 500,000-1,000,000 LOC equivalent |

mature ecosystem target は handwritten tests、migrated examples、generated tests、fuzz corpus entries、property-based tests、certificate snapshots、artifact snapshots を含む。

### Recommended File Sizes

| Purpose | Lines |
|---|---:|
| parser test | 5-30 |
| type test | 10-50 |
| cluster test | 20-80 |
| theorem test | 30-150 |
| integration test | 100-300 |
| stress test | 500-1,000 |

### Required Fail and Soundness Tests

次の theorem は決して pass してはならない。

```mizar
theorem
  1 = 0;
```

cluster failure tests は infinite chains、cyclic registrations、unintended coercion、hidden transitive expansion を含める。

substitution failure tests は variable capture、binder collision、malformed substitution、alpha-conversion failure を含める。

overload ambiguity tests は ambiguous notation、hidden coercion、unstable resolution order を含める。

certificate tests は malformed certificate structure、invalid substitution、invalid SAT proof、unresolved symbol、timeout、resource exhaustion cases を含める。

### Snapshot Tests

snapshot tests は次を cover する。

- `SurfaceAst`
- `TypedAst`
- `CoreIr`
- `VcIr`
- SAT clauses
- `ProofCertificate`
- `VerifiedArtifact`
- dependency slices and fingerprints
- failure records

snapshot hashes は deterministic でなければならない。

### Property-Based Tests

property-based tests は次を generate する。

- random theorems
- random substitutions
- random binder structures
- random cluster graphs
- malformed certificates
- import graphs and dependency slices

soundness bug または nondeterminism を露出した generated case は minimize し、regression test として commit する。

### Fuzzing

continuous fuzzing は次を cover する。

- lexer preprocessing and raw scanner inputs
- parser inputs
- certificate bytes
- substitution evidence
- overload candidate sets
- import graphs

fuzz failures は [19.failure_semantics.md](./19.failure_semantics.md) に従って classify する。crash、panic、nondeterministic acceptance、unsound success は release blocker である。

### Codex Implementation Guardrails

AI-assisted implementation は次を導入してはならない。

- kernel heuristics
- implicit cluster insertion
- certificate checking なしの ATP success trust
- nondeterministic hash generation
- unordered iteration dependence
- hidden global mutable state
- trace evidence なしの fallback inference

## インターフェイス定義

```text
test case
  -> expected phase
  -> expected pass/fail
  -> expected failure category, if fail
  -> expected snapshot hashes, when applicable
```

## 制約と前提

- clean sequential build を deterministic test expectations の reference とする。
- parallel test run は同一 input に対して byte-identical snapshots を生成しなければならない。
- soundness regression は normal pass coverage が改善していても release を block する。
- generated / fuzz tests は、修正済み soundness bug ごとに minimized reproducer を保持する。
