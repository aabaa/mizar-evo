## 目的

この文書は、Mizar Evo がアルゴリズム本体を検証のために準備し、中核の証明ゴールと正しさの条件を `VcIr` へ変換し、ATP 変換より前に決定的な義務を解消する方法を定義します。

これは [00.pipeline_overview.md](./00.pipeline_overview.md) のフェーズ 10・11・12 を詳細化します。とくに、バックエンド中立な `CoreIr`、アルゴリズム固有の `ControlFlowIr`、証明器に依存しない `VcIr`、そして ATP 変換が受け取る対象を選別する ATP 前の解消結果との境界を定義します。

## 設計判断

### VC 生成は証明器に依存しない

`VcIr` は ATP 変換より前の、最後の Mizar 側の表現です。ゴール・前提・ローカルコンテキスト・ソースマップ・証明ヒントを記録しますが、TPTP・SMT-LIB・具体的なバックエンドプロファイルには確定しません。

フェーズ 11 の VC 生成が所有するもの:

- `ObligationSeedId` を具体的な `VcId` へ変換する
- アルゴリズムの制御フローの事実を準備する
- 証明スケルトンの葉から証明義務を生成する
- 定義・再定義・registration・reduction の正しさの義務を生成する
- アルゴリズムの契約・不変条件・assertion・呼び出し事前条件・停止性の義務を生成する
- ローカルな前提集合と引用された参照を付与する

フェーズ 12 の ATP 前の解消が所有するもの:

- 正準的な `VcIr` に対する決定的な解消を試みる
- ATP 前の `VcStatus` の値を割り当てる
- ATP に送られる VC のコンテキストを弱めずに保持する

このフェーズが所有しないもの:

- 名前・型・registration・オーバーロード・elaboration の判断
- ATP のエンコードやバックエンドの選択
- 外部プロセスの管理
- 証明書の検証
- 安定したアーティファクトの出力

### 義務シードは正確に一度だけ VC になる

フェーズ 9 は義務シードを作りますが `VcId` は作りません。`VcId` の割り当てを所有するのはフェーズ 11 だけです。

各シードは 0 個以上の VC に変換されます。

- シードがポリシーで無効化された、または不正でスキップされた項目に属する場合は 0 個
- 通常の定理／証明／定義のゴールでは 1 個
- ループ不変条件の確立と維持のように、1 つのソース構文要素が複数の検査に展開される場合は複数個

シードと VC の対応付けは、診断・LSP メタデータ・アーティファクト・増分ビルドがすべての VC をソースへ遡れるよう記録します。

### ControlFlowIr はアルゴリズム固有である

アルゴリズム本体は、定理の証明にはない情報を必要とします。すなわち、変更・分岐・ループ・早期脱出・ghost 状態・停止性測度です。フェーズ 10 は VC 生成より前に `ControlFlowIr` を構築します。

`ControlFlowIr` が所有するもの:

- 基本ブロックと辺
- ローカル変数の代入状態
- return・break・continue による脱出
- ループのヘッダとカットポイント
- ループ構造に付与された不変条件と decreasing の注釈
- ghost／ランタイムの使用分類
- 呼び出し箇所と、その契約の実体化

証明結果は所有しません。フェーズ 11 が必要とする構造化されたプログラムの事実を準備するだけです。

### VC はローカルコンテキストを明示的に保つ

すべての VC は自己完結したローカルコンテキストを持ちます。

- 導入された束縛子と型述語
- 証明スケルトン由来の仮定
- アルゴリズムの経路条件
- 利用可能なループ不変条件
- `by` 根拠付け由来の引用された前提
- 型検査と解決トレース由来の事実
- 定義の展開ポリシーと選択された展開

コンテキストは記号的かつ正準的です。ATP 変換は後で前提のエンコードや刈り込みの方法を選んでよいですが、ソース構文から意味論的なコンテキストを再構築してはなりません。

### ATP 前の解消は決定的かつ説明可能である

フェーズ 12 は、外部探索なしに Mizar 側の機構で解ける義務を扱います。

許される解消の出所には次が含まれます。

- 構文的なトートロジーと矛盾の検査
- 反射性と直接的な等号正規化
- `TypeFactTable` にすでに存在する型事実
- 再生可能な `ResolutionTrace` を持つ cluster の事実
- 展開ポリシーによって許可された定義的な簡約
- 設定された gas とタイムアウトのもとでの `by computation`

ATP 前の解消は `AssumedByPolicy` や `PolicyOpen` のようなポリシー状態を割り当ててもよいですが、これらは解消の根拠ではありません。解消は説明を記録しなければなりません。フェーズ 12 は VC を黙って捨ててはなりません。外部探索を必要とする VC は `NeedsAtp` として印付けされ、ソース・ローカルコンテキスト・証明ヒントを保持したまま ATP 変換へ進みます。

### 定義の展開は制御される

VC 生成は、`CoreIr` の明示的な展開ポリシーを通じてのみ定義を展開してよいです。

ポリシーは次を制御します。

- 定義が既定で opaque か transparent か
- reduction registration が簡約を許すか
- 計算可能なアルゴリズムを `by computation` で評価してよいか
- 証明ヒントが特定の VC に対するローカルな展開を要求しているか

これにより、走査順による証明強度や ATP 問題サイズの偶発的な変化を防ぎます。

### アルゴリズムの VC は構造化された制御フローに従う

アルゴリズム検証は、`ControlFlowIr` 上の Hoare スタイルのスキーマを用います。

ジェネレータは次の VC を作ります。

- 呼び出し箇所の `requires`
- 各 return の `ensures`
- プログラム地点の `assert`
- ループ進入前の不変条件の確立
- 各ループの後退辺における不変条件の維持
- 通常のループ脱出における不変条件の利用可能性
- `break` および `continue` における不変条件の保存
- decreasing 測度の非負性と狭義の減少
- コレクションループの有限性と順序非依存性の要件
- ghost 消去の安全性

再帰および相互再帰のアルゴリズムでは、停止性の義務は呼び出しグラフの成分と decreasing 測度ごとにグループ化されます。

### 状態とポリシーは VC に反映される

定理の状態は VC 生成と解消に影響します。

- `assumed` の項目は、ポリシーに従ってアーティファクトメタデータと前提を生成するが、その命題の証明 VC は生成しない。
- `open` の項目は、ATP ディスパッチなしに open な証明状態を生成してよい。
- `conditional` の項目は、clean でない依存を記録しなければならない。
- 通常の定理は、clean でない項目に依存してはならない。

ポリシーによる判断は、欠落した義務として隠すのではなく、`VcStatus` に明示的に表現しなければなりません。

## VC パイプライン

### ステップ 1: ControlFlowIr の構築

入力:

- `CoreIr` 由来の中核アルゴリズムシェル
- 低位化された契約と文の木
- ghost／ランタイムのメタデータ
- ソースマップ

出力:

- アルゴリズムごとの `ControlFlowIr`
- アルゴリズムの診断
- 文からブロックへのソースマップ

責務:

- 基本ブロックと制御フローの辺を構築する。
- 通常・return・break・continue の脱出を解決する。
- ループ不変条件と decreasing 測度をループのヘッダに付与する。
- 代入状態と代入前使用を追跡する。
- ghost 専用の式と、不正な ghost からランタイムへのフローを分類する。
- 再現可能な診断のために文の順序を保持する。

### ステップ 2: 義務シードの収集

入力:

- `CoreIr`
- `ControlFlowIr`
- 義務シードテーブル
- 定理と registration の状態メタデータ

出力:

- VC 生成が可能な、順序付けられた `ObligationSeed`
- シードの依存グラフ

責務:

- 証明スケルトンの末端ゴールを収集する。
- 定義と registration の正しさのシードを収集する。
- CFG の構造が含意するアルゴリズムの契約と停止性のシードを収集する。
- スキップされた不正な項目に属するシードを除外する。
- モジュール・項目・ソース範囲・シード種別で決定的にソートする。

### ステップ 3: 定理と定義の VC の生成

入力:

- 証明木と定理の命題
- 定義と registration の中核項目
- ローカルな証明コンテキスト
- 引用されたラベルと前提

出力:

- 定理・証明ステップ・定義・registration の義務の `VcIr`

責務:

- 定理の束縛子と型述語を実体化する。
- 末端の証明ゴールを VC のゴールへ変換する。
- `by` 根拠付け由来の引用された前提を付与する。
- existence・uniqueness・coherence・compatibility・consistency・reducibility の義務を生成する。
- clean／clean でないの検査のために状態依存を記録する。

### ステップ 4: アルゴリズムの VC の生成

入力:

- `ControlFlowIr`
- アルゴリズムの契約
- 呼び出しグラフと停止性のメタデータ

出力:

- アルゴリズムの義務の `VcIr`
- アルゴリズム固有の診断メタデータ

責務:

- 呼び出しの事前条件 VC を生成する。
- return の事後条件 VC を生成する。
- assertion VC を生成する。
- ループ不変条件の確立と維持の VC を生成する。
- break／continue の不変条件 VC を生成する。
- decreasing 測度と停止性の VC を生成する。
- ghost 消去の安全性 VC を生成する。
- 経路条件を、バックエンド固有の論理式ではなくローカルコンテキストとしてエンコードする。

### ステップ 5: VC の正規化と分類

入力:

- 生の VC
- ローカルコンテキスト
- ソースマップ
- 証明ヒント

出力:

- 正準的な `VcIr`
- `VcIndex`
- 再現可能な VC の順序

責務:

- α 同値の束縛子を正規化する。
- 前提を安定した参照でソートする。
- ソース範囲と義務シードの来歴を付与する。
- VC の種別と優先度を分類する。
- 証明ヒントと計算上限を付与する。
- 決定的な `VcId` を割り当てる。

### ステップ 6: ATP 前の解消

入力:

- 正準的な `VcIr`
- ローカルな型事実と cluster 事実
- 定義の展開ポリシー
- 計算エンジンの設定

出力:

- 説明付きの解消済み VC
- ATP 変換向けの `NeedsAtp` VC
- 決定的な解消の診断

責務:

- 自明なゴールと型由来のゴールを解く。
- `ResolutionTrace` を用いて cluster 由来の事実を再生する。
- 許可された簡約と定義的な簡約を適用する。
- 設定された上限内で `by computation` のゴールを評価する。
- ポリシーで制御された assumed／open の状態を印付けする。
- ATP に送られる VC のコンテキストを弱めずに保持する。

```rust
    algorithm: SymbolId,
    blocks: Vec<BasicBlock>,

```

### 基本ブロック

```rust
struct BasicBlock {
    id: BasicBlockId,
    statements: Vec<CoreStatementId>,
    terminator: Terminator,
    context_in: ProgramContextId,
    context_out: Vec<ProgramContextId>,
}

enum Terminator {
    Goto(BasicBlockId),
    Branch { condition: CoreFormulaId, then_bb: BasicBlockId, else_bb: BasicBlockId },
    Return(Option<CoreTermId>),
    Break(LoopId),
    Continue(LoopId),

```

### 義務シード

```rust
struct ObligationSeed {
    id: ObligationSeedId,

    Termination { algorithm: SymbolId, call_or_loop: TerminationSiteId },

```

### VC IR

```rust
struct VcIr {
    id: VcId,
    seed: ObligationSeedId,
    source_range: SourceRange,
    kind: VcKind,
    local_context: LocalContext,
    premises: Vec<PremiseRef>,
    goal: CoreFormulaId,

    TheoremStep,
    DefinitionCorrectness(CorrectnessKind),

```

### VC の状態

```rust
enum VcStatus {
    PolicyOpen(PolicyRef),
    Discharged(DischargeEvidence),
    NeedsAtp,
    SkippedDueToInvalidInput,
    AssumedByPolicy(PolicyRef),
}

struct DischargeEvidence {
    method: DischargeMethod,
    premises: Vec<PremiseRef>,
    trace: Option<ResolutionTraceRef>,
}
```

`NeedsAtp` はフェーズ 13 の入力状態です。`PolicyOpen` は、有効なポリシーのもとで ATP に送られない、意図的に open な義務を記録します。`AssumedByPolicy` は受理された仮定を記録するもので、証明の根拠ではありません。`VcStatus` はアーティファクト出力までは内部的なものです。カーネルが受理した証明状態は後続で割り当てられます。

## 診断

VC の診断は ATP 変換より前に生成されます。

| 診断クラス | 例 |
|---|---|
| CFG 構築 | 到達不能な文、代入前使用、不正な break |
| ghost／ランタイムの分離 | ghost 値がランタイムの式へ流れ込む |
| 義務生成 | ループ不変条件の欠落、未対応の文の形 |
| 停止性 | decreasing 測度の欠落、Nat 値でない、減少しない |
| コンテキスト構築 | 引用された前提が利用不可、ローカルな事実がスコープを逸脱する |
| ATP 前の解消 | 計算上限の超過、再帰的な定義展開 |
| ポリシー | clean な証明が必要な箇所で open または assumed の項目が使われた |

診断には次を含めるべきです。

- 義務シードのソース位置
- 利用可能なら生成された `VcId`
- デバッグ表示用のローカルコンテキストの要約
- 関連するループ・呼び出し・定理・registration のソース位置
- 決定的な解消の説明、または失敗の理由

## 増分処理と再現性

VC のキャッシュキーには次を含めます。

- `CoreIr` のハッシュ
- アルゴリズムの義務用の `ControlFlowIr` のハッシュ
- 義務シードのハッシュ
- 関連する型事実と解決トレースのハッシュ
- 検証器ポリシーと計算上限の設定
- VC のスキーマバージョン

変更は次を無効化します。

- 変更された義務シードから生成された VC
- CFG・契約・停止性メタデータが変わったアルゴリズムの VC
- 変更された `NeedsAtp` VC に対応する下流の ATP 問題
- 変更された VC の状態や診断を記録するアーティファクト

順序は決定的でなければなりません。

- 義務シードは、モジュール・項目・ソース範囲・由来種別でソートする。
- 生成される `VcId` は、ソート済みのシードリストとシードごとの展開索引から割り当てる。
- 前提は正準的な `PremiseRef` でソートする。
- 解消済み・ポリシー状態・`NeedsAtp` の各 VC リストは VC 順を保持する。
- 同じソース範囲の診断は、VC の種別と診断コードで順序付ける。

## 検討した代替案

### ResolvedTypedAst から直接 VC を生成する

ソースの形を保った構文から VC を生成すると表層の詳細は保持しやすいですが、elaboration のロジックが重複し、証明コンテキストが記法に依存します。採用した設計では、`CoreIr` とソースマップから VC を生成します。

### ControlFlowIr と CoreIr を統合する

CFG のデータを `CoreIr` に直接入れるとアルゴリズム検証は呼び出しやすくなりますが、アルゴリズム以外の消費者がプログラム解析の詳細を扱わなければならなくなります。採用した設計では、アルゴリズム固有のフローを `ControlFlowIr` に分離します。

### すべての VC を ATP へ送る

すべての義務を ATP に委譲するとフェーズ 12 は単純になりますが、遅く、非決定的になり、型／cluster の説明が隠れます。採用した設計では、決定的な義務を ATP 変換より前に解消します。

### VC 生成中に ATP 前の解消を行う

生成と解消を交互に行うとメモリは節約できますが、診断とスケジューリングの再現性が下がります。採用した設計では、まず正準的な `VcIr` を生成し、その後に別ステップとして決定的な解消を行います。

## 採用方針

Mizar Evo は、検証準備の境界を 3 段階で構成します。

フェーズ 10 はアルゴリズム本体の `ControlFlowIr` を構築します。フェーズ 11 は `CoreIr` と `ControlFlowIr` からの義務シードを、決定的な `VcId` を持つ正準的な `VcIr` へ変換します。フェーズ 12 は、決定的な Mizar 側の機構で解ける義務を解消し、設定されたポリシー状態を割り当て、`NeedsAtp` の VC だけを ATP 変換へ渡します。

## 制約と前提

- VC 生成は、elaboration が `CoreIr` を生成した後にのみ開始する。
- フェーズ 10 は、VC が生成される前に不正なアルゴリズム制御フローを却下してよい。
- `VcId` を割り当てるのはフェーズ 11 だけである。
- フェーズ 12 は VC を解消したりポリシー状態を割り当てたりしてよいが、ATP に送られる VC を消去したりコンテキストを弱めたりしてはならない。
- ATP 変換は、`NeedsAtp` 状態を持つ正準的な `VcIr` だけを受け取る。
- ATP 前の解消は、同一のソース・依存・ツールチェイン・ポリシー・計算上限に対して決定的でなければならない。
