# アーキテクチャ: VC 生成

> 正本は英語です。英語版: [../en/07.vc_generation.md](../en/07.vc_generation.md)。

## 目的

この文書は、Mizar Evo が algorithm bodies を verification のために準備し、core proof goals and correctness conditions を `VcIr` に変換し、ATP translation 前に deterministic obligations を discharge する方法を定義する。

これは [00.pipeline_overview.md](./00.pipeline_overview.md) の phase 10、11、12 を詳細化する。特に、backend-neutral な `CoreIr`、algorithm-specific な `ControlFlowIr`、prover-independent な `VcIr`、ATP translation が受け取る対象を選別する pre-ATP discharge result の境界を定義する。

### Pipeline Position

| Phase | Input | Output | This Document Defines |
|---|---|---|---|
| 10. Algorithm Verification Preparation | `CoreIr` algorithm shells | `ControlFlowIr` | CFG construction、ghost/runtime separation、contract placement、termination metadata |
| 11. VC Generation | `CoreIr`, `ControlFlowIr`, obligation seeds | `VcIr` | theorem、proof-step、definition、registration、algorithm obligation generation |
| 12. Pre-ATP Discharge | `VcIr`, local facts, deterministic evaluators | discharged、policy-status、または `NeedsAtp` VCs | trivial、type-derived、cluster-derived、definitional、computation-based discharge |

## 設計判断

### VC Generation Is Prover-Independent

`VcIr` は ATP translation 前の最後の Mizar-side representation である。Goals、premises、local context、source maps、proof hints を記録するが、TPTP、SMT-LIB、concrete backend profile には commit しない。

Phase 11 VC generation は次を所有する。

- `ObligationSeedId`s を concrete `VcId`s に変換する
- algorithm control-flow facts を準備する
- proof skeleton leaves から proof obligations を生成する
- definitions、redefinitions、registrations、reductions の correctness obligations を生成する
- algorithm contract、invariant、assertion、call precondition、termination obligations を生成する
- local premise sets and cited references を付与する

Phase 12 pre-ATP discharge は次を所有する。

- canonical `VcIr` に対する deterministic discharge を試みる
- pre-ATP `VcStatus` values を割り当てる
- ATP-bound VCs の context を弱めずに保持する

これは次を所有しない。

- name、type、registration、overload、elaboration decisions
- ATP encoding or backend selection
- external process management
- certificate validation
- stable artifact emission

### Obligation Seeds Become VCs Exactly Once

Phase 9 は obligation seeds を作るが `VcId`s は作らない。Phase 11 だけが `VcId` assignment を所有する。

各 seed は 0 個以上の VCs に変換される。

- seed が policy-disabled または invalid skipped item に属する場合は 0
- ordinary theorem/proof/definition goals では 1
- loop invariant establishment and maintenance のように 1 つの source construct が複数 checks に展開される場合は複数

Seed-to-VC mapping は、diagnostics、LSP metadata、artifacts、incremental builds がすべての VC を source へ trace できるよう記録する。

### ControlFlowIr Is Algorithm-Specific

Algorithm bodies は theorem proofs にはない情報を必要とする。Mutation、branching、loops、early exits、ghost state、termination measures である。Phase 10 は VC generation 前に `ControlFlowIr` を構築する。

`ControlFlowIr` は次を所有する。

- basic blocks and edges
- local variable assignment state
- return、break、continue exits
- loop headers and cutpoints
- loop structure に付与された invariant and decreasing annotations
- ghost/runtime use classification
- call sites and their contract instantiations

Proof results は所有しない。Phase 11 が必要とする structured program facts を準備するだけである。

### VCs Preserve Local Context Explicitly

すべての VC は self-contained local context を持つ。

- introduced binders and type predicates
- proof skeletons 由来の assumptions
- algorithm path conditions
- available loop invariants
- `by` justifications 由来の cited premises
- type checking and registration traces 由来の facts
- definition expansion policy and selected unfoldings

Context は symbolic and canonical である。ATP translation は後で premises の encode or prune 方法を選んでよいが、source syntax から semantic context を再構築してはならない。

### Pre-ATP Discharge Is Deterministic and Explainable

Phase 12 は external search なしに Mizar-side machinery が解ける obligations を扱う。

Allowed discharge sources は次を含む。

- syntactic tautologies and contradiction checks
- reflexivity and direct equality normalization
- `TypeFactTable` にすでに存在する type facts
- replayable `ResolutionTrace` を持つ cluster facts
- expansion policy により許可された definitional reductions
- configured gas and timeout の下での `by computation`

Pre-ATP discharge は `AssumedByPolicy` or `PolicyOpen` のような policy statuses も割り当ててよいが、これらは discharge evidence ではない。Discharge は explanation を記録しなければならない。Phase 12 は VCs を黙って drop してはならない。External search を必要とする VCs は `NeedsAtp` として mark され、source、local context、proof hints を保持したまま ATP translation へ進む。

### Definition Unfolding Is Controlled

VC generation は `CoreIr` の explicit expansion policy を通じてのみ definitions を unfold してよい。

Policy は次を制御する。

- definition が default で opaque か transparent か
- reduction registration が simplification を許すか
- computable algorithm が `by computation` で evaluate されてよいか
- proof hint が特定 VC に対する local unfolding を要求しているか

これにより traversal order による accidental changes in proof strength or ATP problem size を防ぐ。

### Algorithm VCs Follow Structured Control Flow

Algorithm verification は `ControlFlowIr` 上の Hoare-style schema を使う。

Generator は次の VCs を作る。

- call sites の `requires`
- 各 return の `ensures`
- program point の `assert`
- loop entry 前の invariant establishment
- 各 loop back edge の invariant maintenance
- normal loop exit における invariant availability
- `break` and `continue` における invariant preservation
- decreasing measure non-negativity and strict decrease
- collection loops の finiteness and order-independence requirements
- ghost erasure safety

Recursive and mutually recursive algorithms では、termination obligations は call graph component and decreasing measure ごとに group される。

### Status and Policy Are Reflected in VCs

Theorem statuses は VC generation and discharge に影響する。

- `assumed` items は policy に従って artifact metadata and premises を生成するが、その proposition の proof VC は生成しない
- `open` items は ATP dispatch なしに open proof status を生成してよい
- `conditional` items は non-clean dependencies を記録しなければならない
- ordinary theorems は non-clean items に依存してはならない

Policy decisions は missing obligations として隠すのではなく、`VcStatus` に明示的に表現しなければならない。

## VC パイプライン

### Step 1: Build ControlFlowIr

Input:

- `CoreIr` からの core algorithm shells
- lowered contracts and statement trees
- ghost/runtime metadata
- source maps

Output:

- algorithm ごとの `ControlFlowIr`
- algorithm diagnostics
- statement-to-block source maps

Responsibilities:

- basic blocks and control-flow edges を構築する
- normal、return、break、continue exits を解決する
- loop invariants and decreasing measures を loop headers に付与する
- assignment state and use-before-assignment を track する
- ghost-only expressions and illegal ghost-to-runtime flows を分類する
- reproducible diagnostics のため statement order を保持する

### Step 2: Collect Obligation Seeds

Input:

- `CoreIr`
- `ControlFlowIr`
- obligation seed table
- theorem and registration status metadata

Output:

- VC generation 可能な ordered `ObligationSeed`s
- seed dependency graph

Responsibilities:

- proof skeleton terminal goals を collect する
- definition and registration correctness seeds を collect する
- CFG structure が imply する algorithm contract and termination seeds を collect する
- skipped invalid items に属する seeds を除外する
- module、item、source range、seed kind で deterministic に sort する

### Step 3: Generate Theorem and Definition VCs

Input:

- proof trees and theorem propositions
- definition and registration core items
- local proof contexts
- cited labels and premises

Output:

- theorem、proof-step、definition、registration obligations の `VcIr`

Responsibilities:

- theorem binders and type predicates を instantiate する
- terminal proof goals を VC goals に変換する
- `by` justifications 由来の cited premises を付与する
- existence、uniqueness、coherence、compatibility、consistency、reducibility obligations を生成する
- clean/non-clean checks のため status dependencies を記録する

### Step 4: Generate Algorithm VCs

Input:

- `ControlFlowIr`
- algorithm contracts
- call graph and termination metadata

Output:

- algorithm obligations の `VcIr`
- algorithm-specific diagnostic metadata

Responsibilities:

- calls の precondition VCs を生成する
- returns の postcondition VCs を生成する
- assertion VCs を生成する
- loop invariant establishment and maintenance VCs を生成する
- break/continue invariant VCs を生成する
- decreasing measure and termination VCs を生成する
- ghost erasure safety VCs を生成する
- path conditions を backend-specific formulas ではなく local context として encode する

### Step 5: Normalize and Classify VCs

Input:

- raw VCs
- local contexts
- source maps
- proof hints

Output:

- canonical `VcIr`
- `VcIndex`
- reproducible VC ordering

Responsibilities:

- alpha-equivalent binders を normalize する
- premises を stable references で sort する
- source ranges and obligation seed provenance を付与する
- VC kind and priority を classify する
- proof hints and computation limits を付与する
- deterministic `VcId`s を割り当てる

### Step 6: Pre-ATP Discharge

Input:

- canonical `VcIr`
- local type and cluster facts
- definition expansion policy
- computation engine configuration

Output:

- explanations 付き discharged VCs
- ATP translation 向け `NeedsAtp` VCs
- deterministic discharge diagnostics

Responsibilities:

- trivial and type-derived goals を解く
- `ResolutionTrace` を使って cluster-derived facts を replay する
- allowed reductions and definitional simplifications を適用する
- configured limits 内で `by computation` goals を evaluate する
- policy-controlled assumed/open statuses を mark する
- ATP-bound VCs の context を弱めずに保持する

## インターフェイス定義

### ControlFlowIr

```rust
struct ControlFlowIr {
    algorithm: SymbolId,
    blocks: Vec<BasicBlock>,
    locals: LocalTable,
    contracts: ContractSet,
    loops: Vec<LoopInfo>,
    exits: Vec<ExitInfo>,
    ghost_effects: GhostEffectTable,
    termination: Option<TerminationPlan>,
    source_map: ControlFlowSourceMap,
}
```

### Basic Block

```rust
struct BasicBlock {
    id: BasicBlockId,
    statements: Vec<CoreStatementId>,
    terminator: Terminator,
    context_in: ProgramContextId,
    context_out: Vec<ProgramContextId>,
}

enum Terminator {
    Goto(BasicBlockId),
    Branch { condition: CoreFormulaId, then_bb: BasicBlockId, else_bb: BasicBlockId },
    Return(Option<CoreTermId>),
    Break(LoopId),
    Continue(LoopId),
    Unreachable,
}
```

### Obligation Seed

```rust
struct ObligationSeed {
    id: ObligationSeedId,
    origin: ObligationOrigin,
    source: CoreSourceRef,
    local_context: CoreContextSnapshot,
    goal: CoreFormulaId,
    hints: Vec<ProofHint>,
}

enum ObligationOrigin {
    ProofStep { theorem: SymbolId, proof_node: CoreProofNodeId },
    DefinitionCorrectness { symbol: SymbolId, condition: CorrectnessKind },
    RegistrationCorrectness { registration: RegistrationId, condition: CorrectnessKind },
    AlgorithmContract { algorithm: SymbolId, site: AlgorithmSiteId, kind: AlgorithmVcKind },
    Termination { algorithm: SymbolId, call_or_loop: TerminationSiteId },
}
```

### VC IR

```rust
struct VcIr {
    id: VcId,
    seed: ObligationSeedId,
    source_range: SourceRange,
    kind: VcKind,
    local_context: LocalContext,
    premises: Vec<PremiseRef>,
    goal: CoreFormulaId,
    proof_hint: Option<ProofHint>,
}

enum VcKind {
    TheoremStep,
    DefinitionCorrectness(CorrectnessKind),
    RegistrationCorrectness(CorrectnessKind),
    AlgorithmContract(AlgorithmVcKind),
    Termination,
    TypeOrSethood,
}
```

### VC Status

```rust
enum VcStatus {
    PolicyOpen(PolicyRef),
    Discharged(DischargeEvidence),
    NeedsAtp,
    SkippedDueToInvalidInput,
    AssumedByPolicy(PolicyRef),
}

struct DischargeEvidence {
    method: DischargeMethod,
    premises: Vec<PremiseRef>,
    trace: Option<ResolutionTraceRef>,
}
```

`NeedsAtp` は phase 13 input state である。`PolicyOpen` は active policy のもとで ATP に送られない intentionally open obligation を記録する。`AssumedByPolicy` は accepted assumption を記録し、proof evidence ではない。`VcStatus` は artifact emission までは internal である。Kernel-accepted proof status は後続で割り当てられる。

## 診断

VC diagnostics は ATP translation 前に生成される。

| Diagnostic Class | Example |
|---|---|
| CFG construction | unreachable statement、use before assignment、illegal break |
| ghost/runtime separation | ghost value flows into runtime expression |
| obligation generation | missing loop invariant、unsupported statement form |
| termination | decreasing measure missing、not Nat-valued、not decreasing |
| context construction | cited premise unavailable、local fact escapes scope |
| pre-ATP discharge | computation limit exceeded、deterministic reduction loop |
| policy | open or assumed item used where clean proof is required |

Diagnostics は次を含むべきである。

- obligation seed の source span
- 利用可能なら generated `VcId`
- debug views 用の local context summary
- related loop、call、theorem、registration source spans
- deterministic discharge explanation or failure reason

## 増分処理と再現性

VC cache keys は次を含む。

- `CoreIr` hash
- algorithm obligations 用の `ControlFlowIr` hash
- obligation seed hashes
- relevant type fact and resolution trace hashes
- verifier policy and computation limit settings
- VC schema version

変更は次を invalidate する。

- changed obligation seeds から生成された VCs
- CFG、contracts、termination metadata が変わった algorithm VCs
- changed `NeedsAtp` VCs に対応する downstream ATP problems
- changed VC status or diagnostics を記録する artifacts

Ordering は deterministic でなければならない。

- obligation seeds は module、item、source range、origin kind で sort する
- generated `VcId`s は sorted seed list and per-seed expansion index から割り当てる
- premises は canonical `PremiseRef` で sort する
- discharged、policy-status、`NeedsAtp` VC lists は VC order を保持する
- 同じ source range の diagnostics は VC kind and diagnostic code で order する

## 検討した代替案

### Generate VCs Directly from ResolvedTypedAst

Source-shaped syntax から VCs を生成すると surface details は保持しやすいが、elaboration logic が重複し、proof contexts が notation に依存する。採用案では `CoreIr` and source maps から VCs を生成する。

### Combine ControlFlowIr and CoreIr

CFG data を `CoreIr` に直接入れると algorithm verification は呼び出しやすくなるが、non-algorithm consumers が program-analysis details を扱わなければならなくなる。採用案では algorithm-specific flow を `ControlFlowIr` に分離する。

### Send All VCs to ATP

すべての obligations を ATP に委譲すると phase 12 は単純になるが、遅く、非決定的になり、type/cluster explanations が隠れる。採用案では deterministic obligations を ATP translation 前に discharge する。

### Do Pre-ATP Discharge During VC Generation

Generation and discharge を interleave すると memory は節約できるが、diagnostics and scheduling の再現性が下がる。採用案では canonical `VcIr` をまず生成し、その後 separate step として deterministic discharge を行う。

## 採用方針

Mizar Evo は three-stage verification preparation boundary を採用する。

Phase 10 は algorithm bodies の `ControlFlowIr` を構築する。Phase 11 は `CoreIr` and `ControlFlowIr` からの obligation seeds を deterministic `VcId`s を持つ canonical `VcIr` に変換する。Phase 12 は deterministic Mizar-side machinery で解ける obligations を discharge し、configured policy statuses を割り当て、`NeedsAtp` VCs だけを ATP translation に渡す。

## 制約と前提

- VC generation は elaboration が `CoreIr` を生成した後にだけ開始する。
- Phase 10 は VCs が生成される前に invalid algorithm control-flow を reject してよい。
- Phase 11 だけが `VcId`s を割り当てる。
- Phase 12 は VCs を discharge したり policy statuses を割り当てたりしてよいが、ATP-bound VCs を erase したり context を弱めたりしてはならない。
- ATP translation は `NeedsAtp` status を持つ canonical `VcIr` だけを受け取る。
- Pre-ATP discharge は identical source、dependencies、toolchain、policy、computation limits に対して deterministic でなければならない。
