# アーキテクチャ: cluster と registration のトレース形式

> 正本は英語です。英語版: [../en/17.cluster_trace_format.md](../en/17.cluster_trace_format.md)。

## 目的

この文書は、cluster 展開とリダクション適用のための、正準的で再生可能な `ResolutionTrace` アーティファクトスキーマを定義します。

cluster 探索とリダクション規則の選択はカーネルの外側で行います。カーネルとアーティファクトの消費者が受け取るのは、どの cluster またはリダクションのステップが使われたかを説明する明示的なトレースだけです。

## 設計判断

### cluster ステップ

各 cluster 展開ステップは、次の形で表します。

```text
ClusterStep
  source_type
  applied_cluster
  generated_attribute
  generated_type
  dependency
  source_range
```

`dependency` は、その展開を正当化する定理・定義・registration・mode・属性、またはそれより前の cluster ステップを記録します。

### リダクションステップ

各リダクション展開ステップは、次の形で表します。

```text
ReductionStep
  applied_reduction
  rule_fqn
  enclosing_term_before
  redex_path
  source_redex
  target_term
  substitution
  discharged_guards
  rule_view
  selection_key
  source_range
```

kernel 再生用のフィールドは `applied_reduction`、`rule_fqn`、`source_redex`、`target_term`、`substitution`、`discharged_guards` です。これらは、すでに受理された `reducibility` 証明に照らして、局所的な `LHS -> RHS` インスタンスを検証し、自動正規化で用いられた正確なラベル付き規則を報告するのに十分です。戦略監査用のフィールドは `enclosing_term_before`、`redex_path`、`rule_view`、`selection_key` です。これらにより、決定的な artifact validator は、そのステップが §17.6.4 の規則順序で選ばれた最左最内 redex であったことを再計算できます。最小 kernel は規則探索を行いません。

### cluster 連鎖トレース

推移的な cluster 展開は、明示的な順序付き連鎖として保存します。

```text
A -> B
B -> C
C -> D
```

トレースはすべての中間ステップを保持しなければなりません。アーティファクトは、元の再生ステップをコンテンツアドレスによる参照として保存しない限り、連鎖を `A -> D` に圧縮してはなりません。

### 決定的な走査

cluster の走査順は、次で固定します。

1. ソース型の正準 ID
2. cluster の由来モジュールパス
3. 宣言のソース順
4. 生成される属性の正準 ID
5. registration のフィンガープリント

ワーカーの完了順・ハッシュマップの反復順・キャッシュの挿入順が、選択されるトレースに影響してはなりません。

リダクション正規化の順序は §17.6.4 で固定します。リダクションステップは、有効な rule-view fingerprint と選択規則 key を記録し、決定的な validator が登録順・import 順・キャッシュ挿入順に依存せず戦略を監査できるようにします。

### cluster ループ検出

cluster のループ検出は決定的です。

- アクティブな展開スタック内の循環は `cluster_loop` として却下する。
- すでに導出された事実の繰り返しは、正準的なフィンガープリントが一致した後にのみ無視する。
- 最大展開深さと最大生成事実数は検証器設定であり、キャッシュキーに含める。
- 上限の超過は上限つきの失敗であり、必要な事実を黙って切り詰めてはならない。

これらの展開深さと生成事実数の上限は、cluster 展開だけに適用します。リダクション適用は cluster 展開深さによって制限されません。停止性は、登録時に検査される簡約順序によって保証されます（§17.6.4）。

### トレースの消費者

トレースの消費者は、次を行ってよいです。

- 診断とアーティファクト検証のために cluster の導出を再生する。
- 診断とアーティファクト検証のために局所的なリダクションインスタンスを再生する。
- 生成された型事実を VC に含める。
- 依存スライスをフィンガープリントする。
- `@show_resolution` の説明を表示する。

トレースの消費者は、トレースに存在しない追加の cluster やリダクションを推論してはなりません。

## インターフェイス定義

```rust
struct ResolutionTrace {
    schema_version: SchemaVersion,
    source_file: SourceFileId,
    steps: Vec<ResolutionTraceStep>,
    derived_facts: Vec<ClusterFact>,
    traversal_profile: ClusterTraversalProfile,
}

enum ResolutionTraceStep {
    Cluster(ClusterStep),
    Reduction(ReductionStep),
}

struct ClusterStep {
    source_type: TypeFingerprint,
    applied_cluster: ClusterFingerprint,
    generated_attribute: AttributeFingerprint,
    generated_type: TypeFingerprint,
    dependency: ClusterDependencyRef,
    source_range: SourceRange,
}

struct ReductionStep {
    applied_reduction: ReductionFingerprint,
    rule_fqn: Fqn,
    enclosing_term_before: TermFingerprint,
    redex_path: RedexPath,
    source_redex: TermFingerprint,
    target_term: TermFingerprint,
    substitution: Vec<PatternBinding>,
    discharged_guards: Vec<GuardEvidenceRef>,
    rule_view: RuleViewFingerprint,
    selection_key: ReductionSelectionKey,
    source_range: SourceRange,
}
```

## 制約と前提

- resolution トレースは意味論的なアーティファクトであり、決定的なハッシュに含める。
- `coherence` または `reducibility` 証明が受理されていない registration は、トレースから不可視である。
- 隠れた推移的展開、暗黙の cluster 挿入、未記録のリダクションステップは禁止する。
- 失敗系テストには、無限連鎖・循環する registration・意図しない型強制・隠れた推移的展開・不正なリダクション置換・不一致なリダクション戦略監査 key を含める。
