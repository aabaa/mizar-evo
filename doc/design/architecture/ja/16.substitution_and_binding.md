## 設計判断

### 変数の分類

実装は、次の変数の分類を区別します。

- 束縛子が導入する束縛変数
- ローカルコンテキスト由来の自由変数
- パターン・規則・evidence template で用いる schematic 変数
- elaboration または証明検査の際に導入される、生成された新鮮な変数

各変数は、安定した内部 ID と、ソース側の表示名を持ちます。等価性は束縛の解決後の安定した ID で判定し、表示名では判定しません。

### 束縛変数と自由変数

自由変数の集合は、正規化された束縛子の構造から計算します。

- 束縛子に入ると、その束縛子の変数は本体の自由変数集合から除かれる。
- シャドーイングは、相異なる安定した ID で表す。
- ソース名だけで捕獲を判定しない。
- 受理に必要な自由変数の条件は、kernel evidence にシリアライズする。

### α 同値

2 つの項は、束縛変数の一貫したリネームだけが異なる場合に、α 同値です。

チェッカは、その場限りのテキスト比較ではなく、正準的な束縛子の正規化によって α 同値を判定します。

### 正準的な束縛子の正規化

束縛子の正規化は、字句上の束縛子の順序に従って、束縛子ローカルな正準 ID を割り当てます。

```text
for x being set holds
  ex y being set st P[x, y]
```

これは `x` や `y` という名前ではなく、束縛子の位置によって正規化されます。束縛子の名前だけが異なる同値な論理式は、同一の正規化エンコードとハッシュを生成します。

### 捕獲回避

置換によって、置換される項の自由変数が、対象コンテキストの束縛子に捕獲されてはなりません。

衝突が起こる場合、実装は決定的な新鮮性の戦略により束縛子を α リネームしてから、置換を適用します。決定的なリネームが、記録された副条件を満たせない場合、その置換は却下します。

### 定義時 closure

ローカル abbreviation (`set`、`deffunc`、`defpred`) は、記録された closure を持つ
置換であり、テキストマクロではありません。そのような abbreviation が定義されると、
resolver は定義する項または論理式に現れるすべての自由変数の安定 ID を記録します。
後続の使用箇所では、形式束縛子に実引数を置換しつつ、それらの捕捉済み ID は固定したままにします。

帰結:

- 定義後に捕捉済みの表示名を shadowing しても、abbreviation の意味は変わりません。
- 捕捉を避けるために必要なら、使用箇所の周囲で alpha rename してよいし、そうするべきです。
- 置換ステップが有効なのは、実引数が定義位置で記録された形式型 guard を満たす場合だけです。
- evidence と legacy trace 形式は、元の resolver の外で abbreviation 展開を再生する場合、
  捕捉された自由変数集合を serialize しなければなりません。

たとえば、`defpred P(n be Nat) means n < m;` が、`m` が特定のローカル束縛を
参照している時点で定義された場合、後続の `for m being Nat holds P(m)` は
2 つの異なる variable ID に展開されます。すなわち、実引数としての量化された `m` と、
本体内に捕捉された外側の `m` です。

### リネームの戦略

新鮮な ID は、次から生成します。

- 所有するソースファイルまたは evidence ID
- 正規化された束縛子のパス
- 元の変数の役割
- 正規化された項に閉じた決定的なカウンタ

生成される表示名は診断専用です。ハッシュや証明の受理に影響してはなりません。

Kernel が replay できる evidence では、evidence-visible な freshness check は
normalized evidence だけで完全に決定される。Binder path は source term 内の original
binder と、target term 内の同じ normalized binder position を選択する。Target binder は
generated-fresh binder でなければならず、その variable id は witness の `fresh` id と
一致しなければならない。Avoided-variable set は、source binder body の free variables
から original bound variable を除いた集合と、その binder の下へ挿入される actual term
の free variables の和として再計算する。Candidate stream は、再計算した avoided set を
除いた evidence variable manifest の stable id 昇順である。Deterministic counter は、
その candidate stream における generated id の zero-based index である。

Producer phase が用いる source-file または evidence domain separation は、stable
variable manifest と witness envelope にすでに反映されていなければならない。Kernel
checker は producer state を参照して fresh id を修復または再発見してはならない。
Normalized evidence が欠落、stale、または上記 candidate stream と不整合であれば、その
置換は拒否する。

Free-variable replay では、recorded capture set は self-attesting ではない。Kernel は
recorded target path 上の normalized target binder stack から capture set を再計算する。
Free-variable side condition が valid なのは、recorded set が再計算した set と完全に一致し、
かつ要求された free-variable predicate が normalized target subtree 上で成り立つ場合だけである。

### 置換の再生

置換の再生は純粋関数です。

```text
(normalized_source, normalized_target, binder_context, side_conditions)
  -> accepted normalized result
  -> or invalid_substitution
```

再生は、アクティブな構文解析器の状態・オーバーロードテーブル・cluster テーブル・ATP バックエンドの出力を参照してはなりません。

## 制約と前提

- 束縛子の正規化は、プラットフォームやワーカー数に依存せず安定でなければならない。
- ソースの表示名は診断のために保持するが、解決後の意味論的な等価性からは除外する。
- 不正な置換の根拠はカーネルによる棄却であり、ヒューリスティックな修復の機会ではない。
- ファジングと失敗系テストには、変数の捕獲・束縛子の衝突・不正な置換・α 変換の失敗を含める。
