## 目的

この文書は evo2 の binder、variable、alpha-equivalence、substitution semantics を固定する。

これは name resolution、type checking、elaboration、VC generation、certificate checking、diagnostics に適用される。目的は binding behavior を deterministic かつ kernel-checkable にすることである。

## Design Decisions

### Variable Classes

implementation は次の variable classes を区別する。

- binders が導入する bound variables
- local context 由来の free variables
- patterns、rules、certificate templates で使う schematic variables
- elaboration または proof checking 中に導入される generated fresh variables

各 variable は stable internal id と source-facing display name を持つ。equality は binding resolution 後の stable id で判定し、display name では判定しない。

### Bound and Free Variables

free-variable sets は normalized binder structure から計算する。

- binder に入ると、その binder variable は body の free-variable set から除かれる。
- shadowing は distinct stable ids で表す。
- source name だけでは capture を判定しない。
- acceptance に必要な free-variable conditions は certificate に serialize する。

### Alpha-Equivalence

2 つの term は、bound variables の一貫した renaming だけが異なる場合に alpha-equivalent である。

checker は ad hoc な textual comparison ではなく、canonical binder normalization により alpha-equivalence を判定する。

### Canonical Binder Normalization

binder normalization は lexical binder order に従って binder-local canonical ids を割り当てる。

```text
for x being set holds
  ex y being set st P[x, y]
```

これは `x` や `y` という名前ではなく binder position によって normalize される。binder 名だけが異なる equivalent formula は同一の normalized encoding と hash を生成する。

### Capture Avoidance

substitution により、substituted term の free variable が target context の binder に捕捉されてはならない。

collision が起こる場合、implementation は deterministic freshness strategy により binder を alpha-rename してから substitution を適用する。deterministic renaming が記録された side conditions を満たせない場合、その substitution は reject する。

### Renaming Strategy

fresh ids は次から生成する。

- owning source file または certificate id
- normalized binder path
- original variable role
- normalized term に scoped な deterministic counter

generated display names は diagnostic-only である。hash や proof acceptance に影響してはならない。

### Substitution Replay

substitution replay は pure function である。

```text
(normalized_source, normalized_target, binder_context, side_conditions)
  -> accepted normalized result
  -> or invalid_substitution
```

replay は active parser state、overload tables、cluster tables、ATP backend output を参照してはならない。

## Interface Definitions

```rust
struct BinderContext {
    binders: Vec<BinderFrame>,
    free_variables: Vec<VariableId>,
}

struct BinderFrame {
    binder_id: BinderId,
    canonical_index: u32,
    source_name: Option<String>,
    variable_id: VariableId,
}

struct FreeVariableConstraint {
    variable: VariableId,
    must_remain_free_in: TermPath,
}
```

## Constraints and Assumptions

- binder normalization は platform や worker count に依存せず stable でなければならない。
- source display names は diagnostics のため保持するが、resolution 後の semantic equality からは除外する。
- malformed substitution evidence は kernel rejection であり、heuristic repair の機会ではない。
- fuzzing と fail tests は variable capture、binder collision、malformed substitution、alpha-conversion failure を含める。
