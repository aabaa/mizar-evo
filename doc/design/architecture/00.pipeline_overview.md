# Architecture: Pipeline Overview

## Purpose

Mizar Evo の言語処理系が、ソースコードを読み込んでから検証済み成果物を出力するまでの**全体処理パイプライン**を定義する。

この文書は、個別のアーキテクチャ文書の前提となる「背骨」を提供する。各フェーズの入力、出力、責務、失敗モード、後続フェーズへの受け渡し境界を明確にし、型検査、登録解決、ATP 連携、診断、インクリメンタルビルドが互いに矛盾しないようにする。

## Design Decisions

### Pipeline as the Primary Architectural Boundary

処理系全体は、次の順序付きフェーズで構成する。

| Phase | Input | Output | Main Responsibility | Failure Mode |
|---|---|---|---|---|
| 0. Package / Workspace Resolution | `mizar.pkg`, `mizar.workspace`, `mizar.lock` | `BuildPlan` | package graph, dependency versions, edition, verifier settings の決定 | dependency / manifest errors |
| 1. Source Loading / Preprocessing | `.miz` source files, `BuildPlan` | `SourceUnit`, `PreprocessedSource` | source validation, comment/doc-comment separation, import pre-scan, annotation collection | lexical precondition errors |
| 2. Lexing | `PreprocessedSource`, active lexicon seed | token stream | reserved words, special symbols, user symbols, numerals の tokenization | lexical errors |
| 3. Parsing | token stream | surface AST | declarations, statements, terms, formulas, proofs, algorithms の構文木化 | syntax errors |
| 4. Module / Name Resolution | surface AST, import graph, symbol indexes | resolved AST | namespace, import/export, labels, qualified names, selector disambiguation | unresolved / ambiguous names |
| 5. Signature Collection | resolved AST | symbol tables, signature environment | struct/mode/attr/pred/func/theorem/registration/template の登録 | duplicate / illegal declarations |
| 6. Type Checking / Type Inference | resolved AST, signature environment | typed AST, type facts, initial obligations | soft type checking, coercion, sethood, narrowing checks | type errors |
| 7. Cluster / Registration Resolution | typed AST, imported registration indexes | enriched typed AST, cluster facts, resolution trace | attribute propagation, registration firing, replayable derivation recording | inconsistent / missing registrations |
| 8. Overload Resolution | enriched typed AST, candidate sets, subsumption DAG | fully resolved typed AST | functor/predicate/mode overload selection, `qua` insertion | overload errors |
| 9. Elaboration / Desugaring | fully resolved typed AST | core IR | surface syntax elimination, template instantiation, definitional expansion boundaries | internal elaboration errors |
| 10. Algorithm Verification Preparation | core IR | control-flow IR, algorithm obligations | contracts, invariants, termination measures, ghost rules の整理 | algorithm verification errors |
| 11. VC Generation | core IR, control-flow IR | verification conditions | theorem/proof/assert/contract/registration 由来の proof obligations 生成 | logical well-formedness errors |
| 12. Pre-ATP Discharge | verification conditions, local facts | discharged VCs, open VCs | trivial facts, computation, definitional equality, type-derived facts の処理 | computation limit / unsupported forms |
| 13. ATP Translation / Dispatch | open VCs, cited premises, proof hints | ATP results, proof certificates | TPTP/SMT-LIB 生成、solver portfolio 実行 | timeout / unknown / backend errors |
| 14. Kernel Certificate Check | proof certificates, kernel context | verified proof status | de Bruijn criterion に基づく独立検証 | certificate rejection |
| 15. Artifact Emit / Cache Update | verified AST/IR, metadata, logs | `*.mizir.json`, logs, witnesses, cache | reproducible artifacts, incremental cache, LSP metadata | artifact I/O errors |
| 16. Documentation / Extraction | verified artifacts, doc comments | rendered docs, extracted code | documentation generation, ghost erasure, executable extraction | documentation / extraction errors |

### Phase Ordering

原則として、言語の意味を確定するフェーズは ATP 呼び出しより前に完了する。

- import / namespace / active lexicon は、lexing と parsing の前後で協調して解決する。
- cluster resolution と overload resolution は type-check phase の一部として扱うが、アーキテクチャ上は独立フェーズとして設計する。
- ATP は型推論、登録解決、overload resolution を担当しない。
- ATP へ渡す問題は、型・名前・overload・template が解決済みの core IR / VC IR から生成する。
- kernel certificate check は ATP 結果を信頼せず、検証可能な証明証明書または限定された witness のみを受け入れる。

### IR Layering

各フェーズ間の主な中間表現は以下とする。

| IR | Produced By | Consumed By | Role |
|---|---|---|---|
| `SourceUnit` | Source Loading | Preprocessing, diagnostics | file path, package, source text, line map |
| `TokenStream` | Lexing | Parsing | syntax-level token sequence |
| `SurfaceAst` | Parsing | Name Resolution | source syntax をほぼ保った AST |
| `ResolvedAst` | Name Resolution | Signature Collection, Type Checking | names, labels, imports, namespace references が解決された AST |
| `TypedAst` | Type Checking | Registration / Overload / Metadata | inferred types, coercion candidates, type facts を持つ AST |
| `ResolvedTypedAst` | Cluster / Overload Resolution | Elaboration, LSP metadata | attributes, overload winners, inserted `qua` が確定した AST |
| `CoreIr` | Elaboration | VC Generation, Kernel | surface sugar を除去した論理寄り表現 |
| `ControlFlowIr` | Algorithm Verification Preparation | VC Generation, Extraction | algorithm body の verification / extraction 用表現 |
| `VcIr` | VC Generation | Pre-ATP, ATP Translation | proof obligation と local context |
| `AtpProblem` | ATP Translation | ATP Backends | TPTP / SMT-LIB へ変換可能な prover input |
| `ProofCertificate` | ATP Backends | Kernel Certificate Check | kernel が検証する証明証明書 |
| `VerifiedArtifact` | Kernel / Artifact Emit | LSP, docs, downstream packages | build output と外部 tool integration の単位 |

詳細な IR 定義は別文書 `ir_layers.md` で定義する。

### Diagnostics as Phase Outputs

diagnostics は単なる副作用ではなく、各フェーズの構造化出力として扱う。

- diagnostic code は [doc/spec/22.error_handling_and_diagnostics.md](../../spec/22.error_handling_and_diagnostics.md) の範囲に従う。
- source span は `SourceUnit` の line map に基づいて生成する。
- name/type/proof diagnostics は、可能な限り secondary span と machine-readable detail を持つ。
- `@show_type`, `@show_resolution`, `@show_thesis`, `@eval` は informational diagnostics と metadata artifact の両方に反映する。

### Incrementality and Reproducibility

各フェーズは、入力 hash と出力 artifact を明示できる単位として設計する。

- source hash, dependency hash, toolchain version, verifier settings を cache key に含める。
- import dependency graph に基づき、変更ファイルと reverse dependency cone のみを再処理する。
- cluster-db と registration index は、変更された source unit の contribution を差し替える。
- artifact output は canonical ordering を用いて reproducible にする。
- ATP backend の非決定性は、seed、timeout、certificate metadata、`externally_attested` status として記録する。

## Alternatives Considered

### Alternative 1: Frontend / Middle / Backend Only

処理系を大きく frontend, middle-end, backend の三層に分ける案。

この分類は実装チームの整理には便利だが、Mizar Evo では cluster resolution, overload resolution, ATP dispatch, certificate checking, artifact emission がそれぞれ異なる信頼境界と cache 境界を持つ。そのため三層分類だけでは、どのフェーズがどの情報を確定するかが曖昧になる。

### Alternative 2: Existing Build Lifecycle Table Only

[doc/spec/23.package_management_and_build_system.md](../../spec/23.package_management_and_build_system.md) の build lifecycle をそのままアーキテクチャ境界とする案。

仕様書の lifecycle はユーザー視点の説明として適切だが、実装設計には粒度が粗い。特に type-check phase 内に含まれる cluster resolution, overload resolution, `qua` insertion, metadata output を分離して考える必要がある。

### Alternative 3: Fully Demand-Driven Pipeline

LSP / IDE を主用途とし、必要な情報を要求された時点で部分的に計算する案。

interactive performance には有利だが、初期設計としては correctness boundary と reproducible build を定義しづらい。まず batch build pipeline を基準に設計し、その後に各フェーズを incremental / demand-driven に最適化する。

## Adopted Approach

明示的な多段 pipeline を採用する。

根拠:

- 言語仕様が syntax, type, registration, overload, proof, algorithm, ATP, artifact まで広範にまたがる。
- ATP に渡す前に Mizar 側で確定すべき情報が多い。
- de Bruijn criterion を満たすには、ATP dispatch と kernel certificate check の境界を明確にする必要がある。
- LSP / AI / documentation が利用する metadata を、どのフェーズで生成するか追跡可能にしたい。
- incremental build の cache key と invalidation boundary を定義しやすい。

## Interface Definitions

### BuildPlan

`BuildPlan` は package / workspace resolution の出力であり、以降の全フェーズの共通入力である。

```rust
struct BuildPlan {
    workspace_root: PathBuf,
    packages: Vec<PackagePlan>,
    dependency_graph: DependencyGraph,
    lockfile: Lockfile,
    toolchain: ToolchainInfo,
    verifier_config: VerifierConfig,
    build_config: BuildConfig,
}
```

### PhaseResult

各フェーズは、成功時にも diagnostics を返す。warnings / infos は build を継続させる。

```rust
struct PhaseResult<T> {
    output: Option<T>,
    diagnostics: Vec<Diagnostic>,
    artifacts: Vec<IntermediateArtifact>,
}
```

`output = None` は、そのフェーズで後続処理を継続できない error が発生したことを表す。

### VerificationCondition

VC は ATP だけでなく、pre-ATP discharge と diagnostics にも使われる。

```rust
struct VerificationCondition {
    id: VcId,
    source_range: SourceRange,
    kind: VcKind,
    local_context: LocalContext,
    premises: Vec<PremiseRef>,
    goal: Formula,
    proof_hint: Option<ProofHint>,
}
```

### VerifiedArtifact

`VerifiedArtifact` は、downstream package、LSP、documentation generator、AI tooling に渡す安定境界である。

```rust
struct VerifiedArtifact {
    schema_version: String,
    source_file: PathBuf,
    source_hash: Hash,
    exports: Vec<ExportedItem>,
    expressions: Vec<ExpressionMetadata>,
    obligations: Vec<ObligationMetadata>,
    proof_witnesses: Vec<ProofWitnessRef>,
    diagnostics: Vec<Diagnostic>,
}
```

## Constraints and Assumptions

- `.miz` source file is the module unit; package and namespace rules follow Ch.23.
- Code regions are ASCII; comments and annotations may contain Unicode.
- Import resolution affects the active lexicon and therefore must be available before final token classification of user-defined symbols.
- Type checking, cluster resolution, and overload resolution are Mizar-side responsibilities.
- ATP backends receive only well-typed, name-resolved, elaborated obligations.
- ATP success is not accepted until kernel certificate checking succeeds, except where policy explicitly permits `externally_attested`.
- Artifacts must be reproducible given identical source, lockfile, toolchain, verifier settings, and accepted certificates.
- Diagnostics are stable API surface for IDE / AI tooling and must not depend on unstable message wording alone.
