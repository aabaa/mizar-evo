# Architecture: AI Agent Interface and Context Protocol

> Canonical language: English. Japanese companion: [../ja/21.ai_agent_interface.md](../ja/21.ai_agent_interface.md).

## Purpose

This document defines the **tooling contract** that lets an AI agent operate on a Mizar Evo project safely: which edits an agent may propose, what authorization is required to perform them, and how the agent reads context, proposes patches, and consumes verifier results.

It is deliberately **not** a language-specification chapter. The Mizar Evo language, its surface syntax, FQNs, and kernel soundness are defined elsewhere and are unchanged by anything here. This document describes the layer *around* the verifier that turns "AI can read Mizar Evo artifacts" (passive AI friendliness) into "AI can search, explain, edit, and re-verify a Mizar Evo project under explicit safety controls" (active AI operability).

The originating rationale and the full target outline are recorded in [doc/idea/ai-readiness.md](../../../idea/ai-readiness.md). This document normatively specifies only the two policy items that do not depend on lower-layer wire schemas or on a mature verifier; everything else is framed as a non-normative roadmap so that the wire formats can be frozen later, once the artifacts they wrap are stable.

## Scope and Normative Status

This document mixes normative policy with a non-normative roadmap. The distinction is deliberate and is stated per section.

| Section | Status |
|---|---|
| Trust Boundary and Safe Edit Classes | **Normative** |
| Authorization Scopes | **Normative for the scope vocabulary and its mapping to edit classes**; transport/OAuth binding is non-normative |
| Design Principles | Non-normative framing |
| Framework and Roadmap | Non-normative; each item names its blocking dependency |

The two normative items — the Green/Yellow/Red edit classes and the authorization-scope vocabulary — are policy. They are meaningful even before a patch-verification API exists, because they constrain what an agent is *allowed* to attempt regardless of how the attempt is wired. They are the natural extension of [08.reasoning_boundary.md](./08.reasoning_boundary.md) and of the `require_kernel_certificates` release policy into the agent-facing surface.

Everything else — the context-budget model, resource URI scheme, theorem/symbol indexes, obligation context extraction, patch wire schema, patch verification API, proof-repair state machine, MCP binding, dataset export, and benchmarks — is described only as a framework. **No wire schema is frozen here.** Wire formats are frozen only after the lower-layer artifacts they wrap (`*.mizir.json`, kernel certificates, cross-edit obligation anchors, proof witnesses) are themselves stable.

## Design Principles

These principles are non-normative framing for the normative sections that follow.

- **Correctness stays with the verifier, ATP, and kernel.** The agent searches, explains, plans, and edits; it never asserts proof validity on its own. This is the core of [08.reasoning_boundary.md](./08.reasoning_boundary.md).
- **Edits are small and verifiable, not free-form rewrites.** The unit of agent action is a single bounded edit (add a `by` citation, insert a `qua`, add an `assert`, add an `import`), each of which the verifier independently accepts or rejects.
- **Read minimal context, not the whole library.** An agent is given a bounded, per-obligation projection, consistent with the resident-set memory model in [README.md](./README.md).
- **Dry-run before apply.** A patch is verified (dry-run) before it is ever applied to source.
- **Authority is explicit and least-privilege.** Capabilities that can move the trust boundary are never granted to an ordinary agent.

## Trust Boundary and Safe Edit Classes

**Status: Normative.**

Every edit an AI agent may propose is classified into one of three safety classes. The class governs *policy* (whether the edit may be auto-proposed, whether human review is required, whether an agent may perform it at all). It does **not** replace verification: a Green or Yellow edit is still only accepted when the verifier accepts it, and acceptance still follows the kernel discipline of [08.reasoning_boundary.md](./08.reasoning_boundary.md).

### Green — may be auto-proposed

Edits that are unlikely to change meaning and whose effect the verifier fully checks:

- add a lemma to a `by` clause;
- remove an unused citation from a `by` clause;
- insert a `qua` coercion to resolve an overload;
- insert an informational annotation (`@show_type`, `@show_resolution`, `@show_thesis`, `@proof_hint`);
- add a comment or doc comment;
- apply `mizar refine` to tighten citations.

### Yellow — may be proposed, but verification success still warrants human review

Edits that the verifier can check, but that change proof structure, search space, or library design:

- add an `assert` to split a goal into steps;
- add a local lemma;
- add an `import`;
- add a registration;
- add a cluster;
- add an algorithm invariant.

These are logically sound when they verify, but they have design consequences (visible-symbol growth, search-space changes, library shape), so a human should confirm them.

### Red — forbidden to an AI agent acting alone

Edits that move, weaken, or bypass the trust boundary:

- add an axiom;
- weaken a theorem statement;
- change a definition;
- weaken a `requires` / `ensures` contract;
- edit `proof_status` by hand;
- edit a `proof_witness` directly;
- relax kernel settings;
- set `require_kernel_certificates = false`.

An AI agent may propose anything it likes *outside* the kernel, but it must never weaken what the kernel checks or what the active verifier policy trusts. Red actions are precisely the ones that would let an agent "prove" a goal by changing the goal, the axioms, the definitions, or the acceptance policy. They require the `unsafe-edit` scope (below), which is not granted to ordinary agents, and they fall outside auto-proposal entirely.

The most common agent failure modes — weakening the goal, changing the theorem statement, adding a convenient axiom, or quietly redefining a symbol — are all Red, and the classification exists specifically to make them inexpressible for an ordinary agent.

## Authorization Scopes

**Status: Normative for the vocabulary and its edit-class mapping. Transport and OAuth binding are non-normative and deferred (see Framework and Roadmap).**

Agent capabilities are named by a fixed scope vocabulary. A deployment grants a subset of these scopes to an agent; the scopes bound what the agent may do regardless of transport.

| Scope | Grants | Highest edit class reachable |
|---|---|---|
| `read` | read source, artifacts, theorems, obligations | none (read-only) |
| `explain` | call the explanation API (`explain-type`, `explain-overload`, `explain-failure`, …) | none (read-only) |
| `verify` | run patch verification (dry-run) without applying | none (no source mutation) |
| `write` | apply a verified patch to source | Green, and Yellow with human review |
| `refine` | apply citation refinement / minimization | Green |
| `publish` | publish a package | none directly; gated action |
| `unsafe-edit` | edit axioms, definitions, theorem statements, contracts | Red |

Rules:

- `read`, `explain`, and `verify` are read-only with respect to source and never mutate project state.
- `write` may apply Green edits and may apply Yellow edits, but Yellow edits remain subject to human review per the edit-class policy above; `write` never reaches Red.
- `unsafe-edit` is the only scope that reaches the Red class. **It is not granted to ordinary AI agents.**
- `publish` requires human approval and is never auto-exercised by an agent.
- A default "AI-safe" agent profile grants at most `read, explain, verify, refine` and, where source mutation is intended, `write`; it denies `unsafe-edit` and `publish`.

This vocabulary is intended to map onto a remote MCP server's OAuth scopes and onto a local `--safe-mode` allow/deny list, but those bindings are deferred; only the names above and their relationship to the edit classes are normative here.

## Framework and Roadmap (Non-normative)

The following components complete the AI agent interface but are **not** specified normatively in this revision. Each is sketched only enough to fix vocabulary and to record the dependency that must be resolved before its wire format can be frozen. The overall target outline is in [doc/idea/ai-readiness.md §16](../../../idea/ai-readiness.md).

A guiding implementation reality applies to all of them: the verifier and its artifact generation are still maturing, so a patch dry-run or a theorem index cannot have substance until the verifier and the artifacts they wrap exist. This is why the wire-schema, MCP-binding, and benchmark items are sequenced last — not only out of conceptual caution but because the lower-layer artifacts they wrap are not yet running.

- **Context Budget Model.** An agent receives a token-bounded projection rather than the whole library. Requests carry a budget; responses report an estimate. (Extends the bounded local-subgraph extraction already in `doc/spec/en/23` §23.7.5.)
- **AI Resource URI Scheme.** Read-only context is addressable by URI (e.g. `mizar://obligation/...`, `mizar://theorem/...`, `mizar://diagnostic/...`). Maps to MCP Resources.
- **Theorem and Symbol Index Artifacts.** A search-oriented `theorem-index` / `symbol-index`. **Dependency:** the rich fields (`head_symbols`, normalized/FOL statements, embedding text) are *not* present in `*.mizir.json` today; the thin first version is built only from existing fields (FQN, `source_range`, `used_axioms`), and the extraction path for the rest is a separate task.
- **Obligation Context Extraction.** `mizar context obligation` bundles the per-obligation goal, hypotheses, candidate facts, diagnostics, and `allowed_patch_kinds`. **Dependency:** this is mostly a surface over existing bounded extraction plus a token budget; the `allowed_patch_kinds` field is where the edit classes above become a per-obligation constraint.
- **Patch Proposal Format.** A JSON patch describing one bounded edit against a target obligation and source hash. **Dependency:** requires a **cross-edit-stable obligation anchor**. Today `VcId` numbering is deterministic within a build but index-based, so it shifts when obligations are inserted or removed; the stable anchor must be specified before any patch wire schema is frozen.
- **Patch Verification API.** The dry-run that returns accepted/rejected with changed obligations and new diagnostics. Reachable only with the `verify` scope. Frozen after the obligation anchor and `*.mizir.json` are stable.
- **Proof Repair State Machine.** `OPEN_OBLIGATION → CANDIDATE_FACTS → PATCH_DRY_RUN → {ACCEPTED, REJECTED} → KERNEL_VERIFIED → REFINED → DONE`, with per-state allowed/forbidden actions aligned to the edit classes.
- **MCP Binding.** The mapping of the above onto MCP Resources (read-only context), Tools (search/explain/verify/apply/refine), and Prompts (human-selected templates). **Deferred.**
- **Authorization Binding (OAuth / `--safe-mode`).** The transport realization of the scope vocabulary: OAuth 2.1 protected-resource scopes for remote servers, and an explicit allow/deny `--safe-mode` for local stdio. **Deferred**; only the vocabulary is normative now.
- **Proof-Witness-Derived Data.** Per-axiom instantiation traces feeding agent training data. **Deferred:** the proof-witness format is still tentative (`doc/spec/en/23` §23.5, and [15.kernel_certificate_format.md](./15.kernel_certificate_format.md)). Per-axiom instantiation scoping is already established, so an axiom's variable scope is independent; a `binder_id`-based representation of that scoping is a *future direction* consistent with [16.substitution_and_binding.md](./16.substitution_and_binding.md), not a settled format.
- **Dataset Export Format.** `goal → used_axioms`, `failed_goal → successful_patch`, `overload_error → qua_fix`, etc. **Deferred** until the corpus and witnesses are stable.
- **Benchmarks and Metrics.** Proof-repair, overload-repair, and context-budget benchmarks. **Deferred**; concrete metrics belong to the evaluation strategy ([20.test_strategy.md](./20.test_strategy.md)), not to a normative spec.

## Interface Definitions

Only the normative surface is fixed here: the edit-class table and the scope-to-edit-class mapping.

### Safe edit classes

```text
Green  : auto-proposable, verifier-checked
Yellow : proposable; human review even when verification succeeds
Red    : forbidden to an agent acting alone; requires unsafe-edit scope
```

### Scope → edit-class mapping

```text
read        -> read-only
explain     -> read-only
verify      -> dry-run only (no source mutation)
write       -> Green; Yellow with human review
refine      -> Green (citation refine/minimize)
publish     -> gated, human-approved
unsafe-edit -> Red (not granted to ordinary agents)
```

All other wire-level interfaces (context, patch, verification responses, URIs, indexes) are described in the non-normative roadmap above and are intentionally left unfrozen.

## Constraints and Assumptions

- An AI agent must never weaken what the kernel checks or what the active verifier policy trusts; Red edits are out of reach without `unsafe-edit`.
- The safe edit classes and scope vocabulary are policy and function even without a verifier present; they constrain *intent*, while verification constrains *acceptance*.
- No wire schema is frozen in this revision. Patch, context, index, and witness formats are frozen only after their underlying artifacts stabilize.
- The patch protocol presupposes a cross-edit-stable obligation anchor; that anchor must be specified before the patch wire schema is frozen.
- Agent-facing context is bounded and per-obligation, consistent with the cross-cutting memory model; bulk dumps of the library, cluster-db, or all traces are not provided.
- Concrete budgets and benchmark numbers are out of scope here and belong to the evaluation strategy.
