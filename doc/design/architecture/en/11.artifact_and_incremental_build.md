# Architecture: Artifact and Incremental Build

> Canonical language: English. Japanese companion: [../ja/11.artifact_and_incremental_build.md](../ja/11.artifact_and_incremental_build.md).

## Purpose

This document defines how Mizar Evo emits verified artifacts and updates the incremental build cache.

It refines pipeline phase 15. Earlier phases produce verified semantic data, diagnostics, ATP logs, and kernel-checked proof status. Phase 15 turns those internal results into stable on-disk artifacts, records enough provenance for reproducibility, and updates cache entries used by later builds and IDE workflows.

## Design Decisions

### Artifact Classes

Phase 15 writes three classes of data. They have different compatibility and retention rules.

| Class | Examples | Consumer | Stability |
|---|---|---|---|
| Published artifacts | `*.mizir.json`, exported module summaries, proof witness references | downstream packages, docs, LSP, AI tooling | versioned stable schema |
| Development artifacts | `missing_facts.json`, `*.atp-log`, diagnostic explanations, debug dumps | local developer workflows | versioned where structured, not publication input |
| Build cache entries | phase cache records, `cluster-db/`, import-scoped views, resolution traces | compiler and LSP server | internal schema, invalidated by toolchain/cache version |

Only published artifacts define package compatibility. Development artifacts may be absent in release builds, and cache entries may be deleted at any time without changing package semantics.

### VerifiedArtifact Is a Projection

`*.mizir.json` is emitted from verified phase outputs, but it is not a serialized compiler state.

It includes:

- source identity and hashes;
- public exports and dependency-facing signatures;
- expression metadata needed by IDE and AI tools;
- verification obligations and accepted proof status;
- proof witness references and content hashes;
- diagnostics from the completed build pass.

It excludes:

- raw AST arenas and internal IR layouts;
- full ATP stdout/stderr streams;
- kernel-internal proof state;
- non-deterministic build-session identifiers;
- absolute paths unless explicitly requested for local diagnostics.

The emitter is allowed to compute the artifact from multiple internal inputs, but no downstream tool may depend on those internal inputs directly.

### Publication Boundary

A package can be published only from published artifacts whose proof obligations are accepted under the active release policy.

The publication check rejects:

- any obligation with `proof_status = "open"`;
- externally attested proofs when the kernel-evidence release policy is active;
- artifacts whose schema version is unsupported by the publishing tool;
- artifacts whose recorded source hash no longer matches the source file;
- proof witness references whose content hash does not match the referenced file.

Development artifacts such as `missing_facts.json` are never part of the published package interface.

### Cache Is an Optimization, Not Authority

Incremental cache entries may let a later build skip work, but they do not establish correctness by themselves. A cached proof result can be reused only when the cache key covers every input that could change the accepted result.

Cache keys include:

- source hash and normalized source path;
- package id, module path, language edition, and lockfile hash;
- toolchain version and relevant phase schema versions;
- verifier configuration, including kernel-evidence policy and computation limits;
- dependency artifact hashes and imported symbol summaries;
- registration and cluster index hashes visible to the file;
- VC, ATP problem, backend profile, and kernel evidence hashes where applicable.

If any key component is missing or produced by an unsupported toolchain/cache schema, the entry is treated as a miss.

Per-obligation proof reuse is additionally governed by [22.incremental_verification_contract.md](./22.incremental_verification_contract.md): `ObligationAnchor` is only the candidate match, and reuse requires matching canonical VC fingerprint, local context fingerprint, dependency slice fingerprints, verifier policy, and proof witness or deterministic discharge hash.

### Deterministic Artifact Output

Artifacts must be reproducible for identical source, lockfile, toolchain, verifier settings, and accepted kernel evidence.

Phase 15 enforces this by:

- sorting maps and sets by canonical keys before serialization;
- assigning artifact IDs from deterministic source ranges and stable per-file ordering;
- storing user-visible timestamps such as `verified_at` only as local artifact metadata excluded from canonical content hashes and publication equivalence checks;
- recording relative paths from the workspace or package root;
- hashing referenced witness and log files by content;
- serializing JSON with a canonical formatter.

ATP runtime variability is recorded as provenance, not allowed to reorder or rename stable artifact entries.

### Atomic Writes

Phase 15 writes artifacts with an atomic replace protocol.

1. Serialize the artifact to a temporary file in the target directory.
2. Flush file contents.
3. Rename the temporary file over the previous artifact.
4. Update the per-package artifact manifest.
5. Flush the containing directory when the platform supports it.

Readers either see the previous complete artifact or the new complete artifact. Partially-written JSON files are build errors and must not be silently accepted.

## Incremental Build Model

### Build Graph

The build system maintains a dependency graph with at least these node classes.

| Node | Invalidated By |
|---|---|
| `SourceUnit` | source content or source path change |
| `ModuleSummary` | exports, imports, or public declaration signature change |
| `RegistrationIndexContribution` | registration declaration or accepted coherence proof change |
| `ResolvedTypedArtifactInput` | source, imports, symbol summaries, type rules, registrations |
| `VcSet` | core IR, control-flow IR, type facts, resolution traces, verifier policy |
| `AtpProblemRecord` | VC, premise set, encoding profile, backend profile |
| `KernelProofRecord` | accepted kernel evidence hash, kernel version, trusted environment, dependency proof refs |
| `VerifiedArtifact` | any published projection input |

Invalidation flows through reverse dependencies. A source edit rechecks the edited file and every file whose imported public interface, registration view, or proof context depends on it. A proof-body-only edit may avoid invalidating downstream modules when exported signatures and accepted theorem statements are unchanged.

### Interface Hashes and Implementation Hashes

Each verified module has two important hashes.

```rust
struct ModuleArtifactHashes {
    interface_hash: Hash,
    implementation_hash: Hash,
}
```

`interface_hash` covers data visible to importers:

- exported definitions, theorem statements, modes, predicates, functors, attributes, structs, and registrations;
- exported algorithm signatures and their `requires` / `ensures` contracts (not algorithm bodies, which importers cannot observe — spec §20.9.1);
- proof acceptance status for exported theorems and activated registrations;
- dependency-facing notation, overload summaries, and coherent-refinement metadata needed for `RefinementJoin`;
- schema versions that affect importer interpretation.

`implementation_hash` covers the full verified artifact for the source file, including local expression metadata, local obligations, diagnostics, algorithm bodies, and proof witness references.

Downstream semantic phases depend primarily on `interface_hash`. LSP, documentation, and local rebuild decisions may also depend on `implementation_hash`.

### Cache Reuse Rules

A cached phase output is reusable only when:

- its cache key exactly matches the requested key;
- every referenced dependency artifact still exists and matches its recorded hash;
- the producing toolchain and schema version are compatible;
- the active verifier policy is at least as strict as the policy recorded by the entry;
- the cached output is not marked `uncacheable`;
- the phase carries complete dependency slices/footprints for the cached output;
- reused proof records still point to accepted proof witnesses or deterministic built-in discharge records.

Policy strictness is monotonic only for recognized policy fields. For example,
an entry produced under the kernel-evidence release policy, historically named
`require_kernel_certificates = true`, may be reused under the same policy, but
an externally attested proof cannot be upgraded to kernel-verified by changing a
policy bit.

An `uncacheable` marker, incomplete dependency slices, unsupported schemas, unknown toolchain compatibility, policy incompatibility, witness hash mismatch, or deterministic discharge hash mismatch force a miss. Cache lookup timing and cache hit/miss timing must not affect proof acceptance.

### Registration and Cluster Cache Updates

The `cluster-db/` cache is updated from verified registration contributions, not directly from parsed declarations. A registration whose coherence proof has not been accepted is not visible to importers and is not part of `interface_hash`.

For changed source files, phase 15 updates:

- registration contribution records for the changed module;
- derived graph and index entries whose origins are the changed module;
- import-scoped visibility views that include the changed origins;
- resolution traces for affected files.

The cache stores origin metadata so stale contributions can be removed when a registration is deleted or renamed.

Deleting or renaming a registration must remove stale origin contributions before any dependent cache hit is accepted. Changes to a registration label, target pattern, guard, declared cluster/reduction contribution, proof status, accepted witness, origin identity, or export visibility invalidate import-scoped registration, cluster, and reduction views that can see the origin.

## Interface Definitions

### Artifact Manifest

Each package artifact directory contains a manifest that indexes emitted artifacts.

```rust
struct ArtifactManifest {
    schema_version: String,
    package_id: PackageId,
    package_version: Version,
    lockfile_hash: Hash,
    toolchain: ToolchainInfo,
    verifier_config_hash: Hash,
    modules: Vec<ModuleArtifactEntry>,
}

struct ModuleArtifactEntry {
    module_path: ModulePath,
    source_file: PathBuf,
    source_hash: Hash,
    artifact_file: PathBuf,
    artifact_hash: Hash,
    interface_hash: Hash,
    implementation_hash: Hash,
    proof_witnesses: Vec<ProofWitnessEntry>,
    diagnostics_hash: Option<Hash>,
}
```

The manifest is the first file downstream tools read. It allows them to locate module artifacts without scanning arbitrary build output.

### Verified Artifact Schema

The architectural shape of `VerifiedArtifact` is:

```rust
struct VerifiedArtifact {
    schema_version: String,
    source_file: PathBuf,
    source_hash: Hash,
    verified_at: Option<Timestamp>,
    module_path: ModulePath,
    interface_hash: Hash,
    implementation_hash: Hash,
    exports: Vec<ExportedItem>,
    expressions: Vec<ExpressionMetadata>,
    obligations: Vec<ObligationMetadata>,
    proof_witnesses: Vec<ProofWitnessRef>,
    diagnostics: Vec<Diagnostic>,
    provenance: BuildProvenance,
}

struct BuildProvenance {
    toolchain: ToolchainInfo,
    language_edition: Edition,
    lockfile_hash: Hash,
    verifier_config_hash: Hash,
    dependency_artifact_hashes: Vec<DependencyArtifactHash>,
    cache_key: Option<String>,
}
```

`verified_at` and `BuildProvenance.cache_key` are local metadata. They are
useful for IDEs, humans inspecting `build/`, and cache debugging, but they are
not included in `interface_hash`, `implementation_hash`, artifact publication
equivalence, or canonical reproducibility comparisons.

`BuildProvenance` is part of reproducibility and debugging. It is not a substitute for checking referenced artifacts and proof witnesses.

### Cache Record

```rust
struct CacheRecord<T> {
    cache_schema_version: String,
    key: CacheKey,
    produced_by: ToolchainInfo,
    dependencies: Vec<CacheDependency>,
    output_hash: Hash,
    output: T,
}

struct CacheKey {
    phase: PipelinePhase,
    source_hash: Option<Hash>,
    input_hashes: Vec<Hash>,
    config_hash: Hash,
    dependency_hashes: Vec<Hash>,
    schema_versions: Vec<SchemaVersion>,
}
```

Cache records are internal. They may use binary encodings if the cache schema version and compatibility rules are explicit.

## Alternatives Considered

### Raw IR Artifacts

Serializing internal IRs would make early tooling easy, but it would freeze implementation details and make schema compatibility unmanageable. The adopted design emits stable projections and keeps raw IR dumps as optional debug artifacts.

### Trusting Cached Proof Results Without Witness Hashes

This would make incremental builds faster, but a stale or corrupted cache entry could silently preserve an invalid proof status. The adopted design requires proof witness or deterministic discharge hashes in proof-related cache keys.

### Rebuilding Whole Packages on Any Change

Whole-package rebuilds are simple and acceptable as a fallback, but they make IDE feedback too slow. The adopted design separates interface and implementation hashes so proof-body edits do not unnecessarily invalidate downstream modules.

### Content-Addressed Store Only

A pure content-addressed store is attractive for deduplication, but users still need predictable `build/` paths for LSP, docs, logs, and inspection. The adopted design may store cache blobs content-addressably while emitting stable package-relative artifact paths.

## Adopted Approach

Mizar Evo emits stable, versioned `VerifiedArtifact` projections and keeps incremental cache entries as internal optimization data. Reuse is allowed only through complete cache keys, dependency artifact hashes, dependency slices, verifier policy validation, and proof witness or deterministic discharge hashes. Artifact output is deterministic, package-relative, and written atomically.

This separates three concerns:

1. published artifacts define the external API for downstream tools and packages;
2. development artifacts support local diagnosis and proof repair;
3. cache records accelerate future builds without becoming trusted proof authority.

## Constraints and Assumptions

- Published artifacts are UTF-8 JSON unless a later schema explicitly defines another format.
- Published artifacts must be readable without loading compiler-internal cache records.
- All paths in portable artifacts are package-relative or workspace-relative.
- Local artifact metadata such as `verified_at` may be user-visible, but must be excluded from canonical artifact hashes.
- Cache entries may be deleted without changing source-level semantics.
- Artifact schemas use explicit `schema_version` fields and compatibility checks.
- Reused proof results must be tied to accepted proof witnesses or deterministic built-in discharge records.
- Incremental builds may choose coarser invalidation initially, but must preserve the same externally visible results as a clean build.
- Build failures during phase 15 must not leave partially-written artifacts that readers can mistake for complete output.
