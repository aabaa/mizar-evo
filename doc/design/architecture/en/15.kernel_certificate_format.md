# Architecture: Kernel Certificate Format

> Canonical language: English. Japanese companion: [../ja/15.kernel_certificate_format.md](../ja/15.kernel_certificate_format.md).

## Purpose

This document defines the final proof certificate format consumed by the trusted kernel.

It refines pipeline phases 13-14. ATP backends, deterministic pre-ATP discharge, and Mizar-side proof reconstruction may produce evidence, but the kernel accepts only this normalized, replayable certificate shape.

## Design Decisions

### Certificate Top Level

A certificate is a canonical, versioned object.

```text
Certificate
  version
  target_vc
  kernel_profile
  imported_axioms
  imported_theorems
  generated_clauses
  substitutions
  resolution_trace
  derived_facts
  final_goal
```

`target_vc` binds the certificate to one canonical verification condition. `kernel_profile` records the accepted certificate schema, clause encoding, hash algorithm, and enabled built-in checkers.

### Trust Scope

The kernel checks only:

- certificate parsing and schema validation;
- imported fact availability by stable id and content hash;
- SAT clause well-formedness;
- substitution, alpha-conversion, and free-variable side conditions;
- resolution trace replay;
- derivation of the final goal from checked clauses.

The kernel must not run proof search, select premises, resolve overloads, expand clusters, insert coercions, or consult global mutable compiler state.

### Clause Representation

Clause serialization must be canonical.

Rules:

- variables are ordered by canonical variable id, not allocation address or parser encounter order after normalization;
- literals are ordered by polarity, predicate/functor symbol id, arity, then normalized argument encoding;
- duplicate literals are removed during certificate normalization;
- tautological clauses are either rejected or represented by an explicit `tautology` marker according to the kernel profile;
- stable hashes are computed from the canonical byte encoding plus schema version.

No map or set iteration order may influence clause ids or clause hashes.

### Resolution Trace

Each resolution step records:

```text
ResolutionStep
  step_id
  parent_clause_a
  parent_clause_b
  pivot_literal
  generated_clause
```

The kernel replays each step by looking up both parents, checking that the pivot appears with opposite polarity, resolving the parents, canonicalizing the result, and comparing it to `generated_clause`.

Parent references must point only to imported clauses, generated clauses, or earlier resolution steps. Forward references are malformed certificates.

### Substitution Rule

Substitutions are explicit certificate entries.

```text
SubstitutionStep
  source_term
  target_term
  binder_context
  freshness_witnesses
  free_variable_constraints
```

The kernel verifies capture avoidance, alpha-conversion, free-variable constraints, and binder normalization exactly as defined in [16.substitution_and_binding.md](./16.substitution_and_binding.md).

### Imported Facts

Imported axioms and theorems are referenced by:

- package id;
- module path;
- exported item id;
- statement fingerprint;
- accepted proof status required by policy.

The certificate is invalid if an imported theorem is unavailable, has a mismatched fingerprint, or was accepted only under a weaker policy than the current kernel profile permits.

### Kernel Rejection Semantics

Rejection categories are stable API values:

| Category | Meaning |
|---|---|
| `malformed_certificate` | schema, version, reference, or canonical encoding failure |
| `invalid_substitution` | capture, alpha-conversion, freshness, or free-variable rule failure |
| `invalid_sat_proof` | clause or resolution replay failure |
| `unresolved_symbol` | referenced symbol, theorem, axiom, or VC is unavailable |
| `timeout` | checking exceeded deterministic resource budget |
| `resource_exhaustion` | memory, clause count, trace length, or recursion limit exceeded |

`timeout` and `resource_exhaustion` are not proof acceptance. They leave the obligation unverified.

## Interface Definitions

```rust
struct Certificate {
    version: CertificateVersion,
    target_vc: VcFingerprint,
    kernel_profile: KernelProfile,
    imported_axioms: Vec<ImportedFactRef>,
    imported_theorems: Vec<ImportedFactRef>,
    generated_clauses: Vec<Clause>,
    substitutions: Vec<SubstitutionStep>,
    resolution_trace: Vec<ResolutionStep>,
    derived_facts: Vec<DerivedFact>,
    final_goal: GoalRef,
}
```

## Constraints and Assumptions

- Certificate checking must be deterministic for identical certificate bytes and dependency artifacts.
- Certificate parsing is part of the trusted boundary and must be small, total, and resource-bounded.
- Backend-specific proof formats must be translated into this format before kernel acceptance.
- A certificate hash is computed only from canonical content, never from file paths, timestamps, or backend runtime logs.
