# Architecture: Kernel Certificate Format

> Canonical language: English. Japanese companion: [../ja/15.kernel_certificate_format.md](../ja/15.kernel_certificate_format.md).

## Purpose

This document defines the final proof evidence format consumed by the trusted kernel.

It refines pipeline phases 13-14. ATP backends, deterministic pre-ATP discharge, and Mizar-side proof reconstruction may produce evidence, but the kernel accepts only this normalized, replayable evidence shape.

## Post-Closeout Correction: Formula/Substitution Evidence

The accepted kernel evidence design is no longer a backend-proof certificate
whose trusted payload is a MiniSAT-compatible resolution trace. The legacy
certificate shape below describes the task-22 `mizar-kernel` implementation
and remains useful as a source-drift inventory item, but it is superseded for
new acceptance work.

The kernel evidence consumed by phase 14 is a canonical, versioned object:

```text
KernelEvidence
  schema_version
  encoding_version
  kernel_profile
  target_vc
  symbol_manifest
  variable_manifest
  formula_evidence
  substitutions
  provenance
  final_goal
```

`formula_evidence` records the source formulas that may be used for the
check: local hypotheses, cited premises, generated VC facts, and accepted
dependency facts. Imported axioms and imported theorems are not top-level
evidence sections: they are formula-entry source bindings that must be
validated against the caller-supplied immutable imported-fact context.
`substitutions` records explicit substitution operations and
binder contexts. `final_goal` records the target formula to refute or prove
under the kernel profile. Instantiated formulas and the SAT problem are not
trusted input fields: the kernel derives them from `formula_evidence`,
`substitutions`, and `final_goal` and may record them only as recomputed
diagnostic/check-trace artifacts.

The evidence does not record a backend proof method. Resolution, instance
method, inverse method, SMT proof objects, and backend logs may explain how an
ATP found the formulas and substitutions, but they are not trusted acceptance
material and must not be promoted into an alternate kernel proof-method
replay path. Phase 14 accepts only when the kernel can validate provenance,
validate substitutions, derive the instantiated formulas, encode the SAT
problem deterministically, and obtain an UNSAT/empty-clause result from the
trusted SAT checker.

The trusted SAT check is part of the kernel acceptance boundary, but it is not
ATP search. The kernel must not choose premises, invent substitutions, perform
premise minimization, search clusters, insert coercions, consult global mutable
compiler state, call external provers, or retry with fallback inference.

### Goal Polarity Is Bound By The Target Obligation

`final_goal` records both the goal formula and an assertion polarity. The
polarity is not free evidence: the check kind required for the target VC comes
from the caller's immutable context, never from the evidence producer. Proof
acceptance of a proof obligation requires the refutation polarity — the goal
is asserted false and the trusted SAT checker must report UNSAT over the
premises plus the negated goal. Evidence whose goal polarity does not match
the check kind required for its target VC is rejected as `context_mismatch`.
In particular, a consistency-style polarity (goal asserted true) must never
produce accepted proof status for a proof obligation: premises plus a
true-asserted goal being unsatisfiable proves that the premises refute the
goal, which is the opposite of proving it.

### Context Identity Covers Non-Imported Source Bindings

Formula entries whose source class is a local hypothesis, cited premise, or
generated VC fact bind producer-supplied ids and provenance payloads. Such a
binding is acceptance material only when the kernel can verify it against the
caller-supplied immutable context identity for the target VC — for example, a
canonical fingerprint of the VC's local context and generated-fact set carried
in the check context. A well-shaped local or VC-fact binding that the immutable
context cannot verify must be rejected fail-closed as `missing_provenance`;
otherwise an evidence producer could label an arbitrary formula — including
the target goal itself — as an available hypothesis. The concrete verification
payload is owned by the kernel evidence schema and the `mizar-vc` handoff
specs and must be defined before non-imported source bindings become
acceptance material.

The v1 context identity payload is caller context, not producer evidence. It
contains `schema_version = 1`, the target VC fingerprint, the opaque
`mizar-vc` canonical formula-envelope handoff hash, the task-28
`context_identity_hash()`, and sorted rows for non-imported source bindings.
The kernel does not recompute the canonical handoff hash from
`ParsedKernelEvidence::canonical_hash_input()`: that parser hash input is the
validated `MIZAR_KERNEL_EVIDENCE\0` binary envelope, while the `mizar-vc`
handoff hash is over the producer handoff renderer. The kernel treats the
canonical handoff hash as an immutable context binding and recomputes only the
context-identity hash over the documented v1 payload:

```text
vc-kernel-context-identity-v1
schema-version=1
target-vc={algorithm_id}:{lower_hex_digest}
canonical-evidence-hash={lower_hex_32_byte_hash}
[entries]
source={source_debug}; formula-id={formula_id}; fingerprint={algorithm_id}:{lower_hex_digest}; producer={producer_debug}
```

Rows are sorted exactly as the task-28 producer sorts them: source kind
(`LocalHypothesis`, `CitedPremise`, `GeneratedVcFact`), source id, formula id,
formula fingerprint `(algorithm_id, digest)`, then producer formula ref. Source
spellings are `LocalHypothesis { local_context_id: N }`,
`CitedPremise { local_context_id: N }`, and
`GeneratedVcFact { vc_fact_id: N }`. Producer refs are spelled
`Core(CoreFormulaId(N))` or `Generated(VcGeneratedFormulaId(N))`. Duplicate
matching rows are ambiguous and must reject fail-closed. The hash domain is
`mizar-vc-kernel-context-identity`; task-31 tests must include a golden vector
for this line grammar so the kernel implementation stays compatible with the
task-28 producer without depending on `mizar-vc`.

The bridge hash is the fixed 32-byte stable hash used by the task-28 producer,
specified here so the kernel can duplicate it without depending on `mizar-vc`.
It is an identity/invalidation hash, not a cryptographic acceptance primitive.
All arithmetic is wrapping `u64`. Initialize four lanes:

```text
lanes[0] = 0x6d697a61722d7663
lanes[1] = 0x70726f6f662d6964
lanes[2] = 0x7461736b32302d76
lanes[3] = 0x66696e6765722d31
```

For each byte of `domain.as_bytes()`, followed by one zero byte, followed by
the UTF-8 payload bytes, with zero-based `index`:

```text
lane = index mod 4
mixed_index = rotate_left_u64(index, lane + 1)
lanes[lane] ^= byte + 0x9e3779b97f4a7c15 + mixed_index
lanes[lane] = rotate_left_u64(lanes[lane], 11 + lane) * 0x100000001b3
```

After the byte loop:

```text
lanes[0] ^= payload_length_bytes
lanes[1] ^= rotate_left_u64(domain_length_bytes, 17)
lanes[2] ^= rotate_left_u64(lanes[0], 7)
lanes[3] ^= rotate_left_u64(lanes[1], 13)
```

The digest is `lanes[0] || lanes[1] || lanes[2] || lanes[3]` in big-endian
byte order.

Post-closeout `mizar-kernel` tasks must update the source, module specs,
tests, and rejection names toward this evidence format. Until then, references
to `Certificate`, `generated_clauses`, `resolution_trace`, and
`invalid_sat_proof` below are legacy/source-drift descriptions rather than the
target acceptance contract. The redesigned rejection model should introduce or
alias a stable SAT-check failure reason such as `invalid_sat_refutation`
without silently weakening existing diagnostics.

## Design Decisions

### Legacy Certificate Top Level

The task-22 legacy certificate is a canonical, versioned object.

```text
Certificate
  version
  target_vc
  kernel_profile
  imported_axioms
  imported_theorems
  generated_clauses
  substitutions
  resolution_trace
  derived_facts
  final_goal
```

`target_vc` binds the legacy certificate to one canonical verification condition. `kernel_profile` records the legacy certificate schema, clause encoding, hash algorithm, and enabled built-in checkers.

### Legacy Trust Scope

The kernel checks only:

- certificate parsing and schema validation;
- imported fact availability by stable id and content hash;
- SAT clause well-formedness;
- substitution, alpha-conversion, and free-variable side conditions;
- resolution trace replay;
- derivation of the final goal from checked clauses.

The kernel must not run proof search, select premises, resolve overloads, expand clusters, insert coercions, or consult global mutable compiler state.

### Clause Representation

Clause serialization must be canonical.

Rules:

- variables are ordered by canonical variable id, not allocation address or parser encounter order after normalization;
- literals are ordered by polarity, predicate/functor symbol id, arity, then normalized argument encoding;
- duplicate literals are removed during certificate normalization;
- tautological clauses are either rejected or represented by an explicit `tautology` marker according to the kernel profile;
- a legacy `tautology` marker is not the `empty` contradiction clause and cannot
  satisfy final-goal checking; under the post-closeout corrected path it is
  audit-only and never trusted acceptance material;
- stable hashes are computed from the canonical byte encoding plus schema version.

No map or set iteration order may influence clause ids or clause hashes.

### Resolution Trace

Each resolution step records:

```text
ResolutionStep
  step_id
  parent_clause_a
  parent_clause_b
  pivot_literal
  generated_clause
```

The kernel replays each step by looking up both parents, checking that the pivot appears with opposite polarity, resolving the parents, canonicalizing the result, and comparing it to `generated_clause`.

Parent references must point only to imported clauses, generated clauses, or earlier resolution steps. Forward references are malformed certificates.

### Substitution Rule

Substitutions are explicit certificate entries.

```text
SubstitutionStep
  source_term
  target_term
  binder_context
  freshness_witnesses
  free_variable_constraints
```

The kernel verifies capture avoidance, alpha-conversion, free-variable constraints, and binder normalization exactly as defined in [16.substitution_and_binding.md](./16.substitution_and_binding.md).

### Imported Facts

Imported axioms and theorems are referenced by:

- package id;
- module path;
- exported item id;
- statement fingerprint;
- accepted proof status required by policy.

The certificate is invalid if an imported theorem is unavailable, has a mismatched fingerprint, or was accepted only under a weaker policy than the current kernel profile permits.

On the corrected formula-evidence path, an accepted imported axiom/theorem
source binding additionally carries an imported-statement projection:

- the source `statement_fingerprint` uses the architecture-18 imported
  statement algorithm id `18`;
- the formula entry's tree fingerprint uses the kernel formula fingerprint
  algorithm id `2`;
- `statement_projection.statement_fingerprint` equals the source statement
  fingerprint;
- `statement_projection.formula_fingerprint` equals the formula entry's tree
  fingerprint;
- `statement_projection.payload` is the canonical v1 payload
  `MIZAR_KERNEL_IMPORTED_STATEMENT_PROJECTION\0` followed by the canonical
  bytes of the statement fingerprint and the canonical bytes of the formula
  fingerprint.

The parser rejects unsupported algorithms, empty or noncanonical projection
payloads, stale statement projections, and formula-projection mismatches. The
checker then compares the same projection against the caller's immutable
imported-fact context; absent or mismatched context projection is
`unresolved_symbol` and must occur before SAT encoding.

### Kernel Rejection Semantics

Rejection categories are stable API values:

| Category | Meaning |
|---|---|
| `malformed_certificate` | schema, version, reference, or canonical encoding failure |
| `invalid_substitution` | capture, alpha-conversion, freshness, or free-variable rule failure |
| `invalid_sat_proof` | legacy clause or resolution replay failure |
| `invalid_sat_refutation` | corrected formula/substitution evidence path failed kernel-derived SAT refutation checking |
| `unresolved_symbol` | referenced symbol, theorem, axiom, or VC is unavailable |
| `timeout` | checking exceeded deterministic resource budget |
| `resource_exhaustion` | memory, clause count, trace length, or recursion limit exceeded |

`timeout` and `resource_exhaustion` are not proof acceptance. They leave the obligation unverified.

## Interface Definitions

```rust
struct Certificate {
    version: CertificateVersion,
    target_vc: VcFingerprint,
    kernel_profile: KernelProfile,
    imported_axioms: Vec<ImportedFactRef>,
    imported_theorems: Vec<ImportedFactRef>,
    generated_clauses: Vec<Clause>,
    substitutions: Vec<SubstitutionStep>,
    resolution_trace: Vec<ResolutionStep>,
    derived_facts: Vec<DerivedFact>,
    final_goal: GoalRef,
}
```

## Constraints and Assumptions

- Certificate checking must be deterministic for identical certificate bytes and dependency artifacts.
- Certificate parsing is part of the trusted boundary and must be small, total, and resource-bounded.
- Backend-specific proof formats must be translated into this format before kernel acceptance.
- A certificate hash is computed only from canonical content, never from file paths, timestamps, or backend runtime logs.
- A fingerprint or content hash that participates in imported-fact identity, target binding, or provenance comparison must be either the exact canonical bytes or a collision-resistant digest; registering a spoofable digest algorithm would make imported-fact identity forgeable and is prohibited.
