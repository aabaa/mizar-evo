# Architecture: Documentation and Extraction

> Canonical language: English. Japanese companion: [../ja/13.documentation_and_extraction.md](../ja/13.documentation_and_extraction.md).

## Purpose

This document defines how Mizar Evo generates documentation and extracted runtime code from verified artifacts.

It refines pipeline phase 16. Documentation generation and code extraction are downstream consumers of verified semantic metadata: they do not establish proof validity, change verification results, or read compiler-internal IR directly unless an implementation-specific cache has already been validated against the corresponding `VerifiedArtifact`.

## Design Decisions

### Phase 16 Is a Consumer Phase

Documentation and extraction consume verified outputs.

They may fail with documentation or extraction diagnostics, but they must not:

- reinterpret name, type, registration, overload, or proof decisions;
- accept or reject kernel evidence;
- mutate `*.mizir.json` or proof witnesses;
- make downstream packages depend on raw compiler IR.

If phase 16 needs implementation-specific data such as cached `ControlFlowIr`, it must verify that the cache key matches the artifact manifest, source hash, toolchain, and schema versions. Otherwise it rebuilds the required projection from stable artifacts or reports a cache miss.

### Documentation Uses Public Semantic Metadata

Generated documentation is based on public verified metadata, not a best-effort parse of source files.

Primary documentation inputs:

- artifact manifest;
- `VerifiedArtifact.exports`;
- source-mapped doc comments from preprocessing;
- resolved FQNs and labels;
- rendered signatures, theorem statements, algorithm contracts, and source links;
- `@latex` annotations and default formula rendering metadata;
- package manifest metadata such as name, version, authors, and edition.

Source files are still read to recover documentation comment text and source snippets, but semantic decisions come from artifacts.

### Extraction Uses Verified Executable Subset

Code extraction applies only to algorithms whose runtime behavior is executable and whose required obligations have been accepted under the active extraction policy.

An algorithm is extractable only when:

- its source artifact and dependency artifacts match the manifest;
- all required contract and termination obligations are accepted;
- ghost declarations, ghost assignments, snapshots, proof-only assertions, and annotations can be erased;
- every runtime operation has an extraction lowering for the selected target;
- non-deterministic `Pick` expressions are executable for the selected target or are rejected;
- externally attested proofs are allowed by policy or absent.

Extraction does not prove additional properties. It trusts phase 14 proof acceptance and phase 15 artifact integrity.

### Documentation and Extraction Share Symbol Identity

Both outputs use the same `ModulePath`, FQN, `ItemId`, source ranges, and export summaries from artifacts.

This ensures:

- documentation anchors match exported labels;
- extracted symbol names can be traced back to source items;
- source links and diagnostics point to the same ranges as LSP and CLI output;
- generated docs and extracted code agree about which items are public.

Target-language names may be mangled for legality, but the mapping must be recorded.

### Outputs Are Deterministic

For identical source, lockfile, verified artifacts, extraction target, and documentation options, phase 16 output must be deterministic.

Rules:

- modules and items are emitted in canonical module/FQN order;
- search indexes and cross-reference indexes are sorted by stable keys;
- generated filenames are derived from module paths and stable labels;
- target-language names are produced by deterministic name mangling;
- timestamps are omitted from reproducible outputs or isolated as local metadata excluded from output hashes;
- diagnostics are ordered using the same rules as [12.diagnostics_and_lsp.md](./12.diagnostics_and_lsp.md).

### Documentation Is Not a Publication Gate for Proof Validity

Documentation warnings should not invalidate a verified package unless a publication profile explicitly requires documentation quality checks.

Examples:

- unresolved doc link: warning by default;
- malformed Markdown: warning with degraded rendering;
- invalid LaTeX: warning or rendering-time marker;
- missing doc comment: optional lint;
- source link unavailable: warning if source is not packaged.

Proof validity remains governed by phases 0-15.

## Documentation Pipeline

### Step 1: Load Documentation Inputs

Input:

- package artifact manifest;
- verified module artifacts;
- source files or source bundles;
- package manifest metadata;
- documentation configuration.

Output:

- `DocBuildPlan`;
- per-module `DocInput`;
- diagnostics for missing or stale artifacts.

The loader validates artifact schema versions, source hashes, and package-relative paths before rendering.

### Step 2: Attach Doc Comments

Doc comments are attached to module and item identities produced by the verifier.

Attachment rules:

- module-level comments attach to the module page;
- item comments attach by source range and item identity;
- comments attached to non-exported items may be omitted from public docs unless private-doc output is requested;
- comments for invalid or unverified items are not rendered as verified API documentation.

The doc generator should preserve Markdown source ranges so diagnostics inside comments can point to the relevant `:::` line.

### Step 3: Resolve Links and Render Math

Cross-references resolve through exported labels and FQNs from artifacts.

The renderer:

- resolves inline code references only when they exactly match a visible label or FQN;
- resolves explicit Markdown links using the spec-defined reference forms;
- builds automatic "Used by" indexes from `used_axioms` and citation metadata when available;
- applies `@latex` annotations for symbol rendering;
- falls back to default syntax-to-LaTeX rendering when no annotation exists.

Unresolved or ambiguous documentation links produce diagnostics but are rendered safely as text.

### Step 4: Emit Static Site

The emitter writes:

- package index;
- module pages;
- item pages;
- search index;
- cross-reference index;
- static assets;
- optional machine-readable documentation index.

Output writes follow the same atomic-write principles as phase 15: readers should not observe partially-written pages or indexes.

## Extraction Pipeline

### Step 1: Select Extractable Items

The extractor starts from exported algorithms, explicit extraction configuration, or CLI-selected items.

It classifies each candidate:

| Status | Meaning |
|---|---|
| `Extractable` | all verification, computability, and target constraints are satisfied |
| `SpecOnly` | mathematically valid but not executable for the selected target |
| `BlockedByProof` | required obligations are open or disallowed by policy |
| `BlockedByTarget` | target lacks a lowering for a runtime construct |
| `BlockedByDependency` | dependency artifact or runtime mapping is missing |

Only `Extractable` items are emitted.

### Step 2: Build Runtime IR

Runtime IR is a projection of verified algorithm data.

It removes:

- ghost declarations and assignments;
- assertions that have no runtime checking policy;
- proof hints and source annotations;
- specification-only snapshots and claim blocks;
- theorem/proof-only structures.

It preserves:

- runtime parameters and return values;
- mutable variables and assignments;
- control-flow structure;
- calls to extractable algorithms;
- runtime checks required by the selected policy;
- source maps for diagnostics and debug output.

### Step 3: Lower to Target Code

Each target backend owns syntax emission and runtime-library bindings.

Common lowering responsibilities:

- name mangling and reverse source map emission;
- type representation mapping;
- function/procedure signature emission;
- control-flow lowering;
- runtime failure handling for permitted dynamic checks;
- packaging generated files into the requested output directory.

Backends must reject unsupported constructs explicitly rather than generating partial code.

### Step 4: Emit Extraction Manifest

Extraction writes a manifest connecting generated code to verified sources.

```rust
struct ExtractionManifest {
    schema_version: String,
    target: ExtractionTarget,
    package_id: PackageId,
    artifact_manifest_hash: Hash,
    verifier_config_hash: Hash,
    items: Vec<ExtractedItem>,
    diagnostics: Vec<Diagnostic>,
}

struct ExtractedItem {
    source_item: SymbolId,
    source_range: SourceRange,
    generated_name: String,
    generated_files: Vec<PathBuf>,
    source_artifact_hash: Hash,
    target_options_hash: Hash,
}
```

The manifest is not proof evidence. It is traceability metadata for generated code.

### Runtime IR

```rust
struct RuntimeIr {
    algorithm: SymbolId,
    signature: RuntimeSignature,
    blocks: Vec<RuntimeBlock>,
    source_map: RuntimeSourceMap,
    required_runtime: Vec<RuntimeCapability>,
}
```

`RuntimeIr` is target-neutral. It is not a proof IR and should not be used by the verifier to establish correctness.

## Alternatives Considered

### Generate Documentation Directly from Source

Parsing source alone would make documentation generation independent of verification, but it would lose resolved labels, overload decisions, rendered semantic signatures, and proof status. The adopted design reads source for comments and snippets while using artifacts for semantics.

### Extract Directly from ControlFlowIr

This is tempting because `ControlFlowIr` already contains executable structure. However, it also contains verification-specific facts and ghost metadata. The adopted design projects a smaller `RuntimeIr` before target lowering.

### Treat Documentation Generation as a Verification Phase

Failing the build for every doc warning would improve documentation quality, but it would conflate proof validity with presentation quality. The adopted design keeps documentation diagnostics separate and allows stricter publication profiles.

### One Target Backend First, Then Generalize

Hard-coding one target would speed up early implementation but would leak target assumptions into Runtime IR. The adopted design keeps target-neutral extraction boundaries even if the first implementation supports only one backend.

## Adopted Approach

Mizar Evo treats documentation generation and code extraction as deterministic consumers of verified artifacts. Documentation renders public semantic metadata plus source doc comments. Extraction projects verified executable algorithms into target-neutral `RuntimeIr`, erases proof-only state, and then lowers through target-specific backends.

Both outputs carry traceability metadata back to source artifacts, but neither output is trusted proof evidence.

## Constraints and Assumptions

- Documentation and extraction require verified artifacts for semantic output.
- Documentation may degrade gracefully for presentation errors; extraction must reject unsupported runtime constructs.
- Ghost state and proof-only annotations must not appear in extracted runtime code.
- Extracted code must be traceable to source items and artifact hashes.
- Target backends must be deterministic for identical `RuntimeIr` and target options.
- Phase 16 outputs may be deleted and regenerated without changing proof validity.
- Publication profiles may impose stricter documentation or extraction requirements than normal local builds.
