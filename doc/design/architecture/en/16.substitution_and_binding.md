# Architecture: Substitution and Binding

## Purpose

This document fixes binder, variable, alpha-equivalence, and substitution semantics for evo2.

It applies to name resolution, type checking, elaboration, VC generation, kernel evidence checking, and diagnostics. The goal is to make binding behavior deterministic and kernel-checkable.

## Design Decisions

### Variable Classes

The implementation distinguishes:

- bound variables introduced by binders;
- free variables from local context;
- schematic variables used in patterns, rules, or evidence templates;
- generated fresh variables introduced during elaboration or proof checking.

Each variable has a stable internal id and a source-facing display name. Equality uses the stable id after binding resolution, not the display name.

### Bound and Free Variables

Free-variable sets are computed from normalized binder structure.

Rules:

- entering a binder removes that binder variable from the free-variable set of its body;
- shadowing is represented by distinct stable ids;
- source names are not sufficient to decide capture;
- free-variable conditions are serialized in kernel evidence when they are required for acceptance.

### Alpha-Equivalence

Two terms are alpha-equivalent when they differ only by consistent renaming of bound variables.

The checker decides alpha-equivalence by canonical binder normalization, not by ad hoc textual comparison.

### Canonical Binder Normalization

Binder normalization assigns binder-local canonical ids in lexical binder order.

```text
for x being set holds
  ex y being set st P[x, y]
```

is normalized by binder positions, not by the names `x` and `y`. Equivalent formulas with renamed binders produce identical normalized encodings and hashes.

### Capture Avoidance

Substitution must not cause a free variable in the substituted term to become bound by a binder in the target context.

If a collision would occur, the implementation must alpha-rename the binder using the deterministic freshness strategy before applying the substitution. If deterministic renaming cannot satisfy all recorded side conditions, the substitution is rejected.

### Definition-Time Closures

Local abbreviations (`set`, `deffunc`, and `defpred`) are substitutions with a
recorded closure, not text macros. When such an abbreviation is defined, the
resolver records the stable ids of every free variable in the defining term or
formula. Later uses substitute actual arguments for formal binders while keeping
those captured ids fixed.

Consequences:

- shadowing a captured display name after the definition does not change the
  abbreviation's meaning;
- alpha-renaming around the use site is allowed and expected when needed to
  avoid capture;
- a substitution step is valid only when the actual arguments satisfy the
  formal type guards recorded at the definition site;
- evidence and legacy trace formats must serialize the captured free-variable
  set whenever abbreviation expansion is replayed outside the original resolver.

For example, if `defpred P(n be Nat) means n < m;` is defined while `m` refers
to a particular local binding, then a later `for m being Nat holds P(m)` expands
with two distinct variable ids: the quantified `m` as the actual argument and
the captured outer `m` inside the body.

### Renaming Strategy

Fresh ids are generated from:

- the owning source file or evidence id;
- the normalized binder path;
- the original variable role;
- a deterministic counter scoped to the normalized term.

Generated display names are diagnostic-only. They must not affect hashes or proof acceptance.

For kernel-replayable evidence, the evidence-visible freshness check is
fully determined by the normalized evidence. The binder path selects the
original binder in the source term and the same normalized binder position in
the target term. The target binder must be a generated-fresh binder whose
variable id is the witness `fresh` id. The avoided-variable set is recomputed
from the free variables of the source binder body, excluding the original
bound variable, plus the free variables of actual terms inserted below that
binder. The candidate stream is the evidence variable manifest in ascending
stable-id order after removing the recomputed avoided set. The deterministic
counter is the zero-based index of the generated id in that candidate stream.

The source-file or evidence domain separation used by producer phases must
already be reflected in the stable variable manifest and witness envelope. A
kernel checker must not consult producer state to repair or rediscover a fresh
id; if the normalized evidence is missing, stale, or inconsistent with the
candidate stream above, the substitution is rejected.

For free-variable replay, a recorded capture set is not self-attesting. The
kernel recomputes the capture set at the recorded target path from the
normalized target binder stack. A free-variable side condition is valid only
when the recorded set exactly matches the recomputed set and the requested
free-variable predicate holds over the normalized target subtree.

### Substitution Replay

Substitution replay is a pure function:

```text
(normalized_source, normalized_target, binder_context, side_conditions)
  -> accepted normalized result
  -> or invalid_substitution
```

Replay must not consult active parser state, overload tables, cluster tables, or ATP backend output.

## Constraints and Assumptions

- Binder normalization must be stable across platforms and worker counts.
- Source display names are preserved for diagnostics but excluded from semantic equality after resolution.
- Malformed substitution evidence is a kernel rejection, not an opportunity for heuristic repair.
- Fuzzing and fail tests must include variable capture, binder collision, malformed substitution, and alpha-conversion failure cases.
