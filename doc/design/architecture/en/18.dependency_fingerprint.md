# Architecture: Dependency Fingerprint

> Canonical language: English. Japanese companion: [../ja/18.dependency_fingerprint.md](../ja/18.dependency_fingerprint.md).

## Purpose

This document defines dependency fingerprints and dependency slices for stable incremental rebuilds.

The goal is to rebuild exactly the semantic work affected by a change while preserving deterministic artifacts and proof acceptance.

## Design Decisions

### Fingerprint Targets

The build system fingerprints at least:

- theorem statements and accepted proof status;
- definitions and definitional expansion boundaries;
- exported algorithm signatures and their `requires` / `ensures` contracts (separately from algorithm bodies, mirroring the theorem-statement/proof-body split);
- notation declarations and parse effects;
- clusters;
- registrations;
- modes;
- attributes.

Proof bodies are fingerprinted separately from theorem statements so downstream modules are not rebuilt when only a proof implementation changes and the accepted theorem statement remains unchanged.

### Dependency Slice

Each theorem and verification condition stores a dependency slice.

```text
DependencySlice
  used_theorems
  used_definitions
  used_clusters
  used_notations
  used_modes
  used_attributes
```

The slice records exactly what was used to parse, resolve, type-check, elaborate, generate VCs, translate ATP problems, and accept certificates.

### Stable Fingerprint Inputs

Fingerprints include:

- canonical semantic content;
- schema version;
- language edition;
- dependency-facing policy fields;
- normalized origin id;
- accepted proof status when visibility depends on it.

Fingerprints exclude:

- comments;
- source formatting outside syntax-sensitive notation;
- timestamps;
- absolute local paths;
- worker ids;
- backend runtime duration.

### Rebuild Triggers

The table distinguishes downstream semantic rebuilds from local artifact refreshes. A proof-body-only edit may require rebuilding local proof witnesses or implementation hashes, but it must not invalidate downstream modules when the exported theorem statement and accepted proof status are unchanged.

| Change | Downstream Semantic Rebuild | Local Artifact Refresh |
|---|---|---|
| comment only | no | no |
| theorem proof body only | no | yes, when proof witnesses or implementation hashes change |
| theorem statement | yes | yes |
| cluster semantics | yes | yes |
| notation parse result | yes | yes |
| registration coherence proof status | yes for importers that can see the registration | yes |
| diagnostic wording only | no | yes, for diagnostic snapshots only |

### API Compatibility Diff

The same public-API fingerprints drive the `mizar semver-check` linter (spec §23.10), which classifies a change against the semantic-versioning rules of spec §23.2. The linter uses **two distinct keys per surface element**: the **normalized origin id** is the *identity* that pairs an element across the baseline and the candidate, and the **fingerprint** is the *value* compared once two elements have been paired by identity. It takes the required version bump as the maximum over all per-element deltas. That verdict is **exact for structural deltas** (add, remove, rename, signature/statement/contract change, export visibility, edition) but, for overload-resolution shifts, is only an input-dependent best-effort heuristic (see below and spec §23.10). Keeping identity and value separate is what makes the structural verdict unambiguous: a missing-or-new identity is a remove/rename, while a same-identity / different-value pair is a signature-or-statement change — the two are not conflated even though both are usually MAJOR.

Reusing fingerprints — rather than diffing source text — then gives the linter three properties for free: a rename is a removed id plus an added id rather than an edit, so it is correctly MAJOR; proof-body-only, algorithm-body-only, and comment-only edits are already excluded from the signature/contract fingerprint inputs above, so they never force a bump; and an overload-resolution shift (spec §23.8.3) is visible as a changed fingerprint — of a registration, or of a conditional cluster, attribute, or mode (which shift resolution through the attribute closure, §19.2.1) — even when no symbol was textually removed.

**Overload-resolution shifts are detected by a best-effort local heuristic.** Whether a change actually shifts resolution depends on downstream call sites, which are not inputs to a single-package fingerprint diff. The linter therefore applies a best-effort heuristic over this package's own baseline-vs-candidate surface (spec §23.10, W0303), flagging as MAJOR all of the following: (a) any added or changed ordinary overload registration that is strictly more specific than, or incomparable-minimal with, an existing ordinary root in the same resolution domain, (b) any added or changed coherent redefinition that changes the joined exposed result facts of its root or can introduce an incompatible refinement join, and (c) any added or changed conditional cluster, attribute, or mode that alters the attribute closure — because the closure feeds functional-cluster guard firing, `RefinementJoin` result facts, and subsumption-DAG edge preconditions (spec §19.2.1, §23.7.3, §23.8.2), a cluster change can shift the viability or specificity of existing candidates without registering a new one. Because the precise outcome depends on expressions only dependents can see, this single-package heuristic both over-reports (false positives) and may miss a downstream-only shift (false negatives) — it is **neither sound nor complete**; a reverse-dependency corpus or recorded call-site traces, where available, make the verdict exact.

**Rebuild triggers and semver classification are related but distinct axes.** A downstream semantic rebuild answers "must dependents recompile?"; a semver bump answers "can dependents' version constraints still be satisfied?". They diverge in both directions:

| Change | Downstream rebuild | Required bump |
|---|---|---|
| new exported theorem or definition | yes (new fingerprint visible) | MINOR — additions do not break dependents |
| removed export | yes | MAJOR |
| weakened `ensures` / strengthened `requires` | yes | MAJOR |
| theorem proof body only | no | PATCH |

A MINOR addition still triggers a downstream rebuild because the dependency slice changes, yet it is non-breaking; conversely a removed export is both a rebuild and a MAJOR break. The linter therefore classifies deltas under spec §23.2 directly and does not reuse the Rebuild Triggers table as its compatibility verdict.

### Cache Validation

A cache hit is valid only when every fingerprint in the dependency slice still matches and the active verifier policy is compatible with the recorded policy.

If a slice is missing or was produced by an unsupported schema, the build must conservatively re-run the affected phase.

## Interface Definitions

```rust
struct DependencySlice {
    owner: SemanticOwnerId,
    used_theorems: Vec<TheoremFingerprint>,
    used_definitions: Vec<DefinitionFingerprint>,
    used_clusters: Vec<ClusterFingerprint>,
    used_notations: Vec<NotationFingerprint>,
    used_modes: Vec<ModeFingerprint>,
    used_attributes: Vec<AttributeFingerprint>,
}
```

## Constraints and Assumptions

- Fingerprint computation must use canonical ordering and canonical serialization.
- Incremental rebuild is an optimization; a full clean build remains the reference behavior.
- Missing dependency data must not be interpreted as "no dependency".
- Dependency slices are part of regression snapshots for deterministic build testing.
