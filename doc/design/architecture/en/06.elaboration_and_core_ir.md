# Architecture: Elaboration and Core IR

> Canonical language: English. Japanese companion: [../ja/06.elaboration_and_core_ir.md](../ja/06.elaboration_and_core_ir.md).

## Purpose

This document defines how Mizar Evo lowers fully resolved surface semantics into `CoreIr`, the prover- and kernel-oriented logical representation used by later verification phases.

It refines phase 9 of [00.pipeline_overview.md](./00.pipeline_overview.md). In particular, it defines the boundary between `ResolvedTypedAst`, which preserves source shape for diagnostics and LSP metadata, and `CoreIr`, which normalizes terms, formulas, definitions, proof structure, and semantic annotations for VC generation and kernel checking.

## Design Decisions

### Elaboration Is the Last Source-Shaped Boundary

`ResolvedTypedAst` is still close to the user's source. It preserves notation, inserted views, overload metadata, and recoverable failed sites. `CoreIr` is normalized for semantic consumers.

Elaboration owns:

- lowering resolved terms and formulas into canonical core nodes;
- replacing user-facing symbols with canonical `SymbolId` references;
- making inserted `qua` views and type assumptions explicit;
- expanding syntactic sugar that has no semantic identity of its own;
- recording definition, theorem, registration, and algorithm boundaries;
- extracting proof skeleton structure into core proof nodes;
- producing source maps back to `ResolvedTypedAst`.

Elaboration does not own:

- name, type, registration, or overload resolution;
- proof search or ATP dispatch;
- control-flow graph construction for algorithm bodies;
- final VC generation;
- artifact schema emission.

If a `ResolvedTypedAst` node is unresolved or blocked, elaboration must preserve an explicit error node or skip the affected item. It must not fabricate a core term.

### Core IR Erases Soft Types Deliberately

Mizar soft types guide checking but are not part of the logical foundation. `CoreIr` therefore erases type annotations into explicit logical predicates and assumptions.

Examples:

- a variable declaration `let x be T` lowers to a core binding for `x` plus a local fact `is_T(x)`;
- a type assertion `x is T` lowers to a predicate formula for the normalized type;
- a `qua` view lowers to the same underlying term plus explicit view metadata and any already established type facts needed by later explanation;
- a `reconsider` binding lowers to a new or narrowed binding with its associated obligation seed reference.

Erasure must be explicit. Later phases should not need to inspect source type syntax to know which logical predicates are available.

### Definitions Are Lowered with Stable Expansion Boundaries

Elaboration records definitions as core items with explicit expansion policy. It does not eagerly inline every definition.

Definition records include:

- canonical defining symbol;
- parameter binders and type predicate assumptions;
- definiens formula or term when present;
- existence, uniqueness, coherence, compatibility, or reducibility obligation seed references;
- rewrite or reduction metadata when the definition is allowed to participate in computation or simplification;
- source ranges and documentation attachment ids.

Later phases choose whether to unfold a definition based on proof context, ATP encoding profile, kernel policy, and reduction metadata. This keeps elaboration deterministic and prevents accidental proof-strength changes from inlining order.

### Templates Are Already Concrete

Template inference and overload selection complete before elaboration. `CoreIr` should not contain unresolved template parameters at ordinary use sites.

Elaboration may still record template origin metadata:

- the source template symbol;
- concrete instantiation arguments;
- generated concrete symbol or instantiation key;
- source range of explicit or inferred template arguments.

This metadata supports diagnostics, reproducibility, and artifact output, but core logical nodes refer to concrete symbols and concrete types.

### Proof Skeletons Become Core Proof Trees

Surface proof syntax contains many convenience forms: `thesis`, structured natural-deduction steps, labels, `by` references, and nested proof blocks. Elaboration lowers these into a core proof tree.

The core proof tree records:

- theorem or lemma proposition;
- current thesis at each proof point;
- introduced variables and assumptions;
- statement labels and citation references;
- terminal proof goals and their local premise sets;
- status dependencies for `open`, `assumed`, and `conditional` items.

`thesis` is not preserved as a magic identifier. It is replaced by the current core formula computed from the proof skeleton.

### Algorithms Are Lowered to Core Items, Not Yet CFG

Algorithm definitions cross two later concerns: logical verification and executable extraction. Phase 9 records algorithm items in `CoreIr` but does not build full `ControlFlowIr`.

Elaboration owns:

- algorithm identity and parameter binders;
- resolved return type and result symbol;
- preconditions, postconditions, invariants, assertions, and termination measures as core formulas or terms;
- ghost/runtime classification on bindings and expressions;
- source-shaped statement tree with resolved symbols.

Phase 10 owns:

- basic-block construction;
- mutation and assignment flow;
- loop invariant placement;
- reachability and use-before-assignment checks;
- extraction-oriented runtime/ghost separation.

This keeps pure logical lowering separate from control-flow analysis.

Algorithm `requires`, `ensures`, `assert`, `invariant`, and `decreasing` clauses are lowered to core formulas or terms in phase 9. Their VC placement, path conditions, loop cutpoints, and termination-check structure are decided later by phases 10 and 11.

### Surface Terms Lower Through Canonical Core Symbols

Some surface constructs are syntactically terms but need generated logical
definitions before they can be used as ordinary `CoreTerm`s. Elaboration owns
that naming step.

- A choice term `the T` lowers to a stable generated choice symbol keyed by the
  alpha-normalized target type and by the explicit free parameters of that type.
  Repeated occurrences with the same key in the same elaboration context denote
  the same core term. The elaboration context is the owning core item after
  template instantiation; generated symbols inside an exported definition are
  owned by that definition and reused when it is unfolded. The generated term
  carries a non-emptiness obligation seed and a type fact
  `is_T(choice_T(params), params)`.
  Runtime `the` occurrences in executable algorithm statements are recorded as
  Pick bindings for phase 10 instead of sharing this stable choice symbol.
- A Fraenkel comprehension lowers to a generated set-valued symbol
  `comprehension_H(params)`, where `H` is the alpha-normalized generator,
  mapper, and predicate body, and `params` are the free variables captured from
  the surrounding context. Elaboration emits a definitional obligation seed for
  the membership axiom of that generated set and records the sethood evidence
  used to accept each generator.
- Nested generated terms capture outer binders as parameters. They never reuse a
  source display name to decide identity; binder ids and alpha-normalized bodies
  decide the generated symbol key.

These generated symbols are internal core names. Stable artifacts may expose
their source range and normalized shape, but source programs cannot cite the
generated names directly.

### Local Abbreviations Are Expanded With Definition-Time Closures

`set`, `deffunc`, and `defpred` introduce proof-local abbreviations, not local
FOL function or predicate symbols. Elaboration records an expansion entry that
contains:

- the formal binders and their type guards;
- the defining term or formula;
- the free variables of the defining body as captured at the definition site;
- the source range and label-like identity used for diagnostics.

Every use is expanded by capture-avoiding substitution against that recorded
definition-time closure. If a later binder shadows a captured display name, the
captured variable id still refers to the original binding. If a use does not
satisfy the abbreviation's formal type guards, elaboration emits the same kind
of type obligation or diagnostic as an ordinary application site.

### Conditional Definiens Lower to Guarded Obligations

A definition body with `if` branches and optional `otherwise` lowers to an
ordered guarded definiens record. Elaboration does not leave branch selection
implicit:

- each branch guard becomes an antecedent for the corresponding defining
  equation or predicate equivalence;
- overlapping branches generate consistency obligation seeds so that the
  definitions agree on common inputs;
- an `otherwise` branch is guarded by the negation of all earlier guards;
- if no `otherwise` branch exists, coverage obligations describe exactly where
  the definition is usable.

VC generation later turns these seeds into ordinary definition correctness VCs.

### Source Maps Are Required, Not Debug Extras

Every core item and core expression that can produce a diagnostic, obligation, or metadata entry must map back to source.

Source maps should preserve:

- source range in `ResolvedTypedAst`;
- originating symbol or label;
- inserted view provenance;
- lowered type predicate provenance;
- template instantiation provenance;
- proof skeleton location.

Later phases must use these maps for diagnostics and artifacts instead of reconstructing locations from core structure.

### Core IR Is Backend-Neutral

`CoreIr` is not TPTP, SMT-LIB, or a kernel certificate language. It is a canonical internal logical representation.

Backend-specific concerns are deferred:

- premise selection and VC slicing to phase 11-12;
- ATP symbol naming and encoding to phase 13;
- certificate format and replay to phase 14;
- stable external metadata projection to phase 15.

## Elaboration Pipeline

### Step 1: Prepare Core Context

Input:

- `ResolvedTypedAst`;
- `SymbolEnv`;
- `TypeFactTable`;
- `ResolutionTrace`;
- dependency summaries.

Output:

- `CoreContext`;
- core symbol table;
- elaboration worklist.

Responsibilities:

- validate that required overload resolution results are present;
- load dependency core summaries needed for references;
- assign stable `CoreItemId`s for current-module items;
- initialize source-map builders;
- collect item-level elaboration dependencies.

### Step 2: Lower Types and Facts

Input:

- normalized types;
- type facts;
- inserted views;
- obligation seeds from earlier semantic phases.

Output:

- core type predicates;
- core assumption records;
- view explanations;
- obligation seed references.

Responsibilities:

- lower type expressions to predicate applications;
- lower attribute chains to conjunctions of predicates;
- record positive and negative attribute polarity;
- attach provenance from `ResolutionTrace`;
- preserve obligation seed ids for later VC generation.

### Step 3: Lower Terms and Formulas

Input:

- resolved term and formula nodes;
- overload resolution table;
- inserted views and type facts.

Output:

- `CoreTerm`s;
- `CoreFormula`s;
- expression source maps.

Responsibilities:

- lower variables, constants, functor applications, selectors, aggregates, and set expressions;
- lower predicate applications, equality, type assertions, attributes, connectives, and quantifiers;
- replace overload sites with selected `SymbolId`s;
- replace `qua` with view metadata and the underlying term;
- preserve unresolved failed sites as explicit elaboration errors.

### Step 4: Lower Declarations and Definitions

Input:

- resolved declarations and signatures;
- lowered terms and formulas;
- pending obligation seed metadata.

Output:

- `CoreItem`s;
- `CoreDefinitionTable`;
- obligation seed table.

Responsibilities:

- lower structures, modes, attributes, predicates, functors, theorems, schemes, registrations, and reductions;
- record definition expansion policy;
- record correctness obligation seed references;
- preserve visibility and export metadata needed by artifacts;
- reject or skip items whose semantic prerequisites failed.

### Step 5: Lower Proofs

Input:

- theorem and proof nodes;
- proof labels and citations;
- lowered formulas.

Output:

- `CoreProofTable`;
- core proof trees;
- terminal proof goal seeds.

Responsibilities:

- compute proof skeletons;
- replace `thesis` with the current core formula;
- lower assumptions and introduced variables;
- attach cited premise references;
- produce terminal proof goal seeds for VC generation;
- preserve status dependencies for `open`, `assumed`, and `conditional` items.

### Step 6: Lower Algorithm Shells

Input:

- resolved algorithm definitions;
- lowered contract formulas and terms;
- ghost/runtime metadata.

Output:

- core algorithm items;
- statement-tree shells for phase 10;
- contract and termination metadata.

Responsibilities:

- record algorithm parameter and result binders;
- lower `requires`, `ensures`, `assert`, `invariant`, and `decreasing` clauses;
- preserve statement order and resolved expression metadata;
- distinguish ghost and runtime bindings;
- defer CFG construction to phase 10.

## Interface Definitions

### Core IR

```rust
struct CoreIr {
    module_id: ModuleId,
    items: Vec<CoreItem>,
    definitions: CoreDefinitionTable,
    proofs: CoreProofTable,
    algorithms: CoreAlgorithmTable,
    obligation_seeds: ObligationSeedTable,
    source_map: CoreSourceMap,
}
```

`CoreIr` is compiler-internal. Stable external artifacts are projections emitted later.

### Core Items

```rust
struct CoreItem {
    id: CoreItemId,
    symbol: SymbolId,
    kind: CoreItemKind,
    visibility: Visibility,
    dependencies: Vec<CoreItemId>,
    source: CoreSourceRef,
}

enum CoreItemKind {
    Structure(CoreStructure),
    Mode(CoreMode),
    Attribute(CoreAttribute),
    Predicate(CorePredicate),
    Functor(CoreFunctor),
    Theorem(CoreTheorem),
    Registration(CoreRegistration),
    Reduction(CoreReduction),
    Algorithm(CoreAlgorithm),
}
```

### Core Terms and Formulas

```rust
enum CoreTerm {
    Var(CoreVarId),
    Const(SymbolId),
    Apply { functor: SymbolId, args: Vec<CoreTermId> },
    Select { selector: SymbolId, base: CoreTermId },
    SetEnum(Vec<CoreTermId>),
    Choice { binder: CoreBinder, predicate: CoreFormulaId },
    Error(ElaborationErrorId),
}

enum CoreFormula {
    True,
    False,
    Atom { predicate: SymbolId, args: Vec<CoreTermId> },
    TypePred { subject: CoreTermId, ty: CoreTypePredicate },
    Not(CoreFormulaId),
    And(Vec<CoreFormulaId>),
    Or(Vec<CoreFormulaId>),
    Implies { premise: CoreFormulaId, conclusion: CoreFormulaId },
    Iff { left: CoreFormulaId, right: CoreFormulaId },
    Forall { binders: Vec<CoreBinder>, body: CoreFormulaId },
    Exists { binders: Vec<CoreBinder>, body: CoreFormulaId },
    Error(ElaborationErrorId),
}
```

Core nodes use canonical symbols and explicit binders. Surface precedence, parentheses, and notation forms are not represented.

### Core Definition

```rust
struct CoreDefinition {
    symbol: SymbolId,
    params: Vec<CoreBinder>,
    body: DefinitionBody,
    expansion: ExpansionPolicy,
    obligations: Vec<ObligationSeedId>,
}

enum ExpansionPolicy {
    Opaque,
    Transparent,
    Reducible { reduction: RegistrationId },
    Computable { algorithm: SymbolId },
}
```

Expansion policy is semantic metadata. It does not force eager inlining during elaboration.

### Core Proof

```rust
struct CoreProof {
    theorem: SymbolId,
    proposition: CoreFormulaId,
    root: CoreProofNodeId,
    status: TheoremStatus,
}

enum CoreProofNode {
    IntroduceBinder { binder: CoreBinder, child: CoreProofNodeId },
    Assume { label: Option<LabelRef>, formula: CoreFormulaId, child: CoreProofNodeId },
    Step { label: Option<LabelRef>, formula: CoreFormulaId, justification: CoreJustification },
    Branch(Vec<CoreProofNodeId>),
    TerminalGoal(ObligationSeedId),
    Error(ElaborationErrorId),
}
```

`ObligationSeedId` is not a `VcId`. It identifies a source-mapped proof goal or correctness condition before VC generation. Phase 11 turns these seeds, together with `CoreIr` and `ControlFlowIr`, into concrete `VcId`s.

### Source Map

```rust
struct CoreSourceMap {
    item_sources: Map<CoreItemId, CoreSourceRef>,
    term_sources: Map<CoreTermId, CoreSourceRef>,
    formula_sources: Map<CoreFormulaId, CoreSourceRef>,
    obligation_sources: Map<ObligationSeedId, CoreSourceRef>,
}

struct CoreSourceRef {
    source_id: SourceId,
    resolved_node: AstNodeId,
    range: SourceRange,
    provenance: Vec<CoreProvenance>,
}
```

## Diagnostics

Elaboration diagnostics are boundary diagnostics between surface semantics and core logic.

| Diagnostic Class | Example |
|---|---|
| unresolved semantic input | overload site has no selected root, type prerequisite failed |
| unsupported lowering | surface construct has no core representation yet |
| malformed proof skeleton | proof body cannot be matched to theorem structure |
| invalid erasure | type or view metadata cannot be converted to predicates |
| algorithm shell | ghost/runtime metadata missing, contract lowering failed |
| source mapping | core node cannot be mapped to a source range |

Diagnostics should include:

- primary source span from `ResolvedTypedAst`;
- relevant semantic provenance, such as inserted `qua`, selected overload root, active refinements, or type fact;
- affected core item id when available;
- stable diagnostic code for LSP and artifacts.

## Incrementality and Reproducibility

Elaboration cache keys include:

- `ResolvedTypedAst` hash;
- `TypeFactTable` and `ResolutionTrace` hashes used by core erasure;
- dependency summary hashes;
- elaboration schema version;
- language edition.

Changes invalidate:

- the changed item's `CoreItem`;
- dependent proof skeletons and definition bodies;
- algorithm shells whose contracts or bodies changed;
- VC generation, ATP translation, and artifact outputs that consume changed core nodes.

Ordering must be deterministic:

- core items follow canonical item order after source-order diagnostics;
- binders preserve source order inside each item;
- conjunctions produced by type erasure use stable predicate ordering;
- source-map provenance lists are sorted by phase and source range;
- skipped/error nodes are serialized in AST traversal order.

## Alternatives Considered

### Elaborate Directly to Verification Conditions

Skipping `CoreIr` would reduce one representation, but VC generation, kernel checking, documentation metadata, and algorithm preparation need different projections. `CoreIr` provides a shared semantic base.

### Preserve Surface Syntax in Core

Keeping notation and source sugar in core would simplify source mapping, but it would force every later phase to understand parser-level constructs. The adopted design preserves source maps while normalizing core semantics.

### Eager Definition Inlining

Inlining all definitions during elaboration would simplify some later terms, but it changes proof search shape and can explode problem size. The adopted design records expansion policy and lets later phases choose controlled unfolding.

### Build Algorithm CFG During Elaboration

Building `ControlFlowIr` immediately would combine logical lowering with program analysis. The adopted design lowers algorithm shells in phase 9 and leaves CFG construction to phase 10.

## Adopted Approach

Mizar Evo lowers `ResolvedTypedAst` into a backend-neutral `CoreIr` in a dedicated elaboration phase.

The elaborator erases soft types into explicit predicates, records definition expansion boundaries, lowers terms and formulas to canonical nodes, converts proof structure into core proof trees, and records algorithm shells for later control-flow preparation. It preserves source maps and provenance so diagnostics, VC generation, ATP translation, kernel checking, and artifacts can all refer back to the same semantic source.

## Constraints and Assumptions

- Elaboration starts only after overload resolution has produced `ResolvedTypedAst`.
- Core nodes must use canonical symbol identities, not source spellings.
- Soft type annotations are erased only through explicit type predicates and assumptions.
- Elaboration must not run proof search or activate registrations.
- Failed semantic sites remain explicit error nodes or skipped items; they must not become valid core terms.
- Algorithm control-flow graph construction belongs to phase 10, not phase 9.
- `CoreIr` is backend-neutral and must not contain ATP-specific encoding decisions.
- Phase 9 creates obligation seeds only; concrete `VcId`s are assigned by phase 11.
