## Design Decisions

### Supported Formats

| Format | Target Backends | Primary Use |
|---|---|---|
| TPTP FOF/TFF | Vampire, E | First-order reasoning, equational reasoning, larger premise sets |
| SMT-LIB 2 | CVC5, Z3 | Arithmetic, SMT theories, equality mixed with theory reasoning |

### Backend-Neutral Problem Layer

`AtpProblem` is not concrete TPTP or SMT-LIB text. It is a backend-neutral representation produced from `VcIr`.

```rust
struct AtpProblem {
    problem_id: AtpProblemId,
    vc_id: VcId,
    logic_profile: LogicProfile,
    expected_result: ExpectedBackendResult,
    declarations: Vec<AtpDeclaration>,
    axioms: Vec<AtpFormula>,
    conjecture: AtpFormula,
    type_context: AtpTypeContext,
    properties: Vec<EncodedProperty>,
    symbol_map: AtpSymbolMap,
    provenance: Vec<AtpProvenance>,
}

enum ExpectedBackendResult {
    Unsat,
}

struct AtpProvenance {
    encoded_id: AtpFormulaId,
    source: AtpSourceRef,
}

enum AtpSourceRef {
    Premise(PremiseRef),
    LocalHypothesis(CoreFormulaId),
    TypeFact(TypeFactRef),
    Property(PropertyRef),
    RegistrationTrace(ResolutionTraceRef),
    DependencyArtifact(ArtifactProofRef),
}
```

One `VcIr` may produce multiple `AtpProblem`s when different backend profiles need different encodings.

### Encoding Strategy

- Soft types may be encoded as backend sorts only when the mapping is lossless for that backend profile. Attribute, mode, subtype, coercion, and intersection-like facts that are not represented by the sort must remain as guard predicates or explicit axioms.
- Mizar functors, predicates, modes, and attributes are mapped to backend-safe symbol names. `symbol_map` records the bijection from backend names to canonical Mizar `SymbolId`s or generated binder IDs.
- Premises come from local hypotheses, explicitly cited `by` references, and Mizar-side resolved facts.
- Properties such as commutativity, symmetry, reflexivity, and idempotence are encoded either as backend-native declarations or as axioms.
- Every encoded declaration, axiom, native property declaration, and guard introduced by translation must have `AtpProvenance` so kernel certificate checking can map backend evidence back to available Mizar-side facts.
- Backend-reported used-axiom metadata is advisory. Trusted `used_axioms` are extracted or validated during kernel certificate checking, then recorded in artifacts.

### Validity Checking Polarity

ATP problems encode proof obligations as validity checks.

- TPTP refutation backends receive premises as axioms and the goal as a conjecture or negated conjecture according to the selected TPTP profile. A successful proof must establish that the premises entail the goal.
- SMT-LIB backends assert the premises and the negation of the goal, then expect `unsat`. They must not assert the goal and accept `sat` as proof.
- `expected_result` records this contract explicitly. The only trusted successful result for the current interface is `Unsat`; backend-specific `sat`, `unknown`, timeout, or error results do not prove the VC.

### Property Encoding

| Property | Formula Shape | TPTP Strategy | SMT-LIB Strategy |
|---|---|---|---|
| `commutativity` | `F(a, b) = F(b, a)` | native AC support when available, otherwise axiom | quantified axiom |
| `symmetry` | `P(a, b) -> P(b, a)` | implication axiom | quantified axiom |
| `reflexivity` | `P(a, a)` | universal axiom | quantified axiom |
| `idempotence` | `F(a, a) = a` | equality axiom | equality axiom |
| `involutiveness` | `F(F(a)) = a` | equality axiom | equality axiom |
| `projectivity` | `F(F(a)) = F(a)` | equality axiom | equality axiom |
| `asymmetry` | `P(a, b) -> not P(b, a)` | implication plus negation axiom | implication axiom |
| `connectedness` | `a != b -> P(a, b) or P(b, a)` | implication axiom | implication axiom |
| `irreflexivity` | `not P(a, a)` | negated reflexive axiom | negated reflexive axiom |

Each encoded property records a `PropertyRef` and the defining symbol it applies to. If a backend-native declaration is used instead of an axiom, the declaration still receives an `encoded_id` and provenance entry so certificates and used-property reporting can refer to it.

## Alternatives Considered

1. **TPTP only**: simpler implementation, but loses SMT arithmetic and theory support.
2. **SMT-LIB only**: strong theory support, but weaker fit for general FOL and large equational searches.
3. **Dual format support**: more implementation work, but allows backend selection by obligation shape.

## Adopted Approach

Mizar Evo uses a backend-neutral `AtpProblem` layer and supports both TPTP and SMT-LIB concrete encoders.

## Interface Definitions

Concrete encoders consume `AtpProblem` and produce backend input:

- TPTP encoder: emits `fof(...)` / `tff(...)` declarations with stable names derived from `symbol_map` and provenance-preserving formula labels.
- SMT-LIB encoder: emits `(declare-...)`, premise `(assert ...)` forms, `(assert (not <goal>))`, and `(check-sat)`, with labels or comments sufficient to recover provenance from certificates and logs.

## Constraints and Assumptions

- Encoding does not need to be reversible.
- Certificate validation is performed by the kernel, not by trusting encoder output.
- Backend-specific extensions may be used only when the selected backend profile records them.
- Sort encoding must not erase soft-type facts that are required to justify the VC.
- Every backend-visible formula or native property declaration must be traceable through `AtpProvenance`.
- Backend-reported used axioms do not become artifact `used_axioms` until kernel checking validates them.
