# Architecture: Failure Semantics

> Canonical language: English. Japanese companion: [../ja/19.failure_semantics.md](../ja/19.failure_semantics.md).

## Purpose

This document fixes failure classification, propagation, and ordering for evo2.

Stable failure semantics are required for reproducible builds, reliable LSP behavior, and regression tests focused on unsoundness prevention.

## Design Decisions

### Failure Categories

The implementation must distinguish at least:

- `parse_error`;
- `resolve_error`;
- `type_error`;
- `overload_ambiguity`;
- `cluster_loop`;
- `atp_timeout`;
- `certificate_rejection`;
- `kernel_rejection`.

These categories are machine-readable and remain stable across wording changes.

Kernel and certificate failures use two layers of classification. `FailureCategory` records the phase-level class used for build propagation and diagnostic ordering. A more specific rejection reason, such as `malformed_certificate`, `invalid_substitution`, or `invalid_sat_proof`, is stored as stable structured detail on the failure record.

### Phase Ownership

Each failure is owned by the earliest phase that can soundly detect it.

Examples:

- invalid syntax is a parser failure, not a type failure;
- unresolved symbol is a resolver failure, not an ATP failure;
- ambiguous overload is an overload-resolution failure, not a kernel failure;
- malformed certificate is a certificate rejection;
- invalid resolution replay is a kernel rejection.

### Deterministic Error Ordering

Errors are emitted in canonical order:

1. source file by package id and module path;
2. source range;
3. phase order;
4. severity;
5. diagnostic code;
6. stable detail key.

Parallel verification completion order must not affect diagnostic order.

### Failure Propagation

A failure blocks only dependent work.

Rules:

- parse errors block semantic phases for the same file;
- interface failures block importers that require the failed interface;
- a failed VC does not hide independent VC failures in the same module;
- ATP timeout leaves an obligation open unless another accepted certificate is available;
- certificate and kernel rejection never become warnings under release policy.

Blocked work must be represented explicitly as blocked, not silently omitted.

### Soundness Priority

When classification is ambiguous, the implementation must choose the category that prevents unsound acceptance and preserves the most precise rejection boundary.

No fallback inference, hidden coercion, implicit cluster insertion, or unchecked ATP result may be used to turn a failure into success.

## Interface Definitions

```rust
enum FailureCategory {
    ParseError,
    ResolveError,
    TypeError,
    OverloadAmbiguity,
    ClusterLoop,
    AtpTimeout,
    CertificateRejection,
    KernelRejection,
}

struct FailureRecord {
    category: FailureCategory,
    phase: PipelinePhase,
    owner: SemanticOwnerId,
    source_range: Option<SourceRange>,
    diagnostic_code: DiagnosticCode,
    stable_detail_key: String,
    rejection_reason: Option<RejectionReason>,
}
```

## Constraints and Assumptions

- Diagnostic text may improve, but category, phase ownership, and stable code changes require compatibility review.
- Timeouts and resource exhaustion are non-acceptance outcomes.
- Failure snapshots are part of regression testing.
- Theorem proving success must never depend on ignoring an earlier deterministic failure.
