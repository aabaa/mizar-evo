# Architecture: Parallel Verification and Scheduling

> Canonical language: English. Japanese companion: [../ja/14.parallel_verification_and_scheduling.md](../ja/14.parallel_verification_and_scheduling.md).

## Purpose

This document defines how Mizar Evo schedules verification work across packages, modules, phases, verification conditions, ATP backends, and artifact updates.

It refines pipeline phases 0 and 10-15. The goal is to allow parallel verification and responsive IDE workflows without changing semantic results, diagnostic ordering, proof acceptance, or artifact reproducibility.

## Design Decisions

### Scheduling Is Separate From Semantics

The scheduler decides when work runs, not what work means.

It may:

- choose parallel execution order for independent modules and VCs;
- reuse cache entries when keys match;
- cancel obsolete work;
- enforce resource budgets;
- prioritize interactive files over background work.

It must not:

- change phase semantics;
- reorder source-visible diagnostics in emitted artifacts;
- accept the first nondeterministic ATP result without policy validation;
- make cache hits trusted proof evidence;
- expose partially-written artifacts.

All semantic outputs are sorted and committed according to canonical ordering after worker tasks complete.

### BuildPlan Produces the Initial Task Graph

Phase 0 resolves the workspace and produces a `BuildPlan`. The scheduler turns that plan into a task graph.

Task graph nodes include:

| Node Kind | Unit | Main Dependencies |
|---|---|---|
| `PackageResolve` | workspace/package | manifests, lockfile, registry metadata |
| `SourceLoad` | source file | package plan |
| `Frontend` | source file | source load, active lexical environment |
| `ModuleResolve` | source file | frontend, imports, dependency summaries |
| `CheckAndElaborate` | source file | resolved AST, symbol env, registration view |
| `VcGenerate` | source file | `CoreIr`, `ControlFlowIr`, resolution traces |
| `VcDischarge` | VC | `VcIr`, deterministic facts |
| `AtpSolve` | VC portfolio | ATP problem, prover policy, obligation budget |
| `BackendRun` | VC/backend profile | translated problem, backend availability, backend budget |
| `KernelCheck` | proof candidate | certificate, kernel context, dependency proof refs |
| `ArtifactCommit` | source file/package | verified metadata, logs, witnesses |

The graph is acyclic at the scheduling level. Cyclic language constructs, such as mutually recursive algorithms, are represented as grouped semantic nodes before VC generation.

The task graph serves both parallel scheduling and incremental invalidation. Task dependency edges are correctness edges, not just scheduling hints. A task may run in parallel only when every edge required for clean-build equivalence is satisfied; removing or weakening an edge for performance is a soundness bug.

### Module Dependencies Gate Semantic Phases

A module may be lexed and parsed before all imports finish, but semantic phases require the relevant dependency summaries.

Rules:

- frontend work may run for all changed files immediately;
- final name resolution waits for imported module summaries and active lexical environment data;
- type checking waits for visible symbol summaries and registration indexes;
- VC generation waits for completed elaboration and algorithm preparation;
- artifact commit waits for proof statuses and diagnostics for that source file.

This allows fast syntax feedback while preserving semantic correctness.

### VCs Are Fine-Grained Tasks

After canonical `VcIr` generation, obligations may be scheduled independently when their contexts are complete.

The scheduler may parallelize:

- deterministic pre-ATP discharge across VCs;
- ATP translation across open VCs;
- backend portfolio runs within one VC;
- kernel checking of independent certificates.

However, generated `VcId`s, obligation order, artifact order, and diagnostic order are fixed by VC generation, not by completion order.

`VcId` is the canonical within-snapshot ordering key. `ObligationAnchor` is the cross-edit tracking key for reuse candidates and must not replace `VcId` ordering.

### Resource Budgets Are Hierarchical

Verification uses hierarchical resource controls.

| Budget | Applies To | Examples |
|---|---|---|
| workspace budget | whole build | max worker threads, memory ceiling |
| package budget | package | dependency build concurrency |
| module budget | source file | max active VCs from one module |
| obligation budget | VC | ATP timeout, axiom budget, proof search policy |
| backend budget | child process | wall time, memory, stdout/stderr capture |

Portfolio execution consumes resources from both the VC budget and the global backend worker pool. This prevents one difficult obligation from starving the rest of the build.

### Cancellation Is Cooperative and Versioned

Cancellation is driven by build versions.

Every scheduled task belongs to a `BuildSessionId` and a source/dependency snapshot. When a newer edit or build request supersedes that snapshot, the scheduler cancels obsolete work.

Cancellation rules:

- pure in-process tasks check cancellation tokens at phase boundaries and safe checkpoints;
- child ATP processes are terminated by the backend manager;
- completed obsolete results are discarded unless their cache keys still match the new snapshot;
- artifact commit is not cancellable after the atomic write transaction begins;
- diagnostics from cancelled work are not published as current diagnostics.

LSP may keep stale diagnostics visible until a newer snapshot publishes replacement diagnostics, as defined by [12.diagnostics_and_lsp.md](./12.diagnostics_and_lsp.md).

### Deterministic Result Ordering

Parallel execution must produce the same externally visible result order as a clean sequential build.

Canonical ordering:

- packages by resolved dependency topological order, then package id;
- modules by package id and module path;
- items by source order and stable item id;
- VCs by canonical `VcId` order;
- backend attempts by configured backend priority, then backend profile id;
- diagnostics by source range, severity, phase order, and diagnostic code;
- artifacts by module path and manifest order.

Completion order is never used as an emitted order.

### Failure Propagation Is Bounded

A failure blocks dependent work but does not necessarily stop the whole build.

Examples:

- syntax failure blocks semantic phases for that file but not frontend work for other files;
- module interface failure blocks importers that need its summary;
- one failed VC marks its obligation open or failed but does not prevent independent VCs from reporting;
- artifact I/O failure blocks commit for affected output but does not retroactively invalidate proof results already computed in memory.

The scheduler records blocked tasks explicitly so diagnostics can distinguish "failed" from "not run because dependency failed".

## Scheduling Model

### Task State

```rust
enum TaskState {
    Pending,
    Ready,
    Running,
    Completed,
    Failed,
    Blocked,
    Cancelled,
    CacheHit,
}
```

`CacheHit` is a completed state for scheduling, but the reused record must still pass the cache validation rules in [11.artifact_and_incremental_build.md](./11.artifact_and_incremental_build.md).

`CacheHit` is a task state after validation, not proof evidence.

### Task Graph

```rust
struct VerificationTask {
    id: TaskId,
    kind: TaskKind,
    snapshot: BuildSnapshotId,
    inputs: Vec<TaskInput>,
    dependencies: Vec<TaskId>,
    priority: TaskPriority,
    cache_key: Option<CacheKey>,
    resource_class: ResourceClass,
}

struct TaskResult {
    task_id: TaskId,
    state: TaskState,
    output_refs: Vec<OutputRef>,
    diagnostics: Vec<Diagnostic>,
    timing: TimingSummary,
}
```

Tasks communicate through immutable output references. They do not mutate each other's IRs.

### Priority Policy

The default scheduler prioritizes:

1. tasks needed to unblock edited/open files;
2. dependency summaries needed by many downstream files;
3. cheap deterministic tasks before expensive ATP tasks;
4. tasks with cached dependencies ready;
5. background documentation/extraction work.

Priority affects latency, not observable semantics.

### Work Queues

Different work classes may use separate queues.

| Queue | Work |
|---|---|
| CPU local | parsing, resolving, type checking, VC generation |
| deterministic proof | pre-ATP discharge, computation with gas limits |
| ATP process | external backend child processes |
| kernel | certificate replay/checking |
| I/O commit | artifact and manifest writes |

Queue separation allows the scheduler to prevent unbounded ATP processes or artifact I/O from starving local semantic analysis.

## Incremental and LSP Workflows

### Batch Build

In batch mode, the scheduler attempts to complete the whole task graph.

It may continue after errors when doing so produces useful independent diagnostics. The final exit status is determined after all non-obsolete ready work finishes or becomes blocked.

### Watch / LSP Build

In watch or LSP mode:

- each edit creates a new source snapshot;
- stale tasks are cancelled or deprioritized;
- syntax and local semantic feedback is prioritized;
- background ATP and kernel tasks may continue only if their cache keys remain valid for the newest snapshot;
- result publication happens through immutable LSP snapshots.

The user should never see diagnostics from a newer source version mixed with artifacts from an incompatible older version.

### Cache-Aware Scheduling

Before scheduling a task, the scheduler asks the cache layer whether a valid entry exists.

Cache hits:

- skip task execution;
- still produce task results with output references;
- are sorted with ordinary completed results;
- may trigger downstream tasks immediately.

Cache misses are ordinary ready tasks. Cache lookup failure is diagnostic only when the cache is required by an explicit mode; otherwise it is treated as a miss.

Cache hit/miss timing must not affect proof status, diagnostics, artifact order, selected ATP evidence, or cache publication.

## ATP and Kernel Scheduling

### Portfolio Runs

An `AtpSolve` task is the scheduling subgraph for one VC portfolio. Its child `BackendRun` tasks execute individual backend profiles.

```rust
struct PortfolioTask {
    vc_id: VcId,
    candidates: Vec<BackendRunTask>,
    policy: ProverPolicy,
}
```

Backend candidates may run in parallel. The portfolio task completes only when:

- deterministic policy selection has an acceptable winner and no pending candidate can displace it under the same policy;
- every candidate has returned timeout/unknown/error or failed the required `KernelCheck`; or
- cancellation supersedes the VC snapshot.

Backend runs only collect candidate evidence. When policy requires kernel validation, the scheduler creates `KernelCheck` tasks for eligible certificates and treats the portfolio as complete only after the composed backend-plus-kernel subgraph has either selected an acceptable winner or proved that no acceptable winner remains.

The accepted candidate is selected deterministically by policy, not by raw completion order. For example, a later kernel-verifiable certificate may beat an earlier externally attested result when release policy requires certificates. If multiple acceptable certificates exist, selection uses canonical backend priority, certificate strength, and deterministic tie-breakers; runtime duration may be recorded as provenance but is not part of canonical proof identity.

### Kernel Check Ordering

Kernel checking may run in parallel for independent certificates. The kernel context for each check is immutable and content-addressed by dependency artifact hashes.

Kernel results are committed to artifacts in `VcId` order. A certificate rejection for one VC does not erase accepted results for other VCs.

## Artifact Commit

Artifact commit is the serialization boundary between parallel internal execution and stable external output.

Rules:

- only one commit transaction writes a given module artifact at a time;
- package manifest updates are serialized after module artifacts are ready;
- committed artifact content is sorted canonically;
- obsolete build sessions cannot overwrite artifacts from a newer session;
- commit records include task graph provenance sufficient for post-mortem debugging.
- worker completion order must not determine proof status, diagnostics, artifact order, selected ATP evidence, or cache publication.

If commit fails, the scheduler reports artifact I/O diagnostics and leaves previous complete artifacts intact.

### Build Scheduler

```rust
trait BuildScheduler {
    fn submit(&self, request: BuildRequest) -> BuildSessionId;
```

```rust
struct BuildRequest {
    workspace_root: PathBuf,

    Batch,
    Check,
    Watch,
    Lsp,

```

### Build Events

```rust
enum BuildEvent {
    TaskStarted(TaskId),
    TaskFinished(TaskId, TaskState),

```

Build events are for progress reporting and LSP integration. They are not the canonical ordering used in artifacts.

## Alternatives Considered

### Run Whole Pipeline Sequentially Per File

This is easy to reason about but wastes parallelism and delays feedback for independent modules. The adopted scheduler keeps phase dependencies explicit while allowing independent work to run concurrently.

### Fully Demand-Driven Query Engine

A query engine can provide excellent incremental behavior, but it makes reproducibility and artifact commit boundaries harder to see. The adopted design can use query-style internals while exposing an explicit task graph and deterministic commit model.

### Let ATP Portfolio Completion Order Decide Winners

This minimizes latency but makes proof status depend on machine load and process timing. The adopted design records completion timing but selects winners through policy and, when required, separate kernel acceptance.

### Cancel by Killing Worker Threads

Hard cancellation risks corrupting shared state and partial outputs. The adopted design uses cooperative cancellation for in-process work and process termination only for external backends.

## Adopted Approach

Mizar Evo uses an explicit task graph scheduler with immutable task outputs, hierarchical resource budgets, cooperative cancellation, cache-aware execution, and deterministic result ordering.

Parallelism is allowed wherever dependencies permit it, but artifact emission, diagnostics, VC order, and proof acceptance are determined by canonical IDs and policy rather than worker completion order.

## Constraints and Assumptions

- Parallel scheduling must not change verified semantics or emitted artifact ordering.
- All task outputs are immutable once published to dependent tasks.
- Completion order is never used as semantic or artifact order.
- Cancellation must not leave partially-written artifacts.
- External ATP processes must be resource-limited and terminable.
- Cache hits must satisfy the same validation rules as in a clean build.
- Incremental sequential, incremental parallel, clean sequential, and clean parallel builds for the same `BuildSnapshot` and verifier policy must agree on proof acceptance, published artifacts, interface hashes, dependency-facing summaries, and canonical diagnostics.
- The scheduler may choose coarser task granularity initially, but must preserve the architectural dependency boundaries.
