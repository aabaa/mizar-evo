# Architecture: Cluster and Registration Trace Format

> Canonical language: English. Japanese companion: [../ja/17.cluster_trace_format.md](../ja/17.cluster_trace_format.md).

## Purpose

This document defines the canonical replayable `ResolutionTrace` artifact schema for cluster expansion and reduction applications.

Cluster search and reduction rule selection are outside the kernel. The kernel and artifact consumers receive only explicit traces that explain which cluster or reduction steps were used.

## Design Decisions

### Cluster Step

Each cluster expansion step is represented as:

```text
ClusterStep
  source_type
  applied_cluster
  generated_attribute
  generated_type
  dependency
  source_range
```

`dependency` records the theorem, definition, registration, mode, attribute, or prior cluster step that justifies the expansion.

### Reduction Step

Each reduction expansion step is represented as:

```text
ReductionStep
  applied_reduction
  rule_fqn
  enclosing_term_before
  redex_path
  source_redex
  target_term
  substitution
  discharged_guards
  rule_view
  selection_key
  source_range
```

The kernel-replay fields are `applied_reduction`, `rule_fqn`, `source_redex`, `target_term`, `substitution`, and `discharged_guards`. They are enough to verify the local `LHS -> RHS` instance against the already accepted `reducibility` proof and to report the exact labelled rule used by automatic normalization. The strategy-audit fields are `enclosing_term_before`, `redex_path`, `rule_view`, and `selection_key`; they let deterministic artifact validators recompute that the emitted step was the leftmost-innermost redex selected by the §17.6.4 rule ordering. The minimum kernel does not perform rule search.

### Cluster Chain Trace

Transitive cluster expansions are stored as explicit ordered chains.

```text
A -> B
B -> C
C -> D
```

The trace must preserve every intermediate step. Artifacts must not collapse the chain to `A -> D` unless the collapsed result also stores the original replay steps by content-addressed reference.

### Deterministic Traversal

Cluster traversal order is fixed by:

1. source type canonical id;
2. cluster origin module path;
3. declaration source order;
4. generated attribute canonical id;
5. registration fingerprint.

Worker completion order, hash-map iteration order, and cache insertion order must never affect selected traces.

Reduction normalisation order is fixed by §17.6.4. A reduction step records the active rule-view fingerprint and selected rule key so that deterministic validators can audit the strategy without relying on registration order, import order, or cache insertion order.

### Cluster Loop Detection

Cluster loop detection is deterministic.

Rules:

- cycles in the active expansion stack are rejected as `cluster_loop`;
- repeated already-derived facts are ignored only after their canonical fingerprint matches;
- maximum expansion depth and maximum generated fact count are verifier settings included in cache keys;
- exceeding the maximum is a bounded failure, not an opportunity to silently truncate required facts.

These expansion-depth and generated-fact bounds apply only to cluster expansion. Reduction applications are not capped by cluster expansion depth; their termination is guaranteed by the simplification order checked at registration time (§17.6.4).

### Trace Consumers

Trace consumers may:

- replay cluster derivations for diagnostics and artifact validation;
- replay local reduction instances for diagnostics and artifact validation;
- include generated type facts in VCs;
- fingerprint dependency slices;
- display `@show_resolution` explanations.

Trace consumers must not infer additional clusters or reductions that are absent from the trace.

## Interface Definitions

```rust
struct ResolutionTrace {
    schema_version: SchemaVersion,
    source_file: SourceFileId,
    steps: Vec<ResolutionTraceStep>,
    derived_facts: Vec<ClusterFact>,
    traversal_profile: ClusterTraversalProfile,
}

enum ResolutionTraceStep {
    Cluster(ClusterStep),
    Reduction(ReductionStep),
}

struct ClusterStep {
    source_type: TypeFingerprint,
    applied_cluster: ClusterFingerprint,
    generated_attribute: AttributeFingerprint,
    generated_type: TypeFingerprint,
    dependency: ClusterDependencyRef,
    source_range: SourceRange,
}

struct ReductionStep {
    applied_reduction: ReductionFingerprint,
    rule_fqn: Fqn,
    enclosing_term_before: TermFingerprint,
    redex_path: RedexPath,
    source_redex: TermFingerprint,
    target_term: TermFingerprint,
    substitution: Vec<PatternBinding>,
    discharged_guards: Vec<GuardEvidenceRef>,
    rule_view: RuleViewFingerprint,
    selection_key: ReductionSelectionKey,
    source_range: SourceRange,
}
```

## Constraints and Assumptions

- Resolution traces are semantic artifacts and must be included in deterministic hashes.
- Registrations whose `coherence` or `reducibility` proof is not accepted are invisible to traces.
- Hidden transitive expansion, implicit cluster insertion, and unrecorded reduction steps are forbidden.
- Fail tests must include infinite chains, cyclic registrations, unintended coercion, hidden transitive expansion, invalid reduction substitutions, and mismatched reduction strategy-audit keys.
