# Architecture: Cluster Trace Format

> Canonical language: English. Japanese companion: [../ja/17.cluster_trace_format.md](../ja/17.cluster_trace_format.md).

## Purpose

This document defines the replayable trace format for cluster and registration expansion.

Cluster search is outside the kernel. The kernel and artifact consumers receive only explicit traces that explain which cluster or registration steps were used.

## Design Decisions

### Cluster Step

Each cluster expansion step is represented as:

```text
ClusterStep
  source_type
  applied_cluster
  generated_attribute
  generated_type
  dependency
```

`dependency` records the theorem, definition, registration, mode, attribute, or prior cluster step that justifies the expansion.

### Chain Trace

Transitive expansions are stored as explicit ordered chains.

```text
A -> B
B -> C
C -> D
```

The trace must preserve every intermediate step. Artifacts must not collapse the chain to `A -> D` unless the collapsed result also stores the original replay steps by content-addressed reference.

### Deterministic Traversal

Cluster traversal order is fixed by:

1. source type canonical id;
2. cluster origin module path;
3. declaration source order;
4. generated attribute canonical id;
5. registration fingerprint.

Worker completion order, hash-map iteration order, and cache insertion order must never affect selected traces.

### Loop Detection

Loop detection is deterministic.

Rules:

- cycles in the active expansion stack are rejected as `cluster_loop`;
- repeated already-derived facts are ignored only after their canonical fingerprint matches;
- maximum expansion depth and maximum generated fact count are verifier settings included in cache keys;
- exceeding the maximum is a bounded failure, not an opportunity to silently truncate required facts.

### Trace Consumers

Trace consumers may:

- replay cluster derivations for diagnostics and artifact validation;
- include generated type facts in VCs;
- fingerprint dependency slices;
- display `@show_resolution` explanations.

Trace consumers must not infer additional clusters that are absent from the trace.

## Interface Definitions

```rust
struct ClusterTrace {
    source_item: ItemId,
    steps: Vec<ClusterStep>,
    derived_facts: Vec<ClusterFact>,
    traversal_profile: ClusterTraversalProfile,
}

struct ClusterStep {
    source_type: TypeFingerprint,
    applied_cluster: ClusterFingerprint,
    generated_attribute: AttributeFingerprint,
    generated_type: TypeFingerprint,
    dependency: ClusterDependencyRef,
}
```

## Constraints and Assumptions

- Cluster traces are semantic artifacts and must be included in deterministic hashes.
- Registrations whose coherence proof is not accepted are invisible to cluster traces.
- Hidden transitive expansion and implicit cluster insertion are forbidden.
- Fail tests must include infinite chains, cyclic registrations, unintended coercion, and hidden transitive expansion.
