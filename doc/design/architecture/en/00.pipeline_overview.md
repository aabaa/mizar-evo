## Purpose

This document defines the end-to-end Mizar Evo compiler and verifier pipeline, from `.miz` source files to verified artifacts, documentation, and extracted code.

It is the parent document for the architecture directory. More focused architecture documents refine individual phases or cross-cutting boundaries.

## Design Decisions

### Pipeline as the Primary Architectural Boundary

| Phase | Input | Output | Main Responsibility | Failure Mode |
|---|---|---|---|---|
| 0. Package / Workspace Resolution | `mizar.pkg`, `mizar.workspace`, `mizar.lock` | `BuildPlan` | Resolve package graph, dependency versions, edition, and verifier settings | dependency / manifest errors |
| 1. Source Loading / Preprocessing | `.miz` source files, `BuildPlan` | `SourceUnit`, `PreprocessedSource` | Validate source, split comments/doc comments, pre-scan imports | lexical precondition errors |
| 2. Lexing | `PreprocessedSource`, active lexicon seed | `TokenStream` | Tokenize reserved words, reserved symbols, user symbols, numerals, and contextual string literals | lexical errors |
| 3. Parsing | `TokenStream` | `SurfaceAst` | Parse declarations, statements, terms, formulas, proofs, algorithms, and annotations | syntax errors |
| 4. Module / Name Resolution | `SurfaceAst`, import graph, symbol indexes | `ResolvedAst` | Resolve namespaces, imports/exports, labels, qualified names, and selector ambiguity | unresolved / ambiguous names |
| 5. Signature Collection | `ResolvedAst` | `SymbolEnv` | Register structs, modes, attributes, predicates, functors, theorems, registrations, and templates | duplicate / illegal declarations |
| 6. Type Checking / Type Inference | `ResolvedAst`, `SymbolEnv` | `TypedAst` | Check soft types, coercions, sethood, and narrowing obligations | type errors |
| 7. Cluster / Registration Resolution | `TypedAst`, registration indexes | `ResolutionTrace`, enriched type facts | Propagate attributes and record replayable registration derivations | inconsistent / missing registrations |
| 8. Overload Resolution | enriched typed AST, candidate sets | `ResolvedTypedAst` | Select overload winners and insert `qua` coercions | overload errors |
| 9. Elaboration / Desugaring | `ResolvedTypedAst` | `CoreIr` | Lower surface syntax to logical core representation | elaboration errors |
| 10. Algorithm Verification Preparation | `CoreIr` | `ControlFlowIr` | Prepare contracts, invariants, ghost rules, and termination measures | algorithm verification errors |
| 11. VC Generation | `CoreIr`, `ControlFlowIr` | `VcIr` | Generate proof obligations from theorems, proof steps, assertions, contracts, and registrations | logical well-formedness errors |
| 12. Pre-ATP Discharge | `VcIr`, local facts | discharged/open VCs | Solve deterministic or computation-based obligations before ATP | computation limit / unsupported forms |
| 13. ATP Translation / Dispatch | open VCs, cited premises, proof hints | `AtpProblem`, ATP results, certificates | Encode problems and run backend provers | timeout / unknown / backend errors |
| 14. Kernel Certificate Check | proof certificates, kernel context | verified proof status | Independently validate proof evidence | certificate rejection |
| 15. Artifact Emit / Cache Update | verified AST/IR, metadata, logs | `VerifiedArtifact`, logs, witnesses, cache | Emit reproducible artifacts and update incremental cache | artifact I/O errors |
| 16. Documentation / Extraction | verified artifacts, doc comments | rendered docs, extracted code | Generate documentation and extracted runtime code | documentation / extraction errors |

### Ordering Rules

- Language semantics must be resolved before ATP dispatch.
- Name resolution, type inference, cluster resolution, overload resolution, and template instantiation are Mizar-side responsibilities.
- ATP backends receive only well-typed, name-resolved, elaborated obligations.
- ATP results are accepted only after kernel certificate checking, except where policy explicitly permits externally attested evidence.

### IR Layering

| IR | Produced By | Consumed By | Role |
|---|---|---|---|
| `SourceUnit` | Source Loading | Preprocessing, diagnostics | file path, package, source text, and line map |
| `PreprocessedSource` | Preprocessing | Lexing | lexical text, comments, doc comments, and import stubs |
| `TokenStream` | Lexing | Parsing | source-spanned token sequence |
| `SurfaceAst` | Parsing | Name Resolution | source-shaped syntax tree |
| `ResolvedAst` | Name Resolution | Signature Collection, Type Checking | AST with resolved names, labels, imports, and namespace references |
| `SymbolEnv` | Signature Collection | Resolver and checker phases | visible symbol and signature environment |
| `TypedAst` | Type Checking | Registration and overload phases | inferred types, coercion candidates, and type facts |
| `ResolutionTrace` | Registration Resolution | Kernel, diagnostics, artifacts | replayable cluster/registration derivation trace |
| `ResolvedTypedAst` | Cluster / Overload Resolution | Elaboration, LSP metadata | final surface semantics |
| `CoreIr` | Elaboration | VC Generation, Kernel | desugared logical representation |
| `ControlFlowIr` | Algorithm Preparation | VC Generation, Extraction | algorithm verification/extraction representation |
| `VcIr` | VC Generation | Pre-ATP, ATP Translation | prover-independent verification condition |
| `AtpProblem` | ATP Translation | ATP Backends | backend-neutral prover problem |
| `ProofCertificate` | ATP Backends | Kernel Certificate Check | backend proof evidence |
| `VerifiedArtifact` | Artifact Emit | LSP, docs, downstream packages | stable external metadata projection |

Detailed IR ownership is defined in [ir_layers.md](./ir_layers.md).

### Diagnostics as Phase Outputs

Diagnostics are structured outputs, not incidental side effects.

- Diagnostic codes follow [doc/spec/en/22.error_handling_and_diagnostics.md](../../../spec/en/22.error_handling_and_diagnostics.md).
- Source spans are grounded in `SourceUnit` line maps.
- Name, type, and proof diagnostics should include secondary spans and machine-readable details when possible.
- Informational annotations such as `@show_type`, `@show_resolution`, `@show_thesis`, and `@eval` feed both diagnostics and metadata artifacts.

### Incrementality and Reproducibility

- Cache keys include source hashes, dependency hashes, toolchain version, language edition, and verifier settings.
- Import dependency graphs determine reverse-dependency invalidation.
- Cluster and registration indexes are updated per changed source contribution.
- Artifact output uses canonical ordering for reproducibility.
- ATP nondeterminism is recorded through seeds, timeout settings, backend metadata, certificate metadata, and externally attested status.

## Alternatives Considered

### Frontend / Middle-End / Backend Only

A traditional three-part compiler split is too coarse for Mizar Evo because cluster resolution, overload resolution, ATP dispatch, certificate checking, and artifact emission have different trust and cache boundaries.

### Existing Build Lifecycle Table Only

The lifecycle in the language specification is useful for users, but implementation design needs more precise boundaries inside type checking and proof processing.

### Fully Demand-Driven Pipeline

Demand-driven analysis is attractive for LSP workflows, but a batch pipeline gives a clearer correctness and reproducibility baseline. Demand-driven execution can be added later within these phase boundaries.

## Adopted Approach

Mizar Evo uses an explicit multi-phase pipeline. This makes semantic ownership, trust boundaries, diagnostics, cache invalidation, and artifact generation easier to reason about.

struct BuildPlan {

    dependency_graph: DependencyGraph,

    kind: VcKind,

    schema_version: String,
    source_file: PathBuf,
    source_hash: Hash,

    expressions: Vec<ExpressionMetadata>,
    obligations: Vec<ObligationMetadata>,
    proof_witnesses: Vec<ProofWitnessRef>,

## Constraints and Assumptions

- A `.miz` source file is the module unit.
- Code regions are ASCII; comments and annotations may contain Unicode.
- Import resolution affects the active lexicon and must be available before final tokenization.
- Type checking, cluster resolution, and overload resolution are Mizar-side responsibilities.
- ATP backends receive only well-typed, name-resolved, elaborated obligations.
- Artifacts must be reproducible for identical source, lockfile, toolchain, verifier settings, and accepted certificates.
