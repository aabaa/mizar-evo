## Design Decisions

### Supported Backends

| Backend | Type | Input Format | Candidate Evidence / Diagnostic Proof Output | Priority |
|---|---|---|---|---|
| Vampire | ATP | TPTP | formula/substitution candidate evidence; TSTP diagnostic only | Primary |
| E | ATP | TPTP | formula/substitution candidate evidence; TSTP diagnostic only | Primary |
| CVC5 | SMT | SMT-LIB | formula/substitution candidate evidence; LFSC/Alethe diagnostic only | Primary for theory-heavy goals |
| Z3 | SMT | SMT-LIB | externally attested proof log; diagnostic only | Optional |

Backends that cannot produce formula/substitution evidence candidates sufficient
for kernel checking may still be run in development profiles for diagnostics or
externally attested evidence. They cannot win a release build when
`require_kernel_certificates` (historical name for requiring kernel evidence
acceptance) or an equivalent policy is active. In `solver: auto`,
kernel-verifiable evidence is preferred over externally attested evidence even
when the externally attested backend returns first.

### Process Model

Backends are launched as child processes.

- Input is passed via stdin or a temporary problem file.
- Stdout and stderr are captured for logs.
- Timeout is configured per obligation; the default is `[verifier].atp_timeout` in `mizar.pkg`.
- Crashed backends are reported as backend errors and do not crash the verifier.
- Process exit status, command-line flags, working directory policy, temporary file policy, stdout hash, and stderr hash are recorded for reproducibility and post-mortem diagnostics.

### Portfolio Execution

When the selected solver is `auto`, Mizar Evo may launch multiple backends in parallel. Phase 13 collects candidate evidence from backends; phase 14 decides whether that evidence is accepted by the kernel. A portfolio winner is selected by deterministic policy ranking among acceptable evidence, with kernel-verified evidence preferred over externally attested results. Raw completion order may be recorded as provenance but must not select the canonical winner. Remaining processes are terminated only after the selected winner is final under the policy, or after the policy decides that no remaining candidate can produce an acceptable result.

For reproducibility, portfolio runs must record:

- backend names and versions;
- concrete input hash;
- selected backend profile and command-line flags;
- timeout and resource limits;
- random seed when applicable;
- exit status, stdout hash, and stderr hash;
- candidate evidence status.

### Result Classification

```rust
enum ATPResult {
    Proved {
        evidence: CandidateKernelEvidence,
        observed_result: BackendResultKind,
    },
    Counterexample(CounterexampleEvidence),
    Timeout,
    Unknown,
    Error(ATPError),
}
```

`Proved` means the backend result matched `EncodedProblem.expected_result` and the backend produced candidate formula/substitution evidence. It does not mean the proof has been accepted by the kernel.

For the current ATP interface, the expected successful result is `Unsat`: SMT-LIB backends must report `unsat` for `premises and not goal`, and TPTP backends must report the corresponding refutation or theorem status for the selected profile. `sat`, `counter-satisfiable`, `unknown`, timeout, process success without a proof payload, and parse failures are never classified as `Proved`.

`Counterexample` is diagnostic evidence, not proof acceptance. It may be used for failure triage only when the backend model is compatible with the encoding profile and can be mapped through `AtpProvenance` back to source-level terms. Otherwise the result is treated as `Unknown` or `Error`.

## Alternatives Considered

1. **Library linking**: lower process overhead, but harder licensing and version isolation.
2. **Long-running prover daemon**: lower startup cost, but more complicated resource management.
3. **Child process execution**: simple isolation and version management, with acceptable overhead.

## Adopted Approach

Mizar Evo uses child process execution with optional portfolio parallelism.

## Interface Definitions

```rust
trait ATPBackend {
    fn name(&self) -> &str;
    fn solve(&self, problem: EncodedProblem, timeout: Duration) -> Result<ATPResult, ATPError>;
}
```

`EncodedProblem` carries concrete backend input plus the `AtpProblem` metadata needed after execution:

- expected backend result and logic profile;
- concrete input hash and selected backend profile;
- symbol map, formula labels, and `AtpProvenance`;
- command-line flags, resource limits, and random seed;
- log metadata needed to connect stdout/stderr, candidate evidence, diagnostic proof objects, and artifacts.

Candidate evidence production must preserve formula labels and backend symbol names so phase 14 can validate formula/substitution evidence against the original `AtpProvenance`. Backend proof objects and logs may support diagnostics or migration audits, but they are not trusted acceptance material.

## Constraints and Assumptions

- Backend binaries are expected to be available on `PATH` or configured explicitly.
- Backend versions are recorded in verifier artifacts and ATP logs.
- Backend crashes are handled gracefully.
- Kernel evidence checking is the trusted acceptance boundary.
- `Proved` is emitted only when the observed backend result matches `EncodedProblem.expected_result` and candidate formula/substitution evidence is present.
- Externally attested evidence is policy-controlled and must remain distinct from kernel-verified proof status.
- Counterexample results support diagnostics but do not establish trusted proof status.
