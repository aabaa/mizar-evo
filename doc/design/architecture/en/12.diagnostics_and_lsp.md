# Architecture: Diagnostics and LSP

> Canonical language: English. Japanese companion: [../ja/12.diagnostics_and_lsp.md](../ja/12.diagnostics_and_lsp.md).

## Purpose

This document defines how diagnostics, source metadata, explanation data, and LSP responses flow through Mizar Evo.

Diagnostics are produced across all pipeline phases, but phase 15 is where they become stable artifact data for tools. The LSP server consumes those artifacts and a small set of query services to provide editor feedback without depending on compiler-internal IR layouts.

## Design Decisions

### Diagnostics Are Structured Phase Outputs

Every pipeline phase emits structured `Diagnostic` records instead of formatted strings.

The formatter for CLI output, the artifact emitter, and the LSP server all consume the same canonical record. This prevents drift where the CLI explains an error one way, while the IDE or artifact metadata sees a different error.

### Diagnostic Identity

Each diagnostic has two identifiers.

| Identifier | Scope | Purpose |
|---|---|---|
| `DiagnosticCode` | stable across releases | human-facing and tool-facing classification, e.g. `E0201` |
| `DiagnosticId` | one build snapshot | links LSP diagnostics to explanation queries and code actions |

`DiagnosticCode` is assigned permanently by the diagnostic registry. `DiagnosticId` is deterministic within a build snapshot and is derived from source file, code, primary span, phase, and a stable per-span index. Tools must not persist `DiagnosticId` across unrelated build snapshots.

### Source Spans Are Centralized

Diagnostics do not store ad hoc line/column calculations. They reference `SourceRange` values backed by the `SourceUnit` line map.

Rules:

- primary span is the location the user should edit first;
- secondary spans identify definitions, candidates, generated obligations, or imported facts;
- generated obligations carry `GeneratedFrom` provenance so diagnostics can point to source-level constructs;
- LSP UTF-16 positions are computed only at the LSP boundary from the source text snapshot.

This keeps compiler diagnostics independent of editor protocol encoding details.

### Diagnostics Survive Partial Failure

The compiler should report as many independent diagnostics as possible without inventing semantic facts after a fatal phase failure.

Phase outputs therefore distinguish:

- `fatal`: downstream semantic phases cannot trust the phase output;
- `blocking`: this item or obligation cannot proceed, but other items may continue;
- `recoverable`: later phases may continue with degraded metadata.

For example, parser recovery may allow name resolution on recovered syntax, but a failed import graph blocks semantic checking for the affected module and its reverse dependencies.

### Explanation Data Is Lazy and Bounded

Rich explanations for type inference, cluster resolution, overload resolution, and proof failure can be much larger than normal diagnostics.

The default diagnostic record stores compact summaries and references:

- explanation kind;
- relevant `expr_id`, `vc_id`, or `diagnostic_id`;
- path to an explanation artifact when one was emitted;
- bounded preview text for CLI and editor display.

The LSP server and `mizar explain` resolve detailed explanations on demand. Bulk extraction of every trace is avoided.

### LSP Reads Stable Snapshots

The LSP server reads a consistent build snapshot, not mutable compiler state.

The snapshot consists of:

- artifact manifest;
- `*.mizir.json` files;
- diagnostic explanation artifacts;
- `missing_facts.json` when present;
- `cluster-db/` and import-scoped views when explanation queries need them;
- source text snapshots for open editor buffers.

When a new build completes, the server atomically switches from the previous snapshot to the new snapshot. In-flight requests either use the old snapshot or the new snapshot, never a mixture.

### Open Buffers Override Disk Source

For open editor buffers, LSP requests use the editor-provided text as the current source snapshot. Build artifacts may still reflect the last completed verification pass on disk.

The server therefore labels responses by freshness:

| Freshness | Meaning |
|---|---|
| `current` | response was computed from the current open-buffer text |
| `stale_artifact` | response comes from the last verified artifact and may not match unsaved text |
| `pending` | verification is scheduled or running |

Diagnostics from stale artifacts may remain visible, but the server should avoid offering destructive code actions when the target range no longer matches the current buffer text.

## Diagnostic Pipeline

### Producers

Each phase contributes diagnostics with phase-specific metadata.

| Phase Area | Typical Diagnostics | Extra Metadata |
|---|---|---|
| frontend | malformed tokens, syntax recovery, missing `end` | expected tokens, recovery point |
| resolver | unresolved names, ambiguous imports, duplicate symbols | candidate symbols, import path |
| checker | type mismatch, sethood failure, cluster failure | inferred/expected types, resolution trace ref |
| overload | ambiguous overload, missing `qua` | candidates, selected policy, explanation ref |
| VC / proof | open obligation, contract failure | `vc_id`, goal summary, premise refs |
| ATP / kernel | timeout, backend error, certificate rejection | backend profile, certificate status, witness ref |
| artifact / build | stale artifact, cache/schema mismatch | artifact path, schema version, cache key summary |

Diagnostics emitted by later phases may reference earlier metadata, but they must still be expressible as standalone records.

### Aggregation

Diagnostics are aggregated per source file and per build snapshot.

The aggregator:

- normalizes paths to workspace-relative paths;
- assigns deterministic `DiagnosticId`s;
- deduplicates identical diagnostics produced through multiple paths;
- sorts diagnostics by source range, severity, phase order, and code;
- attaches build freshness and phase status;
- separates user-facing diagnostics from compiler debug notes.

Deduplication must not merge diagnostics that have different primary spans or different actionable fixes.

### Severity Mapping

The compiler severity maps to CLI and LSP severities as follows.

| Compiler Severity | CLI | LSP |
|---|---|---|
| `Error` | `error[...]` | `Error` |
| `Warning` | `warning[...]` | `Warning` |
| `Info` | `info[...]` | `Information` |
| `Hint` | `hint[...]` or inline note | `Hint` |

`I`-series diagnostics requested by annotations such as `@show_type` are LSP `Hint` or `Information` depending on whether they are inline display hints or explicit output messages.

### Fix Suggestions

Fix suggestions are stored as structured edits when the compiler can compute them safely.

```rust
struct FixSuggestion {
    title: String,
    confidence: FixConfidence,
    edits: Vec<TextEdit>,
    command: Option<DiagnosticCommand>,
    applicability: Applicability,
}
```

`help:` text is derived from this structure for CLI output. The LSP server converts the same structure into code actions.

Suggestions that depend on current buffer contents must verify that the target text still matches before applying edits.

## LSP Model

### Responsibilities

The LSP server owns editor protocol concerns.

It is responsible for:

- document synchronization and open-buffer snapshots;
- converting source ranges to LSP positions;
- publishing diagnostics;
- serving hover, go-to-definition, references, document symbols, and semantic tokens;
- serving code actions and explanation requests;
- scheduling verification or asking the build service to schedule it.

It is not responsible for:

- accepting proof results;
- reimplementing type checking or ATP reasoning;
- interpreting raw compiler IR;
- mutating build artifacts directly.

### Request Sources

| LSP Feature | Primary Data Source |
|---|---|
| diagnostics | aggregated `Diagnostic` records from the latest build snapshot |
| hover | `ExpressionMetadata`, symbol summaries, active thesis metadata |
| go-to-definition | resolved symbol metadata and export summaries |
| references | symbol index built from artifacts and source snapshots |
| document symbols | parsed outline for current buffer, enriched by artifacts when fresh |
| code actions | `FixSuggestion`, `missing_facts.json`, explanation refs |
| semantic tokens | lexer/parser output for current buffer plus resolved symbol classes when fresh |
| inlay hints | inferred types, inserted coercions, active thesis, proof status |
| explain commands | explanation artifacts and bounded query services |

When fresh semantic artifacts are unavailable, the server should degrade gracefully to syntax-only features.

### Incremental Update Flow

1. Editor sends a document change.
2. LSP updates its open-buffer snapshot and publishes lightweight syntax diagnostics when available.
3. Build scheduler computes the affected files using the dependency graph.
4. Compiler phases run and emit artifacts for the completed build snapshot.
5. LSP swaps to the new snapshot and republishes diagnostics for affected files.
6. Stale code actions and explanation handles from the previous snapshot are invalidated.

The server may debounce rapid edits, but it must never publish diagnostics for a newer document version using ranges computed for an older version without checking the version.

### Code Actions

Code actions fall into three classes.

| Class | Examples | Safety Rule |
|---|---|---|
| local text edits | insert `qua`, add missing annotation, replace typo | require range/text match |
| artifact-assisted edits | add lemma from `missing_facts`, refine citation | require matching artifact and source hash |
| commands | run `mizar explain`, show VC panel, re-run verification | no source edit unless command later validates |

The server should prefer small explicit edits over broad rewrites. For proof suggestions, it must distinguish backend-derived candidates from kernel-verified `used_axioms`.

## Interface Definitions

### Diagnostic Record

```rust
struct Diagnostic {
    id: DiagnosticId,
    code: DiagnosticCode,
    semantic_name: String,
    severity: DiagnosticSeverity,
    phase: PipelinePhase,
    message: String,
    primary_span: SourceRange,
    secondary_spans: Vec<DiagnosticSpan>,
    notes: Vec<DiagnosticNote>,
    fixes: Vec<FixSuggestion>,
    explanation: Option<ExplanationRef>,
    related_ids: Vec<DiagnosticId>,
}
```

`message` is concise and stable enough for display, but tools should key on `code`, not message text.

### LSP Snapshot

```rust
struct LspSnapshot {
    snapshot_id: SnapshotId,
    artifact_manifest: ArtifactManifestRef,
    source_versions: Vec<SourceVersion>,
    diagnostics: DiagnosticIndex,
    expression_index: ExpressionIndex,
    symbol_index: SymbolIndex,
    explanation_index: ExplanationIndex,
    freshness: SnapshotFreshness,
}
```

The snapshot is immutable once published to request handlers.

### Explanation Reference

```rust
struct ExplanationRef {
    kind: ExplanationKind,
    diagnostic_id: Option<DiagnosticId>,
    expr_id: Option<ExprId>,
    vc_id: Option<VcId>,
    artifact_path: Option<PathBuf>,
    summary_hash: Hash,
}
```

Explanation references are handles, not embedded traces. Large traces remain in dedicated artifacts or query services.

## Alternatives Considered

### Formatting Diagnostics at Each Phase

This would be easy for early CLI output, but it would duplicate logic across CLI, artifacts, and LSP. Structured records keep one source of truth.

### LSP Re-runs Compiler Logic In-Process

Running checker logic directly in the LSP process might reduce latency, but it risks diverging from batch builds and complicates trust boundaries. The adopted design lets the LSP orchestrate and read stable snapshots.

### Store Full Explanation Traces in Every Diagnostic

This makes each diagnostic self-contained but produces large artifacts and poor editor latency. The adopted design stores compact references and fetches detailed explanations lazily.

### Treat Open Buffers as Build Artifacts

Writing unsaved editor buffers into the normal artifact directory would simplify lookup, but it would pollute reproducible build output. The adopted design keeps open-buffer snapshots in LSP memory and marks artifact freshness explicitly.

## Adopted Approach

Mizar Evo uses structured diagnostics as canonical phase outputs, emits compact diagnostic metadata into artifacts, and serves editor features from immutable LSP snapshots. Rich explanations are referenced lazily, and source edits are applied only after range and text validation.

This keeps CLI output, artifact consumers, and IDE features aligned while preserving the batch verifier as the source of semantic truth.

## Constraints and Assumptions

- Diagnostic numeric codes are stable and never reused.
- Diagnostic messages may improve across versions; tools must not key on message text.
- LSP range conversion is performed at the editor boundary using the exact source snapshot for the request.
- Build artifacts remain the semantic source of truth for verified metadata.
- Open-buffer features may be fresher than artifacts but must be marked as such.
- Code actions must validate target ranges before applying edits.
- Explanation APIs are bounded and query-oriented, not bulk dumps of compiler state.
- LSP must degrade gracefully when artifacts are missing, stale, or produced by an unsupported schema.
