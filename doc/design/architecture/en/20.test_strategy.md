# Architecture: Test Strategy

> Canonical language: English. Japanese companion: [../ja/20.test_strategy.md](../ja/20.test_strategy.md).

## Purpose

This document defines the evo2 regression strategy.

The central testing goal is:

```text
Reject what must not pass
before
Accept everything that should pass
```

Soundness bugs are more dangerous than parser gaps. Kernel-adjacent tests therefore emphasize failing and malformed inputs.

## Design Decisions

### Test Mix

Recommended overall ratio:

| Test Kind | Ratio |
|---|---:|
| pass | 40% |
| fail | 60% |

Recommended kernel-adjacent ratio:

| Test Kind | Ratio |
|---|---:|
| pass | 20% |
| fail | 80% |

### Scale Targets

| Stage | Target |
|---|---|
| evo1 prototype | 5,000-20,000 LOC; 100-300 `.miz` files |
| evo2 alpha | 30,000-100,000 LOC; 300-1,000 `.miz` files |
| evo2 beta / release candidate | 100,000-300,000 LOC; 1,000-5,000 `.miz` files |
| mature ecosystem | 500,000-1,000,000 LOC equivalent |

The mature ecosystem target includes handwritten tests, migrated examples, generated tests, fuzz corpus entries, property-based tests, certificate snapshots, and artifact snapshots.

### Performance and Memory Evaluation

Performance and memory evaluation is a **separate track** from the soundness-first regression strategy above. Soundness regressions block release (see Constraints); performance and memory results are measured and tracked but are not, on their own, correctness gates.

This track is the home for the **concrete metrics** that the qualitative resident-set memory model (spec [§12.6.3](../../../spec/en/12.modules_and_namespaces.md#1263-memory-model), [§23.7.9](../../../spec/en/23.package_management_and_build_system.md#2379-memory-design-principles)) deliberately leaves out. Candidate measurements include:

- peak RSS per verified file, and its growth as a function of import-closure size;
- memory per 10k theorem statements and per 1k registrations (index density);
- LSP idle memory and the per-request memory delta for hover / go-to-definition / explanation queries;
- the memory cost of loading a proof witness on demand from an external file versus inlining witness data into `*.mizir.json`;
- `std.all`-style broad imports versus minimal imports.

These benchmarks validate the memory model empirically: they should show peak memory scaling with the active source file, its imported interfaces, the import-filtered indexes, and the active module's verification conditions — and staying independent of imported proof bodies, private lemmas, and registration data outside the import closure. The specifications state this only qualitatively; the numeric budgets and thresholds live here and are non-normative.

### Required Fail and Soundness Tests

The following theorem must never pass:

```mizar
theorem
  1 = 0;
```

Cluster failure tests must include infinite chains, cyclic registrations, unintended coercion, and hidden transitive expansion.

Substitution failure tests must include variable capture, binder collision, malformed substitution, and alpha-conversion failure.

Overload ambiguity tests must include ambiguous notation, hidden coercion, and unstable resolution order.

Certificate tests must include malformed certificate structure, invalid substitution, invalid SAT proof, unresolved symbol, timeout, and resource exhaustion cases.

### Snapshot Tests

Snapshot tests must cover:

- `SurfaceAst`;
- `TypedAst`;
- `CoreIr`;
- `VcIr`;
- SAT clauses;
- `ProofCertificate`;
- `VerifiedArtifact`;
- dependency slices and fingerprints;
- failure records.

Snapshot hashes must be deterministic.

### Property-Based Tests

Property-based tests should generate:

- random theorems;
- random substitutions;
- random binder structures;
- random cluster graphs;
- malformed certificates;
- import graphs and dependency slices.

Generated cases that expose soundness bugs or nondeterminism must be minimized and committed as regression tests.

### Fuzzing

Continuous fuzzing should cover:

- lexer preprocessing and raw scanner inputs;
- parser inputs;
- certificate bytes;
- substitution evidence;
- overload candidate sets;
- import graphs.

Fuzz failures are classified using [19.failure_semantics.md](./19.failure_semantics.md). Crashes, panics, nondeterministic acceptance, and unsound success are release blockers.

### Codex Implementation Guardrails

AI-assisted implementation must not introduce:

- kernel heuristics;
- implicit cluster insertion;
- trust in ATP success without certificate checking;
- nondeterministic hash generation;
- unordered iteration dependence;
- hidden global mutable state;
- fallback inference without trace evidence.

## Interface Definitions

```text
test case
  -> expected phase
  -> expected pass/fail
  -> expected failure category, if fail
  -> expected snapshot hashes, when applicable
```

## Constraints and Assumptions

- Clean sequential builds are the reference for deterministic test expectations.
- Parallel test runs must produce byte-identical snapshots for identical inputs.
- Soundness regressions block release even if normal pass coverage improves.
- Generated and fuzz tests must preserve minimized reproducers for every fixed soundness bug.
