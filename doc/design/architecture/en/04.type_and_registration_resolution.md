# Architecture: Type and Registration Resolution

> Canonical language: English. Japanese companion: [../ja/04.type_and_registration_resolution.md](../ja/04.type_and_registration_resolution.md).

## Purpose

This document defines how Mizar Evo checks soft types, builds type facts, resolves cluster registrations, and records replayable resolution traces.

It refines phases 6 and 7 of [00.pipeline_overview.md](./00.pipeline_overview.md). In particular, it defines the boundary between name-resolved syntax, type checking, automatic attribute propagation, and the later overload phase.

## Design Decisions

### Type Checking Owns Soft Type Facts, Not Logical Proof Search

The checker interprets resolved syntax as soft-typed Mizar surface semantics. It decides whether a term or formula is well-formed under the currently visible signatures and emits proof obligations when a type claim needs justification.

The checker owns:

- normalized type expressions and type heads;
- local variable, constant, and parameter typing contexts;
- inferred term and formula types;
- widening and narrowing checks;
- sethood checks required by type expressions and Fraenkel-style constructs;
- type-derived assumptions added to local proof contexts;
- coercion candidates that are valid before ordinary overload root selection;
- type and registration diagnostics.

The checker does not own:

- choosing ordinary overload roots and active refinements when multiple typed candidates remain;
- inserting final `qua` nodes for overload disambiguation;
- elaborating surface syntax into core logic;
- ATP search for non-trivial proof obligations;
- accepting proof certificates.

This keeps phases 6 and 7 deterministic and local to Mizar semantics, while preserving a clean boundary for overload resolution and proof processing.

### Types Are Represented as Normalized Predicates

Mizar types are soft classifications over untyped logical objects. The checker therefore treats a type as a normalized radix plus a polarity-aware attribute set.

```rust
struct NormalizedType {
    radix: RadixTypeRef,
    args: Vec<TermId>,
    attrs: AttributeSet,
}

struct AttributeSet {
    positive: SortedSet<AttributeInstance>,
    negative: SortedSet<AttributeInstance>,
}
```

Mode types are unfolded into their defining radix and attribute chain during type normalization, but the source-facing mode spelling is preserved for diagnostics and metadata.

Normalization must be deterministic:

- mode unfolding follows canonical `SymbolId` order;
- duplicate attributes collapse to one instance;
- contradictory attributes are recorded as consistency diagnostics instead of silently removed;
- parameterized attributes are keyed by resolved attribute symbol and normalized argument list;
- source spelling is stored separately from semantic identity.

### Local Type Contexts Are Layered

The checker uses layered contexts rather than one mutable global environment.

| Layer | Contains | Lifetime |
|---|---|---|
| module context | imported signatures, exported registrations, built-ins | entire module |
| declaration context | declaration parameters, definition-local assumptions | current item |
| proof context | thesis, assumptions, proof labels, local facts | current proof block |
| block context | `let`, `given`, `consider`, statement-local bindings | lexical block |
| expression context | expected type, expected formula shape, coercion mode | current expression |

Nested contexts may read outer facts but write only to their own layer. Leaving a block freezes the facts that are allowed to escape, such as theorem conclusions or explicitly introduced bindings.

### Cluster Resolution and Reduction Normalization Are Replayable

Cluster resolution computes a closure of type facts by repeatedly applying active cluster registrations until a fixed point is reached. Reduction registrations are handled separately as deterministic term normalization under §17.6.4; they rewrite terms and record evidence, but they do not participate in the type-fact saturation measure.

Cluster fact resolution is monotone:

- applying an active cluster registration may add derived facts;
- it must not remove existing facts;
- it must not depend on source traversal accidents;
- it terminates through canonical fact keys and saturation limits.

Reduction normalization is terminating because every accepted reduction decreases the simplification order defined in §17.6.4. It has no reduction step limit; saturation limits apply only to cluster fact closure.

Both mechanisms are replayable:

- every derived cluster fact records the registration label and inputs used;
- every reduction step records the rule FQN, source redex, target term, substitution, discharged guards, and strategy-audit key;
- traces refer to canonical `SymbolId`, `RegistrationId`, FQNs, and source provenance;
- trace ordering is deterministic;
- kernel and diagnostics can replay or inspect the derivation without rerunning heuristic search.

### Registration Databases Separate Pending and Activated Registrations

A registration declaration becomes active only after its own well-formedness and correctness obligations have been accepted by the configured verifier policy. Phase 7 can type-check the declaration and create pending registration records, but it cannot assume that obligations generated later in the VC/ATP/kernel pipeline have already succeeded.

The checker first builds a `PendingRegistration` from the resolved declaration. It then verifies:

- referenced attributes, modes, functors, and structures resolve to compatible symbols;
- parameters are well-typed;
- the registration pattern is well-formed;
- required `existence`, `coherence`, or `reducibility` obligations can be emitted as initial obligations;
- the pending record has enough stable identity and provenance to be activated after those obligations are accepted.

Only activated registrations enter the active `RegistrationIndex` used for automatic inference and exported to downstream modules. Invalid or unverified registrations may still appear in diagnostics and local metadata, but they must not contribute automatic type facts.

This applies to the declaring module as well as importers: a local registration is not used later in the same unchecked pass merely because it appears earlier in source order. Once its obligations are accepted, a later incremental pass may include it in the activated registration summary.

### Existential Registrations Gate Attributed Type Use

Attributed types must be known inhabited before they can be used in contexts that introduce values of that type.

The checker consults existential registrations for:

- `let x be A T`;
- mode declarations whose definiens contains attributes;
- functor return types with attributes;
- `consider`, `given`, and `take` contexts that claim witnesses of attributed types.

Missing existence evidence is a type error. The checker may continue with a degraded type fact so that later diagnostics remain useful, but no downstream artifact may treat the value as fully verified.

### Reduction Registrations Are Semantic Rewrites with Provenance

Reduction registrations can simplify terms, but they are not parser rewrites and they must not erase provenance.

A reduction result records:

- the source term;
- the reduced term;
- the registration used;
- the side conditions discharged;
- the phase in which the reduction was applied.

Later phases may use reduced terms for equality reasoning and computation-based discharge, while diagnostics and LSP metadata can still point to the original source expression.

### Overload Candidate Filtering Is Allowed, Root Selection Is Deferred

Type checking may discard impossible candidates when a candidate's arity, kind, or mandatory type constraints cannot match the typed arguments. It may also attach type facts to each remaining candidate.

It must not select a final ordinary overload root when the choice depends on:

- most-specific type comparison;
- inserted `qua` views;
- registration-derived preference after candidate subsumption;
- user-facing ambiguity policy.

Those decisions belong to phase 8 and are specified by `05.overload_resolution.md`.

### Diagnostics Preserve Partial Typing

Type checking and registration resolution should continue after recoverable errors. A node with a type error receives a `TypeStatus::Error` or `TypeStatus::Unknown` marker rather than a fabricated successful type.

Later phases must distinguish:

- well-typed nodes;
- nodes typed under recoverable assumptions;
- nodes with unknown type;
- nodes that are skipped because prerequisite resolution failed.

This supports useful editor feedback without letting invalid facts leak into proofs or artifacts.

## Resolution Pipeline

### Step 1: Build the Type Checking Context

Input:

- `ResolvedAst`;
- `SymbolEnv`;
- dependency `ModuleSummary` artifacts;
- activated registration summaries from dependencies.

Output:

- module-level `TypeContext`;
- active `RegistrationIndex`;
- initial `TypeFactTable`.

Responsibilities:

- load built-in type heads and built-in attributes;
- load visible modes, structures, attributes, functors, predicates, and registrations;
- index registrations by trigger keys;
- seed local facts from declarations and imported signatures;
- reject schema-incompatible registration artifacts.

### Step 2: Normalize Type Expressions

Input:

- resolved type syntax;
- `SymbolEnv`;
- local parameter context.

Output:

- `NormalizedType`;
- mode unfolding records;
- type-expression diagnostics.

Responsibilities:

- resolve radix heads to built-ins, modes, or structures;
- unfold modes into radix plus attributes;
- normalize attribute polarity and parameters;
- check type argument arity and kind;
- preserve source spelling for diagnostics.

### Step 3: Check Declarations and Local Bindings

Input:

- `ResolvedAst`;
- normalized type expressions;
- module and declaration contexts.

Output:

- typed declaration shells;
- local binding facts;
- declaration-level obligations.

Responsibilities:

- type-check `let`, `given`, `consider`, `reconsider`, and definition parameters;
- introduce type assumptions into the correct context layer;
- check that attributed declarations have existence evidence;
- emit narrowing obligations for `reconsider`;
- attach source-spanned type information to bindings.

### Step 4: Infer Terms and Formulas

Input:

- resolved terms and formulas;
- local typing context;
- candidate signatures from `SymbolEnv`.

Output:

- `TypedAst`;
- `TypeFactTable`;
- coercion candidate table;
- type diagnostics.

Responsibilities:

- infer types for variables, constants, selectors, functor applications, and aggregates;
- check predicate and formula well-formedness;
- validate `qua` expressions as views, not narrowing proofs;
- keep ambiguous overload sets as typed candidate groups;
- record expected-type constraints for phase 8.

### Step 5: Resolve Registration Closure

Input:

- current `TypeFactTable`;
- active `RegistrationIndex`;
- typed terms and normalized types.

Output:

- enriched `TypeFactTable`;
- `ResolutionTrace`;
- registration diagnostics.

Responsibilities:

- apply existential, conditional, and functorial registrations to attribute facts;
- apply reduction registrations as deterministic term normalization under §17.6.4;
- saturate cluster attribute facts to a deterministic fixed point;
- record every derived fact with its input facts and registration label, and every reduction step with its local rewrite evidence;
- detect cluster cycles, contradictions, and saturation limits, plus reduction simplification-order or guard failures;
- ignore registrations that are not present in the active `RegistrationIndex`.

### Step 6: Validate Pending Registration Declarations

Input:

- typed registration declarations;
- correctness conditions that must become proof obligations;
- source provenance and registration labels.

Output:

- pending local registrations;
- rejected pending registrations;
- initial registration obligations.

Responsibilities:

- verify pattern well-formedness;
- bind registration labels to `RegistrationId`s;
- generate initial `existence`, `coherence`, and `reducibility` obligations;
- record the dependency between each pending registration and its initial obligations;
- leave activation and export to the artifact phase after verifier policy accepts the obligations.

## Interface Definitions

### Type Environment

```rust
struct TypeContext {
    module: ModuleId,
    symbols: SymbolEnv,
    registrations: RegistrationIndex,
    facts: TypeFactTable,
    scopes: Vec<TypeScope>,
}

struct TypeScope {
    kind: TypeScopeKind,
    bindings: BindingTypeTable,
    assumptions: Vec<TypeFactId>,
}
```

`TypeContext` is checker-owned and is not serialized directly. Artifacts receive stable projections of inferred types, facts, and activated registrations.

### Type Facts

```rust
struct TypeFact {
    id: TypeFactId,
    subject: TermId,
    predicate: TypePredicate,
    polarity: Polarity,
    provenance: FactProvenance,
}

enum FactProvenance {
    Declared(SourceRange),
    Assumed(LabelRef),
    Inferred(ResolutionStepId),
    Obligation(InitialObligationId),
    Builtin(BuiltinRuleId),
}
```

Type facts are the common currency between type checking, registration resolution, diagnostics, and VC generation.

### Registration Index

```rust
struct RegistrationIndex {
    existential: TriggerMap<TypeHeadKey, Vec<RegistrationId>>,
    conditional: TriggerMap<AttributeKey, Vec<RegistrationId>>,
    functorial: TriggerMap<FunctorKey, Vec<RegistrationId>>,
    reduction: TriggerMap<TermHeadKey, Vec<RegistrationId>>,
}

struct RegistrationDef {
    id: RegistrationId,
    label: LabelRef,
    kind: RegistrationKind,
    pattern: RegistrationPattern,
    parameters: Vec<TypedParameter>,
    correctness: CorrectnessCondition,
    source_range: SourceRange,
    origin: ModuleId,
}
```

Trigger maps are optimization indexes. They do not define semantics; semantics is defined by the resolved `RegistrationDef` records.

The active `RegistrationIndex` contains only dependency registrations and previously verified local registrations. Newly declared local registrations are represented separately as `PendingRegistration`s until their initial obligations are mapped to `VcId`s, discharged, and accepted.

### Resolution Trace

The concrete `ResolutionTrace` artifact schema is defined in [17.cluster_trace_format.md](./17.cluster_trace_format.md). Phase 7 populates that schema with cluster steps for derived type facts and reduction steps for automatic term rewrites.

At this phase, a cluster step carries the applied registration, the input type facts, the derived fact, and source provenance. A reduction step carries the applied registration and rule FQN, enclosing-term fingerprint, redex path, source redex, target term, substitution, discharged guard evidence, active rule-view fingerprint, selection key, and source provenance. These values are serialized using the canonical field names from the trace-format document.

`ResolutionTrace` is the replayable explanation of automatic typing and reduction normalization. It is consumed by diagnostics, kernel replay, artifacts, and `@show_resolution` metadata.

### Typed AST

```rust
struct TypedAst {
    source_id: SourceId,
    module_id: ModuleId,
    nodes: AstArena<TypedNode>,
    types: TypeTable,
    facts: TypeFactTable,
    coercions: CoercionCandidateTable,
    errors: TypeErrorTable,
}
```

`TypedAst` preserves source shape. It contains enough information for registration and overload resolution, but it is not yet the final surface semantics because overload roots and active refinements may remain open.

## Diagnostics

Type and registration diagnostics are semantic diagnostics over resolved source syntax.

| Diagnostic Class | Example |
|---|---|
| type expression | unknown type head, wrong type argument count, contradictory attributes |
| binding | declared type is not inhabited, duplicate local binding, illegal escape of local fact |
| term typing | selector unavailable, functor argument type mismatch, formula expected |
| conversion | invalid `qua`, narrowing without proof, failed `reconsider` obligation |
| registration | malformed cluster pattern, missing correctness proof, inactive registration used |
| resolution | cluster cycle, saturation limit, contradictory derived attributes |
| reduction | simplification-order violation, side condition not discharged |

Diagnostics should include:

- primary source span at the use site;
- secondary spans for the declaration, registration, or derived fact involved;
- machine-readable expected and actual type summaries;
- resolution trace references when an inferred fact caused the diagnostic;
- stable diagnostic codes from the type or registration ranges.

## Incrementality and Reproducibility

Type and registration cache keys include:

- `ResolvedAst` hash;
- `SymbolEnv` hash;
- visible dependency registration summary hashes;
- active language edition;
- checker schema version;
- verifier settings that affect registration activation.

Changes invalidate:

- the changed module's `TypedAst`, `TypeFactTable`, and `ResolutionTrace`;
- downstream modules that import changed activated registrations;
- downstream modules that depend on changed exported type signatures;
- overload, elaboration, VC, and artifact phases that consume affected typed facts.

Ordering must be deterministic:

- registration triggers are sorted by canonical `RegistrationId`;
- fixed-point worklists use canonical fact keys;
- trace steps are serialized in derivation order with stable tie-breaking;
- diagnostics with the same span are ordered by diagnostic code and stable type key.

## Alternatives Considered

### Inline Cluster Resolution During Every Type Query

Resolving clusters lazily on each query reduces up-front work, but it makes diagnostics, replay, and incremental cache keys harder to reason about. The adopted design computes explicit closures and records traces.

### Treat Registrations as Trusted Declarations Immediately

Immediate activation is simple, but it lets invalid registrations affect later checking before their correctness obligations are accepted. The adopted design separates pending registrations from activated registrations.

### Combine Type, Registration, and Overload Resolution

Combining these phases can reduce intermediate data structures, but it blurs the difference between fact derivation and candidate choice. The adopted design lets type and registration phases compute facts, then leaves final overload root selection and refinement joining to a dedicated phase.

## Adopted Approach

Mizar Evo uses a two-part checker boundary for phases 6 and 7.

Type checking normalizes type expressions, builds local type contexts, checks terms and formulas, and produces `TypedAst` plus type facts. Registration resolution then computes cluster fact closure with activated cluster registrations and applies activated reduction registrations as deterministic term normalization, recording both effects in a replayable `ResolutionTrace`.

The result is deterministic Mizar-side semantic information: typed syntax, explicit facts, and traceable automatic derivations. Overload resolution, elaboration, proof obligation generation, ATP dispatch, and kernel certificate checking consume these outputs without redoing type or registration inference.

## Constraints and Assumptions

- Type checking starts only after module/name resolution and signature collection have produced a usable `SymbolEnv`.
- The type system is soft; all type facts must remain explainable as logical predicates or registration-derived facts.
- Activated registrations from dependencies are trusted only as verified artifact projections with compatible schema versions.
- Unverified local registrations must not affect automatic inference.
- Local registration activation is delayed until proof obligations are accepted by the configured verifier policy.
- Phase 6-7 structures may refer to `InitialObligationId`s; `VcId`s are assigned later by VC generation.
- Registration resolution must terminate deterministically or emit a bounded saturation diagnostic.
- The checker may produce partial typed output after recoverable errors, but invalid facts must not be exported as verified metadata.
- ATP backends do not perform type checking, registration resolution, or overload resolution.
