# Architecture: Overload Resolution

> Canonical language: English. Japanese companion: [../ja/05.overload_resolution.md](../ja/05.overload_resolution.md).

## Purpose

This document defines how Mizar Evo selects overload winners, resolves template-derived candidates, handles `qua` disambiguation, and produces the final source-shaped semantic AST consumed by elaboration.

It refines phase 8 of [00.pipeline_overview.md](./00.pipeline_overview.md). In particular, it defines the boundary between typed surface syntax with candidate sets and `ResolvedTypedAst` with final overload winners and inserted coercion views.

## Design Decisions

### Overload Resolution Finalizes Surface Semantics

Earlier phases may know that a symbol use refers to an overload set, and type checking may remove candidates that are impossible by arity, kind, or mandatory typing constraints. Phase 8 is the first phase allowed to select the final definition used at a call site.

Overload resolution owns:

- expansion of template-derived candidates;
- viability filtering using enriched type facts;
- construction of per-site specificity comparisons;
- selection of a unique overload winner;
- detection of overload, template, and inheritance-path ambiguity;
- insertion of final `qua` views needed by selected candidates;
- construction of `ResolvedTypedAst`.

It does not own:

- name lookup or import visibility;
- type expression normalization;
- cluster or registration closure;
- proof discharge for `reconsider` or registration correctness;
- lowering selected definitions into core logic.

This keeps phase 8 deterministic and source-shaped while giving elaboration a single definition identity for every resolved use site.

### Candidate Construction Preserves Provenance

The resolver and checker provide candidate groups keyed by the resolved surface reference. Phase 8 turns those groups into concrete overload candidates.

Each candidate records:

- defining `SymbolId`;
- declaration kind and arity;
- parameter `NormalizedType`s;
- return type metadata, without using it for overload selection;
- source and import provenance;
- whether it came from a template instantiation;
- required implicit views from arguments to parameters;
- any pending diagnostics that explain why the candidate was dropped.

Return type does not participate in overload selection. Two definitions with identical argument-type signatures but different return types are declaration conflicts, not overload alternatives. Those conflicts are detected during signature collection; phase 8 may surface the pre-existing diagnostic at a use site, but it must not reclassify conflicting declarations as overload candidates.

Candidate groups passed to phase 8 must already respect source-order and scope visibility. For current-module unqualified references, this means only declarations visible before the use site are present. Phase 8 may validate candidate provenance for diagnostics, but it does not widen the candidate set by scanning `SymbolEnv` globally.

### Template Expansion Precedes Ordinary Candidate Ordering

Template handling has two distinct phases.

1. Template parameter inference or explicit template argument checking produces concrete candidates.
2. The standard overload ordering runs over concrete candidates and ordinary non-template candidates together.

The two steps must not be interleaved. Template inference binds omitted template parameters from exact argument types and mode-hierarchy constraints. It does not use cluster expansion, because cluster-derived alternatives can make inferred parameters non-unique. Once template candidates are concrete, ordinary viability and specificity may use full type facts and cluster closure.

When a template-derived candidate and a non-template candidate are otherwise tied, the non-template candidate wins. This lets libraries provide explicit specializations without creating avoidable ambiguity.

### Viability Uses Type Facts, Not New Inference

A candidate is viable when every argument can be viewed as the corresponding parameter type by proof-free widening:

- identical normalized type;
- mode or structure upcast through a unique inheritance path;
- attribute weakening justified by the enriched type facts and registration closure;
- explicit `qua` view supplied by the source.

Phase 8 may consult `TypeFactTable`, `ResolutionTrace`, and the coercion candidates produced by phase 6-7. It must not derive new cluster facts or activate registrations. If a viability check requires facts that are absent, the candidate is rejected or marked blocked with a diagnostic, rather than extending the type environment.

### Specificity Is a Partial Order

The best candidate is the unique viable candidate whose parameter vector is at least as specific as every other viable candidate.

Phase 8 represents this as a per-site specificity graph:

- nodes are viable concrete candidates;
- an edge `A -> B` means `A` is at least as specific as `B`;
- strictness is derived when `A -> B` exists and `B -> A` does not;
- tied candidates remain equivalent unless a specified tie-breaker applies.

The selected candidate must be a unique maximal element after applying allowed tie-breakers. If no such candidate exists, the site remains unresolved with an ambiguity diagnostic.

Allowed tie-breakers are intentionally narrow:

- non-template candidate beats template-derived candidate when their parameter vectors are otherwise equivalent;
- a source-local declaration shadows an imported declaration only when their argument-type signature is the same and shadowing has already been accepted by name/signature collection;
- no tie-breaker may use return type.

### `qua` Is the Only Implicit View Made Explicit in Output

Source `qua` expressions are preserved. Additionally, phase 8 may insert implicit `qua` views into `ResolvedTypedAst` when the selected candidate requires a proof-free widening view that later phases must see explicitly.

Inserted views are allowed for:

- unique structure inheritance upcasts;
- attribute weakening through already recorded type facts;
- template explicit-argument viability after the concrete candidate has been selected.

Inserted views are not allowed for narrowing. Narrowing belongs to `reconsider` and must already have a proof obligation or accepted local fact from type checking.

Every inserted view records:

- source argument expression;
- target parameter type;
- inheritance path or type-fact explanation;
- whether it was source-written or compiler-inserted;
- diagnostic provenance for `@show_resolution` and LSP metadata.

### Multiple Inheritance Paths Are Ambiguity, Not Search

If an argument can reach a parameter type through more than one inheritance path, phase 8 does not choose a path heuristically. The candidate is blocked by an ambiguous inheritance path unless the source uses `qua` to select the path.

This is distinct from overload ambiguity: a single candidate may be blocked because its required argument view is ambiguous.

### Redefinitions Are Ordinary Candidates with Extra Checks

A `redefine` declaration contributes a candidate like an ordinary definition, but signature collection and type checking must have recorded its relationship to the original definition.

Phase 8 uses the same specificity ordering for redefinitions. It does not prove redefine coherence. Coherence is represented as an initial obligation and handled by later proof phases.

### Output Must Preserve Failed Sites

`ResolvedTypedAst` should include explicit unresolved overload nodes for recoverable errors. Elaboration must skip or degrade those nodes instead of receiving fabricated winners.

This supports LSP features such as hover, candidate lists, `@show_resolution`, and code actions for adding `qua`, while preventing invalid selections from entering core logic.

## Resolution Pipeline

### Step 1: Collect Candidate Sites

Input:

- `TypedAst`;
- candidate groups from name and type checking;
- source-order-filtered visible symbol signatures from `SymbolEnv`;
- enriched type facts and `ResolutionTrace`.

Output:

- `OverloadSite`s;
- initial candidate lists;
- site-level expected type metadata.

Responsibilities:

- find overloaded functor, predicate, attribute, mode, selector, and template uses;
- attach source spans and AST node ids;
- preserve qualified-name and import provenance;
- enforce that current-module candidates are already filtered by use-site visibility;
- skip sites whose name or type prerequisites failed unrecoverably.

### Step 2: Expand Templates

Input:

- overload sites;
- template definitions and explicit template arguments;
- exact argument types from `TypedAst`.

Output:

- concrete template-derived candidates;
- template inference diagnostics.

Responsibilities:

- infer omitted template parameters from exact argument types;
- check template constraints using mode hierarchy only during inference;
- instantiate successful templates into concrete candidate signatures;
- reject ambiguous or inconsistent template inference;
- mark candidates as template-derived for later tie-breaking.

### Step 3: Filter Viable Candidates

Input:

- concrete candidates;
- argument types;
- enriched type facts;
- coercion candidate table;
- source-written `qua` views.

Output:

- viable candidates;
- rejected candidate explanations;
- blocked candidate explanations.

Responsibilities:

- check arity and declaration kind;
- check argument-to-parameter compatibility;
- detect missing facts and invalid narrowing;
- detect ambiguous inheritance paths;
- compute required view plans for viable candidates.

### Step 4: Build the Specificity Graph

Input:

- viable candidates;
- normalized parameter types;
- type closure summaries.

Output:

- `SpecificityGraph`;
- equivalence classes;
- comparison diagnostics for ambiguous sites.

Responsibilities:

- compare candidates position-wise;
- compute at-least-as-specific edges;
- detect incomparable candidates;
- record why a comparison succeeded or failed;
- apply allowed tie-breakers after type comparison.

### Step 5: Select Winners and Insert Views

Input:

- specificity graph;
- viable candidates;
- view plans.

Output:

- `OverloadResolutionTable`;
- `CoercionInsertionTable`;
- overload diagnostics.

Responsibilities:

- select the unique maximal candidate;
- report no-match or ambiguity errors;
- insert compiler-generated `qua` views for selected candidates;
- preserve source-written `qua` nodes;
- record candidate lists for diagnostics and LSP metadata.

### Step 6: Build `ResolvedTypedAst`

Input:

- `TypedAst`;
- selected overload winners;
- inserted views;
- enriched type facts and resolution traces.

Output:

- `ResolvedTypedAst`;
- final overload metadata;
- recoverable overload diagnostics.

Responsibilities:

- rewrite candidate-group nodes into resolved-use nodes;
- attach final definition `SymbolId`s;
- attach final argument view metadata;
- keep unresolved overload nodes for recoverable failures;
- produce stable metadata for elaboration and artifacts.

## Interface Definitions

### Overload Site

```rust
struct OverloadSite {
    id: OverloadSiteId,
    node: AstNodeId,
    kind: OverloadSiteKind,
    name_ref: SymbolRef,
    source_range: SourceRange,
    arguments: Vec<ExprId>,
    expected: Option<ExpectedType>,
}
```

`OverloadSite` is a source-shaped use that may require final candidate selection.

### Candidate

```rust
struct OverloadCandidate {
    id: CandidateId,
    symbol: SymbolId,
    origin: CandidateOrigin,
    params: Vec<NormalizedType>,
    result: Option<NormalizedType>,
    template: Option<TemplateInstantiation>,
    provenance: CandidateProvenance,
}

enum CandidateOrigin {
    Ordinary,
    Redefinition { original: SymbolId },
    TemplateDerived { template: SymbolId },
}
```

`result` is recorded for metadata and later elaboration. It is not part of winner selection.

### Viability

```rust
struct ViableCandidate {
    candidate: CandidateId,
    argument_views: Vec<ArgumentView>,
}

enum CandidateRejection {
    ArityMismatch,
    KindMismatch,
    TypeMismatch { argument: ExprId, expected: NormalizedType, actual: NormalizedType },
    AmbiguousInheritancePath { argument: ExprId, target: NormalizedType, paths: Vec<InheritancePath> },
    MissingTypeFact { argument: ExprId, required: TypePredicate },
}
```

Blocked candidates are retained for diagnostics even though they cannot win.

### Specificity Graph

```rust
struct SpecificityGraph {
    site: OverloadSiteId,
    candidates: Vec<CandidateId>,
    edges: Vec<SpecificityEdge>,
    tie_breaks: Vec<TieBreak>,
}

struct SpecificityEdge {
    more_specific: CandidateId,
    less_specific: CandidateId,
    reasons: Vec<SpecificityReason>,
}
```

The graph is per overload site. It is not a global ordering over declarations.

### Resolution Result

```rust
enum OverloadResolution {
    Resolved {
        site: OverloadSiteId,
        winner: CandidateId,
        inserted_views: Vec<InsertedViewId>,
    },
    NoMatch {
        site: OverloadSiteId,
        rejected: Vec<CandidateRejection>,
    },
    Ambiguous {
        site: OverloadSiteId,
        candidates: Vec<CandidateId>,
        graph: SpecificityGraph,
    },
    Blocked {
        site: OverloadSiteId,
        reason: BlockedReason,
        candidates: Vec<CandidateId>,
    },
}

enum BlockedReason {
    TemplateInferenceFailed,
    AmbiguousTemplateInference,
    AmbiguousInheritancePath,
    PreExistingDeclarationConflict,
}
```

`NoMatch` is used when candidates were considered but none is viable. `Blocked` is used when phase 8 cannot safely run ordinary winner selection because a prerequisite choice is itself ambiguous or invalid.

### Inserted View

```rust
struct InsertedView {
    id: InsertedViewId,
    expression: ExprId,
    target: NormalizedType,
    reason: ViewReason,
    source_written: bool,
}

enum ViewReason {
    SourceQua(SourceRange),
    UniqueInheritancePath(InheritancePath),
    AttributeWeakening(Vec<TypeFactId>),
    TemplateExplicitArgument,
}
```

Inserted views are semantic metadata on `ResolvedTypedAst`, not source edits.

## Diagnostics

Overload diagnostics are semantic diagnostics over typed source syntax.

| Diagnostic Class | Example |
|---|---|
| no match | no candidate accepts the argument types |
| ambiguity | two viable candidates are incomparable |
| template inference | omitted template parameter cannot be inferred uniquely |
| inheritance path | multiple implicit paths to parameter type |
| invalid `qua` | source `qua` is not a proof-free widening |
| redefine | redefinition is not strictly more specific than original |
| declaration conflict | pre-existing same-signature conflict prevents candidate construction |

Diagnostics should include:

- primary span at the use site;
- candidate list with declaration spans and import provenance;
- expected and actual type summaries;
- specificity comparison explanation;
- suggested explicit `qua` target when one candidate would become unique;
- stable candidate ordering for machine-readable output.

## Incrementality and Reproducibility

Overload cache keys include:

- `TypedAst` hash;
- `TypeFactTable` and `ResolutionTrace` hashes;
- visible signature and overload-set hashes;
- template definition hashes;
- checker and overload schema versions.

Changes invalidate:

- overload sites whose argument types or type facts changed;
- sites that reference changed overload sets;
- sites whose template definitions or constraints changed;
- elaboration, VC, and artifact outputs that consume changed `ResolvedTypedAst` nodes.

Ordering must be deterministic:

- candidates are sorted by canonical `SymbolId`, then template instantiation key, then import provenance;
- rejected candidate diagnostics use the same candidate order;
- specificity graph edges are serialized by candidate keys;
- ambiguous candidate sets are reported in stable order;
- inserted views are emitted in AST traversal order.

## Alternatives Considered

### Checker-Owned Winner Selection

Selecting winners during type checking would reduce one IR boundary, but it would mix type fact derivation with candidate ordering. The adopted design lets the checker compute facts and leaves final choice to phase 8.

### Return-Type-Based Overloading

Using expected return type could resolve some calls, but it creates circular interactions with coercions and template inference. The adopted design records return type for metadata and elaboration but excludes it from selection.

### Global Specificity DAG

A global DAG over all overload declarations looks reusable, but specificity can depend on instantiated templates, imported visibility, source-written `qua`, and local type facts. The adopted design builds per-site graphs and caches comparison summaries where useful.

### Heuristic Inheritance Path Selection

Choosing the shortest or first inheritance path would avoid some errors, but it would hide semantic choices in declaration order. The adopted design requires explicit `qua` when multiple paths are available.

## Adopted Approach

Mizar Evo performs overload resolution as a dedicated phase after type and registration resolution.

Phase 8 expands templates, filters viable candidates using existing type facts, builds a per-site specificity graph, applies narrow tie-breakers, selects a unique winner when possible, and records inserted `qua` views. The output is `ResolvedTypedAst`, the final source-shaped semantic representation consumed by elaboration and LSP metadata.

## Constraints and Assumptions

- Phase 8 starts only after type checking and registration closure have produced enriched type facts.
- Phase 8 may read `ResolutionTrace` but must not add new registration-derived facts.
- Return type does not participate in overload winner selection.
- Explicit `qua` may disambiguate proof-free widening only; narrowing requires `reconsider`.
- Multiple inheritance paths require explicit `qua`.
- Failed overload sites remain explicit in `ResolvedTypedAst` and must not be elaborated as valid core terms.
- Candidate groups entering phase 8 must already be filtered by lexical scope and source-order visibility.
- Same-signature declaration conflicts are owned by signature collection, though phase 8 may attach their diagnostics to affected use sites.
- Candidate and diagnostic ordering must be reproducible.
