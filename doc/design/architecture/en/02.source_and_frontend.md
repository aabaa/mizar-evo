# Architecture: Source and Frontend

> Canonical language: English. Japanese companion: [../ja/02.source_and_frontend.md](../ja/02.source_and_frontend.md).

## Purpose

This document defines how the Mizar Evo frontend loads `.miz` source files and coordinates source preprocessing, lexing, parsing, and syntax output for module and name resolution.

It refines phases 1-3 of [00.pipeline_overview.md](./00.pipeline_overview.md). In particular, it defines the boundaries for context-sensitive lexing, the active lexical environment, user-defined symbols, dot handling, fully tokenized string literals, doc comments, and annotation attachment.

## Design Decisions

### Frontend Produces Syntax, Not Semantics

The frontend must not perform semantic name resolution or type checking.

It may:

- validate UTF-8 and code-region ASCII constraints;
- identify comments and doc comments;
- pre-scan import declarations enough to request an active lexical environment;
- tokenize using reserved words, reserved symbols, and active user symbols;
- parse source into a surface-shaped AST;
- perform syntax-level error recovery.

It must not:

- decide whether a dotted path denotes a namespace or a selected term when that requires scope knowledge;
- choose overload winners;
- infer expression types;
- fire cluster registrations;
- generate proof obligations.

### Syntax and Parser Are Separate Crate Boundaries

The implementation should keep syntax data structures and parser logic outside the orchestration-focused `mizar-frontend` crate.

| Crate | Responsibility |
|---|---|
| `mizar-syntax` | Owns `SurfaceAst`, syntax node kinds, arenas, trivia, recovery markers, and parser-facing syntax diagnostics. |
| `mizar-parser` | Consumes `TokenStream`, applies the grammar including Pratt parsing for terms and formulas, attaches annotations and doc comments, and emits `SurfaceAst`. |
| `mizar-frontend` | Loads sources, builds source maps, coordinates preprocessing, active lexical environment construction, lexing, parser-seam invocation, and combined `FrontendOutput`. |

This split lets LSP, formatter, diagnostics, resolver, and tests depend on syntax structures without depending on parser internals. It also keeps parser recovery and grammar evolution from bloating the source-loading/orchestration layer.
Until `mizar-syntax` / `mizar-parser` exist, the frontend uses a stub parser seam
that returns `ast = None` so the source → tokens pipeline can be exercised
without hard parser dependencies.

### Import Pre-Scan Is Shallow

The frontend needs import information before final tokenization because imported modules extend the active lexical environment. However, full import resolution belongs to the module resolver.

Therefore, preprocessing performs only a shallow import pre-scan:

- recognizes top-level `import` forms using a restricted lexical mode;
- extracts raw dotted module paths, aliases, overall source spans, and
  `path.source_segments` for branch imports;
- reports only malformed import syntax that prevents lexicon loading;
- delegates package/module existence, visibility, and export checks to later phases.

The import pre-scan consumes raw lexer output. Raw scanning itself does not interpret imports or construct the active lexical environment.
If strict raw scanning cannot produce a complete raw stream, preprocessing records
an import-pre-scan diagnostic and returns no import stubs rather than inferring
imports from partial raw text.

The build planner / module resolver uses these `ImportStub`s to produce resolved imports and lightweight module lexical summaries. The lexer then builds an `ActiveLexicalEnvironment`, and final disambiguation proceeds with the symbols visible from those imports.

### Lexing Is Raw First, Then Contextually Disambiguated

The lexer is split conceptually into raw scanning and final token disambiguation.

The raw scanner preserves source spans and emits coarse units such as `LexemeRun`. These raw units are not final language tokens and may be split later.

The disambiguator applies the longest-match rule against:

1. user-defined symbols from the active lexical environment;
2. reserved special symbols;
3. reserved words;
4. identifier and numeral rules;
5. parser expectation and scoped identifier override rules where the language requires them.

When a spelling matches both identifier syntax and an active user symbol, final classification may require parser position and scope information. A scoped identifier binding may override a symbol according to the language rules. This decision belongs to token disambiguation, not raw scanning.

The scope information used for lexical disambiguation is supplied by a dedicated scope-skeleton pre-scan over raw lexer output. The disambiguator consumes this read-only view; it does not build scopes itself.

The lexer does not decide whether an identifier is defined, whether a symbol is legally applicable in a term/formula context, or which overload is selected. Those are resolver/checker concerns.

### Dot Handling Is Split Across Lexer, Parser, and Resolver

`.` is deliberately not resolved in one place.

| Role | Layer | Rule |
|---|---|---|
| compound reserved tokens `.{`, `.*`, `.=`, `...` | lexer | recognized as single reserved tokens |
| user-defined symbols containing `.` | disambiguator | recognized by the active lexical environment, parser context, scope override rules, and longest-match |
| selector access / update | parser + resolver | parsed as a possible selector chain in term contexts |
| namespace separator | parser + resolver | parsed in import, citation, annotation, and qualified-name contexts |
| variable shadowing of namespace paths | resolver | local variables win over namespace components |

The frontend may produce an ambiguous source-shaped representation such as `DottedPathOrSelection` where semantic scope is required. The resolver later commits to namespace or selector interpretation.

### String Literals Are Fully Tokenized by the Lexer

`"` and `'` are admissible user-symbol characters, so the lexer cannot treat them as string delimiters globally.

The frontend still targets a complete `TokenStream` before parsing. The current
lexer API represents this dependency as a narrow but uniform `ParserLexContext`;
the frontend/parser integration may later supply grammar-derived
position-sensitive string-required context without exposing arbitrary parser
state to the lexer.

String literals are recognized only at grammar positions that explicitly require a string literal, such as:

- annotation arguments like `@latex("...")`;
- operator declaration arguments such as `infix_operator("+", left, 80)`;
- future grammar positions registered as string-valued arguments.

Outside these positions, quote characters remain ordinary symbol characters governed by the active lexical environment and longest-match rule.

The parser consumes already-tokenized `StringLiteral` tokens. It never drives the
lexer cursor directly. Until the parser-assisted lexing contract is finalized,
only general/stub lexing and bounded uniform `StringRequired` lexing are assumed
for frontend implementation tests; annotation/operator-position string literal
tests stay gated.

### Comments and Doc Comments Are Source Metadata

Comments are removed from lexical input, but they are not discarded.

- ordinary comments are retained for formatting/debug tooling only;
- doc comments are attached to the following documentable item when possible;
- doc comment attachment remains syntactic and may later be rejected if the target is not documentable;
- structured doc tags are parsed enough to preserve tag name, raw arguments, and source spans.

### Annotations Are Parser-Owned Syntax

Annotations are part of the surface syntax and must preserve source location and raw arguments.

Preprocessing does not collect annotations into a separate metadata channel. Annotation tokens remain in `lexical_text`, are emitted into `TokenStream`, and are parsed by the parser into `SurfaceAst` annotation nodes.

The parser validates annotation syntax, but semantic effects are deferred:

- `@[...]` library annotations attach to items as raw library labels;
- `@latex`, `@proof_hint`, `@show_*`, `@eval` are represented as annotation nodes;
- annotation registry validation may happen in the parser for syntax, but meaning is interpreted by later phases;
- annotations must not change core language semantics.

## Frontend Pipeline

### Step 1: Load SourceUnit

Input:

- package/workspace `BuildPlan`;
- `.miz` file path;
- expected package root and `src/` root.

Output:

- `SourceUnit`.

Responsibilities:

- read source bytes;
- validate UTF-8;
- strip one leading UTF-8 BOM from package-authored source before creating loaded source text;
- compute `source_hash`;
- derive module path from file path;
- build `LineMap` using Unicode scalar column rules;
- preserve `LoadingMap` from loaded-text offsets back to original file byte offsets or editor-provided text offsets when loading changed offsets;
- emit file-level diagnostics.

Failure examples:

- unreadable source file;
- invalid UTF-8;
- file outside package `src/` root;
- invalid `.miz` module filename.

### Step 2: Preprocess Source

Input:

- `SourceUnit`.

Output:

- `PreprocessedSource`.

Responsibilities:

- validate code-region ASCII while allowing Unicode in comments; Unicode inside
  annotation string arguments is accepted only after parser-assisted lexing can
  identify those string-required spans before ASCII precondition reporting;
- identify and remove ordinary comments from lexical input;
- preserve doc comments with source spans;
- preserve annotation syntax in lexical input for parser ownership;
- shallow-scan top-level imports;
- preserve exact mapping from lexical text back to source ranges.

The current `mizar-lexer` crate provides the comment-stripping, lexical precondition diagnostics, module-source naming, raw-scan, and import pre-scan helper APIs. The frontend remains responsible for file I/O, UTF-8 validation, line maps, and orchestration.

Failure examples:

- unterminated block comment;
- illegal non-ASCII character in code region;
- import pre-scan failure that prevents active lexical environment construction.

### Step 3: Build ActiveLexicalEnvironment

Input:

- `ImportStub`s from preprocessing;
- package/module indexes available from the build plan;
- already-built dependency symbol exports.

Output:

- `ActiveLexicalEnvironment`.

Responsibilities:

- gather user-defined symbolic names exported by imported modules;
- include built-in reserved symbols and keyword table;
- record import order for deterministic fingerprints and provenance;
- reject or diagnose equal-spelling imported user-symbol conflicts according to
  the lexer/spec contract, rather than resolving them by import-order
  tie-breaking;
- record symbol provenance for diagnostics.

This step is a frontend-adjacent service: it uses shallow imports from the frontend, but full import legality remains part of module resolution.

### Step 4: Lex

Input:

- `PreprocessedSource`;
- `ActiveLexicalEnvironment`;
- parser expectation and scoped binding view where final disambiguation requires them.

Output:

- `TokenStream`.

Responsibilities:

- preserve raw `LexemeRun` spans without premature splitting;
- emit reserved words and reserved special symbols;
- emit user-defined symbols under context-sensitive longest-match;
- emit identifiers and numerals;
- emit `StringLiteral` tokens at string-required positions;
- preserve source spans and original spelling;
- expose lexical diagnostics without losing recoverable tokens.

Failure examples:

- unknown or malformed token sequence;
- invalid numeral form;
- malformed string literal in a string-required position.

### Step 5: Parse

Input:

- `TokenStream`.
- parser inputs derived from the active lexical environment, such as operator fixity and grammar contexts that require string literals.

Output:

- `SurfaceAst`.

Responsibilities:

- parse modules, definitions, registrations, statements, terms, formulas, theorems, proofs, algorithms;
- use Pratt parsing or an equivalent table-driven precedence parser for term and formula expressions;
- parse annotation argument lists and attach annotations to syntax nodes;
- attach doc comments to nearby documentable items;
- preserve source order and source ranges;
- recover from syntax errors at synchronization points such as `;`, `end`, and top-level item keywords.

Failure examples:

- unexpected token;
- unmatched delimiter;
- missing `end`;
- expected string literal token is missing.

## Interface Definitions

### SourceUnit

```rust
struct SourceUnit {
    source_id: SourceId,
    package_id: PackageId,
    module_path: ModulePath,
    normalized_path: NormalizedPath,
    edition: Edition,
    file_path: PathBuf,
    source_text: Arc<str>,
    source_hash: Hash,
    line_map: LineMap,
    loading_map: Option<LoadingMap>,
    origin: SourceOrigin,
    generated_anchor: Option<SourceAnchor>,
}
```

### PreprocessedSource

```rust
struct PreprocessedSource {
    source_id: SourceId,
    lexical_text: LexicalText,
    comments: Vec<Comment>,
    doc_comments: Vec<DocComment>,
    import_stubs: Vec<ImportStub>,
    source_map: LexicalSourceMap,
    diagnostics: Vec<PreprocessDiagnostic>,
}
```

### ImportStub

```rust
struct ImportStub {
    path: ImportStubPath,
    alias: Option<ImportStubAlias>,
    span: SourceRange,
}

struct ImportStubPath {
    spelling: String,
    relative: Option<ImportStubRelativePrefix>,
    components: Vec<String>,
    source_segments: Vec<SourceRange>,
    span: SourceRange,
}

enum ImportStubRelativePrefix {
    Current,
    Parent,
}

struct ImportStubAlias {
    spelling: String,
    span: SourceRange,
}
```

`ImportStub` is the frontend-mapped counterpart of the lexer import pre-scan
stub. It mirrors the lexer `RawModulePath` / `RawModuleAlias` shape, including
the distinction between `./` and `../` relative prefixes, but all ranges are
session `SourceRange`s. It is not a resolved import and does not carry an
import-kind field. It is enough to request lexicon entries and to produce good
diagnostics if lexicon loading fails.

### ActiveLexicalEnvironment

```rust
struct ActiveLexicalEnvironment {
    reserved_words: ReservedWordTable,
    reserved_symbols: ReservedSymbolTable,
    user_symbols: UserSymbolIndex,
    fingerprint: LexicalEnvironmentFingerprint,
}

struct UserSymbolCandidate {
    spelling: SymbolText,
    symbol_id: SymbolId,
    source_module: ModuleId,
    imported_module: ModuleId,
    import_ordinal: usize,
    export_rank: ExportRank,
}
```

`ActiveLexicalEnvironment` is built from already-resolved imports plus lightweight `ModuleLexicalSummary` values. It is still lexer-local: it records lexical shape and provenance, not full module IR or semantic applicability.

### TokenStream

```rust
struct TokenStream {
    source_id: SourceId,
    tokens: Vec<Token>,
    diagnostics: Vec<LexingDiagnostic>,
}

struct Token {
    kind: TokenKind,
    span: SourceRange,
    text: InternedText,
}

enum TokenKind {
    Identifier,
    ReservedWord,
    ReservedSymbol,
    Numeral,
    LexemeRun,
    UserSymbol,
    StringLiteral,
    ErrorRecovery,
}
```

`StringLiteral` appears only where the lexer context recognizes a
string-required position. The current lexer context is uniform; exact
grammar-position recognition is gated on the parser-assisted lexing contract.
`LexingDiagnostic` wraps raw-scan, scope-skeleton, and disambiguator diagnostics
with a frontend-mapped primary session `SourceRange` plus secondary source
anchors when the source map is composite or degraded.

### SurfaceAst

Owned by `mizar-syntax`.

```rust
struct SurfaceAst {
    source_id: SourceId,
    module: SurfaceModule,
    nodes: AstArena<SurfaceNode>,
    trivia: TriviaMap,
    diagnostics: Vec<Diagnostic>,
}
```

### FrontendOutput

Produced by `mizar-frontend` after invoking the lexer and configured parser seam.

```rust
struct FrontendOutput {
    source: SourceUnit,
    preprocessed: PreprocessedSource,
    tokens: TokenStream,
    ast: Option<SurfaceAst>,
    diagnostics: Vec<Diagnostic>,
}
```

With the real parser seam, `ast = None` means parsing could not recover enough
structure for later phases. With the stub parser seam, `ast = None` is the
expected placeholder result. Lexical diagnostics are still returned; syntax
diagnostics are present only when the real parser seam runs.

## Error Recovery

Frontend recovery aims to report multiple useful diagnostics without inventing semantic facts.

Lexer recovery:

- emits `TokenKind::ErrorRecovery` for malformed spans when possible;
- resumes at whitespace, reserved delimiters, or line boundaries;
- preserves the malformed source range for diagnostics.

Parser recovery:

- synchronizes at `;`, `end`, `definition`, `registration`, `theorem`, `lemma`, `proof`, `algorithm`, and EOF;
- creates explicit error nodes where a construct is missing;
- avoids creating fake identifiers or fake resolved symbols;
- keeps doc comments near their original source location, even if attachment fails.
- keeps malformed annotations as syntax-level error nodes when possible.

Recovered AST nodes must be marked so later phases can skip or degrade gracefully.

## Diagnostics

Frontend diagnostics use the syntax/lexical ranges from [doc/spec/en/22.error_handling_and_diagnostics.md](../../../spec/en/22.error_handling_and_diagnostics.md).

| Diagnostic Class | Phase | Example |
|---|---|---|
| lexical precondition | source loading / preprocessing | invalid UTF-8, non-ASCII code character |
| comment structure | preprocessing | unterminated block comment |
| import pre-scan | preprocessing | malformed import prelude, raw scan unavailable for imports |
| lexical environment | active lexical environment | unresolved import, missing lexical summary, user-symbol import conflict |
| scope skeleton | lexing pre-disambiguation | malformed binder list, unmatched lexical block end |
| tokenization | lexing | malformed literal, unknown token |
| syntax | parsing | unexpected token, missing `end`, unmatched delimiter |
| annotation syntax | parsing | malformed annotation argument list |

Frontend diagnostics must include:

- stable diagnostic code;
- primary source span;
- secondary source anchor/span where useful, such as the opening delimiter for a
  missing close or a removed-comment/generated anchor from source mapping;
- recovery note when later diagnostics may be affected.

Frontend diagnostics must not claim semantic facts such as "undefined symbol" or "ambiguous overload"; those belong to later phases.

## Incrementality

Frontend cache keys are layered.

| Output | Cache Key |
|---|---|
| `SourceUnit` | normalized path, source bytes, edition, origin |
| `PreprocessedSource` | `source_hash`, frontend version |
| `ActiveLexicalEnvironment` | resolved imports, dependency lexical-summary fingerprints, import order |
| `TokenStream` | preprocessed hash, active lexical environment fingerprint, parser lexing context / parser-assisted lexing plan |
| `SurfaceAst` | token stream hash, parser version, edition |

Important invalidation rules:

- comment-only edits invalidate `PreprocessedSource` and documentation metadata, but may allow semantic outputs to be reused if lexical text is unchanged;
- import edits invalidate the active lexical environment, token stream, AST, and all semantic layers for that source file;
- dependency symbol export edits can invalidate tokenization even when the local source file is unchanged;
- parser lexing context or parser-assisted lexing-plan changes invalidate
  `TokenStream` even when source and imports are unchanged;
- parser version and language edition are part of the AST cache key.

## Alternatives Considered

### Alternative 1: Pure Lexer Before Import Resolution

Lex all source files without considering imported user symbols, then reinterpret tokens during parsing or name resolution.

This would make lexing simpler, but it conflicts with identifier-shaped user symbols and longest-match rules. It would also force later phases to split or merge token sequences, making source spans and parser recovery fragile.

### Alternative 2: Full Module Resolution Before Lexing

Resolve imports completely before lexing each file.

This gives the lexer precise symbol identities, but it creates a cycle: parsing is needed to understand module contents, while lexing needs imported symbols. The adopted shallow pre-scan breaks the cycle while preserving enough information to build the active lexical environment.

### Alternative 3: Lexer Decides All Dot Roles

Classify `.` as selector, namespace separator, or user functor directly in the lexer.

This fails when variable shadowing or term context determines the correct interpretation. The adopted split keeps compound tokens and active user symbols in the lexer, while leaving selector/namespace decisions to parser and resolver.

### Alternative 4: Treat Quotes as Global String Delimiters

Always lex `'...'` and `"..."` as string literals.

This would conflict with user symbols such as postfix inverse notation. The adopted contextual lexer recognizes string literals only at grammar positions that require them while still producing a complete `TokenStream` before parsing.

## Adopted Approach

Use a staged frontend:

1. load source and build source mapping;
2. preprocess comments, doc comments, and shallow imports;
3. build an active lexical environment from shallow imports and dependency lexical summaries;
4. raw-scan lexeme runs and disambiguate them with longest-match, active symbols, parser expectation, and scoped identifier override rules where needed;
5. parse into source-shaped `SurfaceAst` with explicit recovery nodes.

This design keeps frontend output source-faithful while avoiding semantic commitments that belong to module resolution and type checking.

## Constraints and Assumptions

- Source files are UTF-8 and end with `.miz`.
- Code regions are ASCII; comments may contain Unicode. Unicode in annotation
  string arguments is gated on the parser-assisted lexing contract.
- Import pre-scan is intentionally incomplete and cannot replace module resolution.
- The active lexical environment can affect token boundaries, so tokenization depends on dependency lexical-summary fingerprints.
- The lexer recognizes compound reserved tokens and active user symbols, but not semantic selector/namespace roles.
- The lexer emits complete `StringLiteral` tokens only when the parser lexing
  context requires strings; exact grammar-position recognition is gated on the
  parser-assisted lexing contract.
- The parser owns annotation syntax and attachment; preprocessing does not collect annotations as a separate metadata channel.
- `SurfaceAst` is source-shaped and may contain recovery nodes; later semantic phases must tolerate or reject these explicitly.
- `mizar-frontend` orchestrates parsing but does not own `SurfaceAst` node definitions or grammar implementation.
- Frontend artifacts are internal compiler data, not stable public build artifacts.
