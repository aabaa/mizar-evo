# Architecture: Incremental Verification Contract

> Canonical language: English. Japanese companion: [../ja/22.incremental_verification_contract.md](../ja/22.incremental_verification_contract.md).

## Purpose

This document defines the soundness contract for partial-edit incremental verification, cache reuse, proof/VC reuse, and parallel-compatible verification.

Incremental verification and parallel verification are optimizations over the same clean-build semantics. They must never change source-level meaning, proof acceptance, published artifacts, interface hashes, dependency-facing summaries, or canonical diagnostics.

This document is architecture-level. It fixes cross-phase correctness requirements and semantic data shapes, but it does not freeze Rust type names, field layouts, JSON schemas, MCP resources, LSP payloads, proof-witness wire formats, or cache database layouts.

## Normative Status

This document is normative for incremental verification soundness, cache-reuse preconditions, cross-edit obligation identity, snapshot freshness, and parallel-compatible scheduling.

It is non-normative for final crate APIs, Rust representation, JSON schema, MCP/LSP schema, proof-witness external format, and cache storage layout. Illustrative pseudocode below names concepts, not final implementation types.

## Design Decisions

### Clean-Build Equivalence

Incremental verification is an optimization.

For any accepted incremental result, deleting all internal caches and running a clean build on the same source snapshot, dependency artifacts, lockfile, toolchain, language edition, verifier policy, and accepted kernel evidence must produce the same externally visible proof acceptance, published artifacts, interface hashes, dependency-facing summaries, and canonical diagnostics.

Non-canonical local metadata such as timings, worker ids, temporary paths, backend runtime duration, cache lookup timing, and scheduling priority must not affect acceptance or published artifact identity.

If an implementation cannot prove that a cached result applies to the current `BuildSnapshot`, it must treat the cache lookup as a miss.

### Cache-Miss Fallback

Cache is not proof authority.

A cache hit is valid only after all relevant keys, dependency hashes, dependency slices, schema versions, toolchain information, verifier policy, and proof witness or deterministic discharge hashes have been validated.

Missing cache data, unsupported schema, incomplete dependency slices, incompatible verifier policy, unknown toolchain compatibility, or proof witness / deterministic discharge mismatch must force a cache miss.

A cache miss must change performance only. It must not change source-level semantics, proof acceptance, interface hashes, dependency-facing summaries, published artifact order, or canonical diagnostics.

### Snapshot-Scoped Results

Every incremental verification result belongs to exactly one `BuildSnapshotId`.

Diagnostics, proof statuses, phase outputs, cache records, and artifact drafts produced for an obsolete snapshot must not be published as current results.

An obsolete result may be reused only by passing through normal cache validation against the new `BuildSnapshot`.

Open editor buffers are source snapshots, not published artifacts. They may feed LSP builds and dry-runs, but they are not dependency artifacts until the artifact publication boundary accepts them.

### Identity Model: VcId vs ObligationAnchor

`ObligationAnchor` is a best-effort, cross-edit-stable identity for a verification obligation.

It is distinct from `VcId`.

`VcId` is deterministic within one `BuildSnapshot` and is used for canonical ordering, artifact ordering, diagnostic ordering, and result collation.

`ObligationAnchor` is used for cross-edit tracking, proof repair, LSP obligation tracking, AI patch dry-run, and per-obligation cache reuse candidates across edits.

`ObligationAnchor` is not proof evidence and is not trusted by the kernel.

The safety rules are:

- a matching `VcId` alone is never sufficient for cross-edit proof reuse;
- a matching `SourceRange` alone is never sufficient for cross-edit proof reuse;
- a matching `SurfaceNodeId` or rowan node identity alone is never sufficient for cross-edit proof reuse;
- a matching arena id, syntax-node order, or parser node order alone is never sufficient for cross-edit proof reuse;
- a matching `ObligationAnchor` alone is also not sufficient. It is only the first candidate match.

Cross-edit proof result reuse additionally requires all of:

- matching `ObligationAnchor`;
- matching canonical VC fingerprint;
- matching canonical local context fingerprint;
- matching dependency slice fingerprints;
- compatible verifier policy;
- matching proof witness hash or deterministic discharge hash.

Illustrative shape:

```rust
struct ObligationAnchor {
    owner: OwnerOriginId,
    kind: ObligationKind,
    local_path: LocalProofOrProgramPath,
    label_hint: Option<LabelId>,
    source_shape_hash: Hash,
    canonical_goal_hash: Hash,
    canonical_context_hash: Hash,
    generation_schema_version: SchemaVersion,
}
```

Field meaning:

| Field | Meaning |
|---|---|
| `owner` | Normalized public or module-local owner identity, such as theorem, registration, algorithm, or proof block owner |
| `kind` | Theorem proof obligation, registration coherence, reduction reducibility, algorithm assertion, loop invariant, termination obligation, and similar categories |
| `local_path` | Source-shaped path through proof/program structure, such as proof step path, case/suppose branch path, algorithm block path, or generated obligation site |
| `label_hint` | Optional user-written label when available; useful but not required |
| `source_shape_hash` | Hash of the local syntactic shape that generated the obligation |
| `canonical_goal_hash` | Canonical hash of the normalized goal formula |
| `canonical_context_hash` | Canonical hash of local assumptions, visible premises, relevant type facts, active registration view, and other context that affects the obligation |
| `generation_schema_version` | Version of the VC generation scheme |

The exact Rust type may differ. The contract is the semantic split between within-snapshot canonical order (`VcId`) and cross-edit candidate identity (`ObligationAnchor`).

If an anchor cannot be matched confidently after an edit, the obligation is treated as new and previous proof results must not be reused.

### Dependency Slice Completeness

Any phase output reused from cache must carry a complete dependency slice or dependency footprint for its phase. If that completeness cannot be established, the output must be marked `uncacheable` and cache lookup must miss.

Missing dependency data must never be interpreted as "no dependency".

The dependency slice used for theorem/VC reuse is the composition of the phase slices that affected that theorem or VC. A phase may initially use coarser slices than the ideal granularity, but it must not omit dependencies. False-positive rebuilds are acceptable; false-negative reuse is a soundness bug.

Conceptual phase slices:

| Slice | Required footprint |
|---|---|
| `LexicalParseSlice` | source hash; lexical hash; import pre-scan hash; imported lexical summary fingerprints; active lexical environment fingerprint; parser lexing plan hash; parser grammar/schema version; source-position-aware operator view hash; notation parse-effect fingerprints |
| `ResolveSlice` | imported module summaries; import/export edge fingerprints; resolved labels; resolved symbols; namespace and visibility decisions; selected module identities |
| `TypeRegistrationSlice` | modes and attributes consulted; registrations consulted; cluster facts used; reduction rules applied; active registration view hash; `ResolutionTrace` hashes |
| `OverloadSlice` | overload domain view hash; candidates considered at appropriate fingerprint granularity; selected root fingerprint; coherent refinement metadata used; inserted `qua` evidence, if any |
| `ElaborationSlice` | definitions unfolded; opacity/transparency policy; generated obligation seeds; source-to-core provenance; normalized owner ids |
| `VcSlice` | `ObligationAnchor`; `VcId` within the current `BuildSnapshot`; obligation seed hash; canonical local context hash; canonical goal hash; premise fingerprints; proof hints and computation policy |
| `AtpKernelSlice` | ATP problem hash; premise-selection hash; backend profile; encoding profile; kernel evidence hash; imported fact fingerprints; accepted proof witness hash or deterministic discharge hash |

Exact field names remain crate-level decisions. The rule is completeness, not a specific record layout.

### Frontend Lexical/Parse Footprints

Frontend cache footprints are not semantic `DependencySlice`s. They record tokenization/parsing-affecting inputs, not used theorem, definition, registration, or proof facts.

TokenStream reuse requires matching source lexical hash, active lexical environment fingerprint, imported lexical-summary fingerprints, parser lexing plan / filter hash, and lexer/schema versions. Bundle/source-level reuse still requires a compatible language edition before token and AST entries are considered.

SurfaceAst reuse requires matching `TokenStream` content hash, parser schema version, parser inputs hash, source-position-aware operator view hash, and language edition.

A cached `SurfaceAst` must not be reused if notation declarations, synonym/antonym parse effects, predicate/functor operator metadata, or the active operator view visible at the relevant source position changed.

### Registration, Cluster, and Reduction Activation

Parsed registrations are not active facts.

Only accepted registration contributions may enter importer-visible `RegistrationIndex`, `cluster-db`, reduction index, `ResolutionTrace`, `interface_hash`, or downstream summaries.

An unaccepted coherence, existence, compatibility, or reducibility proof must not make the corresponding registration visible to automated reasoning.

Changing any of the following invalidates all import-scoped registration, cluster, or reduction views that can see the origin:

- registration label;
- target pattern;
- guard;
- declared cluster or reduction contribution;
- coherence proof status;
- reducibility proof status;
- accepted witness;
- origin identity;
- export visibility.

Deleting or renaming a registration must remove stale origin contributions before any dependent cache hit is accepted.

`ResolutionTrace` hashes must participate in every VC/cache key that uses derived cluster or reduction facts.

### Proof Witness and Deterministic Discharge Reuse

Proof-related cache records may be reused only when they reference an accepted proof witness or deterministic built-in discharge record valid for the current verifier policy.

A proof witness hash mismatch is a cache miss.

A deterministic discharge hash mismatch is a cache miss.

Externally attested evidence must not be upgraded to kernel-verified status by cache reuse.

A backend success result is not proof acceptance unless the active policy and kernel boundary accept it.

### Parallel Compatibility

Incremental verification and parallel verification share the same correctness contract.

For the same `BuildSnapshot` and verifier policy, all of the following must agree on externally visible proof acceptance, published artifacts, interface hashes, dependency-facing summaries, and canonical diagnostics:

- clean sequential build;
- clean parallel build;
- incremental sequential build;
- incremental parallel build.

Worker completion order, cache lookup timing, cache hit/miss timing, ATP runtime duration, cancellation timing, and scheduling priority must not affect semantic results.

The scheduler task graph is the shared substrate for both incremental invalidation and parallel readiness. A task may run in parallel only when all dependency edges required for clean-build equivalence are satisfied.

Task dependency edges are correctness edges, not just scheduling hints. Removing or weakening a dependency edge for performance is a soundness bug.

Coarser task granularity is allowed initially; missing dependency edges are not.

`VcId` determines canonical order within one `BuildSnapshot`. Parallel workers may complete VCs in any order, but result collation, artifact order, and diagnostic order must use canonical order.

`ObligationAnchor` is the cross-edit tracking key and must not replace `VcId` ordering.

Artifact commit is the serialization boundary for parallel execution. Published artifacts must not be committed in worker completion order.

Portfolio winner selection must be deterministic under the active proof policy. "First backend to finish" must not be the semantic winner rule. If multiple acceptable evidence candidates exist, selection must use canonical backend priority, kernel-evidence strength, and deterministic tie-breakers.

Runtime duration may be recorded as provenance but must not participate in canonical proof identity.

### Edit-Class Invalidation Rules

| Change kind | Downstream semantic rebuild | Local rebuild / artifact refresh |
|---|---:|---:|
| Whitespace/comment outside syntax-sensitive notation | No | Diagnostics/docs only if needed |
| Theorem proof body only | No, if statement and accepted proof status remain unchanged | Yes |
| Theorem statement | Yes | Yes |
| Theorem accepted/open/failing status change | Yes for importers that can see/use the theorem | Yes |
| Definition body or definitional boundary | Yes | Yes |
| Algorithm body only | No, if exported signature and requires/ensures contract remain unchanged | Yes |
| Algorithm requires/ensures contract | Yes | Yes |
| Notation/operator metadata/parse effect | Yes for files whose active lexical/operator view can see it | Yes |
| Registration declaration | Yes for importers that can see it | Yes |
| Registration coherence/reducibility proof status | Yes for importers that can see the registration | Yes |
| Cluster/reduction contribution | Yes | Yes |
| Diagnostic wording only | No | Diagnostic artifact refresh only |

Theorem proof-body-only edits and registration proof-body edits are not equivalent. A registration coherence/reducibility proof status can change active reasoning facts, so it may invalidate downstream semantic work.

Semantic-versioning classification is not a cache-reuse predicate.

A semver heuristic, including overload-resolution shift heuristics, must not be used to skip resolver, checker, VC, ATP, kernel, or artifact work unless the relevant dependency fingerprints and slices match exactly.

### Alternatives Considered

#### Reuse by Source Range or Parser Node Identity

This is fast but unsound. Source ranges, rowan nodes, compatibility node ids, and arena ids are source-shape handles, not proof identities. Edits can preserve a range while changing semantics, or shift ranges while preserving an obligation.

#### Treat VcId as Cross-Edit Stable

`VcId` is excellent for deterministic ordering inside one snapshot, but inserting or deleting obligations can renumber later VCs. Using it as the only cross-edit key would either reuse stale proofs or miss safe reuse opportunities.

#### Trust Cache Hits as Proof Authority

This would make builds faster but would move proof authority out of the verifier/kernel boundary. The adopted contract requires validation of all relevant inputs, dependency slices, policy, and witness/discharge hashes before reuse.

#### First Completed Backend Wins

This maximizes latency improvement but makes proof identity depend on runtime scheduling. The adopted contract records runtime as provenance and selects evidence deterministically under policy.

### Adopted Approach

Mizar Evo treats clean-build behavior as the reference semantics. Incremental verification and parallel execution are allowed only when they reproduce the same externally visible results for the same `BuildSnapshot` and verifier policy.

The adopted identity model separates:

- `VcId`, the deterministic within-snapshot ordering and collation key;
- `ObligationAnchor`, the best-effort cross-edit tracking and reuse-candidate key;
- canonical VC/context/dependency/policy/witness fingerprints, the required proof-reuse validation keys.

Cache records are optimization data. They may accelerate builds only after validation and may be deleted without changing source-level semantics.

## Interface Definitions

The following pseudocode is conceptual. Final Rust names, field layouts, serialization, and storage decisions belong to crate-level design.

```rust
struct IncrementalReuseCandidate {
    snapshot: BuildSnapshotId,
    vc_id: VcId,
    anchor: ObligationAnchor,
    vc_fingerprint: Hash,
    local_context_fingerprint: Hash,
    dependency_slice_fingerprint: Hash,
    verifier_policy_fingerprint: Hash,
    proof_evidence: ProofEvidenceFingerprint,
}

enum ProofEvidenceFingerprint {
    AcceptedWitness(Hash),
    DeterministicDischarge(Hash),
}

struct PhaseDependencyFootprint {
    phase: PipelinePhase,
    schema_version: SchemaVersion,
    completeness: FootprintCompleteness,
    hashes: Vec<NamedHash>,
}

enum FootprintCompleteness {
    Complete,
    IncompleteUncacheable,
}
```

Validation predicate:

```text
Reusable(candidate, current_snapshot) iff
  candidate.anchor matches the current obligation confidently
  and candidate.vc_fingerprint == current canonical VC fingerprint
  and candidate.local_context_fingerprint == current local context fingerprint
  and candidate.dependency_slice_fingerprint == current composed dependency slice fingerprint
  and candidate.verifier_policy_fingerprint is compatible with the active policy
  and candidate.proof_evidence matches the accepted witness or deterministic discharge record
  and every referenced dependency artifact and schema is available and compatible.
```

Anything else is a cache miss.

## Affected Modules

Expected module-level owners include:

- `mizar-session` - owns `BuildSnapshotId`, `SourceId`, source hashes, and source range coordinate systems; it does not own `ObligationAnchor`
- `mizar-lexer` - owns tokenization inputs and lexical fingerprints
- `mizar-syntax` - owns source-shaped syntax and anchor ingredients without semantic identity
- `mizar-parser` - owns syntactic parse shape and source-position-aware parser operator views
- `mizar-frontend` - owns frontend cache footprints such as `LexicalParseSlice`
- `mizar-resolve` - owns normalized semantic owner/origin ids, imports, exports, labels, and symbol identity
- `mizar-core` - owns source-to-core provenance and obligation seeds
- `mizar-vc` - owns within-snapshot `VcId` assignment, `ObligationAnchor` construction, canonical VC fingerprints, and VC-local slice assembly
- `mizar-cache` - owns cache lookup, proof reuse validation, dependency-slice validation, and policy compatibility checks
- `mizar-build` - owns task graph construction, invalidation scheduling, reverse dependencies, and build/session orchestration
- `mizar-artifact` - owns stable published projections, proof witness references, interface and implementation hashes, and manifest publication

## Constraints and Assumptions

- This document does not implement Rust code.
- This document does not define final Rust trait signatures.
- This document does not define final JSON, LSP, MCP, or proof-witness wire schemas.
- This document does not design an incremental parser algorithm or cache database layout.
