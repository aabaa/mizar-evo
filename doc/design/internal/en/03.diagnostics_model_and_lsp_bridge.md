# Internal Design: Diagnostics Model and LSP Bridge

> Canonical language: English. Japanese companion: [../ja/03.diagnostics_model_and_lsp_bridge.md](../ja/03.diagnostics_model_and_lsp_bridge.md).

## Purpose

This document defines the internal diagnostics model and the LSP bridge that publishes diagnostics, editor metadata, code actions, and explanation queries.

It refines the architecture rule that diagnostics are structured phase outputs and that the LSP server reads immutable snapshots and stable artifacts rather than compiler-internal IR.

## Responsibilities

### Diagnostic Registry

`DiagnosticRegistry` owns stable diagnostic codes.

It provides:

- permanent `DiagnosticCode` allocation and retirement rules;
- mapping from numeric code to semantic name, default severity, phase family, and documentation URL;
- compatibility validation so a code is never reused for a different meaning;
- lookup data for CLI rendering, artifact emission, and LSP conversion.

Messages may improve across versions. Tools key on `DiagnosticCode`, not message text or semantic name.

### Diagnostic Producer API

Compiler phases emit structured diagnostic drafts through a `DiagnosticSink`.

Phase services own the local facts needed to create diagnostics, but they do not format CLI output or LSP diagnostics. They may attach structured details, explanation handles, and fix suggestions when those can be computed safely.

### Diagnostic Aggregator

`DiagnosticAggregator` owns build-snapshot normalization, identity assignment, deduplication, sorting, and publication readiness.

It consumes diagnostics from phase results, scheduler/build errors, and cache/artifact errors. It publishes immutable `BuildDiagnosticIndex` values for CLI, artifacts, and LSP snapshots.

Open-buffer diagnostics produced for unsaved editor text are LSP overlay diagnostics. They may reuse the same record shape, but they are not published to CLI output or `VerifiedArtifact`.

### LSP Bridge

`LspBridge` owns editor-facing protocol conversion.

It is responsible for:

- open-buffer source snapshots and document versions;
- converting compiler `SourceRange` values to LSP UTF-16 ranges;
- publishing diagnostics from the current `LspSnapshot`;
- serving hover, definitions, symbols, semantic tokens, inlay hints, code actions, and explanation requests;
- scheduling builds through the compiler driver.

It is not responsible for proof acceptance, type checking, ATP reasoning, or artifact mutation.

### Explanation Store

`ExplanationStore` owns lazy explanation handles and bounded explanation payloads.

Large traces remain in artifacts, cache-backed query data, or dedicated explanation files. Normal diagnostics store only compact references and bounded previews.

## Data Model

### Diagnostic Draft

```rust
struct DiagnosticDraft {
    code: DiagnosticCode,
    severity_override: Option<DiagnosticSeverity>,
    phase: PipelinePhase,
    stable_detail_key: String,
    message: String,
    primary_span: SourceRange,
    secondary_spans: Vec<DiagnosticSpan>,
    notes: Vec<DiagnosticNote>,
    details: DiagnosticDetails,
    fixes: Vec<FixSuggestion>,
    explanation: Option<ExplanationRef>,
}
```

Drafts are emitted by phase services before snapshot-wide identity assignment. They must already use `SourceRange` values grounded in the relevant source snapshot.

### Diagnostic Record

```rust
struct Diagnostic {
    handle: DiagnosticHandle,
    code: DiagnosticCode,
    semantic_name: String,
    severity: DiagnosticSeverity,
    phase: PipelinePhase,
    stable_detail_key: String,
    message: String,
    primary_span: SourceRange,
    secondary_spans: Vec<DiagnosticSpan>,
    notes: Vec<DiagnosticNote>,
    details: DiagnosticDetails,
    fixes: Vec<FixSuggestion>,
    explanation: Option<ExplanationRef>,
    related_ids: Vec<DiagnosticId>,
    freshness: DiagnosticFreshness,
}

struct DiagnosticHandle {
    snapshot: BuildSnapshotId,
    id: DiagnosticId,
}
```

`DiagnosticId` is deterministic within one `BuildSnapshotId`. It is derived from source identity, diagnostic code, phase, primary span, and a stable per-span ordinal after deduplication. Tools must interpret it through `DiagnosticHandle`; a bare `DiagnosticId` is not meaningful outside its snapshot.

### Diagnostic Index

```rust
struct BuildDiagnosticIndex {
    snapshot: BuildSnapshotId,
    by_source: BTreeMap<SourceId, Vec<Diagnostic>>,
    by_id: BTreeMap<DiagnosticId, DiagnosticRef>,
    phase_status: BTreeMap<WorkUnit, PhaseStatus>,
}

struct LspDiagnosticOverlay {
    document: DocumentUri,
    version: SourceVersion,
    diagnostics: Vec<Diagnostic>,
}
```

The build index is immutable after publication. CLI rendering, artifact projection, and LSP semantic diagnostics all read this same index. LSP overlay diagnostics are editor-current additions and are never projected into published artifacts.

### Fix Suggestion

```rust
struct FixSuggestion {
    title: String,
    confidence: FixConfidence,
    edits: Vec<TextEdit>,
    command: Option<DiagnosticCommand>,
    applicability: Applicability,
    required_snapshot: Option<BuildSnapshotId>,
    required_text_hash: Option<Hash>,
}
```

Any edit that depends on source text must validate the target range and expected text before application. Artifact-assisted fixes also validate artifact hash and source hash.

### Explanation Reference

```rust
struct ExplanationRef {
    kind: ExplanationKind,
    diagnostic: Option<DiagnosticHandle>,
    expr_id: Option<ExprId>,
    vc_id: Option<VcId>,
    source: ExplanationSource,
    summary_hash: Hash,
    preview: Option<String>,
}

enum ExplanationSource {
    Artifact(PathBuf),
    CacheRecord(CacheKey),
    QueryService(ExplanationQuery),
}
```

Explanation references are handles. They are invalid outside the snapshot or artifact hash they were created for. `CacheRecord` sources are optional acceleration or backing data for local explanation queries; published diagnostics must remain interpretable without cache records, and cache-backed explanations never determine proof status.

### LSP Snapshot

```rust
struct LspSnapshot {
    snapshot_id: BuildSnapshotId,
    artifact_manifest: ArtifactManifestRef,
    source_versions: Vec<SourceVersion>,
    diagnostics: BuildDiagnosticIndex,
    overlays: Vec<LspDiagnosticOverlay>,
    expression_index: ExpressionIndex,
    symbol_index: SymbolIndex,
    explanation_index: ExplanationIndex,
    missing_facts: Option<MissingFactsRef>,
    freshness: SnapshotFreshness,
}

enum SnapshotFreshness {
    Current,
    StaleArtifact,
    Pending,
}
```

The snapshot is atomically swapped into request handlers. In-flight requests finish on the snapshot they started with.

## Control Flow

### Phase Diagnostic Production

1. A phase service receives a `PhaseContext` with a `DiagnosticSink`.
2. The phase emits `DiagnosticDraft` values with source ranges and structured details.
3. The phase returns drafts in `PhaseResult`.
4. The scheduler forwards phase diagnostics and phase status to the aggregator.
5. The aggregator does not publish diagnostics until it has a coherent snapshot boundary for the target consumer.

Fatal and blocking phase statuses are recorded separately from diagnostic severity. An error diagnostic may be recoverable, and a phase may be blocked because an upstream dependency failed.

### Build Aggregation

1. Normalize paths to workspace-relative `SourceId`s.
2. Drop or mark obsolete diagnostics whose source snapshot no longer matches the target publication.
3. Deduplicate diagnostics that have the same code, phase, primary span, stable detail key, normalized details hash, fix identity, and explanation identity. Message text and localized/rendered strings are never diagnostic identity.
4. Assign deterministic `DiagnosticId`s.
5. Sort by source file, source range, phase order, severity, diagnostic code, stable detail key, and stable ordinal.
6. Attach freshness and phase status.
7. Publish a `BuildDiagnosticIndex` to CLI, artifact commit, and LSP snapshot construction.

Deduplication must not merge diagnostics with different primary spans, different fix edits, or different explanation refs.

Open-buffer overlay diagnostics skip artifact projection. They are stored in `LspDiagnosticOverlay` values keyed by document version and are replaced by newer overlay diagnostics or by a full build snapshot for the same source version.

### Artifact Projection

During phase 15, `ArtifactService` embeds compact diagnostics into `VerifiedArtifact` and writes larger explanation payloads as development artifacts when enabled.

Artifacts store:

- stable diagnostic records needed by downstream tools;
- explanation refs, not bulk traces;
- source ranges in compiler-native coordinates;
- hashes for external explanation files.

LSP range conversion is not stored in artifacts.

### LSP Snapshot Publication

1. The LSP bridge receives `SnapshotPublished` build events as the normal coherent publication boundary.
2. It reads `artifact-manifest.json` first and resolves module artifacts by manifest entries.
3. It builds expression, symbol, diagnostic, and explanation indexes for the snapshot.
4. It overlays open-buffer source versions for files currently edited in the IDE.
5. It atomically swaps the new `LspSnapshot` into request handlers.
6. It publishes diagnostics for affected files if the document version still matches or can be safely mapped.

`ArtifactCommitted` events may warm caches or invalidate per-module artifact readers, but they do not by themselves trigger a full `LspSnapshot` swap unless the build service also marks the package snapshot coherent.

When semantic artifacts are stale relative to an open buffer, the bridge may keep stale diagnostics visible but marks them as `stale_artifact` and suppresses destructive code actions.

### Lightweight Open-Buffer Diagnostics

The LSP bridge may request syntax-only frontend work for open buffers before a full build completes.

These diagnostics:

- use the current editor text snapshot;
- are marked `current`;
- may replace older syntax diagnostics for the same document version;
- do not claim verified semantic status;
- are superseded when a full build snapshot for the same source version is published.

### Code Actions

1. LSP receives a code action request with document version and range.
2. The bridge looks up diagnostics and fix suggestions in the current snapshot.
3. Local text edits validate current range and expected text.
4. Artifact-assisted edits validate source hash, artifact hash, and relevant explanation or `missing_facts` handle.
5. Command actions capture snapshot and diagnostic handles and revalidate when the command executes.

Proof-related suggestions must distinguish backend-derived candidates from kernel-verified `used_axioms`.

### Explanation Queries

1. The request supplies a `DiagnosticId`, `ExprId`, or `VcId` plus a snapshot id.
2. `ExplanationStore` resolves the handle against artifact hashes or cache keys.
3. The query service enforces size and depth bounds.
4. The response includes freshness, source ranges, and bounded structured explanation data.

If the handle belongs to an obsolete snapshot, the LSP bridge returns a stale-handle response and may suggest re-running the query on the current diagnostic when a matching code/span exists.

## API Sketch

### Diagnostic Registry

```rust
trait DiagnosticRegistry {
    fn lookup(&self, code: DiagnosticCode) -> Option<DiagnosticDescriptor>;
    fn validate_descriptor(&self, descriptor: DiagnosticDescriptor) -> Result<(), RegistryError>;
}

struct DiagnosticDescriptor {
    code: DiagnosticCode,
    semantic_name: String,
    default_severity: DiagnosticSeverity,
    phase_family: DiagnosticPhaseFamily,
    documentation: Option<DocUrl>,
}
```

### Diagnostic Sink and Aggregator

```rust
trait DiagnosticSink {
    fn emit(&self, diagnostic: DiagnosticDraft);
}

trait DiagnosticAggregator {
    fn ingest(&self, source: DiagnosticSource, diagnostics: Vec<DiagnosticDraft>, status: PhaseStatus);
    fn publish(&self, snapshot: BuildSnapshotId) -> BuildDiagnosticIndex;
}
```

### LSP Bridge

```rust
trait LspBridge {
    fn open_document(&self, document: TextDocumentItem);
    fn change_document(&self, change: TextDocumentChange);
    fn publish_snapshot(&self, snapshot: LspSnapshot);
    fn diagnostics_for(&self, document: DocumentUri) -> Vec<LspDiagnostic>;
    fn code_actions(&self, request: CodeActionRequest) -> Vec<CodeAction>;
    fn explain(&self, request: ExplainRequest) -> Result<ExplainResponse, ExplainError>;
}
```

### Range Conversion

```rust
trait RangeMapper {
    fn to_lsp_range(&self, source: SourceId, range: SourceRange, version: SourceVersion) -> Result<LspRange, RangeError>;
    fn validate_edit_target(&self, edit: &TextEdit, version: SourceVersion) -> Result<(), RangeError>;
}
```

Range conversion is performed only at the LSP boundary using the exact source snapshot for the request.

## Error Handling

### Diagnostic Registry Errors

Invalid or reused diagnostic codes are build-time registry errors. They are not emitted as normal user diagnostics because they indicate a compiler implementation bug.

### Aggregation Errors

If a diagnostic references a missing source range or incompatible source snapshot, the aggregator reports an internal diagnostic in developer modes and drops the broken record from user-facing publication. It must not publish a range computed for another document version.

### LSP Snapshot Errors

Unsupported artifact schemas, missing manifests, or stale artifact hashes degrade LSP features by freshness class.

| Condition | LSP Behavior |
|---|---|
| missing artifact manifest | syntax-only features for open buffers |
| unsupported artifact schema | publish compatibility diagnostic when source range is known; disable semantic features |
| stale artifact for open buffer | keep stale diagnostics marked `stale_artifact`; suppress unsafe edits |
| missing explanation payload | show compact diagnostic preview and report explanation unavailable |

### Cancellation

Open-buffer diagnostics and explanation queries are cancellable. Published snapshots are immutable and are not cancelled after publication.

## Constraints and Assumptions

- Diagnostic codes are stable and never reused.
- `DiagnosticId` is deterministic only within one build snapshot.
- LSP requests carry diagnostic handles with snapshot identity, not bare diagnostic IDs.
- Compiler diagnostics store source ranges, not LSP UTF-16 positions.
- Diagnostic aggregation is the single source of ordering for CLI, artifacts, and LSP.
- Open-buffer overlay diagnostics are not emitted into `VerifiedArtifact`.
- Cache-backed explanations are optional and never establish proof status.
- LSP never accepts proof results and never mutates build artifacts directly.
- Open-buffer source snapshots override disk text for editor features.
- Code actions validate range and text before applying edits.
- Explanation APIs are bounded and query-oriented, not bulk dumps.
- Missing or stale artifacts must degrade editor features gracefully.
