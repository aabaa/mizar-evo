# Internal Design: Documentation Extraction

> Canonical language: English. Japanese companion: [../ja/05.documentation_extraction.md](../ja/05.documentation_extraction.md).

## Purpose

This document defines the internal design of phase 16 documentation generation and code extraction.

It refines the phase 16 architecture into scheduler tasks, artifact readers, documentation input models, render indexes, runtime extraction projections, output manifests, diagnostics, and cache boundaries. Phase 16 is part of the batch build graph, but documentation rendering and code extraction are optional task classes. They consume verified artifacts and source bundles; they do not establish proof validity or change phase 0-15 results.

## Responsibilities

### Doc Extraction Service

`DocExtractionService` is the phase 16 phase service registered with the scheduler.

It owns:

- expanding one `DocumentationExtract` scheduler task into documentation and extraction subtasks according to the build profile;
- validating that all required verified artifacts are committed in the package artifact manifest;
- choosing cache keys for documentation and extraction outputs;
- routing documentation diagnostics and extraction diagnostics to the build diagnostic sink;
- reporting output manifests to the driver.

It does not own proof acceptance, artifact commit for `*.mizir.json`, or source-level semantic decisions.

### Artifact Documentation Reader

`ArtifactDocReader` owns stable input loading for phase 16.

It reads:

- `artifact-manifest.json`;
- published `*.mizir.json` verified artifacts;
- package manifest metadata;
- source files or packaged source bundles for documentation comments and snippets;
- optional implementation caches only after cache-key validation.

It must prefer stable artifacts over compiler-internal IR. If a source file is unavailable, documentation may still render semantic pages with source-link diagnostics; extraction must fail for items whose required source or source map is missing.

### Documentation Builder

`DocumentationBuilder` owns documentation input normalization, comment attachment, link resolution, math rendering, HTML rendering, search index generation, and staged render outputs.

It is responsible for:

- building a `DocBuildPlan` from package metadata and verified module artifacts;
- attaching `:::` comments to module and item identities;
- resolving inline references, Markdown links, `@see` tags, and section references;
- applying `@latex` metadata and default formula rendering;
- emitting deterministic page paths, anchors, search indexes, and optional machine-readable doc indexes.

Documentation warnings do not invalidate proof results unless the active publication profile promotes them to errors.

### Extraction Builder

`ExtractionBuilder` owns executable item selection, `RuntimeIr` projection, ghost/proof erasure, target lowering, generated source maps, and extraction manifest emission.

It is responsible for:

- selecting extractable algorithms from exports, extraction configuration, or CLI-selected roots;
- validating proof status and extraction policy for required obligations;
- projecting verified executable algorithm data into target-neutral `RuntimeIr`;
- erasing ghost state, proof-only assertions, annotations, proof hints, and theorem/proof structures;
- rejecting unsupported runtime constructs explicitly;
- invoking target backends for deterministic code generation.

Extraction does not prove additional properties. It trusts phase 14 proof acceptance and phase 15 artifact integrity.

### Output Publisher

`Phase16OutputPublisher` owns atomic writes for generated documentation and extracted code.

It writes to phase 16 output directories, not to the phase 15 verified artifact transaction. It must still use temporary files, complete-directory staging where needed, content hashes, and atomic rename so readers never observe partially-written indexes or generated code.

## Data Model

### Phase 16 Request

```rust
struct DocumentationExtractInput {
    package_id: PackageId,
    package_root: PathBuf,
    artifact_manifest: ArtifactManifestRef,
    source_bundle: Option<SourceBundleRef>,
    doc_config: Option<DocConfig>,
    extraction_config: Option<ExtractionConfig>,
    profile: Phase16Profile,
    snapshot: BuildSnapshotId,
}

struct Phase16Profile {
    render_docs: bool,
    extract_code: bool,
    fail_on_doc_warnings: bool,
    fail_on_extraction_warnings: bool,
    include_private_docs: bool,
}
```

`render_docs` and `extract_code` are independent switches. A verification-only build disables both without changing phases 0-15.

`source_bundle` is optional because published artifacts may be available without packaged source. Documentation can still render semantic pages in that case, with diagnostics for unavailable comments, snippets, and source links. Extraction requires the source maps and executable projections needed for selected items; missing required source data blocks those extraction items.

### Artifact Inputs

```rust
struct Phase16ArtifactSet {
    manifest_hash: Hash,
    package_metadata: PackageMetadata,
    modules: Vec<VerifiedModuleInput>,
    dependency_indexes: Vec<DependencyDocIndexRef>,
}

struct VerifiedModuleInput {
    module_path: ModulePath,
    artifact_ref: VerifiedArtifactRef,
    artifact_hash: Hash,
    interface_hash: Hash,
    implementation_hash: Hash,
    source_ref: Option<SourceFileRef>,
    exports: Vec<ExportedItem>,
    proof_summaries: Vec<ProofSummary>,
}
```

Documentation uses `exports`, semantic signatures, theorem statements, labels, source ranges, proof summaries, and `@latex` projections from the verified artifacts. Extraction uses verified executable algorithm projections and source maps.

### Documentation Inputs

```rust
struct DocBuildPlan {
    package_id: PackageId,
    package_version: Option<Version>,
    modules: Vec<DocModuleInput>,
    dependency_indexes: Vec<DocIndexRef>,
    output_dir: PathBuf,
    options: DocOptions,
}

struct DocModuleInput {
    module_path: ModulePath,
    artifact: VerifiedArtifactRef,
    source: Option<SourceFileRef>,
    comments: Vec<DocCommentBlock>,
    items: Vec<DocItemInput>,

    attachment: DocAttachmentCandidate,
    raw_markdown: String,
    source_range: SourceRange,

```

`DocAttachmentCandidate` records both source adjacency and the verifier-provided module/item identity. Attachment succeeds only when both point to the same verified item.


Anchors are derived from exported labels and FQNs, not from rendered notation.

### Extraction Inputs and Runtime IR

```rust
struct ExtractionPlan {
    package_id: PackageId,
    artifact_manifest_hash: Hash,
    target: ExtractionTarget,
    roots: Vec<ExtractionRoot>,
    options: ExtractionOptions,
}

struct ExtractableItem {
    symbol: SymbolId,
    module_path: ModulePath,
    source_range: SourceRange,
    obligations: Vec<VcId>,
    proof_statuses: Vec<ProofStatus>,
    control_flow_ref: ExecutableProjectionRef,
}

    algorithm: SymbolId,
    signature: RuntimeSignature,
    blocks: Vec<RuntimeBlock>,
    source_map: RuntimeSourceMap,
    required_runtime: Vec<RuntimeCapability>,
}
```

`RuntimeIr` is a target-neutral extraction projection. It is not proof evidence and must not be fed back into the verifier as an authority.

### Phase 16 Output Manifest

```rust
struct Phase16OutputManifest {
    schema_version: SchemaVersion,
    package_id: PackageId,
    artifact_manifest_hash: Hash,
    doc_output: Option<DocOutputManifest>,
    extraction_output: Option<ExtractionManifest>,
    diagnostics_hash: Option<Hash>,
}

struct DocOutputManifest {
    output_dir: PathBuf,
    index_file: PathBuf,
    search_index: PathBuf,
    content_hash: Hash,
    options_hash: Hash,
}

struct ExtractionManifest {
    target: ExtractionTarget,
    output_dir: PathBuf,
    items: Vec<ExtractedItem>,
    content_hash: Hash,
    options_hash: Hash,
}
```

The phase 16 output manifest is traceability metadata for generated outputs. It is not a package verification artifact and does not replace `artifact-manifest.json`.

Paths recorded in phase 16 output manifests are package-relative by default. Workspace-relative paths may be used only for cross-package generated outputs. Absolute paths are restricted to local diagnostics and must not participate in reproducible output hashes.

## Control Flow

### Scheduling

1. The scheduler creates `DocumentationExtract` tasks after phase 15 commits verified artifacts.
2. The driver includes phase 16 tasks only when the build profile enables documentation rendering or code extraction.
3. The task depends on the package artifact manifest and all module artifact commits required by the selected package.
4. Cache keys are built per enabled subtask from the artifact manifest hash, source hashes needed for that subtask, documentation options or extraction target options, toolchain versions, and schema versions.
5. Cache hits return output manifests and diagnostics; cache misses execute the requested builders.

Phase 16 tasks may run in parallel across packages. Within one package, documentation link resolution requires a package-wide index, while extraction lowering may run per selected item after the extraction plan is fixed.

### Documentation Pipeline

1. `ArtifactDocReader` loads the artifact set and source comment blocks.
2. `DocumentationBuilder` builds a package-level symbol and label index from exports and dependency doc indexes.
3. Comments are attached to verified module/item identities.
4. Markdown is parsed with source-range preservation.
5. Inline code links, explicit Markdown links, `@see` tags, and section references are resolved.
6. Formula rendering metadata is applied from `@latex` and default syntax rendering.
7. Pages, search index, cross-reference index, used-by index, and optional machine-readable index are rendered.
8. `Phase16OutputPublisher` writes the staged site atomically and records the doc output manifest.

Unresolved or ambiguous links produce diagnostics and render safely as text.

### Extraction Pipeline

1. `ArtifactDocReader` loads verified executable algorithm projections and dependency summaries.
2. `ExtractionBuilder` selects roots from exports, extraction config, or CLI request.
3. Each candidate is classified as `Extractable`, `SpecOnly`, `BlockedByProof`, `BlockedByTarget`, or `BlockedByDependency`.
4. Extractable candidates are projected into `RuntimeIr`.
5. Ghost state, proof-only structures, annotations, proof hints, and specification-only snapshots are erased.
6. The target backend lowers `RuntimeIr` into generated files and reverse source maps.
7. `Phase16OutputPublisher` writes generated files and `ExtractionManifest` atomically.

Unsupported runtime constructs block only the affected extraction item unless the selected profile requires all requested roots to extract.

### Diagnostics

Documentation and extraction diagnostics use the same `Diagnostic` model as phases 0-15, but their phase family is `Documentation` or `Extraction`.

Rules:

- doc comment diagnostics point to preserved `:::` source ranges when available;
- generated-output diagnostics include source ranges and generated file references when both exist;
- missing source for documentation comments is a warning by default;
- stale or incompatible verified artifacts are errors because phase 16 cannot establish semantics itself;
- publication profiles may promote documentation warnings or extraction warnings to errors.

### Cache and Regeneration

Phase 16 outputs are reproducible derived products.

Cache reuse requires:

- same package id and artifact manifest hash;
- same source hashes for comments, snippets, and source maps;
- same dependency documentation index hashes when link resolution uses dependencies;
- same documentation options and extraction target options;
- compatible phase 16 schema versions and renderer/backend versions.

Deleting phase 16 outputs or cache records must not change verification results. They can be regenerated from verified artifacts and source bundles.

## Error Handling

### Documentation Errors

| Condition | Handling |
|---|---|
| unresolved link | warning by default; render text without link |
| ambiguous label | warning; require FQN or explicit reference |
| malformed Markdown | warning with degraded rendering |
| invalid LaTeX | warning or rendering marker |
| missing doc comment | optional lint |
| missing source bundle | warning for documentation; omit source snippets |
| stale artifact manifest | error; do not render semantic docs |

### Extraction Errors

| Condition | Handling |
|---|---|
| open or disallowed proof status | `BlockedByProof`; do not emit item |
| unsupported runtime construct | `BlockedByTarget`; do not emit item |
| ghost value escapes into runtime | extraction error; indicates invalid projection or earlier checker bug |
| missing dependency mapping | `BlockedByDependency`; do not emit item |
| backend emission failure | extraction error for the affected target |
| partial output write | discard staged directory and keep previous output visible |

### Cancellation

Cancellation discards staged phase 16 outputs unless they were already atomically published for the same snapshot and cache key. Cancelled phase 16 work must not roll back verified artifacts.

## Constraints and Assumptions

- Phase 16 requires verified artifacts for semantic output.
- Documentation and extraction are optional batch-graph tasks.
- Phase 16 must not mutate `*.mizir.json`, proof witnesses, or `artifact-manifest.json`.
- Documentation may degrade gracefully for presentation errors.
- Extraction must reject unsupported runtime constructs explicitly.
- Ghost state and proof-only annotations must not appear in extracted runtime code.
- Generated outputs are deterministic for the same artifact manifest, source hashes, options, target, and toolchain versions.
- Phase 16 outputs are derived products and may be deleted and regenerated without changing proof validity.
