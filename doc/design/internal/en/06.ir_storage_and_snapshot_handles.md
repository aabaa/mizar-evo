# Internal Design: IR Storage and Snapshot Handles

> Canonical language: English. Japanese companion: [../ja/06.ir_storage_and_snapshot_handles.md](../ja/06.ir_storage_and_snapshot_handles.md).

## Purpose

This document defines how internal IR outputs are stored, identified, shared, invalidated, cached, and projected across build snapshots.

It refines the architecture IR layering rules into implementation-facing handles and lifetime contracts. IR values are compiler-internal immutable snapshots. Phase tasks exchange handles to those snapshots rather than mutable IR objects. Published artifacts remain stable external projections and are not raw IR dumps.

## Responsibilities

### IR Storage Service

`IrStorageService` owns compiler-internal phase output storage for a workspace build manager.

It is responsible for:

- allocating immutable storage slots for phase outputs;
- returning typed `PhaseOutputRef<T>` handles after a phase output is sealed;
- storing large outputs either in memory or in content-addressed internal blobs;
- enforcing snapshot ownership and lifetime rules;
- retaining outputs across build sessions while watch/LSP snapshots, diagnostics, or explanation requests still reference them;
- dropping unreferenced outputs after dependent tasks, LSP snapshots, diagnostics, or cache writers no longer need them.

It does not own published artifact serialization or cache compatibility decisions.

### Snapshot Handle Registry

`SnapshotHandleRegistry` maps source, build, and phase-output identities to stable handles.

It is responsible for:

- assigning `BuildSnapshotId` values to exact source/dependency/toolchain/config states;
- assigning `SourceId`, `ModuleId`, `ItemId`, `ExprId`, `VcId`, and related identities within one snapshot;
- recording parent/derived relationships between phase outputs;
- preventing handles from being reused across incompatible snapshots without cache validation.

### Phase Output Publisher

`PhaseOutputPublisher` is the narrow API used by phase services to seal output values.

It is responsible for:

- validating that the output belongs to the task's snapshot and work unit;
- computing content hashes using canonical internal encoding for cache purposes;
- attaching source maps and diagnostic side tables;
- publishing immutable handles to dependent tasks.

It must not expose partially-built IR values to other tasks.

### IR Cache Adapter

`IrCacheAdapter` converts internal phase outputs to and from cache records.

It is responsible for:

- deciding which phase outputs are cacheable;
- serializing cache records with schema versions and dependency summaries;
- validating cache hits before reconstructing `PhaseOutputRef<T>` handles;
- refusing to treat cache hits as proof authority.

Cache records may contain raw internal IR encodings. Published artifacts must not.

### Artifact Projection Boundary

`ArtifactProjectionService` reads sealed phase outputs and creates `VerifiedArtifactDraft` values for phase 15.

It is responsible for projecting stable external metadata from internal IR:

- exported symbols, labels, and source ranges;
- rendered or normalized signatures;
- proof status and witness references;
- diagnostics and explanation refs;
- documentation and LSP metadata needed by downstream tools.

It must not publish raw `SurfaceAst`, `TypedAst`, `CoreIr`, `ControlFlowIr`, `VcIr`, `AtpProblem`, or kernel-internal state.

## Data Model

### Snapshot Identity

```rust
struct BuildSnapshot {
    id: BuildSnapshotId,
    workspace_root: PathBuf,
    source_versions: Vec<SourceVersion>,
    dependency_artifacts: Vec<DependencyArtifactRef>,
    lockfile_hash: Hash,
    toolchain: ToolchainInfo,
    verifier_config_hash: Hash,
}

struct SourceVersion {
    source_id: SourceId,
    package_id: PackageId,
    module_path: ModulePath,
    normalized_path: PathBuf,
    source_hash: Hash,
    edition: Edition,
}
```

`BuildSnapshotId` identifies the complete build input state, not merely the source text of one file. Handles derived from one snapshot cannot be used as current results for another snapshot unless a cache adapter validates the equivalence.

### Phase Output Reference

```rust
struct PhaseOutputRef<T> {
    id: PhaseOutputId,
    phase: PipelinePhase,
    unit: WorkUnit,
    snapshot: BuildSnapshotId,
    content_hash: Hash,
    storage: OutputStorageRef<T>,
    side_tables: SideTableRef,
    lineage: PhaseOutputLineage,
}

struct PhaseOutputLineage {
    producer_task: TaskId,
    input_hashes: Vec<NamedHash>,
    dependency_outputs: Vec<PhaseOutputId>,
}
```

`PhaseOutputRef<T>` is immutable after publication. The type parameter is compile-time documentation of the expected output kind; dynamic storage still records a schema version and runtime kind tag. `side_tables` is a stable handle to source maps, diagnostics, explanation refs, and documentation attachment metadata associated with the output.

### Storage References

```rust
enum OutputStorageRef<T> {
    InMemory {
        arena: ArenaId,
        slot: OutputSlotId,
        marker: PhantomData<T>,
    },
    Blob {
        blob: ContentBlobRef,
        schema_version: SchemaVersion,
        marker: PhantomData<T>,
    },
    CacheRecord {
        record: CacheRecordRef,
        output_hash: Hash,
        marker: PhantomData<T>,
    },
}
```

The storage reference is not a publication path. It may point to process memory, temporary files, or internal cache blobs that can be deleted without changing package semantics.

### IR Identity Tables

```rust
struct IrIdentityTables {
    source_ids: IdTable<SourceId, SourceIdentity>,
    module_ids: IdTable<ModuleId, ModuleIdentity>,
    item_ids: IdTable<ItemId, ItemIdentity>,
    expr_ids: IdTable<ExprId, ExprIdentity>,
    vc_ids: IdTable<VcId, VcIdentity>,
}
```

Identity tables are scoped to a `BuildSnapshotId`. Stable public identities are projected through artifact schemas; internal ids are not compatibility promises.

### Source Maps and Side Tables

```rust
struct IrSourceMap {
    source_id: SourceId,
    ranges: Vec<IrRangeEntry>,
}

struct IrSideTables {
    source_maps: Vec<IrSourceMap>,
    diagnostics: Vec<DiagnosticDraft>,
    explanation_refs: Vec<ExplanationDraftRef>,
    doc_attachments: Vec<DocAttachmentId>,
}
```

Source maps and side tables are stored alongside the output value and referenced from the output handle so diagnostics, artifacts, LSP, and documentation can agree on source ranges without mutating the IR value.

## Control Flow

### Phase Output Publication

1. A phase task receives input handles whose snapshots match the task snapshot.
2. The phase service builds an output value and side tables in task-local mutable storage.
3. `PhaseOutputPublisher` validates work-unit identity, source-map coverage, and declared output kind.
4. The publisher computes a content hash from canonical internal encoding, named input hashes, and dependency output content hashes.
5. The output is sealed into `IrStorageService`.
6. The scheduler receives a `PhaseOutputRef<T>` and passes it to dependent tasks.

Dependent tasks never observe partially-built IR.

### Cache Hit Rehydration

1. The scheduler asks `CacheStore` for a record using the task cache key.
2. `IrCacheAdapter` validates schema versions, toolchain compatibility, dependency hashes, source hashes, and policy fingerprints.
3. The adapter reconstructs an output handle in the current `BuildSnapshotId`.
4. The reconstructed handle records `OutputStorageRef::CacheRecord` lineage and side-table refs rehydrated from the cache record.
5. Downstream tasks consume it like an ordinary completed output.

For proof-related outputs, cache validation must still validate accepted proof witnesses or deterministic discharge records. The cache hit itself is not proof authority.

### Snapshot Replacement for Watch and LSP

1. A source edit creates a new `BuildSnapshotId`.
2. Handles from the old snapshot remain readable for stale diagnostics and explanation requests while referenced.
3. New tasks may reuse old data only through cache validation or explicit unchanged-source checks.
4. The LSP bridge atomically switches current semantic snapshots after diagnostic aggregation.
5. Unreferenced old handles are eligible for collection.

Open buffers are source snapshots, not published artifacts. LSP features must report whether data comes from the current snapshot, a stale snapshot, or syntax-only fallback.

### Artifact Projection

1. `ArtifactProjectionService` reads sealed phase outputs for one module.
2. It projects stable public metadata into `VerifiedArtifactDraft`.
3. Internal ids are mapped to artifact ids, FQNs, labels, and source ranges.
4. Raw IR storage refs are dropped from the draft.
5. `ArtifactService` serializes and publishes the artifact through phase 15 manifest transactions.

After artifact publication, downstream packages, documentation, and LSP artifact readers consume artifacts rather than internal IR storage.

## API Sketch

### IR Storage Service

```rust
trait IrStorageService {
    fn seal<T>(&self, output: SealOutputInput<T>) -> Result<PhaseOutputRef<T>, IrStorageError>;
    fn get<T>(&self, handle: &PhaseOutputRef<T>) -> Result<Arc<T>, IrStorageError>;
    fn side_tables(&self, handle: SideTableRef) -> Result<Arc<IrSideTables>, IrStorageError>;
    fn retain(&self, handle: PhaseOutputId, owner: RetainOwner) -> RetainGuard;
    fn collect(&self, snapshot: BuildSnapshotId) -> CollectionSummary;
}
```

### Snapshot Handle Registry

```rust
trait SnapshotHandleRegistry {
    fn create_snapshot(&self, input: SnapshotInput) -> BuildSnapshot;
    fn source_identity(&self, snapshot: BuildSnapshotId, source: SourcePath) -> Option<SourceVersion>;
    fn derive_output_id(&self, input: OutputIdentityInput) -> PhaseOutputId;
    fn is_current(&self, snapshot: BuildSnapshotId) -> bool;
}
```

### Phase Output Publisher

```rust
trait PhaseOutputPublisher {
    fn publish<T>(&self, task: TaskId, output: PhaseOutputDraft<T>) -> Result<PhaseOutputRef<T>, PublishError>;
}

struct PhaseOutputDraft<T> {
    phase: PipelinePhase,
    unit: WorkUnit,
    snapshot: BuildSnapshotId,
    value: T,
    side_tables: IrSideTables,
    dependencies: Vec<AnyPhaseOutputRef>,
}
```

The publisher derives output hashes from `value`, `side_tables`, named input hashes, and dependency `content_hash` values. It must not include `PhaseOutputId`, `TaskId`, memory addresses, or session-local storage ids in the content hash.

### IR Cache Adapter

```rust
trait IrCacheAdapter<T> {
    fn cache_key(&self, input: CacheKeyInput) -> CacheKey;
    fn encode(&self, output: &PhaseOutputRef<T>) -> Result<CacheRecordDraft, CacheEncodeError>;
    fn rehydrate(&self, record: CacheRecordRef, snapshot: BuildSnapshotId) -> Result<PhaseOutputRef<T>, CacheReuseError>;
}
```

### Artifact Projection Service

```rust
trait ArtifactProjectionService {
    fn project_module(&self, input: ArtifactProjectionInput) -> Result<VerifiedArtifactDraft, ProjectionError>;
}

struct ArtifactProjectionInput {
    module_path: ModulePath,
    snapshot: BuildSnapshotId,
    outputs: ModulePhaseOutputs,
    diagnostics: Vec<DiagnosticRecord>,
}
```

## Error Handling

### Storage Errors

| Condition | Handling |
|---|---|
| missing storage slot | treat as internal error or cache miss depending on origin |
| schema mismatch | reject cache reuse; rerun task if possible |
| corrupted internal blob | discard cache record and report cache integrity diagnostic when configured |
| requested type mismatch | internal API error; do not reinterpret bytes as another IR kind |

### Snapshot Errors

| Condition | Handling |
|---|---|
| handle from obsolete snapshot used as current | reject and request current-snapshot task or cache validation |
| dependency artifact hash mismatch | invalidate dependent handles |
| source hash mismatch | invalidate source-derived handles |
| missing source map | allow computation only if no diagnostics/artifact projection need it; otherwise fail task |

### Projection Errors

Projection failure blocks artifact commit for the affected module, but it does not mutate sealed IR outputs. The previous artifact manifest remains authoritative until phase 15 commits a valid replacement.

## Constraints and Assumptions

- Internal IR values are immutable after their producing phase publishes them.
- Internal ids are scoped to a `BuildSnapshotId` unless explicitly projected into artifact identities.
- Phase tasks exchange handles, not mutable IR objects.
- Phase output content hashes are based on content and named inputs, not snapshot-local ids.
- Cache records are optimization data and not proof authority.
- Published artifacts are stable projections and must not contain raw compiler-internal IR storage refs.
- Old snapshots may remain readable while referenced by LSP or diagnostics, but they cannot be reported as current.
- Storage collection must not change published artifacts or source-level semantics.
