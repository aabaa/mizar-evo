# Internal Design: Artifact Store, Cache Key, and Manifest

> Canonical language: English. Japanese companion: [../ja/02.artifact_store_cache_key_and_manifest.md](../ja/02.artifact_store_cache_key_and_manifest.md).

## Purpose

This document defines the internal design for phase 15 artifact storage, cache key construction, cache record lookup/update, and package artifact manifest transactions.

It refines the artifact architecture into implementation-facing services while preserving the architecture rule that published artifacts are stable external projections and cache records are internal optimizations, not proof authority.

## Responsibilities

### Artifact Store

`ArtifactStore` owns stable build output under each package `artifact_dir`. In a multi-package workspace, every package keeps its own artifact directory; the workspace root is used for path normalization, dependency discovery, and shared cache configuration.

It writes:

- published `*.mizir.json` artifacts;
- package artifact manifests;
- proof witness files referenced by published artifacts;
- development artifacts such as `missing_facts.json`, ATP logs, diagnostic explanation previews, and optional debug dumps.

It must not expose compiler-internal cache records as published artifacts.

### Cache Store

`CacheStore` owns internal build cache records under `cache_dir`.

It stores:

- phase output cache records keyed by complete `CacheKey`s;
- content-addressed blobs for large phase outputs;
- `cluster-db/` indexes, registration contributions, and import-scoped views;
- resolution traces and diagnostic explanation backing data when they are internal cache data.

Cache records may be deleted at any time. Deleting them must not change source-level semantics or published package compatibility.

### Manifest Manager

`ManifestManager` owns atomic updates to the per-package artifact manifest.

It is the only writer allowed to publish a new manifest version. It validates that every manifest entry points to an already-written complete artifact and that recorded hashes match the files on disk before the manifest becomes visible to readers.

### Cache Key Builder

`CacheKeyBuilder` constructs cache keys from normalized identities, hashes, schema versions, and verifier policy.

It does not decide whether a cache entry is trusted. It only produces keys and dependency summaries. `CacheStore` compatibility checks and proof witness validation decide reuse.

## Data Model

### Store Layout

The default physical layout follows the package manifest settings from the spec.

The standard package artifact manifest file name is `artifact-manifest.json`.

```text
build/
  artifact-manifest.json
  <module>.mizir.json
  proof-witnesses/
    <module>/
      vc-<id>.json
  logs/
    <module>.atp-log
  missing_facts.json
  explanations/
    <module>/
      <diagnostic-id>.json

.mizar-cache/
  records/
    <phase>/
      <cache-key>.bin
  blobs/
    sha256/
      <hash>
  cluster-db/
  resolution-trace/
  diagnostic-explanation/
  tmp/
```

The layout is not itself a compatibility promise, except for published artifact paths recorded in the manifest and paths specified by the user-facing spec. Readers use the manifest first.

### Artifact Store Root

```rust
struct ArtifactStoreRoot {
    workspace_root: PathBuf,
    package_root: PathBuf,
    artifact_dir: PathBuf,
    cache_dir: PathBuf,
    path_policy: PathPolicy,
}

enum PathPolicy {
    PackageRelative,
    WorkspaceRelative,
    LocalAbsoluteForDiagnostics,
}
```

Portable artifacts use package-relative or workspace-relative paths. Absolute paths may appear only in explicitly local diagnostics or debug artifacts.

### Artifact Write Input

```rust
struct ArtifactCommitInput {
    module_path: ModulePath,
    source_file: SourcePath,
    source_hash: Hash,
    verified_artifact: VerifiedArtifactDraft,
    proof_witnesses: Vec<ProofWitnessDraft>,
    diagnostics: Vec<Diagnostic>,
    development_outputs: Vec<DevelopmentArtifactDraft>,
    provenance: BuildProvenance,
}
```

`VerifiedArtifactDraft` is an in-memory projection assembled from phase outputs. It becomes a published `VerifiedArtifact` only after canonical serialization, content hashing, and manifest commit.

### Manifest Transaction

```rust
struct ManifestTransaction {
    package_id: PackageId,
    base_manifest_hash: Option<Hash>,
    session: BuildSessionId,
    snapshot: BuildSnapshotId,
    module_updates: Vec<ModuleArtifactUpdate>,
}

struct ModuleArtifactUpdate {
    module_path: ModulePath,
    artifact_file: PathBuf,
    artifact_hash: Hash,
    interface_hash: Hash,
    implementation_hash: Hash,
    proof_witnesses: Vec<ProofWitnessEntry>,
    diagnostics_hash: Option<Hash>,
}
```

The transaction is committed by writing all referenced files first, validating their hashes, writing a temporary manifest, and atomically renaming it over the previous manifest.

### Cache Key

```rust
struct CacheKey {
    cache_schema_version: SchemaVersion,
    phase: PipelinePhase,
    work_unit: WorkUnit,
    source_identity: Option<SourceIdentity>,
    input_hashes: Vec<NamedHash>,
    dependency_hashes: Vec<DependencyHash>,
    dependency_slices: Vec<DependencySliceHash>,
    config_hash: Hash,
    schema_versions: Vec<SchemaVersion>,
    policy_fingerprint: PolicyFingerprint,
}

struct SourceIdentity {
    package_id: PackageId,
    module_path: ModulePath,
    normalized_source_path: PathBuf,
    source_hash: Hash,
    language_edition: Edition,
}
```

All vectors are sorted by canonical key before hashing. `dependency_slices` stores explicit hashes for the theorem, definition, cluster, notation, mode, and attribute slices used by the cached output. The final key is a domain-separated hash of the canonical encoding. `cache_schema_version` identifies the cache record encoding and compatibility rules; `schema_versions` identifies phase output schemas and published artifact schemas that affect the meaning of the cached value.

### Cache Record

```rust
struct CacheRecordHeader {
    cache_schema_version: SchemaVersion,
    key: CacheKey,
    produced_by: ToolchainInfo,
    output_hash: Hash,
    dependencies: Vec<CacheDependency>,
    proof_reuse: Option<ProofReuseEvidence>,
}

struct CacheRecord<T> {
    header: CacheRecordHeader,
    output: T,
}
```

Proof-related cache records must reference accepted proof witnesses or deterministic built-in discharge records. They must validate `ObligationAnchor`, canonical VC fingerprint, dependency slices, verifier policy, and witness/discharge hashes before reuse. A cache record cannot upgrade externally attested evidence to kernel-verified status.

## Control Flow

### Cache Lookup Before Task Execution

1. The scheduler asks the phase service for a `CacheKey` when all required input hashes are available.
2. `CacheStore` looks up the record by exact key.
3. The record header is checked for cache schema compatibility, toolchain compatibility, dependency artifact availability, dependency hash equality, and verifier policy strictness.
4. Proof-related records also validate proof witness or deterministic discharge hashes.
5. A validated hit returns immutable output handles to the scheduler.
6. Any failed check is treated as a miss unless the build mode explicitly requires cache integrity diagnostics.

Cache lookup failure does not change proof status. It only changes whether the phase must execute. `CacheHit` is a scheduling state after validation, not proof evidence.

### Artifact Commit

1. `ArtifactCommit` receives verified phase outputs and diagnostics for one module.
2. `ArtifactStore` canonicalizes the `VerifiedArtifactDraft`.
3. Proof witnesses and development artifacts are serialized to temporary files.
4. The published `*.mizir.json` is serialized with canonical ordering and hashed.
5. Temporary files are flushed before becoming visible.
6. `ArtifactStore` atomically renames completed temporary files into their final paths.
7. Containing directories are flushed where the platform supports it.
8. `ManifestManager` validates file hashes and updates the package manifest transaction.
9. The temporary `artifact-manifest.json` is flushed and atomically renamed over the previous manifest.
10. `CacheStore` promotes cache records and cache indexes that depend on published artifacts only after the manifest commit succeeds and producing phase outputs are complete and validated.
11. Build events publish `ArtifactCommitted` only after the manifest references the new artifact.

If any step before manifest commit fails, the previous manifest remains valid. If manifest commit fails, the new files may remain as unreferenced files, but readers must ignore them because they are not reachable from the manifest. Cache updates that depend on the failed manifest remain staged or are discarded.

Two workers racing to produce the same cache entry must either publish byte-identical content or one writer must lose the atomic commit race without observable semantic difference. Divergent cache contents for the same validated key are a cache integrity error and must not become visible as independent proof outcomes.

### Cluster and Registration Cache Update

The cluster cache is updated from verified contributions only.

- changed module contributions are written with origin metadata;
- stale origins are removed when a registration disappears or is renamed;
- derived indexes are rebuilt only for affected origins when possible;
- import-scoped views are invalidated by visible origin set changes;
- resolution traces are keyed by source snapshot, registration view hash, and phase schema version.

An unaccepted registration coherence proof never enters importer-visible indexes or `interface_hash`.

### Manifest Read Path

Readers such as downstream builds, LSP, docs, and AI tooling read the manifest first.

1. Read and parse `artifact-manifest.json`.
2. Check manifest schema compatibility.
3. Resolve module entries by package-relative paths.
4. Verify artifact hashes when required by the consumer.
5. Load `*.mizir.json` and proof witness references by manifest entry.

Readers must not discover published artifacts by scanning arbitrary files in `build/`.

## API Sketch

### Artifact Store

```rust
trait ArtifactStore {
    fn commit_module(&self, input: ArtifactCommitInput) -> Result<ModuleArtifactUpdate, ArtifactError>;
    fn read_manifest(&self, package: PackageId) -> Result<ArtifactManifest, ArtifactError>;
    fn read_module_artifact(&self, entry: &ModuleArtifactEntry) -> Result<VerifiedArtifact, ArtifactError>;
}
```

### Cache Store

```rust
trait CacheStore {
    fn lookup<T>(&self, key: &CacheKey, policy: CacheReusePolicy) -> Result<Option<CacheRecord<T>>, CacheError>;
    fn insert<T>(&self, record: CacheRecord<T>) -> Result<(), CacheError>;
    fn invalidate_origins(&self, origins: &[ArtifactOrigin]) -> Result<(), CacheError>;
}
```

### Cache Key Builder

```rust
trait CacheKeyBuilder {
    fn key_for_phase(&self, request: CacheKeyRequest) -> CacheKey;
    fn config_hash(&self, config: &VerifierConfig, build: &BuildConfig) -> Hash;
    fn policy_fingerprint(&self, policy: &VerifierPolicy) -> PolicyFingerprint;
}
```

### Manifest Manager

```rust
trait ManifestManager {
    fn begin(&self, package: PackageId, snapshot: BuildSnapshotId) -> Result<ManifestTransaction, ArtifactError>;
    fn stage_module(&self, tx: &mut ManifestTransaction, update: ModuleArtifactUpdate);
    fn commit(&self, tx: ManifestTransaction) -> Result<ArtifactManifest, ArtifactError>;
}
```

The manifest manager is serialized per package. Parallel module artifact writes produce independent `ModuleArtifactUpdate` values, and the manager folds those updates into one package-level publication transaction.

## Error Handling

### Artifact Errors

Artifact errors produce phase 15 diagnostics and prevent publication of the affected manifest update.

| Error | Handling |
|---|---|
| serialization failure | report artifact diagnostic; do not write final file |
| hash mismatch after write | report artifact integrity diagnostic; do not update manifest |
| rename or flush failure | report artifact I/O diagnostic; previous manifest remains authoritative |
| unsupported manifest schema | report compatibility diagnostic; do not reuse or publish through that manifest |
| missing proof witness file | reject publication for the affected module |

### Cache Errors

Cache errors are misses by default.

Corruption, unsupported cache schema, missing blobs, or dependency hash mismatches cause the lookup to return `None` and may emit a warning in verbose modes. They become hard diagnostics only for explicit cache audit or reproducibility modes.

### Concurrency

- one manifest transaction may commit per package at a time;
- temporary files include session and task identifiers but those identifiers are excluded from canonical artifact content;
- obsolete sessions cannot overwrite a manifest from a newer snapshot;
- readers either see the old complete manifest or the new complete manifest.

## Constraints and Assumptions

- Published artifacts are UTF-8 JSON unless a later published schema explicitly defines another format.
- Published artifacts must be readable without loading internal cache records.
- Cache records may use binary encodings but must include explicit cache schema versions.
- Cache keys are complete, canonical, and domain-separated by phase and work unit.
- Cache entries are never trusted proof authority.
- Proof-related cache reuse must validate proof witness or deterministic discharge hashes.
- Manifest updates are atomic package-level publication points.
- Unreferenced files in `build/` are not published artifacts.
- `verified_at` and temporary file names are excluded from canonical content hashes.
- The implementation may choose coarse invalidation initially, but it must produce the same externally visible artifacts as a clean build.
