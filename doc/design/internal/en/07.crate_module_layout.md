# Internal Design: Crate and Module Layout

> Canonical language: English. Japanese companion: [../ja/07.crate_module_layout.md](../ja/07.crate_module_layout.md).

## Purpose

This document summarizes the expected crate and module layout for implementing the architecture and internal design documents.

It is not a Cargo workspace manifest. It is an ownership map that keeps module-level design work aligned with the evo2 architecture guardrails: small kernel verification, immutable IR snapshots, deterministic builds, local verification artifacts, dependency fingerprints, stable failure semantics, and fail-heavy testing.

### Incremental Verification Ownership Notes

- `mizar-session` owns `BuildSnapshotId`, `SourceId`, source hashes, and source range coordinate systems. It does not own `ObligationAnchor`; `BuildSnapshotId` is a freshness/content-state token, not proof authority.
- `mizar-lexer` owns tokenization and active lexical environment inputs. It provides lexical fingerprints/provenance needed for frontend cache keys and does not own semantic resolution or proof dependencies.
- `mizar-syntax` owns source-shaped `SurfaceAst` and syntax node accessors. It preserves labels, proof-step structure, algorithm statement structure, registration/redefinition nodes, source ranges, and recovery markers. It does not own semantic anchors, dependency slices, proof obligations, or proof reuse identity.
- `mizar-parser` owns syntactic parse shape and Pratt/formula parsing. It must consume source-position-aware operator views and does not own VC generation, `ObligationAnchor` construction, dependency slices, or proof obligations.
- `mizar-frontend` owns frontend cache footprints / `LexicalParseSlice`-like data for source, preprocessing, lexing, and parsing. It does not own semantic `DependencySlice`.
- `mizar-resolve` owns normalized semantic origin ids, module graph, imports/exports, labels, namespace resolution, and symbol identity.
- `mizar-core` owns source-to-core provenance and obligation seeds.
- `mizar-vc` owns `VcId` assignment within a snapshot, `ObligationAnchor` construction, canonical VC fingerprints, and VC-local slice assembly.
- `mizar-cache` owns cache lookup, proof reuse validation, dependency-slice validation, and policy compatibility checks.
- `mizar-build` owns task graph construction, invalidation scheduling, reverse dependencies, and build/session orchestration.
- `mizar-artifact` owns stable published projections, proof witness references, `interface_hash`, `implementation_hash`, and manifest publication.

Lexer module specs must preserve the split between raw scanning and context-sensitive disambiguation. They must not require the lexer to load full imported module IR or decide undefined-name diagnostics.

Syntax module specs must keep `SurfaceAst` semantic-free and source-faithful. Parser module specs must consume parser-facing token transfer objects and produce `SurfaceAst` without resolving names, inferring types, choosing overload resolution results, or generating proof obligations. Frontend module specs own orchestration across source loading, preprocessing, active lexicon construction, lexing, and parsing; they must not become the home of AST node definitions or grammar logic.

## Constraints and Assumptions

- Crate boundaries may change, but ownership boundaries must remain explicit.
- Module specs must link back to the architecture documents they refine.
- Any module that affects proof acceptance must state whether it is trusted kernel code, untrusted evidence production, or policy projection.
- Any module that computes hashes must state its canonical ordering and excluded nondeterministic inputs.
- Any module that reports failures must emit stable categories and structured rejection reasons where applicable.
