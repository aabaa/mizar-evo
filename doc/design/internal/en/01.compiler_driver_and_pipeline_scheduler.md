# Internal Design: Compiler Driver and Pipeline Scheduler

> Canonical language: English. Japanese companion: [../ja/01.compiler_driver_and_pipeline_scheduler.md](../ja/01.compiler_driver_and_pipeline_scheduler.md).

## Purpose

This document defines the internal design of the compiler driver and pipeline scheduler.

It refines the architecture pipeline into implementation services that accept build requests, create versioned task graphs, invoke phase services, reuse cache records, aggregate diagnostics, run ATP/kernel work, and commit artifacts without changing the semantic results defined by the architecture layer.

## Responsibilities

### Compiler Driver

The compiler driver is the front door for all build modes.

It owns:

- parsing CLI/watch/LSP build requests into `BuildRequest`;
- invoking phase 0 package/workspace resolution as the build bootstrap;
- creating `BuildSession` values with source and dependency snapshots;
- registering phase service implementations;
- submitting the initial task graph to the scheduler;
- publishing build events to CLI progress reporters and LSP bridges.

It does not own:

- phase semantics such as type checking, overload resolution, VC generation, or proof acceptance;
- cache compatibility decisions after a concrete cache key is built;
- artifact serialization details;
- editor protocol range conversion.

### Pipeline Scheduler

The scheduler owns execution order under the dependency constraints.

It may:

- run independent tasks in parallel;
- prioritize open files and dependency summaries;
- perform cache lookup before scheduling expensive work;
- cancel obsolete tasks by build snapshot;
- route tasks to CPU, ATP, kernel, and I/O queues;
- emit progress events.

It must not:

- change phase semantics;
- accept proof evidence without the configured policy and kernel boundary;
- publish diagnostics from obsolete snapshots as current;
- commit artifacts in completion order;
- expose partial artifact writes.

### Phase Services

Each phase service is an implementation of one architecture phase or a small phase group.

| Service | Phases | Main Output |
|---|---:|---|
| `WorkspacePlanner` | 0 | `BuildPlan`, dependency graph, verifier/build config |
| `SourceFrontend` | 1-3 | `SourceUnit`, `PreprocessedSource`, `TokenStream`, `SurfaceAst` |
| `ModuleResolver` | 4-5 | `ResolvedAst`, `SymbolEnv`, `ModuleSummary` |
| `SemanticChecker` | 6-8 | `TypedAst`, `ResolutionTrace`, `ResolvedTypedAst` |
| `Elaborator` | 9-10 | `CoreIr`, `ControlFlowIr` |
| `VcService` | 11-12 | `VcIr`, deterministic discharge status |
| `AtpService` | 13 | `AtpProblem`, backend candidate evidence |
| `KernelService` | 14 | kernel-accepted proof status |
| `ArtifactService` | 15 | `VerifiedArtifact`, proof witness refs, manifest entries |
| `DocExtractionService` | 16 | rendered docs and extracted code outputs |

Services return structured phase outputs. They do not mutate scheduler state directly.

## Data Model

### Build Session

```rust
struct BuildSession {
    id: BuildSessionId,
    request: BuildRequest,
    snapshot: BuildSnapshot,
    plan: BuildPlan,
    phase_registry: PhaseRegistry,
    event_sink: BuildEventSink,
}

struct BuildSnapshot {
    id: BuildSnapshotId,
    workspace_root: PathBuf,
    source_versions: Vec<SourceVersion>,
    dependency_artifacts: Vec<DependencyArtifactRef>,
    lockfile_hash: Hash,
    toolchain: ToolchainInfo,
    verifier_config_hash: Hash,
}
```

`BuildSessionId` identifies one scheduler run. `BuildSnapshotId` identifies the exact source, dependency, lockfile, toolchain, and verifier settings used by that run. A task result belongs to exactly one `BuildSnapshotId` and may be reused across sessions only through cache validation, not by session id.

### Task Graph

```rust
struct PipelineTask {
    id: TaskId,
    session: BuildSessionId,
    snapshot: BuildSnapshotId,
    kind: PipelineTaskKind,
    unit: WorkUnit,
    dependencies: Vec<TaskId>,
    cache_key: Option<CacheKey>,
    priority: TaskPriority,
    resource_class: ResourceClass,
}

enum PipelineTaskKind {
    PackageResolve,
    SourceLoad,
    Frontend,
    ModuleResolve,
    CheckAndElaborate,
    VcGenerate,
    VcDischarge,
    AtpSolve,
    BackendRun,
    KernelCheck,
    ArtifactCommit,
    DocumentationExtract,
}
```

The scheduler may initially collapse adjacent phases into coarser tasks, such as `CheckAndElaborate`, but each task must still record which architecture phases produced its outputs.

`PackageResolve` represents the phase 0 bootstrap output. In the default implementation, the driver runs `WorkspacePlanner` before submitting the graph and records `PackageResolve` as the completed root input for the scheduled tasks. A later implementation may schedule phase 0 itself, but it must still produce exactly one `BuildPlan` for the session before downstream tasks are created.

### Phase Output Handle

```rust
struct PhaseOutputRef<T> {
    phase: PipelinePhase,
    unit: WorkUnit,
    snapshot: BuildSnapshotId,
    content_hash: Hash,
    storage: OutputStorageRef<T>,
}
```

Tasks exchange output handles rather than mutable IR values. Output handles are immutable once published to dependent tasks.

Shared mutable semantic state between workers is forbidden. Shared indexes used by workers must be immutable snapshot views or transactionally published validated artifacts.

### Task Result

```rust
struct PipelineTaskResult {
    task_id: TaskId,
    state: TaskState,
    phase_outputs: Vec<AnyPhaseOutputRef>,
    diagnostics: Vec<Diagnostic>,
    cache_record: Option<CacheRecordRef>,
    timing: TimingSummary,
}
```

`TaskState::CacheHit` means the scheduler skipped execution after cache validation. It is still recorded as a completed dependency and sorted with ordinary task results.

`CacheHit` is a scheduling state after validation, not proof evidence. Obsolete task results may be reused only through normal cache validation against the target `BuildSnapshotId`.

## Control Flow

### Batch Build

1. The driver creates a `BuildRequest` from CLI options and package defaults.
2. `WorkspacePlanner` resolves manifests, dependencies, lockfile, edition, and verifier settings.
3. The scheduler creates source-file tasks for phases 1-10, VC-level tasks for phases 11-14, artifact commit tasks for phase 15, and documentation/extraction tasks for phase 16.
4. Cache keys are built as soon as all required input hashes are known.
5. Cache hits produce validated output handles; cache misses enter the ready queue.
6. Diagnostics are aggregated per source file and build snapshot.
7. Artifact commit tasks run after proof status and diagnostics for a module are complete.
8. Documentation/extraction tasks run after verified artifacts are committed when enabled by the build profile.
9. The driver reports the final build status after ready work is completed, failed, blocked, or cancelled.

Phase 16 is part of the batch graph, but it remains a separate task class. Profiles that only need verification may disable documentation rendering, code extraction, or both without changing phases 0-15.

### Watch and LSP Build

Watch and LSP builds use the same driver and scheduler, but create new snapshots for edits.

- frontend tasks for changed open files are prioritized;
- semantic tasks wait for dependency summaries that match the new snapshot;
- obsolete tasks are cancelled cooperatively;
- cached outputs may be reused only when their cache keys match the new snapshot;
- diagnostics are published with freshness information;
- LSP snapshots are switched atomically after aggregation.

Open editor buffers are source snapshots, not published artifacts. The LSP bridge converts source positions and applies code actions at the editor boundary.

### ATP and Kernel Subgraph

For each open `VcId`, the scheduler creates an `AtpSolve` task. The ATP service expands it into backend `BackendRun` tasks according to the configured portfolio policy.

Backend results are candidate evidence. When policy requires kernel validation, eligible candidates create `KernelCheck` tasks. The accepted proof status is selected by policy and kernel result, not by backend completion order.

### Artifact Commit Boundary

`ArtifactCommit` is the serialization boundary.

It receives verified phase outputs, diagnostics, proof witness references, and provenance. It writes artifacts through the artifact service using an atomic manifest transaction. Commit order follows canonical module order, not worker completion order.

## API Sketch

### Driver API

```rust
trait CompilerDriver {
    fn submit(&self, request: BuildRequest) -> BuildSessionId;
    fn cancel(&self, session: BuildSessionId, reason: CancelReason);
    fn events(&self, session: BuildSessionId) -> BuildEventStream;
}
```

### Phase Service API

```rust
trait PhaseService<I, O> {
    fn phase(&self) -> PipelinePhase;
    fn cache_key(&self, input: &I, context: &PhaseContext) -> Option<CacheKey>;
    fn execute(&self, input: I, context: PhaseContext) -> PhaseResult<O>;
}

struct PhaseResult<O> {
    status: PhaseStatus,
    output: Option<O>,
    diagnostics: Vec<DiagnosticDraft>,
    output_refs: Vec<AnyPhaseOutputRef>,
}

struct PhaseContext {
    session: BuildSessionId,
    snapshot: BuildSnapshotId,
    cancellation: CancellationToken,
    diagnostics: DiagnosticSink,
    output_store: OutputStoreHandle,
}
```

`cache_key` must not read mutable task state. It is a pure projection from input identities, configuration, schema versions, and dependency hashes.

### Scheduler API

```rust
trait PipelineScheduler {
    fn schedule(&self, session: BuildSession, graph: PipelineGraph);
    fn cancel_snapshot(&self, snapshot: BuildSnapshotId, reason: CancelReason);
    fn subscribe(&self, session: BuildSessionId) -> BuildEventStream;
}
```

### Build Events

```rust
enum BuildEvent {
    SessionStarted(BuildSessionId),
    TaskStarted(TaskId),
    TaskFinished(TaskId, TaskState),
    DiagnosticsReady(SourceId, BuildSnapshotId),
    ArtifactCommitted(ModulePath, Hash),
    SnapshotPublished(BuildSnapshotId),
    SessionFinished(BuildSessionId, BuildStatus),
}
```

Events are progress and integration signals. They are not artifact ordering rules.

## Error Handling

### Phase Failure

Phase services return diagnostics and a phase status.

```rust
enum PhaseStatus {
    Complete,
    Recoverable,
    Blocking,
    Fatal,
}
```

- `Recoverable` permits only downstream work whose semantics do not depend on the degraded metadata, such as syntax-only LSP features, development explanations, or documentation previews. Semantic acceptance and proof status must not depend on recoverable fallback data.
- `Blocking` stops dependent work for the affected unit but does not stop independent tasks.
- `Fatal` stops the session when the global build state cannot be trusted.

### Cancellation

Cancellation is versioned by snapshot.

- in-process tasks check tokens at phase boundaries and safe checkpoints;
- ATP child processes are terminated by the backend manager;
- obsolete completed results are discarded unless cache validation proves they apply to the new snapshot;
- artifact commit is not cancelled after the atomic transaction begins.

Task outputs are immutable after publication. Diagnostics and artifacts are collated by canonical order, not worker completion order.

### Diagnostics

Diagnostics flow through a per-session aggregator. The aggregator deduplicates records, assigns deterministic `DiagnosticId`s, sorts by the architecture-defined order, and publishes only diagnostics whose snapshot is current for the target consumer.

## Constraints and Assumptions

- The scheduler controls execution order, not phase semantics.
- All task outputs are immutable after publication.
- Shared mutable semantic state between workers is forbidden.
- Shared worker indexes must be immutable snapshot views or transactionally published validated artifacts.
- Cache hits must satisfy the validation rules of the artifact and incremental build architecture.
- Diagnostics and artifacts are sorted by canonical order, never by task completion order.
- Kernel certificate checking remains the trusted proof acceptance boundary.
- LSP builds may prioritize open files but must not mix diagnostics from incompatible snapshots.
- Documentation and extraction are scheduled after verified artifacts are available.
