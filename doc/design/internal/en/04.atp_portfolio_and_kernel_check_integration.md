# Internal Design: ATP Portfolio and Kernel Check Integration

> Canonical language: English. Japanese companion: [../ja/04.atp_portfolio_and_kernel_check_integration.md](../ja/04.atp_portfolio_and_kernel_check_integration.md).

## Purpose

This document defines how phase 13 ATP portfolio execution connects to phase 14 kernel evidence checking.

It refines the reasoning boundary into internal task APIs, evidence records, portfolio winner selection, proof witness storage, diagnostics, cache reuse, and artifact-facing proof status. ATP backends produce candidate evidence; only kernel checking or explicitly permitted policy outcomes produce accepted proof status.

## Evidence-Format Correction

Phase 14's target input is formula/substitution kernel evidence, not a
backend-method proof trace. A backend may search by resolution, an instance
method, an inverse method, SMT tactics, or another method, but the trusted
handoff to the kernel records only the formulas, substitutions, provenance,
and target binding needed for an independent kernel check. Instantiated
formulas and the SAT problem are kernel-derived artifacts, not trusted
handoff payload.

The kernel check service is therefore allowed to call the trusted in-process
Rust SAT checker over the SAT problem it derives from supplied
formula/substitution evidence. That call is part of acceptance checking, not
ATP search. The service must not select premises, invent substitutions,
minimize premises, run portfolio search, resolve overloads, search clusters,
insert coercions, or consult global mutable compiler state.

Historical names such as `AtpCertificate`, `ResolutionTrace`, and
`InvalidSatProof` describe the legacy task-22 integration model and must be
classified as `design_drift` / `source_drift` until the kernel evidence schema
redesign lands. New `mizar-atp` work is deferred until it can produce
formula/substitution evidence candidates rather than MiniSAT-compatible
resolution traces.

## Responsibilities

### ATP Portfolio Service

`AtpPortfolioService` owns phase 13 portfolio construction and candidate evidence collection for one open VC.

It is responsible for:

- selecting backend profiles from verifier policy, source hints, VC shape, backend availability, and resource limits;
- requesting concrete encodings from ATP encoders;
- spawning `BackendRun` tasks through the scheduler;
- collecting backend results, stdout/stderr hashes, candidate evidence, counterexamples, and timing;
- applying early-stop rules that are allowed by policy;
- producing a deterministic `PortfolioResult` independent of raw completion order.

It does not accept proof status. It only produces evidence.

### Backend Runner

`BackendRunner` owns one backend child process invocation.

It is responsible for:

- setting command line, working directory, stdin or temporary problem file, timeout, memory limit, and random seed;
- capturing stdout, stderr, exit status, wall time, and backend version;
- classifying backend output as `Proved`, `Counterexample`, `Timeout`, `Unknown`, or `Error`;
- parsing or extracting proof payloads into backend-specific candidate evidence when present;
- terminating the process on cancellation or portfolio stop.

`Proved` means the backend observed the expected result and returned candidate
evidence. It is not kernel acceptance, and later redesign work should prefer a
name such as `EvidenceFound` where that avoids implying trust.

### Kernel Check Service

`KernelCheckService` owns phase 14 proof evidence acceptance.

It is responsible for:

- constructing immutable kernel contexts from `CoreIr`, dependency artifact hashes, premise refs, registration traces, and `AtpProvenance`;
- checking formula/substitution evidence and replayable kernel primitives through a shared kernel evidence API;
- extracting trusted `used_axioms` from accepted formula evidence, accepted built-in evidence, and allowed kernel primitives;
- producing `KernelCheckResult` values for artifact emission and diagnostics.

The kernel service never reruns ATP search, never trusts backend-reported used axioms without validation, and does not make build-policy decisions. It returns policy-independent validation results.

### Proof Policy Evaluator

`ProofPolicyEvaluator` owns policy decisions that are not kernel proof checking.

It decides:

- whether externally attested evidence may be recorded;
- whether `require_kernel_certificates` rejects externally attested evidence as an error;
- whether an open obligation is allowed in the current build mode;
- which accepted candidate wins when phase 12 built-in discharge results and phase 13/14 ATP results are projected into artifact-facing proof status.

Policy outcomes are distinct from trusted proof status.

### Proof Witness Store

`ProofWitnessStore` owns witness files and references passed to phase 15 artifact commit.

It stores kernel-checkable formula/substitution evidence material, accepted
built-in evidence or allowed kernel primitives, and provenance metadata. It
records content hashes before artifact commit, but the witness becomes
publication-reachable only after the artifact manifest references it.

### Portfolio Task


The profile priority is part of deterministic winner selection. It does not override release policy.

### Backend Run Result

```rust
struct BackendRunResult {
    profile: BackendProfileId,
    encoded_problem_hash: Hash,
    backend_version: BackendVersion,
    command_fingerprint: Hash,
    status: BackendRunStatus,

    counterexample: Option<CounterexampleEvidence>,

    Counterexample,

    Unknown,

    Cancelled,

```

`BackendRunStatus::Proved` must only be produced when the observed backend result matches `EncodedProblem.expected_result` and candidate evidence is present.

### Candidate Evidence

```rust
struct CandidateEvidence {
    evidence_id: KernelEvidenceCandidateId,
    format: EvidenceFormat,
    payload_ref: EvidenceBlobRef,
    atp_provenance_hash: Hash,
    symbol_map_hash: Hash,
    backend_formula_labels: Vec<AtpFormulaId>,
}
```

Candidate payloads may be stored as temporary evidence blobs before kernel checking. They are not proof witnesses until accepted or recorded as policy evidence.

### Kernel Check Input and Result

```rust
struct KernelCheckInput {
    vc_id: VcId,
    evidence: KernelEvidenceInput,
    kernel_context: KernelContextRef,
    expected_problem_hash: Hash,
}

enum KernelEvidenceInput {
    FormulaSubstitutionEvidence(CandidateEvidence),
    BuiltinEvidence(BuiltinEvidence),
    KernelPrimitive(KernelPrimitiveEvidence),
    LegacyCertificate(LegacyEvidenceRef),
    LegacyResolutionTrace(LegacyEvidenceRef),
}

struct BuiltinEvidence {
    witness_id: BuiltinWitnessId,
    evidence_ref: EvidenceBlobRef,
    evidence_hash: Hash,
    kernel_profile: KernelProfile,
}

struct LegacyEvidenceRef {
    evidence_ref: EvidenceBlobRef,
    evidence_hash: Hash,
    legacy_format: LegacyEvidenceFormat,
}

struct KernelCheckResult {
    vc_id: VcId,
    evidence: KernelEvidenceId,
    status: KernelCheckStatus,
    used_axioms: Vec<PremiseRef>,
    proof_witness: Option<ProofWitnessDraft>,
    diagnostics: Vec<DiagnosticDraft>,
}

enum KernelCheckStatus {
    Accepted,
    Rejected {
        reason: RejectionReason,
    },
}

enum RejectionReason {
    MalformedEvidence,
    MalformedWitnessData,
    InvalidSubstitution,
    InvalidSatRefutation,
    InvalidClusterTrace,
    UnresolvedSymbol,
    UnsupportedEvidenceFormat,
    ContextMismatch,
    MissingProvenance,
    Timeout,
    ResourceExhaustion,
}
```

`Accepted` is the only kernel check status that can produce a trusted artifact
proof status. For `KernelEvidenceInput::FormulaSubstitutionEvidence`, it is
projected as `proof_status = "kernel_verified"`. For
`KernelEvidenceInput::BuiltinEvidence`, it is projected as
`proof_status = "discharged_builtin"` rather than `"kernel_verified"`.
Built-in evidence must be encoded as formula/substitution evidence or an
explicit kernel primitive allowed by the small-kernel architecture; arbitrary
deterministic rule traces are policy evidence, not trusted kernel evidence.
`LegacyCertificate` and `LegacyResolutionTrace` are source-drift compatibility
inputs only until the post-closeout migration retires or gates them. The
kernel check service must reject them as `UnsupportedEvidenceFormat` under
normal proof policy; an explicit migration/audit mode may inspect them, but it
must not produce trusted `Accepted` results from them.

### ATP Portfolio Result

```rust
struct PortfolioResult {
    vc_id: VcId,
    backend_results: Vec<BackendRunResult>,
    kernel_results: Vec<KernelCheckResult>,
    selected: AtpPortfolioSelection,
    diagnostics: Vec<DiagnosticDraft>,
}

enum AtpPortfolioSelection {
    KernelVerified {
        candidate: KernelEvidenceCandidateId,
        witness: ProofWitnessDraft,
        used_axioms: Vec<PremiseRef>,
    },
    ExternallyAttested {
        backend: BackendProfileId,
        evidence: EvidenceBlobRef,
    },
    Open {
        reason: OpenReason,
    },
    RejectedByPolicy {
        reason: PolicyRejection,
    },
}
```

`PortfolioResult` covers only VCs that reached phase 13 as `NeedsAtp`. Phase 12 built-in discharge results are not inserted into the ATP portfolio; they are checked by the kernel evidence API and merged later during artifact-facing proof status selection.

### Artifact Proof Selection

The selected artifact result is sorted and emitted by canonical `VcId` order during artifact commit.

Policy selection consumes a normalized evidence view:

```rust
enum ProofCandidateEvidence {
    AtpKernel(KernelCheckResult),
    BuiltinKernel(KernelCheckResult),
    External(BackendRunResult),
    Open(OpenReason),
}

enum ProofCandidateSelection {
    KernelVerified {
        candidate: KernelEvidenceId,
        witness: ProofWitnessDraft,
        used_axioms: Vec<PremiseRef>,
    },
    DischargedBuiltin {
        witness: ProofWitnessDraft,
        used_axioms: Vec<PremiseRef>,
    },
    ExternallyAttested {
        backend: BackendProfileId,
        evidence: EvidenceBlobRef,
    },
    Open {
        reason: OpenReason,
    },
    RejectedByPolicy {
        reason: PolicyRejection,
    },
}
```

This keeps policy ranking separate from backend process classification.

## Control Flow

### Portfolio Construction

1. `VcService` emits open VCs in canonical `VcId` order.
2. `AtpPortfolioService` receives one open VC and its `AtpProblem`.
3. The service selects backend profiles from `default_solver`, `@proof_hint`, policy, availability, and VC logic profile.
4. The scheduler creates one `AtpSolve` task and child `BackendRun` tasks.
5. Each backend run receives an immutable encoded problem and backend budget.

If `@proof_hint(solver: ...)` restricts the solver, the portfolio contains only profiles allowed by that hint and policy.

### Backend Execution

1. `BackendRunner` writes or streams the concrete problem input.
2. It launches the backend child process with configured resource limits.
3. It captures stdout/stderr and terminates on timeout or cancellation.
4. It classifies the result against `EncodedProblem.expected_result`.
5. It stores candidate evidence or counterexample evidence as temporary evidence blobs.
6. It returns `BackendRunResult` to the portfolio task.

Backend crashes, parse failures, and malformed output become backend errors, not verifier crashes.

### Kernel Check Scheduling

When a backend result contains candidate evidence, the portfolio service asks the scheduler to create `KernelCheck` work if the policy can use kernel-checked evidence from that candidate.

Kernel checks may run in parallel for independent candidates. Their contexts are immutable and content-addressed by source snapshot, dependency artifacts, VC hash, symbol map hash, and ATP provenance hash.

Built-in discharge evidence produced before ATP search is not part of the
`AtpSolve` subgraph. It uses the same `KernelCheckService` through
`KernelEvidenceInput::BuiltinEvidence`, then joins the proof selection stage
as `ProofCandidateEvidence::BuiltinKernel`. If a pre-ATP discharge cannot be
represented as formula/substitution evidence or an allowed kernel primitive,
it may be recorded only as policy-controlled evidence.

### Winner Selection

ATP portfolio winner selection is deterministic and policy-driven.

Ordering rules:

1. accepted kernel-verified evidence satisfying release policy;
2. externally attested evidence only when policy permits it;
3. open result with the best diagnostic explanation.

Within the same class, candidates are ordered by backend profile priority,
evidence format priority, encoded problem hash, then backend profile id. Raw
completion time is recorded but never used as the semantic winner order.

When `require_kernel_certificates = true` or its successor kernel-evidence
policy is enabled, externally attested evidence cannot win. It may still be
recorded as development evidence if the build profile allows it, but the
obligation remains rejected by policy or open for publication.

Artifact proof selection then merges the ATP portfolio result with any phase
12 built-in discharge result for the same `VcId`. Accepted built-in discharge
evidence is emitted as `discharged_builtin`; accepted ATP formula/substitution
evidence is emitted as `kernel_verified`. Policy-controlled external and open
statuses remain distinguishable from both trusted statuses.

### Early Stop and Cancellation

The portfolio may terminate remaining backend processes only after policy proves they cannot produce a better acceptable result.

Examples:

- after a kernel-verified winner is accepted under a policy that has no higher class;
- after all remaining candidates lack evidence formats required by release policy;
- after cancellation supersedes the build snapshot.

In-process portfolio and kernel work uses cooperative cancellation. Backend child processes are terminated by the backend manager.

### Proof Witness and Artifact Flow

1. Accepted ATP formula/substitution kernel results produce `ProofWitnessDraft` values, trusted `used_axioms`, and `kernel_verified` selections.
2. Accepted built-in discharge kernel results produce `ProofWitnessDraft` values, trusted `used_axioms`, and `discharged_builtin` selections.
3. Externally attested results produce evidence refs and diagnostics, but no trusted `used_axioms`.
4. Open or rejected obligations produce proof diagnostics and optional explanation refs.
5. `ArtifactService` writes proof witnesses and records content hashes during phase 15.
6. The package manifest publishes witness references only after the artifact commit transaction succeeds.

Proof witness hashes participate in proof-related cache keys. Cache hits may reuse proof results only when witness hashes, dependency artifact hashes, policy, and schema versions match.

## Error Handling

### Backend Errors

Backend errors become proof diagnostics attached to the relevant `VcId`.

| Condition | Handling |
|---|---|
| timeout | emit ATP timeout diagnostic and keep other candidates running if useful |
| process crash | emit backend error diagnostic; do not fail unrelated VCs |
| malformed backend output | classify as backend error or unknown |
| counterexample | record diagnostic evidence only when provenance mapping succeeds |
| cancellation | discard result unless cache validation proves it applies to the newer snapshot |

### Kernel Errors

Kernel rejection is a proof error for that candidate, not a verifier crash.

Evidence rejection, context mismatch, missing provenance, unsupported evidence
format, malformed witness data, timeout, and resource exhaustion all prevent
trusted proof status. `KernelCheckResult.status` carries the stable
`RejectionReason`, and diagnostics project it as structured detail under the
phase-level `kernel_evidence_rejection` or `kernel_rejection` failure
category. The portfolio may continue if other candidates remain.

### Policy Errors

When policy rejects externally attested evidence, the result remains distinct
from kernel rejection. Diagnostics should explain whether the obligation
failed because no prover succeeded, the kernel rejected every candidate
evidence package, or policy required kernel evidence.

### Cache Reuse

Proof cache reuse is allowed only through complete cache validation:

- same `VcId` and VC hash;
- same accepted witness hash, built-in evidence hash, or allowed kernel primitive witness hash;
- compatible kernel version and evidence checker schema;
- same dependency artifact hashes and trusted environment;
- active policy at least as strict as the cached policy.

Cached proof results are reused evidence records, not independent proof authority.

## Constraints and Assumptions

- ATP backend success is candidate evidence, not accepted proof status.
- `kernel_verified` is emitted only after successful kernel checking.
- Externally attested evidence is policy-controlled and never equivalent to kernel verification.
- Portfolio completion order is not semantic ordering.
- Remaining backend processes may be stopped only after policy-safe winner selection or cancellation.
- Kernel contexts are immutable and content-addressed by source, dependency, VC, symbol map, and provenance hashes.
- Backend-reported used axioms are advisory until validated by kernel checking.
- Proof-related cache hits must validate witness hashes and policy strictness.
- Counterexamples and missing-fact suggestions support diagnostics but do not establish proof status.
