> Canonical language: English. Japanese companion: [../ja/00.internal_overview.md](../ja/00.internal_overview.md).

## Purpose

This document defines the scope of the internal design layer for Mizar Evo.

Internal design documents refine the architecture documents into implementation-facing subsystems, APIs, data structures, and execution contracts. They are more concrete than the architecture layer, but still broader than 1:1 module specifications.

## Responsibilities

The internal design layer owns the mapping from architectural boundaries to implementation services.

| Concern | Architecture Defines | Internal Design Defines |
|---|---|---|
| Pipeline phases | semantic phase boundaries and ordering | phase service APIs, driver state, task dependencies |
| IR layers | ownership and consumers | snapshot handles, storage lifetime, identity rules |
| Scheduling | allowed parallelism and canonical ordering | task graph representation, queues, cancellation tokens, event stream |
| Artifacts and cache | publication boundary and reuse rules | store APIs, cache key builders, manifest transaction protocol |
| Diagnostics and LSP | stable records and snapshot model | diagnostic aggregation, explanation handles, LSP bridge services |
| ATP and kernel | trust boundary and backend protocol | portfolio tasks, proof witness flow, kernel check integration |
| Documentation | artifact-based extraction boundary | doc extraction passes, render inputs, generated output manifests |

### Memory Model (Cross-Cutting)

Memory scalability is a cross-cutting concern realized by several internal subsystems rather than a single one. The resident-set strategy — keep interfaces and indexes resident; load proof bodies, traces, and witnesses lazily; never duplicate global indexes per import closure — is stated normatively in spec [§12.6.3](../../../spec/en/12.modules_and_namespaces.md#1263-memory-model) and [§23.7.9](../../../spec/en/23.package_management_and_build_system.md#2379-memory-design-principles), and as a cross-cutting concern in the [architecture README](../../architecture/en/README.md#memory-model). The internal layer implements it as follows:

| Resident-set rule | Internal mechanism | Document |
|---|---|---|
| Hold imported interfaces, not bodies | Imported public interfaces (`ModuleSummary`, published `*.mizir.json`) are read through the manifest reader and keyed by `interface_hash`; proof bodies and witnesses stay external | [02.artifact_store_cache_key_and_manifest.md](./02.artifact_store_cache_key_and_manifest.md) |
| Keep only active work resident; spill and collect the rest | `IrStorageService` stores large outputs as content-addressed blobs and drops unreferenced outputs via `retain`/`collect` | [06.ir_storage_and_snapshot_handles.md](./06.ir_storage_and_snapshot_handles.md) |
| Do not duplicate global indexes per closure | `cluster-db/` indexes plus import-scoped views in the cache store | [02.artifact_store_cache_key_and_manifest.md](./02.artifact_store_cache_key_and_manifest.md) |
| Externalize witnesses and traces | proof-witness files referenced by hash; cache blobs and traces deletable without changing semantics | [02.artifact_store_cache_key_and_manifest.md](./02.artifact_store_cache_key_and_manifest.md) |
| Bound in-flight discharge | per-VC portfolio tasks with backend memory limits; candidate evidence kept as temporary blobs until accepted | [04.atp_portfolio_and_kernel_check_integration.md](./04.atp_portfolio_and_kernel_check_integration.md) |
| Serve explanations lazily | `ExplanationStore` resolves lazy handles and bounded previews on demand, not bulk-loaded | [03.diagnostics_model_and_lsp_bridge.md](./03.diagnostics_model_and_lsp_bridge.md) |

This is a qualitative resident-set discipline, not a performance guarantee; concrete budgets and benchmark metrics live in the [test strategy](../../architecture/en/20.test_strategy.md#performance-and-memory-evaluation).

## Internal Subsystem Boundaries

Expected subsystem crates are named here for design clarity. The final crate layout may differ, but the responsibility boundaries should remain stable.

| Subsystem | Primary Responsibility |
|---|---|
| `mizar-driver` | build requests, pipeline orchestration, phase service registry |
| `mizar-build` | workspace planning, task graph scheduler, resource budgets, cancellation |
| `mizar-session` | immutable source and build snapshots shared by batch, watch, and LSP modes |
| `mizar-lexer` | source preprocessing helpers, raw scanning, import pre-scan, and token disambiguation |
| `mizar-syntax` | source-shaped syntax structures, trivia, and parser recovery markers |
| `mizar-parser` | grammar implementation and `TokenStream` to `SurfaceAst` parsing |
| `mizar-frontend` | source loading and phase 1-3 orchestration across lexer and parser services |
| `mizar-cache` | cache key construction, cache record lookup, compatibility checks |
| `mizar-artifact` | verified artifact projection, proof witness references, manifest transactions |
| `mizar-diagnostics` | diagnostic registry, aggregation, sorting, rendering, explanation handles |
| `mizar-lsp` | editor protocol bridge and current-buffer freshness handling |
| `mizar-atp` | ATP problem encoding, backend process execution, portfolio candidate collection |
| `mizar-kernel` | proof certificate checking and trusted proof status |
| `mizar-proof` | proof policy evaluation, proof status projection, and witness selection |
| `mizar-doc` | documentation rendering and extraction from verified artifacts |
| `mizar-test` | pass/fail tests, snapshots, fuzz/property harnesses, and soundness regressions |

## Control Flow

Internal designs use the same phase numbering as [Pipeline Overview](../../architecture/en/00.pipeline_overview.md).

1. The driver receives a `BuildRequest` from CLI, watch, or LSP.
2. Phase 0 produces a `BuildPlan` and dependency graph.
3. The scheduler expands the plan into versioned tasks.
4. Phase services produce immutable outputs, diagnostics, and cacheable records.
5. Proof-related tasks dispatch ATP backends and kernel certificate checks through the same scheduler.
6. Artifact commit serializes verified outputs through manifest transactions.
7. LSP and documentation tools consume published snapshots and artifacts rather than compiler-internal IR.

## Constraints and Assumptions

- Internal design must preserve all architecture phase boundaries and trust boundaries.
- Internal APIs may evolve, but published artifact schemas and diagnostic codes require compatibility policy.
- Cache hits are optimization results, not proof authority.
- Parallel execution must not change diagnostic order, artifact order, proof acceptance, or reproducibility.
- LSP workflows may prioritize latency, but batch verification remains the semantic baseline.
