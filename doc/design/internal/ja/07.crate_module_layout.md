# 内部設計: crate とモジュールの構成

> 正本は英語です。英語版: [../en/07.crate_module_layout.md](../en/07.crate_module_layout.md)。

## 目的

この文書は、アーキテクチャ文書と内部設計文書を実装するために想定する、crate とモジュールの構成をまとめる。

これは Cargo のワークスペースマニフェストではない。モジュール単位の設計作業を evo2 のアーキテクチャ上のガードレール、すなわち小カーネルによる検証、不変な IR スナップショット、決定的なビルド、ローカルな検証アーティファクト、依存フィンガープリント、安定した失敗セマンティクス、失敗重視のテストに沿わせるための、責務の対応表である。

## 制約と前提

- crate の境界は変わってよいが、責務の境界は明示し続けなければならない。
- モジュール仕様は、自身が詳細化するアーキテクチャ文書へリンクし返さなければならない。
- 証明の受理に影響するモジュールは、それが信頼されたカーネルコードか、信頼されないエビデンス生成か、ポリシーの射影かを明記しなければならない。
- ハッシュを計算するモジュールは、その正準順序と、除外する非決定的な入力を明記しなければならない。
- 失敗を報告するモジュールは、安定した分類と、必要に応じて構造化された拒否理由を出力しなければならない。
