# Internal Design: Artifact Store, Cache Key, and Manifest

> Canonical language: English. English canonical version: [../en/02.artifact_store_cache_key_and_manifest.md](../en/02.artifact_store_cache_key_and_manifest.md).

## 目的

この文書は、phase 15 artifact storage、cache key construction、cache record lookup/update、package artifact manifest transactions の internal design を定義する。

これは artifact architecture を implementation-facing services へ詳細化する。ただし、published artifacts は stable external projections であり、cache records は proof authority ではなく internal optimization である、という architecture rule を保持する。

## Responsibilities

### Artifact Store

`ArtifactStore` は各 package の `artifact_dir` 以下にある stable build output を所有する。Multi-package workspace でも各 package はそれぞれの artifact directory を持つ。Workspace root は path normalization、dependency discovery、shared cache configuration のために使う。

書き込むもの:

- published `*.mizir.json` artifacts
- package artifact manifests
- published artifacts から参照される proof witness files
- `missing_facts.json`、ATP logs、diagnostic explanation previews、optional debug dumps のような development artifacts

Compiler-internal cache records を published artifacts として expose してはならない。

### Cache Store

`CacheStore` は `cache_dir` 以下の internal build cache records を所有する。

保存するもの:

- complete `CacheKey`s で key 付けされた phase output cache records
- large phase outputs 用の content-addressed blobs
- `cluster-db/` indexes、registration contributions、import-scoped views
- internal cache data としての resolution traces and diagnostic explanation backing data

Cache records はいつ削除されてもよい。削除しても source-level semantics や published package compatibility を変えてはならない。

### Manifest Manager

`ManifestManager` は per-package artifact manifest の atomic updates を所有する。

新しい manifest version を publish してよい唯一の writer である。Manifest が readers から見える前に、すべての manifest entry が already-written complete artifact を指すことと、recorded hashes が disk 上の files と一致することを検証する。

### Cache Key Builder

`CacheKeyBuilder` は normalized identities、hashes、schema versions、verifier policy から cache keys を構築する。

Cache entry が trusted かどうかは判断しない。Keys and dependency summaries だけを生成する。Reuse は `CacheStore` compatibility checks and proof witness validation が決定する。

## Data Model

### Store Layout

Default physical layout は spec の package manifest settings に従う。

Standard package artifact manifest file name は `artifact-manifest.json` である。

```text
build/
  artifact-manifest.json
  <module>.mizir.json
  proof-witnesses/
    <module>/
      vc-<id>.json
  logs/
    <module>.atp-log
  missing_facts.json
  explanations/
    <module>/
      <diagnostic-id>.json

.mizar-cache/
  records/
    <phase>/
      <cache-key>.bin
  blobs/
    sha256/
      <hash>
  cluster-db/
  resolution-trace/
  diagnostic-explanation/
  tmp/
```

Layout 自体は、manifest に記録された published artifact paths と user-facing spec が定める paths を除き compatibility promise ではない。Readers は manifest を最初に読む。

### Artifact Store Root

```rust
struct ArtifactStoreRoot {
    workspace_root: PathBuf,
    package_root: PathBuf,
    artifact_dir: PathBuf,
    cache_dir: PathBuf,
    path_policy: PathPolicy,
}

enum PathPolicy {
    PackageRelative,
    WorkspaceRelative,
    LocalAbsoluteForDiagnostics,
}
```

Portable artifacts は package-relative または workspace-relative paths を使う。Absolute paths は明示的に local diagnostics or debug artifacts の場合に限り出現してよい。

### Artifact Write Input

```rust
struct ArtifactCommitInput {
    module_path: ModulePath,
    source_file: SourcePath,
    source_hash: Hash,
    verified_artifact: VerifiedArtifactDraft,
    proof_witnesses: Vec<ProofWitnessDraft>,
    diagnostics: Vec<Diagnostic>,
    development_outputs: Vec<DevelopmentArtifactDraft>,
    provenance: BuildProvenance,
}
```

`VerifiedArtifactDraft` は phase outputs から組み立てられる in-memory projection である。Canonical serialization、content hashing、manifest commit の後にのみ published `VerifiedArtifact` になる。

### Manifest Transaction

```rust
struct ManifestTransaction {
    package_id: PackageId,
    base_manifest_hash: Option<Hash>,
    session: BuildSessionId,
    snapshot: BuildSnapshotId,
    module_updates: Vec<ModuleArtifactUpdate>,
}

struct ModuleArtifactUpdate {
    module_path: ModulePath,
    artifact_file: PathBuf,
    artifact_hash: Hash,
    interface_hash: Hash,
    implementation_hash: Hash,
    proof_witnesses: Vec<ProofWitnessEntry>,
    diagnostics_hash: Option<Hash>,
}
```

Transaction は、すべての referenced files を先に書き、hashes を検証し、temporary manifest を書き、previous manifest の上へ atomic rename することで commit される。

### Cache Key

```rust
struct CacheKey {
    cache_schema_version: SchemaVersion,
    phase: PipelinePhase,
    work_unit: WorkUnit,
    source_identity: Option<SourceIdentity>,
    input_hashes: Vec<NamedHash>,
    dependency_hashes: Vec<DependencyHash>,
    config_hash: Hash,
    schema_versions: Vec<SchemaVersion>,
    policy_fingerprint: PolicyFingerprint,
}

struct SourceIdentity {
    package_id: PackageId,
    module_path: ModulePath,
    normalized_source_path: PathBuf,
    source_hash: Hash,
    language_edition: Edition,
}
```

すべての vectors は hashing 前に canonical key で sort される。Final key は canonical encoding の domain-separated hash である。`cache_schema_version` は cache record encoding and compatibility rules を識別する。`schema_versions` は cached value の意味に影響する phase output schemas and published artifact schemas を識別する。

### Cache Record

```rust
struct CacheRecordHeader {
    cache_schema_version: SchemaVersion,
    key: CacheKey,
    produced_by: ToolchainInfo,
    output_hash: Hash,
    dependencies: Vec<CacheDependency>,
    proof_reuse: Option<ProofReuseEvidence>,
}

struct CacheRecord<T> {
    header: CacheRecordHeader,
    output: T,
}
```

Proof-related cache records は accepted proof witnesses または deterministic built-in discharge records を参照しなければならない。Cache record は externally attested evidence を kernel-verified status へ upgrade できない。

## Control Flow

### Cache Lookup Before Task Execution

1. Scheduler は required input hashes が揃った時点で phase service に `CacheKey` を要求する。
2. `CacheStore` は exact key で record を lookup する。
3. Record header について cache schema compatibility、toolchain compatibility、dependency artifact availability、dependency hash equality、verifier policy strictness を確認する。
4. Proof-related records は proof witness または deterministic discharge hashes も検証する。
5. Validated hit は scheduler に immutable output handles を返す。
6. いずれかの check が失敗した場合、build mode が cache integrity diagnostics を明示要求していない限り miss として扱う。

Cache lookup failure は proof status を変えない。Phase を実行する必要があるかだけを変える。

### Artifact Commit

1. `ArtifactCommit` は一つの module について verified phase outputs and diagnostics を受け取る。
2. `ArtifactStore` は `VerifiedArtifactDraft` を canonicalize する。
3. Proof witnesses and development artifacts を temporary files へ serialize する。
4. Published `*.mizir.json` を canonical ordering で serialize し hash する。
5. Temporary files は visible になる前に flush する。
6. `ArtifactStore` は completed temporary files を final paths へ atomic rename する。
7. Platform が support する場合、containing directories を flush する。
8. `ManifestManager` は file hashes を検証し package manifest transaction を更新する。
9. Temporary `artifact-manifest.json` を flush し、previous manifest の上へ atomic rename する。
10. `CacheStore` は published artifacts に依存する cache records and cache indexes を manifest commit 成功後にのみ promote する。
11. Build events は manifest が new artifact を参照した後にのみ `ArtifactCommitted` を publish する。

Manifest commit 前のどの step が失敗しても previous manifest は valid のままである。Manifest commit が失敗した場合、新しい files が unreferenced files として残ることはあるが、manifest から到達できないため readers は無視しなければならない。Failed manifest に依存する cache updates は staged のまま残すか discard する。

### Cluster and Registration Cache Update

Cluster cache は verified contributions からのみ更新される。

- changed module contributions は origin metadata とともに書かれる
- registration が disappear または rename された場合 stale origins を remove する
- derived indexes は可能な場合 affected origins だけ rebuild する
- import-scoped views は visible origin set changes により invalidate される
- resolution traces は source snapshot、registration view hash、phase schema version で key 付けされる

Accepted されていない registration coherence proof は importer-visible indexes や `interface_hash` に入らない。

### Manifest Read Path

Downstream builds、LSP、docs、AI tooling のような readers は manifest を最初に読む。

1. `artifact-manifest.json` を read and parse する。
2. Manifest schema compatibility を確認する。
3. Package-relative paths により module entries を resolve する。
4. Consumer が要求する場合 artifact hashes を検証する。
5. Manifest entry により `*.mizir.json` and proof witness references を load する。

Readers は `build/` 内の arbitrary files を scan して published artifacts を発見してはならない。

## API Sketch

### Artifact Store

```rust
trait ArtifactStore {
    fn commit_module(&self, input: ArtifactCommitInput) -> Result<ModuleArtifactUpdate, ArtifactError>;
    fn read_manifest(&self, package: PackageId) -> Result<ArtifactManifest, ArtifactError>;
    fn read_module_artifact(&self, entry: &ModuleArtifactEntry) -> Result<VerifiedArtifact, ArtifactError>;
}
```

### Cache Store

```rust
trait CacheStore {
    fn lookup<T>(&self, key: &CacheKey, policy: CacheReusePolicy) -> Result<Option<CacheRecord<T>>, CacheError>;
    fn insert<T>(&self, record: CacheRecord<T>) -> Result<(), CacheError>;
    fn invalidate_origins(&self, origins: &[ArtifactOrigin]) -> Result<(), CacheError>;
}
```

### Cache Key Builder

```rust
trait CacheKeyBuilder {
    fn key_for_phase(&self, request: CacheKeyRequest) -> CacheKey;
    fn config_hash(&self, config: &VerifierConfig, build: &BuildConfig) -> Hash;
    fn policy_fingerprint(&self, policy: &VerifierPolicy) -> PolicyFingerprint;
}
```

### Manifest Manager

```rust
trait ManifestManager {
    fn begin(&self, package: PackageId, snapshot: BuildSnapshotId) -> Result<ManifestTransaction, ArtifactError>;
    fn stage_module(&self, tx: &mut ManifestTransaction, update: ModuleArtifactUpdate);
    fn commit(&self, tx: ManifestTransaction) -> Result<ArtifactManifest, ArtifactError>;
}
```

Manifest manager は package ごとに serialized される。Parallel module artifact writes は independent な `ModuleArtifactUpdate` values を生成し、manager がそれらを一つの package-level publication transaction に fold する。

## Error Handling

### Artifact Errors

Artifact errors は phase 15 diagnostics を生成し、affected manifest update の publication を防ぐ。

| Error | Handling |
|---|---|
| serialization failure | artifact diagnostic を報告し final file を書かない |
| hash mismatch after write | artifact integrity diagnostic を報告し manifest を更新しない |
| rename or flush failure | artifact I/O diagnostic を報告し previous manifest を authoritative のままにする |
| unsupported manifest schema | compatibility diagnostic を報告し、その manifest で reuse or publish しない |
| missing proof witness file | affected module の publication を reject する |

### Cache Errors

Cache errors は default では misses である。

Corruption、unsupported cache schema、missing blobs、dependency hash mismatches は lookup に `None` を返させ、verbose modes では warning を出してよい。Explicit cache audit or reproducibility modes でのみ hard diagnostics になる。

### Concurrency

- 一度に package あたり一つの manifest transaction だけが commit してよい
- temporary files は session and task identifiers を含むが、それらの identifiers は canonical artifact content から除外される
- obsolete sessions は newer snapshot の manifest を overwrite できない
- readers は old complete manifest または new complete manifest のどちらかを見る

## Constraints and Assumptions

- Published artifacts は、後続 published schema が別 format を明示しない限り UTF-8 JSON である。
- Published artifacts は internal cache records を load せずに読めなければならない。
- Cache records は binary encodings を使ってよいが、explicit cache schema versions を含まなければならない。
- Cache keys は complete、canonical であり、phase and work unit により domain-separated される。
- Cache entries は決して trusted proof authority ではない。
- Proof-related cache reuse は proof witness または deterministic discharge hashes を検証しなければならない。
- Manifest updates は atomic package-level publication points である。
- `build/` 内の unreferenced files は published artifacts ではない。
- `verified_at` and temporary file names は canonical content hashes から除外される。
- 実装は初期段階で coarse invalidation を選んでよいが、clean build と同じ externally visible artifacts を生成しなければならない。
