## 目的

この文書は、フェーズ 15 のアーティファクト保存、キャッシュキーの構築、キャッシュレコードの参照・更新、パッケージのアーティファクトマニフェストトランザクションについて、内部設計を定義する。

ここでは、アーティファクトのアーキテクチャを実装に近いサービスへと詳細化する。その際、公開アーティファクトは安定した外部射影であり、キャッシュレコードは証明の根拠ではなく内部的な最適化にすぎない、というアーキテクチャ上の規則を保持する。

## 責務

### アーティファクトストア

`ArtifactStore` は、各パッケージの `artifact_dir` 配下にある安定したビルド出力を管理する。複数パッケージのワークスペースでは、各パッケージがそれぞれのアーティファクトディレクトリを持つ。ワークスペースルートは、パス正規化、依存の発見、共有キャッシュ設定のために用いる。

書き込むもの:

- 公開される `*.mizir.json` アーティファクト
- パッケージのアーティファクトマニフェスト
- 公開アーティファクトから参照される証明ウィットネスファイル
- `missing_facts.json`、ATP ログ、診断説明のプレビュー、任意のデバッグダンプといった開発用アーティファクト

コンパイラ内部のキャッシュレコードを、公開アーティファクトとして外部へ見せてはならない。

### キャッシュストア

`CacheStore` は、`cache_dir` 配下にある内部ビルドキャッシュレコードを管理する。

保存するもの:

- 完全な `CacheKey` でキー付けされたフェーズ出力のキャッシュレコード
- 大きなフェーズ出力のためのコンテンツアドレス方式の blob
- `cluster-db/` のインデックス、登録の寄与、インポートスコープのビュー
- 内部キャッシュデータとしての解決トレースと診断説明の裏付けデータ

キャッシュレコードはいつ削除されてもよい。削除しても、ソースレベルの意味論や公開パッケージの互換性を変えてはならない。

### マニフェストマネージャ

`ManifestManager` は、パッケージごとのアーティファクトマニフェストへのアトミックな更新を管理する。

新しいマニフェストバージョンを公開できる唯一の書き手である。マニフェストが読み手から見えるようになる前に、すべてのマニフェストエントリが既に書き込み済みの完全なアーティファクトを指していること、および記録されたハッシュがディスク上のファイルと一致することを検証する。

### キャッシュキー構築器

`CacheKeyBuilder` は、正規化された同一性、ハッシュ、スキーマバージョン、検証器ポリシーからキャッシュキーを構築する。

キャッシュエントリが信頼できるかどうかは判断しない。キーと依存サマリを生成するだけである。再利用の可否は、`CacheStore` の互換性チェックと証明ウィットネスの検証が決める。

## データモデル

### ストアのレイアウト

既定の物理レイアウトは、仕様で定めるパッケージマニフェストの設定に従う。

標準的なパッケージのアーティファクトマニフェストのファイル名は `artifact-manifest.json` である。

```text
build/
  artifact-manifest.json
  <module>.mizir.json
  proof-witnesses/
    <module>/
      vc-<id>.json
  logs/
    <module>.atp-log
  missing_facts.json
  explanations/
    <module>/
      <diagnostic-id>.json

.mizar-cache/
  records/
    <phase>/
      <cache-key>.bin
  blobs/
    sha256/
      <hash>
  cluster-db/
  resolution-trace/
  diagnostic-explanation/
  tmp/
```

レイアウト自体は互換性の約束ではない。例外は、マニフェストに記録される公開アーティファクトのパスと、ユーザー向け仕様が定めるパスである。読み手はまずマニフェストを参照する。

### アーティファクトストアルート

```rust
struct ArtifactStoreRoot {
    workspace_root: PathBuf,
    package_root: PathBuf,
    artifact_dir: PathBuf,
    cache_dir: PathBuf,
    path_policy: PathPolicy,
}

enum PathPolicy {
    PackageRelative,
    WorkspaceRelative,
    LocalAbsoluteForDiagnostics,
}
```

可搬なアーティファクトは、パッケージ相対またはワークスペース相対のパスを用いる。絶対パスは、明示的にローカルな診断またはデバッグ用アーティファクトの場合に限って現れてよい。

### アーティファクト書き込み入力

```rust
struct ArtifactCommitInput {
    module_path: ModulePath,
    source_file: SourcePath,
    source_hash: Hash,
    verified_artifact: VerifiedArtifactDraft,
    proof_witnesses: Vec<ProofWitnessDraft>,
    diagnostics: Vec<Diagnostic>,
    development_outputs: Vec<DevelopmentArtifactDraft>,
    provenance: BuildProvenance,
}
```

`VerifiedArtifactDraft` は、フェーズ出力から組み立てられるメモリ上の射影である。正準的なシリアライズ、コンテンツハッシュの計算、マニフェストのコミットを経て初めて、公開される `VerifiedArtifact` となる。

### マニフェストトランザクション

```rust
struct ManifestTransaction {
    package_id: PackageId,
    base_manifest_hash: Option<Hash>,
    session: BuildSessionId,
    snapshot: BuildSnapshotId,
    module_updates: Vec<ModuleArtifactUpdate>,
}

struct ModuleArtifactUpdate {
    module_path: ModulePath,
    artifact_file: PathBuf,
    artifact_hash: Hash,
    interface_hash: Hash,
    implementation_hash: Hash,
    proof_witnesses: Vec<ProofWitnessEntry>,
    diagnostics_hash: Option<Hash>,
}
```

トランザクションは、参照されるすべてのファイルを先に書き込み、それらのハッシュを検証し、一時マニフェストを書き、それを以前のマニフェストの上へアトミックにリネームすることでコミットされる。

### キャッシュキー

```rust
struct CacheKey {
    cache_schema_version: SchemaVersion,
    phase: PipelinePhase,
    work_unit: WorkUnit,
    source_identity: Option<SourceIdentity>,
    input_hashes: Vec<NamedHash>,
    dependency_hashes: Vec<DependencyHash>,
    dependency_slices: Vec<DependencySliceHash>,
    config_hash: Hash,
    schema_versions: Vec<SchemaVersion>,
    policy_fingerprint: PolicyFingerprint,
}

struct SourceIdentity {
    package_id: PackageId,
    module_path: ModulePath,
    normalized_source_path: PathBuf,
    source_hash: Hash,
    language_edition: Edition,
}
```

すべてのベクタは、ハッシュ計算の前に正準キーで整列される。`dependency_slices` は、キャッシュされた出力が用いた定理、定義、クラスタ、記法、モード、属性の各スライスについて、明示的なハッシュを保持する。最終的なキーは、正準エンコーディングをドメイン分離してハッシュしたものである。`cache_schema_version` はキャッシュレコードのエンコーディングと互換性規則を識別し、`schema_versions` はキャッシュ値の意味に影響するフェーズ出力スキーマと公開アーティファクトスキーマを識別する。

### キャッシュレコード

```rust
struct CacheRecordHeader {
    cache_schema_version: SchemaVersion,
    key: CacheKey,
    produced_by: ToolchainInfo,
    output_hash: Hash,
    dependencies: Vec<CacheDependency>,
    proof_reuse: Option<ProofReuseEvidence>,
}

struct CacheRecord<T> {
    header: CacheRecordHeader,
    output: T,
}
```

証明に関わるキャッシュレコードは、受理された証明ウィットネス、または決定的な組み込み義務解消のレコードを参照しなければならない。再利用前には `ObligationAnchor`、canonical VC fingerprint、dependency slice、verifier policy、witness/discharge hash を検証しなければならない。キャッシュレコードは、外部が証言したエビデンスを、カーネル検証済みのステータスへ格上げすることはできない。

## 制御フロー

### タスク実行前のキャッシュ参照

1. スケジューラは、必要な入力ハッシュがすべて揃った時点で、フェーズサービスに `CacheKey` を要求する。
2. `CacheStore` が、厳密なキーでレコードを参照する。
3. レコードヘッダについて、キャッシュスキーマの互換性、ツールチェインの互換性、依存アーティファクトの利用可能性、依存ハッシュの一致、検証器ポリシーの厳格さを確認する。
4. 証明に関わるレコードでは、証明ウィットネスまたは決定的な義務解消のハッシュも検証する。
5. 検証に通ったヒットは、不変の出力ハンドルをスケジューラへ返す。
6. いずれかの確認が失敗した場合は、ビルドモードがキャッシュ整合性の診断を明示的に要求しない限り、ミスとして扱う。

キャッシュ参照の失敗は、証明ステータスを変えない。フェーズを実行する必要があるかどうかだけを変える。`CacheHit` は validation 後の scheduling state であり、proof evidence ではない。

### アーティファクトのコミット

1. `ArtifactCommit` が、1 つのモジュールについて、検証済みのフェーズ出力と診断を受け取る。
2. `ArtifactStore` が `VerifiedArtifactDraft` を正準化する。
3. 証明ウィットネスと開発用アーティファクトを一時ファイルへシリアライズする。
4. 公開する `*.mizir.json` を正準順序でシリアライズし、ハッシュを計算する。
5. 一時ファイルは、可視になる前にフラッシュする。
6. `ArtifactStore` が、完成した一時ファイルを最終パスへアトミックにリネームする。
7. プラットフォームが対応していれば、格納先ディレクトリをフラッシュする。
8. `ManifestManager` がファイルハッシュを検証し、パッケージマニフェストトランザクションを更新する。
9. 一時の `artifact-manifest.json` をフラッシュし、以前のマニフェストの上へアトミックにリネームする。
10. `CacheStore` は、公開アーティファクトに依存するキャッシュレコードとキャッシュインデックスを、マニフェストのコミットが成功し、生成元 phase output が完了・検証された後にのみ昇格させる。
11. ビルドイベントは、マニフェストが新しいアーティファクトを参照した後にのみ `ArtifactCommitted` を公開する。

マニフェストのコミット前のいずれかの手順が失敗しても、以前のマニフェストは有効なままである。マニフェストのコミットが失敗した場合、新しいファイルが参照されないファイルとして残ることはあるが、マニフェストから到達できないため、読み手はそれらを無視しなければならない。失敗したマニフェストに依存するキャッシュ更新は、ステージのまま残すか破棄する。

同じ cache entry を生成しようとする 2 つの worker は、byte-identical な内容を公開するか、一方の writer が atomic commit race に負け、観測可能な意味論差を生まないかのどちらかでなければならない。同じ validated key に対する divergent cache contents は cache integrity error であり、独立した proof outcome として見えてはならない。

### クラスタと登録のキャッシュ更新

クラスタキャッシュは、検証済みの寄与からのみ更新される。

- 変更されたモジュールの寄与は、由来メタデータとともに書き込まれる。
- 登録が消失または改名された場合、古い由来を除去する。
- 派生インデックスは、可能であれば影響を受けた由来についてのみ再構築する。
- インポートスコープのビューは、可視な由来集合の変化によって無効化される。
- 解決トレースは、ソーススナップショット、登録ビューのハッシュ、フェーズスキーマバージョンでキー付けされる。

受理されていない登録の整合性証明は、インポート側から見えるインデックスや `interface_hash` には入らない。

### マニフェストの読み取り経路

下流のビルド、LSP、ドキュメント、AI ツールといった読み手は、まずマニフェストを参照する。

1. `artifact-manifest.json` を読み込んで解析する。
2. マニフェストスキーマの互換性を確認する。
3. パッケージ相対パスでモジュールエントリを解決する。
4. 消費者が要求する場合は、アーティファクトのハッシュを検証する。
5. マニフェストエントリに従って `*.mizir.json` と証明ウィットネス参照を読み込む。

読み手は、`build/` 内の任意のファイルを走査して公開アーティファクトを発見してはならない。

## API 概要

### アーティファクトストア

```rust
trait ArtifactStore {
    fn commit_module(&self, input: ArtifactCommitInput) -> Result<ModuleArtifactUpdate, ArtifactError>;
    fn read_manifest(&self, package: PackageId) -> Result<ArtifactManifest, ArtifactError>;
    fn read_module_artifact(&self, entry: &ModuleArtifactEntry) -> Result<VerifiedArtifact, ArtifactError>;
}
```

### キャッシュストア

```rust
trait CacheStore {
    fn lookup<T>(&self, key: &CacheKey, policy: CacheReusePolicy) -> Result<Option<CacheRecord<T>>, CacheError>;
    fn insert<T>(&self, record: CacheRecord<T>) -> Result<(), CacheError>;
    fn invalidate_origins(&self, origins: &[ArtifactOrigin]) -> Result<(), CacheError>;
}
```

### キャッシュキー構築器

```rust
trait CacheKeyBuilder {
    fn key_for_phase(&self, request: CacheKeyRequest) -> CacheKey;
    fn config_hash(&self, config: &VerifierConfig, build: &BuildConfig) -> Hash;
    fn policy_fingerprint(&self, policy: &VerifierPolicy) -> PolicyFingerprint;
}
```

### マニフェストマネージャ

```rust
trait ManifestManager {
    fn begin(&self, package: PackageId, snapshot: BuildSnapshotId) -> Result<ManifestTransaction, ArtifactError>;
    fn stage_module(&self, tx: &mut ManifestTransaction, update: ModuleArtifactUpdate);
    fn commit(&self, tx: ManifestTransaction) -> Result<ArtifactManifest, ArtifactError>;
}
```

マニフェストマネージャはパッケージごとに直列化される。並列なモジュールアーティファクト書き込みは、それぞれ独立した `ModuleArtifactUpdate` 値を生成し、マネージャがそれらを 1 つのパッケージ単位の公開トランザクションへ畳み込む。

## エラー処理

### アーティファクトエラー

アーティファクトエラーはフェーズ 15 の診断を生成し、影響を受けるマニフェスト更新の公開を防ぐ。

| エラー | 処理 |
|---|---|
| シリアライズの失敗 | アーティファクト診断を報告し、最終ファイルを書かない |
| 書き込み後のハッシュ不一致 | アーティファクト整合性の診断を報告し、マニフェストを更新しない |
| リネームまたはフラッシュの失敗 | アーティファクト I/O 診断を報告し、以前のマニフェストを権威として残す |
| 非対応のマニフェストスキーマ | 互換性の診断を報告し、そのマニフェストでの再利用や公開を行わない |
| 証明ウィットネスファイルの欠落 | 当該モジュールの公開を拒否する |

### キャッシュエラー

キャッシュエラーは、既定ではミスとして扱う。

破損、非対応のキャッシュスキーマ、欠落した blob、依存ハッシュの不一致は、参照に `None` を返させ、詳細出力モードでは警告を出してよい。明示的なキャッシュ監査モードや再現性モードに限り、これらはハードな診断となる。

### 並行性

- パッケージあたり同時にコミットできるマニフェストトランザクションは 1 つだけである。
- 一時ファイルはセッション識別子とタスク識別子を含むが、それらの識別子は正準的なアーティファクト内容からは除外される。
- 古くなったセッションは、より新しいスナップショットのマニフェストを上書きできない。
- 読み手は、古い完全なマニフェストか、新しい完全なマニフェストのいずれかを見る。

## 制約と前提

- 公開アーティファクトは、後続の公開スキーマが別の形式を明示しない限り、UTF-8 の JSON である。
- 公開アーティファクトは、内部キャッシュレコードを読み込まずに読めなければならない。
- キャッシュレコードはバイナリエンコーディングを用いてよいが、明示的なキャッシュスキーマバージョンを含まなければならない。
- キャッシュキーは完全かつ正準的であり、フェーズと作業単位によってドメイン分離される。
- キャッシュエントリは、決して信頼された証明の根拠ではない。
- 証明に関わるキャッシュ再利用は、証明ウィットネスまたは決定的な義務解消のハッシュを検証しなければならない。
- マニフェスト更新は、パッケージ単位のアトミックな公開ポイントである。
- `build/` 内の参照されないファイルは、公開アーティファクトではない。
- `verified_at` と一時ファイル名は、正準的なコンテンツハッシュから除外される。
- 実装は初期段階で粒度の粗い無効化を選んでよいが、クリーンビルドと同一の、外部から見えるアーティファクトを生成しなければならない。
