# Internal Design: Compiler Driver and Pipeline Scheduler

> Canonical language: English. English canonical version: [../en/01.compiler_driver_and_pipeline_scheduler.md](../en/01.compiler_driver_and_pipeline_scheduler.md).

## 目的

この文書は compiler driver and pipeline scheduler の internal design を定義する。

Architecture pipeline を、build requests を受け取り、versioned task graphs を作成し、phase services を呼び出し、cache records を再利用し、diagnostics を集約し、ATP/kernel work を実行し、architecture layer が定義する semantic results を変えずに artifacts を commit する implementation services へ詳細化する。

## Responsibilities

### Compiler Driver

Compiler driver はすべての build modes の front door である。

Driver が所有するもの:

- CLI/watch/LSP build requests を `BuildRequest` に変換する
- phase 0 package/workspace resolution を build bootstrap として呼び出す
- source and dependency snapshots を持つ `BuildSession` values を作成する
- phase service implementations を登録する
- initial task graph を scheduler に submit する
- build events を CLI progress reporters and LSP bridges へ publish する

Driver が所有しないもの:

- type checking、overload resolution、VC generation、proof acceptance のような phase semantics
- concrete cache key が構築された後の cache compatibility decisions
- artifact serialization details
- editor protocol range conversion

### Pipeline Scheduler

Scheduler は dependency constraints の下で execution order を所有する。

Scheduler がしてよいこと:

- independent tasks を parallel に実行する
- open files and dependency summaries を優先する
- expensive work を schedule する前に cache lookup を行う
- obsolete tasks を build snapshot により cancel する
- tasks を CPU、ATP、kernel、I/O queues へ route する
- progress events を emit する

Scheduler がしてはならないこと:

- phase semantics を変更する
- configured policy and kernel boundary なしに proof evidence を accept する
- obsolete snapshots からの diagnostics を current として publish する
- artifacts を completion order で commit する
- partial artifact writes を expose する

### Phase Services

各 phase service は一つの architecture phase、または小さな phase group の実装である。

| Service | Phases | Main Output |
|---|---:|---|
| `WorkspacePlanner` | 0 | `BuildPlan`, dependency graph, verifier/build config |
| `SourceFrontend` | 1-3 | `SourceUnit`, `PreprocessedSource`, `TokenStream`, `SurfaceAst` |
| `ModuleResolver` | 4-5 | `ResolvedAst`, `SymbolEnv`, `ModuleSummary` |
| `SemanticChecker` | 6-8 | `TypedAst`, `ResolutionTrace`, `ResolvedTypedAst` |
| `Elaborator` | 9-10 | `CoreIr`, `ControlFlowIr` |
| `VcService` | 11-12 | `VcIr`, deterministic discharge status |
| `AtpService` | 13 | `AtpProblem`, backend candidate evidence |
| `KernelService` | 14 | kernel-accepted proof status |
| `ArtifactService` | 15 | `VerifiedArtifact`, proof witness refs, manifest entries |
| `DocExtractionService` | 16 | rendered docs and extracted code outputs |

Services は structured phase outputs を返す。Scheduler state を直接 mutate しない。

## Data Model

### Build Session

```rust
struct BuildSession {
    id: BuildSessionId,
    request: BuildRequest,
    snapshot: BuildSnapshot,
    plan: BuildPlan,
    phase_registry: PhaseRegistry,
    event_sink: BuildEventSink,
}

struct BuildSnapshot {
    id: BuildSnapshotId,
    workspace_root: PathBuf,
    source_versions: Vec<SourceVersion>,
    dependency_artifacts: Vec<DependencyArtifactRef>,
    lockfile_hash: Hash,
    toolchain: ToolchainInfo,
    verifier_config_hash: Hash,
}
```

`BuildSessionId` は一つの scheduler run を識別する。`BuildSnapshotId` は、その run が使った exact source、dependency、lockfile、toolchain、verifier settings を識別する。Task result は session id ではなく cache validation を通してのみ session 間で reuse される。

### Task Graph

```rust
struct PipelineTask {
    id: TaskId,
    session: BuildSessionId,
    snapshot: BuildSnapshotId,
    kind: PipelineTaskKind,
    unit: WorkUnit,
    dependencies: Vec<TaskId>,
    cache_key: Option<CacheKey>,
    priority: TaskPriority,
    resource_class: ResourceClass,
}

enum PipelineTaskKind {
    PackageResolve,
    SourceLoad,
    Frontend,
    ModuleResolve,
    CheckAndElaborate,
    VcGenerate,
    VcDischarge,
    AtpSolve,
    BackendRun,
    KernelCheck,
    ArtifactCommit,
    DocumentationExtract,
}
```

Scheduler は初期実装では `CheckAndElaborate` のように隣接 phase を coarse task にまとめてよい。ただし各 task は、どの architecture phases が outputs を生成したかを記録しなければならない。

`PackageResolve` は phase 0 bootstrap output を表す。Default implementation では driver が graph submission 前に `WorkspacePlanner` を実行し、scheduled tasks の completed root input として `PackageResolve` を記録する。後続実装では phase 0 自体を schedule してもよいが、downstream tasks を作成する前に session につき一つの `BuildPlan` を必ず生成しなければならない。

### Phase Output Handle

```rust
struct PhaseOutputRef<T> {
    phase: PipelinePhase,
    unit: WorkUnit,
    snapshot: BuildSnapshotId,
    content_hash: Hash,
    storage: OutputStorageRef<T>,
}
```

Tasks は mutable IR values ではなく output handles を交換する。Output handles は dependent tasks に publish された後 immutable である。

### Task Result

```rust
struct PipelineTaskResult {
    task_id: TaskId,
    state: TaskState,
    phase_outputs: Vec<AnyPhaseOutputRef>,
    diagnostics: Vec<Diagnostic>,
    cache_record: Option<CacheRecordRef>,
    timing: TimingSummary,
}
```

`TaskState::CacheHit` は、cache validation 後に scheduler が execution を skip したことを意味する。それでも completed dependency として記録され、通常の task results とともに sort される。

## Control Flow

### Batch Build

1. Driver は CLI options and package defaults から `BuildRequest` を作成する。
2. `WorkspacePlanner` が manifests、dependencies、lockfile、edition、verifier settings を解決する。
3. Scheduler は phases 1-10 について source-file tasks を、phases 11-14 について VC-level tasks を、phase 15 について artifact commit tasks を、phase 16 について documentation/extraction tasks を作成する。
4. 必要な input hashes が揃い次第 cache keys を構築する。
5. Cache hits は validated output handles を生成し、cache misses は ready queue に入る。
6. Diagnostics は source file and build snapshot ごとに aggregate される。
7. Artifact commit tasks は module の proof status and diagnostics が完了した後に実行される。
8. Documentation/extraction tasks は build profile で有効な場合、verified artifacts の commit 後に実行される。
9. Driver は ready work が completed、failed、blocked、cancelled になった後 final build status を報告する。

Phase 16 は batch graph の一部だが、separate task class のままである。Verification だけを必要とする profile は、phases 0-15 を変えずに documentation rendering、code extraction、またはその両方を disable してよい。

### Watch and LSP Build

Watch and LSP builds は同じ driver and scheduler を使うが、edits ごとに new snapshots を作成する。

- changed open files の frontend tasks を優先する
- semantic tasks は new snapshot に一致する dependency summaries を待つ
- obsolete tasks は cooperative に cancel する
- cached outputs は cache keys が new snapshot に一致する場合のみ reuse する
- diagnostics は freshness information とともに publish する
- LSP snapshots は aggregation 後に atomically switch する

Open editor buffers は source snapshots であり published artifacts ではない。LSP bridge は editor boundary で source positions を変換し code actions を適用する。

### ATP and Kernel Subgraph

各 open `VcId` について scheduler は `AtpSolve` task を作成する。ATP service は configured portfolio policy に従い backend `BackendRun` tasks へ展開する。

Backend results は candidate evidence である。Policy が kernel validation を要求する場合、eligible candidates は `KernelCheck` tasks を作成する。Accepted proof status は backend completion order ではなく、policy and kernel result により選択される。

### Artifact Commit Boundary

`ArtifactCommit` は serialization boundary である。

これは verified phase outputs、diagnostics、proof witness references、provenance を受け取り、artifact service を通して atomic manifest transaction で artifacts を書く。Commit order は worker completion order ではなく canonical module order に従う。

## API Sketch

### Driver API

```rust
trait CompilerDriver {
    fn submit(&self, request: BuildRequest) -> BuildSessionId;
    fn cancel(&self, session: BuildSessionId, reason: CancelReason);
    fn events(&self, session: BuildSessionId) -> BuildEventStream;
}
```

### Phase Service API

```rust
trait PhaseService<I, O> {
    fn phase(&self) -> PipelinePhase;
    fn cache_key(&self, input: &I, context: &PhaseContext) -> Option<CacheKey>;
    fn execute(&self, input: I, context: PhaseContext) -> PhaseResult<O>;
}

struct PhaseResult<O> {
    status: PhaseStatus,
    output: Option<O>,
    diagnostics: Vec<DiagnosticDraft>,
    output_refs: Vec<AnyPhaseOutputRef>,
}

struct PhaseContext {
    session: BuildSessionId,
    snapshot: BuildSnapshotId,
    cancellation: CancellationToken,
    diagnostics: DiagnosticSink,
    output_store: OutputStoreHandle,
}
```

`cache_key` は mutable task state を読んではならない。Input identities、configuration、schema versions、dependency hashes からの pure projection である。

### Scheduler API

```rust
trait PipelineScheduler {
    fn schedule(&self, session: BuildSession, graph: PipelineGraph);
    fn cancel_snapshot(&self, snapshot: BuildSnapshotId, reason: CancelReason);
    fn subscribe(&self, session: BuildSessionId) -> BuildEventStream;
}
```

### Build Events

```rust
enum BuildEvent {
    SessionStarted(BuildSessionId),
    TaskStarted(TaskId),
    TaskFinished(TaskId, TaskState),
    DiagnosticsReady(SourceId, BuildSnapshotId),
    ArtifactCommitted(ModulePath, Hash),
    SnapshotPublished(BuildSnapshotId),
    SessionFinished(BuildSessionId, BuildStatus),
}
```

Events は progress and integration signals である。Artifact ordering rules ではない。

## Error Handling

### Phase Failure

Phase services は diagnostics and phase status を返す。

```rust
enum PhaseStatus {
    Complete,
    Recoverable,
    Blocking,
    Fatal,
}
```

- `Recoverable` は degraded metadata による downstream work を許す。
- `Blocking` は affected unit の dependent work を止めるが、independent tasks は止めない。
- `Fatal` は global build state が信頼できない場合に session を止める。

### Cancellation

Cancellation は snapshot により versioned される。

- in-process tasks は phase boundaries and safe checkpoints で tokens を確認する
- ATP child processes は backend manager が terminate する
- obsolete completed results は cache validation により new snapshot へ適用可能と証明されない限り discard する
- artifact commit は atomic transaction 開始後には cancel しない

### Diagnostics

Diagnostics は per-session aggregator を通る。Aggregator は records を deduplicate し、deterministic `DiagnosticId`s を割り当て、architecture-defined order で sort し、target consumer に対して current な snapshot の diagnostics だけを publish する。

## Constraints and Assumptions

- Scheduler は execution order を制御するが phase semantics は制御しない。
- すべての task outputs は publication 後 immutable である。
- Cache hits は artifact and incremental build architecture の validation rules を満たさなければならない。
- Diagnostics and artifacts は task completion order ではなく canonical order で sort する。
- Kernel certificate checking は trusted proof acceptance boundary のままである。
- LSP builds は open files を優先してよいが、incompatible snapshots の diagnostics を混ぜてはならない。
- Documentation and extraction は verified artifacts が利用可能になった後に schedule される。
