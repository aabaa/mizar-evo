# 内部設計: コンパイラドライバとパイプラインスケジューラ

> 正本は英語です。英語版: [../en/01.compiler_driver_and_pipeline_scheduler.md](../en/01.compiler_driver_and_pipeline_scheduler.md)。

## 目的

この文書は、コンパイラドライバとパイプラインスケジューラの内部設計を定義する。

ここでは、アーキテクチャ上のパイプラインを、ビルド要求を受け取り、バージョン付きタスクグラフを作成し、フェーズサービスを呼び出し、キャッシュレコードを再利用し、診断を集約し、ATP／カーネル処理を実行し、アーティファクトをコミットする一連の実装サービスへと詳細化する。その際、アーキテクチャ層が定義する意味的な結果は変えない。

## 責務

### コンパイラドライバ

コンパイラドライバは、すべてのビルドモードの入口である。

ドライバが担うもの:

- CLI／ウォッチ／LSP のビルド要求を `BuildRequest` へ変換する。
- ビルドの起点として、フェーズ 0 のパッケージ／ワークスペース解決を呼び出す。
- ソースと依存のスナップショットを持つ `BuildSession` 値を作成する。
- フェーズサービスの実装を登録する。
- 初期タスクグラフをスケジューラへ投入する。
- ビルドイベントを CLI の進捗レポーターと LSP ブリッジへ公開する。

ドライバが担わないもの:

- 型検査、オーバーロード解決、VC 生成、証明の受理といったフェーズの意味論。
- 具体的なキャッシュキーが構築された後の、キャッシュ互換性の判断。
- アーティファクトのシリアライズの詳細。
- エディタプロトコルにおける範囲変換。

### パイプラインスケジューラ

スケジューラは、依存制約のもとでの実行順序を担う。

スケジューラが行ってよいこと:

- 独立したタスクを並列に実行する。
- 開いているファイルと依存サマリを優先する。
- 高コストな処理をスケジュールする前にキャッシュ参照を行う。
- 古くなったタスクをビルドスナップショット単位でキャンセルする。
- タスクを CPU、ATP、カーネル、I/O の各キューへ振り分ける。
- 進捗イベントを発行する。

スケジューラが行ってはならないこと:

- フェーズの意味論を変更する。
- 設定されたポリシーとカーネル境界を経ずに証明エビデンスを受理する。
- 古いスナップショットの診断を、現在のものとして公開する。
- アーティファクトを完了順にコミットする。
- 書き込み途中のアーティファクトを外部に見せる。

### フェーズサービス

各フェーズサービスは、1 つのアーキテクチャフェーズ、または小さなフェーズ群の実装である。

| サービス | フェーズ | 主な出力 |
|---|---:|---|
| `WorkspacePlanner` | 0 | `BuildPlan`、依存グラフ、検証器／ビルド設定 |
| `SourceFrontend` | 1〜3 | `SourceUnit`、`PreprocessedSource`、`TokenStream`、`SurfaceAst` |
| `ModuleResolver` | 4〜5 | `ResolvedAst`、`SymbolEnv`、`ModuleSummary` |
| `SemanticChecker` | 6〜8 | `TypedAst`、`ResolutionTrace`、`ResolvedTypedAst` |
| `Elaborator` | 9〜10 | `CoreIr`、`ControlFlowIr` |
| `VcService` | 11〜12 | `VcIr`、決定的な義務解消ステータス |
| `AtpService` | 13 | `AtpProblem`、バックエンドの候補エビデンス |
| `KernelService` | 14 | カーネルが受理した証明ステータス |
| `ArtifactService` | 15 | `VerifiedArtifact`、証明ウィットネス参照、マニフェストエントリ |
| `DocExtractionService` | 16 | レンダリング済みドキュメントと抽出されたコード出力 |

各サービスは構造化されたフェーズ出力を返す。スケジューラの状態を直接書き換えることはない。

## データモデル

### ビルドセッション

```rust
struct BuildSession {
    id: BuildSessionId,
    request: BuildRequest,
    snapshot: BuildSnapshot,
    plan: BuildPlan,
    phase_registry: PhaseRegistry,
    event_sink: BuildEventSink,
}

struct BuildSnapshot {
    id: BuildSnapshotId,
    workspace_root: PathBuf,
    source_versions: Vec<SourceVersion>,
    dependency_artifacts: Vec<DependencyArtifactRef>,
    lockfile_hash: Hash,
    toolchain: ToolchainInfo,
    verifier_config_hash: Hash,
}
```

`BuildSessionId` は 1 回のスケジューラ実行を識別する。`BuildSnapshotId` は、その実行が用いた厳密なソース、依存、ロックファイル、ツールチェイン、検証器設定を識別する。タスク結果がセッションをまたいで再利用されるのは、セッション ID によってではなく、キャッシュ検証を通した場合に限られる。

### タスクグラフ

```rust
struct PipelineTask {
    id: TaskId,
    session: BuildSessionId,
    snapshot: BuildSnapshotId,
    kind: PipelineTaskKind,
    unit: WorkUnit,
    dependencies: Vec<TaskId>,
    cache_key: Option<CacheKey>,
    priority: TaskPriority,
    resource_class: ResourceClass,
}

enum PipelineTaskKind {
    PackageResolve,
    SourceLoad,
    Frontend,
    ModuleResolve,
    CheckAndElaborate,
    VcGenerate,
    VcDischarge,
    AtpSolve,
    BackendRun,
    KernelCheck,
    ArtifactCommit,
    DocumentationExtract,
}
```

スケジューラは初期実装において、`CheckAndElaborate` のように隣接するフェーズを粒度の粗いタスクへまとめてよい。ただし各タスクは、どのアーキテクチャフェーズが出力を生成したかを必ず記録しなければならない。

`PackageResolve` は、フェーズ 0 の起点となる出力を表す。既定の実装では、ドライバがグラフ投入の前に `WorkspacePlanner` を実行し、`PackageResolve` をスケジュール対象タスクの完了済みルート入力として記録する。後続の実装ではフェーズ 0 自体をスケジュールしてもよいが、その場合も下流のタスクを作成する前に、セッションごとに必ず 1 つの `BuildPlan` を生成しなければならない。

### フェーズ出力ハンドル

```rust
struct PhaseOutputRef<T> {
    phase: PipelinePhase,
    unit: WorkUnit,
    snapshot: BuildSnapshotId,
    content_hash: Hash,
    storage: OutputStorageRef<T>,
}
```

タスクは可変な IR 値ではなく、出力ハンドルをやり取りする。出力ハンドルは、依存タスクへ公開された時点で不変となる。

### タスク結果

```rust
struct PipelineTaskResult {
    task_id: TaskId,
    state: TaskState,
    phase_outputs: Vec<AnyPhaseOutputRef>,
    diagnostics: Vec<Diagnostic>,
    cache_record: Option<CacheRecordRef>,
    timing: TimingSummary,
}
```

`TaskState::CacheHit` は、キャッシュ検証の後にスケジューラが実行を省略したことを意味する。それでも完了済みの依存として記録され、通常のタスク結果と同じように整列される。

## 制御フロー

### バッチビルド

1. ドライバが、CLI オプションとパッケージの既定値から `BuildRequest` を作成する。
2. `WorkspacePlanner` が、マニフェスト、依存、ロックファイル、エディション、検証器設定を解決する。
3. スケジューラが、フェーズ 1〜10 についてソースファイル単位のタスクを、フェーズ 11〜14 について VC 単位のタスクを、フェーズ 15 についてアーティファクトコミットタスクを、フェーズ 16 についてドキュメント／抽出タスクを作成する。
4. 必要な入力ハッシュがすべて判明し次第、キャッシュキーを構築する。
5. キャッシュヒットは検証済みの出力ハンドルを生み、キャッシュミスは実行待ちキューに入る。
6. 診断は、ソースファイルとビルドスナップショットごとに集約される。
7. アーティファクトコミットタスクは、当該モジュールの証明ステータスと診断が確定した後に実行される。
8. ドキュメント／抽出タスクは、ビルドプロファイルで有効な場合に、検証済みアーティファクトのコミット後に実行される。
9. ドライバは、実行待ち処理が完了・失敗・ブロック・キャンセルのいずれかに至った後、最終的なビルドステータスを報告する。

フェーズ 16 はバッチグラフの一部だが、独立したタスク種別のままである。検証のみを必要とするプロファイルは、フェーズ 0〜15 を変えずに、ドキュメントレンダリング、コード抽出、またはその両方を無効化してよい。

### ウォッチビルドと LSP ビルド

ウォッチビルドと LSP ビルドは同じドライバとスケジューラを用いるが、編集ごとに新しいスナップショットを作成する。

- 変更された、開いているファイルのフロントエンドタスクを優先する。
- 意味解析タスクは、新しいスナップショットと一致する依存サマリを待つ。
- 古くなったタスクは協調的にキャンセルする。
- キャッシュ済みの出力は、そのキャッシュキーが新しいスナップショットと一致する場合にのみ再利用する。
- 診断は鮮度情報を伴って公開する。
- LSP スナップショットは集約後にアトミックに切り替える。

エディタで開いているバッファはソーススナップショットであり、公開アーティファクトではない。LSP ブリッジは、エディタとの境界でソース位置を変換し、コードアクションを適用する。

### ATP・カーネルのサブグラフ

スケジューラは、未解消の `VcId` ごとに `AtpSolve` タスクを作成する。ATP サービスは、設定されたポートフォリオポリシーに従って、これをバックエンドの `BackendRun` タスクへ展開する。

バックエンドの結果は候補エビデンスである。ポリシーがカーネル検証を要求する場合、対象となる候補は `KernelCheck` タスクを生成する。受理される証明ステータスは、バックエンドの完了順ではなく、ポリシーとカーネルの結果によって選択される。

### アーティファクトコミットの境界

`ArtifactCommit` はシリアライズの境界である。

ここでは、検証済みのフェーズ出力、診断、証明ウィットネス参照、由来情報を受け取り、アーティファクトサービスを通じてアトミックなマニフェストトランザクションでアーティファクトを書き込む。コミットの順序は、ワーカーの完了順ではなく、正準的なモジュール順に従う。

## API 概要

### ドライバ API

```rust
trait CompilerDriver {
    fn submit(&self, request: BuildRequest) -> BuildSessionId;
    fn cancel(&self, session: BuildSessionId, reason: CancelReason);
    fn events(&self, session: BuildSessionId) -> BuildEventStream;
}
```

### フェーズサービス API

```rust
trait PhaseService<I, O> {
    fn phase(&self) -> PipelinePhase;
    fn cache_key(&self, input: &I, context: &PhaseContext) -> Option<CacheKey>;
    fn execute(&self, input: I, context: PhaseContext) -> PhaseResult<O>;
}

struct PhaseResult<O> {
    status: PhaseStatus,
    output: Option<O>,
    diagnostics: Vec<DiagnosticDraft>,
    output_refs: Vec<AnyPhaseOutputRef>,
}

struct PhaseContext {
    session: BuildSessionId,
    snapshot: BuildSnapshotId,
    cancellation: CancellationToken,
    diagnostics: DiagnosticSink,
    output_store: OutputStoreHandle,
}
```

`cache_key` は可変なタスク状態を読んではならない。これは、入力の同一性、設定、スキーマバージョン、依存ハッシュから決まる純粋な射影である。

### スケジューラ API

```rust
trait PipelineScheduler {
    fn schedule(&self, session: BuildSession, graph: PipelineGraph);
    fn cancel_snapshot(&self, snapshot: BuildSnapshotId, reason: CancelReason);
    fn subscribe(&self, session: BuildSessionId) -> BuildEventStream;
}
```

### ビルドイベント

```rust
enum BuildEvent {
    SessionStarted(BuildSessionId),
    TaskStarted(TaskId),
    TaskFinished(TaskId, TaskState),
    DiagnosticsReady(SourceId, BuildSnapshotId),
    ArtifactCommitted(ModulePath, Hash),
    SnapshotPublished(BuildSnapshotId),
    SessionFinished(BuildSessionId, BuildStatus),
}
```

イベントは進捗と連携のためのシグナルであり、アーティファクトの順序を定める規則ではない。

## エラー処理

### フェーズの失敗

フェーズサービスは、診断とフェーズステータスを返す。

```rust
enum PhaseStatus {
    Complete,
    Recoverable,
    Blocking,
    Fatal,
}
```

- `Recoverable` は、構文のみの LSP 機能、開発用の説明、ドキュメントのプレビューのように、劣化したメタデータに意味論が依存しない下流処理だけを許す。意味的な受理と証明ステータスは、回復用のフォールバックデータに依存してはならない。
- `Blocking` は、当該作業単位の依存処理を止めるが、独立したタスクは止めない。
- `Fatal` は、グローバルなビルド状態が信頼できない場合にセッションを止める。

### キャンセル

キャンセルはスナップショット単位でバージョン管理される。

- プロセス内のタスクは、フェーズ境界と安全なチェックポイントでトークンを確認する。
- ATP の子プロセスはバックエンドマネージャが終了させる。
- 古くなった完了済み結果は、キャッシュ検証によって新しいスナップショットに適用可能だと証明されない限り破棄する。
- アーティファクトのコミットは、アトミックなトランザクションが開始した後はキャンセルしない。

### 診断

診断は、セッションごとの集約器を通る。集約器はレコードを重複除去し、決定的な `DiagnosticId` を割り当て、アーキテクチャが定める順序で整列し、対象の消費者にとって現在のものであるスナップショットの診断だけを公開する。

## 制約と前提

- スケジューラは実行順序を制御するが、フェーズの意味論は制御しない。
- すべてのタスク出力は、公開後は不変である。
- キャッシュヒットは、アーティファクトとインクリメンタルビルドのアーキテクチャが定める検証規則を満たさなければならない。
- 診断とアーティファクトは、タスクの完了順ではなく正準順序で整列する。
- カーネル証明書の検査は、信頼された証明受理の境界であり続ける。
- LSP ビルドは開いているファイルを優先してよいが、互換性のないスナップショットの診断を混在させてはならない。
- ドキュメント生成と抽出は、検証済みアーティファクトが利用可能になった後にスケジュールされる。
