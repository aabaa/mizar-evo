## 目的

この文書は、Mizar Evo における内部設計層の範囲を定義する。

内部設計文書は、アーキテクチャ文書を、実装に近いサブシステム、API、データ構造、実行契約へと詳細化する。アーキテクチャ文書より具体的でありながら、モジュールと1対1で対応する仕様よりは広い粒度を保つ。

## 責務

内部設計層は、アーキテクチャ上の境界から実装サービスへの対応付けを担う。

| 関心事 | アーキテクチャが定義するもの | 内部設計が定義するもの |
|---|---|---|
| パイプラインのフェーズ | 意味的なフェーズ境界と順序 | フェーズサービスの API、ドライバの状態、タスク依存関係 |
| IR の層 | 所有関係と消費者 | スナップショットハンドル、ストレージのライフタイム、同一性規則 |
| スケジューリング | 許容される並列性と正準順序 | タスクグラフの表現、キュー、キャンセルトークン、イベントストリーム |
| アーティファクトとキャッシュ | 公開境界と再利用規則 | ストア API、キャッシュキー構築器、マニフェストトランザクションのプロトコル |
| 診断と LSP | 安定したレコードとスナップショットモデル | 診断の集約、説明ハンドル、LSP ブリッジサービス |
| ATP とカーネル | 信頼境界とバックエンドプロトコル | ポートフォリオタスク、証明ウィットネスの流れ、カーネル検査の統合 |
| ドキュメント | アーティファクトベースの抽出境界 | ドキュメント抽出パス、レンダリング入力、生成出力のマニフェスト |

### メモリモデル（横断的）

メモリのスケーラビリティは、単一のサブシステムではなく複数の内部サブシステムによって実現される横断的な関心事である。常駐集合戦略——インターフェイスと index は常駐させ、証明本体・トレース・witness は遅延的に読み込み、global な index を import closure ごとに複製しない——は、spec [§12.6.3](../../../spec/ja/12.modules_and_namespaces.md#1263-メモリモデル) と [§23.7.9](../../../spec/ja/23.package_management_and_build_system.md#2379-メモリ設計原則) で規範的に、また[アーキテクチャ README](../../architecture/ja/README.md#メモリモデル) で横断的関心事として述べている。内部層はこれを次のように実装する。

| 常駐集合のルール | 内部の機構 | 文書 |
|---|---|---|
| import したインターフェイスを保持し、本体は保持しない | import した public インターフェイス（`ModuleSummary`、公開された `*.mizir.json`）は manifest リーダー経由で読み、`interface_hash` で鍵付けする。証明本体と witness は外部に置く | [02.artifact_store_cache_key_and_manifest.md](./02.artifact_store_cache_key_and_manifest.md) |
| active な作業のみ常駐させ、残りは退避・回収する | `IrStorageService` は大きな出力をコンテンツアドレス方式の blob として保存し、参照されない出力を `retain`／`collect` で破棄する | [06.ir_storage_and_snapshot_handles.md](./06.ir_storage_and_snapshot_handles.md) |
| global な index を closure ごとに複製しない | キャッシュストアにおける `cluster-db/` index と import スコープのビュー | [02.artifact_store_cache_key_and_manifest.md](./02.artifact_store_cache_key_and_manifest.md) |
| witness とトレースを外部化する | hash で参照される proof-witness file。キャッシュ blob とトレースは意味論を変えずに削除できる | [02.artifact_store_cache_key_and_manifest.md](./02.artifact_store_cache_key_and_manifest.md) |
| 進行中の discharge を有界化する | VC ごとの portfolio タスクとバックエンドのメモリ上限。候補となる evidence は受理されるまで一時 blob として保持する | [04.atp_portfolio_and_kernel_check_integration.md](./04.atp_portfolio_and_kernel_check_integration.md) |
| 説明を遅延的に供給する | `ExplanationStore` が遅延ハンドルと有界プレビューを必要時に解決し、一括ロードしない | [03.diagnostics_model_and_lsp_bridge.md](./03.diagnostics_model_and_lsp_bridge.md) |

これは定性的な常駐集合の規律であり、性能保証ではない。具体的な予算とベンチマーク指標は[テスト戦略](../../architecture/ja/20.test_strategy.md#性能とメモリの評価)に置かれる。

## 内部サブシステムの境界

設計上の明確さのため、ここで想定するサブシステム crate を挙げる。最終的な crate 構成は変わってもよいが、責務の境界は安定させる。

| サブシステム | 主な責務 |
|---|---|
| `mizar-driver` | ビルド要求、パイプラインの統括、フェーズサービスのレジストリ |
| `mizar-build` | ワークスペースの計画立案、タスクグラフスケジューラ、リソース予算、キャンセル |
| `mizar-session` | バッチ・ウォッチ・LSP の各モードが共有する、不変のソース／ビルドスナップショット |
| `mizar-lexer` | ソース前処理ヘルパー、生スキャン、インポート事前走査、トークン曖昧性解消 |
| `mizar-syntax` | ソースの形を保った構文構造、trivia、パーサー回復マーカー |
| `mizar-parser` | 文法実装と `TokenStream` から `SurfaceAst` への構文解析 |
| `mizar-frontend` | ソース読み込みと、レキサ／パーサーサービスをまたぐフェーズ 1-3 の統制 |
| `mizar-cache` | キャッシュキーの構築、キャッシュレコードの参照、互換性チェック |
| `mizar-artifact` | 検証済みアーティファクトの射影、証明ウィットネスの参照、マニフェストトランザクション |
| `mizar-diagnostics` | 診断レジストリ、集約、整列、レンダリング、説明ハンドル |
| `mizar-lsp` | エディタプロトコルのブリッジと現在バッファの鮮度管理 |
| `mizar-atp` | ATP 問題のエンコード、バックエンドプロセスの実行、ポートフォリオ候補の収集 |
| `mizar-kernel` | 証明書の検査と、信頼された証明ステータスの判定 |
| `mizar-proof` | 証明ポリシーの評価、証明ステータスの射影、ウィットネスの選択 |
| `mizar-doc` | 検証済みアーティファクトからのドキュメントレンダリングと抽出 |
| `mizar-test` | 合否テスト、スナップショット、ファズ／プロパティハーネス、健全性回帰 |

## 制御フロー

内部設計は [パイプライン概要](../../architecture/ja/00.pipeline_overview.md) と同じフェーズ番号を用いる。

1. ドライバが CLI、ウォッチ、LSP のいずれかから `BuildRequest` を受け取る。
2. フェーズ 0 が `BuildPlan` と依存グラフを生成する。
3. スケジューラが計画をバージョン付きタスクへ展開する。
4. フェーズサービスが、不変の出力、診断、キャッシュ可能なレコードを生成する。
5. 証明関連のタスクは、同じスケジューラを通じて ATP バックエンドとカーネル証明書検査をディスパッチする。
6. アーティファクトのコミットは、マニフェストトランザクションを通じて検証済み出力をシリアライズする。
7. LSP やドキュメントツールは、コンパイラ内部の IR ではなく、公開済みのスナップショットとアーティファクトを参照する。

## 制約と前提

- 内部設計は、アーキテクチャのフェーズ境界と信頼境界をすべて保持しなければならない。
- 内部 API は進化してよいが、公開アーティファクトのスキーマと診断コードには互換性ポリシーが必要である。
- キャッシュヒットは最適化の結果であって、証明の根拠ではない。
- 並列実行は、診断の順序、アーティファクトの順序、証明の受理、再現性を変えてはならない。
- LSP のワークフローはレイテンシを優先してよいが、意味的な基準はあくまでバッチ検証である。
