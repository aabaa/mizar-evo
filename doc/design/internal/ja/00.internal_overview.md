## 目的

この文書は Mizar Evo の internal design layer の scope を定義する。

Internal design documents は architecture documents を implementation-facing subsystems、APIs、data structures、execution contracts へ詳細化する。Architecture documents より具体的だが、1:1 の module specifications よりは広い粒度を保つ。

## Responsibilities

Internal design layer は、architectural boundaries から implementation services への mapping を所有する。

| Concern | Architecture Defines | Internal Design Defines |
|---|---|---|
| Pipeline phases | semantic phase boundaries and ordering | phase service APIs、driver state、task dependencies |
| IR layers | ownership and consumers | snapshot handles、storage lifetime、identity rules |
| Scheduling | allowed parallelism and canonical ordering | task graph representation、queues、cancellation tokens、event stream |
| Artifacts and cache | publication boundary and reuse rules | store APIs、cache key builders、manifest transaction protocol |
| Diagnostics and LSP | stable records and snapshot model | diagnostic aggregation、explanation handles、LSP bridge services |
| ATP and kernel | trust boundary and backend protocol | portfolio tasks、proof witness flow、kernel check integration |
| Documentation | artifact-based extraction boundary | doc extraction passes、render inputs、generated output manifests |

## Internal Subsystem Boundaries

ここでは設計上の明確さのため expected subsystem crates を名付ける。最終的な crate layout は変わってよいが、責務境界は安定させる。

| Subsystem | Primary Responsibility |
|---|---|
| `mizar-driver` | build requests、pipeline orchestration、phase service registry |
| `mizar-build` | workspace planning、task graph scheduler、resource budgets、cancellation |
| `mizar-session` | batch、watch、LSP modes が共有する immutable source and build snapshots |
| `mizar-cache` | cache key construction、cache record lookup、compatibility checks |
| `mizar-artifact` | verified artifact projection、proof witness references、manifest transactions |
| `mizar-diagnostics` | diagnostic registry、aggregation、sorting、rendering、explanation handles |
| `mizar-lsp` | editor protocol bridge and current-buffer freshness handling |
| `mizar-atp` | ATP problem encoding、backend process execution、portfolio candidate collection |
| `mizar-kernel` | proof certificate checking and trusted proof status |
| `mizar-proof` | proof policy evaluation、proof status projection、witness selection |
| `mizar-doc` | verified artifacts からの documentation rendering and extraction |
| `mizar-test` | pass/fail tests、snapshots、fuzz/property harnesses、soundness regressions |

## Control Flow

Internal designs は [Pipeline Overview](../../architecture/ja/00.pipeline_overview.md) と同じ phase numbering を使う。

1. Driver は CLI、watch、LSP から `BuildRequest` を受け取る。
2. Phase 0 が `BuildPlan` and dependency graph を生成する。
3. Scheduler が plan を versioned tasks へ展開する。
4. Phase services が immutable outputs、diagnostics、cacheable records を生成する。
5. Proof-related tasks は同じ scheduler を通して ATP backends and kernel certificate checks を dispatch する。
6. Artifact commit は manifest transactions を通して verified outputs を serialize する。
7. LSP and documentation tools は compiler-internal IR ではなく published snapshots and artifacts を読む。

## Constraints and Assumptions

- Internal design はすべての architecture phase boundaries and trust boundaries を保持しなければならない。
- Internal APIs は進化してよいが、published artifact schemas and diagnostic codes には compatibility policy が必要である。
- Cache hits は optimization results であり proof authority ではない。
- Parallel execution は diagnostic order、artifact order、proof acceptance、reproducibility を変えてはならない。
- LSP workflows は latency を優先してよいが、batch verification が semantic baseline である。
