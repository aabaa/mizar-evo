# 内部設計: ドキュメント抽出

> 正本は英語です。英語版: [../en/05.documentation_extraction.md](../en/05.documentation_extraction.md)。

## 目的

この文書は、フェーズ 16 のドキュメント生成とコード抽出の内部設計を定義する。

ここでは、フェーズ 16 のアーキテクチャを、スケジューラタスク、アーティファクト読み取り、ドキュメント入力モデル、レンダリングインデックス、ランタイム抽出の射影、出力マニフェスト、診断、キャッシュ境界へと詳細化する。フェーズ 16 はバッチビルドグラフの一部だが、ドキュメントレンダリングとコード抽出は任意のタスク種別である。これらは検証済みアーティファクトとソースバンドルを消費するが、証明の妥当性を確立することはなく、フェーズ 0〜15 の結果を変更することもない。

## 責務

### ドキュメント抽出サービス

`DocExtractionService` は、スケジューラに登録されるフェーズ 16 のフェーズサービスである。

担うもの:

- ビルドプロファイルに応じて、1 つの `DocumentationExtract` スケジューラタスクを、ドキュメントと抽出のサブタスクへ展開する。
- 必要な検証済みアーティファクトが、パッケージのアーティファクトマニフェストにコミット済みであることを検証する。
- ドキュメント出力と抽出出力のキャッシュキーを選ぶ。
- ドキュメントの診断と抽出の診断を、ビルドの診断シンクへ流す。
- 出力マニフェストをドライバへ報告する。

証明の受理、`*.mizir.json` のアーティファクトコミット、ソースレベルの意味的な判断は担わない。

### アーティファクトドキュメント読み取り

`ArtifactDocReader` は、フェーズ 16 における安定した入力の読み込みを担う。

読むもの:

- `artifact-manifest.json`
- 公開された `*.mizir.json` 検証済みアーティファクト
- パッケージマニフェストのメタデータ
- ドキュメントコメントやスニペットのためのソースファイル、またはパッケージ化されたソースバンドル
- キャッシュキー検証後の、任意の実装キャッシュ

コンパイラ内部の IR より、安定したアーティファクトを優先しなければならない。ソースファイルが利用できない場合でも、ドキュメントはソースリンクの診断を付けて意味的なページをレンダリングしてよい。抽出は、必要なソースまたはソースマップが欠けた項目については失敗しなければならない。

### ドキュメントビルダー

`DocumentationBuilder` は、ドキュメント入力の正規化、コメントの付与、リンク解決、数式レンダリング、HTML レンダリング、検索インデックスの生成、ステージングされたレンダリング出力を担う。

責務:

- パッケージメタデータと検証済みモジュールアーティファクトから `DocBuildPlan` を構築する。
- `:::` コメントを、モジュールと項目の同一性に付与する。
- インライン参照、Markdown リンク、`@see` タグ、節参照を解決する。
- `@latex` メタデータと既定の数式レンダリングを適用する。
- 決定的なページパス、アンカー、検索インデックス、任意の機械可読ドキュメントインデックスを出力する。

ドキュメントの警告は、現在の公開プロファイルがそれらをエラーへ格上げしない限り、証明結果を無効にすることはない。

### 抽出ビルダー

`ExtractionBuilder` は、実行可能項目の選択、`RuntimeIr` への射影、ゴースト／証明の消去、ターゲットへのローワリング、生成ソースマップ、抽出マニフェストの出力を担う。

責務:

- エクスポート、抽出設定、または CLI で選択されたルートから、抽出可能なアルゴリズムを選ぶ。
- 必要な義務について、証明ステータスと抽出ポリシーを検証する。
- 検証済みの実行可能アルゴリズムデータを、ターゲット中立な `RuntimeIr` へ射影する。
- ゴースト状態、証明専用のアサーション、アノテーション、証明ヒント、定理／証明の構造を消去する。
- 非対応のランタイム構成要素を明示的に拒否する。
- 決定的なコード生成のためにターゲットバックエンドを呼び出す。

抽出は追加の性質を証明しない。フェーズ 14 の証明受理と、フェーズ 15 のアーティファクト整合性を信頼する。

### 出力パブリッシャー

`Phase16OutputPublisher` は、生成されたドキュメントと抽出されたコードのアトミックな書き込みを担う。

書き込み先は、フェーズ 15 の検証済みアーティファクトトランザクションではなく、フェーズ 16 の出力ディレクトリである。ただし、一時ファイル、必要に応じたディレクトリ単位のステージング、コンテンツハッシュ、アトミックなリネームを用い、読み手が書き込み途中のインデックスや生成コードを観測しないようにしなければならない。

## データモデル

### フェーズ 16 の要求

```rust
struct DocumentationExtractInput {
    package_id: PackageId,
    package_root: PathBuf,
    artifact_manifest: ArtifactManifestRef,
    source_bundle: Option<SourceBundleRef>,
    doc_config: Option<DocConfig>,
    extraction_config: Option<ExtractionConfig>,
    profile: Phase16Profile,
    snapshot: BuildSnapshotId,
}

struct Phase16Profile {
    render_docs: bool,
    extract_code: bool,
    fail_on_doc_warnings: bool,
    fail_on_extraction_warnings: bool,
    include_private_docs: bool,
}
```

`render_docs` と `extract_code` は独立したスイッチである。検証のみのビルドは、フェーズ 0〜15 を変えずに両方を無効化する。

`source_bundle` が任意なのは、パッケージ化されたソースがなくても公開アーティファクトが利用できる場合があるためである。その場合でも、ドキュメントは、利用できないコメント、スニペット、ソースリンクについて診断を出しつつ、意味的なページをレンダリングできる。抽出は、選択された項目に必要なソースマップと実行可能射影を要求し、必要なソースデータが欠けた抽出項目はブロックする。

### アーティファクト入力

```rust
struct Phase16ArtifactSet {
    manifest_hash: Hash,
    package_metadata: PackageMetadata,
    modules: Vec<VerifiedModuleInput>,
    dependency_indexes: Vec<DependencyDocIndexRef>,
}

struct VerifiedModuleInput {
    module_path: ModulePath,
    artifact_ref: VerifiedArtifactRef,
    artifact_hash: Hash,
    interface_hash: Hash,
    implementation_hash: Hash,
    source_ref: Option<SourceFileRef>,
    exports: Vec<ExportedItem>,
    proof_summaries: Vec<ProofSummary>,
}
```

ドキュメントは、検証済みアーティファクトの `exports`、意味的なシグネチャ、定理の言明、ラベル、ソース範囲、証明サマリ、`@latex` の射影を用いる。抽出は、検証済みの実行可能アルゴリズム射影とソースマップを用いる。

### ドキュメント入力

```rust
struct DocBuildPlan {
    package_id: PackageId,
    package_version: Option<Version>,
    modules: Vec<DocModuleInput>,
    dependency_indexes: Vec<DocIndexRef>,

}

struct DocModuleInput {
    module_path: ModulePath,
    artifact: VerifiedArtifactRef,
    source: Option<SourceFileRef>,

struct DocCommentBlock {

    raw_markdown: String,

```

`DocAttachmentCandidate` は、ソース上の隣接関係と、検証器が提供するモジュール／項目の同一性の両方を記録する。付与が成功するのは、その両方が同じ検証済み項目を指す場合だけである。

アンカーは、レンダリングされた記法ではなく、エクスポートされたラベルと FQN から導出する。

### 抽出入力とランタイム IR

`RuntimeIr` は、ターゲット中立な抽出の射影である。証明エビデンスではなく、根拠として検証器へ戻してはならない。

### フェーズ 16 の出力マニフェスト

```rust
struct Phase16OutputManifest {
    schema_version: SchemaVersion,
    package_id: PackageId,
    artifact_manifest_hash: Hash,
    doc_output: Option<DocOutputManifest>,
    extraction_output: Option<ExtractionManifest>,
    diagnostics_hash: Option<Hash>,

    output_dir: PathBuf,
    index_file: PathBuf,
    search_index: PathBuf,
    content_hash: Hash,
    options_hash: Hash,
}

    target: ExtractionTarget,
    output_dir: PathBuf,
    items: Vec<ExtractedItem>,
    content_hash: Hash,
    options_hash: Hash,

```

フェーズ 16 の出力マニフェストは、生成された出力の追跡用メタデータである。パッケージの検証アーティファクトではなく、`artifact-manifest.json` を置き換えることもない。

フェーズ 16 の出力マニフェストに記録するパスは、既定でパッケージ相対である。ワークスペース相対のパスは、パッケージをまたぐ生成出力に限って用いてよい。絶対パスはローカルな診断に限定し、再現可能な出力ハッシュに加えてはならない。

## 制御フロー

### スケジューリング

1. スケジューラは、フェーズ 15 が検証済みアーティファクトをコミットした後、`DocumentationExtract` タスクを作成する。
2. ドライバは、ビルドプロファイルがドキュメントレンダリングまたはコード抽出を有効にしている場合にのみ、フェーズ 16 タスクを含める。
3. タスクは、パッケージのアーティファクトマニフェストと、選択されたパッケージに必要なすべてのモジュールアーティファクトコミットに依存する。
4. キャッシュキーは、有効なサブタスクごとに、アーティファクトマニフェストハッシュ、そのサブタスクに必要なソースハッシュ、ドキュメントオプションまたは抽出ターゲットのオプション、ツールチェインバージョン、スキーマバージョンから構築する。
5. キャッシュヒットは出力マニフェストと診断を返し、キャッシュミスは要求されたビルダーを実行する。

フェーズ 16 タスクは、パッケージをまたいで並列に実行してよい。1 つのパッケージ内では、ドキュメントのリンク解決はパッケージ全体のインデックスを必要とし、抽出のローワリングは、抽出計画が確定した後に選択された項目ごとに実行してよい。

### ドキュメントパイプライン

1. `ArtifactDocReader` が、アーティファクト集合とソースのコメントブロックを読み込む。
2. `DocumentationBuilder` が、エクスポートと依存のドキュメントインデックスから、パッケージレベルのシンボル・ラベルインデックスを構築する。
3. コメントを、検証済みのモジュール／項目の同一性に付与する。
4. Markdown を、ソース範囲を保持したまま解析する。
5. インラインコードリンク、明示的な Markdown リンク、`@see` タグ、節参照を解決する。
6. 数式レンダリングのメタデータを、`@latex` と既定の構文レンダリングから適用する。
7. ページ、検索インデックス、相互参照インデックス、被参照インデックス、任意の機械可読インデックスをレンダリングする。
8. `Phase16OutputPublisher` が、ステージングされたサイトをアトミックに書き込み、ドキュメント出力マニフェストを記録する。

未解決または曖昧なリンクは診断を生成し、安全にテキストとしてレンダリングされる。

### 抽出パイプライン

1. `ArtifactDocReader` が、検証済みの実行可能アルゴリズム射影と依存サマリを読み込む。
2. `ExtractionBuilder` が、エクスポート、抽出設定、CLI 要求からルートを選ぶ。
3. 各候補を `Extractable`、`SpecOnly`、`BlockedByProof`、`BlockedByTarget`、`BlockedByDependency` のいずれかに分類する。
4. 抽出可能な候補を `RuntimeIr` へ射影する。
5. ゴースト状態、証明専用の構造、アノテーション、証明ヒント、仕様専用のスナップショットを消去する。
6. ターゲットバックエンドが、`RuntimeIr` を生成ファイルと逆方向ソースマップへローワリングする。
7. `Phase16OutputPublisher` が、生成ファイルと `ExtractionManifest` をアトミックに書き込む。

非対応のランタイム構成要素は、選択されたプロファイルが要求されたすべてのルートの抽出を要求しない限り、影響を受ける抽出項目だけをブロックする。

### 診断

ドキュメントと抽出の診断は、フェーズ 0〜15 と同じ `Diagnostic` モデルを用いるが、そのフェーズ系統は `Documentation` または `Extraction` である。

規則:

- ドキュメントコメントの診断は、利用可能であれば保持された `:::` のソース範囲を指す。
- 生成出力の診断は、両方が存在する場合、ソース範囲と生成ファイル参照を含む。
- ドキュメントコメントのソース欠落は、既定では警告である。
- 古い、または互換性のない検証済みアーティファクトは、フェーズ 16 が意味論を確立できないため、エラーである。
- 公開プロファイルは、ドキュメントの警告や抽出の警告をエラーへ格上げしてよい。

### キャッシュと再生成

フェーズ 16 の出力は、再現可能な派生成果物である。

キャッシュ再利用に必要なもの:

- 同じパッケージ ID とアーティファクトマニフェストハッシュ。
- コメント、スニペット、ソースマップについての同じソースハッシュ。
- リンク解決が依存を用いる場合、同じ依存ドキュメントインデックスのハッシュ。
- 同じドキュメントオプションと抽出ターゲットのオプション。
- 互換性のあるフェーズ 16 スキーマバージョンと、レンダラ／バックエンドのバージョン。

フェーズ 16 の出力やキャッシュレコードを削除しても、検証結果が変わってはならない。これらは、検証済みアーティファクトとソースバンドルから再生成できる。

## API 概要

### ドキュメント抽出サービス

```rust
trait DocExtractionService {
    fn plan(&self, input: DocumentationExtractInput) -> Result<Phase16Plan, Phase16Error>;

```

## エラー処理

### ドキュメントエラー

| 条件 | 処理 |
|---|---|
| 未解決のリンク | 既定では警告とし、リンクなしのテキストとしてレンダリングする |
| 曖昧なラベル | 警告とし、FQN または明示的な参照を要求する |
| 不正な Markdown | 警告とし、縮退したレンダリングを行う |
| 不正な LaTeX | 警告またはレンダリングマーカー |
| ドキュメントコメントの欠落 | 任意の lint |
| ソースバンドルの欠落 | ドキュメントについては警告とし、ソーススニペットを省く |
| アーティファクトマニフェストの陳腐化 | エラーとし、意味的なドキュメントをレンダリングしない |

### 抽出エラー

| 条件 | 処理 |
|---|---|
| 未解消または許可されない証明ステータス | `BlockedByProof`。項目を出力しない |
| 非対応のランタイム構成要素 | `BlockedByTarget`。項目を出力しない |
| ゴースト値がランタイムへ漏れ出す | 抽出エラー。射影の不正、またはより前段のチェッカーのバグを示す |
| 依存の対応付けの欠落 | `BlockedByDependency`。項目を出力しない |
| バックエンド出力の失敗 | 当該ターゲットについての抽出エラー |
| 部分的な出力の書き込み | ステージングディレクトリを破棄し、以前の出力を可視のまま保つ |

### キャンセル

キャンセルは、同じスナップショットとキャッシュキーに対して既にアトミックに公開されたものを除き、ステージングされたフェーズ 16 出力を破棄する。キャンセルされたフェーズ 16 処理は、検証済みアーティファクトをロールバックしてはならない。

## 制約と前提

- フェーズ 16 は、意味的な出力のために検証済みアーティファクトを必要とする。
- ドキュメントと抽出は、任意のバッチグラフタスクである。
- フェーズ 16 は、`*.mizir.json`、証明ウィットネス、`artifact-manifest.json` を変更してはならない。
- ドキュメントは、表示上のエラーに対して緩やかに縮退してよい。
- 抽出は、非対応のランタイム構成要素を明示的に拒否しなければならない。
- ゴースト状態と証明専用のアノテーションは、抽出されたランタイムコードに現れてはならない。
- 生成出力は、同じアーティファクトマニフェスト、ソースハッシュ、オプション、ターゲット、ツールチェインバージョンに対して決定的である。
- フェーズ 16 の出力は派生成果物であり、証明の妥当性を変えずに削除・再生成できる。
