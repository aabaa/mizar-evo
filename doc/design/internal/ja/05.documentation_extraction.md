# Internal Design: Documentation Extraction

> Canonical language: English. English canonical version: [../en/05.documentation_extraction.md](../en/05.documentation_extraction.md).

## 目的

この文書は、phase 16 documentation generation and code extraction の内部設計を定義する。

Phase 16 architecture を scheduler tasks、artifact readers、documentation input models、render indexes、runtime extraction projections、output manifests、diagnostics、cache boundaries へ詳細化する。Phase 16 は batch build graph の一部だが、documentation rendering と code extraction は optional task classes である。Verified artifacts and source bundles を consume し、proof validity を確立せず、phase 0-15 results を変更しない。

## Responsibilities

### Doc Extraction Service

`DocExtractionService` は scheduler に登録される phase 16 phase service である。

所有するもの:

- build profile に応じて一つの `DocumentationExtract` scheduler task を documentation and extraction subtasks へ展開する
- 必要な verified artifacts が package artifact manifest に commit 済みであることを検証する
- documentation and extraction outputs の cache keys を選ぶ
- documentation diagnostics and extraction diagnostics を build diagnostic sink へ流す
- output manifests を driver に報告する

Proof acceptance、`*.mizir.json` の artifact commit、source-level semantic decisions は所有しない。

### Artifact Documentation Reader

`ArtifactDocReader` は phase 16 の stable input loading を所有する。

読むもの:

- `artifact-manifest.json`
- published `*.mizir.json` verified artifacts
- package manifest metadata
- documentation comments and snippets 用の source files or packaged source bundles
- cache-key validation 後の optional implementation caches

Compiler-internal IR より stable artifacts を優先しなければならない。Source file が利用できない場合でも、documentation は source-link diagnostics 付きで semantic pages を render してよい。Extraction は required source or source map が欠けた items について fail しなければならない。

### Documentation Builder

`DocumentationBuilder` は documentation input normalization、comment attachment、link resolution、math rendering、HTML rendering、search index generation、staged render outputs を所有する。

責務:

- package metadata and verified module artifacts から `DocBuildPlan` を構築する
- `:::` comments を module and item identities に attach する
- inline references、Markdown links、`@see` tags、section references を resolve する
- `@latex` metadata and default formula rendering を適用する
- deterministic page paths、anchors、search indexes、optional machine-readable doc indexes を emit する

Active publication profile が errors に promote しない限り、documentation warnings は proof results を invalid にしない。

### Extraction Builder

`ExtractionBuilder` は executable item selection、`RuntimeIr` projection、ghost/proof erasure、target lowering、generated source maps、extraction manifest emission を所有する。

責務:

- exports、extraction configuration、CLI-selected roots から extractable algorithms を選ぶ
- required obligations の proof status and extraction policy を検証する
- verified executable algorithm data を target-neutral `RuntimeIr` へ project する
- ghost state、proof-only assertions、annotations、proof hints、theorem/proof structures を erase する
- unsupported runtime constructs を明示的に reject する
- deterministic code generation のため target backends を呼び出す

Extraction は追加の properties を証明しない。Phase 14 proof acceptance and phase 15 artifact integrity を信頼する。

### Output Publisher

`Phase16OutputPublisher` は generated documentation and extracted code の atomic writes を所有する。

Phase 15 verified artifact transaction ではなく、phase 16 output directories へ書く。ただし temporary files、必要に応じた complete-directory staging、content hashes、atomic rename を使い、readers が partially-written indexes or generated code を観測しないようにしなければならない。

## Data Model

### Phase 16 Request

```rust
struct DocumentationExtractInput {
    package_id: PackageId,
    package_root: PathBuf,
    artifact_manifest: ArtifactManifestRef,
    source_bundle: Option<SourceBundleRef>,
    doc_config: Option<DocConfig>,
    extraction_config: Option<ExtractionConfig>,
    profile: Phase16Profile,
    snapshot: BuildSnapshotId,
}

struct Phase16Profile {
    render_docs: bool,
    extract_code: bool,
    fail_on_doc_warnings: bool,
    fail_on_extraction_warnings: bool,
    include_private_docs: bool,
}
```

`render_docs` and `extract_code` は独立した switch である。Verification-only build は phases 0-15 を変えずに両方を disable する。

`source_bundle` は optional である。Published artifacts は packaged source なしで利用可能な場合があるためである。その場合でも documentation は unavailable comments、snippets、source links について diagnostics を出しつつ semantic pages を render できる。Extraction は selected items に必要な source maps and executable projections を要求し、必要な source data が欠けた extraction items は block する。

### Artifact Inputs

```rust
struct Phase16ArtifactSet {
    manifest_hash: Hash,
    package_metadata: PackageMetadata,
    modules: Vec<VerifiedModuleInput>,
    dependency_indexes: Vec<DependencyDocIndexRef>,
}

struct VerifiedModuleInput {
    module_path: ModulePath,
    artifact_ref: VerifiedArtifactRef,
    artifact_hash: Hash,
    interface_hash: Hash,
    implementation_hash: Hash,
    source_ref: Option<SourceFileRef>,
    exports: Vec<ExportedItem>,
    proof_summaries: Vec<ProofSummary>,
}
```

Documentation は verified artifacts の `exports`、semantic signatures、theorem statements、labels、source ranges、proof summaries、`@latex` projections を使う。Extraction は verified executable algorithm projections and source maps を使う。

### Documentation Inputs

```rust
struct DocBuildPlan {

```

`DocAttachmentCandidate` は source adjacency と verifier-provided module/item identity の両方を記録する。Attachment は両方が同じ verified item を指す場合にのみ成功する。

### Extraction Inputs and Runtime IR

```rust
struct ExtractionPlan {
    package_id: PackageId,
    artifact_manifest_hash: Hash,
    target: ExtractionTarget,
    roots: Vec<ExtractionRoot>,
    options: ExtractionOptions,
}

struct ExtractableItem {
    symbol: SymbolId,
    module_path: ModulePath,
    source_range: SourceRange,
    obligations: Vec<VcId>,
    proof_statuses: Vec<ProofStatus>,

    algorithm: SymbolId,
    signature: RuntimeSignature,
    blocks: Vec<RuntimeBlock>,
    source_map: RuntimeSourceMap,
    required_runtime: Vec<RuntimeCapability>,

```

`RuntimeIr` は target-neutral extraction projection である。Proof evidence ではなく、verifier に authority として戻してはならない。

### Phase 16 Output Manifest

```rust
struct Phase16OutputManifest {
    schema_version: SchemaVersion,
    package_id: PackageId,
    artifact_manifest_hash: Hash,
    doc_output: Option<DocOutputManifest>,
    extraction_output: Option<ExtractionManifest>,
    diagnostics_hash: Option<Hash>,
}

struct DocOutputManifest {
    output_dir: PathBuf,
    index_file: PathBuf,
    search_index: PathBuf,
    content_hash: Hash,
    options_hash: Hash,
}

struct ExtractionManifest {
    target: ExtractionTarget,
    output_dir: PathBuf,
    items: Vec<ExtractedItem>,
    content_hash: Hash,
    options_hash: Hash,
}
```

Phase 16 output manifest は generated outputs の traceability metadata である。Package verification artifact ではなく、`artifact-manifest.json` を置き換えない。

Phase 16 output manifests に記録する paths は default で package-relative である。Workspace-relative paths は cross-package generated outputs に限って使ってよい。Absolute paths は local diagnostics に限定し、reproducible output hashes に参加してはならない。

## Control Flow

### Scheduling

1. Scheduler は phase 15 が verified artifacts を commit した後、`DocumentationExtract` tasks を作成する。
2. Driver は build profile が documentation rendering または code extraction を enable している場合だけ phase 16 tasks を含める。
3. Task は package artifact manifest と、選択 package に必要なすべての module artifact commits に依存する。
4. Cache keys は enabled subtask ごとに、artifact manifest hash、その subtask に必要な source hashes、documentation options または extraction target options、toolchain versions、schema versions から構築する。
5. Cache hits は output manifests and diagnostics を返し、cache misses は requested builders を実行する。

Phase 16 tasks は packages 間で parallel に実行してよい。一つの package 内では documentation link resolution は package-wide index を必要とし、extraction lowering は extraction plan が固定された後 selected item ごとに実行してよい。

### Documentation Pipeline

1. `ArtifactDocReader` は artifact set and source comment blocks を load する。
2. `DocumentationBuilder` は exports and dependency doc indexes から package-level symbol and label index を構築する。
3. Comments を verified module/item identities に attach する。
4. Markdown を source-range preservation 付きで parse する。
5. Inline code links、explicit Markdown links、`@see` tags、section references を resolve する。
6. Formula rendering metadata を `@latex` and default syntax rendering から適用する。
7. Pages、search index、cross-reference index、used-by index、optional machine-readable index を render する。
8. `Phase16OutputPublisher` は staged site を atomically に書き、doc output manifest を記録する。

Unresolved or ambiguous links は diagnostics を生成し、安全に text として render する。

### Extraction Pipeline

1. `ArtifactDocReader` は verified executable algorithm projections and dependency summaries を load する。
2. `ExtractionBuilder` は exports、extraction config、CLI request から roots を選ぶ。
3. 各 candidate を `Extractable`、`SpecOnly`、`BlockedByProof`、`BlockedByTarget`、`BlockedByDependency` に classify する。
4. Extractable candidates を `RuntimeIr` へ project する。
5. Ghost state、proof-only structures、annotations、proof hints、specification-only snapshots を erase する。
6. Target backend は `RuntimeIr` を generated files and reverse source maps へ lower する。
7. `Phase16OutputPublisher` は generated files and `ExtractionManifest` を atomically に書く。

Unsupported runtime constructs は、selected profile が all requested roots の extraction を要求しない限り、affected extraction item だけを block する。

### Diagnostics

Documentation and extraction diagnostics は phases 0-15 と同じ `Diagnostic` model を使うが、phase family は `Documentation` または `Extraction` である。

Rules:

- doc comment diagnostics は、利用可能なら保持された `:::` source ranges を指す
- generated-output diagnostics は、両方が存在する場合 source ranges and generated file references を含む
- documentation comments の source missing は default では warning
- stale or incompatible verified artifacts は、phase 16 が semantics を確立できないため error
- publication profiles は documentation warnings or extraction warnings を errors に promote してよい

### Cache and Regeneration

Phase 16 outputs は reproducible derived products である。

Cache reuse に必要なもの:

- same package id and artifact manifest hash
- comments、snippets、source maps 用の same source hashes
- link resolution が dependencies を使う場合、same dependency documentation index hashes
- same documentation options and extraction target options
- compatible phase 16 schema versions and renderer/backend versions

Phase 16 outputs or cache records を削除しても verification results は変わってはならない。Verified artifacts and source bundles から再生成できる。

## Error Handling

### Documentation Errors

| Condition | Handling |
|---|---|
| unresolved link | warning by default; render text without link |
| ambiguous label | warning; require FQN or explicit reference |
| malformed Markdown | warning with degraded rendering |
| invalid LaTeX | warning or rendering marker |
| missing doc comment | optional lint |
| missing source bundle | warning for documentation; omit source snippets |
| stale artifact manifest | error; do not render semantic docs |

### Extraction Errors

| Condition | Handling |
|---|---|
| open or disallowed proof status | `BlockedByProof`; do not emit item |
| unsupported runtime construct | `BlockedByTarget`; do not emit item |
| ghost value escapes into runtime | extraction error; indicates invalid projection or earlier checker bug |
| missing dependency mapping | `BlockedByDependency`; do not emit item |
| backend emission failure | extraction error for the affected target |
| partial output write | discard staged directory and keep previous output visible |

### Cancellation

Cancellation は、同じ snapshot and cache key に対してすでに atomically published されたものを除き、staged phase 16 outputs を discard する。Cancelled phase 16 work は verified artifacts を rollback してはならない。

## Constraints and Assumptions

- Phase 16 は semantic output のため verified artifacts を必要とする。
- Documentation and extraction は optional batch-graph tasks である。
- Phase 16 は `*.mizir.json`、proof witnesses、`artifact-manifest.json` を mutate してはならない。
- Documentation は presentation errors に対して degrade gracefully してよい。
- Extraction は unsupported runtime constructs を明示的に reject しなければならない。
- Ghost state and proof-only annotations は extracted runtime code に現れてはならない。
- Generated outputs は同じ artifact manifest、source hashes、options、target、toolchain versions に対して deterministic である。
- Phase 16 outputs は derived products であり、proof validity を変えずに削除 and regenerate できる。
