# Internal Design: Diagnostics Model and LSP Bridge

> Canonical language: English. English canonical version: [../en/03.diagnostics_model_and_lsp_bridge.md](../en/03.diagnostics_model_and_lsp_bridge.md).

## 目的

この文書は、diagnostics model と、diagnostics、editor metadata、code actions、explanation queries を publish する LSP bridge の internal design を定義する。

これは diagnostics が structured phase outputs であり、LSP server は compiler-internal IR ではなく immutable snapshots and stable artifacts を読む、という architecture rule を詳細化する。

## Responsibilities

### Diagnostic Registry

`DiagnosticRegistry` は stable diagnostic codes を所有する。

提供するもの:

- permanent `DiagnosticCode` allocation and retirement rules
- numeric code から semantic name、default severity、phase family、documentation URL への mapping
- code が異なる意味で reuse されないことを保証する compatibility validation
- CLI rendering、artifact emission、LSP conversion 用の lookup data

Messages は version 間で改善され得る。Tools は message text や semantic name ではなく `DiagnosticCode` を key にする。

### Diagnostic Producer API

Compiler phases は `DiagnosticSink` を通して structured diagnostic drafts を emit する。

Phase services は diagnostics の生成に必要な local facts を所有するが、CLI output や LSP diagnostics を format しない。安全に計算できる場合は structured details、explanation handles、fix suggestions を付与してよい。

### Diagnostic Aggregator

`DiagnosticAggregator` は build snapshot ごとの normalization、identity assignment、deduplication、sorting、publication readiness を所有する。

Phase results、scheduler/build errors、cache/artifact errors から diagnostics を受け取り、CLI、artifacts、LSP snapshots 用の immutable `BuildDiagnosticIndex` values を publish する。

Unsaved editor text について生成される open-buffer diagnostics は LSP overlay diagnostics である。同じ record shape を reuse してよいが、CLI output や `VerifiedArtifact` へは publish しない。

### LSP Bridge

`LspBridge` は editor-facing protocol conversion を所有する。

責務:

- open-buffer source snapshots and document versions
- compiler `SourceRange` values から LSP UTF-16 ranges への変換
- current `LspSnapshot` からの diagnostics publication
- hover、definitions、symbols、semantic tokens、inlay hints、code actions、explanation requests の提供
- compiler driver を通した builds の scheduling

Proof acceptance、type checking、ATP reasoning、artifact mutation は責務ではない。

### Explanation Store

`ExplanationStore` は lazy explanation handles and bounded explanation payloads を所有する。

Large traces は artifacts、cache-backed query data、または dedicated explanation files に残す。通常の diagnostics は compact references and bounded previews だけを保存する。

## Data Model

### Diagnostic Draft

```rust
struct DiagnosticDraft {
    code: DiagnosticCode,
    severity_override: Option<DiagnosticSeverity>,
    phase: PipelinePhase,
    message: String,
    primary_span: SourceRange,
    secondary_spans: Vec<DiagnosticSpan>,
    notes: Vec<DiagnosticNote>,
    details: DiagnosticDetails,
    fixes: Vec<FixSuggestion>,
    explanation: Option<ExplanationRef>,
}
```

Drafts は snapshot-wide identity assignment 前に phase services が emit する。Relevant source snapshot に基づく `SourceRange` values をすでに使っていなければならない。

### Diagnostic Record

```rust
struct Diagnostic {
    handle: DiagnosticHandle,
    code: DiagnosticCode,
    semantic_name: String,
    severity: DiagnosticSeverity,
    phase: PipelinePhase,
    message: String,
    primary_span: SourceRange,
    secondary_spans: Vec<DiagnosticSpan>,
    notes: Vec<DiagnosticNote>,
    details: DiagnosticDetails,
    fixes: Vec<FixSuggestion>,
    explanation: Option<ExplanationRef>,
    related_ids: Vec<DiagnosticId>,
    freshness: DiagnosticFreshness,
}

struct DiagnosticHandle {
    snapshot: BuildSnapshotId,
    id: DiagnosticId,
}
```

`DiagnosticId` は一つの `BuildSnapshotId` 内で deterministic である。Deduplication 後、source identity、diagnostic code、phase、primary span、stable per-span ordinal から導出する。Tools は `DiagnosticHandle` を通して解釈しなければならない。Bare `DiagnosticId` は snapshot の外では意味を持たない。

### Diagnostic Index

```rust
struct BuildDiagnosticIndex {
    snapshot: BuildSnapshotId,
    by_source: BTreeMap<SourceId, Vec<Diagnostic>>,
    by_id: BTreeMap<DiagnosticId, DiagnosticRef>,
    phase_status: BTreeMap<WorkUnit, PhaseStatus>,
}

struct LspDiagnosticOverlay {
    document: DocumentUri,
    version: SourceVersion,
    diagnostics: Vec<Diagnostic>,
}
```

Build index は publication 後 immutable である。CLI rendering、artifact projection、LSP semantic diagnostics はすべて同じ index を読む。LSP overlay diagnostics は editor-current additions であり、published artifacts へ project しない。

### Fix Suggestion

```rust
struct FixSuggestion {
    title: String,
    confidence: FixConfidence,
    edits: Vec<TextEdit>,
    command: Option<DiagnosticCommand>,
    applicability: Applicability,
    required_snapshot: Option<BuildSnapshotId>,
    required_text_hash: Option<Hash>,
}
```

Source text に依存する edit は、適用前に target range and expected text を検証しなければならない。Artifact-assisted fixes は artifact hash and source hash も検証する。

### Explanation Reference

```rust
struct ExplanationRef {
    kind: ExplanationKind,
    diagnostic: Option<DiagnosticHandle>,
    expr_id: Option<ExprId>,
    vc_id: Option<VcId>,
    source: ExplanationSource,
    summary_hash: Hash,
    preview: Option<String>,
}

enum ExplanationSource {
    Artifact(PathBuf),
    CacheRecord(CacheKey),
    QueryService(ExplanationQuery),
}
```

Explanation references は handles である。作成元の snapshot または artifact hash の外では invalid である。`CacheRecord` sources は local explanation queries の optional acceleration または backing data である。Published diagnostics は cache records なしで解釈できなければならず、cache-backed explanations は proof status を決定しない。

### LSP Snapshot

```rust
struct LspSnapshot {
    snapshot_id: BuildSnapshotId,
    artifact_manifest: ArtifactManifestRef,
    source_versions: Vec<SourceVersion>,
    diagnostics: BuildDiagnosticIndex,
    overlays: Vec<LspDiagnosticOverlay>,
    expression_index: ExpressionIndex,
    symbol_index: SymbolIndex,
    explanation_index: ExplanationIndex,
    missing_facts: Option<MissingFactsRef>,
    freshness: SnapshotFreshness,
}

enum SnapshotFreshness {
    Current,
    StaleArtifact,
    Pending,
}
```

Snapshot は request handlers へ atomically swapped される。In-flight requests は開始時の snapshot 上で完了する。

## Control Flow

### Phase Diagnostic Production

1. Phase service は `DiagnosticSink` を持つ `PhaseContext` を受け取る。
2. Phase は source ranges and structured details を持つ `DiagnosticDraft` values を emit する。
3. Phase は `PhaseResult` の中で drafts を返す。
4. Scheduler は phase diagnostics and phase status を aggregator へ forward する。
5. Aggregator は target consumer 用の coherent snapshot boundary ができるまで diagnostics を publish しない。

Fatal and blocking phase statuses は diagnostic severity とは別に記録される。Error diagnostic であっても recoverable な場合があり、phase は upstream dependency failure により blocked になる場合がある。

### Build Aggregation

1. Paths を workspace-relative `SourceId`s へ normalize する。
2. Source snapshot が target publication と一致しない obsolete diagnostics を drop または mark する。
3. Same code、phase、primary span、message identity、details hash、fix identity を持つ diagnostics を deduplicate する。
4. Deterministic `DiagnosticId`s を割り当てる。
5. Source range、severity、phase order、diagnostic code、stable ordinal で sort する。
6. Freshness and phase status を attach する。
7. `BuildDiagnosticIndex` を CLI、artifact commit、LSP snapshot construction へ publish する。

Deduplication は different primary spans、different fix edits、different explanation refs を持つ diagnostics を merge してはならない。

Open-buffer overlay diagnostics は artifact projection を skip する。Document version で key 付けされた `LspDiagnosticOverlay` values に保存され、newer overlay diagnostics または same source version の full build snapshot により replace される。

### Artifact Projection

Phase 15 で `ArtifactService` は compact diagnostics を `VerifiedArtifact` に embed し、enabled の場合は larger explanation payloads を development artifacts として書く。

Artifacts が保存するもの:

- downstream tools が必要とする stable diagnostic records
- bulk traces ではなく explanation refs
- compiler-native coordinates の source ranges
- external explanation files の hashes

LSP range conversion は artifacts に保存しない。

### LSP Snapshot Publication

1. LSP bridge は normal coherent publication boundary として `SnapshotPublished` build events を受け取る。
2. `artifact-manifest.json` を最初に読み、manifest entries で module artifacts を resolve する。
3. Snapshot 用の expression、symbol、diagnostic、explanation indexes を構築する。
4. IDE で編集中の files について open-buffer source versions を overlay する。
5. New `LspSnapshot` を request handlers へ atomically swap する。
6. Document version が一致する、または安全に map できる場合、affected files の diagnostics を publish する。

`ArtifactCommitted` events は caches を warm したり per-module artifact readers を invalidate したりしてよいが、build service が package snapshot coherent とも mark しない限り、それだけで full `LspSnapshot` swap を trigger しない。

Semantic artifacts が open buffer より stale な場合、bridge は stale diagnostics を表示し続けてよいが、`stale_artifact` として mark し、destructive code actions を suppress する。

### Lightweight Open-Buffer Diagnostics

LSP bridge は full build completion の前に open buffers へ syntax-only frontend work を要求してよい。

These diagnostics:

- current editor text snapshot を使う
- `current` と mark される
- same document version の older syntax diagnostics を replace してよい
- verified semantic status を主張しない
- same source version の full build snapshot が publish されると supersede される

### Code Actions

1. LSP は document version and range を持つ code action request を受け取る。
2. Bridge は current snapshot 内の diagnostics and fix suggestions を lookup する。
3. Local text edits は current range and expected text を検証する。
4. Artifact-assisted edits は source hash、artifact hash、relevant explanation or `missing_facts` handle を検証する。
5. Command actions は snapshot and diagnostic handles を capture し、command execution 時に revalidate する。

Proof-related suggestions は backend-derived candidates と kernel-verified `used_axioms` を区別しなければならない。

### Explanation Queries

1. Request は `DiagnosticId`、`ExprId`、または `VcId` と snapshot id を渡す。
2. `ExplanationStore` は artifact hashes または cache keys に対して handle を resolve する。
3. Query service は size and depth bounds を enforce する。
4. Response は freshness、source ranges、bounded structured explanation data を含む。

Handle が obsolete snapshot に属する場合、LSP bridge は stale-handle response を返し、matching code/span が current diagnostic にあるなら current diagnostic 上で query を再実行する提案をしてよい。

## API Sketch

### Diagnostic Registry

```rust
trait DiagnosticRegistry {
    fn lookup(&self, code: DiagnosticCode) -> Option<DiagnosticDescriptor>;
    fn validate_descriptor(&self, descriptor: DiagnosticDescriptor) -> Result<(), RegistryError>;
}

struct DiagnosticDescriptor {
    code: DiagnosticCode,
    semantic_name: String,
    default_severity: DiagnosticSeverity,
    phase_family: DiagnosticPhaseFamily,
    documentation: Option<DocUrl>,
}
```

### Diagnostic Sink and Aggregator

```rust
trait DiagnosticSink {
    fn emit(&self, diagnostic: DiagnosticDraft);
}

trait DiagnosticAggregator {
    fn ingest(&self, source: DiagnosticSource, diagnostics: Vec<DiagnosticDraft>, status: PhaseStatus);
    fn publish(&self, snapshot: BuildSnapshotId) -> BuildDiagnosticIndex;
}
```

### LSP Bridge

```rust
trait LspBridge {
    fn open_document(&self, document: TextDocumentItem);
    fn change_document(&self, change: TextDocumentChange);
    fn publish_snapshot(&self, snapshot: LspSnapshot);
    fn diagnostics_for(&self, document: DocumentUri) -> Vec<LspDiagnostic>;
    fn code_actions(&self, request: CodeActionRequest) -> Vec<CodeAction>;
    fn explain(&self, request: ExplainRequest) -> Result<ExplainResponse, ExplainError>;
}
```

### Range Conversion

```rust
trait RangeMapper {
    fn to_lsp_range(&self, source: SourceId, range: SourceRange, version: SourceVersion) -> Result<LspRange, RangeError>;
    fn validate_edit_target(&self, edit: &TextEdit, version: SourceVersion) -> Result<(), RangeError>;
}
```

Range conversion は request の exact source snapshot を使って LSP boundary でのみ行われる。

## Error Handling

### Diagnostic Registry Errors

Invalid or reused diagnostic codes は build-time registry errors である。Compiler implementation bug を示すため、通常の user diagnostics としては emit しない。

### Aggregation Errors

Diagnostic が missing source range または incompatible source snapshot を参照する場合、aggregator は developer modes で internal diagnostic を報告し、broken record を user-facing publication から drop する。別 document version 用に計算された range を publish してはならない。

### LSP Snapshot Errors

Unsupported artifact schemas、missing manifests、stale artifact hashes は freshness class によって LSP features を degrade する。

| Condition | LSP Behavior |
|---|---|
| missing artifact manifest | open buffers について syntax-only features |
| unsupported artifact schema | source range が分かる場合 compatibility diagnostic を publish し、semantic features を disable |
| stale artifact for open buffer | stale diagnostics を `stale_artifact` として保持し、unsafe edits を suppress |
| missing explanation payload | compact diagnostic preview を表示し、explanation unavailable を報告 |

### Cancellation

Open-buffer diagnostics and explanation queries は cancellable である。Published snapshots は immutable であり、publication 後には cancel しない。

## Constraints and Assumptions

- Diagnostic codes は stable で reuse されない。
- `DiagnosticId` は一つの build snapshot 内でのみ deterministic である。
- LSP requests は bare diagnostic IDs ではなく snapshot identity を含む diagnostic handles を持つ。
- Compiler diagnostics は LSP UTF-16 positions ではなく source ranges を保存する。
- Diagnostic aggregation は CLI、artifacts、LSP の ordering の single source である。
- Open-buffer overlay diagnostics は `VerifiedArtifact` へ emit しない。
- Cache-backed explanations は optional であり proof status を確立しない。
- LSP は proof results を accept せず、build artifacts を直接 mutate しない。
- Open-buffer source snapshots は editor features について disk text を override する。
- Code actions は edits 適用前に range and text を validate する。
- Explanation APIs は bounded and query-oriented であり、bulk dumps ではない。
- Missing or stale artifacts は editor features を graceful に degrade しなければならない。
