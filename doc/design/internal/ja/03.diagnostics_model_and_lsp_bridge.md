# 内部設計: 診断モデルと LSP ブリッジ

> 正本は英語です。英語版: [../en/03.diagnostics_model_and_lsp_bridge.md](../en/03.diagnostics_model_and_lsp_bridge.md)。

## 目的

この文書は、診断モデルと、診断、エディタメタデータ、コードアクション、説明クエリを公開する LSP ブリッジの内部設計を定義する。

ここでは、診断は構造化されたフェーズ出力であり、LSP サーバーはコンパイラ内部の IR ではなく不変のスナップショットと安定したアーティファクトを読む、というアーキテクチャ上の規則を詳細化する。

## 責務

### 診断レジストリ

`DiagnosticRegistry` は、安定した診断コードを管理する。

提供するもの:

- `DiagnosticCode` の恒久的な割り当てと廃止の規則
- 数値コードから、意味的な名前、既定の重大度、フェーズ系統、ドキュメント URL への対応付け
- あるコードが別の意味で再利用されないことを保証する互換性検証
- CLI レンダリング、アーティファクト出力、LSP 変換のための参照データ

メッセージはバージョンを追うごとに改善されてよい。ツールは、メッセージ文や意味的な名前ではなく `DiagnosticCode` をキーとする。

### 診断生成 API

コンパイラの各フェーズは、`DiagnosticSink` を通じて構造化された診断ドラフトを発行する。

フェーズサービスは、診断の生成に必要なローカルな事実を保持するが、CLI 出力や LSP 診断の整形は行わない。安全に計算できる場合は、構造化された詳細、説明ハンドル、修正候補を付与してよい。

### 診断集約器

`DiagnosticAggregator` は、ビルドスナップショットごとの正規化、同一性の割り当て、重複除去、整列、公開準備を管理する。

フェーズ結果、スケジューラ／ビルドのエラー、キャッシュ／アーティファクトのエラーから診断を受け取り、CLI、アーティファクト、LSP スナップショット向けに不変の `BuildDiagnosticIndex` 値を公開する。

未保存のエディタテキストに対して生成される、開いているバッファの診断は、LSP オーバーレイ診断である。同じレコード形を再利用してよいが、CLI 出力や `VerifiedArtifact` へは公開しない。

### LSP ブリッジ

`LspBridge` は、エディタ向けのプロトコル変換を管理する。

責務:

- 開いているバッファのソーススナップショットとドキュメントバージョン
- コンパイラの `SourceRange` 値から LSP の UTF-16 範囲への変換
- 現在の `LspSnapshot` からの診断の公開
- ホバー、定義、シンボル、セマンティックトークン、インレイヒント、コードアクション、説明要求の提供
- コンパイラドライバを通じたビルドのスケジューリング

証明の受理、型検査、ATP による推論、アーティファクトの変更は、LSP ブリッジの責務ではない。

### 説明ストア

`ExplanationStore` は、遅延評価される説明ハンドルと、サイズに上限のある説明ペイロードを管理する。

大きなトレースは、アーティファクト、キャッシュを裏付けとするクエリデータ、または専用の説明ファイルに残す。通常の診断は、コンパクトな参照とサイズに上限のあるプレビューだけを保持する。

## データモデル

### 診断ドラフト

```rust
struct DiagnosticDraft {
    code: DiagnosticCode,
    severity_override: Option<DiagnosticSeverity>,
    phase: PipelinePhase,
    stable_detail_key: String,
    message: String,
    primary_span: SourceRange,
    secondary_spans: Vec<DiagnosticSpan>,
    notes: Vec<DiagnosticNote>,
    details: DiagnosticDetails,
    fixes: Vec<FixSuggestion>,
    explanation: Option<ExplanationRef>,
}
```

ドラフトは、スナップショット全体にわたる同一性の割り当て前に、フェーズサービスが発行する。ドラフトの時点で、該当するソーススナップショットに基づく `SourceRange` 値をすでに用いていなければならない。

### 診断レコード

```rust
struct Diagnostic {
    handle: DiagnosticHandle,
    code: DiagnosticCode,
    semantic_name: String,
    severity: DiagnosticSeverity,
    phase: PipelinePhase,
    stable_detail_key: String,
    message: String,
    primary_span: SourceRange,
    secondary_spans: Vec<DiagnosticSpan>,
    notes: Vec<DiagnosticNote>,
    details: DiagnosticDetails,
    fixes: Vec<FixSuggestion>,
    explanation: Option<ExplanationRef>,
    related_ids: Vec<DiagnosticId>,
    freshness: DiagnosticFreshness,
}

struct DiagnosticHandle {
    snapshot: BuildSnapshotId,
    id: DiagnosticId,
}
```

`DiagnosticId` は、1 つの `BuildSnapshotId` 内で決定的である。重複除去の後、ソースの同一性、診断コード、フェーズ、主スパン、およびスパンごとの安定した序数から導出される。ツールは `DiagnosticHandle` を通じて解釈しなければならない。`DiagnosticId` 単体では、スナップショットの外では意味を持たない。

### 診断インデックス

```rust
struct BuildDiagnosticIndex {
    snapshot: BuildSnapshotId,
    by_source: BTreeMap<SourceId, Vec<Diagnostic>>,
    by_id: BTreeMap<DiagnosticId, DiagnosticRef>,
    phase_status: BTreeMap<WorkUnit, PhaseStatus>,
}

struct LspDiagnosticOverlay {
    document: DocumentUri,
    version: SourceVersion,
    diagnostics: Vec<Diagnostic>,
}
```

ビルドインデックスは公開後に不変となる。CLI レンダリング、アーティファクトへの射影、LSP の意味診断は、いずれも同じインデックスを読む。LSP オーバーレイ診断はエディタの現在状態に対する追加分であり、公開アーティファクトへは決して射影されない。

### 修正候補

```rust
struct FixSuggestion {
    title: String,
    confidence: FixConfidence,
    edits: Vec<TextEdit>,
    command: Option<DiagnosticCommand>,
    applicability: Applicability,
    required_snapshot: Option<BuildSnapshotId>,
    required_text_hash: Option<Hash>,
}
```

ソーステキストに依存する編集は、適用前に対象範囲と期待テキストを検証しなければならない。アーティファクトを利用する修正は、アーティファクトハッシュとソースハッシュも検証する。

### 説明参照

```rust
struct ExplanationRef {
    kind: ExplanationKind,
    diagnostic: Option<DiagnosticHandle>,
    expr_id: Option<ExprId>,
    vc_id: Option<VcId>,
    source: ExplanationSource,
    summary_hash: Hash,
    preview: Option<String>,
}

enum ExplanationSource {
    Artifact(PathBuf),
    CacheRecord(CacheKey),
    QueryService(ExplanationQuery),
}
```

説明参照はハンドルである。作成元のスナップショットまたはアーティファクトハッシュの外では無効である。`CacheRecord` の出所は、ローカルな説明クエリのための任意の高速化手段または裏付けデータである。公開診断はキャッシュレコードなしで解釈できなければならず、キャッシュを裏付けとする説明が証明ステータスを決定することはない。

### LSP スナップショット

```rust
struct LspSnapshot {
    snapshot_id: BuildSnapshotId,
    artifact_manifest: ArtifactManifestRef,
    source_versions: Vec<SourceVersion>,
    diagnostics: BuildDiagnosticIndex,
    overlays: Vec<LspDiagnosticOverlay>,
    expression_index: ExpressionIndex,
    symbol_index: SymbolIndex,
    explanation_index: ExplanationIndex,
    missing_facts: Option<MissingFactsRef>,
    freshness: SnapshotFreshness,
}

enum SnapshotFreshness {
    Current,
    StaleArtifact,
    Pending,
}
```

スナップショットは、要求ハンドラへアトミックに差し替えられる。処理中の要求は、開始時のスナップショット上で完了する。

## 制御フロー

### フェーズによる診断の生成

1. フェーズサービスが、`DiagnosticSink` を持つ `PhaseContext` を受け取る。
2. フェーズが、ソース範囲と構造化された詳細を持つ `DiagnosticDraft` 値を発行する。
3. フェーズが `PhaseResult` の中でドラフトを返す。
4. スケジューラが、フェーズの診断とフェーズステータスを集約器へ転送する。
5. 集約器は、対象の消費者向けに整合したスナップショット境界が整うまで、診断を公開しない。

致命的・ブロッキングのフェーズステータスは、診断の重大度とは別に記録される。エラー診断であっても回復可能な場合があり、また上流の依存が失敗したためにフェーズがブロックされる場合もある。

### ビルド集約

1. パスをワークスペース相対の `SourceId` へ正規化する。
2. ソーススナップショットが対象の公開と一致しなくなった、古い診断を破棄またはマークする。
3. 同じコード、フェーズ、主スパン、メッセージの同一性、詳細ハッシュ、修正の同一性を持つ診断を重複除去する。
4. 決定的な `DiagnosticId` を割り当てる。
5. ソースファイル、ソース範囲、フェーズ順、重大度、診断コード、安定した詳細キー、安定した序数で整列する。
6. 鮮度とフェーズステータスを付与する。
7. `BuildDiagnosticIndex` を CLI、アーティファクトコミット、LSP スナップショット構築へ公開する。

重複除去は、主スパンが異なる、修正編集が異なる、または説明参照が異なる診断を併合してはならない。

開いているバッファのオーバーレイ診断は、アーティファクトへの射影を行わない。ドキュメントバージョンでキー付けされた `LspDiagnosticOverlay` 値に保存され、より新しいオーバーレイ診断、または同じソースバージョンの完全なビルドスナップショットによって置き換えられる。

### アーティファクトへの射影

フェーズ 15 で、`ArtifactService` はコンパクトな診断を `VerifiedArtifact` に埋め込み、有効な場合は、より大きな説明ペイロードを開発用アーティファクトとして書き込む。

アーティファクトが保存するもの:

- 下流ツールが必要とする安定した診断レコード
- 一括トレースではなく、説明参照
- コンパイラ固有の座標によるソース範囲
- 外部の説明ファイルのハッシュ

LSP の範囲変換はアーティファクトに保存しない。

### LSP スナップショットの公開

1. LSP ブリッジは、通常の整合した公開境界として `SnapshotPublished` ビルドイベントを受け取る。
2. まず `artifact-manifest.json` を読み、マニフェストエントリでモジュールアーティファクトを解決する。
3. そのスナップショット用に、式・シンボル・診断・説明の各インデックスを構築する。
4. IDE で現在編集中のファイルについて、開いているバッファのソースバージョンをオーバーレイする。
5. 新しい `LspSnapshot` を要求ハンドラへアトミックに差し替える。
6. ドキュメントバージョンが一致する、または安全に対応付けられる場合、影響を受けるファイルの診断を公開する。

`ArtifactCommitted` イベントは、キャッシュを温めたり、モジュール単位のアーティファクト読み取りを無効化したりしてよいが、ビルドサービスがパッケージスナップショットを整合済みとマークしない限り、それ単体で完全な `LspSnapshot` の差し替えを引き起こすことはない。

意味アーティファクトが開いているバッファに対して古くなっている場合、ブリッジは古い診断を表示し続けてよいが、それらを `stale_artifact` としてマークし、破壊的なコードアクションを抑制する。

### 軽量な、開いているバッファの診断

LSP ブリッジは、完全なビルドが完了する前に、開いているバッファに対して構文のみのフロントエンド処理を要求してよい。

これらの診断は次の性質を持つ。

- 現在のエディタテキストのスナップショットを用いる。
- `current` としてマークされる。
- 同じドキュメントバージョンの、より古い構文診断を置き換えてよい。
- 検証済みの意味的なステータスを主張しない。
- 同じソースバージョンの完全なビルドスナップショットが公開されると置き換えられる。

### コードアクション

1. LSP が、ドキュメントバージョンと範囲を持つコードアクション要求を受け取る。
2. ブリッジが、現在のスナップショット内の診断と修正候補を参照する。
3. ローカルなテキスト編集は、現在の範囲と期待テキストを検証する。
4. アーティファクトを利用する編集は、ソースハッシュ、アーティファクトハッシュ、および関連する説明または `missing_facts` ハンドルを検証する。
5. コマンドアクションは、スナップショットと診断のハンドルを取り込み、コマンド実行時に再検証する。

証明に関わる候補は、バックエンド由来の候補と、カーネル検証済みの `used_axioms` を区別しなければならない。

### 説明クエリ

1. 要求は、`DiagnosticId`、`ExprId`、または `VcId` と、スナップショット ID を渡す。
2. `ExplanationStore` が、アーティファクトハッシュまたはキャッシュキーに対してハンドルを解決する。
3. クエリサービスがサイズと深さの上限を強制する。
4. 応答は、鮮度、ソース範囲、上限付きの構造化された説明データを含む。

ハンドルが古いスナップショットに属する場合、LSP ブリッジは古いハンドルである旨の応答を返し、一致するコード／スパンが現在の診断に存在するなら、現在の診断上でクエリを再実行するよう提案してよい。

範囲変換は、要求に対応する厳密なソーススナップショットを用いて、LSP 境界でのみ行われる。

## エラー処理

### 診断レジストリのエラー

無効な、または再利用された診断コードは、ビルド時のレジストリエラーである。これらはコンパイラの実装バグを示すため、通常のユーザー診断としては発行しない。

### 集約のエラー

診断が、欠落したソース範囲や互換性のないソーススナップショットを参照する場合、集約器は開発者モードで内部診断を報告し、壊れたレコードをユーザー向けの公開から除外する。別のドキュメントバージョン向けに計算された範囲を公開してはならない。

### LSP スナップショットのエラー

非対応のアーティファクトスキーマ、欠落したマニフェスト、古いアーティファクトハッシュは、鮮度クラスに応じて LSP 機能を縮退させる。

| 条件 | LSP の挙動 |
|---|---|
| アーティファクトマニフェストの欠落 | 開いているバッファに対し構文のみの機能を提供する |
| 非対応のアーティファクトスキーマ | ソース範囲が判明していれば互換性の診断を公開し、意味的な機能を無効化する |
| 開いているバッファに対するアーティファクトの陳腐化 | 古い診断を `stale_artifact` として保持し、安全でない編集を抑制する |
| 説明ペイロードの欠落 | コンパクトな診断プレビューを表示し、説明が利用できない旨を報告する |

### キャンセル

開いているバッファの診断と説明クエリはキャンセル可能である。公開済みのスナップショットは不変であり、公開後にキャンセルされることはない。

## 制約と前提

- 診断コードは安定しており、再利用されない。
- `DiagnosticId` は、1 つのビルドスナップショット内でのみ決定的である。
- LSP 要求は、`DiagnosticId` 単体ではなく、スナップショットの同一性を含む診断ハンドルを持つ。
- コンパイラの診断は、LSP の UTF-16 位置ではなくソース範囲を保持する。
- 診断集約は、CLI、アーティファクト、LSP に共通する順序の唯一の源である。
- 開いているバッファのオーバーレイ診断は、`VerifiedArtifact` へは出力されない。
- キャッシュを裏付けとする説明は任意であり、証明ステータスを確立することはない。
- LSP は証明結果を受理せず、ビルドアーティファクトを直接変更することもない。
- 開いているバッファのソーススナップショットは、エディタ機能においてディスク上のテキストに優先する。
- コードアクションは、編集を適用する前に範囲とテキストを検証する。
- 説明 API は上限付きかつクエリ指向であり、一括ダンプではない。
- アーティファクトが欠落または陳腐化した場合は、エディタ機能を緩やかに縮退させなければならない。
