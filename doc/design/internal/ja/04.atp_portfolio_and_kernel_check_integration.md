# Internal Design: ATP Portfolio and Kernel Check Integration

> Canonical language: English. English canonical version: [../en/04.atp_portfolio_and_kernel_check_integration.md](../en/04.atp_portfolio_and_kernel_check_integration.md).

## 目的

この文書は、phase 13 ATP portfolio execution が phase 14 kernel certificate checking へどのように接続されるかを定義する。

Reasoning boundary を internal task APIs、evidence records、portfolio winner selection、proof witness storage、diagnostics、cache reuse、artifact-facing proof status へ詳細化する。ATP backends は candidate evidence を生成する。Accepted proof status を生成できるのは kernel checking または明示的に許可された policy outcomes だけである。

## Responsibilities

### ATP Portfolio Service

`AtpPortfolioService` は、一つの open VC について phase 13 portfolio construction and candidate evidence collection を所有する。

責務:

- verifier policy、source hints、VC shape、backend availability、resource limits から backend profiles を選ぶ
- ATP encoders に concrete encodings を要求する
- scheduler を通して `BackendRun` tasks を spawn する
- backend results、stdout/stderr hashes、certificates、counterexamples、timing を収集する
- policy が許す early-stop rules を適用する
- raw completion order に依存しない deterministic `PortfolioResult` を生成する

Proof status は受理しない。Evidence だけを生成する。

### Backend Runner

`BackendRunner` は一つの backend child process invocation を所有する。

責務:

- command line、working directory、stdin or temporary problem file、timeout、memory limit、random seed を設定する
- stdout、stderr、exit status、wall time、backend version を capture する
- backend output を `Proved`、`Counterexample`、`Timeout`、`Unknown`、`Error` に classify する
- proof payload がある場合 backend-specific certificate candidates として parse する
- cancellation または portfolio stop に応じて process を terminate する

`Proved` は backend が expected result を観測し proof evidence を返したことを意味する。Kernel acceptance ではない。

### Kernel Check Service

`KernelCheckService` は phase 14 proof evidence acceptance を所有する。

責務:

- `CoreIr`、dependency artifact hashes、premise refs、registration traces、`AtpProvenance` から immutable kernel contexts を構築する
- normalized certificates と replayable kernel primitives を shared kernel evidence API で check する
- accepted certificates、accepted built-in certificates、allowed kernel primitives から trusted `used_axioms` を抽出する
- artifact emission and diagnostics 用の `KernelCheckResult` values を生成する

Kernel service は ATP search を再実行せず、backend-reported used axioms を validation なしに信頼せず、build-policy decisions を行わない。Policy-independent validation results を返す。

### Proof Policy Evaluator

`ProofPolicyEvaluator` は kernel proof checking ではない policy decisions を所有する。

決定するもの:

- externally attested evidence を記録してよいか
- `require_kernel_certificates` が externally attested evidence を error として reject するか
- current build mode で open obligation を許すか
- phase 12 built-in discharge results と phase 13/14 ATP results を artifact-facing proof status へ project するとき、どの accepted candidate が winner になるか

Policy outcomes は trusted proof status とは別である。

### Proof Witness Store

`ProofWitnessStore` は phase 15 artifact commit に渡す witness files and references を所有する。

Kernel-checkable certificate material、accepted built-in certificates または allowed kernel primitives、provenance metadata を保存する。Artifact commit 前に content hashes を記録するが、witness は artifact manifest が参照して初めて publication-reachable になる。

## Data Model

### Portfolio Task

```rust
struct PortfolioTaskInput {
    vc_id: VcId,
    vc_hash: Hash,
    atp_problem: AtpProblem,
    premise_set_hash: Hash,
    proof_hint: Option<ProofHint>,
    policy: ProverPolicy,
    budget: ObligationBudget,
    snapshot: BuildSnapshotId,
}

struct BackendProfile {
    id: BackendProfileId,
    backend: BackendKind,
    input_format: ProblemFormat,
    certificate_formats: Vec<CertificateFormat>,
    priority: u32,
    resource_limits: BackendResourceLimits,
    requires_kernel_certificate: bool,
}
```

Profile priority は deterministic winner selection の一部である。Release policy を override しない。

### Backend Run Result

```rust
struct BackendRunResult {
    profile: BackendProfileId,
    encoded_problem_hash: Hash,
    backend_version: BackendVersion,
    command_fingerprint: Hash,
    status: BackendRunStatus,
    certificate: Option<CertificateCandidate>,
    counterexample: Option<CounterexampleEvidence>,
    stdout_hash: Hash,
    stderr_hash: Hash,
    timing: TimingSummary,
}

enum BackendRunStatus {
    Proved,
    Counterexample,
    Timeout,
    Unknown,
    Error,
    Cancelled,
}
```

`BackendRunStatus::Proved` は observed backend result が `EncodedProblem.expected_result` と一致し、proof evidence が存在する場合にのみ生成されなければならない。

### Certificate Candidate

```rust
struct CertificateCandidate {
    certificate_id: CertificateId,
    format: CertificateFormat,
    payload_ref: EvidenceBlobRef,
    atp_provenance_hash: Hash,
    symbol_map_hash: Hash,
    backend_formula_labels: Vec<AtpFormulaId>,
}
```

Candidate payloads は kernel checking 前には temporary evidence blobs として保存してよい。Accepted または policy evidence として記録されるまでは proof witnesses ではない。

### Kernel Check Input and Result

```rust
struct KernelCheckInput {
    vc_id: VcId,
    evidence: KernelEvidenceInput,
    kernel_context: KernelContextRef,
    expected_problem_hash: Hash,
}

enum KernelEvidenceInput {
    AtpCertificate(CertificateCandidate),
    BuiltinCertificate(BuiltinCertificateEvidence),
    ResolutionTrace(ResolutionTraceEvidence),
}

struct BuiltinCertificateEvidence {
    witness_id: BuiltinWitnessId,
    certificate_ref: EvidenceBlobRef,
    certificate_hash: Hash,
    kernel_profile: KernelProfile,
}

struct KernelCheckResult {
    vc_id: VcId,
    evidence: KernelEvidenceId,
    status: KernelCheckStatus,
    used_axioms: Vec<PremiseRef>,
    proof_witness: Option<ProofWitnessDraft>,
    diagnostics: Vec<DiagnosticDraft>,
}

enum KernelCheckStatus {
    Accepted,
    Rejected {
        reason: RejectionReason,
    },
}

enum RejectionReason {
    MalformedCertificate,
    InvalidSubstitution,
    InvalidSatProof,
    UnresolvedSymbol,
    UnsupportedCertificateFormat,
    ContextMismatch,
    Timeout,
    ResourceExhaustion,
}
```

`Accepted` だけが trusted artifact proof status を生成できる kernel check status である。`KernelEvidenceInput::AtpCertificate` では `proof_status = "kernel_verified"` として project される。`KernelEvidenceInput::BuiltinCertificate` では `"kernel_verified"` ではなく `proof_status = "discharged_builtin"` として project される。built-in evidence は normalized certificate または small-kernel architecture が許す explicit kernel primitive として encode しなければならない。任意の deterministic rule trace は trusted kernel evidence ではなく policy evidence である。

### ATP Portfolio Result

```rust
struct PortfolioResult {
    vc_id: VcId,
    backend_results: Vec<BackendRunResult>,
    kernel_results: Vec<KernelCheckResult>,
    selected: AtpPortfolioSelection,
    diagnostics: Vec<DiagnosticDraft>,
}

enum AtpPortfolioSelection {
    KernelVerified {
        candidate: CertificateId,
        witness: ProofWitnessDraft,
        used_axioms: Vec<PremiseRef>,
    },
    ExternallyAttested {
        backend: BackendProfileId,
        evidence: EvidenceBlobRef,
    },
    Open {
        reason: OpenReason,
    },
    RejectedByPolicy {
        reason: PolicyRejection,
    },
}
```

`PortfolioResult` は phase 13 に `NeedsAtp` として到達した VCs だけを扱う。Phase 12 built-in discharge results は ATP portfolio へ挿入せず、kernel evidence API で check した後、artifact-facing proof status selection で merge する。

### Artifact Proof Selection

Selected artifact result は artifact commit 時に canonical `VcId` order で sort and emit される。

Policy selection は正規化された evidence view を消費する。

```rust
enum ProofCandidateEvidence {
    AtpKernel(KernelCheckResult),
    BuiltinKernel(KernelCheckResult),
    External(BackendRunResult),
    Open(OpenReason),
}

enum ProofCandidateSelection {
    KernelVerified {
        candidate: KernelEvidenceId,
        witness: ProofWitnessDraft,
        used_axioms: Vec<PremiseRef>,
    },
    DischargedBuiltin {
        witness: ProofWitnessDraft,
        used_axioms: Vec<PremiseRef>,
    },
    ExternallyAttested {
        backend: BackendProfileId,
        evidence: EvidenceBlobRef,
    },
    Open {
        reason: OpenReason,
    },
    RejectedByPolicy {
        reason: PolicyRejection,
    },
}
```

これにより、policy ranking は backend process classification から分離される。

## Control Flow

### Portfolio Construction

1. `VcService` は open VCs を canonical `VcId` order で emit する。
2. `AtpPortfolioService` は一つの open VC とその `AtpProblem` を受け取る。
3. Service は `default_solver`、`@proof_hint`、policy、availability、VC logic profile から backend profiles を選ぶ。
4. Scheduler は一つの `AtpSolve` task と child `BackendRun` tasks を作成する。
5. 各 backend run は immutable encoded problem and backend budget を受け取る。

`@proof_hint(solver: ...)` が solver を制限する場合、portfolio はその hint and policy が許す profiles だけを含む。

### Backend Execution

1. `BackendRunner` は concrete problem input を write または stream する。
2. Configured resource limits で backend child process を launch する。
3. Stdout/stderr を capture し、timeout or cancellation 時に terminate する。
4. `EncodedProblem.expected_result` に照らして result を classify する。
5. Certificate candidates または counterexample evidence を temporary evidence blobs として保存する。
6. `BackendRunResult` を portfolio task に返す。

Backend crashes、parse failures、malformed output は verifier crash ではなく backend errors になる。

### Kernel Check Scheduling

Backend result が certificate candidate を含む場合、portfolio service は、その candidate 由来の kernel-checked evidence を policy が利用できるなら scheduler に `KernelCheck` work の作成を依頼する。

Kernel checks は independent candidates について parallel に実行してよい。Contexts は source snapshot、dependency artifacts、VC hash、symbol map hash、ATP provenance hash により immutable and content-addressed である。

ATP search 前に生成された built-in discharge evidence は `AtpSolve` subgraph の一部ではない。同じ `KernelCheckService` を `KernelEvidenceInput::BuiltinCertificate` として使い、`ProofCandidateEvidence::BuiltinKernel` として proof selection stage に合流する。pre-ATP discharge が normalized certificate または allowed kernel primitive として表現できない場合、それは policy-controlled evidence としてのみ記録できる。

### Winner Selection

ATP portfolio winner selection は deterministic and policy-driven である。

Ordering rules:

1. release policy を満たす accepted kernel-verified certificate
2. policy が許可する場合のみ externally attested evidence
3. best diagnostic explanation を持つ open result

同じ class 内では、backend profile priority、certificate format priority、encoded problem hash、backend profile id の順で candidates を order する。Raw completion time は記録するが semantic winner order には決して使わない。

`require_kernel_certificates = true` の場合、externally attested evidence は winner になれない。Build profile が許すなら development evidence として記録してよいが、publication では obligation は rejected by policy または open のままである。

Artifact proof selection は、その後 ATP portfolio result と同じ `VcId` の phase 12 built-in discharge result を merge する。Accepted built-in discharge evidence は `discharged_builtin`、accepted ATP certificate evidence は `kernel_verified` として emit される。Policy-controlled external and open statuses はどちらの trusted status とも区別されたまま残る。

### Early Stop and Cancellation

Portfolio は policy が「残りの backend processes がより良い acceptable result を生成できない」と証明した後に限り remaining backend processes を terminate してよい。

Examples:

- higher class が存在しない policy のもとで kernel-verified winner が accepted された後
- release policy が要求する certificate formats を remaining candidates が持たない場合
- cancellation が build snapshot を supersede した場合

In-process portfolio and kernel work は cooperative cancellation を使う。Backend child processes は backend manager が terminate する。

### Proof Witness and Artifact Flow

1. Accepted ATP certificate kernel results は `ProofWitnessDraft` values、trusted `used_axioms`、`kernel_verified` selections を生成する。
2. Accepted built-in discharge kernel results は `ProofWitnessDraft` values、trusted `used_axioms`、`discharged_builtin` selections を生成する。
3. Externally attested results は evidence refs and diagnostics を生成するが、trusted `used_axioms` は持たない。
4. Open or rejected obligations は proof diagnostics and optional explanation refs を生成する。
5. `ArtifactService` は phase 15 で proof witnesses を書き、content hashes を記録する。
6. Package manifest は artifact commit transaction 成功後にのみ witness references を publish する。

Proof witness hashes は proof-related cache keys に参加する。Cache hits が proof results を reuse できるのは witness hashes、dependency artifact hashes、policy、schema versions が一致する場合だけである。

## Error Handling

### Backend Errors

Backend errors は relevant `VcId` に attached された proof diagnostics になる。

| Condition | Handling |
|---|---|
| timeout | ATP timeout diagnostic を emit し、有用なら他 candidates を running のままにする |
| process crash | backend error diagnostic を emit し、unrelated VCs は fail させない |
| malformed backend output | backend error or unknown として classify する |
| counterexample | provenance mapping が成功した場合だけ diagnostic evidence として記録する |
| cancellation | cache validation が newer snapshot に適用可能と証明しない限り result を discard する |

### Kernel Errors

Kernel rejection はその candidate の proof error であり、verifier crash ではない。

Certificate rejection、context mismatch、missing provenance、unsupported certificate format、malformed witness data、timeout、resource exhaustion はすべて trusted proof status を防ぐ。`KernelCheckResult.status` は stable `RejectionReason` を保持し、diagnostics はそれを phase-level `certificate_rejection` または `kernel_rejection` failure category の structured detail として project する。Other candidates が残っている場合 portfolio は続行してよい。

### Policy Errors

Policy が externally attested evidence を reject する場合、その result は kernel rejection とは区別されたままである。Diagnostics は obligation が no prover succeeded、kernel rejected every certificate、または policy required kernel certificates のどれで失敗したのかを説明するべきである。

### Cache Reuse

Proof cache reuse は complete cache validation を通してのみ許可される。

- same `VcId` and VC hash
- same accepted witness hash、built-in certificate hash、or allowed kernel primitive witness hash
- compatible kernel version and certificate checker schema
- same dependency artifact hashes and trusted environment
- active policy at least as strict as the cached policy

Cached proof results は reused evidence records であり、independent proof authority ではない。

## Constraints and Assumptions

- ATP backend success は candidate evidence であり accepted proof status ではない。
- `kernel_verified` は successful kernel checking の後にのみ emit される。
- Externally attested evidence は policy-controlled であり kernel verification と同等ではない。
- Portfolio completion order は semantic ordering ではない。
- Remaining backend processes は policy-safe winner selection または cancellation の後にのみ停止してよい。
- Kernel contexts は source、dependency、VC、symbol map、provenance hashes により immutable and content-addressed である。
- Backend-reported used axioms は kernel checking で validate されるまでは advisory である。
- Proof-related cache hits は witness hashes and policy strictness を validate しなければならない。
- Counterexamples and missing-fact suggestions は diagnostics を支援するが proof status を確立しない。
