# 内部設計: ATP ポートフォリオとカーネル検査の統合

> 正本は英語です。英語版: [../en/04.atp_portfolio_and_kernel_check_integration.md](../en/04.atp_portfolio_and_kernel_check_integration.md)。

## 目的

この文書は、フェーズ 13 の ATP ポートフォリオ実行が、フェーズ 14 のカーネルエビデンス検査へどのように接続されるかを定義する。

ここでは、推論境界を、内部タスク API、エビデンスレコード、ポートフォリオの勝者選択、証明ウィットネスの保存、診断、キャッシュ再利用、アーティファクト向けの証明ステータスへと詳細化する。ATP バックエンドは候補エビデンスを生成する。受理された証明ステータスを生成できるのは、カーネル検査、または明示的に許可されたポリシー上の結果だけである。

## エビデンス形式の修正

phase 14 の target input は formula/substitution kernel evidence であり、
backend 固有の proof trace ではない。backend は resolution、instance method、
inverse method、SMT tactic、その他の方式で探索してよいが、kernel への
trusted handoff が記録するのは、独立した kernel check に必要な formula、
substitution、provenance、target binding だけである。instantiated formula と
SAT problem は kernel-derived artifact であり、trusted handoff payload ではない。

したがって kernel check service は、与えられた formula/substitution evidence から
導出した SAT problem に対して、信頼された in-process Rust SAT checker を呼び出して
よい。この呼び出しは acceptance checking の一部であり、ATP 探索ではない。
service は前提選択、置換の発明、premise minimization、portfolio search、
overload resolution、cluster search、coercion insertion、大域的な mutable compiler
state 参照を行ってはならない。

`AtpCertificate`、`ResolutionTrace`、`InvalidSatProof` などの歴史的な名前は
legacy task-22 integration model を説明するものであり、kernel evidence schema
redesign が着地するまで `design_drift` / `source_drift` として分類しなければ
ならない。新しい `mizar-atp` 作業は、MiniSAT 互換 resolution trace ではなく
formula/substitution evidence candidate を生成できるまで deferred とする。

## 責務

### ATP ポートフォリオサービス

`AtpPortfolioService` は、1 つの未解消 VC について、フェーズ 13 のポートフォリオ構築と候補エビデンスの収集を担う。

責務:

- 検証器ポリシー、ソース上のヒント、VC の形、バックエンドの利用可能性、リソース上限から、バックエンドプロファイルを選ぶ。
- ATP エンコーダに具体的なエンコーディングを要求する。
- スケジューラを通じて `BackendRun` タスクを起動する。
- バックエンドの結果、標準出力／標準エラーのハッシュ、候補エビデンス、反例、所要時間を収集する。
- ポリシーが許す早期停止規則を適用する。
- 生の完了順に依存しない、決定的な `PortfolioResult` を生成する。

証明ステータスは受理しない。エビデンスを生成するだけである。

### バックエンドランナー

`BackendRunner` は、1 つのバックエンド子プロセスの起動を担う。

責務:

- コマンドライン、作業ディレクトリ、標準入力または一時問題ファイル、タイムアウト、メモリ上限、乱数シードを設定する。
- 標準出力、標準エラー、終了ステータス、実時間、バックエンドバージョンを取り込む。
- バックエンド出力を `Proved`、`Counterexample`、`Timeout`、`Unknown`、`Error` のいずれかに分類する。
- 証明ペイロードがある場合は、バックエンド固有の候補エビデンスとして解析または抽出する。
- キャンセルまたはポートフォリオの停止に応じてプロセスを終了させる。

`Proved` は、バックエンドが期待される結果を観測し、候補エビデンスを返した
ことを意味する。カーネルによる受理ではない。信頼を含意しないようにする
必要がある場合、後続の再設計作業では `EvidenceFound` のような名前を優先する。

### カーネル検査サービス

`KernelCheckService` は、フェーズ 14 の証明エビデンス受理を担う。

責務:

- `CoreIr`、依存アーティファクトのハッシュ、前提参照、登録トレース、`AtpProvenance` から、不変のカーネルコンテキストを構築する。
- formula/substitution エビデンスと再生可能なカーネルプリミティブを、共有のカーネルエビデンス API で検査する。
- 受理された formula evidence、受理された built-in evidence、許可されたカーネルプリミティブから、信頼された `used_axioms` を抽出する。
- アーティファクト出力と診断のための `KernelCheckResult` 値を生成する。

カーネルサービスは、ATP 探索を再実行せず、バックエンドが報告した使用公理を検証なしに信頼せず、ビルドポリシーの判断も行わない。ポリシーに依存しない検証結果を返す。

### 証明ポリシー評価器

`ProofPolicyEvaluator` は、カーネルによる証明検査ではないポリシー判断を担う。

判断するもの:

- 外部証言エビデンスを記録してよいか。
- `require_kernel_certificates` が、外部証言エビデンスをエラーとして拒否するか。
- 現在のビルドモードで未解消の義務を許すか。
- フェーズ 12 の組み込み義務解消の結果と、フェーズ 13／14 の ATP 結果をアーティファクト向けの証明ステータスへ射影する際に、どの受理済み候補を勝者とするか。

ポリシー上の結果は、信頼された証明ステータスとは別物である。

### 証明ウィットネスストア

`ProofWitnessStore` は、フェーズ 15 のアーティファクトコミットへ渡すウィットネスファイルと参照を担う。

カーネルで検査可能な formula/substitution evidence 素材、受理された built-in
evidence または許可されたカーネルプリミティブ、由来メタデータを保存する。
アーティファクトコミット前にコンテンツハッシュを記録するが、ウィットネスが
公開から到達可能になるのは、アーティファクトマニフェストがそれを参照した後である。

### ポートフォリオタスク

```rust

```

プロファイルの優先度は、決定的な勝者選択の一部である。リリースポリシーを上書きすることはない。

### バックエンド実行結果

```rust
struct BackendRunResult {
    profile: BackendProfileId,
    encoded_problem_hash: Hash,
    backend_version: BackendVersion,
    command_fingerprint: Hash,
```

`BackendRunStatus::Proved` は、観測されたバックエンド結果が `EncodedProblem.expected_result` と一致し、かつ候補エビデンスが存在する場合にのみ生成されなければならない。

### 候補エビデンス

```rust
struct CandidateEvidence {
    evidence_id: KernelEvidenceCandidateId,
    format: EvidenceFormat,
    payload_ref: EvidenceBlobRef,
    atp_provenance_hash: Hash,

    backend_formula_labels: Vec<AtpFormulaId>,

```

候補のペイロードは、カーネル検査の前は一時的なエビデンス blob として保存してよい。受理されるか、ポリシーエビデンスとして記録されるまでは、証明ウィットネスではない。

### カーネル検査の入力と結果

```rust
struct KernelCheckInput {
    vc_id: VcId,
    evidence: KernelEvidenceInput,
    kernel_context: KernelContextRef,
    expected_problem_hash: Hash,
}

enum KernelEvidenceInput {
    FormulaSubstitutionEvidence(CandidateEvidence),
    BuiltinEvidence(BuiltinEvidence),
    KernelPrimitive(KernelPrimitiveEvidence),
    LegacyCertificate(LegacyEvidenceRef),
    LegacyResolutionTrace(LegacyEvidenceRef),
}

struct BuiltinEvidence {
    witness_id: BuiltinWitnessId,
    evidence_ref: EvidenceBlobRef,
    evidence_hash: Hash,
    kernel_profile: KernelProfile,
}

struct LegacyEvidenceRef {
    evidence_ref: EvidenceBlobRef,
    evidence_hash: Hash,
    legacy_format: LegacyEvidenceFormat,
}

struct KernelCheckResult {
    vc_id: VcId,
    evidence: KernelEvidenceId,
    status: KernelCheckStatus,
    used_axioms: Vec<PremiseRef>,
    proof_witness: Option<ProofWitnessDraft>,
    diagnostics: Vec<DiagnosticDraft>,
}

enum KernelCheckStatus {
    Accepted,
    Rejected {
        reason: RejectionReason,
    },
}

enum RejectionReason {
    MalformedEvidence,
    MalformedWitnessData,
    InvalidSubstitution,
    InvalidSatRefutation,
    InvalidClusterTrace,
    UnresolvedSymbol,
    UnsupportedEvidenceFormat,
    ContextMismatch,
    MissingProvenance,
    Timeout,
    ResourceExhaustion,
}
```

`Accepted` は、信頼されたアーティファクトの証明ステータスを生成できる唯一の
カーネル検査ステータスである。`KernelEvidenceInput::FormulaSubstitutionEvidence`
では `proof_status = "kernel_verified"` として射影される。
`KernelEvidenceInput::BuiltinEvidence` では、`"kernel_verified"` ではなく
`proof_status = "discharged_builtin"` として射影される。組み込みエビデンスは、
formula/substitution evidence、または小カーネルアーキテクチャが許す明示的な
カーネルプリミティブとしてエンコードしなければならない。任意の決定的な
規則トレースは、信頼されたカーネルエビデンスではなく、ポリシーエビデンスである。
`LegacyCertificate` と `LegacyResolutionTrace` は、closeout 後の migration が retire
または gate するまでの source-drift compatibility input に限られる。kernel check
service は通常の proof policy の下ではそれらを `UnsupportedEvidenceFormat` として
拒否しなければならない。明示的な migration/audit mode はそれらを inspection して
よいが、そこから trusted `Accepted` result を生成してはならない。

### ATP ポートフォリオ結果

```rust
struct PortfolioResult {

        candidate: KernelEvidenceCandidateId,
```

`PortfolioResult` は、`NeedsAtp` としてフェーズ 13 に到達した VC だけを扱う。フェーズ 12 の組み込み義務解消の結果は ATP ポートフォリオへ挿入されず、カーネルエビデンス API で検査された後、アーティファクト向けの証明ステータス選択の段階で併合される。

### アーティファクトの証明選択

選択されたアーティファクト結果は、アーティファクトコミット時に正準的な `VcId` 順で整列・出力される。

ポリシー選択は、正規化されたエビデンスビューを入力とする。

```rust
enum ProofCandidateEvidence {
    AtpKernel(KernelCheckResult),
    BuiltinKernel(KernelCheckResult),
    External(BackendRunResult),
    Open(OpenReason),
}

enum ProofCandidateSelection {
    KernelVerified {
        candidate: KernelEvidenceId,
        witness: ProofWitnessDraft,
        used_axioms: Vec<PremiseRef>,
    },
    DischargedBuiltin {
        witness: ProofWitnessDraft,
        used_axioms: Vec<PremiseRef>,
    },
    ExternallyAttested {
        backend: BackendProfileId,
        evidence: EvidenceBlobRef,
    },
    Open {
        reason: OpenReason,
    },
    RejectedByPolicy {
        reason: PolicyRejection,
    },
}
```

これにより、ポリシーによる序列付けは、バックエンドプロセスの分類から分離される。

## 制御フロー

### ポートフォリオの構築

1. `VcService` が、未解消の VC を正準的な `VcId` 順で発行する。
2. `AtpPortfolioService` が、1 つの未解消 VC とその `AtpProblem` を受け取る。
3. サービスが、`default_solver`、`@proof_hint`、ポリシー、利用可能性、VC の論理プロファイルから、バックエンドプロファイルを選ぶ。
4. スケジューラが、1 つの `AtpSolve` タスクと、子の `BackendRun` タスクを作成する。
5. 各バックエンド実行は、不変のエンコード済み問題と、バックエンドの予算を受け取る。

`@proof_hint(solver: ...)` がソルバーを制限する場合、ポートフォリオには、そのヒントとポリシーが許すプロファイルだけが含まれる。

### バックエンドの実行

1. `BackendRunner` が、具体的な問題入力を書き込む、またはストリーミングする。
2. 設定されたリソース上限でバックエンド子プロセスを起動する。
3. 標準出力／標準エラーを取り込み、タイムアウトまたはキャンセル時に終了させる。
4. `EncodedProblem.expected_result` に照らして結果を分類する。
5. 候補エビデンスまたは反例エビデンスを、一時的なエビデンス blob として保存する。
6. `BackendRunResult` をポートフォリオタスクへ返す。

バックエンドのクラッシュ、解析失敗、不正な出力は、検証器のクラッシュではなく、バックエンドエラーとなる。

### カーネル検査のスケジューリング

バックエンド結果が候補エビデンスを含む場合、ポートフォリオサービスは、その候補由来のカーネル検査済みエビデンスをポリシーが利用できるなら、スケジューラに `KernelCheck` 処理の作成を依頼する。

カーネル検査は、独立した候補について並列に実行してよい。そのコンテキストは、ソーススナップショット、依存アーティファクト、VC ハッシュ、シンボルマップハッシュ、ATP 由来ハッシュによって、不変かつコンテンツアドレス方式である。

ATP 探索より前に生成された組み込み義務解消エビデンスは、`AtpSolve`
サブグラフの一部ではない。同じ `KernelCheckService` を
`KernelEvidenceInput::BuiltinEvidence` として用い、その後
`ProofCandidateEvidence::BuiltinKernel` として証明選択の段階に合流する。ATP 前の
義務解消が、formula/substitution evidence または許可されたカーネルプリミティブとして
表現できない場合は、ポリシー管理下のエビデンスとしてのみ記録できる。

### 勝者の選択

ATP ポートフォリオの勝者選択は、決定的かつポリシー駆動である。

順序付けの規則:

1. リリースポリシーを満たす、受理済みのカーネル検証済みエビデンス。
2. ポリシーが許可する場合に限り、外部証言エビデンス。
3. 最良の診断説明を持つ、未解消の結果。

同じクラス内では、候補をバックエンドプロファイルの優先度、エビデンス形式の
優先度、エンコード済み問題のハッシュ、バックエンドプロファイル ID の順で
並べる。生の完了時刻は記録するが、意味的な勝者順には決して用いない。

`require_kernel_certificates = true` またはその successor kernel-evidence policy が
有効な場合、外部証言エビデンスは勝者になれない。ビルドプロファイルが許すなら
開発用エビデンスとして記録してよいが、公開上は、義務はポリシーにより拒否された
ままか、未解消のままである。

その後、アーティファクトの証明選択は、ATP ポートフォリオ結果と、同じ `VcId`
に対するフェーズ 12 の組み込み義務解消結果とを併合する。受理された組み込み
義務解消エビデンスは `discharged_builtin` として、受理された ATP
formula/substitution evidence は `kernel_verified` として出力される。ポリシー管理下の
外部ステータスや未解消ステータスは、いずれの信頼ステータスとも区別されたまま残る。

### 早期停止とキャンセル

ポートフォリオが残りのバックエンドプロセスを終了させてよいのは、「残りのバックエンドプロセスではこれ以上良い受理可能な結果を生成できない」とポリシーが証明した後に限られる。

例:

- 上位のクラスが存在しないポリシーのもとで、カーネル検証済みの勝者が受理された後。
- 残りの候補が、リリースポリシーの要求するエビデンス形式を持たない場合。
- キャンセルがビルドスナップショットを置き換えた後。

プロセス内のポートフォリオ処理とカーネル処理は、協調的キャンセルを用いる。バックエンド子プロセスはバックエンドマネージャが終了させる。

### 証明ウィットネスとアーティファクトの流れ

1. 受理された ATP formula/substitution kernel result は、`ProofWitnessDraft` 値、信頼された `used_axioms`、`kernel_verified` の選択を生成する。
2. 受理された組み込み義務解消のカーネル結果は、`ProofWitnessDraft` 値、信頼された `used_axioms`、`discharged_builtin` の選択を生成する。
3. 外部証言の結果は、エビデンス参照と診断を生成するが、信頼された `used_axioms` は持たない。
4. 未解消または拒否された義務は、証明の診断と、任意の説明参照を生成する。
5. `ArtifactService` が、フェーズ 15 で証明ウィットネスを書き込み、コンテンツハッシュを記録する。
6. パッケージマニフェストは、アーティファクトコミットトランザクションが成功した後にのみ、ウィットネス参照を公開する。

証明ウィットネスのハッシュは、証明に関わるキャッシュキーに加わる。キャッシュヒットが証明結果を再利用できるのは、ウィットネスハッシュ、依存アーティファクトのハッシュ、ポリシー、スキーマバージョンが一致する場合だけである。

## エラー処理

### バックエンドエラー

バックエンドエラーは、該当する `VcId` に付与される証明の診断となる。

| 条件 | 処理 |
|---|---|
| タイムアウト | ATP タイムアウトの診断を発行し、有用であれば他の候補を実行のまま続ける |
| プロセスのクラッシュ | バックエンドエラーの診断を発行し、無関係な VC は失敗させない |
| 不正なバックエンド出力 | バックエンドエラーまたは不明として分類する |
| 反例 | 由来の対応付けが成功した場合のみ、診断エビデンスとして記録する |
| キャンセル | キャッシュ検証がより新しいスナップショットに適用可能と証明しない限り、結果を破棄する |

### カーネルエラー

カーネルによる拒否は、その候補についての証明エラーであり、検証器のクラッシュではない。

エビデンスの拒否、コンテキストの不一致、由来の欠落、非対応のエビデンス形式、
不正なウィットネスデータ、タイムアウト、リソース枯渇は、いずれも信頼された
証明ステータスを妨げる。`KernelCheckResult.status` は安定した `RejectionReason`
を保持し、診断はそれを、フェーズレベルの `kernel_evidence_rejection` または
`kernel_rejection` という失敗分類のもとで構造化された詳細として射影する。他の
候補が残っていれば、ポートフォリオは続行してよい。

### ポリシーエラー

ポリシーが外部証言エビデンスを拒否する場合、その結果はカーネルによる拒否とは
区別されたまま残る。診断は、義務が失敗したのは、どのソルバーも成功しなかった
ためか、カーネルがすべての candidate evidence package を拒否したためか、それとも
ポリシーが kernel evidence を要求したためかを説明するべきである。

### キャッシュ再利用

証明のキャッシュ再利用は、完全なキャッシュ検証を通してのみ許可される。

- 同じ `VcId` と VC ハッシュ。
- 同じ受理済みウィットネスハッシュ、組み込みエビデンスハッシュ、または許可されたカーネルプリミティブのウィットネスハッシュ。
- 互換性のあるカーネルバージョンとエビデンスチェッカーのスキーマ。
- 同じ依存アーティファクトのハッシュと信頼された環境。
- キャッシュ時のポリシーと少なくとも同等以上に厳格な、現在のポリシー。

キャッシュされた証明結果は、再利用されるエビデンスレコードであり、独立した証明の根拠ではない。

## 制約と前提

- ATP バックエンドの成功は候補エビデンスであり、受理された証明ステータスではない。
- `kernel_verified` は、カーネル検査が成功した後にのみ出力される。
- 外部証言エビデンスはポリシー管理下にあり、カーネル検証と同等ではない。
- ポートフォリオの完了順は、意味的な順序ではない。
- 残りのバックエンドプロセスを停止してよいのは、ポリシー上安全な勝者選択の後か、キャンセルの後だけである。
- カーネルコンテキストは、ソース、依存、VC、シンボルマップ、由来の各ハッシュによって、不変かつコンテンツアドレス方式である。
- バックエンドが報告した使用公理は、カーネル検査で検証されるまでは参考情報にすぎない。
- 証明に関わるキャッシュヒットは、ウィットネスハッシュとポリシーの厳格さを検証しなければならない。
- 反例や不足事実の提案は診断を支援するが、証明ステータスを確立することはない。
