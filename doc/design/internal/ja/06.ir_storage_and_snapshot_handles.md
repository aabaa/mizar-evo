# 内部設計: IR ストレージとスナップショットハンドル

> 正本は英語です。英語版: [../en/06.ir_storage_and_snapshot_handles.md](../en/06.ir_storage_and_snapshot_handles.md)。

## 目的

この文書は、internal IR outputs をどのように store、identify、share、invalidate、cache、project するかを定義する。

Architecture の IR layering rules を implementation-facing handles and lifetime contracts へ詳細化する。IR values は compiler-internal immutable snapshots である。Phase tasks は mutable IR objects ではなく、それらの snapshots への handles を交換する。Published artifacts は stable external projections であり raw IR dumps ではない。

## 責務

### IR Storage Service

`IrStorageService` は workspace build manager における compiler-internal phase output storage を所有する。

責務:

- phase outputs 用の immutable storage slots を割り当てる
- phase output が sealed された後、typed `PhaseOutputRef<T>` handles を返す
- large outputs を memory または content-addressed internal blobs に保存する
- snapshot ownership and lifetime rules を enforce する
- watch/LSP snapshots、diagnostics、explanation requests が参照している間、build sessions をまたいで outputs を retain する
- dependent tasks、LSP snapshots、diagnostics、cache writers が不要になった unreferenced outputs を drop する

Published artifact serialization や cache compatibility decisions は所有しない。

### Snapshot Handle Registry

`SnapshotHandleRegistry` は source、build、phase-output identities を stable handles へ map する。

責務:

- exact source/dependency/toolchain/config states に `BuildSnapshotId` values を割り当てる
- 一つの snapshot 内で `SourceId`、`ModuleId`、`ItemId`、`ExprId`、`VcId` などの identities を割り当てる
- phase outputs 間の parent/derived relationships を記録する
- cache validation なしに incompatible snapshots 間で handles が reuse されることを防ぐ

### Phase Output Publisher

`PhaseOutputPublisher` は phase services が output values を seal するための narrow API である。

責務:

- output が task の snapshot and work unit に属することを検証する
- cache purposes のため canonical internal encoding で content hashes を計算する
- source maps and diagnostic side tables を attach する
- immutable handles を dependent tasks に publish する

Partially-built IR values を他 tasks に expose してはならない。

### IR Cache Adapter

`IrCacheAdapter` は internal phase outputs と cache records の変換を所有する。

責務:

- どの phase outputs が cacheable かを決める
- schema versions and dependency summaries 付きで cache records を serialize する
- cache hits を検証してから `PhaseOutputRef<T>` handles を reconstruct する
- cache hits を proof authority として扱うことを拒否する

Cache records は raw internal IR encodings を含んでよい。Published artifacts は含んではならない。

### Artifact Projection Boundary

`ArtifactProjectionService` は sealed phase outputs を読み、phase 15 用の `VerifiedArtifactDraft` values を作る。

Internal IR から stable external metadata を project する責務:

- exported symbols、labels、source ranges
- rendered or normalized signatures
- proof status and witness references
- diagnostics and explanation refs
- downstream tools が必要とする documentation and LSP metadata

Raw `SurfaceAst`、`TypedAst`、`CoreIr`、`ControlFlowIr`、`VcIr`、`AtpProblem`、kernel-internal state を publish してはならない。

## データモデル

### Snapshot Identity

```rust
struct BuildSnapshot {
    id: BuildSnapshotId,
    workspace_root: PathBuf,
    source_versions: Vec<SourceVersion>,
    dependency_artifacts: Vec<DependencyArtifactRef>,
    lockfile_hash: Hash,
    toolchain: ToolchainInfo,
    verifier_config_hash: Hash,
}

struct SourceVersion {
    source_id: SourceId,
    package_id: PackageId,
    module_path: ModulePath,
    normalized_path: PathBuf,
    source_hash: Hash,
    edition: Edition,
}
```

`BuildSnapshotId` は一つの file の source text だけでなく、complete build input state を識別する。ある snapshot から派生した handles は、cache adapter が equivalence を検証しない限り、別 snapshot の current results として使えない。

### Phase Output Reference

```rust
struct PhaseOutputRef<T> {
    id: PhaseOutputId,
    phase: PipelinePhase,
    unit: WorkUnit,
    snapshot: BuildSnapshotId,
    content_hash: Hash,
    side_table_hash: Hash,
    storage: OutputStorageRef<T>,
    side_tables: SideTableRef,
    lineage: PhaseOutputLineage,
}

struct PhaseOutputLineage {
    producer_task: TaskId,
    input_hashes: Vec<NamedHash>,
    dependency_outputs: Vec<PhaseOutputId>,
}
```

`PhaseOutputRef<T>` は publication 後 immutable である。型 parameter は expected output kind の compile-time documentation であり、dynamic storage も schema version and runtime kind tag を記録する。`side_tables` は output に関連する source maps、diagnostics、explanation refs、documentation attachment metadata への stable handle である。

### Storage References

```rust
enum OutputStorageRef<T> {
    InMemory {
        arena: ArenaId,
        slot: OutputSlotId,
        marker: PhantomData<T>,
    },
    Blob {
        blob: ContentBlobRef,
        schema_version: SchemaVersion,
        marker: PhantomData<T>,
    },
    CacheRecord {
        record: CacheRecordRef,
        output_hash: Hash,
        marker: PhantomData<T>,
    },
}
```

Storage reference は publication path ではない。Process memory、temporary files、または source-level semantics を変えずに削除可能な internal cache blobs を指してよい。

### IR Identity Tables

```rust
struct IrIdentityTables {
    source_ids: IdTable<SourceId, SourceIdentity>,
    module_ids: IdTable<ModuleId, ModuleIdentity>,
    item_ids: IdTable<ItemId, ItemIdentity>,
    expr_ids: IdTable<ExprId, ExprIdentity>,
    vc_ids: IdTable<VcId, VcIdentity>,
}
```

Identity tables は `BuildSnapshotId` に scope される。Stable public identities は artifact schemas を通して project される。Internal ids は compatibility promises ではない。

### Source Maps and Side Tables

```rust
struct IrSourceMap {
    source_id: SourceId,
    ranges: Vec<IrRangeEntry>,
}

struct IrSideTables {
    source_maps: Vec<IrSourceMap>,
    diagnostics: Vec<DiagnosticDraft>,
    explanation_refs: Vec<ExplanationDraftRef>,
    doc_attachments: Vec<DocAttachmentId>,
}
```

Source maps and side tables は output value と一緒に保存され、output handle から参照される。これにより diagnostics、artifacts、LSP、documentation は IR value を mutate せずに source ranges について合意できる。

## 制御フロー

### Phase Output Publication

1. Phase task は task snapshot と一致する input handles を受け取る。
2. Phase service は task-local mutable storage 内で output value and side tables を構築する。
3. `PhaseOutputPublisher` は work-unit identity、source-map coverage、declared output kind を検証する。
4. Publisher は canonical internal encoding、named input hashes、dependency output content hashes から content hash を計算する。
5. Output は `IrStorageService` に seal される。
6. Scheduler は `PhaseOutputRef<T>` を受け取り、dependent tasks へ渡す。

Dependent tasks は partially-built IR を観測しない。

### Cache Hit Rehydration

1. Scheduler は task cache key で `CacheStore` に record を問い合わせる。
2. `IrCacheAdapter` は schema versions、toolchain compatibility、dependency hashes、source hashes、policy fingerprints を検証する。
3. Adapter は current `BuildSnapshotId` 内に output handle を reconstruct する。
4. Reconstructed handle は `OutputStorageRef::CacheRecord` lineage と、cache record から rehydrate された side-table refs を記録する。
5. Downstream tasks は ordinary completed output と同様に consume する。

Proof-related outputs では、cache validation は accepted proof witnesses or deterministic discharge records も検証しなければならない。Cache hit 自体は proof authority ではない。

### Snapshot Replacement for Watch and LSP

1. Source edit は新しい `BuildSnapshotId` を作る。
2. Old snapshot の handles は、参照されている間 stale diagnostics and explanation requests のため readable のままでよい。
3. New tasks は cache validation または explicit unchanged-source checks を通してのみ old data を reuse してよい。
4. LSP bridge は diagnostic aggregation 後に current semantic snapshots を atomically に switch する。
5. Unreferenced old handles は collection 対象になる。

Open buffers は source snapshots であり published artifacts ではない。LSP features は data が current snapshot、stale snapshot、syntax-only fallback のどれから来たかを report しなければならない。

### Artifact Projection

1. `ArtifactProjectionService` は一つの module の sealed phase outputs を読む。
2. Stable public metadata を `VerifiedArtifactDraft` へ project する。
3. Internal ids を artifact ids、FQNs、labels、source ranges へ map する。
4. Raw IR storage refs は draft から drop する。
5. `ArtifactService` は phase 15 manifest transactions を通して artifact を serialize and publish する。

Artifact publication 後、downstream packages、documentation、LSP artifact readers は internal IR storage ではなく artifacts を consume する。

## API 概要

### IR Storage Service

```rust
trait IrStorageService {
    fn seal<T>(&self, output: SealOutputInput<T>) -> Result<PhaseOutputRef<T>, IrStorageError>;
    fn get<T>(&self, handle: &PhaseOutputRef<T>) -> Result<Arc<T>, IrStorageError>;
    fn side_tables(&self, handle: SideTableRef) -> Result<Arc<IrSideTables>, IrStorageError>;
    fn retain(&self, handle: PhaseOutputId, owner: RetainOwner) -> RetainGuard;
    fn collect(&self, snapshot: BuildSnapshotId) -> CollectionSummary;
}
```

### Snapshot Handle Registry

```rust
trait SnapshotHandleRegistry {
    fn create_snapshot(&self, input: SnapshotInput) -> BuildSnapshot;
    fn source_identity(&self, snapshot: BuildSnapshotId, source: SourcePath) -> Option<SourceVersion>;
    fn derive_output_id(&self, input: OutputIdentityInput) -> PhaseOutputId;
    fn is_current(&self, snapshot: BuildSnapshotId) -> bool;
}
```

### Phase Output Publisher

```rust
trait PhaseOutputPublisher {
    fn publish<T>(&self, task: TaskId, output: PhaseOutputDraft<T>) -> Result<PhaseOutputRef<T>, PublishError>;
}

struct PhaseOutputDraft<T> {
    phase: PipelinePhase,
    unit: WorkUnit,
    snapshot: BuildSnapshotId,
    value: T,
    side_tables: IrSideTables,
    dependencies: Vec<AnyPhaseOutputRef>,
}
```

Publisher は semantic `content_hash` を `value`、named input hashes、dependency `content_hash` values から導出する。diagnostics、explanation previews、source-map side tables、documentation attachment metadata からは別に `side_table_hash` を導出する。diagnostic wording、explanation preview、development-only side-table changes は、その phase が side tables を semantic inputs と明示しない限り semantic cache entries を invalidate してはならない。`PhaseOutputId`、`TaskId`、memory addresses、session-local storage ids をどちらの hash にも含めてはならない。

### IR Cache Adapter

```rust
trait IrCacheAdapter<T> {
    fn cache_key(&self, input: CacheKeyInput) -> CacheKey;
    fn encode(&self, output: &PhaseOutputRef<T>) -> Result<CacheRecordDraft, CacheEncodeError>;
    fn rehydrate(&self, record: CacheRecordRef, snapshot: BuildSnapshotId) -> Result<PhaseOutputRef<T>, CacheReuseError>;
}
```

### Artifact Projection Service

```rust
trait ArtifactProjectionService {
    fn project_module(&self, input: ArtifactProjectionInput) -> Result<VerifiedArtifactDraft, ProjectionError>;
}

struct ArtifactProjectionInput {
    module_path: ModulePath,
    snapshot: BuildSnapshotId,
    outputs: ModulePhaseOutputs,
    diagnostics: Vec<DiagnosticRecord>,
}
```

## エラー処理

### Storage Errors

| Condition | Handling |
|---|---|
| missing storage slot | treat as internal error or cache miss depending on origin |
| schema mismatch | reject cache reuse; rerun task if possible |
| corrupted internal blob | discard cache record and report cache integrity diagnostic when configured |
| requested type mismatch | internal API error; do not reinterpret bytes as another IR kind |

### Snapshot Errors

| Condition | Handling |
|---|---|
| handle from obsolete snapshot used as current | reject and request current-snapshot task or cache validation |
| dependency artifact hash mismatch | invalidate dependent handles |
| source hash mismatch | invalidate source-derived handles |
| missing source map | allow computation only if no diagnostics/artifact projection need it; otherwise fail task |

### Projection Errors

Projection failure blocks artifact commit for the affected module, but it does not mutate sealed IR outputs. The previous artifact manifest remains authoritative until phase 15 commits a valid replacement.

## 制約と前提

- Internal IR values は producing phase が publish した後 immutable である。
- Internal ids は artifact identities へ明示的に project されない限り `BuildSnapshotId` に scope される。
- Phase tasks は mutable IR objects ではなく handles を交換する。
- Phase output content hashes は snapshot-local ids ではなく content and named inputs に基づく。
- Cache records は optimization data であり proof authority ではない。
- Published artifacts は stable projections であり raw compiler-internal IR storage refs を含んではならない。
- Old snapshots は LSP or diagnostics から参照されている間 readable でよいが、current として report してはならない。
- Storage collection は published artifacts or source-level semantics を変更してはならない。
