# Crate Plan: mizar-parser

> Canonical language: English. Japanese companion:
> [../ja/00.crate_plan.md](../ja/00.crate_plan.md).

## Responsibility

`mizar-parser` consumes frontend tokens and emits parser-owned syntax events,
recovery diagnostics, and a `mizar-syntax` surface tree. It owns concrete
grammar recognition, source-order preservation, and parser recovery at grammar
boundaries.

It does not own name resolution, type checking, proof acceptance, semantic
evidence, CoreIr, ControlFlowIr, VC generation, or public diagnostic-code
allocation outside the existing parser diagnostic vocabulary.

Required parser behavior:

- Dispatch a property-shaped `definition` to the dedicated top-level producer
  before the generic `DefinitionBlockItem` producer. The bounded discriminator
  accepts a direct `property` start or an initial `let` region containing a
  top-level `property` before the outer `end`/next top-level boundary; it tracks
  delimiter and nested-block depth, so a missing parameter semicolon still
  selects the dedicated recovery path without matching `property` inside a
  nested proof. A `definition` with no `property` keyword is indistinguishable
  from an ordinary definition block and remains on the generic producer; Task
  48 does not promise specialized missing-`property` recovery. Tests pin the
  missing parameter semicolon path, bounded negative discriminator cases, and
  following-declaration preservation. Preserve any canonical top-level
  annotation prefix in the same source order and ownership style as other
  declaration producers.
- Produce exactly one specialized `DefinitionParameter` child for
  `let identifier be mode_application;`. It accepts one ordinary identifier,
  exact `be`, and a `mode_ref_name` represented by a `TypeHead` containing a
  qualified constructor symbol plus optional existing `TypeArguments`. It
  rejects multiple binders, `being`, `such`/`by` constraints, missing pieces,
  builtin `object`/`set`, attribute chains, and trailing parameter material.
  The generic definition-parameter parser remains unchanged.
- Preserve `property`, owner identifier, dot, property identifier, and the
  exact `means` or `equals` keyword in the owning node. `means` owns one
  `FormulaDefiniens`, then mandatory ordered `existence` and `uniqueness`
  `CorrectnessCondition` children, then optional `coherence`. `equals` owns one
  `TermDefiniens`, permits only optional `coherence`, and diagnoses
  `existence`/`uniqueness` as invalid. Both definiens branches reuse simple and
  case/otherwise parsing.
- Correctness conditions reuse the existing general justification grammar,
  including omitted simple justification, `by` references,
  `by computation(...)`, and `proof ... end`. Task 48 only recognizes and
  preserves those syntax shapes; it does not accept or discharge proofs.
- Preserve the outer `end;` on `PropertyImplementation`. When the bounded
  discriminator sees `property`, recover missing or malformed
  parameter/owner/property-name/dot/body/correctness/terminator pieces with the
  existing parser diagnostic and recovery vocabulary, source-order invalid
  tokens, and synchronization at correctness keywords, outer `end`, the next
  top-level item, or EOF. Recovery must not consume the following declaration.
