# Source/Spec Correspondence Audit: mizar-vc

> Canonical language: English. Japanese companion:
> [../ja/source_spec_audit.md](../ja/source_spec_audit.md).

Task 18 audits the current `mizar-vc` public surface and promised behavior after
the public-enum policy gate. It changes no source behavior, `.miz` fixtures,
expectations, public APIs, or language specifications. Remaining unavailable
behavior is recorded as explicit `external_dependency_gap`, `test_gap`, or
`deferred` work instead of treating current implementation limits as normative.

## Scope And Method

The inventory covers the public modules exported by
`crates/mizar-vc/src/lib.rs`, top-level public items in each module, and public
methods exposed by those public data types. Public newtypes generated by the
crate-local `dense_id!` and `string_key!` macros are grouped with `vc_ir`
because the module spec owns their constructors and accessors.

Module specifications audited:

- [vc_ir.md](./vc_ir.md)
- [generator.md](./generator.md)
- [discharge.md](./discharge.md)
- [dependency_slice.md](./dependency_slice.md)
- [todo.md](./todo.md)

Result: no unclassified `source_drift`, `design_drift`,
`source_undocumented_behavior`, `test_expectation_drift`,
`boundary_violation`, or `repo_metadata_conflict` is observed for the current
explicit-payload implementation. Remaining gaps are the known deferred or
external seams listed below.

## Remaining Classified Follow-Ups

No new source/spec correspondence gap is introduced by Task 18. Existing
classified records remain:

- `external_dependency_gap`: active `proof_verification` runner support and
  source-to-core / source-to-VC extraction seams are absent from `mizar-test`;
  Task 15 records concrete deferred corpus obligations.
- `external_dependency_gap`: `mizar-atp`, `mizar-kernel`, `mizar-proof`, and
  `mizar-cache` are not active workspace consumers, so ATP translation,
  certificate acceptance, proof policy, cache lookup/reuse, and artifact
  persistence remain outside this crate.
- `external_dependency_gap`: upstream explicit payloads are still incomplete
  for some registration/redefinition/reduction details, call-precondition,
  branch, match, range-loop, collection-loop, term-only termination, partial
  termination, Pick non-emptiness, ghost-erasure, and complete trace families.
- `deferred`: Task 20 owns `ObligationAnchor` and cross-edit reuse identity
  follow-through against architecture 22 once canonical VC/context identity and
  consumer policy seams are in place.
- `deferred`: Task 21 records the architecture-22 follow-up audit, Task 22
  records the module-boundary refactor gate, and closeout records final quality
  review and crate-exit status.

No `repo_metadata_conflict` was observed.
