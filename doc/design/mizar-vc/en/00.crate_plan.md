## Responsibility

Owned behavior:

- implement pipeline phases 11-12: VC generation and deterministic pre-ATP
  discharge;
- consume `mizar_core::core_ir::CoreIr`,
  `mizar_core::control_flow::ControlFlowIr`, and the core obligation-seed
  handoff;
- own prover-independent `VcIr`, deterministic `VcId` assignment,
  `ObligationAnchor` construction, canonical VC fingerprints, and local-context
  fingerprints;
- generate theorem, proof-step, definition-correctness,
  registration/redefinition/reduction correctness, checker-initial, algorithm
  contract, assertion, invariant, ghost-erasure, and termination VCs from
  explicit core/control-flow inputs;
- preserve symbolic premise references, proof hints, source/core provenance,
  local proof/program paths, policy statuses, and stable diagnostic ordering;
- perform deterministic pre-ATP discharge with replayable evidence and stable
  computation limits;
- compute conservative per-VC dependency slices and fingerprints that can feed
  cache miss/reuse decisions without hiding incomplete dependency data;
- produce prover-independent kernel evidence handoff packages from existing
  `VcIr` formula, substitution, provenance, and target-binding data once task
  25 implements the builder, without treating the package as acceptance.

Out of scope:

- source loading, lexing, parsing, name resolution, type checking,
  registration activation, overload resolution, elaboration, and control-flow
  construction;
- changing `doc/spec`, existing `.miz` fixtures, or expectation sidecars merely
  to match current implementation behavior;
- ATP encoding, backend process execution, portfolio scheduling, certificate
  validation, kernel proof acceptance, SAT checking, artifact publication,
  cache storage, and proof-reuse validation;
- assigning user-facing public diagnostic codes before the diagnostics registry
  owns that policy;
- promoting deterministic discharge, ATP success, policy-open, assumed, or
  externally attested states to kernel-verified proof status.

Boundary rules:

- `mizar-vc` assigns `VcId`s only within the current build snapshot; `VcId` is
  never cross-edit proof-reuse identity;
- `ObligationAnchor` is a reuse-candidate key, not proof evidence, and reuse
  additionally requires canonical VC, context, dependency-slice, policy, and
  witness/discharge hashes;
- every ATP-bound VC remains present with full local context and `NeedsAtp`
  status;
- unsupported or incomplete dependency information is represented
  conservatively so downstream cache consumers can force misses;
- this crate may produce untrusted deterministic evidence, but trusted
  acceptance remains owned by later proof/kernel policy.
