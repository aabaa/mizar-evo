# Crate Plan: mizar-atp

> Canonical language: English. Japanese companion:
> [../ja/00.crate_plan.md](../ja/00.crate_plan.md).

## Responsibility

Owned behavior:

- phase 13 ATP candidate production for VC obligations with
  `VcStatus::NeedsAtp` after deterministic discharge;
- backend-neutral `AtpProblem` data, deterministic encodings, backend process
  execution, and portfolio candidate collection;
- provenance that ties every backend-visible formula to VC/kernel evidence
  inputs;
- untrusted formula/substitution evidence candidates for later kernel checking;
- diagnostic and reproducibility metadata for backend runs.

Out of scope:

- trusted proof acceptance, kernel evidence checking, SAT checking, and proof
  policy decisions;
- winner selection, proof-status projection, witness publication, and proof
  reuse policy, which belong to downstream proof/cache/artifact integration;
- premise invention, substitution invention, overload resolution, cluster
  search, implicit coercion insertion, or fallback inference;
- treating backend proof methods, resolution traces, SMT proof objects, backend
  logs, or instantiated formulas as trusted acceptance material.

## Exit Criteria

Hard gates:

- no semantic ATP source module exists without a paired English/Japanese module
  spec;
- all crate-owned behavior is documented in module specs, `doc/spec`, tests, or
  traceability metadata before implementation;
- generated evidence remains untrusted until `mizar-kernel` checks it;
- backend proof methods, logs, resolution traces, instantiated formulas, and SAT
  problems never become trusted acceptance material;
- `mizar-atp` does not own proof policy, winner selection, proof witness
  publication, or cache promotion;
- each completed task has its own commit;
- remaining gaps are classified as `deferred`, `external_dependency_gap`, or
  human-owned.

Verification commands:

- For every Rust source task:
  - `cargo test -p mizar-atp`
  - `cargo clippy -p mizar-atp --all-targets -- -D warnings`
  - `cargo fmt --check`
- For tasks that touch VC/kernel boundaries:
  - `cargo test -p mizar-vc`
  - `cargo test -p mizar-kernel`
- For broad crate completion:
  - `cargo clippy --all-targets --all-features -- -D warnings`
  - `cargo test`

Review expectations:

- review-only agent for implementation specification and documentation;
- review-only agent for test sufficiency;
- review-only agent for full implementation;
- review-only agent for source/documentation consistency.
