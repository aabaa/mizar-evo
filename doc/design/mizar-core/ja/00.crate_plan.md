## 責務

所有する挙動:

- `mizar_checker::resolved_typed_ast::ResolvedTypedAst` から backend-neutral
  `CoreIr` への pipeline phase 9 elaboration。
- core algorithm shell から `ControlFlowIr` への pipeline phase 10 control-flow
  preparation。
- core item、term、formula、definition、proof skeleton、algorithm shell、
  source map、provenance、obligation seed の data shape。
- core term / formula の binder normalization、alpha equivalence、
  capture-avoiding substitution、決定的な canonical form。
- soft type を明示的な type predicate、assumption、view metadata、
  obligation seed を通じてのみ消去すること。
- 失敗した semantic site を valid core term に捏造せず、明示的 error node または
  skipped item として保持すること。
- generated choice / comprehension symbol、local abbreviation expansion、
  proof thesis tracking、source/core provenance、debug rendering、CFG block
  ordering を決定的に保つこと。
- obligation seed のみを生成すること。具体的な `VcId` と cross-edit
  `ObligationAnchor` は `mizar-vc` の責務。

対象外:

- lexical、syntax、parser、frontend、resolver、checker、artifact、cache、ATP、
  kernel、proof policy、build、diagnostics registry、LSP、documentation の core
  handoff 外の責務。
- 欠けている checker payload を補うための source-to-checker payload extraction や
  raw syntax walking。
- name/type/registration/cluster/overload/proof search。
- registration activation、proof acceptance、kernel certificate replay、VC
  generation、ATP encoding、artifact schema emission、cache/proof reuse。
- `mizar-diagnostics` の stable registry より前の public diagnostic code 割り当て。
- 未完成の `mizar-vc`、`mizar-kernel`、`mizar-proof` に対する仮 downstream schema。

境界規則:

- semantic input は checker/resolver/session-owned id と explicit payload table
  を通じて届かなければならず、source text の再 parse / 再 scan はしない。
- core logical node は raw source spelling ではなく canonical symbol / binder
  identity を使う。
- 消去される type annotation は、predicate、assumption、view explanation、
  diagnostic、または obligation seed を必ず残す。
- definition unfolding は明示的で、boundary を保存する。body があるという理由だけで
  elaboration が eager inline してはならない。
- algorithm shell lowering は source order を保存し、CFG analysis は phase 10 で
  初めて始める。
- malformed substitution evidence は reject または error として保持し、
  heuristic repair はしない。
- backend-specific encoding と proof acceptance は downstream crate に deferred。

## exit criteria

hard gates:

- blocking/high specification inconsistency が残っていない。
- 実装済み behavior に必要な module spec が英日で存在し、source と一致している。
- source behavior が `doc/spec`、test、module spec、または explicit deferred row なしで
  残っていない。
- milestone-owned behavior に Rust test、active `.miz` test、snapshot test、または
  explicit deferred reason がある。
- test expectation を implementation に合わせる目的だけで rebaseline していない。
- boundary discipline を守る。proof search、VC generation、artifact schema fabrication、
  source-to-checker extraction、kernel checking が `mizar-core` に現れない。
- すべての task が exactly one task commit と ledger entry を持つ。
- 各 task の required verification が commit 前に通過している。
- final broad workspace verification が通過する。未実行 command があれば blocker と
  risk statement を明示する。
- read-only crate quality review の valid score が 90/100 以上。

verification commands:

- docs-only/spec-only/audit task: 少なくとも stage 前の `git diff --check` と、
  task path を明示 stage した後の `git diff --cached --check`。これにより新規 file
  も commit 前に検査される。
- Rust source task: 少なくとも `cargo test -p mizar-core`、
  `cargo clippy -p mizar-core --all-targets -- -D warnings`、
  `cargo fmt --check`。
- checker-boundary / corpus task: 触れた surface に応じて
  `cargo test -p mizar-checker` または `cargo test -p mizar-test` も実行する。
- final closeout: `cargo fmt --check`、
  `cargo clippy --all-targets --all-features -- -D warnings`、`cargo test`。

review expectations:

- implementation-spec/documentation review、test sufficiency review、full
  implementation review、source/documentation consistency review、final quality
  review に review-only agent を使う。
- blocking/high finding は commit 前に修正する。
- medium finding は修正するか、理由付きで明示 deferred とする。
- low note は ledger または exit report に記録した場合のみ残してよい。
