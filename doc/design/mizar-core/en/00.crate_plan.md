## Responsibility

Owned behavior:

- implement pipeline phase 9 elaboration from
  `mizar_checker::resolved_typed_ast::ResolvedTypedAst` into backend-neutral
  `CoreIr`;
- implement pipeline phase 10 control-flow preparation from core algorithm
  shells into `ControlFlowIr`;
- own core item, term, formula, definition, proof-skeleton, algorithm-shell,
  source-map, provenance, and obligation-seed data shapes;
- own binder normalization, alpha-equivalence, capture-avoiding substitution,
  and deterministic canonical forms for core terms and formulas;
- erase soft types only through explicit type predicates, assumptions, view
  metadata, and obligation seeds;
- preserve failed semantic sites as explicit error nodes or skipped items
  instead of fabricating valid core terms;
- keep generated choice/comprehension symbols, local abbreviation expansion,
  proof thesis tracking, source/core provenance, debug renderings, and CFG
  block ordering deterministic;
- produce obligation seeds only. Concrete `VcId`s and cross-edit
  `ObligationAnchor`s are owned by `mizar-vc`.

Out of scope:

- lexical, syntax, parser, frontend, resolver, checker, artifact, cache, ATP,
  kernel, proof-policy, build, diagnostics-registry, LSP, or documentation
  ownership outside the core handoff;
- source-to-checker payload extraction or raw syntax walking to fill missing
  checker payloads;
- name, type, registration, cluster, overload, or proof search;
- registration activation, proof acceptance, kernel certificate replay, VC
  generation, ATP encoding, artifact schema emission, and cache/proof reuse;
- assigning public diagnostic codes before `mizar-diagnostics` owns a stable
  registry;
- inventing placeholder downstream schemas for unfinished `mizar-vc`,
  `mizar-kernel`, or `mizar-proof` work.

Boundary rules:

- all semantic input must arrive through checker/resolver/session-owned ids and
  explicit payload tables, never by reparsing or rescanning source text;
- core logical nodes use canonical symbol and binder identities, not raw source
  spellings;
- every erased type annotation must leave an explicit predicate, assumption,
  view explanation, diagnostic, or obligation seed;
- definition unfolding is explicit and boundary-preserving. Elaboration must
  not eagerly inline definitions merely because bodies are available;
- algorithm shell lowering remains source-order preserving. CFG analysis starts
  only in phase 10;
- malformed substitution evidence is rejected or preserved as an error, not
  repaired heuristically;
- backend-specific encoding and proof acceptance remain deferred to downstream
  crates.

## Exit Criteria

Hard gates:

- no blocking/high specification inconsistency remains;
- all module specs needed by implemented behavior exist in English and
  Japanese and agree with source;
- no source behavior remains undocumented by `doc/spec`, tests, module specs,
  or explicit deferred rows;
- milestone-owned behavior has Rust tests, active `.miz` tests, snapshot tests,
  or explicit deferred reasons;
- test expectations are not rebaselined merely to match implementation;
- boundary discipline is preserved: no proof search, VC generation, artifact
  schema fabrication, source-to-checker extraction, or kernel checking appears
  inside `mizar-core`;
- every task has exactly one task commit and a ledger entry with commit hash,
  reviews, verification, and deferred reasons;
- required verification passes for each task before commit;
- final broad workspace verification passes, or any unrun command has an
  explicit blocker and risk statement;
- read-only crate quality review assigns a valid score of at least 90/100.

Verification commands:

- docs-only/spec-only/audit tasks: at least `git diff --check` before staging
  and `git diff --cached --check` after explicitly staging the task paths, so
  newly added files are checked before commit;
- Rust source tasks: at least `cargo test -p mizar-core`,
  `cargo clippy -p mizar-core --all-targets -- -D warnings`, and
  `cargo fmt --check`;
- checker-boundary or corpus tasks: additionally run relevant
  `cargo test -p mizar-checker` or `cargo test -p mizar-test` when the task
  touches those surfaces;
- final closeout: `cargo fmt --check`,
  `cargo clippy --all-targets --all-features -- -D warnings`, and
  `cargo test`.

Review expectations:

- use review-only agents for implementation-spec/documentation review, test
  sufficiency review, full implementation review, source/documentation
  consistency review, and final quality review;
- blocking/high findings must be fixed before commit;
- medium findings must be fixed or explicitly deferred with a reason;
- residual low notes may remain only if documented in the ledger or exit
  report.
