# Crate Plan: mizar-build

> Canonical language: English. Japanese companion:
> [../ja/00.crate_plan.md](../ja/00.crate_plan.md).

## Responsibility

Owned behavior:

- pipeline phase 0 workspace planning: package/workspace manifests, lockfile
  validation, dependency graph resolution, and deterministic `BuildPlan`
  production;
- phase-0 module-index construction from `BuildPlan` plus source layout or
  dependency artifact summaries;
- the versioned task graph used by parallel verification and incremental
  invalidation;
- scheduling, deterministic result collation, resource budgets, cooperative
  cancellation, bounded failure propagation, and deterministic artifact commit
  ordering;
- a cache-aware scheduling seam that can consume `mizar-cache` decisions before
  execution while preserving clean-build equivalence.

Out of scope:

- build requests, CLI/watch/LSP entry points, build sessions, event streams,
  phase-service registry, and the `salsa` query/database boundary, all of which
  are owned by `mizar-driver`;
- raw IR storage, output handles, and snapshot-handle rehydration, which remain
  `mizar-ir` work;
- `mizar-cache` `CacheKey` construction, dependency fingerprint construction,
  cache-store compatibility checks, or proof-reuse validation;
- artifact schema ownership, manifest transaction internals, proof witness
  publication tokens, and producer-owned artifact projection;
- proof authority, proof search, ATP/backend winner selection, kernel
  acceptance, or proof-status promotion.

`mizar-build` may skip task execution after a validated cache hit, but the hit
is an optimization only. It must never become semantic acceptance, proof
authority, or a reason to upgrade a task, artifact record, backend result, or
externally attested proof to a more trusted status.

`mizar-build` must not depend on `mizar-driver`. The dependency direction is
driver-to-build.

## Boundary Invariants

| ID | Boundary | Rule |
|---|---|---|
| BUILD-I-001 | cache/proof authority | Scheduling or cache reuse would be unsound if treated as proof authority. Every scheduler/cache/commit task must preserve clean-build equivalence and non-promotion of trust. |
| BUILD-I-002 | driver/build dependency | Internal 01 assigns build requests and registry ownership to `mizar-driver`. `mizar-build` must not depend on `mizar-driver`; any reverse dependency is a boundary violation. |

## Exit Criteria

Hard gates:

- no blocking/high specification inconsistency remains;
- no `mizar-build` source behavior remains absent from its module specs or
  recorded external gap list;
- source, tests, English docs, and Japanese companion docs are synchronized;
- every unavailable downstream integration is classified as
  `external_dependency_gap` rather than stubbed;
- the crate has no dependency on `mizar-driver`;
- cache-aware scheduling is documented and implemented as an optimization-only
  execution-skip seam;
- cache hits and artifact records never promote proof trust or semantic
  acceptance;
- task graph, scheduler, resource budget, cancellation, failure propagation,
  deterministic commit boundary, and cache seam behavior are covered by focused
  tests or explicit deferred reasons;
- review-only spec/test/full/source-doc reviews report no unresolved
  blocking/high findings;
- required verification commands pass, or any unrun command is explicitly
  justified.

Verification commands:

```text
cargo test -p mizar-build
cargo clippy -p mizar-build --all-targets -- -D warnings
cargo fmt --check
```

When scheduler/cache/artifact/commit boundaries are touched, also run the
available adjacent crate checks:

```text
cargo test -p mizar-cache
cargo test -p mizar-artifact
cargo test -p mizar-vc
cargo test -p mizar-proof
cargo test -p mizar-driver
```

For crate completion, run:

```text
cargo fmt --check
cargo clippy --all-targets --all-features -- -D warnings
cargo test
```

Review expectations:

- every task follows the AGENTS.md review loop for spec/documentation, test
  sufficiency, full implementation, and source/documentation consistency;
- remaining low/medium notes must be recorded with a concrete deferred reason
  or fixed before closeout;
- crate completion requires a read-only quality score of at least 90/100.
