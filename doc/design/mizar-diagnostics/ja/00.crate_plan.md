## 責務

所有する挙動:

- CLI、artifact、LSP、tool が利用する安定した `DiagnosticCode` identity と
  registry metadata。
- 割り当て済みおよび retired diagnostic code の互換性検証。
- `SourceId`、`SourceRange`、snapshot freshness、failure category、
  structured detail、fix suggestion、lazy explanation handle を持つ
  structured diagnostic draft と immutable diagnostic record。
- CLI formatting、LSP payload、phase ownership を持たない producer sink の
  draft collection。
- 正規化、重複排除、canonical ordering、snapshot-scoped deterministic
  diagnostic id、stale snapshot suppression を含む immutable
  `BuildDiagnosticIndex` への deterministic aggregation。
- diagnostic record と code metadata からの projection としての
  deterministic CLI rendering。
- consumer が help text、LSP action、agent patch proposal に射影できる
  structured fix suggestion payload。ただし validation は consumer 側の責務。
- lazy explanation handle、bounded preview、fail-closed lookup status。

範囲外:

- syntax、static semantics、proof semantics、type behavior、name resolution、
  overload resolution、parser recovery、proof acceptance、trusted status、
  kernel acceptance、verifier policy、ATP search、artifact mutation、build
  scheduling、driver session orchestration、LSP protocol conversion、
  open-buffer document synchronization、artifact publication authority。
- real consumer adoption seam が存在する前の既存 lexer/frontend/parser/resolver
  diagnostics の migration。
- placeholder adapter、fake LSP record、stub driver event、invented resolver
  diagnostic code、provisional artifact projection API。

Tool と consumer は `DiagnosticCode` をキーにし、diagnostic message text、
render text、localized text、collection order、`DiagnosticId` を identity として
扱わない。message は改善してよいが、code は恒久的であり、異なる意味に
再利用してはならない。

## Exit Criteria

Hard gates:

- blocking/high specification inconsistency が残っていない。
- crate-owned public API と behavior は EN/JA design docs と tests に表現されるか、
  明示的に deferred されている。
- crate-owned surface に undocumented source behavior が残っていない。
- `DiagnosticCode` が stable identity であり、message text が authority ではない。
- diagnostics は snapshot freshness に言及し、stale-snapshot diagnostics は
  current として publish されない。
- producer sink は draft を保存し、CLI output や LSP payload を format しない。
- aggregation は deterministic、order-independent、publication 後 immutable。
- render、fix、explain は records からの projection であり、proof、driver、
  LSP、cache、artifact authority を持たない。
- real adoption seam なしに既存 lexer/frontend/parser/resolver diagnostics を
  migrate しない。
- required verification が通るか、未実行 command が正当化されている。
- remaining risks は protocol gap class を使い、deferred、out of scope、
  external dependency gap、または report-only metadata conflict として
  disposition されている。

Verification commands:

```text
cargo test -p mizar-diagnostics
cargo clippy -p mizar-diagnostics --all-targets -- -D warnings
cargo fmt --check
git diff --check
git diff --cached --check
cargo clippy --all-targets --all-features -- -D warnings
cargo test
```

Consumer-boundary commands は implemented consumer seam に触れる task のみで実行する:

```text
cargo test -p mizar-resolve
cargo test -p mizar-frontend
cargo test -p mizar-parser
cargo test -p mizar-lexer
cargo test -p mizar-lsp
cargo test -p mizar-build
```

将来の task が real `mizar-driver` seam に触れ、かつその crate が存在する場合は、
対応する driver verification も実行する。名前が挙がっている seam が存在しない場合は
external dependency gap として記録し、placeholder crate/API は追加しない。

Review expectations:

- 各 task は specification/doc consistency、test sufficiency、full
  implementation または docs-only substance、source/documentation synchronization
  について review-only agent coverage を受ける。
- すべての hard gates が通った後、closeout を complete と扱う前に read-only
  quality review が 90/100 以上の score を付ける。
