# Crate Plan: mizar-diagnostics

> Canonical language: English. Japanese companion:
> [../ja/00.crate_plan.md](../ja/00.crate_plan.md).

## Responsibility

Owned behavior:

- stable `DiagnosticCode` identity and the registry metadata used by CLI,
  artifacts, LSP, and tools;
- compatibility validation for allocated and retired diagnostic codes;
- structured diagnostic drafts and immutable diagnostic records with
  `SourceId`, `SourceRange`, snapshot freshness, failure categories,
  structured details, fix suggestions, and lazy explanation handles;
- producer sink collection of drafts without CLI formatting, LSP payloads, or
  phase ownership;
- deterministic aggregation into immutable `BuildDiagnosticIndex` values,
  including normalization, deduplication, canonical ordering, stable
  snapshot-scoped diagnostic ids, and stale-snapshot suppression;
- deterministic CLI rendering as a projection from diagnostic records and code
  metadata;
- structured fix-suggestion payloads that consumers may project into help text,
  LSP actions, or agent patch proposals after their own validation;
- lazy explanation handles, bounded previews, and fail-closed lookup status.

Out of scope:

- syntax, static semantics, proof semantics, type behavior, name resolution,
  overload resolution, parser recovery, proof acceptance, trusted status,
  kernel acceptance, verifier policy, ATP search, artifact mutation, build
  scheduling, driver session orchestration, LSP protocol conversion, open-buffer
  document synchronization, or artifact publication authority;
- migration of existing lexer/frontend/parser/resolver diagnostics before a
  real consumer adoption seam exists;
- placeholder adapters, fake LSP records, stub driver events, invented resolver
  diagnostic codes, or provisional artifact projection APIs.

Tools and consumers must key on `DiagnosticCode`, never on diagnostic message
text, render text, localized text, collection order, or `DiagnosticId`.
Messages may evolve; codes are permanent and are never reused for different
meanings.

## Exit Criteria

Hard gates:

- no blocking/high specification inconsistency remains;
- every crate-owned public API and behavior is represented in EN/JA design docs
  and tests, or explicitly deferred;
- no source behavior remains undocumented for the crate-owned surface;
- `DiagnosticCode` remains the stable identity and message text is not an
  authority;
- diagnostics mention snapshot freshness and stale-snapshot diagnostics are not
  published as current;
- producer sinks preserve drafts and do not format CLI output or LSP payloads;
- aggregation is deterministic, order-independent, and immutable after
  publication;
- render, fix, and explain are projections from records and do not own proof,
  driver, LSP, cache, or artifact authority;
- existing lexer/frontend/parser/resolver diagnostics are not migrated without a
  real adoption seam;
- required verification passes, or unrun commands are justified;
- remaining risks use protocol gap classes and are dispositioned as deferred,
  out of scope, external dependency gaps, or report-only metadata conflicts.

Verification commands:

```text
cargo test -p mizar-diagnostics
cargo clippy -p mizar-diagnostics --all-targets -- -D warnings
cargo fmt --check
git diff --check
git diff --cached --check
cargo clippy --all-targets --all-features -- -D warnings

```

Consumer-boundary commands are run only when the task touches an implemented
consumer seam:

```text

cargo test -p mizar-frontend
cargo test -p mizar-parser
cargo test -p mizar-lexer
cargo test -p mizar-lsp

```

If a future task touches a real `mizar-driver` seam after that crate exists, run
the corresponding driver verification too. If a named seam is absent, record the
external dependency gap and do not add a placeholder crate or API.

Review expectations:

- each task receives review-only agent coverage for specification/doc
  consistency, test sufficiency, full implementation or docs-only substance,
  and source/documentation synchronization;
- after all hard gates pass, a read-only quality review assigns a score of at
  least 90/100 before closeout is treated as complete.
